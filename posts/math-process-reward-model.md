## 核心结论

过程奖励模型，PRM，指“不给整条解法只打一个总分，而是给推理过程中的每一步打分”的奖励模型。它解决的核心问题不是“答案有没有对”，而是“这条推理路径靠不靠谱”。

对数学推理来说，这个区别很关键。结果奖励模型，ORM，只看最终答案是否正确。这样会出现一种常见情况：前面几步已经错了，但后面碰巧把答案猜对，ORM 仍然可能给高分。PRM 用步骤级监督把这种“偶然对”与“过程可靠”区分开。

OpenAI 在论文《Let’s Verify Step by Step》中展示了这件事的工程价值。PRM800K 数据集提供了约 80 万条步骤级正确性标签；过滤后的训练数据大约覆盖 7.5 万条解题轨迹，来自约 1.2 万道 MATH 题。基于这类数据训练出的 PRM，在 Best-of-N 采样里，比只看最终答案的 ORM 更会挑路径。在论文的大规模验证中，PRM 选择器在代表性 MATH 测试集上把准确率从 ORM 的 72.4% 提升到 78.2%。

新手先抓住一个最直观的结论：如果模型一次会写出多条解法，PRM 不是问“哪条最后答案像对的”，而是问“哪条从头到尾每一步都更像对的”。在多步推理里，后者通常更稳。

| 维度 | ORM | PRM |
| --- | --- | --- |
| 监督信号 | 只看最终答案 | 看每一步是否合理 |
| 能否发现中途错误 | 弱 | 强 |
| 对“蒙对答案”的容忍 | 高 | 低 |
| 适合场景 | 标签稀缺、快速上线 | 多步推理、可收集步骤标签 |
| 典型用途 | 最终结果打分 | 路径筛选、过程校验、训练反馈 |

## 问题定义与边界

这里的问题可以写得很精确：

给定一道数学题，模型会生成一条由多个步骤组成的解题路径
$$
\tau = (x_1, x_2, \dots, x_T)
$$
我们希望训练一个评分器，判断这条路径是否值得保留。难点在于，路径的可靠性不只体现在最后一句，而体现在中间每一步是否延续了正确推理。

这件事的边界也很明确。

第一，PRM 依赖步骤级标签。步骤级标签就是“老师批改草稿”式的数据：第 3 步对不对，第 4 步是否有实质推进，第 5 步是否已经犯了致命错误。没有这类标签，就很难训练出真正的 PRM。

第二，步骤切分必须清楚。“一步”不是任意长度的文本块，而是一个可判断对错的推理单元。切得太粗，一个错误会淹没在长段文字里；切得太细，标注成本会迅速上升。

第三，PRM 最适合“错误会沿路径传播”的任务。数学证明、方程推导、程序合成这类任务，一步错后面常常全错，所以过程监督价值高。对于短问答、事实检索这类“中间步骤不稳定或不显式”的任务，PRM 的优势会缩小。

可以用一个玩具例子理解。

题目：解方程 $2x+3=11$。

路径 A：
1. $2x=8$
2. $x=4$

路径 B：
1. $2x=11-2$
2. $x=4$

两条路径最终答案都写成 4，但路径 B 第一步已经错了，因为 $11-3$ 才是 8。ORM 只看最后答案时，可能把 A 和 B 都当成好解法；PRM 会在第 1 步就把 B 压低。

PRM800K 的价值就在这里。它不是只记录“这道题最后答没答对”，而是记录“这条轨迹中的哪些步骤正确、哪些步骤无进展、哪些步骤错误”。这种标签密度直接决定了 PRM 能不能学到“错误在何处开始出现”。

| 数据项 | 规模 | 作用 |
| --- | --- | --- |
| 题目数 | 约 12,000 | 提供问题分布 |
| 解题轨迹数 | 约 75,000（过滤后） | 提供多条候选路径 |
| 步骤级标签 | 约 800,000（过滤后） | 提供密集监督 |
| 原始未过滤标签 | 超过 100 万 | 含质检与无效样本 |

边界条件也要说清：如果你的业务里拿不到步骤级标签，或者步骤本身不可观察，那 PRM 不是不能做，而是很容易退化成“伪过程评分”，最后又回到只看最终结果。

## 核心机制与推导

PRM 的基本形式可以理解成一个步骤分类器或步骤排序器。

对每一步 $x_i$，模型输出一个分数 $s_i$，表示“这一步是合理推进”的概率或置信度。于是，一条路径就得到一个步骤分数序列：
$$
(s_1, s_2, \dots, s_T)
$$

接下来要做的是路径聚合，也就是把步骤分数合成总分 $S(\tau)$。常见做法有三类：

$$
S_{\min}(\tau) = \min_i s_i
$$

$$
S_{\prod}(\tau) = \prod_{i=1}^{T} s_i
$$

$$
S_{\text{sum}}(\tau) = \sum_{i=1}^{T} s_i
$$

它们的含义不同。

- `min` 表示“短板原则”。只要有一步很差，整条路径就被拉低。
- `prod` 表示“全链路可信度”。每一步都要过关，路径越长越难维持高分。
- `sum` 表示“累计质量”。适合长度较接近、希望保留整体贡献时使用。

还是看一个玩具例子。

路径 A 的步骤分数是 `[0.9, 0.8]`，路径 B 是 `[0.7, 0.95]`。

则：
- $S_{\min}(A)=0.8$
- $S_{\min}(B)=0.7$

所以按 `min` 聚合时选 A。原因很直白：A 两步都还不错，B 虽然最后一步高，但前面有明显短板。

如果按乘积：
- $S_{\prod}(A)=0.72$
- $S_{\prod}(B)=0.665$

仍然选 A。

这个例子体现了 PRM 的核心倾向：优先选择“整条链更平滑、更少硬伤”的路径，而不是“局部看起来很像对的路径”。

| 聚合方式 | 公式 | 优点 | 风险 | 适用场景 |
| --- | --- | --- | --- | --- |
| `min` | $\min_i s_i$ | 能快速惩罚早期错误 | 对单步噪声敏感 | 数学证明、严谨推导 |
| `prod` | $\prod_i s_i$ | 符合链式可靠性直觉 | 长路径天然吃亏 | 步长较稳定的任务 |
| `sum` | $\sum_i s_i$ | 简单、平滑 | 可能掩盖局部严重错误 | 需要容忍探索步骤时 |

PRM 常与 Best-of-N 联合使用。Best-of-N 的意思是“先生成 N 条候选路径，再选最好的一条”。如果单条候选中存在好路径的概率是 $p$，那么至少出现一条好路径的概率是
$$
1-(1-p)^N
$$
这不是 PRM 独有的公式，但 PRM 的价值在于：当好路径真的出现在候选集里时，它更有能力把那条路径挑出来。

例如，若单条候选里含有可接受解法的概率是 $p=0.2$：
- $N=1$ 时，成功率是 0.2
- $N=5$ 时，成功率是 $1-0.8^5 \approx 0.672$
- $N=10$ 时，成功率是 $1-0.8^{10} \approx 0.893$

这说明采样数增加会放大“挑选器”的价值。ORM 和 PRM 都能当挑选器，但 PRM 更擅长识别推理链内部的裂缝，因此在大样本候选里收益更明显。

真实工程例子是论文里的 MATH 评估。系统先为每道题生成大量候选解法，再让不同评分器做 rerank，意思是“重排序并选最优路径”。在这种设置下，PRM 比 ORM 更稳定地把高质量推理链排到前面，于是最终准确率从 72.4% 提升到 78.2%。

## 代码实现

工程上可以把 PRM 看成两个阶段：

1. 训练阶段：学习“某一步是否正确或是否有推进”。
2. 推理阶段：对多条候选路径逐步打分并聚合排序。

下面给一个可运行的简化版 Python 示例。它不依赖深度学习框架，只模拟 PRM 的核心逻辑：步骤打分、路径聚合、Best-of-N 选择。

```python
from math import prod

def aggregate(scores, method="min"):
    assert scores, "scores cannot be empty"
    if method == "min":
        return min(scores)
    if method == "prod":
        return prod(scores)
    if method == "sum":
        return sum(scores)
    raise ValueError(f"unknown method: {method}")

def select_best_path(candidates, method="min"):
    scored = []
    for path in candidates:
        total = aggregate(path["step_scores"], method=method)
        scored.append((total, path["name"], path))
    scored.sort(reverse=True)
    return scored[0]

# 玩具例子
candidates = [
    {"name": "A", "step_scores": [0.9, 0.8], "final_answer_correct": True},
    {"name": "B", "step_scores": [0.7, 0.95], "final_answer_correct": True},
    {"name": "C", "step_scores": [0.95, 0.2], "final_answer_correct": False},
]

best_min = select_best_path(candidates, method="min")
best_prod = select_best_path(candidates, method="prod")

assert best_min[1] == "A"
assert best_prod[1] == "A"
assert aggregate([0.9, 0.8], "min") == 0.8
assert round(aggregate([0.9, 0.8], "prod"), 2) == 0.72

print("best by min:", best_min)
print("best by prod:", best_prod)
```

如果上升到真实训练管线，伪代码大致是下面这样：

```python
# train
for sample in dataset:
    # sample.steps: ["step1", "step2", ...]
    # sample.labels: [1, 1, 0, -1, ...]
    for step_text, label in zip(sample.steps, sample.labels):
        logits = prm(step_text)
        loss = step_loss(logits, label)
        loss.backward()
        optimizer.step()
        optimizer.zero_grad()

# inference
candidates = generator.sample_n(problem, n=64, temperature=0.8)
best_score = -1e9
best_path = None

for path in candidates:
    step_scores = []
    for step in path.steps:
        score = prm(step)
        step_scores.append(score)
    total = aggregate(step_scores, method="min")
    if total > best_score:
        best_score = total
        best_path = path
```

真实工程里有两个实现点很重要。

第一，缓存中间表示。中间表示就是模型处理某一步后得到的隐状态或编码结果。因为同一条路径可能被不同聚合器、不同阈值反复评估，不缓存会重复算很多次。

第二，步骤切分要稳定。训练和推理必须使用近似一致的 step segmentation，意思是“如何把长解法切成一步一步”。训练时按句子切，推理时按换行切，PRM 学到的分布就会错位。

| 工程模块 | 需要做什么 | 原因 |
| --- | --- | --- |
| 数据读取 | 读取题目、步骤、标签 | 构造 step-wise 样本 |
| 步骤切分 | 统一切分规则 | 保证训练推理分布一致 |
| 表示缓存 | 缓存 step embedding/logit | 降低重复评分成本 |
| 聚合策略 | `min/prod/sum` 或混合 | 决定排序行为 |
| 候选生成 | 采样 N 条路径 | 给 PRM 提供可筛选集合 |

## 工程权衡与常见坑

PRM 的主要成本不是“训练一个分类器”，而是“让整条推理流水线都围绕步骤级评价运转”。这会带来几类现实权衡。

第一类是计算量。Best-of-N 本身就要生成 N 条路径，PRM 还要对每条路径的每一步打分。若平均每条路径有 $T$ 步，则额外评分成本大约与 $N \times T$ 成正比。N 很大时，推理费用会上升得很快。

第二类是标签成本。最终答案标签通常可通过自动判题得到，步骤标签往往需要人工或高质量教师模型。人工标注 1 个最终答案，比人工判 10 个中间步骤便宜得多。

第三类是退化风险。一个常见失败模式是：表面上训练了 PRM，实际上模型只学会盯着最后一步。这时它虽然也输出步骤分数，但本质行为已经接近 ORM。

一个常见缓解方法是混合打分：
$$
\text{Score}_{mix} = \alpha \cdot \text{PRM\_score} + (1-\alpha)\cdot \text{ORM\_score}
$$
其中 $\alpha \in [0,1]$。当步骤监督足够可靠时，$\alpha$ 可以更高；当步骤标签有噪声、但最终答案判定很稳时，可以降低 $\alpha$。

例如：
- $\alpha=0.8$：更信任过程
- $\alpha=0.5$：过程和结果各占一半
- $\alpha=0.2$：把 PRM 当辅助信号

再看一个真实工程视角。论文中的大规模评估会为每道题生成多达 1860 条候选样本，再由验证器筛选。直白地说，这就是“多试很多次，再让评分器挑最靠谱的”。这种方案能换来更高准确率，但前提是你接受更高延迟和更高算力开销。离线评测或高价值推理任务可以接受，在线低延迟场景就未必合适。

| 权衡项 | 收益 | 成本 | 常见缓解办法 |
| --- | --- | --- | --- |
| 步骤级监督 | 更早发现错误 | 标注贵 | 用教师模型预标注，人类抽检 |
| Best-of-N | 提升上限 | 推理贵 | 并行生成、减少无效候选 |
| 严格聚合 | 路径更稳 | 可能错杀探索路径 | 调整阈值或改用混合聚合 |
| 混合打分 | 更鲁棒 | 参数更多 | 在验证集上调 $\alpha$ |

常见坑有四个。

1. 步骤定义不一致。训练集的一步是“一行变形”，线上的一步是“一整段解释”，评分会漂。
2. 只标第一处错误。这样能省成本，但后半段分布会缺失，模型不一定学到“错误如何扩散”。
3. 长路径偏置。用乘积聚合时，长路径即使每步都不错，也可能因连乘而低分。
4. 误把 PRM 当生成器。PRM 更适合“评判和筛选”，不等于它本身就会生成好解法。

## 替代方案与适用边界

如果没有步骤标签，ORM 仍然是最现实的 fallback。fallback 的意思是“退而求其次但可落地的方案”。它便宜、清晰、容易自动化，尤其适合有标准答案的任务。

但 ORM 的边界很明显：它无法可靠区分“好过程得出对答案”和“坏过程碰巧得出对答案”。对短答案题影响可能有限，对长链数学推理影响很大。

另一种折中方案是混合策略：先用 ORM 快速过滤明显错误的候选，再用 PRM 对少量高潜力路径精排。这样能把计算压力从“全量逐步评分”降低到“局部逐步评分”。

还有更轻量的替代方法，例如启发式评分。启发式就是“不是通过专门训练的奖励模型，而是用规则或便宜模型做近似打分”。例如：
- 检查每步是否包含合法公式变形
- 检查变量是否前后一致
- 检查最终答案是否能回代验证

这类方法在实时系统里很有价值，但它们通常不如 PRM 通用，也不如 PRM 能覆盖复杂隐性错误。

| 方案 | 是否需要步骤标签 | 计算成本 | 识别早期错误能力 | 适用场景 |
| --- | --- | --- | --- | --- |
| ORM | 否 | 低 | 弱 | 低资源、快速上线 |
| PRM | 是 | 高 | 强 | 数学、代码、多步推理 |
| 混合策略 | 部分需要 | 中 | 中到强 | 线上折中方案 |
| 启发式评分 | 否 | 低到中 | 中 | 固定领域、规则明确 |

对初级工程师，一个实用判断标准是：

- 没有步骤标签，先上 ORM。
- 有少量步骤标签，先做 ORM + PRM 混合。
- 任务本身高度依赖中间过程，且候选生成成本可接受，再投入 PRM。

PRM 不是“所有任务都必须上”的银弹。它最强的地方是：当正确答案依赖一条长而脆弱的推理链时，它能把监督信号前移到错误刚发生的位置。
