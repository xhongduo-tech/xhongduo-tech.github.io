## 核心结论

知识库覆盖补全的目标，是在不完整知识图谱里预测缺失三元组。知识图谱可以理解为“用点和边表达事实的数据结构”，三元组一般写成 $(h,r,t)$，分别表示头实体、关系、尾实体。若把全部实体集合记为 $E$，关系集合记为 $R$，已知三元组集合记为 $T$，那么图可以写成：

$$
G=(E,R,T)
$$

知识库补全本质上是链接预测。链接预测可以理解为“判断图里还可能缺哪条边”。它的输入不是自然语言问答，而是一个不完整的多关系图；它的输出也不是最终真值，而是一组按分数排序的候选事实。统一记一个候选三元组为 $x=(h,r,t)$，模型会给它一个分数：

$$
s(x)\in \mathbb{R}
$$

分数越高，表示这个三元组越像真的，但不代表已经被证明是真的。

一个最小直观例子：库里已有“阿司匹林 - 治疗 - 疼痛”，当模型看到“布洛芬 - 治疗 - ?”时，会枚举很多候选疾病，给每个候选打分，再把高分项排到前面。这个过程像做排序题，不是一次性判“对/错”。

| 问题 | 内容 |
|---|---|
| 输入是什么 | 不完整知识图谱中的已知三元组，加上一个待补全查询，如 $(h,r,?)$ |
| 输出是什么 | 一组候选实体或候选三元组的排序结果 |
| 为什么不是最终真值 | 因为知识库通常遵循开放世界假设，未出现不等于错误 |

---

## 问题定义与边界

知识库覆盖补全只解决“缺失三元组预测”，不等于事实核验，也不等于自动纠错。事实核验可以理解为“拿外部证据判断一句话真假”；自动纠错则是“发现并修复库里已有的错误事实”。而补全只回答一个更窄的问题：在当前已知图的基础上，哪些缺失边最值得怀疑为真。

常见任务有两类：

$$
\hat{t}=\arg\max_{e\in E} s(h,r,e)
$$

这表示尾实体预测，即给定头实体和关系，找最可能的尾实体。

$$
\hat{h}=\arg\max_{e\in E} s(e,r,t)
$$

这表示头实体预测，即给定关系和尾实体，找最可能的头实体。

这里最容易混淆的点，是开放世界假设。开放世界假设可以理解为“库里没写出来，只表示目前不知道，不表示一定是假”。例如知识图谱里没有 `(新药A, 治疗, 某疾病)`，只能说明当前库没有这条记录，不能直接把它当负例。很多初学者在训练和评估时忽略这一点，会导致模型学到错误边界。

下面这个对照表很重要：

| 对比项 | 左边 | 右边 |
|---|---|---|
| 任务性质 | 链接预测 | 事实核验 |
| 系统角色 | 候选生成 | 真值判定 |
| 写回策略 | 先入候选层 | 不应直接写主库 |
| 世界假设 | 开放世界 | 常需外部证据收敛 |
| “未出现”的含义 | 未知 | 不能直接当错误 |

玩具例子：如果图里有 `(北京大学, 位于, 北京)`、`(清华大学, 位于, 北京)`、`(复旦大学, 位于, 上海)`，现在查询 `(上海交通大学, 位于, ?)`，模型会对“北京、上海、广州”等候选地名分别打分。即使训练集中没有 `(上海交通大学, 位于, 上海)`，也不能因为“没见过”就说它一定错。

这也决定了边界：补全模型擅长做“召回”，不擅长单独承担“最终裁决”。

## 核心机制与推导

所有方法都可以统一成一件事：定义一个打分函数 $s(h,r,t)$，再用它做排序。差别只在于分数从哪里来。主流路线大致有三类：嵌入法、路径法、GNN/文本法。

### 1. 嵌入法：在向量空间里做关系匹配

嵌入可以理解为“把实体和关系压成稠密向量表示”。最经典的是 TransE，它假设关系像一个平移向量：

$$
s(h,r,t)=-\|e_h+e_r-e_t\|_1
$$

如果 $e_h+e_r$ 离 $e_t$ 很近，分数就高。它简单、快、易扩展，适合关系模式较简单的数据。

RotatE 把关系表示为复数空间中的旋转，适合对称、反对称、组合关系：

$$
s=\gamma-\|e_h\odot r-e_t\|
$$

其中 $\odot$ 表示逐元素乘法。直觉上，它比 TransE 更灵活，但训练和理解成本更高。

### 2. 路径法：沿图中路径寻找规则证据

路径可以理解为“从一个实体走到另一个实体时经过的关系序列”。例如：

`(药物A) -[适应症]-> (疾病B) -[伴随症状]-> (症状C)`

如果某种路径经常支持某个目标关系，就可以把它抽成规则。规则置信度常写成：

$$
conf(p\rightarrow r)=\frac{support(p\rightarrow r)}{support(p)}
$$

这里的 support 可以理解为“在数据里出现了多少次”。如果一条路径经常对应目标关系，那么它的解释性就强。路径法的优点是可解释，缺点是长路径稀疏、枚举成本高。

### 3. GNN/文本法：融合邻居和文本再打分

GNN 是图神经网络，可以理解为“让节点从邻居收集信息后更新自己的表示”。文本编码则是“把实体描述、别名、上下文文档编码成向量”。统一写法可以是：

$$
z_v = Encoder(v, N(v), text_v)
$$

其中 $N(v)$ 是邻居集合，$text_v$ 是实体文本描述。得到节点表示 $z_v$ 后，再送入三元组打分层。这样做的动机很直接：长尾实体在图里连边少，但往往还有描述文本可用。

### 4. 为什么几乎都做排序，而不是硬分类

因为候选空间很大。一个查询 $(h,r,?)$ 往往要在整个实体集合里找尾实体。与其对每个候选独立做“真/假”分类，不如直接做排序，取 top-k 作为召回结果。这更符合工程目标，也更适合开放世界设定。

下面用一个汇总表收束：

| 方法 | 输入信号 | 打分方式 | 优点 | 缺点 |
|---|---|---|---|---|
| TransE/RotatE | 图结构 | 向量距离或几何变换 | 快，易批量计算 | 解释性弱，长尾吃亏 |
| Path Ranking | 图路径 | 规则置信度/路径权重 | 可解释，便于审核 | 稀疏时效果差 |
| R-GCN | 邻居结构 | 邻居聚合后打分 | 能利用局部结构 | 过平滑、训练重 |
| KG-BERT | 三元组文本化 | 文本编码后分类/排序 | 文本丰富时强 | 推理慢，成本高 |

真实工程例子：医疗知识库需要补全 `(药物, 治疗, 疾病)`、`(疾病, 伴随症状, 症状)`。若一个新药在图中连接很少，但有说明书、适应症文本、临床术语同义词，那么纯嵌入法容易漏掉，文本增强或 GNN+文本联合编码通常更稳。

## 代码实现

下面用一个最小可运行例子展示“数据怎么进来、分数怎么出来、候选怎么排序”。这里用的是 TransE 风格的 $L_1$ 距离分数。

```python
from typing import Dict, List, Tuple

Vector = List[float]

def score_transe(e_h: Vector, e_r: Vector, e_t: Vector) -> float:
    return -sum(abs(h + r - t) for h, r, t in zip(e_h, e_r, e_t))

def rank_tail_candidates(
    e_h: Vector,
    e_r: Vector,
    candidates: Dict[str, Vector]
) -> List[Tuple[str, float]]:
    scored = [(name, score_transe(e_h, e_r, vec)) for name, vec in candidates.items()]
    scored.sort(key=lambda x: x[1], reverse=True)
    return scored

e_h = [1, 2]
e_r = [0, 1]

candidates = {
    "t1": [1, 3],
    "t2": [2, 3],
    "t3": [1, 4],
}

ranking = rank_tail_candidates(e_h, e_r, candidates)

assert score_transe(e_h, e_r, candidates["t1"]) == 0
assert score_transe(e_h, e_r, candidates["t2"]) == -1
assert ranking[0][0] == "t1"
assert ranking[0][1] > ranking[1][1]

print(ranking)
```

这个玩具例子的推导很直接。先算：

$$
e_h + e_r = (1,2) + (0,1) = (1,3)
$$

于是：

- 对 `t1=(1,3)`，距离为 $|(1-1)| + |(3-3)| = 0$，分数是 $0$
- 对 `t2=(2,3)`，距离为 $|(1-2)| + |(3-3)| = 1$，分数是 $-1$
- 对 `t3=(1,4)$，距离为 $|(1-1)| + |(3-4)| = 1$，分数也是 $-1$

结果表如下：

| 候选实体 | 距离 $\|e_h+e_r-e_t\|_1$ | 分数 | 排名 |
|---|---:|---:|---:|
| t1 | 0 | 0 | 1 |
| t2 | 1 | -1 | 2 |
| t3 | 1 | -1 | 3 |

这个例子说明了一件事：补全系统最核心的输出是“排序”，不是一句绝对判断。

如果放到真实工程里，流程通常不是“全量实体暴力枚举后直接入库”，而是：

| 步骤 | 作用 |
|---|---|
| 输入查询三元组模板 | 例如 `(布洛芬, 治疗, ?)` |
| 生成候选集 | 先用类型约束把候选限制为“疾病”类实体 |
| 批量打分 | 用嵌入、路径或文本模型计算分数 |
| Top-k 返回 | 保留前若干候选事实 |
| 后验校验 | 做规则检查、术语对齐、人工审核 |

伪代码可以写成：

```text
for query in incomplete_triples:
    candidates = generate_candidates_by_type(query)
    scores = model.batch_score(query, candidates)
    topk = select_topk(candidates, scores)
    validated = rule_check_and_entity_alignment(topk)
    send_to_review_queue(validated)
```

真实工程例子：医疗库中补全 `(药物X, 治疗, ?)` 时，不会把“疾病、医院、化合物、城市”全混在一起打分，而会先用实体类型约束把候选压到“疾病”集合，再做模型排序。这样既更快，也更不容易出明显错误。

## 工程权衡与常见坑

真正落地时，难点往往不在公式，而在数据假设和写回流程。

第一类坑是长尾实体。长尾可以理解为“出现次数很少的实体”。这类实体邻居少、训练信号弱，纯结构模型容易学不准。常见缓解方法是加文本描述、同义词、类型约束、邻居聚合。

第二类坑是负样本误标。负样本就是训练时当作错误例子的样本。很多系统会把“没出现过的三元组”随机采样成负例，但在开放世界下，这些样本只是“未标注”，不一定真错。误标过多会直接污染训练目标。

第三类坑是评估泄漏。泄漏可以理解为“测试信息提前混进了训练或特征”。例如按随机切分而不是按时间切分，或者训练里见过逆关系、近重复关系，都会让离线指标虚高。

第四类坑是直接自动写回主库。这是最危险的一类。高分并不等于高可信，更不等于高可用。尤其是医疗、金融、法律等高风险领域，一条错误事实写回后会污染下游检索、推荐和问答。

下面给出一个工程上更实用的规避表：

| 常见坑 | 风险 | 常见规避方式 |
|---|---|---|
| 长尾实体少样本 | 召回差、误判高 | 加文本、类型、邻居特征 |
| 负样本误标 | 训练目标被污染 | 过滤已知真值，谨慎构造硬负例 |
| 忽略开放世界假设 | 把未知当错误 | 把未出现视为未标注，而非天然负例 |
| 评估泄漏 | 离线虚高，线上失真 | 时间切分、filtered ranking、查逆关系泄漏 |
| 直接写回主库 | 错误事实扩散 | 候选层隔离，规则校验，人工审核 |

一个更稳妥的线上流程通常是：

| 顺序 | 环节 | 目标 |
|---:|---|---|
| 1 | 模型召回 | 提供高覆盖候选 |
| 2 | 规则校验 | 去掉违反类型或约束的候选 |
| 3 | 术语对齐 | 合并同义词、别名、缩写 |
| 4 | 人工审核 | 处理高风险或高价值事实 |
| 5 | 入主库 | 只写回通过校验的结果 |

真实工程例子：模型给出 `(药物A, 治疗, 疾病X)` 排名第一，但如果“疾病X”只是“疾病X综合征”的别名未对齐，或者该药物只在体外实验中有效而非临床适应症，直接写回主库就会把“候选证据”误写成“正式事实”。

## 替代方案与适用边界

没有一种补全方法适合所有数据条件。选型时应先问：结构是否稠密，路径是否明显，文本是否丰富，是否要求可解释，是否处在高风险场景。

如果图结构比较规整、关系模式较简单、需要高吞吐召回，TransE/RotatE 这类嵌入法通常是第一选择。它们训练和推理都较高效，适合先做候选召回层。

如果业务要求能解释“为什么推荐这条边”，路径法更合适。它的解释性强，适合辅助审核，但面对超稀疏图或超长依赖时可能不稳。

如果文本信息丰富，或者长尾实体很多，GNN/文本联合方法更有优势。医疗、法律、科研文献等领域常常就是这种情况，因为实体名、定义、上下文文本本身携带了大量信息。

还要强调一个边界：知识库补全不是万能方案。若关系是实时变化的、证据强依赖外部文档、或错误成本极高，那么补全系统应该只做召回，不应独立作最终判定。

| 方法 | 适用数据 | 解释性 | 维护成本 | 适用边界 |
|---|---|---|---|---|
| TransE | 图较稠密、关系简单 | 低 | 低 | 适合高效召回 |
| RotatE | 关系模式更复杂 | 低 | 中 | 适合更灵活关系建模 |
| Path Ranking | 路径规律明显 | 高 | 中 | 适合需要可解释性 |
| R-GCN | 邻居结构有信息 | 中 | 高 | 适合图结构增强 |
| KG-BERT | 文本丰富、实体描述强 | 中 | 高 | 适合文本证据重的场景 |
| Open-world 设定下的联合方案 | 长尾多、未标注多 | 中 | 高 | 适合高不完备图谱 |

对零基础读者，最实用的判断规则是：

- 图很强、文本很弱，先试嵌入法。
- 图里路径很有业务含义，优先考虑路径法。
- 文本丰富、术语复杂、长尾严重，优先考虑文本增强或 GNN。
- 高风险行业里，补全只做候选层，不直接当真值层。
