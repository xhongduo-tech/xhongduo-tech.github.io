## 核心结论

排序模型的特征工程，本质上是在回答一个很具体的问题：对于当前用户、当前候选物品、当前场景，哪些信号能稳定地说明“这个物品现在更值得排前面”。

这里的“特征工程”，可以先理解成“把原始数据翻译成模型能比较的数字信号”。排序模型并不直接读取“用户昨天看了三篇视频”“这个商品价格较高”“现在是晚高峰”这种业务描述，它只接收一个向量，也就是一串按固定顺序排列的数值。特征工程的职责，就是把用户意图、物品属性、行为统计和上下文约束编码进这个向量里。

工业排序里最常见的四类特征如下：

| 特征大类 | 白话解释 | 典型信号 | 常见来源 |
| --- | --- | --- | --- |
| 用户画像 | 描述“这个人是谁” | 用户 ID、注册天数、活跃等级、消费层级 | 用户中心、画像服务 |
| 物品画像 | 描述“这个内容或商品是什么” | 类目、作者、价格、发布时间、质量分 | 内容库、商品库、质检系统 |
| 行为统计 | 描述“最近发生了什么” | 7 天点击率、停留时长、转化率、序列偏好 | 日志系统、聚合任务 |
| 场景上下文 | 描述“此刻环境是什么” | 小时段、设备、网络、入口页、地理区域 | 请求上下文、端上信息 |

排序特征工程的核心不是“特征越多越好”，而是“信号越准确、越一致、越可上线越好”。一个好的特征要同时满足四件事：

1. 能表达业务规律。
2. 训练和线上能一致生成。
3. 缺失时有默认值，不会把服务打崩。
4. 时效性、维度和延迟可控。

玩具例子可以先用一个极小向量理解。设用户 embedding 是 $[0.7, 0.3]$，embedding 可以理解成“把离散 ID 或类别压缩成低维数字表示”；物品质量与热度是 $[0.6, 0.4]$；午间标记与高峰标记是 $[1, 0]$；再构造一个交叉特征 $0.21$。最终向量是：

$$
\phi = [0.7, 0.3, 0.6, 0.4, 1, 0, 0.21]
$$

这串数字的含义不是“绝对好坏”，而是“当前候选相对其他候选是否更匹配”。排序模型学到的，就是如何给这串数字加权。

## 问题定义与边界

排序阶段处理的是“候选集合内部的优先级问题”，不是“从全量库里找候选”。这两个环节容易混淆，但工程职责不同。

“召回”可以先理解成“从海量物品中粗筛出几百到几千个候选”；“排序”则是在这些候选里做精细比较，决定谁在前、谁在后。因此，排序特征工程的输入对象是 $(u, i, c)$ 三元组：

- $u$：用户，user
- $i$：物品，item
- $c$：上下文，context

排序模型的目标是学习一个打分函数：

$$
s(u, i, c) = f(\phi(u, i, c))
$$

其中 $\phi(u, i, c)$ 是特征向量，$s(u, i, c)$ 是分值。分值越高，候选越可能排前。

边界必须说清楚，否则团队很容易把各种数据处理都叫“特征工程”。

| 数据类别 | 是否属于排序特征工程 | 处理阶段 | 可变性 |
| --- | --- | --- | --- |
| 用户静态画像 | 是 | 离线产出，在线读取 | 低到中 |
| 用户近 1 小时/1 天行为统计 | 是 | 离线聚合或流式更新 | 高 |
| 请求时间、设备、页面入口 | 是 | 在线实时生成 | 高 |
| 原始曝光/点击日志 | 否，属于上游原始数据 | 日志采集 | 高 |
| 全量物品索引构建 | 否，偏召回/检索 | 离线索引 | 中 |
| 标注点击标签、转化标签 | 部分相关，但更偏训练样本构造 | 训练数据处理 | 中 |

换句话说，排序特征工程只负责“把可用于训练和推理的信号稳定地准备出来”，不负责替代整个推荐系统的数据链路。

一个新手容易理解的请求例子是：某次推荐请求到来时，系统同时拿到这些信息：

- 用户画像：性别、注册时间、是否高活跃
- 用户行为：近 7 天点击率、近 3 天视频偏好
- 物品属性：内容类型、作者质量分、发布时间
- 场景信息：当前时间段、设备类型、入口页

然后构造特征如：

- 用户是否偏好该类目
- 用户过去 7 天对该作者的点击率
- 当前是否处于高峰时段
- 物品发布时间是否过旧
- 用户偏好视频且当前候选是视频

这些都属于排序特征工程；但“日志怎么上报”“召回为什么没把这个物品找出来”不属于这里的边界。

## 核心机制与推导

排序模型最基础的形式，是线性打分：

$$
s(u, i, c) = w^\top \phi(u, i, c)
$$

这里的 $w$ 是模型参数，可以理解成“每个特征的重要性”；$\phi(u, i, c)$ 是特征向量；$w^\top \phi$ 就是点积，也就是“逐项相乘再求和”。

如果用上面的玩具向量：

$$
\phi = [0.7, 0.3, 0.6, 0.4, 1, 0, 0.21]
$$

再设权重：

$$
w = [1.0, 0.8, 0.5, 0.3, 0.2, 0.1, 0.4]
$$

则分值为：

$$
s = 0.7\times1.0 + 0.3\times0.8 + 0.6\times0.5 + 0.4\times0.3 + 1\times0.2 + 0\times0.1 + 0.21\times0.4
$$

$$
s = 1.644
$$

这个数字本身没有业务语义，关键在于它能和其他候选比较。比如另一个候选只得到 $1.31$，那前者应排在前面。

特征向量内部通常可以分成四段：

| 向量位置 | 类型 | 作用 |
| --- | --- | --- |
| $\phi_u$ | 用户特征 | 表达长期偏好与用户身份 |
| $\phi_i$ | 物品特征 | 表达内容质量、属性与供给信息 |
| $\phi_c$ | 上下文特征 | 表达当前时刻与当前场景 |
| $\phi_{uic}$ | 交叉特征 | 表达“这个用户在这个场景下对这个物品”的特殊关系 |

可以写成：

$$
\phi(u, i, c) = [\phi_u; \phi_i; \phi_c; \phi_{uic}]
$$

其中“交叉特征”可以先理解成“两个或多个信号组合后才有意义的特征”。例如：

- 用户偏爱短视频 × 当前候选是短视频
- 用户过去常买高价商品 × 当前商品价格高
- 晚间访问 × 娱乐类内容

为什么不能只靠用户特征和物品特征？因为排序问题本质是匹配问题。用户喜欢篮球，不代表他在工作日上午就想看篮球集锦；可能晚上才点。于是上下文与交叉项必不可少。

再看训练目标。排序不是普通分类，因为它更关心“相对顺序”而不是“单点预测”。常见目标有三类：

| 目标类型 | 白话解释 | 典型形式 |
| --- | --- | --- |
| Pointwise | 把每个样本单独当回归或分类 | 预测点击概率 |
| Pairwise | 让正样本分数高于负样本 | $s^+ > s^-$ |
| Listwise | 直接优化整个列表顺序 | NDCG、ListNet 等 |

NDCG 可以先理解成“前排位置是否被真正相关的物品占据”的指标。它对头部位置更敏感，因为用户更容易看到前几个结果。设位置折损为 $\log_2(1+k)$，则：

$$
DCG@K = \sum_{k=1}^{K} \frac{rel_k}{\log_2(k+1)}
$$

其中 $rel_k$ 是第 $k$ 个位置物品的相关性。再除以理想排序的 $IDCG@K$，得到：

$$
NDCG@K = \frac{DCG@K}{IDCG@K}
$$

这说明一件重要的事：特征工程不是孤立的。你设计什么特征，最终要看它是否提升了列表顺序质量，而不是只看某个单特征是否“看起来合理”。

真实工程例子可以用信息流推荐说明。一个短视频 feed 排序请求到来时，系统会对每个候选视频构造如下信号：

- 用户长期兴趣 embedding：用户历史上更偏向搞笑、科技还是游戏
- 用户短期序列统计：最近 20 次滑动中对哪种内容停留更久
- 作者质量分：作者历史完播率、举报率、内容稳定性
- 新鲜度：发布时间距离当前多久
- 场景特征：网络状态、是否 Wi-Fi、当前是否夜间
- 用户-作者交叉：该用户过去是否常看该作者
- 用户-类目交叉：该用户对“科技”类内容近期点击率是否上升

这些信号一起进入模型，模型并不是在学“科技视频总是更好”，而是在学“对这个用户、这个时间、这个候选集，什么更值得放前面”。

## 代码实现

工程实现通常分三步：

1. 拉取用户、物品、上下文原始信号。
2. 做 embedding、归一化、分桶与交叉。
3. 组装成模型输入，对候选逐个打分。

“归一化”可以先理解成“把不同量纲的数拉到可比较范围内”；“分桶”可以理解成“把连续值切成离散区间”；“hash bucket”可以理解成“把高维离散值映射到固定桶数里，降低维度爆炸”。

下面给一个可运行的 Python 玩具实现，演示从原始信号到排序分值的过程：

```python
from math import log1p

def min_max(value, lower, upper):
    if upper <= lower:
        raise ValueError("upper must be greater than lower")
    clipped = max(lower, min(value, upper))
    return (clipped - lower) / (upper - lower)

def bucket_hour(hour):
    # 早高峰、午间、晚高峰、其他
    if 7 <= hour <= 9:
        return [1.0, 0.0, 0.0, 0.0]
    if 11 <= hour <= 13:
        return [0.0, 1.0, 0.0, 0.0]
    if 18 <= hour <= 21:
        return [0.0, 0.0, 1.0, 0.0]
    return [0.0, 0.0, 0.0, 1.0]

def embed_user(user_id, is_high_active):
    # 玩具 embedding，真实工程会来自训练好的参数表
    base = {
        1: [0.7, 0.3],
        2: [0.2, 0.8],
    }
    vec = base.get(user_id, [0.5, 0.5])
    return vec + [1.0 if is_high_active else 0.0]

def embed_item(item_type, quality_score):
    type_map = {
        "video": [1.0, 0.0],
        "article": [0.0, 1.0],
    }
    type_vec = type_map.get(item_type, [0.5, 0.5])
    quality_norm = min_max(quality_score, 0.0, 100.0)
    return type_vec + [quality_norm]

def build_feature_vector(user, item, context):
    u_vec = embed_user(user["id"], user["is_high_active"])
    i_vec = embed_item(item["type"], item["quality_score"])

    # 行为统计做对数压缩，避免长尾数值过大
    ctr_7d = min_max(user["ctr_7d"], 0.0, 0.2)
    dwell_log = min_max(log1p(user["avg_dwell_sec"]), 0.0, log1p(300.0))

    c_vec = bucket_hour(context["hour"])

    # 交叉特征：用户视频偏好 × 当前物品是否视频
    user_video_pref = min_max(user["video_ratio_7d"], 0.0, 1.0)
    is_video = 1.0 if item["type"] == "video" else 0.0
    cross_1 = user_video_pref * is_video

    # 交叉特征：高活跃用户 × 高质量内容
    cross_2 = (1.0 if user["is_high_active"] else 0.0) * i_vec[-1]

    return u_vec + i_vec + [ctr_7d, dwell_log] + c_vec + [cross_1, cross_2]

def score(feature_vector, weights):
    assert len(feature_vector) == len(weights)
    return sum(x * w for x, w in zip(feature_vector, weights))

user = {
    "id": 1,
    "is_high_active": True,
    "ctr_7d": 0.12,
    "avg_dwell_sec": 80,
    "video_ratio_7d": 0.75,
}

item_a = {"type": "video", "quality_score": 82}
item_b = {"type": "article", "quality_score": 70}
context = {"hour": 12}

features_a = build_feature_vector(user, item_a, context)
features_b = build_feature_vector(user, item_b, context)

weights = [0.6, 0.4, 0.3, 0.5, 0.2, 0.8, 0.7, 0.4, 0.1, 0.2, 0.0, 0.1, 0.9, 0.6]

score_a = score(features_a, weights)
score_b = score(features_b, weights)

assert len(features_a) == len(weights)
assert score_a > score_b
assert 0.0 <= features_a[5] <= 1.0  # 质量分已归一化
print(round(score_a, 4), round(score_b, 4))
```

这段代码里有几个关键点：

- `embed_user` 和 `embed_item` 模拟离散特征转低维向量。
- `min_max` 和 `log1p` 处理数值尺度，避免点击率和停留时长量纲冲突。
- `bucket_hour` 把时间离散化，避免模型直接学习脆弱的“小时数大小关系”。
- `cross_1`、`cross_2` 显式表达用户与物品之间的匹配关系。

真实工程里，不会手写这么小的向量，而是会构造成类似 `ExampleListWithContext` 的结构。它可以理解成“一个请求级上下文，加上一组候选级样本”的输入格式。上下文特征如时间、设备、入口页放在 request 级；用户-物品特征与物品特征放在 candidate 级；最终形成一个列表，模型一次性对整组候选打分。

一个真实工程版本的处理链通常是：

| 步骤 | 典型动作 | 关注点 |
| --- | --- | --- |
| 离线层 | 聚合 CTR、CVR、停留、质量分，训练 embedding | 覆盖率、时效窗口、版本管理 |
| 在线特征层 | 读取画像、统计、上下文 | 低延迟、默认值、缓存命中 |
| 特征转换层 | 归一化、分桶、交叉、缺失填充 | 与训练一致 |
| 排序服务层 | 构造候选列表输入并预测 | 吞吐、超时、降级 |
| 监控层 | 监控均值、方差、NULL 率、延迟 | 漂移告警 |

## 工程权衡与常见坑

排序特征工程真正难的地方，不在“能不能造出特征”，而在“能不能长期稳定地用这些特征”。

最常见的问题如下：

| 常见坑 | 表现 | 根因 | 规避方式 |
| --- | --- | --- | --- |
| 时效性不足 | 离线效果好，线上掉点 | 特征生成滞后，窗口与线上数据可得性不一致 | 预留延迟窗口，如“过去 1 天且延迟 3 小时” |
| 特征穿越 | 测试指标异常高，线上失效 | 用到了未来信息 | 严格按事件时间切分训练样本 |
| 训练/线上不一致 | 同样样本线上线下分数不同 | 两套转换逻辑、默认值不同 | 共享同一套特征流水线 |
| 高维稀疏 | 参数爆炸、泛化差 | ID 和交叉组合太多 | embedding、hash bucket、特征选择 |
| 缺失值混乱 | 新用户或新物品表现极差 | 默认值设计不合理 | 明确 missing 语义，区分“真 0”和“缺失” |
| 无效交叉过多 | 系统变复杂，但收益很小 | 人工堆特征无约束 | 做增益评估，删除贡献低于阈值的特征 |

“特征穿越”需要重点解释。它可以理解成“模型在训练时偷看了未来信息”。例如你构造“用户未来 1 天点击率”作为特征，离线评估会很好，因为它已经包含答案；但线上根本拿不到这个值，所以模型一上线就崩。

小白版例子如下：

- 错误做法：用“某内容未来 24 小时总点击量”做质量分。
- 结果：训练集表现很好，因为热门内容在未来本来就会有更多点击。
- 问题：线上打分时，未来 24 小时还没发生。
- 正确做法：改成“过去 24 小时点击量，且统计延迟 3 小时落地”。

另一个典型坑是“特征漂移”，英文常叫 feature drift，可以理解成“线上输入的分布和训练时不一样了”。比如节假日期间用户访问时段变化很大，导致“小时段”特征分布整体偏移；或者内容供给结构变化，导致作者质量分的均值大幅抬升。此时即使代码没问题，模型效果也会掉。

因此监控不能只看最终 CTR，还应看特征本身：

- 均值、方差是否突变
- NULL 率是否升高
- Top bucket 占比是否异常
- 新用户、新物品覆盖率是否下降
- 训练集与线上实时分布的 PSI 或 KL 差异是否过大

还有一个经常被忽略的权衡是“表达力”和“可维护性”的冲突。交叉特征越多，模型理论上表达越强，但工程成本也越高：

- 字段更多，链路更长
- 线上读取更慢
- 解释性更差
- 默认值设计更复杂
- 排查问题更困难

因此工业上常见做法不是无限扩特征，而是分层治理：

1. 保留高价值基础特征：画像、质量分、近窗统计、上下文。
2. 用 embedding 吸收部分高维稀疏问题。
3. 对人工交叉做增量实验。
4. 定期淘汰低收益、难维护特征。

## 替代方案与适用边界

“排序模型的特征工程”并不等于“永远做单目标点击排序”。在不同业务里，目标和约束不同，方案也会变化。

最基础的方案是单目标排序，即围绕点击、转化或完播率构造特征，再用一个打分函数排序。这种方案简单、稳定、易解释，是大多数团队的基线。

但如果业务希望同时优化多个目标，比如点击率、多样性、时长、GMV、投诉率，单一分值就不一定够。此时会进入“重排”问题，也就是在初步排序结果上再做一次多目标调整。

可对比如下：

| 方案 | 适用条件 | 复杂度 | 可控性 | 局限 |
| --- | --- | --- | --- | --- |
| 单目标线性/深度排序 | 先做稳定基线，延迟预算紧 | 低到中 | 中 | 容易偏向单一目标 |
| MPAD 类多目标序列重排 | feed 流需兼顾相关性与多样性 | 高 | 中 | 序列建模重，算力成本高 |
| CMR 类可控多目标重排 | 策略经常变化，需要在线调偏好 | 高 | 高 | 系统复杂，调参难 |

MPAD 这类方法，可以理解成“按顺序挑选物品时，同时考虑前面已经放了什么”，所以它更适合信息流、短视频流这类用户连续消费的场景。比如前两个内容已经都是篮球视频，第三个即使也高分，也可能故意插入一个科技内容，提升多样性。

CMR 这类可控方法，则更适合策略频繁变化的业务。例如运营今天更重点击，明天更重 GMV，后天又希望控制用户疲劳度，那么可以通过外部偏好参数动态调整重排目标。

但对大多数零基础到初级工程师来说，正确路径不是一开始就上多目标重排，而是：

1. 先把单目标排序特征工程做扎实。
2. 保证训练线上一致。
3. 建立时效、缺失、漂移监控。
4. 再考虑多目标扩展。

因为如果基础特征链路本身不稳定，多目标方法只会把问题放大。简单说，$w^\top \phi$ 这种基线并不过时。只要特征做得准、数据链路一致、评估闭环完整，它依然是大量系统的有效起点。
