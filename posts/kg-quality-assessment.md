## 核心结论

知识图谱质量评估，指的是系统地判断图谱里的三元组是否“对、全、一致、不过期”。三元组就是一条最基本的事实记录，可以理解为“主语-谓语-宾语”这样的断言，例如“北京-是中国首都-中国”。如果评估做得不准，后续检索、问答、推荐、风控、知识融合都会建立在不可靠事实之上，工程风险会直接放大。

工程上最常用的四个维度是准确性、完整性、一致性、时效性。准确性看“写进去的事实有多少是真的”，完整性看“应该有的事实漏了多少”，一致性看“不同事实之间是否互相冲突”，时效性看“事实是否已经过期”。这四个维度里，准确性通常先作为中心指标，因为错误事实会直接污染下游系统。

常见评估方法有三类：抽样检测、规则约束检查、与金标准对比。抽样检测就是从图谱里随机抽一部分事实做人审，成本低，适合先摸清整体噪声；规则检查就是把类型、取值范围、唯一性等约束写成规则，自动批量扫描；金标准对比就是拿一部分高可信标注或高质量外部图谱做参照，直接算指标。三类方法互补，不是替代关系。

一个最直观的数字例子是：如果抽检 1,000 个映射三元组，其中 924 个正确，76 个错误，那么精度就是 $924/1000=92.4\%$。这可以理解为“每 100 个断言里，大约有 7 个需要人工重点核查”。对工程团队来说，这个数字不是学术装饰，而是上线阈值、人工复核预算和告警策略的依据。

| 质量维度 | 白话解释 | 代表性检查手段 | 常用指标 |
|---|---|---|---|
| 准确性 | 已有事实里有多少是真的 | 抽样、金标准对比 | precision |
| 完整性 | 应该有的事实漏了多少 | 金标准对比、覆盖分析 | recall、coverage |
| 一致性 | 事实之间是否互相打架 | 规则约束检查 | violation rate |
| 时效性 | 事实是否已经过期 | 时间规则、版本差异检查 | freshness、staleness |

## 问题定义与边界

知识图谱质量评估的输入，是一个已经存在的三元组集合。输出，不是“把图谱修好”，而是“告诉你哪里可能错、哪里可能缺、哪里可能过期”。这一步的职责是标记状态和量化风险，不负责自动补全所有缺失数据。

可以把它理解成质检，而不是生产。生产阶段负责采集、抽取、融合、对齐；质检阶段负责回答两个问题：第一，这批事实能不能安全进入工程系统；第二，如果不能，问题主要集中在哪些关系、哪些实体、哪些来源。

下面这张表有助于把边界看清楚。

| 项目 | 内容 |
|---|---|
| 输入 | 已有 KG、待评估三元组、部分人工标注、规则集、外部金标准 |
| 输出 | 错误率、精度、召回、规则违例、过期比例、待复核清单 |
| 边界 | 只评估已有断言的可靠性与覆盖情况 |
| 不能做的事 | 不能直接代替数据采集、知识融合、实体消歧、事实修复 |

对新手最直观的说法是：从图谱里随便抽 50 条事实，让人判断“对、错、过期”，再把结果统计成指标。如果系统要求某类事实精度必须超过 95%，那这一步就是拿证据去判断它有没有达标。

这里还要明确一个边界：完整性往往很难在“全图”上严格计算，因为你通常不知道真实世界的全部事实是什么。所以工程里常说的完整性，很多时候是“相对于某个任务范围”或“相对于某个金标准子集”的完整性，而不是宇宙真值上的完整性。

玩具例子：有一个电影小图谱，只有 5 条三元组。你发现“《盗梦空间》-导演-诺兰”是对的，“《盗梦空间》-上映年份-2012”是错的，“《星际穿越》-导演-诺兰”缺失了，“爱因斯坦-职业-电影导演”明显不一致。这个例子很小，但已经覆盖了错、漏、冲突三类典型问题。

真实工程例子：一个电商知识图谱里，商品属性、品牌归属、适用品类每天都在变化。如果“某手机支持卫星通信”这类事实错误，搜索和推荐会直接误导用户；如果“库存状态”过期，运营系统会做出错误决策。此时质量评估不是锦上添花，而是上线前的必要门槛。

## 核心机制与推导

最常见的两个指标是 precision 和 recall。precision，精度，白话解释是“系统判成正确的事实里，真正正确的比例”；recall，召回，白话解释是“真实应该被找出来的正确事实里，系统找出来了多少”。

公式是：

$$
precision = \frac{TP}{TP+FP}
$$

$$
recall = \frac{TP}{TP+FN}
$$

其中，$TP$ 是真阳性，表示被判为正确且实际正确的三元组；$FP$ 是假阳性，表示被判为正确但实际错误的三元组；$FN$ 是假阴性，表示实际正确但系统漏掉的三元组。

如果精度低，说明噪声多，图谱里混进了太多错事实；如果召回低，说明覆盖弱，该有的事实没有补齐。两者通常不能只看一个。一个系统可以把标准设得极严，只保留最确定的事实，从而得到很高精度，但召回会下降；也可以疯狂并入数据，把很多候选事实都收进来，召回会上升，但精度会下滑。

最小数值例子：假设抽样评估 200 条，人工或金标准确认其中 180 条是正确匹配，5 条是错误断言，15 条是应该存在但当前系统没有覆盖到的事实。那么：

$$
precision = \frac{180}{180+5} = 97.3\%
$$

$$
recall = \frac{180}{180+15} = 92.3\%
$$

这组数字的含义很直接：已有结果整体较准，但覆盖还不够，应该优先补漏，而不是继续放松阈值引入更多噪声。

可以用一个简化关系来理解：

| 集合 | 含义 |
|---|---|
| Gold | 金标准里确认正确的事实集合 |
| Predicted | 当前图谱或当前规则产出的事实集合 |
| Gold ∩ Predicted | $TP$ |
| Predicted - Gold | $FP$ |
| Gold - Predicted | $FN$ |

DBpedia 的质量评估框架 QAF 就是沿着这个思路做的。它用高可信映射集合作为参照，持续跟踪不同版本的映射三元组质量，并长期把精度作为核心指标。原因很现实：对于大规模开放图谱，先保证“写进去的大多数是对的”，比一开始追求极限覆盖更可控。2011 年 DBpedia 3.7 的映射三元组精度约为 92.4%，对应的直觉就是“每 100 条事实里约有 7 条可能有问题”。

一致性和时效性虽然不直接落在 precision/recall 里，但依然可以通过规则计数转成可操作指标。例如：

- 一致性违例率 $= \frac{\text{违反规则的三元组数}}{\text{被检查三元组总数}}$
- 过期率 $= \frac{\text{已过有效期的三元组数}}{\text{含时间约束的三元组总数}}$

所以真正可落地的评估体系不是只算两个公式，而是“精度/召回 + 规则违例 + 时间状态”组合起来看。

## 代码实现

一个可执行的最小实现，通常要把三种证据合并起来：抽样标签、规则日志、金标准对比。抽样负责看整体噪声，规则负责批量发现结构错误，金标准负责给出更稳定的基准。

流程可以压缩成四步：

1. 从待评估三元组中抽样或全量遍历关键关系。
2. 对样本做人工标注，或接入高可信 oracle。
3. 跑规则引擎，检查类型、取值范围、唯一性、时间约束。
4. 与金标准做交集比对，统计 $TP/FP/FN$，输出指标与问题清单。

下面是一个能运行的 Python 玩具实现。它不依赖 RDF 框架，但把核心机制完整保留了。

```python
from dataclasses import dataclass
from typing import List, Set, Tuple, Dict

Triple = Tuple[str, str, str]

@dataclass
class EvalResult:
    tp: int
    fp: int
    fn: int
    precision: float
    recall: float
    rule_violations: int

def run_rules(triples: List[Triple]) -> List[Triple]:
    """
    一个极简规则检查器：
    1. bornYear 必须是纯数字
    2. spouse 关系不允许对象为空
    """
    violations = []
    for s, p, o in triples:
        if p == "bornYear" and not o.isdigit():
            violations.append((s, p, o))
        if p == "spouse" and o.strip() == "":
            violations.append((s, p, o))
    return violations

def evaluate(predicted: Set[Triple], gold: Set[Triple]) -> EvalResult:
    tp = len(predicted & gold)
    fp = len(predicted - gold)
    fn = len(gold - predicted)
    precision = tp / (tp + fp) if (tp + fp) else 0.0
    recall = tp / (tp + fn) if (tp + fn) else 0.0
    violations = run_rules(list(predicted))
    return EvalResult(
        tp=tp,
        fp=fp,
        fn=fn,
        precision=precision,
        recall=recall,
        rule_violations=len(violations),
    )

predicted = {
    ("Inception", "director", "Christopher Nolan"),
    ("Inception", "releaseYear", "2010"),
    ("Albert Einstein", "occupation", "Film Director"),  # 错误事实
    ("Ada Lovelace", "bornYear", "1815"),
    ("Marie Curie", "bornYear", "unknown"),  # 规则违例
}

gold = {
    ("Inception", "director", "Christopher Nolan"),
    ("Inception", "releaseYear", "2010"),
    ("Ada Lovelace", "bornYear", "1815"),
    ("Albert Einstein", "occupation", "Physicist"),
    ("Interstellar", "director", "Christopher Nolan"),
}

result = evaluate(predicted, gold)

assert result.tp == 3
assert result.fp == 2
assert result.fn == 2
assert round(result.precision, 2) == 0.60
assert round(result.recall, 2) == 0.60
assert result.rule_violations == 1

print(result)
```

这段代码对应的意思是：

- `predicted` 表示当前图谱里的事实。
- `gold` 表示人工确认或高可信来源中的事实。
- `predicted & gold` 算出正确命中的部分，也就是 $TP$。
- `predicted - gold` 算出当前图谱中多出来但不在金标准里的部分，也就是 $FP$。
- `gold - predicted` 算出该有但没有的部分，也就是 $FN$。
- `run_rules` 模拟规则引擎，例如 RDFUnit 这类工具做的事情。

如果把它放到真实系统中，通常还会增加统一的数据结构，记录每条三元组来自哪里、谁标注过、触发了哪些规则、最后质量分是多少。一个常见结构如下：

| 字段 | 含义 |
|---|---|
| triple_id | 三元组唯一标识 |
| source | 来源系统或抽取器 |
| human_label | 人工标签：correct / wrong / stale |
| rule_flags | 触发的规则列表 |
| in_gold | 是否命中金标准 |
| quality_score | 聚合后的质量分 |

真实工程例子可以想成这样：做医疗知识融合时，药物-适应症、药物-禁忌症、疾病-并发症这些关系不能只靠一个数据源直接并入。通常先做抽样，看错误率是否已经失控；再写规则，如“儿童禁用药物不能同时标记为新生儿推荐”；最后只对关键关系与高可信医学词库或人工审定集做对照。这样才能把可上线部分和待复核部分拆开。

## 工程权衡与常见坑

质量评估最容易被低估的地方，不在公式，而在采样偏差、规则盲区和人工成本。

一个典型问题是：小样本金标准经常偏向“热门实体”或“已知高质量区域”。这样算出来的精度会偏高，因为冷门实体、长尾关系、跨源融合边界本来就更容易出错。如果你只拿热门电影、知名城市、著名科学家去验，结果通常会比真实全局质量好看很多。

另一个常见问题是：规则检查很强，但只能检查“你已经明确写出来的约束”。如果两个图之间语义冲突，但没有显式规则，它就可能完全放过。比如一个图说“某公司总部在上海”，另一个图说“总部在深圳”，单看类型都合法，但语义上明显冲突。

还有众包误判问题。很多 RDF 资源本身是 URI，普通审核者看不懂 `owl:sameAs`、`rdf:type`、YAGO 类路径。没有上下文渲染时，人容易把“表达难懂”误判成“事实错误”。所以真实项目里，审核界面必须把实体名、关系解释、来源文本片段都展示出来。

下面这张表总结常见坑。

| 问题 | 后果 | 规避办法 |
|---|---|---|
| 只用小样本金标准 | 精度被高估 | 增加冷启动实体、多轮采样、分层抽样 |
| 只做规则检查 | 放过跨图语义冲突 | 结合金标准和差异测试 |
| 审核界面只展示 URI | 众包误判率升高 | 渲染实体标签、来源上下文、关系说明 |
| 抽样不分关系类型 | 关键关系风险被平均掉 | 对核心关系单独评估 |
| 过分追求精度 | 图谱覆盖不足 | 为关键任务补看 recall 与 coverage |
| 人工复核比例过低 | 边缘错误持续漏网 | 对高风险来源和高影响关系保留人工后门 |

TripleCheckMate 的经验很有代表性。它在 DBpedia 3.9 上处理了 38,633 个三元组，组织了 58 名专家和众包参与者共同检查，并借助 RDFUnit 先筛出对象和类型层面的可疑项，再交给人工复核。这个流程的核心不是“机器替代人工”，而是“机器先把低成本问题筛一遍，人工只处理边缘和高风险情况”。这才是大规模图谱质检能跑起来的现实路径。

还有一个工程取舍是阈值设定。若你的系统是内部搜索推荐，某些关系精度 90% 可能还能接受；若你的系统进入医疗、金融、风控、合规，95% 甚至 99% 才有讨论空间。阈值不是学术常数，而是和业务风险绑定的工程参数。

## 替代方案与适用边界

除了抽样、规则、金标准，工程里还会用差异测试和基于嵌入的异常检测。

差异测试，白话解释是“拿新旧两个版本对比，看哪里突然变化”。它特别适合做持续集成里的回归检查。例如昨天“1000 个品牌-国家关系”里只有 5 个变化，今天突然变了 300 个，那多半不是世界在一天内剧烈变化，而是抽取流程或映射规则出了问题。它擅长发现异常变化，但不一定直接告诉你真实精度是多少。

基于嵌入的异常检测，白话解释是“把实体和关系压缩成向量，再找语义上看起来不合群的事实”。例如大量“科学家-职业-物理学家”都聚在一起，而“爱因斯坦-职业-电影导演”会成为离群点。它适合做大规模预筛，但通常产出的是“可疑分数”，不是可直接上线汇报的 precision/recall。

选择方案时，关键不是哪个最先进，而是哪个最能回答当前工程问题。

| 方案 | 成本 | 准确度特点 | 所需资源 | 适用场景 |
|---|---|---|---|---|
| 抽样检测 | 低到中 | 对全局噪声判断快，但依赖采样质量 | 标注人力 | 新图谱 sanity check、版本抽检 |
| 规则检查 | 中 | 结构错误检出强，但有规则盲区 | 领域规则、规则引擎 | 类型稳定、约束明确的领域 |
| 金标准对比 | 中到高 | 指标稳定、解释性强 | 高质量标注或外部 KG | 关键关系验真、版本基线 |
| 差异测试 | 低 | 善于发现变化，不直接给真实精度 | 历史版本 | 持续发布、回归监控 |
| 嵌入异常检测 | 中到高 | 擅长发现潜在语义异常 | 训练数据、模型资源 | 大规模预筛、长尾异常发现 |

一个适合新项目的组合策略通常是：

1. 先做抽样，快速知道整体噪声处在什么量级。
2. 再对关键关系写规则，把显式错误批量清掉。
3. 最后拿高信度映射或人工标注子集做金标准对照，给出能汇报的 precision/recall。

这三步可以概括成一句工程话：先快速看一下，再逐条检查，再拿最可靠部分做对比。

适用边界也要说清楚。抽样不适合替代持续监控，因为它难以及时发现局部突变；规则不适合覆盖开放世界中的所有语义冲突，因为你不可能预先写完所有规则；金标准不适合全图依赖，因为高质量标注往往只覆盖局部；嵌入异常检测不适合直接作为上线阈值，因为可解释性和误报控制通常不如规则与金标准。

所以，知识图谱质量评估不是找一个“万能方法”，而是围绕业务风险，把准确性、完整性、一致性、时效性四个维度拆开，用不同证据组合成一个能长期运转的质检体系。
