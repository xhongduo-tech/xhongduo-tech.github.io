## 核心结论

BM25 是一种**稀疏检索**方法。稀疏检索的白话解释是：系统不理解整句话，只看哪些词出现了、出现几次、这些词在全库里稀不稀有。它之所以能在 RAG 中长期存在，不是因为“高级”，而是因为它稳定、快、可解释，适合做第一层候选召回。

BM25 的核心价值有三点：

1. 它把“词频”“稀有度”“文档长度”放进同一个公式里，避免只按出现次数粗暴排序。
2. 它对重复词做**饱和**处理。饱和的白话解释是：某个词第一次、第二次出现很有价值，继续重复的收益会越来越小，不能靠刷词无限加分。
3. 它对长文做**长度归一化**。长度归一化的白话解释是：长文天然更容易碰巧包含查询词，所以需要自动打一点折扣，避免长文总占便宜。

在真实 RAG 系统里，BM25 通常不是最终排序器，而是第一阶段召回器：先从几万到几百万段文本里快速找出 top-k 候选，再交给向量检索、cross-encoder reranker 或 LLM 做更细的判断。

一个新手能立刻理解的玩具例子是查询“合同违约”。如果三个段落分别是：

- A: “合同违约责任包括赔偿损失”
- B: “合同 合同 合同 的签署流程”
- C: “违约后的证据保存与通知义务”

BM25 往往会优先给 A 更高分，因为它同时命中“合同”和“违约”，而 B 虽然重复很多次“合同”，但“刷词”收益会被压平，C 只命中一个词，因此通常排在后面。

## 问题定义与边界

本文讨论的是：如何在一个文档库中，用 BM25 为查询找到更相关的文档或段落，并解释它为什么适合做 RAG 的稀疏召回层。

这里先给出边界。BM25 解决的是**关键词匹配排序**问题，不是语义理解问题。语义理解的白话解释是：系统知道“协议违规”和“合同违约”意思接近；BM25 默认做不到这件事，除非分词、同义词扩展或索引设计提前帮它。

适合 BM25 的场景通常有这些特征：

| 场景 | 为什么适合 |
|---|---|
| 合同、制度、政策、手册 | 术语稳定，关键词直接决定相关性 |
| 企业知识库 FAQ | 问题里常出现明确名词、编号、产品名 |
| 日志、告警、错误码检索 | 精确 token 很关键，不能只靠语义近似 |
| RAG 第一阶段召回 | 需要低延迟、可解释、可增量更新 |

不适合 BM25 的场景通常有这些特征：

| 场景 | 为什么不适合 |
|---|---|
| 同义改写很多 | “协议违规”可能找不到“合同违约” |
| 强依赖上下文理解 | 词都命中，但真正语义不相关 |
| 非常短的模糊查询 | 词太少，区分力不足 |
| 跨语言或口语化表达 | 表达漂移大，词面不稳定 |

必须把“问题定义”和“适用边界”分开看。BM25 不关心句子是否自然，也不关心推理链条是否完整，它只关心查询词和文档词之间的统计关系。因此它常见的失败模式也很直接：

| 边界点 | 表现 |
|---|---|
| 同义词 | “停机”与“服务不可用”不会自动互通 |
| 词序 | “解除合同”与“合同解除条件”区分有限 |
| 语义消歧 | “苹果”可能是水果，也可能是公司 |
| 跨句推理 | 关键信息分散在多句时，单段命中未必充分 |

新手版本可以这样理解：你问“合同违约”，BM25 会去找同时包含“合同”和“违约”的文本；如果文档写的是“协议违规”，那它可能根本看不出来这是同一类问题。所以它做的是“关键词配对”，不是“理解句子”。

## 核心机制与推导

BM25 的常见打分公式是：

$$
\text{score}(D, Q)=\sum_{q_i \in Q} \text{IDF}(q_i)\cdot
\frac{f(q_i,D)\cdot (k_1+1)}
{f(q_i,D)+k_1\cdot \left(1-b+b\cdot \frac{|D|}{avgdl}\right)}
$$

其中：

- $D$ 是一个文档或一个 chunk。
- $Q$ 是查询词集合。
- $f(q_i,D)$ 是词频。词频的白话解释是：某个查询词在文档里出现了多少次。
- $|D|$ 是文档长度。
- $avgdl$ 是全库平均文档长度。
- $k_1$ 控制词频饱和速度。
- $b$ 控制长度归一化强度。

逆文档频率 IDF 常写成：

$$
\text{IDF}(q_i)=\ln\left(\frac{N-n(q_i)+0.5}{n(q_i)+0.5}+1\right)
$$

其中：

- $N$ 是文档总数。
- $n(q_i)$ 是包含词 $q_i$ 的文档数。
- IDF 的白话解释是：越少见的词，区分力越强，权重越高。

参数含义可以压缩成下面这张表：

| 参数 | 作用 | 常见取值 | 直观理解 |
|---|---|---|---|
| `k1` | 控制词频饱和 | 1.2 到 2.0 | 越大越接近线性累加，越小越快“涨不动” |
| `b` | 控制长度惩罚 | 约 0.75 | 越大越强调长文折扣 |
| `avgdl` | 平均长度基线 | 由语料决定 | 用来判断当前文档是长还是短 |

为什么这个公式合理，可以拆成三步理解。

第一步，看 IDF。  
如果一个词在几乎所有文档里都出现，比如“系统”“方法”“处理”，它不能说明太多相关性，所以 IDF 低。反过来，“违约金”“QPS”“E11000”这类词在全库里更稀有，命中后更有区分力。

第二步，看词频饱和。  
如果一个段落出现了 1 次“违约”，和出现 3 次“违约”，后者应该更相关；但出现 20 次未必比 3 次强很多，因为可能只是重复表述。因此 BM25 不是简单线性累加，而是“先涨后平”。

第三步，看长度归一化。  
长文天然含词更多。如果不做长度归一化，长文几乎总能凭覆盖面获胜。BM25 用 $\frac{|D|}{avgdl}$ 给长文加一个分母，防止长文本因为篇幅大而占便宜。

下面给一个最小数值例子。设：

- 总文档数 $N=100$
- 查询词“违约”出现在 5 篇文档中，$n(q)=5$
- 当前文档中该词出现 3 次，$f(q,D)=3$
- 文档长度 $|D|=200$
- 平均长度 $avgdl=150$
- $k_1=1.5, b=0.75$

则：

$$
\text{IDF}=\ln\left(\frac{100-5+0.5}{5+0.5}+1\right)
=\ln\left(\frac{95.5}{5.5}+1\right)\approx 2.91
$$

长度项：

$$
1-b+b\cdot \frac{|D|}{avgdl}
=1-0.75+0.75\cdot \frac{200}{150}=1.25
$$

词频部分：

$$
\frac{3\cdot (1.5+1)}{3+1.5\cdot 1.25}
=
\frac{7.5}{4.875}
\approx 1.54
$$

总得分约为：

$$
2.91 \times 1.54 \approx 4.48
$$

这个结果说明一件很重要的事：即使文档不是最短、词也没有重复很多次，只要查询词足够稀有，并且确实出现了几次，它仍然会拿到较高分。

## 代码实现

实现 BM25 不需要复杂框架。最小流程只有四步：

1. 对每个文档分词并统计长度。
2. 统计每个词出现在多少篇文档中，得到文档频率。
3. 由文档频率计算 IDF。
4. 查询时按公式给每篇文档打分，返回 top-k。

下面给一个可运行的 Python 玩具实现。为了把重点放在 BM25 逻辑上，分词直接用空格切分。在中文真实工程里，你会替换成自己的中文分词器或规则切词器。

```python
import math
from collections import Counter, defaultdict

class BM25:
    def __init__(self, tokenized_docs, k1=1.5, b=0.75):
        self.docs = tokenized_docs
        self.k1 = k1
        self.b = b
        self.N = len(tokenized_docs)
        self.doc_lens = [len(doc) for doc in tokenized_docs]
        self.avgdl = sum(self.doc_lens) / max(self.N, 1)

        self.term_freqs = []
        self.doc_freq = defaultdict(int)

        for doc in tokenized_docs:
            tf = Counter(doc)
            self.term_freqs.append(tf)
            for term in tf:
                self.doc_freq[term] += 1

    def idf(self, term):
        n_q = self.doc_freq.get(term, 0)
        return math.log((self.N - n_q + 0.5) / (n_q + 0.5) + 1)

    def score(self, query_terms, doc_id):
        score = 0.0
        tf = self.term_freqs[doc_id]
        dl = self.doc_lens[doc_id]
        norm = 1 - self.b + self.b * dl / self.avgdl

        for term in query_terms:
            f = tf.get(term, 0)
            if f == 0:
                continue
            score += self.idf(term) * (f * (self.k1 + 1)) / (f + self.k1 * norm)
        return score

    def topk(self, query_terms, k=3):
        scored = [(i, self.score(query_terms, i)) for i in range(self.N)]
        scored.sort(key=lambda x: x[1], reverse=True)
        return scored[:k]

docs = [
    "合同 违约 责任 包括 赔偿 损失".split(),
    "合同 合同 合同 签署 流程".split(),
    "违约 后 证据 保存 通知 义务".split(),
]

bm25 = BM25(docs)
result = bm25.topk("合同 违约".split(), k=3)

# 同时命中两个词的文档应排在最前
assert result[0][0] == 0

# 只刷“合同”的文档不应超过真正同时命中的文档
assert bm25.score("合同 违约".split(), 0) > bm25.score("合同 违约".split(), 1)

# 至少命中一个词的文档分数应大于 0
assert bm25.score("合同".split(), 1) > 0

print(result)
```

这个版本已经覆盖了 BM25 的核心逻辑：`idf()` 计算稀有度，`score()` 做单文档打分，`topk()` 返回候选列表。

如果把它映射到真实工程，一般会多出三层：

| 层 | 职责 |
|---|---|
| 预处理层 | 分词、停用词、字段切分、chunk 切片 |
| 索引层 | 存储倒排索引或词频统计，支持快速召回 |
| 检索层 | query 解析、BM25 打分、top-k、后续 rerank |

真实工程例子：企业法务 RAG。  
一个系统收录合同模板、补充协议、争议处理手册、历史案例摘要。用户问“合同违约后的通知期限怎么算”。第一阶段可以先按段落切片，用 BM25 从几十万段里找出包含“合同”“违约”“通知期限”的候选段。因为这些词在法务语料里术语稳定，BM25 常能迅速把真正相关段落提到前列。随后再用 reranker 判断哪个段落真正在讨论“通知期限”，最后交给 LLM 生成答案。这里 BM25 的价值不是替代 LLM，而是给 LLM 提供更干净的上下文。

## 工程权衡与常见坑

BM25 的公式不复杂，难点在工程配置。真正影响质量的，往往不是公式本身，而是 chunk、分词、字段权重、融合策略。

最常见的坑如下：

| 常见坑 | 现象 | 原因 | 对策 |
|---|---|---|---|
| chunk 太大 | 明明包含关键词，却排不到前面 | 关键词被长文本长度惩罚稀释 | 按段落或语义片段切分 |
| chunk 太小 | 命中了词，但上下文不完整 | 单片段信息不足 | 控制在能独立表达一个事实的粒度 |
| 中文分词不稳 | “违约金”被错切或漏切 | token 质量差 | 自定义词典、规则切词、保留术语 |
| 只索引正文 | 标题信息没参与排序 | 高价值字段缺失 | 标题、摘要、标签单独加权 |
| 只用 BM25 直接喂 LLM | 召回看似高，但答案跑偏 | 词面匹配和语义需求脱节 | 加 rerank 或与向量检索混合 |
| 停用词处理粗糙 | “不支持”“禁止”等关键信号丢失 | 误删功能词或否定词 | 按领域定制停用词表 |
| 参数默认不调 | 某些库表现差 | 语料长度分布与默认值不匹配 | 评估后微调 `k1`、`b` |

有两个权衡最值得单独讲。

第一，chunk 大小。  
如果把整篇 30 页报告当一个 chunk，长度惩罚会很重，查询词也被稀释；如果切得过小，一个句子虽然命中关键词，却可能缺乏支撑上下文。常见实践是按段落、标题块、条款块切分，而不是机械按固定字符数硬切。

第二，BM25 在链路中的角色。  
BM25 更像 recall 层，不像 final rank 层。recall 的白话解释是：宁可先多找一些可能相关的候选，不急着最终定输赢。如果你把 BM25 的 top-3 直接塞给 LLM，系统会出现“召回命中但答案仍不准”的错觉。更稳妥的链路通常是：

`Query -> BM25 top-50 -> Dense retrieval / Rerank -> top-5 -> LLM`

这样做的原因很直接：BM25 擅长精确词面命中，向量检索擅长语义补全，reranker 擅长细粒度排序，LLM 擅长生成。四者职责不同，不应混成一个黑盒。

## 替代方案与适用边界

BM25 最常被拿来和向量检索比较。向量检索的白话解释是：把文本映射成一组数值，让“意思接近”的句子在向量空间里靠得更近，因此能处理同义词、改写和语义近邻。

两者的核心差异如下：

| 维度 | BM25 | 向量检索 |
|---|---|---|
| 匹配基础 | 关键词统计 | 语义相似度 |
| 对同义词 | 弱 | 强 |
| 对精确术语 | 强 | 有时不稳定 |
| 可解释性 | 高 | 相对低 |
| 更新成本 | 低 | 较高，需要重新 embedding |
| 延迟 | 通常更低 | 取决于向量库与 ANN 配置 |
| 适合场景 | 法务、制度、错误码、产品名 | 问答改写、自然语言描述、模糊查询 |

因此，替代方案并不是“用向量替掉 BM25”，而是看业务边界：

- 如果语料术语稳定、用户查询经常带关键词，BM25 仍然值得保留。
- 如果用户表达自由、改写多、同义词多，仅靠 BM25 往往不够。
- 如果系统要求既能精确命中术语，又能补足语义近邻，混合检索通常更稳。

一个简单的混合架构可以写成：

`Query -> Sparse(BM25) + Dense(Embedding ANN) -> Merge -> Rerank -> LLM`

新手可以把它理解为两套网同时捞鱼：  
BM25 那张网孔更规则，擅长抓“名字对得上”的内容；向量检索那张网更擅长抓“说法不同但意思接近”的内容。两者合并后，再用 reranker 做最后排序，效果通常比单独一条链路稳定。

但混合方案也有边界。它不是免费午餐，会带来更多索引、更多在线计算、更多调参点。对于一个几千篇文档的小知识库，如果问题本身就是“某错误码是什么意思”“某合同条款在哪”，单独 BM25 往往已经足够。只有当你观察到明显的语义漏召回，比如用户常说“协议违规”而文档里只写“合同违约”，再上向量补召回才更合理。
