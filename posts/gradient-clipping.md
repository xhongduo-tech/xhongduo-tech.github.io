## 核心结论

梯度裁剪的目标很具体：当某一步反向传播得到的梯度异常大时，把这一步参数更新限制在可控范围内，避免训练直接发散。这里要先区分两个量：

- 梯度 `g`：损失函数对参数的导数，表示“往哪个方向改、改多大”
- 参数更新 `\Delta\theta=-\eta g`：真正执行到参数上的变化量，其中 `\eta` 是学习率

所谓梯度爆炸，本质上不是“梯度有点大”，而是“梯度大到让单步更新失控”。一旦这件事发生，常见现象是：

| 现象 | 表现 |
| --- | --- |
| loss 突然跳高 | 前几步正常，某一步后从平滑下降变成剧烈震荡 |
| 参数更新过猛 | 一次更新把参数推离原本的可行区域 |
| 数值不稳定 | 出现 `Inf`、`NaN`，训练中断 |
| 优化器状态被污染 | 后续即使学习率不变，训练也很难恢复 |

梯度裁剪通常只在“异常步”触发，不会改变大多数正常训练步的行为。主流做法有两类。

按值裁剪限制单个分量：
$$
g_{\text{clip}}=\operatorname{clip}(g,-c,c)
$$

按范数裁剪限制整体大小：
$$
g_{\text{clip}}=g\times \min\left(1,\frac{\tau}{\|g\|_2}\right)
$$

其中
$$
\|g\|_2=\sqrt{\sum_i g_i^2}
$$
表示把整组梯度看成一个向量后，它的 L2 范数，也就是“向量长度”。

工程上更常用的是按范数裁剪，因为它通常保留梯度方向，只缩短更新步长。可以把它理解成：

- 原始梯度方向没变
- 只是把“车速”从危险区降回安全区

例如在 Transformer 配合 AdamW 训练时，若大多数 step 的全局梯度范数在 `0.5` 到 `3` 之间，某次长序列 batch 突然把范数推到 `12`，设置 `\tau=5` 后，这一步会被统一缩放到范数 `5`。训练轨迹可能有波动，但不会因为这一次异常更新直接崩掉。

| 方法 | 触发条件 | 数学形式 | 主要效果 | 典型阈值 |
| --- | --- | --- | --- | --- |
| 按值裁剪 | 某个分量超出区间 | $\operatorname{clip}(g_i,-c,c)$ | 限制局部异常维度 | $c=0.1,1,5$ |
| 按范数裁剪 | 整体范数超阈值 | $g\times \min(1,\tau/\|g\|_2)$ | 保持方向，缩小整体步长 | $\tau=1,5,10$ |

一句话概括：梯度裁剪不是为了把所有梯度都变小，而是为了防止极端大梯度破坏训练动态。

## 问题定义与边界

问题定义可以直接写成更新公式：
$$
\Delta\theta=-\eta g
$$

如果梯度 `g` 很大，那么即使学习率 `\eta` 不算夸张，参数更新 `\Delta\theta` 也可能过大。训练不稳定的直接原因，往往不是“学习率高”或“模型深”，而是两者和梯度幅度一起作用，使得单步更新跨得太远。

为什么深层网络、RNN、长序列 Transformer 更容易出现这个问题？原因在于反向传播不是一次简单求导，而是链式法则在很多层、很多时间步上的连续传递。写成简化形式，可以把总梯度看成多个局部贡献的叠加：
$$
g_{\text{total}}=\sum_{i=1}^{L} g_i
$$

这里的 `L` 可以理解为层数、时间步数，或者若干局部梯度贡献的数量。如果其中某些项很大，总梯度就会突然被放大。更进一步，在真实神经网络里，梯度传播还会涉及雅可比矩阵连乘。若这些矩阵的谱半径偏大，梯度增长会比简单求和更剧烈。

这个机制对新手容易抽象，下面给一个数值直觉。

假设某个参数方向上，5 层网络分别贡献梯度：
$$
[8,\ 11,\ 9,\ 10,\ 12]
$$

那么总梯度大致是：
$$
g_{\text{total}}\approx 8+11+9+10+12=50
$$

如果学习率为
$$
\eta=0.1
$$
那么这一步更新量约为
$$
\Delta\theta=-0.1\times 50=-5
$$

对很多已经接近较优区域的参数来说，一步跳 `5` 通常不是正常优化，而是直接跳出稳定区间。你可以把正常训练理解成“连续小步修正”，而梯度爆炸对应的是“突然大跨步”。

以下场景更容易遇到这个问题：

| 场景 | 是否容易爆炸 | 原因 |
| --- | --- | --- |
| 长序列 RNN | 高 | 时间步长，链式法则连乘更长 |
| 深层 Transformer | 中到高 | 层数多，注意力分布可能尖锐，长上下文会放大波动 |
| 刚开始训练 | 高 | 参数尚未进入稳定区域，激活和梯度分布都在快速变化 |
| 学习率偏高 | 高 | 大梯度再乘大学习率，单步更新容易失控 |
| 浅层 CNN 且归一化充分 | 低 | 梯度传播链条较短，数值分布通常更稳 |

边界也必须说清楚。梯度裁剪解决的是“更新量瞬时过大”，它不解决下面这些问题：

| 问题类型 | 梯度裁剪能否解决 | 原因 |
| --- | --- | --- |
| 数据里有 `NaN/Inf` | 不能 | 输入已经坏了，梯度再裁剪也无效 |
| loss 实现有 bug | 不能 | 例如除零、`log(0)`、错误广播都会直接制造异常值 |
| 混合精度缩放配置错误 | 不能直接解决 | 可能是数值上溢/下溢，不是普通梯度过大 |
| 学习率整体过高 | 只能部分缓解 | 只能拦住异常步，拦不住“每一步都太大” |
| 长尾 batch 偶发异常 | 能较好缓解 | 这是梯度裁剪最典型的适用场景 |

所以，梯度裁剪是稳定训练的安全阀，不是通用修复器。

## 核心机制与推导

梯度裁剪发生在优化器更新之前。顺序是：

1. 前向传播，计算 loss
2. 反向传播，得到梯度
3. 对梯度做裁剪
4. 用裁剪后的梯度更新参数

最常见的两种形式分别是按值裁剪和按范数裁剪。

### 1. 按值裁剪

按值裁剪逐元素处理：
$$
g_i^{\text{clip}}=\min(\max(g_i,-c),c)
$$

意思是：每个分量都不能超过阈值 `c`。如果某一维特别大，它会被直接截断到 `c` 或 `-c`。

例子：
$$
g=[12,-5,0.3,-20],\quad c=6
$$

裁剪后：
$$
g_{\text{clip}}=[6,-5,0.3,-6]
$$

优点是简单直接，特别适合“少数维度出现尖峰”的情况。缺点是会改变向量方向，因为不同分量被压缩的比例不同。

### 2. 按范数裁剪

按范数裁剪先计算整体长度：
$$
\|g\|_2=\sqrt{\sum_i g_i^2}
$$

如果
$$
\|g\|_2\le\tau
$$
则不处理；

如果
$$
\|g\|_2>\tau
$$
则统一缩放：
$$
g_{\text{clip}}=g\times \frac{\tau}{\|g\|_2}
$$

这表示整根向量乘同一个比例，因此方向不变。

为了让这个结论更直观，可以写出方向向量：
$$
u=\frac{g}{\|g\|_2}
$$

那么原始梯度可以写成：
$$
g=\|g\|_2 u
$$

裁剪后变成：
$$
g_{\text{clip}}=\tau u
$$

这说明按范数裁剪只改变长度，不改变方向。这也是它在深度学习训练里更常见的原因。

### 3. 最小数值例子

设
$$
g=[12,-5]
$$

按值裁剪，取 `c=6`：
$$
g_{\text{clip}}=[6,-5]
$$

按范数裁剪，取 `\tau=10`。先算原始范数：
$$
\|g\|_2=\sqrt{12^2+(-5)^2}=\sqrt{169}=13
$$

因为 `13 > 10`，所以按比例缩放：
$$
g_{\text{clip}}=[12,-5]\times \frac{10}{13}\approx [9.23,-3.85]
$$

对比两者：

| 比较项 | 按值裁剪 | 按范数裁剪 |
| --- | --- | --- |
| 控制对象 | 单个分量 | 整体向量 |
| 是否保持方向 | 不一定 | 是 |
| 对局部异常维度 | 更敏感 | 可能不够直接 |
| 对整体更新强度 | 间接控制 | 直接控制 |
| 训练中常见程度 | 次常见 | 更常见 |

还可以再看一次“更新量”差异。若学习率为 `0.1`：

- 不裁剪时，更新为 `[-1.2, 0.5]`
- 范数裁剪后，更新为 `[-0.923, 0.385]`

方向一致，但步长缩短了。这正是裁剪要做的事。

### 4. 为什么工程上更偏好全局范数

真实模型不是一个向量，而是很多层、很多张量。工程里真正关心的不是“某一层看起来合不合格”，而是“这一步总更新量会不会太猛”。因此更常见的是全局范数裁剪，也就是先把所有参数张量的梯度一起看作一个大向量，再计算整体范数。

设模型有 `K` 个参数张量，梯度分别为 `g^{(1)}, g^{(2)}, \dots, g^{(K)}`，则全局范数可以写成：
$$
\|g\|_{\text{global}}=
\sqrt{
\sum_{k=1}^{K}
\|g^{(k)}\|_2^2
}
$$

若全局范数超过阈值 `\tau`，则所有张量统一乘同一个缩放因子：
$$
\text{scale}=\min\left(1,\frac{\tau}{\|g\|_{\text{global}}}\right)
$$

然后
$$
g_{\text{clip}}^{(k)}=\text{scale}\cdot g^{(k)}
$$

这样做的好处是：

| 做法 | 含义 | 风险 |
| --- | --- | --- |
| 每个张量单独裁剪 | 每层各管各的 | 可能每层都合格，但总更新仍偏大 |
| 全局范数裁剪 | 直接限制整步更新强度 | 更符合“防止单步发散”的目标 |

真实工程里，训练一个中等规模 Transformer，batch 中大多数样本长度在 `256` token 左右，偶尔混入接近 `2k` token 的长序列。前 300 步统计显示，全局梯度范数通常落在 `0.7` 到 `2.8`，但有少数 step 跳到 `8` 以上。某次异常 batch 把范数推到 `12`，若直接更新，loss 会从 `2.x` 跳到十几，并伴随溢出告警。加入 `\tau=5` 的全局范数裁剪后，这一步虽然仍异常，但不会把训练轨迹整体毁掉。

## 代码实现

实现位置要固定在反向传播之后、优化器更新之前。不要在前向传播阶段裁剪，因为裁剪的对象不是模型输出，而是用于更新参数的梯度。

先给一个完全可运行的 Python 例子，分别演示：

- 按值裁剪
- 按范数裁剪
- 全局范数裁剪
- 验证裁剪后范数是否满足约束

```python
import math


def clip_by_value(grad, c):
    return [max(-c, min(x, c)) for x in grad]


def l2_norm(vec):
    return math.sqrt(sum(x * x for x in vec))


def clip_by_norm(grad, tau):
    norm = l2_norm(grad)
    if norm == 0.0 or norm <= tau:
        return grad[:]
    scale = tau / norm
    return [x * scale for x in grad]


def global_norm(list_of_grads):
    return math.sqrt(sum(x * x for grad in list_of_grads for x in grad))


def clip_by_global_norm(list_of_grads, tau):
    norm = global_norm(list_of_grads)
    if norm == 0.0 or norm <= tau:
        return [grad[:] for grad in list_of_grads], norm, 1.0

    scale = tau / norm
    clipped = [[x * scale for x in grad] for grad in list_of_grads]
    return clipped, norm, scale


def sgd_step(params, grad, lr):
    return [p - lr * g for p, g in zip(params, grad)]


# 1) 按值裁剪示例
g = [12.0, -5.0]
g_value = clip_by_value(g, c=6.0)
assert g_value == [6.0, -5.0]

# 2) 按范数裁剪示例
g_norm = clip_by_norm(g, tau=10.0)
assert abs(l2_norm(g_norm) - 10.0) < 1e-8

# 同比例缩放，方向保持
scale_0 = g_norm[0] / g[0]
scale_1 = g_norm[1] / g[1]
assert abs(scale_0 - scale_1) < 1e-8

# 3) 模拟参数更新
params = [1.0, -2.0]
lr = 0.1
updated_no_clip = sgd_step(params, g, lr)
updated_clip = sgd_step(params, g_norm, lr)

step_no_clip = l2_norm([a - b for a, b in zip(updated_no_clip, params)])
step_clip = l2_norm([a - b for a, b in zip(updated_clip, params)])
assert step_clip < step_no_clip

# 4) 全局范数裁剪示例：假设模型有两个参数张量
grads = [
    [3.0, 4.0],      # 范数 5
    [12.0, 0.0],     # 范数 12
]

clipped_grads, before_norm, scale = clip_by_global_norm(grads, tau=10.0)
after_norm = global_norm(clipped_grads)

assert before_norm > 10.0
assert abs(after_norm - 10.0) < 1e-8
assert 0.0 < scale < 1.0

print("All assertions passed.")
```

上面这段代码能直接运行，覆盖了裁剪最核心的数值逻辑。

如果放进真实训练循环，PyTorch 写法通常是这样：

```python
import torch
from torch import nn, optim

model = nn.Linear(8, 1)
optimizer = optim.AdamW(model.parameters(), lr=1e-3)
criterion = nn.MSELoss()

x = torch.randn(16, 8)
y = torch.randn(16, 1)

optimizer.zero_grad()
pred = model(x)
loss = criterion(pred, y)
loss.backward()

# 在 optimizer.step() 之前执行
total_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

optimizer.step()

print(f"loss={loss.item():.4f}, grad_norm_before_clip={float(total_norm):.4f}")
```

这段代码的关键点：

| 代码 | 作用 |
| --- | --- |
| `loss.backward()` | 计算梯度 |
| `clip_grad_norm_(...)` | 若全局梯度范数超阈值，则统一缩放 |
| `optimizer.step()` | 用裁剪后的梯度更新参数 |

TensorFlow 中则常见于 `GradientTape` 之后：

```python
import tensorflow as tf

model = tf.keras.Sequential([
    tf.keras.layers.Dense(16, activation="relu"),
    tf.keras.layers.Dense(1),
])

optimizer = tf.keras.optimizers.Adam(learning_rate=1e-3)
x = tf.random.normal((16, 8))
y = tf.random.normal((16, 1))

with tf.GradientTape() as tape:
    pred = model(x, training=True)
    loss = tf.reduce_mean(tf.square(pred - y))

grads = tape.gradient(loss, model.trainable_variables)

# 每个张量单独按范数裁剪
tau = 1.0
clipped_grads = [tf.clip_by_norm(g, clip_norm=tau) for g in grads]

optimizer.apply_gradients(zip(clipped_grads, model.trainable_variables))

print("loss:", float(loss))
```

不过更贴近工程实践的通常是“全局范数裁剪”而不是“逐张量裁剪”，因为全局范数更接近“总更新量”的概念。

### 一个常见实现错误

很多新手会把顺序写成：

1. `optimizer.step()`
2. 再裁剪梯度

这是无效的。因为参数已经更新完了。正确顺序必须是：

1. `backward()`
2. `clip_grad_*`
3. `step()`
4. `zero_grad()`

### 训练日志建议一起加上

如果你已经决定用梯度裁剪，最好同时记录：

- 裁剪前全局梯度范数
- 是否触发裁剪
- 缩放比例
- loss 是否仍然出现异常尖峰

这样你才能判断裁剪是在“偶尔救场”，还是“几乎每一步都在压制训练”。

## 工程权衡与常见坑

最关键的超参数是阈值。阈值太低，训练会被过度限制；阈值太高，异常步又拦不住。

这个权衡可以用一句话理解：梯度裁剪本质上是在给更新步长设置上限。上限设得过紧，相当于隐式降低学习率；上限设得过松，则安全阀几乎不起作用。

推荐做法不是凭感觉直接填 `1` 或 `5`，而是先统计一段训练中的梯度范数分布，再决定阈值。例如先记录前 `200` 到 `1000` 步的全局梯度范数，观察：

- 正常区间在哪里
- 尾部异常值有多大
- 裁剪希望拦住的是前 5% 还是前 1% 的异常步

经验上常把 `\tau` 设在梯度范数分布的 `75%` 到 `95%` 分位附近。这样多数正常更新原样通过，只拦截尾部尖峰。

| $\tau$ | 触发频率 | 收敛速度 | 稳定性 | 适用判断 |
| --- | --- | --- | --- | --- |
| 0.5 | 很高 | 慢 | 很稳，但常过度抑制 | 通常偏低 |
| 2.5 | 低到中 | 较快 | 稳定 | 常见可用区间 |
| 5.0 | 很低 | 快 | 对偶发尖峰有效 | Transformer 常见起点之一 |
| 20.0 | 极低 | 快 | 几乎无保护 | 通常偏高 |

举个更具体的例子。若某次训练前 400 步的全局范数大多在 `0.3` 到 `2.8` 之间：

- 设 `\tau=2.5`，可能只有约 5% 的 step 会触发
- 设 `\tau=0.5`，几乎每一步都会触发
- 设 `\tau=20`，基本相当于没开裁剪

这就是为什么阈值不能脱离分布去讨论。

常见坑主要有下面几类。

### 1. 误以为梯度裁剪能替代学习率调整

不能。若学习率整体偏高，即使没有爆炸，训练也可能持续震荡。梯度裁剪只能拦截异常步，不能修复所有步都太大的情况。

判断方法很直接：

| 现象 | 更像哪类问题 |
| --- | --- |
| 只有少数 step 突然尖峰 | 更像异常 batch 或梯度长尾，裁剪有效 |
| 几乎每一步都震荡 | 更像学习率偏高，优先调学习率 |
| 一开始就连续发散 | 可能是学习率、初始化、实现 bug，而不是单纯爆炸 |

### 2. 只看 loss，不看梯度范数

只看 loss 曲线，很难判断梯度裁剪是否真的起作用。loss 跳动可能来自很多原因，而梯度范数能直接告诉你“更新强度”是否失控。

至少建议记录：

| 指标 | 含义 |
| --- | --- |
| `grad_norm_before_clip` | 原始梯度是否异常大 |
| `grad_norm_after_clip` | 裁剪是否生效 |
| `clip_trigger_rate` | 多久触发一次裁剪 |
| `loss` | 裁剪后训练是否更稳定 |

### 3. 裁剪后仍有 `NaN` 却继续跑

这通常说明根因不只是梯度过大。需要优先排查下面几项：

| 排查项 | 要看什么 | 目的 |
| --- | --- | --- |
| 输入数据 | 是否含 `NaN/Inf`、异常长度、异常标签 | 排除脏数据 |
| loss 计算 | 是否有除零、对负数开方、`log(0)` | 排除数学实现错误 |
| 学习率 | 是否过高、warmup 是否缺失 | 排除整体步长过大 |
| 混合精度 | 是否使用了正确的 loss scaling | 排除数值上溢/下溢 |
| 梯度日志 | 裁剪前后范数是否仍持续爆炸 | 判断裁剪是否真的有效 |

### 4. 不加区分地使用按值裁剪

按值裁剪并不是错，但它更容易扭曲方向。若问题是“整体更新太大”，按值裁剪未必是最佳手段。只有在你有明确证据表明“个别维度经常出现离群尖峰”时，它才更有针对性。

### 5. 混淆“逐张量裁剪”和“全局裁剪”

这两个实现的行为不同。

| 方式 | 优点 | 风险 |
| --- | --- | --- |
| 逐张量裁剪 | 简单，易实现 | 每个张量单独合格，不代表总更新量合格 |
| 全局裁剪 | 直接限制总步长 | 实现上多一步全局统计 |

大型模型训练里，更常见的选择是全局范数裁剪。

### 6. 阈值一旦设定就长期不看

训练不同阶段的梯度分布可能会变化。比如 warmup 后、切换数据阶段后、上下文长度增加后，范数分布都可能改变。阈值不是永远固定的常数，而是一个要结合训练日志复核的工程参数。

## 替代方案与适用边界

梯度裁剪不是唯一稳定训练的方法。它更像安全阀，负责拦截偶发大更新；而很多其他方法是在源头上降低梯度波动。

| 方法 | 适用场景 | 优点 | 局限 |
| --- | --- | --- | --- |
| 学习率 warmup | 训练初期不稳定 | 降低刚开始的更新冲击 | 对后期偶发尖峰不够直接 |
| 学习率衰减 | 中后期震荡 | 逐步减小步长 | 不专门处理异常单步 |
| AdamW | 通用训练 | 自适应步长，通常更稳 | 仍可能遇到极端梯度 |
| LayerNorm / RMSNorm | Transformer | 稳定激活与梯度分布 | 不是专门为爆炸设计 |
| 更稳的初始化 | 深层网络 | 从源头改善传播尺度 | 对训练中后期尖峰无直接保护 |
| 梯度监控但不裁剪 | 浅层稳定网络 | 不干扰优化过程 | 出现爆炸时没有保护 |

实际工程里常见的是组合策略，而不是二选一。例如：

| 场景 | 常见组合 |
| --- | --- |
| 长上下文 Transformer | `warmup + AdamW + 全局范数裁剪` |
| 深层 RNN | `较保守学习率 + 梯度裁剪 + 归一化` |
| 浅层 CNN | `warmup/衰减 + 监控梯度`，必要时再加裁剪 |
| 强化学习高方差训练 | `裁剪 + 归一化 + 更保守的优化器配置` |

边界同样要明确。

如果模型已经具备：

- 充分的归一化
- 保守的学习率策略
- 稳定的初始化
- 梯度监控中几乎没有长尾尖峰

那么“只监控，不裁剪”是合理选择。工程目标不是把所有稳定性技巧都打开，而是在最小干预下获得可重复的训练过程。

相反，下列情况更建议默认打开梯度裁剪：

| 场景 | 建议 |
| --- | --- |
| 长上下文 Transformer | 建议默认启用全局范数裁剪 |
| 深层 RNN/LSTM | 建议默认启用 |
| 强化学习高方差回报网络 | 建议默认启用 |
| 刚开始尝试新模型、新数据管线 | 建议先开，等分布稳定后再评估是否保留 |

所以更准确的说法不是“梯度裁剪该不该用”，而是“你的训练过程是否存在少量异常大更新，且这些异常更新足以破坏整体轨迹”。如果答案是是，那么梯度裁剪就是低成本且高收益的工具。
