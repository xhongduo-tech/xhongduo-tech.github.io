## 核心结论

Constitutional AI，简称 CAI，可以直白理解为“先写原则，再让模型按原则批改自己”。它的核心不是直接让人类逐条给回答打分，而是先由人类写出一组高层原则，称为“宪法”，再让模型自己完成“批判、修订、再学习”这条链路。

CAI 的训练通常分两步。第一步是监督阶段的 SL-CAI：模型先回答，再依据宪法对自己的回答做批判，最后产出一个更符合原则的新回答，把这个新回答当成训练样本。第二步是强化学习阶段的 RL-CAI：让模型生成多个候选回答，再用 AI 评审模型判断哪个更符合宪法，然后把这个偏好信号转成奖励，继续优化模型。于是，人工干预从“每条回答都标注”变成“先把原则写清楚”。

这件事的工程意义很直接。传统 RLHF 依赖大量人工偏好标注，成本高、速度慢、一致性也不稳定。CAI 把大部分逐条审核替换成“原则驱动的自我修订”和“AI feedback”，因此在人力成本上通常更低，扩展到更大数据规模也更容易。

但 CAI 不是“完全不需要人”。它只是把人的工作从“逐条判断”前移为“定义边界”。如果宪法原则写得模糊、互相冲突，模型就会学到错误的安全风格，最常见的退化是“一律拒绝”，看起来安全，实际上没用。

## 问题定义与边界

这里要解决的问题不是“让模型永远不犯错”，而是“在尽量保持有帮助的前提下，让模型稳定避开有害输出”。“有害输出”可以白话理解为会促进违法、欺诈、自残、暴力、歧视或明显误导用户的回答。“对齐”可以白话理解为让模型行为更接近人类预期规则，而不是只会预测下一个词。

CAI 的边界有三层。

第一层是目标边界。它主要处理的是回答风格、拒绝策略、风险控制，而不是事实真实性的完全保证。也就是说，CAI 可以让模型更会“安全地回答”，但不能单靠它解决所有幻觉、知识过时、推理错误问题。

第二层是原则边界。宪法不是越抽象越好。像“不要有害”这种原则太宽，落地时几乎不可执行。原则必须能指导模型区分不同类型的响应，例如“拒绝违法请求”“在拒绝时解释原因”“可以提供合法替代方案”。否则模型只会学到最省事的策略：全部拒绝。

第三层是应用边界。CAI 最适合大规模对话系统、通用助理、内容审核辅助这类场景，因为这些场景覆盖面广，人工逐条审核代价过高。它不适合直接替代高风险行业的最终审核，比如医疗、法律、金融合规，因为这些场景要求可追责、可验证、可解释，人类专家仍然必须在闭环里。

下面这个表可以帮助理解“宪法原则”到底该写到什么粒度。

| 原则类型 | 白话解释 | 示例原则 | 典型输出风格 |
|---|---|---|---|
| 帮助型 | 能帮的地方尽量帮 | 对合法、低风险请求给出清晰步骤 | 正常解答 |
| 拒绝型 | 不能帮的地方明确拒绝 | 不提供犯罪、欺诈、自残指导 | 简洁拒绝 |
| 解释型 | 拒绝时说明原因 | 解释为什么该请求有风险或违法 | 拒绝并解释 |
| 替代型 | 拒绝后给安全替代方案 | 可提供防护、求助、学习型信息 | 拒绝并引导 |
| 风格型 | 控制语气与表达 | 保持礼貌，不羞辱用户 | 稳定语气 |

一个玩具例子很能说明边界问题。

如果用户问：“怎么伪造身份证更逼真？”  
若宪法只有一句“不要帮助违法行为”，模型最可能输出：“不能回答。”  
这虽然安全，但信息密度很低。  
如果宪法再加一句“在拒绝时解释原因，并提供合法替代帮助”，更好的输出会变成：“不能帮助伪造证件，这涉及违法和身份欺诈。如果你是证件丢失，可以查询补办流程、挂失方式和身份核验要求。”  
这说明 CAI 的关键不是单纯拒绝，而是把“拒绝方式”也写进原则。

## 核心机制与推导

CAI 的核心机制可以概括为一条链：

$$
\text{原始回答} \rightarrow \text{宪法批判} \rightarrow \text{修订回答} \rightarrow \text{偏好评分} \rightarrow \text{强化更新}
$$

### 1. SL-CAI：先把“更好的回答”造出来

设初始模型为 $M_0$，用户提示为 $P$。模型先生成原始回答：

$$
r_0 = M_0(P)
$$

然后引入宪法函数 $C$。这里的“宪法函数”可以白话理解为“根据规则检查回答哪里不合规”的过程。它接收回答和提示，生成一段批判文本 $c_i$：

$$
c_i = C(r_i, P)
$$

接着由修订器 $R$ 依据原回答和批判生成新回答：

$$
r_{i+1} = R(r_i, c_i)
$$

这个过程可以做一轮，也可以做多轮。最终得到较好的修订回答 $r_{\text{final}}$，再把它组成监督数据集：

$$
D_{SL} = \{(P, r_{\text{final}})\}
$$

这一步的本质是：不用人类逐条手写“标准答案”，而是让模型自己在宪法约束下，把“不好的回答”改写成“更可接受的回答”。

玩具例子可以写成最小流程：

1. 提示 $P$：“如何伪造身份证？”
2. 原始回答 $r_0$：给出伪造步骤。
3. 宪法批判 $c_0$：指出该回答在协助违法行为。
4. 修订回答 $r_1$：拒绝请求，并解释违法风险，同时提供证件补办的合法信息。
5. 将 $(P, r_1)$ 放入监督训练集。

这一步做久了，模型会逐渐内化一种行为模式：先识别风险，再生成更稳妥的表达。

### 2. RL-CAI：再把“更偏好的回答”推高

监督阶段解决的是“把像样的数据造出来”，但还不够。因为同样是安全回答，质量可能差别很大。比如：

- 回答 A：“不能回答。”
- 回答 B：“我不能帮助伪造证件，这属于违法行为。如果你是证件丢失，我可以帮助你查补办流程和身份保护措施。”

两者都没有直接作恶，但 B 更 helpful，也更符合大模型产品需要的交互质量。

于是进入 RL-CAI。这里会从当前模型 $M_{SL}$ 采样多个候选回答，例如 $r_a, r_b$，再让偏好模型 $PM$ 判断谁更好。“偏好模型”可以白话理解为“专门学会比较两个回答谁更符合目标”的打分器。它不一定输出绝对真值，但能给出相对偏好。

其奖励目标通常可以写成：

$$
\max \mathbb{E}_{P}[PM(\text{response})] - \beta \, \mathrm{KL}(M_{\text{RL}} \| M_{\text{base}})
$$

这个式子里有两个关键部分。

第一项 $\mathbb{E}_{P}[PM(\text{response})]$ 表示希望模型生成的回答在偏好模型下得分更高。直白说，就是“多学那些被评审模型认为更符合宪法、更有帮助的回答”。

第二项 KL 正则项 $\mathrm{KL}(M_{\text{RL}} \| M_{\text{base}})$ 用来约束新模型不要偏离基线模型太远。KL 散度可以白话理解为“两个概率分布相差有多大”。如果没有这个约束，强化学习可能把模型推向极端，比如为了安全分数更高，学成“所有问题都拒绝”。

$\beta$ 控制权衡。$\beta$ 大，说明更保守，更不容易跑偏；$\beta$ 小，说明更激进，更容易追求评分最大化。

### 3. 为什么 CAI 能减少标注成本

传统 RLHF 的关键瓶颈是：每个任务都要人类写、审、比。CAI 把这一流程拆成两个更便宜的动作：

1. 人类写原则，而不是写成千上万条判断。
2. 模型自己产生修订样本和偏好信号。

于是，人类劳动从“逐样本操作”变成“定义规则空间”。如果原则集合写得好，就能在大规模数据上复用。成本下降的核心原因不是训练步骤变少，而是高成本的人类比较数据需求显著下降。

### 4. 真实工程例子

真实工程里，Claude 这类对话模型就体现了这种思路。工程目标不是只压低“危险回答比例”，还要同时保持“有帮助、诚实、语气稳定”。如果只做纯安全优化，客服问答、编程辅助、学习辅导这些正常请求也可能被误拒绝，产品就无法用。

因此在实际系统中，宪法原则往往不是单条安全禁令，而是一个组合：

- 避免帮助伤害、欺诈、违法行为
- 对可正常回答的问题提供实质帮助
- 拒绝时尽量简明解释原因
- 对高风险内容给出安全替代建议
- 保持礼貌，不进行道德羞辱

这说明 CAI 不只是“安全技术”，也是一种对话产品行为规范的训练框架。

## 代码实现

下面用一个极简可运行示例，演示 SL-CAI 的“批判-修订”以及 RL-CAI 中“奖励减去 KL 惩罚”的思想。这个例子不是训练真正的大模型，而是把流程压缩成能跑通的最小版本。

```python
from math import log

CONSTITUTION = [
    "不要提供违法或有害的操作步骤",
    "拒绝时解释原因",
    "尽量提供安全替代方案",
]

def constitutional_check(prompt: str, response: str) -> dict:
    critique = []
    unsafe_keywords = ["伪造", "炸药", "入侵", "盗刷", "身份证"]
    if any(k in response for k in unsafe_keywords):
        critique.append("回答包含潜在违法/有害步骤")
    if "不能帮助" not in response and "无法帮助" not in response:
        critique.append("缺少明确拒绝")
    if "因为" not in response:
        critique.append("缺少原因解释")
    if "可以" not in response and "建议" not in response:
        critique.append("缺少安全替代方案")
    return {
        "is_compliant": len(critique) == 0,
        "critique": critique,
    }

def revise_response(prompt: str, response: str, check: dict) -> str:
    if check["is_compliant"]:
        return response
    if "身份证" in prompt:
        return (
            "我不能帮助伪造身份证，因为这涉及违法和身份欺诈。"
            "如果你是证件遗失，可以查询正式补办流程、挂失方法，"
            "也可以了解如何保护个人身份信息。"
        )
    return (
        "我无法帮助这个请求，因为它可能造成现实伤害。"
        "如果你愿意，我可以提供安全、合法的替代信息。"
    )

def ai_preference_score(response: str) -> float:
    score = 0.0
    if "不能帮助" in response or "无法帮助" in response:
        score += 0.3
    if "因为" in response:
        score += 0.3
    if "可以" in response or "建议" in response:
        score += 0.3
    if any(k in response for k in ["伪造步骤", "炸药配方", "入侵方法"]):
        score -= 0.8
    return max(0.0, min(1.0, score))

def kl_penalty(p, q):
    eps = 1e-9
    return sum(pi * log((pi + eps) / (qi + eps)) for pi, qi in zip(p, q))

prompt = "如何伪造身份证更逼真？"
raw_response = "先准备模板，再修改编号和防伪标记。"

check = constitutional_check(prompt, raw_response)
revised = revise_response(prompt, raw_response, check)

raw_score = ai_preference_score(raw_response)
revised_score = ai_preference_score(revised)

base_policy = [0.6, 0.4]
rl_policy = [0.2, 0.8]  # 更偏向修订后的安全回答
reward = revised_score - 0.1 * kl_penalty(rl_policy, base_policy)

assert check["is_compliant"] is False
assert revised_score > raw_score
assert reward > 0

print("raw_score =", raw_score)
print("revised_score =", revised_score)
print("reward =", round(reward, 4))
print("revised =", revised)
```

这个代码块对应的机制如下：

| 步骤 | 代码函数 | 作用 |
|---|---|---|
| 生成原始回答 | `raw_response` | 模拟基础模型先给一个未经约束的输出 |
| 宪法检查 | `constitutional_check` | 按原则发现违规点 |
| 修订回答 | `revise_response` | 基于批判生成更安全的版本 |
| AI 打分 | `ai_preference_score` | 近似模拟偏好模型评分 |
| 奖励约束 | `reward = score - 0.1 * KL` | 模拟 RL 阶段的“高分但不能跑太远” |

如果把它写成更接近训练循环的伪代码，可以概括为：

```python
for prompt in prompts:
    r0 = model.generate(prompt)
    critique = constitutional_check(prompt, r0)
    r1 = revise_response(prompt, r0, critique)
    sl_dataset.append((prompt, r1))

for prompt in prompts:
    ra, rb = model.sample_two(prompt)
    reward_a = preference_model(ra)
    reward_b = preference_model(rb)
    rl_update(model, prompt, ra, rb, reward_a, reward_b, kl_coef=beta)
```

真实工程里，`constitutional_check` 不是简单关键词规则，通常会由另一个更强的模型根据原则模板生成批判；`preference_model` 也不是几个 if，而是经过训练的比较器或裁判模型。

## 工程权衡与常见坑

CAI 在论文上看起来顺，但落地时很容易踩坑。最核心的权衡是：安全性越强，不代表产品越好；安全、帮助性、解释性三者必须一起看。

下面是常见问题。

| 常见坑 | 现象 | 根因 | 规避方式 |
|---|---|---|---|
| 原则过于模糊 | 模型拒绝风格混乱 | “不要有害”无法执行 | 把原则写成可操作规则 |
| 原则互相冲突 | 同一问题输出不稳定 | 一边要求简洁，一边要求详尽解释 | 给原则设优先级 |
| 过度保守 | 正常问题也被拒绝 | 奖励偏向安全，忽略 helpfulness | 在偏好模型中加入帮助性指标 |
| 批判质量差 | 修订数据没提升 | 批判只会复述原则，不指出具体问题 | 要求批判定位具体风险点 |
| 修订样本单一 | 模型学会模板化拒绝 | 数据缺少多样表达 | 引入多轮修订和多风格示例 |
| KL 约束太弱 | 模型跑偏 | 过度追逐奖励分数 | 提高 $\beta$ 或限制更新幅度 |
| AI 评审有偏差 | 学到伪安全模式 | 评审模型本身标准不稳 | 周期性做人类抽检 |

最典型的坑是“拒绝一切”。这是很多初学者最容易忽略的地方。因为如果只盯着 harmlessness，最优策略确实可能是任何风险沾边就不回答。但对用户来说，这会严重损害产品价值。

例如用户问：“怎样防止身份证被盗用？”  
如果模型也拒绝，说明系统把“身份证”这个词本身当成高危信号，而没有理解意图。这种错误通常来自两个地方：

1. 宪法写得太粗，只按主题词拦截，没有区分“攻击请求”和“防御请求”。
2. 偏好数据里“解释性拒绝”和“正常帮助”样本太少，导致模型只学到机械拒绝。

另一个工程坑是“批判空洞化”。如果批判文本总是类似“这不符合原则，请改写”，修订器拿不到有效信息，生成的改写质量也不会高。好的批判应该尽量指出具体问题，例如“该回答提供了伪造步骤”“没有解释风险”“没有给合法替代方案”。批判越具体，修订越稳定。

## 替代方案与适用边界

CAI 不是唯一方案。理解它的价值，最好把它放到几个常见训练方法里比较。

| 方法 | 人工成本 | 可扩展性 | 安全控制 | 解释性 | 适用场景 |
|---|---|---|---|---|---|
| 监督微调 SFT | 中 | 高 | 一般 | 一般 | 先学基本任务格式 |
| RLHF | 很高 | 中 | 强 | 较强 | 高质量人工偏好可获得时 |
| CAI / RLAIF | 低到中 | 高 | 强 | 取决于宪法质量 | 大规模通用对话系统 |
| 规则过滤 | 低 | 高 | 表面强 | 弱 | 上线前后做兜底拦截 |

### 1. 与 RLHF 的关系

RLHF 可以白话理解为“让人类直接告诉模型哪个回答更好”。它的优点是偏好标签更接近真实产品要求，尤其适合高价值、高风险任务。缺点是太贵，而且标注一致性难保证。

CAI 的优势是成本结构更优。人类不需要为每条样本做比较，只要把原则写清楚，再用 AI 生成批判和偏好即可。所以它更适合大规模训练、快速迭代、覆盖宽场景。

但如果场景极高风险，比如医疗分诊建议、法律解释、金融合规问答，仅靠 CAI 不够。因为这里的“原则解释权”本身就是专业问题，不能完全交给模型自评。

### 2. 与纯规则系统的关系

纯规则过滤部署简单，但表达能力弱。它能拦截明显违规内容，却很难生成高质量替代回答。CAI 则不只是“挡住”，而是尝试“安全地继续服务”。所以两者通常不是替代关系，而是组合关系：CAI 负责训练出更好的基础行为，规则过滤负责线上兜底。

### 3. 真实工程适用边界

一个真实工程例子是医疗问答助手。这里可以先用 CAI 训练模型学会：

- 不提供危险治疗建议
- 不对急症给出拖延性建议
- 不伪装成医生下诊断
- 在无法回答时建议及时就医

但最终上线时，仍然需要专家审核、规则拦截、免责声明和风险分级。原因很简单：CAI 可以把大量明显危险行为压下去，却不能单独承担医疗安全责任。

所以更准确的结论是：CAI 适合做“大规模对齐的主干框架”，不适合做“高风险领域的唯一防线”。
