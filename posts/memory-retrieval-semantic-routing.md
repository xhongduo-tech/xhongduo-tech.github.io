## 核心结论

当 Agent 同时维护事实记忆、对话记忆、技能记忆等多个索引时，不能默认“把查询丢给所有库再统一排序”就是最稳的方案。更常见的结果是噪声增多、延迟上升、错误召回混进上下文，最终让回答更像“知道很多，但答非所问”。

语义路由就是在检索前先做一次分流。白话说，它先判断“这句话更像在问什么”，再决定该查哪类记忆。一个实用流水线通常是两步：

1. 先做意图路由，选出最可能相关的子记忆。
2. 再在该子记忆内执行专用检索策略。

例如一个有“事实 + 对话 + 技能”三类索引的 Agent，面对“退款政策里配件能退吗”，应该优先唤起事实记忆；面对“你上次答应帮我整理会议纪要”，应该优先唤起对话记忆；面对“帮我按公司模板写周报”，则更像技能记忆。先分流，再检索，通常比全库盲检更准。

需要特别纠正一个常见误读：iCARE 论文里常被转述的 `92.19%`，对应的是 **3-shot 的 LLM-as-Router** 在消歧后文本上的准确率；同文中的 **ProtoKNN 3-shot** 最好结果是 `73.44%`。这不意味着原型路由没价值，恰好相反，论文同时指出 ProtoKNN 延迟更低，适合作为低成本前置路由，再把低置信度样本交给更重的 LLM 校验。

| 意图类型 | 目标索引 | 典型内容 | 示例检索策略 |
| --- | --- | --- | --- |
| 事实查询 | 事实记忆 | 政策、参数、定义、配置项 | 向量检索 + BM25 混合召回 |
| 上下文追问 | 对话记忆 | 最近几轮对话、承诺、偏好 | 时间排序 + thread 过滤 |
| 执行动作/方法 | 技能记忆 | SOP、模板、工具使用步骤 | 标签过滤 + 过程片段召回 |

## 问题定义与边界

问题不是“能不能检索到”，而是“该先去哪一类记忆里检索”。

多索引系统里，索引本身代表不同语义职责。事实记忆保存稳定知识；对话记忆保存会话历史；技能记忆保存可执行流程。它们都可能与同一查询在字面上相似，但只有一类最接近用户真实意图。

如果不做路由，系统就会把多个库一起搜出来，再希望重排器自己消化差异。这在小规模 demo 可以工作，在真实工程里会遇到三类问题：

1. 噪声：技能文档里出现“退款”二字，不代表它比政策条款更适合回答退款规则。
2. 延迟：并行查询所有索引，成本随索引数近似线性增长。
3. 可解释性差：答错时很难知道是“检索错”还是“查错库”。

Azure 关于 agent memory 的资料本质上就在讲这个边界：有些场景优先按时间取最近记忆，有些场景优先按语义取相似记忆，还有些场景需要混合检索。它没有说“所有记忆统一一个查法”，而是在提醒你：**记忆类型不同，访问模式就应不同**。

题设给了一个经验关系：误路由率每降低 `1%`，任务完成率大约提升 `2%`。这更适合当作工程经验式，而不是通用物理定律。可以写成：

$$
\Delta Success \approx -2 \times \Delta Error_{route}
$$

如果误路由率从 `8%` 降到 `7%`，则任务完成率可近似提升 `2` 个百分点。这个式子不保证在所有系统都成立，但它强调了一个事实：在多记忆系统中，前面的路由错误会被后面的检索和生成放大。

玩具例子很直观：

- 查询 A：“帮我查一下 PostgreSQL 默认端口”
- 查询 B：“你刚才说的默认端口是多少来着”
- 查询 C：“帮我写个连接数据库的排障步骤”

三句话都包含“端口”或“数据库”，但 A 更像事实记忆，B 更像对话记忆，C 更像技能记忆。只靠关键词，很容易混。

## 核心机制与推导

最常见的语义路由做法，是为每个路由维护一组代表性示例，也叫意图原型。原型这个词的白话解释是：它不是全部样本，只是这一类问题的“中心样子”。

设查询为 $q$，某个路由为 $r$，该路由有若干示例 utterance。先把文本编码成向量，再计算查询与该路由原型的相似度：

$$
score_r = \cos\left(emb(q), \frac{1}{|U_r|}\sum_{u \in U_r} emb(u)\right)
$$

其中：

- $emb(\cdot)$ 是文本嵌入模型，把句子变成向量。
- $\cos$ 是余弦相似度，白话说就是“两个向量方向有多像”。

然后对所有路由求分数，取 top-1、top-k，或设阈值分发。

一个简化示意：

```text
                [事实路由中心]
                     ^
                     |
         q1 ---------|
                     |
[技能路由中心] ------ + ------ [对话路由中心]
                     |
         q2 ---------|
```

- `q1 = "退款政策里配件能退吗"` 更靠近事实路由中心。
- `q2 = "按我们上次说的模板继续写"` 更靠近对话或技能路由中心，具体取决于原型设计。

再看一个数值化玩具例子。假设：

- “事实路由”原型由“查参数”“查定义”“查规则”等句子平均得到。
- “诗歌路由”原型由“写诗”“写一首短诗”“来个押韵句子”等句子平均得到。

对查询“帮我写个诗”，得到：

- `score_诗歌 = 0.91`
- `score_事实 = 0.28`

对查询“帮我查事实”，得到：

- `score_事实 = 0.88`
- `score_诗歌 = 0.19`

这就是语义路由的最小闭环。

真实工程例子可以用客服 Agent。用户问“我上周买的耳机能不能退”。这句话同时含有：

- “上周买的”这种时间线索，像对话或订单上下文；
- “能不能退”这种规则判断，像事实记忆；
- 若系统还支持“发起退款”，又会牵涉动作流。

因此工程上常把路由拆成两层：

1. 一级判断主索引：事实 / 对话 / 技能。
2. 二级判断是否需要多路并发，或进入 LLM 校验。

iCARE 的结果说明了一个关键点：当输入先经过消歧，路由准确率会明显提高。也就是说，语义路由并不只是“算相似度”，前面还可以加轻量澄清，比如把“140 正常吗”补成“血压 140 正常吗”，再送去路由。

## 代码实现

下面给一个可运行的最小 Python 版本。它没有依赖真实 embedding 服务，而是用手工向量模拟“意图原型 + 余弦相似度 + 阈值 fallback”的流程，便于理解。

```python
from math import sqrt

def cosine(a, b):
    dot = sum(x * y for x, y in zip(a, b))
    na = sqrt(sum(x * x for x in a))
    nb = sqrt(sum(y * y for y in b))
    return dot / (na * nb)

# 这里假设向量维度含义分别偏向：事实、对话、技能
ROUTES = {
    "facts": {
        "prototype": [0.95, 0.10, 0.05],
        "index": "facts_index",
    },
    "dialogue": {
        "prototype": [0.10, 0.95, 0.15],
        "index": "dialogue_index",
    },
    "skills": {
        "prototype": [0.10, 0.20, 0.95],
        "index": "skills_index",
    },
}

def fake_embed(text: str):
    text = text.lower()
    if any(k in text for k in ["规则", "政策", "定义", "参数", "默认", "事实"]):
        return [0.92, 0.12, 0.08]
    if any(k in text for k in ["刚才", "上次", "之前", "你说过", "聊天"]):
        return [0.10, 0.93, 0.12]
    if any(k in text for k in ["步骤", "模板", "怎么做", "流程", "写个"]):
        return [0.12, 0.18, 0.91]
    return [0.45, 0.40, 0.35]  # 模糊查询

def route_query(query: str, threshold: float = 0.80):
    q_emb = fake_embed(query)
    scored = []
    for route_name, route in ROUTES.items():
        s = cosine(q_emb, route["prototype"])
        scored.append((route_name, route["index"], round(s, 4)))
    scored.sort(key=lambda x: x[2], reverse=True)

    winners = [x for x in scored if x[2] >= threshold]
    return winners if winners else [("fallback", "llm_router", scored[0][2])]

assert route_query("帮我查一下 PostgreSQL 默认端口")[0][0] == "facts"
assert route_query("你刚才说的默认端口是多少来着")[0][0] == "dialogue"
assert route_query("给我一个数据库排障步骤")[0][0] == "skills"

print(route_query("退款政策是什么"))
print(route_query("你上次怎么回答我的"))
print(route_query("按模板写个周报"))
```

生产版流程通常是：

1. 维护每个意图的示例集。
2. 离线计算原型向量，存入配置。
3. 在线把查询嵌入为向量。
4. 计算所有路由分数。
5. 若 top-1 高于阈值，直接分发。
6. 若 top-1 与 top-2 接近，走多路分发或 LLM 校验。

对应的 JavaScript 伪代码如下：

```ts
async function routeQuery(query) {
  const qEmb = await embed(query);
  const scores = routes.map(r => ({
    name: r.name,
    score: cosine(qEmb, mean(r.examplesEmbeddings))
  }));

  const winners = selectTopK(scores, { k: 2, threshold: 0.82 });

  if (winners.length > 0) return winners;
  return [{ name: "fallback_llm_router", score: scores[0].score }];
}
```

这里最重要的不是代码复杂度，而是原型数据的维护方式。示例集应该来自真实日志，而不是只靠工程师拍脑袋写几句“标准问法”。

## 工程权衡与常见坑

最大的坑不是模型不够强，而是样本不够像真实用户。

短句、否定句、省略句最容易误路由。比如“不是让我写步骤，我是问规则”同时触发技能词和事实词；“就按上次那个来”如果脱离线程上下文，几乎无法仅靠文本判断。

另一个常见坑是把路由类别切得太细。类别太多时，路由边界会重叠，准确率反而下降。对初级工程师来说，先把一级路由控制在 `3-7` 类，通常比一开始就拆成二十类更稳。

下面是三类实现的典型权衡：

| 方案 | 准确率潜力 | 延迟 | 成本 | 常见问题 |
| --- | --- | --- | --- | --- |
| 无路由，直接全库检索 | 中 | 高 | 中到高 | 噪声大，难解释 |
| 单次 LLM 路由 | 高 | 中到高 | 高 | 成本波动，输出不稳定 |
| 预嵌入原型路由 | 中到高 | 低 | 低 | 新意图覆盖不足 |

BSWEN 的例子本质上就是在说明：单一大 Prompt 让一个 Agent 同时处理诗歌、故事、笑话、问答时，行为会不稳定；拆成“分类器 + 专用子链”后，系统更容易调试。多记忆检索也是同一个逻辑：**不要让一个检索器在没有边界信息的前提下同时理解所有记忆职责**。

工程上建议至少做三件事：

1. 给每条路由保存置信度与命中日志。
2. 维护 fallback 路径，接收模糊查询。
3. 每周从误路由样本里反向补示例。

## 替代方案与适用边界

语义路由不是唯一答案，它只是多索引记忆里很常用的一层。

| 方案 | 适合场景 | 优点 | 局限 |
| --- | --- | --- | --- |
| 规则匹配路由 | 意图空间固定、可枚举 | 简单、可控、便宜 | 泛化差，维护硬 |
| LLM 直接路由 | 原型期、类别常变 | 灵活、零样本能力强 | 成本高，延迟不稳 |
| 预嵌入原型路由 | 低延迟、类别稳定 | 快、便宜、易部署 | 依赖示例覆盖 |
| 端到端多索引统一检索 | 索引少、数据同质 | 架构简单 | 噪声与成本会放大 |
| 混合路由 | 中大型生产系统 | 稳定性最好 | 实现更复杂 |

适用边界也要说清楚。

语义路由最适合：

- 记忆类型相对固定；
- 能持续收集真实查询日志；
- 系统对延迟敏感；
- 允许把复杂样本交给 fallback。

它不太适合：

- 意图空间每天都在变；
- 几乎没有标注或示例可积累；
- 查询天然是多意图混合，且必须一步完成；
- 数据规模太小，小到全库检索也几乎没有噪声。

因此更稳妥的结论不是“以后都用语义路由”，而是：**把语义路由当成默认前置层，把 LLM 校验当成低置信度兜底层**。这通常比二选一更符合工程现实。

## 参考资料

1. Intent Recognition and Auto-Routing in Multi-Agent Systems
作用：总结了 LLM 路由、专用路由器、语义路由的常见实现方式，适合作为入门全景。

2. Microsoft Learn: Agent memories in Azure Cosmos DB for NoSQL
作用：说明不同记忆类型对应不同检索模式，例如按时间取最近记忆、按语义取相似记忆、按关键词做精确过滤，支持“先路由再检索”的工程设计。

3. iCARE: Ontology-Guided Intent Routing for Multi-Agent LLM-Based Dialogue Systems
作用：给出一个更严格的实验背景。关键结论有两点：一是消歧后的输入能显著提高路由效果；二是 3-shot 的 GPT-4o mini 在消歧文本上达到 `92.19%` 准确率，而 ProtoKNN 在同设定下最好为 `73.44%`，但延迟更低，适合作为轻量前置路由。

4. BSWEN: AI Agent Routing: A Practical Guide to Intent Classification and Routing Implementation
作用：提供了“先分类、再交给专用子模块”的工程示例，说明为什么单一大 Prompt 在多任务场景下容易失稳。

5. Emergent Mind: Retrieval-Augmented Routing
作用：把路由问题放到更一般的检索增强框架里讨论，强调目标不是只追求准确率，而是在准确率、延迟、计算成本之间寻找最优路径。
