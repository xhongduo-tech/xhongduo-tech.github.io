## 核心结论

指令数据去重与多样性控制，是指在指令微调前先删除重复、近重复、模板化和低增益样本，再按任务类型、领域、难度重新控制数据分布。它的目标不是“数据尽量多”，而是“样本尽量少而有效”。

去重解决的是“模型反复学习同一模式”的问题。多样性控制解决的是“模型只会少数高频任务”的问题。两者应放在同一条数据流水线上：先去重，再采样；先减少冗余，再调整分布。

玩具例子：

- `把下面句子改写得更正式`
- `将下列句子转换为正式表达`

这两条指令表达不同，但任务意图相同，都是“正式化改写”。它们应视为近重复样本，只保留一条。

同时，下面这条指令不应被删掉：

- `给出 SQL 左连接查询`

它与“改写”任务距离很远，属于不同任务类型。即使它在数据集中数量少，也可能是有价值的长尾样本。长尾样本是指出现频率低、但对覆盖真实能力有帮助的数据。

| 样本类型 | 判断依据 | 处理方式 |
|---|---|---|
| 重复样本 | 文本几乎完全相同 | 删除，只保留一条 |
| 近重复样本 | 语义和任务意图高度相同 | 聚类后保留代表样本 |
| 长尾样本 | 数量少但任务独立 | 保留，并设置最小配额 |
| 目标保留样本 | 符合目标任务、质量高、覆盖稀缺能力 | 优先保留 |

## 问题定义与边界

指令微调，是用“指令、输入、回答”形式的数据继续训练模型，使模型更好地遵循任务要求。这里讨论的对象是指令微调数据，不是通用预训练语料。预训练语料更关注语言覆盖和规模，指令微调数据更关注任务覆盖、表达方式覆盖和难度覆盖。

去重关注样本之间是否过于相似。多样性关注采样后的整体分布是否合理。二者边界不同：

- 去重问的是：这两条样本是不是在教模型同一件事？
- 多样性问的是：最终留下的数据是否覆盖了目标能力？

例如：

- `改写成正式表达`
- `转成正式写法`

这两条回答内容可能不同，但任务意图相同，属于近重复。

再看：

- `请总结以下内容`
- `请翻译以下内容`

这两条模板很像，都是“请 + 动词 + 以下内容”，但任务语义不同。它们属于模板重复风险，但不一定是语义重复。模板是指一批样本共享的固定句式或结构。

| 边界项 | 定义 | 是否直接删除 |
|---|---|---|
| 语义重复 | 指令含义和任务意图高度一致 | 通常删除或聚类保留 |
| 模板重复 | 指令句式高度相似 | 不一定删除，需结合任务意图 |
| 低质量样本 | 指令不清、答案错、格式乱、不可执行 | 应删除 |
| 长尾样本 | 低频但有独立能力价值 | 应保留 |
| 真实业务分布偏移 | 数据比例与目标使用场景不一致 | 通过采样修正 |

真实工程例子：做客服/办公助手时，原始数据中可能有大量“总结一段话”“把语气改正式”“提取关键信息”。如果不处理，模型会在这些高频任务上表现不错，但遇到“根据表格判断异常原因”“生成 SQL 查询”“诊断接口错误”时能力不足。问题不是模型完全没学会语言，而是指令数据分布把模型推向了少数任务模式。

## 核心机制与推导

一个稳妥的工程流程通常包含三层去重机制：语义相似度、ROUGE-L 字符串重叠、模板规则。

语义相似度，是把文本转换成向量后比较方向是否接近。向量是数字列表，用来表示文本含义。设第 \(i\) 条指令为 \(x_i\)，它的嵌入向量为 \(e_i\)，则两条指令的语义相似度为：

$$
s_{ij}=\cos(e_i,e_j)
$$

这里的 \(\cos(e_i,e_j)\) 是余弦相似度，值越高，表示两个文本在语义空间中越接近。

ROUGE-L 用最长公共子序列衡量文本重叠。最长公共子序列是两个字符串中按原顺序共同出现的最长片段，不要求连续。设 \(\mathrm{LCS}\) 是两条文本的最长公共子序列长度，则：

$$
P=\frac{\mathrm{LCS}}{|x_i|},\quad R=\frac{\mathrm{LCS}}{|x_j|},\quad R_L=\frac{2PR}{P+R}
$$

其中 \(P\) 是相对第一条文本的覆盖比例，\(R\) 是相对第二条文本的覆盖比例，\(R_L\) 是二者的调和平均。它适合发现“换少量词但结构几乎一样”的近重复。

模板命中 \(t_{ij}\) 用于兜底。它通常取 \(0\) 或 \(1\)：如果两条指令命中同一组模板规则，则为 \(1\)，否则为 \(0\)。例如“请总结以下内容”“请概括以下文本”可能都命中“摘要类模板”。

综合重复分数可以写成：

$$
d_{ij}=\alpha s_{ij}+\beta R_L+\gamma t_{ij}
$$

其中 \(\alpha,\beta,\gamma\) 是权重，表示语义、字符串重叠、模板规则的重要程度。若：

$$
d_{ij}\ge \tau
$$

则认为两条样本重复或近重复，\(\tau\) 是阈值。

最小数值例子：

- \(x_1\)：`把下面句子改写得更正式`
- \(x_2\)：`将下列句子转换为正式表达`

假设 \(\cos(e_1,e_2)=0.95\)，\(R_L(x_1,x_2)=0.87\)，模板命中 \(t_{12}=1\)，取 \(\alpha=0.5,\beta=0.3,\gamma=0.2\)，则：

$$
d_{12}=0.5\times0.95+0.3\times0.87+0.2\times1=0.936
$$

若阈值 \(\tau=0.90\)，这对样本应去重，只保留一条。另一条 SQL 指令与改写指令的综合分数可能只有 \(0.14\)，应保留。

多样性控制的本质是分桶采样。分桶是把样本按属性归入不同组，例如任务类型、领域、难度。设分桶变量为 \(z\)，采样后的分布为 \(p_S(z)\)，目标分布为 \(q(z)\)。目标是让：

$$
p_S(z)\approx q(z)
$$

工程上可以用配额采样，也可以最小化分布差异，例如：

$$
D_{\mathrm{KL}}(p_S\|q)
$$

KL 散度用于衡量两个分布的差异。它越小，说明采样后的数据分布越接近目标分布。但这里的目标不是机械平均，而是贴近训练目标或业务目标。

## 代码实现

实现顺序应是：

数据清洗 -> 向量化 -> 近邻检索 -> 相似度打分 -> 去重 -> 分桶采样。

不要把去重和重采样混在一起做。去重阶段决定“哪些样本冗余”，采样阶段决定“哪些样本比例合适”。如果顺序混乱，很容易先抽走少量长尾样本，再错误地认为它们不重要。

| 步骤 | 输入 | 输出 | 用途 |
|---|---|---|---|
| 数据清洗 | 原始样本 | 规范化文本 | 删除空样本、坏格式、明显错误 |
| 向量化 | 指令文本 | embedding | 支持语义近邻检索 |
| 近邻检索 | embedding 集合 | 候选相似样本对 | 避免全量两两比较 |
| 相似度打分 | 样本对 | 综合重复分数 | 判断重复或近重复 |
| 去重 | 重复样本簇 | 代表样本 | 删除冗余 |
| 分桶采样 | 去重后样本 | 最终训练集 | 控制任务、领域、难度分布 |

下面是一段可运行的最小 Python 代码。为了避免依赖外部模型，示例使用字符集合构造玩具向量。真实工程中应替换为 Sentence-BERT、embedding API 或内部向量模型。

```python
from collections import Counter, defaultdict
from math import sqrt

samples = [
    {"id": 1, "text": "把下面句子改写得更正式", "task": "rewrite", "domain": "office", "difficulty": "easy"},
    {"id": 2, "text": "将下列句子转换为正式表达", "task": "rewrite", "domain": "office", "difficulty": "easy"},
    {"id": 3, "text": "给出 SQL 左连接查询", "task": "sql", "domain": "database", "difficulty": "medium"},
]

cfg = {
    "alpha": 0.5,
    "beta": 0.3,
    "gamma": 0.2,
    "tau": 0.72,
}

def embed(text):
    return Counter(ch for ch in text if not ch.isspace())

def cosine(a, b):
    keys = set(a) | set(b)
    dot = sum(a[k] * b[k] for k in keys)
    na = sqrt(sum(v * v for v in a.values()))
    nb = sqrt(sum(v * v for v in b.values()))
    return dot / (na * nb) if na and nb else 0.0

def lcs_len(a, b):
    dp = [[0] * (len(b) + 1) for _ in range(len(a) + 1)]
    for i, ca in enumerate(a, 1):
        for j, cb in enumerate(b, 1):
            dp[i][j] = dp[i - 1][j - 1] + 1 if ca == cb else max(dp[i - 1][j], dp[i][j - 1])
    return dp[-1][-1]

def rouge_l(a, b):
    lcs = lcs_len(a, b)
    p = lcs / len(a) if a else 0
    r = lcs / len(b) if b else 0
    return 2 * p * r / (p + r) if p + r else 0.0

def template_hit(a, b):
    rewrite_words = ("改写", "转换", "正式", "表达", "写法")
    sql_words = ("SQL", "查询", "连接")
    a_rewrite = any(w in a for w in rewrite_words)
    b_rewrite = any(w in b for w in rewrite_words)
    a_sql = any(w in a for w in sql_words)
    b_sql = any(w in b for w in sql_words)
    return int((a_rewrite and b_rewrite) or (a_sql and b_sql))

def score(x, y):
    s = cosine(embed(x), embed(y))
    r = rouge_l(x, y)
    t = template_hit(x, y)
    return cfg["alpha"] * s + cfg["beta"] * r + cfg["gamma"] * t

duplicate_ids = set()

for i in range(len(samples)):
    for j in range(i + 1, len(samples)):
        d = score(samples[i]["text"], samples[j]["text"])
        if d >= cfg["tau"]:
            duplicate_ids.add(samples[j]["id"])

deduped = [x for x in samples if x["id"] not in duplicate_ids]

def assign_bucket(x):
    return (x["task"], x["domain"], x["difficulty"])

buckets = defaultdict(list)
for x in deduped:
    buckets[assign_bucket(x)].append(x)

assert 2 in duplicate_ids
assert 1 not in duplicate_ids
assert 3 not in duplicate_ids
assert len(deduped) == 2
assert ("sql", "database", "medium") in buckets
```

真实工程实现时，近邻检索通常不会全量两两比较，因为 \(n\) 条样本两两比较需要 \(O(n^2)\) 次计算。数据量达到几十万时，应使用向量索引，例如 FAISS、HNSW 或数据库内置向量检索。先召回每条样本最相似的 \(k\) 个候选，再计算 ROUGE-L 和模板规则，可以把成本控制在可接受范围。

配置项也不应写死。不同业务的 \(\tau\)、\(\alpha\)、\(\beta\)、\(\gamma\) 需要单独校准。客服摘要、代码生成、法律问答、SQL 查询的重复模式不同，直接复用同一套参数会产生明显误删或漏删。

## 工程权衡与常见坑

阈值不能全局通吃。不同领域、不同任务的重复模式不同。代码类指令中，两个样本只改了变量名，可能仍然是重复；法律类指令中，两个问题词面相近，但法规条款不同，可能不能删除。

多样性也不是无脑均匀分布。目标不是让每个任务类型数量一样，而是让数据分布接近业务目标。如果客服场景里“总结”和“改写”占比本来就高，可以适当压缩冗余，但不能把“表格推理”“错误诊断”“流程生成”压到几乎没有。否则模型会在常见任务上稳定，在稍复杂的真实工单里失效。

| 常见坑 | 后果 | 规避方式 |
|---|---|---|
| 只看词面重叠 | 可能误删合法改写，或漏掉语义重复 | 结合语义嵌入、ROUGE-L、模板规则 |
| 先随机切分再去重 | 训练集和验证集可能出现近重复，评估虚高 | 先全量去重，再划分数据集 |
| 模板规则过死 | 只能抓固定句式，泛化弱 | 规则检测与语义聚类并行 |
| 均匀采样破坏真实分布 | 模型能力偏离业务需求 | 用目标分布 \(q(z)\) 做软约束 |
| 阈值不分领域 | 误删和漏删都变多 | 按任务或领域校准阈值 |
| 只保留高频任务 | 长尾能力不足 | 给长尾任务设置最小配额 |
| 只保留长尾任务 | 主任务质量下降 | 主任务保留合理上限，不强行清空 |

一个实用校准方式是人工抽样。先按候选相似度区间分层抽样，例如 \(0.7-0.8\)、\(0.8-0.9\)、\(0.9-1.0\)，人工标注“应删、可删、应留”。再根据误删率和漏删率选择阈值。误删率是本该保留却被删除的比例，漏删率是本该删除却被保留的比例。对微调数据来说，误删长尾任务通常比保留少量重复更危险。

## 替代方案与适用边界

如果数据量很小，先做轻量规则去重和人工抽查，未必需要复杂的向量检索和聚类。复杂系统的收益来自规模，当数据只有几百条时，人工看一遍通常更可靠。

如果任务分布天然不均衡，但业务目标就是不均衡，应保留这种偏斜，只做去冗余，不做强平衡。例如垂直办公场景里，80% 的需求都是摘要和改写，就不应强行把摘要、改写、SQL、代码、推理拉成 1:1:1:1:1。更合理的做法是保留主任务占比，同时给长尾任务设置最小配额。

| 方案 | 优点 | 缺点 | 适用边界 |
|---|---|---|---|
| 纯规则法 | 快、可解释、实现简单 | 召回低，抓不住语义近重复 | 小规模、强模板数据 |
| 语义聚类法 | 能发现换说法的重复样本 | 依赖 embedding 质量，成本更高 | 中大规模指令数据 |
| 规则 + 向量联合 | 准确率和召回较均衡 | 工程复杂度更高 | 大多数工程场景 |
| 强制均衡采样 | 分布清晰，便于控制 | 可能破坏真实业务比例 | 目标分布本身接近均匀 |
| 人工审核 | 质量高，适合定标 | 成本高，规模受限 | 小数据集或阈值校准阶段 |

最推荐的默认方案是“规则 + 向量联合”。先用向量召回语义近邻，再用 ROUGE-L 和模板规则降低误判，最后按任务、领域、难度分桶采样。它不是最简单的方案，但在多数指令微调工程中足够稳健。

最终要记住一个原则：去重不是为了让数据看起来干净，多样性也不是为了让统计表好看。它们共同服务于微调目标：让模型用有限训练预算学习更多有差异、有价值、会在真实场景出现的任务模式。
