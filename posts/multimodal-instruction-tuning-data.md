## 核心结论

多模态指令微调数据构建，核心不是把图片和文字放进同一个样本，而是构造一组真正对齐的监督信号，让模型学习条件分布
$$
p(a \mid x,q,i)
$$
其中，`x` 是图像证据，`q` 是问题或指令，`a` 是参考答案，`i` 是输出意图或格式约束。意图可以理解为“希望模型怎么答”，比如“只输出数字”“用一句话总结”“按 JSON 返回”。

训练目标通常写成：
$$
\mathcal{L}=-\sum_{j=1}^{N}\log p_{\theta}(a_j\mid x_j,q_j,i_j)
$$

这条公式的含义很直接：对每个样本，如果模型在“给定这张图、这个问题、这个格式要求”后，越能把正确答案的概率压高，损失就越低；反之，样本错配、答案伪造、格式混乱，都会把损失抬高，并把错误监督写进模型参数。

一个新手能立刻理解的判断标准是：如果不看图也能把答案猜对，这条多模态样本大概率有问题。多模态助手的上限，往往不是先被模型规模卡住，而是先被数据对齐质量卡住。

下面是“好样本 vs 坏样本”的最小对照：

| 样本类型 | 图像 | 问题 | 答案 | 是否合格 | 原因 |
|---|---|---|---|---|---|
| 好样本 | 发票截图 | “本单应付多少” | “128.50” | 是 | 答案可从图中金额字段验证 |
| 坏样本 | 发票截图 | “本单应付多少” | “通常是 100 左右” | 否 | 答案来自经验猜测，不来自图像 |
| 坏样本 | 猫的照片 | “右上角日期是什么” | “2024-10-01” | 否 | 图中没有该字段，监督无来源 |
| 坏样本 | 收据截图 | “请只输出数字” | “总金额是 128.50 元。” | 否 | 格式约束与答案不一致 |

玩具例子里，如果题目问的是图里右上角数字，就不能拿左下角文字当答案。真实工程里，如果问题问“应付金额”，答案就必须对应金额字段本身，而不是历史订单均值、商户常见价格或标注员的主观猜测。

## 问题定义与边界

多模态指令微调数据构建，解决的是“监督信号如何可靠来自图像”的问题，不直接解决模型架构设计、推理加速、部署成本，也不等于纯文本指令微调。

四个核心变量如下：

| 变量 | 含义 | 例子 |
|---|---|---|
| `x` | 图像证据 | 发票截图、表格照片、UI 界面图 |
| `q` | 问题/指令 | “总金额是多少” |
| `a` | 目标答案 | “128.50” |
| `i` | 意图/格式 | “只输出数字，保留两位小数” |

这四个变量必须同时成立。少一个，监督关系就会变形。

一个合格正例是：图片是一张收据，问题是“总金额多少”，答案是图中的总金额字段，意图是“只输出数字”。  
一个典型反例是：问题写“请总结这张图的主题”，但图像只是一张模糊背景图，关键信息根本不在视觉证据中，模型只能靠语言先验胡乱生成。

可以把它理解成“出题、看图、作答、规定答案格式”四件事必须同时对上。

边界上要分清三类场景：

| 类型 | 是否属于本文重点 | 说明 |
|---|---|---|
| 图像问答、票据理解、界面助手 | 是 | 图像证据直接决定答案 |
| 纯文本客服、代码问答 | 否 | 没有视觉证据，不是多模态数据构建核心问题 |
| 图像生成、图文检索预训练 | 部分相关 | 也有图文配对，但目标函数和监督形式不同 |

容易混淆的地方有三个。

第一，图文同时出现，不代表就是有效多模态样本。社交媒体帖子里有图有文，但如果答案只依赖文字说明，那它对视觉学习帮助有限。  
第二，有图像依赖，不代表不需要格式约束。很多工程任务不仅要答对，还要“按字段答”“按 JSON 答”。  
第三，样本多，不代表监督强。大规模弱对齐数据可以提供覆盖面，但如果核心任务要求高精度字段抽取，弱对齐会成为主要噪声源。

适用任务、非适用任务、易混场景可以简单记成下面三行：

| 类别 | 代表任务 | 判断标准 |
|---|---|---|
| 适用 | VQA、OCR 问答、图表问答、GUI 助手 | 答案必须依赖图像 |
| 非适用 | 纯文本 SFT、代码补全 | 没有视觉证据 |
| 易混淆 | 图文帖子总结、商品描述生成 | 可能主要依赖语言而非图像 |

## 核心机制与推导

数据集统一记作：
$$
\mathcal{D}=\{(x_j,q_j,a_j,i_j)\}_{j=1}^{N}
$$

训练目标是：
$$
\mathcal{L}=-\sum_{j=1}^{N}\log p_{\theta}(a_j\mid x_j,q_j,i_j)
$$

多任务时常写成：
$$
\mathcal{L}=\sum_t \lambda_t \,\mathbb{E}_{(x,q,a)\sim \mathcal{D}_t}[-\log p_{\theta}(a\mid x,q,i_t)]
$$
这里的 $\lambda_t$ 是任务权重，可以理解为“不同任务在总训练里的分量”。如果票据 OCR 问答、图表问答、图像描述都混在一起训练，但模板和权重不统一，模型会把不同任务的监督信号互相污染。

核心机制其实只有一句话：让问题确实依赖图像，而不是只依赖语言先验。

一个玩具数值例子最能说明问题：

| 样本 | 正确答案概率 | 交叉熵损失 |
|---|---:|---:|
| 1 | 0.9 | 0.105 |
| 2 | 0.1 | 2.303 |

平均损失是：
$$
(0.105 + 2.303)/2 = 1.204
$$

第二条样本如果是错配样本，比如图片里明明写着 `128.50`，却把答案标成 `98.00`，它会显著抬高训练损失。更关键的是，这种损失不是“有益的难样本”，而是“错误监督”。模型会被迫同时学习两件相互冲突的事：一方面看图，另一方面忽略图。

可以把流程理解成下面这个简化“流程图”：

| 阶段 | 要做什么 | 目标 |
|---|---|---|
| 样本构造 | 组织 `(x,q,a,i)` | 建立监督单位 |
| 对齐校验 | 检查答案能否从图像验证 | 去掉伪相关 |
| 模板统一 | 统一 prompt 和 target 形式 | 稳定训练分布 |
| 损失优化 | 最小化交叉熵 | 提高条件生成概率 |

如果训练时把“看图题”做成了“背答案题”，模型学到的就不是视觉能力，而是语言猜测能力。这也是很多早期多模态数据集的问题：图文表面相关，但答案并不真正扎根于视觉证据。

真实工程例子是发票助手。任务看起来很简单：给一张发票图，回答“本单应付多少”。但真正对齐时至少要保证以下几件事：

| 校验项 | 为什么要做 |
|---|---|
| 金额字段来自图中可见区域 | 防止答案来自外部系统 |
| 币种与金额一致 | 防止把 USD 和 CNY 混写 |
| 税前/税后定义清楚 | 防止监督口径不一致 |
| 输出格式固定 | 防止同义答案影响训练稳定性 |

没有这些约束，模型很容易学会固定模板，比如“金额通常在右下角”“发票金额往往是两位小数”，但这不是稳健视觉理解。

## 代码实现

代码层面，重点不是写一个能把图片和文本读进来的 `dataloader`，而是写一个能保证监督有效的数据管线。数据管线可以理解为“样本进入训练前的质检流水线”。

下面给出一个可运行的 Python 玩具实现。它用结构化元数据模拟 OCR 和字段校验，展示四件事：样本构造、配对、模板统一、低质量过滤。

```python
from dataclasses import dataclass
from math import log
from typing import List, Dict


@dataclass
class RawRecord:
    image_id: str
    image_meta: Dict[str, str]   # 模拟 OCR/字段抽取结果
    question: str
    answer: str
    intent: str


def build_sample(record: RawRecord) -> Dict[str, str]:
    prompt = (
        f"<image:{record.image_id}>\n"
        f"问题：{record.question}\n"
        f"输出要求：{record.intent}"
    )
    return {
        "image": record.image_id,
        "prompt": prompt,
        "target": record.answer.strip()
    }


def check_alignment(image_meta: Dict[str, str], question: str, answer: str) -> bool:
    q = question.lower()
    a = answer.strip()

    if "总金额" in question or "应付" in question:
        return image_meta.get("total_amount") == a

    if "日期" in question:
        return image_meta.get("date") == a

    if "商户" in question:
        return image_meta.get("merchant") == a

    # 未覆盖的问题类型默认不过滤为真，而是判为需人工审核
    return False


def normalize_answer(question: str, answer: str) -> str:
    if "金额" in question or "应付" in question:
        return answer.replace("元", "").strip()
    return answer.strip()


def deduplicate(records: List[RawRecord]) -> List[RawRecord]:
    seen = set()
    result = []
    for r in records:
        key = (r.image_id, r.question.strip(), r.answer.strip(), r.intent.strip())
        if key not in seen:
            seen.add(key)
            result.append(r)
    return result


def build_dataset(records: List[RawRecord]) -> List[Dict[str, str]]:
    dataset = []
    for r in deduplicate(records):
        norm_answer = normalize_answer(r.question, r.answer)
        normalized = RawRecord(
            image_id=r.image_id,
            image_meta=r.image_meta,
            question=r.question,
            answer=norm_answer,
            intent=r.intent,
        )
        if check_alignment(normalized.image_meta, normalized.question, normalized.answer):
            dataset.append(build_sample(normalized))
    return dataset


def cross_entropy(prob: float) -> float:
    return -log(prob)


records = [
    RawRecord(
        image_id="invoice_001",
        image_meta={"total_amount": "128.50", "date": "2026-04-01", "merchant": "ACME"},
        question="本单应付多少",
        answer="128.50元",
        intent="只输出数字，保留两位小数"
    ),
    RawRecord(
        image_id="invoice_001",
        image_meta={"total_amount": "128.50", "date": "2026-04-01", "merchant": "ACME"},
        question="本单应付多少",
        answer="99.00",
        intent="只输出数字，保留两位小数"
    ),
]

dataset = build_dataset(records)

assert len(dataset) == 1
assert dataset[0]["target"] == "128.50"
assert "输出要求：只输出数字" in dataset[0]["prompt"]
assert round(cross_entropy(0.9), 3) == 0.105
assert round(cross_entropy(0.1), 3) == 2.303
```

这个例子虽然小，但已经体现了真实数据构建的基本原则。

1. 原始数据不是直接进训练，而是先过一致性检查。  
2. 答案先标准化，再校验是否能从图像证据验证。  
3. 格式约束进入 prompt，而不是放在标注文档里不进模型。  
4. 去重不是可选项，否则验证集容易被训练集污染。

如果类比本博客仓库里的 `posts.json` 索引流程，可以把原始样本索引理解为一张“多模态 `posts.json`”：先有样本目录，再按 `image_id` 找图，按记录找问题和答案，最后导出统一训练格式。

一个可落地的数据处理步骤表如下：

| 步骤 | 内容 | 输出 |
|---|---|---|
| 1 | 收集原始图文 | 原始图片、OCR、问答草稿 |
| 2 | 对齐问题和图像 | `(x,q)` 配对记录 |
| 3 | 标准化答案格式 | 统一数值、日期、枚举项 |
| 4 | 过滤噪声样本 | 去掉错配、重复、不可验证样本 |
| 5 | 划分训练/验证集 | 按图像哈希或来源切分 |
| 6 | 导出训练格式 | 统一 prompt/target JSONL |

真实工程里，还常见一个增强步骤：把“必须看图才能答”的负样本混进去。例如同一张发票，随机替换问题中的字段名称，若答案还能被模型轻易猜中，说明样本设计仍然过于依赖语言模板。

## 工程权衡与常见坑

真正影响效果的，通常不是“模型是 7B 还是 13B”，而是数据是否可学习、可验证、可复现。

下面是常见风险与规避方式：

| 常见坑 | 表现 | 规避方式 |
|---|---|---|
| 视觉-文本错位 | 问题和图不对应 | 人工抽检 + 自动一致性过滤 |
| 只学语言先验 | 不看图也能答 | 硬负样本 + 必须看图任务 |
| 输出格式漂移 | 一会儿短答一会儿长答 | 固定模板 |
| 细粒度信息缺失 | OCR 小字识别差 | 高分辨率 + 裁剪 + 区域标注 |
| 数据泄漏/重复 | 评测虚高 | 去重 + 按图像哈希切分 |

“错误样本如何污染训练”可以压缩成一个短流程：

| 错误链路 | 后果 |
|---|---|
| 图像与问题错配 | 模型学到忽略图像 |
| 答案不可验证 | 监督目标变随机噪声 |
| 模板不统一 | 同类任务分布破碎 |
| 评测集泄漏 | 指标虚高，线上失效 |

发票助手是典型真实工程例子。很多团队一开始会把后端数据库里的金额直接拿来当答案，觉得这样标注快。但如果不反查图中金额、币种、税额字段，模型学到的往往是“商户名到价格区间”的统计关系，而不是 OCR 与视觉定位能力。上线后只要版式一变、字段位置一变、币种一变，性能就会掉。

另一个常见坑是“格式正确但语义错误”。比如要求“只输出数字”，模型输出了 `128.50`，看起来格式没问题，但图中真正的“应付金额”是 `138.50`。如果评测只做字符串格式检查，不做字段级语义核对，错误会被掩盖。

因此工程上通常要在成本和精度之间做权衡：

| 选择 | 成本 | 收益 | 适合场景 |
|---|---|---|---|
| 纯人工标注 | 高 | 准确 | 小规模高质量数据 |
| OCR 辅助校验 | 中 | 提升字段一致性 | 文本密集图像 |
| 区域级框标注 | 很高 | 最强局部监督 | 合同、票据、表格 |
| 弱监督爬取 | 低 | 快速扩量 | 通用预热数据 |

结论是：高质量对齐样本不一定要求全量人工，但一定要有可验证闭环。没有验证闭环，规模越大，噪声越系统化。

## 替代方案与适用边界

不是所有任务都适合用同一种数据构建方式。关键要看四件事：需不需要读图中文字、需不需要定位局部区域、需不需要强格式控制、需不需要多任务统一模板。

常见方案对比如下：

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 统一多模态指令微调 | 泛化强 | 构建成本高 | 通用 VQA / 助手 |
| OCR 辅助标注 | 字段准确 | 管线更复杂 | 发票、合同、表格 |
| 区域级监督 | 细粒度强 | 标注成本高 | 局部对象问答 |
| 纯文本指令微调 | 简单 | 无视觉能力 | 文本任务 |

可以把选型过程看成一棵简化决策树：

| 决策问题 | 是 | 否 |
|---|---|---|
| 需要读图中文字吗 | 优先考虑 OCR 辅助 | 可先尝试 OCR-free |
| 需要定位局部区域吗 | 加区域级监督 | 全图语义即可 |
| 需要固定输出格式吗 | 明确加入 `i` 约束 | 可做开放生成 |
| 需要多任务统一模板吗 | 统一 prompt 和 target 规范 | 可按任务分开构建 |

适合 OCR-free 的场景，一般是图像结构清晰、文字不密集、问题依赖整体语义，比如“图中是什么场景”“折线图总体趋势如何”。  
适合 OCR 或区域标注的场景，则是合同、发票、票据、表格截图这类文本密集、字段敏感任务。  
新手可以记成一句话：有些题靠“看大意”就够，有些题必须“逐字读图”。

统一多模态指令微调不是唯一答案，但它适合做通用助手底座。OCR 辅助和区域监督的价值，不是替代它，而是为需要精确字段理解的任务提供更强监督。纯文本指令微调则适合完全不依赖图像的任务，简单但不具备视觉能力。
