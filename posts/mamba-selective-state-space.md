## 核心结论

Mamba 的选择性状态空间机制，核心不是“把 RNN 再做一遍”，而是把传统状态空间模型中的关键参数从固定常量改成当前输入的函数。最重要的变化是：

$$
\Delta_t = \operatorname{softplus}(W_\Delta x_t + b_\Delta)
$$

其中 $\Delta_t$ 是离散化步长，白话说就是“本步更新要走多快”。它大，表示更快响应当前输入、更多遗忘历史；它小，表示更新更慢、更多保留过去状态。

传统线性时不变状态空间模型，简称 LTI SSM，指系统参数不随时间和输入变化。它的优点是稳定、容易并行；缺点是面对内容差异很大的序列时，反应方式太固定。Mamba 把 $\Delta,B,C$ 做成输入相关，于是模型可以按内容决定“什么该记、什么该忘”。

可以把这个机制先粗略理解成一个“动态档位系统”：输入 $x_t$ 决定当前档位 $\Delta_t$。档位高，系统迅速切到当前路况；档位低，系统延续之前的惯性。这个比喻只用于建立直觉，正式定义仍然是输入依赖的状态更新参数。

一个最简流程可以写成：

| 步骤 | 作用 |
|---|---|
| $x_t \rightarrow \Delta_t$ | 决定更新快慢 |
| $x_t \rightarrow B_t$ | 决定当前输入如何写入状态 |
| $x_t \rightarrow C_t$ | 决定当前状态如何读出输出 |
| 状态更新 $h_{t-1}\rightarrow h_t$ | 同时融合历史和当前内容 |

## 问题定义与边界

先定义问题。状态空间模型，简称 SSM，可以看成一种“用隐状态压缩历史，再递推更新”的序列建模方法。它通常写成连续或离散形式，本质都是维护一个状态 $h_t$，再从状态生成输出 $y_t$。

传统 LTI SSM 的典型问题不是“不会建模长序列”，而是“对不同内容一视同仁”。例如：

- 遇到平稳背景时，理想模型应当慢更新，维持长期信息。
- 遇到突发信号时，理想模型应当快更新，迅速把注意力切到当前输入。

但固定参数模型只能用同一套更新规则处理全部 token。这样会出现两个常见矛盾：

| 维度 | 固定参数 SSM | 输入依赖参数 SSM（Mamba） |
|---|---|---|
| 更新速度 | 固定 | 随内容变化 |
| 长期记忆 | 强，但不够灵活 | 可保留，也可主动遗忘 |
| 局部突发响应 | 往往不够敏感 | 更容易快速切换 |
| 计算结构 | 更适合固定卷积化 | 不能直接用固定卷积 |
| 工程代价 | 较低 | 更高 |

这就是 Mamba 要解决的边界：它不是要完全替代所有 SSM，也不是要在所有任务上胜过 Transformer，而是要在“序列很长，但局部内容又很关键”的场景中，提供一种更灵活的状态更新机制。

玩具例子可以先看一句文本流：

| 位置 | 输入内容 | 理想行为 |
|---|---|---|
| 1-20 | 普通背景描述 | 缓慢更新，保留整体语境 |
| 21 | `not` 或 `但是` | 快速更新，因为语义发生反转 |
| 22-30 | 后续解释 | 在新语义下继续积累 |

固定步长模型只能一直按同一节奏更新；Mamba 则希望在第 21 个位置自动把 $\Delta_t$ 拉大。

真实工程例子是视频理解。视频大部分帧可能变化很小，但某几帧出现目标进入、遮挡或动作突变。固定参数模型对所有帧使用同样的状态更新强度，往往难以同时兼顾稳定背景与突发动作。Mamba 的目标就是让模型对“静态背景”和“突发 motion”生成不同更新策略。

## 核心机制与推导

先从标准递推写法出发。离散状态更新可以抽象成：

$$
h_t = \bar{A}_t h_{t-1} + \bar{B}_t x_t,\qquad y_t = C_t h_t
$$

这里：

- $h_t$ 是隐状态，白话说是“到当前为止压缩下来的历史记忆”。
- $\bar{A}_t$ 是状态转移，决定旧记忆如何延续。
- $\bar{B}_t$ 是输入写入，决定当前输入怎么写进记忆。
- $C_t$ 是读出矩阵，决定从记忆里读出什么作为输出。

Mamba 的关键是让这些量依赖于当前输入 $x_t$。最核心的是：

$$
\Delta_t = \operatorname{softplus}(W_\Delta x_t + b_\Delta)
$$

softplus 的定义是：

$$
\operatorname{softplus}(z)=\log(1+e^z)
$$

它的作用是把任意实数映射到正数，保证步长 $\Delta_t > 0$。这很重要，因为离散化步长不能是负数。

一个常见的表达思路是：

$$
s_t = W_s x_t
$$

$$
h_t = (\operatorname{diag}(s_t)A)h_{t-1} + (\operatorname{diag}(s_t)B_t)x_t
$$

$$
y_t = C_t h_t
$$

其中 $s_t$ 是选择向量，白话说是“每个状态维度本步开多大门”。$\operatorname{diag}(s_t)$ 表示把向量放到对角线上，等价于对各维度做独立缩放。于是，不同隐藏维度可以在同一步里有不同的更新强度。

### 为什么 $\Delta_t$ 能控制“记忆”与“遗忘”

直觉来自离散化。连续系统里常见状态矩阵为 $A$，离散后可写成近似形式：

$$
\bar{A}_t \approx e^{\Delta_t A}
$$

如果 $A$ 的特征值大多为负，那么：

- $\Delta_t$ 小时，$e^{\Delta_t A}$ 更接近恒等映射，旧状态保留更多。
- $\Delta_t$ 大时，衰减更强，旧状态更快被冲淡。

所以 $\Delta_t$ 本质上控制的是“历史影响衰减的速度”。

### 两步数值玩具例子

假设一维状态满足：

$$
h_t = \alpha_t h_{t-1} + (1-\alpha_t)x_t
$$

并令：

$$
\alpha_t = e^{-\Delta_t}
$$

这不是 Mamba 全部细节，但足够解释选择性机制。

若 $W_\Delta x_t + b_\Delta = -2$，则：

$$
\Delta_t = \operatorname{softplus}(-2)\approx 0.13
$$

此时：

$$
\alpha_t \approx e^{-0.13}\approx 0.88
$$

说明旧状态保留约 88%，当前输入只小幅写入。

若 $W_\Delta x_t + b_\Delta = 3$，则：

$$
\Delta_t = \operatorname{softplus}(3)\approx 3.05
$$

此时：

$$
\alpha_t \approx e^{-3.05}\approx 0.047
$$

说明旧状态几乎被清空，模型主要响应当前输入。

这就是“选择性”最核心的推导：同一个状态更新框架，因为 $\Delta_t$ 由输入控制，系统能在不同位置自动切换“记住”与“忘掉”。

## 代码实现

下面给一个可运行的极简 Python 版本。它不是论文原始实现，但保留了选择性机制的核心：输入先生成 $\Delta_t$，再决定状态保留比例。

```python
import math

def softplus(x: float) -> float:
    return math.log1p(math.exp(x))

def selective_ssm(xs, w_delta=1.0, b_delta=0.0):
    """
    极简一维 selective SSM:
    h_t = alpha_t * h_{t-1} + (1 - alpha_t) * x_t
    alpha_t = exp(-softplus(w_delta * x_t + b_delta))
    """
    h = 0.0
    states = []
    deltas = []
    alphas = []

    for x in xs:
        delta = softplus(w_delta * x + b_delta)
        alpha = math.exp(-delta)
        h = alpha * h + (1.0 - alpha) * x

        deltas.append(delta)
        alphas.append(alpha)
        states.append(h)

    return states, deltas, alphas

# 玩具例子：小输入时 delta 小，保留旧状态；大输入时 delta 大，快速切换
states1, deltas1, alphas1 = selective_ssm([-2.0], w_delta=1.0, b_delta=0.0)
states2, deltas2, alphas2 = selective_ssm([3.0], w_delta=1.0, b_delta=0.0)

assert deltas1[0] < deltas2[0]
assert alphas1[0] > alphas2[0]

# 两步序列：先稳定背景，再突发强信号
states, deltas, alphas = selective_ssm([0.1, 3.0], w_delta=1.0, b_delta=0.0)

# 第二步应更强调当前输入，因此 alpha 更小
assert alphas[1] < alphas[0]

# 状态应向第二步输入 3.0 明显靠近
assert states[1] > states[0]
assert states[1] > 1.0

print("deltas_small_big =", deltas1[0], deltas2[0])
print("alphas_small_big =", alphas1[0], alphas2[0])
print("states =", states)
```

如果把一维推广到多维，可以写成更接近论文表达的伪代码：

```python
# x_t: [d_model]
# h: [d_state]

delta_t = softplus(W_delta @ x_t + b_delta)   # [d_state] or [1]
s_t     = W_s @ x_t                           # [d_state]
B_t     = W_B @ x_t                           # [d_state]
C_t     = W_C @ x_t                           # [d_state]

A_bar_t = exp(delta_t * A)                    # 离散化后的状态转移
h       = (s_t * A_bar_t) * h + (s_t * B_t) * x_t_scalar_or_proj
y_t     = C_t * h
```

这里的关键不是某个具体矩阵写法，而是“当前 token 先生成控制量，再更新状态”。

### selective scan 为什么必要

如果参数固定，整段序列常可转成卷积，进一步使用 FFT 加速。FFT，白话说是“把固定卷积从时域搬到频域做更快乘法”。但 Mamba 的参数每步都变，卷积核不再固定，FFT 这条路就断了。

因此实现上要用 selective scan。它可以理解成“带输入依赖系数的并行前缀扫描”：

1. 每个位置先计算本地参数 $\Delta_t,B_t,C_t$。
2. 把递推式改写成可组合的局部变换。
3. 用 scan 并行合并多个局部变换，而不是严格一步一步串行。
4. 对长序列分块，并在反向传播时做 recomputation。

recomputation，白话说是“反向时重算部分中间结果，换取更低显存”。这是工程上必须接受的折中，因为 selective 机制会让中间激活很多。

真实工程例子是 Vision Mamba。图像常被展开成多条扫描路径，例如按行、按列、正向、反向做 cross-scan。每个 patch 都会生成自己的 $\Delta_t,B_t,C_t$，因此不仅要处理长距离依赖，还要处理多方向扫描的同步与显存开销。

## 工程权衡与常见坑

Mamba 的优势很清楚，但代价也很明确：你获得了内容感知更新，失去了固定核带来的简洁加速路径。

| 常见坑 | 为什么会出问题 | 规避方式 |
|---|---|---|
| 误以为还能直接用 FFT | 参数随输入变化，核不固定 | 使用 selective scan |
| 逐 token 纯串行实现 | 长序列吞吐量会明显下降 | 做并行 scan 或分块 scan |
| 保存全部中间态 | 显存会随序列长度快速增长 | 使用 recomputation |
| $\Delta_t$ 数值过大 | 旧状态可能过快衰减，训练不稳 | 控制初始化与归一化 |
| $\Delta_t$ 数值过小 | 长期不更新，局部响应迟钝 | 调整偏置和门控范围 |
| 视觉任务直接照搬语言实现 | 图像是二维结构，扫描路径不同 | 使用 cross-scan 或多方向路径 |

一个常见误解是：“既然 Mamba 还是递推，速度一定比 Transformer 慢。”这不准确。正确说法是：Mamba 失去了固定卷积核的 FFT 加速，但仍可通过硬件友好的 scan 内核获得很强吞吐。它慢不慢，取决于实现质量，而不是公式本身。

另一个坑是把选择性机制理解成“简单门控”。它当然有门控成分，但不只是乘一个门值。真正重要的是，它同时控制离散化步长、输入写入和输出读出，因此影响的是整个状态演化过程，而不是输出端的单点开关。

在视觉场景里，问题会更复杂。假设一张大图按行扫描，如果每个像素都动态生成不同步长，那么你不能再把整行当成统一时间尺度处理。若还把所有方向的中间状态都完整缓存，显存会很快爆掉。所以工业实现通常会结合：

- 分块扫描
- 融合 kernel
- 反向重算
- 限制状态维度和投影维度

## 替代方案与适用边界

Mamba 不是唯一可用的状态空间路线。它最主要的替代者是固定参数 SSM，如 S4、S5。

S4、S5 的核心思想是设计结构化状态矩阵，使长序列建模稳定且可高效实现。它们擅长“固定动力系统下的长程依赖建模”。Mamba 则进一步追求“内容决定动力系统”。

| 方案 | 参数是否依赖输入 | 是否容易卷积化 | 计算负担 | 更适合的任务 |
|---|---|---|---|---|
| S4 | 否 | 是 | 较低 | 长序列、模式较稳定 |
| S5 | 否或较弱依赖 | 较容易 | 较低到中等 | 稳定长程依赖建模 |
| Mamba | 是 | 否，不能直接固定卷积化 | 中到较高 | 长序列且局部内容变化大 |
| Transformer | 注意力依赖输入 | 不走卷积路径 | 序列长时开销高 | 需要全局内容交互的任务 |

可以用一句更直观的话总结：

- S4/S5 更像固定镜头，拍摄规则稳定。
- Mamba 更像可变焦镜头，当前画面内容会改变系统响应方式。

这个比喻只帮助理解差异，不替代正式定义。正式边界是：

1. 如果任务主要依赖稳定的长程模式，输入间差异不大，固定参数 SSM 往往已经足够。
2. 如果任务既有长上下文，又常出现局部突发信号，Mamba 的选择性机制更有价值。
3. 如果序列不长，或者内容调制并不重要，Mamba 可能是过度设计，因为它带来了额外实现复杂度。
4. 如果任务强依赖任意两位置之间的直接交互，注意力机制仍然有明显优势。

一个真实工程判断标准是：看任务是否同时满足“长序列”和“内容切换频繁”。例如：

- 条件语言建模：更适合 Mamba。
- 长视频事件检测：更适合 Mamba。
- 短文本分类：未必值得。
- 小分辨率、短上下文视觉分类：额外选择机制可能收益有限。
