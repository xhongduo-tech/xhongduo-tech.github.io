## 核心结论

RAG 的上下文压缩，本质上是在“检索之后、生成之前”做一次信息提纯。信息提纯的意思是：不是把检索到的整段文档原样交给主模型，而是只保留与当前问题强相关的句子、段落或字段，再把这些高密度内容拼成最终上下文。

它成立的原因很直接。标准 RAG 的主要问题通常不是“完全没检索到”，而是“检索到了太多半相关内容”。主模型看到过长、过杂的上下文时，会出现三类退化：一是注意力被噪音分散，二是 token 成本和延迟上升，三是答案更容易混入无关信息。上下文压缩针对的正是这三件事。

压缩比例常用下面这个量表示：

$$
r=\frac{\text{压缩后 tokens}}{\text{原始 tokens}}
$$

如果目标是 $r \approx 25\%-30\%$，意味着只保留原始上下文的四分之一到三分之一，压缩倍数大约是 3 到 4 倍。这不是一个数学定理，而是工程上常见的经验区间：压得太少，收益不明显；压得太狠，容易删掉关键事实。

先看一个玩具例子。假设检索器找回了 2 段材料，每段约 1000 tokens。用户问的是“Redis 的 AOF 重写为什么能减少磁盘占用”。这两段材料里真正相关的，可能各只有 300 tokens。压缩器把这 600 tokens 提出来交给主模型，效果通常会比直接塞 2000 tokens 更稳，因为主模型收到的是“问题相关材料”，不是“和 Redis 大致有关的一切”。

真实工程里，这一步通常由一个小模型或专门的压缩链完成，例如先 Top-k 检索，再对每个候选 chunk 做相关性抽取，最后把抽取结果交给回答模型。核心收益不是“让模型更聪明”，而是“让模型更少看到无关内容”。

## 问题定义与边界

要先把问题定义清楚。这里讨论的不是摘要系统，也不是训练阶段的长上下文建模，而是在线推理时的 RAG 管线优化。管线顺序通常是：

检索 $\rightarrow$ 可选重排 $\rightarrow$ 上下文压缩 $\rightarrow$ 生成

其中，“压缩”不是把文档概括成一段泛泛总结，而是围绕当前 query 做条件化保留。条件化的意思是：同一份文档，面对不同问题，保留的内容应当不同。

标准 RAG 的痛点可以用一个表概括：

| 方案 | 输入给主模型的内容 | token 成本 | 上下文质量 | 常见问题 |
| --- | --- | --- | --- | --- |
| 未压缩 RAG | 整段检索结果 | 高 | 中等，容易混噪音 | 答非所问、延迟高 |
| 压缩后 RAG | 只保留相关片段 | 中低 | 高，信息密度更高 | 可能误删细节 |
| 只做截断 | 前 N 个 tokens | 低 | 不稳定 | 容易把答案截掉 |

边界也要明确。上下文压缩并不是所有 RAG 系统都必须有。

适合使用的场景：

| 场景特征 | 是否适合压缩 | 原因 |
| --- | --- | --- |
| 文档长、chunk 大、查询细 | 适合 | 检索召回后噪音通常很多 |
| 多文档拼接、上下文预算紧 | 适合 | 压缩能提升单位 token 信息密度 |
| 法规、手册、知识库问答 | 适合 | 文档结构长而事实点分散 |
| chunk 本身很短 | 不一定 | 再压缩收益有限 |
| 极低延迟要求 | 不一定 | 多一次调用可能不划算 |
| 已经是高质量结构化字段检索 | 不一定 | 字段本身就是压缩后的信息 |

对初学者可以这样理解：检索器像是在书架上找出几本“可能有答案的书”，上下文压缩不是重新写书评，而是把每本书里和问题有关的几页折出来，最后只把折出来的页交给主模型。

这里还有一个常被忽略的边界：压缩只能解决“检索到了但太杂”的问题，不能解决“根本没检索到”或“召回本身错了”的问题。如果召回阶段就把关键文档漏掉，压缩再精细也没有用。

## 核心机制与推导

上下文压缩的最小机制可以拆成三步：

1. 对检索回来的候选文档逐个看它和 query 的局部相关性。
2. 从每个候选里抽取最相关的句子、段落或字段。
3. 把抽取结果按来源和顺序拼接，交给主模型回答。

“局部相关性”可以理解为：不是问“这篇文档总体相关吗”，而是问“这篇文档里哪一部分和当前问题最相关”。

从形式上看，假设原始检索结果一共有 $n$ 个片段，每个片段 token 数为 $t_i$，压缩后保留 token 数为 $c_i$，那么总压缩率是：

$$
r=\frac{\sum_{i=1}^{n} c_i}{\sum_{i=1}^{n} t_i}
$$

压缩倍数 $m$ 则可以写成：

$$
m=\frac{1}{r}
$$

如果 $r=0.30$，那么 $m\approx 3.33$，表示原来需要 2000 tokens，现在只需要约 600 tokens。

这个公式的重要性不在于计算本身，而在于它把工程问题量化了。你可以把压缩器看成一个受预算约束的选择器：在保真度尽量不变的前提下，把 token 预算压到目标区间内。

再看一个具体数值例子。假设有两个候选段落：

- 段落 A：1000 tokens，其中 280 tokens 讲“数据库连接池泄漏的检测”
- 段落 B：1000 tokens，其中 320 tokens 讲“连接池超时参数和泄漏表现”

用户问“连接池泄漏在日志上通常怎么暴露”。如果直接拼接，主模型看到 2000 tokens；如果压缩后只保留 600 tokens，那么：

$$
r=\frac{600}{2000}=0.3
$$

这意味着输入长度缩减了 70%。对主模型来说，这不是简单的“少看一点”，而是“少看无关部分，多看相关部分”。

上下文压缩之所以能改善回答质量，核心不是摘要能力，而是减少上下文污染。上下文污染可以白话解释为：模型虽然看到了答案，但同时也看到了很多干扰项，导致注意力分配变差，最终输出混入次要甚至错误内容。

真实工程中的机制通常比这个抽象过程更细一点。常见做法包括：

| 压缩方式 | 白话解释 | 优点 | 风险 |
| --- | --- | --- | --- |
| 句子抽取 | 从长段里挑出最相关句子 | 可追溯、稳定 | 句间衔接可能断裂 |
| 段落抽取 | 保留局部完整语义块 | 保真更好 | 压缩率较低 |
| 字段过滤 | 只保留 title、error_code、policy 等字段 | 高效 | 依赖结构化数据 |
| 生成式压缩 | 用模型重写成更短文本 | 更灵活 | 容易引入幻觉 |

对于零基础读者，最关键的认知是：压缩器最好优先做“抽取”，其次才是“重写”。因为抽取保留了原文证据，方便核对来源；重写虽然更短，但更容易把原文没有说过的话混进去。

## 代码实现

下面先给一个可运行的玩具实现。它不依赖大模型，只用最简单的词匹配模拟“按 query 抽取相关句子”的过程，用来说明输入输出关系。

```python
import re
from typing import List, Tuple

def split_sentences(text: str) -> List[str]:
    parts = re.split(r"[。！？\n]", text)
    return [p.strip() for p in parts if p.strip()]

def tokenize(text: str) -> List[str]:
    return re.findall(r"[a-zA-Z0-9_]+|[\u4e00-\u9fff]{2,}", text.lower())

def overlap_score(query: str, sentence: str) -> int:
    q_tokens = set(tokenize(query))
    s_tokens = set(tokenize(sentence))
    return len(q_tokens & s_tokens)

def compress_context(query: str, docs: List[str], keep_per_doc: int = 2) -> List[Tuple[int, List[str]]]:
    results = []
    for idx, doc in enumerate(docs):
        sentences = split_sentences(doc)
        ranked = sorted(
            ((overlap_score(query, s), s) for s in sentences),
            key=lambda x: x[0],
            reverse=True,
        )
        kept = [s for score, s in ranked[:keep_per_doc] if score > 0]
        results.append((idx, kept))
    return results

query = "AOF 重写 为什么 减少 磁盘 占用"

docs = [
    "Redis 提供 RDB 和 AOF 两种持久化机制。AOF 会记录写命令。AOF 重写会用当前数据状态生成更短的新命令序列，因此通常能减少磁盘占用。AOF 还能配置同步策略。",
    "连接池泄漏会导致活跃连接长期不释放。排查时要看超时日志、线程堆栈和慢请求。和 Redis AOF 无关。",
]

compressed = compress_context(query, docs, keep_per_doc=2)

assert len(compressed) == 2
assert "AOF 重写会用当前数据状态生成更短的新命令序列，因此通常能减少磁盘占用" in compressed[0][1]
assert compressed[1][1] == []

print(compressed)
```

这个例子对应的流程是：

检索结果文档列表 $\rightarrow$ 逐句打分 $\rightarrow$ 每篇保留若干句 $\rightarrow$ 把保留结果交给主模型

在真实系统中，词匹配通常不够，需要用更强的压缩器。以 LangChain 风格的伪代码为例：

```python
from langchain_openai import ChatOpenAI
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor

# 基础检索器：先从向量库取 Top-k
base_retriever = vectorstore.as_retriever(search_kwargs={"k": 6})

# 压缩器：围绕 query 抽取相关内容
llm = ChatOpenAI(
    model="gpt-4o-mini",
    temperature=0,
    max_tokens=4000,
)

compressor = LLMChainExtractor.from_llm(llm)

compression_retriever = ContextualCompressionRetriever(
    base_retriever=base_retriever,
    base_compressor=compressor,
)

docs = compression_retriever.invoke("AOF 重写为什么能减少磁盘占用？")
answer = qa_chain.invoke({"input_documents": docs, "question": "AOF 重写为什么能减少磁盘占用？"})
```

这里每一步的职责要分清：

| 步骤 | 输入 | 输出 | 作用 |
| --- | --- | --- | --- |
| 检索 | query | Top-k 文档 | 先尽量召回可能有用的材料 |
| 压缩 | query + 文档 | 相关片段 | 去掉与当前问题无关的内容 |
| 生成 | query + 压缩片段 | 最终答案 | 基于高密度上下文作答 |

真实工程例子可以看企业内部知识库问答。用户问：“支付系统重试机制为什么会导致重复扣款，应该看哪几个日志字段？”向量检索可能召回《支付网关设计》《订单状态机》《消息队列重试规范》《退款补偿手册》四份文档。若直接拼接，主模型会看到大量背景知识；若压缩器按 query 只保留“幂等键”“request_id”“重试次数”“最终状态写入时序”这些局部片段，主模型更容易给出可执行答案，而且可以把来源文档一并返回，便于排查。

实现上还有一个实用原则：压缩器输出最好保留来源元数据，例如文档 id、段落位置、链接。这样一旦回答有争议，可以回溯是“检索错了”“压缩错了”还是“生成错了”。

## 工程权衡与常见坑

上下文压缩的工程价值很高，但它不是免费收益。最核心的权衡，是“压缩率”和“保真度”之间的冲突。保真度可以白话理解为：压缩之后，关键事实是否还在，含义是否没有被改坏。

下面这个表能帮助做决策：

| 策略 | 准确率风险 | 成本 | 延迟 | 适用情况 |
| --- | --- | --- | --- | --- |
| 不压缩 | 中 | 高 | 高 | 文档少、预算宽松 |
| 适度压缩 | 低到中 | 中低 | 中 | 大多数知识库问答 |
| 过度压缩 | 高 | 低 | 中 | 只适合信息高度冗余场景 |

最常见的坑有五类。

第一，压缩过头，删掉关键细节。  
比如用户问“某 API 在 429 时是否应该指数退避，并且最大等待多久”，压缩器只保留了“应该指数退避”，把“最大等待上限 32 秒”删了。主模型最后会答对原则，但答错参数。

第二，多跳问题被拆坏。  
多跳问题的意思是：答案需要跨多个片段组合。例如“某个异常是由配置变更引起，还是由数据库版本升级引起”。如果压缩器只按局部相似度抽句子，可能把两段因果链拆散，导致主模型看不到完整推导。

第三，生成式压缩引入幻觉。  
如果压缩器不是抽取原文，而是重写原文，它可能“补全”一些原文并未明确给出的结论。这在合规、医疗、金融等高风险场景尤其危险。

第四，压缩成本本身反而不划算。  
如果原始 chunk 很短，比如每个 chunk 只有 150 到 200 tokens，再加一个压缩步骤，很可能省下的 token 不足以覆盖额外模型调用，系统总延迟还更高。

第五，评估方式错误。  
很多团队只看最终问答准确率，不看压缩阶段是否丢了证据。更稳妥的做法是同时监控：
- 压缩率 $r$
- 答案正确率
- 引用证据覆盖率
- Faithfulness，也就是答案是否忠于输入上下文

一个对初学者很有用的实践是：先不要一上来追求 20% 的极限压缩率，而是先从 50%-60% 开始，确认关键事实不会丢，再逐步收紧预算。因为一开始最重要的是证明“压缩不伤答案”，而不是证明“压得足够狠”。

还要注意语料异构问题。异构语料的意思是：你的知识库不是同一种文档风格，有规范文档、聊天记录、错误日志、SQL 变更单、产品公告。对一种文档上表现好的压缩 prompt，到了另一种文档上可能失效。所以压缩器不要只在单一数据集上调参后直接全量上线。

## 替代方案与适用边界

上下文压缩不是唯一方案，很多时候它应该和别的方法组合，而不是单独使用。

最容易混淆的是 reranking。rerank 的作用是“在 chunk 之间排序”，压缩的作用是“在 chunk 内部提纯”。前者回答“哪几段最值得看”，后者回答“每一段里哪几句最值得保留”。

可以用下面这张表区分：

| 方法 | 作用位置 | 做什么 | 是否改写内容 |
| --- | --- | --- | --- |
| Query transformation | 检索前 | 重写查询，使召回更准 | 改 query，不改文档 |
| Reranking | 检索后 | 给候选 chunk 重新排序 | 不改文档 |
| Chunk filtering | 检索后 | 直接过滤掉低相关 chunk | 不改文档 |
| Context compression | 生成前 | 从保留 chunk 中再抽取关键信息 | 可能抽取，也可能改写 |

实际管线常见为：

检索 $\rightarrow$ rerank $\rightarrow$ 可选过滤 $\rightarrow$ 可选压缩 $\rightarrow$ 生成

什么情况下优先用 rerank 而不是压缩？

- 你的 chunk 已经很短，比如每段 200 tokens。
- 你的延迟预算极紧，只能接受一次检索后直出。
- 你的场景需要完整证据段，不能只看被摘出的局部句子。

什么情况下压缩更值得上？

- 单个 chunk 很长，里面主题混杂。
- 多个 chunk 都“部分相关”，但无一段可以整段保留。
- 主模型上下文预算昂贵，token 成本敏感。
- 希望把证据密度做高，让回答更聚焦。

还有一种替代思路是结构化检索。例如对日志、工单、商品信息、监控指标，不一定需要“压缩长文本”，而是直接按字段检索：error_code、service_name、timestamp、owner、region。字段检索本质上已经把文本预先压缩成结构化信息，这时再做文本压缩通常收益有限。

所以，适用边界可以总结成一句话：当问题主要出在“召回内容太杂”时，用上下文压缩；当问题主要出在“召回不准”时，先改检索和重排；当数据本身可以结构化时，优先结构化，不要把一切都交给压缩器。
