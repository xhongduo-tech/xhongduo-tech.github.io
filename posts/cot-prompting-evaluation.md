## 核心结论

Chain-of-Thought Prompting，简称 CoT，可以直译为“链式思维提示”。它的做法不是只让模型给最终答案，而是要求模型先写出中间推理步骤，再输出结论。对评测的直接影响是：**在需要多步推理的 benchmark 上，CoT 往往显著提高分数；在主要考事实回忆或概念识别的 benchmark 上，收益通常很小，甚至不值得付出额外成本。**

这里的“多步推理”可以先理解成：答案不是输入里直接写着的，需要模型把几个中间步骤串起来才能得到。例如数学题、符号推导、复杂逻辑判断、需要多跳组合的信息抽取。GSM8K、MATH、BBH 这类数据集大量包含这种题型，因此 CoT 容易带来明显提升。相反，MMLU 中有不少题目更接近知识检索或概念判断，只要模型知道事实，直接答就够了，显式展开推理链不一定增加正确率。

一个最小玩具例子能说明边界。问“3+4=？”时，直接提示输出 `7` 就够了；改成 CoT 后，模型可能先写“3+4=7”，再写“答案：7”。这个过程更长，但结果没有更好。可一旦题目变成“Roger 有 3 个球，又买 2 包，每包 3 个，最后一共有几个球？”，CoT 往往更有价值，因为它把“先算 $2\times 3$，再加 3”显式拆开，减少跳步。

| 任务类型 | 代表 benchmark | 直接回答 | CoT 回答 | 常见效果 |
|---|---|---:|---:|---|
| 多步算术/推导 | GSM8K、MATH | 容易跳步 | 更容易稳定展开步骤 | 提升明显 |
| 复杂规则或逻辑组合 | BBH 部分子任务 | 容易漏条件 | 更容易逐项检查 | 常有提升 |
| 事实回忆/单跳知识 | MMLU 部分题型 | 已足够 | 常只增加长度 | 收益有限 |
| 极简单题 | 基础加法、常识判断 | 已最优或接近 | 冗余 | 通常无益 |

## 问题定义与边界

讨论 “CoT 对评测的影响” 时，先要把问题边界划清楚。这里的核心不是“模型会不会思考”，而是：**当评测协议要求或鼓励模型显式输出中间推理时，分数、成本、可解释性会怎样变化。**

更形式化地说，设问题为 $q$，最终答案为 $a$，中间推理序列为 $r_1, r_2, \dots, r_k$。普通 prompting 只关心输出 $a$；CoT prompting 则让模型同时生成 $r_1\ldots r_k$ 和 $a$。因此它改变的不是任务本身，而是**输出结构**与**解题路径**。

这带来两个边界条件：

1. 如果任务本身需要多步组合，CoT 有机会让模型把隐含过程显式化，从而降低遗漏步骤的概率。
2. 如果任务本身接近“看到问题就能直接检索答案”，CoT 只是把一个本来可直接完成的动作变成长输出，收益很可能不足以覆盖成本。

还是看一个新手可理解的例子。

玩具例子：
- 问题：Roger 有 3 个球，又买了 2 包，每包 3 个，一共有多少个？
- 直接答案：9
- CoT 过程：先算买了多少个球，$2\times 3=6$；再算总数，$3+6=9$；答案是 9。

这个题的关键不在于数字大小，而在于它至少包含两个依赖顺序的步骤。若模型直接从题面跳到最终答案，中间任何一步都可能算错；若模型被要求写出步骤，评测时就更容易对齐“有没有按正确路径解”。

| 问题类别 | 输入中是否直接包含答案 | 是否需要中间步骤 | 是否推荐 CoT |
|---|---|---|---|
| 单跳事实问答 | 通常是 | 否或极少 | 不优先 |
| 简单模板匹配 | 通常接近是 | 很少 | 多数不需要 |
| 多步算术 | 否 | 是 | 推荐 |
| 多条件逻辑判断 | 否 | 是 | 推荐 |
| 需要程序执行或工具计算 | 否 | 可能很多 | 仅 CoT 不够，常要结合工具 |

这也解释了为什么不能把 CoT 当成“通用加分按钮”。评测集的题型组成不同，CoT 的平均收益就会不同。拿同一个模型去测 GSM8K 和 MMLU，观察到的提升幅度本来就不该一样。

## 核心机制与推导

CoT 的机制可以用一个很简单的概率分解来理解。普通提示通常让模型最大化：

$$
P(a \mid q, prompt)
$$

也就是给定问题 $q$ 和提示模板 `prompt`，直接生成答案 $a$。而 CoT 会把答案前面插入一段推理链：

$$
P(r_1, r_2, \ldots, r_k, a \mid q, prompt)
$$

如果再按自回归生成，也可以写成：

$$
P(r_1,\ldots,r_k,a \mid q,prompt)
=
P(r_1\mid q,prompt)\prod_{i=2}^{k}P(r_i\mid q,prompt,r_{<i})\cdot P(a\mid q,prompt,r_{\le k})
$$

白话解释是：模型先生成第一步推理，再基于前面的推理生成下一步，最后在已有推理链的条件下生成最终答案。于是“答案”不再是一个孤立 token 序列，而是被放在一个逐步展开的上下文后面。

这类提示为什么会在推理 benchmark 上有效，通常有三个原因。

第一，**它改变了解码轨迹**。  
“解码轨迹”可以理解成模型生成文字时走过的路径。直接回答时，模型可能在前几个 token 就走向错误答案；CoT 强制它先走过若干中间节点，使得正确路径更容易被延续。

第二，**它把隐含结构显式化**。  
原本“先乘再加”“先列条件再判断”这些步骤可能只存在于模型内部的激活状态里，CoT 把它们搬到输出中。对多步题，这种显式化经常有助于稳定表现。

第三，**few-shot 示例会把“如何解题”一起教给模型**。  
few-shot 的意思是“给少量示例”。如果提示中不仅有题目和答案，还有完整解题链，那么示例传递的不只是结果映射，还包括过程模板。模型会模仿这种格式，例如：

- 题目：Roger 有 3 个球，又买 2 包，每包 3 个。
- 推理：$2\times3=6$；$3+6=9$
- 答案：9

这与普通 few-shot 的差别很大。普通 few-shot 更像“看几个输入输出对”；CoT few-shot 则更像“看别人怎么一步一步做题”。

但这里要注意一个误区：**CoT 有时提升的是“更会按格式展开”，不等于模型真的具备了更强的底层推理能力。** 在评测里，这种差别非常重要。因为 benchmark 分数上升，可能来源于：
- 模型更容易命中正确中间步骤；
- 模型更容易在长输出中自我纠错；
- 提示模板恰好和数据集的解题形式高度匹配。

真实工程例子：做数学题自动评测平台时，系统常常不只需要知道“对/错”，还需要知道“错在哪一步”。这时 CoT 的价值不只是提升准确率，还包括提供可验证的中间步骤。例如模型输出：
- 第一步：$2\times3=6$
- 第二步：$3+6=8$
- 最终答案：8

后台验证器一看就知道乘法对了，加法错了。对教学系统、调试系统、错误归因系统，这种结构化错误定位非常有用。

## 代码实现

工程里实现 CoT，核心通常只有两步：**构造 prompt**，以及**解析输出中的推理链和最终答案**。最小版本甚至不依赖特定模型 SDK，也可以先在本地把输入输出协议设计清楚。

下面给一个可运行的 Python 玩具实现。它不调用真实模型，而是演示“如何组装 CoT prompt”和“如何从响应里拆出 reasoning 与 answer”。这部分逻辑在接入任何大模型 API 时都能复用。

```python
import re

def build_cot_prompt(question: str, examples: list[dict]) -> str:
    blocks = []
    for ex in examples:
        blocks.append(
            "Q: " + ex["q"] + "\n"
            "Reasoning: " + ex["reasoning"] + "\n"
            "Answer: " + ex["answer"]
        )
    demo = "\n\n".join(blocks)
    return (
        "Solve the problem by showing reasoning first, then final answer.\n\n"
        f"{demo}\n\n"
        f"Q: {question}\n"
        "Reasoning:"
    )

def parse_chain(response_text: str) -> tuple[str, str]:
    m = re.search(r"Reasoning:\s*(.*?)\s*Answer:\s*(.+)\s*$", response_text, re.S)
    if not m:
        raise ValueError("response format invalid")
    reasoning = m.group(1).strip()
    answer = m.group(2).strip()
    return reasoning, answer

examples = [
    {
        "q": "Roger has 3 balls and buys 2 packs of 3. How many balls now?",
        "reasoning": "He buys 2*3=6 balls. Total is 3+6=9.",
        "answer": "9",
    }
]

prompt = build_cot_prompt(
    "A box has 4 apples. You buy 3 more boxes. How many apples total?",
    examples,
)

fake_model_response = (
    "Reasoning: 3 more boxes means 3*4=12 apples. "
    "Original 4 apples plus 12 gives 16.\n"
    "Answer: 16"
)

reasoning, answer = parse_chain(fake_model_response)

assert "3*4=12" in reasoning
assert answer == "16"
assert prompt.startswith("Solve the problem")
```

这个例子里有几个工程上值得保留的点：

| 字段 | 作用 | 为什么重要 |
|---|---|---|
| `question` | 用户题目 | 原始输入 |
| `examples` | CoT few-shot 示例 | 告诉模型“过程该怎么写” |
| `reasoning` | 中间推理链 | 便于人工查看或程序验证 |
| `answer` | 最终答案 | 供评测脚本打分 |

如果接入真实模型，伪代码通常就是：

```python
prompt = build_prompt(question, chain_examples)
response = model.generate(prompt)
reasoning, answer = parse_chain(response.text)
score = judge(answer, gold_answer)
```

真实工程例子可以再往前走一步：如果你在做题目评测服务，可以把 `reasoning` 中的算式抽取出来送给一个“小验证器”，例如只允许四则运算和变量替换。这样模型即使最后答错，也能告诉你它是“算错了”还是“理解错了题目”。这比只看最终 accuracy 更适合分析模型失败模式。

## 工程权衡与常见坑

CoT 在评测中最常见的代价是 **token 成本**。token 可以先理解成“模型计费和上下文长度的基本单位”。推理链越长，请求越贵，延迟越高，吞吐越低。于是工程上不能只看分数，还要看性价比。

| 维度 | 直接回答 | CoT |
|---|---|---|
| 输出 token | 少 | 多 |
| 延迟 | 低 | 更高 |
| 评测成本 | 低 | 更高 |
| 可解释性 | 弱 | 强 |
| 中间步骤可验证性 | 几乎没有 | 较强 |
| 对多步题准确率 | 一般 | 常更好 |
| 对事实题准确率 | 常已足够 | 常无明显增益 |

常见坑主要有五类。

第一，**把 CoT 当成普适优化**。  
如果你的评测集大量是单跳知识题，CoT 可能几乎不提高分数，却把 token 用量拉高很多。对线上系统，这会直接变成成本问题。

第二，**把“会写过程”误认为“过程一定可信”**。  
模型可能生成一条看起来合理、实际上错误的推理链。这类现象有时被称为“事后合理化”，意思是模型先得到或猜到答案，再补一段像样的解释。评测如果只看最终答案，会忽略这个问题；如果要使用 CoT 作为可解释依据，最好加验证。

第三，**解析格式不稳定**。  
模型可能输出：
- `Final answer: 9`
- `答案为 9`
- `Therefore, the answer is 9`
如果你的评测脚本强依赖固定模板，解析就会脆弱。工程里通常需要更强约束的输出协议，例如明确使用 JSON、XML 标签，或者把最终答案单独放进结构化字段。

第四，**训练分布与评测分布不一致**。  
有些 benchmark 的题目天然适合逐步演算；有些则更多依赖知识覆盖或阅读理解。你在某一类数据上的 CoT 收益，不能直接外推到另一类任务。

第五，**中间步骤泄漏和评分偏差**。  
如果评测系统把完整 CoT 暴露给判题器，或者下游模型拿到前一轮推理结果再打分，可能引入额外偏差。评测设计时要区分“生成答案的模型”和“判定答案的系统”，避免让中间过程影响本不该影响的评分逻辑。

一个典型真实工程例子是在线数学辅导系统。你希望学生不仅看到答案，还看到每一步推导；同时后台需要知道模型哪一步最常错。CoT 很适合这个场景。但如果把同样协议直接套到“百科问答机器人”，系统就会无意义地变慢，还可能生成一大段并不可靠的解释。

## 替代方案与适用边界

CoT 不是唯一方案，也不是所有任务的默认选择。更合理的做法是按任务结构选策略。

| 任务特征 | 推荐方案 | 原因 |
|---|---|---|
| 单跳事实问答 | 直接 prompt | 最短路径，成本最低 |
| 简单分类/抽取 | 直接输出标签或字段 | 易解析，没必要展开 |
| 多步算术/逻辑 | CoT | 显式步骤有帮助 |
| 高精度计算 | 工具调用或程序执行 | 仅靠自然语言推理不稳 |
| 需要多条候选路径比较 | Self-Consistency 等方法 | 单条 CoT 可能不够稳 |

这里的 Self-Consistency 可以先理解成“采样多条推理链，再对最终答案投票”。它常用于进一步提升多步推理稳定性，但代价比单条 CoT 更高。

实践里可以用一个简单决策原则：

1. 如果答案主要依赖事实记忆，先试直接回答。
2. 如果错误主要来自漏步骤、算错、漏条件，优先试 CoT。
3. 如果错误主要来自算术不稳或外部知识缺失，只加 CoT 往往不够，需要工具或检索。
4. 如果成本敏感，先测小样本，确认 CoT 的增益是否覆盖额外 token。

新手可理解的对比例子：
- 问“美国总统是谁？”这是事实问答。若只要当前答案，直接 prompt 更合适。
- 问“如果 A 的速度是 B 的 2 倍，谁更快？”这需要关系推导。即便题目很简单，CoT 也更符合任务结构。
- 问“一个订单系统中，满减、优惠券、会员折扣按什么顺序结算后最终价格最低？”这已经接近真实工程规则推理。直接回答很容易漏规则，CoT 或程序化规则执行会更可靠。

从评测角度看，Sprague 等人的核心提醒可以概括成一句话：**不是所有 benchmark 都应该默认启用 CoT，是否使用应由任务是否需要多步推理来决定。** 这也是为什么讨论 CoT 时，必须同时报告任务类型、prompt 模板、输出长度和评分方式，而不是只报一个总分提升。
