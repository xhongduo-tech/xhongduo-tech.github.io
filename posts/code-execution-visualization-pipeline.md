## 核心结论

代码执行结果要变成“人能读懂且尽量可信的可视化结论”，不能只做两步：先画图，再让模型看图。更稳的做法是把流程拆成一个可检查的渲染管线：

$$
C = G(D),\quad I = R(C),\quad U = S(I),\quad D' = M(U),\quad \text{NL} = L(D')
$$

这里，$D$ 是原始数据，白话说就是程序真正算出来的数；$G$ 是代码生成器，负责把数据组织成 Matplotlib 或 Plotly 脚本；$R$ 是渲染器，负责执行脚本得到图像；$S$ 是序列化器，负责把图像转成 base64 或文件路径；$M$ 是多模态理解器，负责“看图”后输出结构化结果；$L$ 是语言生成器，负责把结构化结果写成自然语言。

这条管线的核心价值不在“能看图”，而在“每一步都能验”。单个多模态模型对图表细节的读数会有误差，用户给出的经验值是 Matplotlib 图上约 5% 到 15%，但对整体趋势的判断通常更稳，约在 85% 左右。因此，工程上不应把视觉模型当作唯一真值源，而应把它当作“语义解释器”，把数值真值留在代码执行和验证环节。

玩具例子最能说明问题。设数据集 $D=\{(1,2),(2,4)\}$。程序先生成代码，代码渲染出折线图，图像再被编码成 base64，随后交给多模态模型读取。模型如果返回“两个点近似满足 $y\approx 2x$，趋势线性上升”，这个结论不是凭空来的，而是前面每个算子顺序传递的结果。如果中间加入 validator，白话说就是“专门检查数字和几何关系的小模块”，还可以验证斜率、坐标范围、轴标签是否和原始数据一致。

真实工程例子比玩具例子更严格。MatPlotAgent 这一类系统并不是“让大模型直接猜一张图”，而是采用“代码生成 -> 图像渲染 -> 视觉模型评分 -> 代码修正”的反馈闭环。这样做的含义是，视觉模型不只是终点解释器，还是渲染质量的评估器。这个设计能显著提升图表的正确率和表达质量，尤其适合科研图表、报表自动生成、智能分析助手这类不能只靠感觉判断的场景。

## 问题定义与边界

问题定义可以写得很直接：代码执行 Agent 需要把数值结果转成图，再把图转成自然语言结论。这里至少同时存在两类任务。

第一类是数值保真。白话说，图里显示的点、线、柱子，必须对应程序真实算出来的值。第二类是语义表达。白话说，系统最终要能说清楚“是在上升、下降、波动，还是两组数据存在对比关系”。

如果只做“直接问图像”，问题会立刻出现。视觉模型可能能说出“整体在上升”，但不一定能稳定读出精确坐标，也不一定知道横轴和纵轴到底代表什么。对零基础读者来说，这一点尤其重要，因为新手最容易把“模型说得像”误认为“模型说得对”。

下面这个表格能看出两种方案的边界差异：

| 方案 | 输入 | 输出 | 精度来源 | 可控性 | 可验证性 | 适用场景 |
|---|---|---|---|---|---|---|
| 直接问图像 | 图片 | 自然语言描述 | 主要依赖视觉模型 | 低 | 低 | 快速演示、低风险原型 |
| Agent 渲染 + Schema | 数据、代码、图片、结构化约束 | 结构化结果 + 自然语言 | 数值执行 + 视觉理解共同决定 | 高 | 高 | 科研图表、报表系统、智能体工具链 |

这里的 Schema，白话说就是“强制输出格式”。例如，不允许模型只回答“趋势不错”，而必须输出：

| 字段 | 含义 |
|---|---|
| `points_detected` | 识别到的关键点 |
| `trend` | 上升/下降/震荡 |
| `x_label` | 横轴含义 |
| `y_label` | 纵轴含义 |
| `confidence` | 模型自报置信度 |
| `needs_validation` | 是否需要数值复核 |

边界也要讲清楚。这个管线解决的是“数值结果如何可靠地变成图表和文字”。它不解决上游数据本身是否正确，也不保证视觉模型能像 OCR 那样无误读出每个像素对应的数值。对于科研结论、医学图表、金融图表这类高风险场景，视觉结果必须回到原始数据做复核，不能把图片解释当作最终事实。

## 核心机制与推导

把整条链路拆开看，工程设计会清楚很多。

### 1. 代码生成算子 $G$

$G(D)$ 的作用是把原始数据转换成绘图代码。这里的输入最好不是自然语言段落，而是结构化数据，例如：

```text
D = {
  "x": [1, 2],
  "y": [2, 4],
  "chart": "line",
  "title": "Toy Example"
}
```

这样做的原因很简单：结构化输入比自由文本更容易验证，也更适合自动生成代码。

### 2. 渲染算子 $R$

$R(C)$ 执行绘图代码，得到图像 $I$。这个阶段常被低估，但实际非常关键，因为 DPI、画布尺寸、坐标轴范围、颜色对比度、标记点大小都会影响后续视觉识别。

如果图中点很小、标签重叠、坐标轴范围不合理，多模态模型即使“看懂了趋势”，也可能读错具体值。也就是说，渲染质量会直接影响理解质量。

### 3. 序列化算子 $S$

$S(I)$ 把图像变成下游接口能消费的形式，常见有两种：

| 形式 | 白话解释 | 优点 | 缺点 |
|---|---|---|---|
| base64 | 把图片字节编码成文本 | 易于内嵌到 API 请求 | 文本体积大 |
| 文件路径/URL | 传文件位置 | 更省 token | 依赖存储和访问权限 |

这一层看起来只是“搬运图片”，但其实是接口边界的核心。因为从这里开始，后面的模型已经不再接触原始数值，只能通过图像进行理解。

### 4. 多模态理解算子 $M$

$M(U)$ 负责把视觉输入变成结构化理解结果 $D'$。这个结果不应该是直接一段散文，而应该先是中间表示，例如：

```json
{
  "chart_type": "line",
  "points_detected": [[1, 2], [2, 4]],
  "trend": "increasing",
  "slope_estimate": 2.0,
  "possible_relation": "y≈2x",
  "confidence": 0.86
}
```

这样设计的意义是，语言输出和视觉理解解耦。白话说，先让模型“报结构化事实”，再让另一个模块“写人话”。如果直接让模型同时负责看图和写结论，错误会混在一起，很难定位。

### 5. 语言输出算子 $L$

$L(D')$ 把结构化结果转成自然语言。它本身并不新增事实，只做重述和组织。例如把上面的结果写成：

“图中两个点分别接近 $(1,2)$ 和 $(2,4)$，整体呈线性上升，关系近似满足 $y\approx 2x$。”

玩具例子完整走一遍会更直观：

1. 输入数据：$D=\{(1,2),(2,4)\}$
2. 生成代码：$C=G(D)$
3. 渲染图片：$I=R(C)$
4. 编码图片：$U=S(I)$
5. 图像理解：$D'=M(U)$
6. 语言输出：$\text{NL}=L(D')$

如果把每一步都写成合同，白话说就是“输入输出协议”，那么系统可验证性就会明显增强。比如：

| 模块 | 输入 | 输出 | 验收条件 |
|---|---|---|---|
| `Generator` | 结构化数据 | 绘图代码 | 能运行、图类型正确 |
| `Renderer` | 绘图代码 | PNG/SVG | 无异常、尺寸符合要求 |
| `Serializer` | 图片字节 | base64/路径 | 可被下游成功读取 |
| `Vision Parser` | 图片 | JSON 结果 | 字段完整、单位明确 |
| `Narrator` | JSON 结果 | 文字总结 | 不新增未验证事实 |

真实工程里还会插入额外 Agent。比如 Numeric Calculator，白话说就是“只负责算数，不负责看图”；Geometry Validator，白话说就是“只负责检查图形上的几何关系”；Visualizer，白话说就是“只负责生成和修图”。多 Agent 的价值不在“模块多”，而在“职责单一”，这样每个环节出错时更容易定位和修正。

## 代码实现

下面给一个最小可运行版本，演示“数据 -> Matplotlib -> base64 -> 结构化描述”的主干流程。这个例子不调用真实多模态 API，而是先把接口边界搭好，方便新手理解系统结构。

```python
import io
import base64
import matplotlib.pyplot as plt

def render_plot_to_base64(x, y, title="Toy Example"):
    assert len(x) == len(y) and len(x) > 0

    fig, ax = plt.subplots(figsize=(4, 3), dpi=120)
    ax.plot(x, y, marker="o")
    ax.set_title(title)
    ax.set_xlabel("x")
    ax.set_ylabel("y")
    ax.grid(True, alpha=0.3)

    buf = io.BytesIO()
    fig.savefig(buf, format="png", bbox_inches="tight")
    plt.close(fig)

    raw = buf.getvalue()
    b64 = base64.b64encode(raw).decode("utf-8")

    assert len(raw) > 0
    assert len(b64) > 0
    return b64

def numeric_validator(x, y):
    assert len(x) == len(y)
    if len(x) < 2:
        return {"trend": "unknown", "slope": None}

    dx = x[-1] - x[0]
    dy = y[-1] - y[0]
    slope = dy / dx if dx != 0 else None
    trend = "increasing" if dy > 0 else "decreasing" if dy < 0 else "flat"

    return {"trend": trend, "slope": slope}

def build_natural_language(parsed):
    trend_map = {
        "increasing": "线性上升",
        "decreasing": "线性下降",
        "flat": "基本持平",
        "unknown": "趋势未知"
    }
    slope = parsed["slope"]
    trend = trend_map[parsed["trend"]]
    if slope is None:
        return f"图表整体{trend}。"
    return f"图表整体{trend}，斜率约为 {slope:.2f}。"

x = [1, 2]
y = [2, 4]

b64 = render_plot_to_base64(x, y)
validated = numeric_validator(x, y)
text = build_natural_language(validated)

assert validated["trend"] == "increasing"
assert abs(validated["slope"] - 2.0) < 1e-9
assert "线性上升" in text
assert isinstance(b64, str) and len(b64) > 100

print(text)
print(b64[:60] + "...")
```

这段代码里有三个关键点。

第一，`render_plot_to_base64` 对应的是 $R$ 和 $S$。它把图渲染到内存，再编码成 base64。这样不依赖文件系统，适合直接发给 API。

第二，`numeric_validator` 不看图片，只看原始数据。它的作用是给视觉模型提供“地面真值”。白话说，就算视觉模型把图上的斜率看成 1.9，只要数值验证器算出真实值是 2.0，系统就知道应该以哪个为准。

第三，`build_natural_language` 只根据已验证的结构化结果写自然语言。这样能避免“看图模块顺手脑补”。

如果要接入真实多模态模型，通常会在 `b64` 生成后再发一个请求，请求内容包括：

| 顺序 | 操作 | 数据格式 |
|---|---|---|
| 1 | 执行绘图脚本 | Python 对象 |
| 2 | 保存 PNG 到内存 | 二进制字节 |
| 3 | 转 base64 | 字符串 |
| 4 | 发送给多模态模型 | JSON + 图片字段 |
| 5 | 返回结构化理解 | JSON |
| 6 | 对比 validator 结果 | Python 对象/JSON |
| 7 | 生成自然语言结论 | 文本 |

一个真实工程例子是自动分析销售报表。假设某电商 Agent 先从数据库拉取最近 30 天销售额，再生成折线图，然后由多模态模型给出“后半段增速放缓，但整体仍上升”的总结。这里如果没有 validator，模型可能因为坐标轴压缩或标签密集，把“增速放缓”误说成“开始下降”。因此生产系统通常会同时保留原始时序数据、统计摘要和图像解释结果，三者交叉验证后才输出。

## 工程权衡与常见坑

第一类坑是把视觉模型当测量仪。它适合判断趋势，不适合单独承担精确读数任务。尤其当坐标轴刻度密、点位重叠、线条很细时，5% 到 15% 的误差并不意外。工程上应该把“读图”定位成语义增强，而不是数值主源。

第二类坑是渲染参数不稳定。同一份数据，如果画布宽高、DPI、颜色、字体、线宽不同，模型读取效果可能显著变化。很多“模型理解错了”的根因，其实是渲染质量太差。

第三类坑是输出格式过松。如果 prompt 只写“请分析这张图”，模型会自由发挥；如果明确要求输出字段、单位、趋势、置信度、是否需要复核，结果会稳定很多。

下面是常见问题与缓解方式：

| 常见坑 | 表现 | 根因 | 缓解措施 |
|---|---|---|---|
| 坐标被忽略 | 只说“上涨”，不说涨多少 | 模型偏语义，不偏测量 | 强制输出点位和坐标范围 |
| 比例失真 | 斜率判断偏差大 | 轴范围、纵横比不合理 | 固定坐标范围和画布比例 |
| 标签重叠 | 轴含义识别错误 | 文本太密、分辨率低 | 提高 DPI，减少标签密度 |
| 置信过高 | 说得很确定但与数据不符 | 模型自信与事实脱钩 | 增加 validator 和人工复核 |
| 直接问图像得分低 | 结果不可重复 | 缺少 schema 和中间态 | 先结构化，再自然语言 |

Plot2Code 这类基准的启发很直接：直接让模型从图像到代码或从图像到解释，虽然可用，但稳定性和可控性不够。尤其对零基础工程师来说，最大的风险不是“不会用”，而是“以为结果可信”。因此，系统设计必须显式区分三件事：

1. 哪些事实来自原始数据。
2. 哪些事实来自视觉模型观察。
3. 哪些文字只是对已有事实的重述。

一个实用做法是给每条结论都附上来源标签。例如：

- `source=data`: 斜率精确为 2.0
- `source=vision`: 图中趋势呈上升
- `source=language`: 可总结为“近似满足 y≈2x”

这样当三者冲突时，系统可以优先信任 `source=data`。

## 替代方案与适用边界

不是所有场景都需要完整渲染管线。选择方案时，核心标准是风险和可验证性要求。

| 方案 | 核心思路 | 适用场景 | 优点 | 风险 |
|---|---|---|---|---|
| Direct Asking | 直接把图发给模型提问 | Demo、原型、教学演示 | 快 | 不稳定、难验证 |
| MatPlotAgent 式反馈 | 代码生成后让视觉模型评分并修正 | 科研图表、复杂自动化分析 | 可迭代优化 | 系统更复杂 |
| TiInsight 式对话分析 | 结构化数据 + 图表推荐 + 交互问答 | BI、数据分析助手 | 交互性强 | 依赖数据接入和语义层 |
| 纯规则系统 | 不看图，只根据数据模板写结论 | 固定报表、强监管场景 | 最稳定 | 表达能力有限 |

对零基础到初级工程师，一个实用判断标准是：

如果你只是想“把结果解释给人看”，并且图表不承担审计责任，Direct Asking 可以作为起点。

如果你要做的是“代码执行 Agent 自动生成分析报告”，尤其还要让报告进入业务流程，那么至少要有三层：绘图代码、数值验证、结构化视觉输出。

如果你面对的是科研、医疗、金融、工业仪表这类高风险场景，视觉模型最多只能做辅助解释，最终结论必须以原始数据和规则校验为准。

MatPlotAgent/MatPlotBench 这类工作说明，多模态反馈式管线在简单和困难案例上都比“直接解码”更稳。原因并不神秘：它不是让一个模型一次性做完所有事，而是把问题分解成“能执行、能渲染、能评估、能修正”的闭环。对工程系统来说，这种可分解性比单次回答更重要。

最终可以把适用边界归纳为一句话：当你需要“可解释”和“可复核”时，渲染管线必须显式存在；当你只需要“快速得到一个大致说法”时，直接问图像才有成本优势。
