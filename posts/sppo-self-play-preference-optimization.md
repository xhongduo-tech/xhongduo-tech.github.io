## 核心结论

SPPO（Self-Play Preference Optimization，自博弈偏好优化）可以看作一种“少依赖人工标注的对齐训练”方法。它不先要求人工把所有回答排好优劣，而是让模型在同一条 prompt 上生成多个候选回答，再由偏好模型判断这些回答谁更优，最后把“更常赢”的回答概率直接往上推。

它和 DPO、IPO 的关键区别不在于“也用了偏好数据”，而在于它直接拟合回答本身应当有多大概率，而不是只拟合 winner 和 loser 之间的相对差距。对零基础读者可以先这样理解：DPO 更像“让 A 比 B 更容易被选中”，SPPO 更像“让 A 的真实出现概率提高到一个更合理的位置”。

新手版玩具例子可以直接类比成双人零和棋局。双人零和博弈，意思是一方赢多少，另一方就输多少，总收益相反。RLHF 在这里被写成“当前模型”和“它的历史版本或其他采样策略”之间的对抗：如果某个回答在多轮比较里经常赢，下一轮训练后它就更容易被模型再次采样出来。

下表先给出最重要的差异：

| 方法 | 训练信号来源 | 主要更新目标 | 对数据稀疏的表现 |
|---|---|---|---|
| SPPO | 自博弈生成的偏好比较 | 直接提高被偏好回答的概率 | 通常更稳 |
| DPO | 偏好对 `(winner, loser)` | 拉开 winner/loser 的对数概率差 | 稀疏时可能只学到“相对更好” |
| IPO | 偏好对 + 概率间隔约束 | 控制 preference margin | 轻量，但仍偏相对比较 |
| 传统 RLHF | 奖励模型 + PPO | 最大化奖励期望 | 成本高、训练更复杂 |

## 问题定义与边界

SPPO 要解决的问题很具体：给定一条 prompt，模型可能写出多种回答，我们希望它更常输出“在人类偏好或偏好模型看来更好”的那一些。

这里的“偏好模型”可以理解成一个自动裁判，比如 PairRM。它不是直接生成答案，而是接收多个候选回答，判断谁更好。SPPO 关心的核心量是胜率。胜率就是某个回答在候选池中与其他回答两两比较时，赢的频率。

一个新手版例子：

同一条 prompt 下采样 5 个回答，偏好模型两两比较后，得到：

| 回答 | 胜场 | 总对局 | 胜率 |
|---|---:|---:|---:|
| A | 4 | 4 | 1.00 |
| B | 3 | 4 | 0.75 |
| C | 2 | 4 | 0.50 |
| D | 1 | 4 | 0.25 |
| E | 0 | 4 | 0.00 |

SPPO 的目标不是只记住 “A 比 E 强”，而是把这组胜率转成训练信号，让模型下次更倾向于生成 A、B，降低 D、E 的概率。

它的边界也很明确：

| 边界条件 | 为什么重要 |
|---|---|
| 偏好模型要尽量稳定 | 如果裁判自己漂移很大，训练方向会偏 |
| 每个 prompt 最好有多个候选 | 候选太少时，胜率估计会非常粗糙 |
| 需要历史对手池 | 否则模型容易只在当前分布里自我强化 |
| 不等于“完全不需要人工” | 偏好模型本身通常仍来自人工偏好数据 |

简化流程可以写成：

`prompt -> 采样候选 -> 偏好评分 -> 计算胜率 -> 权重更新`

真实工程例子里，论文做法不是让模型永远和“当前自己”比，而是和历史策略、不同温度采样策略形成连续对局池。这么做的原因很直接：如果你只和当下的自己比，模型可能会把一种局部风格不断放大，最后在更广泛分布上退化。

## 核心机制与推导

SPPO 的核心更新式是：

$$
\pi_{t+1}(y|x)\propto \pi_t(y|x)\cdot \exp(\eta \cdot P(y \succ \pi_t|x))
$$

这里：

- $\pi_t(y|x)$ 是当前策略在 prompt $x$ 下生成回答 $y$ 的概率。
- $\eta$ 是学习率，意思是每轮更新迈多大步。
- $P(y \succ \pi_t|x)$ 是回答 $y$ 相对当前策略样本池的平均胜率。

白话解释是：原本就有一定概率的回答，如果它在比较里更常赢，就乘上更大的指数权重；赢得少，就乘上较小权重。之后再归一化，得到下一轮概率分布。

最经典的玩具例子是两个回答 A/B。初始概率都为 0.5，胜率分别是 0.8 和 0.2，取 $\eta=1$：

$$
w_A = 0.5 \cdot e^{0.8} \approx 1.11
$$

$$
w_B = 0.5 \cdot e^{0.2} \approx 0.61
$$

归一化后：

$$
\pi_{t+1}(A|x)=\frac{1.11}{1.11+0.61}\approx 0.65,\quad
\pi_{t+1}(B|x)=\frac{0.61}{1.11+0.61}\approx 0.35
$$

表格看更直观：

| 回答 | 旧概率 | $\exp(\eta \cdot P)$ | 归一化前权重 | 新概率 |
|---|---:|---:|---:|---:|
| A | 0.50 | $e^{0.8}\approx 2.23$ | 1.11 | 0.65 |
| B | 0.50 | $e^{0.2}\approx 1.22$ | 0.61 | 0.35 |

为什么论文又写成平方差回归形式？因为直接在大模型上操作完整分布不现实，所以可以转成对 `log π` 的拟合：

$$
\mathbb{E}_{x,y\sim \pi_t}\left[\left(\log \pi(y|x)-(\eta\cdot P(y\succ \pi_t|x)-\log Z_{\pi_t}(x))\right)^2\right]
$$

这里的 $\log Z_{\pi_t}(x)$ 是归一化项，可以理解成“把所有回答概率重新缩放到和为 1 所需的校正量”。没有它，单个回答的权重能涨，但整个分布不一定合法。

这也是 SPPO 理论上比只比较 winner/loser 更完整的原因。DPO/IPO 更像学习“相对顺序”，而 SPPO 在拟合“顺序 + 绝对概率位置”。当数据很稀疏时，这个差别会非常关键。

## 代码实现

实现 SPPO 不需要先理解全部博弈论细节，但必须抓住四个步骤：

1. 同一 prompt 采样多个回答。
2. 用偏好模型评估候选之间的相对优劣。
3. 把比较结果变成每个回答的胜率。
4. 用胜率构造目标，回归到策略网络的对数概率。

下面是一个可运行的 Python 玩具实现，演示“两个回答按胜率做一次 SPPO 更新”的核心逻辑：

```python
import math

def sppo_step(probs, win_rates, eta=1.0):
    assert len(probs) == len(win_rates)
    assert abs(sum(probs) - 1.0) < 1e-9
    assert all(p > 0 for p in probs)
    assert all(0.0 <= w <= 1.0 for w in win_rates)

    unnormalized = [p * math.exp(eta * w) for p, w in zip(probs, win_rates)]
    z = sum(unnormalized)
    new_probs = [u / z for u in unnormalized]

    assert abs(sum(new_probs) - 1.0) < 1e-9
    return new_probs

old_probs = [0.5, 0.5]
win_rates = [0.8, 0.2]

new_probs = sppo_step(old_probs, win_rates, eta=1.0)

assert round(new_probs[0], 2) == 0.65
assert round(new_probs[1], 2) == 0.35
assert new_probs[0] > old_probs[0]
assert new_probs[1] < old_probs[1]

print(new_probs)
```

更接近训练流程的伪代码如下：

```python
for prompt in batch:
    responses = [model.sample(prompt, strategy=s) for s in strategy_pool]
    pair_scores = pairrm.compare_all(prompt, responses)
    win_rates = estimate_win_rates(pair_scores)
    targets = eta * win_rates - log_Z_estimate(prompt, responses)
    log_probs = model.log_probs(prompt, responses)
    loss = mse(log_probs, targets)
    loss.backward()
```

输入输出关系可以整理成表：

| 输入/中间量 | 含义 | 输出作用 |
|---|---|---|
| `prompt` | 用户问题 | 触发采样 |
| `responses` | 同一 prompt 的多个候选回答 | 构造对局池 |
| `pair_scores` | 偏好模型比较结果 | 转成胜率 |
| `win_rates` | 每个回答赢的频率 | 作为主要训练信号 |
| `log_Z_estimate` | 归一化项估计 | 修正目标尺度 |
| `loss` | 平方误差目标 | 更新模型参数 |

真实工程例子里，假设你在做一个客服回答模型。单一 prompt “用户要求退款但订单已超时” 可以采样 6 个版本：严格拒绝、政策解释、先道歉再解释、给出人工渠道、混合方案、过度承诺。PairRM 或内部偏好模型会更偏向“合规且语气好”的回答。SPPO 会让模型提高这类回答的生成概率，而不是只记住“方案 4 比方案 1 好”。

## 工程权衡与常见坑

最常见的误解是：既然都是偏好学习，为什么不直接上 DPO？答案是 DPO 足够轻，但它优化的是成对差值，不一定能恢复每个回答应该占多大概率。

| 问题 | 影响 | 解决思路 |
|---|---|---|
| 每个 prompt 只有一对偏好 | 容易只学到相对顺序，不学绝对概率 | 尽量同 prompt 采样多候选，用 SPPO 胜率回归 |
| 忽略 $\log Z$ 动态 | 概率更新方向可能偏 | 用历史策略或当前候选池估计归一化项 |
| 对手池过窄 | 模型在单一分布上自我强化 | 混入历史 checkpoint、不同温度策略 |
| 偏好模型不稳定 | 错把噪声当信号 | 固定裁判版本，定期做一致性检查 |
| 候选回答太同质 | 胜率区分度低 | 提高采样多样性，如调温度/top-p |

一个新手常见坑是把 tie 当作“不更新”。这在工程上很诱人，因为省事，但含义其实很强：你等于假设偏好模型在这一段分布上没有提供任何额外信息。如果裁判模型本身在漂移，或者候选池变化很大，这种偷懒做法会让训练逐渐偏离真实优势方向。

另一个坑是只保留 winner/loser 二值结果。数据稀疏时，DPO/IPO 会把“winner 比 loser 好”这个关系学得很强，但 winner 的绝对概率可能仍然不够高。SPPO 的价值正是在这里：它不是只拉开差值，而是试图把 winner 从 0.5 真正推到 0.7 一类更合理的位置。

估计 $\log Z$ 的一个简化流程是：

1. 对同一 prompt 保留当前轮候选集合。
2. 用这些候选的指数权重求局部归一化常数。
3. 再结合历史策略样本做平滑。
4. 避免把单轮候选池的波动直接当成全局真值。

## 替代方案与适用边界

SPPO 不是“全面替代 DPO”。它更像在偏好样本有限、又希望策略更新更稳定时的一种更强方法。

| 方法 | 样本需求 | 更新目标 | 泛化与稳定性 |
|---|---|---|---|
| SPPO | 中等，需要多候选或历史池 | 直接拟合回答概率 | 对自博弈泛化更友好 |
| DPO | 较低，只要偏好对 | 拉开 winner/loser 差距 | 轻量，但可能偏相对排序 |
| IPO | 较低 | 优化偏好间隔 | 比 DPO 更平滑一些 |
| 传统 RLHF(PPO) | 高，需要奖励模型和在线训练 | 最大化奖励 | 灵活，但训练成本最高 |

如果你的偏好模型非常强，数据也很丰富，DPO/IPO 往往是更便宜的第一步，因为实现简单、调参直接。反过来，如果你希望模型在“与不同历史版本对抗”时保持稳定，并且不想过度依赖人工标注，SPPO 更合适。

新手版选择建议可以写得很实用：

- 算力紧、先求能跑：先用 DPO。
- 有偏好模型、想减少人工标注依赖：考虑 SPPO。
- 需要更强理论保证，尤其关心自博弈稳定性：优先 SPPO。
- 奖励函数清晰、能承受在线 RL 成本：传统 RLHF 仍然有效。

更准确地说，SPPO 的适用边界不是“它总是更强”，而是“它在多候选、自博弈、概率拟合这三个条件都成立时更占优”。如果你根本拿不到可靠偏好裁判，或者每个 prompt 只能生成一个回答，那它的优势就很难发挥出来。
