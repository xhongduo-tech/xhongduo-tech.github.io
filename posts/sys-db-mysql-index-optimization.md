## 核心结论

MySQL 索引优化的核心，不是“给常用列都加索引”，而是让查询条件尽可能沿着 B+ 树的有序键值向下定位，减少扫描行数、减少回表次数、减少随机 I/O。

B+ 树可以理解为“按顺序组织的多层目录”。查询时，数据库不是一行一行试，而是先根据索引键找到大致区间，再缩小到更小范围，最后拿到目标记录。只要查询条件与索引顺序匹配，代价通常远小于全表扫描。

真正有效的优化通常只有三类：

| 优化点 | 本质 | 直接收益 |
|---|---|---|
| 联合索引顺序合理 | 让等值条件先定位、范围条件后裁剪 | 扫描更少 |
| 覆盖索引 | 查询所需列都在索引里 | 减少回表随机 I/O |
| 避免索引失效 | 不让函数、隐式转换、错误列顺序破坏有序性 | 保持执行计划稳定 |

“有索引”和“没索引”的差别，不只是快一点，而是查询路径完全不同：

| 场景 | 读取路径 | I/O 特征 |
|---|---|---|
| 无索引 | 全表顺序扫描，逐行判断条件 | 扫描行数大 |
| 有合适索引 | 先定位索引页，再顺着叶子节点读区间 | 定位快，扫描集中 |
| 有覆盖索引 | 只读索引页，不回主键记录 | 随机 I/O 更少 |

玩具例子：查询 `user_id = 1001`，如果 `user_id` 有索引，就像查字典时先按首字母跳到某一页，再在这一页附近顺着找；如果没有索引，就像把整本字典从第一页翻到最后一页。

优化目标始终是平衡三件事：读性能、写放大、维护成本。索引越多，查询未必越快，写入通常一定更慢。

## 问题定义与边界

索引优化要解决的主要问题有三类：

1. 查询退化成全表扫描，扫描行数过大。
2. 命中了索引，但回表次数太多，随机 I/O 成本高。
3. 建了联合索引，但查询条件没有按最左前缀匹配，索引利用率低。

这里的“最左前缀”是指：联合索引按列顺序组织，只有从最左边开始连续匹配，数据库才能沿着这棵树持续向下定位。比如索引是 `(a, b, c)`，那么 `a`、`a+b`、`a+b+c` 都可能有效，但单独查 `b` 或 `b+c` 一般不能直接利用这棵索引完成定位。

索引能稳定发挥作用的典型场景：

| 条件类型 | 是否容易利用索引 | 说明 |
|---|---|---|
| 等值条件 `=` | 是 | 最适合做精确定位 |
| 范围条件 `> < BETWEEN` | 是，但会截断后续列 | 能划区间，但右侧列通常不能继续用于定位 |
| 排序 `ORDER BY` | 有条件可用 | 需与索引顺序兼容 |
| 覆盖查询 | 是 | 返回列都在索引中时最优 |
| 函数包裹列 | 常失效 | 破坏原始有序键值 |
| 隐式类型转换 | 常失效 | 让比较方式变化 |

边界也要讲清楚。索引不是万能的。下面这条 SQL 是初学者最容易误判的：

```sql
WHERE a = 1 AND b > 2 AND c = 3
```

如果索引是 `(a, b, c)`，那么通常能稳定利用的是 `a` 和 `b`，因为走到 `b > 2` 时，索引访问已经进入一个范围；一旦进入范围，`c` 往往不能继续用于“缩小索引定位区间”，只能在扫描到的记录上继续判断。

多列索引本质上是在有序空间里切片。对于索引 `(k1, k2, k3, ...)`，若条件是 `k1 = v1`，那么索引扫描区间可以表示为：

$$
(k_1, k_2, k_3, \dots) \in [(v_1, -\infty, -\infty, \dots), (v_1, +\infty, +\infty, \dots))
$$

这表示“先把第一列固定在 `v1` 这段范围里”，然后才可能继续利用第二列、第三列去细分。若第二列变成范围条件，例如 `k2 > v2`，区间会进一步收缩，但第三列之后通常不再用于继续定位。

所以索引优化的边界不是“有没有索引”，而是“查询条件能否与索引的有序结构对齐”。

## 核心机制与推导

先看联合索引 `(a, b, c, d)`。它不是四棵独立索引拼起来，而是一棵按 `(a, b, c, d)` 字典序排序的 B+ 树。所谓“字典序”，就是先比 `a`，`a` 相同再比 `b`，再比 `c`，最后比 `d`。

可以把这棵树理解成多维有序空间：

1. 第一层先按 `a` 分组。
2. 在某个 `a` 组内，再按 `b` 排序。
3. 在某个 `(a,b)` 组内，再按 `c` 排序。
4. 在某个 `(a,b,c)` 组内，再按 `d` 排序。

这就是为什么“列顺序决定命中层数”。

看一个最典型的玩具例子。索引为 `(a, b, c, d)`，查询为：

```sql
WHERE a = 1 AND b = 2 AND c > 3 AND d = 4
```

执行逻辑可以拆成三步：

1. `a = 1`：先定位到 `a=1` 这一大段。
2. `b = 2`：再缩小到 `(a=1,b=2)` 这一小段。
3. `c > 3`：从 `(a=1,b=2,c>3)` 这个起点开始顺着扫。

问题出在第 3 步。`c > 3` 不再是一个点，而是一个区间。既然现在扫的是一个区间，那么 `(a=1,b=2,c=4,d=?)`、`(a=1,b=2,c=5,d=?)`、`(a=1,b=2,c=6,d=?)` 都可能被读到。`d = 4` 不能再帮你继续“按树向下缩小范围”，只能在读出来的记录里再过滤。

可用一个更形式化的区间表示：

$$
(a,b,c,d) \in [(1,2,3,+\epsilon,-\infty), (1,2,+\infty,+\infty))
$$

意思是：扫描从 `a=1,b=2,c>3` 的起点开始，一直扫到 `(1,2)` 这一段结束。这个过程中，`d=4` 不是导航条件，而是过滤条件。

下面这张表可直接帮助判断联合索引命中层数：

| 索引列顺序 | 查询条件 | 可用于定位的列 | 说明 |
|---|---|---|
| `(a,b,c)` | `a=1` | `a` | 命中 1 层 |
| `(a,b,c)` | `a=1 AND b=2` | `a,b` | 命中 2 层 |
| `(a,b,c)` | `a=1 AND b>2` | `a,b` | `b` 作为范围，`c` 通常失效 |
| `(a,b,c)` | `a=1 AND c=3` | `a` | 缺失中间列 `b`，`c` 难继续利用 |
| `(a,b,c)` | `b=2 AND c=3` | 无或很弱 | 不满足最左前缀 |

覆盖索引也必须单独理解。所谓“覆盖索引”，就是查询要返回的列已经都在索引叶子节点里，不需要再回到主键记录里取值。这里的“回表”是指：先通过二级索引找到主键，再根据主键去聚簇索引或数据页拿完整行。

例如索引是 `(user_id, order_status, created_at, order_id, total_amount)`，查询是：

```sql
SELECT order_id, total_amount
FROM orders
WHERE user_id = 10 AND order_status = 'paid'
ORDER BY created_at DESC
LIMIT 10;
```

如果返回列只有 `order_id` 和 `total_amount`，而这两列已在索引中，那么数据库可以只读索引，不回表。这就是覆盖索引的价值：不仅“命中索引”，而且“结果直接从索引返回”。

可以把代价粗略理解为：

$$
查询总成本 \approx 索引定位成本 + 索引扫描成本 + 回表次数 \times 单次随机I/O成本
$$

覆盖索引的优化点，就是尽量把最后一项压到接近 0。

## 代码实现

下面用一个常见的订单表说明索引设计。先建表，再建索引。

```sql
CREATE TABLE orders (
  order_id BIGINT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  order_status VARCHAR(20) NOT NULL,
  total_amount DECIMAL(10,2) NOT NULL,
  created_at DATETIME NOT NULL,
  remark VARCHAR(255) DEFAULT NULL
);

-- 用于“查某用户某状态下最近订单”
CREATE INDEX idx_user_status_created
ON orders (user_id, order_status, created_at);

-- 用于覆盖查询的扩展版本
CREATE INDEX idx_user_status_created_cover
ON orders (user_id, order_status, created_at, order_id, total_amount);
```

查询写法：

```sql
SELECT order_id, total_amount
FROM orders
WHERE user_id = 1001
  AND order_status = 'paid'
ORDER BY created_at DESC
LIMIT 10;
```

这个查询的目标很明确：

1. `user_id` 和 `order_status` 做等值定位。
2. `created_at` 支持按时间顺序取最近 10 条。
3. `order_id, total_amount` 如果包含在索引里，就能覆盖返回。

`EXPLAIN` 的简化示意可以写成：

```sql
EXPLAIN
SELECT order_id, total_amount
FROM orders
WHERE user_id = 1001
  AND order_status = 'paid'
ORDER BY created_at DESC
LIMIT 10;
```

可能看到类似结果：

| id | table | type | key | rows | Extra |
|---|---|---|---|---|---|
| 1 | orders | ref | idx_user_status_created_cover | 10 | Using where; Using index |

这里要读懂两个关键词：

- `Using index`：表示覆盖索引，返回列直接从索引读取。
- `Using where`：表示命中索引后，仍有条件在存储引擎或服务器层继续判断，这并不一定是坏事。

如果你改成下面这个查询：

```sql
SELECT order_id, total_amount, remark
FROM orders
WHERE user_id = 1001
  AND order_status = 'paid'
ORDER BY created_at DESC
LIMIT 10;
```

因为 `remark` 不在索引里，通常就不能覆盖了，需要回表。

再给一个可运行的 Python 玩具程序，模拟“最左前缀 + 范围截断”的匹配规则：

```python
from typing import List, Tuple

def used_prefix(index_cols: List[str], eq_cols: List[str], range_col: str | None = None) -> List[str]:
    """
    简化模拟:
    1. 从最左列开始连续匹配等值条件
    2. 第一个范围列可以使用
    3. 范围列之后停止
    """
    used = []
    eq_set = set(eq_cols)

    for col in index_cols:
        if col in eq_set:
            used.append(col)
            continue
        if range_col == col:
            used.append(col)
        break
    return used

assert used_prefix(["a", "b", "c"], ["a", "b"], None) == ["a", "b"]
assert used_prefix(["a", "b", "c"], ["a"], "b") == ["a", "b"]
assert used_prefix(["a", "b", "c", "d"], ["a", "b"], "c") == ["a", "b", "c"]
assert used_prefix(["a", "b", "c"], ["b"], None) == []
assert used_prefix(["user_id", "order_status", "created_at"], ["user_id", "order_status"], "created_at") == [
    "user_id", "order_status", "created_at"
]

print("all assertions passed")
```

这个程序当然不是 MySQL 执行器，但它把最核心的判断规则表达清楚了：联合索引能用到哪里，取决于“是否从最左连续匹配”以及“范围条件出现在哪一列”。

真实工程例子：电商订单中心经常要查“某用户最近已支付订单”。这时 `(user_id, order_status, created_at)` 通常比 `(order_status, created_at, user_id)` 更合理，因为前者先按用户切分，再按状态缩小，再按时间取最近记录，和查询模式一致。若高频接口只返回 `order_id、total_amount`，则可进一步把这两列并入索引，减少回表。

## 工程权衡与常见坑

索引优化是典型的工程权衡。每新增一个索引，读取可能变快，但写入一定变重。因为 `INSERT`、`UPDATE`、`DELETE` 不仅要改数据，还要维护对应 B+ 树结构。页分裂、页合并、额外日志、缓存占用，都会带来成本。

下面这张表概括常见策略与影响：

| 索引策略 | 选择性 | 写放大 | 是否易覆盖 | 风险 |
|---|---|---|---|---|
| 单列高选择性索引 | 高 | 低到中 | 一般 | 不能兼顾排序与多条件 |
| 合理联合索引 | 中到高 | 中 | 高 | 设计错误会浪费 |
| 覆盖索引追加返回列 | 中 | 高 | 很高 | 索引体积膨胀 |
| 低选择性列单独建索引 | 低 | 中 | 低 | 可能几乎无收益 |
| 冗余重复索引 | 不一定 | 高 | 不一定 | 白白增加维护成本 |

常见坑有五类。

第一，低选择性列。  
“选择性”可以理解为“这一列能把数据区分开的能力”。例如性别、布尔状态、是否删除，这类列往往只有几个取值，单独建索引时很容易扫描出大量记录，优化器可能直接放弃不用。

第二，范围条件放得太早。  
如果查询经常是：

```sql
WHERE user_id = ? AND created_at >= ? AND order_status = 'paid'
```

而索引是 `(user_id, created_at, order_status)`，那么 `created_at` 作为范围列后，`order_status` 往往无法继续用于索引定位。若业务更常见的是先按用户和状态过滤，再取一段时间，索引 `(user_id, order_status, created_at)` 更稳。

第三，对索引列做函数或表达式运算。  
例如：

```sql
WHERE DATE(created_at) = '2026-03-12'
WHERE UPPER(email) = 'A@B.COM'
```

这里函数作用在列上，数据库无法直接利用原始索引有序性，往往退化。更好的写法通常是改写条件，使列保持原值比较，例如把日期条件改成时间范围。

第四，隐式类型转换。  
例如 `user_id` 是整数列，却写成：

```sql
WHERE user_id = '1001'
```

某些情况下优化器仍能处理，但复杂表达式、字符集差异或跨类型比较时可能让索引利用变差。工程上应尽量保证参数类型与列类型一致。

第五，误把“命中索引”当成“查询已经优化好”。  
很多慢查询其实已经用了索引，但 `rows` 依然很大，或者回表次数很多。真正要看的不是有没有 `key`，而是扫描多少行、是否覆盖、排序是否借助索引完成。

真实工程例子继续展开。订单表上有索引 `(user_id, order_status, created_at, order_id, total_amount)`，用于接口“查询某用户最近 10 条已支付订单”。这个设计的收益是：

1. `user_id` 定位单个用户数据范围。
2. `order_status` 再缩小到已支付订单。
3. `created_at` 直接支持时间顺序访问。
4. `order_id, total_amount` 作为返回列实现覆盖。

但代价也明确：

1. 索引更宽，占用更多磁盘和缓冲池。
2. 下单、改状态时维护成本增加。
3. 如果接口后来频繁增加 `remark`、`coupon_name` 等返回列，覆盖能力又会丢失，不能无限追加列来补。

所以工程上不要追求“把所有接口都做成覆盖索引”，而要抓住真正热点查询。

## 替代方案与适用边界

并不是所有性能问题都该靠单表索引解决。若场景本身不适合，继续堆索引只会增加复杂度。

下面是常见替代方案：

| 替代策略 | 适用边界 | 优势 | 代价 |
|---|---|---|---|
| 缓存 | 热点读多、重复查询多 | 延迟低，减数据库压力 | 一致性处理复杂 |
| 聚合统计表 | 报表、计数、汇总类 | 查询极快 | 需要异步维护 |
| 物化视图式结果表 | 写多读少、离线分析 | 避免在线复杂索引 | 实时性较差 |
| 分库分表 | 数据量极大、单表已成瓶颈 | 降低单表规模 | 开发运维复杂 |
| 搜索引擎 | 模糊搜索、多字段检索 | 检索能力强 | 架构更复杂 |

初级例子：日志表写入极其频繁，如果你试图给 `level`、`service_name`、`created_at`、`trace_id` 建很多二级索引，写入吞吐通常会明显下降。这类表更常见的做法是：

1. 在线表只保留极少索引，保障写入。
2. 定期把数据汇总到统计表或物化结果表。
3. 查询统计时走汇总表，而不是在原始日志表上做复杂过滤。

如果需求是“查看最近 100 条错误日志”，有时甚至可以只保留基于时间的关键索引，再结合 `ORDER BY created_at DESC LIMIT 100` 做近似读取，而不是试图把所有过滤模式都编码进联合索引。

适用边界可以简单概括为：

1. 单点查找、范围查找、排序分页，优先考虑 B+ 树索引。
2. 高频热点重复读，优先考虑缓存。
3. 大量统计聚合，优先考虑汇总表。
4. 写入极重且查询模式分散，减少索引数量比盲目加索引更重要。
5. 模糊检索、多条件自由组合搜索，不应勉强用普通联合索引硬扛。

索引设计前，必须先看 `EXPLAIN`、慢查询日志、实际访问模式。脱离真实 SQL 讨论“最佳索引”没有意义。索引永远是为查询服务，而不是为表结构服务。
