## 核心结论

CLIPA 可以理解成一种“保持 CLIP 训练目标不变，但重写训练成本结构”的方法。CLIP 是一种图文对比学习模型，白话说，就是让正确的图片和文本在向量空间里更接近，让错误配对更远。CLIPA 没有发明新的损失函数，也没有把任务改成别的东西，它做的核心优化只有两件事：一是在大部分预训练阶段把图像和文本的 token 长度压短，二是在训练后段切回完整 token 做短程微调。

这件事成立的关键，不是“短 token 本身更聪明”，而是“大模型对短 token 更耐受”。论文把这个现象概括为逆缩放规律：模型容量越大，输入被压短后带来的性能损失越小，因此可以先用便宜的短序列完成大部分训练，再用少量全 token 训练把表达能力补回来。

如果用一句适合新手的描述，标准 CLIP 更像“从头到尾完整读材料再做对齐”，CLIPA 更像“先只看关键信息完成大部分学习，最后再补一次完整阅读”。目标还是图文对齐，但大多数算力花费已经被压缩掉了。

下面这张表可以先建立整体印象：

| 维度 | 标准 CLIP | CLIPA |
|---|---|---|
| 训练目标 | 图文双向对比学习 | 图文双向对比学习 |
| 图像 token | 通常全长度 | 预训练阶段压短，后期恢复 |
| 文本 token | 通常全长度 | 预训练阶段截短，后期恢复 |
| 训练策略 | 单阶段 | 两阶段或分阶段 |
| 算力开销 | 高 | 明显更低 |
| 主要收益来源 | 无特殊压缩 | token 缩短 + 阶段化训练 |

## 问题定义与边界

CLIP 训练最贵的部分，不是“算一个相似度矩阵”，而是编码图片和文本时的 Transformer。Transformer 是一种依赖自注意力机制的网络，白话说，它会让每个 token 和其他 token 两两交互；token 一多，计算量就会快速上涨。图像分支尤其昂贵，因为视觉 Transformer 往往要处理更多 token。

以视觉编码器为例，图像 token 数近似可以写成：

$$
N_v = \left(\frac{R}{p}\right)^2 + 1
$$

其中 $R$ 是输入分辨率，$p$ 是 patch size，也就是把图像切成多大一块一块再送进模型。比如在 $224 \times 224$ 分辨率下，`ViT-L/16` 的 patch size 是 16，那么 token 数就是：

$$
N_v = (224/16)^2 + 1 = 14^2 + 1 = 197
$$

这 197 个 token 进入注意力层后，单层复杂度近似与 $N_v^2$ 成正比。也就是说，图像 token 不是线性变贵，而是近似平方级变贵。

CLIPA 解决的问题边界非常明确：它解决的是“训练阶段太贵”，不是“CLIP 目标函数不够强”，也不是“互联网图文数据噪声太大”。它通过缩短输入长度和拆分训练阶段来降本，而不是通过改任务定义来提效。

| 项目 | CLIPA 要解决的 | CLIPA 不直接解决的 |
|---|---|---|
| 图像 token 过长 | 是 |  |
| 文本 token 过长 | 是 |  |
| 预训练总算力过高 | 是 |  |
| 数据噪声 |  | 是 |
| 损失函数设计 |  | 是 |
| 下游微调策略 |  | 是 |

边界也要说清楚。短 token 训练有效，不等于 token 可以无限压缩。压缩过头后，模型看到的信息不够，图文对齐能力就会明显掉下来。尤其是文本侧，如果只剩极短片段，很多决定语义的限定词、关系词和属性词会直接丢失。

## 核心机制与推导

CLIPA 沿用的仍然是 CLIP 的基本框架。设图像编码器为 $f_v$，文本编码器为 $f_t$。给定一组图文对 $(x_i, y_i)$，分别得到向量表示：

$$
z_i = f_v(x_i), \quad t_i = f_t(y_i)
$$

然后用点积加温度系数计算相似度：

$$
s_{ij} = \tau z_i^\top t_j
$$

这里的温度系数 $\tau$ 可以理解成一个缩放参数，白话说，它控制“相似和不相似之间拉开的力度”。训练时通常做双向对比学习，也就是“图到文”和“文到图”各算一次交叉熵损失：

$$
L = \frac{CE_{i2t} + CE_{t2i}}{2}
$$

关键点在于：CLIPA 没改上面这套东西。它改的是送进 $f_v$ 和 $f_t$ 的 token 长度，以及训练分成几个阶段。

先看一个玩具例子。假设标准 CLIP 的图像分支在某个配置下处理 197 个 token，而 CLIPA 在预训练阶段把它压到 17 个 token，那么单层注意力的理论主项会从：

$$
197^2 = 38809
$$

变成：

$$
17^2 = 289
$$

二者比例为：

$$
\left(\frac{17}{197}\right)^2 \approx 0.00745
$$

也就是大约只剩原来的 $1/134$。新手只要抓住一句话就够了：注意力矩阵从“接近两百乘两百”变成了“十几乘十几”，所以便宜很多。

但这里必须加一句工程上的修正：端到端训练速度不会真的提升 134 倍，因为总开销还包括 MLP、embedding、文本塔、数据加载、通信同步，以及最后全 token 微调的成本。所以这个比值是“注意力主项的理论上限”，不是整套训练的实际加速倍数。

再看 token 长度和代价的直观关系：

| 图像 token 数 $N_v$ | 单层注意力主项 $N_v^2$ | 相对 197 token 的比例 |
|---|---:|---:|
| 197 | 38809 | 1.00 |
| 65 | 4225 | 0.109 |
| 17 | 289 | 0.007 |

文本侧逻辑类似。文本 token 是把一句描述切成一个个词片后送进文本 Transformer。截短文本，本质上也是减少自注意力中的两两交互。论文结论表明，大模型对这种截短更有韧性，但也存在阈值，压得过狠仍然会损失语义。

真实工程里，CLIPA 的完整思路不是“一次性用短 token 训到底”，而是：

1. 预训练阶段：用更短的图像 token 和文本 token，以更低成本学习大规模图文对齐。
2. 恢复阶段：切回更高分辨率或更完整的文本长度，做短程微调，把前面压缩掉的一部分表达能力补回来。

这就是它能在明显更低算力下接近甚至达到标准 CLIP 性能的原因。省掉的是“大部分训练过程中的昂贵全 token 计算”，而不是最后完全不付恢复成本。

## 代码实现

实现 CLIPA 时，最重要的不是重写模型结构，而是把训练配置拆成两个阶段。工程上真正要控制的变量通常是 `image_size`、`patch_size`、`text_len`、`stage_steps` 和阶段切换点。

下面给一个可运行的 Python 玩具实现。它不训练模型，只计算不同配置下图像 token 数和注意力主项比例，帮助你验证“为什么缩短 token 会便宜很多”。

```python
def vit_tokens(image_size: int, patch_size: int) -> int:
    assert image_size % patch_size == 0
    return (image_size // patch_size) ** 2 + 1

def attention_cost(tokens: int, hidden_dim: int = 1024) -> int:
    return tokens * tokens * hidden_dim

# 标准配置：224 分辨率 + 16 patch
full_tokens = vit_tokens(224, 16)
assert full_tokens == 197

# 一种更短序列的玩具配置：64 像素输入，仍用 16 patch
short_tokens = vit_tokens(64, 16)
assert short_tokens == 17

full_cost = attention_cost(full_tokens)
short_cost = attention_cost(short_tokens)

ratio = short_cost / full_cost
assert round(ratio, 4) == round((17 / 197) ** 2, 4)
assert ratio < 0.01

print({
    "full_tokens": full_tokens,
    "short_tokens": short_tokens,
    "relative_attention_cost": round(ratio, 6),
})
```

如果把这个思路映射到训练流程，伪代码通常长这样：

```python
# stage 1: short-token pretraining
image_size = 112
patch_size = 16
text_len = 16
train_clip_like_model(
    image_size=image_size,
    patch_size=patch_size,
    text_len=text_len,
    stage="pretrain"
)

# stage 2: full-token finetuning
image_size = 224
patch_size = 16
text_len = 77
finetune_clip_like_model(
    image_size=image_size,
    patch_size=patch_size,
    text_len=text_len,
    stage="finetune"
)
```

这里的 `text_len` 指文本最大长度，白话说，就是一句话最多保留多少个 token；超出的部分会被截断。`stage="pretrain"` 和 `stage="finetune"` 不是论文规定的固定字段，而是工程上用来区分配置阶段的常见写法。

一个实现配置表可以帮助把论文机制映射到代码参数：

| 参数 | 预训练阶段 | 微调阶段 | 预期效果 |
|---|---|---|---|
| `image_size` | 小 | 大 | 图像 token 减少，训练更便宜 |
| `patch_size` | 固定或少改 | 固定 | 决定 token 网格密度 |
| `text_len` | 短 | 完整 | 文本塔更省算力 |
| `batch_size` | 可更大 | 通常回落 | 提高吞吐 |
| `stage_steps` | 长 | 短 | 大部分训练在低成本阶段完成 |
| `resume_from` | 无 | 预训练权重 | 保留已学到的对齐能力 |

真实工程例子可以这样理解：如果你手里只有 8 张 A100，直接从零训练完整 CLIP 往往代价过高；但如果先用较小分辨率和短文本长度做大规模预训练，再切到 224 或 336 分辨率做短程恢复，往往更有机会在预算内拿到可用的零样本检索或分类能力。这就是 CLIPA 这类方案最有现实价值的地方。

## 工程权衡与常见坑

第一个常见坑，是把 `ViT-L/14` 误解成“天然比 `ViT-L/16` 更省”。这是错的。`/14` 和 `/16` 说的是 patch size，不是模型自动压缩 token 的能力。在同样分辨率下，patch 越小，切出来的 patch 越多，token 往往越多。所以 `/14` 不会因为名字更大就更省，很多时候反而更贵。真正决定 CLIPA 省算力的，是训练阶段把输入长度压短，而不是单看模型命名。

第二个坑，是把文本长度压得过狠。文本截断会直接丢语义，尤其是在电商描述、长 caption、属性密集句子里更明显。经验上，过短如 4 token 往往风险很高，8 到 16 这类区间通常更稳，具体还要看数据分布。

第三个坑，是混淆不同论文或不同表格里的“加速倍数”。有的对比是只算预训练，有的是算总 GPU hours，有的是对比不同 baseline。`1/15`、`1/33`、`39x` 这类数字如果不看分母和实验设定，几乎一定会误读。

| 坑 | 影响 | 规避方式 |
|---|---|---|
| 把 `/14` 误认为更省 | 算力估计错误 | 看 token 数，不只看模型名 |
| 文本截断过狠 | 语义信息损失 | 先从 8 或 16 token 试验 |
| 只看“x 倍加速” | 收益判断失真 | 明确 baseline 和核算口径 |
| 忽略微调成本 | 低估总训练时间 | 预训练和恢复阶段分开算 |
| 把短 token 当万能解 | 高分辨率细节丢失 | 结合任务需求设置下限 |

还有一个实际权衡是：短 token 训练更便宜，但也会削弱细粒度信息。对粗粒度图文对齐，比如“猫”“狗”“雪山”“城市夜景”，模型通常还能学得不错；但对细粒度视觉差异，比如药片刻字、工业零件缺陷、长文本里的否定关系和属性组合，压缩过度就会明显伤性能。

## 替代方案与适用边界

CLIPA 最适合的场景，是“预算有限，但仍想保留 CLIP 式图文对齐能力”。如果你的目标是做一个强于入门 baseline、又不想支付完整 CLIP 训练成本的系统，它很有吸引力。

但它并不适合所有任务。对于强依赖长文本细节的任务，比如复杂商品描述理解、长指令图像检索，或者强依赖高分辨率局部信息的任务，比如遥感细粒度识别、病理图像分析，过短 token 可能让关键信息在预训练阶段就被丢掉，后面的短程微调也未必补得回来。

它也不是唯一的降本路径。蒸馏是把大模型知识压到小模型里；混合精度是用更低数值精度减少显存和算力；梯度累积是用多次小批次模拟大 batch；数据裁剪是减少训练样本或过滤低质量数据。这些方法和 CLIPA 并不互斥，但关注点不同。CLIPA 更像“从序列长度和训练阶段设计入手改成本结构”。

| 方案 | 适用场景 | 主要代价 |
|---|---|---|
| 标准 CLIP | 资源充足，追求上限 | 训练昂贵 |
| CLIPA | 算力受限，仍想保留较强性能 | 需要阶段化训练 |
| 蒸馏 | 已有教师模型，希望部署更轻 | 上限受教师和学生容量约束 |
| 混合精度 | 通用训练降本 | 不改变任务结构 |
| 数据裁剪 | 数据量过大或质量不稳 | 可能损失覆盖度 |

如果只做一个可用的图文检索 baseline，CLIPA 往往比完整 CLIP 更划算。如果目标是追求长文本语义完整性或高分辨率局部辨别能力，标准 CLIP 或更针对性的结构优化通常更稳。
