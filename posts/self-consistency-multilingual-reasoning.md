## 核心结论

Self-Consistency，简称 SC，可以直白理解为：同一道题让模型独立作答多次，再用多数票决定最终答案。它在多语言推理中的收益并不均匀。真正决定收益的，不是“采样次数变多了”，而是“某种语言内部是否本来就更容易收敛到同一个答案”。

在 MGSM 这类多语言小学数学推理基准上，高资源语言通常表现出更强的路径收敛性。这里的“路径收敛性”可以理解为：模型多次生成不同推理过程时，最终答案仍然更容易重复落到同一个值。于是，在相同采样预算下，英语、中文、孟加拉语这类训练覆盖较高的语言，往往更能从 SC 中获益；而泰语、斯瓦希里语、泰卢固语等低资源语言，即使增加采样，收益也常常更有限。

可以把这个现象压缩成一句话：SC 不是单纯的“采样放大器”，而是“错误分布过滤器”。如果正确答案本来就更稳定、错误答案更分散，那么投票就能把正确答案放大出来；如果错误答案同样稳定，或者错误答案虽然彼此不同但足以冲散正确票，SC 的增益就会明显缩小。

跨语言版本的 SC 进一步把“同一种语言内的多次采样”扩展为“多种语言、多条路径联合投票”。这类方法常见于 CLC、CLSP 一类工作。它们的价值在于：不同语言会暴露不同的推理偏差，联合投票能降低单一语言局部失稳的风险。但它们也有清晰边界：语言不是越多越好，低质量语言加入过多时，噪声会积累，最终反而拉低整体准确率。

## 问题定义与边界

本文讨论的问题很具体：在多语言数学推理中，SC 的提升是否在不同语言之间均衡；如果不均衡，差异来自哪里；在工程系统里又该如何利用这种差异，而不是被这种差异拖累。

这里的“多语言数学推理”主要指 MGSM 一类任务。MGSM 可以理解为：把同一批小学数学应用题翻译成多种语言，用来测试模型在不同语言下的数学推理能力。它特别适合研究语言差异，因为题目语义理论上保持不变，主要变化来自两部分：一是语言表达本身，二是模型对该语言的掌握程度。

为了避免讨论发散，本文边界固定在四点。

1. 只讨论生成式推理场景，也就是模型先生成一段思路，再输出最终答案。
2. 只讨论固定采样预算下的比较。例如每种语言都采样 $k$ 次，而不是某些语言额外获得更高预算。
3. 重点看答案级投票，不展开过程级验证器、程序执行器或外部裁判模型。
4. 讨论中的“高资源”和“低资源”是相对概念，指模型训练语料中覆盖更充分或更稀缺的语言，而不是语言本身的社会价值判断。

一个简化后的语言视图如下。它不是绝对结论，而是工程上常见的经验模式。

| 语言 | 资源级别 | 单语 SC 常见表现 | 跨语言投票中的角色 |
|---|---|---:|---|
| 英语 | 高 | 增益大，路径更集中 | 常作为锚点语言 |
| 中文 | 高 | 多数模型下较稳定 | 常与英语共同构成主投票源 |
| 孟加拉语 | 中高 | 常有明显增益 | 可提供有效补充证据 |
| 法语 | 中 | 表现较平衡 | 适合与英语配对 |
| 德语 | 中 | 通常优于低资源语言 | 可稳定参与投票 |
| 日语 | 中 | 对提示模板较敏感 | 适合作为辅助语言 |
| 俄语 | 中 | 有一定收益 | 适合做次级补充 |
| 泰语 | 低 | 提升偏小，路径分散 | 易引入噪声 |
| 斯瓦希里语 | 低 | 增益有限 | 需谨慎加权 |
| 泰卢固语等低资源语言 | 低 | 易受翻译和词法差异影响 | 只建议在验证后接入 |

采样预算也是一个硬约束。假设每种语言采样 $k=5$ 次，如果选用 6 种语言，总共就会得到 30 条推理路径。预算一旦固定，问题就不再是“能不能投票”，而是“预算更该分配给更多语言，还是该分配给更少语言但每种语言采样更多次”。这直接决定系统是在“扩展信息来源”，还是在“强化单语稳定性”。

一个最简单的玩具例子足以说明本质。题目是“12 个苹果分给 3 人，每人几个”。英语采样 5 次得到 `[4, 4, 4, 5, 4]`，泰语采样 5 次得到 `[4, 3, 5, 4, 6]`。两种语言都出现了正确答案 `4`，但英语已经接近共识，泰语则明显分裂。此时 SC 对英语更有效，因为多数答案天然稳；对泰语帮助较小，因为票被不同错误答案稀释了。

对新手而言，可以把它想成一个班级投票问题：如果一个班里大部分人都算对了，投票能很快选出正确答案；如果一个班里每个人都在犯不同的错，投票并不会自动变聪明。

## 核心机制与推导

SC 的基础形式可以写成：

$$
\hat y_{\text{SC}}=\arg\max_y \sum_{j=1}^{k}\mathbf{1}[y_j=y]
$$

其中，$y_j$ 表示第 $j$ 次采样得到的最终答案，$\mathbf{1}[\cdot]$ 是指示函数。它的含义很简单：如果第 $j$ 次采样的答案等于候选答案 $y$，就记 1 分，否则记 0 分。最后分数最高的答案获胜。

这个公式背后依赖一个关键前提：正确推理路径虽然表述形式可能不同，但更容易收敛到相同的最终答案；而错误路径更分散，彼此不容易形成稳定多数。只要这个前提成立，多采样加投票就能有效压低随机波动。

跨语言 SC 只是把“单语言内的多次采样”扩展为“多语言、多路径联合统计”：

$$
\hat y=\arg\max_y \sum_{i=1}^{m}\sum_{j=1}^{k}\mathbf{1}[y_{ij}=y]
$$

这里：

| 符号 | 含义 |
|---|---|
| $m$ | 使用的语言数量 |
| $k$ | 每种语言的采样次数 |
| $y_{ij}$ | 第 $i$ 种语言第 $j$ 条路径产生的最终答案 |
| $\hat y$ | 投票后的最终答案 |

这个式子的直观解释是：每种语言像一个小评审组，每组内部先产生若干票，最后把所有票汇总，选出票数最多的答案。

SC 与跨语言 SC 的差别可以压缩成下面这个表。

| 方法 | 投票范围 | 解决的问题 | 主要风险 |
|---|---|---|---|
| 单语 SC | 一种语言内的多条路径 | 降低同语种随机波动 | 被该语言能力上限锁住 |
| 跨语言 SC / CLC / CLSP | 多语言、多路径联合 | 利用语言互补性纠错 | 噪声语言会稀释共识 |

为什么高资源语言往往增益更高，可以先用一个最简概率视角理解。设单次采样得到正确答案的概率为 $p$，采样 $k$ 次后用多数投票，则最终投票正确的概率近似为：

$$
P(\text{majority correct})=\sum_{r=\lceil (k+1)/2\rceil}^{k}\binom{k}{r}p^r(1-p)^{k-r}
$$

这个式子不是在说真实推理路径彼此独立。真实系统里，多条路径常常相关，提示词、模型偏好、语言模板都会让采样之间带有耦合。这个公式的作用只是提供一个清晰直觉：当单次正确率 $p$ 已经高于 0.5 时，多数投票会迅速放大优势；当 $p$ 接近 0.5 或低于 0.5 时，投票就很难带来大幅收益。

例如，若 $k=5$：

| 单次正确率 $p$ | 多数投票正确率近似 |
|---:|---:|
| 0.56 | 0.61 |
| 0.60 | 0.68 |
| 0.68 | 0.81 |
| 0.75 | 0.90 |

这个表很关键。它说明 SC 收益不是线性增加的，而是对单次正确率非常敏感。高资源语言常见的情况是：单次正确率更高、答案抽取更稳定、路径更容易汇聚，于是 SC 会给出更明显的收益。低资源语言则常常处在“略高于随机但不够稳”的区间，所以投票能帮一点，但帮不到同样的幅度。

再看一个玩具推导。假设英语单次正确率约为 0.68，泰语约为 0.56，且都采样 5 次。英语更容易出现“5 条里至少 3 条答对”，因此多数票通常能强化正确答案；泰语更容易出现“2 条对、3 条错”或“正确答案与多个错误答案互相分裂”，于是投票只能部分缓解噪声，而无法稳定翻盘。

工程上更接近下面这个流程：

`语言输入 -> 每种语言采样 k 条 CoT -> 提取最终答案 -> 归一化答案 -> 汇总票数 -> 选择最高票答案`

注意中间多了一个“归一化答案”步骤。因为在真实系统里，模型可能输出 `4`、`4.0`、`4 apples`、`答案是 4`。如果不做归一化，这些本应属于同一票的结果会被拆开，投票统计就会失真。

如果把英语、中文、孟加拉语、泰语同时接入，一个常见现象是：英语与中文会在最终答案上彼此支持，孟加拉语偶尔提供有效增量，而泰语既可能帮助纠错，也可能带来额外分裂。这就是为什么“增加语言数”并不自动等于“提高准确率”。

## 代码实现

下面给出一个可以直接运行的 Python 示例。它不调用真实大模型，而是模拟“不同语言下多次采样得到的答案分布”，用来展示三件事：

1. 单语 SC 如何工作。
2. 跨语言联合投票如何工作。
3. 加权投票和答案归一化为什么在工程上很重要。

```python
from __future__ import annotations

import re
from collections import Counter, defaultdict
from typing import Dict, Iterable, List, Tuple


def normalize_answer(text: str) -> str:
    """
    归一化最终答案，避免 '4', '4.0', '答案是4个' 被当成三类。
    这里只处理数值类答案，足够演示 SC 的核心逻辑。
    """
    text = text.strip().lower()

    # 提取第一个数字，支持整数和小数
    match = re.search(r"-?\d+(?:\.\d+)?", text)
    if not match:
        return text

    value = float(match.group())
    if value.is_integer():
        return str(int(value))
    return str(value)


def self_consistency_vote(samples: Iterable[str]) -> Tuple[str, Counter]:
    normalized = [normalize_answer(x) for x in samples]
    counter = Counter(normalized)
    answer, _ = counter.most_common(1)[0]
    return answer, counter


def cross_language_vote(paths: Dict[str, List[str]]) -> Tuple[str, Counter]:
    merged = Counter()
    for answers in paths.values():
        normalized = [normalize_answer(x) for x in answers]
        merged.update(normalized)
    answer, _ = merged.most_common(1)[0]
    return answer, merged


def weighted_vote(
    paths: Dict[str, List[str]], weights: Dict[str, float] | None = None
) -> Tuple[str, Dict[str, float]]:
    if weights is None:
        weights = {lang: 1.0 for lang in paths}

    score = defaultdict(float)
    for lang, answers in paths.items():
        for ans in answers:
            score[normalize_answer(ans)] += weights.get(lang, 1.0)

    final_answer = max(score.items(), key=lambda item: item[1])[0]
    return final_answer, dict(score)


def accuracy_gain(before: float, after: float) -> float:
    return round(after - before, 1)


def main() -> None:
    # 玩具例子：英语更集中，泰语更分散
    english_paths = ["4", "4 apples", "4", "5", "4.0"]
    thai_paths = ["4", "3", "5", "答案是 4", "6"]

    en_ans, en_counter = self_consistency_vote(english_paths)
    th_ans, th_counter = self_consistency_vote(thai_paths)

    assert en_ans == "4"
    assert th_ans == "4"

    # 再加入一种中高资源语言
    multi_paths = {
        "en": english_paths,
        "th": thai_paths,
        "bn": ["4", "4", "4", "4", "7"],
    }

    final_ans, merged_counter = cross_language_vote(multi_paths)

    assert final_ans == "4"
    assert merged_counter["4"] == 10

    # 加权投票：假设泰语历史稳定性较弱，因此降低票权
    weights = {"en": 1.0, "bn": 0.9, "th": 0.5}
    weighted_ans, weighted_scores = weighted_vote(multi_paths, weights)

    assert weighted_ans == "4"
    assert weighted_scores["4"] > weighted_scores["5"]

    # 模拟不同语言在 SC 前后的准确率
    baseline = {
        "en": 60.2,
        "bn": 66.2,
        "th": 64.8,
    }
    after_sc = {
        "en": 75.2,
        "bn": 75.2,
        "th": 68.0,
    }

    gains = {
        lang: accuracy_gain(baseline[lang], after_sc[lang])
        for lang in baseline
    }

    assert gains["en"] == 15.0
    assert gains["bn"] == 9.0
    assert gains["th"] == 3.2

    print("English SC:", en_counter)
    print("Thai SC:", th_counter)
    print("Cross-language vote:", merged_counter)
    print("Weighted vote:", weighted_scores)
    print("Accuracy gains:", gains)


if __name__ == "__main__":
    main()
```

这段代码可以直接运行，输出会清楚展示三层信息。

| 组件 | 作用 | 为什么重要 |
|---|---|---|
| `normalize_answer` | 统一答案格式 | 避免同一答案被拆票 |
| `self_consistency_vote` | 单语内部多数投票 | 对应基础 SC |
| `cross_language_vote` | 跨语言合并投票 | 对应跨语言 SC |
| `weighted_vote` | 给不同语言设置票权 | 对应工程中的可靠性控制 |
| `accuracy_gain` | 计算净增 | 便于比较语言间收益差异 |

如果把它映射到真实模型系统，伪代码结构通常是这样：

```python
def collect_paths(model, question_by_lang, langs, k):
    outputs = {}
    for lang in langs:
        outputs[lang] = []
        prompt = question_by_lang[lang]
        for _ in range(k):
            cot = model.generate(
                prompt=prompt,
                temperature=0.7,
                top_p=0.95
            )
            answer = extract_final_answer(cot)
            outputs[lang].append(answer)
    return outputs


def extract_final_answer(text):
    """
    真实系统里这里通常最容易出错。
    需要统一处理数值、单位、货币、分数、小数和多余文本。
    """
    return normalize_answer(text)


def choose_answer(paths, mode="weighted", weights=None):
    if mode == "single":
        raise ValueError("single mode requires one language only")
    if mode == "equal":
        ans, _ = cross_language_vote(paths)
        return ans
    if mode == "weighted":
        ans, _ = weighted_vote(paths, weights=weights)
        return ans
    raise ValueError(f"unknown mode: {mode}")
```

新手最容易忽略的是：真实工程的难点通常不在“投票公式”，而在三个基础步骤。

1. `extract_final_answer` 是否稳定。
2. 各语言 prompt 是否对齐。
3. 不同语言是否真的值得等权投票。

如果这三步没做好，SC 看起来只是“多跑几次模型”，但最终提升会很不稳定。

再给一个更贴近工程的例子。假设题目是：

> A shop sells 3 notebooks for 12 dollars. How much does 1 notebook cost?

如果英语输出 `4`，中文输出 `4 元`，孟加拉语输出 `答案是 4`，泰语输出 `3`、`4`、`5` 混杂，那么系统真正需要做的不是简单计票，而是：

1. 抽取每条路径的最终数值。
2. 把不同语言中的同义答案统一。
3. 再决定是否让某些语言降权。

也就是说，SC 的核心不是“让模型想很多次”，而是“让可比较的答案聚合起来”。

## 工程权衡与常见坑

真实系统里，跨语言 SC 的难点不在公式，而在噪声管理。公式只负责聚合；系统成败往往取决于你把什么送进聚合器。

| 常见坑 | 现象 | 原因 | 规避策略 |
|---|---|---|---|
| 盲目增加语言数 | 语言越多反而越差 | 低质量语言把错误票带入多数 | 先做 3 到 6 种语言的消融实验 |
| 低资源语言 prompt 不稳 | 同题多次采样严重分裂 | 模型语言能力弱，提示模板不一致 | 先统一英语模板，再做受控翻译 |
| 答案抽取不统一 | 投票统计失真 | 同义答案被拆成多类 | 做数值、单位、格式归一化 |
| 翻译引入语义偏差 | 某些语言系统性出错 | 题意在翻译中变形 | 原题保留，翻译只作对齐视图 |
| 预算分配失衡 | 成本升高但收益有限 | 预算花在低价值语言上 | 高资源语言多采样，低资源语言少量补充 |
| 稳定但稳定地错 | 某语言内部高度一致，但答案错 | 翻译错、提示错或模型偏差固定 | 增加外部校验，不只看投票 |
| 多数票打平 | 两个答案票数相同 | 语言间意见分裂 | 设计 tie-break 规则，如优先高权重语言 |

一个常见误区是：“既然跨语言可以互补，那就把所有可用语言都加进去。”这在直觉上诱人，但在实践中经常失败。多一个语言并不只是多一份信息，也多一份误差源。若新增语言单次正确率低、输出格式不稳、答案抽取困难，它贡献的很可能不是纠错票，而是噪声票。

工程上更稳的流程通常是：

`英语模板对齐 -> 选择少量高价值语言 -> 各语言多路径采样 -> 归一化答案 -> 加权投票 -> 必要时外部校验`

假设你在做一个面向东南亚市场的数学问答代理，需要支持英语、泰语、印尼语和中文。一个更稳的设计不是四种语言完全等权，而是：

| 语言 | 角色 | 典型策略 |
|---|---|---|
| 英语 | 主锚点 | 使用最成熟的提示模板，多采样 |
| 中文 | 强辅助 | 与英语并行投票 |
| 印尼语 | 目标市场语言 | 保留适中票权，观察增量价值 |
| 泰语 | 易波动语言 | 少量采样，必要时降权 |

这类设计承认一个事实：不同语言的票不是同质的。只要系统目标是提高准确率，而不是追求形式上的“语种公平”，那就应该按可靠性分配预算和票权。

另一个很重要的坑是把“路径一致”误当成“推理正确”。一致只表示模型稳定，不表示它稳定地对。比如题目被翻译错了，某种语言内 5 条路径全部一致输出同一个错误答案，那么 SC 只会更坚定地选错。因此在高风险任务中，SC 最好与外部校验结合，例如：

| 校验方式 | 适用题型 | 价值 |
|---|---|---|
| 数值代入检查 | 小学数学、代数题 | 快速验证结果是否满足条件 |
| 程序执行器 | 可形式化表达的问题 | 可直接检查计算是否正确 |
| 规则验证器 | 单位换算、约束判断 | 能拦截明显格式或逻辑错误 |
| 二次裁判模型 | 输出较长、答案非纯数值 | 可比较多个候选答案合理性 |

一句话总结这部分：SC 只能聚合已有信号，不能凭空创造正确性。

## 替代方案与适用边界

如果场景不同，不一定非要用跨语言 SC。常见替代方案至少有四类。

| 方案 | 适用边界 | 优点 | 缺点 |
|---|---|---|---|
| 纯英文 SC | 模型英语最强、成本有限 | 实现简单，稳定性高 | 放弃跨语言互补信息 |
| 多语言等权投票 | 各语种质量接近 | 结构简单，易扩展 | 易受低质量语言拖累 |
| 多语言加权投票 | 已有历史评估数据 | 能抑制低资源噪声 | 需要维护权重 |
| 过程验证器 + 单语生成 | 高风险任务 | 可直接检查推理合法性 | 实现复杂、成本更高 |

如果你的系统没有足够语种支持，优先考虑纯英文 SC 往往是更稳的工程选择。原因不是“英语天然最好”，而是大多数通用模型在英语上的训练覆盖更充分，答案抽取工具链也最成熟。在预算紧张时，把预算集中到英语内部多采样，通常比浅层覆盖很多语言更有效。

如果你已经确认多种语言都有可用质量，那么多语言加权投票通常优于等权投票。一个直接做法是根据历史验证集准确率设置权重，例如：

$$
w_i = \frac{\text{Acc}_i}{\sum_{t=1}^{m}\text{Acc}_t}
$$

其中，$\text{Acc}_i$ 表示第 $i$ 种语言在验证集上的历史准确率，$w_i$ 是该语言的归一化权重。于是最终投票可以写成：

$$
\hat y=\arg\max_y \sum_{i=1}^{m}\sum_{j=1}^{k} w_i \cdot \mathbf{1}[y_{ij}=y]
$$

这个公式表达的不是复杂技巧，而是一个很朴素的工程原则：历史上更可靠的语言，票权就应该更高。

它们的适用边界也需要说清楚。

1. 当题目高度依赖文化背景、本地表达或隐含常识时，跨语言转换可能改变问题含义，SC 收益会下降。
2. 当模型输出不是离散答案，而是长文本解释时，简单多数投票往往不够，需要语义聚类、过程比对或裁判模型。
3. 当低资源语言的单次正确率过低，且输出格式不稳定时，把它纳入投票通常只会增加噪声。
4. 当采样预算极低，例如每种语言只能采样 1 到 2 次时，跨语言投票的统计意义本身就有限。
5. 当任务允许外部验证时，过程校验往往比单纯增加采样更划算。

因此，真正实用的判断标准不是“方法是否看起来更先进”，而是两个更基础的问题：

1. 该语言内部的路径是否足够稳定。
2. 该语言能否提供独立且可靠的纠错信息。

只有这两个问题的答案都接近“能”，跨语言 SC 才值得投入。
