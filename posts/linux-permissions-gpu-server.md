## 核心结论

Linux 权限管理的核心不是“用户名”，而是 **UID/GID**。UID 是用户编号，GID 是用户组编号，内核只根据这两个数字判断“你是谁”。`sudo`、传统 `rwx` 权限、ACL、GPU 资源隔离，都是围绕这套身份编号继续加规则。

在单机或共享服务器上，最稳定的做法是三层联动：

| 授权层级 | 解决的问题 | 控制对象 | 典型工具 | 适合场景 |
| --- | --- | --- | --- | --- |
| UID/GID + `rwx` | 基础身份与目录归属 | 用户、组、文件 | `chown` `chmod` | 大多数普通目录 |
| `sudoers` | 谁能临时执行特权命令 | 命令、目标用户、主机 | `visudo` `sudo` | 运维脚本、服务重启 |
| ACL | 对单个 UID/GID 精细授权 | 文件和目录 | `setfacl` `getfacl` | 跨组协作、共享数据目录 |
| GPU 分区与配额 | 谁能占用哪些计算资源 | GPU、显存、进程、容器 | MIG、cgroup、调度器 | 多用户 GPU 服务器 |

最小权限原则的意思是：只给完成任务所需的最少权限，不给额外能力。它不是一句口号，而是一套具体操作习惯：

1. 用 UID/GID 管身份边界。
2. 用 `sudoers` 管特权命令白名单。
3. 用 ACL 给少量例外用户补权限，而不是放大整个目录权限。
4. 用 GPU 配额、MIG 或调度器限制显卡资源，不让一个人的训练任务拖垮整台机器。

一个新手能立刻理解的玩具例子是：`/srv/data` 目录归 `root:dev`，普通开发者都在 `dev` 组，可以读写；但 `bob` 不在 `dev` 组，却临时要维护某个数据目录。这时不应该把目录改成 `777`，而应该单独给 `bob` 的 UID 增加 ACL，再让管理员用 `sudoers` 只开放“重启 GPU 管理服务”这一条命令。

## 问题定义与边界

先把问题说清楚。Linux 权限系统主要解决四类边界：

1. 谁能登录并拥有某个身份。
2. 谁能读写哪个文件或目录。
3. 谁能执行 root 才能做的操作。
4. 在共享 GPU 服务器里，谁能占用多少算力、显存和磁盘。

这里有一个容易混淆的点：**用户名只是可读标签，UID/GID 才是内核真正使用的身份凭据**。例如 `/etc/passwd` 里记录了 `alice:x:1001:1001:...`，真正生效的是 `1001` 这个 UID，而不是字符串 `alice`。同理，用户组 `dev` 本质上也是一个 GID。

所以，传统权限判断其实很简单：内核拿“当前进程的 UID/GID”和“文件 inode 上的 owner/group/mode”做比对，最后决定读、写、执行是否放行。

但这套机制有边界。它擅长处理“文件属于谁、组里的人能做什么”，不擅长处理以下情况：

- 某个不在本组的用户，需要临时读写某个目录。
- 某个开发者可以重启服务，但不能安装软件、不能编辑系统文件。
- 多个团队共用 GPU，彼此不能抢显存、不能误杀对方进程、不能把公共磁盘写满。

这就是为什么只靠 `chmod` 和“把人都加进 sudo 组”通常不够。尤其在 GPU 服务器里，权限问题不只是文件访问，还包括资源归属。你可以把它理解成三种对象同时需要授权：

| 对象 | 典型权限问题 | 单靠 `rwx` 是否足够 |
| --- | --- | --- |
| 人 | 谁是谁、属于哪个组 | 否，只解决基础身份 |
| 文件 | 谁能读写执行 | 部分足够 |
| 特权命令 | 谁能以 root 执行什么 | 不够，需要 `sudoers` |
| 设备与资源 | 谁能用哪块 GPU、多少显存 | 不够，需要资源隔离 |

ACL 的核心公式必须记住：

$$
\text{最终权限} = \text{条目权限} \cap \text{mask}
$$

这里的 `mask` 可以理解成“ACL 的总闸门”。就算某个 ACL 条目写了 `rwx`，如果 `mask` 只有 `r--`，最终也只有读权限。

## 核心机制与推导

### 1. UID/GID 是最底层身份

当进程访问文件时，系统按这个顺序判断：

1. 进程 UID 是否等于文件 owner。
2. 若不等于，进程所属 GID 是否匹配文件 group。
3. 若都不匹配，就看 other 权限。
4. 若文件存在 ACL，还要继续套用 ACL 规则。

这意味着“把用户加组”本质上是给他增加一类身份标签。比如 `/data/projectA` 归 `root:dev`，权限是 `770`，那 `dev` 组成员可以读写，其他人不行。

一个玩具例子：

- `alice` 的 UID/GID 是 `1001/1001`
- `bob` 的 UID/GID 是 `1002/1002`
- `dev` 组 GID 是 `2001`
- `/srv/data` 归属 `root:dev`，模式是 `750`

结果是：

- `alice` 如果在 `dev` 组，可以读和执行目录。
- `bob` 如果不在 `dev` 组，就只能落到 `other` 权限，通常什么都做不了。

### 2. `sudo` 不是“变成 root”，而是“按策略借用特权”

`sudo` 的白话解释是：普通用户请求系统按规则临时提升身份去执行某个命令。关键点不在“提升”，而在“按规则”。

`sudoers` 文件定义了三件事：

| 维度 | 含义 | 示例 |
| --- | --- | --- |
| 谁可以执行 | 用户或组 | `%dev` |
| 以谁的身份执行 | 通常是 root，也可指定别的用户 | `(root)` `(ALL)` |
| 能执行什么 | 命令白名单 | `/usr/bin/systemctl restart gpu-manager` |

例如：

```sudoers
%dev ALL=(root) /usr/bin/systemctl restart gpu-manager
```

含义是：`dev` 组成员在所有主机上，都可以用 root 身份执行这一条命令，但不能执行其他 root 命令。这样做的价值在于审计。日志里仍然能看到是谁发起了 sudo，而不是一群人共用 root 密码。

### 3. ACL 在 `rwx` 之外增加“例外名单”

ACL 的白话解释是：在 owner/group/other 这三档之外，再补一张更细的授权表。

典型命令：

```bash
id -u bob
setfacl -m u:1002:rw /srv/data
getfacl /srv/data
```

这里 `u:1002:rw` 表示给 UID `1002` 的用户增加读写权限。很多系统文档会直接写用户名，但理解层面最好先记住：**底层最终仍然落到 UID/GID**。

ACL 的关键不是“能加权限”，而是“加权限时仍有总上限”。例如：

```bash
setfacl -m u:1002:rwx /srv/data
setfacl -m m:r-- /srv/data
```

此时虽然给 `1002` 写了 `rwx`，但实际结果只有 `r--`，因为：

$$
rwx \cap r-- = r--
$$

`getfacl` 的输出通常会看到类似内容：

```text
# file: srv/data
# owner: root
# group: dev
user::rwx
user:1002:rw-
group::r-x
mask::rw-
other::---
```

这里 `mask::rw-` 就是所有命名用户、命名组以及 group 类权限的上限。

### 4. 多用户 GPU 服务器为什么必须把“文件权限”和“资源权限”分开看

共享 GPU 的真实问题往往不是“能不能进入目录”，而是：

- 某人训练任务吃满整卡显存，别人作业启动失败。
- 某人误杀了不是自己 UID 的进程。
- 某项目日志无限增长，公共磁盘被写满。
- 某容器拿到过宽的设备权限，能探测或干扰别的任务。

所以 GPU 服务器规范至少要覆盖：

| 资源类型 | 约束目标 | 常用手段 |
| --- | --- | --- |
| GPU 设备 | 每个用户/任务可用哪些卡 | `CUDA_VISIBLE_DEVICES`、调度器 |
| 显存 | 避免一个任务吞掉整卡 | MIG、显存隔离、容器资源策略 |
| CPU/内存 | 防止训练进程拖垮系统 | cgroup |
| 磁盘空间 | 防止数据和 checkpoint 爆盘 | 项目目录配额、定期清理 |
| 进程归属 | 只能管理自己的任务 | 按 UID 启动、审计日志 |

真实工程例子：一台 8 卡服务器供 4 个项目组共用。管理员把 `/projects/A`、`/projects/B` 分别归到不同组，用 ACL 给跨组协作成员补读权限；同时在调度器或容器层限制每个任务只能看见指定 GPU；高端卡如果支持 MIG，还可以把单卡切成多个隔离实例，进一步把显存和计算资源边界硬化。

## 代码实现

先给出最小可用的 `sudoers` 配置。编辑时必须用 `visudo`，因为它会校验语法，避免把 sudo 配坏。

```sudoers
# dev 组成员只能重启 GPU 管理服务，不能执行任意 root 命令
%dev ALL=(root) /usr/bin/systemctl restart gpu-manager

# ops 组成员可以查看服务状态，但不能修改配置
%ops ALL=(root) /usr/bin/systemctl status gpu-manager
```

下面是一个批量设置 ACL 的脚本思路。场景是 `/projects/projectA/team.txt` 里放了成员 UID，每行一个 UID，目录 `/srv/data/projectA` 需要给这些人读写权限，并用 mask 把总上限锁成 `rw-`。

```bash
#!/usr/bin/env bash
set -euo pipefail

TEAM_FILE="/projects/projectA/team.txt"
TARGET_DIR="/srv/data/projectA"

# 先确保目录存在
test -d "$TARGET_DIR"

# 统一设置 ACL mask，避免有人误配成更高权限
setfacl -m m:rw "$TARGET_DIR"

while IFS= read -r uid; do
  # 跳过空行和注释
  [[ -z "$uid" || "$uid" =~ ^# ]] && continue

  # 给指定 UID 添加读写权限
  setfacl -m "u:${uid}:rw" "$TARGET_DIR"
done < "$TEAM_FILE"

# 打印最终 ACL，便于审计
getfacl "$TARGET_DIR"
```

如果你想把目录设置成“以后新文件也继承类似 ACL”，还需要默认 ACL：

```bash
setfacl -m d:m:rw /srv/data/projectA
setfacl -m d:u:1002:rw /srv/data/projectA
```

`d:` 表示 default，也就是新建文件/目录时继承的默认规则。

下面用一个可运行的 Python 玩具程序模拟 ACL 中“条目权限与 mask 相交”的计算逻辑。它不是系统命令替代品，但能帮助新手理解为什么 ACL 条目写得再大，mask 仍然能把结果压下来。

```python
def apply_mask(entry: str, mask: str) -> str:
    """
    entry 和 mask 形如 'rwx'、'rw-'、'r--'
    最终权限 = 条目权限 ∩ mask
    """
    out = []
    for e, m in zip(entry, mask):
        out.append(e if e == m and e != "-" else "-")
    return "".join(out)

assert apply_mask("rwx", "r--") == "r--"
assert apply_mask("rw-", "r--") == "r--"
assert apply_mask("r-x", "rw-") == "r--"
assert apply_mask("---", "rwx") == "---"

# 玩具例子：给 bob 配了 rwx，但管理员把 mask 锁成 rw-
bob_entry = "rwx"
mask = "rw-"
effective = apply_mask(bob_entry, mask)
assert effective == "rw-"
print("bob effective:", effective)
```

真实工程里，建议把“用户管理”和“目录授权”拆成两个步骤：

1. 人员归组走标准流程，比如新成员先进入 `dev` 或 `research`。
2. 个别项目目录的跨组例外，才用 ACL 补充。

这样做的原因是：组策略负责大面，ACL 负责局部例外，维护成本最低。

## 工程权衡与常见坑

最常见的错误，是把“方便”误当成“正确”。

### 1. `chmod 777` 是最典型的失控入口

`777` 的意思是 owner、group、other 全部可读写执行。它确实能快速“解决权限不够”，但代价是完全失去边界。任何本机用户、某些服务账号、甚至被入侵的低权限进程，都可能借这个目录写入恶意内容。

更好的做法是：

- 目录主体权限留给 owner/group。
- 个别例外用户用 ACL 精确加 `rw`。
- 如果只是共享协作，优先加组，不要先开全局写权限。

### 2. 把人随意加入 sudo 组，等价于扩大 root 面积

很多发行版里，`sudo` 组成员默认可以执行任意 root 命令。这意味着：

- 可以安装任意软件。
- 可以改系统服务。
- 可以读很多本不该看的数据。
- 可以绕过你在文件层做的大部分限制。

如果你的真实需求只是“允许某人重启一个服务”，那就应该写命令白名单，不应该把他直接变成全能管理员。

### 3. ACL 配了条目，却忘了检查 mask

这是 ACL 最容易让新手困惑的地方。表现通常是：

- `setfacl -m u:1002:rwx file`
- 结果对方还是不能写或不能执行

原因往往是 `mask` 太小。排查步骤应该固定成：

1. `getfacl <path>`
2. 看 `user:1002` 条目
3. 看 `mask::`
4. 用“条目权限 ∩ mask”心算最终结果

### 4. 默认 ACL 和普通 ACL 混淆

给目录加普通 ACL，只影响当前目录本身；给目录加默认 ACL，才会影响新建文件的继承行为。如果共享项目目录里总是“新文件权限不对”，多数不是 Linux 抽风，而是默认 ACL 没配。

### 5. GPU 服务器只管文件，不管资源，会出现“权限没问题但机器不可用”

下面是高频坑与规避方法：

| 常见坑 | 后果 | 规避措施 |
| --- | --- | --- |
| 直接 `chmod 777` | 任意用户可写，审计失效 | 用组权限或 ACL 精确授权 |
| 随意加入 `sudo` 组 | root 权限泛化 | 用 `sudoers` 白名单 |
| 只配 ACL 不看 mask | 看似授权成功，实际不生效 | 每次用 `getfacl` 核对 |
| 只配当前 ACL 不配默认 ACL | 新文件继承失败 | 给共享目录补 `d:` 规则 |
| 不限制 GPU 显存占用 | 一人吃满整卡，别人作业失败 | MIG、显存隔离、调度配额 |
| 不限制磁盘空间 | checkpoint 写爆盘 | 项目目录配额、清理策略 |
| 用 root 启动训练脚本 | 进程归属混乱，误操作风险大 | 一律按个人 UID 启动任务 |

一个真实工程例子是：研究团队常把训练脚本写成 `sudo python train.py`，因为“这样最省事”。这会直接破坏审计和文件归属。模型输出、缓存、日志都可能归 root，下一次普通用户再跑时就会遇到新一轮权限报错。正确做法是：普通训练任务永远用个人 UID 启动，只有系统级维护动作才使用受控的 `sudo`。

## 替代方案与适用边界

不是所有环境都需要把 ACL、MIG、复杂调度全上齐。方案选择要看规模和风险。

| 方案 | 适用场景 | 优点 | 局限 |
| --- | --- | --- | --- |
| 传统组权限 + `chmod` | 小团队、目录边界清晰 | 简单、直观、易维护 | 对跨组例外支持差 |
| 组权限 + `sudoers` 白名单 | 少量运维动作需要授权 | 审计清晰、风险可控 | 不能解决复杂文件共享 |
| 组权限 + ACL | 多项目共享目录、跨组协作频繁 | 文件授权精确 | 维护复杂度上升，需要管 mask |
| 容器/cgroup 配额 | 多任务混跑，需要隔离 CPU/内存/GPU 可见性 | 资源边界更强 | 运维门槛更高 |
| MIG 或硬件级 GPU 分区 | 高价值 GPU、多租户并发 | 隔离最强，显存边界硬 | 依赖硬件和驱动支持 |

小型研发组的典型方案通常已经够用：

1. 项目目录按组划分。
2. 用 `chmod g+s` 保持新文件继承组归属。
3. 用 `sudoers` 给少数维护命令开白名单。
4. 遇到个别跨组访问，再补 ACL。

这里的 `g+s` 是 setgid 位，白话解释是“目录下新建文件自动继承父目录的组”。它对协作目录很实用，因为能减少“文件属于个人私组，别人打不开”的问题。

大型团队或公共 GPU 集群则不同。此时仅靠 ACL 解决不了资源争抢，必须上调度器、cgroup 或 MIG。尤其当多个训练任务共享高端卡时，文件权限只决定“能不能读数据”，但显存隔离才决定“作业会不会互相踩死”。

如果你的环境有旧驱动、图形应用兼容要求，或者硬件不支持 MIG，那么容器级资源限制和调度策略往往比强推硬件分区更现实。这类方案的边界是：隔离足够强，但未必像硬件分区那样彻底。
