## 核心结论

`Depth-Down Architecture` 可以理解为一种“逐层下采样 + 通道压缩”的编码器式结构。白话说，它不是一次性把图像缩小，而是让模型沿着深度方向，连续把高分辨率细节压成更紧凑的表示，再在更小的特征图上继续计算。

它的核心价值不是“压得越狠越先进”，而是用可控的信息损失换三件事：更大的感受野、更多的语义聚合、更低的计算成本。感受野指一个神经元能“看到”的输入范围；语义聚合指模型把局部纹理逐步合并成“边缘、部件、目标”这类更抽象概念的过程。

最直观的理解是：把一张大图先压成更小的特征图，再在小图上判断内容。这样模型更快，也更容易利用全局上下文，但边界和纹理会变糊。因此它适合分类这类“只要最后结论”的任务，不适合直接拿去做要求像素级精度的分割或重建；后者通常必须配合 `skip connection`。`skip connection` 是一条把浅层特征直接送到后层的捷径，作用是保留细节和稳定训练。

优缺点可以先看总表：

| 维度 | 优势 | 代价 |
|---|---|---|
| 计算量 | 特征图变小后，后续卷积更便宜 | 早期压缩错误会一路传播 |
| 表达层次 | 更容易形成局部到全局的语义层次 | 细粒度空间信息减少 |
| 感受野 | 同样深度下更快覆盖大范围区域 | 小目标和边界更容易丢 |
| 工程实现 | 模块化明确，易做编码器主干 | 往往需要配套上采样或跳连 |

“分类 vs 分割”的区别可以用下面这个简图理解：

| 任务 | 输出目标 | 是否能接受强下采样 | 典型结构 |
|---|---|---|---|
| 图像分类 | 一张图一个类别 | 通常可以 | 编码器、ResNet、Inception |
| 图像分割 | 每个像素一个类别 | 通常不可以单独使用 | U-Net、编码器-解码器 |
| 目标检测 | 边框 + 类别 | 中等，需多尺度补偿 | FPN、带下采样主干 |
| 图像重建/超分 | 还原细节 | 通常不可以激进压缩 | 对称结构、保细节分支 |

## 问题定义与边界

先限定术语边界：`Depth-Down Architecture` 不是一个严格统一的标准术语。本文把它作为归纳性命名，用来描述“以连续下采样和表达压缩为主线的编码器式架构”。这意味着它不是某一篇论文的专有名词，而是对一类常见设计模式的抽象。

一句话定义是：把特征从高分辨率、较丰富细节的表示，逐层压缩为低分辨率、较强语义的表示，以换取效率和全局信息整合能力。

这里最容易混淆的是“网络里出现过下采样”和“整个架构以逐层压缩为主线”不是一回事。比如很多卷积网络都含有池化层，但只有当下采样构成连续主路径，且深层计算主要建立在缩小后的特征图上时，才更符合本文讨论的 `Depth-Down`。

术语边界如下：

| 术语 | 关注点 | 是否等同于 Depth-Down | 说明 |
|---|---|---|---|
| `Depth-Down` | 连续压缩链条 | 否，本文是上位归纳 | 强调“逐层压缩”的主线 |
| `Encoder` | 编码成紧凑表示 | 高度相关，但不完全等同 | 许多编码器都属于此类 |
| `Pooling` | 单次局部汇聚 | 否 | 只是一个操作，不是架构 |
| `Stride Conv` | 用步长卷积下采样 | 否 | 只是实现下采样的一种方法 |
| `Residual` | 恒等捷径与残差学习 | 否 | 解决优化问题，不等于压缩主线 |

判断规则可以写成两条：

1. 如果网络只是局部用了 `pooling` 或 `stride conv`，但主干设计目标不是“逐级缩小并在小图上持续计算”，它不应直接被称为 `Depth-Down`。
2. 如果网络前半段形成了稳定的“提特征 -> 压缩 -> 再提特征 -> 再压缩”的链条，它基本属于本文所说的 `Depth-Down` 范畴。

玩具例子：一个三层网络，第一层把 `32×32` 图像做卷积，第二层直接池化成 `16×16`，第三层就输出分类结果。这更像“带一次下采样的普通 CNN”，不够典型。若网络持续经历 `64×64 -> 32×32 -> 16×16 -> 8×8`，并在每一级都扩展语义表示，就更符合定义。

真实工程例子：`ResNet` 的某些 stage 会做下采样，但 `ResNet` 的核心创新是残差学习，不是“压缩本身”。相反，`U-Net` 的编码器半段就是典型的 `Depth-Down`：不断降低分辨率、增加语义抽象，然后再由解码器恢复空间细节。

## 核心机制与推导

形式化地写，设第 $l$ 层特征为：

$$
x_l \in \mathbb{R}^{H_l \times W_l \times C_l}
$$

其中 $H_l, W_l, C_l$ 分别表示高、宽、通道数。通道数可以理解为“同一位置上不同类型的特征描述器”。

一个典型的 depth-down 更新可写成：

$$
x_{l+1} = D_l\big(\phi(W_l * x_l + b_l)\big)
$$

其中：

- $W_l * x_l + b_l$ 是线性变换，通常由卷积实现。
- $\phi$ 是非线性激活，用来提高表达能力。
- $D_l$ 是压缩算子，负责空间下采样、通道压缩，或二者同时发生。

若下采样步长为 $s$，通道压缩比例为 $r$，则常见变化是：

$$
H_{l+1} = \frac{H_l}{s}, \quad W_{l+1} = \frac{W_l}{s}, \quad C_{l+1} = r C_l,\; 0<r\le 1
$$

尺寸变化表如下：

| 层级 | 空间尺寸 | 通道数 | 含义 |
|---|---|---|---|
| 输入层 | $H \times W$ | $C_0$ | 保留最多细节 |
| 中间层 1 | $\frac{H}{2}\times\frac{W}{2}$ | $C_1$ | 局部纹理开始汇聚 |
| 中间层 2 | $\frac{H}{4}\times\frac{W}{4}$ | $C_2$ | 目标部件信息更明显 |
| 深层 | $\frac{H}{8}\times\frac{W}{8}$ 或更小 | $C_3$ | 更偏全局语义 |

这个过程带来的第一个结论是：信息损失不是 bug，而是结构性结果。只要 $D_l$ 降低了维度，它通常就不是单射。单射指“不同输入一定对应不同输出”；非单射意味着多个输入可能压成同一个输出，因此不可逆。

玩具例子最直接：

$$
x=[1,2,3,4], \quad D(x)=[1,2]
$$

那么

$$
x'=[1,2,9,10]
$$

也满足 $D(x')=[1,2]$。白话说，只保留前半部分以后，后半部分已经丢掉了，后面算得再复杂也补不回来。

第二个结论是：下采样前若不做低通滤波，就容易混叠。混叠指高频细节在采样后伪装成错误的低频模式。若输入近似带限，也就是高频成分受控，那么先低通再采样，更符合采样定理，信息保留会更稳定。这也是 `BlurPool` 之类方法存在的原因。

第三个结论是：`1×1` bottleneck 只在“通道冗余较多”时才有意义。它本质是位置不变的线性投影，用较低维通道先压缩，再做昂贵卷积。

参数量对比如下。设输入输出通道都为 64，中间压到 16：

| 方案 | 参数量公式 | 参数量 |
|---|---|---|
| 直接 `3×3, 64→64` | $3^2 \times 64 \times 64$ | 36864 |
| `1×1, 64→16` + `3×3, 16→16` + `1×1, 16→64` | $64\times16 + 3^2\times16\times16 + 16\times64$ | 4352 |
| `1×1, 64→16` + `3×3, 16→64` | $64\times16 + 3^2\times16\times64$ | 10240 |

这说明 bottleneck 的收益可能非常大，但前提是压缩后的 16 个通道仍足以表达有效信息。否则参数省下来了，表达能力也被一起砍掉。

## 代码实现

实现时，最清楚的拆法是三个模块：

1. 特征提取块：先在当前分辨率上提局部模式。
2. 降采样块：把空间尺寸减小，并可选压缩通道。
3. 跳连块：把浅层高分辨率特征留给后面的解码或融合模块。

模块结构图可以写成：

```text
Input
  -> FeatureBlock(3x3 conv + act)
  -> Bottleneck(1x1 conv, optional)
  -> Downsample(stride conv or pool)
  -> Next Stage
            \
             -> Skip Connection (optional)
```

下面先给一个可运行的 Python 玩具实现，用数组模拟“特征提取 + 压缩 + 下采样”，并用 `assert` 验证不可逆和尺寸变化：

```python
from typing import List

def relu(xs: List[float]) -> List[float]:
    return [max(0.0, x) for x in xs]

def bottleneck(xs: List[float], keep: int) -> List[float]:
    # 玩具版 1x1 压缩：这里只是保留前 keep 个“通道”
    return xs[:keep]

def downsample_1d(xs: List[float], stride: int) -> List[float]:
    return xs[::stride]

def depth_down_block(xs: List[float], keep_channels: int, stride: int) -> List[float]:
    feat = relu(xs)
    compact = bottleneck(feat, keep_channels)
    return downsample_1d(compact, stride)

x = [1.0, 2.0, 3.0, 4.0]
y = depth_down_block(x, keep_channels=4, stride=2)
assert y == [1.0, 3.0]

x1 = [1.0, 2.0, 3.0, 4.0]
x2 = [1.0, 9.0, 3.0, 10.0]
assert downsample_1d(x1, 2) == downsample_1d(x2, 2)

z = depth_down_block([1.0, -2.0, 5.0, 6.0, 7.0, 8.0], keep_channels=4, stride=2)
assert len(z) == 2
```

上面代码不是深度学习框架实现，但表达了结构事实：先变换，再压缩，再按步长丢点，输出维度更小，而且通常不可逆。

再给一个最小 PyTorch 风格示意：

```python
import torch
import torch.nn as nn

class DepthDownBlock(nn.Module):
    def __init__(self, in_ch, mid_ch, out_ch):
        super().__init__()
        self.feature = nn.Sequential(
            nn.Conv2d(in_ch, mid_ch, kernel_size=3, padding=1),
            nn.ReLU(inplace=True),
        )
        self.bottleneck = nn.Conv2d(mid_ch, out_ch, kernel_size=1)
        self.down = nn.Conv2d(out_ch, out_ch, kernel_size=3, stride=2, padding=1)

    def forward(self, x):
        feat = self.feature(x)
        compact = self.bottleneck(feat)
        y = self.down(compact)
        return y, feat  # feat 可作为 skip connection

x = torch.randn(2, 32, 64, 64)
block = DepthDownBlock(32, 64, 16)
y, skip = block(x)
assert y.shape == (2, 16, 32, 32)
assert skip.shape == (2, 64, 64, 64)
```

这里的位置关系要看清：

- `3×3 Conv` 负责提局部模式。
- `1×1 Conv` 负责通道投影，不改变空间尺寸。
- `stride=2` 的卷积负责空间下采样。
- 返回的 `feat` 可以送往跳连分支，供后面恢复细节。

输入输出尺寸变化示例：

| 模块 | 输入 | 输出 |
|---|---|---|
| `FeatureBlock` | `64×64×32` | `64×64×64` |
| `1×1 Bottleneck` | `64×64×64` | `64×64×16` |
| `Stride Conv` | `64×64×16` | `32×32×16` |

真实工程例子是医学分割中的 `U-Net` 编码器。前半段连续做 depth-down，快速扩大感受野，提取器官和病灶的整体形状；后半段再用上采样和 skip connection 把边界补回来。如果只保留前半段而没有后半段，最终输出往往“知道哪里有东西”，但轮廓不够准。

## 工程权衡与常见坑

工程上最重要的不是会不会写下采样，而是压缩应该发生在什么阶段、压多少、是否需要补偿路径。

第一类权衡是“压缩早晚”。压缩过早，浅层边缘和纹理还没稳定就被抹掉；压缩过晚，算力消耗过高。分类任务往往可以更早进入压缩链条，因为最终只输出一个标签；分割任务则需要更保守。

第二类权衡是“空间压缩”和“通道压缩”不是同一个问题。空间压缩降低定位精度，通道压缩降低表示容量。两者同时做时，风险会叠加。

第三类权衡是“更小参数量”不等于“更强工程效果”。很多新手看到 `1×1` 或 bottleneck 节省参数，就到处加，最后把本来就不富余的通道进一步压扁，反而损失性能。

常见坑清单如下：

| 常见坑 | 问题本质 | 后果 | 规避策略 |
|---|---|---|---|
| 无损误解 | 误把下采样当压缩编码 | 以为后面能恢复全部细节 | 预先承认信息损失，必要时加 skip |
| 混叠 | 直接 stride/pooling 未低通 | 位移不稳定、细节伪影 | 先平滑再采样，或用 anti-aliasing |
| 过早压缩 | 低层纹理还未形成就被砍掉 | 小目标、边界变差 | 先做少量特征提取，再降采样 |
| 任务不匹配 | 分类结构硬套到分割/重建 | 输出粗糙、不准 | 根据输出粒度选择结构 |
| 误用 `1×1` | 通道本就不冗余仍强行降维 | 表达能力不足 | 只在通道冗余明显时使用 |

医学分割是一个典型反例。病灶边界往往窄、弱、对比度低，如果一开始就用大步长强压缩，后面即使上采样回原尺寸，也常常只能恢复“模糊位置”，难以恢复真实边界。白话说，图缩得太快，轮廓就来不及学清楚。

另一个常见问题是把 `Residual` 和 `skip connection` 混为一谈。两者都提供捷径，但用途不同：残差块里的恒等映射主要解决深层优化问题；编码器-解码器之间的跨层跳连主要解决细节回流问题。它们可以一起出现，但不是同一件事。

## 替代方案与适用边界

如果任务重视空间精度，不应把纯 `Depth-Down` 当成终点，而应把它当作编码器的一半。最典型的替代方案是 `U-Net` 式编码器-解码器：前半段压缩，后半段恢复，并通过跳连补细节。

如果任务更重视分类鲁棒性和效率，可以更激进地使用 `bottleneck`、`stride conv`、`pooling`，甚至加抗混叠下采样，使模型在有限算力内看见更大范围。

替代方案对照如下：

| 方法 | 核心思想 | 适用任务 | 限制 |
|---|---|---|---|
| `Inception bottleneck` | 先用 `1×1` 降维，再做较贵卷积 | 分类、检测 | 依赖通道冗余假设 |
| `ResNet` | 用残差学习稳定深层训练 | 分类、检测、通用主干 | 不自动解决细节恢复 |
| `U-Net` | 编码压缩 + 解码恢复 + 跳连 | 分割、重建 | 显存和结构复杂度更高 |
| `BlurPool / anti-aliasing` | 先低通再下采样 | 对位移稳定性敏感的视觉任务 | 额外计算与实现成本 |
| `SliceSamp` | 用替代性采样保留更多信息 | 研究和实验性下采样设计 | 工程生态与验证较少 |

适用边界可以概括成一句话：只要结论，不太要细节，可以压得更狠；既要结论，还要像素级细节，就必须给细节留回路。

因此：

- 图像分类通常适合更强的 depth-down，因为最终输出是类别，不要求边界。
- 图像分割不适合激进单向压缩，因为输出本身就是空间结构。
- 目标检测处于中间地带，既要语义也要定位，所以常引入多尺度特征融合。
- 重建、超分、医学影像分析等任务，对早期细节损失尤其敏感，需要更保守的下采样策略。
