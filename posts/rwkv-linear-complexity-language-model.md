## 核心结论

RWKV 是一种把 Transformer 和 RNN 结合起来的语言模型结构。这里的“结构”可以理解为模型处理序列数据的计算组织方式。它的核心目标很明确：训练时尽量保留 Transformer 的并行能力，推理时尽量保留 RNN 的低内存递推能力。

更具体地说，RWKV 用一个按通道定义的衰减率 $w$，把历史 token 的信息压缩进固定大小的状态里。所谓“通道”，就是隐藏向量的每一维可以独立学习自己的衰减速度。这样做的结果是：

| 维度 | Transformer | RWKV |
|---|---|---|
| 训练方式 | 全序列并行 | 全序列并行 |
| 推理方式 | 依赖 KV cache | 依赖固定状态递推 |
| 单步状态大小 | 随上下文长度增长 | 固定不变 |
| 长上下文内存 | 高 | 低 |
| 适合场景 | GPU 高吞吐服务 | 长上下文、CPU、边缘设备 |

RWKV 最重要的结论不是“它比 Transformer 更先进”，而是“它把成本重新分配了”。训练阶段仍然需要大量矩阵乘法，工程形态仍接近 Transformer；但推理阶段只维护少量状态，不需要保存整段历史的 KV cache，因此能在长上下文和低内存环境里获得明显优势。

对初学者，一个最直观的理解是：RWKV 不保留所有历史 token 的键值对，而是把历史压成两组滚动更新的加权和。新 token 到来时，只要更新这两组累积量，就能算出当前输出。

## 问题定义与边界

语言模型要解决的问题，是根据前文预测下一个 token。这里的 token 可以理解为被切分后的最小文本单元，比如一个词片段。传统 Transformer 在做这件事时，会让当前位置和所有历史位置做注意力计算，因此推理时通常要保留历史的 Key/Value 缓存，也就是 KV cache。

RWKV 试图解决的是另一个更工程化的问题：

1. 训练时仍然要吃满 GPU 并行能力。
2. 推理时不要让内存随上下文长度线性增长。
3. 最好还能支持很长上下文，比如 64K 甚至更长。

它的边界也要说清楚：

| 问题 | RWKV 的回答 | 边界 |
|---|---|---|
| 能否像 Transformer 一样并行训练？ | 可以 | 训练实现仍是大矩阵计算 |
| 能否像 RNN 一样固定状态推理？ | 可以 | 解码仍然按时间步串行 |
| 能否完全替代注意力？ | 在部分任务上可行 | 表达能力与优化难度仍受结构限制 |
| 能否免费获得长上下文能力？ | 内存侧更友好 | 长程建模质量仍依赖参数和训练方法 |

玩具例子可以先看“滚动加权平均”：

假设你在统计一段数据的趋势，不想保存全部历史，只想保留“加权和”和“总权重”。每来一个新数，就把旧统计量衰减一点，再把新数加进去。这样你不需要回头看所有历史，也能持续得到当前估计。RWKV 的 WKV 本质上就是这种思想的高维版本。

真实工程例子是长对话推理。比如一个本地部署的聊天系统持续处理用户多轮输入，Transformer 通常需要越来越大的 KV cache；RWKV 则只维护固定宽度的状态向量。对 CPU、本地设备、嵌入式设备，这种内存形态更友好。它尤其适合“单条流持续向前”的场景，而不适合需要大量并行解码、依赖成熟 Transformer 推理基础设施的系统。

## 核心机制与推导

RWKV 的核心算子常写成 WKV。它本质上是在时间轴上对历史 value 做指数衰减加权平均。

设第 $t$ 个位置的 key 和 value 分别为 $k_t, v_t$，通道衰减率为 $w$。则某一时刻的历史加权平均可写为：

$$
\mathrm{wkv}_t =
\frac{\sum_{i=1}^{t-1} e^{-(t-1-i)w + k_i}\cdot v_i}
{\sum_{i=1}^{t-1} e^{-(t-1-i)w + k_i}}
$$

这个公式的意思是：越早的历史会被 $e^{-w}$ 连续衰减，越重要的 token 还会通过更大的 $k_i$ 获得更高权重。这里的“权重”就是模型自动学出来的关注强度。

关键点在于，上式可以递推。定义：

$$
a_t = e^{-w}\odot a_{t-1} + e^{k_t}\odot v_t
$$

$$
b_t = e^{-w}\odot b_{t-1} + e^{k_t}
$$

那么就有：

$$
\mathrm{wkv}_t = \frac{a_{t-1}}{b_{t-1}}
\quad \text{或在带 bonus 项时} \quad
\mathrm{WKV}_t =
\frac{a_{t-1}+e^{u+k_t}\odot v_t}{b_{t-1}+e^{u+k_t}}
$$

其中 $\odot$ 表示逐元素乘法，也就是向量每一维分别相乘。$u$ 是一个额外的时间混合参数，用来让当前 token 的贡献有单独的调节项。

为什么递推成立？把 $a_t$ 展开就能看清：

$$
a_t
= e^{-w}a_{t-1}+e^{k_t}v_t
= e^{-2w}a_{t-2}+e^{-w}e^{k_{t-1}}v_{t-1}+e^{k_t}v_t
$$

继续展开可得：

$$
a_t = \sum_{i=1}^{t} e^{-(t-i)w+k_i}v_i
$$

同理：

$$
b_t = \sum_{i=1}^{t} e^{-(t-i)w+k_i}
$$

所以 $\frac{a_t}{b_t}$ 就是历史 value 的衰减加权平均。也就是说，RWKV 不是“近似忘掉历史”，而是把“显式遍历历史”改写成“维护可递推的统计量”。

对新手，一个足够准确的白话解释是：

- $a_t$ 是“历史 value 的加权和”。
- $b_t$ 是“历史权重总和”。
- 两者相除，就是当前时刻对历史的归一化汇总。
- 新 token 到来时，旧状态先衰减，再加入新贡献。

下面用一个标量玩具例子直观说明。

设 $w=1$，所以 $e^{-w}\approx 0.3679$。再设：

- $k_1=0,\ v_1=1$
- $k_2=0.5,\ v_2=2$

初始化 $a_0=b_0=0$。

第 1 步：

$$
a_1 = 0.3679\cdot 0 + e^0\cdot 1 = 1
$$

$$
b_1 = 0.3679\cdot 0 + e^0 = 1
$$

所以输出约为 $1$。

第 2 步：

$$
a_2 = 0.3679\cdot 1 + e^{0.5}\cdot 2 \approx 3.665
$$

$$
b_2 = 0.3679\cdot 1 + e^{0.5} \approx 2.0166
$$

因此：

$$
\frac{a_2}{b_2}\approx 1.817
$$

这个数字比第一个 value 更靠近第二个 value，因为第二个 token 更新、更“新”，且 $k_2$ 更大，权重也更高。

RWKV-6 的进一步发展，是把原本相对固定的衰减设计，扩展成更强的 data-dependent decay，也就是“衰减速度依赖当前数据”。这让模型不只是按固定节奏遗忘历史，而是能根据输入内容动态决定保留多久，从而提升表达能力。

## 代码实现

下面给出一个最小可运行的 Python 版本。它先用直接求和实现 WKV，再用递推实现 WKV，并用 `assert` 验证两者一致。

```python
import math

def wkv_direct(keys, values, w):
    out = []
    for t in range(len(keys)):
        num = 0.0
        den = 0.0
        for i in range(t + 1):
            weight = math.exp(-(t - i) * w + keys[i])
            num += weight * values[i]
            den += weight
        out.append(num / den)
    return out

def wkv_recurrent(keys, values, w):
    a = 0.0
    b = 0.0
    out = []
    decay = math.exp(-w)
    for k, v in zip(keys, values):
        a = decay * a + math.exp(k) * v
        b = decay * b + math.exp(k)
        out.append(a / b)
    return out

keys = [0.0, 0.5, -0.2, 0.3]
values = [1.0, 2.0, -1.0, 3.0]
w = 1.0

direct = wkv_direct(keys, values, w)
recurrent = wkv_recurrent(keys, values, w)

for x, y in zip(direct, recurrent):
    assert abs(x - y) < 1e-9

assert recurrent[0] == 1.0
print(recurrent)
```

这段代码说明了两件事：

1. 递推形式和显式求和是等价的。
2. 推理时你不需要保存整个历史序列，只需要状态 `a` 和 `b`。

如果扩展到向量版本，逻辑不变，只是把标量换成逐通道向量：

```python
import numpy as np

def rwkv_step(a, b, k, v, w):
    # a: 历史加权和
    # b: 历史总权重
    # k, v, w: 当前 token 的 key/value/衰减
    decay = np.exp(-w)
    ek = np.exp(k)
    a = decay * a + ek * v
    b = decay * b + ek
    y = a / b
    return a, b, y

a = np.zeros(3)
b = np.zeros(3)
k = np.array([0.0, 0.2, -0.1])
v = np.array([1.0, 2.0, 3.0])
w = np.array([1.0, 0.5, 0.8])

a, b, y = rwkv_step(a, b, k, v, w)
assert y.shape == (3,)
assert np.all(b > 0)
```

真实工程里不会这样直接裸算 `exp`，因为数值稳定性会出问题。更稳妥的做法有两类：

| 做法 | 思路 | 适用性 |
|---|---|---|
| 减最大值缩放 | 先减去当前最大 log 权重，再做 `exp` | 简单，常用于实现 |
| log-space 状态 | 在对数空间维护状态，避免直接爆炸 | 更稳，但实现更复杂 |

一个简单的稳定化思路是：如果要计算 $\exp(k)$，先找一个参考值 $m$，改算 $\exp(k-m)$，最后把分子分母同时按同样比例缩放。因为归一化比值不会变：

$$
\frac{e^{x}A}{e^{x}B}=\frac{A}{B}
$$

所以很多实现都会在分子分母上同步做缩放，避免上溢或下溢。

## 工程权衡与常见坑

RWKV 的工程价值非常清楚，但它不是“无代价替代”。主要权衡如下：

| 维度 | 优势 | 代价 |
|---|---|---|
| 推理内存 | 固定状态，无需完整 KV cache | 单条序列仍按时间步递推 |
| 长上下文 | 更容易把内存控制住 | 不代表长程建模质量自动更强 |
| 部署 | 适合 CPU、本地设备、边缘设备 | 工具链生态不如 Transformer 成熟 |
| 训练 | 保留并行训练特性 | 训练成本并没有变成 RNN 级别 |

常见坑主要集中在数值稳定性和参数约束。

第一类坑是 `exp` 溢出。假设某一维的 $k_t=80$，那么 $\exp(80)$ 已经非常大；如果再叠加历史状态，分子分母都可能变成 `inf`。这时即使理论公式正确，实际实现也会得到 `nan`。

第二类坑是过强衰减或过弱衰减。比如 $w$ 过大时，$e^{-w}$ 非常接近 0，模型几乎只看最新 token；$w$ 过小时，历史衰减太慢，旧信息会长期占据状态，更新变钝。

第三类坑是忘记保证 $w>0$。如果 $w<0$，那么 $e^{-w}>1$，历史不是衰减而是放大，状态会快速失控。工程上通常会用参数化方式确保这一点，比如令：

$$
w = \mathrm{softplus}(\theta)
$$

这样无论 $\theta$ 取什么值，$w$ 都是正数。

第四类坑是把“固定状态”误解成“信息无损压缩”。固定状态意味着模型必须学会把历史压进有限维度里，这当然会带来表达瓶颈。RWKV 强在工程形态，不代表它在所有任务上都能无损替代全注意力。

下面给一个稳定与不稳定的直观对比：

| 场景 | 直接计算 | 稳定化计算 |
|---|---|---|
| $k=5$ | 通常正常 | 正常 |
| $k=50$ | 可能接近数值极限 | 仍可控 |
| $k=100$ | 很容易溢出 | 若做 log-space，通常可算 |
| $w<0$ | 状态可能爆炸 | 通过参数约束避免 |

真实工程例子是本地长文本生成。如果一台设备只有有限内存，而你又要做长文续写或多轮对话，Transformer 的 KV cache 会不断增长；RWKV 只维护固定宽度状态，内存更稳定。但如果你的场景是高吞吐批量推理，现有 GPU Transformer 内核和 KV cache 优化已经非常成熟，RWKV 未必占优。

## 替代方案与适用边界

RWKV 最合适的理解方式，不是和所有模型争“谁最先进”，而是看它位于什么折中点。

| 方案 | 训练特征 | 推理特征 | 适用场景 |
|---|---|---|---|
| Transformer + KV cache | 高并行 | 内存随上下文增长 | 云端 GPU、高吞吐服务 |
| LSTM/GRU | 递推训练较强 | 固定状态 | 小模型、传统时序任务 |
| RWKV | 并行训练 + 递推推理 | 固定状态 | 长上下文、低内存部署 |

对初学者，可以记一句最准确的白话：

RWKV 是“训练像 Transformer，推理像 RNN”。

这句话为什么成立？

- 训练像 Transformer：因为整个序列可以并行计算各位置的表示，适合 GPU。
- 推理像 RNN：因为每一步只依赖上一步状态，不需要回看全部历史缓存。

如果把三者的单步推理流程写成伪代码，大致是这样：

```text
Transformer:
  cache.append(K_t, V_t)
  y_t = attend(Q_t, cache)

LSTM:
  h_t, c_t = f(x_t, h_{t-1}, c_{t-1})
  y_t = g(h_t)

RWKV:
  a_t = decay * a_{t-1} + exp(k_t) * v_t
  b_t = decay * b_{t-1} + exp(k_t)
  y_t = a_t / b_t
```

适用边界也要明确：

1. 如果你最关心推理内存和长上下文部署，RWKV 很有吸引力。
2. 如果你最关心生态成熟度、推理框架支持、现成模型规模，Transformer 仍然更主流。
3. 如果任务本身序列不长、模型很小，LSTM 之类传统递推模型可能已经足够。
4. 如果需要极强的全局 token 交互能力，固定状态递推结构通常会更受限制。

因此，RWKV 更像一条工程导向的路线：它不是否定注意力，而是在“并行训练”和“低内存推理”之间找到了一个可实现的中间点。
