## 核心结论

SimHash 是一种**文档指纹**算法。文档指纹可以理解为“把一篇很长的文本压缩成一个固定长度的二进制编号”，后续只比较编号，就能大致判断两篇文档像不像。它的核心做法不是直接比较原文，而是先把文档拆成特征，再把特征按权重投票到每一位，最后把每一位的正负号变成 `1/0`。

它适合处理“**几乎一样，只改了很少部分**”的文档。例如网页正文只改了日期、广告位、页脚、少量措辞，这类近似重复文档通常会得到非常接近的 SimHash，海明距离（Hamming Distance，指两个二进制串有多少位不同）往往只差 `1~3` 位。工程上常把“海明距离 ≤ 3”作为严格去重门槛，把“≤ 5` 或 `≤ 7`”作为更宽松的候选门槛。

一个最直接的玩具例子是：如果文档 A 和文档 B 的 64 位指纹只在 3 位以内不同，那么它们通常可以被视为近似重复，适合爬虫阶段快速丢弃重复页面。这里的“64 位指纹”就是一个长度固定为 64 的二进制串，比较时只需要做一次异或和一次统计不同位数。

SimHash 与 MinHash 经常被放在一起讨论，但它们解决的问题并不完全相同。MinHash 更接近“估计两个集合的 Jaccard 相似度”，适合关心集合重叠比例的场景；SimHash 更接近“把整体特征方向压成一个签名”，适合高吞吐、低延迟的近重复检测。前者通常需要保存多组哈希值，后者往往只需一个 64 或 128 位签名，存储和比较都更轻。

| 方法 | 主要相似度 | 存储量 | 比对复杂度 | 更适合的场景 |
|---|---|---:|---:|---|
| SimHash | 近似角度相似、近重复 | 低，常见 64/128 位 | 很低，`XOR + popcount` | 搜索引擎、爬虫去重、索引去重 |
| MinHash | Jaccard 集合相似度 | 较高，多组签名 | 中等，需要逐组比较 | 集合相似度估计、宽松文本相似 |

## 问题定义与边界

我们要解决的问题不是“判断两篇文章主题是否类似”，而是“在海量文档里，快速找出几乎一样的重复内容”。这两个问题差别很大。主题相似可能意味着两篇文章都在讲数据库，但用词、结构、篇幅都不同；近似重复则意味着它们大部分文本相同，只改了很少一部分。

对零基础读者，可以把 SimHash 想成下面这个过程：

1. 把文档拆成很多小特征，比如词、短语、行块、句子 n-gram。
2. 每个特征先变成一个固定长度的二进制串。
3. 每个二进制位都参与投票：该位是 `1` 就加分，是 `0` 就减分。
4. 最后每个位看总分正负，正数记成 `1`，负数记成 `0`。

这就像每个特征都在对“最终指纹的每一位”投票，权重大、出现频繁、信息量高的特征会影响更多。最终得到的不是原文拷贝，而是一个压缩后的方向签名。

它的边界必须说清楚：

| 边界项 | 适用情况 | 不适用情况 |
|---|---|---|
| 文档差异大小 | 差异很小，局部改动 | 大面积改写、重组、摘要式改写 |
| 特征类型 | 词、短语、shingle、行块稳定 | 特征抽取得很噪，随机词过多 |
| 阈值规模 | 常见 `k <= 3` 或 `k <= 7` | 需要容忍大量不同位 |
| 检索目标 | 快速筛出候选重复 | 追求高语义理解能力 |

因此，SimHash 不是“万能相似度算法”。它最强的地方是：在极大规模数据中，用极低成本识别**字面差异不大的近重复**。如果两篇文档只是主题相近、措辞不同、段落顺序变化很大，SimHash 往往会漏掉。

文本图示可以这样理解：

文档原文  
→ 特征提取  
→ 每个特征做固定长度哈希  
→ 按权重对每一位加减  
→ 得到 64/128 位指纹  
→ 通过海明距离做近重复判断

## 核心机制与推导

SimHash 的核心公式可以写成：

$$
v_j=\sum_i w_i \cdot s_{i,j}
$$

其中：

- $w_i$ 是第 $i$ 个特征的权重。权重就是“这个特征有多重要”的分数。
- $s_{i,j}\in\{-1,+1\}$ 表示第 $i$ 个特征哈希后在第 $j$ 位上的符号。如果该位是 `1`，记为 `+1`；如果该位是 `0`，记为 `-1`。
- $v_j$ 是第 $j$ 位累计后的总分。

最后，第 $j$ 位的指纹由 $v_j$ 的符号决定：

- $v_j > 0$，该位记为 `1`
- $v_j \le 0$，该位记为 `0`

这一步的含义很直接：每一位都让全部特征来投票，最终保留多数方向。

两个指纹的距离用海明距离定义：

$$
HD(h_1, h_2)=\sum_j \mathbb{1}(h_1[j]\neq h_2[j])
$$

这里的 $\mathbb{1}$ 是指示函数，可以理解为“如果这一位不同就记 1，否则记 0”。所有位加起来，就是总共有多少位不同。

### 4 位玩具例子

假设只看 4 位，文档 1 有两个特征：

- 特征 A：`1010`，权重 1
- 特征 B：`1100`，权重 1

按位累加时，`1` 记 `+1`，`0` 记 `-1`：

| 位次 | A=1010 | B=1100 | 累加结果 |
|---|---:|---:|---:|
| 第1位 | +1 | +1 | +2 |
| 第2位 | -1 | +1 | 0 |
| 第3位 | +1 | -1 | 0 |
| 第4位 | -1 | -1 | -2 |

如果把 `0` 也归到 `0`，最终指纹是 `1000`。

现在把文档 1 的特征 B 替换成特征 C：

- 特征 C：`1110`，权重 1

重新累加：

| 位次 | A=1010 | C=1110 | 累加结果 |
|---|---:|---:|---:|
| 第1位 | +1 | +1 | +2 |
| 第2位 | -1 | +1 | 0 |
| 第3位 | +1 | +1 | +2 |
| 第4位 | -1 | -1 | -2 |

最终指纹变成 `1010`。

此时：

- 文档 1 指纹：`1000`
- 文档 2 指纹：`1010`

两者只在第 3 位不同，所以海明距离为 `1`。在 4 位空间里，差 1 位已经说明它们非常接近；在真实工程里的 64 位空间中，差 `1~3` 位通常就是典型近重复候选。

### 为什么它能工作

SimHash 本质上是在近似保留“高维特征向量的方向”。方向可以理解为“整体分布偏向哪里”，而不是每个特征是否完全相同。特征接近、权重分布接近的两个文档，它们在很多位上的投票结果会一致，因此最终指纹也接近。

但这也是它的限制来源：如果大量特征变了，或者特征抽取方法本身不稳定，那么最终投票方向会被打乱，海明距离就会迅速增大。

## 代码实现

实现流程可以压缩成四步：

1. 提取特征
2. 对每个特征做固定长度哈希
3. 按位累加权重
4. 取符号得到指纹，再用异或统计海明距离

下面给出一个可以直接运行的 Python 版本。它用 `blake2b` 生成稳定哈希，用空格分词作为玩具特征，重点是展示机制，不依赖第三方库。

```python
import hashlib
from collections import Counter

def stable_hash64(text: str) -> int:
    digest = hashlib.blake2b(text.encode("utf-8"), digest_size=8).digest()
    return int.from_bytes(digest, "big")

def simhash(tokens, bits=64):
    weights = Counter(tokens)  # 词频作为权重
    vector = [0] * bits

    for token, weight in weights.items():
        h = stable_hash64(token)
        for j in range(bits):
            bit = (h >> j) & 1
            vector[j] += weight if bit else -weight

    fingerprint = 0
    for j, score in enumerate(vector):
        if score > 0:
            fingerprint |= (1 << j)
    return fingerprint

def hamming_distance(x: int, y: int) -> int:
    return (x ^ y).bit_count()

doc_a = "python simhash dedup search engine"
doc_b = "python simhash dedup search engine"
doc_c = "python simhash dedup crawler engine"
doc_d = "database transaction isolation mvcc lock"

f_a = simhash(doc_a.split())
f_b = simhash(doc_b.split())
f_c = simhash(doc_c.split())
f_d = simhash(doc_d.split())

assert hamming_distance(f_a, f_b) == 0
assert hamming_distance(f_a, f_c) >= 0
assert hamming_distance(f_a, f_d) > hamming_distance(f_a, f_c)

print("A/B:", hamming_distance(f_a, f_b))
print("A/C:", hamming_distance(f_a, f_c))
print("A/D:", hamming_distance(f_a, f_d))
```

伪代码和真实实现是一致的：

```text
for each feature:
    hash = hash_func(feature)
    for j in range(bits):
        v[j] += weight * (bit_is_set(hash, j) ? 1 : -1)

fingerprint = sign(v)
distance = popcount(f1 XOR f2)
```

每一步分别表示：

- `hash_func(feature)`：把特征变成固定位数的二进制串
- `v[j] += ...`：对第 `j` 位累计投票
- `sign(v)`：把正负号压成最终 `1/0`
- `popcount(f1 XOR f2)`：先异或，再统计有多少位不同

真实工程里通常直接保存 64 位或 128 位整数。查询时不需要恢复原文，只需要加载指纹并计算海明距离。`popcount` 是“统计二进制里有多少个 1”的操作，现代 CPU 对它有很强的指令级支持，因此速度很高。

一个真实工程例子是搜索引擎去重。抓取网页后，系统会抽取正文特征，计算 64 位 SimHash。如果新页面与已有页面的海明距离 ≤ 3，就把它标为近重复候选，不再重复建立完整索引。这能直接减少抓取、存储和排序成本。

## 工程权衡与常见坑

最常见的工程选择是位数。64 位更省空间，也更容易放进内存和索引结构；128 位区分能力更强，但存储、网络传输、索引结构都会更重。对于网页去重、爬虫去重，64 位往往已经够用；对误报特别敏感的场景，才考虑更长位数。

真正的难点不在“算一个 SimHash”，而在“**如何在海量指纹里快速找到海明距离 ≤ k 的候选**”。因为如果对每个新文档都和全部旧文档逐个比较，复杂度仍然太高。

常见做法是**分段匹配**。以 64 位指纹、阈值 `k = 3` 为例，可以把指纹拆成 4 段，每段 16 位。原理是：如果两个 64 位指纹最多只差 3 位，那么把它们切成 4 段后，至少有 1 段会完全相同。这叫鸽巢原理的工程应用。

于是可以建立 4 张倒排表：

- 第 1 段值 → 文档列表
- 第 2 段值 → 文档列表
- 第 3 段值 → 文档列表
- 第 4 段值 → 文档列表

查询新文档时：

1. 计算它的 64 位 SimHash
2. 切成 4 段
3. 去 4 张表里分别查同段值的候选
4. 合并候选集合
5. 对候选做精确海明距离计算

这就是“多表 + 分段”的典型做法。它不是直接得到答案，而是先把全量比对缩小成很小的候选集，再做精确判断。

Google 搜索引擎在爬虫和索引阶段处理近重复网页时，常被引用的实践就是类似思路：64 位 SimHash 配合分段检索，先快速缩小候选，再用海明距离 ≤ 3 之类的阈值做最终去重。它的价值不只是“找重复”，而是减少重复抓取、重复索引、重复排序。

常见坑和对策如下：

| 常见坑 | 结果 | 对策 |
|---|---|---|
| 特征切得太碎 | 噪声过多，误报上升 | 用词组、shingle、正文块，不要只看单字 |
| 权重设计不合理 | 页脚、导航、模板词占主导 | 对停用词、模板词降权或过滤 |
| 阈值设太大 | 候选暴涨，误报严重 | 从 `k=3` 或 `k=4` 起做离线评估 |
| 阈值设太小 | 漏掉轻微改写 | 用业务样本调参，不要凭感觉定 |
| 只做分段召回，不做精确校验 | 错误候选直接通过 | 候选召回后必须再算精确海明距离 |
| 哈希函数不稳定 | 结果不可复现 | 使用稳定、跨进程一致的哈希实现 |

还有一个经常被忽略的问题：模板网页。很多站点正文只占页面的一小部分，其余是导航、推荐、版权、广告。如果特征提取阶段没有先抽正文，SimHash 算出来的其实是“模板指纹”，不是“内容指纹”，结果会把不相关页面误判成重复。

## 替代方案与适用边界

如果你关心的是“两个集合重叠比例有多大”，而不是“它们是否几乎一样”，MinHash 往往更合适。MinHash 会对集合做多次随机投影，保留多个最小哈希值，再用这些签名估计 Jaccard 相似度。Jaccard 可以理解为“交集占并集的比例”。

对新手来说，可以这样区分：

- SimHash：看整体方向是否几乎一致，适合差异很小的近重复
- MinHash：看集合重叠比例是否高，适合差异更大的相似集合

例如两篇长文档讨论同一主题，但一篇顺序重组了，删改了很多句子，还替换了不少词。它们的词集合可能仍然高度重叠，MinHash 还能给出较高相似度估计；但 SimHash 因为很多投票位已经翻转，海明距离可能变得很大，从而漏检。

| 维度 | SimHash | MinHash / LSH |
|---|---|---|
| 核心思想 | 加权投票后取位符号 | 多组最小哈希估计 Jaccard |
| 主要输入 | 带权特征向量 | 集合或 shingles 集合 |
| 存储成本 | 低，一个短指纹 | 较高，多组签名 |
| 召回范围 | 适合很小差异 | 可容忍更大差异 |
| 常见错误 | 差异稍大就漏检 | 存储和计算更重 |
| 典型场景 | 搜索去重、网页近重复 | 文档集合相似、宽松匹配 |

什么时候该换方案：

- 文档改写幅度大，SimHash 海明距离经常超过 `7`
- 更关心“集合重叠比例”而不是“字面近重复”
- 特征频率变化大，整体方向不稳定
- 需要做近邻搜索而不是简单重复过滤

因此，SimHash 不是 MinHash 的低配版，它们是偏好不同的工具。前者强调高吞吐、短签名、快速判重；后者强调相似度估计的稳定性和更宽的适用边界。
