## 核心结论

Gemini 的跨模态注意力融合策略，可以概括为一句话：**不同模态先变成同一种“token 序列”，再进入同一套 Transformer 层做联合建模。**这里的“模态”是数据类型，例如文本、图像、音频、视频；“联合建模”是指模型不是先分别理解再拼接结果，而是在每一层里直接让不同模态互相影响。

这属于 **early fusion，早期融合**。白话解释是：文字、图片 patch、音频帧不是分开走两条流水线，而是一开始就排进同一个长队列。这样做的直接结果是，每层自注意力都共享同一个键值池，也就是所有 token 都能在同一轮计算里彼此查询。文本可以“看”图像，图像也可以“问”文本，音频和视频同样参与。

Gemini 的设计重点不只是“能融合”，而是**在长上下文里持续融合**。公开材料把它描述为结合了长上下文机制、局部窗口与全局 summary token，以及高容量结构，从而让跨模态信息在多层网络中持续流动，而不是只在输入端做一次粗糙对齐。这样才能支撑“根据一句文本描述，在超长视频上下文中定位某一帧”这类任务。

| 特征 | 贡献 |
| --- | --- |
| 早期融合 | 所有模态进入同一 Transformer stack，减少后端补丁式拼接 |
| 共享自注意力 | 每层都允许跨模态查询，信息流动不是单次发生 |
| 局部 + 全局机制 | 兼顾细节理解与长距离一致性 |
| 高容量结构 | 在长上下文和多模态同时存在时维持表达能力 |

---

## 问题定义与边界

问题本质是：**如何让一个 Transformer 同时处理文本、图像、音频、视频，并且让它们在同一层级上交换语义信息。**“语义信息”就是“这段内容到底在表达什么”。如果不同模态各算各的，最后再拼接，模型往往只能做浅层对应，难以形成稳定的跨模态推理链。

Gemini 选择的边界很明确：

1. 先把不同输入都表示成 token。
2. 这些 token 必须被映射到同一个隐藏空间，也就是同一种向量坐标系。
3. 后续层共享同一套注意力机制，而不是为每种模态维护完全独立的主干网络。
4. 在超长上下文下，不可能让每个 token 永远全连接，所以需要局部窗口、全局 token 或摘要路径控制计算量。

对新手来说，可以把它理解成一个“统一会场”。文本 token 是一句句文字拆开的单位，图像 patch token 是图片切成小块后的单位，音频 token 是一小段一小段时间窗的表示。Gemini 的做法不是让这些人先在各自房间开会，再把会议纪要送给总经理，而是让大家从一开始就在一个大会场里发言。

| 输入类型 | 处理方式 | 备注 |
| --- | --- | --- |
| 文本 token | 词或子词 embedding | 通常最接近语言输出端 |
| 图像 patch | 视觉编码后映射到共享空间 | 不直接保留像素，而是保留语义块 |
| 音频/视频 token | 按时间分块后编码 | 既有局部时间结构，也要接入全局上下文 |
| Global token | 摘要性记忆单元 | 用于长序列的信息汇总 |

这里还要强调一个边界：**Gemini 的优势不等于“所有多模态模型都该这样设计”。**如果任务只是图片检索、图文匹配，late fusion 也可能足够；只有当任务要求跨层、跨模态、长链条推理时，early fusion 的收益才明显。

---

## 核心机制与推导

自注意力的核心是：每个 token 生成查询向量 $Q$，再与全体 token 的键向量 $K$ 做匹配，得到权重后去加权值向量 $V$。白话解释是：当前 token 会问一句“我现在该参考谁”，然后从别人那里取信息。

Gemini 关键不在注意力公式本身，而在 **$K,V$ 来自全模态共享池**。如果某个文本 token 在第 $\ell$ 层更新时，既能访问文本 token，也能访问图像 token、音频 token，那么跨模态对齐就不是额外插件，而是主网络的默认行为：

$$
h_{\text{text}}^{(\ell+1)}=\mathrm{LayerNorm}\left[h_{\text{text}}^{(\ell)}+\mathrm{MultiHeadAttn}(Q=W_Q h_{\text{text}}^{(\ell)},K=W_K[h_{\text{text}}^{(\ell)};h_{\text{vis}}^{(\ell)}],V=W_V[h_{\text{text}}^{(\ell)};h_{\text{vis}}^{(\ell)}])\right]
$$

其中

$$
\mathrm{Attn}(Q,K,V)=\mathrm{softmax}\left(\frac{QK^\top}{\sqrt{d}}\right)V
$$

这里的 $d$ 是向量维度，用来做缩放，避免数值过大导致 softmax 过尖锐。

### 玩具例子

假设当前层只有 3 个 token：两个文本 token，记作 $T_1,T_2$；一个图像 patch，记作 $V_1$。某个注意力头计算后得到权重：

$$
[0.2, 0.3, 0.5]
$$

这表示当前更新的文本 token 会从三个位置取信息，其中 50% 来自视觉 token。于是这一层输出可以写成：

$$
0.2T_1 + 0.3T_2 + 0.5V_1
$$

这个例子说明了两点：

1. 跨模态信息不是在最后一步才接入，而是在每层都能接入。
2. 一旦某层已经混入视觉信息，下一层继续传播时，这部分视觉语义会和文本语义一起参与后续推理。

### 真实工程例子

考虑一个长视频问答系统。用户提问：“视频里出现红色警示灯并伴随高频报警音的是哪一段？”这不是单一图像理解，也不是单一语音识别。模型需要同时做三件事：

1. 理解文本查询“红色警示灯”“高频报警音”是什么意思。
2. 在大量视频帧里找到视觉上对应的红色闪烁区域。
3. 在大量音频片段里找到与“报警音”匹配的频谱模式。
4. 把视觉位置、音频时间、文本语义统一到一个答案里。

如果是 early fusion，那么文字、视频帧 token、音频 token 在主干网络中持续共存，文本查询可以直接引导视觉和音频 token 的激活分布。这就是跨模态检索能在超长上下文下仍然成立的原因。

## 代码实现

最小实现思路不是“写出完整 Gemini”，而是抓住它的结构要点：**不同模态先投影到同一维度，再拼接成一个序列，随后在共享注意力里更新。**

下面这个玩具实现只保留核心思想。它不是训练级代码，但能运行，并验证“文本 query 可以从视觉 token 获取信息”。

```python
import math

def softmax(xs):
    m = max(xs)
    exps = [math.exp(x - m) for x in xs]
    s = sum(exps)
    return [x / s for x in exps]

def dot(a, b):
    return sum(x * y for x, y in zip(a, b))

def cross_modal_update(query_token, all_tokens):
    scores = [dot(query_token, tok) / math.sqrt(len(query_token)) for tok in all_tokens]
    weights = softmax(scores)
    out = [0.0] * len(query_token)
    for w, tok in zip(weights, all_tokens):
        for i, v in enumerate(tok):
            out[i] += w * v
    return weights, out

# 两个文本 token，一个视觉 token
T1 = [1.0, 0.0]
T2 = [0.8, 0.2]
V1 = [0.0, 1.0]

weights, updated = cross_modal_update(T1, [T1, T2, V1])

assert len(weights) == 3
assert abs(sum(weights) - 1.0) < 1e-9
assert updated[0] > 0.0 and updated[1] > 0.0
assert weights[2] > 0.15  # 视觉 token 的确参与了更新

print("weights =", weights)
print("updated =", updated)
```

上面这段代码省略了多头注意力、残差连接、LayerNorm、位置编码和 mask，但保留了真正重要的部分：

| 实现步骤 | 作用 |
| --- | --- |
| 不同模态映射到同一维度 | 保证可以进入同一注意力计算 |
| `all_tokens = text + vision + audio` | 形成共享键值池 |
| `query_token` 单独更新 | 模拟某一模态读取全模态上下文 |
| `weights[2] > 0` | 证明视觉信息不是被隔离的 |

真实工程里还会再加几层机制。

第一层是 **modality id**，也就是模态标识。白话解释是给每个 token 加一个“我是文本/图像/音频”的标签，避免模型只看到向量值而分不清来源。

第二层是 **位置结构**。文本是线性顺序，图像常有二维位置，视频还有时间维。统一成 token 序列之后，位置编码必须补偿这些结构，不然“左上角 patch”和“下一秒音频帧”的关系会丢掉。

第三层是 **长上下文控制**。如果把百万级 token 全做全连接注意力，成本会过高，所以工程上通常要配合局部窗口、全局摘要 token 或稀疏模式。

## 工程权衡与常见坑

early fusion 的收益很大，但代价也明确。

第一，**文本偏置**很容易出现。所谓“偏置”，就是模型更相信某一类输入。因为训练语料常常语言监督更强，文本 token 更容易主导注意力分配，结果是模型明明看到了图像，却回答了“常识上更可能”的内容。

一个典型问题是：图里明明是异常场景，例如大象站在客厅里，但模型输出“客厅通常有沙发和电视”。这说明它没有真正被视觉证据约束，而是被语言先验带偏了。

第二，**不同模态的密度不一致**。一句文本可能只有十几个 token，一张高清图切 patch 后可能有上百个 token，一段视频再乘上时间轴会更大。如果不控制采样和权重，某一模态会在数量上压倒其他模态。

第三，**长上下文下的信息漂移**。漂移就是局部证据在很多层传播后被冲淡。工程上必须给“关键证据”保留稳定的传播路径，例如局部窗口保细节，全局 token 保摘要。

| 常见坑 | 现象 | 规避策略 |
| --- | --- | --- |
| 文本主导偏置 | 看图回答成常识续写 | 增加视觉 grounding loss、在指令中强调以视觉为准 |
| 模态数量失衡 | 视频 token 淹没文本 token | 分层采样、下采样、摘要 token |
| 长上下文漂移 | 前面出现的视觉证据后面失效 | 局部窗口 + 全局记忆路径 |
| 统一空间不稳 | 不同模态向量语义对不齐 | 对比学习、跨模态配对训练 |

真实工程里，复杂 UI 理解、自动驾驶、医疗影像问答都属于高风险场景。原因不是模型“看不见”，而是**模型可能看见了，但没有把视觉证据放在最终决策的最高优先级上**。这类任务不能只靠通用预训练后的自然表现，通常要补充显式对齐训练。

## 替代方案与适用边界

early fusion 不是唯一方案。常见替代方案至少有三类。

| 方案 | 核心思路 | 适用边界 | 主要问题 |
| --- | --- | --- | --- |
| Early fusion | 一开始就统一 token，主干层层联合建模 | 原生多模态推理、长上下文检索 | 训练和算力成本高 |
| Late fusion | 先各自编码，最后再合并表示或结果 | 快速给已有 LLM 加视觉能力 | 跨层对齐弱 |
| Cascade / adapter | 用额外模块让一种模态查询另一种模态 | 轻量升级、模块化部署 | 交互不够深，易有信息延迟 |

### 玩具对比

如果任务只是“图片和标题是否匹配”，late fusion 很自然：文本编码器出一个向量，图像编码器出一个向量，最后算相似度即可。因为这里只需要“匹不匹配”，不需要复杂的跨层推理。

但如果任务换成“请根据这段长文本说明，从 2 小时视频里找到对应画面，并解释前后因果”，late fusion 就明显吃亏。因为文本和视频没有在同一堆隐藏状态里持续交互，很多细粒度线索无法自然传播。

### 真实工程边界

在边缘设备或成本敏感系统里，常见做法是：先用视觉编码器抽特征，再把摘要喂给语言模型。这是合理的，因为部署目标首先是便宜和稳定，不是最强跨模态推理。反过来，像 Gemini 这类原生多模态大模型，更适合高复杂度问答、长视频理解、图文音频联合检索，因为它从架构上就为“多模态共同推理”付出了成本。

因此，适用边界可以直接记成一句话：**需要深度联合推理时选 early fusion，需要低成本快速落地时选 late fusion 或 adapter。**
