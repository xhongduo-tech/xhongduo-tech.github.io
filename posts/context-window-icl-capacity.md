## 核心结论

Transformer 的上下文窗口决定“能放多少 token”，但不直接决定“能学会多少个 in-context 示例”。这里的 in-context learning，简称 ICL，指模型不更新参数，只靠提示词里的示例临时完成任务。真正限制 ICL 有效示例数的，更接近一种“工作记忆”容量。

这个容量可以先用一个工程上很实用的近似式理解：

$$
H \approx \frac{d_{\text{model}}}{d_{\text{head}}}
$$

其中，$d_{\text{model}}$ 是模型隐藏维度，白话说就是每个 token 的主表示宽度；$d_{\text{head}}$ 是单个注意力头的维度，白话说就是每个头能独立处理信息的宽度；$H$ 是注意力头数，可以把它先看成并行的“概念槽”数量。

桌布类比最直接。上下文窗口像桌布，token 是桌面面积；注意力头像桌上的活页笔记本，负责临时记住不同概念。桌布再大，只会让已有笔记本摆得更稀，不能凭空多出新的笔记本。于是，长窗口解决的是“放得下”，不是“记得住”。

一个最小玩具例子：假设模型配置是 $d_{\text{model}}=4096,\ d_{\text{head}}=128$，那么

$$
H=4096/128=32
$$

这不代表模型只能看 32 个 token，而是说它能并行维护的独立注意力通道大致是 32 个。工程上如果你往上下文里塞 500 个高度相似、同标签的示例，模型并不会得到 500 份独立新信息，很多示例只是在重复争抢同一批槽位。

真实工程例子来自 Bertsch 等对长上下文 ICL 的系统实验。以 Clinic-150 为例，Llama2-80k 虽然最多能放下 2000 个示例，但达到最大精度 95% 时只需要约 1000 个示例；Llama2-32k 在同任务上约 750 个示例就达到 95% 饱和点。继续塞示例，收益明显变小，说明“窗口还没满”不等于“记忆还没满”。

```text
概念图
长桌布 = 上下文窗口 token 数
笔记本 = 注意力头 / 概念槽

桌布变大:  [ notebook ]     [ notebook ]        [ notebook ]
槽位不变:     还是这几本，只是排得更开
```

## 问题定义与边界

这篇文章讨论的不是“模型总共能记住多少知识”，而是“单次推理时，提示词里到底能放多少示例才还有效”。这是两个不同问题。前者是参数记忆，后者是工作记忆。

这里把“ICL 示例容量”定义成：模型在一次上下文中，仍能稳定利用的有效示例数。它不等于总 token 数，而更接近：

$$
\text{有效示例容量} \approx \text{概念槽数} \times \text{每槽可容纳的有效区分信息}
$$

边界主要有两个。

第一，token 增多不一定带来新概念。很多示例只是同一标签、同一模板、同一表达方式的重复，它们会挤进同一类注意力模式，带来的只是冗余。

第二，当示例继续增加时，注意力分布会变“摊薄”。注意力熵，白话说就是“注意力有多分散”，如果持续升高，通常意味着模型越来越难把注意力锁定到少数真正有用的位置。

下面用论文中的真实数据看这个边界。Bertsch 等把“饱和点”定义为：达到该模型在该数据集最大准确率的 95% 所需的示例数。对于分类任务，长窗口确实有帮助，但帮助并不是无限增长。

| 数据集 | 模型 | 窗口内最多示例数 | 达到 95% 最大精度所需示例数 | 观察 |
|---|---:|---:|---:|---|
| Banking-77 | Llama2-80k | 1750 | 750 | 远早于满窗就饱和 |
| Clinic-150 | Llama2-80k | 2000 | 1000 | 只用约 50% 窗口就接近最优 |
| Clinic-150 | Llama2-32k | 1169 | 750 | 约 64% 窗口处已接近饱和 |
| TREC | Llama2-80k | 2000 | 75 | 标签空间小，更早饱和 |

这组结果说明两个事实。第一，任务的标签空间越大，通常需要的示例越多；第二，即使是 80k 长窗口，真正有价值的示例数也常常远小于“塞得下的总数”。

如果再把示例按标签集中摆放，而不是随机混合，性能还会掉。Clinic-150 上，Llama2-32k 在 1169-shot 时，按标签排序会让准确率下降 25.7 个百分点。这说明问题不只是“多少示例”，还有“示例如何占用槽位”。

## 核心机制与推导

多头注意力可以先粗略看成并行的多个选择器。每个 head 都会在上下文中寻找“和当前问题最匹配的位置”。head 首次出现时，可以把它理解成“一个独立的小检索器”。

标准多头配置满足：

$$
d_{\text{model}} = H \cdot d_{\text{head}}
$$

所以

$$
H = \frac{d_{\text{model}}}{d_{\text{head}}}
$$

这就是为什么工程上很多人说“头数决定并行检索槽位”。这不是说每个头严格对应一个语义概念，而是说在许多 ICL 场景下，它们表现得像有限个可并行工作的概念槽。

Mahdavi 等给出了更理论化的结果：在一定假设下，一个带 $H$ 个头、上下文长度为 $n$ 的多头注意力层，可以记住的样本数满足

$$
T = \Omega(Hn)
$$

并且在特定设定下，参数规模是 $\Theta(Hd^2)$ 时，能记住 $\Omega(Hn)$ 个例子。这里的 $\Omega(\cdot)$ 是下界记号，白话说就是“至少能随这个量级增长”。

为什么头数重要？因为 softmax 饱和时，一个 head 会倾向把大部分权重压到少数位置。softmax 首次出现时，可以把它理解成“把分数变成概率，并放大高分项”。当某个位置分数显著更高，这个 head 就近似成一个定向读取槽。

但随着示例数变多，两个变化会同时出现。

第一，单个 head 需要在更多候选位置间分配概率。  
第二，不同示例之间更容易互相干扰，尤其是标签相同、格式相似时。

Gong 与 Zhang 在 N-back 任务中观察到，随着 $N$ 增大，注意力总熵上升，而准确率下降。N-back 是一种工作记忆任务，白话说就是“当前位置要回忆前面第 N 个位置的内容”。他们定义的总熵为：

$$
H_N(A)=-\sum_i\sum_{j=1}^{i} A_{i,j}\log A_{i,j}
$$

如果 $H_N(A)$ 上升，说明注意力更分散，模型更难稳定锁定真正该看的位置。把这个现象映射到 ICL，就是：示例太多时，模型不是得到更多独立记忆，而是在已有槽位上被迫更平均地分配注意力。

可以把机制总结成下面这条链路：

```text
示例数增加
-> 候选位置增多
-> 每个 head 的注意力更难尖锐集中
-> 熵上升，检索更分散
-> 有效示例利用率下降
-> 准确率进入平台区
```

所以“长窗口提升 ICL”这句话只说对了一半。更准确的说法是：长窗口先提供更大的检索池，但真正能把多少示例转化成有效任务信息，还受可并行槽位和注意力集中能力限制。

## 代码实现

工程上最实用的做法不是把上下文塞满，而是先根据模型配置估算“概念槽预算”，再按标签多样性组装示例，并监控注意力熵或其代理指标。

下面给一个可以直接运行的 Python 小脚本。它做三件事：

1. 由 $d_{\text{model}}$ 和 $d_{\text{head}}$ 估算 head 数。  
2. 用 `k * H` 作为示例上限，避免盲目堆样本。  
3. 用一组模拟注意力分布计算熵，演示“分布越散，熵越高”。

```python
import math
from collections import defaultdict

def head_count(d_model: int, d_head: int) -> int:
    assert d_model > 0 and d_head > 0
    assert d_model % d_head == 0
    return d_model // d_head

def shannon_entropy(probs):
    assert probs, "empty probability list"
    s = sum(probs)
    assert abs(s - 1.0) < 1e-9
    return -sum(p * math.log(p) for p in probs if p > 0)

def choose_balanced_examples(pool, max_examples):
    buckets = defaultdict(list)
    for item in pool:
        buckets[item["label"]].append(item)

    labels = sorted(buckets.keys())
    chosen = []
    i = 0
    while len(chosen) < max_examples and labels:
        label = labels[i % len(labels)]
        if buckets[label]:
            chosen.append(buckets[label].pop(0))
        i += 1
        if all(len(v) == 0 for v in buckets.values()):
            break
    return chosen

if __name__ == "__main__":
    d_model = 4096
    d_head = 128
    H = head_count(d_model, d_head)
    assert H == 32

    # 工程上可调: 每个 head 先给 8 个示例预算
    max_examples = 8 * H
    assert max_examples == 256

    pool = [
        {"id": i, "label": f"intent_{i % 10}"}
        for i in range(1000)
    ]
    chosen = choose_balanced_examples(pool, max_examples)

    label_counts = defaultdict(int)
    for x in chosen:
        label_counts[x["label"]] += 1

    # 验证大致均衡
    counts = sorted(label_counts.values())
    assert counts[-1] - counts[0] <= 1

    # 两组注意力分布: 尖锐 vs 分散
    sharp = [0.88, 0.07, 0.03, 0.02]
    flat = [0.30, 0.25, 0.23, 0.22]
    sharp_H = shannon_entropy(sharp)
    flat_H = shannon_entropy(flat)

    assert sharp_H < flat_H
    print("H =", H)
    print("max_examples =", max_examples)
    print("sharp_entropy =", round(sharp_H, 4))
    print("flat_entropy =", round(flat_H, 4))
```

玩具例子可以这样理解。假设你做 10 类意图分类，示例库有 1000 条。如果模型只有 32 个头，你先给每个头分配 8 条预算，总共选 256 条，并尽量做标签均衡。这个策略通常比“随机塞 1000 条、其中大部分都是高频标签”更稳。

真实工程例子是 Banking-77 或 Clinic-150 这类大标签空间任务。一个常见做法是：离线挑出 500 到 750 条标签分布均衡的示例，作为固定缓存；在线请求时，不再每次全量检索和重编码，而是只拼接缓存和当前 query。这样做的好处是推理成本更低，而且通常不会明显损失长窗口 ICL 已经达到的平台区精度。

| 模型配置 | $H=d_{model}/d_{head}$ | 建议起始示例预算 | 适用情况 |
|---|---:|---:|---|
| 4096 / 128 | 32 | 128 到 256 | 中等复杂分类 |
| 5120 / 128 | 40 | 160 到 320 | 标签更多的意图任务 |
| 8192 / 128 | 64 | 256 到 512 | 复杂标签空间或多任务提示 |

## 工程权衡与常见坑

最常见的错误是把“长窗口”直接等同于“多示例一定更好”。这会在两个层面出问题。

第一，同标签堆积会浪费槽位。示例看起来很多，但语义覆盖很窄，模型只是在重复看到同一种模式。  
第二，示例排序不当会破坏局部上下文化。Bertsch 等已经证明，按标签聚类排序会显著伤害长上下文 ICL。

下面是更实用的坑位表。

| 常见坑 | 现象 | 规避策略 |
|---|---|---|
| 盲目填满窗口 | token 成本高，但准确率几乎不再涨 | 先做饱和点实验，找到 95% 最大精度对应示例数 |
| 同标签样本过多 | 高频标签占满工作记忆，长尾标签识别差 | 强制标签均衡或按语义簇做配额 |
| 示例按标签成组排列 | 模型更难做跨标签对比，泛化下降 | 随机打散，或做受控交错排序 |
| 每次都重检索重编码 | 延迟和成本高，结果波动大 | 固定一组高质量缓存示例，再做少量增量检索 |
| 只看窗口长度，不看 head 结构 | 以为 80k 一定优于 32k | 同时评估头数、任务标签空间、示例冗余度 |
| 只堆数量，不筛质量 | 噪声示例把注意力拉散 | 优先保留边界样本、难样本、标签代表样本 |

真实工程里，Banking-77 和 Clinic-150 这类任务很适合“固定工作记忆组合”。做法不是每次都从几万条库里动态找 1000 条，而是先离线缓存 500 到 750 条代表性示例，保证标签覆盖、表达多样、格式统一。这样往往已经接近长窗口 ICL 的有效上限，再继续加示例，收益通常不值额外成本。

## 替代方案与适用边界

如果概念槽是瓶颈，替代方案就不该只是“再加 token”，而应该改变信息进入槽位的方式。

第一类方案是提升示例质量而不是数量。对分类、路由、抽取这类任务，优先保留标签边界清晰、格式稳定、覆盖长尾标签的示例，通常比多放几十条同类样本更有效。

第二类方案是分层处理长文本。比如先 chunk，再 summary，再把摘要层结果喂给最终决策提示。这种方法不是突破了 $H$ 的限制，而是把原本同一层里争抢槽位的信息，改造成多轮复用槽位。

第三类方案是外部记忆缓存。也就是把检索、摘要、缓存三步放到上下文外部完成，最终只把已经压缩过的信息送进模型工作区。

| 方案 | 做法 | 优点 | 边界 |
|---|---|---|---|
| 直接塞示例 | 把大量原始示例放进单次提示 | 简单直接 | 很快碰到槽位饱和 |
| 分层提问 | chunk -> summary -> final answer | 降低单层竞争 | 摘要质量差会丢关键信息 |
| 记忆缓存 | 离线选代表样本并复用 | 成本低、稳定性好 | 对分布漂移敏感 |
| 检索增强 | 只取与 query 最相关的少量示例 | token 利用率高 | 检索误差会直接传递 |

什么时候该用哪种？

如果任务是标签固定的意图分类，优先用“均衡缓存 + 少量动态补充”。  
如果任务是超长文档问答，优先用“分块总结 + 最终汇总”。  
如果任务经常出现新概念、新标签、新实体，仅靠缓存不够，需要检索增强。  
如果任务本身要比较多个概念，而这些概念彼此相似，就要特别警惕槽位被重复样本占满。

核心判断标准不是“窗口还剩多少”，而是“当前上下文里还有没有新的、可区分的概念信息值得占一个槽”。
