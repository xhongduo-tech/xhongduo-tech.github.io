## 核心结论

ALiBi（Attention with Linear Biases，在线性偏置下的注意力）不是给 token 再加一个位置向量，而是在注意力打分阶段直接加入一个“距离惩罚”。白话说，模型先判断两个位置的内容像不像，再额外扣掉“它们离得有多远”带来的分数。

标准公式可以写成：

$$
s_{ij}^{(h)}=\frac{q_i k_j^\top}{\sqrt d}-m_h|i-j|
$$

其中，$q_i k_j^\top / \sqrt d$ 是内容相似度，表示“这两个 token 在语义上是否相关”；$m_h|i-j|$ 是距离惩罚，表示“即使相关，离得越远也默认越不优先”。$m_h$ 是第 $h$ 个注意力头的斜率，不同头会用不同斜率覆盖不同距离尺度。

一个最直接的理解方式是：ALiBi 解决的是“位置偏置怎么注入”，不是“注意力如何降复杂度”。它不会把 $O(T^2)$ 的注意力复杂度变成更低，只是让注意力在不引入额外位置参数的前提下，天然偏向近邻，同时保留远距离依赖的可能性。

| 组成部分 | 负责什么 | 不负责什么 |
|---|---|---|
| 内容相似度 $\frac{q_i k_j^\top}{\sqrt d}$ | 判断语义相关性 | 不知道先后顺序 |
| 距离惩罚 $m_h|i-j|$ | 给近邻更高默认优先级 | 不改变注意力复杂度 |

玩具例子：假设当前位置要决定看三个候选 token，内容分数分别是 `8.0, 8.2, 8.5`，距离分别是 `0, 2, 8`，斜率 $m=0.2$。加入 ALiBi 后分数变成 `8.0, 7.8, 6.9`。原本最远的那个 token 内容最相似，但距离太远，被线性惩罚后不再是首选。这个例子说明 ALiBi 不是禁止远距离，而是让远距离必须“更有价值”才值得被关注。

## 问题定义与边界

Transformer 的自注意力本身只看 token 之间的内容匹配，不天然知道“谁在前、谁在后、谁离谁更近”。白话说，如果不给任何位置相关信息，模型看到的是一个集合，而不是一条有顺序的序列。

这会带来三个常见问题：

| 问题 | 传统注意力的表现 | ALiBi 的处理 |
|---|---|---|
| 不知道顺序 | 只能看内容 | 加距离惩罚 |
| 长度外推差 | 超出训练长度容易退化 | 用距离规则直接泛化 |
| 近邻偏好不足 | 需要模型自己学 | 直接编码偏好 |

为什么“长度外推”重要？长度外推的意思是：训练时只见过较短序列，推理时希望模型还能处理更长序列。很多绝对位置编码依赖一个固定长度范围，超出后要么没有定义，要么效果明显下降。ALiBi 没有位置向量表，它只依赖相对距离 $|i-j|$，所以在训练长度之外仍然能按照同样规则工作。

真实工程例子：代码补全任务里，当前行通常最依赖最近几行局部上下文，例如变量赋值、函数入参、缩进块状态；但更远处的函数签名、类定义、import 也可能有用。ALiBi 的目标不是让模型只看近处，而是让模型默认更看近处，同时保留“远处如果信息足够关键，仍然能被选中”的机制。

它的适用边界也要说清楚：

- 适合：因果语言模型、长上下文生成、训练长度与推理长度不一致的场景。
- 不适合：强依赖绝对位置编号的任务，例如“第 3 个字段”和“第 8 个字段”语义完全不同，且这种差异不能只用相对距离表达。
- 也不适合直接处理二维或图结构位置。比如图像 patch 的二维相对位置、表格单元格的行列关系，仅靠一维线性距离通常不够。

## 核心机制与推导

先看标准注意力。对第 $i$ 个查询位置和第 $j$ 个键位置，标准打分是：

$$
a_{ij}^{(h)}=\frac{q_i k_j^\top}{\sqrt d}
$$

这个分数只反映内容相关性，不包含位置信息。ALiBi 的做法是在 softmax 之前直接加一个偏置项：

$$
s_{ij}^{(h)}=a_{ij}^{(h)}-m_h|i-j|
$$

这里的“偏置”可以理解成“在内容分数之外，额外施加的固定规则”。因为是线性的，距离每增加 1，分数就固定减少 $m_h$。这比学习一个庞大的位置表更简单，也更容易外推。

先看最小推导。假设当前位置 $i=3$，可看的历史位置是 $j=3,2,1,0$，那么距离分别是：

- $|3-3|=0$
- $|3-2|=1$
- $|3-1|=2$
- $|3-0|=3$

如果某个头的斜率 $m_h=0.5$，那么距离偏置就是：

- $0$
- $-0.5$
- $-1.0$
- $-1.5$

这说明即使四个位置内容分数一样，softmax 之前近处位置也会更高。

为什么每个头要用不同斜率？多头注意力的“头”可以理解成并行的多个观察视角。ALiBi 给不同头分配不同的 $m_h$，本质上是在做多尺度建模：

| 斜率大小 | 效果 |
|---|---|
| 大 | 更偏向近邻 |
| 中 | 兼顾局部和中程 |
| 小 | 更容易保留远距离信息 |

这很重要。因为如果所有头都用同一个大斜率，远距离信息会被整体压得太狠；如果都用同一个小斜率，位置偏好又太弱。多头不同斜率相当于让一部分头负责局部模式，另一部分头负责更长的依赖。

在因果注意力里，未来位置本来就不可见，所以通常写成等价形式：

$$
B_{ij}^{(h)}=
\begin{cases}
m_h(j-i), & j\le i \\
-\infty, & j>i
\end{cases}
$$

因为当 $j\le i$ 时，$|i-j|=i-j$，所以

$$
-m_h|i-j| = -m_h(i-j)=m_h(j-i)
$$

这样写有两个工程好处。第一，因果场景下不需要显式算绝对值；第二，偏置矩阵天然与下三角 mask 对齐。要注意，$j>i$ 的 $-\infty$ 来自因果 mask，不是 ALiBi 自己产生的。ALiBi 负责“离得远扣分”，causal mask 负责“未来完全不可见”，两者不是一回事。

## 代码实现

工程实现的核心只有一句话：在 attention logits 上加一个按“头、查询位置、键位置”组织的 bias 矩阵，然后再做 mask 和 softmax。

实现流程可以概括成下面四步：

| 步骤 | 输入 | 输出 |
|---|---|---|
| 计算内容分数 | `q, k` | `scores` |
| 加距离偏置 | `scores, alibi_bias` | `biased_scores` |
| 应用 causal mask | `biased_scores` | 可见位置分数 |
| softmax | 可见位置分数 | 注意力权重 |

最小可运行 Python 例子如下。这个例子不依赖深度学习框架，只用 `math`，目的不是训练模型，而是把机制跑通。

```python
import math

def softmax(xs):
    m = max(xs)
    exps = [math.exp(x - m) for x in xs]
    s = sum(exps)
    return [x / s for x in exps]

def alibi_scores(content_scores, query_pos, slopes):
    # content_scores: [num_heads][seq_len]
    out = []
    for h, row in enumerate(content_scores):
        m_h = slopes[h]
        biased = []
        for key_pos, score in enumerate(row):
            distance = abs(query_pos - key_pos)
            biased.append(score - m_h * distance)
        out.append(biased)
    return out

# 玩具例子：2 个头，当前位置 query_pos=3，只看 4 个历史位置
content_scores = [
    [1.0, 1.5, 2.0, 2.2],  # head 0
    [1.0, 1.5, 2.0, 2.2],  # head 1
]
slopes = [1.0, 0.1]  # head 0 强局部，head 1 更保留远距离
query_pos = 3

biased = alibi_scores(content_scores, query_pos, slopes)

head0_weights = softmax(biased[0])
head1_weights = softmax(biased[1])

# 距离最近的位置 index=3 应更受偏好
assert head0_weights[3] > head0_weights[2] > head0_weights[1] > head0_weights[0]

# 小斜率的头对远距离更宽容，因此最远位置权重更高
assert head1_weights[0] > head0_weights[0]

print("head0 biased scores:", biased[0])
print("head0 weights:", head0_weights)
print("head1 biased scores:", biased[1])
print("head1 weights:", head1_weights)
```

如果放回真实模型，伪代码通常就是：

```python
scores = q @ k.transpose(-2, -1) / math.sqrt(d)
scores = scores - alibi_bias
scores = scores.masked_fill(causal_mask == 0, float("-inf"))
attn = scores.softmax(dim=-1)
```

这里有三个实现要点：

- `alibi_bias` 可以按最大序列长度预先构造，再在实际前向时切片复用。
- `alibi_bias` 通常形状是 `[num_heads, seq_len, seq_len]` 或能广播到这个形状。
- 它不需要训练额外参数，训练中只更新正常的 `q/k/v` 和其他模型参数。

真实工程例子：做长上下文代码模型时，训练窗口可能是 4K token，但线上希望支持 16K 或更长。如果你用一个固定训练长度的 learned positional embedding，超出部分要么没有训练过，要么需要额外插值和外推处理。ALiBi 直接按距离扣分，不依赖“第 4097 个位置的专属向量”，因此迁移到更长上下文时实现更直接，行为也更连续。

## 工程权衡与常见坑

ALiBi 的优势很明确：简单、零额外位置参数、长度外推自然、实现位置低侵入。它常被喜欢的原因，不是它在所有任务上都最强，而是它用很低的复杂度换来不错的长度泛化能力。

但工程里最常见的问题不是公式背错，而是把三件不同的事混在一起：

- 距离惩罚：决定默认更看近处。
- causal mask：决定能不能看未来。
- 长度外推：决定训练长度外是否还能工作。

这三者逻辑不同，必须分开理解。

| 常见坑 | 后果 | 怎么避免 |
|---|---|---|
| 没加 causal mask | 看到未来 token | mask 与 bias 分开实现 |
| 斜率过大 | 远距离信息被压死 | 用多尺度斜率 |
| 斜率过小 | 位置偏好太弱 | 检查 bias 是否生效 |
| 误解“可外推” | 以为无限长度都稳定 | 做真实长序列回归测试 |
| 只关注位置编码 | 忽略 $O(T^2)$ 计算 | 明确它不解决复杂度 |

一个容易忽视的点是：可外推不等于无限稳定。ALiBi 的规则在长序列上仍然成立，但模型其他部分是否能真正利用超长上下文，仍然受训练分布、优化过程、数据覆盖和数值稳定性影响。比如模型训练时大多数样本都在 1K 以内，即使位置机制能支持 8K，模型也不一定在 8K 上真的学会了有效检索。

再看一个权衡表：

| 维度 | ALiBi | 典型结论 |
|---|---|---|
| 简单性 | 高 | 改 logits 即可 |
| 外推性 | 较强 | 通常优于固定长度绝对位置表 |
| 位置表达能力 | 中等 | 更偏“距离偏好”，不是丰富结构建模 |
| 与 RoPE 对比 | 更直接外推 | 但几何结构表达通常不如 RoPE 丰富 |

实际调参时，斜率设计是关键。斜率过大时，softmax 前的分差会迅速拉开，远距离 token 即使内容很相关，也很难翻盘；斜率过小时，ALiBi 接近“几乎没加位置偏置”，收益有限。所以很多实现会采用按头分布的一组固定斜率，而不是手工给所有头同一个值。

## 替代方案与适用边界

ALiBi 不是唯一的位置方案，它的核心优势是“用相对距离偏好换取外推和实现简单”。如果任务更看重复杂的位置结构，其他方法可能更合适。

| 方案 | 优点 | 局限 | 适用场景 |
|---|---|---|---|
| ALiBi | 零参数、易外推 | 位置表达较简单 | 长上下文生成 |
| RoPE | 相对位置信号强 | 超长时常需要额外缩放策略 | 通用 Transformer |
| 绝对位置编码 | 实现简单 | 外推弱 | 固定长度任务 |
| Learned position | 灵活 | 长度泛化差 | 训练长度稳定场景 |

可以这样理解它们的差异：

- ALiBi 直接修改打分，表达的是“离得越远，默认越不重要”。
- RoPE（旋转位置编码，给向量做位置相关旋转）把位置信息编码到查询和键本身里，表达更细，但超长外推通常要额外处理。
- 绝对位置编码给每个位置一个独立表示，简单直接，但很依赖训练长度范围。
- Learned positional embedding 最灵活，但最容易受“没见过的位置”限制。

什么时候优先考虑 ALiBi？

- 任务是因果语言建模。
- 推理长度很可能超过训练长度。
- 更关心长距离泛化，而不是复杂结构位置建模。
- 希望实现改动少，不想引入额外位置参数表。

什么时候要优先比较 RoPE 或其他方案？

- 任务更看重细粒度相对位置关系。
- 模型体系或现有基础设施已经深度依赖 RoPE。
- 需要处理比“一维线性序列距离”更复杂的结构，例如二维布局、多模态对齐、图结构邻接。

短文本分类就是一个典型边界例子。若输入长度很短，且结果更多由整体语义决定，位置机制往往不是主要瓶颈，此时 ALiBi 的收益可能有限。相反，在长上下文生成、代码建模、长文问答这类任务里，它更有价值，因为这些任务确实需要“默认近邻优先，但远处信息仍可用”的偏置。
