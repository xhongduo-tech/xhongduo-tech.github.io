## 核心结论

ALiBi 的核心做法不是“给 token 加一个位置向量”，而是“在注意力打分时按距离直接扣分”。注意力分数原本是

$$
\frac{QK^T}{\sqrt{d}}
$$

加入 ALiBi 后变成

$$
\mathrm{softmax}\left(\frac{QK^T}{\sqrt{d}} + B^{(h)}\right)
$$

其中 $B^{(h)}$ 是第 $h$ 个注意力头的线性偏置矩阵。白话说，**模型在选上下文时，不再靠嵌入里的位置信息记住顺序，而是在“打分阶段”直接对远距离 token 罚分**。

最常见的因果自注意力场景里，若查询位置是 $i$、键位置是 $j$，且只能看过去位置 $j \le i$，则常写为：

$$
B_{ij}^{(h)} = m_h \cdot (j - i)
$$

因为 $j - i \le 0$，所以这本质上等价于减去 $m_h \cdot |i-j|$。距离越远，值越负，罚分越大。

一个最小玩具例子：

- 第 3 个 token 查询第 1、2、3 个键
- 某个头的斜率 $m=0.1$
- 则 bias 分别是 $-0.2,-0.1,0$

这意味着即便 $QK^T$ 的原始相似度差不多，模型也会更偏向离当前位置更近的上下文。新手可以直接记成一句话：**距离越远，扣分越多**。

不同头使用不同斜率，是 ALiBi 的第二个关键点。白话说，**多头注意力是让多个“观察者”从不同角度看上下文**。斜率大的头更看近邻，斜率小的头保留更长视野，于是模型同时拥有“局部语法感”和“长程依赖感”。

| 斜率 $m_h$ | 关注倾向 | 直观理解 |
|---|---|---|
| 大 | 更近 | 远处稍微不相似就被压下去 |
| 中 | 中等距离 | 兼顾局部与全局 |
| 小 | 更远 | 对长距离信息更宽容 |

ALiBi 最大的工程价值在于：**它不引入新的可学习参数，却通常比绝对位置编码更容易做长度外推**。训练时见过 1024 长度，推理时跑到 2048 或更长，偏置函数仍然成立，因为“距离惩罚”本身没有依赖一个固定训练长度表。

## 问题定义与边界

Transformer 本身只会做向量相似度匹配，不天然知道“谁在前、谁在后”。位置编码就是为了解决这个问题。白话说，**如果不告诉模型顺序，句子里的 token 对它来说像一袋散乱的词片段**。

传统做法主要有两类：

- 绝对位置编码：给每个位置一个固定或可学习向量，加到 token embedding 上
- 相对位置编码：让模型在注意力里感知 token 之间的相对距离

问题在于，很多绝对位置方案在训练长度之外不稳定。因为模型在训练时只见过有限长度位置，推理时如果突然出现更长序列，位置模式可能失真。ALiBi 试图解决的就是这个边界：**让模型在不增加额外参数的前提下，尽量平滑地从短序列泛化到长序列**。

它的适用边界也很明确：**优先面向因果自注意力，也就是 GPT 类“只能看左边”的生成模型**。因果 mask 是一种遮罩规则，表示当前位置只能看自己和之前的位置，不能偷看未来 token。此时 $j \le i$ 恒成立，距离定义清晰，ALiBi 的偏置矩阵也最自然。

在这种设置下，矩阵结构可以写成：

$$
B_{ij}^{(h)}=
\begin{cases}
m_h (j-i), & j \le i \\
-\infty, & j > i
\end{cases}
$$

其中 $-\infty$ 不是 ALiBi 本身，而是因果 mask 的遮挡效果，表示未来位置完全不可见。

可以把它想成一个简化示意：

| 查询位置 $i$ \\ 键位置 $j$ | 1 | 2 | 3 | 4 |
|---|---:|---:|---:|---:|
| 1 | 0 | mask | mask | mask |
| 2 | $-m$ | 0 | mask | mask |
| 3 | $-2m$ | $-m$ | 0 | mask |
| 4 | $-3m$ | $-2m$ | $-m$ | 0 |

这张表说明两件事：

- 只能看左下三角区域，因为有因果 mask
- 在可见区域里，越远的位置偏置越负

一个新手友好的真实使用场景是 GPT 写代码补全。模型生成当前位置 token 时，只能看当前之前的代码片段。ALiBi 让它在“最近几行”和“更远函数定义”之间自动形成偏好层次，而不需要把每个位置写死到 embedding 中。

但如果是双向编码器任务，比如 BERT 做句子分类，token 既要看左边也要看右边，此时 ALiBi 的原始因果定义就不能直接照搬。不是说完全不能改，而是**它不是默认适配的设计目标**。

---

## 核心机制与推导

先看原始注意力。设查询矩阵为 $Q$、键矩阵为 $K$、值矩阵为 $V$，则单头注意力是

$$
\mathrm{Attention}(Q,K,V)=\mathrm{softmax}\left(\frac{QK^T}{\sqrt{d}}\right)V
$$

这里的 $QK^T/\sqrt{d}$ 可以理解成“相关性分数表”，每一项表示当前位置该不该关注另一个位置。

ALiBi 只改这一步分数，不改 $Q,K,V$ 的构造：

$$
\mathrm{Attention}(Q,K,V)=\mathrm{softmax}\left(\frac{QK^T}{\sqrt{d}} + B^{(h)}\right)V
$$

其中

$$
B_{ij}^{(h)} = m_h (j - i)
$$

在因果场景下 $j \le i$，所以它总是非正数。等价地看，也就是减去 $m_h |i-j|$。这就是“线性衰减”的含义：距离每增加 1，分数固定减少 $m_h$。

继续看前面的玩具例子。假设某头在位置 $i=3$ 时，对三个可见键的原始分数是：

| 键位置 $j$ | 原始 $QK^T/\sqrt d$ | ALiBi bias | 相加后 |
|---|---:|---:|---:|
| 1 | 1.2 | -0.2 | 1.0 |
| 2 | 1.1 | -0.1 | 1.0 |
| 3 | 0.95 | 0 | 0.95 |

如果没有 ALiBi，位置 1 的分数最高；加入后，1 和 2 被拉平，3 也没有被甩得太远。再经过 softmax，注意力分布会更偏近邻。**注意，ALiBi 不是强行禁止远距离依赖，而是给远距离依赖增加额外门槛**。远处 token 如果语义上非常重要，原始相似度仍然可以战胜这部分罚分。

为什么多头要用不同斜率？因为多头的目的不是重复做同一件事，而是分工。一个常见做法是让斜率按几何级数递减。白话说，**每往后一个头，对距离的惩罚缩小一点**。

示意表：

| Head Index | 斜率示意 | 倾向 |
|---|---:|---|
| 0 | 1/2 | 强烈偏近邻 |
| 1 | 1/4 | 偏近邻 |
| 2 | 1/8 | 中等 |
| 3 | 1/16 | 保留更长距离 |

这样会形成一种自然分层：

- 大斜率头负责局部模式，比如相邻 token、短词组、括号闭合
- 小斜率头负责长程模式，比如变量定义与使用、段落主题延续

简化后的注意力偏置矩阵看起来像这样：

```text
  j=1   2    3    4
i=1   0
  2  -m    0
  3 -2m   -m    0
  4 -3m  -2m   -m    0
```

它本质上是一张“越远越吃亏”的下三角表，而且每个头各自有一张，因为 $m_h$ 不同。

为什么这能帮助长度外推？因为 ALiBi 学到的不是“第 513 个位置长什么样”，而是“远一点就减几分”这种规则。前者依赖训练里是否见过具体位置，后者只依赖距离函数本身。只要推理时还是同样的因果结构，长度变长不会破坏这个规则。

一个真实工程例子是长文档生成或代码仓库问答。模型训练时如果上下文窗口是 1024，部署时希望输入 2048 甚至更长。使用绝对位置嵌入时，超出训练范围的位置常常是风险点；使用 ALiBi 时，位置偏置仍可直接按距离算出来，所以通常更平滑。

## 代码实现

工程上，ALiBi 的实现非常直接：**先算普通 attention scores，再加上按 head 和位置构造的 bias tensor**。这个 bias 往往可以预计算，并在 batch 维上广播复用。

下面给出一个可运行的 Python 例子，展示如何构造因果 ALiBi 偏置，并验证玩具例子中的数值。

```python
import math

def build_alibi_bias(num_heads: int, seq_len: int, slopes):
    assert len(slopes) == num_heads
    bias = []
    for h in range(num_heads):
        m = slopes[h]
        head_bias = []
        for i in range(seq_len):
            row = []
            for j in range(seq_len):
                if j <= i:
                    row.append(m * (j - i))   # <= 0
                else:
                    row.append(float("-inf")) # causal mask
            head_bias.append(row)
        bias.append(head_bias)
    return bias

# 玩具例子：1 个头，长度 4，斜率 0.1
bias = build_alibi_bias(num_heads=1, seq_len=4, slopes=[0.1])

# 第 3 个 token（下标 2）查看第 1/2/3 个键（下标 0/1/2）
row = bias[0][2]
assert abs(row[0] - (-0.2)) < 1e-9
assert abs(row[1] - (-0.1)) < 1e-9
assert abs(row[2] - 0.0) < 1e-9
assert math.isinf(row[3]) and row[3] < 0

def add_bias_to_scores(scores, bias_row):
    out = []
    for s, b in zip(scores, bias_row):
        out.append(s + b if not math.isinf(b) else b)
    return out

raw_scores = [1.2, 1.1, 0.95, 0.7]
new_scores = add_bias_to_scores(raw_scores, row)

assert abs(new_scores[0] - 1.0) < 1e-9
assert abs(new_scores[1] - 1.0) < 1e-9
assert abs(new_scores[2] - 0.95) < 1e-9
assert math.isinf(new_scores[3]) and new_scores[3] < 0

print("ALiBi bias 构造与示例验证通过")
```

如果映射到深度学习框架，伪代码通常是：

```python
# Q: [B, H, T, D]
# K: [B, H, T, D]
scores = (Q @ K.transpose(-1, -2)) / sqrt(D)   # [B, H, T, T]
scores = scores + alibi_bias                    # [1, H, T, T] 广播到 batch
scores = scores + causal_mask
weights = softmax(scores, dim=-1)
output = weights @ V
```

这里的广播意思是：**同一个偏置模板可同时作用于一个 batch 里的所有样本**，因为只要序列长度相同，距离结构就相同。

一个实际工程上的配置表可以这样理解：

| 头编号 | 斜率选择 | 作用 |
|---|---:|---|
| 0 | 0.5 | 最强调局部 |
| 1 | 0.25 | 强调局部 |
| 2 | 0.125 | 中距离 |
| 3 | 0.0625 | 长距离更宽容 |

如果头数更多，常见做法是继续按几何级数扩展，而不是让多个头完全共享同一个斜率。

## 工程权衡与常见坑

ALiBi 代码短，不代表没有工程权衡。

第一，**斜率分布必须和头数匹配**。如果 16 个头只重复 4 种斜率，等于很多头在做高度相似的事情，多头分工会变差。极端情况下，一批头全部只盯近邻，远距离信息就只能靠少数头承担，训练会不稳定。

第二，**斜率过大或过小都可能有问题**。

- 过大：远距离 token 很难翻盘，头会变得“近视”
- 过小：偏置几乎不起作用，和没加差不多

第三，**必须和因果 mask 配合正确**。ALiBi 不是 mask 的替代品。它只是在允许可见的位置上调整分数，不会自动阻止模型看未来 token。实现时如果忘了 causal mask，会直接破坏自回归训练。

第四，**长序列显存问题没有被根治**。ALiBi 解决的是位置表示，不是注意力复杂度。序列从 1024 扩到 4096，$T \times T$ 的注意力矩阵仍然更大。它让“表示规则”更容易外推，但不让标准注意力免费变便宜。

第五，**训练外推不等于无限外推**。训练到 1024，测试到 2048 往往还合理；测试到 32768 是否稳定，要看模型容量、数据分布、优化设置。ALiBi 提供的是更平滑的归纳偏置，不是数学保证。

一个常见真实工程坑是代码模型训练。团队给 16 个头只配了少数几个手工斜率，结果前几层大量头都聚集在局部 token，模型对函数跨文件引用的追踪能力明显弱。修正方法通常很直接：**每个头独立斜率，并保证从大到小平滑覆盖多个尺度**。

上线前至少检查这几项：

| 检查项 | 为什么要查 |
|---|---|
| 每个头是否有独立斜率 | 防止多头退化成重复分工 |
| 斜率是否单调递减 | 保证近邻到远距视野连续覆盖 |
| causal mask 是否单独存在 | 防止误看未来 token |
| bias shape 是否正确广播 | 防止 batch/head 维度错位 |
| 长度外推是否做过离线评测 | 防止“理论能推，实际崩掉” |

## 替代方案与适用边界

ALiBi 不是唯一位置方案，更不是所有任务的默认最优解。它的优势主要集中在“因果生成 + 希望长度外推 + 不想增加参数”这组条件上。

下面做一个简表对比：

| 方案 | 位置注入方式 | 额外参数 | 长度外推 | 典型适用场景 |
|---|---|---:|---|---|
| 绝对位置编码 | 加到 embedding 上 | 可有可无 | 通常较弱 | 固定长度输入、实现简单 |
| 相对位置 bias | 注意力里建模相对距离 | 通常少量或无 | 中等到较好 | 编码器/解码器都可设计 |
| RoPE | 对 $Q,K$ 做旋转变换 | 无 | 通常较好 | 现代表征模型、双向或因果都常见 |
| ALiBi | 对 score 加线性距离偏置 | 无 | 对因果外推友好 | GPT 类自回归模型 |

和 RoPE 比较最容易理解。RoPE 是 Rotary Position Embedding，白话说，**它通过对查询和键做角度旋转，把位置信息写进向量几何关系里**。ALiBi 则不改向量，只改最终打分。

两者差异可以记成：

- RoPE：位置进入 $Q,K$ 表达层
- ALiBi：位置进入 attention score 层

这带来不同边界：

- 如果你做的是 GPT 类生成模型，而且特别在意训练短、测试长，ALiBi 很有吸引力
- 如果你做的是双向理解任务、检索编码器、通用表征模型，RoPE 或相对位置 bias 往往更常见
- 如果输入长度稳定且不长，绝对位置编码仍然足够简单直接

一个新手容易误解的点是：“既然 ALiBi 不加参数，是不是就一定更好？”答案是否定的。**位置方案的选择取决于模型类型、任务目标、长度范围和工程约束**。ALiBi 的强项是因果外推，不是通吃所有架构。
