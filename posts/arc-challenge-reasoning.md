## 核心结论

ARC（Abstraction and Reasoning Corpus，抽象与推理语料）是一个专门测“少样本抽象推理”的基准。它不是让系统看很多标签后做分类，而是让系统只看 2 到 5 个输入输出示例，就推断一个隐藏规则，再把这个规则应用到新输入上。

它的重要性不在于分数本身，而在于它把“是否真的学会了规则”与“是否只是记住了模式”分开了。很多模型在大数据分类上表现很好，但在 ARC 上会暴露出一个问题：见过很多图，不等于会从新题里归纳规则。

先看一个玩具例子。颜色用整数表示，`0` 可理解为空白，`1` 和 `2` 是两种颜色。

| 项目 | 网格 |
|---|---|
| 训练输入 | `[[0,0,1],[0,1,1],[0,0,0]]` |
| 训练输出 | `[[0,0,2],[0,2,2],[0,0,0]]` |
| 测试输入 | `[[0,1,0],[1,1,0],[0,0,0]]` |
| 预期输出 | `[[0,2,0],[2,2,0],[0,0,0]]` |

这里的规则不是“记住这张图”，而是“把属于同一邻接块的 `1` 替换成 `2`”。邻接块的白话解释是：上下左右连在一起的一团格子。测试图虽然长得不一样，但只要满足同样的结构，规则就要迁移过去。

因此，ARC 真正考的是：系统能否从极少示例中抽出几何、计数、对象关系、对称、复制、扩散、替换等规则，并在新输入上泛化。

## 问题定义与边界

一个 ARC 任务通常由 2 到 5 个训练示例和 1 个或多个测试输入组成。每个示例都是一个输入网格和输出网格。网格可以看成整数矩阵，数字范围通常是 0 到 9，每个数字对应一种颜色或符号。

形式化地说，任务可写成：

$$
T=\{(x_i,y_i)\}_{i=1}^n,\quad x_i,y_i\in \mathbb{Z}^{H_i\times W_i}
$$

目标是找到一个变换函数 $f$，使得：

$$
f(x_i)=y_i,\quad i=1,\dots,n
$$

并且对新输入 $x_*$ 也成立：

$$
f(x_*)=?
$$

这里 $\mathbb{Z}^{H_i\times W_i}$ 的意思是一个 $H_i\times W_i$ 的整数网格。白话说，就是“高为 $H_i$、宽为 $W_i$ 的彩色格子图”。

看一个最小边界例子。假设训练对是一个 \(3\times3\) 网格：

- 输入：`1` 表示红色块，`0` 表示空白
- 输出：把所有红色块改成蓝色 `2`

那么测试时即使位置变了、块形状变了，只要规则仍是“把该对象整体换色”，系统就应该做同样的事。

ARC 的边界也要说清楚：

| 边界 | 含义 |
|---|---|
| 不是大规模监督学习 | 每题样本极少，不能靠几万条同类标注稳稳拟合 |
| 不是记忆题库 | 不能依赖看过评估题再调规则 |
| 不保证输入输出同尺寸 | 有些题会扩展、裁剪、拼接网格 |
| 不只考像素统计 | 许多题的关键在“对象”和“关系”，不是颜色频率 |

所以，ARC 不是“看图识别”，而是“看少量例子猜程序”。

## 核心机制与推导

理解 ARC，最直接的方式是把它看成“从示例反推一个小程序”。

如果输入网格是一个矩阵，那么很多 ARC 规则可以写成：

$$
f:\mathbb{Z}^{H\times W}\rightarrow \mathbb{Z}^{H'\times W'}
$$

这个函数 $f$ 往往不是单一步骤，而是由若干操作组合而成，例如：

1. 识别对象
2. 计算对象属性
3. 判断对象之间的关系
4. 选择一个变换
5. 生成输出网格

“对象”的白话解释是：在网格里可以被当成一个整体处理的图形块，比如一条线、一个方块、一组连通单元。

例如一个常见推理链路是：

`示例集 -> 找共同模式 -> 提炼规则 -> 应用到测试输入`

假设训练示例中都出现如下现象：

- 输入中有一个非零色块
- 输出中该色块向右扩展 2 格
- 背景保持不变

此时合理的候选规则不是“右边这两列经常变色”，而是“检测目标对象并做平移或扩张”。这就是抽象层级的差别。前者是像素统计，后者是对象规则。

再看一个稍复杂的玩具例子。输入：

```text
0 3 0
3 3 0
0 0 0
```

输出：

```text
0 4 0
4 4 4
0 4 0
```

如果只看颜色频率，会得到“3 变成 4，而且 4 变多了”。但真正的规则可能是：“找到连通的 `3` 形状，然后沿四邻域扩散一圈并重涂为 `4`。”这类题要求系统显式表示连通性、边界、扩展方向。

ARC 难就难在：满足训练示例的函数 $f$ 往往不止一个。真正有效的系统需要偏向更简单、更结构化、更可迁移的解释。可以把这种偏好理解为一种先验，即系统更愿意相信“对象复制”“轴对称”“最大块保留”这类结构规则，而不是一堆只对当前样本成立的局部例外。

真实工程例子是：当团队做通用推理系统时，会把 ARC 当成“规则发现能力”的回归测试。新模型上线前，不只看最终是否答对，还会看它失败在什么阶段：是没找到对象，还是找到对象但关系推错，还是变换选对了却生成错尺寸。2025 年发布的 H-ARC 提供了大量人类作答轨迹和自然语言描述，正适合做这种误差定位。

## 代码实现

工程上，一个 ARC 求解器通常不是一个单函数，而是一条流水线：

| 模块 | 作用 | 白话解释 |
|---|---|---|
| 任务加载 | 读取 `train` 与 `test` | 把每道题的示例取出来 |
| 对象解析 | 找连通块、颜色、边界框 | 先看图里有哪些“东西” |
| 规则模板匹配 | 尝试替换、平移、镜像、计数等模板 | 猜这题属于哪类规律 |
| 候选生成 | 对测试输入应用规则 | 按猜到的规则生成答案 |
| 验证与排序 | 检查是否解释全部训练对 | 留下最能统一解释示例的规则 |

核心伪代码通常长这样：

```python
for pattern in template_library:
    if match(pattern, train_examples):
        candidate = apply(pattern, test_input)
        if candidate is valid:
            return candidate
```

下面给一个可运行的最小 Python 例子，实现“把所有值为 1 的格子替换为 2”。它不覆盖真实 ARC 的复杂性，但能让新手先看到“从训练对抽规则，再对测试图应用”的最小闭环。

```python
from copy import deepcopy

train_input = [
    [0, 0, 1],
    [0, 1, 1],
    [0, 0, 0],
]

train_output = [
    [0, 0, 2],
    [0, 2, 2],
    [0, 0, 0],
]

test_input = [
    [0, 1, 0],
    [1, 1, 0],
    [0, 0, 0],
]

expected_output = [
    [0, 2, 0],
    [2, 2, 0],
    [0, 0, 0],
]

def infer_replace_rule(inp, out):
    mapping = {}
    for r in range(len(inp)):
        for c in range(len(inp[0])):
            src = inp[r][c]
            dst = out[r][c]
            if src in mapping and mapping[src] != dst:
                raise ValueError("rule is not a simple per-color replacement")
            mapping[src] = dst
    return mapping

def apply_rule(grid, mapping):
    result = deepcopy(grid)
    for r in range(len(grid)):
        for c in range(len(grid[0])):
            result[r][c] = mapping.get(grid[r][c], grid[r][c])
    return result

rule = infer_replace_rule(train_input, train_output)
pred = apply_rule(test_input, rule)

assert rule[1] == 2
assert pred == expected_output
print(pred)
```

真实系统当然不会只做“按颜色替换”。更常见的做法是：

1. 先把网格拆成对象
2. 为对象计算属性：颜色、面积、方向、相对位置
3. 在模板库里搜索规则
4. 选择能统一解释全部训练对的最短程序或最高分规则
5. 把规则应用到测试输入

如果把 ARC 文件结构类比成博客里的索引加载，它也很像“按任务文件名读取，再对每道题单独推理”。不同的是，这里的“slug”对应任务 JSON 文件名，而不是文章 URL。

## 工程权衡与常见坑

ARC 最容易犯的工程错误，不是模型太弱，而是评测流程不干净。

第一类坑是规则泄露。官方仓库明确把 `training` 和 `evaluation` 分开，并要求开发时不要把 evaluation 信息泄漏进算法设计过程。如果团队在训练时反复查看评估题，哪怕没有直接记忆答案，也会把规则分布偷偷学进去。这样得到的分数不能说明泛化能力。

第二类坑是把 ARC 当成像素拟合问题。很多纯统计模型会学到“某颜色出现后，另一颜色概率上升”之类的浅层模式，但一旦测试输入换了布局，就失效了。

| 策略 | 优点 | 常见失败模式 |
|---|---|---|
| 像素频率/端到端统计 | 实现快 | 训练对解释得过去，测试一换形状就崩 |
| 符号搜索 | 规则可解释 | 搜索空间大，速度慢 |
| 模块化规则库 | 工程上可控 | 规则库不全时会卡在覆盖率 |
| 神经+符号混合 | 兼顾感知与搜索 | 系统复杂，调试成本高 |

第三类坑是只看“答没答对”，不看“怎么错”。H-ARC 的价值就在这里。它不只给最终提交，还给人类逐步操作轨迹和文字描述。一个系统如果总是在“先复制输入，再局部修补”这一步出错，而人类普遍会先“找主对象再扩展”，那就说明系统缺的是对象级规则，而不是颜色识别能力。

一个真实工程场景是：团队做 ARC solver 时，训练准确率上升很快，但在陌生题型上始终不稳定。接入 H-ARC 后发现，人类错误往往是“结构大体对，细节偏一格”，而模型错误常常直接破坏对象边界。这说明模型没有形成稳定的对象表示，只是在局部像素上蒙对过一部分题。

## 替代方案与适用边界

如果目标是普通视觉识别，ARC 往往不是合适选择。它样本少、题型离散、强调规则发现，不适合作为通用图像分类基准。

下面把 CIFAR、ARC、H-ARC 放在一起比较：

| 特征 | CIFAR | ARC | H-ARC |
|---|---|---|---|
| 基本单位 | 带类别标签的图片 | 输入输出网格任务 | 人类在 ARC 上的行为数据 |
| 每类样本规模 | 大 | 每题只有少量示例 | 覆盖 800 个公开 ARC 任务的人类尝试 |
| 主要能力 | 统计识别 | 抽象推理与规则泛化 | 人类策略、错误模式、轨迹分析 |
| 输出形式 | 分类标签 | 生成完整输出网格 | 动作序列、提交结果、文字描述 |
| 适合问题 | 识别“这是什么” | 推断“规则是什么” | 分析“人为什么这样解题” |

因此，选择基准时要看目标：

- 如果你要评估卷积网络或视觉分类器，CIFAR 这类数据集更合适。
- 如果你要评估系统能否从极少样本中归纳规则，ARC 更合适。
- 如果你要研究人机差异、对齐推理过程、分析失败机制，H-ARC 更合适。

ARC 也有自己的适用边界。它非常适合测“抽象视觉规则发现”，但不直接覆盖长期记忆、开放世界知识、持续交互学习等能力。任务规模一大、环境一变，单纯依赖手写规则库会很快遇到瓶颈。这也是为什么很多研究开始转向神经+符号混合，或者借助 H-ARC 的行为数据来约束搜索和解释。
