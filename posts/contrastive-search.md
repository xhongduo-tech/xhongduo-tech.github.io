## 核心结论

Contrastive Search 是一种解码策略。解码策略解决的问题不是“模型学到了什么”，而是“模型已经给出下一步候选后，最后到底选哪个 token”。它不再像 Greedy Search 那样每一步都直接拿概率最大的 token，也不像 Beam Search 那样主要围绕累计概率维护多条高分路径，而是先把候选范围限制在前 $k$ 个高概率 token 里，再在这些“本来就比较可信”的候选中加入一项重复惩罚。

这个惩罚不是简单检查词面是否重复，不是看“这个词之前有没有出现过”，而是看隐藏状态是否和历史内容过于相似。隐藏状态可以先理解为“模型内部对一个 token 及其上下文意义的向量表示”。如果某个候选 token 的表示和历史某一段内容非常接近，通常说明它更可能把文本继续推向“重复、绕圈、换个说法复读同一件事”的方向。

它的核心公式可以先压缩成一句话：

$$
x_t=\arg\max_{v\in V_k}\Big[(1-\alpha)\,p_\theta(v\mid x_{<t})-\alpha\max_{1\le j<t}\cos(h_v,h_{x_j})\Big]
$$

其中：

- $V_k$ 是当前时刻概率最高的前 $k$ 个候选 token 集合。
- $p_\theta(v\mid x_{<t})$ 是候选 token $v$ 在当前上下文下的条件概率。
- $h_v$ 是候选 token 的隐藏表示。
- $h_{x_j}$ 是历史第 $j$ 个 token 的隐藏表示。
- $\cos(\cdot,\cdot)$ 是余弦相似度，用来衡量两个向量方向是否接近。
- $\alpha$ 控制“保流畅”和“防重复”之间的权重。

一句话直觉是：不是只选“当前最响亮”的词，而是在前 $k$ 个靠谱候选里，挑一个“既像人话，又别老重复自己”的词。对于开放域生成，这种做法通常比纯贪心和束搜索更容易同时保住连贯性与信息推进。

| 策略 | 选词依据 | 优点 | 常见问题 |
|---|---|---|---|
| Greedy Search | 只看当前最大概率 | 快、稳定、实现简单 | 容易重复、容易落入局部最优 |
| Beam Search | 看多条高概率路径的累计分数 | 句法常较稳，适合受约束任务 | 开放生成里常乏味、保守、模板化 |
| Contrastive Search | 高概率候选 + 相似度惩罚 | 连贯且更少复读，结果稳定 | 需要调 $k,\alpha$，效果依赖表示空间质量 |

再补一个最常见的误解：Contrastive Search 不是“故意选冷门词”，也不是“为了多样性强行反常识”。它的工作方式更接近于在一个本来就高质量的候选池里，剔除那些虽然概率高、但会把文本带回老路的选项。

## 问题定义与边界

它要解决的问题很具体：大语言模型在开放域生成时，经常出现退化。这里的退化不是指语法错误，而是指一种更隐蔽的问题：文本表面依然通顺，但内容开始绕圈、重复、套模板，信息量不再前进。

典型症状包括：

- 连续几句话都在重述同一结论。
- 不断使用“然后”“另外”“总之”这类安全连接词撑长文本。
- 回答看起来礼貌完整，但新增信息很少。
- 句法没问题，语义却像在原地打转。

例如，一个模型回答“为什么接口超时”时，可能连续输出：

> 可能是网络问题。  
> 也可能是服务问题。  
> 同时也可能和网络或服务状态有关。

这三句都不算错，但几乎没有推进。问题不在于单句概率，而在于每一步都只追求局部最安全的 token，最后把整段文本推向了低信息密度区域。

Contrastive Search 的出发点就是：训练分布里，高频、安全、局部看起来合理的 token 往往概率更高。如果解码阶段每一步都只盯着最大概率，那么模型会反复回到这些“安全词”和“安全句式”上。结果就是语言越来越平滑，内容越来越空。

这里要明确它的适用边界。Contrastive Search 主要适合开放域生成，例如：

- 对话系统
- 续写
- 长答案解释
- 写作辅助
- 开放问题问答

这些任务的共同点是：既要求自然、连贯，又要求内容能持续推进，不能只是同义复述。

但它并不天然适合所有任务。以下场景要更谨慎：

- 术语必须严格重复的摘要任务
- 结构固定的抽取任务
- 代码补全中要求精确 token 序列的场景
- 机器翻译这类更强调条件似然最优的任务

原因很简单：有些任务里，重复并不是坏事，而是正确表达的一部分。比如医学术语、法条名称、接口字段名，本来就应该在多句中稳定重复。如果惩罚过强，模型可能为了“别像之前”而压掉本来应该保留的关键词，反而损失准确性。

下面这张表把问题与边界压缩一下：

| 退化症状 | 常见触发策略 | Contrastive Search 如何干预 |
|---|---|---|
| 高频词反复出现 | Greedy 只追局部最高概率 | 从 top-$k$ 中挑“不太像历史”的候选 |
| 语义绕圈 | Beam 偏向安全高概率路径 | 惩罚与历史表示高度相似的候选 |
| 句子通顺但没新信息 | 只优化似然，不管表达冗余 | 在每步解码中加入多样性约束 |
| 输出风格过于模板化 | 解码太保守 | 允许略低概率但更有信息增量的 token 入选 |

一个新手更容易理解的玩具例子是：

“这个问题需要分几步看。然后……然后……然后……”

这里“然后”在局部上并不一定低概率，但它和前文的表达功能过于接近，继续选它大概率不会带来新信息。Contrastive Search 会把这种“局部合理、全局没推进”的候选往下压。

再看一个更工程化的例子。假设你在做客服助手，用户问：

> 为什么接口会超时？

纯 Greedy 很可能给出这种答案：

> 可能是网络问题，也可能是服务问题，也可能是下游问题。

这段话每句都成立，但用户没法据此排查。Contrastive Search 更容易继续生成：

> 先检查网关超时阈值是否小于下游平均响应时间，再确认是否开启了指数退避重试，以及连接池是否被打满。

这里的词不一定每一步都是绝对最大概率，但它们提供了明显的信息增量。

所以它解决的不是“模型会不会说通顺的话”，而是“模型会不会持续说有推进的话”。

## 核心机制与推导

先把符号讲清楚，避免公式看起来像黑箱。

设当前已经生成了前缀 $x_{<t}=(x_1,x_2,\dots,x_{t-1})$，现在要决定第 $t$ 个 token。模型会给出整个词表上的条件分布：

$$
p_\theta(v\mid x_{<t})
$$

这表示“在前文已经确定的条件下，下一个 token 是 $v$ 的概率”。

如果用 Greedy Search，做法很直接：

$$
x_t=\arg\max_{v\in V} p_\theta(v\mid x_{<t})
$$

也就是直接在整个词表里选概率最大的 token。问题是，这个目标只关心“当前一步最像训练数据的是什么”，并不关心“这个选择会不会让文本陷入重复”。

Contrastive Search 先加一个限制：不在整个词表上乱选，而是只看当前概率最高的前 $k$ 个候选：

$$
V_k=\text{TopK}\big(p_\theta(\cdot\mid x_{<t}),k\big)
$$

这一步很关键。它保证了搜索空间仍然落在“高可信候选”内，不会因为追求多样性而去选明显不自然的 token。可以把它理解为先做一道概率门槛过滤，流畅性主要靠这一步保住。

接着，对每个候选 $v\in V_k$，模型拿到它的隐藏表示 $h_v$，再与历史 token 的隐藏表示 $\{h_{x_1},h_{x_2},\dots,h_{x_{t-1}}\}$ 做比较。比较指标是余弦相似度：

$$
\cos(h_v,h_{x_j})=\frac{h_v^\top h_{x_j}}{\|h_v\|\|h_{x_j}\|}
$$

余弦相似度的值通常在 $[-1,1]$ 之间。值越接近 1，说明两个向量方向越接近，也就是在表示空间里越“像”。

Contrastive Search 不取平均相似度，而是取最大相似度：

$$
\max_{1\le j<t}\cos(h_v,h_{x_j})
$$

为什么是最大值，不是平均值？因为退化通常不是“和所有历史都稍微有点像”，而是“和某一段历史特别像”。重复往往由一个局部尖峰主导。若改成平均值，最危险的那种强相似会被其他低相似项冲淡，惩罚就不够敏感。

于是得到每个候选的综合得分：

$$
\text{score}(v)=(1-\alpha)\,p_\theta(v\mid x_{<t})-\alpha\max_{1\le j<t}\cos(h_v,h_{x_j})
$$

最终选择分数最高的候选：

$$
x_t=\arg\max_{v\in V_k}\text{score}(v)
$$

这个式子可以拆成两部分理解：

- 第一项 $(1-\alpha)p_\theta(v\mid x_{<t})$：保留语言模型本来的流畅性偏好。
- 第二项 $\alpha\max_{1\le j<t}\cos(h_v,h_{x_j})$：抑制和历史过于接近的候选。

$\alpha\in[0,1]$ 决定两者谁更重要：

- $\alpha$ 很小：更接近 Greedy，只是轻微抑制重复。
- $\alpha$ 适中：通常能在连贯与多样性之间取得平衡。
- $\alpha$ 很大：会强烈回避相似表达，可能导致不稳或压掉必要术语。

为了让公式更落地，继续用一个数值例子。设：

- $k=4$
- $\alpha=0.6$

当前 top-$4$ 候选如下：

| 候选 | 概率 $p$ | 与历史最大相似度 |
|---|---:|---:|
| A | 0.45 | 0.30 |
| B | 0.42 | 0.05 |
| C | 0.39 | 0.10 |
| D | 0.36 | 0.22 |

逐个计算：

$$
\text{score}(A)=0.4\times 0.45-0.6\times 0.30=0.18-0.18=0
$$

$$
\text{score}(B)=0.4\times 0.42-0.6\times 0.05=0.168-0.03=0.138
$$

$$
\text{score}(C)=0.4\times 0.39-0.6\times 0.10=0.156-0.06=0.096
$$

$$
\text{score}(D)=0.4\times 0.36-0.6\times 0.22=0.144-0.132=0.012
$$

所以最后选 B。

把它整理成表更直观：

| 候选 | 概率 $p$ | 最大相似度 | $(1-\alpha)p$ | $\alpha\cdot sim$ | 总分 | 结论 |
|---|---:|---:|---:|---:|---:|---|
| A | 0.45 | 0.30 | 0.180 | 0.180 | 0.000 | 被重复惩罚抵消 |
| B | 0.42 | 0.05 | 0.168 | 0.030 | 0.138 | 选中 |
| C | 0.39 | 0.10 | 0.156 | 0.060 | 0.096 | 可选，但不如 B |
| D | 0.36 | 0.22 | 0.144 | 0.132 | 0.012 | 概率不低，但太像历史 |

这个例子说明的重点不是“它总会选低概率词”，而是：

1. 它先用 top-$k$ 保证候选本身就比较可信。
2. 然后只在这些高质量候选内部做再排序。
3. 最终选的是“概率还不错，同时又不太重复”的那个。

这也是它和纯采样方法的根本区别。采样是按分布随机抽，Contrastive Search 是在高概率区间内做确定性的对比选择。

再补一个新手常见疑问：这里为什么能用“候选 token 的隐藏状态”来衡量未来是否会重复？因为隐藏状态不是简单的词向量，它携带了当前位置、上下文条件以及模型对当前候选的语义判断。两个 token 在词面上不同，隐藏状态仍然可能很像，例如“因此”和“所以”在某些上下文下就可能位于相近区域。Contrastive Search 正是利用了这种“语义层面的相似”去识别换词复读。

## 代码实现

实现流程可以压成四步：

1. 模型前向，拿到当前 step 的 logits，以及与候选相关的隐藏表示。
2. 取 top-$k$ 候选。
3. 计算每个候选与历史隐藏状态的最大余弦相似度。
4. 用混合分数重新排序，选分数最高者。

先给一个完全可运行的 Python 玩具实现。它不依赖任何深度学习框架，只模拟 Contrastive Search 的核心逻辑：候选概率、候选向量、历史向量、再排序。

```python
from math import sqrt
from typing import Dict, List


def cosine(a: List[float], b: List[float]) -> float:
    if len(a) != len(b):
        raise ValueError("Embedding dimensions must match")

    dot = sum(x * y for x, y in zip(a, b))
    na = sqrt(sum(x * x for x in a))
    nb = sqrt(sum(y * y for y in b))

    if na == 0.0 or nb == 0.0:
        return 0.0

    return dot / (na * nb)


def contrastive_pick(
    candidates: List[Dict],
    history_embeddings: List[List[float]],
    alpha: float = 0.6,
    k: int = 4,
) -> Dict:
    if not 0.0 <= alpha <= 1.0:
        raise ValueError("alpha must be in [0, 1]")
    if k <= 0:
        raise ValueError("k must be positive")
    if not candidates:
        raise ValueError("candidates must not be empty")

    topk = sorted(candidates, key=lambda x: x["prob"], reverse=True)[:k]

    best = None
    scored = []

    for item in topk:
        if history_embeddings:
            max_sim = max(cosine(item["embedding"], h) for h in history_embeddings)
        else:
            max_sim = 0.0

        score = (1.0 - alpha) * item["prob"] - alpha * max_sim

        candidate = {
            "token": item["token"],
            "prob": item["prob"],
            "max_sim": max_sim,
            "score": score,
        }
        scored.append(candidate)

        if best is None or candidate["score"] > best["score"]:
            best = candidate

    return {
        "best": best,
        "topk_scored": scored,
    }


if __name__ == "__main__":
    history = [
        [1.0, 0.0],   # 历史表示 1
        [0.9, 0.1],   # 历史表示 2
    ]

    candidates = [
        {"token": "A", "prob": 0.45, "embedding": [0.95, 0.05]},
        {"token": "B", "prob": 0.42, "embedding": [0.0, 1.0]},
        {"token": "C", "prob": 0.39, "embedding": [0.4, 0.9]},
        {"token": "D", "prob": 0.36, "embedding": [0.8, 0.2]},
    ]

    result = contrastive_pick(candidates, history, alpha=0.6, k=4)

    best = result["best"]
    print("Best candidate:", best)
    print("All scored candidates:")
    for item in result["topk_scored"]:
        print(item)

    assert best["token"] == "B"
    assert round(best["score"], 3) == 0.138
```

这段代码运行后会选出 `B`，结果和前面的表格一致。它有几个刻意保留的细节：

- 检查了 `alpha` 范围和 `k` 的合法性，避免参数错了但静默运行。
- 检查向量维度是否一致，避免余弦计算出错。
- 在没有历史时把 `max_sim` 设为 0，这对应生成初期还没有可比较对象的情况。
- 返回了 `topk_scored`，方便调试每个候选为何被选中或淘汰。

如果你是第一次接触这个算法，建议先在这段玩具代码上做三种实验：

1. 把 `alpha` 从 `0.1` 调到 `0.8`，观察结果如何变化。
2. 把候选 B 的 `prob` 再调低一点，看它什么时候不再被选中。
3. 把 A 的 embedding 改得更不像历史，看惩罚如何减轻。

这样能很快建立直觉：Contrastive Search 不是单纯偏爱低相似度，而是在“概率收益”和“重复代价”之间找平衡点。

如果放进真实模型，伪代码通常是这样：

```python
def contrastive_search_step(model, input_ids, history_hidden, alpha, k):
    logits, candidate_hidden = model.forward_for_next_token(input_ids)
    probs = softmax(logits)
    topk_ids, topk_probs = topk(probs, k)

    scores = []
    for token_id, prob in zip(topk_ids, topk_probs):
        h_v = candidate_hidden[token_id]
        if history_hidden:
            max_sim = max(cosine_similarity(h_v, h) for h in history_hidden)
        else:
            max_sim = 0.0

        score = (1 - alpha) * prob - alpha * max_sim
        scores.append((token_id, score, h_v))

    next_token_id, _, next_hidden = max(scores, key=lambda x: x[1])
    history_hidden.append(next_hidden)

    return next_token_id, history_hidden
```

不过真实系统里还有几个实现细节，文章里不能省略，否则“能看懂公式，但落不了地”。

第一，`prob` 在工程上通常不是直接拿 softmax 后的概率做标量融合，而是遵循具体实现选择 logits、log-prob 或规范化后的分数。不同实现的数值尺度不同，所以 $\alpha$ 的可用范围也会跟着变。论文或教程里的公式写成概率是为了说明思想，工程实现更关心“两个量的尺度是否可比较”。

第二，`candidate_hidden` 并不总是“词表中每个 token 的固定向量”。在实际模型里，它常常依赖当前上下文，是本轮前向中与候选相关的表示。这也是 Contrastive Search 与简单词面去重不同的地方，它惩罚的是上下文条件下的表示相似，而不是词典里静态 embedding 的相似。

第三，最重要的是缓存。因为生成是逐步进行的，如果每一步都重新把全部历史 token 再跑一遍隐藏状态，再与全部 top-$k$ 候选逐个比较，成本会快速变高。实际做法通常是：

- 每生成一个新 token，就把它对应的隐藏状态追加到缓存。
- 下一步直接复用这个缓存，而不是重算整段历史。
- 如果上下文很长，再结合窗口截断，只对最近一段历史做比较。

下面这张表把核心变量与工程含义对齐一下：

| 变量 | 含义 | 常见取值 | 作用 |
|---|---|---|---|
| `alpha` | 重复惩罚权重 | 0.1 到 0.8 | 越大越抑制重复 |
| `k` | top-$k$ 候选数 | 4、8、16 | 越大探索越多，计算也更贵 |
| `history_hidden` | 历史 token 的隐藏表示缓存 | 长度随生成步数增长 | 避免每步重复算历史 |
| `candidate_hidden` | 当前候选 token 的表示 | 由模型前向给出 | 用来算相似度 |
| `max_sim` | 与历史的最大余弦相似度 | $[-1,1]$ | 直接决定惩罚项 |

再给一个更实际的产品例子。假设你在做一个写作辅助系统，前端实时显示模型续写：

- 如果用 Greedy，输出通常稳定，但很容易出现“首先、其次、最后”这类模板词串。
- 如果用 Nucleus Sampling，多样性强，但同一提示词每次结果可能差别很大，调试困难。
- 如果用 Contrastive Search，往往能保持较强稳定性，同时减少复读，更适合线上回归测试和问题复现。

所以从实现角度看，它并不复杂，复杂的是把“表示质量、参数尺度、缓存成本”一起控制住。

## 工程权衡与常见坑

Contrastive Search 不是“默认比其他解码都好”。它有效的前提之一，是隐藏表示空间本身要有足够的区分度。表示空间可以先理解为：模型把不同 token、不同上下文映射到向量后的几何分布。如果这个空间里很多向量都挤在相近方向上，那么余弦相似度就很难准确区分“合理延续”和“重复复读”。

这也是为什么同样的公式，在不同模型上效果差异可能很大：

- 在表示能力更强的模型上，相似度更容易对准“语义重复”。
- 在较小或较旧的模型上，相似度有时会把正常推进也误判成重复。

这时不能简单说“算法失效”，更准确的说法是：表示空间质量不足以支撑这种对比式再排序。

下面把常见坑直接列清楚：

| 坑 | 为什么发生 | 解决策略 |
|---|---|---|
| $\alpha$ 太大，生成变怪 | 过度强调“不像过去” | 从小值开始调，如 0.2、0.4 |
| $\alpha$ 太小，几乎没效果 | 退化惩罚太弱，近似 Greedy | 逐步上调并观察重复率 |
| 用平均相似度替代最大相似度 | 强重复被稀释 | 保留 `max` 形式 |
| 小模型表示空间太挤 | 相似度不可靠 | 调低 $\alpha$，缩短历史窗口 |
| `k` 太小 | 候选集合过窄，近似 Greedy | 适度增大到 8 或 16 |
| `k` 太大 | 计算开销上升，低质候选变多 | 结合延迟预算调参 |
| 全历史都参与比较 | 长文本下成本高，早期上下文干扰大 | 使用滑动窗口或截断历史 |
| 在高术语密度任务直接套用 | 必要重复也会被惩罚 | 先做任务验证，再决定是否启用 |

这里尤其要讲两个新手最容易踩的误区。

第一个误区：把 Contrastive Search 理解成“尽量选陌生词”。不对。它从头到尾都不是为了追求花哨和猎奇，而是为了减少无效重复。它仍然只在高概率候选里工作。如果你把 $\alpha$ 拉得过高，模型确实可能开始回避本来应该重复的关键词，比如函数名、术语、实体名，那就不是算法变聪明了，而是约束过强导致准确性下降。

第二个误区：把最大相似度换成平均相似度，觉得这样更平滑。这个改法在直觉上很“温和”，但通常会直接削弱算法效果。因为真正危险的重复不是“整体都略像”，而是“某一段特别像”。平均值会把这种尖峰洗平，抑制就不够锋利。

还要补一个工程上很现实的问题：延迟。Contrastive Search 每一步都要对 top-$k$ 候选做额外相似度比较，复杂度大于纯 Greedy。在长文本生成里，如果历史长度记为 $T$，候选数为 $k$，那么额外比较大致是 $O(kT)$。虽然通过缓存能明显降低重算成本，但它仍然比“只拿 argmax”贵。在在线系统中，这意味着：

- 低延迟要求极高的场景未必适合直接使用。
- 生成长度越长，额外代价越明显。
- 参数搜索时不能只看质量，也要看吞吐和时延。

一个实用调参顺序通常是：

1. 先固定一个较小的 $k$，例如 4 或 8。
2. 再从较小的 $\alpha$ 开始，例如 0.2 或 0.3。
3. 观察三类指标：重复率、人工可读性、推理延迟。
4. 如果重复仍多，先调大 $\alpha$；如果输出开始发散或术语不稳，再回调。
5. 只有当候选池明显太窄时，再增大 $k$。

如果你面对的是长回答、解释类任务，这个顺序通常比盲目搜参数更稳。

## 替代方案与适用边界

最常见的替代方案是采样类方法，例如 Top-$k$ Sampling 和 Nucleus Sampling。采样的核心思想是：不是每次都拿最高概率 token，而是在一个截断后的分布里按概率随机抽样。这样天然会带来多样性，但代价是稳定性下降。

Contrastive Search 与它们的差别，不在于“谁更高级”，而在于优化目标不同。

先看一个总览：

| 策略 | 是否确定性 | 连贯性 | 多样性 | 典型适用场景 |
|---|---|---|---|---|
| Beam Search | 是 | 高 | 低 | 机器翻译、受约束生成 |
| Top-$k$ Sampling | 否 | 中 | 中到高 | 一般开放生成 |
| Nucleus Sampling | 否 | 中 | 高 | 创意写作、风格化生成 |
| Contrastive Search | 是 | 高 | 中到高 | 对话、解释、写作辅助 |

这里“确定性”很重要。确定性指的是：同样的输入、同样的参数、同样的模型，输出基本固定。这一点在工程上比论文里看起来更重要，因为它直接影响：

- 问题复现
- A/B 对比
- 回归测试
- 线上故障排查

如果一个客服模型今天答得很好、明天同输入又跑偏，而你用的是采样策略，那么很难判断问题来自模型变化还是随机性本身。Contrastive Search 的确定性在这类场景里是明显优势。

但它也不是所有情况下都优于采样。

如果你要的是：

- 稳定
- 不复读
- 表达自然
- 信息能持续推进

那么 Contrastive Search 通常很合适。

如果你要的是：

- 更有创意
- 更发散
- 风格变化更丰富
- 允许偶尔冒险甚至跑偏

那么 Nucleus Sampling 往往更合适。

如果你的任务高度受约束，例如：

- 机器翻译
- 固定格式抽取
- 结构化表单生成
- 某些信息必须原样保留的摘要

那 Beam Search 或任务定制解码往往更稳。因为这些任务的重点不是“避免说得像之前”，而是“尽量接近目标分布中的最优序列”。

可以把几种方法的直觉差别记成一句话：

- Greedy Search：最稳，但最容易保守和复读。
- Nucleus Sampling：最灵活，但结果波动更大。
- Contrastive Search：在稳定性和抗退化之间取中间平衡。

再给一个产品选择例子：

- 小说续写、广告文案、诗歌生成：采样类方法通常更合适，因为任务允许甚至鼓励意外性。
- 问答助手、企业知识库解释、写作辅助：Contrastive Search 更常见，因为用户更在意清楚、稳定、不绕圈。
- 翻译、代码自动补全中的精确场景：往往要谨慎使用 Contrastive Search，因为必要重复和格式精确性比“避免相似”更重要。

所以实际判断标准不是看哪篇论文更热，而是看你的产品目标到底是什么。你要的是创造性，还是稳定的信息推进；要的是每次都不同，还是每次都能复现。Contrastive Search 的价值，主要就在后者。
