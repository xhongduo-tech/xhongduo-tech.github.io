## 核心结论

双塔模型是一种召回模型：它把`用户`和`物品`分别编码成同一向量空间中的向量，再用点积或余弦相似度计算相关性，最后取 TopK 候选。

新手版解释：先把“用户画像”压缩成一个向量，再把“物品特征”压缩成一个向量，最后看两个向量像不像。像的排前面，不像的排后面。

业务版解释：短视频首页召回时，用户侧信息实时计算，物品侧向量离线预计算，然后在向量库里找最相近的几百到几千条候选，再交给排序模型。

双塔模型的核心公式是：

$$
u = f_u(x_u), \quad v = f_i(x_i)
$$

其中，$x_u$ 是用户特征，$x_i$ 是物品特征，$f_u$ 是用户塔，$f_i$ 是物品塔。`塔`是一个白话说法，指一组把输入特征变成向量的神经网络或编码函数。

相似度通常写成：

$$
s(u,v)=u^T v
$$

或：

$$
s(u,v)=\cos(u,v)
$$

结构可以简化成：

```text
用户特征 xu ──> 用户塔 fu ──> 用户向量 u ┐
                                             ├─> 相似度计算 ──> TopK 检索
物品特征 xi ──> 物品塔 fi ──> 物品向量 v ┘
```

它的价值不在于“建模最复杂”，而在于“检索最可扩展”。当候选物品有百万、千万甚至更多时，系统不能把每个用户和每个物品都送进复杂模型逐个判断。双塔把物品向量提前算好，线上只需要算一个用户向量，再用 ANN 近似近邻搜索快速找相似物品。

ANN 是 Approximate Nearest Neighbor，中文通常叫近似最近邻搜索。它不是精确遍历所有向量，而是用索引结构快速找到“足够接近”的向量，换取大规模检索速度。

## 问题定义与边界

双塔模型解决的问题是：从海量候选物品中快速找出一批相关候选。它主要服务召回阶段，不负责最终排序。

推荐系统常分为召回、粗排、精排、重排。`召回`是第一步，目标是从巨大候选池里先筛出几百到几千个可能相关的物品；`精排`是后面的步骤，目标是对少量候选做更精细的排序。

问题定义表：

| 任务 | 双塔是否适合 | 原因 |
|---|---:|---|
| 大规模召回 | 是 | 可以用 ANN 快速检索 |
| 精排 | 否 | 需要复杂特征交互 |
| 实时推荐召回 | 是 | 用户塔可在线计算 |
| 强交互匹配 | 一般不够 | 交互信息在检索前被压缩 |

新手版解释：像在图书馆找书。先把“你想要什么”变成一个查询向量，再把“每本书的内容”变成向量，先找最接近的书架位置，再逐本细看。双塔负责“先找接近的书”，不是负责“逐页判断哪本书最适合”。

双塔模型有三个关键边界：

| 约束 | 含义 | 工程影响 |
|---|---|---|
| 用户塔只看用户侧特征 | 用户塔输入用户历史、画像、上下文 | 线上可实时计算用户向量 |
| 物品塔只看物品侧特征 | 物品塔输入标题、类目、作者、内容 embedding | 物品向量可以离线预计算 |
| 检索阶段不做交叉网络计算 | 用户和物品只通过相似度匹配 | 检索快，但细粒度交互弱 |

`交叉特征`是指用户特征和物品特征组合后产生的信息。例如“这个用户最近连续看篮球视频”和“这个视频是某个球队夺冠集锦”之间的匹配，不只是用户向量和物品向量各自能完整表达的内容。

边界示例：如果任务需要精细理解“用户当前点击了哪个商品、候选商品和当前商品之间有什么互补关系、价格区间是否匹配”，单纯双塔往往不够。因为它在检索前已经把用户和物品分别压缩成向量，很多细粒度组合信息没有显式建模。

## 核心机制与推导

双塔的核心是“表示学习 + 相似度检索”。`表示学习`是把原始特征变成向量的过程，目标是让向量保留对任务有用的信息。训练时，模型要让正样本更近，让负样本更远。

正样本是用户真实点击、观看、购买、收藏过的物品。负样本是用户没有点击、跳过、曝光未点击，或从候选池中采样出来的物品。

示意图：

```text
训练前：
用户 u      正样本 v+       负样本 v-
  ●             ●              ●

训练目标：
用户 u  <──拉近──  正样本 v+
用户 u  ──推远──>  负样本 v-

训练后：
用户 u  ●● v+                         ● v-
```

最基础的打分函数是点积：

$$
s(u,v)=u^T v
$$

余弦相似度是：

$$
\cos(u,v)=\frac{u^T v}{\|u\|_2\|v\|_2}
$$

其中，$\|u\|_2$ 表示向量的 L2 范数，也就是向量长度。如果用户向量和物品向量都做 L2 归一化，即：

$$
\|u\|_2 = 1, \quad \|v\|_2 = 1
$$

那么：

$$
\cos(u,v)=u^T v
$$

也就是说，归一化之后，点积和余弦相似度等价。这个结论很重要，因为很多向量检索系统对点积或 L2 距离优化更成熟。先归一化，再用点积检索，可以让训练目标和检索实现更统一。

玩具例子：

$$
u=(0.6,0.8), \quad v_1=(0.6,0.8), \quad v_2=(0.0,1.0)
$$

三个向量都已经归一化。点积为：

$$
s(u,v_1)=0.6 \times 0.6 + 0.8 \times 0.8 = 1.00
$$

$$
s(u,v_2)=0.6 \times 0.0 + 0.8 \times 1.0 = 0.80
$$

所以 `v1` 排在 `v2` 前。这个例子说明：双塔不是直接判断“喜欢或不喜欢”，而是把候选映射到向量空间后，按相似度排序。

常见训练目标是对比损失：

$$
L = -\log \frac{e^{s(u,v^+)}}{e^{s(u,v^+)} + \sum e^{s(u,v^-)}}
$$

这个公式的意思是：让正样本 $v^+$ 的得分在所有候选中占比更高。负样本 $v^-$ 得分越高，损失越大，模型就会被迫把它推远。

训练和推理的差异：

| 阶段 | 计算内容 | 目的 |
|---|---|---|
| 训练 | 用户向量、正样本向量、负样本向量、对比损失 | 学表达 |
| 推理 | 用户向量 + 向量库近邻搜索 | 找候选 |

真实工程例子：短视频首页召回中，用户塔输入最近观看序列、停留时长、点赞、关注、设备、地域、时间段等特征。物品塔输入视频标题、作者、类目、标签、文本 embedding、图像 embedding、热视频统计特征等。训练时用“曝光后长观看”作为强正样本，用曝光未点击、短停留、同类热门未点击视频作为负样本。上线时，物品向量离线写入 Faiss 或其他向量库，用户请求到达后实时生成用户向量，再检索 TopK。

## 代码实现

实现上通常分成三步：编码用户、编码物品、向量检索。

伪代码如下：

```python
# 用户塔
user_vec = user_encoder(user_features)

# 物品塔
item_vecs = item_encoder(all_item_features)  # offline

# 召回
scores = ann_index.search(user_vec, top_k=200)
candidates = [item_id for item_id, score in scores]
```

新手版解释：先把所有商品“拍照存档”，用户来了以后现场做一张“兴趣照片”，然后在库存里快速找最像的那批。

离线和在线链路可以画成：

```text
离线链路：
物品库 ──> 物品特征抽取 ──> 物品塔 ──> 物品向量 ──> ANN 索引

在线链路：
用户请求 ──> 用户特征抽取 ──> 用户塔 ──> 用户向量 ──> ANN 检索 ──> 候选列表
```

下面是一个可运行的最小 Python 例子。它不用深度学习框架，只演示双塔召回的核心：向量归一化、点积打分、TopK 返回。

```python
import numpy as np

def l2_normalize(x):
    x = np.asarray(x, dtype=np.float64)
    norm = np.linalg.norm(x)
    if norm == 0:
        return x
    return x / norm

def user_encoder(features):
    # 玩具用户塔：把两个兴趣强度映射成用户向量
    return l2_normalize([features["sports"], features["music"]])

def item_encoder(features):
    # 玩具物品塔：把物品的体育/音乐属性映射成物品向量
    return l2_normalize([features["sports"], features["music"]])

def top_k(user_vec, item_vecs, k):
    scores = []
    for item_id, item_vec in item_vecs.items():
        score = float(np.dot(user_vec, item_vec))
        scores.append((item_id, score))
    return sorted(scores, key=lambda x: x[1], reverse=True)[:k]

user = {"sports": 0.6, "music": 0.8}
items = {
    "basketball_mix": {"sports": 0.6, "music": 0.8},
    "pure_music": {"sports": 0.0, "music": 1.0},
    "football_news": {"sports": 1.0, "music": 0.0},
}

u = user_encoder(user)
item_vecs = {item_id: item_encoder(feat) for item_id, feat in items.items()}
result = top_k(u, item_vecs, k=2)

assert result[0][0] == "basketball_mix"
assert round(result[0][1], 2) == 1.00
assert round(result[1][1], 2) == 0.80

print(result)
```

训练时会变成类似流程：

```python
for batch in dataloader:
    u = user_encoder(batch.user_features)
    v_pos = item_encoder(batch.pos_item_features)
    v_neg = item_encoder(batch.neg_item_features)

    loss = contrastive_loss(u, v_pos, v_neg)
    loss.backward()
    optimizer.step()
```

工程里不会用 Python 循环逐个算所有物品分数。物品向量会批量构建索引，ANN 系统负责检索。Faiss 是常见选择，它提供向量索引、近邻搜索、GPU 加速等能力。实际系统还会加入过滤条件，例如地区、年龄分层、内容安全、已看去重、库存状态等。

## 工程权衡与常见坑

双塔的工程难点不在模型结构，而在训练目标、负采样、向量更新、检索一致性。

常见坑如下：

| 问题 | 现象 | 规避方式 |
|---|---|---|
| 向量范数失控 | 点积被大范数投机 | 归一化或加范数约束 |
| 负采样太简单 | 召回学不到细粒度区分 | 混合 hard negatives / in-batch negatives |
| 物品向量更新慢 | 线上分布漂移 | 增量重建、分层索引 |
| 训练指标和召回指标不一致 | AUC 高但 Recall@K 低 | 同时看 Recall@K、NDCG@K、CTR |
| 单向量表达不足 | 长序列、多兴趣用户效果差 | 多兴趣塔、兴趣路由 |

`hard negative` 是困难负样本，指看起来和正样本很像、但用户没有选择的物品。它比随机负样本更能训练模型的区分能力。`in-batch negative` 是把同一个 batch 中其他用户的正样本当作当前用户的负样本，优点是计算效率高。

新手版解释：模型看起来能把正确物品排前面，但如果负样本太简单，它可能只学会“区分明显不相关的东西”。例如用户喜欢篮球视频，随机负样本全是美妆视频，模型很容易学会区分；但上线时真正难的是区分“篮球集锦”“篮球教学”“篮球八卦”“篮球游戏解说”。

训练目标和检索目标也可能不一致：

| 对比项 | 训练时关注 | 检索时关注 |
|---|---|---|
| 样本范围 | 一个 batch 或采样候选 | 全量物品库 |
| 优化目标 | 损失下降、AUC 提升 | Recall@K、NDCG@K、线上 CTR |
| 负样本 | 人工采样或 batch 内构造 | 真实候选池中的所有竞争物品 |
| 更新频率 | 训练周期决定 | 索引构建和发布频率决定 |

线上/线下链路可以简化成：

```text
训练数据 ──> 样本构造 ──> 双塔训练 ──> 模型发布
                                      │
                                      ├─> 离线生成物品向量 ──> 构建 ANN 索引
                                      │
用户请求 ──> 实时用户特征 ──> 用户塔 ──> ANN 检索 ──> 召回候选
```

真实工程例子：短视频首页里，如果物品向量一天才更新一次，但热点视频的内容分布小时级变化，检索结果会滞后。新发布的视频可能因为没有及时进入索引而召回不足，旧热门视频可能继续占据候选。解决方式通常包括小时级增量索引、热门内容单独通道、实时特征补充、索引灰度发布。

另一个常见问题是向量范数投机。如果直接用点积，模型可能通过增大向量长度来提高分数，而不是学习更好的方向。归一化可以让模型主要比较方向相似度，减少这种投机。也可以在训练中加入温度参数、范数正则或向量裁剪。

## 替代方案与适用边界

双塔不是万能方案。它适合快速筛候选，不适合承担所有推荐判断。

新手版解释：如果你要的是“快速筛一批候选”，双塔够用；如果你要的是“认真判断这个候选到底是不是最优”，需要更复杂的模型。

替代方案对比如下：

| 方案 | 优点 | 缺点 | 适用阶段 |
|---|---|---|---|
| 双塔模型 | 快、可扩展、易 ANN | 交互弱 | 召回 |
| 交叉编码器 | 交互强、表达细 | 计算重 | 排序 |
| 多兴趣模型 | 能表示多个兴趣点 | 架构更复杂 | 长序列召回 |
| 对比学习预训练双塔 | 表示更稳健 | 训练流程更复杂 | 冷启动/泛化 |

`交叉编码器`是把用户和物品特征一起输入模型，让模型直接学习它们之间的交互。它表达能力更强，但不能提前预计算所有物品向量，因此很难用于超大规模召回。

`多兴趣模型`是为一个用户生成多个兴趣向量，而不是只生成一个用户向量。例如一个用户同时喜欢足球、摄影、编程，单个向量可能把这些兴趣混在一起，多兴趣表示可以分别召回不同兴趣方向的候选。

双塔架构的演进通常有三条线：

| 演进方向 | 解决的问题 | 典型做法 |
|---|---|---|
| 特征交互增强 | 双塔交互弱 | 在塔内加入序列模型、注意力、特征组合 |
| 对比学习预训练 | 数据稀疏、冷启动 | 用行为序列或内容增强学习稳定表示 |
| 多兴趣建模 | 单向量表达不足 | 生成多个用户兴趣向量分别检索 |

边界判断可以很直接：

| 场景 | 推荐选择 |
|---|---|
| 候选规模很大，需要毫秒级召回 | 优先双塔 |
| 候选已经很少，需要精细判断 | 使用交叉模型或精排模型 |
| 用户兴趣明显多峰 | 考虑多兴趣表示 |
| 新物品、新用户多，行为数据稀疏 | 考虑内容特征和对比学习预训练 |
| 业务强依赖上下文组合 | 双塔召回后接更强排序模型 |

对比示例：短视频首页候选生成适合双塔，因为系统需要从海量视频里快速找一批相关内容；商品详情页的“相似商品排序”可能需要更强交叉模型，因为当前商品、价格、品牌、库存、促销、用户偏好之间有复杂组合关系。

最终工程形态通常不是“只用双塔”，而是“双塔召回 + 多路召回 + 排序模型”。双塔负责提供高覆盖、高效率的候选来源，后面的排序模型负责更精细的业务目标优化。
