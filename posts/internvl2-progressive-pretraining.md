## 核心结论

InternVL 2.0 的核心不是“把图片直接丢给大语言模型”，而是先把图像压成一串可计算的视觉 token，再把这串 token 映射到语言模型能理解的向量空间，最后按文本生成的方式输出答案。这里的 token 可以先理解成“模型内部处理信息的最小块”，和文本里的词片段作用类似，只是来源从文字变成了图像。

它的架构可以写成一个很短的式子：

$$
Z = P(f_V(I))
$$

其中，$I$ 是输入图像，$f_V$ 是视觉编码器 InternViT，作用是“把图片编码成视觉特征”；$P$ 是一个两层 MLP projector，作用是“把视觉特征翻译到语言模型的向量空间”；$Z$ 是送入 LLM 的视觉上下文。

对新手来说，可以把它理解成一条明确的数据管线：

```text
image -> InternViT -> visual tokens -> MLP projector -> text-space embeddings -> LLM.generate()
```

这和“直接喂图”的差别很大：

| 方案 | 图像进入模型的方式 | 是否有专门视觉编码阶段 | 是否便于高分辨率扩展 | 适合的任务 |
| --- | --- | --- | --- | --- |
| 直接喂图的直觉想象 | 把像素当成普通输入 | 否 | 差 | 不成立，工程上基本不可用 |
| InternVL 2.0 | 先编码，再投影，再交给 LLM | 是 | 强 | 文档问答、表格解析、图文问答 |

更重要的是，InternVL 2.0 的优势不只来自模型参数量，而来自三件事同时成立：第一，`ViT-MLP-LLM` 融合接口足够简单；第二，训练按阶段推进，而不是一次性全量乱训；第三，输入图像采用动态分辨率策略，不把整页文档粗暴缩成一张低清图。

## 问题定义与边界

InternVL 2.0 要解决的问题，不是“识别图片里有什么物体”这么简单，而是更接近“让模型同时理解图像内容、版面结构、细粒度文字和自然语言指令”。版面结构可以先理解成“文字、表格、标题、图例在页面上的空间关系”，这正是文档、截图、报表任务的难点。

一个真实工程例子是财务报表截图。传统 OCR 的主要目标是抽取字符，多数时候只能告诉你“这一格里写了什么字”。但如果问题变成“2024 年 Q3 的净利润是否低于 Q2，原因可能来自哪个费用项”，系统不仅要读数字，还要知道行列对齐关系、表头归属、注释位置和上下文语义。这个时候，纯 OCR 往往不够，多模态模型才是主力。

这里必须先划清版本边界，避免把后续模型配置误记到 2.0：

| 版本 | 架构接口 | 300M 视觉编码器常见配对 | 是否出现 72B 语言模型配对 | 重点变化 |
| --- | --- | --- | --- | --- |
| InternVL 2.0 | `ViT-MLP-LLM` | `Qwen2-0.5B`、`internlm2-1.8B`、`Phi-3-mini`、`internlm2.5-7B` | 否 | 动态分辨率、两阶段训练 |
| InternVL 2.5 | `ViT-MLP-LLM` | `Qwen2.5-0.5B/3B` 等 | 72B 出现在 6B ViT 配对中 | 渐进式扩展、数据质量治理 |
| InternVL 3.0 | `ViT-MLP-LLM` 的延续 | 300M ViT 的新版本继续使用 | 更大规模模型更成熟 | 原生多模态预训练和更强推理 |

所以，“InternViT-300M 与 Qwen2-72B 融合”这件事，不应被表述为 InternVL 2.0 的官方标准配置。更准确的说法是：这种大语言模型配对主要出现在后续 2.5/3.0 系列，但接口设计延续了同一条 `ViT-MLP-LLM` 主线。

适用任务可以明确列出来：

| 任务 | 是否适合 InternVL 2.0 | 原因 |
| --- | --- | --- |
| 文档问答 | 适合 | 既要读字，也要看版面 |
| 票据识别 | 适合 | 字段位置和键值关系重要 |
| 表格解析 | 适合 | 行列结构不能丢 |
| 长图理解 | 适合 | 需要动态分辨率保细节 |
| 极端模糊小图 | 不适合 | 输入信息本身就丢失 |
| 只做字符抽取 | 通常不必用 | OCR 更便宜直接 |

## 核心机制与推导

InternVL 2.0 的核心机制可以拆成四步：切图、编码、投影、生成。

第一步是动态分辨率切图。它不是把任意图片强行缩放到固定尺寸，而是按长宽比把图片切成若干个 `448×448` 的 tile。tile 可以先理解成“按块切出来的小图”。这样做的原因很直接：文档场景里的信息密度不均匀，缩整页会让小字和细表格一起糊掉。

第二步是视觉编码。每个 tile 进入 InternViT，得到一组视觉特征。官方又通过 `pixel unshuffle` 把 token 数进一步压缩。这个操作可以先理解成“把局部空间信息重新排布，用更少 token 表示同一块图像”。

因此，每个 `448×448` tile 最终大约对应 256 个视觉 token。设图片被切成 $n$ 个 tile，则视觉序列长度近似为：

$$
L_v = 256n
$$

这条式子很重要，因为它直接告诉你高分辨率输入的成本。

玩具例子：一张 `896×448` 的图片，如果按宽度切成 2 个 tile，那么视觉 token 数大约就是：

$$
L_v = 256 \times 2 = 512
$$

再看一个更接近实际部署的例子。假设测试时允许最多 40 个 tile，那么视觉 token 数大约是：

$$
L_v = 256 \times 40 = 10240
$$

这意味着模型能看更细的 4K 级图像，但也会大量占用上下文窗口。上下文窗口可以先理解成“模型一次能同时处理的总输入长度预算”。视觉 token 越多，给文本 prompt 和输出答案留下的空间就越少。

第三步是 projector 融合。InternVL 2.0 不让语言模型直接理解像素，而是让两层 MLP 把视觉特征映射到文本词向量空间。这样做的工程价值很大：视觉模块和语言模块可以相对解耦，换不同 LLM 时不必重做整套视觉理解。

第四步是自回归生成。自回归可以先理解成“模型每次预测下一个 token，并依赖前面已经生成的内容”。公式写成：

$$
p(y|I,q)=\prod_t p(y_t|y_{<t}, Z, q)
$$

这里 $q$ 是用户问题，$Z$ 是视觉上下文，$y_t$ 是第 $t$ 个输出 token。它表示：模型并不是“先看完图得到一个固定答案”，而是在生成每个词时都同时参考问题、视觉信息和此前生成内容。

把整个过程压成表格就是：

| 阶段 | 输入 | 输出 | 作用 |
| --- | --- | --- | --- |
| 切图 | 原始图片 | 多个 `448×448` tile | 保留长图和文档细节 |
| 编码 | tile | 视觉特征 | 提取图像语义 |
| 投影 | 视觉特征 | 语言空间向量 | 让 LLM 能消费视觉信息 |
| 生成 | 视觉向量 + 文本问题 | 文本答案 | 按语言模型方式回答 |

训练策略也要分开说清。若按 2.0 官方表述，它主要是两阶段：先训 MLP 对齐，再联合训练全模型。若按你给定的“三阶段渐进式”口径来理解，更准确的归纳是：先做视觉-语言对齐，再做多任务指令微调，最后用高质量数据把行为稳定下来。这个“三阶段”是对公开训练流程的工程化总结，不应误解成 2.0 官方逐字写出的独立三段命名。

## 代码实现

工程上，InternVL 2.0 不是一个黑盒 API，而是一条明确的数据处理流水线。下面先给一段最小可运行的 Python，模拟 tile 数和视觉 token 数之间的关系：

```python
from math import ceil

def estimate_tiles(width: int, height: int, tile_size: int = 448) -> int:
    cols = ceil(width / tile_size)
    rows = ceil(height / tile_size)
    return cols * rows

def estimate_visual_tokens(width: int, height: int, tile_size: int = 448, tokens_per_tile: int = 256) -> int:
    return estimate_tiles(width, height, tile_size) * tokens_per_tile

# 玩具例子：896x448 -> 2 个 tile -> 512 个视觉 token
assert estimate_tiles(896, 448) == 2
assert estimate_visual_tokens(896, 448) == 512

# 真实工程上常见的宽长文档图：1792x1344 -> 12 个 tile
assert estimate_tiles(1792, 1344) == 12
assert estimate_visual_tokens(1792, 1344) == 3072
```

如果把流程写成伪代码，基本就是：

```text
image
-> split_tiles(image, image_size=448, tile_max=12)
-> vision_encoder(tiles)
-> projector(visual_features)
-> concat_with_text(question)
-> LLM.generate(max_new_tokens=512)
```

一个真实工程例子是“PDF 页面问答”。输入不是一整本 PDF，而是先把单页渲染成图片，再交给动态切图逻辑。原因很简单：你真正想保留的是页面结构，而不是字符流。

| 环节 | 输入 | 输出 | 常见参数 |
| --- | --- | --- | --- |
| 页面渲染 | PDF 页 | RGB 图像 | `dpi` |
| 动态切图 | 图像 | tile 列表 | `image_size=448`, `tile_max` |
| 视觉编码 | tile 张量 | 视觉特征 | `bf16/fp16` |
| 投影融合 | 视觉特征 | LLM 输入向量 | projector hidden size |
| 文本生成 | 问题 + 视觉向量 | 答案 | `max_new_tokens` |

推理参数里最值得先盯住的通常只有三个：

| 参数 | 含义 | 调大后的效果 | 风险 |
| --- | --- | --- | --- |
| `tile_max` | 最多允许多少块 tile | 看得更细 | 更慢、更占显存 |
| `image_size` | 每块 tile 的边长 | 改变切图粒度 | 与训练设定偏离会退化 |
| `max_new_tokens` | 最长输出长度 | 答案更完整 | 延迟升高、啰嗦概率上升 |

## 工程权衡与常见坑

InternVL 2.0 最核心的工程权衡，是“分辨率收益”和“上下文成本”绑定在一起。图越清楚，视觉 token 越多；视觉 token 越多，显存、延迟和上下文压力越大。这不是实现细节，而是系统设计的主约束。

| 选择 | 收益 | 代价 |
| --- | --- | --- |
| 高分辨率、多 tile | 小字、表格、版面更稳 | token 成本高，推理慢 |
| 低分辨率、少 tile | 省显存、省上下文 | 细节丢失，文档任务易错 |
| 数据覆盖更广 | 泛化更强 | 噪声样本更多 |
| 严格数据过滤 | 输出更稳定 | 数据构建成本更高 |

一个常见坑是把整页文档先压成一张低分辨率图。这样确实省 token，但表格列对齐、脚注、小字号、复选框状态都容易消失。很多团队以为“模型不懂文档”，实际问题是输入阶段已经把文档信息毁掉了。

第二个坑是版本混淆。2.0、2.5、3.0 共用大体一致的 `ViT-MLP-LLM` 架构，容易让人误以为“任意视觉编码器都能直接配任意 LLM”。接口能迁移，不等于官方配置边界不存在。尤其是 `300M + 72B` 这种说法，如果不标明版本，很容易误导读者。

第三个坑是训练和推理的 tile 上限不一致。官方文档里，2.0 训练常见是最多 12 个 tile，而测试可以零样本扩展到 40 个 tile；2.5 的训练上限又更高。能扩不代表无限扩。若部署时把 `tile_max` 设置得远高于训练分布，模型虽然“看见”更多图块，但未必学会稳定利用这些额外视觉上下文。

第四个坑来自数据噪声。2.5 官方文档明确强调，少量异常样本也可能诱发重复生成或异常行为。这件事对 2.0 系列的工程实践同样成立。对于文档和 OCR 混合数据，如果标注错位、框坐标漂移、表格答案不一致，模型会学到非常奇怪的输出习惯。

可以用一个排错表快速定位问题：

| 现象 | 常见原因 | 处理方式 |
| --- | --- | --- |
| 小字读不清 | tile 太少或图像压缩过度 | 提高 `tile_max`，保留原始分辨率 |
| 回答只抽字，不理解结构 | 输入被当成 OCR 任务设计 | 改写指令，保留版面上下文 |
| 输出重复片段 | 训练数据有异常样本 | 做规则过滤和质量打分 |
| 高分辨率下延迟暴涨 | 视觉 token 过多 | 控制 tile 上限，必要时先裁感兴趣区域 |

## 替代方案与适用边界

如果任务只是“把图片里的字符抠出来”，OCR 仍然是更直接、更便宜、更稳定的选择。OCR 是“光学字符识别”，白话讲就是把图里的字转成文本，不负责深层语义推理。

但如果任务变成“理解图文关系”，InternVL 2.0 这类多模态模型才开始有明显优势。比如识别快递单号，OCR 足够；判断一份合同截图中某条款是否与附录冲突，就需要同时看标题层级、段落位置、引用关系和文字内容，这已经超出纯 OCR 的舒适区。

| 方案 | 擅长什么 | 不擅长什么 | 成本 |
| --- | --- | --- | --- |
| OCR | 字符抽取、版面检测 | 图文联合推理 | 低 |
| 传统 VLM | 一般图像问答 | 长文档、细粒度小字 | 中 |
| InternVL 2.0 | 文档、表格、长图、复杂截图理解 | 极端长上下文和强行动推理仍有限 | 中到高 |

适用边界可以直接判断：

| 场景 | 选型建议 |
| --- | --- |
| 单纯字符识别 | 先选 OCR |
| 版面理解 + 字段关系抽取 | 优先多模态模型 |
| 复杂图文问答 | InternVL 2.0 类方案更合适 |
| 极低质量模糊输入 | 先做图像增强或重采集 |
| 超长文档批处理 | 需要分页、裁块和检索配合 |

因此，最准确的结论不是“InternVL 2.0 一定优于 OCR”，而是：当问题需要同时利用视觉结构和语言语义时，`ViT-MLP-LLM + 动态分辨率` 这条路线明显更合适；当问题只是字符提取时，使用多模态大模型通常是在为不需要的能力买单。
