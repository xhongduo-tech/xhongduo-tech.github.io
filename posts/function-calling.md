## Function Calling 的核心，不是“模型会调 API”，而是接口被结构化了

让 LLM 调用外部工具，真正困难的并不是生成一段类似函数名的文本，而是确保：

- 工具选择正确
- 参数格式合法
- 返回结果能继续参与后续推理

Function Calling 的关键价值，是把原本依赖自由文本约定的工具交互，变成结构化接口协议。

---

## 为什么 JSON Schema 很重要

工具描述通常包含：

- 名称
- 功能说明
- 参数字段及类型

这相当于给模型提供一份紧凑的接口文档。  
一旦 schema 足够清晰，模型就更容易学会：

- 什么时候该调用哪个工具
- 每个工具需要哪些参数
- 哪些字段是必填、可选或枚举

---

## 为什么仅靠自由文本不够

若只让模型输出类似：

```text
call_weather(city=Shanghai)
```

那么：

- 容易拼错字段
- 容易缺参数
- 难以并行与自动校验

结构化工具调用通过约束输出格式，把“模型是否记得语法”问题，转成“模型是否理解接口语义”问题。这大幅提升了系统可靠性。

---

## 小结

Function Calling 的本质，是把 LLM 与外部系统之间的交互，从自由文本协商升级为结构化协议协作。JSON Schema 让工具含义和参数边界清晰可校验，约束解码又进一步保证了输出合法性。

因此，Function Calling 不是一个 UI 级能力，而是现代 Agent 系统可靠落地的接口基础。
