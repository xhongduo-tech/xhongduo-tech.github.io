## 核心结论

指令数据的多样性，指的是训练样本在“任务类型、输入长度、语言风格、输出约束、安全与拒答”这些维度上的覆盖面，而不是单纯把题目数量做大。对零基础读者可以把它理解成：模型不是只要“见得多”，而是要“见得全”。

---

在指令微调里，真正决定泛化能力的，通常不是再多堆几万条相似问答，而是让训练分布覆盖更多语义方向。Only-IF 的核心发现就是：在总数据量相同的前提下，只要提高语义域多样性，模型在未见任务上的表现就会明显更稳。这里的“未见任务”可以理解为：训练时没怎么见过，但线上会真实遇到的指令类型，比如长文本改写、带格式约束的输出、拒答场景、跨语言请求。

一个直接的玩具例子：

- 模型 A 的训练集只有“闲聊 + 小学算术”
- 模型 B 的训练集在相同总量下，额外加入“写作、法务问答、命令行解释、长输入总结、JSON 输出、拒答示例”

当用户输入一句风格很奇怪、要求又很细的指令时，模型 A 往往会把它错判成聊天或问答任务；模型 B 更容易先识别“这是什么任务”，再选对回答策略。前者像只会两种做法，后者像掌握了一组做法。

经验上，`1 万条单一风格数据` 往往不如 `5 千条高覆盖数据`。前者容易让模型在头部任务上过拟合，后者更容易形成“意图识别 + 策略选择”的能力。工程上常见做法是对高频任务降权，对低频任务增采样，例如若某类样本频次为 $f_i$，可设
$$
w_i \propto \frac{1}{\sqrt{f_i}}
$$
再配合温度采样控制头尾比例。

下面这张表可以直观看出“覆盖”与“泛化”的关系。数值是示意，不是统一基准测试结果，但趋势符合论文结论与工程经验。

| 训练集方案 | 任务类别覆盖 | 输入长度覆盖 | 格式约束覆盖 | 未见任务准确率 | 长输入稳定性 | 拒答正确率 |
| --- | --- | --- | --- | --- | --- | --- |
| 只堆聊天问答 | 低 | 低 | 低 | 低 | 低 | 低 |
| 聊天 + 算术 + 少量写作 | 中 | 低 | 低 | 中低 | 低 | 低 |
| 多语义域 + 长短输入 + 结构化输出 + 拒答 | 高 | 高 | 高 | 高 | 高 | 高 |

结论可以压缩成一句话：多样性不是“题目多”，而是“训练分布覆盖广”；覆盖不够时，模型会把新问题强行映射到高频旧套路里。

## 问题定义与边界

先把概念定清楚。指令微调是指：用“用户指令 -> 理想回答”的监督数据，训练模型更像一个会执行任务的助手，而不只是续写文本。这里的“多样性”不是 prompt 换几个说法，而是训练分布在多个轴上都有覆盖。

可以用一个定义框来理解：

| 多样性维度 | 白话解释 | 作用 | 样本举例 | 可测指标 |
| --- | --- | --- | --- | --- |
| 任务类型 | 让模型见过不同“工作类型” | 学会先判断意图 | 分类、抽取、总结、翻译、代码解释 | 各任务占比、任务熵 |
| 输入长度 | 让模型见过短问题和长上下文 | 避免长 prompt 崩掉 | 一句话问答、2k token 文档总结 | 长度分桶覆盖率 |
| 语言风格 | 让模型见过正式、口语、命令式、模糊表达 | 提升鲁棒性 | “请总结”/“给我压缩成三点” | 风格分桶、模板多样度 |
| 输出约束 | 让模型学会按格式交付 | 减少跑题 | JSON、表格、要点列表、固定字段 | 格式正确率 |
| 安全与拒答 | 让模型学会何时不能答、如何拒答 | 避免有害输出 | 医疗处方、越权指令、危险操作 | 拒答准确率、误拒率 |

边界也要讲清楚。

第一，多样性不等于“随机打乱话术”。如果 1 万条样本都还是同一类客服问答，只是换了不同说法，它的语义覆盖仍然很窄。模型更可能学到“这类问题常见回答模板”，而不是更高层的任务抽象。

第二，多样性不等于“标签种类多就够了”。有的团队会说自己有 20 个任务标签，但每个任务都只有短输入、中文、自由文本输出、没有拒答场景。这种数据在维度上仍然是塌缩的。模型上线后碰到英文、长上下文、严格 JSON 输出、敏感请求时，仍然会掉线。

第三，这篇文章讨论的是“指令微调数据分布设计”，不是预训练语料设计。预训练解决的是广义语言与世界知识覆盖；指令微调更像“把模型教成会做事的助手”。这里关注的是任务执行分布，而不是原始互联网文本覆盖。

再用一个零基础能理解的例子说明：

如果训练集只有“聊天”和“数学题”，模型会逐渐形成一个隐含假设：用户输入不是来聊天，就是来算数。于是当用户说“请把这段合同改写成风险清单，并按 JSON 输出”，模型可能会做三种错误中的一种：

- 当成聊天，给一段泛泛建议
- 当成问答，只答一个点
- 忽略 JSON 约束，输出一段自由文本

这不是模型“笨”，而是训练分布没有教它这种指令应该映射到哪种策略。加入写作、法务、结构化输出、长输入和拒答样本后，模型才更可能学到：先识别任务，再调用正确的输出模式。

## 核心机制与推导

为什么高覆盖数据更容易泛化？核心原因是：模型在监督训练中学到的，不只是“答案内容”，还包括“什么指令该用什么策略”。如果头部任务频率过高，梯度会被它们主导，模型更容易记住高频套路，而不是形成均衡的任务边界。

设任务类别为 $i$，其样本频次为 $f_i$。最简单的去头部化方法是按频次反比降权，例如：
$$
w_i = \frac{1}{\sqrt{f_i}}
$$
直觉上，样本越多的任务，新增一条数据带来的边际收益越小，所以不应继续按原频率主导训练。

更平滑的做法来自 class-balanced loss。它先定义“有效样本数”：
$$
E_n = \frac{1-\beta^n}{1-\beta}, \quad \beta \in [0,1)
$$
再令类别权重与 $1/E_n$ 成正比。白话说：不是每多一条数据，信息量都一样大；同类样本越多，重复信息越多。

把权重得到后，还可以再做温度采样。设原始打分为 $w_i$，采样概率写成：
$$
P(i|T)=\frac{\exp(w_i/T)}{\sum_j \exp(w_j/T)}
$$
其中温度 $T$ 用来控制“拉平程度”：

- $T<1$：放大差异，更偏向高权重任务
- $T>1$：压平差异，让采样更均匀

一个玩具数值例子。假设两类任务 A 和 B：

- A 频次 $f_A=10000$
- B 频次 $f_B=400$

先做频次反平方根降权：
$$
w_A=\frac{1}{\sqrt{10000}}=0.01,\quad
w_B=\frac{1}{\sqrt{400}}=0.05
$$

如果直接按权重归一化：
$$
p_A=\frac{0.01}{0.01+0.05}\approx 0.167,\quad
p_B\approx 0.833
$$

也就是说，虽然原始数据里 A 是绝对头部，但重加权后，B 在采样里反而成为主角。这样做的目的不是“偏爱冷门”，而是修正训练分布，让模型别一直盯着已经学会的部分。

如果再做温度平滑，比如对对数权重做 softmax 并取 $T=2$，概率会向更均匀的方向移动，近似可以理解为从 `0.17/0.83` 拉回到更平滑的 `0.33/0.67` 一带。这样既能补尾部，又不会过度牺牲头部任务。

可以把整个过程写成一条链：

$$
f_i \rightarrow w_i \propto \frac{1}{\sqrt{f_i}}
\rightarrow \frac{w_i}{T}
\rightarrow P(i|T)
$$

真实工程例子比玩具例子更能说明问题。假设一个通用助手要覆盖：

- 日常问答
- 文档总结
- 表格抽取
- 命令行解释
- 安全拒答
- 中英混合输入

如果训练数据 70% 都是开放问答，模型会在评测集上看起来“很会聊天”，但一遇到下面这些情况就容易翻车：

- 4k token 的长输入总结
- “必须输出合法 JSON”
- “这个危险操作能不能一步步教我”
- “帮我解释这段 bash 错误日志”

这类失败，本质上不是参数量问题，而是训练分布没有覆盖到这些策略切换点。

## 代码实现

下面给一个最小可运行的 Python 例子，展示如何从“任务频次”得到“重加权后的采样概率”。代码目标不是生产可用，而是把机制讲清楚。

```python
import math

def normalized_inverse_sqrt_weights(freqs):
    weights = {k: 1.0 / math.sqrt(v) for k, v in freqs.items()}
    total = sum(weights.values())
    return {k: weights[k] / total for k in weights}

def softmax(xs):
    m = max(xs)
    exps = [math.exp(x - m) for x in xs]
    s = sum(exps)
    return [x / s for x in exps]

def temperature_sampling_from_probs(probs, temperature=1.0):
    # 对概率取 log 后再做温度 softmax，避免直接在原始频次上操作
    items = list(probs.items())
    names = [k for k, _ in items]
    logits = [math.log(v) / temperature for _, v in items]
    out = softmax(logits)
    return dict(zip(names, out))

freqs = {"qa_head": 10000, "cmd_tail": 400}

base_probs = normalized_inverse_sqrt_weights(freqs)
temp_probs = temperature_sampling_from_probs(base_probs, temperature=2.0)

assert round(base_probs["qa_head"], 3) == 0.167
assert round(base_probs["cmd_tail"], 3) == 0.833
assert 0 < temp_probs["qa_head"] < temp_probs["cmd_tail"] < 1
assert abs(sum(temp_probs.values()) - 1.0) < 1e-9

print("base_probs =", base_probs)
print("temp_probs =", temp_probs)
```

对应的简化流水线可以写成：

```text
频次统计 -> 计算任务权重 -> 温度采样任务 -> 抽样组成 batch -> 训练 -> 重新统计弱项
```

如果进一步写成伪代码，就是：

```python
freqs = count_task_freq(training_pool)
weights = {t: 1 / sqrt(freqs[t]) for t in freqs}
probs = temperature_softmax(weights, T=2.0)
sampled_tasks = sample_by_probs(probs, batch_size)

for task in sampled_tasks:
    sample = sample_example(task)
    train_on(sample)
```

更真实一点的工程版本，通常不是只看任务标签，而是同时看多个字段：

- `task_type`
- `input_length_bucket`
- `language`
- `output_format`
- `safety_label`

也就是说，你真正采样的对象往往不是“问答类”这么粗的标签，而是类似：

- `summarization + long + zh + bullet`
- `extraction + medium + en + json`
- `safety_refusal + short + zh + freeform`

这样才能避免一种常见假象：任务标签看起来很全，但长输入、结构化输出、拒答场景仍然严重缺样。

再给一个真实工程例子。假设你在做企业内部知识助手，原始日志里 80% 是简单 FAQ。若直接全量微调，模型会在 FAQ 指标上很好看，但在下面三类真实请求上不稳：

- “根据这份长文档提炼 5 个风险点”
- “把结果按固定 JSON schema 输出”
- “涉及内部权限操作时必须拒答”

这时更合理的做法不是继续堆 FAQ，而是：

1. 对 FAQ 降采样或降权。
2. 补充长输入总结、结构化输出、拒答样本。
3. 按多维桶做温度采样。
4. 每轮训练后看哪些桶仍然弱，再定向补数据。

这套流程的重点不是“更复杂”，而是让训练分布更像线上真实分布。

## 工程权衡与常见坑

工程上最常见的误区，是把“类别均衡”误当成“多样性足够”。比如每个任务标签都取 1000 条，看起来很公平，但如果这 1000 条都很短、都只用中文、都不给格式约束、都不含拒答，那上线后仍然会在长文本、跨语言、结构化输出和安全场景上集体掉线。

下面这张表总结常见坑：

| 问题类型 | 传统采样后果 | 更合理的对策 |
| --- | --- | --- |
| 只看任务标签，不看长度 | 长 prompt 一到就崩 | 按长度分桶采样 |
| 只看语义域，不看格式 | JSON、表格输出错误多 | 把格式约束当成独立维度 |
| 头部问答过多 | 模型学会套模板 | 对高频任务降权或温度平滑 |
| 没有拒答数据 | 危险请求也认真回答 | 混入明确拒答与边界说明样本 |
| 只保留主流语言 | 非英语或混合输入失稳 | 语言维度单独建桶 |
| 一次性静态抽样 | 后续弱点无法补齐 | 迭代重估弱项并补样 |

Low Training Data 一类工作说明了一点：不是非得用全集，选择一个高覆盖的小子集，往往就能达到接近甚至超过全量的效果。比如 DiverseEvol 的思路，就是在嵌入空间中迭代挑选“与已有样本差异更大”的数据点。白话说，它不是问“哪个样本最常见”，而是问“哪个样本能把当前覆盖面再撑开一点”。

但这里也有权衡：

- 如果过度追求尾部，头部常见任务可能退步。
- 如果只追求嵌入距离，可能选到“看起来不同、但实际价值不高”的样本。
- 如果拒答样本过多，模型可能变得保守，误拒率上升。

所以工程上通常不是单一指标，而是几种信号一起用：

- 频次重加权：防头部垄断
- 温度采样：控制平滑程度
- 多维桶覆盖：保证长度、语言、格式、安全都被覆盖
- 迭代评估：发现弱桶后再补样

可以把风险缓解流程写得很简单：

1. 先统计线上请求分布，确定必须覆盖的桶。
2. 对训练池做多维标注，而不是只打任务标签。
3. 用降权或温度采样控制头尾比例。
4. 单独监控长输入、格式正确率、拒答准确率、误拒率。
5. 每轮只补最弱的几个桶，不盲目扩大全集。

## 替代方案与适用边界

频次重加权和温度采样解决的是“分布失衡”问题，但它们不直接回答一个更尖锐的问题：在预算很小的时候，到底该保留哪些样本？这时就要看替代方案。

第一类方案是多样性优先，比如 DiverseEvol、K-center、Vendi-Score 这一类。核心思想是从嵌入空间里选“彼此更不相似”的样本，保证覆盖面。适用场景是：

- 数据池很大，但训练预算有限
- 需要先快速做出一个广覆盖版本
- 还没有明确知道模型的具体弱点

第二类方案是弱点优先，比如 MIWV。它的思路是衡量“哪些指令最能暴露模型当前能力缺口”，再优先选这些样本。适用场景是：

- 已有一个基础模型
- 已知模型在某些类型上明显偏弱
- 希望用极小预算做定向补强

可以用一张判断表快速区分：

| 方法 | 预算区间 | 核心指标 | 适用场景 |
| --- | --- | --- | --- |
| 频次重加权 + 温度采样 | 中到大 | 类别频次、采样概率 | 已有较大数据池，先修正头尾失衡 |
| DiverseEvol / K-center | 小到中 | 嵌入空间覆盖 | 预算有限，先做高覆盖子集 |
| MIWV | 很小到中 | 模型弱点增益 | 已知模型短板，做针对性补强 |
| 全量直接训练 | 大 | 无特殊选择 | 数据质量高且预算充足 |

真实工程里，经常会面对两个选择：

- 方案 A：继续扩大全量数据，保持原始分布
- 方案 B：只取 5% 高覆盖子集，再做温度采样

很多团队直觉上会选 A，因为“数据更多更安全”。但如果新增的 95% 仍然主要来自头部问答，A 的收益可能非常有限。B 虽然数据更少，却可能因为覆盖了低频格式、长输入、拒答场景和弱项任务，在 few-shot 或真实线上任务上更稳。

适用边界也必须说清楚。多样性不是万能药：

- 如果基础模型太弱，单靠高覆盖小数据也救不了核心能力。
- 如果任务本身极窄，比如只做一种固定格式抽取，过多跨域样本反而会稀释目标。
- 如果线上需求高度集中，追求极端均衡可能不如针对主任务做更细的数据清洗。

所以最实用的原则不是“永远追求最大多样性”，而是“用最小成本覆盖线上真正会出现的关键维度”。
