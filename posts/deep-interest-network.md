## 核心结论

DIN（Deep Interest Network，深度兴趣网络）解决的不是“怎么把用户历史行为喂给模型”这个表面问题，而是“同一个用户面对不同候选物品时，应该激活哪部分历史行为”这个核心问题。

传统 CTR 排序模型常把用户历史点击、浏览、收藏等行为做 `sum` 或 `mean pooling`，也就是直接求和或求平均。这种做法默认一个用户只有一个固定兴趣向量。问题在于，真实用户通常同时有多种兴趣，且当前展示的候选物品只和其中一部分兴趣相关。DIN 的关键改动是：不再生成一个对所有候选都相同的用户向量，而是针对每个候选物品，动态计算一组行为权重，再得到一个“候选相关”的兴趣表示。

其核心公式是：

$$
V_u(A)=\sum_i g(V_i,V_a)\cdot V_i
$$

其中，$V_i$ 是第 $i$ 个历史行为的向量，$V_a$ 是当前候选物品向量，$g(V_i,V_a)$ 是激活单元输出的权重。白话解释是：历史行为不是平均对待，而是“谁和当前候选更像，谁权重更高”。

这带来三个直接收益：

| 结论 | 技术含义 | 新手理解 |
| --- | --- | --- |
| 兴趣表示动态变化 | 用户向量依赖候选物品 | 同一个人看手机广告和耳机广告，模型关注的历史记录不同 |
| 多兴趣可区分 | 不再被单一平均向量抹平 | 用户既喜欢数码也喜欢运动，两类兴趣不会互相覆盖 |
| 排序判别力更强 | CTR 模型看到更精细的匹配信号 | 更容易区分“会不会点这个具体物品” |

玩具例子最容易说明这一点。假设用户历史里有两类行为：看过很多手机，也看过几次笔记本。如果当前候选是手机壳，那么和手机相关的行为会被放大；如果当前候选是运动耳机，那么手机和笔记本行为都不一定强相关，最终兴趣向量会明显不同。DIN 的价值不在“加了注意力”这四个字，而在于把“候选相关性”直接放进了用户兴趣建模。

## 问题定义与边界

DIN 主要用于 CTR 预估，也就是预测“某个用户看到某个物品后点击的概率”。CTR 是 click-through rate，点击率，这里更准确地说是点击概率建模。

问题边界可以写成一句话：已知用户画像、上下文特征、候选物品特征、用户历史行为序列，输出当前候选的点击概率。

DIN 特别针对序列行为特征，也就是用户过去的点击、浏览、加购、收藏等离散事件。它假设用户行为有两个关键属性：

| 特征 | 含义 | 新手理解 |
| --- | --- | --- |
| Diversity | 用户同时存在多种兴趣 | 既买数码，也看运动装备 |
| Local activation | 只有一部分历史对当前候选真正相关 | 展示跑鞋时，不需要过度关注用户看过的手机壳 |

如果用平均池化处理历史序列，会出现一个结构性问题：同一用户的兴趣向量固定不变，于是模型只能依赖候选物品本身去补救。这样会让“用户对不同候选的兴趣差异”表达不足。

举一个新手版例子。某电商用户最近看过“运动鞋”和“数码手环”。现在平台有两个候选广告：

1. 候选 A：运动手环
2. 候选 B：手机充电器

对候选 A，模型应该更重视“运动鞋”和“数码手环”两类行为，尤其是与穿戴设备接近的部分；对候选 B，历史中的“数码手环”可能仍有一点帮助，但“运动鞋”就不该占主要权重。若直接求平均，这两种候选最终看到的用户表示会过于接近，排序效果会受限。

DIN 也有明确边界。它不是万能的序列建模框架，主要适合以下场景：

| 适合 | 原因 |
| --- | --- |
| 用户行为较丰富的推荐/广告排序 | 有足够多历史行为可供动态加权 |
| 候选相关性强的任务 | 当前物品与历史行为的匹配关系决定点击 |
| 工业级稀疏特征系统 | Embedding + MLP 是标准工程形态 |

不太适合的情况包括：

| 不太适合 | 原因 |
| --- | --- |
| 用户历史极短 | 动态加权没有足够序列可用 |
| 目标更偏长期价值而非即时点击 | DIN 更偏当前候选的即时匹配 |
| 强时间演化任务 | DIN 不显式建模兴趣迁移过程 |

真实工程例子是信息流广告排序。用户上午连续浏览母婴用品，下午开始搜索空气净化器。若只保留一个固定兴趣向量，模型很难表达“这个用户现在对家电更敏感”。DIN 至少能在候选是空气净化器时，把相关行为激活出来，比简单平均更接近真实点击决策。

## 核心机制与推导

DIN 的核心部件是 activation unit，激活单元。它本质上是一个小型前馈网络，输入是“历史行为向量”和“候选物品向量”的交互特征，输出一个标量权重。

术语解释：注意力机制就是“给输入的不同部分分配不同权重”。在 DIN 里，被分配权重的是用户历史行为。

### 1. 候选相关兴趣向量

设用户历史行为向量为 $V_1,V_2,\dots,V_n$，候选物品向量为 $V_a$，则：

$$
V_u(A)=\sum_{i=1}^n g(V_i,V_a)\cdot V_i
$$

这里的 $g$ 不是固定相似度函数，而是可学习的神经网络。工程上常把以下特征拼起来送入 MLP：

$$
[V_i,\ V_a,\ V_i-V_a,\ V_i \odot V_a]
$$

其中 $\odot$ 表示逐元素乘。白话解释是：不仅看两个向量本身，还看它们的差异和逐维匹配情况。

数值玩具例子如下。

候选广告：

$$
V_a=[0.6,0.4]
$$

两个历史行为分别为：

$$
V_1=[0.2,0.8],\quad V_2=[0.9,0.1]
$$

激活单元输出权重：

$$
g(V_1,V_a)=0.3,\quad g(V_2,V_a)=0.7
$$

则候选相关兴趣向量为：

$$
V_u(A)=0.3\cdot[0.2,0.8]+0.7\cdot[0.9,0.1]=[0.69,0.29]
$$

这个结果说明：虽然用户有两个历史行为，但模型认为第二个行为和当前候选更相关，所以最终表示更靠近 $V_2$。这就是“局部激活”的具体含义。

### 2. 为什么不强制做 softmax

标准注意力常把权重归一化到和为 1。DIN 论文中的一个重要点是，注意力分数不一定必须做 softmax。原因是 CTR 场景下，用户历史长度和行为强度本身可能有信息量。若强行归一化，可能丢掉“这个用户整体历史信号强弱”的差异。

因此工程实现中常有一个开关：`att_weight_normalization`。开启时更像标准 attention；关闭时更保留原始打分尺度。两者没有绝对优劣，要用离线指标和在线实验决定。

### 3. Dice 激活函数

Dice（Data Adaptive Activation Function）是 DIN 另一个常被忽略但工程上很重要的设计。激活函数就是神经网络里控制非线性的函数。ReLU 的分界点固定在 0，而 Dice 会根据 mini-batch 的统计量自适应调整。

公式是：

$$
p_i=\frac{1}{1+\exp\left(-\frac{y_i-\mathrm{E}[y_i]}{\sqrt{\mathrm{Var}[y_i]+\epsilon}}\right)}
$$

$$
y_i'=a_i(1-p_i)y_i+p_i y_i
$$

其中，$\mathrm{E}[y_i]$ 是 mini-batch 均值，$\mathrm{Var}[y_i]$ 是 mini-batch 方差，$a_i$ 是可学习参数。

白话解释：Dice 不直接把“小于 0 的值砍掉”，而是根据当前批次的数据分布，平滑决定某个神经元该通过多少、保留多少负半轴信息。对大规模稀疏特征来说，这能让训练更稳，因为不同 batch 的激活分布差异很大。

### 4. 为什么 Dice 对稀疏 CTR 有帮助

CTR 任务的输入通常是高维稀疏特征。稀疏的意思是大部分维度都为 0，只有少量维度有值。此时，如果用固定阈值激活函数，某些神经元可能长期处于“几乎不更新”状态，收敛会慢，也更容易出现训练不稳定。

Dice 的好处可以概括成一条：

| 机制 | 作用 |
| --- | --- |
| 基于 batch 均值和方差调整分界点 | 适应不同特征分布 |
| 负半轴保留可学习比例 | 减少信息硬截断 |
| 更适合稀疏输入 | 训练过程更平滑 |

因此，DIN 不是只有“候选相关注意力”一个改进，而是“动态兴趣建模 + 更稳的激活函数”一起构成了完整方案。

## 代码实现

下面先用一个纯 Python 玩具实现说明 DIN 的核心计算。它不是训练完整模型，而是复现“给定候选，计算动态兴趣向量”的关键过程。

```python
import math

def dot(a, b):
    return sum(x * y for x, y in zip(a, b))

def attention_score(hist_vec, candidate_vec):
    # 玩具版打分：直接用点积后过 sigmoid
    score = dot(hist_vec, candidate_vec)
    return 1.0 / (1.0 + math.exp(-score))

def din_interest(history, candidate):
    weights = [attention_score(h, candidate) for h in history]
    interest = [
        sum(w * h[d] for w, h in zip(weights, history))
        for d in range(len(candidate))
    ]
    return weights, interest

history = [
    [0.2, 0.8],  # 行为1
    [0.9, 0.1],  # 行为2
]
candidate_phone = [0.6, 0.4]
candidate_sport = [0.1, 0.9]

w1, interest1 = din_interest(history, candidate_phone)
w2, interest2 = din_interest(history, candidate_sport)

assert len(w1) == 2
assert len(interest1) == 2
assert interest1 != interest2  # 不同候选应得到不同兴趣向量
assert sum(w1) > 0
assert sum(w2) > 0

print("candidate_phone weights:", w1)
print("candidate_phone interest:", interest1)
print("candidate_sport weights:", w2)
print("candidate_sport interest:", interest2)
```

这段代码的重点不是打分函数有多复杂，而是验证一个性质：同样的历史序列，在不同候选下，输出的兴趣向量应该不同。只要这个性质不成立，就不是 DIN 的核心思想。

如果在 Keras/DeepCTR 中落地，通常会直接使用现成实现。DeepCTR 是一个面向 CTR 场景的深度学习库，封装了 DIN、DIEN 等常见结构。典型写法如下：

```python
from deepctr.models import DIN

model = DIN(
    dnn_feature_columns,
    history_feature_list=["item_id"],
    att_hidden_units=(80, 40),
    att_activation="dice",
    att_weight_normalization=True,
)
```

这段配置背后的含义是：

| 参数 | 作用 | 工程含义 |
| --- | --- | --- |
| `history_feature_list` | 指定哪些字段属于历史序列 | 告诉模型哪些是用户行为轨迹 |
| `att_hidden_units` | 注意力 MLP 的层宽 | 控制激活单元表达能力 |
| `att_activation="dice"` | 使用 Dice 激活 | 提升稀疏输入训练稳定性 |
| `att_weight_normalization` | 是否归一化注意力权重 | 控制是否更接近标准 attention |

真实工程里，DIN 一般不会单独存在，而是嵌在完整排序链路中：

1. 离散特征做 embedding
2. 候选物品和历史序列进入 attention pooling
3. 得到候选相关兴趣向量
4. 与用户画像、上下文、统计特征拼接
5. 送入 DNN 输出 CTR

一个真实工程例子是电商推荐排序。输入特征往往包括：

| 特征组 | 示例 |
| --- | --- |
| 用户侧 | 年龄段、设备、地域、会员等级 |
| 候选侧 | item_id、shop_id、brand_id、类目 |
| 上下文 | 时间、流量位置、渠道 |
| 历史序列 | 最近点击 item_id、最近浏览类目、最近加购 shop_id |

在这里，DIN 的 attention pooling 层最主要负责“把长序列压缩成一个候选相关向量”，否则后面的 DNN 无法直接消费变长行为序列。

## 工程权衡与常见坑

DIN 在线上通常比“平均池化 + DNN”更强，但它绝不是无代价升级。工程上要同时看收益、代价、回滚条件。

### 1. 计算成本上升

DIN 需要对“候选物品 × 历史行为”逐一计算 attention 分数。候选越多、历史越长，计算量越大。若一个请求要对几百个候选打分，且每个用户保留 100 条以上行为，线上延迟会明显上升。

常见做法包括：

| 方法 | 目的 |
| --- | --- |
| 截断历史序列长度 | 控制延迟和显存 |
| 只保留高价值行为 | 降噪并减少计算 |
| 分层召回后再做 DIN 排序 | 把 DIN 留给少量高质量候选 |

### 2. 注意力并不等于可解释性

很多人会把高 attention 权重直接解释为“模型认为这条行为导致点击”。这在工程上很危险。attention 分数更多是中间表示学习结果，不应等价于严格因果解释。它可以做分析参考，但不应直接作为业务结论。

### 3. 长序列噪声问题

用户历史里常混有误点击、低意图浏览、活动流量、异常曝光。DIN 会对每条历史分配权重，如果输入序列噪声很多，注意力可能学会放大错误模式。

因此序列特征治理很重要：

| 常见坑 | 表现 | 处理方式 |
| --- | --- | --- |
| 历史过长 | 训练慢、噪声多 | 截断最近 N 条或按行为强度采样 |
| 行为类型混杂 | 点击、曝光、加购语义不同 | 分开建模或加行为类型 embedding |
| 负样本采样失真 | 离线效果虚高 | 保持采样策略与线上分布接近 |
| 只看 AUC 不看 GAUC | 局部用户群退化被掩盖 | 同时监控 GAUC、分桶指标 |

术语解释：GAUC 是 group AUC，按用户或请求分组后再计算的 AUC，更适合排序场景，因为它能反映组内排序质量。

### 4. 自动化检查、核心指标和回滚条件

DIN 上线前不能只看离线 AUC。更稳妥的检查集至少包括：

| 检查项 | 目的 | 最低要求 |
| --- | --- | --- |
| 离线 AUC/GAUC | 验证整体排序能力 | 新模型高于基线 |
| 分桶 GAUC | 看不同用户群是否退化 | 新老用户、高活跃低活跃都不能明显劣化 |
| 推理延迟 P95/P99 | 控制线上 SLA | 不超过系统预算 |
| 权重分布监控 | 防止注意力塌缩 | 不能长期集中到单一位置 |
| 训练稳定性 | 防止发散 | loss、AUC 曲线平滑 |

回滚条件必须事先写清楚，而不是上线后临时判断。一个典型规则可以是：

| 版本 | GAUC | 说明 |
| --- | --- | --- |
| 基线模型 | 59.59% | 当前稳定版本 |
| DIN + Dice | 60.83% | 可上线候选版本 |
| 新上线版本 | ≥60.3% | 若连续低于阈值则回滚 |

真实工程例子：假设你在广告排序系统中把平均池化替换成 DIN，离线 GAUC 提升 0.8 个点，但线上首日发现高峰期 P99 延迟上涨 25%，同时低活跃用户桶的 GAUC 下滑。此时即使总 GAUC 仍略高，也不应贸然全量。更稳妥的做法是按流量分层灰度，设置“连续若干个监控窗口低于阈值自动回滚”的策略。

### 5. Dice 和正则化要一起看

DIN 原论文还强调 mini-batch aware regularization。它的核心思想是：稀疏场景下，不是每次都该对所有 embedding 参数施加同样的正则，而应该更关注当前 batch 真正被激活的参数。否则要么正则太弱导致过拟合，要么正则太强压制有效特征。

简单说，Dice 解决的是“激活分布不稳定”，自适应正则更多解决“稀疏参数过拟合”。二者配合，训练通常更稳。

## 替代方案与适用边界

DIN 解决的是“候选相关兴趣激活”，但它没有完整建模兴趣随时间的演化，也没有显式利用 session 结构。若任务特征不同，可以考虑替代方案。

| 方法 | 关注点 | 适用场景 |
| --- | --- | --- |
| DIN | 候选相关的历史行为加权 | 常规 CTR、多兴趣用户 |
| DIEN | 兴趣演化，显式建模时间过程 | 用户意图变化快 |
| DSIN | 会话级兴趣提取再聚合 | 行为天然按 session 分组 |

DIEN（Deep Interest Evolution Network，深度兴趣演化网络）可以看成在 DIN 基础上更进一步。它先提取兴趣，再建模兴趣如何随时间演化。白话解释：DIN 更像“从历史里挑相关片段”，DIEN 更像“先理解兴趣轨迹，再判断当前候选是否匹配这个轨迹”。

DSIN（Deep Session Interest Network，深度会话兴趣网络）适合 session 结构明显的场景。session 就是一段连续行为会话，例如一次旅游搜索、一轮晚间购物浏览。它先在每个 session 内提取兴趣，再让候选去激活相关 session。对于旅游、内容消费、搜索推荐这类“短期主题非常集中”的业务，DSIN 往往比直接把所有行为混在一起更合适。

新手版例子如下：

1. 常规电商首页推荐：DIN 往往够用，因为候选相关性最重要。
2. 短视频推荐：若用户兴趣切换很快，DIEN 更可能有价值。
3. 旅游搜索推荐：用户常围绕一次出行形成若干 session，DSIN 更合适。

DIN 的适用边界还包括一个现实约束：如果你的系统还处在“特征工程很弱、日志质量差、样本链路不稳定”的阶段，直接上 DIN 不一定最划算。因为模型越复杂，对数据质量越敏感。很多团队先把样本、特征、监控、延迟链路打稳，再引入 DIN，整体收益更可控。
