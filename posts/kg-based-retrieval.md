## 核心结论

基于 Knowledge Graph 的召回，核心不是“找相似物品”，而是把用户历史行为映射到知识图谱上，再沿语义关系扩展候选集。

知识图谱，简称 KG，是由实体和关系组成的图结构。实体可以是商品、品牌、类目、演员、疾病、论文；关系可以是“属于”“适配”“同品牌”“导演是”“上位概念是”。在推荐召回里，KG 的价值是把物品背后的语义结构显式建出来。

普通召回主要依赖点击、购买、共现、向量相似度。KG 召回额外利用“这个物品是什么、属于什么、能配什么、和谁有上下位关系”。因此它最适合解决稀疏和冷启动问题，尤其是在物品有丰富属性、关系和概念体系时。

玩具例子：用户只点过 `iPhone 15`。如果只看点击共现，系统可能只能召回其他热门手机或同价位手机。但在 KG 里，`iPhone 15` 可以连到 `Apple`、`MagSafe`、`手机壳`、`兼容配件`、`Lightning/USB-C 配件` 等实体。即使某个新上架的手机壳没有点击数据，只要它通过“适配机型=iPhone 15”“品牌=Apple 生态”“类型=保护壳”接入图，也有机会被召回。

| 维度 | 普通召回 | KG 召回 |
|---|---|---|
| 召回信号 | 点击、购买、收藏、共现、向量相似 | 实体、属性、关系、多跳语义路径 |
| 候选来源 | 相似用户、相似物品、热门池、双塔近邻 | 历史物品在 KG 上的多跳邻居 |
| 适用场景 | 行为数据充足、物品变化稳定 | 冷启动、长尾物品、语义关系丰富 |
| 局限性 | 对新物品、新用户弱 | 依赖图质量，容易受噪声和延迟影响 |

真实工程例子：电商平台上架一批 `iPhone 15` 手机壳。新商品没有点击和购买记录，协同过滤难以召回。但商品结构化信息里有 `适配机型=iPhone 15`、`材质=silicone`、`品类=手机壳`、`兼容=MagSafe`。用户最近浏览了 `iPhone 15` 和 `MagSafe 充电器`，KG 召回可以沿“机型-配件-兼容关系”扩展候选，把新手机壳放进召回集，再交给排序模型判断是否展示。

## 问题定义与边界

本文默认只讨论召回，不讨论排序。召回阶段的目标是从海量物品中快速找出几百到几千个可能相关的候选；排序阶段才负责精细比较点击率、转化率、价格、库存、时效等因素。

新手版解释：普通召回只看“谁和谁买过”“哪些物品经常一起出现”；KG 召回还看“这个东西属于什么、能配什么、和谁有上下位关系”，所以能更早发现候选。

符号先定义清楚：

| 符号 | 含义 | 白话解释 |
|---|---|---|
| $u$ | 用户 | 当前要做推荐的人 |
| $i$ | 物品 | 候选商品、文章、视频或音乐 |
| $S_u$ | 用户历史点击集合 | 用户过去交互过的物品集合 |
| $G$ | 知识图谱，$G=(E,R)$ | 由实体集合 $E$ 和关系集合 $R$ 组成的图 |
| $K$ | 扩展跳数 | 从历史物品出发向外走几层 |
| $score(u,i)$ | 用户与物品的召回分数 | 分数越高，越可能进入候选集 |

更具体地，知识图谱里常用三元组表示关系：

$$
(h, r, t)
$$

其中 $h$ 是头实体，$r$ 是关系，$t$ 是尾实体。例如：

$$
(\text{iPhone 15}, \text{compatible\_with}, \text{MagSafe})
$$

用户历史点击集合记为：

$$
S_u = \{i_1, i_2, ..., i_n\}
$$

第 $k$ 跳邻域记为 $R_u^{(k)}$，意思是从用户历史物品出发，沿图走到第 $k$ 层时能触达的关系集合。KG 召回要做的事，就是从 $S_u$ 出发，在 $G$ 上扩展 $K$ 跳，然后给触达的候选物品计算 $score(u,i)$。

边界也要明确：KG 召回不等于完整推荐系统。它不负责最终展示顺序，不直接处理页面多样性，也不单独决定商业策略。它只是在召回阶段提供一批“有语义依据的候选”。

## 核心机制与推导

多跳传播是 KG 召回的核心机制。多跳传播，就是用户兴趣从已交互物品出发，沿图上的实体关系逐层扩散，最终把远处但语义相关的物品带入候选集。

一个统一的抽象写法如下：

```text
R_u^(0) = {(h, r, t) | h ∈ S_u}
α_{k,j} = softmax(q_u^(k-1) · e_{h_{k,j}})
q_u^(k) = Σ_j α_{k,j} φ(e_{t_{k,j}}, e_{r_{k,j}})
s(u,i) = σ(q_u^(K) · e_i)
```

逐行解释：

`R_u^(0) = {(h, r, t) | h ∈ S_u}` 表示第 0 层从用户点过的物品开始。只要三元组的头实体 $h$ 属于用户历史集合 $S_u$，这条边就会进入初始传播范围。

`α_{k,j} = softmax(q_u^(k-1) · e_{h_{k,j}})` 表示第 $k$ 跳第 $j$ 条边的重要性。$q_u^{(k-1)}$ 是上一跳得到的用户兴趣向量，$e_{h_{k,j}}$ 是当前头实体的向量。点积越大，说明这条边和当前兴趣越匹配。`softmax` 会把多个边的权重归一化。

`q_u^(k) = Σ_j α_{k,j} φ(e_{t_{k,j}}, e_{r_{k,j}})` 表示聚合第 $k$ 跳的信息。$e_{t_{k,j}}$ 是尾实体向量，$e_{r_{k,j}}$ 是关系向量，$\phi$ 是组合函数，用来表达“通过某种关系到达某个实体”这件事。加权求和后得到新的用户兴趣向量。

`s(u,i) = σ(q_u^(K) · e_i)` 表示最终候选物品得分。$q_u^{(K)}$ 是传播 $K$ 跳后的用户兴趣，$e_i$ 是候选物品向量。两者越接近，分数越高。$\sigma$ 通常是 sigmoid 函数，用来把分数压到 $0$ 到 $1$ 附近。

数值例子：用户最近点过物品 `A`。KG 里有两条两跳路径：

| 路径 | 边权 | 路径分数 |
|---|---:|---:|
| `A -> B -> D` | `0.7 × 0.9` | `0.63` |
| `A -> C -> E` | `0.3 × 0.8` | `0.24` |
| `A -> F -> D` | `0.1 × 0.5` | `0.05` |

`D` 的总分约为 $0.63 + 0.05 = 0.68$，高于 `E` 的 $0.24$。所以 `D` 比 `E` 更容易进入召回集。关键点是：不是单条边决定结果，而是多条语义路径一起加权。

不同方法的差异，主要在传播方式和训练目标上：

| 方法 | 核心思想 | 训练目标 | 适合场景 |
|---|---|---|---|
| RippleNet | 从用户历史物品出发，像涟漪一样逐跳扩散兴趣 | 点击预测或评分预测 | 需要解释多跳兴趣传播的推荐场景 |
| KGAT | 在协同知识图上做注意力消息传递 | 推荐损失 + 图注意力传播 | 用户-物品行为和 KG 都较丰富的场景 |
| MKM-SR | 把下一物品预测和知识图谱嵌入联合训练 | $L = L_{next-item} + \lambda L_{KG}$ | 会话推荐、短序列推荐、行为稀疏场景 |

KGAT 经常被放进“路径或高阶关系方法”里讨论，但它的本质更接近图上的注意力消息传递，不是简单枚举所有路径。

## 代码实现

工程实现通常分四步：构图、实体对齐、邻域扩展、候选打分。

实体对齐，是把业务系统里的商品 ID、内容 ID、品牌 ID 和 KG 里的实体 ID 对上。例如商品表里的 `sku_123` 要能映射到图里的 `iPhone 15 case` 实体，否则用户行为无法进入图。

新手版伪代码如下：

```python
# 1. 取用户历史点击
# 2. 从历史物品查 KG 邻域
# 3. 按边权/注意力聚合路径分数
# 4. 对候选物品排序并返回 topK
```

一个可运行的最小 Python 例子：

```python
from collections import defaultdict

KG = {
    "iPhone 15": [
        ("compatible_with", "MagSafe", 0.9),
        ("made_by", "Apple", 0.8),
        ("has_accessory", "Phone Case", 0.7),
    ],
    "MagSafe": [
        ("related_item", "MagSafe Case", 0.8),
        ("related_item", "MagSafe Charger", 0.6),
    ],
    "Apple": [
        ("brand_item", "AirPods", 0.4),
        ("brand_item", "Apple Watch", 0.5),
    ],
    "Phone Case": [
        ("instance", "New iPhone 15 Silicone Case", 0.9),
        ("instance", "Generic Phone Case", 0.3),
    ],
}

ITEMS = {
    "MagSafe Case",
    "MagSafe Charger",
    "AirPods",
    "Apple Watch",
    "New iPhone 15 Silicone Case",
    "Generic Phone Case",
}

def kg_recall(user_history, kg, items, top_k=3, max_hop=2):
    scores = defaultdict(float)
    frontier = [(item, 1.0) for item in user_history]

    for _ in range(max_hop):
        next_frontier = []
        for entity, path_score in frontier:
            for relation, target, weight in kg.get(entity, []):
                new_score = path_score * weight
                if target in items:
                    scores[target] += new_score
                next_frontier.append((target, new_score))
        frontier = next_frontier

    ranked = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    return ranked[:top_k]

result = kg_recall(["iPhone 15"], KG, ITEMS, top_k=3)
assert result[0][0] == "New iPhone 15 Silicone Case"
assert result[0][1] > result[-1][1]
print(result)
```

这个例子没有训练 embedding，也没有注意力网络，只保留 KG 召回最小逻辑：从历史物品出发，查邻居，累积路径分数，输出 topK。

真实系统不会在线临时遍历全图。常见做法是把“离线图处理”和“在线召回”分开：

| 模块 | 离线职责 | 在线职责 |
|---|---|---|
| 离线预计算 | 构建 KG、清洗边、计算多跳邻居、训练 embedding | 不在线构图 |
| 线上查询 | 预先产出 item-to-candidates 或 entity-to-candidates | 根据用户历史快速查候选 |
| 缓存 | 缓存热门实体邻域、item embedding、topN 候选 | 减少图查询和向量检索开销 |
| topK 截断 | 离线控制每个实体最多保留多少候选 | 在线合并、去重、截断 |

线上召回通常只做轻量操作：取用户最近行为，查预计算候选，合并分数，过滤已曝光或不可售物品，返回 topK。

## 工程权衡与常见坑

KG 召回的收益来自结构信息，风险也来自结构信息。KG 噪声、实体错配、未来信息泄漏，会直接污染召回结果，必须优先控制数据质量。

| 问题 | 现象 | 影响 | 规避策略 |
|---|---|---|---|
| KG 噪声大 | 错关系、错属性、低质量自动抽取边 | 候选偏移，召回看似丰富但不相关 | 关系白名单、置信度过滤、人工抽检 |
| 多跳过扩展 | 跳数到 4-5 后候选暴涨 | 噪声放大，召回延迟和排序压力上升 | 常限制 2-3 跳，按边权采样和截断 |
| 训练泄漏 | 把未来才出现的关系边放进训练图 | 模型提前看到答案，离线指标虚高 | 严格按时间切分构图和训练数据 |
| 线上延迟高 | 实时查图、实时扩邻居、候选过多 | P99 延迟不可控 | 离线预计算、缓存、ANN 检索、topK 截断 |
| 把 KGAT 误当纯路径模型 | 只枚举路径，不做消息聚合 | 复现偏差，效果不稳定 | 按图注意力消息传递理解和实现 |

训练泄漏尤其隐蔽。例如某商品在 6 月才被标注为 `爆款配件`，但训练 5 月模型时已经把这条边放进 KG，模型就等于提前知道了未来结果。这不是模型能力强，而是数据切分错了。

多跳扩展也要克制。1 跳候选通常语义很近，2 跳能发现隐式关系，3 跳开始需要强过滤。跳数开到 4-5 后，候选集会迅速膨胀，很多只是弱相关实体。例如 `iPhone 15 -> Apple -> 科技公司 -> 电子产品 -> 智能家居`，后面的候选未必适合当前用户。

还有一个常见误区：把 KG 召回当作排序模型的替代品。召回只需要高覆盖和基本相关，排序才处理细粒度偏好。KG 召回把新手机壳召回来，并不代表它一定要排在第一位。

## 替代方案与适用边界

KG 召回不是通用最优解。只有当图结构足够丰富、实体关系质量较高，并且业务确实需要语义扩展时，它的收益才明显。

新手版解释：如果商品没有稳定的属性和关系，比如大量长尾 UGC 内容，强行上 KG 可能不如用点击共现；如果商品属性很完整，比如电商、内容、知识类平台，KG 更有价值。

| 方案 | 依赖数据 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| 共现召回 | 用户行为日志 | 简单、稳定、可解释 | 冷启动弱，依赖历史行为 | 点击量充足、业务快速迭代 |
| 协同过滤 | 用户-物品交互矩阵 | 能发现相似用户和相似物品 | 稀疏时效果下降 | 中等规模推荐系统 |
| 双塔召回 | 用户特征、物品特征、行为标签 | 在线检索快，可大规模部署 | 训练和特征工程成本较高 | 大规模推荐、广告、内容分发 |
| KG 召回 | 实体、关系、属性、行为数据 | 缓解稀疏和冷启动，可做语义扩展 | 依赖图质量，工程复杂度高 | 电商、影视、音乐、知识内容、医疗知识库 |

如果业务目标强时效，比如新闻流、短视频热点、直播推荐，KG 的静态语义关系可能不如实时行为信号有效。如果物品之间天然关系弱，比如用户随机发布的短文本，构图成本可能高于收益。

更稳妥的工程策略是混合召回：共现召回保证行为相关，双塔召回保证规模化检索，KG 召回补充语义和冷启动候选。最终由排序模型统一比较。
