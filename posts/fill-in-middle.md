## 核心结论

Fill-in-the-Middle，简称 FIM，是一种“补中间”的训练目标：给定前缀 `P` 和后缀 `S`，让模型生成中间缺失段 `M`。白话说，它不是只看左边往后续写，而是先看“前面写了什么”和“后面应该接什么”，再把中间空缺补出来。

这件事对代码模型尤其重要。因为真实编程很少总是“从文件第一行一路往下写”，更常见的是在已有代码中间插入一段逻辑、补一个条件分支、替换一个函数体、修一个局部 bug。普通 next-token 训练目标擅长“继续往后写”，但对“中间插入”并不天然友好；FIM 则直接把这种任务放进训练目标里。

一个新手可以直接记住的结论是：普通续写是在猜“下一个 token”，FIM 是在补“中间缺掉的那块”。这里的 token 可以理解为“模型处理文本时切出来的基本片段”，可能是一个词、半个词，也可能是一个符号。

下面这个对比表可以先建立最基本的直觉：

| 对比项 | 普通 next-token | FIM |
|---|---|---|
| 输入上下文 | 只看左侧前缀 | 同时看左侧前缀和右侧后缀 |
| 目标 | 预测下一个 token | 预测中间缺失段 |
| 擅长任务 | 从头续写、长文本延展 | 代码插入、局部编辑、缺口补全 |
| 训练视角 | 左到右生成 | 条件化的中间生成 |
| 典型场景 | 写完整函数 | 已有函数签名和结尾，补中间函数体 |

玩具例子最容易理解。假设原序列是 `[1,2,3,4,5]`，把中间的 `3` 挖掉。普通续写是给 `[1,2]`，让模型继续猜后面；FIM 是给 `[1,2]` 和 `[4,5]`，要求模型补回 `[3]`。两者目标不同，模型学到的能力也不同。

真实工程例子更贴近代码编辑。比如用户已经写好：

```python
def clamp(x, low, high):
    # 中间逻辑缺失
    return x
```

如果模型只会左到右续写，它更像在“继续写注释之后的文本”。但 FIM 模型可以同时看到函数签名和后面的 `return x`，于是更容易补出中间逻辑，例如先约束边界，再返回结果。这就是 Code Llama、DeepSeek-Coder 这类代码模型大量采用 FIM 的原因之一。

## 问题定义与边界

形式化地看，原始序列记作 $X=(x_1,\dots,x_n)$。我们从中选两个切点，把它拆成三段：

- 前缀 `P`：左侧已知上下文，也就是缺口前面的内容
- 中间段 `M`：要预测的缺失部分
- 后缀 `S`：右侧已知上下文，也就是缺口后面的内容

白话说，就是把一整段内容挖出一个洞，洞左边叫前缀，洞本身叫中间段，洞右边叫后缀。

最小示例：

- 原序列：`[1,2,3,4,5]`
- 前缀 `P=[1,2]`
- 中间段 `M=[3]`
- 后缀 `S=[4,5]`

这就是“把中间的 3 挖掉，让模型同时看见前面的 1、2 和后面的 4、5，再补 3”。

切分关系可以用表格固定下来：

| 角色 | 内容 |
|---|---|
| 前缀 `P` | 已知的左侧上下文 |
| 中间段 `M` | 需要生成的缺失部分 |
| 后缀 `S` | 已知的右侧上下文 |

FIM 的边界也要说清楚。

第一，FIM 的前提是推理时真的能同时拿到 `P` 和 `S`。如果你只有前缀、没有后缀，那它就退化成普通续写。也就是说，FIM 不是神奇地让模型“自动知道后面会是什么”，而是训练它在“后面已知”的条件下更好地补中间。

第二，FIM 解决的是“局部补全”问题，不等于“任何生成任务都更强”。如果任务是从空白文件开始生成一整段长代码，那么右侧后缀往往根本不存在，此时 FIM 的优势会下降。

第三，`P`、`M`、`S` 是基于 token 序列切分，不一定刚好对应一行代码、一个语句块或一个函数。这个细节很重要，因为切得太碎会破坏语法结构，后面会讲到工程后果。

---

## 核心机制与推导

FIM 的训练目标是：在给定前缀 `P` 和后缀 `S` 的条件下，自回归生成中间段 `M`。自回归的意思是“一个 token 一个 token 地往外吐，每一步依赖前面已经生成出的内容”。

对应的损失函数通常写成：

\[
\mathcal{L}_{FIM}(\theta) = -\sum_{t=1}^{|M|}\log p_\theta(m_t \mid P, S, m_{<t})
\]

这行公式的白话解释是：模型在生成中间段的第 $t$ 个 token 时，条件不是只有左边内容，而是“前缀 `P` + 后缀 `S` + 已经生成出的中间前半段 `m_{<t}`”。然后把每一步预测正确 token 的对数概率加起来，取负号，作为训练损失。概率越高，损失越小；概率越低，损失越大。

继续用玩具例子说明。假设：

- `P=[1,2]`
- `M=[3]`
- `S=[4,5]`

模型看到 `P` 和 `S` 后，需要预测 `M` 的唯一一个元素 `3`。如果模型给出：

$$
p_\theta(3 \mid P,S)=0.9
$$

那么这一个位置的损失就是：

$$
-\log 0.9
$$

如果它把 `2` 或 `4` 的概率打得更高，损失就会更大。训练过程的本质，就是反复调整参数 $\theta$，让模型更擅长在左右文约束下补出中间正确内容。

一个关键点是：FIM 通常不要求改 Transformer 结构。Transformer 可以理解为“处理序列上下文关系的主干网络”。很多实现里，变化主要发生在训练样本的构造方式，而不是重新发明一套网络。也就是：

1. 从原始代码样本里随机切出 `P/M/S`
2. 用特殊标记把三段重新拼成模型认识的输入格式
3. 让模型只对 `M` 这段承担主要预测目标

所以 FIM 更像“训练数据重排 + 目标重定义”，而不是“模型架构彻底换代”。

真实工程里，这个设计非常实用。IDE 代码补全时，用户常常已经写了函数头、后面的调用方式、甚至 return 语句，只缺中间实现。FIM 训练过的模型更容易理解“后面必须对齐什么语义”，因此补出来的代码更符合局部上下文，而不是只会从左往右瞎延长。

## 代码实现

实现 FIM 的核心通常不是训练一个全新模型，而是把已有代码样本改造成 infill 格式。infill 可以理解为“带洞填空格式”，通过特殊 token 告诉模型哪里是前缀、哪里是缺口、哪里是后缀。

不同模型的格式不同，这一点必须严格匹配。

- Code Llama 一类实现常见为“前缀 + 后缀 + 中间”的重排思路
- DeepSeek-Coder 一类实现会显式使用 `<｜fim▁begin｜>`、`<｜fim▁hole｜>`、`<｜fim▁end｜>` 这样的特殊 token

新手可以把它理解成：先给模型贴好标签，明确告诉它“这里开始是一段 FIM 样本，这里是洞，这里是右侧上下文，这里该开始补答案”。

下面给一个可运行的 Python 玩具实现，展示“随机切点 + 重新拼接 + 校验”的基本流程：

```python
import random

FIM_BEGIN = "<FIM_BEGIN>"
FIM_HOLE = "<FIM_HOLE>"
FIM_END = "<FIM_END>"

def build_fim_sample(tokens, i, j):
    assert 0 <= i < j <= len(tokens)
    P = tokens[:i]
    M = tokens[i:j]
    S = tokens[j:]

    # 一种常见拼接思路：BEGIN + P + HOLE + S + END + M
    input_tokens = [FIM_BEGIN] + P + [FIM_HOLE] + S + [FIM_END] + M
    return P, M, S, input_tokens

def demo():
    tokens = ["def", "add", "(", "a", ",", "b", ")", ":", "return", "a", "+", "b"]
    i, j = 8, 12  # 把 return a + b 作为中间段
    P, M, S, input_tokens = build_fim_sample(tokens, i, j)

    assert P == ["def", "add", "(", "a", ",", "b", ")", ":"]
    assert M == ["return", "a", "+", "b"]
    assert S == []
    assert input_tokens[0] == FIM_BEGIN
    assert FIM_HOLE in input_tokens
    assert FIM_END in input_tokens
    assert input_tokens[-4:] == M

demo()
print("fim sample ok")
```

上面这个例子只是演示格式，不涉及真正训练。真实训练时，流程大致如下：

1. 把代码 tokenize
2. 采样切点 `i, j`
3. 构造 `P/M/S`
4. 用目标模型要求的 FIM token 重排输入
5. 根据实现决定 labels 如何对齐，通常只对 `M` 计算 loss

再给一个更贴近工程的例子。原始代码是：

```python
def normalize_score(x, low, high):
    # 缺口
    return x
```

可以把“缺口”当作 `M`，函数签名当作 `P`，`return x` 当作 `S`。模型训练时会看到类似“左边是规范化函数，右边最终返回 `x`”，于是更容易补出中间逻辑，比如边界检查、线性缩放、异常处理。对 IDE 来说，这比纯续写更接近真实使用方式。

伪代码可以概括为：

```python
# x 是 token 序列
# 只演示构造逻辑，不绑定具体模型
x = tokenize(code)
i, j = sample_cut_points(x)

P = x[:i]
M = x[i:j]
S = x[j:]

input_ids = [FIM_BEGIN] + P + [FIM_HOLE] + S + [FIM_END] + M
labels = make_labels_for_middle_only(input_ids, M)

assert len(input_ids) == len(labels)
```

真正落地时，最容易出错的不是“采样切点”本身，而是“你以为自己用了 FIM，实际 token 格式和模型预训练时的不一致”。一旦格式错位，模型并不会自动适配，训练和推理效果都会明显变差。

## 工程权衡与常见坑

FIM 很有价值，但不是比例越高越好。很多代码模型会把 FIM 样本和普通 next-token 样本混合训练。原因很直接：如果只训练“补中间”，模型在“从头往后写完整文件”这类任务上的能力可能下降。工程上常见做法是保留一定比例的普通自回归样本，让两类能力同时存在。

第二个关键权衡是切分策略。代码不是自然语言段落，它有明确的语法单元，比如函数名、括号、缩进、关键字、字符串边界。如果切点刚好把这些结构切碎，模型学到的条件关系会变差。

例如你把下面代码切成非常碎的段：

```python
def f(
```

和

```python
):
```

中间缺口再单独留一个变量名，这种样本虽然形式上合法，但训练信号弱，因为前后文都不构成完整语义单元。白话说，连人都读不顺，模型也更难学顺。

常见坑可以直接列出来：

| 常见坑 | 直接后果 | 规避方法 |
|---|---|---|
| 只给前缀，不给后缀 | 退化成普通续写 | 保证推理时同时提供 `P` 和 `S` |
| FIM 特殊 token 用错 | 格式失效 | 严格按模型官方格式拼接 |
| 切点太碎 | 语法和 token 结构被破坏 | 优先按语义边界切分 |
| 只训 FIM | 左到右能力下降 | 与 next-token 混合训练 |
| 训练/推理上下文不一致 | 长代码补全明显变差 | 统一上下文长度策略 |

还有几个代码场景中特别实际的问题。

第一，缩进和换行。Python 这种依赖缩进的语言，如果把缺口切在缩进敏感位置，模型即使语义对，也可能生成错误层级。训练样本最好尽量保留完整代码块。

第二，token 边界不等于语义边界。分词器可能把一个标识符切成多个子词，但工程上最好优先按“语句块、表达式、函数体”切，而不是只看 token 下标随机乱切。

第三，训练分布与推理分布要接近。如果你训练时大多数样本都是“短前缀 + 短后缀 + 很短中间段”，上线后却拿它去做“长文件中间插入一大段复杂逻辑”，效果通常不会好。

## 替代方案与适用边界

FIM 适合“中间插入”和“局部编辑”，但它不是所有生成任务的默认最优解。判断标准很简单：任务是否天然依赖双侧上下文。

如果用户在 IDE 里补一个函数体、补一个 `if` 分支、修复被删掉的一段代码，这类任务前后文约束都很强，FIM 很合适。因为右侧后缀不是噪声，而是强约束，模型知道自己生成的中间内容最后必须和后面的已有代码接得上。

如果任务是从零开始生成一个新文件，或者先写一个大纲再一路往下展开，这时普通自回归训练往往已经足够。因为右侧后缀根本不存在，FIM 的结构优势发挥不出来。

可以直接对比：

| 场景 | 是否适合 FIM | 原因 |
|---|---|---|
| 代码插入 | 适合 | 需要双侧上下文 |
| 局部重写 | 适合 | 前后文约束很强 |
| 从头生成 | 一般 | 只靠左侧上下文也够用 |
| 长仓库级补全 | 取决于上下文长度 | 对上下文质量要求高 |

与 FIM 对应的替代思路主要有三类。

第一，纯 next-token 自回归。这是最基础的方案，优点是训练和推理都简单，适合从头生成；缺点是对中间插入不天然友好。

第二，编辑模型或指令式重写。也就是直接输入“原代码 + 修改要求”，输出修改后代码。它更像“整体改写”，不一定显式建模 `P/M/S` 三段。

第三，检索增强再生成。检索增强是“先去外部知识库或代码库找相关上下文，再让模型生成”。它解决的是“上下文不够”问题，不等于 FIM，但在仓库级补全里常常和 FIM 配合使用。

因此更准确的说法是：FIM 不是替代所有代码生成技术，而是在“局部编辑、缺口补全、双侧约束强”的任务里，提供更贴近使用场景的训练目标。
