## 核心结论

MMC4 和 OBELICS 的关键价值，不是提供更多“图像 + 一句话”的样本，而是提供“图像嵌在文本流里的网页级样本”。交错数据，白话说，就是图片和文字不是分开的，而是像真实网页那样穿插出现，并且保留前后顺序。对多模态模型来说，这比单张图配单句描述更接近真实输入。

一个最直接的区别是：图文对数据训练模型学“这张图像大概在说什么”，交错数据训练模型学“这张图在整页内容里处于什么位置、和哪段文字有关、和前后图片有没有关系”。如果一个商品页有 3 张图、5 段文字，模型不仅要识别图片内容，还要知道第一张是外观图，第二张是接口细节，第三张是安装示意图，并把它们和对应段落连起来。

这也是为什么 MMC4 和 OBELICS 对 in-context 多模态学习特别重要。in-context，白话说，就是模型不重新训练，只靠当前上下文里的几个图文例子就能临时学会任务。要让模型具备这种能力，训练时就不能只看孤立样本，而要让它见过“多图、多段落、长上下文”的真实排列方式。

| 维度 | 普通图文对数据 | 交错图文数据 |
| --- | --- | --- |
| 基本单位 | 单张图 + 单句/单段文字 | 一个网页或文档中的图文序列 |
| 是否保留上下文 | 通常不保留 | 保留前后文与相对位置 |
| 是否支持多图关系 | 弱 | 强 |
| 训练目标 | 图像描述、检索、单图问答 | 多图理解、页面问答、长上下文推理 |
| 典型问题 | “这张图是什么” | “这张图对应哪一段，和前一张图关系是什么” |

从构建方式看，MMC4 更像“先找到候选图文，再用全局匹配把图放回最合适位置”；OBELICS 更像“把网页按结构清洗后尽量原样保留下来”。两者都服务于交错语料，但一个更强调对齐算法，一个更强调网页抽取和过滤流水线。

## 问题定义与边界

本文讨论的对象，是“从开放网页中抽取、过滤、保留顺序的多模态预训练语料”。重点不在模型结构，而在训练数据长什么样、是怎么构建出来的。边界要先说清楚，否则很容易把交错数据和别的多模态数据混为一谈。

先看一个最小网页例子。假设某个网页包含以下结构：标题、一段导语、一张产品图、第二段说明、一张安装图、右侧广告块、底部导航链接。我们真正想保留的是“标题 → 导语 → 产品图 → 说明 → 安装图”这条主内容链，而不是把两张图随便各配一句话，更不是把广告图也当训练信号喂给模型。

这里会反复用到几个记号：

| 术语 | 含义 | 白话解释 |
| --- | --- | --- |
| $v_i$ | 第 $i$ 张图像 | 页面里的某一张图片 |
| $t_j$ | 第 $j$ 个文本块 | 某一段文字或某一句 |
| $s_{ij}$ | 图像与文本的相似度 | 这张图和这段话有多像 |
| $x_{ij}$ | 是否把 $v_i$ 分配给 $t_j$ | 这张图最后是不是放到这段附近 |
| DOM | Document Object Model | 浏览器眼里的网页结构树 |
| interleaved | 图文交错 | 图片和文字按原顺序穿插出现 |

再看 MMC4 和 OBELICS 的边界差异。

MMC4 的核心问题是：网页抓下来以后，图片和文字不一定已经是可靠配对，怎么把图片插回最合理的文本位置。它大量依赖视觉文本编码器的相似度来做全局分配，因此重点是“对齐”。

OBELICS 的核心问题是：网页原始结构里有大量噪声，比如广告、导航、模板脚注、重复图片，怎样把真正有信息的文本和图片从 HTML/DOM 中抽出来，并按原顺序保留下来。因此重点是“抽取、过滤、去重、保序”。

这和下面几类数据不同：

| 数据类型 | 关注点 | 与交错数据的区别 |
| --- | --- | --- |
| caption 数据 | 单图配描述 | 不保留多图上下文 |
| OCR 文档数据 | 文字识别 | 重点在页面文字，不一定有图文语义绑定 |
| 纯文档数据 | 长文本结构 | 没有视觉信号 |
| 单图图文对 | 图文匹配 | 只有局部对应，缺少页面级关系 |

所以，交错数据不是“更大的 caption 数据”，也不是“带图片的 OCR 页面”。它真正解决的是：模型如何在训练阶段就见过“真实网页里的图文组织方式”。

## 核心机制与推导

MMC4 的代表性思路可以抽象成四步：抽取图像候选、抽取文本块、计算相似度矩阵、做全局分配并重建文本流。这里的“全局分配”是核心，因为如果只做逐图最近邻，多个图片可能会同时抢同一段文字，监督信号就会互相打架。

设网页中有图像集合 $\{v_i\}$ 和文本块集合 $\{t_j\}$。先用视觉编码器 $f_v$ 和文本编码器 $f_t$ 把它们映射到同一个向量空间，相似度定义为：

$$
s_{ij} = \cos(f_v(v_i), f_t(t_j))
$$

其中 $\cos(\cdot,\cdot)$ 是余弦相似度，白话说，就是两个向量方向有多接近。相似度越高，说明这张图越可能属于这段文字。

然后做一个二值分配优化：

$$
\max \sum_{i,j} x_{ij} s_{ij}
$$

满足：

$$
x_{ij} \in \{0,1\}
$$

并且通常要加约束，使每张图最多分配给一个文本块，每个文本块也不要被过多图片占用。这样做的原因很直接：局部最优不一定等于整体最优。

看一个玩具例子。网页有 2 张图 $v_1, v_2$，3 段文本 $t_1, t_2, t_3$，相似度矩阵是：

$$
S=
\begin{bmatrix}
0.244 & 0.318 & 0.277 \\
0.223 & 0.324 & 0.261
\end{bmatrix}
$$

如果对每张图独立做最近邻，$v_1$ 会选 $t_2$，$v_2$ 也会选 $t_2$。这意味着两张图都试图贴到同一段文字附近，数据构造上会冲突。更合理的做法是整体求一个总分最大的组合。此时一个可行解是：

- $v_1 \rightarrow t_3$
- $v_2 \rightarrow t_2$

总分是：

$$
0.277 + 0.324 = 0.601
$$

虽然 $v_1$ 没选自己的局部最高分 $0.318$，但整体组合更稳定，因为它避免了两张图压到同一位置。这就是“全局最优比逐图最近邻更合理”的原因。

MMC4 的价值不只在数学形式，更在输出格式。分配完成后，它不是生成若干独立图文对，而是把图片重新插回文本流，得到类似下面的训练样本：

1. 段落 A
2. 图像 1
3. 段落 B
4. 段落 C
5. 图像 2
6. 段落 D

模型看到的不是碎片，而是一整个交错文档。

OBELICS 的机制则不同。它不强调“重新算哪张图最像哪段话”，而是强调“尽量保留网页原结构，只把明显低质量部分去掉”。因此它更接近一个高质量数据工程流水线：

1. 解析 HTML 和 DOM
2. 识别正文区域、文本节点、`<img>` 节点
3. 过滤广告、导航、页脚、社交按钮、追踪像素
4. 去除重复图、重复段落、模板文本
5. 按原始顺序输出交错文档

如果说 MMC4 的核心是“对齐”，OBELICS 的核心就是“保序 + 清洗”。两者都能生成交错样本，但出发点不同。MMC4 假设“网页原位置不完全可信，需要再对齐”；OBELICS 更相信“网页结构本身有价值，应尽量保留”。

真实工程例子可以看新闻专题页。一个页面可能包含封面图、现场图、统计图、引述段、背景说明、时间线。单图 caption 数据最多教模型“这是一张现场照片”，而交错数据能教模型“统计图对应的是第三段里的伤亡数据，现场图对应导语，时间线图和后文背景解释相关”。这类关系才是页面问答、多图比较、长上下文理解真正需要的监督。

## 代码实现

实现层面可以拆成两条流水线：MMC4 风格的“匹配与重建顺序”，以及 OBELICS 风格的“HTML 清洗与保序抽取”。

先看 MMC4 风格的最小实现。下面的代码不用外部库，直接对一个小相似度矩阵做全局枚举，找出总分最高且不冲突的图文分配。它是玩具实现，但逻辑和真实流水线一致。

```python
from itertools import permutations

def best_alignment(similarity):
    n_img = len(similarity)
    n_txt = len(similarity[0])
    best_score = float("-inf")
    best_assign = None

    for cols in permutations(range(n_txt), n_img):
        score = sum(similarity[i][cols[i]] for i in range(n_img))
        if score > best_score:
            best_score = score
            best_assign = list(enumerate(cols))
    return best_assign, round(best_score, 3)

S = [
    [0.244, 0.318, 0.277],
    [0.223, 0.324, 0.261],
]

assign, score = best_alignment(S)

# 返回 [(0, 1), (1, 2)] 或 [(0, 2), (1, 1)] 取决于是否允许同分近似，
# 这里这个矩阵的全局最优是不冲突分配中的最大值。
assert score == 0.601
assert assign == [(0, 2), (1, 1)]

print(assign, score)
```

这段代码的意义不在性能，而在说明数据构建逻辑：先有相似度矩阵，再做全局约束下的最优分配，最后把结果写回文档顺序。

如果把它扩展成工程流水线，通常会变成下面这样：

```text
输入页面
-> 提取 text_blocks
-> 提取 image_urls
-> 下载并过滤图片
-> 编码得到 image_embeddings / text_embeddings
-> 计算 similarity_matrix
-> 做全局匹配
-> 将图片插入对应文本位置
-> 输出 interleaved_document
```

再看 OBELICS 风格的实现。它更像网页内容抽取器，而不是匹配器。

```python
from bs4 import BeautifulSoup

def extract_interleaved_blocks(html):
    soup = BeautifulSoup(html, "html.parser")
    blocks = []

    for node in soup.body.descendants:
        if getattr(node, "name", None) == "img":
            src = node.get("src", "")
            if "ad" in src or "logo" in src:
                continue
            blocks.append({"type": "image", "value": src})

        elif getattr(node, "name", None) in {"p", "h1", "h2", "li"}:
            text = node.get_text(" ", strip=True)
            if len(text) < 20:
                continue
            blocks.append({"type": "text", "value": text})

    return blocks
```

这段代码也只是简化版，但它准确体现了 OBELICS 路线的重点：解析网页、过滤噪声、保留原始顺序。第一段代码是“配对”，第二段代码是“整理网页”。

工程上常见的数据结构大致如下：

| 名称 | 类型 | 作用 |
| --- | --- | --- |
| `page_html` | 原始字符串 | 网页源码 |
| `text_blocks` | 文本块列表 | 候选段落、标题、列表项 |
| `image_urls` | URL 列表 | 页面中的图片候选 |
| `filtered_blocks` | 过滤后块序列 | 去掉广告、导航后的内容 |
| `aligned_sample` | 对齐结果 | 图像与文本块的匹配关系 |
| `interleaved_document` | 最终样本 | 可直接送入训练的交错文档 |

真实工程里还要补几类异常处理。比如图片下载失败时要不要丢整页，坏链接是否回退到纯文本样本，超长页面是否截断，重复图是否只保留第一次出现，模板段落如“相关阅读”“点击订阅”是否用规则或分类器过滤。没有这些步骤，交错数据很快就会被网页噪声淹没。

## 工程权衡与常见坑

交错数据最大的工程误区，是把它重新压扁成普通图文对。这样做看似更简单，实际上会直接损失页面级监督。模型本来可以学到“第二张图对应第三段说明”，压扁后只剩“图和一句话是否相关”，跨图关系和上下文链路都消失了。

另一个高频错误是只做局部最近邻匹配。上面的玩具例子已经说明，多张图可能同时指向同一段文字。结果就是训练信号冲突：一段文字被迫解释多张不完全相同的图片，模型学到的是模糊、稀释甚至自相矛盾的对应关系。

真实工程中，噪声通常比算法更难处理。以一个新闻页坏样本为例：页面里有 1 张主图、3 张广告图、2 个导航块、1 个站点 logo、若干“上一篇/下一篇”推荐卡片。如果不过滤，模型原本该学的是“主图和正文导语的关系”，结果却被广告图和模板文字主导。

下面这个表最值得在工程里反复检查：

| 问题 | 表现 | 影响 | 规避方式 |
| --- | --- | --- | --- |
| 把交错数据当图文对 | 样本被拆成独立对 | 丢失跨图关系和顺序信息 | 保留 DOM 顺序，输出整页序列 |
| 局部最近邻匹配 | 多图抢同一段 | 监督冲突 | 用全局匹配或加唯一性约束 |
| 广告/logo 污染 | 大量无信息图片进入样本 | 训练噪声高 | URL 关键字过滤、尺寸过滤 |
| 重复图片/重复段落 | 同一内容反复出现 | 数据分布偏斜 | 图像、段落、文档级去重 |
| 长页面过载 | 单页图片和文本过多 | 显存和训练预算浪费 | 截断、抽样、限制每页图片数 |
| 不良内容 | NSFW 或低质页面进入语料 | 安全与质量风险 | NSFW 过滤、站点级黑名单 |
| DOM 结构被破坏 | 抽取后顺序错乱 | 模型学不到真实布局 | 保留节点遍历顺序 |

这些对策里，最关键的不是某一个 fancy 算法，而是“保序”和“多级过滤”。尺寸和长宽比过滤可以去掉追踪像素、小图标、横幅广告；URL 关键字过滤能快速筛掉 `ad`、`banner`、`logo` 一类资源；图像、段落、文档级去重分别对应不同粒度的重复问题；DOM 顺序保留保证最后输出还是“网页流”，不是“随机包”。

还有一个常被忽略的权衡：交错数据不是越长越好。页面特别长时，后半部分可能是评论区、推荐区、脚注、版权说明，它们对视觉语言预训练贡献很小，却会吞掉大量 token 和图片预算。因此工程上常常要做正文提取、块级打分和上限裁剪。

## 替代方案与适用边界

交错网页语料不是通用最优解，它只是更适合某一类任务。理解这一点，比盲目追求“网页级多模态”更重要。

先看几类常见替代方案：

| 方案 | 优点 | 缺点 | 数据成本 | 适用场景 |
| --- | --- | --- | --- | --- |
| 单图 caption | 成本低、样本清晰 | 没有上下文 | 低 | 图像描述、基础识别 |
| 单图图文对 | 检索和对齐效果好 | 多图关系弱 | 中 | 跨模态检索、单图问答 |
| OCR 文档数据 | 擅长读页面文字 | 图文语义绑定弱 | 中到高 | 表单、票据、扫描件 |
| 交错网页语料 | 保留顺序和多图关系 | 清洗复杂、噪声高 | 高 | 页面问答、长上下文、多图理解 |
| 纯网页文本 | 文本长上下文强 | 没有视觉监督 | 低到中 | 文本检索、语言建模 |

一个典型对比例子是“说明书问答”。如果训练数据主要是 caption，模型可能只学会“这是一台设备”“这是安装图”。但如果训练数据是交错式网页或手册页面，它更有机会学到“这张图对应第 3 步安装说明”“第 2 张图和第 4 段都在解释接线顺序”。前者教的是一句话标签，后者教的是整页结构关系。

所以什么时候应该优先考虑 MMC4 或 OBELICS 这类数据？

1. 任务输入天然是多图 + 长文本。
2. 问题依赖图像在上下文中的位置。
3. 需要模型比较多张图、跨段落找依据。
4. 目标是页面级问答、商品页理解、专题页分析、手册或教程理解。

反过来，如果任务只是单图分类、简单描述、轻量级图像问答，交错式预训练未必划算。因为它的数据清洗成本高，训练输入更长，工程复杂度明显大于普通图文对。

再强调一次适用边界。MMC4 和 OBELICS 解决的是“让模型习惯真实网页式输入”，不是“替代所有多模态数据”。在实际训练里，它们往往更适合作为数据配方中的一个组成部分，和 caption、单图图文对、OCR 文档数据一起使用，而不是孤立存在。
