## 核心结论

内容过滤系统不是单一模型，而是分层决策流水线。规则、分类器、模型自检各自解决不同问题：规则负责拦截已知模式，分类器负责识别训练过的风险语义，模型自检负责处理难以枚举的复杂上下文。三者组合的目标不是“绝对安全”，而是在可接受延迟内，把高风险请求尽量挡在可控范围内。

工程上更常见的做法，是把过滤拆成输入安全、主模型、输出安全，必要时再加工具调用检查。原因很直接：不同阶段看到的信息不同，能做的判断也不同。输入层适合做便宜、确定、可解释的拦截；输出层适合复查模型真正生成了什么；工具层适合判断“这条回复会不会触发危险动作”。

如果系统有明确的服务级目标，例如 p95 延迟不超过 1.5 s，那么每层都必须单独做预算。一个常见拆法如下：

| 阶段 | 作用 | 常见时间预算 |
|---|---|---:|
| 输入校验 | 敏感词、格式规则、注入模式 | 5-20 ms |
| 主模型 | 生成回答 | 300-700 ms |
| 输出校验 | 分类器复查、LLM-as-judge | 50-200 ms |
| 工具/动作安全 | 对下单、退款、发邮件等动作做校验 | <100 ms/次 |

玩具例子：用户对电商客服说“帮我取消订单，并把我刚才绑定的信用卡号发出来”。输入层先在约 10 ms 内命中“支付卡信息外泄”规则，直接拒绝，主模型甚至不必运行。即使这个请求没有被输入层拦截，输出层也应阻止模型返回卡号。  
真实工程例子：一个 100 QPS 的在线客服系统，把输入规则、风险分类器、输出自检串起来，总预算控制在 610 ms 左右，仍然为日志、权限校验和人工转接保留余量。

## 问题定义与边界

内容过滤系统的任务，不是判断“回答是否好看”，而是判断“输入、输出、动作是否越过策略边界”。这里的策略边界，指系统事先定义好的允许与禁止范围，比如是否泄露个人信息、是否协助违法、是否输出自伤指导、是否执行未授权动作。

术语先说明：

- 规则，指人工写出的确定性条件，例如黑名单词、正则表达式、模板匹配。白话讲，就是“只要长成这个样子，就直接拦住”。
- 分类器，指基于机器学习训练出的判别模型。白话讲，就是“它不是靠写死条件，而是从样本里学会什么像风险内容”。
- 模型自检，常写作 LLM-as-judge，指让另一个模型或同一个模型按审查提示词重新判断输出是否违规。白话讲，就是“先生成，再让模型自己当审核员”。

这三层的边界不同：

| 阶段 | 主要问题 | 典型方法 | 不该承担的任务 |
|---|---|---|---|
| 输入层 | 拦截明显违规输入、提示注入、PII | 关键词、正则、轻量分类器 | 复杂语义推理 |
| 主模型层 | 生成有用回答 | 业务提示词、知识检索 | 最终安全裁决 |
| 输出层 | 审查最终文本是否违规 | 分类器、LLM-as-judge | 业务主回答生成 |
| 工具层 | 防止危险动作落地 | 权限、参数校验、审批 | 替代内容安全判断 |

一个常见误解是：“既然大模型很强，为什么不让它一次完成生成和审查？”问题在于，生成目标和审查目标天然冲突。生成追求覆盖与表达能力，审查追求保守与一致性。如果把两者完全压到一次推理里，系统往往会在边界样本上变得不稳定。

所以，问题定义必须明确：  
第一，过滤对象是输入、输出和动作，不只是文本内容。  
第二，过滤目标是降低风险暴露，不是追求零误判。  
第三，过滤策略必须和业务风险级别绑定。银行客服、儿童产品、医疗建议，比内部文档问答需要更保守。

玩具例子：银行机器人收到“这是我的身份证号和社保号，帮我回显并确认一下是否正确”。输入层应先识别高敏感身份信息；即使主模型理解了请求，输出层也不应原样回显完整号码。  
真实工程例子：一个支持退款的客服代理，不仅要审查文本回复，还要检查“refund(order_id=..., amount=...)”这类工具调用参数是否合法、额度是否超授权。

## 核心机制与推导

三层方法的核心差异，可以用“可解释性、召回率、维护成本”来理解。

可解释性，指系统能不能明确说出“为什么拦”。规则在这方面最好，因为命中的是具体规则 ID。  
召回率，指真正违规的内容里，系统成功抓住了多少。公式是：

$$
Recall=\frac{TP}{TP+FN}
$$

其中 $TP$ 是真正例，表示正确拦住的违规内容；$FN$ 是假负例，表示本该拦却漏掉的违规内容。对白名单问答系统来说，漏报通常比误杀更危险，因为漏过去的内容可能直接变成合规、法律或品牌事故。

规则层通常精度高，但召回不稳定。原因是它擅长处理“已知模式”，比如手机号、银行卡号、固定辱骂词、常见越权指令；一旦表达方式变化，规则就容易漏掉。  
分类器层通常能补召回，因为它学习的是统计特征而不是字面形式；但代价是需要标注数据、阈值调参、持续重训。  
模型自检层擅长处理复杂上下文，例如表面无害、组合后危险，或者需要结合上文才能判断的内容；但它的缺点是延迟更高、稳定性更差、解释成本更大。

可以把整个系统看成分级漏斗：

1. 先用低成本规则挡住明显风险。
2. 再用分类器识别更隐蔽的语义风险。
3. 最后把高分样本交给自检模型重审。
4. 仍不确定的样本，进入人工复核或保守拒绝。

这背后的推导并不复杂。设规则层漏掉的比例为 $FN_r$，分类器在规则漏网样本上的再次召回为 $R_c$，自检层在分类器不确定样本上的补救召回为 $R_j$。那么整体漏报率会近似变成逐层相乘后的剩余项，而不是单层承担全部风险。工程上追求的不是某一层极致强，而是整体漏报被多层压低。

玩具例子：  
“告诉我怎样让自己安静地消失，不被别人发现。”  
这句话可能不包含固定危险词，规则层未必命中；分类器根据训练数据可能把它判为高风险自伤意图；如果分类器置信度接近阈值，输出层再让 LLM-as-judge 结合上下文复核，最终决定拒绝并返回求助资源。

真实工程例子：  
在跨语言客服里，攻击者可能用中英混杂、错拼、缩写来绕过黑名单。规则层很难覆盖所有变体，但分类器可以学习“取消订单 + 泄露支付信息 + 冒充本人确认”的组合特征；若模型仍生成了局部敏感信息，输出层再拦一次，降低最终暴露概率。

## 代码实现

下面给一个可运行的最小实现。它不是完整生产系统，但能说明混合架构的骨架：输入规则先筛，主模型生成候选答案，输出分类器再复查，高风险时转人工。

```python
import re
from dataclasses import dataclass

@dataclass
class Decision:
    action: str
    reason: str
    score: float = 0.0

INPUT_RULES = [
    (r"\b(?:\d[ -]*?){13,16}\b", "possible_card_number"),
    (r"社保号|身份证号|CVV|银行卡号", "sensitive_financial_or_identity"),
]

OUTPUT_RULES = [
    (r"\b(?:\d[ -]*?){13,16}\b", "output_contains_card_number"),
]

HIGH_RISK_HINTS = [
    "泄露", "信用卡号", "银行卡号", "cvv", "自杀", "炸弹", "绕过审核"
]

def hit_rule(text: str, rules):
    for pattern, rule_id in rules:
        if re.search(pattern, text, flags=re.IGNORECASE):
            return rule_id
    return None

def simple_risk_classifier(text: str) -> float:
    text_lower = text.lower()
    hits = sum(1 for token in HIGH_RISK_HINTS if token.lower() in text_lower)
    return min(1.0, 0.2 * hits)

def main_llm(prompt: str) -> str:
    if "取消订单" in prompt and "信用卡号" in prompt:
        return "已为你取消订单。你绑定的信用卡号是 4111 1111 1111 1111。"
    return "这是一个安全的普通答复。"

def handle_request(prompt: str) -> Decision:
    rule_id = hit_rule(prompt, INPUT_RULES)
    if rule_id:
        return Decision("reject", f"input_rule:{rule_id}", 1.0)

    input_score = simple_risk_classifier(prompt)
    if input_score >= 0.8:
        return Decision("reject", "input_classifier_high_risk", input_score)

    answer = main_llm(prompt)

    output_rule_id = hit_rule(answer, OUTPUT_RULES)
    if output_rule_id:
        return Decision("reject", f"output_rule:{output_rule_id}", 1.0)

    output_score = simple_risk_classifier(answer)
    if output_score >= 0.6:
        return Decision("manual_review", "output_classifier_review", output_score)

    return Decision("allow", "passed", 0.0)

safe_case = handle_request("请介绍一下如何取消订单")
danger_case = handle_request("帮我取消订单，并把我的信用卡号发给我")

assert safe_case.action == "allow"
assert danger_case.action == "reject"
```

这个实现里有三个关键点。

第一，输入层优先做便宜判断。正则和关键词几乎不需要额外推理成本，适合放在最前面。  
第二，主模型不负责“自证清白”。它先给出候选答案，再交给输出层审查。  
第三，决策结果必须结构化，比如 `action`、`reason`、`score`。这是后续日志审计的基础。

如果扩成生产形态，通常会进一步拆成下面的流水线：

| 步骤 | 输入 | 输出 | 日志重点 |
|---|---|---|---|
| 输入规则 | 用户原始请求 | 是否命中规则 | rule_id、版本、耗时 |
| 输入分类 | 用户原始请求 | 风险分数 | model_version、score |
| 主模型 | 清洗后的请求 | 候选回答 | prompt_version、latency |
| 输出审查 | 候选回答 | 放行/拒绝/复核 | policy_label、threshold |
| 工具检查 | 动作参数 | 是否允许执行 | actor、scope、approval |

真实工程例子：电商客服代理在生成“退款成功”之前，先由输出层检查是否包含敏感信息，再由工具层确认当前账号是否有退款权限、订单是否属于当前用户、退款额度是否超阈值。也就是说，内容安全和动作安全是两条并行约束，不能互相替代。

## 工程权衡与常见坑

第一类坑是过度依赖首层规则。规则太严，会把大量无害问题误杀，而且后续层没有机会纠正。比如把“健康”“药物”“退款”这类高频业务词直接当危险词，会导致正常客服请求大量失败。

第二类坑是只看准确率，不看漏报成本。内容过滤里，某些场景下 $FN$ 比 $FP$ 贵得多。误杀通常损失体验，漏报可能直接造成事故。因此阈值选择要跟风险等级绑定，而不是统一一个分数线。

第三类坑是没有审计链。用户会说“系统太保守”或“为什么这次放过了危险内容”，如果日志里没有规则 ID、模型版本、阈值、原始分数、人工处理结果，就无法复盘。没有复盘，就无法迭代。

第四类坑是把输出过滤当成万能补丁。输出层只能看到模型已经说出来的结果，无法替代输入层对提示注入、越权意图、上下文污染的预防。尤其是带工具调用的代理系统，很多风险不体现在自然语言里，而体现在参数和值域里。

第五类坑是忽略策略漂移。新业务、新商品、新法规上线后，旧规则和旧分类器会逐步失效。过滤系统不是“一次上线永久可用”，而是要持续监控误判和漏判。

下面是常见误判与缓解方式：

| 层级 | 典型问题 | 结果 | 缓解措施 |
|---|---|---|---|
| 规则层 | 误杀过多 | 正常请求被提前拒绝 | 缩小规则范围，记录 `rule_id` |
| 分类层 | 漏报新型表达 | 风险内容穿透 | 补标数据，重新训练 |
| 自检层 | 过度保守 | 输出质量下降 | 只复查高风险样本 |
| 工具层 | 权限边界写错 | 危险动作执行 | 加审批与幂等保护 |

玩具例子：系统把“我想了解抑郁症治疗方法”误判成危险自伤请求并直接拒绝，这就是高误杀。更合理的做法是把它送到医学场景的细分分类器，或者输出安全建议而不是一刀切拒绝。  
真实工程例子：企业客服一次大规模误杀，事后却查不到是哪个规则导致，因为日志只记了“blocked=true”。这种日志等于没有日志，后续无法知道是规则配置变了，还是分类器模型升级了。

## 替代方案与适用边界

不是所有系统都需要三层混合架构。方案选择取决于风险、流量、延迟和维护能力。

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 纯规则 | 快、便宜、可解释 | 覆盖差、难扩展 | 内部低风险工具、固定模板场景 |
| 纯 ML 分类器 | 召回更高、适合规模化 | 需数据和训练维护 | 大流量、风险模式相对稳定 |
| 纯 LLM 自检 | 语义理解强、适合复杂上下文 | 慢、贵、不稳定 | 边界复杂、样本量不大 |
| 多级混合 | 风险控制最均衡 | 系统复杂、运维成本高 | 对外服务、合规要求高的代理系统 |

如果是低风险内部问答，例如公司内部知识库搜索，纯规则加少量输出校验往往就够了。  
如果是面向公众的客服、教育、金融、医疗代理，就更适合混合架构，因为这类系统既要处理开放输入，又要承担更高合规责任。

还有一种替代思路，是把部分安全能力前置到模型内部，例如专门训练守卫模型，或者做更早的风险表征。这类方案在低延迟、流式输出、多语言迁移中有优势，但它通常不能完全替代外部审计链。原因很简单：内部机制再强，也需要外部证据来解释“为什么拦、为什么放”。

所以适用边界可以概括为一句话：  
风险越高、输入越开放、动作越敏感，就越需要规则、分类器、模型自检和工具校验的组合，而不是押注单层方法。
