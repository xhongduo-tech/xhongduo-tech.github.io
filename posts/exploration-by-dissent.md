## 核心结论

集成不确定性估计的核心做法是：不用一个 Q 网络或策略网络，而是同时训练多个成员，让它们对同一个状态或动作分别给出判断，再用这些判断之间的分歧来近似“不确定性”。这里的“不确定性”主要指认知不确定性，也就是模型因为见得不够多、学得不够稳而产生的犹豫。

对强化学习来说，这件事的价值很直接。单模型容易出现“自信地错”，它给出一个很高或很低的值，但你并不知道这个值到底是因为真的学会了，还是因为函数逼近误差碰巧把它推高了。集成方法把“值有多高”与“模型有多确定”分开了：平均值描述利用价值，分歧描述探索价值。于是可以把动作打分写成

$$
\mu(s,a)=\frac1K\sum_{k=1}^{K}Q_k(s,a),\qquad
\sigma^2(s,a)=\frac1K\sum_{k=1}^{K}(Q_k(s,a)-\mu(s,a))^2
$$

再构造一个偏探索的上置信界分数：

$$
u(s,a)=\mu(s,a)+\beta \sigma(s,a)
$$

其中 $\beta$ 是探索强度系数，白话说就是“你愿意为不确定性多付多少分”。

一个玩具例子就够说明问题。假设 3 个 Q head 对动作 A 的输出是 $[10,12,14]$，对动作 B 的输出是 $[13,13,13]$。A 的平均值是 12，B 的平均值是 13，看起来 B 更好；但 A 的分歧更大，说明模型对 A 还没学稳。如果取 $\beta=1$，A 的 UCB 分数可能反而更高，于是策略会先去试 A。这就是“Exploration by Dissent”的字面含义：因为成员意见不一致，所以值得探索。

| 动作 | 成员输出 | 平均值 $\mu$ | 标准差 $\sigma$ | UCB 分数 $\mu+\beta\sigma$（$\beta=1$） | 倾向 |
|---|---|---:|---:|---:|---|
| A | [10, 12, 14] | 12 | 1.63 | 13.63 | 先探索 |
| B | [13, 13, 13] | 13 | 0 | 13 | 先利用 |

一句话概括：分歧越大，通常越值得探索；分歧越小，通常越适合利用。

## 问题定义与边界

这类方法要解决的不是“怎样把 Q 值拟合得更准”这个单点问题，而是“怎样知道自己哪里还不会”。强化学习的难点常常不在优化器，而在数据分布。你只能学到自己去过的状态，没去过的状态没有监督信号；如果策略一直待在熟悉区域，值函数会在陌生区域长期空白，进而形成探索不足。

这里需要先区分两类不确定性。

| 类型 | 定义 | 白话解释 | 能否通过更多数据降低 |
|---|---|---|---|
| Epistemic uncertainty | 认知不确定性 | 模型没见过、没学会 | 通常可以 |
| Aleatoric uncertainty | 随机不确定性 | 环境本身就有噪声 | 通常不行 |

集成分歧主要想刻画第一类，也就是 epistemic uncertainty。比如机器人第一次看到某种很偏的抓取姿态时，不同成员可能给出完全不同的成功率判断，这说明模型对这个区域缺经验；如果它已经在标准姿态上看过很多数据，多个成员的输出就会更一致。

但边界也很重要。若环境本身强随机，比如同一个状态动作在不同时间得到的回报差异就很大，那么成员分歧变大不一定代表“没学会”，也可能只是这个环境天生吵。此时如果机械地把高方差当成高探索价值，就会把噪声误判成知识空白。

本文讨论的边界是：用 Q 集成或策略集成的分歧，构造探索信号。它不是完整的贝叶斯强化学习推导，也不保证方差就是严格校准后的后验不确定性。工程上我们把它当成一个有效近似。

常见的分歧指标除了方差，还有最大分歧：

$$
d_{\max}(s,a)=\max_{i,j}|Q_i(s,a)-Q_j(s,a)|
$$

它的直觉更粗暴：只要成员里有人强烈不同意，这个点就可能值得看一眼。方差更平滑，最大分歧更敏感，二者都在工程里常见。

## 核心机制与推导

底层逻辑可以拆成三步。

第一步，训练多个成员。成员可以是完全独立的网络，也可以是共享 backbone、只在最后几层分叉的多 head 结构。backbone 是共享特征提取器，白话说就是“前面几层大家共用，后面每个成员自己给答案”。为了让分歧有意义，这些成员不能完全看相同数据、走相同初始化、用完全同步的更新路径，否则它们很快会塌缩成同一个模型。

第二步，计算分歧。对每个状态动作对 $(s,a)$，都拿到 $K$ 个输出 $Q_1,\dots,Q_K$，计算均值、方差或最大分歧。均值代表利用价值，分歧代表探索收益。于是就能把探索融入动作选择、奖励塑形或采样优先级。

第三步，把分歧注入决策。最常见有三种：

| 用法 | 公式 | 作用 |
|---|---|---|
| UCB 选动作 | $u(s,a)=\mu(s,a)+\beta \sigma(s,a)$ | 高值且高不确定的动作更容易被选中 |
| 内在奖励 | $r_{\text{int}}(s,a)=\alpha \sigma(s,a)$ | 把“陌生感”直接加到奖励里 |
| 最大分歧筛选 | $d_{\max}(s,a)$ | 快速找到成员最不一致的区域 |

为什么这套逻辑能工作？因为强化学习里的探索问题，本质上是“信息价值”问题。一个动作除了可能带来外部奖励，还可能带来对环境的新认知。集成分歧相当于给“新认知的潜在价值”做了一个近似分数。它不是在问“这个动作现在值不值”，而是在问“试了这个动作后，我会不会学到更多”。

玩具例子可以再推一次。动作 A 输出 $[10,12,14]$，动作 B 输出 $[13,13,13]$。

$$
\mu_A=12,\quad \sigma_A=\sqrt{\frac{(10-12)^2+(12-12)^2+(14-12)^2}{3}}\approx1.63
$$

$$
\mu_B=13,\quad \sigma_B=0
$$

若 $\beta=1$，则

$$
u_A\approx13.63,\qquad u_B=13
$$

因此系统优先尝试 A，不是因为 A 已经更优，而是因为 A 可能隐藏着尚未确认的高价值区域。这个机制与多臂老虎机里的 UCB 思路一致，只不过这里的置信度不再来自解析统计量，而是来自深度网络集成的经验分歧。

真实工程例子更能看出价值。以机器人抓取为例，训练初期模型在“桌面中央、无遮挡、标准姿态”的样本上很快收敛，因为这些样本常见且回报密集；但在“边缘摆放、半遮挡、斜角抓取”这些长尾状态上，单模型往往要么盲目保守，要么胡乱高估。若使用集成 Q 网络，系统会发现这些长尾状态对应的成员方差更高，于是优先把采样预算投到这些区域。这种做法不是随机乱试，而是有针对性地去补知识空白。

## 代码实现

工程上最常见的结构是“共享 backbone + 多个 Q head”。它的原因很现实：完全独立的 $K$ 个网络更干净，但计算和显存成本都更高；共享 backbone 会引入成员相关性，但通常能把成本控制在可接受范围内。

下面先给一个最小可运行的 Python 例子，演示如何根据集成输出计算均值、标准差和 UCB 分数，并验证上面的玩具例子。

```python
import math

def mean(xs):
    return sum(xs) / len(xs)

def variance(xs):
    m = mean(xs)
    return sum((x - m) ** 2 for x in xs) / len(xs)

def std(xs):
    return math.sqrt(variance(xs))

def ucb_score(q_values, beta=1.0):
    m = mean(q_values)
    s = std(q_values)
    return m + beta * s

action_a = [10.0, 12.0, 14.0]
action_b = [13.0, 13.0, 13.0]

mu_a = mean(action_a)
mu_b = mean(action_b)
std_a = std(action_a)
std_b = std(action_b)

ucb_a = ucb_score(action_a, beta=1.0)
ucb_b = ucb_score(action_b, beta=1.0)

assert mu_a == 12.0
assert mu_b == 13.0
assert round(std_a, 2) == 1.63
assert std_b == 0.0
assert ucb_a > ucb_b

print("A:", mu_a, round(std_a, 2), round(ucb_a, 2))
print("B:", mu_b, round(std_b, 2), round(ucb_b, 2))
```

如果你在 PyTorch 里写多 head Q 网络，推理逻辑通常类似下面这样。`q_values` 的形状是 `[K, batch_size, num_actions]`，也就是每个成员都对每个动作给一个值。

```python
import torch

def select_action_with_ucb(q_values, beta: float):
    # q_values: [K, B, A]
    mean_q = q_values.mean(dim=0)                       # [B, A]
    std_q = q_values.std(dim=0, unbiased=False)        # [B, A]
    ucb_q = mean_q + beta * std_q                      # [B, A]
    action = ucb_q.argmax(dim=-1)                      # [B]
    return action, mean_q, std_q, ucb_q
```

仅有多 head 还不够，关键在于让成员看到不同的数据子集。Bootstrap DQN 的核心思想就是 bootstrap sample，也就是“有放回抽样形成的不同训练集”；深度实现里常见简化做法是 Bernoulli mask：每条 transition 对每个 head 掷一次硬币，决定它是否参与该 head 的更新。

```python
import torch

def bootstrap_mask(num_heads: int, p: float = 0.8):
    probs = torch.full((num_heads,), p)
    return torch.bernoulli(probs).bool()

mask = bootstrap_mask(num_heads=5, p=0.8)
assert mask.shape == (5,)
assert mask.dtype == torch.bool
```

一个简化训练循环可以概括为：

1. 从 replay buffer 采样 batch。
2. 为每条样本、每个 head 生成 bootstrap mask。
3. 每个 head 只在自己被分配到的样本上计算 TD loss。
4. 共享参数部分联合更新，head 参数各自更新。
5. 推理时对所有 head 输出求均值与方差，再做 UCB 选动作或构造内在奖励。

实现方案可以这样比较：

| 方案 | 特点 | 优点 | 代价 |
|---|---|---|---|
| 完全独立网络 | 每个成员整网独立 | 分歧最真实 | 最贵 |
| 共享 backbone + 多 head | 前面共享，后面分叉 | 最常用，性价比高 | 成员相关性更强 |
| 单网络 + dropout 近似 | 通过随机失活制造差异 | 实现轻量 | 不确定性质量通常较弱 |

如果是策略梯度或 actor-critic，思想也一样。你可以集成 critic 来指导 UCB 样式探索，也可以集成 dynamics model 或 policy 输出，把成员分歧作为探索信号。形式不同，核心不变：分歧代表尚未确认。

## 工程权衡与常见坑

第一类坑是把噪声当成不确定性。环境如果本身有高随机回报，成员可能全都在该区域学得很难，但那不等于多试几次就会消失。比如广告投放、推荐系统、市场反馈这类场景，外界扰动很强，单看 ensemble variance 容易把 aleatoric 噪声误当成可学习空白。修复思路通常是结合更长时间尺度的统计、分离 reward noise，或者只把分歧作为辅助信号而非唯一决策依据。

第二类坑是成员不独立。若所有 head 同初始化、同 batch、同 target、同优化步，它们很快就会高度相关，最后方差接近 0，但这不是因为学稳了，而是因为大家被你训练成了同一个人。Bootstrap mask、不同随机种子、不同数据顺序、不同 target network 更新时机，都能帮助保留成员差异。

第三类坑是探索过猛。公式

$$
u(s,a)=\mu(s,a)+\beta\sigma(s,a)
$$

里的 $\beta$ 决定了你对不确定性的偏好。$\beta$ 太小，分歧形同虚设；$\beta$ 太大，策略会长期追逐高方差区域，即使那些区域外部回报很差，也会被反复访问。工程上常见修复是让 $\beta$ 随训练退火，或者对 $\sigma$ 做归一化，让早期和后期的尺度更稳定。

第四类坑是计算开销。$K$ 越大，推理和训练都越贵。真实工程里常见做法不是盲目堆大集成，而是用 5 到 10 个成员的小规模 ensemble，搭配共享 backbone 和混合精度训练。因为超过某个点后，额外成员带来的不确定性收益会递减，但算力成本几乎线性上涨。

第五类坑出现在离线强化学习。离线 RL 只有固定数据集，数据之外的动作往往既高不确定，又不可行。此时若用 UCB 偏向高方差动作，模型反而更容易被 OOD 区域误导。高不确定在这里不是“该探索”，而可能是“别碰”。因此离线场景常要结合保守约束、行为策略约束或 OOD 过滤。

| 问题 | 现象 | 后果 | 常见修复 |
|---|---|---|---|
| 噪声误判 | 高方差区域反复被访问 | 训练效率下降 | 区分 epistemic 与 aleatoric |
| 成员塌缩 | head 输出高度一致 | 不确定性失效 | bootstrap、不同 seed |
| $\beta$ 过大 | 一直追高方差 | 利用不足 | 退火或归一化 |
| 成本过高 | 显存和时延上升 | 难上线 | 小集成、共享 backbone |
| 离线误用 | OOD 动作被高估 | 策略崩坏 | 保守约束、OOD 过滤 |

真实工程例子可以看自动驾驶或机器人导航。系统如果把每个“高方差路况”都当成必须尝试的目标，可能会过度进入边缘状态，导致收敛变慢，甚至在安全约束下根本不可执行。因此高风险任务里，集成不确定性更适合作为“优先评估信号”，而不是无条件动作驱动器。

## 替代方案与适用边界

集成方法很强，但不是探索的默认答案。它适合那些“需要知道自己哪里不会”的任务，尤其是稀疏奖励、长时序依赖、长尾状态多、单步随机探索效率很差的场景。比如机器人抓取、长链条导航、复杂控制、自动决策系统，这些任务都容易因为探索不足而停在局部最优。

如果任务本身奖励密集、状态空间小、随机试错成本低，那么更轻量的方法通常更划算。

| 方法 | 核心思路 | 适合场景 | 局限 |
|---|---|---|---|
| $\epsilon$-greedy | 直接随机选动作 | 小规模、奖励密集任务 | 不区分哪里值得探索 |
| Entropy bonus | 让策略分布更随机 | 策略梯度方法 | 随机不等于有信息量 |
| Parameter noise | 扰动参数而非动作 | 连续控制 | 不确定性表达较弱 |
| Ensemble/UCB | 用分歧引导探索 | 稀疏奖励、长链条任务 | 成本更高 |

一个新手可理解的判断标准是：如果你只是玩一个简单环境，走错一步成本不高，而且随机试几次很快就能覆盖全局，那么普通 $\epsilon$-greedy 往往够用；如果你需要连续很多步都走对，才能到达一个有奖励的新区域，那么“盲随机”通常不够，这时分歧驱动的深度探索更有价值。

适用边界还包括风险约束。集成分歧高，只能说明模型没把握，不自动等价于“值得执行”。在真实世界系统里，策略往往还要同时满足安全约束、成本约束、行为先验。尤其在离线 RL、高风险控制、医疗或金融决策里，探索不再是随便去试，而是受控地评估和筛选。

因此更准确的说法是：ensemble disagreement 是一个很强的认知不确定性代理信号，但它应当与任务风险、数据来源和部署约束一起使用，而不是单独决定一切。
