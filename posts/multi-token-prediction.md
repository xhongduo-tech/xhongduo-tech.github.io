## 核心结论

Multi-Token Prediction，简称 MTP，可以直接理解为“同一个上下文，一次监督多个未来位置”。标准自回归语言模型只学习一步预测，也就是给定前缀 $x_{1:t}$，只优化下一个 token 的分布；MTP 则在共享 backbone 上增加多个预测头，同时学习 $t+1,t+2,\dots,t+n$ 这些位置的目标。

它的价值主要有两个。

第一，训练时让模型显式学习短期多步前瞻。这里的“前瞻”指模型不仅关心“下一步是什么”，还要对后面几步的局部结构负责。第二，推理时这些额外头可以作为 speculative decoding 的候选生成器。speculative decoding 可以直译为“推测式解码”，意思是先批量提出若干 token 候选，再由更严格的验证过程决定接受多少个，从而减少严格串行的一步一步生成。

新手可以先看一个玩具例子。输入前缀是 `I like`。标准语言模型通常只猜下一个词，比如大概率是 `tea`。MTP 则会同时判断后面的多个位置，例如第 1 个头预测 `tea`，第 2 个头预测 `with`，第 3 个头预测 `milk`。它不是把一句话一次性完整生成出来，而是在同一个隐藏表示上，对多个未来位置分别做预测。

核心形式可以写成：

$$
p_k(\cdot \mid x_{1:t})=\mathrm{softmax}(W_k h_t),\quad k=1,\dots,n
$$

其中 $h_t$ 是共享上下文表示，$W_k$ 是第 $k$ 个预测头的参数。训练目标通常写成：

$$
\mathcal L=\frac{1}{n}\sum_{k=1}^{n}\mathrm{CE}\big(y_{t+k},\,p_k(\cdot\mid x_{1:t})\big)
$$

这表示：同一个前缀下，多个未来位置的交叉熵一起平均。

| 维度 | 标准 LM | MTP |
|---|---|---|
| 训练目标 | 只预测 $t+1$ | 同时预测 $t+1,\dots,t+n$ |
| 表示来源 | 单个共享 backbone | 同一个共享 backbone |
| 输出头 | 一个输出头 | 多个未来步长输出头 |
| 训练信号 | 一步监督 | 多步监督 |
| 推理默认方式 | 标准自回归 | 通常仍用第 1 头自回归 |
| 潜在加速来源 | 无额外提议能力 | 可为 speculative decoding 提供候选 |

先把结论立住：MTP 的本质不是“模型直接知道更远的答案”，而是“模型被训练去同时关心多个紧邻的未来位置”。

## 问题定义与边界

先定义标准自回归语言模型的边界。它优化的是一步条件概率：

$$
p(x_{t+1}\mid x_{1:t})
$$

也就是说，训练时只要求模型把“下一个 token”猜准。这里的 token 可以理解为模型处理文本的最小离散单位，可能是一个字、一个词片段，或者标点。MTP 做的不是改掉自回归这个大框架，而是把一步监督扩展成多步监督。

共享上下文表示记为：

$$
h_t=f_\theta(x_{1:t})
$$

这句话的含义很关键：所有预测头都共享同一个前缀表示 $h_t$。也就是说，MTP 不是为每个未来步长单独跑一遍完整模型，而是在一个共享表示上挂多个头，各自预测不同距离的未来 token。

看一个边界更清楚的例子。输入是 `The capital of France is`。标准 LM 只要把下一个 token 猜对即可，比如 `Paris`。MTP 则还会同时对更后面的局部结构负责，例如后面更可能接句号、括号中的解释、还是继续补充信息。它扩展的是局部多步预测能力，而不是把“事实检索”“规划推理”“工具调用”这些能力自动补齐。

再看一个工程边界。聊天续写时，MTP 适合做局部连续文本生成，因为它关心的是接下来几个 token 的局部结构是否连贯。但如果任务是“先查数据库，再调工具，再根据结果规划下一步”，那主要瓶颈不是局部 token 预测，而是外部信息获取与全流程决策。MTP 不能替代这些模块。

| 项目 | 定义 |
|---|---|
| 输入 | 前缀 $x_{1:t}$ |
| 输出 | 多个未来 token 分布 |
| 训练 | 多头交叉熵平均 |
| 适用范围 | 局部多步预测、候选提议 |
| 不适用范围 | 直接替代全流程推理、检索、规划 |

因此，问题定义要收紧到一句话：MTP 研究的是“如何让一个共享表示同时承载多个近未来位置的监督”，而不是“如何让模型跳过推理直接输出最终答案”。

## 核心机制与推导

机制本身并不复杂，复杂的是不要把它理解错。

第一步，输入前缀 $x_{1:t}$，backbone 编码得到共享隐藏状态：

$$
h_t=f_\theta(x_{1:t})
$$

这里的 backbone 就是“主干网络”，可以理解为负责抽取上下文语义的主体模型，比如 Transformer 的大部分层。

第二步，第 $k$ 个 head 输出第 $k$ 步未来 token 的分布：

$$
p_k(\cdot \mid x_{1:t})=\mathrm{softmax}(W_k h_t)
$$

这里的 head 可以理解为“接在共享表示后的一个小预测层”，最常见就是线性层加 softmax。不同 head 的差别不在于输入，而在于它们的参数不同、负责的未来步长不同。

第三步，对每个未来位置计算交叉熵，再平均：

$$
\mathcal L=\frac{1}{n}\sum_{k=1}^{n}\mathrm{CE}(y_{t+k},p_k)
$$

交叉熵可以白话理解为“模型分布和真实答案之间的偏差损失”。真实 token 的概率越高，损失越小。

推导链路可以直接写成：

前缀 $x_{1:t}$  
$\downarrow$  
backbone 编码得到 $h_t$  
$\downarrow$  
head 1 预测 $p_1$，head 2 预测 $p_2$，…，head $n$ 预测 $p_n$  
$\downarrow$  
分别与 $y_{t+1},y_{t+2},\dots,y_{t+n}$ 做 CE  
$\downarrow$  
平均得到总损失 $\mathcal L$

符号含义如下。

| 符号 | 含义 |
|---|---|
| $x_{1:t}$ | 输入前缀 |
| $h_t$ | 共享隐藏状态 |
| $W_k$ | 第 $k$ 个预测头参数 |
| $y_{t+k}$ | 第 $k$ 步真实 token |

玩具例子最适合理解监督方式。输入 `I like`，设 $n=3$，真实后续是 `(tea, with, milk)`。这意味着第 1 个头要学会预测 `tea`，第 2 个头要学会预测 `with`，第 3 个头要学会预测 `milk`。模型不是只对第一个词负责，而是同一个上下文表示要承载三步局部未来信息。

如果三个头对正确 token 的概率分别是 $0.7,0.5,0.25$，那么单样本平均损失是：

$$
-\frac{\ln 0.7+\ln 0.5+\ln 0.25}{3}\approx 0.812
$$

这个值的直观意义是：第 1 个头已经比较有把握，第 2 个头一般，第 3 个头明显更难。也正因为如此，远期头通常更难学，收益不会线性增长。

真实工程例子可以看代码补全。编辑器里已经输入：

```text
def add(a, b):
    return
```

标准 LM 只需要把下一个 token，例如空格或 `a` 猜准。MTP 则会对后续几个 token 一起施压，例如 `a`、` +`、` b` 这些局部结构。对代码这种局部规律强、短距离依赖明显的场景，多步监督常常比纯一步监督更容易转化成候选质量提升。

## 代码实现

工程实现通常是在主模型最后一层 hidden state 后面挂多个线性层。每个线性层对应一个未来步长。训练时关键不是“多几个层”本身，而是“目标怎么对齐、哪些位置要 mask、序列尾部怎么处理”。

下面先看一个可运行的最小 Python 例子。它不依赖深度学习框架，只演示 MTP 的损失计算逻辑。

```python
import math

def mtp_average_nll(probs):
    # probs[k] 表示第 k 个未来位置对真实 token 给出的概率
    assert len(probs) > 0
    assert all(0.0 < p <= 1.0 for p in probs)
    return -sum(math.log(p) for p in probs) / len(probs)

toy_probs = [0.7, 0.5, 0.25]
loss = mtp_average_nll(toy_probs)

assert round(loss, 3) == 0.812
assert mtp_average_nll([1.0]) == 0.0
assert mtp_average_nll([0.5, 0.5]) > mtp_average_nll([0.9, 0.9])

print(loss)
```

这个例子只做一件事：验证“多个未来位置的负对数似然取平均”这件事在数值上到底怎么算。

再看实现思路的伪代码：

```python
h = backbone(x_prefix)             # [B, T, D]
logits = []
for k in range(1, n + 1):
    logits_k = head[k](h[:, -1])   # [B, V]，只取最后一个位置
    logits.append(logits_k)

loss = 0
for k in range(1, n + 1):
    loss += cross_entropy(logits[k - 1], target[:, t + k])
loss = loss / n
```

如果是完整序列训练，通常不是只取最后一个位置，而是对所有有效时间步都计算多头损失。此时张量形状大致是：

- `hidden`: `[batch, seq_len, d_model]`
- `head_k logits`: `[batch, seq_len, vocab_size]`
- `n` 个头：`n` 组 logits

实现注意事项如下。

| 项目 | 说明 |
|---|---|
| 训练目标对齐 | 第 $k$ 个头对应 `target[t+k]` |
| 序列末尾 | 超出长度的未来位置必须 mask |
| 显存峰值 | 多头 logits 同时保留会抬高显存 |
| 推理默认 | 通常只用第 1 头做标准自回归 |
| 加速模式 | 配合 speculative decoding 使用 |

这里最容易踩的坑是 target shift，也就是“标签平移”。标准 LM 只要把输入右移一位；MTP 需要为每个 head 分别平移 $k$ 位。如果一个 batch 中某些序列已经到尾部，那么第 2、3、4 个头对应的监督就可能无效，这些位置必须 mask 掉，否则损失会混入假标签。

再看推理。最保守的做法是：训练时用多头，推理时仍然只拿第 1 个头走标准自回归。这么做不一定加速，但可能改善表示学习。若要追求速度，则把多个头当作候选提议器。例如一次先提议 3 个 token，再让验证步骤判断能接受前几个。只有接受的 token 才能真正跳过串行步数。

## 工程权衡与常见坑

最需要纠正的误解是：MTP 不等于天然加速。

它提供的是“多步候选能力”，而不是“无条件跳过验证的权力”。真正把多头预测转成 wall-clock latency，也就是端到端实际耗时下降，通常要依赖 speculative decoding。没有 accept/reject 机制，多头只是多了一些预测结果，不会自动变成更快的最终输出。

看一个简单例子。假设 3 个头一次提议 3 个 token，但验证模型最终只接受了第 1 个 token，那么后两个候选实际上没有带来收益，还额外消耗了提议与验证成本。这就是为什么远期头并非越多越好。

真实工程中，这个问题在不同任务上的表现差异很大。代码补全往往局部规则强，例如括号、缩进、关键字、常见 API 模式都比较稳定，因此短距离未来 token 的接受率更高，MTP 更可能带来收益。反过来，在强推理任务中，未来 token 受中间思考路径影响更大，远期位置更不稳定，接受率通常更低，额外头的价值会下降。

| 误区/风险 | 事实/结果 |
|---|---|
| 加头就等于加速 | 加速依赖 speculative decoding |
| 头数越多越强 | 远期头更难预测 |
| 多头输出可直接拼接 | 必须经过 accept/reject 验证 |
| 训练几乎零成本 | 参数、显存、实现复杂度都会增加 |
| 比较实验直接比效果 | 需要控制参数预算公平比较 |

还要注意参数公平性。加了多个 head，本质上增加了参数量。如果拿“更大参数的 MTP 模型”去和“更小参数的标准 LM”直接比较，结论很容易失真。更公平的做法是控制总预算，例如适当减少 trunk 层数，用节省出来的参数补给额外 heads。这样比较才更接近“训练目标不同”而不是“模型更大”。

实现上还有三个常见坑。

第一，多头输出不要直接拼成最终文本。这样做会导致分布偏移，也就是生成分布不再对应原本想要的目标模型。第二，序列尾部 mask 不能省。否则远期头会学到大量无效信号。第三，训练日志要分 head 监控。只看总 loss 容易掩盖问题，因为第 1 头可能很好，第 4 头已经明显崩掉。

## 替代方案与适用边界

如果目标只是提升推理速度，MTP 不是唯一方案。它只是“在同一个 backbone 上顺手扩出多步候选”的一条路线。其他常见方向还有独立 draft model 的 speculative decoding、早退机制、分层解码，以及更偏系统优化的缓存与并行策略。

标准自回归最简单。它的优点是训练目标直接、实现稳定、推理分布清晰；缺点是一步一步串行，天然受限于 token 级串行依赖。MTP 的优点是可以把多步监督放进训练目标，并在推理时提供候选；缺点是远期头难学、实现更复杂、收益依赖验证接受率。独立 draft model speculative decoding 则更灵活，因为提议器和验证器分离，可以单独设计更便宜的草稿模型，但系统集成复杂度更高。早退或分层解码适合某些结构固定、层级特征明显的场景，但不一定和所有模型结构兼容。

| 方案 | 训练复杂度 | 推理收益 | 实现成本 | 适用场景 |
|---|---|---|---|---|
| 标准自回归 | 低 | 低 | 低 | 通用基线 |
| MTP | 中 | 中到高，依赖接受率 | 中 | 局部连续生成、同 backbone 扩展 |
| 独立 draft model speculative decoding | 中到高 | 中到高 | 高 | 需要更灵活提速路径 |
| 早退/分层解码 | 中 | 视模型而定 | 中到高 | 结构适配明确的系统 |

适用边界可以直接判断：

- 局部连续生成：适合
- 需要提速：适合，但要配合验证
- 强全局推理：谨慎
- 低接受率场景：谨慎

一个新手容易误判的地方是，把“同时预测多个未来 token”理解成“模型更懂整体规划”。这两者不是一回事。MTP 更像“让模型更会提议未来”，不是“直接替代主模型决策”。如果任务核心矛盾在全局计划、外部检索、工具链协作，MTP 只能改善局部候选质量，不能单独解决主问题。
