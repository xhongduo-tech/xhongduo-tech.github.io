## 核心结论

多模态知识图谱的目标，不是“把图像贴到实体旁边当附件”，而是把图像、文本和结构三元组映射到同一个向量空间。向量空间可以理解为“机器内部用来比较语义远近的坐标系”。一旦对齐成功，系统就能同时利用“这个实体和谁有关”与“这个实体长什么样、呈现什么外观”两类信息，完成实体补全、图文互检索和噪声纠错。

新手版可以直接理解成一句话：把“苹果”这条知识图谱既用三元组讲明，又用实拍图展示，再把两者投到一个空间里让“苹果”图像和文字互相靠近，方便图搜文、文搜图。

IKRL 这类方法的关键，不是单独训练一个图像编码器，而是把结构表示和图像表示共同放进统一能量函数里。能量函数可以理解为“判断一个三元组是否合理的打分规则，距离越小越合理”。这样做的结果是，图像不会脱离知识图谱单独学习，而会被结构语义持续约束。TEGAH 之类更偏检索的方法，则进一步用图注意力和多源特征融合，提升大规模跨模态检索时的稳定性。

下面先用一个概括表压缩核心能力：

| 输入模态 | 整合手段 | 输出能力 |
|---|---|---|
| 结构三元组 | TransE/图嵌入等结构编码 | 实体关系推理、补全 |
| 图像 | CNN/视觉编码器得到图像向量 | 外观信息建模、视觉区分 |
| 文本描述 | 文本编码或属性编码 | 名称、属性、说明补充 |
| 图像 + 结构 | 注意力聚合 + 联合能量函数 | 图文对齐、鲁棒实体表示 |
| 图像 + 文本 + 结构 | 共享子空间或图注意力融合 | 跨模态检索、补全、去噪 |

## 问题定义与边界

问题定义可以写得很准确：给定一个知识图谱中的实体 $e$，它可能同时拥有结构信息、文本信息和多张图像，我们希望学习一个统一表示，使这些模态在语义上可比、在训练上可联合、在检索上可互相映射。

这里的“模态”就是信息来源的类型。文本是一种模态，图像也是一种模态，结构三元组同样可以看成一种模态。所谓“模态鸿沟”，就是不同来源的数据虽然描述的是同一个对象，但数值形式完全不同，不能直接比较。

边界必须先讲清楚，否则实现时很容易失控。以电商知识图谱为例，一件商品可能有标题、品牌、颜色、材质等属性三元组，也有 3 到 10 张图片。真正需要定义的问题有三个：

1. 哪些图片参与对齐。主图、细节图、包装图、用户晒单图，信息质量并不一致。
2. 哪些结构属性参与注意力计算。颜色、品类、品牌通常有效，但库存、上架时间这类字段往往不适合作为视觉对齐信号。
3. 对齐目标是什么。是为了补全图谱、做图搜文，还是为了跨模态召回，不同目标对应不同损失函数和评估指标。

边界可以用表格明确：

| 维度 | 可纳入范围 | 不建议混在一起的范围 | 原因 |
|---|---|---|---|
| 图像模态 | 主图、标准实拍图、细节图 | 用户晒单、广告图、拼接海报 | 噪声大，语义不稳定 |
| 文本模态 | 标题、属性、短描述 | 评论、长文案、营销语 | 描述不一致，噪声高 |
| 结构模态 | 品类、品牌、属性关系、上下位关系 | 日志字段、运营字段 | 与视觉语义弱相关 |
| 粒度 | 实体级、属性级 | 句子级和像素级混训 | 对齐成本高，收益未必成正比 |
| 场景 | 实体对齐、跨模态检索、补全 | 实时生成式问答直接混用 | 目标函数不同 |

玩具例子可以先看“苹果”这个词。它既可能表示水果，也可能表示公司。如果知识图谱只有文本名词“苹果”，歧义很大；如果加入图像，红色圆形果实会把它往水果方向拉近，手机或 logo 会把它往公司方向拉近；如果再加入结构三元组，如 `(苹果, 属于, 水果)` 或 `(Apple, 创始人, Steve Jobs)`，歧义基本就能被消掉。这个例子说明，多模态融合不是为了“多放点数据”，而是为了降低单一模态的不确定性。

真实工程例子是电商商品图谱。一个商品实体可能有：
- 标题：`2026 新款轻量跑鞋`
- 属性三元组：`(跑鞋A, 颜色, 白蓝)`、`(跑鞋A, 适用场景, 跑步)`、`(跑鞋A, 品牌, X)`
- 图像：正面图、侧面图、鞋底纹路图

系统要决定哪些图最能代表该商品，哪些属性最能帮助图像对齐，以及检索时是“输入图片找商品文本”，还是“输入商品文本找最像的图”。这些都属于问题边界，而不是模型细节。

## 核心机制与推导

IKRL 的核心思路是：先学习结构表示，再让图像表示在结构约束下聚合，最后把结构和图像一起放进统一能量函数。

“结构表示”可以理解为实体在知识图谱关系网络中的向量。比如 TransE 假设一个合理三元组 $(h,r,t)$ 满足：

$$
h + r \approx t
$$

这里 $h$ 是头实体向量，$r$ 是关系向量，$t$ 是尾实体向量。距离越小，表示三元组越可信。

IKRL 不是只算结构项，而是定义总能量：

$$
E(h,r,t)=E_{SS}+E_{SI}+E_{IS}+E_{II}
$$

其中：

$$
E_{SS}=||h_S+r-t_S||
$$

$$
E_{SI}=||h_S+r-t_I||
$$

$$
E_{IS}=||h_I+r-t_S||
$$

$$
E_{II}=||h_I+r-t_I||
$$

下标 $S$ 表示结构表示，$I$ 表示图像表示。这个式子的意义非常直接：

| 项 | 含义 | 作用 |
|---|---|---|
| $E_{SS}$ | 纯结构到纯结构 | 保留原始知识图谱语义 |
| $E_{SI}$ | 结构头到图像尾 | 约束图像尾实体与结构一致 |
| $E_{IS}$ | 图像头到结构尾 | 约束图像头实体与结构一致 |
| $E_{II}$ | 纯图像到纯图像 | 让视觉信息也能独立参与判断 |

为什么这样设计有效？因为它同时优化“同模态一致性”和“跨模态一致性”。如果只优化 $E_{SS}$，图像不会真正融入图谱；如果只优化图像检索损失，结构语义会被弱化。四项相加，才形成统一约束。

接着是图像聚合。一个实体往往不止一张图，所以不能直接把任意一张图片当实体图像表示。IKRL 的做法是先把每张图片编码成向量 $p_i$，再根据它和结构表示 $e_S$ 的相似度算注意力。注意力可以理解为“这张图在当前实体上有多可信的权重”。

$$
att(p_i, e_S)=\frac{\exp(p_i \cdot e_S)}{\sum_j \exp(p_j \cdot e_S)}
$$

再把所有图片加权求和：

$$
e_I = \sum_i att(p_i,e_S)\, p_i
$$

新手版可以这样理解：先看结构决定哪张图更可信，再按比例混合，生成一个实体级图像表示。

把这个过程画成简化箭头图就是：

`结构向量 e_S -> 与每张图像向量做相似度 -> softmax 得到注意力 -> 加权聚合 -> 图像实体表示 e_I -> 进入统一能量函数`

研究摘要里给了一个最小数值例子，可以直接拿来帮助理解。设实体“苹果”的结构嵌入为：

$$
e_S=[0.5,0.1]
$$

两张候选图像为：

$$
p_1=[0.4,0.2],\quad p_2=[0.1,0.6]
$$

点积相似度分别是：

$$
p_1 \cdot e_S = 0.4 \times 0.5 + 0.2 \times 0.1 = 0.22
$$

$$
p_2 \cdot e_S = 0.1 \times 0.5 + 0.6 \times 0.1 = 0.11
$$

经过 softmax 后，第一张图权重更高，说明它更符合“苹果”当前的结构语义。于是聚合结果更偏向 $p_1$。这意味着，系统不是盲目平均所有图，而是让结构信号参与图像筛选。

真实工程里，TEGAH 这类方法会在这个基础上进一步处理两个问题。第一，单个图片注意力容易受噪声影响；第二，文本和标签本身也可能不一致。图注意力机制可以理解为“在多种节点和边的图结构上，再做一层带权信息传播”，它不只看实体与图像，还看标签、文本、多源特征之间的依赖关系，因此更适合大规模检索场景。

## 代码实现

实现上可以拆成四步：读结构索引、编码图像、按结构计算注意力、用联合损失训练。这里先用一个简化的可运行 Python 例子说明机制，不依赖深度学习框架，也能验证注意力聚合是否成立。

```python
import math

def dot(a, b):
    return sum(x * y for x, y in zip(a, b))

def softmax(xs):
    m = max(xs)
    exps = [math.exp(x - m) for x in xs]
    s = sum(exps)
    return [x / s for x in exps]

def aggregate_image_embedding(struct_emb, image_embs):
    # 按结构向量与每张图像向量的相似度分配权重
    scores = [dot(img, struct_emb) for img in image_embs]
    att = softmax(scores)
    out = [0.0] * len(struct_emb)
    for w, img in zip(att, image_embs):
        for i, v in enumerate(img):
            out[i] += w * v
    return att, out

def l2_distance(a, b):
    return math.sqrt(sum((x - y) ** 2 for x, y in zip(a, b)))

def transe_energy(h, r, t):
    hr = [x + y for x, y in zip(h, r)]
    return l2_distance(hr, t)

def ikrl_energy(h_s, h_i, r, t_s, t_i):
    e_ss = transe_energy(h_s, r, t_s)
    e_si = transe_energy(h_s, r, t_i)
    e_is = transe_energy(h_i, r, t_s)
    e_ii = transe_energy(h_i, r, t_i)
    return e_ss + e_si + e_is + e_ii

# 玩具例子：苹果
e_s = [0.5, 0.1]
p1 = [0.4, 0.2]
p2 = [0.1, 0.6]

att, e_i = aggregate_image_embedding(e_s, [p1, p2])

assert len(att) == 2
assert abs(sum(att) - 1.0) < 1e-9
assert att[0] > att[1]  # 第一张图更接近结构语义

# 构造一个简单关系：苹果 属于 水果
h_s = e_s
h_i = e_i
r = [0.1, 0.0]
t_s = [0.6, 0.1]
t_i = [0.55, 0.15]

energy = ikrl_energy(h_s, h_i, r, t_s, t_i)
assert energy >= 0.0

print("attention:", att)
print("image embedding:", e_i)
print("ikrl energy:", energy)
```

这段代码对应的不是完整训练流程，而是最小可验证单元。它说明三件事：

1. 结构向量可以参与图像选择。
2. 多张图片可以聚合成实体级图像表示。
3. 结构项和图像项可以一起进入统一能量函数。

如果类比本博客仓库的 `posts/posts.json`，可以把它看成“先读结构知识”。索引文件里定义了每篇文章的标题、标签、日期、摘要，相当于实体的结构化属性；真正的文章正文 `.md` 则更像补充模态。在多模态知识图谱里，做法也类似：先读结构索引，再取外部模态补充，但训练时必须回到统一表示空间，而不是把外部模态孤立使用。

真实工程里的伪代码大致如下：

```python
# 1. 读取实体及结构三元组
entities = load_entity_index("entities.json")
triples = load_triples("kg_triples.json")

# 2. 用视觉编码器得到每张图片向量 p_i
# 真实项目可用 ResNet、ViT 或 CLIP visual encoder
image_vectors = {
    entity_id: [vision_encoder(img) for img in images_of(entity_id)]
    for entity_id in entities
}

# 3. 学习结构嵌入 e_S
struct_vectors = train_struct_embeddings(triples)

# 4. 对每个实体，按结构向量对图片做注意力聚合，得到 e_I
image_entity_vectors = {}
for entity_id in entities:
    e_s = struct_vectors[entity_id]
    pics = image_vectors.get(entity_id, [])
    if not pics:
        continue
    att, e_i = aggregate_image_embedding(e_s, pics)
    image_entity_vectors[entity_id] = e_i

# 5. 用 shared-subspace / ranking loss / IKRL energy 联合训练
loss = train_joint_objective(struct_vectors, image_entity_vectors, triples)
```

真实工程例子继续沿用电商商品图谱。假设“某款跑鞋”有 5 张图，其中 2 张是标准主图，1 张是细节图，2 张是营销拼图。结构表示会更偏向“鞋型、颜色、品类”相关视觉特征，因此标准主图和细节图的权重通常更高。营销拼图因为背景复杂、文字遮挡多，注意力应被压低。训练后，用户输入“白蓝轻量跑鞋”，系统不只依赖文本匹配，还能通过共享子空间召回最接近的商品图。

## 工程权衡与常见坑

工程上最容易低估的问题，不是模型结构，而是数据噪声。注意力机制本身并不会自动“聪明”，它只会把训练里学到的偏好放大。如果图片质量差、标签错、文本不一致，注意力就会被带偏。

下面直接给问题与规避对照表：

| 问题 | 规避方式 |
|---|---|
| 图像模糊、背景复杂，注意力错误集中到坏图 | 用 `ATT + AVG` 混合聚合，避免单图支配结果 |
| 图片数量差异大，有的实体 1 张图，有的 20 张图 | 设定上限采样，或按图片类型分组聚合 |
| 文本描述与图片不一致 | 引入文本置信度、标签清洗或图注意力二次融合 |
| 结构属性过多，弱相关字段干扰图像对齐 | 只保留与视觉相关的属性关系做注意力条件 |
| 训练目标只优化结构补全，不优化检索 | 联合使用 ranking loss 和跨模态检索指标 |
| 实体长尾严重，很多实体图片很少 | 对长尾实体退化为结构主导或文本增强方案 |

新手最容易踩的坑，是以为“注意力一定优于平均”。这不对。若商品图很模糊，ATT 恰好把高权重给了那张坏图，整体表示就会偏掉。加一个平均分支或门控补偿，相当于告诉模型：即使某张图看起来很突出，也不要完全忽略其他图。这也是为什么在不少场景里，ATT 比 MAX 稳，而 ATT+AVG 又比纯 ATT 更抗噪。

另一个坑是文本不一致。比如商品标题写“轻量运动鞋”，属性里写“休闲鞋”，图片又更像跑鞋。此时模态间熵会上升。熵这里可以白话理解为“系统面对同一实体时感到不确定的程度”。不一致越多，向量空间里的对齐就越散，检索和补全都会变差。解决办法通常不是单纯换更大的模型，而是增强约束：引入图注意力、做标签层级聚合、对文本进行归一化。

评估也经常做错。跨模态检索不能只看训练 loss，至少要看 `Hit@1`、`Hit@10`、`Recall@K`。`Hit@1` 可以理解为“第一名就命中的比例”，它最能反映用户真实体验。一个模型即使训练 loss 下降，如果 `Hit@1` 没提升，通常说明共享子空间学得并不好。

再补一个真实工程例子。某电商平台做“拍图找同款”，商品知识图谱中鞋子类目图片很多，但服饰类图片背景和模特姿态差异大。结果模型在鞋类检索上表现不错，在服饰上明显变差。原因不是服饰更难，而是服饰图片里的非商品信息更多，注意力被姿态、背景、光照干扰。后来通过“先做人像区域裁剪，再做属性约束聚合”，效果才稳定下来。这说明多模态融合不是纯算法题，前处理和数据治理同样决定上限。

## 替代方案与适用边界

不是所有场景都适合直接上 IKRL。是否使用图像主导融合，关键看图像量、图像质量和任务目标。

如果图像稀少，或者图片和实体关系不稳定，那么结构主导、文本增强通常更合适。比如工业设备图谱中，很多实体没有标准图，只有规格说明书和属性字段，此时强行上图像对齐收益有限，文本和属性更可靠。

如果任务重点是大规模跨模态检索，而不是单纯知识补全，那么 TEGAH 这类带图注意力和多源融合的方法更适合。它的优势不在于“更复杂”，而在于更适合处理标签、文本、图像同时存在且规模较大的场景。

可以用一个简化决策流程来记：

`图像是否充足且质量高？ -> 是 -> 是否需要图文互检索？ -> 是 -> IKRL / TEGAH`
`图像是否充足且质量高？ -> 是 -> 主要做实体补全？ -> 是 -> IKRL 更直接`
`图像是否稀少或噪声大？ -> 是 -> 文本增强 + 结构主导`
`文本是否丰富但图片弱？ -> 是 -> 属性/描述多视角融合`
`数据规模是否很大且检索优先？ -> 是 -> 图注意力 + 多源特征方案`

新手版可以概括成两句：
- 没有图时，就用属性加文本做融合，结构仍然是主骨架。
- 图很多、还要做图搜文时，再考虑 IKRL 或 TEGAH 这种跨模态统一空间方案。

适用边界也要说清楚。IKRL 适合“实体有多张相对可靠图片，且结构图谱本身可用”的场景，例如商品、地标、动植物、医疗影像实体等。它不太适合“图像和实体关系非常松散”的场景，例如社交媒体帖子图谱，因为帖子配图未必对应核心实体语义。TEGAH 更适合检索导向任务，但如果你的主要目标是知识推理而非召回，投入复杂的哈希与图注意力模块未必划算。
