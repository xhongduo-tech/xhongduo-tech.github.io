## 核心结论

指令微调的数据规模，决定效果的不是“样本条数”本身，而是**有效 token**。有效 token 可以理解为“真正给模型提供任务信号的词元”，例如约束条件、字段定义、推理步骤、错误边界，而不是“好的，下面我来回答”这类模板填充。

工程上，这条收益曲线通常可以近似写成：

$$
\text{Gain} \approx a + b \log(T_{\text{eff}})
$$

其中 $T_{\text{eff}}$ 是有效 token 总量。这个式子不是严格自然定律，而是一个很实用的近似：前期增加数据收益很快，后期明显变慢。对应到指令微调，常见现象是：

- 1k 到 10k 条，提升通常最明显。
- 10k 到 50k 条，仍然常有收益，但已经不是“翻倍提升”。
- 超过 50k，如果数据质量、任务多样性和回答长度都没有同步提高，边际收益会迅速下降。

一个对新手更有用的判断是：**10 万条平均 80 token 的短答数据，往往不如 3 万条结构完整、包含约束和较长答案的数据。** 因为前者样本很多，但每条信号弱；后者样本更少，但每条都在教模型“怎么答、答到什么粒度、遵守什么格式”。

先看一个玩具例子。假设你要把基座模型微调成“能回答制造业质检问答”的助手，有三版数据：

| 版本 | 样本数 | 平均回答 token | 估计有效 token 密度 | 示意评测分 |
| --- | ---: | ---: | ---: | ---: |
| A | 1,000 | 220 | 0.72 | 68 |
| B | 10,000 | 120 | 0.51 | 78 |
| C | 60,000 | 80 | 0.27 | 81 |

这个表不是论文原始表，而是工程示意图。它表达的重点只有一个：样本数从 1k 涨到 60k，看起来扩了 60 倍，但模型真正吸收到的“高价值监督信号”并没有按 60 倍增长，所以分数只小幅上升。

再看一个真实工程例子。IgnitionGPT 做燃料性质抽取时，针对结构明确、标签一致的专业任务，在 GPT-4.1-mini 基础上微调，只用全部训练数据中的 10% 就达到 100% 测试准确率，零样本基线约为 47.8%。这说明在专业窄任务里，只要数据覆盖了主要表达变体，小而准的数据可以非常高效。

## 问题定义与边界

这里说的“数据规模”，至少包含两个维度：

| 维度 | 含义 | 常见误判 |
| --- | --- | --- |
| 样本数 | 一共有多少条 instruction-response | 误以为条数越多越好 |
| 有效 token | 真正推动模型学习任务模式的 token 总量 | 忽略回答长度、约束完整性、重复度 |

**边际收益**，就是“再多加一批数据时，模型多提升多少”。在指令微调里，边际收益下降很快，因为基座模型在预训练阶段已经学会了大量通用语言模式；SFT 真正在补的是“任务格式、响应风格、约束遵循、领域术语映射”。

所以，数据规模讨论必须加上边界条件：

- 在**同等质量**下，样本越多通常越好，但收益会饱和。
- 在**质量下降**时，更多样本不一定更好，因为噪声会稀释梯度信号。梯度可以理解为“模型参数朝哪个方向更新”的数值指引。
- 在**任务很窄、输出格式强约束**时，几千条高质量样本可能已经足够。
- 在**任务很广、风格很多、场景很多**时，规模上限会更高，但关键仍然是覆盖面和有效 token，而不是盲目堆条数。

可以把训练曲线想成一条先陡后平的线：前 1k、5k、10k 条数据让损失下降很明显，到了 50k 左右，如果后续新增样本大多只是重复措辞、短答复述或模板变体，曲线就会逐渐变平。

一个边界判断表如下：

| 数据形态 | 可行性 | 原因 |
| --- | --- | --- |
| 高质量小量 | 高 | 信号密度高，容易快速对齐任务格式 |
| 高质量中量 | 很高 | 同时兼顾覆盖面与稳定性 |
| 低质量大量 | 中到低 | 容易把训练预算花在重复模式上 |
| 低质量且大量重复 | 很低 | 有效 token 增长很慢，甚至引入错误偏好 |

## 核心机制与推导

理解这个问题，核心不是“样本有多少条”，而是“每个 token 对学习有没有贡献”。

**token**，就是模型处理文本时的基本单位，可以粗看成词、词片段或符号。  
**token 影响力**，就是某个 token 在训练后，是否让模型更会做当前任务。

Token Cleaning 这条思路给出了一种更细粒度的看法。论文中把 token 分数建立在“模型更新前后损失差”上。损失可以理解为“模型犯错的程度”，越低越好。其核心写法可以整理为：

$$
\mathrm{Infl}(x_{i,j})=\ell(x_{i,j};\theta')-\ell(x_{i,j};\theta)
$$

再把 token 质量分数定义为影响力的相反数：

$$
\mathrm{Score}(x_{i,j})=-\mathrm{Infl}(x_{i,j})
$$

如果把 $\theta$ 视为基座模型，把 $\theta'$ 视为参考模型，那么也可以直观写成：

$$
\Delta_{\text{token}} \approx \mathcal{L}_{\text{base}}-\mathcal{L}_{\text{reference}}
$$

含义很直接：如果参考模型在这个 token 上比基座模型更不容易出错，说明这个 token 更可能带有任务信息。

玩具例子如下。假设一条训练数据是：

“抽取订单字段：客户名、SKU、数量。输出 JSON，不要解释。”

其中不同 token 的信息密度并不一样：

| token 类型 | 例子 | 典型作用 |
| --- | --- | --- |
| 高影响 token | `SKU`、`数量`、`JSON`、`不要解释` | 定义任务目标和输出约束 |
| 低影响 token | `请`、`下面`、`进行` | 语言通顺，但任务信息弱 |
| 低频噪声 token | 无意义口头语、错误标点、脏标签 | 可能干扰训练 |

这就解释了为什么“短答、模板化、重复客套”会拖累性价比。它们占了 token 预算，却没有提供同等强度的任务监督。

把这个机制推到数据规模上，就会得到一个很实用的结论：如果新增 5 万条数据里，大部分只是把同一种短答模板重复很多次，那么新增的不是 5 万条独立信号，而是少量模式的重复曝光。于是：

$$
T_{\text{eff}} \ll T_{\text{raw}}
$$

也就是有效 token 总量远小于原始 token 总量。收益自然不会线性增长。

LIMIT 的结果也支持这个方向：MPT-7B 和 MPT-30B 在 1k 到 60k 的指令集上微调时，1k 到 6k 的小而精数据已经可以取得很强结果；其中一组实验里，MPT-30B 用 1,000 条 Instruct 子集，在 Eval Gauntlet 上能与用 56.2k 全量 Instruct 微调的模型打平。这说明很多场景下，真正稀缺的不是“更多条数”，而是“更好的覆盖与更强的信号”。

一个实用步骤清单可以写成：

1. 先统计每条样本的回答长度、约束密度、字段密度、重复度。
2. 再估计 token 或样本级质量分数。
3. 过滤低分 token 或低分样本。
4. 检查任务多样性是否覆盖主要场景。
5. 最后再决定是扩样本，还是先补长答与约束样本。

## 代码实现

下面给一个可运行的 Python 玩具实现。它不依赖大模型，只演示“如何根据 token 分数过滤低价值 token，并估算有效 token 规模”。这里把 `base_loss` 和 `ref_loss` 当成已经离线统计好的值。

```python
from math import log

samples = [
    {
        "id": 1,
        "tokens": ["抽取", "订单号", "并", "输出", "JSON"],
        "base_loss": [1.8, 2.2, 0.3, 0.7, 1.9],
        "ref_loss":  [1.1, 0.9, 0.2, 0.5, 0.6],
    },
    {
        "id": 2,
        "tokens": ["请", "帮我", "分析", "原因", "并", "给出", "三步", "方案"],
        "base_loss": [0.2, 0.4, 1.0, 1.3, 0.3, 0.6, 1.5, 1.7],
        "ref_loss":  [0.2, 0.3, 0.8, 0.8, 0.3, 0.5, 0.7, 0.9],
    },
]

def score_tokens(tokens, base_loss, ref_loss, threshold=0.4):
    token_scores = {}
    filtered_tokens = []
    for tok, b, r in zip(tokens, base_loss, ref_loss):
        score = b - r  # 近似 Delta_token = L_base - L_reference
        token_scores[tok] = score
        if score >= threshold:
            filtered_tokens.append(tok)
    return token_scores, filtered_tokens

all_kept = []
for sample in samples:
    token_scores, filtered_tokens = score_tokens(
        sample["tokens"],
        sample["base_loss"],
        sample["ref_loss"],
        threshold=0.4,
    )
    sample["token_scores"] = token_scores
    sample["filtered_tokens"] = filtered_tokens
    all_kept.extend(filtered_tokens)

effective_tokens = len(all_kept)
estimated_gain = round(log(effective_tokens + 1), 4)

assert "订单号" in samples[0]["filtered_tokens"]
assert "JSON" in samples[0]["filtered_tokens"]
assert "请" not in samples[1]["filtered_tokens"]
assert effective_tokens > 0

print("effective_tokens =", effective_tokens)
print("estimated_gain =", estimated_gain)
print(samples)
```

这段代码对应的训练前流程是：

1. 先加载 1k 条指令数据。
2. 为每个 token 计算 `token_scores`。
3. 用阈值筛掉低影响 token，得到 `filtered_tokens`。
4. 再把保留下来的内容组装成训练 batch。

真正工程里，`base_loss` 与 `ref_loss` 可以来自两种方式：

- 用基座模型和一个参考模型分别跑前向，记录每个 token 的负对数似然。
- 不做 token 级清理，只把这些统计量聚合到样本级，先筛样本再训练。

如果你是新手，一个足够实用的起点是：**先做样本级统计，再决定是否值得上 token 级清理**。因为后者更细，但实现成本更高。

## 工程权衡与常见坑

最常见的错误不是“数据不够多”，而是“错误地把低密度数据堆得很多”。

| 坑 | 影响 | 规避措施 |
| --- | --- | --- |
| 盲目堆短答、重复答 | 有效 token 密度低，收益很快饱和 | 提高回答完整度，减少模板复述 |
| 任务过于单一 | 模型学会模板，不学会迁移 | 补多任务样本，覆盖不同输入形式 |
| 只看样本数不看 token | 误判数据规模 | 统计平均长度、约束词、字段词 |
| 只追求长答 | 长但空也没用 | 长答必须有步骤、边界、约束 |
| 标签风格不一致 | 模型输出抖动 | 统一格式、口吻、字段命名 |

对比两个实操方案就很清楚：

- 方案 A：6 万条“短答复述”，大多是两三句口头化回答。
- 方案 B：2 万条“多任务长答”，每条都包含背景、约束、步骤、输出格式。

A 看起来更大，B 实际更强。因为 B 教的是“任务行为”，A 教的更多是“常见说法”。

一个真实工程判断标准是：如果你的数据里，高频出现“当然可以”“下面是答案”“希望对你有帮助”这类词，而字段定义、错误条件、输出约束却很少出现，那么数据规模大概率被高估了。

建议的简单流程是：

1. 先做 token 清理或至少做模板短语统计。
2. 再做多样性检查，确认任务类型不是单模板重复。
3. 最后才决定是否继续扩到 50k 以上。

## 替代方案与适用边界

如果预算很低，不必一开始就追求“大规模指令库”。更稳妥的是按项目形态选策略。

**策略一：精选 1k 到 6k 条**

适合场景：

- 任务边界明确
- 输出格式固定
- 领域术语清楚
- 训练预算有限

这类项目优先做高质量采样，收益最高。LIMIT 的经验就很符合这个区间。

**策略二：高质量核心集 + 少量弱监督扩展**

弱监督可以理解为“标签没那么强，但能补覆盖面”的数据，比如规则生成答案、模型自蒸馏答案、半人工校验样本。适合场景：

- 主任务已经有 2k 到 10k 条强监督
- 还需要补长尾场景
- 预算不够支持大规模人工标注

新人可理解的对比是：

- “只用精选 2k”：适合先把主任务打稳。
- “2k 精选 + 8k 弱标签”：适合在主任务稳定后补边角覆盖。

一个简化决策树如下：

| 项目形态 | 优先策略 |
| --- | --- |
| 窄任务、强格式 | 精选小数据 |
| 通用问答、场景多 | 高质量中等规模 |
| 长尾很多、人工贵 | 核心集 + 弱监督混合 |
| 数据重复严重 | 先清理，后扩量 |
| 新增数据质量无法保证 | 暂停扩量，改做筛选 |

什么时候继续增量采样，什么时候应该停？

- 如果新增样本能引入新任务、新约束、新输出结构，可以继续加。
- 如果新增样本只是改写已有模板，应该停。
- 如果离线评测已经平稳，而失败案例集中在少数长尾类型，应该改做定向补样，而不是全量扩样。
