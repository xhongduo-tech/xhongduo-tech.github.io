## 核心结论

在 ML 项目里，`pytest` 最有价值的用法不是“把 Python 代码测一遍”，而是把“模型是否还能稳定工作”变成可重复执行的检查。对初级工程师来说，可以先记住两件事。

第一，**用 `fixture` 管理共享资源**。`fixture` 可以理解为“测试前准备好的公共对象”。模型加载、分词器初始化、验证集读取，往往都很慢。如果每个测试都重新加载一次模型，测试会非常慢，也容易因为环境抖动产生无关失败。`@pytest.fixture(scope="session")` 的含义是：在整轮测试开始时加载一次，后面的测试都复用这一个对象。

第二，**用参数化和近似断言把“多样输入 + 浮点误差”一起处理掉**。参数化测试 `@pytest.mark.parametrize(...)` 的作用是：同一套断言自动跑多组输入。`pytest.approx(...)` 的作用是：允许浮点数存在很小的计算误差，避免 `0.30000000000000004` 这种数值细节把测试误判为失败。

把这两点放在一起，常见流程就是：

模型加载一次 → 多组样本重复跑 → 输出用 `approx` 或指标门槛校验 → CI 决定是否放行

这套流程对“模型测试”和“回归测试”都成立。前者关注单个样本或单个函数是否正确，后者关注新代码是否让整体指标下降。对于分类模型，常见守门指标是准确率：

$$
Accuracy=\frac{TP+TN}{TP+TN+FP+FN}
$$

其中，准确率可以理解为“预测对的样本占总样本的比例”。如果历史上稳定版本在固定验证集上的准确率长期在 0.87 左右，那么把门槛设成 0.85，就是一个常见而实用的回归测试做法。

## 问题定义与边界

先区分两个容易混淆的概念。

**模型测试**，重点是“这个模型接口和输出行为是否符合预期”。例如，输入一条固定文本，输出概率是否仍然接近历史值；输入一个二维特征向量，预测标签是否仍然正确。

**回归测试**，这里的“回归”不是回归模型，而是“防回退”。也就是防止你改了数据预处理、改了特征顺序、改了阈值、升级了依赖之后，模型指标悄悄变差。

这类问题在 ML 项目里比普通业务代码更棘手，原因主要有三点：

| 问题 | 为什么会发生 | 直接后果 |
|---|---|---|
| 模型加载昂贵 | 权重文件大、初始化慢、依赖重 | 测试变慢，开发者不愿意跑 |
| 输出常是浮点数 | 概率、损失、embedding 都有数值误差 | 精确相等判断经常误杀 |
| 指标依赖数据集 | 不同验证集分布不同 | 新旧模型结果不可比 |

这就决定了边界条件。

第一，**回归测试必须依赖固定验证集**。固定验证集可以理解为“一份长期不变的参考试卷”。如果你今天用 A 数据集、明天用 B 数据集，哪怕模型代码完全没变，最终准确率也可能变化，这时你无法判断到底是模型退化了，还是数据集变了。

第二，**门槛来自历史稳定版本，而不是随手拍一个数**。例如在 CI 里写：

```python
assert accuracy_score(y_true, y_pred) >= 0.85
```

这行代码本身没问题，但 `0.85` 必须有来源。通常做法是统计过去若干次稳定发布在固定验证集上的表现，再留一点安全余量。

第三，**并不是所有输出都适合 `approx`**。如果你测的是标签，如 `"spam"` 或 `"ham"`，那就应直接比较相等；如果你测的是概率、损失值、向量元素，才适合近似比较。

一个“玩具例子”可以帮助建立直觉。假设你有一个二分类器，固定验证集上共 10 条样本，其中 8 条预测正确，那么准确率就是：

$$
Accuracy=\frac{8}{10}=0.8
$$

如果新代码上线后只剩 7 条预测正确，准确率变成 0.7。这就是典型的精度回退。即使模型还能运行、接口也没报错，回归测试仍然应该失败，因为“能跑”不等于“质量没掉”。

一个“真实工程例子”是推荐系统或风控模型的 CI 流水线。工程里常见做法是：代码提交后，先加载一个已经训练好的基线模型或快速训练一个小模型，再在固定验证集上评估 Accuracy、AUC 或 F1。一旦指标低于门槛，合并请求就不通过。这样做的目的不是保证模型最优，而是阻止明显劣化进入主分支。

## 核心机制与推导

先看 `fixture`。它的核心价值是**把昂贵准备步骤从“每个测试一次”变成“整轮测试一次”**。

如果模型文件有 500MB，加载一次要 8 秒，而你有 30 个测试，那么不使用共享加载时，理论上仅初始化就可能消耗 240 秒。把作用域设为 `session` 后，初始化只做一次，后续测试都拿同一个实例。这里的 `session` 可以理解为“本次 pytest 运行周期”。

再看参数化。参数化测试的本质是：**把同一条测试规则应用到多组样本上**。这在 ML 里非常自然，因为模型几乎总是“同一接口，很多输入”。你不需要复制粘贴 20 个 `test_xxx_case1`、`test_xxx_case2`，只要写一套逻辑，然后把数据表传进去。

第三个关键机制是 `pytest.approx`。它解决的是浮点数比较问题。

浮点数可以理解为“计算机里对小数的近似表示”。因为二进制表示有限，像 `0.1`、`0.2` 这种十进制小数往往不能被精确表示，所以：

```python
0.1 + 0.2
```

结果通常不是数学上的精确 `0.3`，而是接近 `0.3` 的一个数。如果你直接写 `assert 0.1 + 0.2 == 0.3`，测试可能失败；但这不是业务错误，而是表示误差。

`pytest.approx(expected, rel=rtol, abs=atol)` 的容差规则可以写成：

$$
|actual-expected| \le \max(rel \cdot |expected|, abs)
$$

这条公式的意思是：实际值和期望值的差，只要没有超过“相对容差”和“绝对容差”中的较大者，就认为测试通过。

这里两个术语需要先说清楚。

- 相对容差 `rel`：允许误差随着数值大小按比例放大。数值越大，可接受误差越大。
- 绝对容差 `abs`：给一个固定误差上限，适合接近 0 的数。

常见默认值是 `rel=1e-6`、`abs=1e-12`。这不是“永远正确”的设置，只是通用起点。对于模型概率输出，通常足够；对于经过量化、GPU 非确定性较强、或跨平台数值差异明显的场景，往往需要放宽。

下表可以直观看到容差的效果：

| 期望值 | 实际值 | `rel` | `abs` | 是否通过 | 原因 |
|---|---:|---:|---:|---|---|
| 0.3 | 0.30000000000000004 | 1e-6 | 1e-12 | 通过 | 误差极小 |
| 0.3 | 0.301 | 1e-6 | 1e-12 | 不通过 | 相对误差过大 |
| 1000.0 | 1000.0005 | 1e-6 | 1e-12 | 通过 | 相对误差仅 5e-7 |
| 0.0 | 1e-9 | 1e-6 | 1e-12 | 不通过 | 期望值为 0 时主要看绝对容差 |
| 0.0 | 1e-9 | 1e-6 | 1e-8 | 通过 | 放宽了绝对容差 |

因此，`approx` 适合回答的问题是：“这个浮点输出是否仍然落在合理误差范围内？”  
而准确率门槛适合回答的问题是：“这次整体模型质量是否低于上线底线？”  
前者看单点或局部行为，后者看整体结果，两者并不冲突。

## 代码实现

下面先给一个可运行的“玩具例子”。它不依赖真实模型文件，但完整演示了“浮点近似判断”和“准确率门槛”的思路。

```python
def predict_score(x):
    # 一个非常简单的打分函数，模拟模型输出概率
    return 0.1 * x + 0.2

def accuracy_score(y_true, y_pred):
    correct = sum(int(a == b) for a, b in zip(y_true, y_pred))
    return correct / len(y_true)

# 浮点输出不应直接用 == 比较
score = predict_score(1)
assert abs(score - 0.3) < 1e-12

# 固定验证集上的分类结果检查
y_true = [1, 0, 1, 1, 0]
y_pred = [1, 0, 1, 0, 0]
acc = accuracy_score(y_true, y_pred)
assert acc == 0.8
assert acc >= 0.75
```

上面只是说明思想。真正写 `pytest` 时，通常会拆成 `fixture`、参数化测试、回归测试三层。

```python
import math
import pytest

class SimpleModel:
    # 用一个简单的线性函数模拟模型
    def predict_score(self, x):
        return 0.1 * x + 0.2

    def predict_label(self, x, threshold=0.5):
        return 1 if self.predict_score(x) >= threshold else 0

def accuracy_score(y_true, y_pred):
    correct = sum(int(a == b) for a, b in zip(y_true, y_pred))
    return correct / len(y_true)

@pytest.fixture(scope="session")
def model():
    # session 作用域：整轮测试只初始化一次
    return SimpleModel()

@pytest.mark.parametrize(
    "x,expected_score",
    [
        (1, 0.3),
        (2, 0.4),
        (3, 0.5),
    ],
)
def test_predict_score(model, x, expected_score):
    # 浮点数用 approx，避免微小误差导致假失败
    actual = model.predict_score(x)
    assert actual == pytest.approx(expected_score, rel=1e-6, abs=1e-12)

@pytest.mark.parametrize(
    "x,expected_label",
    [
        (1, 0),
        (2, 0),
        (3, 1),
        (5, 1),
    ],
)
def test_predict_label(model, x, expected_label):
    # 离散标签直接比较相等
    assert model.predict_label(x) == expected_label

def test_regression_accuracy_gate(model):
    # 固定验证集：不要每次都换数据，否则无法和历史基线比较
    x_valid = [1, 2, 3, 4, 5]
    y_true = [0, 0, 1, 1, 1]

    y_pred = [model.predict_label(x) for x in x_valid]
    acc = accuracy_score(y_true, y_pred)

    # 基线门槛应来自历史稳定版本
    min_accuracy = 0.8
    assert acc >= min_accuracy
```

这段代码体现了三件事。

第一，`model()` 通过 `session` 作用域复用模型对象。如果换成真实工程，里面通常是 `load_model("model.pkl")`、初始化 tokenizer，或者读取固定验证集索引。

第二，`test_predict_score` 用参数化覆盖多组输入。这样你只维护一套断言逻辑，但可以扩充很多样本。

第三，`test_regression_accuracy_gate` 用固定验证集和门槛值做整体回归测试。这类测试通常不会验证每个样本都对，而是验证整体指标没有跌破底线。

一个更接近真实工程的例子如下。假设你的文本分类模型部署前会在 CI 中检查精度：

- `fixture` 加载 `model.pkl` 和 `validation.csv`
- 参数化测试抽查若干高风险样本，比如空文本、超长文本、特殊符号文本
- 回归测试在固定验证集上跑完整评估
- 若 `accuracy < 0.85`，CI 失败，阻止合并

这种结构的价值在于：它既测“局部行为没坏”，也测“整体质量没掉”。

## 工程权衡与常见坑

在实际项目里，测试不是写出来就结束，关键在于门槛是否稳、结果是否可解释、维护成本是否可控。

最常见的坑是把 `approx` 当成万能工具。它并不是“只要不确定就放进去”。如果你比较的是分类标签、字符串、布尔值，直接相等判断更清晰。只有浮点数输出才适合近似比较。

第二个常见坑是**容差设置失真**。容差太紧，会出现假阴性，也就是模型其实没问题，但测试老是红；容差太松，则会把真实问题放过去。经验上，概率输出、loss、均值等结果可以从 `rel=1e-6` 起步，再根据平台和数值稳定性调整。

第三个坑是 **`Decimal` 类型直接和 `pytest.approx` 混用**。`Decimal` 可以理解为“十进制定点风格的高精度数值类型”。如果项目里有财务或计费逻辑，常会用到它。但 `pytest.approx` 主要面向普通数值比较，和 `Decimal` 直接混用可能报错，工程上最简单的做法是先统一为 `float`，或者全链路统一数值类型。

第四个坑是**验证集不固定**。很多团队一开始会在测试里“随机抽一些最新数据”，看起来很真实，实际上会破坏可比性。回归测试的首要目标不是模拟线上全部变化，而是稳定比较“同一把尺子上的前后差异”。

第五个坑是**只看 Accuracy**。准确率适合类分布相对均衡的任务，但在严重类别不平衡时可能会误导。例如 95% 样本都是负类，模型全部预测负类，也能拿到 95% Accuracy。这时应同时检查 AUC、F1、召回率等指标。

下表汇总几个典型问题：

| 问题 | 表现 | 规避方式 |
|---|---|---|
| `approx` 用在标签比较 | 断言语义不清，测试可读性差 | 标签直接 `==`，概率才用 `approx` |
| 容差过紧 | 不同平台偶发失败 | 从默认值起步，再按任务调整 |
| `Decimal` 混用 | 报 `TypeError` 或比较行为异常 | 统一转换为 `float` 或统一类型 |
| 验证集每次变化 | 指标忽高忽低，无法判断回退 | 固定留存验证集 |
| 只看一个指标 | 掩盖不平衡数据问题 | Accuracy 与 AUC/F1 并行检查 |

真实工程里，一个稳妥做法是把测试分层：

- 快速单元测试：几秒内完成，覆盖预处理、后处理、少量固定样本
- 中速模型测试：加载模型，检查若干参数化样本与概率输出
- 回归测试：在固定验证集上算指标门槛，通常放在 CI 或 nightly job

这样既不会让本地开发太慢，也能保住主分支质量。

## 替代方案与适用边界

`pytest.approx`、指标门槛、标签精确比较，这三种方法并不是竞争关系，而是对应不同问题。

| 方法 | 适用对象 | 优点 | 局限 |
|---|---|---|---|
| `pytest.approx` | 概率、损失、向量元素等浮点输出 | 能容忍正常数值误差 | 不适合离散标签和复杂分布变化 |
| 指标门槛 | Accuracy、AUC、F1 等整体指标 | 能直接防止整体精度回退 | 依赖固定验证集和稳定基线 |
| 直接相等比较 | 分类标签、规则输出、固定文本 | 语义清晰，失败原因明确 | 对浮点输出过于严格 |

什么时候优先用 `approx`？  
当你关注的是某个函数、某个样本、某个局部数值行为是否稳定。例如，模型预处理后某个特征值应接近历史结果；推理后某条样本的概率输出应维持在预期范围内。

什么时候优先用指标门槛？  
当你真正关心的是“改代码后整体效果有没有掉”。这时通常写：

```python
assert accuracy_score(y_true, y_pred) >= 0.85
```

或者：

```python
assert roc_auc_score(y_true, y_score) >= 0.90
```

AUC 可以理解为“模型把正样本排在负样本前面的能力”。当类别不平衡较严重时，它往往比 Accuracy 更稳定。

什么时候直接比较标签？  
当模型输出是确定性的离散结果，并且你希望检查业务规则没有变化。例如垃圾邮件分类器在固定样本上必须输出 `"spam"`，这时再用 `approx` 就没有意义。

还要看到它们的边界。

第一，**非确定性模型会让测试抖动**。如果推理过程包含随机采样、dropout 未关闭、GPU 内核非确定性等因素，那么同一输入每次输出都可能略不同。此时要先尽量固定随机种子、关闭训练态，再决定用近似比较还是统计指标。

第二，**回归测试不能替代离线评估**。`pytest` 里的门槛测试适合做“守门员”，不适合做完整模型评估报告。前者回答“有没有明显变差”，后者回答“当前模型到底多好”。

第三，**门槛不应频繁跟着新模型上涨**。如果每次模型略微变好，你就立刻把 CI 门槛改高，团队很快会陷入“任何小改动都过不了”的状态。更稳妥的做法是按版本周期调整基线，而不是每次提交都改规则。
