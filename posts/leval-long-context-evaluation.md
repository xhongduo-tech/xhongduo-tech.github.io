## 核心结论

L-Eval 是一个面向长上下文语言模型的评测基准。长上下文，指模型一次要读入几千到几十万 token 的文本，再完成问答、摘要、信息抽取等任务。它的关键价值不只是“文本更长”，而是把“长文本理解”和“指令跟随”绑在一起评测：模型既要读懂整段材料，也要按要求输出指定长度、指定形式、指定风格的答案。

这件事很重要，因为传统评测常常默认“答对就行”，但真实工程里不是这样。用户会说“请在 150 字内总结这场 7K 字会议”“请给出两条风险，不要扩写背景”“请输出一个关键词，不要解释”。如果模型内容基本正确，但长度失控、格式错误、重点跑偏，工程上仍然不可用。L-Eval 正是在评测里显式加入这类约束。

论文版本的 L-Eval 包含 20 个子任务、508 篇长文档、2000 多组人工标注问答，输入长度覆盖约 3K 到 200K tokens。OpenCompass 文档中的可运行实现则介绍了 18 个子任务、411 篇文档、5 个闭合任务和 13 个开放任务。这里要区分“论文定义的完整基准”和“某个工具链里当前集成的任务子集”，两者不是同一层面的数字。

一个直观例子是“概括 7K 字的会议记录，并控制在 120 到 150 字”。如果模型只写了 40 字，哪怕句句都对，也不满足需求；如果写了 500 字，信息更全，也不满足需求。L-Eval 用 LIE 和 LLM Judge 的组合处理这个问题：先检查长度与指令是否对齐，再看内容质量，最后用更接近人工判断的方式复核。

下表可以先把它和传统 ROUGE 思路区分开：

| 维度 | 传统 ROUGE | LIE + LLM Judge |
| --- | --- | --- |
| 关注点 | 参考答案与生成文本的重叠程度 | 先看是否满足长度/格式约束，再看内容质量 |
| 对长度指令的处理 | 通常不显式处理 | 长度不满足可直接降分或不计分 |
| 对“短而准”的偏差 | 容易给高分 | 会惩罚不满足尺幅要求的输出 |
| 对“长而跑题”的偏差 | 可能部分重叠仍拿分 | 会结合指令和整体语义复核 |
| 与真实使用场景的贴合度 | 中等 | 更高 |

## 问题定义与边界

L-Eval 要回答的问题是：当输入不再是几句话，而是几十页材料时，模型到底是在“真正理解长上下文”，还是只是在前后文里撞到局部答案？这里的“评测”不是泛泛比较模型谁更强，而是要在不同任务形态、不同文本领域、不同长度区间下，比较模型的稳定性。

它覆盖的场景比较接近真实业务。法律文书、财经报告、会议纪要、技术材料、小说章节，这些文本的共同点是长、信息密度不均匀、关键点分散。模型如果只能处理短提示词，在这些任务上会暴露出明显问题，比如忘记前文约束、只记住开头和结尾、忽略中段证据。

L-Eval 里的任务大致可以分成两类。

第一类是闭合任务。闭合，意思是答案空间比较窄，通常能用精确匹配或规则判断。比如 `topic_retrieval` 让模型在给定长文后，从候选主题里输出目标关键词。这里评测关注的是“找没找对”，不太关心生成文采。

第二类是开放任务。开放，意思是合理答案不止一种，通常是摘要、开放问答、段落生成。这类任务难点不只是答对事实，还要遵守具体指令。比如“对 10K 字小说写一个 150 字结尾总结，语气保持中性，不要剧透次要支线”。这时单纯看关键词重叠远远不够。

可以把两类任务放在一张表里理解：

| 维度 | 闭合任务 | 开放任务 |
| --- | --- | --- |
| 答案空间 | 窄，常有标准答案或关键词 | 宽，存在多种合理表达 |
| 常见指标 | Exact Match、规则匹配 | ROUGE、LLM Judge、人工评审 |
| 指令特点 | 明确要求输出某个词、选项、短句 | 明确要求长度、结构、风格、覆盖范围 |
| 典型长度 | 长输入 + 短输出 | 长输入 + 中长输出 |
| 示例 | 从长文中找主题词 | 将 7K 字会议压缩成 150 字摘要 |

玩具例子可以这样看。给你一篇 5000 字会议记录，里面反复讨论预算、排期、招聘、合规。问题是：“请在 20 到 30 字内总结本次会议最核心的决策。”这不是普通摘要，而是“摘要 + 长度控制 + 主次判断”。模型若输出“会议讨论了很多问题，涉及预算和排期等多个方面”，信息不能算错，但没有抓住“最核心决策”，也没有满足精炼要求。

真实工程例子更接近企业内部知识库。比如法务系统里有一份长合同，产品团队只想知道“自动续约条款是否存在，以及通知期限是多少”。如果模型答成一段泛化解释，甚至把其他条款也一起写进来，用户仍然要手工再读一次合同。对这种场景，L-Eval 的价值在于把“能否直接交付给用户”变成可量化问题。

它的边界也很明确。L-Eval 不是训练方法，不解决模型如何扩容上下文；它是评测方法，用来比较模型在长上下文下的行为。它也不是短文本通用基准，若任务本身没有长度约束、没有长文依赖，那么 GLUE、MMLU、普通摘要集可能更合适。

## 核心机制与推导

L-Eval 的核心思路可以概括成一句话：长文本评测不能只看“答得像不像参考答案”，还要看“有没有按用户要求回答”。

这里的 LIE，Length-Instruction-Enhanced，可以理解成“把长度要求和指令要求显式并入评分过程”。最简单的形式可以写成：

$$
\text{Score} = \text{ROUGE} \times \mathbf{1}[\text{length satisfied}]
$$

其中 $\mathbf{1}[\cdot]$ 是指示函数。指示函数的白话解释是：条件满足时取 1，不满足时取 0。也就是说，如果用户要求 120 到 150 字，而模型输出 60 字，那么即便 ROUGE 很高，也可以直接被压成 0，或者至少大幅降权。

进一步地，可以把它写成更接近实际系统的形式：

$$
\text{FinalScore} = \text{Judge}(\text{ContentScore}, \text{LengthConstraint}, \text{FormatConstraint})
$$

这里 `Judge` 通常由更强的语言模型或人工规则承担。白话讲，就是先算内容接近程度，再让一个“裁判”检查它有没有按规则答题。

为什么要这样做？因为传统 n-gram 指标有两个典型偏差。

第一，偏爱短答案。参考答案如果本来就短，模型只抓几个高频关键词，可能就拿到不错分数，但实际没有完成摘要任务。

第二，忽略格式与结构。如果用户要求三点列表、限定字数、保持正式语气，ROUGE 通常不直接感知这些约束。

玩具例子很容易说明。参考答案是“项目延期因预算审批未完成，决定推迟两周并冻结招聘”。模型 A 输出“预算审批未完成，项目推迟两周”，模型 B 输出“会议主要结论是因预算审批未完成，项目整体延期两周，同时新增岗位招聘暂时冻结，后续以财务审批结果为准”。A 可能因为更短、更贴近关键词而有较高重叠，但若指令要求“100 字内完整总结核心决策”，B 更可能是工程上可用的答案。

真实工程里，这个机制常用于评估长文摘要服务。假设你在做会议纪要自动生成，业务要求最终结果必须控制在企业 IM 卡片的一屏内，例如 120 到 180 字。如果评测只看 ROUGE，模型会学到“越短越稳”；如果加入 LIE，它才会学到“在受限长度下保留关键事实”。

下面是一个最小可运行的 Python 例子，展示如何做长度检查，并在开放任务里决定是否继续评分。这里为了可运行，我们用一个极简重叠率代替真实 ROUGE-L，但流程与 LIE 思路一致。

```python
import re

def char_len(text: str) -> int:
    return len(re.sub(r"\s+", "", text))

def lie_gate(text: str, min_len: int, max_len: int) -> bool:
    n = char_len(text)
    return min_len <= n <= max_len

def toy_overlap_score(pred: str, ref: str) -> float:
    pred_tokens = set(pred)
    ref_tokens = set(ref)
    if not ref_tokens:
        return 0.0
    return len(pred_tokens & ref_tokens) / len(ref_tokens)

def lie_score(pred: str, ref: str, min_len: int, max_len: int) -> float:
    if not lie_gate(pred, min_len, max_len):
        return 0.0
    return toy_overlap_score(pred, ref)

reference = "预算审批未完成，项目延期两周并冻结招聘。"
good = "预算审批未完成，项目延期两周，并暂时冻结招聘。"
too_short = "项目延期两周。"

assert lie_gate(good, 18, 30) is True
assert lie_gate(too_short, 18, 30) is False
assert lie_score(too_short, reference, 18, 30) == 0.0
assert lie_score(good, reference, 18, 30) > 0.0
```

这个例子故意简单，但它说明了一个关键判断顺序：先看约束是否满足，再决定内容分数是否有效。对长上下文任务，这是比“先打内容分再补充分析”更合理的流程。

## 代码实现

在工程实现里，L-Eval 通常不是一个单独的评分函数，而是一条完整流水线：读取样本、拼接指令、调用模型、按任务类型分流评分、必要时送入 Judge 复核。

OpenCompass 的实现思路可以概括为四步。

第一步，加载数据项。每个样本通常包含文档内容、问题、参考答案，以及 `length_instruction` 之类的约束字段。

第二步，拼接 prompt。也就是把“长文内容”“具体问题”“输出长度要求”组合成最终输入。

第三步，调用模型生成答案。闭合任务一般希望输出短且确定，开放任务允许自然语言生成。

第四步，按任务类型评分。闭合任务走 exact match 或关键词判断；开放任务先看长度要求，再算 ROUGE-L，最后可接 LLM Judge。

可以先看一个新手更容易理解的伪实现：

```python
from typing import Dict, List

CLOSE_ENDED_TASKS = {
    "topic_retrieval",
    "coursera",
    "gsm100",
    "codeU",
    "sci_fi",
}

def build_prompt(document: str, question: str, min_len: int, max_len: int) -> str:
    return (
        f"请基于给定材料回答问题。\n"
        f"输出要求：控制在{min_len}-{max_len}字内，直接给答案，不要添加无关说明。\n\n"
        f"材料：\n{document}\n\n"
        f"问题：{question}\n"
    )

def score_close_ended(pred: str, gold_keywords: List[str]) -> float:
    pred_norm = pred.strip().lower()
    return 1.0 if any(k.lower() in pred_norm for k in gold_keywords) else 0.0

def score_open_ended(pred: str, ref: str, min_len: int, max_len: int) -> float:
    if not (min_len <= len(pred.replace(" ", "")) <= max_len):
        return 0.0
    overlap = len(set(pred) & set(ref)) / max(len(set(ref)), 1)
    return overlap

def evaluate_item(item: Dict, model_generate) -> float:
    prompt = build_prompt(
        item["document"],
        item["question"],
        item["min_len"],
        item["max_len"],
    )
    pred = model_generate(prompt)

    if item["task_name"] in CLOSE_ENDED_TASKS:
        return score_close_ended(pred, item["gold_keywords"])
    return score_open_ended(pred, item["reference"], item["min_len"], item["max_len"])
```

这段代码里最值得注意的不是“怎么打分”，而是“长度约束在 prompt 和 evaluator 两边都要出现”。只在 prompt 里写“请控制在 150 字内”是不够的，因为模型可能不听；只在 evaluator 里做硬裁剪也不够，因为模型没有被提前告知约束。L-Eval 的做法是两边同时约束。

可以用一张表理解 OpenCompass 里常见的任务划分方式：

| 任务类别 | 类型 | 常见指标 | 说明 |
| --- | --- | --- | --- |
| `topic_retrieval` 一类 | 闭合 | Exact Match / 关键词命中 | 检查是否从长文中找对目标主题 |
| 基于长文的事实问答 | 闭合或半闭合 | 精确匹配 / 规则判断 | 输出通常较短 |
| 会议/报告摘要 | 开放 | ROUGE-L + LIE + Judge | 关注压缩率与信息覆盖 |
| 小说/长篇内容总结 | 开放 | ROUGE-L + Judge | 更考验全局理解与风格控制 |
| 多领域长文问答 | 开放 | 文本相似度 + Judge | 需要判断回答是否忠于原文 |

玩具例子里，`prompt` 可能很简单：

```python
prompt = f"请在 120-150 字内总结以下会议记录的核心决策：\n\n{document}"
```

真实工程例子会复杂得多。比如你在做一个评测平台，需要兼容 8K、32K、128K 上下文的不同模型。这时 prompt 还会携带系统角色、few-shot 示例、输出格式模板、停止词约束，甚至模型专属的 chat 模板。评测不是“把文档扔进去”这么简单，而是要尽量确保不同模型在公平、可复现的条件下比较。

## 工程权衡与常见坑

L-Eval 之所以有现实意义，恰恰因为长上下文评测里坑很多，而且很多坑不是模型结构问题，而是评测设计问题。

第一个坑是只看 ROUGE 或 accuracy。这样做便宜、快、可批量，但很容易奖励错误行为。模型可能学会输出极短模板句，从而稳定命中高频词；也可能把参考答案中的关键词堆起来，看起来相似，实际上没有完成任务。LIE 和 Judge 的存在，就是为了减少这类“指标投机”。

第二个坑是上下文截断。不同模型支持的上下文长度不同。一个 100K token 的样本，丢给只支持 8K 的模型时，必然要裁剪。OpenCompass 文档提到一种常见策略：截掉中间部分，尽量保留开头指令和结尾问题。原因很简单，开头常有任务定义，结尾常有具体问题，二者比正文中间的很多细节更关键。

这个逻辑可以写成一个简化检查式：

$$
\text{if } \text{len(prompt)} > \text{model\_ctx}: \text{truncate\_middle(prompt)}
$$

第三个坑是把“论文基准”和“工具实现”混为一谈。论文说 20 个子任务、508 篇文档；OpenCompass 当前文档介绍的运行版本是 18 个子任务、411 篇文档。工程上你必须写清楚自己跑的是哪一版，否则结果没法复现，也没法和别人直接对齐。

第四个坑是字数、词数、token 数混用。中文里“150 字”和“150 tokens”不是一回事，英文里“100 words”和“100 tokens”也不是一回事。如果 `length_instruction` 是按字数写的，评测器就不该偷偷按 token 统计。否则会出现模型明明遵守了用户要求，却被评测器误判。

下表总结一些高频问题与规避策略：

| 问题 | 表现 | 规避策略 |
| --- | --- | --- |
| n-gram 偏差 | 短答案拿高分，长答案被低估 | 引入 LIE 和 Judge，先检查约束再评分 |
| 上下文超限 | 关键问题被截断，结果随机 | 优先保留系统指令、任务要求、末尾问题 |
| 长度单位不一致 | 按字要求却按 token 打分 | 明确长度口径，评测与 prompt 保持一致 |
| 模型输出格式漂移 | 要列表却输出散文 | 在 prompt 中写清结构，并加规则校验 |
| 版本不一致 | 论文数字和代码数字对不上 | 在报告中标注数据版本与实现版本 |

真实工程里，模型比较还涉及性价比。论文中的实验表明，商用模型在超长上下文下通常仍领先，而一些开源模型即使上下文上限较小，在较短区间内也能逼近商用模型表现。对白盒团队来说，这意味着不能只看“最长支持多少 token”，还要看“在你真实任务长度分布内，单位成本的有效得分是多少”。

## 替代方案与适用边界

如果不使用 L-Eval，还有几类常见替代方案。

第一类是只用传统指标。比如闭合任务用 accuracy，开放任务用 ROUGE 或 BLEU。这种方法优点是简单、便宜、跑得快，适合做早期粗筛。但一旦任务强调“按指令输出”，它的缺点就很明显，因为它不会认真处理长度和格式要求。

第二类是只做人评。人评最接近真实体验，但成本高、速度慢、复现性一般，不适合频繁回归测试。大规模模型迭代时，完全依赖人评很难持续。

第三类是短文本基准，例如 GLUE、MMLU 这类任务。它们很有价值，但主要评估短输入下的分类、推理、知识能力，不适合衡量“给你一篇 7K 字会议纪要，按 150 字限制总结”的能力。也就是说，它们回答的是另一个问题。

下面这张表能更清楚地看出边界：

| 方案 | 是否支持长度指令 | 评分流程 | 典型场景 |
| --- | --- | --- | --- |
| GLUE/MMLU 等短文本基准 | 基本不支持 | 标准分类/选择题指标 | 短文本理解、知识与推理 |
| 传统长文摘要 ROUGE | 弱支持 | 直接比参考答案重叠 | 长文摘要粗评 |
| L-Eval | 明确支持 | 指令约束 + 内容评分 + Judge | 长文问答、摘要、指令跟随 |
| 纯人工评测 | 可支持 | 人工打分 | 小规模高精度验收 |

所以，L-Eval 适合什么场景？适合那些输出不是“随便说说”，而是必须同时满足“读懂长文”和“遵守用户要求”的场景，比如企业知识库问答、法务文档分析、长会纪要摘要、长篇材料的结构化提取。

它不太适合什么场景？不太适合短文本任务，也不太适合完全没有指令约束的自由生成任务。如果你的业务只需要“这段话情感是正还是负”，那就没必要引入一整套长上下文评测体系。
