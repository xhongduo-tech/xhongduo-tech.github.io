## 核心结论

视频时序建模的目标，不是把很多图片堆起来分类，而是让模型理解“前后帧怎样变化”。时序建模可以直白理解为：模型不只看这一帧长什么样，还要看前一帧和后一帧发生了什么。

从方法演进看，主线很清楚：

| 方法 | 怎么引入时间信息 | 参数/计算代价 | 时间感受野 | 适合场景 |
| --- | --- | --- | --- | --- |
| C3D / I3D 等 3D 卷积 | 在时间维也做卷积 | 较高 | 局部 | 短动作、固定长度 clip |
| TSM | 沿通道搬运相邻帧特征 | 几乎不增加参数 | 中等，靠网络层叠扩展 | 实时推理、边缘设备 |
| Video Transformer | 用注意力直接连接所有帧 token | 高，显存压力大 | 全局 | 长依赖、多阶段事件 |

结论可以压缩成一句话：短视频、短动作、预算有限时，3D 卷积和 TSM 更实用；长视频、跨片段依赖强时，Video Transformer 更有上限，但必须做分段设计。

一个玩具例子是“举手”。单看某一帧，手已经在空中，模型可能把它误判成“打招呼”或“伸懒腰”；只有看到“手从身体两侧逐渐抬起”的连续变化，动作语义才稳定。

## 问题定义与边界

视频时序建模的输入通常写成一段帧序列：

$$
X = \{x_{t-n}, \dots, x_t, \dots, x_{t+n}\}
$$

模型要学习一个映射：

$$
y_t = f(X)
$$

这里的 $x_t$ 是第 $t$ 帧图像，$y_t$ 可以是动作类别、事件边界、下一步预测结果等。

“边界”必须先说清，否则讨论架构没有意义：

| 约束变量 | 含义 | 为什么重要 |
| --- | --- | --- |
| 帧数 | 一次送入模型的帧数量 | 决定能看多长时间 |
| 动作长度 | 目标事件持续多久 | 决定局部窗口是否够用 |
| 实时性 | 是否要边采集边预测 | 决定能否接受大模型 |
| 显存/算力 | GPU 或边缘设备资源上限 | 决定能否使用全局注意力 |
| 采样策略 | 连续采样还是稀疏采样 | 决定时间信息是否被破坏 |

零基础读者最容易混淆的一点是：视频任务不等于“多张图像分类”。如果任务是“识别一个人在做什么”，静态外观有时够用；但如果任务是“区分坐下和站起”“区分拿起和放下”，关键不在单帧内容，而在变化方向。

再看一个玩具例子：识别“举手”至少要看前后 2 到 3 帧。只看中间某一帧，手在半空，信息不足；看连续帧，才能看到速度和方向。

因此，视频时序建模讨论的核心不是“模型是否足够大”，而是“模型能否在给定资源下保留足够长、足够有效的时间依赖”。

## 核心机制与推导

### 1. 3D 卷积：把时间当作第三个卷积维度

3D 卷积可以理解为：卷积核不只在高和宽上滑动，也在时间轴上滑动。感受野这个词的白话解释是：某一层神经元到底能看到输入中的多大范围。

如果时间卷积核大小是 $k_t$，时间步长是 $\text{stride}_t$，单层局部感受范围可以写成：

$$
R_t = (k_t - 1)\times \text{stride}_t + 1
$$

更常见的直观推导是：若两层都用时间核大小 3、步长 1，则第一层能看 3 帧，第二层叠加后能看 5 帧，第三层能看 7 帧。也就是局部窗口会扩展，但扩展速度不快。

这解释了 3D 卷积的优势和限制：

- 优势是局部时空模式抓得很稳，比如“嘴开始张开”“球刚离手”。
- 限制是长动作容易丢失，比如“先走向门，再停下，再转身离开”这种跨很多秒的依赖。

一个常见误区是以为“堆更多 3D 卷积层就自然等于长时建模”。理论上感受野会变大，但信息路径变长、池化变粗、显存变高，实际效果并不等价于真正的全局建模。

### 2. TSM：不加参数地借邻帧信息

TSM 全称 Temporal Shift Module，白话解释是：不新增卷积权重，只把一部分通道的特征往前一帧或后一帧挪，让当前层在 2D backbone 里也能接触邻近时间信息。

设某层特征为 $F \in \mathbb{R}^{T \times C \times H \times W}$，通道维按比例 $r$ 划分：

- 前 $rC$ 个通道向前移到 $t-1$
- 中间 $rC$ 个通道向后移到 $t+1$
- 剩余 $(1-2r)C$ 保持在当前时刻

可以写成一个示意：

$$
F'_t = [F_{t-1}^{1:rC},\; F_{t+1}^{rC:2rC},\; F_t^{2rC:C}]
$$

它的关键点不是“更强表达”，而是“更低代价的时序接入”。因为搬运通道本身不引入新参数，所以它能直接插到 2D ResNet 等主干里。

直观理解：

- 3D 卷积像把连续三帧打包后一起卷。
- TSM 像在特征层偷偷把前后页内容挪一点过来。
- 当前帧卷积虽然还是 2D，但输入里已经混入邻帧线索。

真实工程例子是边缘设备实时动作识别。Jetson Nano 或手机 SoC 上，完整 Video Transformer 往往太重；TSM 只改特征流动方式，能保留 2D CNN 的大部分吞吐，同时补上基础时序能力。

### 3. Video Transformer：直接建立全局时间依赖

Transformer 里的自注意力，白话解释是：每一帧都可以主动决定“我应该参考哪些别的帧”。

标准公式是：

$$
\text{Attention}(Q,K,V)=\text{softmax}\left(\frac{QK^\top}{\sqrt{d}}\right)V
$$

其中 $Q,K,V$ 分别是 query、key、value 映射。对视频来说，token 可以来自帧 patch，也可以来自更粗的 tubelet。核心优势是：任意两个时刻都能直接交互，不需要靠很多层局部卷积慢慢传。

这就是为什么 Transformer 天然适合长依赖问题，例如：

- “人先拿钥匙，再开门，再进屋”
- “篮球战术里先挡拆，再分球，再空切”
- “监控视频里可疑动作在 20 秒前就埋下线索”

但代价同样明确。若 token 数量为 $N$，注意力矩阵规模通常是 $N \times N$，时间和显存开销近似二次增长。视频的 $N$ 往往远大于图像，因为多了时间维。

所以实际工程不会把 30 秒原视频完整喂给标准全局注意力，而是先做 chunk。chunk 的白话解释是：把长视频切成多个小片段，先局部建模，再逐层汇总。

## 代码实现

下面先用一个可运行的 Python 玩具实现说明 TSM 的通道移动。这里不依赖深度学习框架，只演示时序搬运的核心逻辑。

```python
import numpy as np

def temporal_shift(x, fold_div=4):
    """
    x: [T, C]
    将一部分通道左移/右移，其余保持不变
    """
    T, C = x.shape
    fold = C // fold_div
    out = np.zeros_like(x)

    # 前 fold 个通道：来自下一时刻，等价于向左看
    out[:-1, :fold] = x[1:, :fold]

    # 中间 fold 个通道：来自上一时刻，等价于向右看
    out[1:, fold:2*fold] = x[:-1, fold:2*fold]

    # 其余通道保留当前时刻
    out[:, 2*fold:] = x[:, 2*fold:]
    return out

x = np.arange(4 * 8).reshape(4, 8)
y = temporal_shift(x, fold_div=4)

# 当前帧后半部分通道保持不变
assert np.array_equal(y[:, 4:], x[:, 4:])

# 第 0 帧前两列来自原第 1 帧前两列
assert np.array_equal(y[0, :2], x[1, :2])

# 第 1 帧第 2~3 列来自原第 0 帧第 2~3 列
assert np.array_equal(y[1, 2:4], x[0, 2:4])
```

如果把它放进 CNN 主干，典型顺序是：

```python
# 伪代码
def block(x):
    # x: [B, T, C, H, W]
    x = temporal_shift(x, fold_div=8)   # 在卷积前注入邻帧信息
    x = conv2d_bn_relu(x)               # 仍然是普通 2D 卷积
    x = conv2d_bn_relu(x)
    return x
```

为什么要“只移动部分通道”而不是全部移动？因为当前帧信息也必须保留。若全部通道都被邻帧覆盖，模型会失去当前时刻的静态外观。

再看长视频 Transformer 的 chunk 化伪代码：

```python
# 伪代码
def encode_long_video(frames, chunk_size=16):
    chunks = split_into_chunks(frames, chunk_size)
    local_tokens = []

    for chunk in chunks:
        # 每个 chunk 内做局部时空编码
        z = video_transformer(chunk)
        # 保留 chunk 摘要，避免后续只剩平均池化
        summary = z.mean(axis=0)
        local_tokens.append(summary)

    # 第二层聚合器处理跨 chunk 关系
    global_repr = temporal_aggregator(local_tokens)
    return classifier(global_repr)
```

这里的关键不是“切块”本身，而是“切完后如何保留跨块关系”。只做独立 chunk 分类再平均，往往会把“先发生 A，再发生 B”的顺序信息抹平。

## 工程权衡与常见坑

工程里最常见的取舍不是精度，而是“延迟、显存、稳定性”三者怎么平衡。

| 场景 | 候选方法 | 优点 | 主要问题 |
| --- | --- | --- | --- |
| 边缘设备实时识别 | TSM + 2D ResNet | 快，改造小 | 长依赖有限 |
| 标准短视频分类 | I3D / (2+1)D / 小型 ViViT | 精度和成本平衡 | clip 长度有限 |
| 长视频理解 | Video Transformer + chunk | 跨段能力强 | 显存高，设计复杂 |

常见坑主要有五类：

1. 只采样稀疏帧，导致动作方向丢失。  
很多初学者把视频每秒采 1 帧，再说模型“不会时序”。问题可能不是模型，而是采样已经把时间结构毁了。

2. 误把局部建模当全局建模。  
3D 卷积能看连续变化，但不代表它天然适合 30 秒长事件。

3. TSM 插入位置不对。  
若每一层都大比例 shift，静态表征会被破坏。常见做法是在残差块前、且只移动少量通道。

4. Transformer 直接堆长序列，显存爆炸。  
因为注意力复杂度随 token 数增长很快。30 秒视频如果保持高帧率和高分辨率，序列长度会失控。

5. chunk 后只做平均池化，跨段逻辑被抹平。  
解决方法通常是保留每个 chunk 的 summary token、时间位置编码，或者做第二层时序聚合。

真实工程例子：做 30 秒监控视频告警，如果每秒 25 帧、每帧再切成很多 patch，标准全局注意力通常不可承受。更稳妥的路线是：

- 先按 1 秒或 2 秒切 chunk
- chunk 内做局部编码
- 输出 chunk summary
- 用轻量 Transformer 或时序池化做第二层聚合

这类层级结构的本质，是把“细粒度局部动作”和“粗粒度全局事件”拆成两级处理。

## 替代方案与适用边界

没有一种方法对所有视频都最优，重点是动作时长、资源预算、是否要求实时。

| 方法 | 适用视频长度 | 资源需求 | 跨段能力 | 典型任务 |
| --- | --- | --- | --- | --- |
| C3D / I3D | 短 | 中到高 | 弱 | 短动作分类 |
| TSM | 短到中等 | 低到中 | 中 | 实时识别、边缘部署 |
| Video Transformer | 中到长 | 高 | 强 | 长视频理解、复杂事件分析 |
| Transformer + chunk | 长 | 中到高 | 强 | 会议视频、监控、体育战术 |

可以用三个例子理解边界：

- “眨眼”这类短时动作，C3D 或 I3D 往往够用，因为关键信息集中在很短窗口内。
- “跑步”这类持续动作，TSM 往往更合适，因为它在保持帧率的同时能看到邻帧变化。
- “篮球战术分析”这类跨很多秒、跨多个角色的任务，更适合 Transformer 加 chunk，因为需要远距离依赖和全局关系。

因此，替代方案不是谁淘汰谁，而是谁在什么边界条件下最划算。若你的任务只有 8 到 16 帧、设备还受限，直接上大 Transformer 通常不是工程上最优解。
