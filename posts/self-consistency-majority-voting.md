## 核心结论

自洽性解码（Self-Consistency）是一种解码策略。解码策略就是“模型已经会回答后，最后到底怎么把答案生出来”的方法。它不改模型参数，不额外训练，而是把“只走一条推理路径”改成“采样多条推理路径，再对最终答案投票”。

它解决的不是“模型完全不会”，而是“模型有时会、但单次推理容易偶发走偏”。单条链式思维（Chain of Thought，CoT，指模型把中间推理步骤也写出来）常见问题是：某一步算错、某个假设用错，最后答案就偏了。自洽性的核心做法是保留这些中间推理的随机性，但只在最终答案层面做聚合：

$$
\hat a = \arg\max_{a\in\mathcal{A}} \sum_{i=1}^{N}\mathbb{1}(a_i=a)
$$

这里 $\mathbb{1}(a_i=a)$ 是指示函数，白话就是“第 $i$ 条路径的最终答案是不是 $a$，是就记 1，不是就记 0”。

最重要的工程结论有三条：

| 对比项 | 单路径 CoT | 自洽性多路径投票 |
|---|---:|---:|
| 解码方式 | 贪心或单次采样 | 多次采样后多数票 |
| 训练成本 | 无 | 无 |
| 推理成本 | 1 倍 | 约 $N$ 倍 |
| 稳定性 | 较差 | 明显更高 |
| GSM8K, PaLM 540B | 56.6% | 74.4%（40 路径） |

玩具例子：同一道小学算术题，让模型独立回答 40 次，每次都允许它写完整思路，但系统只提取最后答案。如果 24 出现 25 次，26 出现 8 次，28 出现 7 次，最终返回 24。错误路径通常分散，不会集中成一个高频答案；正确推理如果更稳定，就会在投票里胜出。

这本质上是一种“内集成”。集成就是把多个弱一些的结果合起来得到更稳的结果。传统集成要多个模型，自洽性只用一个模型反复采样。

## 问题定义与边界

问题定义可以写得很严格：

1. 对同一个输入问题，用随机解码采样 $N$ 条推理路径。
2. 从每条路径中提取最终答案 $a_i$。
3. 在答案空间 $\mathcal A$ 中做边际化投票，也就是只看“最终答案出现了几次”。
4. 输出票数最高的答案 $\hat a$。

这里“随机解码”通常指 temperature、top-k、top-p 这类采样控制。temperature 可以理解为“放大或缩小概率分布的随机性”；top-k 就是“每一步只允许从概率最高的前 k 个 token 里抽样”。

它的适用边界很明确：问题必须能归约到一个可比较的最终结论。所谓“可归约”，就是不同路径最后都能落到同一种答案格式里，比如整数、选项、布尔值、结构化标签。

| 问题类型 | 是否适合自洽投票 | 原因 |
|---|---|---|
| 算术题 | 适合 | 最终答案是确定数字 |
| 多步符号推理 | 适合 | 结论可标准化 |
| 多选题 | 适合 | 答案空间离散 |
| 代码修 bug | 部分适合 | 可对测试是否通过投票，但不能只看文字答案 |
| 开放式摘要 | 不适合 | 每条输出都不同，难直接多数票 |
| 创意写作 | 不适合 | 没有唯一正确答案 |

新手版理解：像 5 个朋友各自算同一道题，最后谁的人数多就听谁的。如果 3 个人都算出 24，就输出 24。但这个比喻有一个严格边界：前提是“24”这个结果能被直接比对。如果 5 个人写了 5 段完全不同的产品总结，就没有自然的“多数票”可做。

另一个边界是“集体一致错”。如果模型的错误不是偶发噪声，而是系统性偏差，比如所有路径都受同一个错误模板影响，那么多采样不会修复问题，反而会把错误稳定下来。白话说：自洽性只擅长消除随机错，不擅长消除共性错。

## 核心机制与推导

为什么它有效，核心不是“投票本身神奇”，而是“多条路径在错误上不完全相关”。相关性就是“几次错误是不是总一起发生”。如果每条推理都完全一样，那采样 40 次和采样 1 次没有本质区别；只有当随机采样真的带来路径差异时，投票才有意义。

把单路径正确率记为 $p$。如果先假设每条路径独立，且 $p > 0.5$，那么多数票正确的概率会随着 $N$ 增大而上升。这和经典统计里的伯努利多数表决一致。虽然真实 LLM 路径并不独立，但只要“部分独立”，多数票仍然能降低噪声。

经验上，很多任务的错误率满足近似幂律：

$$
\mathrm{Err}(N) \propto N^{-\alpha}
$$

其中 $\alpha$ 是经验指数，常见在 $0.3 \sim 0.5$ 左右。幂律的意思是：采样次数翻倍，收益不是线性翻倍，而是按稳定比例减少错误。

把上式取对数：

$$
\log \mathrm{Err}(N) = -\alpha \log N + C
$$

这说明在对数坐标下，$\log N$ 和 $\log \mathrm{Err}(N)$ 近似线性。白话解释：横轴每翻一倍，纵轴大致按固定斜率下降，所以前几次加样本最值钱，后面越来越贵。

可以用一个描述性图示理解：

```text
错误率
^
|\
| \
|  \
|   \
|    \
|      \
+-----------------> log(N)
  1  2  4  8 16 32
```

这条线不是严格直线，但通常前期下降明显，后期逐渐变缓。

玩具例子：假设单路径答对率约为 0.6。不是每次都答对，但“偏正确”。这时采样 1 次可能常错，采样 5 次后多数票会明显更稳；采样 40 次时还会更好，但边际收益已经比 1 到 5 小很多。这就是为什么工程上常先试 $N=5, 10, 20$，而不是默认拉满到 100。

真实工程例子：在 Amazon Bedrock 的案例里，同一类算术推理任务，30 条路径的自洽投票把准确率从 51.7% 提升到 68%，而只用 5 条路径就已经获得约 7.5 个百分点的增益。这说明部署层能直接利用该机制，不需要改训练流程，但也同时说明“多采样会持续涨”不等于“无限采样都划算”。

## 代码实现

实现可以拆成三步：采样、解析、聚合。

1. 采样：对同一 prompt 调用模型 $N$ 次，开启随机解码。
2. 解析：从每条输出中提取最终答案。
3. 聚合：统计频次，返回众数；必要时给出置信度。

最小伪代码如下：

```python
answers = []
for i in range(N):
    trace = sample_chain_of_thought(prompt, temperature=0.7, top_k=40)
    answer = parse_final_answer(trace)
    answers.append(answer)
return mode(answers)
```

真正难点不在“投票”，而在“解析”。如果输出里有“最终答案是 24”“Answer: 24”“所以结果为 24 元”，解析函数必须能统一提取出 `24`。否则错误不是出在模型，而是出在后处理。

下面给一个可运行的 Python 玩具实现。它不依赖外部模型，而是模拟多条路径和答案提取逻辑：

```python
import re
from collections import Counter

def parse_final_answer(text: str):
    patterns = [
        r"最终答案是[:：]?\s*(-?\d+(?:\.\d+)?)",
        r"答案[:：]?\s*(-?\d+(?:\.\d+)?)",
        r"结果[:：]?\s*(-?\d+(?:\.\d+)?)",
    ]
    for p in patterns:
        m = re.search(p, text)
        if m:
            num = m.group(1)
            return int(num) if num.isdigit() or (num.startswith("-") and num[1:].isdigit()) else float(num)
    nums = re.findall(r"-?\d+(?:\.\d+)?", text)
    if nums:
        num = nums[-1]
        return int(num) if num.isdigit() or (num.startswith("-") and num[1:].isdigit()) else float(num)
    return None

def self_consistency_vote(traces):
    answers = [parse_final_answer(t) for t in traces]
    answers = [a for a in answers if a is not None]
    counts = Counter(answers)
    winner, votes = counts.most_common(1)[0]
    confidence = votes / len(answers)
    return winner, confidence, counts

traces = [
    "先算单价再乘数量，最终答案是: 24",
    "设总价为 x，可得结果: 24",
    "中间一步算错了，答案: 26",
    "重新整理后，最终答案是 24",
    "最后得到 24",
]

winner, confidence, counts = self_consistency_vote(traces)

assert winner == 24
assert counts[24] == 4
assert round(confidence, 2) == 0.80

print(winner, confidence, counts)
```

这个例子刻意展示了一个事实：路径内容可以不同，甚至有路径中间过程混乱，但只要最终答案解析正确，投票仍然有效。

真实工程实现会多加三层：

| 步骤 | 关键实现 | 常见耗时来源 | 并行化建议 |
|---|---|---|---|
| 单路径采样 | 1 次模型调用 | 推理延迟 | 无需特别处理 |
| N=5 投票 | 5 次模型调用 + 解析 | 网络往返、生成 token | 并发请求 |
| N=30 投票 | 30 次模型调用 + 解析 + 统计 | 推理成本线性增长 | 批推理或队列并行 |
| 结果聚合 | 众数/加权投票 | 很低 | 本地完成 |

在 AWS Bedrock、批处理推理平台或自建并发 worker 上，比较合理的实现方式是：

1. 固定同一 prompt 模板。
2. 为每个请求设置不同随机种子或采样流。
3. 批量发起推理。
4. 用统一解析器抽取最终答案。
5. 返回 `winner + vote_distribution + confidence`。

`confidence` 可以定义为最高票占比，例如 $c = \frac{\max_a \mathrm{count}(a)}{N}$。它不是概率保证，但能作为是否早停、是否继续采样的工程信号。

## 工程权衡与常见坑

第一类权衡是成本。自洽性的收益几乎总伴随线性成本增长。采样 30 次，模型推理费用和延迟大致就是单次的 30 倍，除非平台支持高效批推理。

| 路径数 N | 预期收益 | 成本倍率 | 工程建议 |
|---:|---|---:|---|
| 1 | 基线 | 1x | 仅作对照 |
| 5 | 通常已有明显提升 | 5x | 最值得先试 |
| 15 | 继续上涨但开始递减 | 15x | 适合中高价值任务 |
| 30 | 常见上限区间 | 30x | 需验证 ROI |
| 40+ | 可能继续提升 | 40x+ | 通常只用于高风险场景 |

第二类权衡是多样性。路径必须“足够不同”，投票才有意义。如果 temperature 太低，30 条路径可能只是同一句话的轻微改写；如果 temperature 太高，又可能让正确路径本身变少。这里要调的是“可控随机性”，不是盲目加噪声。

第三类权衡是答案标准化。很多失败案例不是模型没推出来，而是输出格式不统一，导致 `24元`、`24.0`、`二十四` 被当成三个答案。对于数值类任务，应该先做规范化，例如单位剥离、数值四舍五入、分数和小数统一。

常见坑可以直接列出来：

- 路径不独立：temperature 太低，采样只是重复同一条错误。
- 解析不健壮：正则只支持一种格式，导致大量有效答案丢失。
- 投票粒度错了：对整段文本投票，而不是对最终结论投票。
- 问题不适配：开放生成任务硬做多数票，结果全是低频答案。
- 预算失控：没有先测 $Err(N)$ 曲线，直接把 N 拉到很高。
- 集体一致错：模型在某类题目上有系统性偏差，多采样无效。

真实工程例子：如果你在一个客服质检或教育题库系统里做“步骤型数学判题”，那么自洽性很适合，因为每道题最终都有标准答案；如果你在做“让模型写一段解释邮件”，简单多数票往往没有意义，更适合引入打分器（reranker，重排序器，即先生成多个候选，再用另一个规则或模型评分）来选优。

一个实用策略是早停：先采样 5 条，如果同一答案已经拿到 4 票，且占比达到阈值，比如 80%，直接停止；只有当票型分散时，再继续采样到 15 或 30。这样可以把平均成本压下来。

## 替代方案与适用边界

自洽性不是唯一方案，它只是“零训练、低改造成本”这一象限里非常强的一种。

| 方案 | 是否需额外训练 | 推理成本 | 实现复杂度 | 适用场景 |
|---|---|---:|---:|---|
| 自洽性投票 | 否 | 中到高 | 低 | 有确定最终答案的推理任务 |
| Beam Search + Re-rank | 否 | 中到高 | 中 | 可定义候选评分函数的任务 |
| 奖励模型/重排序模型 | 通常需要 | 中 | 高 | 有质量判别标准但投票无效 |
| 微调/强化学习 | 是 | 单次推理可低 | 很高 | 高频稳定场景，值得前置投入 |
| 多模型集成 | 否或是 | 高 | 高 | 高风险、高价值任务 |

Beam Search（束搜索，指同时保留若干高概率前缀）的问题是它偏向高概率路径，未必能带来真正的推理多样性。自洽性则主动引入随机采样，更适合“正确思路不止一条，但错误也很多”的场景。

微调和强化学习适合另一种问题：模型对这类任务普遍不行，靠投票也投不出正确答案。这时要修的是模型能力本身，而不是解码策略。

新手版边界可以这样记：

- 如果问题像“这道题答案是多少”，自洽性通常有价值。
- 如果问题像“写一篇更好的摘要”，简单投票通常没有价值。
- 如果问题像“给 10 个候选 SQL，哪一个能跑通”，那就更适合执行验证或 rerank，而不是看哪个文字答案出现次数最多。

因此，自洽性的核心边界不是“能不能多采样”，而是“最终结果能不能被稳定比较”。能比较，就适合投票；不能比较，就需要评分器、执行器或者训练手段。
