## 核心结论

Multi-Agent 协作模式，直白说就是“把一个大任务拆给多个有分工的智能体，并规定它们怎么沟通、谁做决定、什么时候同步”。它成立的根本原因，不是“模型变多了”，而是把单体系统最容易失控的三件事拆开处理了：记忆压力、规划深度、执行反馈。

单体大模型面对长链任务时，通常同时承担“理解问题、拆解步骤、调用工具、检查结果、修正路线”五类职责。任务一长，提示词就会膨胀，状态就会丢失，局部错误还会沿链条放大。多智能体模式通过角色化分工，把这些职责拆成多个局部闭环，例如“感知、规划、执行”三角色：感知角色收集上下文和外部信号，规划角色生成可执行方案，执行角色调用工具并回传结果。对新手来说，可以把它理解成“多个专业兵协同出击”，而不是“一个人同时扮演侦察员、参谋和操作员”。

它的第二个关键点是通信结构。通信结构，白话说就是“谁能和谁说话、按什么顺序说、每次传多少信息”。多智能体不是所有角色随意聊天，而是预先约束参与者、互动类型、组织结构、策略和协调协议。这样做的价值是可控扩展：增加角色时，不必把整个系统重写，只要把新角色接到既有通信图里。

从理论上看，多体协作在在线学习和元学习场景下能获得额外收益。若单体平均遗憾度是 $O(\sqrt{T})$，那么在图连通且通信可用的多智能体设定下，可降到

$$
r_N = O\left(\sqrt{T/N}\right)
$$

这里的遗憾度，白话说就是“系统在一段时间里因为没做到最优而多付出的损失”。$N$ 个智能体并行后，收益不是线性翻倍，但能得到 $\sqrt{1/N}$ 级别的改进。比如 $T=100, N=2$ 时，粗略从 $10$ 降到 $7.07$，相当于降低约 29%。

下面这张表先给出最重要的整体对比：

| 维度 | 单体 Agent | Multi-Agent 协作 |
|---|---|---|
| 记忆管理 | 全部状态堆在一个上下文里，容易溢出 | 分角色持有局部状态，分布式记忆更稳 |
| 规划能力 | 长链任务易混淆中间目标 | 可分层规划，局部计划更清晰 |
| 鲁棒性 | 单点失误会拖垮全局 | 某一角色失误可局部回滚或替换 |
| 并行性 | 基本串行 | 可并行处理子任务 |
| 理论遗憾 | 常见为 $O(\sqrt{T})$ 量级 | 可达 $O(\sqrt{T/N})$ |
| 通信负担 | 几乎没有内部通信 | 需要协议、同步和带宽预算 |
| 控制难度 | 简单 | 高，需要治理与审计 |

## 问题定义与边界

要讨论 Multi-Agent 协作，先要把“多”定义清楚。只要多了几个提示词模板，不等于多智能体。一个可讨论的多智能体系统，至少要明确五个要素：

| 要素 | 定义 | 典型取值 |
|---|---|---|
| 参与智能体数 | 系统里有多少个独立角色 | 2 个、3 个、N 个 |
| 互动类型 | 角色之间怎么交换信息 | 广播、点对点、轮询、层级汇报 |
| 组织结构 | 谁对谁负责 | 中心式、分布式、层级式 |
| 策略 | 每个角色如何决策 | 固定规则、规划驱动、学习驱动 |
| 协调协议 | 冲突时怎么收敛 | 投票、仲裁、优先级、同步屏障 |

边界也要说清。Multi-Agent 不是“角色越多越好”。它通常只在以下条件下更有价值：

| 边界条件 | 为什么重要 | 超出边界会怎样 |
|---|---|---|
| 通信频率可控 | 频繁同步会吞掉计算资源 | 延迟变高，吞吐下降 |
| 图连通性存在 | 信息要能传到关键角色 | 局部最优、全局失配 |
| 目标可模块化 | 任务能被拆成相对独立子问题 | 分工后反而增加协调成本 |
| 模型能力有互补性 | 不同角色确实各有所长 | 只是复制同类错误 |
| 语义约束清晰 | 角色对术语和状态有共同理解 | 输出互相矛盾 |

一个玩具例子最容易看清差异。

假设任务是“根据用户一句自然语言指令，完成一个促销策略配置”。  
单体做法：一个 LLM 直接读输入，自己解析约束、决定策略、生成配置、校验结果。问题是它必须同时记住业务规则、优惠条件、库存约束和输出格式，步骤一多就容易漏。

双智能体做法：  
第一个智能体负责提示解析，也就是把“满 200 减 30，限华东地区，新客优先”转成结构化约束。  
第二个智能体负责策略执行，也就是根据结构化约束生成具体规则。  
两者按轮询通信：解析器输出约束，执行器发现缺失字段再回问，直到收敛。

这个例子里，系统并没有变“更聪明”，而是把错误分成了两类可诊断问题：解析错，还是执行错。对工程系统来说，这比单次回答更重要。

但边界同样明显。如果任务本身很短，比如“把一句话翻译成英文”，多角色会纯增开销；如果角色间术语不一致，例如一个角色把“优先级”理解为调度优先级，另一个理解为商业优先级，那么通信越多，错误越系统化。

## 核心机制与推导

Multi-Agent 协作的机制可以分成三层理解：分工、通信、收敛。

第一层是分工。分工的意义不是把任务随便切块，而是让每个角色只维护自己必须知道的局部状态。局部状态，白话说就是“这个角色为了继续工作必须记住的信息”。感知角色关心输入变化，规划角色关心目标和约束，执行角色关心工具调用结果。这样做的直接效果是上下文不再全部挤在一个窗口里。

第二层是通信。通信不是附属功能，而是多体成立的核心。没有通信，多角色只是多个孤立模块。通信协议需要回答三个问题：发什么、发给谁、什么时候发。工程上常见三种结构：

```text
中心式:
感知 -> 协调器 -> 规划 -> 协调器 -> 执行

层级式:
总协调
  -> 规划组长 -> 执行组
  -> 审计组长 -> 校验组

分布式:
A <-> B <-> C <-> D
```

第三层是收敛，也就是“多个局部决策最终为什么不会越走越散”。这里最有代表性的机制之一是梯度追踪。梯度，白话说就是“当前参数往哪个方向改，损失会下降得更快”。梯度追踪的关键点在于：每个智能体不需要拿到全局全部数据，只需和邻居做一次通信，就能逐步逼近全局梯度方向。

理论上，单体在 $T$ 轮中的平均遗憾常写成：

$$
r_1 = O(\sqrt{T})
$$

多智能体在 $N$ 个节点协同、图连通、通信受限但可持续进行时，可达到：

$$
r_N = O\left(\sqrt{T/N}\right)
$$

为什么会多出一个 $1/\sqrt{N}$？直观上，因为每轮并不是只有一个体在积累经验，而是 $N$ 个体同时采样、同时更新，再通过通信把局部信息汇总。并行贡献了更多样本，通信抵消了各自只看到局部数据的偏差，所以总误差下降更快。

把它写成最小推导很直观：

$$
\frac{r_N}{r_1}
\approx
\frac{\sqrt{T/N}}{\sqrt{T}}
=
\sqrt{\frac{1}{N}}
$$

当 $N=2$ 时：

$$
\sqrt{\frac{1}{2}} \approx 0.707
$$

这表示多体系统的遗憾大约是单体的 70.7%，也就是降低约 29.3%。

玩具例子直接代数值：

- 单体：$T=100$，遗憾近似 $\sqrt{100}=10$
- 双智能体：$\sqrt{100/2} \approx 7.07$
- 降幅：$(10-7.07)/10 \approx 29.3\%$

这不是说“多一个智能体就一定快 29%”，而是说在满足通信与连通条件时，理论上存在这种量级收益。真实系统里还要扣除通信成本、等待时间和语义对齐失败带来的损耗。

真实工程例子来自个性化制造。一个定制生产系统里，部分环节必须满足严格 PID 时序约束。PID，白话说就是一种根据误差连续调节控制量的反馈控制方法，常见于工业控制。此时可把系统拆成两类协作组：

- 同步协作组：负责强时序、强一致性的控制环节，要求在固定窗口内同步更新。
- 异步协作组：负责跨工位、跨订单的一致性调整，不要求每一步同时完成，但要持续共享状态。

再结合图聚类和梯度追踪：同步组内部高频同步，保证本地控制一致；异步组低频交换摘要，维护全局元模型。这样才能在有限延迟内同时满足“局部控制稳定”和“全局定制响应快”。

## 代码实现

代码层面，Multi-Agent 协作至少要把三个对象建清楚：角色、通信、协调器。

角色，白话说就是“一个有职责、有工具、有输入输出契约的执行单元”。  
通信接口，白话说就是“角色之间怎么发消息和收消息”。  
协调器，白话说就是“谁负责推进流程、处理冲突和做超时控制”。

核心接口可以先抽象成下面这个表：

| 接口 | 输入 | 输出 | 作用 |
|---|---|---|---|
| `observe(task)` | 任务或外部事件 | 局部状态 | 读取上下文 |
| `plan(state)` | 局部状态 | 子目标/动作 | 形成下一步决策 |
| `act(action)` | 动作 | 工具结果 | 执行工具或生成内容 |
| `send(peer, msg)` | 邻居与消息 | 无 | 发通信 |
| `recv()` | 无 | 消息列表 | 收通信 |
| `update(shared)` | 共享摘要 | 新状态 | 做一致性更新 |

新手版的 agent loop 可以写成下面这样：

```text
loop(task):
    local_state = observe(task)
    inbox = recv()
    local_state = merge(local_state, inbox)

    action = plan(local_state)
    result = act(action)

    summary = summarize(result, local_state)
    for neighbor in neighbors:
        send(neighbor, summary)

    if need_sync():
        local_state = update(global_or_neighbor_summary)

    return result
```

如果要把“通信 + 角色 + 更新”写成一个最小可运行的 Python 例子，可以做一个双智能体玩具模拟：两个 agent 各自持有局部梯度估计，每轮先本地更新，再做一次邻居平均，观察误差下降。

```python
from math import sqrt

class Agent:
    def __init__(self, name, x, lr=0.2):
        self.name = name
        self.x = x
        self.lr = lr
        self.track = 0.0

    def local_grad(self, target):
        # 最小二乘损失 L=(x-target)^2 的梯度
        return 2 * (self.x - target)

    def step(self, target):
        g = self.local_grad(target)
        self.track = g
        self.x -= self.lr * self.track

def communicate(a, b):
    # 一次点对点通信：平均梯度追踪量
    avg = (a.track + b.track) / 2.0
    a.track = avg
    b.track = avg

def sync_update(a, b):
    a.x -= a.lr * a.track
    b.x -= b.lr * b.track

a = Agent("planner", x=10.0)
b = Agent("executor", x=0.0)
target = 4.0

before = abs(a.x - target) + abs(b.x - target)

for _ in range(5):
    a.step(target)
    b.step(target)
    communicate(a, b)
    sync_update(a, b)

after = abs(a.x - target) + abs(b.x - target)

assert after < before
assert abs(a.track - b.track) < 1e-9

single_regret = sqrt(100)
multi_regret = sqrt(100 / 2)
assert round(single_regret, 2) == 10.00
assert round(multi_regret, 2) == 7.07
```

这个代码不是工业实现，但足够表达三个核心点：

1. 每个 agent 有自己的局部状态和局部梯度。
2. 通信是显式动作，不是默认共享内存。
3. 更新分为本地更新和同步更新两段。

真实系统里，一般还要补三层工程结构：

- 角色层：封装工具、提示模板、状态格式。
- 协议层：定义消息 schema、重试、超时、幂等。
- 调度层：管理同步组、异步通道、优先级和资源配额。

## 工程权衡与常见坑

Multi-Agent 最常见的问题，不是“模型不够强”，而是“系统不够稳”。稳不稳主要取决于三种权衡：通信与计算、自治与控制、速度与可解释性。

第一类权衡是通信与计算。同步通信会占用算力和等待时间，尤其在在线学习场景里，它会和训练、推理、工具调用直接争资源。一个角色等别人返回时，GPU、队列槽位、数据库连接都可能被占着。解决办法通常不是“全局同步更频繁”，而是把同步范围缩小。图聚类，白话说就是“先把强相关节点分成小组，只在组内高频同步，组间低频交流”。这样能显著降低同步风暴。

第二类权衡是自治与控制。角色越自治，局部效率越高；但自治过强，系统越难保证全局一致。很多团队一开始喜欢让所有 agent 都能直接调用工具、修改计划、发起回滚，最后得到的是一个谁都能动全局状态的系统。更稳妥的办法是：把可写权限和可建议权限拆开，执行角色可提议修改，协调器负责提交生效。

第三类权衡是速度与可解释性。多体系统很容易形成“共识黑箱”，即最后大家得出一个结论，但没人说得清这个结论是怎么形成的。对人机混合系统，这是信任问题。TB-CSPN 中的主题阈值可以理解为“按主题相关性动态决定谁该加入这次协作”；层次 Petri 网可以理解为“把复杂协作流程画成可追踪的状态迁移图”。它们的价值不是更智能，而是更透明。

下面这张表总结常见坑和规避手段：

| 常见坑 | 现象 | 规避手段 |
|---|---|---|
| 资源冲突 | 同步时阻塞工具调用和训练 | 图聚类缩小同步组，异步通道分流 |
| 语义碎片 | 同一术语在不同角色中含义不同 | 统一 schema、状态字典、消息模板 |
| 静态协调 | 角色固定，任务变化时低效 | 动态组建，按主题阈值拉起角色 |
| 不可解释共识 | 得到结论但无法追责 | 层次流程图、审计日志、决策快照 |
| 过度并行 | 子任务互相覆盖，结果冲突 | 明确写权限和提交仲裁 |
| 过度通信 | 消息洪泛，吞吐下降 | 摘要传输、批量同步、频率上限 |

制造场景能把这些问题看得最清楚。比如定制生产线上，订单变化会影响排产、工艺参数、库存和交付承诺。若同步组直接覆盖全局计划，PID 时序控制可能稳定，但跨组资源冲突会升高；若完全异步，局部响应快，但客户承诺和现场状态可能不一致。更合理的做法是：

- 同步组只负责强时序局部控制。
- 异步组维护跨组一致性的摘要状态。
- 协调器按延迟预算选择“立即同步”还是“延后合并”。
- 梯度追踪或状态追踪只传摘要，不传全部历史。

这样做的本质，是承认“不可能所有地方都强一致”，然后把一致性预算花在最重要的位置。

## 替代方案与适用边界

不是所有系统都值得上 Multi-Agent。判断标准很简单：拆分收益是否大于协调成本。

如果任务规模小、链路短、审计要求高但实时性一般，中心式协调通常更合适。中心式协调，白话说就是“一个主控负责调度，其他角色只提供能力，不自己组织协作”。它的优点是控制简单、故障定位容易；缺点是主控会成为瓶颈。

如果资源更紧张，甚至可以退回“单一协调器 + 多个工具/提示串联”。这不是严格意义的多智能体，而是工具编排。它适合耗时小、通信少、流程比较固定的场景，例如文档抽取、规则校验、固定格式生成。问题是它对长链规划和异常恢复的能力有限，一旦中间步骤偏了，后续链条大概率一起偏。

下面给出一个简化对比：

| 方案 | 协作复杂度 | 控制粒度 | 资源要求 | 适用场景 |
|---|---|---|---|---|
| Multi-Agent 协作 | 高 | 高 | 高 | 长链任务、模块化目标、需并行 |
| 中心式协调 | 中 | 很高 | 中 | 审计优先、流程稳定、角色有限 |
| 工具串联 | 低 | 中 | 低 | 短流程、固定模板、通信极少 |

所以适用边界可以归纳成三条：

1. 当任务天然可模块化，且子问题之间需要有限但持续的信息交换时，多智能体最有价值。
2. 当系统无法承受频繁通信、角色语义又难统一时，应回退到中心式协调并强化审计。
3. 当目标只是把几个已有工具按固定顺序串起来时，不要硬上多智能体，维护成本会超过收益。

一个非常实用的判断法是问自己两个问题：

- 如果删掉通信层，这些角色还能独立贡献价值吗？
- 如果增加一个角色，系统复杂度是否可预测地上升，而不是失控地上升？

若两个问题都答不上来，说明当前更像“堆模块”，还不是成熟的 Multi-Agent 架构。
