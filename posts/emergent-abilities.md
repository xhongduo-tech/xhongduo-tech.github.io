## 核心结论

涌现能力（Emergent Abilities）指的是：模型规模增长到某个区间后，某些任务能力不是“继续慢慢变好”，而是从接近随机猜测，突然跳到可用水平。这里的“涌现”不是哲学词，它在工程上有明确含义：小模型上的实验结果，不能可靠外推到大模型。

这件事的重要性在于，很多人默认“多加一点参数，效果多好一点”。Wei 等人在 2022 年总结的大模型实验说明，至少在部分任务和部分评测方式下，这个假设不成立。算术推理、多步推理、上下文学习（in-context learning，指模型不更新参数、只靠提示里的示例当场学会任务）都出现过类似现象。

一个玩具例子可以这样理解。假设你在“鱼缸里捞银币”：10B 以下模型像是在水里乱抓，偶尔碰对；到 70B 以上，模型突然能同时处理“先找鱼缸里最大的银币，再判断它来自哪个主题故事”这种多主题问答。它不是每一步都线性变强，而像“前面一直在门外徘徊，突然跨过门槛”。

下面这张表把“跳跃”直观化。数值是示意性的，重点是趋势而不是某一条具体曲线：

| 模型规模 | 3位加法精确匹配 | 多主题问答精确匹配 | 连续文本相似度 |
|---|---:|---:|---:|
| 1B | 接近随机 | 接近随机 | 略高于随机 |
| 10B | 仍接近随机 | 仍接近随机 | 平滑提升 |
| 70B | 明显可用 | 明显可用 | 继续提升 |
| 280B | 高准确率 | 接近人类水平 | 继续提升 |

核心判断是两层：

| 结论 | 含义 |
|---|---|
| 规模可能带来新能力 | 有些任务必须跨过临界规模，才会从“不会”变成“会” |
| 涨幅可能是指标制造的 | 如果评测是精确匹配，曲线会比底层能力更“陡” |

## 问题定义与边界

讨论涌现，先要把边界画清楚。否则任何“效果涨了很多”都能被叫做涌现，这会把概念冲淡。

第一，涌现不是普通提升。普通提升是模型从 40 分涨到 45 分，再到 50 分。涌现强调的是非连续行为：小模型几乎没有任务能力，过了某个规模后突然进入可用区间。

第二，涌现必须绑定具体任务和具体指标。离开任务谈“模型有涌现能力”是空话。比如：

| 任务 | 常见指标 | 为什么容易看到“跳跃” |
|---|---|---|
| 3位加法 | Exact Match，精确匹配 | 结果只要错一位就记 0 分 |
| 多步数学推理 | 最终答案精确匹配 | 中间过程稍错，整题归零 |
| 多主题问答 | 文本完全匹配或分类准确率 | 离散标签导致分段式变化 |
| 上下文学习 | few-shot 准确率 | 小模型可能完全不会利用示例 |

第三，涌现不等于“模型突然理解世界”。更严格的说法是：在某个任务定义和评测定义下，性能曲线表现出阈值式增长。Schaeffer 等人在 2023 年提出争议：所谓“涌现”可能不是底层能力真的不连续，而是因为用了非连续指标，比如 exact match。模型底层 token 概率可能一直在平滑改进，只是到了某一段后，终于足以把整道题完整答对，于是表面上看像突然跳起来。

这个机制可以用一个常见近似解释：

$$
\mathrm{Accuracy}(N)\approx \exp(-L\cdot \mathcal{L}_{CE}(N))
$$

其中：

- $N$ 是模型参数量
- $\mathcal{L}_{CE}(N)$ 是交叉熵损失，白话说，就是模型平均“猜错下一个词”的程度
- $L$ 是目标序列长度
- Accuracy 用的是精确匹配这类离散指标

如果 $L$ 很长，那么即使每个 token 的预测只变好一点，整段序列“全部正确”的概率也会发生陡增。这就是“看起来突然会了”的来源之一。

给零基础读者一个直观版本：你坐在小火车上看站牌，车厢短时看不清；多挂几节车厢后，你不是“一点点更清楚”，而是突然能完整读出站名。底层视野是在连续变好，但“能不能完整读出来”这个判定是离散的。

## 核心机制与推导

Wei 2022 的观察是经验现象，Schaeffer 2023 进一步解释了为什么这种现象在某些指标下会特别明显。核心机制是：底层损失随规模按幂律下降，而离散指标会把这种平滑下降放大成“阈值跳跃”。

先写出经验形式：

$$
\mathcal{L}_{CE}(N)=\left(\frac{N}{c}\right)^{\alpha}, \quad \alpha<0
$$

这里：

- $c$ 是一个缩放常数
- $\alpha<0$ 表示模型越大，损失越低

再看序列级精确匹配。若我们粗略假设长度为 $L$ 的目标序列，每个位置都要预测正确，且整体正确率与总负对数概率相关，则可得近似：

$$
\mathrm{Accuracy}(N)\approx \exp(-L\cdot \mathcal{L}_{CE}(N))
$$

代入上式：

$$
\mathrm{Accuracy}(N)\approx \exp\left(-L\cdot \left(\frac{N}{c}\right)^{\alpha}\right)
$$

因为 $\alpha<0$，所以当 $N$ 增大时，$\left(\frac{N}{c}\right)^{\alpha}$ 会下降。问题在于它被放进了指数函数里。指数函数会把“损失的小幅下降”映射成“准确率的大幅上升”。

推导的工程含义可以拆成三点：

| 变量 | 变化 | 对结果的影响 |
|---|---|---|
| 参数量 $N$ | 增大 | 交叉熵下降 |
| 序列长度 $L$ | 增大 | 跳跃更尖锐 |
| 评测是否离散 | 越离散 | 越容易看到“突然出现” |

这解释了为什么长链路任务更容易出现涌现表象。因为长度 $L$ 越大，只要单 token 预测还有一点不稳，整个序列就容易全盘归零；一旦单 token 概率再提高一点，整段 suddenly pass。

可以把曲线想象成一张草图：

- 横轴：模型参数量，从 1B 到 300B
- 纵轴：精确匹配准确率
- 前半段：几乎贴着地面，像没学会
- 靠近阈值：曲线迅速抬头
- 后半段：继续上升，但边际增幅变缓

这不是说模型内部必然发生了“神秘相变”。更谨慎的说法是：缩放带来的底层能力提升，被评测函数非线性放大了。

真实工程例子是 GSM8K 一类小学数学题。Chain-of-Thought（链式思维提示，指要求模型显式写出中间推理步骤）在超大模型上能显著提升结果，但在小模型上常常没用，甚至更差。原因并不神秘：小模型连终点答案分布都不稳，再强迫它生成长推理链，只会把错误扩大。

## 代码实现

下面给一个可运行的 Python 玩具实现。它不依赖真实大模型，而是模拟“模型越大，交叉熵越低；精确匹配呈现阈值跳跃”的现象，同时比较连续指标和离散指标的差异。

```python
import math

def cross_entropy_loss(params_billion, c=20.0, alpha=-0.55):
    """
    params_billion: 模型参数量，单位 B
    返回一个简化后的交叉熵近似值
    """
    return (params_billion / c) ** alpha

def exact_match_accuracy(params_billion, seq_len=12):
    """
    近似公式：Accuracy(N) ≈ exp(-L * L_CE(N))
    这里只做演示，不代表真实论文中的精确数值
    """
    ce = cross_entropy_loss(params_billion)
    acc = math.exp(-seq_len * ce)
    return max(0.0, min(1.0, acc))

def continuous_score(params_billion):
    """
    连续指标示意：把损失映射成一个平滑上升分数
    """
    ce = cross_entropy_loss(params_billion)
    score = 1.0 / (1.0 + ce)
    return max(0.0, min(1.0, score))

sizes = [1, 3, 10, 30, 70, 130, 280]

results = []
for n in sizes:
    em = exact_match_accuracy(n, seq_len=12)
    cont = continuous_score(n)
    results.append((n, round(em, 4), round(cont, 4)))

for row in results:
    print(f"{row[0]:>4}B | exact_match={row[1]:.4f} | continuous={row[2]:.4f}")

# 基本断言：规模越大，两个指标都不应下降
ems = [exact_match_accuracy(n, seq_len=12) for n in sizes]
conts = [continuous_score(n) for n in sizes]

assert all(x <= y for x, y in zip(ems, ems[1:])), "精确匹配应单调不减"
assert all(x <= y for x, y in zip(conts, conts[1:])), "连续分数应单调不减"

# 精确匹配的“跳跃”应该比连续指标更明显
jump_em = exact_match_accuracy(70, 12) - exact_match_accuracy(10, 12)
jump_cont = continuous_score(70) - continuous_score(10)
assert jump_em > jump_cont, "离散指标的跳跃应更明显"
```

如果你把这段代码画成图，通常会看到：

- `continuous_score` 平滑上升
- `exact_match_accuracy` 在某一段突然变陡

这正是“Mirage”争议的核心：底层能力可能一直在涨，只是 exact match 把它显示成了断崖式跳跃。

如果换成真实工程流程，最小实现通常是下面三步：

1. 准备同一模型家族的多个 checkpoint，比如 7B、13B、34B、70B。
2. 在同一数据集上跑同一提示模板，例如 GSM8K 的 standard prompt 和 chain-of-thought prompt。
3. 同时计算两类指标：
   - 离散指标：最终答案精确匹配
   - 连续指标：token 对数概率、编辑距离、Brier 分数

一个真实工程例子是客服问答或数学辅导系统。你可能有一个“先读题，再分步求解，再给最终答案”的提示模板。若模型只有 20B 左右，却强行启用 CoT，它看起来会更流畅，但最终答案可能更差。此时应该先验证它是否已经跨过任务阈值，而不是默认“推理链越长越好”。

## 工程权衡与常见坑

第一类坑是把指标当能力本身。精确匹配很适合自动评测，但它会把“差一点正确”和“完全胡说”都记成 0。于是小模型看起来像完全不会，大模型看起来像突然学会。换成编辑距离、token 级对数概率后，曲线经常平滑得多。

| 指标 | 类型 | 优点 | 风险 |
|---|---|---|---|
| Exact Match | 离散 | 易解释、易自动化 | 容易制造“突跃” |
| 编辑距离 | 连续 | 能区分“差一点” | 对语义等价不敏感 |
| Token 概率 | 连续 | 反映底层建模能力 | 不直观 |
| Brier 分数 | 连续 | 适合概率校准 | 需要概率输出稳定 |

第二类坑是误用 Chain-of-Thought。它的结论不是“所有模型都应该写步骤”，而是“足够大的模型在推理任务上能从步骤提示中获益”。对于小模型，CoT 常见问题是：

| 现象 | 本质 |
|---|---|
| 文字更长 | 不等于推理更强 |
| 步骤看起来像真的 | 可能只是语言模式更顺滑 |
| 最终答案更差 | 说明中间链条在放大误差 |

第三类坑是跨模型家族乱比较。不同架构、训练语料、数据清洗、tokenizer、训练步数都可能改变曲线。如果你把“70B 的 A 模型”与“13B 的 B 模型”对比后说发现了涌现，结论通常不稳。更合理的做法是比较同一家族 checkpoint。

第四类坑是把“能力阈值”当成固定参数量。不存在一个全行业统一的“60B 就会推理”。阈值依赖任务难度、提示方式、答案长度、评测标准。更准确的表达是：在某些任务和某些评测下，该家族模型在这个规模区间附近出现陡升。

## 替代方案与适用边界

如果你没有 100B 以上模型，不应该期待“小模型多写几步就能补上差距”。更现实的替代方案是把任务拆开，让小模型只做它擅长的部分。

| 任务类型 | 是否强依赖涌现 | 推荐策略 |
|---|---|---|
| 长链数学推理 | 高 | 大模型 + CoT，或外部求解器 |
| 多跳问答 | 高 | 大模型，或检索增强后分阶段求解 |
| 简单分类 | 低 | 小模型即可 |
| 信息抽取 | 低 | 小模型 + 模板提示 |
| 固定流程问答 | 中 | 小模型 + 工具调用 + 规则约束 |

对初级工程师来说，可以把选择规则记成一句话：如果任务要求“多步稳定推理”，先假设它需要更大模型或外部工具；如果任务只是“格式化、抽取、改写、检索后重组”，小模型通常已经够用。

常见替代路径有三类：

| 方案 | 适用边界 | 代价 |
|---|---|---|
| 小模型 + 模板提示 | 结构化任务 | 泛化有限 |
| 小模型 + 检索/计算器/代码执行 | 有外部知识或明确规则的任务 | 系统复杂度上升 |
| 蒸馏或微调 | 任务分布稳定 | 维护成本增加 |

例如你只有 20B 模型，又要做财务公式问答。正确策略不是期待它自然涌现出多步算术能力，而是：

1. 把问题解析成结构化字段。
2. 用外部计算模块完成公式求值。
3. 让模型只负责解释结果和组织语言。

这类设计本质上是在绕开“必须等规模跨阈值”的部分，把最难的推理外包给确定性工具。

因此，涌现能力的适用边界可以总结成一句更严格的话：它提醒我们“规模可能是某些能力的必要条件”，但不意味着所有能力都只能靠堆参数获得。
