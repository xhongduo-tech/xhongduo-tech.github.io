## 核心结论

信道容量是“一个带噪传输系统在误码率可以压到任意小时，最多还能稳定传多少信息”的上限。形式化地写：

$$
C=\max_{p(x)} I(X;Y)
$$

其中，互信息 $I(X;Y)$ 表示“输入 $X$ 里有多少信息真正传到了输出 $Y$”。白话说，输入发了很多内容，但经过噪声后，只有一部分还能被接收端可靠区分，这部分就是有效信息。

香农信道编码定理给出两个结论：

1. 若传输率 $R<C$，存在某种编码方法，使误码率随着码长增加而趋近于 0。
2. 若传输率 $R>C$，不管编码多复杂，错误率都不可能被压到 0，且会有正下界。

这说明容量不是“某种编码的性能”，而是“整个信道物理条件决定的极限”。

常见模型里，容量可以明确写出来：

| 信道模型 | 参数含义 | 容量表达 |
| --- | --- | --- |
| BSC$(p)$ 二进制对称信道 | $p$ 是比特翻转概率 | $C=1-H_{\mathrm b}(p)$ |
| AWGN 加性高斯白噪声信道 | SNR 是信噪比 | $C=\frac12\log_2(1+\mathrm{SNR})$ bit/实维 |
| 带宽为 $B$ 的 AWGN | $B$ 是频带宽度 | $C=B\log_2(1+S/N)$ |

其中二元熵函数为：

$$
H_{\mathrm b}(p)=-p\log_2 p-(1-p)\log_2(1-p)
$$

玩具例子：BSC$(0.1)$ 表示每个比特有 10% 概率翻转。它的容量约为 $0.531$ bit/符号，意思不是“每次只能发半个比特”，而是“长期平均下来，每个信道使用只能可靠携带 0.531 bit 的信息”。

真实工程例子：电话线路若带宽约 4 kHz，SNR 约 20 dB，则 Shannon-Hartley 上限约为 $26.6$ kb/s。这个值不是某个 modem 的宣传参数，而是这条线路在该噪声条件下的理论极限。

## 问题定义与边界

“信道”是输入到输出的随机映射。白话说，同样的输入发两次，输出可能不完全一样，因为中间有噪声。

### 1. 离散信道：BSC

BSC，Binary Symmetric Channel，二进制对称信道，表示输入只有 0 和 1，且每个符号独立地以概率 $p$ 翻转。

- 发 0，收到 1 的概率是 $p$
- 发 1，收到 0 的概率也是 $p$

它适合讲清楚容量的最基本思想，因为模型简单、公式闭合。

对新手最重要的一点是：容量不是“无错传输”，而是“允许先编码、再传输、最后解码，并且码长可以很长时，能达到的稳定速率上限”。

### 2. 连续信道：AWGN

AWGN，Additive White Gaussian Noise，加性高斯白噪声信道，表示接收端看到的是：

$$
Y=X+N
$$

其中 $N$ 是高斯噪声。白话说，发送的模拟信号会被叠加一层随机扰动。

AWGN 模型常用于无线通信、光纤、部分有线系统的基础分析。它的容量公式：

$$
C=B\log_2(1+S/N)
$$

这里：

- $B$ 是带宽，表示单位时间可用的频率范围
- $S/N$ 是信噪比，表示信号强度相对噪声强度的比例

如果把容量按“每个实数维度”来计量，可以写成：

$$
C=\frac12\log_2(1+\mathrm{SNR})
$$

### 3. 边界条件

香农定理成立有前提，不能忽略：

| 条件 | 含义 | 如果不满足会怎样 |
| --- | --- | --- |
| 足够长的码块 | 一次不是发几个比特，而是发很长的编码块 | 误码率可能远高于理论极限 |
| 合适的编码与解码 | 要有能逼近容量的码，例如 LDPC、Polar | 仅有“低于容量”的码率仍不够 |
| 模型近似成立 | 实际信道应近似独立、平稳、已知统计规律 | 公式可能失效 |
| 功率或带宽限制明确 | 容量总是在约束下定义 | 不加约束时容量问题本身不完整 |

所以，“$R<C$ 就一定没问题”是错误理解。正确说法是：在满足假设并允许足够复杂编码时，存在方法让误差任意小。

## 核心机制与推导

容量公式的核心是互信息。

### 1. 互信息为什么是核心量

熵 $H(X)$ 表示“不确定性大小”。白话说，某个随机变量越难预测，熵越高。

条件熵 $H(Y|X)$ 表示“知道输入后，输出还剩多少不确定性”。在信道问题里，这部分主要来自噪声。

于是：

$$
I(X;Y)=H(Y)-H(Y|X)
$$

含义是：输出里总共有多少信息，减去噪声造成的那部分不可控不确定性，剩下的才是输入真正传过来的信息。

但输入分布 $p(x)$ 可以由发送端设计，所以容量定义为：

$$
C=\max_{p(x)} I(X;Y)
$$

即“找一种最合适的输入分布，让信道尽可能多地传信息”。

### 2. BSC 容量为什么是 $1-H_{\mathrm b}(p)$

对 BSC，若输入均匀分布，即：

$$
P(X=0)=P(X=1)=\frac12
$$

则输出也均匀，因此：

$$
H(Y)=1
$$

而给定输入后，输出只剩“翻还是不翻”这一个随机性，所以：

$$
H(Y|X)=H_{\mathrm b}(p)
$$

因此：

$$
I(X;Y)=H(Y)-H(Y|X)=1-H_{\mathrm b}(p)
$$

进一步可以证明，BSC 上均匀输入就是最优分布，所以：

$$
C=1-H_{\mathrm b}(p)
$$

#### 玩具例子：$p=0.1$

先算二元熵：

$$
H_{\mathrm b}(0.1)=-(0.1)\log_2 0.1-(0.9)\log_2 0.9 \approx 0.469
$$

因此容量：

$$
C=1-0.469\approx 0.531
$$

这表示每次使用这个信道，平均最多只能可靠带走 0.531 bit 信息。

如果你硬要按 1 bit/符号发原始比特，不做编码，系统就会直接承受 10% 的翻转错误；如果你加冗余编码，才有机会把有效信息率控制在 0.531 以下并把误码率压低。

### 3. AWGN 容量为什么和 $\log(1+\mathrm{SNR})$ 有关

AWGN 的输出是“信号 + 噪声”。容量推导需要连续随机变量和微分熵，这里给直观机制：

1. 噪声方差固定时，发送功率越大，信号和噪声越容易区分。
2. 但区分能力不是线性增长，而是对数增长，因为高 SNR 区间里继续增大发送功率，收益会逐渐变小。
3. 在平均功率受限条件下，高斯输入分布最优，因此得到 Shannon-Hartley 公式。

对应表达式：

$$
C=B\log_2(1+S/N)
$$

如果只看每个实数自由度：

$$
C=\frac12\log_2(1+\mathrm{SNR})
$$

一个常用近似是：SNR 每提升 3 dB，大约乘以 2，因此容量增加约：

$$
\frac12\log_2 2=0.5
$$

bit/实维；如果按复维或每赫兹双边定义，常见口语会说“每增加 3 dB，容量约增加 1 bit/s/Hz”，本质上取决于你采用哪种维度记法。工程里必须先说明单位，否则容易混淆。

### 4. 为什么 $R<C$ 可低误，而 $R>C$ 不可避免出错

正向部分的直觉是“随机编码 + 典型集解码”。

- 随机编码：从巨大码本中挑出彼此足够可分离的码字
- 典型集：长序列会表现出稳定统计规律
- 解码：接收端选择与输出最匹配的那个码字

当 $R<C$ 时，码字之间“间距”足够，噪声不容易把一个码字伪装成另一个。随着码长 $n$ 增大，误码概率可以做到极小。

反向部分说明：若 $R>C$，码字塞得太密，输出中可区分的信息量不够，多个候选消息会落入相同的“可观察区域”，错误就不可避免。

可以把它理解成一个边界：

| 码率区间 | 结论 |
| --- | --- |
| $R<C$ | 理论上可构造低误码编码 |
| $R=C$ 附近 | 需要很长码长和复杂编码，工程实现困难 |
| $R>C$ | 不存在任意低误码方案 |

## 代码实现

下面给出一个可运行的 Python 例子，计算 BSC 容量和 Shannon-Hartley 容量，并用简单重复码做一个 Monte Carlo 小实验。重复码不是好码，它只是为了让新手看到“码率下降后，错误可能下降”。

```python
import math
import random


def binary_entropy(p: float) -> float:
    """二元熵 H_b(p)，p 是比特翻转概率。"""
    if p < 0 or p > 1:
        raise ValueError("p must be in [0, 1]")
    if p == 0 or p == 1:
        return 0.0
    return -(p * math.log2(p) + (1 - p) * math.log2(1 - p))


def bsc_capacity(p: float) -> float:
    """BSC(p) 的容量，单位 bit/符号。"""
    return 1.0 - binary_entropy(p)


def shannon_capacity(bandwidth_hz: float, snr_linear: float) -> float:
    """Shannon-Hartley 容量，单位 bit/s。"""
    if bandwidth_hz <= 0 or snr_linear < 0:
        raise ValueError("bandwidth must be > 0 and snr must be >= 0")
    return bandwidth_hz * math.log2(1 + snr_linear)


def db_to_linear(db: float) -> float:
    """dB 转线性信噪比。"""
    return 10 ** (db / 10.0)


def send_bsc_bit(bit: int, p: float) -> int:
    """通过 BSC 发送单个 bit。"""
    flipped = random.random() < p
    return bit ^ int(flipped)


def repetition_encode(bit: int, n: int) -> list[int]:
    """n 重复码，码率 R = 1/n。"""
    return [bit] * n


def repetition_decode(bits: list[int]) -> int:
    """多数投票解码。"""
    ones = sum(bits)
    zeros = len(bits) - ones
    return 1 if ones > zeros else 0


def simulate_repetition_ber(p: float, n: int, trials: int = 20000) -> float:
    """估计 n 重复码在 BSC(p) 上的块错误率。"""
    errors = 0
    for _ in range(trials):
        msg = random.randint(0, 1)
        codeword = repetition_encode(msg, n)
        received = [send_bsc_bit(b, p) for b in codeword]
        decoded = repetition_decode(received)
        if decoded != msg:
            errors += 1
    return errors / trials


# 数值检查
cap_bsc_01 = bsc_capacity(0.1)
assert abs(cap_bsc_01 - 0.5310) < 0.01

snr_20db = db_to_linear(20)  # 20 dB -> 100
cap_phone = shannon_capacity(4000, snr_20db)
assert abs(cap_phone - 26631.0) < 200.0  # 约 26.6 kb/s

# 简单模拟：重复码越长，码率越低，错误率通常越低
random.seed(0)
ber_n1 = simulate_repetition_ber(p=0.1, n=1, trials=5000)  # 无编码
ber_n3 = simulate_repetition_ber(p=0.1, n=3, trials=5000)
ber_n5 = simulate_repetition_ber(p=0.1, n=5, trials=5000)

assert ber_n3 < ber_n1
assert ber_n5 < ber_n3

print("BSC(0.1) capacity =", round(cap_bsc_01, 4), "bit/use")
print("Phone-line capacity ≈", round(cap_phone / 1000, 2), "kb/s")
print("repetition n=1 error =", round(ber_n1, 4))
print("repetition n=3 error =", round(ber_n3, 4))
print("repetition n=5 error =", round(ber_n5, 4))
```

这段代码展示了三件事：

1. BSC$(0.1)$ 的容量确实接近 0.531。
2. 4 kHz、20 dB 的电话线路容量约为 26.6 kb/s。
3. 即使是很低效的重复码，只要牺牲码率，错误率也会下降。

但必须强调：重复码远远达不到容量极限。现实系统会用卷积码、Turbo、LDPC、Polar 等更高效的编码。

如果要做“误码率随 $R/C$ 变化”的实验，可以用这样的伪流程：

1. 固定信道参数，如 BSC 的 $p$ 或 AWGN 的 SNR
2. 选择一组编码方案，对应不同码率 $R$
3. 对每个 $R$ 做大量随机消息仿真
4. 统计块错误率或比特错误率
5. 观察当 $R$ 接近或超过 $C$ 时，误码率如何恶化

真实工程例子：通信芯片设计里，工程师会先根据目标 SNR 算理论容量，再决定用什么调制阶数和编码率。比如 5G 里高 SNR 用户可以上更高码率，低 SNR 用户则要退回更保守的 MCS（调制编码方案），本质上就是在追随“不能超过当前有效容量”这个原则。

## 工程权衡与常见坑

### 1. “$R<C$ 就稳了”是最常见误解

理论结论说的是“存在某种编码”。工程系统还要考虑：

- 码长越长，性能越接近容量，但延迟越高
- 解码算法越强，复杂度和功耗越高
- 实际信道可能时变，容量不是固定常数

| 常见误解 | 实际问题 | 规避方式 |
| --- | --- | --- |
| $R<C$ 就一定低误码 | 编码不够强、块长太短 | 选现代编码并做链路仿真 |
| 容量公式就是吞吐量 | 还受协议开销、重传、同步损失影响 | 区分物理层容量与系统吞吐 |
| 只看平均 SNR 就够 | 深衰落、突发噪声会破坏假设 | 用 outage、链路自适应、交织 |
| 高 SNR 就无限增速 | 容量是对数增长，收益递减 | 结合带宽、功率、复杂度做优化 |

### 2. 容量逼近区很贵

越接近容量边界，系统越“脆”。

- 需要更长块长
- 需要更复杂解码
- 对信道估计误差更敏感
- 实际实现损失更显著

因此工程上往往不会把系统设计在“紧贴容量”的位置，而是保留 margin，也就是安全余量。

### 3. LLM 里的“上下文容量”类比

这个类比不是说大模型真的是香农信道，而是说它也存在“有限资源下的信息传递上限”。

- 上下文窗口有限：能直接放进去的 token 数有限
- 注意力和隐藏状态有限：不是所有输入都能完整保留
- 浮点精度有限：信息在多层传播中会压缩、混叠、衰减

可以把 prompt 到最终隐藏表示看作一个“记忆通道”。如果任务需要的信息熵大于这个通道能稳定保留的信息量，就会出现遗忘、错配、细节丢失。

玩具类比：你往一个只能装 1 升水的容器里灌 2 升水，不是“装得更满”，而是直接溢出。长 prompt 里塞太多同等级信息，后面的关键信号不一定能被稳定传递。

真实工程例子：检索增强生成里，如果一次塞入 20 段检索文本，而其中只有 3 段真正相关，那么大量冗余文本会占用上下文“容量”，使模型对关键证据的注意力稀释，答案反而变差。好的 RAG 系统不会只追求“召回更多”，而是追求“在有限上下文里传递更多有效信息”。

### 4. 联邦学习和蒸馏里的通信上限

- 联邦学习中，客户端上传梯度或模型更新，本质上是在有限带宽上发送训练信息。
- 模型蒸馏中，教师模型向学生模型传递软标签、特征或中间表示，也是在有限表示空间里传信息。

如果压缩过猛，量化噪声过大，等价于“信道太差”，有效互信息下降，最终会影响收敛速度和性能上界。

## 替代方案与适用边界

当系统逼近容量上限时，出路通常不是“硬顶”，而是改变约束条件或改进编码。

| 方案 | 作用 | 适用边界 |
| --- | --- | --- |
| 更强编码，如 LDPC、Polar | 在相同信道下更接近容量 | 可接受较高编解码复杂度 |
| 增加带宽 | 直接提高 AWGN 容量 | 频谱资源允许时 |
| 增加发射功率 | 提高 SNR | 受能耗、法规、干扰限制 |
| 多天线 MIMO | 增加并行空间通道 | 需要多天线和良好信道条件 |
| 自适应调制编码 | 根据当前信道切换码率 | 信道状态可测且变化不太快 |
| 交织与重传 | 对抗突发错误 | 时延可容忍时 |

### 1. 不同场景下的替代思路

对通信系统，接近容量的主流路径是“现代编码 + 自适应链路 + 多路复用”。

对 LLM，上下文容量受限时，更有效的做法不是无脑加长 prompt，而是做“互信息导向筛选”：

- 保留与目标最相关的上下文
- 删除重复、模板化、低增益信息
- 对长材料先压缩成结构化摘要
- 让证据按任务相关性排序

可以写成一个非常粗糙的评分思想：

$$
\text{score}(d)=\alpha \cdot \text{relevance}(d,q)-\beta \cdot \text{redundancy}(d,\mathcal D)
$$

其中 $d$ 是候选片段，$q$ 是问题。意思是：选择“相关性高、冗余低”的内容，尽量把有限上下文留给高信息密度片段。

真实工程例子：在客服问答系统中，普通提示可能把整篇产品文档都塞进去；更好的做法是先检索出最相关段落，再做摘要与去重，只把与当前问题直接相关的规格、约束和异常条件放入 prompt。这样不是“传了更多 token”，而是“传了更多有效信息”。

### 2. 适用边界

容量理论很强，但不要滥用。

- 它能给出“上限”和“趋势”，但不直接给出最优工程实现
- 它适合分析长码、平均性能、统计信道
- 它不自动覆盖强时变、对抗性扰动、非平稳数据分布

LLM 类比尤其要克制。上下文、注意力、梯度通信都可以借用“容量约束”和“互信息瓶颈”来理解，但不能把香农公式原封不动套到模型上。正确用法是把它当成分析框架：资源有限，信息可传递量就有限，因此需要做筛选、压缩、编码和冗余控制。
