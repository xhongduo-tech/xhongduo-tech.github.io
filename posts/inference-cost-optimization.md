## 核心结论

推理成本优化的目标，不是把实例小时单价压到最低，而是把单位有效输出的成本压到最低。这里的“有效输出”指真正成功交付给调用方、没有因为中断、重试、冷启动和排队而被浪费的 token。

用统一符号表示，最核心的两个公式是：

$$
r_{eff}=r \times a \times (1-\rho)
$$

$$
C_{1M}=\frac{10^6 \times P}{3600 \times r_{eff}}
$$

其中，`有效吞吐`就是扣除不可用时间和额外损失后的真实吞吐；`每百万 token 成本`就是交付 100 万个有效 token 需要花多少钱。

这两个公式说明一件事：推理成本由 `P`、`r`、`a`、`ρ` 共同决定，而不是只由 GPU 单价决定。

- `spot` 实例单价低，但可能被回收，中断会拉低 `a`，还会抬高 `ρ`
- `预留实例` 单价通常更高，但稳定性更强，适合把 `a` 做高、把波动做低
- `混合部署` 不是折中说法，而是把稳定负载和弹性负载分开承接，从而同时优化成本和可用性

最容易误判的点是：便宜资源不一定带来更低有效成本。单价低但频繁中断，最后可能更贵。

| 方案 | 小时单价 `P` | 标称吞吐 `r` | 可用率 `a` | 损失率 `ρ` | 每百万有效 token 成本 `C_1M` |
|---|---:|---:|---:|---:|---:|
| 稳定实例 | 24 $/h | 400 tok/s | 1.00 | 0.00 | 16.67 $ |
| 低价 spot | 14 $/h | 400 tok/s | 0.55 | 0.10 | 19.64 $ |

上表已经足够说明问题：裸价便宜 41.7%，不代表单位有效输出更便宜。

## 问题定义与边界

本文讨论的是**推理服务成本**。推理服务就是模型已经训练好以后，对外提供文本生成、问答、摘要、分类等在线或离线能力的那一层系统。它不是训练成本，也不是单纯的云账单统计。

更准确地说，本文讨论的是“为了交付有效 token，系统总共要承担多少资源成本”。这比“某台机器多少钱一小时”更接近工程现实。

先定义变量：

| 符号 | 含义 | 单位 | 增大后对成本的影响 |
|---|---|---|---|
| `P` | 实例小时成本 | $/h | 越大，成本越高 |
| `r` | 稳定吞吐 | tokens/s | 越大，成本越低 |
| `a` | 有效可用率 | 0~1 | 越大，成本越低 |
| `ρ` | 额外损失率 | 0~1 | 越大，成本越高 |
| `C_1M` | 每百万有效 token 成本 | $/1M tokens | 目标最小化 |

这里有两个术语需要先讲清楚。

`吞吐`，白话说，就是系统每秒能稳定吐出多少 token。  
`可用率`，白话说，就是资源真正能工作、能接单、能持续完成请求的时间占比。

`ρ` 不是一个单一来源，它通常包含：

- 冷启动损失：实例刚拉起时，模型加载、权重映射、KV cache 预热都会浪费时间
- 重试损失：请求执行到一半失败，又重新做一遍
- 回放丢失：流式输出中断后，前面部分不能直接复用
- 排队超时：用户等不到结果就超时退出，后续计算价值下降
- 缓存失效：实例被回收后，局部缓存和批处理上下文丢失

一个在线问答服务就很典型。白天有稳定流量，要求低延迟；夜里可能跑一批长文本摘要，要求是吞吐和成本优先。这两类负载不能用同一套策略硬套。前者更在意 `a` 和尾延迟，后者更能容忍 `spot` 的中断。

本文不讨论的边界也要明确：

| 不讨论项 | 原因 |
|---|---|
| 模型训练成本 | 目标不同，训练关注收敛成本，不是在线交付成本 |
| 模型效果优劣 | 默认模型能力已选定，本文只比部署与交付效率 |
| 多租户复杂计费细节 | 可并入 `P` 或 `ρ`，但不是主线 |
| 人力运维成本精算 | 重要，但难统一量化，本文只点到服务复杂度影响 |

所以本文的边界很清楚：讨论同一类推理任务，如何比较不同资源策略的单位有效输出成本。

## 核心机制与推导

先从公式本身出发。假设某实例理论吞吐是 `r`，但它不是 100% 时间都在稳定输出。中断、迁移、冷启动会让部分时间不可用，因此要乘一个 `a`。即使资源可用，也可能因为重试、回放、预热而产生浪费，所以还要再乘上 $(1-\rho)$。

于是：

$$
r_{eff}=r \times a \times (1-\rho)
$$

这一步的含义很直接：你真正卖给业务的不是理论吞吐，而是有效吞吐。

再看成本。一个实例每小时花费 `P` 美元，一小时有 3600 秒，因此一小时可以交付的有效 token 数是：

$$
3600 \times r_{eff}
$$

那交付 100 万有效 token 的成本自然是：

$$
C_{1M}=\frac{10^6 \times P}{3600 \times r_{eff}}
$$

把前一个公式代入，得到：

$$
C_{1M}=\frac{10^6 \times P}{3600 \times r \times a \times (1-\rho)}
$$

这个式子揭示了资源类型的真正影响路径：

| 资源因素 | 主要影响变量 | 典型结果 |
|---|---|---|
| 实例单价变化 | `P` | 裸价下降不一定带来有效成本下降 |
| GPU/专用芯片性能 | `r` | 吞吐更高，单位输出成本通常下降 |
| 被回收、中断、抢占 | `a` | 可用率下降，成本迅速上升 |
| 冷启动、重试、回放 | `ρ` | 损失变大，成本上升 |
| 抖动、排队、尾延迟 | `a` 与 `ρ` 的实际有效值 | 理论值和线上值偏离 |

### 玩具例子

假设两个方案的标称吞吐都一样，都是 `400 tokens/s`。

- 方案 A：稳定实例，`P=24`，`a=1.0`，`ρ=0`
- 方案 B：spot 实例，`P=14`，`a=0.55`，`ρ=0.10`

计算：

- A：$r_{eff}=400 \times 1.0 \times 1=400$
- B：$r_{eff}=400 \times 0.55 \times 0.9=198$

于是：

- A：$C_{1M}=10^6 \times 24 / (3600 \times 400)=16.67$
- B：$C_{1M}=10^6 \times 14 / (3600 \times 198)\approx 19.64$

结论反转了。虽然 B 单价低，但有效成本更高。

### 真实工程例子

考虑一个 LLM 在线问答系统。白天有持续用户请求，要求首 token 延迟低；晚上还要跑客服对话摘要和知识库重写。很多团队一开始会犯一个错误：把所有流量都压到最便宜的 spot 池上，以为账面价格最好看。

结果通常是：

1. 白天高峰期 spot 被回收，在线请求重试
2. 新实例拉起要重新加载模型，TTFT 明显恶化
3. 流式回答中断，用户侧感知为卡顿或超时
4. 为了兜底，又临时扩稳定池，调度变复杂

最后账单未必最低，SLO 还先坏了。

更合理的做法是：

- 稳定底座流量放在预留实例或长期稳定资源上
- 峰值增量和夜间批任务放到 spot 池
- 无状态请求支持自动重放
- 监控 `tokens/s`、`queue_duration`、`TTFT`、中断率和重试率

这样做的本质，不是“同时用了两种资源”，而是用资源类型去匹配负载特征。

简化成本分解可以写成：

| 成本项 | 是否直接体现在单价 | 对有效成本的影响方式 |
|---|---|---|
| 实例账单 | 是 | 进入 `P` |
| 中断停机 | 否 | 拉低 `a` |
| 模型重载 | 否 | 抬高 `ρ` |
| 请求重试 | 否 | 抬高 `ρ` |
| 排队超时 | 否 | 让有效交付下降，等价于拉低 `a` 或抬高 `ρ` |

这就是为什么“只看 GPU 价格”是错误抽象。工程里真正付钱买的是稳定交付能力。

## 代码实现

把公式落成代码后，容量规划会清晰很多。下面给一个最小可运行的 Python 版本。它不依赖第三方库，输入 4 个核心参数，输出每百万有效 token 成本。

```python
from dataclasses import dataclass

@dataclass
class Plan:
    name: str
    price_per_hour: float   # P, $/h
    tokens_per_sec: float   # r, tokens/s
    availability: float     # a, 0~1
    loss_rate: float        # rho, 0~1

def effective_throughput(plan: Plan) -> float:
    assert plan.price_per_hour > 0
    assert plan.tokens_per_sec > 0
    assert 0 <= plan.availability <= 1
    assert 0 <= plan.loss_rate < 1
    return plan.tokens_per_sec * plan.availability * (1 - plan.loss_rate)

def cost_per_million_tokens(plan: Plan) -> float:
    r_eff = effective_throughput(plan)
    return 1_000_000 * plan.price_per_hour / (3600 * r_eff)

stable = Plan(
    name="reserved-base",
    price_per_hour=24.0,
    tokens_per_sec=400.0,
    availability=1.0,
    loss_rate=0.0,
)

spot = Plan(
    name="spot-pool",
    price_per_hour=14.0,
    tokens_per_sec=400.0,
    availability=0.55,
    loss_rate=0.10,
)

stable_cost = cost_per_million_tokens(stable)
spot_cost = cost_per_million_tokens(spot)

print(stable.name, round(stable_cost, 2))
print(spot.name, round(spot_cost, 2))

assert round(stable_cost, 2) == 16.67
assert round(spot_cost, 2) == 19.64
assert stable_cost < spot_cost
```

这个例子做了三件事：

- 把抽象公式变成可计算函数
- 用 `assert` 固化最基本的输入约束
- 直接比较两个方案的单位有效输出成本

如果想更贴近工程，可以把多种候选方案放进列表里排序：

```python
plans = [
    stable,
    spot,
    Plan("hybrid-estimated", 19.0, 400.0, 0.92, 0.03),
]

ranked = sorted(
    [(p.name, round(cost_per_million_tokens(p), 2)) for p in plans],
    key=lambda x: x[1]
)

print(ranked)
assert ranked[0][0] in {"reserved-base", "hybrid-estimated"}
```

计算结果可以整理成表：

| 方案 | `P` | `r` | `a` | `ρ` | `r_eff` | `C_1M` |
|---|---:|---:|---:|---:|---:|---:|
| reserved-base | 24 | 400 | 1.00 | 0.00 | 400.0 | 16.67 |
| spot-pool | 14 | 400 | 0.55 | 0.10 | 198.0 | 19.64 |
| hybrid-estimated | 19 | 400 | 0.92 | 0.03 | 356.96 | 14.79 |

一个很常见的工程化伪逻辑是：

1. 先估稳定底座流量，配稳定资源
2. 再估可中断负载，挂到 spot 或更便宜池
3. 最后用统一公式比较 `C_1M`，不是比较裸价

这类函数可以放进内部容量规划脚本、压测报告或自动选型面板里。只要参数来自真实观测而不是拍脑袋，结论通常会比“看单价下单”可靠得多。

## 工程权衡与常见坑

推理成本优化不是会算一个公式就结束。真正难的是，线上系统里的 `a` 和 `ρ` 不是常数，而是会随着流量形态、调度策略、缓存状态和扩缩容动作变化。

首先，不要只看平均吞吐。平均值会掩盖问题，尤其在在线服务里，用户感知更接近尾延迟。`P95 TTFT` 的意思是 95% 请求在某个时间内拿到首 token，它比平均值更能反映卡顿风险。如果平均吞吐很好，但 P95 首 token 很差，业务照样会认为系统不可用。

其次，混合部署的关键不在“混”，而在“分层”。稳定底座负责主流在线流量，spot 负责峰值和离线任务。如果没分层，贵资源可能跑了低价值批任务，便宜资源反而顶着关键链路。

下面是常见坑：

| 常见坑 | 具体表现 | 后果 | 修正方式 |
|---|---|---|---|
| 只看单价 | 只比较 `$ / h` | 低价高损耗，结论失真 | 统一换算成 `$/1M tokens` |
| 只看平均吞吐 | 忽略 `P95 TTFT` 和排队 | 用户侧仍超时 | 同时纳入延迟与队列指标 |
| 忽略中断恢复成本 | 不统计重载和重试 | `ρ` 被低估 | 把冷启动、回放、重试计入损失 |
| 预留买多了 | 按峰值买长期资源 | 闲置严重 | 只买稳定底座容量 |
| 混合部署没分层 | 所有流量共享同池 | 调度混乱，收益不稳定 | 在线/离线、底座/峰值拆开 |
| 只看模型层不看服务层 | 只讨论模型速度 | 忽略扩缩容、KV cache、网关排队 | 端到端测量真实交付成本 |

一个错误方案可以这样概括：所有流量都走低价 spot。表面看单价低，实际上因为中断频繁，用户请求多次重试，TTFT 飙升，最终既伤体验又伤成本。

对应的正确方案通常是：

- 稳定底座池：承接主流在线请求，目标是高 `a`
- 弹性池：吸收活动峰值，目标是便宜扩容
- 离线池：承接批任务，允许中断，目标是最低 `C_1M`

还要注意一件事：有时吞吐 `r` 本身也会随资源策略变化。比如批处理深度不同、cache 命中率不同、上下文长度分布不同，都会让理论吞吐和线上吞吐偏离。所以压测时最好分场景，而不是只测一个“平均请求”。

## 替代方案与适用边界

不存在适合所有业务的唯一最优策略。资源策略必须服从业务目标，而业务目标通常由三件事决定：负载稳定性、延迟要求、可中断容忍度。

先看几类常见方案：

| 方案 | 适用条件 | 优点 | 风险 | 适用边界 |
|---|---|---|---|---|
| `spot` | 可中断、可重试、批量任务明显 | 单价低，适合拉低账面成本 | 中断、重载、抖动大 | 不适合核心在线低延迟链路 |
| `预留实例` | 稳定长期负载、SLO 严格 | `a` 高、容量稳、易做基线规划 | 长期承诺可能带来闲置 | 适合底座流量，不适合大幅波动峰值 |
| `混合部署` | 峰谷明显、负载分层清楚 | 同时兼顾稳定性与成本 | 调度与监控更复杂 | 需要有明确流量分层能力 |
| 专用推理硬件 | 模型和框架匹配、规模较大 | 吞吐高，单位推理成本可能更低 | 迁移成本、生态约束 | 适合长期大规模稳定业务 |

可以用三个真实场景来理解。

**在线问答服务**  
特点是实时、低延迟、不可频繁失败。推荐策略是“预留实例做底座，必要时少量弹性补峰”。原因不是保守，而是用户请求对 `a` 和 TTFT 很敏感。

**夜间离线摘要**  
特点是批量、可重跑、对秒级延迟不敏感。推荐策略是优先使用 spot 或更便宜的弹性资源，因为中断的业务代价低，`ρ` 可以通过任务切片和断点续跑控制。

**活动流量或热点突发**  
特点是峰值不稳定，平时又不值得长期买满。推荐策略是混合部署。稳定资源吃住常态流量，spot 或其他弹性池承接突发流量。如果再配合排队控制和优先级调度，通常比纯稳定池更省钱。

所以策略选择不是问“哪个最便宜”，而是问“哪种资源最适合这段负载”。当业务高度稳定时，预留实例可能更便宜；当业务高度可中断时，spot 可能更便宜；当峰谷波动明显时，混合部署通常最合理。
