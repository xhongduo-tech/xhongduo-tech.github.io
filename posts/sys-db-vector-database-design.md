## 核心结论

向量数据库的设计核心，不是“把向量存进去”，而是“在可接受延迟内找到足够接近的邻居”。这里的“邻居”指距离查询向量最近的一批向量，也就是语义上最相似的对象。真正决定系统性能的部分，通常是近似最近邻检索，英文叫 Approximate Nearest Neighbor，简称 ANN，意思是“允许少量误差来换速度”。

如果直接暴力扫描，每次查询都要对全部 $N$ 个向量做一次距离计算，复杂度近似为 $O(Nd)$，其中 $d$ 是向量维度。数据量到百万、千万、亿级后，这种方法很快失去实时性。向量数据库因此会引入索引，把“查所有人”改成“先快速定位候选，再精查候选”。

工业界最常见的三类索引是 HNSW、IVF、PQ：

| 索引 | 核心思路 | 优点 | 代价 | 常见适用场景 |
| --- | --- | --- | --- | --- |
| HNSW | 用多层图结构跳跃搜索 | 高召回、低延迟 | 内存占用高、构建慢 | 百万到千万级、内存较充足 |
| IVF | 先聚类分桶，再只搜少数桶 | 查询范围可控、实现成熟 | 召回依赖聚类质量 | 中大规模数据集 |
| PQ | 把向量压缩成短码 | 大幅省内存、省磁盘 | 有量化误差 | 亿级以上、成本敏感场景 |

实际系统里，常见组合不是三选一，而是混搭。比如先用 IVF 缩小搜索范围，再用 PQ 压缩候选向量；或者在较小数据规模下直接用 HNSW 追求更高召回率。

结论可以压缩成一句话：向量数据库的设计，本质是在召回率、延迟、内存、构建成本、更新成本之间选一个工程平衡点。

## 问题定义与边界

向量数据库解决的问题，可以先写成一个标准形式：给定查询向量 $q \in \mathbb{R}^d$，在数据库中的向量集合 $\{x_1, x_2, ..., x_N\}$ 里，找出距离 $q$ 最近的前 $k$ 个向量。

这里“距离”不是日常说的物理距离，而是“相似度的计算规则”。常见有三种：

| 距离度量 | 白话解释 | 常见用途 |
| --- | --- | --- |
| L2 距离 | 看两个点在空间里有多远 | 图像、度量学习 |
| 余弦相似度 | 只看方向像不像，不太看长度 | 文本 embedding |
| 内积 | 同时受方向和长度影响 | 推荐系统、检索打分 |

如果向量都做过归一化，即长度被缩放到 1，那么余弦相似度和内积在排序上通常等价。这是很多文本检索系统喜欢先归一化 embedding 的原因之一。

评估 ANN 检索时，最重要的指标之一是 Recall@k。它表示“近似检索返回的前 k 个结果里，找回了多少个真实最近邻”。公式是：

$$
Recall@k = \frac{|Topk_{ANN} \cap Topk_{exact}|}{k}
$$

其中 $Topk_{exact}$ 是暴力精确搜索得到的标准答案，$Topk_{ANN}$ 是 ANN 索引返回的结果。Recall@10 = 0.9，意思是精确答案前 10 个里，ANN 平均能找回 9 个。

问题边界也要讲清楚。向量数据库不是所有场景都需要：

1. 如果数据量只有几万，暴力扫描可能已经足够快。
2. 如果业务要求“绝对精确”，ANN 天生不满足，因为它允许漏掉少量真邻居。
3. 如果查询量很低，但离线批处理时间充足，也不一定需要复杂索引。
4. 如果数据更新极其频繁，某些索引的重建成本会成为瓶颈。

一个玩具例子可以帮助理解边界。

假设有 5 本书，每本书都被编码成二维向量：

- A: $(1, 1)$
- B: $(1, 2)$
- C: $(2, 2)$
- D: $(8, 8)$
- E: $(9, 8)$

查询向量是 $q=(1.5, 1.5)$。肉眼就能看出，最近的应该是 A、B、C，D 和 E 明显太远。只有 5 个点时，直接算完全部距离最简单，不需要任何索引。向量数据库真正开始体现价值，是在点的数量从 5 变成 500 万以后。

## 核心机制与推导

### 1. 为什么暴力检索会失效

暴力检索要对每个向量都做一次距离计算。以 L2 距离为例：

$$
\|q - x\|_2^2 = \sum_{i=1}^{d}(q_i - x_i)^2
$$

如果 $N=10^8$，$d=768$，一次查询大约要做 $10^8 \times 768$ 量级的基础运算。即使单次运算很快，总成本仍然过高。于是索引的目标就变成：尽量少看无关向量。

### 2. HNSW：用图结构“跳着找”

HNSW 全称 Hierarchical Navigable Small World。白话解释是：它把所有向量连成一张“多层导航图”，查询时先在高层快速跳大步，再在低层细化。

可以把它想成城市导航：

- 顶层像高速公路，只保留少量远距离连接，负责快速跨城。
- 底层像街道网络，连接更密，负责最后一公里。

查询流程大致是：

1. 从顶层入口点开始。
2. 每次走向“更接近查询向量”的邻居。
3. 到达当前层局部最优后，下沉到下一层继续找。
4. 最后一层保留一批候选，再输出 top-k。

HNSW 的关键参数通常有两个：

| 参数 | 含义 | 调大后的效果 |
| --- | --- | --- |
| `M` | 每个节点保留的邻居数 | 召回更高，但索引更大 |
| `ef_search` | 查询时保留的候选池大小 | 召回更高，但查询更慢 |

它适合高召回、低延迟场景，但代价是内存重。因为图结构不仅要存原始向量，还要存大量边。

### 3. IVF：先分桶，再局部搜索

IVF 全称 Inverted File。白话解释是：先把向量按位置分成很多桶，查询时只搜最相关的几个桶。

典型流程：

1. 用聚类算法把全量向量分成 `lists` 个簇。
2. 每个向量只挂到自己所属的簇下面。
3. 查询时，先找离查询向量最近的若干个簇。
4. 只在这些簇内部做距离比较。

这相当于“先找街区，再进楼找人”。如果总共有 1000 个簇，而查询只扫描 10 个簇，那么搜索范围就从全库压缩到了约 1%。

IVF 的主要参数是：

- `lists`：总簇数。越大，桶越细，但训练和管理成本更高。
- `nprobe` 或 `probes`：查询时扫描多少个簇。越大，召回更高，但延迟也更高。

IVF 的本质是用聚类把“全局搜索”变成“局部搜索”。它比 HNSW 更容易控制内存，但召回通常更依赖数据分布。

### 4. PQ：用压缩码替代原始向量

PQ 全称 Product Quantization。白话解释是：把一个长向量切成几段，每段都用一个短编号表示，这样就不用完整存原始浮点数。

假设一个 128 维向量被切成 $M=4$ 段，每段 32 维。每一段都单独训练一个码本，码本里有若干“代表向量”。原始子向量不再直接存储，而是只存“它最接近第几个代表向量”的编号。

于是原始向量：

$$
x = [x^{(1)}, x^{(2)}, ..., x^{(M)}]
$$

会被近似为：

$$
\hat{x} = [c_{1,k_1}, c_{2,k_2}, ..., c_{M,k_M}]
$$

其中 $c_{m,k_m}$ 是第 $m$ 个子空间里的第 $k_m$ 个码字。

查询时通常使用 ADC，Asymmetric Distance Computation，意思是“查询向量不压缩，数据库向量压缩后再算距离”。公式是：

$$
\hat d(q,x)=\sum_{m=1}^{M}\|q^{(m)}-c_{m,k_m}\|^2
$$

它的重要意义在于：查询时不必重建完整向量，只需要对每个子空间查表并求和。这样距离计算从“浮点向量计算”变成“查表加法”，速度和存储都会明显改善。

PQ 的代价也很直接：它引入量化误差。压缩越狠，误差通常越大，召回率越容易下降。

### 5. 三者组合的系统思路

真实系统常见的是 IVFPQ：

1. IVF 先把全库缩小到少数几个桶。
2. PQ 再把桶内向量压缩存储。
3. 先用压缩距离快速筛出候选。
4. 必要时再用原始向量做 rerank，也就是“对少量候选重新精排”。

这是一种典型的两阶段检索架构：第一阶段求快，第二阶段补精度。

真实工程例子可以用推荐系统理解。比如音乐推荐里，库中有上亿首歌的 embedding。查询向量代表“当前用户偏好”。系统不可能把全部歌曲逐一比对，而会先用 ANN 索引找到几百个候选，再叠加版权、语言、地区、年龄分级等过滤条件，最后才把十几个结果返回给用户。

## 代码实现

下面先给一个可运行的 Python 玩具实现。它不实现 HNSW 或 IVF，而是展示“精确检索”和“聚类缩小搜索范围”的差别，帮助理解 ANN 的基本思想。

```python
import math

def l2(a, b):
    return sum((x - y) ** 2 for x, y in zip(a, b))

def exact_topk(query, vectors, k=2):
    scored = [(name, l2(query, vec)) for name, vec in vectors.items()]
    scored.sort(key=lambda x: x[1])
    return [name for name, _ in scored[:k]]

def cluster_topk(query, vectors, centroids, assignments, probe=1, k=2):
    centroid_scored = [(cid, l2(query, cvec)) for cid, cvec in centroids.items()]
    centroid_scored.sort(key=lambda x: x[1])
    selected = {cid for cid, _ in centroid_scored[:probe]}

    candidates = []
    for name, vec in vectors.items():
        if assignments[name] in selected:
            candidates.append((name, l2(query, vec)))

    candidates.sort(key=lambda x: x[1])
    return [name for name, _ in candidates[:k]]

vectors = {
    "A": (1.0, 1.0),
    "B": (1.0, 2.0),
    "C": (2.0, 2.0),
    "D": (8.0, 8.0),
    "E": (9.0, 8.0),
}

centroids = {
    "c1": (1.3, 1.7),
    "c2": (8.5, 8.0),
}

assignments = {
    "A": "c1",
    "B": "c1",
    "C": "c1",
    "D": "c2",
    "E": "c2",
}

query = (1.5, 1.5)

exact = exact_topk(query, vectors, k=2)
approx = cluster_topk(query, vectors, centroids, assignments, probe=1, k=2)

assert exact == ["A", "B"] or exact == ["B", "A"]
assert set(approx) == {"A", "B"}
print("exact:", exact)
print("approx:", approx)
```

这个例子里，`cluster_topk` 先比较查询向量和两个簇中心的距离，再只在最接近的那个簇里搜。它就是 IVF 的极简版思路。

如果把它映射到数据库产品，接口通常不是自己手写，而是通过向量索引完成。以 SQL 风格伪代码表示，常见写法类似：

```sql
CREATE INDEX docs_emb_hnsw ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 32, ef_construction = 128);

SET hnsw.ef_search = 200;

SELECT id, title
FROM documents
ORDER BY embedding <=> :query_vector
LIMIT 10;
```

如果数据规模更大，可能切到 IVF 或 IVFPQ：

```sql
CREATE INDEX docs_emb_ivf ON documents
USING ivf (embedding vector_l2_ops)
WITH (
  lists = 1000,
  quantization = 'pq',
  pq_m = 64,
  pq_k = 256
);

SET ivf.probes = 20;

SELECT id, title
FROM documents
ORDER BY embedding <-> :query_vector
LIMIT 10;
```

参数和效果的对应关系可以先记成下面这张表：

| 参数 | 作用 | 调大后的典型影响 |
| --- | --- | --- |
| `ef_construction` | HNSW 建索引时的候选池 | 构建更慢，图质量更好 |
| `ef_search` | HNSW 查询候选池 | 召回上升，查询变慢 |
| `lists` | IVF 聚类桶数量 | 桶更细，训练和维护更复杂 |
| `probes` | IVF 查询扫描桶数 | 召回上升，延迟上升 |
| `pq_m` | PQ 子空间数量 | 压缩更细，但编码更复杂 |
| `pq_k` | 每个子空间的码字数 | 表达能力更强，但训练成本更高 |

生产实现一般还会加两件事：

1. 过滤条件下推，比如只搜某租户、某语言、某时间范围。
2. 候选重排，也就是 ANN 先召回 100 到 1000 个候选，再用原始向量或更重的模型重新排序。

## 工程权衡与常见坑

向量数据库落地后，最常见的问题不是“能不能查”，而是“查得是否稳定”。这里的稳定，指的是在数据规模增长、分布变化、模型更新之后，延迟和召回不要突然恶化。

### 1. 只看延迟，不看召回

这是最典型的坑。把 `ef_search` 或 `probes` 调低，延迟确实会下降，但 Recall@k 也可能明显下降。很多团队上线后只监控 p95 延迟，却没有对比 exact baseline，结果是系统“很快”，但找错了内容。

建议至少保留一份小规模精确索引或离线精确评测集，定期计算 Recall@10、Recall@100。

### 2. HNSW 内存膨胀

HNSW 的图边很多，节点越多，内存越容易膨胀。数据到亿级时，如果还坚持用纯 HNSW，内存成本会非常高，构建时间也会拉长。它不是不能用，而是要确认预算是否允许。

### 3. IVF 聚类失真

如果训练聚类中心时样本不够，或者数据分布已经漂移，查询可能经常被路由到错误桶，导致召回下降。这种问题在 embedding 模型升级后尤其常见。模型一换，原有聚类边界可能不再合适。

### 4. PQ 压缩过度

PQ 省内存最明显，但它的风险也最直白：压缩得越狠，误差越大。尤其是在高维、细粒度语义区分场景里，过度压缩会让很多“本来很近”的向量被排错顺序。

### 5. 忘记 rerank

ANN 第一阶段的职责是“快速缩小候选范围”，不是直接输出最终最准结果。对于 RAG、推荐、广告等质量敏感业务，常见做法是“宽召回 + 精排”。例如先取 top-100 候选，再用原始向量精确算距离，或者用 cross-encoder 重排前几十个文档。

下面是一张常见问题巡检表：

| 问题 | 现象 | 监控指标 | 常见处理方式 |
| --- | --- | --- | --- |
| 参数过小 | 延迟低但结果质量差 | Recall@10, NDCG | 提高 `ef_search` 或 `probes` |
| HNSW 过大 | 内存占用异常 | 内存峰值、构建时长 | 降低 `M`，或切 IVF/PQ |
| 聚类过时 | 新数据召回下降 | 分桶命中率、Recall 走势 | 重训 IVF 聚类中心 |
| PQ 失真 | top-k 排序抖动 | exact 对比误差 | 增加码字、减小压缩率 |
| 无基线 | 无法判断质量退化 | exact recall baseline | 建离线评测集 |
| 过滤后空结果 | 候选被过滤掉太多 | 过滤后候选数 | 扩大初始召回集 |

一个真实工程例子是 RAG 文档检索。假设企业知识库有 5000 万段文本，先用 IVFPQ 找到 200 个候选片段，再对这 200 个片段做精确余弦计算，最后把前 20 个送入重排模型。这样做的原因不是“算法更复杂”，而是粗排阶段负责吞吐量，精排阶段负责质量，职责分开后更容易调优。

## 替代方案与适用边界

向量数据库没有单一最优解，只有和规模、预算、更新模式匹配的方案。

先看一个粗略选型表：

| 方案 | 数据规模 | 内存要求 | 召回上限 | 更新友好性 | 适用边界 |
| --- | --- | --- | --- | --- | --- |
| 精确扫描 | 小规模 | 低 | 最高 | 高 | 数据少、强精确要求 |
| HNSW | 百万到中千万 | 高 | 很高 | 中等 | 低延迟、高召回 |
| IVF | 千万到亿级 | 中 | 中高 | 中高 | 规模扩大后的通用方案 |
| IVFPQ | 亿级到十亿级 | 低到中 | 中高 | 中等 | 成本敏感、存储受限 |
| 磁盘型 ANN | 十亿级以上 | 更低内存 | 中高 | 视实现而定 | 内存装不下全量索引 |
| 混合检索 | 各规模 | 取决于组合 | 更稳定 | 中等 | 关键词和语义都重要 |

### 1. 什么时候继续用暴力检索

如果只有几十万数据，QPS 不高，向量维度也不大，暴力检索加 SIMD 或 GPU 可能已经够用。此时引入复杂索引，收益未必大于维护成本。

### 2. 什么时候优先 HNSW

当业务追求高召回和低延迟，而且数据规模还没大到内存爆炸时，HNSW 往往是最直接的选择。比如商品相似推荐、站内语义搜索、个人知识库检索。

### 3. 什么时候切 IVF 或 IVFPQ

当数据到亿级，或者单机内存预算明确受限时，通常要考虑 IVF 或 IVFPQ。它们更像“以可控误差换规模能力”的方案。

### 4. 混合检索不是备选，而是常态

很多业务并不是纯向量问题。比如用户搜“iPhone 15 Type-C 充电慢”，关键词“iPhone 15”“Type-C”很重要，语义相似也重要。这时常见做法是 BM25 先召回一批关键词命中文档，向量索引再召回一批语义相关文档，最后合并重排。对知识库和电商搜索，这往往比纯向量更稳。

### 5. 适用边界要跟着业务目标走

如果系统目标是“对话问答尽量别答错”，那召回率通常比极限低延迟更重要。如果系统目标是“首页推荐 20ms 内返回”，那吞吐和尾延迟优先级会更高。索引设计要服从业务目标，而不是反过来。
