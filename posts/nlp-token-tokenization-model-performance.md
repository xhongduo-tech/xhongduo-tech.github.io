## 核心结论

分词本质上是在决定 token 的粒度。token 可以理解为“模型一次读取的最小文本块”。词表越大，单个 token 往往越长，序列会被压缩；词表越小，文本会被拆得更细，序列会变长。

这件事直接影响两类成本：

1. 嵌入层参数量，公式是 $E = |V| \times d$  
词表大小 $|V|$ 乘嵌入维度 $d$，就是嵌入表的参数量。嵌入层可以理解为“给每个 token 存一份向量字典”。

2. 自注意力计算成本，近似看作 $C_{attn} \propto n^2 \times d$  
其中 $n$ 是序列长度，$d$ 是隐藏维度。自注意力可以理解为“每个 token 都要和其他 token 两两比较一次”。

因此，词表大小不是越大越好，也不是越小越好。它是在“压缩序列长度”和“增大嵌入表”之间找平衡。

对多数英文或以英文为主的通用语料，32K 到 64K 的 subword 词表通常是比较稳妥的工程区间。subword 指“介于字符和完整单词之间的子词单元”，例如把 `playing` 切成 `play` 和 `ing`。把词表从 50K 扩到 100K，平均压缩率常常只提升约 10% 到 20%，但嵌入层参数会直接翻倍。字符级方案虽然词表极小，但序列会显著变长，通常需要更深的网络才能补回语义建模能力。

一个玩具例子：

- 小词表把 `transformer` 切成 `trans` + `former`
- 大词表直接把 `transformer` 当一个 token

后者序列更短，但必须给 `transformer` 单独存一个向量；如果这种长词很多，词表就会迅速膨胀。

## 问题定义与边界

这里讨论的“性能”不是单一指标，而是至少包含三部分：

| 维度 | 含义 | 主要受什么影响 |
| --- | --- | --- |
| 推理速度 | 单次生成或编码有多快 | 序列长度 $n$，尤其是注意力部分 |
| 内存占用 | 模型加载和运行需要多少显存/内存 | 词表大小、嵌入维度、KV cache |
| 训练稳定性 | loss 是否平稳下降，稀有 token 是否学得动 | 数据规模、token 频率分布、词表是否过大 |

词表大小只是一部分变量，不能脱离以下边界单独谈：

| 变量 | 为什么重要 |
| --- | --- |
| 嵌入维度 $d$ | 同样是 50K 词表，256 维和 2048 维的参数量差很多 |
| 上下文长度 | 同样的压缩率变化，对 512 token 和 32K token 的收益不同 |
| 训练语料规模 | 大词表会引入更多低频 token，没有足够样本就学不稳 |
| 硬件类型 | 手机、边缘设备、云 GPU 对内存和吞吐的敏感点不同 |
| 语言类型 | 英文、中文、多语言的词形结构差异很大 |

下面给一个粗略对比，帮助建立量级感。假设参数使用 FP32，每个参数 4 字节：

| 词表与维度 | 嵌入参数量 | 嵌入内存约值 | 序列压缩效果 | 常见感受 |
| --- | --- | --- | --- | --- |
| 16K × 256 | 4.1M | 16MB | 一般 | 表小，省内存，但序列偏长 |
| 32K × 512 | 16.4M | 64MB | 较好 | 常见平衡点 |
| 50K × 768 | 38.4M | 150MB | 更好 | 通用模型常见区间 |
| 100K × 768 | 76.8M | 300MB | 略优于 50K | 序列更短，但表明显更重 |

真实工程里，经常会出现一种现象：32K 比 16K 更快，但 100K 不一定比 50K 更划算。原因不是 100K 没有压缩收益，而是收益开始递减。

## 核心机制与推导

先把文本长度写成一个简单关系。假设一段文本有 $w$ 个词，平均每个词被切成 $r$ 个 token，那么序列长度近似是：

$$
n \approx w \times r
$$

这里的 $r$ 就是 tokens/word。它可以理解为“每个词平均会被切成几块”。

代入注意力成本：

$$
C_{attn} \propto n^2 \times d = (w \times r)^2 \times d
$$

这说明注意力成本对分词压缩率是平方敏感的。只要 $r$ 降低，$n$ 降低，注意力成本会更明显下降。

但嵌入层不是这样。它对词表大小是线性增长：

$$
E = |V| \times d
$$

所以有两个方向相反的力：

- 增大词表：$r$ 下降，$n$ 下降，注意力更便宜
- 增大词表：$|V|$ 上升，嵌入层更大，低频 token 更多

这就是分词和模型性能关系的核心。

一个常见经验值是：

| 词表大小 | 英文常见 tokens/word 区间 | 含义 |
| --- | --- | --- |
| 32K | 1.3 到 1.5 | 压缩效果已经不错 |
| 50K | 1.2 到 1.4 | 比 32K 更短一些 |
| 100K | 1.1 到 1.3 | 继续变短，但收益有限 |

玩具例子：假设有 1000 个英文词。

- 用 32K 词表，平均 $r = 1.4$，则 $n = 1400$
- 用 100K 词表，平均 $r = 1.2$，则 $n = 1200$

注意力成本近似比值是：

$$
\frac{1200^2}{1400^2} \approx 0.735
$$

也就是注意力部分大约下降 26.5%。这不是小数，但也不是数量级变化。与此同时，若嵌入维度不变，从 50K 扩到 100K，嵌入参数直接翻倍。

再看训练数据约束。大词表会包含更多稀有 token。稀有 token 可以理解为“训练中很少见到的词块”。如果某些 token 在 10B token 的训练语料里都很少出现，它们的梯度更新次数就很少，学习会不稳定。结果往往是：

- 词表大了，但很多 token 没学好
- loss 降不下来或尾部震荡
- 罕见 token 拖慢整体收敛

这也是为什么“词表大小要与语料量匹配”。经验上，10B token 规模的数据，32K 到 64K 往往更稳；多语言场景因为不同语言都需要覆盖，词表常要扩大到单语的 2 到 4 倍，否则小语种会被过度拆分。

真实工程例子：做一个云端文本生成服务，如果 API 成本主要受输入 token 数影响，那么较大的词表可能通过压缩序列降低账单和延迟；但如果是边缘设备部署，启动时必须把嵌入表加载进内存，那么过大的词表反而首先撞上内存上限。

## 代码实现

下面用一个最小 Python 脚本估算三件事：

- 嵌入层参数量与内存
- 不同压缩率下的序列长度
- 自注意力相对成本

```python
from dataclasses import dataclass

@dataclass
class TokenizerPlan:
    vocab_size: int
    dim: int
    tokens_per_word: float

def embedding_params(vocab_size: int, dim: int) -> int:
    return vocab_size * dim

def embedding_bytes(vocab_size: int, dim: int, bytes_per_param: int = 4) -> int:
    return embedding_params(vocab_size, dim) * bytes_per_param

def seq_len(words: int, tokens_per_word: float) -> int:
    return int(round(words * tokens_per_word))

def attention_cost(seq_len_value: int, dim: int) -> int:
    return seq_len_value * seq_len_value * dim

def mb(x: int) -> float:
    return x / (1024 * 1024)

plan_32k = TokenizerPlan(vocab_size=32_000, dim=768, tokens_per_word=1.4)
plan_100k = TokenizerPlan(vocab_size=100_000, dim=768, tokens_per_word=1.2)

words = 1000

seq_32k = seq_len(words, plan_32k.tokens_per_word)
seq_100k = seq_len(words, plan_100k.tokens_per_word)

emb_32k_mb = mb(embedding_bytes(plan_32k.vocab_size, plan_32k.dim))
emb_100k_mb = mb(embedding_bytes(plan_100k.vocab_size, plan_100k.dim))

attn_32k = attention_cost(seq_32k, plan_32k.dim)
attn_100k = attention_cost(seq_100k, plan_100k.dim)

print("32K seq_len:", seq_32k)
print("100K seq_len:", seq_100k)
print("32K embedding MB:", round(emb_32k_mb, 1))
print("100K embedding MB:", round(emb_100k_mb, 1))
print("attn cost ratio 100K/32K:", round(attn_100k / attn_32k, 3))

assert seq_100k < seq_32k
assert emb_100k_mb > emb_32k_mb
assert attn_100k < attn_32k
```

如果按这组参数估算：

- 32K 词表时，序列约 1400 token
- 100K 词表时，序列约 1200 token
- 注意力成本下降
- 但嵌入内存显著上升

还可以把常见方案放在一起看：

| 方案 | 词数 | tokens/word | 序列长度 $n$ | 嵌入内存约值 | 注意力相对成本 |
| --- | --- | --- | --- | --- | --- |
| 16K, 256维 | 1000 | 1.7 | 1700 | 16MB | 高 |
| 32K, 768维 | 1000 | 1.4 | 1400 | 94MB | 中 |
| 50K, 768维 | 1000 | 1.3 | 1300 | 147MB | 中低 |
| 100K, 768维 | 1000 | 1.2 | 1200 | 293MB | 低 |

这个脚本虽然简化了很多细节，但足够用于做初步选型。真正工程里还要加入 batch size、KV cache、量化精度和 FFN 开销，但分词带来的方向性影响已经能看清。

## 工程权衡与常见坑

最常见的误区是只盯着一种成本。

第一类坑是 vocabulary bloat，也就是“词表膨胀”，指词表里混入了很多低频、收益很小的 token。表面上看分词更细致，实际上这些 token 根本没有足够数据学习。

第二类坑是误以为字符级一定更省。字符级模型的确词表很小，但序列会大幅变长。模型必须自己从字符组合里重新学出词和短语，这通常要求更深或更宽的网络。

第三类坑是忽略部署场景。同样一个词表，在训练集群和手机端的优先级完全不同。

| 常见坑 | 现象 | 原因 | 应对策略 |
| --- | --- | --- | --- |
| 词表膨胀 | 参数增多但效果不升 | 低频 token 太多 | 控制词表上限，过滤极低频片段 |
| 梯度稀疏 | 某些 token 学不动 | 训练语料不足 | 让词表规模与数据量匹配 |
| 字符级过长序列 | 训练和推理都慢 | $n$ 变长，注意力变贵 | 用 subword，或改更适合长序列的结构 |
| 多语言失衡 | 小语种被拆得很碎 | 词表被高频语言占满 | 扩大词表或做语言平衡采样 |
| 只看 token 压缩率 | 账面更短但总成本更高 | 忽略嵌入内存和加载成本 | 同时评估 $E$ 与 $C_{attn}$ |

真实工程例子：边缘设备上的离线分类模型，常会选 16K 左右词表加 256 维嵌入，把嵌入表控制在约 16MB。这样启动稳，内存压力可控。但如果迁移到云端摘要服务，同样的 16K 词表会让输入序列偏长，导致吞吐下降，最终更可能改到 50K 甚至更大。

一个实用原则是：如果你已经发现序列长度是主要瓶颈，再考虑增大词表；如果模型首先卡在内存、冷启动或训练样本不足，就不该靠继续扩大词表解决问题。

## 替代方案与适用边界

不是所有任务都该用常规 subword。不同分词方案的重点不同。

| 方案 | 适用场景 | 主要优点 | 主要限制 |
| --- | --- | --- | --- |
| Subword | 通用 NLP、大多数 LLM | 平衡序列长度和词表大小 | 仍需选择合适词表上限 |
| Character | 极低资源语言、强鲁棒拼写任务 | 词表极小，几乎无 OOV | 序列长，常需更深网络 |
| Byte-level BPE | 多语言、混合编码、脏文本 | 基础覆盖稳定，不怕生僻字符 | token 数可能偏多 |
| Charformer 类重标度方法 | 想保留细粒度输入又控制序列长度 | 让字符输入先局部聚合，再送主干网络 | 结构更复杂，工程实现成本更高 |

这里的 OOV 是 out-of-vocabulary，意思是“词表外词”，也就是训练时没收录进去的片段。

对初级工程师最实用的选型建议是：

- 常规英文或中英混合通用任务，优先从 32K 到 64K 的 subword 起步
- 多语言任务，词表通常要比单语大 2 到 4 倍，否则小语种切分会过碎
- 数据量不大时，不要盲目上超大词表，因为很多 token 学不到稳定表示
- 如果序列极长，可以先考虑更高效的注意力结构、缩短上下文、联合调宽调深，而不是只扩大词表
- 如果必须用字符级输入，最好同时引入更强的局部组合模块，否则很难追上合理的 subword 基线

真正有效的优化通常是联合调整，而不是只调一个旋钮。也就是同时考虑：

- 词表大小
- 模型宽度与深度
- 上下文长度
- 训练语料规模
- 部署硬件约束

只扩大词表，往往只能解决“序列偏长”这一类问题，解决不了“数据太少”或“模型主干不够强”这类问题。
