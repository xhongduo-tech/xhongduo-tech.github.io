## 核心结论

多模态融合的新范式，当前可以压缩成两条主线：组合式多模态与原生多模态。

组合式多模态，白话说，就是把“看图”“听音”“转文字”“最终推理”拆给不同专用模型做，再把结果串起来。它像一条工程流水线：视觉编码器负责图像，ASR 负责语音转文字，TTS 负责语音生成，最后由大语言模型或多模态大模型做综合判断。它的优势是成熟、可替换、易部署，代价是前面步骤一旦错了，后面通常救不回来。

原生多模态，白话说，就是不再先拆成多个单独模型，而是把图像、文本、音频都变成统一的 token 序列，让同一个 Transformer 从第一层开始一起处理。它的目标不是“先转成文本再思考”，而是“在同一套表示里直接联合理解和生成”。这条路线在细粒度跨模态对齐、实时交互、统一推理上更有潜力，但训练成本高，数据要求也高。

一个直接判断标准是：如果你的系统目标是“先把活干完，成本可控，出问题能快速替换模块”，优先选组合式；如果你的目标是“让模型真的跨模态思考，而不是把多模态问题偷偷改写成文本问题”，原生路线更值得投入。

下表先给出总对比：

| 维度 | 组合式多模态 | 原生多模态 |
| --- | --- | --- |
| 训练方式 | 各模块分别预训练，再串联 | 在统一架构里联合训练 |
| 推理方式 | 分阶段处理，按需调用大模型 | 端到端统一推理 |
| 工程成熟度 | 高 | 仍在快速演进 |
| 模块可替换性 | 强 | 弱 |
| 跨模态对齐能力 | 依赖接口质量 | 理论上更强 |
| 延迟控制 | 容易分层优化 | 需要更强系统优化 |
| 数据门槛 | 中等 | 高 |
| 算力门槛 | 可控 | 高 |

玩具例子可以这样理解。

假设你做一个“看图回答”系统。组合式做法是：先用视觉模型识别“图里有一只猫和一张桌子”，再把这句文本交给语言模型回答。原生做法是：图像 patch token 和问题文本 token 一起输入模型，模型直接在内部完成“看到猫在桌上”这个推理。两者都能输出答案，但前者更像“先翻译，再理解”，后者更像“直接理解”。

真实工程例子更能说明差异。内容审核场景里，组合式已经很成熟：先用轻量视觉模型和 ASR 粗筛，只把高风险样本送给大型 MLLM 复核。根据 EmergentMind 对 COEF-VQ 的整理，该类两阶段策略可以把大型模型调用比例压到 4.45%，平均延迟从 430ms 降到 50ms。另一边，AGI House 在 2025 年 12 月对 Gemini 3 的总结则把它归为“born-together”的原生路线：图、文、声在统一架构内共同处理，优势体现在细粒度 UI 理解、视频理解和更长链条的跨模态推理。

## 问题定义与边界

这篇文章讨论的问题不是“多模态能不能做”，而是更具体的问题：

如何在视觉、语言、音频联合任务里，同时解决三件事：

1. 跨模态对齐  
跨模态对齐，白话说，就是图像里看到的东西、语音里说的内容、文本里的词，最终能不能被模型当成同一件事理解。

2. 推理延迟  
推理延迟，白话说，就是用户发起请求后，系统要等多久才有结果。

3. 训练与部署成本  
训练成本是把模型训出来要花多少 GPU 和数据，部署成本是上线后每次调用要花多少钱。

边界也要先说清楚。这里讨论的是视觉、文本、音频三种主流模态，不展开机器人控制、触觉、3D 点云等更广义模态。因为对“零基础到初级工程师”来说，这三种模态已经覆盖大多数实际产品：图文问答、语音助手、视频理解、内容审核、会议摘要、客服机器人。

适用边界可以先用一张表看清：

| 判断维度 | 更偏组合式 | 更偏原生 |
| --- | --- | --- |
| 任务复杂度 | 中低，规则清晰 | 高，需要细粒度联合推理 |
| 部署资源 | 分布式、异构设备 | 集中式高算力 |
| 延迟容忍度 | 严格，要求稳定毫秒级到亚秒级 | 可接受较高延迟换更强能力 |
| 数据规模 | 中小数据也能工作 | 需要大规模多模态数据 |
| 系统维护方式 | 团队按模块分工 | 团队围绕统一模型迭代 |
| 输出形态 | 常是文本结论或审核标签 | 常包含理解、生成、交互统一能力 |

组合式的典型边界，是“成熟工程系统”。例如视频审核：先 ASR 转文本，再用视觉模型识别关键帧，再把摘要交给 LLM 做规则判断。这类系统的目标不是证明模型有多先进，而是保证吞吐、成本、回滚能力。

原生路线的典型边界，是“追求统一能力的前沿系统”。例如一个真正的多模态助手，需要一边听用户说话，一边看屏幕截图，一边理解界面元素关系，再给出下一步操作建议。这种任务里，单纯把图像和语音都翻译成文本，往往会丢掉布局、时序和细节。

所以，问题不是“哪条路线绝对更先进”，而是“你的约束条件在哪”。如果你缺的是预算和交付周期，组合式通常更现实；如果你缺的是跨模态统一建模能力，原生路线才是长期解法。

## 核心机制与推导

先看原生多模态的核心机制。

原生路线的第一步，是统一 token 空间。token，白话说，就是模型处理信息时使用的最小离散单元。文本里的词片可以是 token，图像切成 patch 后也可以编码成 token，音频也可以切成时间片或离散码本 token。这样模型看到的输入，不再是“这是一张图，那是一段声音”，而是“这是一串混合 token”。

统一后，所有 token 会进入同一个 Transformer。Transformer，白话说，就是靠注意力机制决定“当前位置该看谁”的序列模型。原生路线的关键不是后面加一层融合，而是从第一层开始，图、文、声就在同一注意力图里互相影响。

如果写成简化表达，输入可以是：

$$
X = [x_1^{text}, x_2^{text}, v_1^{img}, v_2^{img}, a_1^{audio}, a_2^{audio}, \dots]
$$

模型训练目标通常还是下一 token 预测：

$$
L = - \sum_{t=1}^{T}\log p(x_t \mid x_{<t})
$$

只是这里的 $x_t$ 不再只来自文本词表，而是来自跨模态联合词表。

原生路线为什么成本高？因为它不仅要扩大参数量，还要扩大多模态 token 数据量。EmergentMind 总结的经验缩放律可以写成：

$$
L(N,D)=E+\frac{A}{N^\alpha}+\frac{B}{D^\beta}
$$

其中：

- $N$ 是模型参数规模，白话说，就是模型“大脑容量”
- $D$ 是训练 token 规模，白话说，就是喂给模型的训练素材总量
- $E,A,B,\alpha,\beta$ 是由数据和架构决定的常数

这个式子表达的不是“模型越大越好”，而是“参数和数据都要投，且要平衡投”。如果只有大模型没有足够多的高质量多模态数据，损失不会按预期下降；如果只有海量数据但模型容量不够，也吃不下这些信息。

再看组合式多模态的核心机制。

组合式的关键不是“把所有模态都接上”，而是“分层处理，只把难样本交给贵模型”。它常用一个轻量前置模型做 gating。gating，白话说，就是决定“当前样本要不要升级处理”的门控规则。

最常见的门控信号是不确定度。若轻量模型输出类别分布 $p_x$，可以计算熵：

$$
H(p_x) = - \sum_i p_i \log p_i
$$

当

$$
H(p_x) > H_{th}
$$

时，说明模型“没把握”，才把样本送去更重的大模型阶段。

资源节省可以写成：

$$
R = 1 - \alpha
$$

其中 $\alpha$ 是被送到重模型的样本比例。如果只有 4.45% 的样本进入重模型，那么理论上的大模型调用削减比例就是 $1 - 0.0445 = 95.55\%$。

这背后的推导很直接：

- 单次大模型成本极高
- 大多数样本其实是“容易样本”
- 用轻量模型先把容易样本处理掉，贵资源只留给模糊、冲突、边界样本

这也是组合式能大幅省 GPU 的原因。

下表把两条机制并排看，会更清楚：

| 问题 | 组合式 | 原生 |
| --- | --- | --- |
| 信息流形式 | 模块间逐步传递 | 全模态统一流动 |
| 何时调用大模型 | 只在高不确定样本上调用 | 默认整次推理都在统一大模型内 |
| 对齐发生在哪里 | 接口层、对齐层、rerank 层 | 主干网络每一层 |
| 可解释性 | 更强，容易定位哪一段错 | 更弱，错误更难拆解 |
| 主要优化手段 | 门控、筛选、rerank、缓存 | MoE、统一 tokenizer、并行注意力、缓存 |

玩具例子：

你做“图片是否含违规广告语”判断。轻量模型对一张图输出 `[正常: 0.52, 风险: 0.48]`，熵很高，说明没把握，这时再把原图、OCR 文本、页面上下文一起送给 MLLM。另一张图输出 `[正常: 0.99, 风险: 0.01]`，熵很低，直接放行。这个过程就是典型的熵门控。

真实工程例子：

TikTok 类视频审核场景里，轻量视觉模型、Whisper 类语音识别、文本特征抽取先做低成本预筛，再把高不确定视频交给大型多模态模型做复核。与此相对，原生路线试图在统一模型里直接对视频帧、字幕、音频事件做联合理解，避免“先转文本再丢细节”的损失。

## 代码实现

先给一个可运行的玩具实现，模拟组合式门控。这个例子不依赖深度学习框架，但把关键思想写清楚了。

```python
import math

def entropy(probs):
    total = sum(probs)
    assert abs(total - 1.0) < 1e-9, "probabilities must sum to 1"
    return -sum(p * math.log(p, 2) for p in probs if p > 0)

def should_call_big_model(class_probs, conf_threshold=0.85, entropy_threshold=0.8):
    top_conf = max(class_probs)
    h = entropy(class_probs)
    # 置信度低或熵高，说明轻量模型没把握，需要升级到大模型
    return top_conf < conf_threshold or h > entropy_threshold

def pipeline_predict(sample):
    # 轻量模型输出，真实系统里这里可能来自视觉分类器或 ASR+分类头
    class_probs = sample["class_probs"]

    if should_call_big_model(class_probs):
        # 大模型阶段：这里用规则代替真实 MLLM
        return {
            "route": "big_model",
            "label": sample["big_model_label"]
        }
    else:
        label = "risk" if class_probs[1] > class_probs[0] else "normal"
        return {
            "route": "small_model",
            "label": label
        }

easy = {"class_probs": [0.98, 0.02], "big_model_label": "normal"}
hard = {"class_probs": [0.52, 0.48], "big_model_label": "risk"}

assert pipeline_predict(easy)["route"] == "small_model"
assert pipeline_predict(hard)["route"] == "big_model"
assert entropy([0.5, 0.5]) > entropy([0.99, 0.01])

print("ok")
```

这个代码说明三件事：

1. 轻量模型不是为了最终最强精度，而是为了先做便宜判断。
2. `entropy` 可以当“不确定度”指标。
3. 升级逻辑要写成明确阈值，才能根据业务目标调参。

如果把它画成流程，就是：

```text
输入(图/音/文)
  -> 轻量模型
  -> 计算置信度/熵
  -> 若高风险或高不确定:
         进入大模型复核
     否则:
         直接输出
```

真实工程例子里，这个流程会更复杂：

```text
视频
  -> 抽帧
  -> 轻量视觉分类器
  -> ASR 转录
  -> OCR 抽字
  -> 风险聚合打分
  -> rerank / 复核
  -> MLLM 终审
```

这里的 rerank，白话说，就是在多个候选结果里重新排序，避免前一阶段的粗糙判断直接决定最终结果。它常插在“轻量筛选”和“大模型终审”之间。

再看原生多模态的伪代码。注意这是结构示意，不是完整训练代码。

```python
class UnifiedTokenizer:
    def encode_text(self, text):
        return ["T:" + w for w in text.split()]

    def encode_image(self, image_patches):
        return [f"I:{p}" for p in image_patches]

    def encode_audio(self, audio_units):
        return [f"A:{u}" for u in audio_units]

def native_multimodal_forward(text, image_patches, audio_units, transformer):
    tokenizer = UnifiedTokenizer()

    text_tokens = tokenizer.encode_text(text)
    image_tokens = tokenizer.encode_image(image_patches)
    audio_tokens = tokenizer.encode_audio(audio_units)

    # 所有模态直接拼到同一序列里
    joint_tokens = text_tokens + image_tokens + audio_tokens

    # 同一个 Transformer 从第一层开始联合处理
    hidden = transformer(joint_tokens)

    # 统一解码头可以输出文本，也可以路由到图像/音频生成头
    return hidden
```

这个示意里最重要的不是“拼接”本身，而是“拼接之后仍由同一主干处理”。如果你只是把图像编码成向量，再在最后一层塞给 LLM，那通常仍更接近组合式或适配式系统，不是严格意义上的原生统一建模。

对于新手，一个容易混淆的点是：统一输入不等于原生多模态。判断标准应该是：

- 是否共享主干网络
- 是否从早期层开始联合建模
- 是否以统一目标训练
- 是否支持跨模态统一生成或推理

只满足“支持图文输入”，还不够。

## 工程权衡与常见坑

组合式路线最大的坑，是级联误差。级联误差，白话说，就是前面模块的错误被后面模块放大。

典型情况有三个：

| 坑点 | 组合式表现 | 原生表现 | 常见规避手段 |
| --- | --- | --- | --- |
| 错误传播 | 严重，前段错后段难救 | 相对少，但可能整体误判 | 增加 rerank、复核、回传机制 |
| 延迟 | 易拆分优化，但链路多 | 单次链路短，但模型重 | 缓存、批处理、分级推理 |
| 自适应部署 | 强，可按设备切换模块 | 弱，统一模型部署重 | 模块量化、边缘/云协同 |
| 训练资源 | 低到中 | 高 | MoE、混合精度、蒸馏 |

组合式的第一个大坑是漏召回。比如视频分类器没有识别出一个危险镜头，下游 MLLM 可能永远看不到这一帧，也就谈不上纠正。这个问题不是“LLM 再强一点就能解决”，而是“错误信息根本没传下去”。

第二个坑是接口失真。ASR 把一句话识别错一个否定词，文本语义可能完全相反。图像 OCR 少识别一个数字，后续风控规则也会跑偏。流水线越长，接口越多，失真点越多。

第三个坑是阈值调错。门控阈值太保守，会把大量样本送入大模型，成本直接失控；门控阈值太激进，会把真正困难样本挡在外面，召回率下降。

因此组合式系统里，rerank 和复核的插入点很关键：

```text
轻量筛选
  -> 候选结果
  -> rerank / 置信度校准
  -> 大模型复核
  -> 最终输出
```

原生路线的坑则完全不同。

第一是训练成本。原生模型要学的是“跨模态共同表示”，而不是“把模态都接上”。这要求更大规模的数据清洗、统一 token 设计、长上下文训练和更高通信成本。

第二是推理开销。一个统一模型虽然减少了模块切换，但如果上下文里同时塞入图像 patch、长文本、音频 token，序列长度会暴涨。注意力计算成本通常随序列长度增加而上升，这会直接带来延迟问题。

第三是调试困难。组合式系统可以直接定位“ASR 错了”“OCR 错了”“图像分类器错了”。原生模型则更像一个大黑盒，出了错常常只能从样本和日志反推，很难像流水线那样逐段修。

因此原生路线通常会配合三类工程手段：

1. MoE  
MoE，白话说，就是很多“专家子网络”里每次只激活少数几个，避免每个 token 都跑完整大模型。

2. KV Cache 或流式缓存  
缓存前面已经算过的上下文状态，减少重复计算。

3. speculative execution  
也叫推测执行，白话说，就是先让小模型快速猜，再让大模型只验证或修正，目的是降低平均延迟。

真实工程例子里，原生系统如果做实时语音视频助手，通常不会“裸跑”统一大模型，而会在前面挂小模型预取、流式缓存和分段上下文管理。否则交互体验很容易掉到秒级甚至更慢。

## 替代方案与适用边界

很多团队最后不会纯选一条路线，而是走混合方案。

混合方案，白话说，就是前面仍用组合式做低成本过滤，后面在关键环节用原生模型做统一推理。这样做的原因很现实：你既想控制成本，又不想在最关键判断上被级联误差限制死。

MMEdge 一类工作可以看作这种思想的工程化版本。它不是固定一条流水线，而是根据设备资源、当前延迟预算、模态质量动态切换路径。比如边缘设备先跑轻量模型，如果发现样本复杂、网络允许、预算够，再把样本升级给更强模型。

可以用下表快速判断三类路线：

| 方案 | 资源需求 | 可解释性 | 延迟容忍度 | 数据门槛 | 适合场景 |
| --- | --- | --- | --- | --- | --- |
| 组合式 | 低到中 | 高 | 低 | 中 | 审核、检索、客服、规则明确业务 |
| 原生 | 高 | 中到低 | 中到高 | 高 | 统一助手、复杂跨模态推理、前沿产品 |
| 混合式 | 中 | 中 | 中 | 中到高 | 既要成本控制又要关键环节强能力 |

混合式的典型流程是：

```text
输入
  -> first-pass 轻量筛选
  -> 若简单: 直接输出
  -> 若复杂: 原生多模态模型统一推理
  -> 必要时回退到规则/人工复核
```

玩具例子：

一个会议助手要做“边看 PPT 边听发言边生成摘要”。如果只是普通会议，组合式足够：ASR 转录 + OCR 抽字 + 文本摘要。如果遇到大量指代，比如“这一页左下角那个公式”和“刚才第三分钟提到的实验曲线”，组合式很容易丢失空间关系和时间关系，这时原生统一模型更合适。混合方案就是：先默认走组合式，检测到复杂跨模态指代再升级。

真实工程例子：

企业知识库问答里，普通 PDF 问答常常是 OCR + chunk + 文本检索就能解决；但如果问题要求同时理解页面布局、图表趋势、截图中的按钮状态、旁白音频说明，这时“全部转文本”会损失大量信息。最佳做法常常不是全量改造为原生系统，而是在复杂查询上引入原生模型作为 fallback。

所以，适用边界最后可以归纳成一句话：

- 当系统主要矛盾是“成本、稳定、可运维”，组合式优先。
- 当系统主要矛盾是“统一理解、细粒度对齐、交互连贯性”，原生优先。
- 当两类矛盾都存在，混合式通常最现实。
