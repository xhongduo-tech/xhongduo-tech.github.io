## 核心结论

SIQA（Social IQa，Social Interaction Question Answering）是一个专门测量“社交常识推理”能力的数据集。社交常识推理，白话说，就是模型能不能根据人与人互动的场景，判断一个人为什么这么做、接下来大概率会发生什么、另一个人会有什么感受。它不是考百科知识，也不是考算术，而是考“人情世故”背后的可推断规律。

这个基准包含约 3.76 万条三选一题目。每道题由 `context`、`question`、`choices`、`label` 组成：先给一个简短社交场景，再问一个“为什么”或“接下来怎样”的问题，最后给三个候选答案，其中 1 个正确、2 个是对抗式构造的错误答案。对抗式，白话说，就是错误答案看起来也很像真的，目的是防止模型靠关键词投机。

它的评测核心很直接：模型给三个选项分别打分，经过 softmax 变成概率，选最高的那个，最后看准确率。公式通常写成：

$$
p(y=i \mid x)=\frac{e^{s_i}}{\sum_{j=1}^{3} e^{s_j}}
$$

$$
L=-\log p(y^* \mid x), \quad Acc=\frac{\text{正确预测数}}{N}
$$

其中 $s_i$ 是第 $i$ 个选项的分数，$y^*$ 是正确答案。训练时最小化交叉熵损失 $L$，评估时看准确率 $Acc$。

一个新手能立刻看懂的玩具例子是：

| 场景 | 问题 | A | B | C | 正确 |
|---|---|---|---|---|---|
| 朋友在公交车上轻声低语 | 她为什么不想让别人听到？ | 在讲惊喜计划 | 在听广播 | 在检查手机 | A |

这里不需要复杂世界知识，只需要理解常见社交意图：人故意压低声音，往往是因为内容不想被旁人听见。

## 问题定义与边界

SIQA 的问题定义很窄，但正因为窄，才有评测价值。它关注的是“社会互动中的可推断关系”，主要包括三类：

| 类型 | 含义 | 典型问法 |
|---|---|---|
| 意图推理 | 推断角色为什么这样做 | Why did X do this? |
| 结果推理 | 推断接下来可能发生什么 | What will happen next? |
| 情绪推理 | 推断角色会有什么感受 | How would Y feel? |

这里有两个边界要先说清。

第一，它是多选题，不是开放生成。也就是说，SIQA 衡量的是“在有限候选里能否识别更合理答案”，而不是“能否自由写出高质量社交解释”。这两者相关，但不等价。一个生成模型回答得很流畅，不代表它在标准化多选基准上一定高分；反过来也一样。

第二，它只覆盖“常见社交情境”，不覆盖复杂制度、法律、专业流程。比如“同事在会议中被打断会不会不舒服”属于它能测的范围；“跨国公司股权激励谈判中的心理博弈”则超出它的自然分布。

可以把一个样本抽象成下面这样：

| 字段 | 含义 | 约束 |
|---|---|---|
| `context` | 社交场景描述 | 通常很短，聚焦一个互动片段 |
| `question` | 针对场景的推理问题 | 多为 why / what happens next / how feel |
| `choices` | 三个候选答案 | 固定三选一 |
| `label` | 正确选项索引 | 每题只有一个正确答案 |

例如：

- `context`: `Jordan 想告诉 Tracy 一个秘密。`
- `question`: `Jordan 为什么这么做？`
- `choices`:
  - A. 为了保证没人听见
  - B. 想听音乐
  - C. 检查设备
- `label`: `A`

这个例子看起来简单，但它已经体现出边界：模型必须从“秘密”这个语义和“社交保密”这个规范之间建立联系，而不是只匹配词面。

工程上还有一个重要边界：不要自己“顺手造”错误选项。因为一旦错误选项风格太假，比如句子明显更短、用词更怪、语法更差，模型就可能学到“选最像人话的那个”，而不是学到社交推理本身。这会把基准做坏。

## 核心机制与推导

SIQA 的核心机制可以概括成一句话：把每个选项都当成“这个场景下是否合理”的候选解释，让模型做相对比较，而不是做绝对生成。

设输入样本为 $x=(context, question, choices)$。模型分别为三个选项算出分数 $s_1,s_2,s_3$。这个分数也叫 logit，白话说，就是“模型内部觉得这个选项有多像正确答案”的未归一化值。然后用 softmax 变成概率：

$$
p(y=i \mid x)=\frac{e^{s_i}}{e^{s_1}+e^{s_2}+e^{s_3}}
$$

softmax 的作用是把三个任意实数压成 $[0,1]$ 区间内的概率，并保证总和为 1。分数差距越大，概率分布越尖锐。

如果正确答案是第 1 项，那么训练损失是：

$$
L=-\log p(y=1 \mid x)
$$

为什么这么定义？因为当正确答案概率越高时，$\log p$ 越接近 0，前面加负号后损失越小；当正确答案概率很低时，损失会快速变大，训练就会强烈惩罚这种情况。

看一个最小数值例子。假设模型给三个选项的分数是：

- 正确项：$s_1=2.0$
- 错误项：$s_2=0.5$
- 错误项：$s_3=-0.2$

则：

$$
p(y=1 \mid x)=\frac{e^{2.0}}{e^{2.0}+e^{0.5}+e^{-0.2}} \approx 0.74
$$

对应损失：

$$
L=-\log(0.74)\approx 0.30
$$

这说明模型已经较相信正确项，因此损失较小。

再看一个玩具推理流程。上下文是“她转头离开”，问题是“下一步会怎么做？”，三个候选是：

- A. 继续说话
- B. 离开
- C. 拿出电话

如果模型对 B 的分数最高，那么 softmax 后 B 的概率最大，最终预测 B。这个流程的关键不是理解单句语法，而是理解行为连续性：既然“转头离开”已经发生，那么“离开”比“继续说话”更符合社交动作的后续展开。

为什么 SIQA 比很多通用 QA 更难？因为它要求模型把“社会规范”和“行为意图”编码进表示中。比如“轻声说话”可能对应保密，“道歉”可能对应补救关系，“沉默”可能对应尴尬、害怕或不愿冲突。这里不存在一个固定模板，而是统计上常见、语境敏感的推断。

更重要的是，错误选项不是随便编的，而是经过对抗式构造。对抗式构造的意义在于：错误答案表面上也要自然、通顺、像人会写的话。否则模型只会抓表层信号。换句话说，SIQA 试图把“会读人话”和“会推理人际互动”这两件事拆开，让后者真的被测试到。

真实工程例子可以看客服系统。用户说：“我已经等了一周，你们还是没有回复。”如果模型只懂字面，它可能给出泛化回复；如果模型经过类似 SIQA 的社交推理训练，它更可能先识别出用户在表达不满与失望，再选择道歉、解释和补救。这不是把 SIQA 直接拿去上线，而是用它量化模型有没有最基础的社交理解能力。

## 代码实现

最常见的实现方式，是在预训练语言模型上加一个三分类头。分类头，白话说，就是把模型最后一层表示映射成分数的一个小模块。做法通常不是一次性把三个选项同时拼进去，而是把每个选项都和同一个 `context + question` 组成一条输入，分别得到一个分数，最后拼成 `batch_size x 3` 的 logits。

一个简化的数据流如下：

| 阶段 | 输入 | 输出 |
|---|---|---|
| 编码 | `[context; question; choice_i]` | 单个选项表示 |
| 打分 | 单个选项表示 | `logit_i` |
| 聚合 | 3 个 `logit_i` | `logits=[s_1,s_2,s_3]` |
| 训练 | `logits + label` | 交叉熵损失 |
| 评估 | `argmax(logits)` | 预测类别 |

下面给一个可运行的最小 `python` 例子，不依赖深度学习框架，只模拟“打分 -> softmax -> loss -> accuracy”的评测核心：

```python
import math

def softmax(logits):
    m = max(logits)
    exps = [math.exp(x - m) for x in logits]
    s = sum(exps)
    return [x / s for x in exps]

def cross_entropy_loss(logits, label):
    probs = softmax(logits)
    return -math.log(probs[label])

def predict(logits):
    return max(range(len(logits)), key=lambda i: logits[i])

samples = [
    {
        "context": "朋友在公交车上轻声低语。",
        "question": "她为什么不想让别人听到？",
        "choices": ["在讲惊喜计划", "在听广播", "在检查手机"],
        "label": 0,
        "logits": [2.0, 0.3, -0.2],
    },
    {
        "context": "Jordan 想告诉 Tracy 一个秘密。",
        "question": "Jordan 为什么这么做？",
        "choices": ["为了保证没人听见", "想听音乐", "检查设备"],
        "label": 0,
        "logits": [1.7, 0.1, -0.4],
    },
]

correct = 0
losses = []

for sample in samples:
    y_hat = predict(sample["logits"])
    correct += int(y_hat == sample["label"])
    losses.append(cross_entropy_loss(sample["logits"], sample["label"]))

acc = correct / len(samples)
avg_loss = sum(losses) / len(losses)

assert predict([2.0, 0.3, -0.2]) == 0
assert 0.0 < avg_loss < 1.0
assert acc == 1.0

print("accuracy =", acc)
print("avg_loss =", round(avg_loss, 4))
```

如果换成真实训练，伪代码通常长这样：

```python
def score_sample(model, tokenizer, context, question, choices):
    texts = [context + " " + question + " " + c for c in choices]
    inputs = tokenizer(texts, padding=True, truncation=True, return_tensors="pt")
    logits = model(**inputs).logits.squeeze(-1)   # shape: [3]
    pred = logits.argmax().item()
    return pred, logits
```

训练循环的核心则是：

```python
logits = []  # shape: [batch, 3]
for i in range(3):
    # 每个 choice 单独编码并打分
    logits_i = scorer(input_ids_for_choice_i)  # shape: [batch]
    logits.append(logits_i)

logits = stack(logits, dim=1)  # [batch, 3]
loss = cross_entropy(logits, labels)
loss.backward()
optimizer.step()
```

这里最重要的不是 API 细节，而是 shape 设计要清楚。最终用于交叉熵的张量必须是 `batch x 3`，标签是 `batch`，每个值取 `0/1/2` 之一。

## 工程权衡与常见坑

SIQA 真正有价值的地方，是它对“负样本质量”要求高。负样本，白话说，就是错误答案。如果错误答案太差，整个任务就退化成“辨认假话”，而不是“理解社交”。

常见坑可以直接列出来：

| 坑 | 为什么危险 | 避免方式 |
|---|---|---|
| 自行合成错误选项 | 风格偏差明显，模型学会投机 | 保留原始三选项，不随意重写 |
| 把生成式回答硬映射成多选 | 评测口径混乱 | 明确使用 ranking 或 classification |
| 只看总准确率 | 看不到哪类社交能力弱 | 按意图/结果/情绪分桶统计 |
| 忽略数据泄漏 | 同类模板可能出现在训练和验证中 | 严格按官方 split 跑 |
| 过度依赖词面线索 | 模型其实没学到推理 | 做对抗分析和错误案例检查 |

新手最容易犯的错误，是觉得“我用大模型随便生成两个错误答案就行”。问题在于，这两个错误答案很可能在语气、长度、具体程度上明显不同于正确答案。比如正确答案是“为了保证没人听见”，你生成的错误答案却是“因为天空是蓝色的”。这种错误项太假，模型不需要理解社交，只要识别荒谬度就够了。

另一个工程坑是把 SIQA 分数直接等同于“助手有社会智能”。这也不对。SIQA 测的是离线多选推理，不是在线对话控制。真实部署时，一个客服机器人还需要安全策略、事实一致性、拒答机制、长上下文管理。SIQA 更像“基础体检”，不是完整验收。

真实工程里更稳妥的做法是两步走：

1. 先在 SIQA 上评估或微调，确认模型至少能识别基本社交意图。
2. 再把这种能力接到对话系统中，例如作为 reranker、奖励模型特征、或响应前校验层。

比如陪伴型对话系统中，用户说“他刚才那样说让我很难受”。如果系统只追求流畅，很可能转去闲聊；如果加入社交推理校验层，就会先判断“这里最重要的是识别情绪并做情感回应”，而不是立即给知识性答案。

还有一个常见坑是任务迁移过度。SIQA 偏向短文本、单轮场景、有限候选。如果把它学到的模式直接外推到长对话、多角色冲突、文化差异强的场景，效果会显著下降。工程上要把它视为“局部有效能力”，而不是通用解法。

## 替代方案与适用边界

SIQA 不是唯一的常识基准，但它的定位很明确：更偏“社会互动理解”，而不是“实体世界知识问答”。

可以和常见基准做一个对比：

| 数据集 | 主要考什么 | 负样本特点 | 更适合什么场景 |
|---|---|---|---|
| SIQA | 意图、情绪、后果 | 三选一，对抗式自然错误项 | 社交理解、客服、陪伴对话 |
| CommonsenseQA | 通用常识 | 多选，偏实体与关系常识 | 宽泛世界知识推理 |
| ATOMIC | 事件因果与人类需求 | 更像知识图谱/推断库 | 构建推理知识底座 |
| HellaSwag | 常识续写与干扰项辨别 | 强对抗负样本 | 事件续写、语境自然性判断 |

如果你的目标是判断“用户为什么生气”“某句话会不会冒犯对方”“下一句应该先安抚还是先解释”，SIQA 比很多通用常识集更贴近任务。因为它的问题形式本来就是围绕社会意图和情绪展开的。

但如果你的目标是做百科问答、物理常识、程序推理，SIQA 就不够用了。它不能替代更广义的推理数据，也不能证明模型有深层心理理论。它能说明的是：在有限、常见、短上下文的社会互动题里，这个模型是否更容易做出符合人类直觉的判断。

资源不足时，一个实用策略是“先小后大”：

- 先用少量 SIQA 微调，给模型注入基本社交判断偏好。
- 再迁移到更大规模的通用常识或指令数据。
- 最后用真实业务对话做验证，而不是停留在 benchmark 分数。

这种顺序的好处是，模型先学会“什么叫社交上更合理”，再去处理更复杂的开放任务。否则直接在大而杂的 QA 语料上训练，模型可能学到很多表面模式，却没有稳定的社会互动偏好。

不建议把 SIQA 用在这些场景上：

- 需要长程多轮博弈的复杂谈判
- 高文化依赖、强地域差异的礼仪判断
- 需要法律、医学等专业规范优先的高风险决策
- 开放生成质量的最终验收

这些场景要么超出它的标签定义，要么需要额外维度的评测。
