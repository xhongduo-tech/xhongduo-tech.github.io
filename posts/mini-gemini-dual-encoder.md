## 核心结论

Mini-Gemini 是一种面向多模态理解的双视觉编码器框架。多模态理解指模型同时处理图像和文本，并根据两者生成回答。它的核心做法不是把高清图像完整展开后直接送入大语言模型，而是让低分辨率视觉编码器负责全局语义，让高分辨率视觉编码器负责局部细节，再通过补丁信息挖掘把细节融合回低分辨率 token。

视觉 token 可以理解为图像被模型切分和编码后得到的一组向量。大语言模型不能直接“看图片”，它实际接收的是这些向量。问题在于：图像分辨率越高，切出的 patch 越多，视觉 token 数越容易膨胀。Mini-Gemini 的价值就在于，它尽量保持送入 LLM 的视觉 token 数不变，同时把高清细节补回来。

| 方案 | 全局语义 | 局部细节 | token 成本 | 适用场景 |
|---|---:|---:|---:|---|
| 低分辨率单编码器 | 强 | 弱 | 低 | 普通图像理解 |
| 高清全量输入 | 强 | 强 | 很高 | 小图或离线分析 |
| Mini-Gemini | 强 | 强 | 中低 | 文档、OCR、图表、细粒度问答 |

玩具例子：一张票据里，低分辨率图可以先判断“这是发票，里面有金额、日期、编号”。高分辨率图再补回小字里的具体金额、税号和发票代码。如果只看低分辨率，模型知道图像大概是什么，但读不清字；如果把高清图完整切成大量 token，成本会快速上升，并且占用语言模型上下文。

真实工程例子：在企业报销系统里，用户上传发票、行程单、合同截图。系统需要回答“含税金额是多少”“供应商名称是什么”“合同编号是哪一行”。这类任务既需要整体版面判断，也需要局部小字识别。Mini-Gemini 适合这种“先看版面，再按位置补细节”的任务。

## 问题定义与边界

传统视觉语言模型通常先用视觉编码器把图像变成视觉 token，再把这些 token 接到语言模型前面。视觉编码器是把图像转换成向量表示的神经网络。低分辨率输入计算便宜，但会损失小字、细线、表格边界和图表标注。高分辨率输入细节更多，但会产生更多 patch，进一步产生更多 token。

如果一张图像被切成 $P \times P$ 的 patch，图像宽高从 $224 \times 224$ 增加到 $448 \times 448$，patch 大小不变时，token 数通常会变成原来的 4 倍。更一般地，token 数近似满足：

$$
N \approx \frac{H}{p} \times \frac{W}{p}
$$

其中 \(H,W\) 是图像高宽，\(p\) 是 patch 尺寸。分辨率翻倍时，面积变成 4 倍，token 数也接近 4 倍。这就是高分辨率输入的 token 爆炸问题。

| 问题 | 低分辨率方案的缺点 | 高清全量方案的缺点 |
|---|---|---|
| 文档问答 | 看不清小字 | token 太多 |
| 表格理解 | 容易丢列对齐 | 计算成本高 |
| 图表问答 | 读不清坐标轴 | 上下文压力大 |
| 截图分析 | 细小按钮和数值容易丢失 | 推理延迟增加 |

新手例子：看一张会议室白板照片。低分辨率能知道“这是白板，上面有文字和箭头”，但看不清每行写了什么。高清图可以看清文字，但如果整张图都拆成很多视觉 token，语言模型会被大量图像信息占满，推理成本也会上升。问题的本质是：要细节，但不能把成本一起放大。

Mini-Gemini 的边界也要明确。它主要优化的是高分辨率细节提取效率，不是让模型自动获得更强的逻辑推理能力，也不是完全解决幻觉问题。幻觉指模型生成了看似合理但与输入不一致的内容。Mini-Gemini 可以减少“没看清导致答错”的情况，但不能保证模型在复杂推理、常识判断和长文本生成中完全正确。

## 核心机制与推导

Mini-Gemini 的机制可以概括为三步：低分辨率 token 表示全局结构；高分辨率局部块提供细节候选；局部 cross-attention 把相关细节融合回低分辨率 token。cross-attention 是一种让一组向量去查询另一组向量的注意力机制。这里，低分辨率 token 是 query，高分辨率局部特征是 key 和 value。

核心公式可以写成：

$$
t_i=\mathrm{MLP}\Big(q_i+\mathrm{softmax}(\phi(q_i)\phi(K_i)^\top)\,\phi(V_i)\Big)
$$

| 符号 | 含义 |
|---|---|
| \(q_i\) | 第 \(i\) 个低分辨率视觉 token |
| \(K_i\) | 与该 token 对齐的高分辨率候选块 key |
| \(V_i\) | 与该 token 对齐的高分辨率候选块 value |
| \(\phi\) | 投影层，把向量映射到适合计算注意力的空间 |
| softmax | 把相似度转换成权重，权重和为 1 |
| MLP | 多层感知机，用于融合后的非线性变换 |
| \(t_i\) | 注入局部细节后的新视觉 token |

这个公式的含义很直接。第 \(i\) 个低分辨率 token 先带着全局语义，例如“这里大概是表格右下角”。然后它只去看自己空间位置对应的高清候选块，而不是看整张高清图。attention 会计算 query 和每个高清候选块之间的相关性，再按权重聚合细节。最后，聚合出的细节通过残差连接加回原 token，再交给 MLP 更新表示。

残差连接指把原始输入直接加到变换结果里。它的作用是保留原 token 已有的全局信息，避免细节融合时把原来的语义冲掉。

玩具数值例子：设 \(q=[1,0]\)，两块高分辨率 key 分别为 \(k_1=[1,0]\)、\(k_2=[0,1]\)，value 分别为 \(v_1=[2,0]\)、\(v_2=[0,4]\)，投影层 \(\phi\) 暂时看成恒等映射。query 与两个 key 的相似度是 \([1,0]\)。softmax 后约为 \([0.731,0.269]\)。细节聚合结果为：

$$
0.731[2,0]+0.269[0,4]=[1.462,1.076]
$$

再加上原始 \(q\)，得到 \([2.462,1.076]\)。这说明新 token 仍保留原来的方向，同时吸收了更匹配的高清局部细节。

这不是全局跨图检索，而是局部补丁信息挖掘。补丁信息挖掘指从与当前位置相关的高分辨率 patch 中提取有用细节。它的工程意义是：把“高清全图输入”改成“按需取细节”，从而把计算压力从全量 token 爆炸变成局部候选聚合。

## 代码实现

下面的代码是一个可运行的最小例子，用 NumPy 模拟 Mini-Gemini 的局部 mining。它不训练模型，只演示“每个低分辨率 token 只融合自己对应的高分辨率候选块”。

```python
import numpy as np

def softmax(x):
    x = x - np.max(x)
    e = np.exp(x)
    return e / e.sum()

def local_mining(lr_tokens, hr_tokens):
    """
    lr_tokens: [N, D]
    hr_tokens: [N, M, D]
    N: 低分辨率 token 数
    M: 每个低分辨率 token 对应的高分辨率候选块数
    D: 向量维度
    """
    n, d = lr_tokens.shape
    fused = []

    for i in range(n):
        q = lr_tokens[i]          # [D]
        k = hr_tokens[i]          # [M, D]
        v = hr_tokens[i]          # [M, D]

        scores = (k @ q) / np.sqrt(d)   # [M]
        attn = softmax(scores)          # [M]
        detail = attn @ v               # [D]

        # 这里用 tanh 模拟 MLP 的非线性变换
        t = np.tanh(q + detail)
        fused.append(t)

    return np.stack(fused)

lr = np.array([
    [1.0, 0.0],
    [0.0, 1.0],
])

hr = np.array([
    [[2.0, 0.0], [0.0, 4.0]],
    [[3.0, 0.0], [0.0, 2.0]],
])

out = local_mining(lr, hr)

assert out.shape == (2, 2)
assert out[0, 0] > out[0, 1]   # 第一个 LR token 更偏向第一块 HR 细节
assert out[1, 1] > out[1, 0]   # 第二个 LR token 更偏向第二块 HR 细节

print(out)
```

完整工程里，流程通常是这样的：

| 步骤 | 输入 | 输出 | 作用 |
|---|---|---|---|
| 低分辨率编码 | 整图缩小版 | LR tokens | 保留全局语义 |
| 高分辨率编码 | 局部高清块 | HR tokens | 提供细节候选 |
| 局部 attention | LR query + HR key/value | 细节聚合结果 | 挖掘局部信息 |
| 残差融合 | 原 token + 细节 | 新 token | 稳定更新表示 |
| MLP | 融合 token | 最终 token | 非线性增强 |

实现时最关键的是空间对齐。空间对齐指低分辨率 patch 和高分辨率局部区域之间有稳定的位置映射。例如，低分辨率第 3 行第 5 列的 token，应该对应高清图中相同相对位置的一组候选块。如果这个映射不稳定，attention 算得再准确，也会在错误区域里找细节。

真实工程实现通常会有两个编码器。低分辨率图像进入主视觉编码器，得到固定数量的视觉 token。高分辨率图像被切成局部区域，进入高分辨率编码器。然后系统按网格位置建立 LR token 到 HR patch 的对应关系，完成局部融合。最后，融合后的 token 再送入大语言模型。

## 工程权衡与常见坑

Mini-Gemini 的收益主要来自细粒度任务。细粒度任务指答案依赖局部小信息的任务，例如读票据金额、识别表格列名、查看图表坐标轴、分析仪表盘数值。如果数据主要是普通自然图像，比如“图中是不是一只猫”“场景是在室内还是室外”，低分辨率编码器可能已经足够，双编码器带来的收益不一定覆盖成本。

| 常见坑 | 后果 | 规避方式 |
|---|---|---|
| HR 与 LR 不对齐 | 细节错位，模型问错区域 | 固定空间映射规则 |
| 直接全量送入 HR token | token 再次爆炸 | 只做局部 mining |
| 训练集缺少 OCR、图表、文档 | 学不到细节能力 | 补足细粒度数据 |
| 只看整体指标 | 掩盖小字识别失败 | 增加细粒度样例评测 |
| 裁剪策略随数据变化 | 推理阶段分布漂移 | 训练和推理保持同一预处理流程 |

新手例子：如果你把高清图随便裁，左边的小字被分到右边 token 的候选块里，模型就会“问错地方”。这不是模型不会看，而是输入组织错了。多模态模型里的很多错误，表面看是模型理解失败，实际是预处理、裁剪、缩放和对齐规则不稳定。

真实工程例子：一个客服质检系统要分析坐席截图，判断“当前订单状态是否异常”。低分辨率能看出这是订单页面，高分辨率能读到状态栏里的“小字：退款中”。如果 HR patch 和 LR token 对齐错误，模型可能把订单编号区域当成状态区域，最后回答看似合理但完全错位。

另一个常见权衡是延迟。Mini-Gemini 虽然避免把全部高清 token 送入 LLM，但仍然需要额外的高分辨率编码器和局部融合模块。对于移动端实时识别、低成本批量分类这类任务，它可能仍然太重。工程判断标准很简单：如果任务答案经常依赖小字、密集结构、局部数值，Mini-Gemini 更值得；如果任务主要是粗粒度分类，简单视觉编码器更直接。

评估也不能只看平均准确率。平均指标可能被大量简单样本拉高，却掩盖关键失败样本。更可靠的做法是单独构造文档小字、表格错列、图表坐标、截图按钮、票据编号等评测集，观察模型是否真正从高清细节中受益。

## 替代方案与适用边界

Mini-Gemini 不是唯一解。它是一种在“全局语义”和“局部细节”之间做折中的方案。替代方案包括低分辨率单编码器、高清全量输入、图像切块全量编码、层级视觉编码、稀疏视觉 token 压缩等。

| 方案 | 优点 | 缺点 | 适用边界 |
|---|---|---|---|
| 高清全量输入 | 细节完整 | 代价高 | 小图、离线处理 |
| 低分辨率单编码器 | 简单高效 | 细节差 | 粗粒度任务 |
| 图像切块全量编码 | 细节较好 | token 多 | 局部分析较多 |
| 层级视觉编码 | 结构清晰 | 训练和实现复杂 | 需要多尺度表示 |
| 稀疏 token 压缩 | 成本可控 | 可能丢关键信息 | 冗余区域较多的图像 |
| Mini-Gemini | 平衡全局与细节 | 对齐和训练更复杂 | 文档、表格、图表、截图 |

新手对比可以这样理解：直接高清输入像把整本书所有页面一次性摊开，信息完整但很重；低分辨率方案像只看目录，知道大概内容但看不见细节；Mini-Gemini 像先看目录，再只翻相关页。这个比喻只用于理解信息组织方式，真正的技术机制仍然是视觉 token、局部 cross-attention 和残差融合。

Mini-Gemini 的适用边界包括三点。第一，它不适合只追求最低延迟的轻量应用，因为双编码器会增加额外计算。第二，它不适合没有高分辨率细节价值的任务，因为此时局部 mining 带来的收益有限。第三，它不会自动消除多模态模型的幻觉问题，只是降低“看不清细节”造成的错误概率。

在工程选型时，可以按问题类型判断：如果输入是票据、合同、表格、图表、网页截图、仪表盘截图，且问题经常指向局部文字或数值，Mini-Gemini 的设计很匹配。如果输入是普通照片，问题多为“这是什么”“图中有几个人”“场景在哪里”，低分辨率视觉编码器通常已经能覆盖大部分需求。
