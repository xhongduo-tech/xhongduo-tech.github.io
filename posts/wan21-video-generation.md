## 核心结论

Wan 2.1 可以把它看成一套“面向视频生成的开源模型族”。模型族的意思是，不是单个权重，而是一组面向不同任务和规模的模型。按公开资料，它的核心路线是 `DiT + Flow Matching + 3D causal VAE`，覆盖文生视频（`T2V`）、图生视频（`I2V`）、首尾帧生成视频（`FLF2V`）等任务。

这里先给一个最重要的判断：Wan 2.1 的价值不只是“能生成视频”，而是把中文提示词、多任务输入形式、视频潜空间建模、显存优化策略放到了同一条工程链路里。对零基础到初级工程师来说，这意味着它不是只适合看演示，而是适合做“先低成本验证，再逐步升配”的实际工作流。

新手版玩具例子可以这样理解。输入一句中文提示词：“电商口红在桌面旋转，镜头慢慢推进，画面出现可读的中文品牌字样。” 模型不是先画第 1 帧，再画第 2 帧，直到把视频拼出来；它更像是先把整段视频压缩到一个更小、更好建模的“潜空间”，然后在这个潜空间里逐步生成完整运动，最后再还原成视频。

下表先看总览：

| 能力 | 典型输入 | 典型输出 | 中文能力 | 典型分辨率 | 常见模型规模 |
|---|---|---|---|---|---|
| `T2V` | 文本提示词 | 从零生成视频 | 中文提示词更友好 | `480P`、`720P` | `1.3B`、`14B` |
| `I2V` | 单张图片 + 提示词 | 让静态图动起来 | 可结合中文描述控制运动 | `480P`、`720P` | `14B` 为主 |
| `FLF2V` | 首帧 + 尾帧 + 文本 | 生成中间运动过程 | 中文图文任务更有针对性 | `480P`、`720P` | `14B` |
| 中文文字渲染 | 中文提示或含文字场景 | 视频中可见中英文文字 | 官方强调支持中英文文字生成 | 依任务而定 | 依任务而定 |

如果只记一句话，可以记成：Wan 2.1 适合做“中文视频生成的通用底座”，尤其适合先在 `1.3B + 480P` 上跑草图，再往更高质量版本升级。

## 问题定义与边界

视频生成不是“生成很多张图片”这么简单。它本质上是在时间维度上联合建模三个目标：画面一致性、运动连续性、条件对齐。画面一致性指的是同一个物体在连续帧里不能忽然变形或换身份；运动连续性指的是镜头运动和物体运动要自然；条件对齐指的是输出视频要符合文本、图片或首尾帧这类输入条件。

可以用一个最简形式写成：

$$
V\in\mathbb{R}^{T\times H\times W\times 3},\quad c=\text{condition}
$$

这里的 $V$ 是视频张量，也就是一个带时间维度的图像序列；$T$ 表示帧数，$H,W$ 表示分辨率，$3$ 是 RGB 三个通道；$c$ 是条件，可以是文本、图片、首尾帧，或者它们的组合。

边界也要讲清楚。Wan 2.1 强项是通用视频生成、多任务条件控制、中文提示词和中文字渲染，但它不是所有视频任务的统一答案。它不是专门做长视频编辑的系统，也不是视频理解模型，更不能保证“任意中文都稳定、任意长视频都不炸显存、任意复杂镜头都一步成功”。

一个真实工程例子是电商短视频。你要生成 5 秒商品广告，不是要求“每一帧都像海报”，而是要求商品不要变形、镜头能推进、文字尽量可读、动作不要突然跳变。Wan 2.1 解决的正是这种联合目标，而不是替代剪辑软件做完整后期。

边界表如下：

| 能做什么 | 不能直接保证什么 | 使用时要注意什么 |
|---|---|---|
| 从文本生成短视频 | 任意长视频都稳定生成 | 帧数和分辨率一上去，显存迅速增加 |
| 从图片生成动态片段 | 任意复杂中文都稳定可读 | 中文文字能力强，不等于长句字幕必然稳定 |
| 用首尾帧约束中间运动 | 严格影视级镜头语法 | 最好先用低分辨率验证运动路径 |
| 兼顾中文场景与多任务输入 | 替代专业视频编辑流程 | 更适合原型生成，不是后期工业管线全包 |

还有一个容易混淆的点：中文社区里有时会把 Wan 2.1 说成“某公司的视频模型”。做技术判断时，应以官方仓库、技术报告、模型卡为准，避免把二手传播中的归属说法当成结论。

## 核心机制与推导

Wan 2.1 的第一层关键机制，是不直接在像素空间生成视频。像素空间就是我们看到的原始帧序列，维度很高、冗余很多、时间依赖也很重，直接生成的成本非常高。它先用 `3D causal VAE` 把视频压缩到潜空间。VAE 可以先理解成“把复杂数据压缩成更好建模的表示”的编码器-解码器结构；`3D` 表示同时处理时间和空间；`causal` 表示在时间方向上要遵守前后依赖，不随意看未来帧。

公式先写成：

$$
z=E(V)
$$

这里的 $E$ 是编码器，$V$ 是原视频，$z$ 是压缩后的潜变量。后续生成不是直接生成 $V$，而是在潜空间里生成 $z$，最后再由解码器把 $z$ 还原成视频。

第二层关键机制，是它采用 `Flow Matching`。这个术语第一次出现可以白话解释成：它不是把视频看成“反复去噪”的离散步骤，而是学习一条从噪声流向真实数据的连续轨迹。对应公式是：

$$
\frac{dx_t}{dt}=v_\theta(x_t,t,c)
$$

这里 $x_t$ 表示时刻 $t$ 的中间状态，$v_\theta$ 是模型学到的速度场，也就是“当前应该朝哪个方向走”。条件 $c$ 仍然来自文本、图片或首尾帧。直观上，它学的不是“这一小步该擦掉多少噪声”，而是“从当前位置往哪个方向移动，才能更快走到目标视频”。

这套路线的好处，是把高维视频生成转成“潜空间上的连续动力系统求解”。对新手来说，可以用一个玩具例子理解。

假设你要生成一个 5 秒、16 fps 的短视频，总帧数是：

$$
T=5\times 16=80
$$

如果编码器按约 4 帧为一个处理块，那么大约会有 $80/4=20$ 个时间块。这样做的意义不是数学上更优雅，而是工程上更省显存。因为编码和解码不必把全部 80 帧一次性展开到显存里，而是按块缓存和处理，把峰值显存压下来。

机制链路可以画成下面这个简单示意：

| 阶段 | 输入 | 输出 | 作用 |
|---|---|---|---|
| 条件编码 | 文本 / 图片 / 首尾帧 | 条件特征 `c` | 把用户输入变成模型可用信号 |
| 视频编码 | 原视频或训练样本 | 潜变量 `z` | 用 `3D causal VAE` 压缩视频 |
| 潜空间生成 | 噪声 + 条件 `c` | 生成的潜变量 | 用 `Flow Matching` 学习连续流 |
| 视频解码 | 潜变量 | 最终视频帧 | 从潜空间还原出可播放视频 |

再看分块处理的工程含义。公开资料里常见的思路是把长视频切成 `1 + T/4` 级别的块做缓存。这里不必死记公式细节，重点是理解“前面可能有一个初始化块，后面按约 4 帧为单位递推”。它的作用有三点：

| 分块思路 | 直观含义 | 工程收益 |
|---|---|---|
| 初始化块 | 建立起始状态或上下文 | 避免一开始就展开全部序列 |
| 每块约 4 帧 | 时间上切片处理 | 降低峰值显存 |
| 特征缓存 | 前面块的中间结果可复用 | 减少重复计算，提升长视频可行性 |

如果把整个机制压缩成一句话，就是：文本或图片先变成条件，视频先被压缩到潜空间，模型在潜空间里沿着学习到的速度场从噪声流向目标，再把结果解码回视频。这也是它和“直接生成像素序列”的本质区别。

## 代码实现

实际落地时，最重要的不是把论文术语背下来，而是把推理流程跑通。新手最容易犯的错，是上来就选大模型、高分辨率、长时长，最后直接 OOM。正确顺序通常是：先用小模型验证镜头和构图，再逐步升配。

下面给一个最小可运行的 Python 玩具代码。它不调用真实 Wan 2.1 权重，而是用参数检查模拟一个“生成请求是否合法”的流程。这样做的目的是把推理时最核心的工程约束先讲清楚。

```python
def estimate_video_spec(seconds: int, fps: int, chunk_size: int = 4):
    assert seconds > 0
    assert fps > 0
    assert chunk_size > 0

    num_frames = seconds * fps
    num_chunks = (num_frames + chunk_size - 1) // chunk_size

    return {
        "num_frames": num_frames,
        "num_chunks": num_chunks,
    }


def choose_profile(model: str, resolution: str, offload_model: bool, t5_cpu: bool):
    assert model in {"T2V-1.3B", "T2V-14B", "FLF2V-14B"}
    assert resolution in {"480P", "720P"}

    if model == "T2V-1.3B" and resolution == "720P":
        return "warning: 1.3B at 720P may be unstable"

    if model in {"T2V-14B", "FLF2V-14B"} and not (offload_model or t5_cpu):
        return "warning: high OOM risk on single GPU"

    return "ok"


spec = estimate_video_spec(seconds=5, fps=16)
assert spec["num_frames"] == 80
assert spec["num_chunks"] == 20

profile = choose_profile(
    model="T2V-14B",
    resolution="720P",
    offload_model=True,
    t5_cpu=False,
)
assert profile == "ok"
```

上面这个玩具例子对应的真实含义很直接：5 秒、16 fps 就是 80 帧；如果按 4 帧一个块处理，就是 20 个块；当你把模型换到 `14B + 720P` 时，就必须优先考虑显存卸载和文本编码放 CPU。

再给一段更贴近真实推理流程的伪代码：

```python

# 3. prepare prompt / reference image

resolution = "480P"
fps = 16
num_frames = 81
offload_model = False
t5_cpu = False

# text_features = text_encoder(prompt)
# wan = load_model(model, offload_model=offload_model, t5_cpu=t5_cpu)
# latent = wan.sample(text_features, resolution=resolution, fps=fps, num_frames=num_frames)
# video = vae.decode(latent)
# save_mp4(video, "output.mp4", fps=fps)
```

关键参数可以先按下面理解：

| 参数 | 典型值 | 作用 | 推荐理解 |
|---|---|---|---|
| `model` | `T2V-1.3B` / `T2V-14B` / `FLF2V-14B` | 选择任务与规模 | 先小后大，先通用后专用 |
| `resolution` | `480P` / `720P` | 输出分辨率 | 新手默认先 `480P` |
| `offload_model` | `True/False` | 把部分模型权重卸载到 CPU | 用时间换显存 |
| `t5_cpu` | `True/False` | 把文本编码器放到 CPU | 降低 GPU 压力 |
| `fps` | 如 `16` | 每秒帧数 | 越高越平滑，也越贵 |
| `num_frames` | 如 `81` | 总帧数 | 决定时长和序列成本 |

推荐配置思路不是“哪个参数最强就选哪个”，而是按任务阶段来选：

| 场景 | 推荐配置 | 原因 |
|---|---|---|
| 单卡草图验证 | `T2V-1.3B + 480P` | 成本低，先验证镜头、构图、文案 |
| 单卡高风险尝试 | `14B + offload_model=True + t5_cpu=True` | 显存吃紧时先求能跑通 |
| 多卡正式生成 | `14B + 720P + 并行策略` | 面向最终质量版本 |

真实工程例子可以这样设计。你要做一条中文电商口红视频，先在单卡上跑 `T2V-1.3B`，提示词直接写中文，重点看三件事：商品形态是否稳定、镜头推进是否符合预期、短中文文案是否可读。如果这三点都不对，直接切 `14B` 只会更贵，不会自动变正确。只有当低成本版本把镜头语言验证清楚后，再切到 `14B` 或 `FLF2V-14B` 做正式版，才是合理路径。

## 工程权衡与常见坑

Wan 2.1 的第一大工程约束是显存。模型规模、分辨率、帧数，这三项不是线性叠加，而是会一起抬高显存和时延。初学者最容易把 `14B + 720P + 长序列` 直接绑定在一起，这通常是最容易失败的组合。

第二个常见误区，是把“支持中文提示词和中文字渲染”理解成“任意中文都稳定”。这是不准确的。能力强，表示在任务设定和数据分布覆盖内更有优势；不表示任何长句字幕、复杂排版、快速镜头切换都能稳定保真。

坑位表可以先记下面这些：

| 常见坑 | 表现 | 原因 | 规避方式 |
|---|---|---|---|
| `1.3B` 跑 `720P` 不稳定 | 质量波动、速度慢、可能失败 | 小模型容量有限 | 默认先用 `480P` |
| `14B` 单卡 OOM | 直接爆显存 | 参数量和序列长度都大 | 开 `offload_model`、`t5_cpu`，必要时多卡 |
| 长帧数导致成本飙升 | 推理很慢或无法运行 | 时间维度被拉长 | 先缩短时长验证脚本 |
| 中文长句字幕失真 | 字不稳、变形、难读 | 文本渲染本身仍是高难任务 | 用短词、短句、低运动背景先验证 |
| 并行策略配置不当 | 程序报错或性能差 | 分片策略有整除约束 | 检查 head 数、序列长度与并行切分是否匹配 |

再看场景化配置：

| 场景 | 目标 | 推荐做法 |
|---|---|---|
| 单卡预览 | 看镜头方向和主体一致性 | `1.3B + 480P + 短时长` |
| 多卡正式生成 | 拿高质量成片 | `14B + 720P + 并行推理` |
| 长视频推理 | 拉长时间维度 | 分块处理、缓存特征、降低单次峰值 |

这里的“先低配验证”不是保守，而是标准工程做法。比如你想做“中文海报视频”，如果一次性要求长句字幕、复杂镜头、`720P`、单卡推理，那么失败概率很高。更靠谱的顺序是：先做 `480P` 草图验证运动，再验证短文案是否可读，最后再升到高质量版本。

## 替代方案与适用边界

Wan 2.1 不是唯一选择。视频生成模型的选型要看任务，而不是只看谁更新、谁更大。和它最常被拿来对比的一类方案是 `SVD`，也就是 Stable Video Diffusion 这一类图生视频路线。

先给结论：如果任务主要是“单张图变成一个短动态片段”，而且不强调中文文字渲染和多任务覆盖，`SVD` 仍然是合理选择。如果任务是“文本生成视频、图片生成视频、首尾帧控制、中文场景、文字可读性”这些需求叠加在一起，Wan 2.1 更贴近目标。

对比如下：

| 任务类型 | 典型输入 | 输出长度 | 中文能力 | 适合场景 | 推荐模型 |
|---|---|---|---|---|---|
| 短片段图生视频 | 单张图片 | 典型短序列，如 14/25 帧 | 不以中文文字为主打 | 图片轻动态、短演示 | `SVD` |
| 文生视频 | 文本提示词 | 可覆盖更完整短视频 | 更适合中文提示场景 | 中文脚本原型、广告草图 | `Wan 2.1` |
| 首尾帧约束生成 | 首帧 + 尾帧 + 文本 | 中间过程补全 | 更强调任务覆盖 | 镜头过渡、动作桥接 | `Wan 2.1` |
| 需要中英文可见文字 | 文本或图片条件 | 短视频 | 官方强调支持中英文文字 | 中文电商、字幕化封面 | `Wan 2.1` |

两者路线上的差别也要知道。`SVD` 主要是 `image-to-video`，属于 latent diffusion 路线，典型输出是较短片段；Wan 2.1 更强调 `T2V + I2V + FLF2V` 的任务覆盖，以及 `3D causal VAE + Flow Matching` 的视频潜空间生成路线。

给一个新手版决策例子。如果你只有一张产品图，只想做一个 14 到 25 帧的轻微镜头运动展示，`SVD` 这类方案可能已经够用；但如果你要的是“中文封面文案可读、镜头推进、商品姿态稳定、甚至还要首尾帧控制”，Wan 2.1 更适合，因为它的任务定义本来就更宽。

所以最终不是“谁绝对更强”，而是“哪个模型和你的任务边界更匹配”。
