## 核心结论

Codex 的训练流程可以压缩成一条因果链：先收集“自然语言说明 + 代码实现”的大规模配对数据，再把这种配对作为监督信号去微调 GPT-3，最后用标准化单元测试评估模型是否真的会“按说明写函数”。

这里的“微调”是指：拿一个已经学会通用语言模式的基础模型，再用某类专门数据继续训练，让它更擅长特定任务。对 Codex 而言，这个特定任务就是从自然语言描述生成代码。

公开资料里，Codex 常被描述为基于 GPT-3 12B 参数版本，使用约 159GB 过滤后的 GitHub 代码数据进一步训练。训练重点不是随便喂整仓库文本，而是尽量让模型反复看到“docstring 到函数实现”的映射。这里的 docstring 可以理解为“写在函数前面的功能说明文字”，它既像需求，也像输入条件。

评估阶段的核心指标是 HumanEval 上的 pass@k。HumanEval 是一个小而标准的代码生成测试集，每道题给出函数签名和文字说明，再用隐藏测试判断生成代码是否正确。公开结果中，Codex 的 pass@1 达到 28.8%，意思是“每题只取 1 次生成结果时，约有 28.8% 的题能直接通过测试”。这个成绩不能说明模型已经会做完整软件工程，但足以说明它学到了相当强的“从说明到中小函数实现”的能力。

玩具例子可以这样理解：你写一段说明“返回斐波那契数列第 n 项”，模型生成一个 `fib(n)`。如果生成代码能通过测试，例如 `fib(0)==0`、`fib(1)==1`、`fib(7)==13`，那这次样本就算成功。Codex 的训练目标，本质上就是把这种成功概率尽可能提高。

## 问题定义与边界

Codex 解决的不是“自动完成整个软件项目”，而是一个更窄、更清晰的问题：给定自然语言说明和局部上下文，生成满足语义要求的代码片段，尤其是函数级代码。

“边界”很重要，因为很多对代码模型的误解都来自把函数补全能力误当成完整项目设计能力。Codex 训练中最典型的数据形式，是这样的配对：

| 维度 | 内容 | 作用 |
| --- | --- | --- |
| 输入形式 | docstring、注释、函数签名 | 把需求翻译成可条件生成的上下文 |
| 目标输出 | 函数体或紧接其后的实现代码 | 让模型学习“说明对应什么实现” |
| 测试集 | HumanEval 单元测试 | 判断代码是否语义正确，而不是只看像不像代码 |

一个面向新手的最小例子是“根据说明写平均值函数”。输入可能是：

```python
def mean(xs):
    """返回列表平均值；空列表返回 0.0"""
```

目标输出则是某个合法实现，例如判断空列表、做求和、返回浮点结果。这里模型学的不是“平均值”这个数学概念本身，而是“当说明里出现这些词时，什么代码结构最可能正确”。

因此，Codex 的能力边界至少包括三层。

第一层是任务边界：更适合 docstring 到中小函数，而不是跨十几个文件的系统级重构。

第二层是上下文边界：它依赖 prompt 中给出的局部信息。如果依赖关系、类型约束、外部 API 行为没有写进上下文，模型只能靠训练分布猜。

第三层是评估边界：HumanEval 主要测函数正确性，不直接测长期可维护性、性能、架构一致性，也不保证安全性。

真实工程里，这意味着一个模型可以在标准题上分数不错，但在公司私有 SDK、跨服务调用、旧版本 API 兼容这类场景里明显退化。因为这些信息往往不在训练分布里，也不在给定 prompt 里。

## 核心机制与推导

Codex 的核心机制，是把“自然语言描述代码”的过程转化成标准的条件语言建模。条件语言建模可以直白理解为：前面给定一段上下文，模型按概率一个 token 一个 token 地续写后面的内容。

如果把 docstring 记为 $d$，代码序列记为 $y_1, y_2, ..., y_T$，那么训练目标可以写成：

$$
P(y \mid d) = \prod_{t=1}^{T} P(y_t \mid d, y_{<t})
$$

这表示：在给定说明文字和已生成前缀的条件下，最大化正确代码 token 的概率。换句话说，模型并不是先“理解完需求”再整体输出代码，而是在每一步预测下一个最合理的代码符号。

训练流程可以概括成四步。

1. 从 GitHub 代码中抽取高质量样本。
2. 对样本做过滤、去重、清洗，尽量保留“说明 + 实现”关系清晰的片段。
3. 把这些片段拼成 Transformer 可读的 token 序列。
4. 用监督信号更新模型参数，让正确实现的 token 概率升高。

这里的 Transformer 可以理解为“通过注意力机制同时看上下文不同位置关系的序列模型”。它适合代码，是因为代码对远距离依赖很敏感，例如前面定义的变量、缩进结构、函数签名都会影响后续 token。

评估时，Codex 不是只生成 1 个答案，而是通常采样多个候选，再看其中有多少能通过测试。于是就引出了 pass@k。

设一题总共采样 $n$ 个候选，其中有 $c$ 个通过测试。如果最终从中取出 $k$ 个候选，那么至少选中 1 个正确答案的概率为：

$$
\text{pass@k} = 1 - \frac{\binom{n-c}{k}}{\binom{n}{k}}
$$

式子的含义很直接：先算“一个正确答案都没选中”的概率，再用 1 去减。

玩具例子如下。

如果某题采样 $n=200$ 个候选，只有 $c=3$ 个通过测试，那么：

$$
\text{pass@1} = 1 - \frac{\binom{197}{1}}{\binom{200}{1}} = 1 - \frac{197}{200} = 1.5\%
$$

这说明只给你 1 次机会时，命中正确实现的概率很低。

如果同样是 $n=200$，但可通过测试的候选有 $c=60$ 个，允许看 $k=10$ 个样本，则：

$$
\text{pass@10} = 1 - \frac{\binom{140}{10}}{\binom{200}{10}}
$$

这个值大约在 56% 左右，明显高得多。直观含义是：不是模型“单发必中”，而是模型分布里已经包含不少正确答案，多采样就能把它们捞出来。

所以，Codex 的真实能力通常要分成两层看。

一层是“分布质量”：正确答案在不在模型生成分布里。

另一层是“搜索能力”：你采样多少次、怎么排序、是否执行测试，决定你能不能把正确答案找出来。

## 代码实现

下面用一个可运行的 Python 玩具脚本模拟 HumanEval 风格的评估流程。它不是训练 Codex，而是复现“多候选生成 + 自动测试 + 统计 pass@k”这条主线。

```python
from math import comb

def estimate_pass_at_k(n: int, c: int, k: int) -> float:
    assert 0 <= c <= n
    assert 1 <= k <= n
    if n - c < k:
        return 1.0
    return 1.0 - comb(n - c, k) / comb(n, k)

def run_tests(candidate, tests):
    for x, expected in tests:
        if candidate(x) != expected:
            return False
    return True

def fib_bad(n: int) -> int:
    return n  # 错误实现

def fib_ok(n: int) -> int:
    if n < 2:
        return n
    a, b = 0, 1
    for _ in range(2, n + 1):
        a, b = b, a + b
    return b

tests = [(0, 0), (1, 1), (2, 1), (7, 13)]

candidates = [fib_bad, fib_bad, fib_ok, fib_bad]
passed = sum(run_tests(fn, tests) for fn in candidates)
n = len(candidates)

assert run_tests(fib_ok, tests) is True
assert run_tests(fib_bad, tests) is False
assert passed == 1
assert abs(estimate_pass_at_k(n=4, c=1, k=1) - 0.25) < 1e-9

print("n =", n)
print("c =", passed)
print("pass@1 =", estimate_pass_at_k(n, passed, 1))
print("pass@2 =", estimate_pass_at_k(n, passed, 2))
```

这段代码对应的机制是：

1. 先准备多个候选函数，模拟模型多次采样。
2. 再用测试用例逐个执行，统计通过数量 $c$。
3. 最后根据 $n, c, k$ 计算 pass@k。

如果把它对应到真实 Codex 流程，大致可以写成下面的伪代码：

```python
prompt = build_prompt(docstring, signature, few_shot_examples)
samples = sample_model(prompt, num_samples=n, temperature=t)

passed = 0
scored_candidates = []

for code in samples:
    logp = model_logprob(code, prompt)
    ok = run_in_sandbox(code, unit_tests)
    if ok:
        passed += 1
    scored_candidates.append((code, logp, ok))

scored_candidates.sort(key=lambda x: x[1], reverse=True)
result = {
    "pass@k": estimate_pass_at_k(n, passed, k),
    "top_candidate": scored_candidates[0],
}
```

这里有三个工程上不能省的点。

第一，多候选采样。因为正确实现往往不是概率最高的那个，单次贪心解码会错过本来“会做但没一次写对”的题。

第二，自动测试。代码生成和普通文本不同，语义正确性必须靠执行结果判定，不能只看表面像不像。

第三，候选排序。`logp` 就是模型给一段输出分配的对数概率，可理解为“模型自己有多确信这段输出自然”。它不等于正确性，但常常可以作为排序信号。

真实工程例子是 GitHub Copilot 一类产品。用户在编辑器里写出 docstring 或函数签名后，后端模型会根据上下文生成若干补全候选，再按概率或策略排序，把最可能有用的结果展示给用户。更成熟的系统还会接入静态分析、沙箱测试或 lint 规则做二次筛选。

## 工程权衡与常见坑

Codex 训练和使用的关键权衡，不是“要不要生成代码”，而是“用多大代价换多高的正确率”。

最典型的权衡是多候选采样。采样次数增加，pass@k 通常上升，但推理成本、测试成本、延迟也同步上升。对在线补全产品而言，这会直接影响编辑器响应时间。

常见风险可以整理成下面这张表：

| 风险类型 | 典型表现 | 规避方式 |
| --- | --- | --- |
| 未定义 API | 生成看起来合理但项目里不存在的方法名 | 检查项目符号表，结合静态分析 |
| 链式调用错误 | 多个对象方法串联时顺序或返回类型错 | 提供更完整上下文，加入类型信息 |
| 变量未绑定 | 使用未声明变量、作用域错误 | 约束生成范围，补充签名与局部变量 |
| 不安全调用 | 执行 shell、网络、文件操作时越权 | 在沙箱中运行，限制系统调用 |
| 排序失真 | logp 高但语义错误 | 用测试结果或规则校验覆盖排序偏差 |

一个新手容易误解的问题是：为什么模型“看起来懂了需求”，却还会写出不存在的 API？原因是语言模型本质上在做概率续写。只要某种写法在训练分布中常见，或者和上下文词面强相关，它就可能被续写出来，即使当前项目并没有这个接口。

另一个常见坑是长链依赖。比如 docstring 写“读取数据库记录，过滤状态，再调用外部支付 API，最后回写缓存”。这类任务包含多个外部系统、多个中间对象、多个失败分支。模型可能每一步都写得像样，但整体一执行就断，因为真实依赖约束没有全部进入上下文。

真实工程里，比较稳妥的做法通常是：

1. 把生成目标限制在单函数或单模块。
2. 给足签名、类型、已有变量、示例调用。
3. 生成多个候选。
4. 先做静态检查，再做沙箱执行。
5. 人工审查外部调用、权限操作和资源释放。

也就是说，Codex 适合作为“高效草拟器”，不适合作为“无人值守提交器”。

## 替代方案与适用边界

Codex 风格训练最适合的场景，是“需求说明较明确、代码范围较局部、可以写出自动测试”的任务。比如工具函数、数据清洗函数、简单算法题、常规 API 包装层。

如果任务超出这个边界，单纯依赖 docstring 到函数的训练范式就不够了。

| 场景 | 是否适合 Codex 式 docstring 生成 | 更稳妥的做法 |
| --- | --- | --- |
| 中小函数补全 | 适合 | 直接生成并用测试筛选 |
| 标准算法题 | 适合 | 多采样 + 单元测试 |
| 跨文件重构 | 不太适合 | 增加仓库级上下文，分步生成 |
| 框架迁移 | 不太适合 | 检索文档 + 规则迁移 + 人工 review |
| 高安全代码 | 边界很窄 | 沙箱、审计、最小权限、人工批准 |

例如，要让模型生成一个跨多个文件的大功能，只给一句 docstring 往往不够。更合理的方法是把任务拆成几个层次：先生成接口定义，再生成模块实现，再生成测试，最后做集成验证。这里已经不是单纯的“docstring 到函数”，而是“规划 + 检索 + 生成 + 验证”的复合流程。

替代方案主要有三类。

第一，更大模型。更大模型通常有更强的上下文建模和更广的代码先验，但成本也更高，而且不自动解决私有代码库知识缺失问题。

第二，多轮交互。不是一次把函数写完，而是先让模型提炼需求、确认边界、生成草稿，再根据报错继续修正。这种方式更接近真实协作。

第三，检索增强或工具增强。把项目文档、API 定义、类型信息、测试模板先检索出来，再与 prompt 一起喂给模型。这样模型不是纯猜，而是在“看资料后生成”。

所以，Codex 的最佳适用边界不是“万能写代码”，而是“在局部、可验证、上下文清晰的任务上，大幅降低首版代码草拟成本”。
