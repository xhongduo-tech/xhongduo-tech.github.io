## 核心结论

构建文档摘要系统，不是“把一段文本丢给模型，拿回几句话”这么简单。工程级系统的正确抽象是：它是一条端到端流水线，输入是原始文档，输出是受约束的摘要结果，中间要同时满足质量、结构和资源三类目标。

这里的“端到端流水线”可以直白理解为：文档先被识别类型，再被切分、理解、压缩、重组，最后被评估和反馈修正。只有这条链条稳定，系统才可上线。

最常见的失败，不是模型完全不会总结，而是三个目标只顾了一个：

| 维度 | 目标值示例 | 实现手段 |
| --- | --- | --- |
| 质量（ROUGE/BERTScore） | 覆盖核心事实，和参考摘要有较高一致性 | 分段总结、事实约束、自动评测 |
| 结构一致性 | 输出固定为“关键点 + 影响”或“术语 + 解释” | 模板化解码、层级聚合 |
| 资源/延时 | 200 ms 到 2 s 内返回，成本可控 | 小模型路由、缓存、分层裁剪 |

玩具例子：团队要把一份 2000 字设备规范压缩成 5 条“关键特性 + 影响”短句，并要求 200 ms 内返回。这个任务表面是摘要，实际上同时包含三条约束：不能漏核心特性，格式必须统一，响应还必须足够快。只要漏掉其中任意一条，它都不是一个可用产品。

结论可以写得更硬一点：文档摘要系统的价值，不在“模型能不能写摘要”，而在“系统能不能稳定地产出符合业务格式、覆盖关键内容、且延时和成本可接受的摘要”。

## 问题定义与边界

文档摘要系统的定义是：在给定长度、格式、语言和业务约束下，抽取或生成文档核心语义的系统。这里“核心语义”可以直白理解为：读完原文后，用户最不该错过的信息。

这个定义必须带边界，否则讨论没有工程意义。至少要先固定四件事：

| 输入特征 | 控制变量 | 输出形式 |
| --- | --- | --- |
| 文档类型：技术文档、会议纪要、新闻、法规 | 延时预算：200 ms、1 s、5 s | 一段式摘要、要点列表、术语表 |
| 文档长度：500 字、5000 字、5 万字 | 成本预算：单次 API 成本、GPU 占用 | 摘要 + 解释、摘要 + 翻译 |
| 文档语言：中文、英文、多语言混合 | 可解释性：是否保留中间结果 | 面向新人或面向专家 |
| 文档结构：标题清晰或纯长文本 | 评估方式：自动指标或人工审核 | 固定格式或自由生成 |

为什么一定要先定边界？因为“好摘要”没有统一标准。新闻摘要强调事件和时间，技术文档摘要强调术语和依赖关系，会议纪要摘要强调责任人和决策。输入类型不同，摘要模板、评估指标、甚至模型选择都会变化。

新手可以先记住一个简单判断：摘要系统不是先“总结”，而是先“识别任务类型”。TecAce 的案例就很典型。它不是直接把三星开发者门户页面压成一段短文，而是先识别这是技术文档，再输出关键词、解释和翻译。这里的关键不是模型更聪明，而是系统先判断了边界，知道用户要的是“快速理解技术页面”，不是“文学性缩写”。

真实工程里，常见的误设边界包括：

1. 把所有文档都走同一个 prompt，结果新闻可以用，法规全错。
2. 没定义摘要长度，只写“尽量简洁”，导致输出忽长忽短。
3. 没定义目标读者，结果新手看不懂，专家嫌太浅。
4. 没定义延时预算，离线方案被误拿去做在线接口。

## 核心机制与推导

主流摘要模型的基本骨架仍然可以写成 `encoder -> attention -> decoder`。`encoder` 可以直白理解为“把原文编码成机器可处理表示”；`attention` 是“让模型决定当前该关注原文哪些部分”；`decoder` 是“把内部表示重新写成摘要”。

短文场景下，这套结构常常够用。长文场景下，它会碰到两个硬问题：

1. 输入太长，模型上下文窗口不够。
2. 就算能塞进去，跨段信息也容易断裂，导致摘要像一串没有连接的句子。

所以工程上通常会引入“层级分段”。“层级”可以直白理解为：先按小块理解，再按大块整合，而不是一口气读完整篇。

如果原文记为 $D$，分成 $m$ 个片段：

$$
D = [d_1, d_2, \dots, d_m]
$$

每个片段先得到局部摘要 $s_i$，再把这些局部摘要聚合成最终摘要 $S$。这比单次直接生成更稳，因为每一步都只处理有限上下文。

评价时，ROUGE-L 常被用来衡量结构覆盖。它基于 LCS，Longest Common Subsequence，白话就是“两个文本按原顺序能对上的最长公共片段”。

$$
ROUGE\text{-}L = \frac{LCS(summary, reference)}{|reference|}
$$

它的直观含义是：生成摘要在多大程度上保留了参考摘要的主要结构顺序。

玩具例子可以非常小。原文是：

- 设备 A 可移植
- 设备 A 耗电低
- 设备 A 延迟 20 ms

参考摘要是：“设备 A 延迟 20 ms”。

如果系统输出“设备 A 性能好”，它虽然听上去合理，但没有准确保留“20 ms”这个关键事实。也就是说，语义看似接近，业务价值却可能很低。这也是为什么只靠流畅度不够，必须加覆盖和事实约束。

CoTHSSum 提出的思路值得工程上借鉴：先做层级切分，再在每段上做 CoT 推理。CoT，Chain-of-Thought，白话就是“让模型先写中间思路，再给最终答案”。在摘要任务里，这不是为了展示推理过程给用户看，而是为了让系统内部先形成提纲、关键点和局部结论，减少跨段跳跃。

一个好理解的分层流程是：

1. 段落层：每段提炼关键事实。
2. 小节层：把多段事实整理成主题要点。
3. 全文层：按预定结构输出最终摘要。

这样做的本质，不是“多调用几次模型”，而是把大任务拆成一组更稳定的小任务。

## 代码实现

一个最小可运行版本，不需要上来就接大模型。先用规则分段、句长打分和模板聚合，就能把系统骨架跑起来。下面的代码实现了一个玩具版流水线：分句、提取高分句、按固定格式输出，并带 `assert` 校验。

```python
import re
from collections import Counter

def split_sentences(text: str):
    parts = re.split(r"[。！？!?]\s*", text.strip())
    return [p.strip() for p in parts if p.strip()]

def sentence_score(sentence: str, global_freq: Counter):
    words = re.findall(r"[\u4e00-\u9fffA-Za-z0-9]+", sentence)
    return sum(global_freq[w] for w in words)

def summarize(text: str, top_k: int = 2):
    sentences = split_sentences(text)
    words = []
    for s in sentences:
        words.extend(re.findall(r"[\u4e00-\u9fffA-Za-z0-9]+", s))
    freq = Counter(words)
    ranked = sorted(sentences, key=lambda s: sentence_score(s, freq), reverse=True)
    selected = ranked[:top_k]
    return [f"关键点{i+1}: {s}" for i, s in enumerate(selected)]

doc = (
    "设备A面向边缘部署，整机重量低于2kg。"
    "它在持续负载下平均延迟20ms。"
    "设备A支持远程升级，便于批量维护。"
    "文档要求上线后7天内完成首轮稳定性验证。"
)

result = summarize(doc, top_k=2)
assert len(result) == 2
assert any("20ms" in x for x in result)
print("\n".join(result))
```

这个版本很粗糙，但它说明了工程骨架先于模型复杂度。真正上线时，管线一般会演化成下面这样：

```python
segments = hierarchical_segment(doc)
partials = [cot_summary(seg) for seg in segments]
final = structure_guided_decode(partials, schema="关键特性+影响")
metrics = evaluate(final, reference=None, rubrics=["coverage", "coherence", "latency"])
```

这里每个函数都对应一个明确职责：

- `hierarchical_segment`：按标题、段落或 token 长度切分。
- `cot_summary`：先抽关键点，再压缩成片段摘要。
- `structure_guided_decode`：把片段摘要填进固定格式。
- `evaluate`：同时计算自动指标和业务规则。

真实工程例子可以参考“技术文档摘要 + 术语解释 + 翻译”的系统。比如读取开发者门户页面后，先识别为技术文档，再做三层处理：

1. 提炼主旨：这个页面解决什么问题。
2. 抽术语：识别 API、协议、参数名，并给简短解释。
3. 生成目标语言摘要：例如英文原文输出中文简述。

这样用户拿到的不是一段空泛摘要，而是一组可直接阅读和检索的结构化结果。

## 工程权衡与常见坑

摘要系统最常见的坑，不是“模型性能差”，而是“系统设计把问题简化错了”。

| Pitfall | Why it happens | Mitigation |
| --- | --- | --- |
| 信息跳跃 | 长文跨段依赖被打断，attention 对远距离信息不稳定 | 分层切分 + 中间提纲 |
| 漏关键事实 | 直接压缩时优先保留高频信息，低频但关键的数字被丢掉 | 事实槽位约束，优先保留数字、时间、实体 |
| 指标好看但不好用 | ROUGE 重词面重叠，忽略同义表达和业务需求 | 加 BERTScore、人评和任务规则 |
| 延时过高 | 每层都调用大模型，链路过长 | 小模型预筛选、缓存中间结果 |
| 输出不稳定 | prompt 没固定格式，模型自由发挥过大 | schema 解码、模板填充 |
| 用户不信任 | 没有证据链，不知道摘要来自哪 | 保留引用句、片段来源、可回溯链接 |

一个典型错误是只相信 attention。attention 的直白理解是“模型当前看哪里”，但它不等于“模型一定整体理解了文档结构”。所以只靠一次性注意力聚合，常见现象就是：句子都通顺，但逻辑连接很弱。

例如原文前半段讲“设备 A 低功耗”，后半段讲“电池设计限制了峰值性能”。若系统直接输出“设备 A 性能强且功耗低”，这就是典型的跨段事实融合错误。它不是胡言乱语，而是把互相制约的两段信息错误合并了。

另一个坑是过度依赖 ROUGE。ROUGE 仍然重要，因为它便宜、可自动化、可批量跑，但它不能代表全部质量。尤其是技术文档里，同义表达很多，真正关键的往往是术语解释、依赖关系和数字约束，这些并不总能通过 n-gram 重叠被准确衡量。

工程上更稳的做法是三层评估一起上：

1. 自动指标：ROUGE、BERTScore。
2. 规则校验：数字、实体、术语是否保留。
3. 人工抽检：是否可读、是否满足场景模板。

## 替代方案与适用边界

不是所有摘要任务都需要层级 CoT。系统设计的关键不是“最先进”，而是“约束匹配”。

| 场景 | 适合方案 | 关键理由 |
| --- | --- | --- |
| 短新闻、短邮件 | 轻量模型 + prompt | 输入短，结构简单，延时优先 |
| 会议纪要 | 说话人切分 + 要点抽取 | 重点是责任人和决策，不是长文推理 |
| 多语言技术文档 | 多模型路由 + 术语表 + 翻译 | 同时要术语解释和跨语言输出 |
| 法规、科研、长报告 | 层级分段 + CoT + 结构化解码 | 长依赖强，漏一点就可能失真 |
| 企业内部定制摘要 | Prompt 管理 + 用户反馈闭环 | 场景固定，长期可迭代优化 |

替代路径主要有三类。

第一类是抽取式方案。抽取式，白话就是“直接挑原文句子，不自己重写”。它实现简单、可解释性强、事实风险低，适合保守场景，但可读性通常一般。

第二类是端到端生成式方案。生成式，白话就是“模型自己重新组织并写出摘要”。它可读性好，但最容易出现事实偏移，适合短文或对可解释性要求不高的场景。

第三类是混合式方案：先抽取候选事实，再生成结构化摘要。这通常是工程上最稳的一类，因为它在真实性和可读性之间更容易取得平衡。

如果预算有限，一个现实策略是：

1. 短文走小模型或规则模板。
2. 长文先分段，再只对高价值片段调用强模型。
3. 用 like/dislike 和人工修改记录持续更新 prompt。

这也是 TecAce 案例值得借鉴的地方：不是假设存在一个万能 prompt，而是把“文档类型识别 + prompt 管理 + 用户反馈”做成闭环。这样系统不是一次性优化，而是随业务慢慢收敛。
