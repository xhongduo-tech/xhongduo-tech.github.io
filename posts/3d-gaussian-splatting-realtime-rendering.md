## 核心结论

3D Gaussian Splatting，简称 3DGS，本质上是把场景表示成一组可优化的 3D 高斯原语，再把这些原语直接光栅化到屏幕上。高斯原语可以先理解成“带体积、带方向、带透明度的模糊小椭球”，它不是单个点，也不是网格三角形。

它比经典 NeRF 更适合“高质量且可交互”的新视角合成。原因不在于网络更大，而在于计算路径变了：NeRF 主要靠“沿射线采样并反复查询 MLP”，3DGS 主要靠“显式图元投影并做 alpha 合成”。前者更像连续函数求值，后者更像一套可微分的实时渲染管线。

官方项目页给出的核心卖点很明确：在完整场景和 1080p 分辨率下，3DGS 可以做到高质量实时新视角合成；论文与项目页强调的关键三件事分别是 3D 高斯表示、交替优化与密度控制、以及可见性感知的快速渲染。

| 方案 | 场景表示 | 渲染方式 | 速度特征 |
|---|---|---|---|
| NeRF | 隐式 MLP | 射线采样 + 网络查询 | 慢，通常偏离线 |
| Instant-NGP | 隐式哈希编码 + 小网络 | 射线采样 + 加速查询 | 比 NeRF 快，但仍是体渲染 |
| 3DGS | 显式高斯集合 | 光栅化 + alpha 合成 | 快，适合实时预览 |

如果目标是“从一组照片恢复可漫游场景，并在 viewer 里实时看结果”，3DGS 是目前最有工程落地价值的一类表示方法之一。

## 问题定义与边界

它解决的问题可以写成一句话：给定多张已知相机位姿的图像，学习一个场景表示，使系统能够合成任意新视角图像。

这里有三个边界需要先说清楚。

第一，它主要面向静态或弱动态场景。静态场景就是拍摄期间结构基本不变，例如房间、展厅、街景立面。人群快速走动、树叶剧烈摆动、布料形变明显，都不是它最稳的输入。

第二，它追求的是“视觉重建”，不是“精确 CAD 几何”。视觉重建的意思是从图像看起来对，而不是每个表面法向、孔洞厚度、刚体边界都精确到工程测量级。

第三，它依赖相机位姿。相机位姿就是每张图是从哪里、朝哪个方向拍的。位姿估计错，后面所有高斯都会在错误几何上优化，结果常见为重影、漂浮、拉花。

| 输入 | 目标 | 输出 |
|---|---|---|
| 多视角图像 + 相机位姿 | 学习可渲染场景表示 | 任意新视角图像 |

玩具例子：你拿手机绕一张桌子拍 80 张照片，系统知道每张照片相机在哪里，就要学出一个“桌子场景”，以后你从任意角度看，系统都能重新渲染这张桌子。

真实工程例子：园区级数字孪生里，采集团队先用视频或相机环拍建筑与走廊，再用 COLMAP 做 SfM。SfM 是 Structure from Motion，白话就是“从多张图反推出相机位置和稀疏三维结构”。接着 3DGS 训练高斯场景，最后在实时 viewer 中做漫游、导览或 VR 预览。

| 适合 | 不适合 |
|---|---|
| 室内重建、展厅漫游、建筑可视化、VR 预览 | 高速动态场景、强拓扑变化、精细 CAD 编辑、严格几何测量 |

## 核心机制与推导

第 $i$ 个高斯通常包含位置 $\mu_i$、协方差 $\Sigma_i$、不透明度 $o_i$ 和颜色参数。协方差可以理解成“这个高斯在三维空间里长什么形状、朝什么方向扩散”。

统一记号如下：

| 符号 | 含义 |
|---|---|
| $\mu_i$ | 第 $i$ 个高斯中心 |
| $\Sigma_i$ | 3D 协方差矩阵 |
| $o_i$ | 不透明度 |
| $b_{i,lm}$ | 球谐系数 |
| $\omega$ | 视线方向 |
| $L$ | 球谐阶数 |

单个 3D 高斯可写成：

$$
G_i(x)=\exp\left(-\frac{1}{2}(x-\mu_i)^T\Sigma_i^{-1}(x-\mu_i)\right)
$$

为保证 $\Sigma_i$ 正定，工程上通常不直接优化满矩阵，而是写成：

$$
\Sigma_i = R_i \, \mathrm{diag}(s_i^2)\, R_i^T
$$

其中 $s_i$ 是三个主轴尺度，$R_i$ 是旋转。这样做的好处是参数更稳定，也更容易约束。

颜色不是固定 RGB，而常常用球谐函数表示视角相关颜色。球谐可以先理解成“用一组方向基函数近似不同视角下的颜色变化”。

$$
c_i(\omega)=\sum_{l=0}^{L}\sum_{m=-l}^{l} b_{i,lm}Y_{lm}(\omega)
$$

渲染时，每个 3D 高斯会投影到图像平面，变成一个 2D 椭圆斑点。常见近似写成：

$$
\Sigma_i^{2D}=J_i W_i \Sigma_i W_i^T J_i^T
$$

其中 $W_i$ 是世界到相机变换，$J_i$ 是投影雅可比。直觉上，它表示“原来三维里的椭球，被当前相机看过去后，在屏幕上变成什么椭圆”。

对某个像素 $p$，第 $i$ 个高斯的贡献近似为：

$$
\alpha_i(p)=o_i \cdot G_i^{2D}(p)
$$

多个高斯按深度排序后做前向 alpha 合成：

$$
C(p)=\sum_i T_i \alpha_i(p)c_i(\omega), \quad T_i=\prod_{j<i}(1-\alpha_j(p))
$$

这里 $T_i$ 是透过率，白话就是“前面那些高斯还剩多少光没挡住”。

玩具例子最容易理解这一层。假设某像素上只有两个已投影高斯：

- 前景：$\alpha_1=0.4$，颜色 $c_1=(1,0,0)$
- 背景：$\alpha_2=0.5$，颜色 $c_2=(0,1,0)$

则

$$
C = \alpha_1 c_1 + (1-\alpha_1)\alpha_2 c_2
= 0.4(1,0,0) + 0.6 \times 0.5 (0,1,0)
= (0.4, 0.3, 0)
$$

这就是“前面的先贡献，后面的乘上剩余透过率”。

3DGS 快，核心也在这里：它避免了在空空间里做大量无效采样，而是只处理那些真正有贡献的显式高斯。

## 代码实现

官方实现的训练主线可以概括为：

```text
多视角图像
-> COLMAP 估计位姿与稀疏点
-> 初始化高斯集合
-> 用 GaussianRasterizer 渲染当前视图
-> 计算重建损失
-> 反向传播更新 means / scales / rotations / opacities / SH
-> densify / prune
```

其中 `densify` 是增密，白话就是“在误差高或梯度大的地方复制或拆分高斯”；`prune` 是裁剪，意思是删掉贡献很低、透明度很低或明显噪声化的高斯。

关键参数可以这样理解：

| 参数 | 作用 | 常见风险 |
|---|---|---|
| `means3D` | 高斯中心位置 | 位姿差时容易漂 |
| `scales` | 主轴尺度 | 过大变糊，过小变碎 |
| `rotations` | 椭球朝向 | 参数不稳会拉花 |
| `opacities` | 遮挡强度 | 过高会堆积脏雾 |
| `shs` | 视角相关颜色 | 阶数高会增内存和噪声 |

下面这个 Python 例子不是完整 3DGS，而是把“按深度排序的 alpha 合成”单独抽出来，便于验证核心渲染逻辑：

```python
from typing import List, Tuple

Color = Tuple[float, float, float]

def composite(front_to_back: List[Tuple[float, Color]]) -> Color:
    transmittance = 1.0
    out = [0.0, 0.0, 0.0]
    for alpha, color in front_to_back:
        weight = transmittance * alpha
        out[0] += weight * color[0]
        out[1] += weight * color[1]
        out[2] += weight * color[2]
        transmittance *= (1.0 - alpha)
    return tuple(round(v, 6) for v in out)

# 玩具例子：前红后绿
c = composite([
    (0.4, (1.0, 0.0, 0.0)),
    (0.5, (0.0, 1.0, 0.0)),
])
assert c == (0.4, 0.3, 0.0)

# 不透明前景应完全遮挡后景
c2 = composite([
    (1.0, (0.2, 0.4, 0.6)),
    (0.9, (1.0, 0.0, 0.0)),
])
assert c2 == (0.2, 0.4, 0.6)

print("ok", c, c2)
```

真实工程里，像素级重活不是 Python 做，而是 CUDA 栅格化器做。官方仓库把可微渲染核心单独拆成 `diff-gaussian-rasterization`，这说明它的主计算瓶颈已经不是“网络推理”，而是“高斯投影、排序、覆盖率与合成”。

## 工程权衡与常见坑

3DGS 最大的工程现实是：效果往往先输在前端，而不是输在优化器。

第一坑是 COLMAP 位姿不稳。症状通常是同一物体边缘出现双层轮廓、玻璃附近出现漂浮色块、地面像被抹开。修法不是先调学习率，而是回去检查拍摄覆盖、运动模糊、重复纹理和相机内参。

第二坑是 densify 太激进。高斯越多不等于越好。过度增密会导致显存暴涨、训练变慢、局部噪声堆积。很多新手看到细节不够，就直接让系统“长更多点”，结果长出来的是雾状伪影。

第三坑是 opacity 管不好。opacity 是不透明度，决定一个高斯挡住后景的能力。如果只增密不裁剪，低价值高斯会像浮尘一样残留，画面看起来脏。

第四坑是 SH 阶数和分辨率同时拉高。球谐阶数高，表达能力更强，但内存、计算和噪声风险也上升。官方实现常见默认是 `sh_degree=3`，这在大多数真实场景上已经是合理起点。

| 问题 | 现象 | 规避方式 |
|---|---|---|
| SfM/位姿不稳 | 重影、漂浮、几何扭曲 | 先修数据与位姿，再谈训练 |
| densify 过强 | 显存爆、噪声堆积 | 控制阈值和数量上限 |
| prune 不足 | 浮尘残留、边缘脏 | 定期裁剪低贡献高斯 |
| SH 阶数过高 | 内存增、颜色噪声 | 先从 `3` 起步 |
| 输入分辨率过高 | 训练慢、调参成本高 | 先在 1K 到 1.6K 级别验证 |

一个实用调参顺序是：先看位姿，再看 densify，再看 SH 和分辨率，最后再处理 opacity reset 与 prune 策略。这个顺序的原因很简单，前面几项通常决定了 80% 的上限。

## 替代方案与适用边界

3DGS 不是替代一切，而是在一组具体目标上更优。

和 NeRF 相比，3DGS 的优势是渲染快、交互性强、工程体验好。NeRF 的优势是表示连续体积场更直观，很多后续工作在几何、动态、可编辑性和物理建模上更容易接理论链条。

和 Instant-NGP 相比，3DGS 也更偏“显式图元渲染”，而 Instant-NGP 仍然属于“高效隐式场查询”。因此两者虽然都追求速度，但计算组织方式不同。

| 方法 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| NeRF | 连续表示直观，研究扩展丰富 | 训练和渲染慢 | 离线重建、算法研究 |
| Instant-NGP | 训练明显加速 | 仍需体渲染采样 | 快速隐式表示实验 |
| 3DGS | 实时性强，viewer 友好 | 对位姿和动态更敏感 | 实景漫游、VR/AR、数字孪生 |

一句话总结边界：3DGS 的优势在“快速、高质量、可交互”，而不是“最通用、最精确、最可编辑”。

如果任务是展厅预览、室内漫游、建筑展示，3DGS 通常是优先候选。如果任务是复杂动态人体、强时序编辑、可控重拓扑生成，就需要看动态 NeRF、4D 表示、mesh-based 方法或后续层级高斯变体。
