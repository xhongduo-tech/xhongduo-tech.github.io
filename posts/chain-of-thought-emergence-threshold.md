## 核心结论

思维链，英文是 Chain-of-Thought，简称 CoT，可以理解为“把中间推理步骤显式写出来”。它降低多步推理任务的涌现临界规模，核心原因不是“模型突然更聪明了”，而是“原本一次性求解的大问题，被拆成了若干个局部可解的小问题”。

对零基础读者，可以先抓住一句话：**CoT 不是凭空增加参数，而是在推理时把同一份参数用得更深、更有顺序。**

Wei 等人在 2022 年展示了一个关键现象：对于 GSM8K 这类小学数学文字题，带有 few-shot 思维链示例的 540B PaLM 能显著提升表现。后续 Tay 等人在 U-PaLM 上进一步展示：改进训练后的模型在更小规模上也能在若干需要推理的任务中出现原本更大模型才明显出现的能力，论文中给出的代表性数字是 **62B 与 540B** 的对比。这类结果支持一个工程上很有用的判断：**把问题拆成链条，能把“必须靠更大参数才能触发”的能力，部分转化为“靠更好的推理过程就能触发”的能力。**

可以把它写成一个简化关系：

$$
P_{\text{crit}} \approx P_{\text{step}} \cdot N
$$

这里：

- $P_{\text{crit}}$ 表示任务整体的临界参数规模，也就是模型要“突然变得会做”大致需要多大。
- $P_{\text{step}}$ 表示单个子步骤所需的参数规模。
- $N$ 表示推理链被拆成多少步。

这不是严格定律，而是帮助理解的近似模型。它表达的意思是：**整体难题的门槛，往往来自多个子步骤叠加；如果每一步都被提示得足够清楚，那么每一步的局部门槛就会低很多。**

玩具例子很简单：

题目：小明有 17 个苹果，送出 5 个，又买回 8 个，现在有多少个？

直接问模型：“现在有多少个？”  
模型可能跳算，甚至把“送出”和“买回”混在一起。

如果改成三步：

1. 列出初始数量  
2. 算送出后的数量  
3. 算买回后的最终数量

模型更容易稳定输出：
$17-5=12$，$12+8=20$，所以答案是 20。

这就是 CoT 的最小直觉版本：**不是让模型凭感觉撞答案，而是让它沿着局部正确路径前进。**

## 问题定义与边界

“涌现能力”，英文是 emergent ability，可以直白理解为：**模型规模没到某个点之前几乎不会，到了某个点之后表现突然上来。**

本文讨论的不是所有任务，而是有明确中间步骤的多步推理任务，例如：

| 任务类型 | 直接求解特征 | CoT 后变化 |
| --- | --- | --- |
| 算术文字题 | 需要连续变换变量 | 拆成逐步计算 |
| 常识多跳问答 | 需要串联多个事实 | 拆成逐条判断 |
| 符号推理 | 需要保持状态一致 | 拆成状态更新序列 |

边界也要讲清楚。

第一，CoT 主要帮助“需要过程”的任务，不是所有任务都该上 CoT。  
如果题目只是“法国首都是哪里”，直接回答通常更快、更便宜。

第二，CoT 讨论的是**推理时的组织方式**，不是训练时参数真的减少了。  
所以“临界规模下降”更准确的说法是：**在给定任务和给定提示方式下，达到可用精度所需的模型规模下降了。**

第三，62B 对 540B 不是普适公式。  
更准确地说，Tay 等人的 U-PaLM 结果说明：在改进训练和 CoT 共同作用下，若干任务的能力出现点向更小模型移动；这是一种“阈值左移”的现象，而不是“任何 62B 都等于任何 540B”。

一个真实工程例子是客服质检。  
任务不是简单分类，而是判断“用户是否先表达不满，再提出退款要求，最后客服是否给出有效方案”。这类判断本质上是三步：

1. 抽取用户情绪
2. 抽取诉求
3. 判断客服响应是否匹配

如果直接问“这段对话是否处理得当”，小模型容易漏掉中间条件。  
如果按三步链条做，它往往不需要更大参数就能达到更稳定的结果。

## 核心机制与推导

从机制上看，CoT 起作用主要有三层原因。

第一层是**问题分解**。  
问题分解就是把一个复杂映射拆成多个简单映射。原本模型要学的是：

$$
x \rightarrow y
$$

CoT 后变成：

$$
x \rightarrow z_1 \rightarrow z_2 \rightarrow \cdots \rightarrow z_N \rightarrow y
$$

这里的 $z_i$ 是中间状态，比如“已知量”“中间计算结果”“候选解释”。

第二层是**有效计算深度放大**。  
有效计算深度可以白话理解为：模型在一次回答里，实际执行了多少层有意义的推理展开。可写成：

$$
D_{\text{eff}} = N \cdot d_{\text{step}}
$$

其中：

- $D_{\text{eff}}$：整体有效深度
- $N$：链条步数
- $d_{\text{step}}$：每一步内部完成的局部推理深度

直觉上，链条越长，模型越像在“分阶段求解”，而不是“一步猜中”。

第三层是**误差局部化**。  
直接回答时，所有难点压在同一次输出里，任何一步没想清楚都会污染最终答案。  
CoT 把错误限制在某个局部步骤，于是模型更容易在后续步骤中维持结构化状态。

可以用一个简化推导来理解。设直接解题的成功率为 $p_{\text{direct}}$，拆成 $N$ 步后，每一步成功率为 $p_{\text{step}}$。如果步骤之间有提示约束和可见中间结果，整体成功率不再是简单的盲乘，而更接近：

$$
p_{\text{cot}} \approx g(N, p_{\text{step}}, c)
$$

这里 $c$ 表示步骤之间的约束强度，例如“先列已知，再计算，再总结”。  
当 $c$ 足够强时，CoT 会提升路径稳定性，使得局部正确更容易传到全局正确。

Jin 等 2024 的结果进一步说明，**只增加推理步骤长度，即使不增加新信息，也可能显著提升性能**。这说明链条长度不只是“解释更详细”，而是在改变模型的求解轨迹。工程上可以把 $N$ 看成一种“预算放大因子”：

$$
\text{Budget} \uparrow \Rightarrow N \uparrow \Rightarrow D_{\text{eff}} \uparrow \Rightarrow \text{Accuracy 常常上升}
$$

但这个上升不是无限的。超过某个长度后，收益会递减，甚至因为错误累积而下降。

用论文中的经验事实可以概括成下面这张表：

| 观察 | 含义 | 对临界规模的影响 |
| --- | --- | --- |
| 大模型加 CoT 明显强于直接回答 | 多步任务受益最大 | 同规模下精度提高 |
| 更长的 reasoning step 常带来更高精度 | 链长本身是有效变量 | 阈值向小模型左移 |
| U-PaLM 在更小规模出现更强推理表现 | 训练改进与 CoT 可叠加 | 所需参数继续下降 |

## 代码实现

工程里最常见的实现方式，不是让模型“自由发挥思考”，而是把链条结构写进 prompt。

一个最小模板：

```text
请按以下步骤回答：

Step 1: 列出已知条件
Step 2: 写出需要计算的中间量
Step 3: 逐步计算
Step 4: 给出最终答案

问题：
{question}
```

玩具例子对应的完整提示可以写成：

```text
问题：小明有 17 个苹果，送出 5 个，又买回 8 个，现在有多少个？

请按步骤回答：
Step 1: 初始数量是多少？
Step 2: 送出后还剩多少？
Step 3: 买回后总数是多少？
Step 4: 用一句话总结答案。
```

下面给一个可运行的 Python 小程序，用最简单的方式演示“把任务拆步后再执行”。这里不是在调用大模型，而是在模拟 CoT 的调度思想。

```python
import re

def solve_apple_problem(text: str) -> int:
    nums = list(map(int, re.findall(r"\d+", text)))
    assert len(nums) == 3, "题目应包含三个数字"
    initial, give_away, buy_back = nums

    # Step 1: 列出已知
    assert initial >= 0 and give_away >= 0 and buy_back >= 0

    # Step 2: 计算送出后
    remain = initial - give_away

    # Step 3: 计算买回后
    final = remain + buy_back

    # Step 4: 返回答案
    return final

q = "小明有17个苹果，送出5个，又买回8个，现在有多少个？"
answer = solve_apple_problem(q)

assert answer == 20
print(answer)
```

真实工程中，调用流程通常是这样：

```python
def build_cot_prompt(question: str) -> str:
    return f"""
你是一个严谨的解题助手。按固定格式输出：

Step 1: 列出已知
Step 2: 分解为子问题
Step 3: 逐步求解
Step 4: 最终答案

问题：
{question}
""".strip()

def parse_final_answer(text: str) -> str:
    lines = [line.strip() for line in text.splitlines() if line.strip()]
    assert len(lines) > 0
    return lines[-1]

prompt = build_cot_prompt("仓库原有120件货，发出35件，补货18件，现有多少件？")
assert "Step 1" in prompt
```

工程上的关键不是模板多华丽，而是两点：

1. 步骤名要稳定，便于日志分析和自动校验。  
2. 每一步最好都能被程序检查，比如是否列出已知、是否出现中间变量、最终答案是否落在合法范围内。

## 工程权衡与常见坑

CoT 的代价很直接：**token 更多，延迟更高，费用更贵。**

一个典型权衡表如下：

| 链长 | token 增量 | 时间增长 | 常见错误率变化 |
| --- | --- | --- | --- |
| 0 步，直接答 | 基线 | 1.0x | 对多步题偏高 |
| 3 步短链 | +20% 到 +60% | 1.2x 到 1.6x | 通常明显下降 |
| 5 到 8 步中链 | +60% 到 +150% | 1.6x 到 2.5x | 常是性价比最优区 |
| 超长链 | +150% 以上 | 2.5x 以上 | 可能因累积误差反弹 |

比如实时问答系统里，提示从 400 token 增到 800 token，端到端延迟接近翻倍并不奇怪。  
所以实际系统常做的是“按题目难度动态决定链长”，而不是所有请求统一上长链。

常见坑主要有四个。

第一，**链条过长但没有新增约束**。  
这会变成无意义复述，只增加成本。

第二，**中间步骤写得像自然语言散文**。  
越散，越难校验。对工程系统，更好的形式是半结构化步骤。

第三，**把错误中间步骤直接当真**。  
一旦 Step 2 算错，后面全错。所以要加中间校验，比如数值范围检查、格式检查、单位检查。

第四，**把 CoT 当成万能增益**。  
如果任务本身是一跳事实查询，CoT 只会拖慢系统。

## 替代方案与适用边界

如果你需要的是更低延迟，而不是最高推理上限，CoT 不是唯一选择。

| 方案 | 适用场景 | 不适用场景 |
| --- | --- | --- |
| 直接 prompt | 单步事实问答、低延迟服务 | 多步推理、状态跟踪 |
| 短 CoT | 中等复杂度任务 | 极低延迟场景 |
| Chain-of-Verification | 先答再校验 | 无法容忍二次调用成本 |
| Self-Consistency | 高价值推理任务 | 预算受限场景 |
| Ensemble Sampling | 需要更高稳健性 | 在线实时系统 |

常识问答就是一个反例。  
问题：“水在标准大气压下多少摄氏度沸腾？”  
直接回答“100 摄氏度”通常就够了。  
如果强行加 3 步 CoT，收益接近零，却多花 token。

所以适用边界可以概括为：

- 适合：数学题、多跳检索、规则判断、状态演化、代码调试。
- 不太适合：短事实问答、固定模板抽取、超低延迟接口。

工程上更稳妥的策略通常不是“永远使用 CoT”，而是：

1. 先判断任务是否多步。  
2. 再决定链长 $N$。  
3. 最后给关键步骤加校验或投票。
