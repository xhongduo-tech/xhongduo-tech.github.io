## 核心结论

PLE（Progressive Layered Extraction）是一个多层 `CGC` 式多任务学习模型：它把专家网络拆成“任务独享专家”和“共享专家”，再用门控网络逐层选择和融合专家输出，用来减少任务之间的负迁移。

负迁移是指：多个任务一起训练时，一个任务学到的表示反而损害另一个任务的效果。推荐系统里很常见，例如点击率、观看时长、完播率都和用户兴趣有关，但它们并不等价。

新手可以把 PLE 理解成“先分组复习，再统一讨论，再分组补课”。每个任务先保留自己的特定知识，再从共享分支吸收通用知识，下一层继续做同样的提取。比如视频推荐里，点击和观看时长相关，但不完全一样：点击更偏“愿不愿意点开”，观看时长更偏“内容能不能持续吸引”。PLE 不会让两个目标完全共用一套表示，而是分别保留各自偏好，再共享一部分通用特征。

| 模型 | 核心结构 | 共享方式 | 抗负迁移能力 |
|---|---|---|---|
| SharedBottom | 底层全共享，上层任务独立 | 所有任务共用同一个底座 | 弱 |
| MMoE | 多个共享专家 + 每任务 gate | 所有专家都共享，任务自己选择 | 中 |
| PLE | 私有专家 + 共享专家 + 多层 gate | 共享与私有分层共存 | 强 |

PLE 适合以下场景：

| 场景 | 是否适合 | 说明 |
|---|---|---|
| 点击、时长、完播联合建模 | 适合 | 目标相关但优化方向不同 |
| 广告 CTR 与 CVR 联合建模 | 适合 | 点击和转化存在因果链，但样本分布不同 |
| 多个几乎相同的分类任务 | 一般不必 | 简单共享模型可能已经足够 |
| 完全无关的任务一起训练 | 不适合 | 共享表示会引入干扰 |

## 问题定义与边界

多任务学习（Multi-Task Learning，MTL）是指一个模型同时学习多个预测目标，并希望这些任务通过共享数据和表示互相帮助。它的目标不是“任务越多越好”，而是让相关任务共享有效信息，同时避免互相干扰。

在视频推荐中，`click`、`watch time`、`completion`、`share` 都是常见任务。新手版解释如下：点击更像“是否愿意点开”，时长更像“内容是否持续吸引”，完播更像“是否看完”，分享更像“是否值得传播”。这些目标有交集，但不能混成一个单目标。一个吸引眼球的视频可能点击率高，但观看时长低；一个教学视频可能点击率一般，但完播和收藏很好。

多任务学习的难点主要来自两类冲突：

| 冲突类型 | 白话解释 | 推荐系统例子 |
|---|---|---|
| 梯度冲突 | 不同任务更新参数的方向相反 | 点击希望模型偏向标题党，时长希望模型惩罚标题党 |
| 表示冲突 | 不同任务需要的特征抽象不同 | 点击依赖封面和标题，完播更依赖内容质量和视频长度 |

用数学语言说，若两个任务的损失分别是 $L_1$ 和 $L_2$，共享参数为 $\theta$，当梯度内积满足：

$$
\nabla_\theta L_1 \cdot \nabla_\theta L_2 < 0
$$

就说明两个任务在当前参数上存在更新方向冲突。它不一定每一步都发生，但在大规模推荐模型中很常见。

PLE 的边界也要明确。它不是所有多任务问题的默认答案，只适合“任务有关联，但不完全同质”的问题。

| 任务关系 | 是否适合 PLE | 原因 |
|---|---|---|
| 强相关但不一致 | 适合 | 需要共享与私有并存 |
| 几乎同质 | 一般不必 | 简单共享模型可能足够 |
| 几乎无关 | 不适合 | 共享带来干扰 |

真实工程例子是腾讯视频推荐的多目标排序。论文中使用真实腾讯视频推荐数据验证 PLE，目标包括用户观看行为相关指标。在线评估里，相比当时的强多任务学习基线，`view-count` 和 `watch time` 都有提升。这类场景里，各目标都服务于推荐排序，但它们对样本、特征和优化方向的要求不同，正好符合 PLE 的适用边界。

## 核心机制与推导

PLE 的核心组件叫 `CGC`（Customized Gate Control），可以理解为“定制化门控控制”：每个任务有自己的私有专家，也能选择共享专家；共享分支能看更广的专家集合，用来提炼通用表示。

先定义符号。设任务数为 $T$，层数为 $l$。第 $l$ 层中，第 $t$ 个任务的输入表示是 $h_t^{(l)}$，共享分支输入是 $h_s^{(l)}$。专家网络是一个小型神经网络，作用是把输入特征变成某种中间表示。

任务 $t$ 的第 $k$ 个私有专家输出为：

$$
z_{t,k}^{(l)} = E_{t,k}^{(l)}(h_t^{(l)})
$$

共享分支的第 $m$ 个共享专家输出为：

$$
u_m^{(l)} = E_{s,m}^{(l)}(h_s^{(l)})
$$

门控网络（gate）是一个输出权重的网络，它决定当前任务更相信哪些专家。softmax 是一种把任意实数向量变成概率分布的函数，输出值非负且总和为 1：

$$
a_{t}^{(l)} = \mathrm{softmax}(G_t^{(l)}(h_t^{(l)}))
$$

如果任务 $t$ 可以选择自己的私有专家和共享专家，则可选专家输出集合记为 $x_i^{(l)}$，任务融合结果为：

$$
h_t^{(l+1)} = \sum_i a_{t,i}^{(l)} x_i^{(l)}
$$

共享分支也有自己的 gate，但它通常能看到更广的专家集合，包括多个任务的私有专家和共享专家。最后一层只保留各任务输出，不再继续向上输出共享分支，因为高层表示应该更偏任务专属决策。

层间输入/输出可以写成：

```text
第 l 层输入:
  h_1^(l), h_2^(l), ..., h_T^(l), h_s^(l)

专家计算:
  任务私有专家: E_1, E_2, ..., E_T
  共享专家:     E_s

门控融合:
  任务 gate 1 -> h_1^(l+1)
  任务 gate 2 -> h_2^(l+1)
  ...
  共享 gate   -> h_s^(l+1)

最后一层:
  只输出 h_1, h_2, ..., h_T，再接各任务 tower
```

玩具例子：2 个任务，1 个私有专家，1 个共享专家。任务 1 私有输出为 `2`，共享输出为 `6`，门控权重是 `[0.75, 0.25]`，则：

$$
0.75 \times 2 + 0.25 \times 6 = 3.0
$$

任务 2 私有输出为 `4`，共享输出为 `6`，门控权重是 `[0.2, 0.8]`，则：

$$
0.2 \times 4 + 0.8 \times 6 = 5.6
$$

共享分支如果对 `[2, 4, 6]` 给出权重 `[0.1, 0.2, 0.7]`，则：

$$
0.1 \times 2 + 0.2 \times 4 + 0.7 \times 6 = 5.2
$$

这说明同一组专家输入，会被不同任务以不同方式组合。所谓“递进式”，就是上一层的融合输出会作为下一层输入，模型逐层筛掉干扰、保留有用信息。

## 代码实现

代码实现的关键不是把公式堆出来，而是把“专家层、门控层、任务分支、共享分支”拆成清晰模块，保证 `forward` 的数据流可读、可调试、可扩展。

模块拆分图如下：

```text
Input
  -> Shared Embedding / Backbone
  -> PLELayer 1
       -> private experts
       -> shared experts
       -> task gates
       -> shared gate
  -> PLELayer 2
       -> private experts
       -> shared experts
       -> task gates
  -> TaskTower(click)
  -> TaskTower(watch_time)
  -> LossAggregator
```

新手版伪代码：

```text
1. 输入特征先过共享底座
2. 每一层分别计算任务私有专家和共享专家输出
3. 每个任务用自己的 gate 给“私有专家 + 共享专家”加权
4. 非最后一层中，共享 gate 对更广的专家集合加权
5. 下一层继续迭代
6. 最后输出各任务预测头
```

张量形状表：

| 模块 | 输入形状 | 输出形状 | 说明 |
|---|---|---|---|
| 任务私有专家 | `[B, D]` | `[B, H]` | 每任务独享 |
| 共享专家 | `[B, D]` | `[B, H]` | 所有任务可见 |
| 任务门控 | `[B, D]` | `[B, E]` | `E` 为可选专家数 |
| 任务融合结果 | `[B, H]` | `[B, H]` | 加权和 |
| 最终预测头 | `[B, H]` | `[B, 1]` 或 `[B, C]` | 各任务独立输出 |

下面是一个可运行的 Python 玩具实现，用纯列表模拟专家加权，重点验证 PLE 的门控融合逻辑：

```python
import math

def softmax(xs):
    m = max(xs)
    exps = [math.exp(x - m) for x in xs]
    s = sum(exps)
    return [x / s for x in exps]

def weighted_sum(values, weights):
    assert len(values) == len(weights)
    return sum(v * w for v, w in zip(values, weights))

def ple_toy_layer():
    task1_private = 2.0
    task2_private = 4.0
    shared = 6.0

    task1_gate = [0.75, 0.25]
    task2_gate = [0.2, 0.8]
    shared_gate = [0.1, 0.2, 0.7]

    task1_out = weighted_sum([task1_private, shared], task1_gate)
    task2_out = weighted_sum([task2_private, shared], task2_gate)
    shared_out = weighted_sum([task1_private, task2_private, shared], shared_gate)

    return task1_out, task2_out, shared_out

t1, t2, s = ple_toy_layer()

assert abs(t1 - 3.0) < 1e-9
assert abs(t2 - 5.6) < 1e-9
assert abs(s - 5.2) < 1e-9

probs = softmax([1.0, 2.0, 3.0])
assert abs(sum(probs) - 1.0) < 1e-9
assert probs[2] > probs[1] > probs[0]

print(t1, t2, s)
```

如果落到 PyTorch 风格，通常可以拆成五类模块：

| 模块 | 责任 |
|---|---|
| `ExpertLayer` | 定义一个 MLP 专家，把 `[B, D]` 变成 `[B, H]` |
| `GateLayer` | 输出 softmax 权重，决定专家组合比例 |
| `PLELayer` | 管理私有专家、共享专家、任务 gate、共享 gate |
| `TaskTower` | 每个任务自己的预测头 |
| `LossAggregator` | 汇总多任务 loss，并处理权重 |

PaddleRec 中可以直接参考多任务 `ple` 模型配置；DeepCTR-Torch 中也有 `PLE` 类实现，适合对照类结构理解工程代码。实际工程中建议先只实现 1 层 PLE、每任务 1 个私有专家、1 个共享专家，指标稳定后再增加专家数和层数。

## 工程权衡与常见坑

PLE 的收益通常来自更强的表示隔离，但代价是参数量、延迟和调参复杂度上升。因此不能默认“层数越深越好、专家越多越好”。

| 配置 | 参数量 | 延迟 | 可能收益 | 建议 |
|---|---|---|---|---|
| 1 层，少量专家 | 低 | 低 | 中 | 适合作为第一版 |
| 多层，少量专家 | 中 | 中 | 中高 | 适合任务冲突明显时尝试 |
| 多层，大量专家 | 高 | 高 | 不稳定 | 需要充足数据和监控 |
| 完全独立模型 | 高 | 高 | 隔离强 | 适合任务相关性弱时 |

常见问题如下：

| 问题 | 现象 | 处理方式 |
|---|---|---|
| 专家数过多 | 参数和延迟上涨 | 先用 `1-2` 个私有专家 + `1` 个共享专家 |
| gate 塌缩 | 单个专家长期占主导 | 加熵正则、warm-up、监控利用率 |
| loss 尺度不一致 | 某任务主导训练 | 先做归一化和权重校准 |
| 训练不稳定 | 指标波动大 | 降低学习率或简化层数 |

gate 塌缩是最常见的问题。比如一个任务的 gate 长期把 95% 权重压在单个专家上，那路由几乎退化成单专家模型。新手可以理解为“老师明明安排了三个组讨论，结果每次都只有一个组在干活”。这时要检查专家利用率，而不是只看最终 AUC 或 loss。

训练时至少监控这些指标：

| 指标 | 作用 |
|---|---|
| 各任务 loss | 判断是否有任务被牺牲 |
| 各任务核心业务指标 | 例如 CTR、时长、完播率 |
| gate 权重分布 | 判断专家是否被充分使用 |
| 专家输出范数 | 判断某些专家是否失效 |
| 梯度范数 | 判断训练是否被单任务主导 |
| 推理延迟 | 判断结构复杂度是否可上线 |

真实工程中，PLE 的收益往往要和 loss 权重一起调。若观看时长 loss 数值远大于点击 loss，模型可能主要优化时长，点击分支即使有私有专家也救不回来。正确顺序通常是：先让 SharedBottom 和 MMoE 基线稳定，再接入 PLE；先处理 loss 尺度和样本权重，再讨论结构是否有效。

## 替代方案与适用边界

PLE 的定位是“比简单共享更稳、比完全独立更省”的中间方案。它不是所有多任务问题的默认答案。如果任务相关性弱、数据规模不足、线上延迟预算很紧，PLE 的结构复杂度可能超过收益。

| 模型 | 结构复杂度 | 抗负迁移能力 | 适用场景 |
|---|---|---|---|
| SharedBottom | 低 | 弱 | 任务高度一致 |
| MMoE | 中 | 中 | 共享需求明确 |
| PLE | 中高 | 强 | 相关但冲突明显 |
| 独立模型 | 高 | 强隔离 | 任务关系很弱或业务完全不同 |

`SharedBottom` 适合任务强相关、希望快速上线的场景。它可以作为多任务学习的第一条基线，因为结构简单、调试成本低。

`MMoE` 适合任务共享程度较高、希望通过门控自适应选择专家的场景。它比 SharedBottom 更灵活，但所有专家仍然主要是共享专家，任务私有模式表达不足。

`PLE` 适合任务既相关又冲突，需要更细粒度隔离的场景。新手可以把三者理解为：“全共享”“共享专家 + 任务选择”“共享和私有分层并存”。

选型时可以用下面的判断清单：

| 判断问题 | 更可能选择 |
|---|---|
| 只是想快速验证多任务是否有效 | SharedBottom |
| 多任务相关，且希望模型自动分配专家 | MMoE |
| 多任务相关，但某些任务明显互相拖累 | PLE |
| 多任务几乎无关，或者上线系统完全不同 | 独立模型 |
| 数据量很小，模型很容易过拟合 | SharedBottom 或简化版 MMoE |
| 延迟预算极紧 | SharedBottom 或轻量 PLE |

工程建议是：不要直接从复杂 PLE 开始。先建 SharedBottom 基线，再建 MMoE 基线，最后用 PLE 验证“私有专家 + 共享专家 + 多层递进”是否真的带来稳定提升。只有当 PLE 在离线指标、线上指标、延迟、可维护性四个维度都能接受时，它才是合理选择。
