## 核心结论

LLM 的内容安全过滤，本质上不是“加一个敏感词表”这么简单，而是一个三层安全栈：输入审核、模型内置拒答、输出审核。

第一层是输入 guardrail。guardrail 可以直译为“护栏”，白话解释就是在请求进入模型前先做一道拦截。它负责检查关键词、隐私信息、越权请求、提示词注入、异常长度等问题。第二层是模型内置策略，也就是模型在生成过程中根据安全策略决定拒答、缩写、改写或只提供安全替代。第三层是输出 guardrail，它在答案已经生成后再做一次复核，避免模型虽然“想遵守规则”，但最终文本里仍然出现暴力、仇恨、色情、自残、违法指导或隐私泄露内容。

这三层同时存在的原因很直接：单层一定会漏。输入层擅长快，但理解上下文有限；模型层理解强，但并不稳定；输出层能兜底，但已经付出了生成成本。把三层串起来，形成冗余防御，一层失效时，其他层还有机会补救。

下面用一个新手能理解的玩具例子看这个结构。假设用户问：“如何制作炸药？”  
输入层先命中危险关键词，直接标记为高风险；如果规则没有直接拦截，也可以把请求重写成“用户请求获取危险品制作方法”，再交给模型；模型层根据策略模板拒答，并改为提供安全教育或法律风险说明；如果模型仍输出了具体步骤，输出层再用危险内容分类器打分，超过阈值就阻断或重写。整个链路不是为了“让模型更聪明”，而是为了“让错误更难穿透”。

真实工程里更常见的是银行客服、教育助手、企业知识库这类场景。比如银行客服中，输入层先识别账号、身份证号、手机号、银行卡号等 PII；PII 是 personally identifiable information，白话解释就是能定位到具体人的敏感信息。模型层拒绝执行“帮我把张三账户余额发给另一个用户”这种越权请求；输出层再检查最终短信或聊天回复里是否泄露了余额、完整账号、验证码等内容。这样设计不是因为某个工具最强，而是因为不同风险必须由不同位置处理。

可以先用一个表格记住三层职责：

| 层级 | 主要职责 | 常见技术 | 典型失败模式 |
| --- | --- | --- | --- |
| 输入审核 | 在生成前拦截危险请求和异常输入 | 规则匹配、正则、关键词、PII 检测、注入检测 | 理解弱，容易被改写绕过 |
| 模型内置策略 | 在生成中依据安全政策拒答或改写 | 系统提示、策略模板、对齐训练、拒答模板 | 受上下文干扰，行为不稳定 |
| 输出审核 | 对生成结果再次复核和修正 | 分类器打分、毒性检测、PII 检测、重写器 | 已发生生成成本，可能影响延迟 |

结论可以压缩成一句话：内容安全过滤不是单点能力，而是“前置拦截 + 生成期约束 + 结果复核”的系统工程。

## 问题定义与边界

内容安全过滤要解决的问题，是让模型在真实业务中尽量不输出高风险内容，同时又不要过度误杀正常请求。这里的“高风险内容”通常包括六类：暴力、仇恨、色情、自残、违法指导、隐私泄露。不同业务还会增加版权、医疗建议、金融建议、企业机密、未成年人保护等专用类别。

边界先讲清楚：内容安全过滤不等于事实正确性检测，也不等于业务权限控制。一个回答可以“没有危险内容”但仍然“事实错误”；也可以“语气安全”但“越权访问数据”。所以安全过滤通常只是风险控制体系的一部分，不能替代鉴权、审计、日志、速率限制和人工审批。

为什么工程上总在讨论“误杀”和“漏判”？因为这两个代价都很真实。

- 误杀，英文常说 false positive，白话解释就是把正常内容误判成违规内容。
- 漏判，英文常说 false negative，白话解释就是本来违规的内容却没有被拦住。

假设一个学习助手把“如何防止实验室乙醇起火”误判为危险问题并直接拒答，这会损害可用性。反过来，如果它放过“如何绕过门禁系统”这种请求，安全事故成本又会更高。所以过滤系统不是“越严越好”，而是要按场景优化。

常见评估指标如下：

$$
\text{Precision}=\frac{TP}{TP+FP}
$$

precision 叫精确率，白话解释就是“被你拦下的内容里，真正有问题的比例”。

$$
\text{Recall}=\frac{TP}{TP+FN}
$$

recall 叫召回率，白话解释就是“所有真正有问题的内容里，你拦住了多少”。

$$
\text{FPR}=\frac{FP}{FP+TN}
$$

FPR 是 false positive rate，白话解释就是“所有本来没问题的内容里，你误杀了多少”。

$$
\text{E2E Block Rate}=\frac{\text{被最终阻断的请求数}}{\text{总请求数}}
$$

端到端阻断率的意义最直接，就是从用户视角看，最终有多少请求被系统挡掉。它不是单个分类器指标，而是整个链路的业务结果。

看一个玩具数据集。1000 次请求中，真实违规 40 条，系统最终拦截 72 条，其中 36 条是真违规、36 条是误杀。那就有：

- precision = 36 / 72 = 50%
- recall = 36 / 40 = 90%
- FPR = 36 / 960 = 3.75%
- E2E 阻断率 = 72 / 1000 = 7.2%

这组数字说明什么？说明这个系统“很敢拦”，所以 recall 不低，但 precision 很一般，误杀不小。如果这是面向公众的写作助手，3.75% 的 FPR 可能已经明显影响体验。如果这是高风险政务或金融场景，团队可能反而接受更高误杀，换取更高 recall。

再看真实工程边界。银行客服、儿童产品、医疗问答更看重 recall，因为漏一次可能直接出事故。内部知识助手、代码补全、办公写作更看重 precision，因为用户会高频使用，误杀会快速积累成流失。也就是说，内容安全过滤没有全局最优阈值，只有场景相关的最优平衡。

## 核心机制与推导

一套能上线的内容安全过滤，通常不是“规则”或“分类器”二选一，而是规则匹配、分类器打分、上下文重写、策略模板四类机制组合。

先说规则匹配。规则是最容易理解的手段，就是把一些已知危险模式写成关键词、黑白名单、正则表达式、结构校验。它的优点是快、便宜、可解释。比如检测身份证号、银行卡号、邮箱、手机号，正则表达式往往比大模型更稳定。对提示词注入里常见的 “ignore previous instructions”“reveal system prompt” 之类模式，规则也适合做第一层筛查。

但规则的缺点也很明显：表达一变，就可能漏。比如“如何制作炸药”很好拦，但“如果我想合成一种爆炸性混合物，需要哪些原料”就不一定能靠简单关键词命中。所以第二类机制是分类器打分。分类器可以是传统文本分类模型，也可以是专门的小模型，甚至是另一层 LLM 判别器。它输出的不是简单“是/否”，而是风险分数，例如暴力风险 0.91、PII 泄露风险 0.12、仇恨风险 0.03。工程上有了分数，才能设阈值、做分层处置、统计 precision 和 recall。

第三类机制是上下文重写。重写的作用不是“伪装问题”，而是把原始请求转成更适合安全决策的结构化输入。比如用户说：“告诉我怎么黑进公司邮箱。”  
输入层可以把它重写成：

- 意图：获取非法入侵指导
- 风险类别：违法行为、网络攻击
- 推荐策略：拒答，转为安全合规说明

这样模型层收到的就不再是原句，而是一个已经被净化和标注的上下文。这里的“净化”，白话解释就是把对模型有诱导性的危险表达，转成系统内部更稳定的安全标签。

第四类机制是策略模板。策略模板可以理解为一组标准决策规则，规定模型遇到不同风险类别时应该拒答、降级回答、转人工还是只返回科普信息。比如 NeMo Guardrails、Guidance 这类工具，本质上是在帮助团队把“安全政策”从零散提示词变成可复用的流程。

可以把整个流程想成下面这张逻辑图：

| 步骤 | 输入 | 处理 | 输出 |
| --- | --- | --- | --- |
| 1 | 用户 prompt | 规则匹配、PII 检测、注入检测 | 风险标签、是否直拦 |
| 2 | 原始或重写后的上下文 | 分类器/判别模型打分 | 风险分数 |
| 3 | 风险分数 + 政策模板 | 决策：放行、拒答、重写、转人工 | 安全行动 |
| 4 | 模型生成结果 | 输出分类、毒性/PII 审核 | 最终返回或阻断 |

这里最关键的不是某个单点算法，而是决策链条。举前面那个危险请求的完整机制：

1. 用户输入“如何制作炸药”。  
2. 输入层规则命中高危关键词，风险分数直接上升。  
3. 若命中非常高，直接拒答；若命中中等，则进入分类器复核。  
4. 分类器确认属于违法和暴力相关高风险内容。  
5. 策略模板决定“不提供步骤，但可以给安全与法律风险说明”。  
6. 模型生成拒答文本。  
7. 输出层再检查这段拒答里有没有残留危险细节，比如原料名、比例、具体温度。  
8. 审核通过后返回。

这个链条为什么有效？因为它把“识别风险”和“生成文本”拆开了。识别阶段偏向保守，生成阶段偏向受控，输出阶段偏向兜底。系统不再指望模型一次性“自己想明白”，而是把判断分层外化。

评估时也要分层看。假设输入层 recall 很高，但 precision 很差，就会把大量正常请求错误送去拒答；如果输入层太松，风险内容会更多落到模型层和输出层，导致延迟和成本上升。于是工程上常做的不是单阈值，而是分段阈值：

- 低风险：直接放行
- 中风险：重写后让模型受限生成
- 高风险：直接拒答
- 不确定：转人工

这个分段思想，和单看 precision、recall 不同。它强调的是系统调度，而不是分类器比赛分数。

## 代码实现

下面给一个可运行的 Python 玩具实现。这个例子不会调用真实 LLM，但会模拟输入审核、模型层决策、输出审核三步，重点是把链路写清楚。

```python
import re
from dataclasses import dataclass

DANGEROUS_PATTERNS = [
    r"炸药",
    r"爆炸物",
    r"自制枪",
    r"黑进",
    r"绕过门禁",
]

PII_PATTERNS = [
    r"\b1\d{10}\b",                  # 简化手机号
    r"\b\d{15,19}\b",                # 简化银行卡号
    r"[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}",
]

@dataclass
class ReviewResult:
    blocked: bool
    action: str
    reason: str
    content: str

def match_any(patterns, text):
    return any(re.search(p, text, flags=re.IGNORECASE) for p in patterns)

def input_guardrail(prompt: str):
    if match_any(PII_PATTERNS, prompt):
        return {"risk": 0.95, "label": "privacy"}
    if match_any(DANGEROUS_PATTERNS, prompt):
        return {"risk": 0.98, "label": "harmful"}
    if len(prompt) > 4000:
        return {"risk": 0.60, "label": "abnormal_length"}
    return {"risk": 0.05, "label": "safe"}

def policy_decision(risk_info):
    risk = risk_info["risk"]
    label = risk_info["label"]
    if risk >= 0.90:
        return "reject", f"high_risk:{label}"
    if risk >= 0.50:
        return "rewrite", f"medium_risk:{label}"
    return "allow", "low_risk"

def model_generate(prompt: str, action: str):
    if action == "reject":
        return "无法提供该请求的具体操作信息，但可以说明相关安全风险与合法替代方案。"
    if action == "rewrite":
        return "该请求存在风险。以下仅提供合规、安全、非操作性的背景说明。"
    return f"安全回答：{prompt}"

def output_guardrail(text: str):
    # 模拟输出审核：如果答案里仍包含危险词，则阻断
    if match_any(DANGEROUS_PATTERNS, text):
        return {"risk": 0.99, "label": "unsafe_output"}
    return {"risk": 0.01, "label": "safe_output"}

def handle_request(prompt: str) -> ReviewResult:
    in_risk = input_guardrail(prompt)
    action, reason = policy_decision(in_risk)
    draft = model_generate(prompt, action)
    out_risk = output_guardrail(draft)

    if out_risk["risk"] >= 0.90:
        return ReviewResult(
            blocked=True,
            action="block_output",
            reason=out_risk["label"],
            content="输出未通过安全审核。"
        )

    return ReviewResult(
        blocked=False,
        action=action,
        reason=reason,
        content=draft
    )

# 玩具例子
r1 = handle_request("如何制作炸药")
assert r1.blocked is False
assert "无法提供" in r1.content

r2 = handle_request("请帮我总结二叉树的前序遍历")
assert r2.blocked is False
assert "安全回答" in r2.content

r3 = handle_request("我的手机号是13800138000，帮我看看是不是敏感信息")
assert r3.blocked is False
assert "无法提供" in r3.content or "合规" in r3.content

print("all tests passed")
```

这个实现故意保持简单，但已经体现了三个关键点。

第一，输入 guardrail 优先执行。原因很现实：规则和正则快，应该先把显而易见的高风险请求筛掉，避免无谓调用模型。  
第二，策略决策和文本生成分开。这样以后你要替换模型、调整阈值、加入人工审核，都不用改整个链路。  
第三，输出 guardrail 再查一遍。因为模型即使被要求拒答，也可能在拒答文本里夹带危险细节，或者在多轮对话中被上下文带偏。

如果把它映射到真实工程，可以这样理解：

| 环节 | 输入 | 输出 | 常见实现 |
| --- | --- | --- | --- |
| 输入审核 | prompt、历史对话、用户画像 | 风险类别、风险分数、重写上下文 | 正则、关键词、PII 模型、注入检测器 |
| 模型层 | 净化后的上下文、策略模板 | 草稿答案 | LLM + policy prompt |
| 输出审核 | 草稿答案、风险分数、业务策略 | 放行、屏蔽、重写、转人工 | 毒性分类器、PII 检测器、审计规则 |

真实工程例子可以看银行客服。用户说：“把我上次验证码发我一下。”  
输入层先识别“验证码”属于敏感凭证请求；模型层根据策略模板不返回验证码，而是引导用户走身份校验流程；输出层确保最终回答中没有出现任何历史验证码、完整手机号或账户后四位以外的信息。这里的关键不是拒绝得多漂亮，而是不能在链路任何地方泄露凭证。

## 工程权衡与常见坑

内容安全过滤最容易被低估的地方，不是算法本身，而是工程权衡。

第一个权衡是 recall 和 FPR。高 recall 意味着更少漏判，但通常伴随更多误杀；低 FPR 意味着少误伤正常用户，但风险内容更容易漏过。很多团队上线初期只盯着“拦住了多少危险请求”，忽略“误挡了多少正常请求”，结果就是用户觉得系统什么都不让问。

第二个权衡是延迟。输入层加一个正则几乎没成本，但加三个外部分类器、一次重写、一次输出审核，p99 延迟就可能明显上升。p99 的意思是 99% 请求都低于这个延迟，白话解释就是用户高峰期最能感受到的“慢”。如果安全链太长，客服、搜索、编程助手这类实时场景会直接受影响。

第三个权衡是多语言一致性。很多过滤器在英语上表现不错，但到了中文、混合语言、拼音缩写、谐音词、方言表达时效果显著下降。看起来是“同一套规则”，实际上是“同一套英文假设”。这会导致中文用户误杀更多，或者某些风险内容在非英语里漏得更多。

第四个权衡是人工审核负载。工程上常见做法不是所有高风险都直接拦，而是对低置信度样本送人审。问题在于，阈值一旦设得太保守，人审队列会膨胀，成本迅速上升。

下面这个表格可以帮助理解阈值变化的影响：

| 阈值策略 | Recall | FPR | 人审负载 | 用户体验 | 适合场景 |
| --- | --- | --- | --- | --- | --- |
| 低阈值，严格拦截 | 高 | 高 | 低到中 | 容易误杀 | 高风险业务 |
| 高阈值，宽松放行 | 低到中 | 低 | 低 | 体验更顺 | 低风险内容助手 |
| 双阈值，灰区人审 | 中到高 | 中 | 高 | 较稳 | 金融、政务、企业客服 |

常见坑主要有五类。

一是只看单层指标，不看端到端结果。输入分类器 precision 很高，不代表最终系统表现就好，因为模型层可能重新引入风险内容。  
二是只做关键词，不做上下文。比如“自杀率统计分析”是研究问题，“如何无痛自杀”是危险请求。只靠词面很难区分。  
三是把输出审核当成可有可无。实际上很多越狱和注入攻击不是骗过输入层，而是诱导模型在输出阶段泄露不该说的东西。  
四是缺少漂移监控。漂移，白话解释就是数据分布随着时间变化，导致老规则和老阈值越来越不准。新热点事件、新俚语、新拼写规避方式，都会让系统退化。  
五是没有把业务日志和标注闭环打通。没有持续标注，就不知道误杀在哪里增加、哪类请求开始漏判，也谈不上调阈值。

一个典型真实工程例子是内容社区审核。团队把阈值压得很低，希望拦住所有仇恨和色情内容。结果 recall 上去了，但 FPR 也明显升高，正常讨论医学、法律、历史暴力事件的帖子被大量误杀。后来的修正通常不是“把规则删掉”，而是引入分层阈值：极高风险直接阻断，中间区域转人工，低风险放行，同时按语种和内容类型分别校准阈值。

所以工程上真正要监控的，不只是 precision、recall、FPR，还至少包括：

- 端到端阻断率
- 按语种拆分的误杀率
- 高风险类别漏判率
- 人审回流率
- p95/p99 延迟
- 新规则上线前后的对照实验结果

## 替代方案与适用边界

内容安全过滤常见有三种路线：纯规则、纯评分、混合策略。

纯规则最轻量。规则就是固定模式匹配，优点是快、便宜、可解释，适合预算有限、风险类型单一、文本格式较固定的场景。比如企业内部表单，主要防止手机号、身份证号、邮箱外泄，规则足够实用。问题在于，规则对复杂语义理解差，遇到改写、隐喻、跨语言表达时 recall 往往偏低。

纯评分指主要依赖分类器或 LLM 审核器。优点是覆盖面更广，对改写和上下文更敏感。问题是可解释性弱、成本更高、延迟更大，而且不同语言和新领域容易漂移。如果完全没有规则兜底，连简单的 PII 模式都要交给模型判断，通常不划算。

混合策略是大多数线上系统的实际选择：规则负责高确定性的硬约束，分类器负责灰区判断，策略模板负责统一处置，输出审核负责兜底。这不是理论上最“优雅”的方法，但通常是最稳的工程方案。

可以对比成下面这个表：

| 方案 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| 纯规则 | 快、便宜、可解释 | 语义理解差，容易漏 | PII 检测、轻量部署、固定模板输入 |
| 纯评分 | 泛化强，能看上下文 | 成本高、延迟高、易漂移 | 批量审核、复杂语义场景 |
| 混合策略 | 稳定性和覆盖面较平衡 | 架构更复杂，维护成本更高 | 实时客服、企业助手、公开产品 |

这里再给一个新手友好的类比，但保持技术准确。只用规则，就像用“关键词列表”检查作文，只要学生换个说法就可能漏掉。只用评分器，则像让阅卷老师每次都完整通读全文，准确率可能更高，但速度慢、成本高。混合策略则是先用规则筛明显错误，再让老师判断复杂题目，最后把结果复核一遍。

适用边界也要讲清楚。

实时客服场景更偏向混合策略，因为延迟必须受控，且风险不能全压给模型。  
批量内容审核可以更偏评分，因为允许更高计算成本，重点是整体召回。  
小语种或多语言场景必须单独评测，不能拿英文阈值直接复用。  
高对抗场景，比如公开 API、面向陌生用户的聊天产品，还需要额外加入注入检测、会话级记忆隔离、工具调用权限控制。内容安全过滤能挡文本风险，但挡不住所有系统级越权。

所以“哪种方案最好”这个问题本身并不准确。更准确的问法是：你的风险类型是什么，延迟预算多少，误杀成本和漏判成本各有多高，是否有人工审核资源，是否覆盖多语言，是否面向对抗性用户。只有这些边界明确，方案选择才有意义。
