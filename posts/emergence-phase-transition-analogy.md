## 核心结论

涌现能力可以用“相变”做一个严格但有限的类比。相变，白话说，就是系统在某个阈值附近从一种整体状态突然切到另一种整体状态。放到大语言模型里，参数量、训练算力或有效数据规模可以看成控制参量，白话说，就是外部可调的旋钮；任务成功率、结构化输出比例、低重复率文本占比可以看成序参量，白话说，就是系统是否进入“新能力状态”的观测指标。

关键点不在于损失函数本身出现真正的不连续，而在于：损失通常是平滑下降的，但下游评测往往经过一个强非线性解码。非线性，白话说，就是输入只变一点，输出可能突然变很多。于是模型内部表示的连续改善，经过阈值型判定后，会表现成准确率的陡升，看起来像“突然会了”。

如果用统计物理语言写，临界点附近可写成
$$
m - m_c \propto (N - N_c)^\beta, \quad N > N_c
$$
其中 $m$ 是序参量，$N$ 是控制参量，$N_c$ 是临界规模，$\beta$ 是临界指数。临界指数，白话说，就是“越过阈值后增长有多快”的幂律刻度。它的价值不在于制造神秘感，而在于帮助工程上预估：到底还差多少规模，能力跳变会不会真的发生，是否值得继续加算力。

一个新手版玩具例子是“烧水”: 水温从 20℃ 到 99℃ 都在连续变热，但到了约 100℃，状态从液态切到气液共存，宏观行为明显不同。模型也类似：参数量从 30 亿到 60 亿也许只带来小幅 loss 改善，但某项任务准确率可能在一个窄区间内从 20% 跳到 80%。

| 物理相变 | 大模型类比 | 含义 |
| --- | --- | --- |
| 温度 | 参数量/算力/数据规模 | 驱动系统变化的控制参量 |
| 磁化强度或密度差 | 准确率/结构化率/低重复率 | 描述系统状态的序参量 |
| 临界温度 $T_c$ | 临界规模 $N_c$ | 跳变附近的阈值 |
| 临界指数 $\beta$ | 能力增长幂律指数 | 描述阈值后的增长速度 |

文字示意图可以这样理解：横轴是模型规模 $N$，纵轴是任务成功率 $m$。loss 曲线缓慢下降，而成功率曲线在 $N_c$ 左右突然变陡，像一条被拉直的 S 曲线中段。

## 问题定义与边界

本文讨论的不是“任何性能提升都叫涌现”，而是更窄的现象：当控制参量逼近某个阈值时，某个任务指标出现明显非线性跃迁。这里的“明显”通常体现在导数变大、局部曲率变陡，或者评测结果从接近随机突然进入可用区间。

边界要先说清：

1. 只讨论大语言模型或类似的自回归生成模型。
2. 控制参量主要看参数量 $N$、训练 token 数 $D$、推理温度 $T$。
3. 序参量主要看任务准确率、格式遵守率、语义一致性、重复率的反向指标。
4. 不把所有离散任务都解释成真正的物理相变，因为很多跳变来自评测指标的阈值化设计。

训练层面的平滑标度通常写成
$$
L(N,D)=E+\frac{A}{N^\alpha}+\frac{B}{D^\beta}
$$
这里的 $L$ 是损失，白话说，就是模型平均犯错的程度；$E$ 是不可约误差，白话说，就是即使无限扩展也难以消除的那部分；$\alpha,\beta$ 则描述规模和数据增加时损失下降的速度。

这个公式说明一件事：从优化视角看，很多量是连续变好的。问题在于用户看到的不是 loss，而是“能不能做成一件事”。一旦观测指标本身带阈值，连续改善就会被投影成离散跃迁。

下面给一个边界表：

| 类型 | 变量 | 可量化指标 | 本文角色 |
| --- | --- | --- | --- |
| 控制参量 | 参数量 $N$ | 7B、13B、70B | 主控制参量 |
| 控制参量 | 数据量 $D$ | 训练 token 数 | 与 $N$ 联合决定 loss |
| 控制参量 | 温度 $T$ | 0.8, 1.0, 1.2 | 推理阶段行为旋钮 |
| 序参量 | 准确率 $m$ | GSM8K/代码题通过率 | 典型能力指标 |
| 序参量 | 结构化率 | JSON 合法输出比例 | 工程常用指标 |
| 序参量 | 重复率反指标 | 去重后 n-gram 比例 | 文本是否失稳 |

新手版边界例子：如果控制参量定义成“模型参数数 + 推理温度”，序参量可定义成“数学题准确率 + JSON 输出合法率”。这比只看 loss 更接近真实使用体验。

## 核心机制与推导

核心机制可以拆成两段：先有平滑标度，再有非线性解码。

第一段是平滑标度。随着 $N$ 和 $D$ 增大，模型内部表示能力通常持续改善。这里可以把某项任务相关的内部可分性写成一个随规模缓慢上升的 logit：
$$
z(N)=a\log \frac{N}{N_0}+b
$$
logit，白话说，就是模型在做二分类决策前的“原始打分”，还没被压到 0 到 1 之间。

第二段是非线性解码。很多任务最终看的是概率或通过/失败，这常用 sigmoid 映射：
$$
p(N)=\sigma(k(z(N)-\theta))=\frac{1}{1+e^{-k(z(N)-\theta)}}
$$
这里 $k$ 是斜率，白话说，就是阈值有多锋利；$\theta$ 是任务门槛，白话说，就是“内部表示至少要好到什么程度”才能稳定过关。

当 $k$ 很大时，哪怕 $z(N)$ 只增加一点点，$p(N)$ 也会从低值迅速跳到高值。这就解释了为什么 loss 只从 1.02 降到 0.98，但准确率可能从 0.1 跳到 0.9。

玩具例子直接代入：
$$
z(N)=0.2\log(N/10^8), \quad p(N)=\sigma(60(z(N)-0.2))
$$
取 $N=3\times10^8$ 与 $6\times10^8$：

- 当 $N=3\times10^8$，$z\approx0.220$，刚过阈值不多。
- 当 $N=6\times10^8$，$z\approx0.358$，超过阈值更多。
- 因为外层有系数 60，两个规模点在概率上会被迅速拉开。

可以把这个过程写成三步推导：

1. 用 scaling law 得到 loss 或内部表征随规模平滑变化。
2. 假设某项能力对应一个阈值化的判定面，即 $z=\theta$ 附近很敏感。
3. 用 sigmoid 或 hard threshold 把连续表征投影到离散任务结果，于是出现“相变样”的跳变。

临界指数进入讨论，是因为临界点附近经常不是简单直线，而更像幂律：
$$
m(N)-m_c \sim (N-N_c)^\beta
$$
如果不同任务在重标度后落到相似曲线，说明它们可能共享某种“指数类”。指数类，白话说，就是不同系统虽然细节不同，但临界附近 obey 同一套增长规律。这并不意味着语言模型真的等同于铁磁体，而是说：在阈值现象的统计描述上，可能存在可复用的缩放结构。

下表把玩具例子数值化：

| $N$ | $z(N)$ | $p(N)$ | loss 示例 |
| --- | --- | --- | --- |
| $2\times10^8$ | 0.139 | 0.025 | 1.05 |
| $3\times10^8$ | 0.220 | 0.770 | 1.02 |
| $4\times10^8$ | 0.277 | 0.990 | 1.00 |
| $6\times10^8$ | 0.358 | 1.000 近似 | 0.98 |

这个表故意展示一个事实：loss 是缓慢下降的，而 $p$ 不是。真正“跳”的往往不是优化本身，而是观测函数。

真实工程例子更有代表性。很多团队在做结构化生成时，会发现推理温度从 0.8 调到 1.1，输出状态像从“可控”切到“发散”。如果把温度 $T$ 看作控制参量，把 JSON 合法率或长段重复率看作序参量，那么也会观察到阈值区：低于某个温度文本太保守，高于某个温度格式开始崩，临界附近最敏感。这里的“相变”不是参数规模驱动，而是采样动力学驱动，但数学形态相似。

## 代码实现

下面给一个最小可运行实现。它做三件事：

1. 用简化的 scaling law 计算 loss。
2. 用对数规模映射得到内部 logit。
3. 用 sigmoid 得到任务成功率，并用临界点附近幂律近似拟合。

```python
import math
import numpy as np

# 1) 简化的 scaling law
def loss_fn(N, D, E=0.8, A=5.0, alpha=0.30, B=2.0, beta=0.20):
    return E + A / (N ** alpha) + B / (D ** beta)

# 2) 内部表征：随 log(N) 平滑增长
def logit_fn(N, N0=1e8, a=0.2, b=0.0):
    return a * math.log(N / N0) + b

# 3) 非线性解码：把连续表征映射成接近阶跃的成功率
def sigmoid(x):
    return 1.0 / (1.0 + math.exp(-x))

def success_prob(N, k=60.0, theta=0.2):
    z = logit_fn(N)
    return sigmoid(k * (z - theta))

# 采样一组规模点
Ns = np.array([2e8, 3e8, 4e8, 5e8, 6e8, 8e8], dtype=float)
D = 5e10

losses = np.array([loss_fn(N, D) for N in Ns])
probs = np.array([success_prob(N) for N in Ns])

# 基本性质检查：loss 随规模下降，成功率随规模上升
assert np.all(np.diff(losses) < 0), "loss 应随 N 增大而下降"
assert np.all(np.diff(probs) > 0), "成功率应随 N 增大而上升"

# 用已知临界点做局部幂律拟合：log(m-mc) = beta * log(N-Nc) + c
Nc = 2.72e8   # 由 logit_fn(Nc)=theta 推出
mc = 0.5      # sigmoid 在阈值点附近取 0.5

mask = Ns > Nc
x = np.log(Ns[mask] - Nc)
y = np.log(probs[mask] - mc)

coef = np.polyfit(x, y, 1)
beta_est = coef[0]

# 对这个玩具模型，beta_est 只是局部近似指数，不要求等于 1
assert beta_est > 0, "临界指数估计应为正"

print("N:", Ns.astype(int))
print("loss:", np.round(losses, 4))
print("prob:", np.round(probs, 4))
print("estimated_beta:", round(beta_est, 4))
```

这段代码不是在证明“真实 LLM 一定 obey 某个固定 $\beta$”，而是在展示工程流程：先测平滑量，再定义非线性观测，再在临界区做有限标度拟合。有限标度，白话说，就是样本和规模有限时，用一套缩放公式估计无限大系统的临界行为。

如果要继续扩展，通常会做两件事：

- 用更多 checkpoint 拟合 $N_c,\beta,k,\theta$。
- 用 `matplotlib` 画两张图：`loss-N` 的平滑曲线，以及 `prob-N` 的陡升曲线。

## 工程权衡与常见坑

最大的坑，是把平滑 scaling law 直接当成能力预测器。它适合预测平均误差下降，不适合直接预测“是否跨过任务门槛”。门槛一旦存在，非线性解码就会放大局部变化。

第二个坑，是忽略数据谱。数据谱，白话说，就是不同频率、难度、模式在数据中如何分布。若训练数据里高价值模式很稀疏，$\alpha$ 的有效值会变，导致你以为再加一点参数就能跨过 $N_c$，实际上还差数据质量或数据覆盖。

第三个坑，是把温度、提示词、输出格式约束当成“无关细节”。在真实系统里，这些量会移动阈值 $\theta$，甚至改变斜率 $k$。同一个底模，在不同采样设置下，看到的“涌现点”可能根本不是同一个。

| 误区 | 影响 | 规避措施 |
| --- | --- | --- |
| 只看 loss 下降 | 误判是否接近能力跃迁 | 同时跟踪序参量曲线 |
| 忽略非线性解码 | 看不见阈值放大效应 | 显式建模 $p=\sigma(k(z-\theta))$ |
| 忽略数据谱变化 | 错估 $\alpha$ 与 $N_c$ | 分任务、分数据簇做拟合 |
| 用单点结果断言涌现 | 容易把噪声当相变 | 至少做多 checkpoint 曲线 |
| 温度设置不统一 | 比较结果失真 | 固定推理协议再比较规模 |

工程上的检查点可以压缩成三条：

- 非线性解码是否显式建模。
- 数据谱是否变化，尤其是高难样本占比是否变了。
- 采样温度和输出约束是否让阈值位置漂移。

一个真实工程判断例子：团队看到某任务 loss 只降了 0.04，就计划把模型从 13B 扩到 34B，认为“快涌现了”。这很危险。正确做法是先拿已有 checkpoints 拟合 $z(N)$ 和 $p(N)$，再看临界区是否真的逼近。如果拟合出来的 $\theta$ 还远，盲目加算力只是昂贵试错。

## 替代方案与适用边界

相变视角不是唯一方法。至少还有两类替代方案。

第一类是纯 scaling law 预测。优点是简单、稳健、数据需求低，适合早期训练资源规划。缺点是它主要对平均损失有效，对离散任务门槛不够敏感。

第二类是经验阈值法。做法是直接看学习曲线的一阶差分或局部斜率，哪里变陡就认为哪里“快涌现”。它实现简单，但解释力弱，容易把评测噪声、prompt 偶然性、温度变化混进去。

| 策略 | 优点 | 缺点 | 适用条件 |
| --- | --- | --- | --- |
| 纯 scaling law | 易拟合，成本低 | 难预测离散能力跳变 | 早期容量规划 |
| 经验阈值法 | 实现快，直观 | 稳定性差，可解释性弱 | 小规模实验筛查 |
| 相变/临界指数法 | 能描述阈值与跳变 | 需要更多 checkpoint 和更严谨观测 | 大模型调优、昂贵训练决策 |

何时适合切换到相变视角，可以用一个简单决策规则：

1. 如果任务指标是连续的，且没有明显门槛，先用 scaling law。
2. 如果任务指标是通过/失败、格式合法/不合法，优先怀疑存在非线性解码。
3. 如果训练成本高，继续扩模前必须判断“会不会跨阈值”，就做临界拟合。
4. 如果模型和数据都很小、任务极稀疏，临界指数估计会很不稳，不建议过度解释。

适用边界也要明确。相变类比最有用的场景，是大模型、数据足够多、任务评测具备清晰门槛、并且训练预算高到需要提前判断是否值得继续扩展。不太适合的小模型场景包括：样本量太小、评测噪声很大、prompt 变化就能改变结果、任务本身定义不稳定。此时看到“跳变”，更可能是评测协议问题，而不是系统真的跨了某个稳健临界点。
