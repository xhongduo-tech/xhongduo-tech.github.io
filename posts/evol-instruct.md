## 核心结论

Evol-Instruct 是一种进化式指令生成方法：从简单种子指令出发，通过多轮改写、加约束、扩展场景、深化推理，把指令逐步变成更复杂、更有信息量的微调样本。

它的重点不是“让模型多生成一些题”，而是“让题目沿着可控方向逐轮变难”。这使它适合构造指令微调数据，尤其适合复杂推理、长尾场景、领域定制任务。WizardLM 系列模型就是基于这类方法构建指令数据的代表工作。

玩具例子：

| 轮次 | 指令 |
|---|---|
| 种子 | 解释什么是过拟合。 |
| 第 1 轮 `in-depth` | 解释什么是过拟合，并说明训练误差和验证误差通常如何变化。 |
| 第 2 轮 `in-breadth` | 比较过拟合、欠拟合和正则化，并给出一个真实建模场景。 |

`in-depth` 指“向深处演化”，也就是增加约束、步骤、推理链长度。`in-breadth` 指“向广处演化”，也就是扩展主题、技能、输入形式和应用场景。

| 演化方向 | 白话解释 | 典型变化 |
|---|---|---|
| `in-depth` | 同一个问题变得更难 | 增加条件、要求比较、加入推理步骤 |
| `in-breadth` | 问题覆盖面变得更广 | 换任务类型、换领域、加入新输入形态 |

研究结论可以概括为：

| 结论 | 含义 |
|---|---|
| 指令难度可以被迭代提升 | 不必一次生成高难题，可以逐轮增加复杂度 |
| 数据覆盖可以被主动扩展 | 不只围绕原问题改写，也能扩展到相邻任务 |
| 过滤器决定下限 | 坏样本不过滤，演化越多污染越大 |
| 种子质量决定上限 | 种子太窄或太错，后续演化也会偏 |

## 问题定义与边界

Evol-Instruct 解决的问题是：高质量指令微调数据不足。指令微调是指用“用户指令 + 模型回答”的样本训练模型，让模型更好地遵循自然语言任务。

普通做法有两个常见问题。第一，人工编写成本高。第二，让模型一次性批量生成指令，容易得到大量简单、重复、模板化的问题。Evol-Instruct 的思路是从已有种子出发，让数据难度和覆盖面逐轮提升。

设初始数据集为：

$$
D^{(0)} = \{(I_k^{(0)}, R_k^{(0)})\}
$$

其中：

| 符号 | 含义 |
|---|---|
| $D^{(0)}$ | 第 0 轮种子数据集 |
| $I_k^{(t)}$ | 第 $t$ 轮第 $k$ 条指令 |
| $R_k^{(t)}$ | 第 $t$ 轮第 $k$ 条回答 |
| $E_p$ | 演化器，即用提示词控制模型改写指令的过程 |
| $M$ | 生成回答的模型 |
| $F(I,R)$ | 过滤器，判断指令和回答是否可用 |

流程可以写成：

```text
种子指令
  -> 演化器改写
  -> 生成新回答
  -> 过滤器筛选
  -> 合格样本进入下一轮
  -> 重复多轮
```

真实工程例子：一个客服系统只有 200 条 FAQ，例如“如何重置密码”。如果直接用这些 FAQ 微调，模型主要学会回答老问题。真实工单往往更复杂，例如用户收不到邮件、手机丢失、账号绑定企业 SSO、管理员权限不足。Evol-Instruct 可以把原始问题演化成：“用户无法接收重置邮件、手机已丢失，并且账号绑定企业 SSO 时，客服应如何分步处理？”这类样本更接近真实工单。

边界也必须明确：

| 场景 | 是否适合 | 原因 |
|---|---:|---|
| 复杂客服工单 | 适合 | 需要多条件、多步骤处理 |
| 代码解释与调试 | 适合 | 可以逐步增加约束、错误日志、边界条件 |
| 基础常识问答补量 | 不一定适合 | Self-Instruct 或模板生成更便宜 |
| 法律、医疗最终建议 | 谨慎 | 错误答案风险高，需要专家过滤 |
| 标准答案极强的考试题 | 谨慎 | 自动演化可能偏离评分标准 |
| 种子本身质量很差 | 不适合 | 错误会被放大 |

Evol-Instruct 不是万能数据生成器。如果种子问题低质、过滤规则粗糙、演化方向脱离真实任务，最后只会得到“看起来复杂但没有训练价值”的数据。

## 核心机制与推导

Evol-Instruct 的核心机制是一个闭环：

```text
I_k^(t)
  -> E_p 改写得到 I_k^(t+1)
  -> M 生成 R_k^(t+1)
  -> F 判断是否保留
  -> 合格样本进入 D^(t+1)
```

形式化写法如下：

$$
D^{(0)} = \{(I_k^{(0)}, R_k^{(0)})\}
$$

$$
I_k^{(t+1)} = E_p(I_k^{(t)})
$$

$$
R_k^{(t+1)} = M(I_k^{(t+1)})
$$

$$
F(I_k^{(t+1)}, R_k^{(t+1)}) = 1
$$

当 $F(I,R)=1$ 时，样本保留；否则丢弃或回池重试。过滤器是指一组自动或人工规则，用来判断样本是否有效，例如回答是否为空、是否拒答、是否复制了提示词模板、是否明显不可回答。

单轮演化示意：

| 步骤 | 输入 | 输出 | 目的 |
|---|---|---|---|
| 读取 | `解释什么是过拟合。` | 原始指令 | 提供起点 |
| 演化 | 原始指令 | 更复杂的新指令 | 增加难度或扩展场景 |
| 回答 | 新指令 | 新回答 | 构造训练样本 |
| 过滤 | 新指令 + 新回答 | 通过或失败 | 控制质量 |
| 入库 | 通过样本 | 下一轮数据 | 支持继续演化 |

两类演化路径的作用不同：

| 类型 | 增加什么 | 示例 |
|---|---|---|
| `in-depth` | 步骤、约束、推理链长度 | 要求解释训练误差和验证误差曲线 |
| `in-breadth` | 主题、技能、场景覆盖 | 从过拟合扩展到欠拟合、正则化和建模案例 |

玩具例子中，原始问题“解释什么是过拟合”只要求定义。`in-depth` 版本加入“训练误差和验证误差如何变化”，要求模型说明现象。`in-breadth` 版本加入“欠拟合、正则化、真实建模场景”，要求模型做比较和迁移。

关键在于“每轮只变难一点”。如果第一轮直接生成一个很长、很多条件、还要求跨学科推理的问题，模型可能答不稳，过滤器也更难判断质量。逐轮演化让数据复杂度更平滑。

## 代码实现

代码实现的重点不是训练模型，而是把数据生成流水线搭起来：读取种子、演化指令、生成回答、过滤失败样本、去重、落盘。

最小伪代码如下：

```python
D = seed_data
for t in range(T):
    D_next = []
    for I, _ in D:
        I_new = evolve_prompt(I)
        R_new = model_answer(I_new)
        if filter_sample(I_new, R_new):
            D_next.append((I_new, R_new))
    D = D_next
```

下面是一个可运行的玩具版本。它不调用真实大模型，而是用规则模拟 `in-depth` 和 `in-breadth`，重点展示 Evol-Instruct 的数据流。

```python
from dataclasses import dataclass

@dataclass
class Sample:
    instruction: str
    response: str
    round: int
    evolution_type: str
    pass_flag: bool

def evolve_instruction(instruction: str, evolution_type: str) -> str:
    if evolution_type == "in-depth":
        return instruction.rstrip("。") + "，并说明训练误差和验证误差通常如何变化。"
    if evolution_type == "in-breadth":
        return "比较过拟合、欠拟合和正则化的区别，并给出一个真实建模场景。"
    raise ValueError("unknown evolution_type")

def model_answer(instruction: str) -> str:
    if "过拟合" in instruction:
        return "过拟合是模型在训练集上表现很好，但在新数据上泛化较差的现象。"
    return ""

def filter_sample(instruction: str, response: str) -> bool:
    if not instruction.strip() or not response.strip():
        return False
    banned = ["Given Prompt", "Rewritten Prompt", "sorry"]
    if any(token.lower() in response.lower() for token in banned):
        return False
    if len(instruction) < 8 or len(response) < 10:
        return False
    return True

seed = [Sample("解释什么是过拟合。", "过拟合是泛化能力不足的现象。", 0, "seed", True)]

generated = []
for item in seed:
    for evo_type in ["in-depth", "in-breadth"]:
        new_i = evolve_instruction(item.instruction, evo_type)
        new_r = model_answer(new_i)
        ok = filter_sample(new_i, new_r)
        generated.append(Sample(new_i, new_r, 1, evo_type, ok))

assert len(generated) == 2
assert all(s.pass_flag for s in generated)
assert generated[0].evolution_type == "in-depth"
assert "验证误差" in generated[0].instruction
```

工程落盘时，建议每条数据至少保留这些字段：

| 字段 | 含义 |
|---|---|
| `instruction` | 演化后的用户指令 |
| `response` | 生成的回答 |
| `round` | 第几轮演化 |
| `evolution_type` | `in-depth` 或 `in-breadth` |
| `pass_flag` | 是否通过过滤 |
| `source_slug` | 来自哪条种子，可选但推荐 |
| `hash` | 去重用指纹，可选但推荐 |

真实工程中还需要几个控制项。第一，去重，避免同一问题换个说法反复进入训练集。第二，失败回池，把可修复样本重新生成，而不是直接丢弃全部。第三，温度设置，温度是控制生成随机性的参数；太低会重复，太高会跑偏。第四，演化提示词和过滤规则要独立配置，方便后续评估和回滚。

## 工程权衡与常见坑

Evol-Instruct 最大风险不是生成不够多，而是生成得更复杂却没有更有价值。长度不是难度，更多条件也不一定等于更好的训练样本。

错误演化：

```text
请用非常非常详细的语言解释什么是过拟合以及过拟合。
```

这只是变长，没有增加新的判断、输入或推理要求。

更好的演化：

```text
解释什么是过拟合，并结合训练误差下降、验证误差上升的曲线现象，说明为什么早停可以缓解过拟合。
```

这增加了现象、原因和方法之间的关系，信息量更高。

常见坑如下：

| 坑 | 表现 | 规避方式 |
|---|---|---|
| 只变长不变难 | 指令越来越啰嗦 | 要求每轮加入新约束、新输入或新推理点 |
| 生成不可回答题 | 条件互相矛盾或缺少必要信息 | 过滤矛盾条件，保留可回答边界 |
| 过滤太弱 | 空回答、拒答、模板泄漏进入训练集 | 设置硬规则和抽样人工检查 |
| 只追求深度 | 主题集中在少数方向 | 混用 `in-depth` 和 `in-breadth` |
| 对比不公平 | 数据量变多导致效果提升 | 做等量样本对比 |
| 提示词泄漏 | 回答里出现改写模板 | 禁止模板标记进入最终数据 |

失败样本规则示例：

| 规则 | 判定为失败的例子 |
|---|---|
| 回答为空 | `""` |
| 明显拒答 | `sorry, I cannot answer` |
| 复制模板 | `Given Prompt: ... Rewritten Prompt: ...` |
| 指令无意义 | `请详细说明这个问题。` 但没有具体问题 |
| 约束矛盾 | 要求“只用一句话”同时“列出 20 个步骤” |
| 答案与指令无关 | 指令问密码重置，回答讲机器学习 |

过滤规则可以从简单硬规则开始：

```text
1. 指令和回答不能为空。
2. 回答不能包含演化提示词模板。
3. 指令长度不能过短，回答长度不能过短。
4. 指令不能包含明显矛盾约束。
5. 同一批次内相似度过高的样本只保留一条。
6. 高风险领域必须人工抽检。
```

真实工程例子：客服工单数据中，如果把“如何重置密码”演化到“用户手机丢失、邮箱不可用、企业 SSO 登录失败、管理员离职”的复杂场景，这有价值。但如果答案只是泛泛说“联系客服处理”，就没有训练价值。复杂指令必须配套具体、可执行、符合业务规则的回答。

## 替代方案与适用边界

Evol-Instruct 适合“需要复杂度渐进提升”的数据构造，但不是所有任务都应该用它。

| 方法 | 做法 | 优点 | 缺点 | 适合场景 |
|---|---|---|---|---|
| Self-Instruct | 让模型直接批量生成新指令 | 快速、便宜、铺量快 | 容易简单和重复 | 基础问答、通用任务补量 |
| Evol-Instruct | 从种子逐轮演化 | 难度分层、长尾覆盖好 | 流程更复杂，依赖过滤 | 复杂推理、领域工单、代码任务 |
| 人工编写 | 专家直接写样本 | 质量高、可控 | 成本高、规模慢 | 高风险任务、标准严格任务 |
| 模板扩增 | 用模板替换变量 | 稳定、便宜 | 多样性有限 | 格式固定、规则明确的任务 |

适用边界可以这样判断：

| 维度 | 适合 Evol-Instruct | 不适合 Evol-Instruct |
|---|---|---|
| 任务类型 | 多步骤、多条件、长尾多 | 简单固定问答 |
| 数据质量要求 | 中高，需要过滤和抽检 | 极高且必须专家确认 |
| 成本 | 能接受生成、过滤、评估流程 | 只需要低成本快速补量 |
| 可扩展性 | 希望持续演化新数据 | 数据规模很小且稳定 |
| 答案标准 | 有一定弹性 | 唯一标准答案强约束 |

什么时候不用 Evol-Instruct：

| 判断问题 | 结论 |
|---|---|
| 只是缺 100 条基础 FAQ 吗 | 优先人工模板或 Self-Instruct |
| 答案必须完全符合法规或医学规范吗 | 不应只靠自动演化 |
| 种子数据本身还没清洗好吗 | 先清洗种子 |
| 没有能力做过滤和抽检吗 | 暂缓使用 |
| 目标是复杂工单、代码调试、长尾问法吗 | 可以考虑 Evol-Instruct |

Self-Instruct 和 Evol-Instruct 的区别可以用一句话概括：Self-Instruct 更像“让模型直接想更多题”，Evol-Instruct 更像“拿已有题逐轮升级”。如果目标是快速补常识问答，Self-Instruct 通常更便宜。如果目标是让模型处理复杂工单、长上下文输入、带约束的推理任务，Evol-Instruct 更合适。
