## 核心结论

中文分词的特殊处理，核心不是“先把中文强行切成词”，而是**在没有空格边界的前提下，设计一个稳定的 token 映射规则**。token 可以理解为“模型真正读取的最小文本单元”。对中文来说，这个单元通常在“单字”和“常见字串”之间摆动，而不是像英文那样天然按空格切开。

主流策略大致有三类：

| 策略 | 直观含义 | 中文上的典型表现 | 主要优点 | 主要风险 |
|---|---|---:|---|---|
| 字级 | 每个汉字基本对应 1 个 token | 约 1.0 字/token | 稳定、无外部分词依赖 | token 数偏多 |
| 词级 + 子词 | 先切词，再做子词切分 | 常见词可合并，整体约 1.1 到 1.5 字/token | 某些词更短 | 误切会向后传播 |
| 子词/字节级 BPE | 用统计合并规则直接学常见片段 | 常见公开经验约 1.5 到 1.8 字/token | 多语言统一、压缩略好 | 词表污染、罕见合并词副作用 |

一个足够实用的估算公式是：

$$
N_{token} \approx \left\lceil \frac{N_{char}}{cpt} \right\rceil
$$

其中 $N_{char}$ 是汉字数，$cpt$ 是“每个 token 平均包含多少个汉字”。如果 $cpt=1.0$，基本就是字级；如果 $cpt=1.8$，说明平均 1.8 个字压成 1 个 token，但压缩空间仍有限。

玩具例子：“今天天气很好”。

- 字级切法：`今 / 天 / 天 / 气 / 很 / 好`
- 更激进的子词切法，最多也只是把少数组合并成 `今天 / 天气 / 很 / 好` 或类似形式
- 结论：中文里很多高频信息仍然落在“字”这个层面，完全脱离字级的收益并不大

BERT-base-chinese 的常用做法，本质上就是**以字级为主的 WordPiece 词表**。WordPiece 可以理解为“一种把高频片段也收入词表的子词方法”。它的词表大小是 21128，这说明中文预训练模型并不是先依赖一个外部中文分词器再喂给模型，而是直接让 tokenizer 处理原始中文字符串。对多数 NLU 任务，这种做法的准确率并不吃亏，却能减少切词错误传播。

## 问题定义与边界

中文分词的特殊处理，指的是：**因为中文文本没有像英文那样天然用空格标记词边界，所以模型必须自己决定，哪些字符该单独编码，哪些字符组合可以一起编码。**

这里要分清三个层次：

| 层次 | 关注对象 | 本文是否讨论 |
|---|---|---|
| 人类阅读层 | “这个句子应该怎么断词才自然” | 部分涉及 |
| NLP 工具层 | jieba、pkuseg 这类传统中文分词器 | 不是重点 |
| 模型输入层 | tokenizer 如何把文本映射为 token ID | 是重点 |

本文讨论的是第三层，也就是**大模型和预训练模型入口处的 tokenizer**。不讨论搜索引擎、规则分词、词法分析器那类专用系统，也不讨论中文排版规范本身。

对新手来说，最容易混淆的一点是：**“中文没有空格”不等于“模型必须先切词”。**  
正确理解是：模型只需要一个稳定的编码方案。这个方案可以按字，也可以按常见词片段，也可以按字节，但必须保证训练和推理时一致。

可以用一个文字版图示理解：

- 原句：`这一段中文`
- 字级方案：`这 | 一 | 段 | 中 | 文`
- 词级方案：`这一段 | 中文`
- 混合方案：`这 | 一段 | 中文`

这三种方案都能工作。差别不在“能不能输入模型”，而在**长度、鲁棒性、跨域适配能力**。

从工程视角，训练和推理的边界也不同：

| 场景 | 更常见策略 | 原因 |
|---|---|---|
| 预训练 | 字级或字级主导的子词 | 覆盖稳定，避免误切 |
| 微调分类/NER | 沿用预训练 tokenizer | 保持分布一致 |
| 超长上下文生成 | 更倾向 BPE/字节级压缩 | 降低 token 成本 |
| 垂直领域术语密集 | 混合策略更常见 | 兼顾术语和后备字级 |

---

## 核心机制与推导

Tokenizer 的基本工作方式是：**拿着词表，从左到右尝试匹配最长的已知片段。**  
BPE 可以理解为“把经常一起出现的字符反复合并成更大的片段”；WordPiece 可以理解为“优先保留能解释训练语料的高频子词”。

中文里的关键现实是：**高频双字词、三字词确实存在，但单字本身的信息密度也很高。**  
这决定了中文的压缩上限不可能像英文那样显著，因为英文靠空格先天给出了词边界，而中文没有这个红利。

仍然用公式估算：

$$
N_{token} \approx \left\lceil \frac{N_{char}}{cpt} \right\rceil
$$

假设输入长度是 300 个汉字：

| 平均字/token（cpt） | 估算 token 数 | 相对字级节省 |
|---:|---:|---:|
| 1.0 | 300 | 0% |
| 1.2 | 250 | 16.7% |
| 1.5 | 200 | 33.3% |
| 1.8 | 167 | 44.3% |

这张表能说明两个事实。

第一，压缩确实存在。  
300 字从 300 token 降到 167 token，不是零收益。

第二，收益递减也很明显。  
从 1.5 提高到 1.8，只少了 33 个 token；但为了这 33 个 token，你往往要引入更大的词表、更复杂的统计合并、更高的词表清洗成本，以及跨域数据的不稳定性。

推导逻辑并不复杂：

1. 中文总长度固定为 $N_{char}$。
2. 每个 token 平均容纳 $cpt$ 个汉字。
3. 那么总 token 数就是总字符数除以平均容纳量。
4. $cpt$ 越大，token 越少，但中文里 $cpt$ 很难无限增大，因为很多字在不同上下文中组合不稳定。

玩具例子可以看得更直观：

- 句子 A：`北京大学生物系`
- 合理分法可能是：`北京大学 / 生物系`
- 也可能是：`北京 / 大学生 / 物系`，这是错误边界
- 如果你先依赖外部分词，错误边界会直接改变后续 token
- 如果保留字级后备，最差也只是退回 `北 / 京 / 大 / 学 / 生 / 物 / 系`

这就是为什么中文模型常偏爱“字级为底，子词为辅”。它不是最短，但更稳。

真实工程例子：中文命名实体识别。  
句子是“患者在华山医院接受检查”。如果词级系统把“华山医院”误切成“华山 / 医院”，而标注体系需要整个机构名作为单个实体，那么标签对齐会立刻变复杂。字符级 tokenizer 虽然 token 更多，但每个字的边界是确定的，实体 span 对齐反而简单。

## 代码实现

下面给出一个**可运行的纯 Python 玩具实现**。它不依赖第三方库，但能说明三种思路：字级、最长匹配词级、以及 token 数估算。

```python
import math

def char_tokenize(text: str):
    return list(text)

def longest_match_tokenize(text: str, vocab):
    tokens = []
    i = 0
    max_len = max(len(x) for x in vocab) if vocab else 1
    while i < len(text):
        matched = None
        for size in range(min(max_len, len(text) - i), 0, -1):
            piece = text[i:i+size]
            if piece in vocab:
                matched = piece
                break
        if matched is None:
            matched = text[i]
        tokens.append(matched)
        i += len(matched)
    return tokens

def estimate_tokens(num_chars: int, cpt: float):
    return math.ceil(num_chars / cpt)

# 玩具例子
text = "今天天气很好"
char_tokens = char_tokenize(text)
vocab = {"今天", "天气", "很好", "很", "好"}
subword_tokens = longest_match_tokenize(text, vocab)

assert char_tokens == ["今", "天", "天", "气", "很", "好"]
assert subword_tokens == ["今天", "天气", "很好"]
assert estimate_tokens(300, 1.5) == 200
assert estimate_tokens(300, 1.8) == 167

print("char:", char_tokens)
print("subword:", subword_tokens)
print("ok")
```

这段代码反映了真实 tokenizer 的两个核心点：

| 步骤 | 作用 | 对中文的意义 |
|---|---|---|
| `char_tokenize` | 按字切开 | 最稳的后备策略 |
| `longest_match_tokenize` | 先匹配最长词表片段 | 模拟 BPE/WordPiece 的局部合并 |
| `estimate_tokens` | 估算长度 | 用于上下文预算和成本预估 |

如果放到真实模型中，流程通常是：

1. 原始中文字符串进入 tokenizer。
2. tokenizer 输出 token 序列和对应 ID。
3. 模型读入 ID，做分类、序列标注或生成。
4. 如果是 Masked LM，还要决定 mask 粒度。

这里顺带解释两个常见术语。

- CLM：这里指字符级掩码思路，白话讲就是“该遮哪个字，就只遮哪个字”。
- WWM：Whole Word Masking，白话讲就是“如果一个词由多个字组成，就把整个词一起遮掉”。

在中文 BERT 里，WWM 听起来更“懂词”，但实证结果并不意味着它总是显著更强。很多句子级任务上，字符级掩码和 WWM 的差距很小。这说明中文语义并不完全依赖预先确定的词边界，很多信息可以在上下文中由字符组合恢复。

## 工程权衡与常见坑

中文 tokenizer 的工程选择，本质是**鲁棒性、长度成本、词表质量**三者之间的权衡。

先看常见坑：

| 常见坑 | 问题表现 | 规避策略 |
|---|---|---|
| 词级误切 | 实体边界错、标签对齐错 | 保留字级后备，不把外部分词作为唯一入口 |
| OOV 问题 | 新词、错别字、口语写法映射失败 | 使用子词或字级 tokenizer |
| 词表污染 | 低质量网页词、垃圾词进入高频合并 | 训练前清洗语料，审核高风险 token |
| 过度压缩 | 少量罕见长 token 在微调时不稳定 | 常用汉字必须保留单字 token |
| 上下文过长 | 字级导致 token 数增加 | sliding window、摘要、检索增强 |

“词表污染”值得单独说明。  
如果一个 tokenizer 在中文语料中吸收了大量垃圾网页、成人站点或 spam 文本，某些不该高频出现的词可能被提前合并成单个 token。这样做的风险不是“模型一定输出脏内容”，而是**这些 token 的统计地位被抬高后，微调和推理的行为会变得更难控**。

一个真实工程例子是客服质检或医疗文本分类。  
假设训练集本来很干净，但底层 tokenizer 里某些敏感词已经是高频单 token。微调时，这些 token 的 logit 变化可能更剧烈。logit 可以理解为“模型给某个候选 token 的原始打分”。如果数据又小，模型就更容易在局部模式上过拟合。

处理方式通常不是“把所有敏感词删掉”，而是：

1. 训练前检查词表中的异常长尾 token。
2. 输入侧做规范化和清洗。
3. 对长文本使用局部窗口，减少单个异常 token 对整段的影响。

下面给一个极简的 sliding window 示例，说明如何在字级输入上控制长度：

```python
def sliding_windows(chars, window_size=6, stride=3):
    windows = []
    for i in range(0, max(len(chars) - window_size + 1, 1), stride):
        windows.append(chars[i:i + window_size])
    if chars and len(chars) < window_size:
        windows.append(chars)
    return windows

text = list("患者在华山医院接受检查")
wins = sliding_windows(text, window_size=6, stride=3)

assert wins[0] == ["患", "者", "在", "华", "山", "医"]
assert len(wins) >= 2
print(wins)
```

这种方法不高级，但在工程上很常用：**承认字级 token 较长，然后用窗口化策略弥补。**

## 替代方案与适用边界

如果任务目标不同，中文分词策略也应该变。

| 方案 | 适用场景 | 优点 | 风险 |
|---|---|---|---|
| 纯字级 | 分类、NER、纠错、短文本理解 | 最稳、边界清晰 | 长文本成本高 |
| 词级 + 子词 | 领域术语稳定、词边界较清楚 | 术语压缩更好 | 依赖切词质量 |
| SentencePiece | 统一训练 tokenizer，便于多语言 | 工具成熟 | 中文收益有限 |
| Byte-level BPE | 跨语言大模型、统一编码 | 覆盖强、无需显式 OOV | 中文压缩有限，需清洗词表 |

可以用一个简单决策树来判断：

- 是否必须跨语言统一？
- 如果是，优先考虑 BPE 或 byte-level BPE。
- 是否担心中文实体边界和误切？
- 如果担心，必须保留常见汉字的单字 token 作为后备。
- 是否追求极致上下文压缩？
- 如果追求，可以扩大词表，但要接受收益递减和清洗成本上升。

所以，像 GPT 系列或 LLaMA 系列这类大模型，即便使用更大的 BPE 词表，在中文上也通常不是“彻底抛弃字级”，而是形成“**字 + 常见组合**”的混合结构。公开经验里，中文大致做到 1.5 到 1.8 字/token 已经算不错，这离“每字一个 token”的上限并不远。

这也给出了适用边界：

- 如果你做的是中文分类、NER、句法相关任务，字级或字级主导方案通常更稳。
- 如果你做的是多语言生成、超长上下文、统一成本控制，BPE/字节级方案更现实。
- 如果你以为“中文一定要先分词再送进模型”，这个前提本身就不成立。
