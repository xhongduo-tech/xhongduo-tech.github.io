## 核心结论

CoT，中文常译为“思维链”，白话说就是让模型把“中间推理过程”也一起写出来。它能提升一部分题目的可读性，但不能保证逻辑正确。尤其在形式逻辑、反事实推理、否定推理这三类任务里，CoT 经常不是“纠错器”，而是“放大器”：原本就存在的偏差，会被一段看起来完整的推理文字包装得更自然、更有说服力。

更准确地说，很多大语言模型并不是先在内部完成抽象演绎，再把过程翻译成文字；更常见的情况是，它先生成一种“像推理的文本结构”，然后顺着这个结构把结论补全。于是，一旦题目稍微偏离训练时常见的表达模板，比如多一个否定词、换一种问法、加入反事实条件，输出就容易从“推得慢”变成“推错但很流畅”。

已有研究里，一个反复出现的现象是：在三段论和类似的逻辑任务上，CoT 并没有稳定达到高可靠性，常见准确率大致只有 60% 到 70%。这意味着即使题目形式很规整，模型也会在相当比例的样本上犯系统性错误。更关键的是，要求模型“再想一遍”或“反思一下”并不一定改善结果，有时反而让它更坚定地把错误写长。

下面这张表可以先把核心现象压缩成一个工程判断：

| 现象 | 直观表现 | 工程含义 |
|---|---|---|
| 训练内有效、训练外脆弱 | 熟悉模板下看起来会推理，换表述就掉点 | 不能把样例成功率当成逻辑能力 |
| 信念偏差 | 先接受“像是真的”结论，再倒推理由 | CoT 可能把错因写得更像对因 |
| 否定词敏感 | `not`、`unless`、`implausible` 一类词容易扰乱结论 | 文本重写、提示词润色都可能引入新错误 |
| 反思不稳定 | 重答、复核后不一定更准 | “让模型再想想”不是可靠补救 |
| 高流畅度低可验证性 | 推理文字完整，但无法形式化检查 | 高风险场景必须加外部校验 |

所以结论不是“CoT 没用”，而是“CoT 不能替代逻辑验证”。它更像一种输出格式优化，而不是形式推理引擎。

## 问题定义与边界

先定义边界。本文讨论的不是“模型会不会写一段解释”，而是“模型能不能在逻辑上稳定地从前提推出结论”。这两件事在自然语言里很像，但在工程上完全不同。

形式逻辑，白话说就是结论能不能只靠规则从前提必然推出，不依赖语气、常识联想和文字感觉。最典型的三段论可以写成：

$$
\forall x (M(x) \rightarrow P(x)),\quad M(a)\quad \therefore \quad P(a)
$$

这里的意思是：如果所有满足 $M$ 的对象都满足 $P$，并且对象 $a$ 满足 $M$，那么 $a$ 必然满足 $P$。这类题目的关键不是“写出几步文字”，而是有没有遵守推出规则。

问题在于，大模型常常不把任务当成“验证可推出性”，而是当成“生成一个最像正确答案的完成文本”。这就会引出两个边界问题。

第一，信念偏差。信念偏差，白话说就是人或模型更容易接受“听起来合理”的结论，而不是“逻辑上有效”的结论。比如：

- 前提 1：所有 3 的倍数都大于 2。
- 前提 2：6 是 3 的倍数。
- 结论：6 大于 2。

这是有效推理，因为结论确实由前提推出。

但如果写成：

- 前提 1：所有 3 的倍数都大于 2。
- 前提 2：6 是 3 的倍数。
- 问题：除非 6 等于 2，否则结论是否不成立？

很多模型会被“除非”“不成立”这些词牵着走，把一个简单的有效推理改写成混乱的判断。真正的逻辑系统会先把条件翻译清楚，再判断；语言模型往往先受表面措辞影响。

第二，表征敏感。表征敏感，白话说就是同一个逻辑问题，只要换一种写法，模型表现就明显波动。比如“哪个结论成立”和“哪个结论不可能成立”，语义上只是判断方向反过来，但模型准确率可能大幅变化。说明它处理的是语言表面模式，而不是稳定的抽象结构。

可以把这个失败路径简化成下面这样：

$$
\text{表面措辞} \rightarrow \text{先验倾向的结论} \rightarrow \text{补写中间步骤}
$$

而不是理想中的：

$$
\text{前提解析} \rightarrow \text{形式验证} \rightarrow \text{唯一允许的结论}
$$

这也是本文的边界：我们讨论的是 CoT 在“需要严格逻辑约束”的任务里为什么不可靠，而不是它在开放问答、解释生成、头脑风暴里有没有表达价值。

## 核心机制与推导

要理解 CoT 为什么会系统性失败，先要把“看起来在推理”和“真的在演绎”分开。

CoT 的典型流程可以写成：

$$
\text{Premises} \rightarrow \text{CoT steps} \rightarrow \text{Conclusion}
$$

很多人直觉上会认为，中间多了 `CoT steps`，说明模型多做了几步思考，所以更接近逻辑推理。问题是，这个箭头未必表示“内部演绎过程”，也可能只是“更长的文本生成过程”。如果中间步骤本身不是被约束出来的，而是被语言概率采样出来的，那么它们只是在增加解释长度，不是在增加正确性证明。

玩具例子最容易看清这个差异。

题目：

- 所有蓝色方块都是金属。
- A 是蓝色方块。
- 所以 A 是金属。

一个真正的规则系统只需要检查模式是否匹配：`所有 M 都是 P；a 是 M；所以 a 是 P`。匹配成功，结论成立。

而 CoT 常见的生成方式是：

1. “蓝色方块属于一个类别。”
2. “题目说这个类别的对象都是金属。”
3. “A 属于这个类别。”
4. “因此 A 可能是金属，结论大概率成立。”

这段文字表面上更长，但第 4 步已经悄悄把“必然成立”写成了“可能成立”。对自然语言读者来说不明显，对形式逻辑来说却是性质变化。也就是说，CoT 很可能把一个精确命题改写成模糊判断。

否定推理里这种问题更严重。否定词，白话说就是 `不`、`不是`、`不能`、`不可能`、`除非` 这类会反转条件方向的词。模型对这些词特别敏感，因为它们在语言里依赖上下文范围，稍有偏移就会翻转真值条件。

例如：

- 所有通过审核的请求都带签名。
- 请求 R 没有签名。
- 能否推出 R 没有通过审核？

这其实是经典的逆否命题问题。若形式化为：

$$
\forall x(\text{Approved}(x) \rightarrow \text{Signed}(x))
$$

那么由 $\neg \text{Signed}(R)$ 可以推出 $\neg \text{Approved}(R)$，这是有效的逆否推理。但模型若只在自然语言表面走，很可能写出：

“没有签名不一定等于没通过审核，因为题目没有明确说只有通过审核的请求才带签名。”

这段话听起来谨慎，实际上是错的。它把“若通过审核则带签名”误读成了“带签名是通过审核的充分必要条件”。CoT 不但没纠正这个错，反而把错解释得更像样。

再看反事实。反事实，白话说就是讨论“如果事实不是现在这样，会怎样”。这类题要求把真实世界状态和假设世界状态分开。语言模型常见错误是把两者混在一起。

比如：

- 事实：数据库当前是只读模式。
- 反事实前提：如果数据库不是只读模式，迁移脚本会成功。
- 事实：迁移脚本失败了。
- 问：能否由此推出数据库一定是只读模式？

答案是否定的，因为脚本失败可能有别的原因。可 CoT 往往会沿着“失败 -> 前提条件不满足 -> 数据库只读”这种最顺口的叙述一路写下去。这不是演绎，而是最短解释偏好。

所以，CoT 在逻辑任务中的系统性失败，不是偶发粗心，而是由三个机制共同推动的：

| 机制 | 白话解释 | 对结果的影响 |
|---|---|---|
| 模式补全 | 按“像推理”的文本模板续写 | 生成过程容易替代验证过程 |
| 信念偏差 | 先接受顺眼结论，再补理由 | 常识或措辞压过形式规则 |
| 表征敏感 | 同题换写法，内部表征就变 | 否定、反事实、量词下波动明显 |

因此，CoT 的核心问题不在“步数不够多”，而在“步骤没有被逻辑约束”。

## 代码实现

工程上正确的做法不是单纯要求模型“请逐步思考”，而是把 CoT 当成候选解释，再用外部验证器检查。外部验证器，白话说就是模型外的一段可执行规则程序，用来独立判断结论是否真的能从前提推出。

下面给一个可运行的玩具实现。它不追求覆盖所有逻辑，只演示一个关键原则：文本生成和逻辑验证必须分层。

```python
from dataclasses import dataclass
from typing import Dict, List


@dataclass
class UniversalRule:
    src: str
    dst: str


@dataclass
class Fact:
    entity: str
    cls: str


def verify_syllogism(rules: List[UniversalRule], facts: List[Fact], conclusion: Fact) -> bool:
    closure: Dict[str, set] = {}

    for fact in facts:
        closure.setdefault(fact.entity, set()).add(fact.cls)

    changed = True
    while changed:
        changed = False
        for entity, classes in list(closure.items()):
            for rule in rules:
                if rule.src in classes and rule.dst not in classes:
                    classes.add(rule.dst)
                    changed = True

    return conclusion.cls in closure.get(conclusion.entity, set())


# 玩具例子：所有 3 的倍数都大于 2；6 是 3 的倍数；因此 6 大于 2
rules = [UniversalRule("multiple_of_3", "gt_2")]
facts = [Fact("6", "multiple_of_3")]
assert verify_syllogism(rules, facts, Fact("6", "gt_2")) is True

# 不能反向推出：6 大于 2，不能说明 6 一定是 3 的倍数
rules = [UniversalRule("multiple_of_3", "gt_2")]
facts = [Fact("6", "gt_2")]
assert verify_syllogism(rules, facts, Fact("6", "multiple_of_3")) is False

# 真实工程味道的例子：通过审核的请求都带签名；请求 R 未带签名
# 这里只验证正向闭包，因此不能直接从“未签名”推出“未审核”，这需要单独实现逆否规则
rules = [UniversalRule("approved", "signed")]
facts = [Fact("R", "approved")]
assert verify_syllogism(rules, facts, Fact("R", "signed")) is True
```

这个例子故意很小，但它揭示了一个重要工程分层：

1. 模型负责把自然语言题目转成结构化前提。
2. 验证器负责判断结构化前提是否支持结论。
3. 只有验证通过，系统才输出“成立”。

如果把它写成伪代码，就是：

```python
premises = extract_premises(prompt)
candidate = model_generate_cot_and_conclusion(prompt)

if verifier_supports(premises, candidate.conclusion):
    return candidate.conclusion
else:
    return "无法由给定前提推出该结论"
```

真实工程例子可以看一个权限系统。

题目：

- 所有管理员都能访问审计日志。
- 用户 Alice 是管理员。
- 因此 Alice 能访问审计日志。

如果让模型直接 CoT，它大概率能答对。但真实系统不会因此就信任它。因为一旦题目变成：

- 所有管理员都能访问审计日志。
- Alice 当前会话未通过 MFA。
- 问：Alice 是否一定不能访问审计日志？

模型很容易开始混合“管理员权限”“MFA 状态”“当前会话策略”这些不同层次的信息，然后写出一段似是而非的解释。正确工程做法是把“角色规则”“会话约束”“资源策略”分别建模，再让验证器求交，而不是让 CoT 自由发挥。

所以代码实现的关键不是“生成更长的链”，而是“让结论必须经过可执行约束”。

## 工程权衡与常见坑

在工程环境里，CoT 最危险的地方不是偶尔答错，而是“错误被解释得很像正确”。这会降低人工审查的警觉性。

先看一个真实工程例子：医疗或合规问答。比如 HIPAA 这类隐私规则场景，任务要求不是“生成一段看起来合理的回答”，而是“绝不能输出违反规则的信息”。如果模型的优化目标还是完成文本，它会优先把答案写完整，再尽量补充合规措辞。CoT 会让这个过程更明显，因为它会把每一步都写出来，但这些步骤本身并没有经过规则引擎校验。

另一个常见例子是 word ladder。word ladder，白话说就是每一步只能改一个字母，从起点词走到终点词的单词接龙。人类解这类题时，核心是每一步都验证“是否只改了一个字母”。模型加了 CoT 后，往往会写：

“先把第一个字母改掉，再调整中间字母，最后得到目标词。”

问题是它写得很顺，却可能某一步一下改了两个字母。也就是说，它完成了“解释任务”，没有完成“规则遵守任务”。

常见坑可以压缩成下面这张表：

| 任务 | 主要约束 | 常见错误 | CoT 是否天然改善 |
|---|---|---|---|
| 三段论判断 | 结论必须由前提推出 | 把常识合理当成逻辑有效 | 否 |
| 否定推理 | 正确处理作用域和方向 | 把否定词范围读错 | 否 |
| 反事实推理 | 分清真实世界与假设世界 | 把两套状态混写 | 否 |
| 合规问答 | 严格遵守外部规则 | 先输出完整答案再补合规解释 | 否 |
| 规则游戏 | 每一步都满足局部约束 | 过程说明正确但中间步骤违规 | 否 |

这里有几个非常具体的坑：

第一，把 CoT 当可解释性证据。很多团队看到“模型把步骤写出来了”，就以为系统更透明。实际上，这只能说明它会生成解释文本，不能说明这些步骤是因果上真实使用过的内部依据。

第二，把重试当纠错。遇到错误时常见策略是“再问一次”或“让它反思”。这在开放生成任务里有时有帮助，但在逻辑约束任务里不稳定，因为模型可能只是换一种说法重复原来的偏差。

第三，用提示词修辞替代验证器。比如加“请仔细处理否定词”“请像数学家一样证明”。这可能局部改善，但本质仍是语言引导，不是规则执行。对于需要低错误率的系统，这种方法不够。

第四，忽略输入重写带来的分布漂移。产品里常见一个流水线：检索结果摘要 -> 提示词模板拼接 -> 模型 CoT 输出。任何一个摘要器或模板器对原问题做了轻微改写，都可能让模型掉出熟悉分布。最后用户看到的是同一个问题，系统内部其实已经换了表达形态。

所以工程判断应当很明确：只要任务要求“规则必须对”，CoT 就不能单独上生产。

## 替代方案与适用边界

如果 CoT 不够，替代方案是什么？不是简单地“不要解释”，而是把解释和验证分开。

第一类是 neuro-symbolic 方案。neuro-symbolic，白话说就是“神经网络负责感知和提取，符号系统负责严格推理”。例如先让模型把自然语言拆成大前提、小前提、候选结论，再交给逻辑模块检查。这样做的价值是：自然语言理解仍由模型承担，但“是否可推出”不再靠语言概率决定。

第二类是外部符号验证。符号验证，白话说就是用明确规则操作结构化表示，而不是靠词语感觉。它最适合模板化、约束清晰、可形式化的任务，比如权限规则、流程审批、配置依赖、有限范围数学题。

第三类是反事实和否定专项训练。这类方法的目标不是完全替代验证器，而是降低模型在特定表达上的脆弱性。它能减少错误，但通常不能把系统提升到“无需外部检查”的水平。

可以把几种策略做一个工程对比：

| 策略 | 透明度 | 泛化能力 | 部署成本 | 适用边界 |
|---|---|---|---|---|
| 原始 CoT | 表面透明，实则不可证 | 对表达变化敏感 | 低 | 解释型、低风险任务 |
| 分阶段 CoT / SR-FoT 类方法 | 比原始 CoT 更可审计 | 中等，仍受抽取质量影响 | 中 | 结构较稳定的逻辑问答 |
| 外部符号验证 | 高，可明确说明为何成立 | 对已建模规则最稳 | 中到高 | 高风险、强约束任务 |
| 完整规则引擎 | 最高 | 对规则内任务最稳，对开放语义最弱 | 高 | 合规、权限、配置、流程系统 |

它们的适用边界也很清楚。

当任务是“给用户讲清楚为什么”时，CoT 可以保留，但应该被视为展示层，而不是裁判层。

当任务是“答案必须符号正确”时，应该优先采用“模型抽取 + 验证器判定”的结构。

当任务既开放又高风险，比如医疗建议、财务合规、法律条款判断，仅靠 CoT 或改良版 CoT 都不够，需要额外检索、规则库、人工复核等多层约束。

换句话说，CoT 最适合做的是“帮助人看懂候选 reasoning”，最不适合做的是“独立承担最终逻辑责任”。
