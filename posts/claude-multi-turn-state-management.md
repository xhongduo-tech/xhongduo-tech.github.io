## 核心结论

Claude 的多轮对话状态，不是“脑子里有一块会自己增长的记忆区”，而是更接近“把已经算过的上下文中间结果缓存起来，再在尾部继续追加”。这里的 **KV cache**，白话说就是“把模型已经读过内容时产生的键和值向量先存起来，下次别重算”。因此，多轮状态延续的核心不是神秘记忆，而是两件事：

1. 固定前缀尽量不变，让缓存可复用。
2. 新消息不断追加，让模型基于旧缓存继续算。

可以把一段对话理解为下面这个结构：

`静态前缀（system/tool 说明） -> 历史消息 -> 本轮新消息`

如果静态前缀不变，Claude 不必每轮都从头重读前面所有 token，而是直接复用前面已经算好的 KV cache，只处理新增部分。这就是“状态叠加”，不是“每轮全量重算”。

但这不等于早期信息永远不会丢。随着轮次增加，早期指令离当前生成位置越来越远，注意力分布会逐渐偏向最近内容，甚至偏向少数“汇流点”token。这类现象常被称为 **attention sink**，白话说就是“注意力被吸到少数固定位置，远处信息虽然还在上下文里，但越来越不容易被真正用到”。

一个可操作的结论是：Claude 的多轮状态管理，本质上同时受两种约束控制。

| 约束 | 作用 | 结果 |
|---|---|---|
| KV cache 复用 | 降低重复计算成本 | 会话可以高效延续 |
| 长距离注意力衰减 | 早期信息权重下降 | 轮次变长后更容易偏离早期规则 |

玩具例子：第一轮你说“后续所有回答都用表格”，第十几轮你连续问了很多实现细节、报错、改写需求。即使这条规则仍在上下文里，模型也更可能优先盯住最近几轮内容，而不是最早那句全局规则。

真实工程例子：在 Claude Code 这类代理式产品里，系统模板、工具协议、输出约束通常放在固定前缀；真正变化的是用户消息、工具结果和 `<system-reminder>` 之类的动态块。这样做的目标不是“更聪明”，而是“让缓存命中并尽量稳住状态”。

## 问题定义与边界

这里要定义清楚“多轮对话状态管理”到底指什么。

**状态管理**，白话说就是“模型怎么在连续多轮里看起来记得前面说过的话，并且不用每次从零开始”。它通常包含三层含义：

1. 计算层：如何减少重复计算。
2. 表示层：历史信息如何继续参与当前注意力。
3. 工程层：哪些内容放前缀，哪些内容放尾部。

边界也要说清楚。Claude 的“状态”不是传统程序里的可随机读写对象，不像数据库里可以按 key 精确更新。它更像“整段上下文共同影响下一 token 概率分布”。所以它擅长延续风格、任务、约束，不擅长长期精确保存大量细碎事实。

下面这个表，可以用来理解轮次增长对指令遵循的影响。这里的 **PIF**，白话说就是“程序化指令遵循分数，用来衡量模型有没有按要求执行指令”。

| 轮次 | PIF 分数 | 现象 |
|---|---:|---|
| 第 1 轮 | 0.81 | 基本能稳定执行初始规则 |
| 第 10 轮 | 约中度下降 | 细节约束开始偶发漏掉 |
| 第 20 轮 | 0.64 | 早期规则稀释明显，偏离更常见 |

从第 1 轮到第 20 轮，分数从 0.81 降到 0.64，绝对下降 0.17，相对约下降：

$$
\frac{0.81 - 0.64}{0.81} \approx 21\%
$$

如果只把它粗略理解成工程经验，可以说：超过 20 轮后，指令遵循度进入了明显下滑区间，10% 到 20% 级别的退化是值得预期的。

这也是问题的真实边界：多轮状态管理不是“让模型永远记住”，而是在**语言建模能力、上下文长度、缓存命中率、响应成本**之间找平衡。

## 核心机制与推导

Claude 延续多轮状态的底层，仍然是 Transformer 的自注意力。**注意力**，白话说就是“当前 token 在决定下一步时，要从历史哪些位置取信息、取多少”。

最基本的注意力打分可以写成：

$$
Attention(i,j)=softmax\left(\frac{Q_i \cdot K_j}{\sqrt d} + sink\_bias_j\right)
$$

含义如下：

- $i$ 是当前位置。
- $j$ 是历史中的某个位置。
- $Q_i$ 是当前位置的查询向量，表示“我现在想找什么信息”。
- $K_j$ 是历史位置的键向量，表示“我这里存了什么信息”。
- $\sqrt d$ 是缩放项，防止数值过大。
- $sink\_bias_j$ 是附加偏置，用来调节某些位置更容易吸收注意力。

这里的 **sink bias**，白话说就是“给少数位置额外的吸引力，让低价值注意力有地方沉下去，别把整个分布搅乱”。它像一个汇流点：如果上下文很长、很多 token 都不重要，模型不必平均关注所有无效位置，而是把一部分无效权重导向少数稳定点。

玩具例子可以这样理解：

- 没有 sink 机制时，100 个历史 token 都会来竞争注意力。
- 其中 80 个其实只是过渡句、格式符、重复说明。
- softmax 仍然要给它们分配概率质量，真正重要的规则 token 会被挤薄。
- 有了 sink bias 后，一部分“无效关注”被导向固定位置，重要 token 的相对区分度更容易保住。

但 sink 只能缓解，不能消除长距离问题。因为另一个关键因素是**位置编码**，白话说就是“告诉模型每个 token 在序列里的相对位置，否则模型只知道词，不知道先后”。长上下文里常见的思路包括 RoPE、ALiBi 一类相对位置方案，它们试图让模型在距离拉大时，仍能维持位置感，不至于完全把远端信息当成噪声。

可以把它们理解为两个协同部件：

| 机制 | 作用 | 解决的问题 |
|---|---|---|
| sink bias | 给低价值注意力提供汇流点 | 减少注意力分布被无效 token 稀释 |
| 相对位置偏置（如 RoPE/ALiBi） | 维持远近关系表达 | 缓解长距离信息快速失真 |

所以，多轮状态为什么会退化，可以用一个简化推导来理解。假设一条早期系统规则在第 1 轮的重要性权重是 $w_0$，每增加一轮，其有效可访问性按某个衰减系数 $\alpha$ 下降，那么第 $t$ 轮时近似可写为：

$$
w_t = w_0 \cdot \alpha^t,\quad 0 < \alpha < 1
$$

这不是精确物理定律，但适合工程直觉：轮次越多、插入内容越杂、远端规则越容易被覆盖。Claude 通过训练和位置设计，把 $\alpha$ 调得没那么小；但只要 $\alpha < 1$，长会话里早期规则仍会逐渐失势。

真实工程例子：一个代码代理连续执行“读文件 -> 调工具 -> 解释错误 -> 生成补丁 -> 再读日志”二十多轮后，即使最开始系统里写着“只能修改指定目录，不可删除用户文件”，模型也更可能优先响应最近的错误修复目标，而不是最早的全局约束。因此工程系统会反复插入 reminder，或者在达到一定轮次后主动开新 session。

## 代码实现

工程实现上，最重要的不是“怎么模拟一个会记忆的模型”，而是“怎么组织 prompt，让缓存尽量稳定命中”。

最简结构如下：

1. 固定前缀：系统规则、工具定义、安全约束。
2. 可变尾部：用户消息、工具返回、会话摘要。
3. 缓存策略：前缀不变就复用，前缀变了就重建。

下面是一个可运行的 Python 玩具实现，用来模拟“固定前缀 + 追加消息 + cache 失效”的逻辑。它不是 Transformer 真正的 KV cache，只是帮助理解生命周期。

```python
from dataclasses import dataclass, field
from hashlib import sha256

@dataclass
class SessionState:
    static_prefix: str
    prefix_cache_key: str = field(init=False)
    turns: list[str] = field(default_factory=list)

    def __post_init__(self):
        self.prefix_cache_key = self._hash(self.static_prefix)

    def _hash(self, text: str) -> str:
        return sha256(text.encode("utf-8")).hexdigest()

    def append_turn(self, message: str) -> None:
        self.turns.append(message)

    def can_reuse_cache(self, new_prefix: str) -> bool:
        return self.prefix_cache_key == self._hash(new_prefix)

    def reset_with_new_prefix(self, new_prefix: str) -> None:
        self.static_prefix = new_prefix
        self.prefix_cache_key = self._hash(new_prefix)
        self.turns = []

    def render_context(self) -> str:
        return self.static_prefix + "\n" + "\n".join(self.turns)

# 固定前缀
prefix = "<system>只输出 JSON</system>\n<tools>search, edit</tools>"
state = SessionState(static_prefix=prefix)

# 多轮追加
state.append_turn("<user>给我 Python 例子</user>")
state.append_turn("<assistant>{\"lang\": \"python\"}</assistant>")

# 前缀没变，cache 可复用
assert state.can_reuse_cache(prefix) is True

# 前缀一旦变动，cache 失效
new_prefix = "<system>只输出 Markdown</system>\n<tools>search, edit</tools>"
assert state.can_reuse_cache(new_prefix) is False

# reset 后旧轮次清空
state.reset_with_new_prefix(new_prefix)
assert state.turns == []
assert "Markdown" in state.render_context()
```

这段代码要表达的核心只有三点：

- `static_prefix` 不变，缓存 key 就不变。
- 每轮只把新增消息 append 到尾部。
- 一旦改前缀，就要 reset，因为旧缓存与新前缀不再一致。

如果写成伪流程，可以更直观：

```text
初始化静态前缀
计算 prefix_cache

for 每一轮对话:
    if 静态前缀未变化:
        复用 prefix_cache
        只编码新增消息
        更新尾部 KV
    else:
        丢弃旧 cache
        从新前缀开始重建
```

真实工程里，一个常见结构是：

```text
[固定系统模板]
[固定工具协议]
[固定输出约束]
[可变 system-reminder]
[历史用户/助手消息]
[本轮工具结果]
[本轮用户新消息]
```

这里为什么常把 `<system-reminder>` 单独放可变区？因为它承担“提醒当前目标”的作用，但又不应污染静态前缀。如果把“当前时间”“当前轮次”“本轮工具摘要”都塞进前缀，那么每轮前缀都不同，缓存命中率会直接掉下来。

## 工程权衡与常见坑

多轮状态管理最大的误区，是把“能放进 prompt 的东西”都一股脑塞进系统前缀。这样做短期看规则更集中，长期看却会破坏缓存稳定性。

最常见的坑是下面这个对比。

| 做法 | 结果 | 是否推荐 |
|---|---|---|
| 动态信息放前缀 | 每轮前缀变化，cache 频繁失效 | 不推荐 |
| 动态信息放尾部 | 前缀稳定，可复用历史计算 | 推荐 |

这里的 **动态信息**，白话说就是“每轮都可能变的内容”，例如：

- 当前时间
- 当前轮次计数
- 最近一次工具调用结果
- 临时任务状态
- 可变业务参数

玩具例子：如果你把“今天是第 13 轮，请优先修复 bug A”写进系统前缀，那么第 14 轮就得改成“今天是第 14 轮”，这等于告诉模型“前面整段基础说明都变了”，缓存自然不能直接复用。

真实工程例子：一个代码代理如果把完整工具历史都塞进前缀，每次调用新工具都要重写前缀，导致两件事同时发生：

1. 性能退化，因为前缀重算。
2. 状态变脆，因为历史组织方式不断变化，模型更难稳定聚焦。

另一个坑是过度相信“上下文窗口够大就没问题”。窗口大，只表示“放得下”；不表示“远处信息仍有足够高的注意力权重”。因此，长对话中要主动做“状态整形”。

推荐做法可以压缩成下面几条：

- 固定前缀只放长期不变规则。
- 所有会变的信息放尾部追加。
- 关键约束隔若干轮重申一次。
- 达到较长轮次后，用摘要开新 session。
- 把高价值状态抽成结构化 memory，而不是依赖原始聊天全文。

其中“摘要开新 session”尤其重要。因为它本质上是在做一次**信息蒸馏**，白话说就是“把长对话里真正重要的规则和结论抽出来，重新放回近处”。这比让模型在二十多轮历史里自己找重点更稳定。

## 替代方案与适用边界

如果你的任务经常超过十几轮，单纯依赖 Claude 的原始多轮上下文，通常不是最稳方案。更可靠的是把“长会话”拆成“短会话 + 结构化状态”。

下面是几种常见替代策略。

| 策略 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| 长会话直接续写 | 短任务、状态简单 | 实现最简单 | 轮次长后退化明显 |
| 定期摘要 + 新 session | 中长任务、规则稳定 | 重置注意力，成本可控 | 摘要质量决定效果 |
| Retrieval 检索注入 | 外部知识多、文档长 | 按需取回，减少噪声 | 需要额外检索系统 |
| Slot-based memory 显式记忆 | 任务状态明确可结构化 | 可控、可校验 | 设计和维护成本更高 |

这里的 **Retrieval**，白话说就是“先去知识库里找相关片段，再把结果塞回 prompt”；**slot-based memory**，白话说就是“把重要状态存在几个明确字段里，比如当前目标、已完成步骤、禁止操作列表”。

玩具例子：写一篇长报告，不要把 30 轮草稿讨论都保留在一个会话里，而是每写完一章，生成“本章摘要 + 剩余待办”，再开新对话继续。这样后面的会话看到的是压缩后的高价值状态，而不是整串冗长历史。

真实工程例子：做代码代理时，可以维护这样一个显式状态对象：

- `goal`: 当前要修的 bug
- `constraints`: 不可改动的目录和接口
- `done`: 已完成步骤
- `open_questions`: 仍待确认的问题

然后每轮把这个状态对象注入尾部，而不是让模型自己从 200KB 历史消息里回忆。这样做的本质，是把“隐式记忆”变成“显式记忆”。

适用边界也很明确：

- 如果任务很短，直接多轮就够了。
- 如果任务很长但状态容易总结，优先“摘要 + 新 session”。
- 如果任务依赖大量外部资料，优先 Retrieval。
- 如果任务是代理执行、步骤清晰、约束严格，优先显式 memory。
