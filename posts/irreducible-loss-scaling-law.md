## 核心结论

扩展定律里的不可约损失项 $E$，可以理解为“这份数据本身自带、任何模型都必须支付的平均不确定性”。在常见写法

$$
L(N,D)=E+\frac{A}{N^\alpha}+\frac{B}{D^\beta}
$$

中，$L$ 是训练交叉熵，$N$ 是参数量，$D$ 是训练 token 数，$\alpha,\beta$ 是缩放指数，表示“规模增长后损失下降有多快”。其中只有后两项会随着模型和数据变大而下降，$E$ 不会。

对初学者，一个最重要的结论是：交叉熵不是“全都能靠堆参数降下去”。它可以拆成“数据熵”和“模型误差”两部分：

$$
\text{CrossEntropy}(p,q)=H(p)+D_{KL}(p\|q)
$$

这里的 $H(p)$ 是数据熵，白话说就是“真实数据自己有多难预测”；$D_{KL}$ 是 KL 散度，白话说就是“模型和真实分布还差多少”。扩展训练主要是在压低 $D_{KL}$，而 $E$ 对应的正是 $H(p)$ 那部分下界。

Chinchilla 一类工作里拟合出的常数项大约是 $E\approx 1.69$ nats/token，这个值常被当作通用参考。但它不是宇宙常数，只对特定训练分布成立。数据更规整、更冗余、更可压缩，$E$ 往往更低；数据更噪声化、更混杂、更难预测，$E$ 往往更高。真正做外推时，先测当前数据分布的 $E_i$，比直接套 1.69 更重要。

## 问题定义与边界

“不可约损失”这个词容易被误解。它不是“模型没训练好还剩下的误差”，而是“即使模型无限强，面对这份数据仍然存在的平均不确定性”。

边界要先划清：

| 概念 | 含义 | 会不会随模型变大下降 |
|---|---|---|
| $E$ | 数据分布自身的熵下界 | 不会 |
| $\frac{A}{N^\alpha}$ | 参数不足带来的可缩减误差 | 会 |
| $\frac{B}{D^\beta}$ | 数据不足带来的可缩减误差 | 会 |
| 总 loss | 上面三项之和 | 会下降，但有底 |

玩具例子最容易说明这一点。假设有两个同样大小的 token 数据集。

| 数据集 | 内容特征 | gzip 可压缩率 | 估计的 $E(H)$ 趋势 |
|---|---|---:|---:|
| 数据集 A | 大量模板化日志，重复字段多 | 4.2x | 较低 |
| 数据集 B | 混杂 OCR 错字、乱码、噪声片段 | 1.3x | 较高 |

“可压缩率”白话说就是“重复模式有多少”。一个数据集越容易压缩，通常说明冗余越多、预测越容易，因此其熵下界更低。反过来，如果数据本身像噪声，模型再大也只能学到“它就是不稳定”。

所以，1.69 nats 不能被机械套用到所有语料。网页语料、代码语料、合成问答、高噪声抓取文本，它们对应的 $E$ 可能都不同。文章讨论的边界是“训练分布上的平均 token 交叉熵下界”，不是某个单样本、某个任务指标、某个 benchmark 分数的统一下限。

## 核心机制与推导

标准扩展定律写成二元形式是：

$$
L(N,D)=E+\frac{A}{N^\alpha}+\frac{B}{D^\beta}
$$

三项的物理含义可以直接读出来：

| 项 | 作用 | 直觉解释 |
|---|---|---|
| $E$ | 下限项 | 数据自己就有的不可预测性 |
| $\frac{A}{N^\alpha}$ | 模型项 | 参数太少，装不下规律 |
| $\frac{B}{D^\beta}$ | 数据项 | 数据太少，规律没见全 |

如果取常见拟合参数

$$
E=1.69,\quad A=406.4,\quad \alpha=0.34,\quad B=410.7,\quad \beta=0.28
$$

并代入 $N=10^9,\ D=10^{12}$，可得：

$$
\frac{A}{N^\alpha}=\frac{406.4}{(10^9)^{0.34}}\approx 0.35
$$

$$
\frac{B}{D^\beta}=\frac{410.7}{(10^{12})^{0.28}}\approx 0.18
$$

因此

$$
L\approx 1.69+0.35+0.18=2.22 \text{ nats/token}
$$

这就是扩展定律最容易被忽略的地方：你现在看到的 2.22，不是“离 0 还有 2.22”；而是“离理论底部只剩约 0.53”。也就是说，继续扩大模型和数据，能压的空间已经只剩可缩减部分，回报会越来越小。

为了更直观看到边际收益递减，可以看一组玩具算例：

| $N$ | $D$ | 参数项 | 数据项 | 总 loss |
|---:|---:|---:|---:|---:|
| $10^8$ | $10^{10}$ | 0.77 | 0.65 | 3.11 |
| $10^9$ | $10^{12}$ | 0.35 | 0.18 | 2.22 |
| $10^{10}$ | $10^{13}$ | 0.16 | 0.09 | 1.94 |

从 3.11 降到 2.22 很明显，但从 2.22 再降到 1.94，投入更大、下降更少，因为越来越接近 $E$。

更进一步，很多工作会把常数项写成 $E(H)$，表示它依赖数据复杂度 $H$。这里的 $H$ 可以用压缩率、经验熵估计或更细粒度的质量指标近似。一个直观表达是：

$$
E(H)\approx c_0+c_1 H
$$

这不是唯一公式，而是一种工程近似，意思是“数据越复杂、越难压缩、越接近高熵源，底噪越高”。

跨数据集迁移时，还会用到 Loss-to-Loss 关系：

$$
L_1=K(L_0-E_0)^\kappa+E_1
$$

其中 $L_0$ 是旧分布上的 loss，$L_1$ 是新分布上的 loss，$K,\kappa$ 是拟合系数，$E_0,E_1$ 是两个分布各自的不可约损失。它的关键意义是：迁移时先把“可缩减部分”与“不可缩减部分”拆开，再讨论映射关系。否则，不同数据集的曲线会被常数项偏移混在一起。

真实工程例子是 FineWeb 和 StarCoder 这类跨分布预测。网页文本和代码文本并不是同一分布，直接拿一个分布上的总 loss 曲线外推到另一个分布，经常会失败。把 $E_0,E_1$ 分离出来后，才可能复用已有扫描结果，减少重跑大规模实验的成本。

## 代码实现

下面给出一个最小可运行实现。第一部分计算扩展定律；第二部分演示 Loss-to-Loss；第三部分给一个非常粗糙的“压缩率转 $E(H)$”示例。这里的压缩率映射只是教学用，不是正式估计器。

```python
import math

def estimate_loss(N, D, E, A, alpha, B, beta):
    """
    N: 参数量
    D: 训练 token 数
    E: 不可约损失，下界项
    A, alpha: 参数项系数与指数
    B, beta: 数据项系数与指数
    """
    if N <= 0 or D <= 0:
        raise ValueError("N and D must be positive.")
    return E + A / (N ** alpha) + B / (D ** beta)

def loss_components(N, D, E, A, alpha, B, beta):
    param_term = A / (N ** alpha)
    data_term = B / (D ** beta)
    total = E + param_term + data_term
    return {
        "E": E,
        "param_term": param_term,
        "data_term": data_term,
        "total": total,
    }

def predict_loss_to_loss(L0, E0, kappa, K, E1):
    """
    L0: 源分布上的 loss
    E0: 源分布不可约损失
    kappa, K: 迁移拟合参数
    E1: 目标分布不可约损失
    """
    if L0 < E0:
        raise ValueError("L0 must be >= E0, otherwise reducible part is negative.")
    return K * ((L0 - E0) ** kappa) + E1

def estimate_E_from_compression(compression_ratio, base_E=1.69, ref_ratio=2.5, slope=0.18):
    """
    教学用近似：
    压缩率越高，说明数据越冗余，E 越可能偏低。
    compression_ratio = 原始大小 / 压缩后大小
    """
    if compression_ratio <= 0:
        raise ValueError("compression_ratio must be positive.")
    return base_E - slope * math.log(compression_ratio / ref_ratio)

cfg = dict(E=1.69, A=406.4, alpha=0.34, B=410.7, beta=0.28)

parts = loss_components(1e9, 1e12, **cfg)
assert abs(parts["param_term"] - 0.356) < 0.03
assert abs(parts["data_term"] - 0.181) < 0.03
assert abs(parts["total"] - 2.227) < 0.05

# 更大的模型和数据，不应低于 E
bigger = estimate_loss(1e12, 1e15, **cfg)
assert bigger > cfg["E"]
assert bigger < parts["total"]

# Loss-to-Loss：目标分布不可约损失更高时，总 loss 会整体抬升
L1 = predict_loss_to_loss(L0=2.3, E0=1.69, kappa=0.9, K=1.1, E1=1.85)
assert L1 > 1.85

# 玩具版 E(H)：更高压缩率 -> 更低估计 E
E_clean = estimate_E_from_compression(4.0)
E_noisy = estimate_E_from_compression(1.4)
assert E_clean < E_noisy

print("baseline total loss:", round(parts["total"], 4))
print("larger-scale loss:", round(bigger, 4))
print("estimated E(clean):", round(E_clean, 4))
print("estimated E(noisy):", round(E_noisy, 4))
print("predicted target-domain loss:", round(L1, 4))
```

这段代码有两个使用原则。

第一，`E` 不是调参时随便拧出来的常数。它应当来自数据分布分析，或者来自该分布上的小规模拟合。第二，Loss-to-Loss 里的 `E0` 和 `E1` 也不是为了把曲线“拟合好看”而自由漂移的项，而是两个数据集自身的底噪估计。

对新手，一个可靠做法是先跑小模型扫描，测出几组 $(N,D,L)$，再固定候选的 $E$ 区间做拟合。如果发现所有曲线都系统性抬高或压低，再检查数据质量、分词器、去重策略，而不是直接把 1.69 当万能答案。

## 工程权衡与常见坑

最常见的坑，是把某篇论文里的 1.69 直接抄到自己的数据集上。这样做的问题不是“略有误差”，而是可能把超大规模外推整体做乐观了。

例如，一个团队在代码预训练项目里直接沿用通用网页语料或另一份代码语料的 $E$，然后发现外推出来的 loss 明显低得不合理。真正原因往往不是拟合公式错了，而是当前分布的 $E_i$ 更高，或者去重、清洗、分词策略改变了熵结构。把 StarCoder 一类分布的 $E$ 直接套到 FineWeb，或者反过来套，本身就缺少依据。

还要区分“$E$ 没变”和“KL 在下降”。如果你继续加算力后，loss 还在下降，但下降幅度越来越小，未必说明数据到头了，可能只是 $\frac{A}{N^\alpha}$ 与 $\frac{B}{D^\beta}$ 已经进入慢衰减区。这时真正要问的是：

1. 当前主导项是参数项还是数据项。
2. 数据质量是否让 $E$ 偏高。
3. 继续堆参数，是否只是更慢地逼近同一个底。

一个实用流程是：

1. 先测当前语料的压缩率、去重率、乱码率等质量指标。
2. 用小模型拟合本分布上的候选 $E_i$。
3. 再判断该优先增加参数、增加 token，还是先做数据清洗。
4. 跨数据集迁移时使用 $E_0,E_1$ 分离后的 Loss-to-Loss，而不是直接平移总 loss 曲线。

另一个坑是把训练 loss 的下限和任务能力下限混为一谈。$E$ 讲的是训练分布上的平均 token 不确定性，不等于“模型理解能力的绝对上限”。例如某任务指标可能仍随更好的对齐、采样、工具调用而变化，但训练交叉熵的扩展回报已经接近底部。这两个层面不能混写。

## 替代方案与适用边界

当数据量有限、数据质量波动很大，或者不同来源语料混合严重时，只用 $D$ 表示“数据规模”通常不够。更合理的做法是加入质量感知项 $Q$：

$$
L = E + \frac{A}{N^\alpha} + \frac{B}{D^\beta Q^\gamma}
$$

这里的 $Q$ 可以理解为“单位数据里有效信息的密度”。白话说，不是每个 token 都同样有价值。一个简单工程近似是把去重率、压缩率、格式完整度、语言纯度组合成质量分数，再让它修正数据项。

例如，如果一份数据 gzip 压缩率明显偏低，说明其随机噪声高、结构差，那么即使 token 数看起来很多，也不应按“高质量大语料”那样乐观估算。对初学者，可以先用粗糙规则：

$$
Q \propto \frac{\text{compressibility}}{D}
$$

它不严格，但能提醒一个事实：低质量数据的“有效样本量”往往小于名义 token 数。

Loss-to-Loss 也有边界。它适合“已经有源分布扫描曲线，且目标分布与源分布存在稳定映射”的场景。如果目标领域非常新，比如专业法律文本、少数语言、特定格式机器日志，那么即使公式成立，也仍需小规模实验先验证 $E_1$ 和映射指数 $\kappa$。没有这些先验时，迁移预测只能当假设，不能当预算依据。

所以，替代方案不是放弃 $E$，而是承认：在真实工程里，$E$ 常常应写成 $E(H)$，数据项也常常应写成带质量修正的形式。越接近异构、多源、强噪声数据，这一点越重要。

1. Hoffmann et al., *Training Compute-Optimal Large Language Models*  
   关键作用：给出 Chinchilla 风格的参数-数据联合扩展公式，是理解 $E + A/N^\alpha + B/D^\beta$ 的基础入口。

2. Subramanyam 等，ICLR 2026 相关工作  
   关键作用：强调不可约损失项与数据分布性质相关，并讨论质量感知扩展写法，适合进一步理解“为什么 1.69 不能乱套”。

3. EmergentMind, *Data Scaling Laws in ML*  
   关键作用：把不可约损失解释为数据熵下界，并讨论 $E(H)$ 与压缩率、复杂度估计的关系，适合从“固定常数”过渡到“分布依赖常数”。

4. Kempner Institute, *Loss-to-Loss Prediction*  
   关键作用：给出 $L_1 = K(L_0-E_0)^\kappa + E_1$ 这类跨分布预测方法，适合工程上复用已有训练扫描结果。

5. 信息论教材中关于交叉熵、熵、KL 散度的章节  
   关键作用：用于理解公式
   $$
   \text{CrossEntropy}=H(p)+D_{KL}(p\|q)
   $$
   这是“$E$ 为什么是不可约项”的理论根基。

推荐阅读顺序是：先读 Chinchilla 公式，建立三项分解直觉；再读数据扩展与 $E(H)$ 的讨论，理解为什么不同数据集的下界不同；最后看 Loss-to-Loss，把这些概念放到跨分布工程预测里。
