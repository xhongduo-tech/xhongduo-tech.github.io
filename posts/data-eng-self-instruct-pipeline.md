## 核心结论

Self-Instruct 是一种**用强模型给自己造训练数据**的方法。它的基本闭环很固定：先准备少量人工种子任务，再让强模型生成新指令，再为这些指令生成回答，最后把低质量、重复、跑题的数据过滤掉，剩下的数据用于指令微调。

它解决的问题不是“凭空创造知识”，而是“把少量高质量示例扩展成大量格式统一、任务多样的指令数据”。对零基础读者，可以把它理解成一个自动化数据工厂：人工先做出几个合格样板，模型再按样板批量生产，过滤器负责质检。

玩具例子很直接。假设你想让模型学会“教人做饭”，但手里只有 5 条人工写的指令，比如“解释如何煮米饭”“给出番茄炒蛋步骤”。接着让强模型基于这些样板再生成几十条类似但不完全重复的新指令，然后再让模型为每条指令写回答，最后删掉重复、太长、格式错误、语言不对的样本。剩下的就是可以拿去训练的指令数据集。

真实工程例子是 Stanford Alpaca。它用约 175 条人工种子任务，通过 `text-davinci-003` 自举生成约 52K 条指令-响应数据，再用于微调 LLaMA。Vicuna 等后续工程并没有推翻这条路线，而是在生成规模更大之后，进一步加强去重、相似性过滤和格式清洗，避免数据质量随着迭代轮次持续下降。

Self-Instruct 的关键不在“生成得多”，而在“多样性和质量同时受控”。多样性主要靠采样策略控制，尤其是温度、top-k、top-p；质量主要靠过滤策略控制，尤其是长度、格式、相似度、语言检测和相关性检查。没有前者，数据会过于单一；没有后者，数据会快速退化。

| 种子规模 | 候选指令生成量 | 过滤后可用量 | 典型用途 |
|---|---:|---:|---|
| 50 | 2,000 | 800-1,200 | 小规模验证流程 |
| 175 | 52,000 | 数万级 | 指令微调冷启动 |
| 500 | 100,000+ | 5万以上 | 多任务覆盖扩展 |

采样温度常用下面的公式描述：

$$
P(x_i)=\frac{\exp(\text{logit}_i/T)}{\sum_j \exp(\text{logit}_j/T)}
$$

其中 $T$ 是温度。温度高，概率分布更平，输出更发散；温度低，概率分布更尖，输出更稳定。

## 问题定义与边界

问题定义可以写得很明确：**在人工标注很少的情况下，如何扩展出足够多、足够多样、又足够规范的指令-响应对，用来训练一个更听指令的模型。**

这里的“指令调优”指的是让模型更擅长按照自然语言任务要求输出结果。它不是继续预训练，也不是知识注入。Self-Instruct 关注的是任务形式、回答风格和指令遵循能力。

它的输入通常只有两类：

| 阶段 | 输入 | 输出 |
|---|---|---|
| 种子准备 | 少量人工任务 | 高质量样板 |
| 指令生成 | 种子任务 + few-shot prompt | 新指令候选 |
| 响应生成 | 新指令候选 | 指令对应回答 |
| 过滤清洗 | 指令与回答候选 | 可训练数据 |
| 微调训练 | 清洗后的数据集 | 指令调优模型 |

边界同样重要。Self-Instruct 并不能保证“生成出来就是真实、正确、可上线”的数据。它特别容易遇到以下问题：

1. 单一采样策略会导致任务分布偏斜。
2. 种子太少时，模型只会围绕已有模板做近邻改写。
3. 多轮自举会把前一轮错误放大，形成质量退化。
4. 回答生成阶段可能语气像样，但事实不准。
5. 如果过滤只看长度，不看语义相似度，重复样本会大量混入。

新手版例子：如果你只有一条人工种子任务“帮我写一段总结”，那模型更可能生成“总结会议内容”“总结文章观点”这类近似任务，而不太可能跳到“写广告文案”或“设计 SQL 查询”。这说明 Self-Instruct 不是无中生有，它受种子分布强烈约束。

工程上常见边界参数大致如下：

| 条件 | 常见设置 | 风险 |
|---|---|---|
| 温度 $T<1$ | 0.5-0.8 | 更稳定，但多样性不足 |
| 温度 $T>1$ | 1.0-1.3 | 更多样，但噪声变大 |
| ROUGE-L 阈值 | `< 0.7` | 阈值过高会放入重复样本 |
| 指令长度下界 | 10-20 字 | 太短通常语义不完整 |
| 指令长度上界 | 120-200 字 | 太长常混入解释和约束噪声 |
| 语言检测 | 目标语言一致 | 混语种会污染训练集 |

因此，Self-Instruct 适合“冷启动”和“低成本扩充”，不适合直接替代高风险场景的人类标注。

## 核心机制与推导

Self-Instruct 的核心机制可以压缩成一条链路：

`seed tasks -> instruction generation -> response generation -> filters -> training data`

其中最关键的是两件事：**如何让新指令足够多样**，以及**如何让生成结果不过度失真**。

先看采样。采样就是模型在多个可能 token 中做选择。温度是控制随机性的参数，白话解释就是“让模型更保守还是更敢发散”。当 $T>1$ 时，原来概率低的 token 会被放大，生成结果更丰富；当 $T<1$ 时，高概率 token 更占优势，输出更集中。

但只调温度还不够。因为温度升高以后，长尾 token 会变多，容易引入胡乱扩展。于是通常还会配合 top-k 或 top-p：

- `top-k`：每一步只在概率最高的前 `k` 个 token 里选。白话说法是“把候选范围先截断”。
- `top-p`：每一步只保留累计概率达到 `p` 的最小 token 集合。白话说法是“保留足够可能的一小堆 token，而不是固定个数”。

top-p 的概念可以简写为：找到最小集合 $S$，使得

$$
\sum_{x_i \in S} P(x_i) \ge p
$$

然后只从这个集合里采样。它比 top-k 更自适应，因为不同位置的分布陡峭程度不一样。

下面这张表可以帮助理解三类采样参数和输出多样性的关系：

| 参数 | 变大后的效果 | 优点 | 风险 |
|---|---|---|---|
| 温度 `T` | 分布变平 | 任务更发散 | 噪声、幻觉增加 |
| `top-k` | 候选集合变大 | 词面变化增加 | 容易放入低质量 token |
| `top-p` | 保留更多累计概率质量 | 兼顾灵活和稳定 | 参数过高时接近无约束 |

玩具例子：如果模型看到种子任务“写一段产品总结”，在 `T=0.6` 时，它大概率会继续生成“总结活动复盘”“总结论文内容”；在 `T=1.2` 时，可能会扩展到“比较两种方案优缺点”“写一封回应投诉邮件”这类更远的任务。如果没有后续过滤，这种扩展既可能带来多样性，也可能带来跑题。

过滤阶段因此必不可少。常见过滤器一般组合使用，而不是只靠一个指标：

| 过滤器 | 白话解释 | 作用 |
|---|---|---|
| ROUGE-L | 看两个文本是否太像 | 去重、去近似改写 |
| SBERT 余弦相似度 | 看语义是否太接近 | 过滤语义重复 |
| 长度规则 | 看是否过短或过长 | 去掉不完整或啰嗦样本 |
| 格式检查 | 看是否符合目标模板 | 保证训练格式统一 |
| 语言检测 | 看是否是目标语言 | 防止中英混杂 |
| 关键词规则 | 看是否包含脏词、拒答模板等 | 去掉明显坏样本 |

其中 ROUGE-L 是基于最长公共子序列的重叠指标，白话解释就是“看两段文本在顺序上有多少相似片段”。它适合过滤字面近似的改写。SBERT 相似度更偏语义层面，能补足“字面不同但意思一样”的情况。

新手版阈值示例可以写成：先生成 1,000 条新指令，然后把每条都和已有种子、已有合格样本做相似度比较。若 ROUGE-L 大于 0.7，就认为太像了直接删除；若长度小于 10 字或大于 150 字，也删除；若不属于目标语言，也删除。最后留下来的样本既更像“新任务”，又更像“规范数据”。

真实工程例子里，Alpaca 之所以能用少量种子扩展到数万样本，不是因为模型天然可靠，而是因为 prompt 设计、采样策略和过滤规则一起工作。Vicuna 及后续项目继续加入更严格的语义相似度过滤，本质上是在解决同一个问题：迭代放大后的重复和漂移。

## 代码实现

实际工程里，Self-Instruct 一般拆成四个模块：读取种子、生成新指令、生成回答、过滤与记录。记录模块很重要，因为你后面调参时需要知道每条样本为什么被保留或被删除。

下面先给一个最小伪代码，结构和真实工程是一致的：

```python
seed_tasks = load_seeds()
new_instructions = gpt3.generate(seed_tasks, temperature=1.2, top_p=0.95)
responses = gpt3.generate(new_instructions, temperature=0.7)
filtered = filter_candidates(responses, rouge_lt=0.7, length=(20, 150))
save(filtered)
```

更接近可运行版本的玩具实现如下。它不调用真实大模型，只模拟“生成后过滤”的核心流程，但接口形状和工程思路是一致的：

```python
from difflib import SequenceMatcher

SEEDS = [
    "解释如何煮米饭",
    "给出番茄炒蛋的步骤",
    "总结一篇短文的中心思想",
]

CANDIDATES = [
    {"instruction": "解释如何煮米饭", "response": "先淘米，再加水蒸煮。"},
    {"instruction": "给出一份简单的炒青菜步骤", "response": "洗净青菜，热锅下油，大火快炒。"},
    {"instruction": "写", "response": "太短了"},
    {"instruction": "请用三句话总结下面的文章观点", "response": "先识别主题，再提炼论点，最后压缩表达。"},
]

def rouge_like(a: str, b: str) -> float:
    return SequenceMatcher(None, a, b).ratio()

def is_valid_length(text: str, low: int = 6, high: int = 60) -> bool:
    return low <= len(text) <= high

def is_duplicate(instr: str, existing: list[str], threshold: float = 0.7) -> bool:
    return any(rouge_like(instr, x) >= threshold for x in existing)

def filter_candidates(candidates, seeds):
    kept = []
    rejected = []

    for item in candidates:
        instr = item["instruction"].strip()
        resp = item["response"].strip()

        if not is_valid_length(instr):
            rejected.append((instr, "bad_length"))
            continue
        if not is_valid_length(resp):
            rejected.append((instr, "bad_response_length"))
            continue
        if is_duplicate(instr, seeds + [x["instruction"] for x in kept]):
            rejected.append((instr, "duplicate"))
            continue

        kept.append(item)

    return kept, rejected

kept, rejected = filter_candidates(CANDIDATES, SEEDS)

assert len(kept) == 2
assert any(reason == "duplicate" for _, reason in rejected)
assert any(reason == "bad_length" for _, reason in rejected)

print("kept:", kept)
print("rejected:", rejected)
```

这段代码体现了三个关键点：

1. 过滤器要串联，而不是单点判断。
2. 新生成样本不仅要和种子比，也要和已保留样本比。
3. 每条拒绝都要记录原因，否则后面无法调阈值。

模块拆分可以按下面的方式设计：

| 模块 | 输入 | 输出 | 作用 |
|---|---|---|---|
| 生成器 | 种子任务、prompt、采样参数 | 新指令或新回答 | 扩展候选数据 |
| 过滤器 | 候选样本、已有样本、阈值 | 保留/拒绝结果 | 质量控制 |
| 持久化模块 | 结构化样本、日志 | JSONL/CSV/DB | 审计与复现 |
| 监控模块 | 每轮统计数据 | 仪表盘或报告 | 发现退化趋势 |

真实工程例子可以这样落地：`seed.json` 存 175 条人工任务；第一阶段用较高温度生成新指令；第二阶段用较低温度生成回答，因为回答通常需要更稳定；第三阶段串上长度、语言、ROUGE、SBERT 和格式检查；第四阶段把每条样本的 `instruction`、`response`、`source_seed`、`generation_round`、`rejection_reason` 一并保存。

这里一个常见实践是“生成参数和过滤参数都要可调”。例如：

- 指令生成：`temperature=1.0~1.3`，`top_p=0.9~0.95`
- 回答生成：`temperature=0.3~0.8`
- 去重阈值：ROUGE-L `< 0.7`
- 长度限制：指令 `10~120` 字，回答 `20~400` 字

如果这些参数写死在代码里，后期调优成本会非常高。

## 工程权衡与常见坑

Self-Instruct 的工程本质不是“多生成”，而是“在成本、覆盖率、质量之间做平衡”。

第一个权衡是温度。高温采样会带来更多任务类型，适合早期扩展；但温度过高后，模型容易生成模糊指令、矛盾条件和无意义约束。低温采样更稳，但会让数据像模板改写机。常见做法是**分阶段调度**：早期生成指令时温度略高，回答生成时温度降低。

第二个权衡是过滤强度。过滤太松，数据污染训练集；过滤太严，有用的长尾任务被删掉。实际项目里经常不是“找一个完美阈值”，而是“给不同轮次设不同阈值”。第一轮扩展可以放松，后续迭代逐步收紧。

第三个权衡是迭代轮次。Self-Instruct 理论上可以一轮轮自举下去，但轮次越多，越容易把前一轮的模式偏差和错误格式复制下去。也就是常说的**质量退化**：样本数量上升，但任务新颖度下降、回答准确度下降、格式漂移增加。

| 问题 | 典型表现 | 对策 |
|---|---|---|
| 高温噪声 | 跑题、胡乱约束、事实漂移 | 降低温度，配合 top-p/top-k |
| 格式漂移 | 回答夹带解释、前缀不统一 | 强制模板检查 |
| 重复堆积 | 大量近义改写 | ROUGE + SBERT 双过滤 |
| 代间退化 | 越迭代越像旧数据重写 | 代间抽检，限制自举轮次 |
| 领域污染 | 混入非目标任务 | 分类器或关键词白名单 |

新手版例子：你一开始把 `T` 设成 1.2，结果生成很多“和做饭无关”的任务，比如写诗、做翻译。说明多样性虽然上来了，但任务边界失控。此时可以把 `T` 调到 0.8，同时增加“必须包含烹饪动作词”的规则，或者加一个语义分类器先筛掉不属于目标域的指令。

真实工程例子：如果你在客服场景中自举数据，模型可能会逐渐学会一种“过度礼貌但无信息量”的回答风格，比如每条都先说“很抱歉给您带来不便”。这种风格一旦进入训练集，会被后续微调放大。解决办法不是只靠 prompt，而是把“模板化空话比例”也纳入过滤指标。

工程流程可以概括成下面这条循环：

`生成 -> 过滤 -> 合入训练集 -> 质量监控 -> 调参 -> 下一轮生成`

每一轮至少要看三类统计：

1. 保留率：判断当前过滤是不是过严或过松。
2. 重复率：判断新样本是不是还在贡献增量信息。
3. 人工抽检准确率：判断模型是否在生成看似合理但实际错误的回答。

## 替代方案与适用边界

Self-Instruct 不是唯一方案，它只是“低人工成本扩充数据”的一条路线。是否该用它，取决于你手里的资源和业务风险。

最直接的替代方案是人工标注。优点是稳定、可控、质量高；缺点是慢、贵、覆盖速度低。第二类方案是检索增强生成，先从知识库或历史语料检索，再构造训练样本。第三类方案是直接用公开指令数据集做微调，例如 Dolly、OpenAssistant 或其他开放语料。

| 方案 | 适用场景 | 成本 | 质量稳定性 |
|---|---|---:|---|
| Self-Instruct | 冷启动、缺数据、有强模型 | 低到中 | 中 |
| 人工标注 | 高风险、高精度要求 | 高 | 高 |
| 检索增量构造 | 企业已有文档或工单 | 中 | 中到高 |
| 公开指令库微调 | 通用助手能力补齐 | 低 | 取决于数据源 |

Self-Instruct 最适合以下边界：

1. 新任务刚起步，几乎没有标注数据。
2. 有一个足够强的教师模型可用于生成。
3. 任务更重视“会不会做”而不是“事实是否零误差”。
4. 可以接受人工抽检，而不是逐条专家审校。

它不太适合以下边界：

1. 医疗、法律、金融合规等高风险领域。
2. 品牌口径极其敏感的客服或公关输出。
3. 需要强事实约束、强可追溯来源的任务。
4. 数据分布必须与真实用户请求高度一致的场景。

玩具例子：一家小团队要做“写客服回复”的模型，但没有现成数据。可以先写几十条种子任务，让强模型扩展出几千条样本，再人工抽检一部分，快速得到第一版训练集。  
真实工程例子：如果是医疗问答，Self-Instruct 可以用于生成“表达方式”和“问法多样性”，但最终答案必须经过专家规则或人工校对，否则高风险错误会被系统性写入训练集。

比较稳妥的 best practice 可以压缩成几条：

1. 先做 5,000 到 10,000 条候选，不要一开始就追求十万级规模。
2. 去重至少做两层：ROUGE-L 和语义相似度。
3. 指令生成与回答生成分开调参，回答温度通常更低。
4. 每一代都做人抽样，不要只看自动指标。
5. 当新增样本的重复率持续上升时，应停止继续自举或回补人工种子。
