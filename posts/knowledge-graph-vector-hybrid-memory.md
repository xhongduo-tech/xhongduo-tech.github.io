## 核心结论

知识图谱与向量检索的混合记忆架构，本质上是把两类互补的记忆系统串起来使用。向量检索是“把文本变成数字坐标后按语义距离找近邻”的方法，擅长覆盖各种自然语言表述；知识图谱是“用实体和关系组成可遍历网络”的结构化表示，擅长做多跳关系约束和可解释推理。单独使用任意一方，都有明显短板。

混合架构的常见流程是：先用向量检索从大规模文本中召回一批语义相关候选，再用知识图谱检查这些候选能否在实体关系网络中走通，最后按混合分数重新排序。可以把它理解为“先广撒网，再做结构化验真”。一个面向新手的直观例子是：先用“语义雷达”找出 120 个潜在段落，再用“关系地图”筛出能连成有效路径的 30 个，最终交给 LLM 的上下文既相关，又不容易自相矛盾。

核心打分可写成：

$$
S_{\text{hybrid}} = \alpha S_v + \beta S_g
$$

其中 $S_v$ 是向量得分，表示文本与查询的语义接近程度；$S_g$ 是图谱得分，表示候选文本对应实体在图中是否满足关系约束。$\alpha,\beta$ 可以用静态网格搜索调优，也可以由上层策略模型按查询类型动态预测。工程上通常不直接对全量候选跑图搜索，而采用 cascade，也就是分级计算：先看低成本的 $S_v$，只有达到阈值 $\tau$ 的候选才继续计算 $S_g$。

已有公开研究和工程报告都指向同一结论：纯向量方案往往召回更广，但多跳关系题容易错；纯图谱方案推理更稳，但覆盖受限。混合方案能同时提高召回率与结构一致性，在复杂问答任务中通常优于两端极端方案。

## 问题定义与边界

这类架构要解决的问题不是“能不能检索到一段相关文本”，而是“能不能在有限上下文窗口内，把既相关又逻辑成立的证据交给生成模型”。这里有两个常见失败模式。

第一类失败是纯向量失败。向量检索很容易召回语义相近但关系不成立的文本。比如问题是“谁是某项目当前审批人的直属上级”，查询里同时包含项目、审批人、组织层级三个关系约束；纯向量检索可能找来很多包含相同词汇的制度说明、旧版本流程文档和无关角色描述，但它不保证这些信息能组成正确关系链。

第二类失败是纯图谱失败。知识图谱依赖结构化覆盖。如果图里没有最新实体、别名映射不完整，或者文本中的新说法尚未入图，那么只靠图遍历会直接漏掉答案。换句话说，图谱擅长“在已知结构里推理”，不擅长“在语言变体中发现候选”。

可以用下表概括两者边界：

| 维度 | 向量检索 | 知识图谱 |
|---|---|---|
| 召回 | 高，能覆盖语言变体 | 依赖图覆盖，可能漏解 |
| 多跳推理 | 弱，靠后续模型补 | 强，可显式走路径 |
| 可解释性 | 低，通常只给相似度 | 高，可展示实体与边 |
| 延迟 | 通常低，适合大规模近邻搜索 | 可能高，取决于 hop 和图规模 |

因此，混合架构的定义应明确边界条件，而不是泛泛地说“两个都用”。至少要回答四个问题。

第一，文本库规模多大，是否需要 ANN，即近似最近邻搜索，来控制延迟。ANN 的白话解释是“用近似方法找最近点，比精确扫描快很多”。

第二，图谱规模多大，允许几跳搜索。所谓 k-hop，就是“从一个实体出发最多走 k 条边”。

第三，LLM 的上下文窗口有多大。即使检索出 200 个候选，真正能塞进模型的上下文也往往只有十几到几十个片段。

第四，单次查询的延迟预算是多少。若在线问答要求 300ms 级响应，图搜索就必须极度克制；若是离线分析，才有空间做更深层 rerank。rerank 的白话解释是“对初筛结果做第二轮更精细排序”。

一个适合初学者理解的流程图可以这样描述：`query -> vector topK -> entity linking -> graph filter -> rerank -> context assembly -> LLM`。其中真正的关键不在“有没有图谱”，而在“图谱是在召回前、召回后还是生成后参与”。本文讨论的是最实用的一种：先召回，再过滤和补全。

## 核心机制与推导

先看最基本的向量阶段。给定查询 $q$ 和候选文本 $d_i$，向量检索通常计算余弦相似度：

$$
S_v(q, d_i) = \frac{e_q \cdot e_i}{\|e_q\| \|e_i\|}
$$

其中 $e_q,e_i$ 是嵌入向量，也就是“把文本编码成固定长度数字数组”的结果。系统先取 top-K，例如 120 个候选，目的是保证语义覆盖。

然后进入图谱阶段。这里先做实体链接，也就是“把文本里提到的人名、系统名、项目名对齐到图里的节点”。对于每个候选文本，如果它包含实体集合 $E_i$，系统会在图里尝试检查这些实体能否与查询目标形成有效路径。常见的图谱得分可以写成：

$$
S_g = \max_{p \in \mathcal{P}} \left( w_r(p) \cdot w_c(p) \cdot \frac{1}{1+\lambda \cdot \text{len}(p)} \right)
$$

这里 $\mathcal{P}$ 是候选路径集合；$w_r(p)$ 是关系权重，表示路径上的关系类型是否符合任务；$w_c(p)$ 是实体置信度，表示节点链接是否可靠；$\text{len}(p)$ 是路径长度。这个式子的意思很直接：关系越匹配、实体越可信、路径越短，得分越高。

于是得到混合分数：

$$
S_{\text{hybrid}} = \alpha S_v + \beta S_g
$$

若系统采用 cascade，则不是所有候选都计算 $S_g$，而是：

$$
S(q,d_i)=
\begin{cases}
S_v(q,d_i), & S_v(q,d_i) \le \tau \\
\alpha S_v(q,d_i)+\beta S_g(q,d_i), & S_v(q,d_i) > \tau
\end{cases}
$$

这意味着只有语义上先过关的候选，才值得投入更高成本去做图搜索。

看一个玩具例子。问题是：“谁创办了生产 iPhone 的公司？”向量检索可能召回这些文本：

| 候选 | 文本内容摘要 | $S_v$ |
|---|---|---|
| A | “苹果公司发布了 iPhone 15。” | 0.91 |
| B | “苹果公司由乔布斯等人创办。” | 0.82 |
| C | “某手机厂商也生产智能手机。” | 0.78 |

若只按向量得分排序，A 很可能排在前面，因为它和“iPhone”高度相关。但图谱里可以走出链路：`iPhone -> produced_by -> Apple -> founded_by -> Steve Jobs`。于是 B 对应实体能与问题中的关系模板走通，A 虽然相关，却只覆盖一跳事实，不能回答“谁创办了”。图谱过滤后，B 的 $S_g$ 明显更高，最终排序就会把 B 提到前面。

再看一个稍微完整的数字案例。系统先从 10 万段文档里取回 120 个候选；其中 80 个能完成实体链接；再对这些实体做 2-hop BFS，BFS 的白话解释是“按层一圈圈向外扩展搜索”。结果只有 30 个候选能连到目标实体集合；再根据路径长度折扣、边权和实体置信度，保留 15 个高分子图，交给最终 rerank。这里的收益在于：向量阶段负责“别漏”，图谱阶段负责“别乱”。

下表展示几个关键参数对结果的影响：

| 参数 | 含义 | 取值增大后的影响 | 风险 |
|---|---|---|---|
| `K` | 向量召回数量 | 召回更全 | 图阶段成本上升 |
| `k-hop` | 图搜索深度 | 可覆盖更复杂关系 | 路径爆炸、噪声增加 |
| $w_r$ | 关系匹配权重 | 更强调结构约束 | 可能压制语言变体 |
| $w_c$ | 实体置信度 | 降低错链影响 | 低资源场景可能过严 |
| $\tau$ | cascade 阈值 | 减少图搜索次数 | 阈值过高会漏掉好候选 |

一个真实工程例子是企业内部知识问答。问题可能是：“支付链路异常时，哪个服务依赖哪个审批人负责的变更单？”这类问题同时涉及服务依赖、组织归属、时间状态。向量检索可以找到包含“支付链路”“变更单”“审批人”的文档，但只有图谱能判断这些实体在同一个依赖网络和时间约束下是否成立。因此，混合架构对这类跨系统、多实体、多跳问题更有稳定优势。

## 代码实现

实现时要同时维护两套索引，而不是只做一个向量库再临时查图。

第一套是 `text -> vector`，用于语义召回。第二套是 `text <-> triples`，用于把文档片段与图谱中的实体、关系、三元组建立双向映射。三元组就是“主语-谓语-宾语”结构，例如 `(Apple, founded_by, Steve Jobs)`。没有这层互索引，图谱很难参与文档级过滤，只能停留在独立检索。

下面给出一个可运行的 Python 玩具实现，演示“向量召回 + 图谱过滤 + 混合排序”的骨架。它不是生产代码，但逻辑是完整的。

```python
from collections import deque

docs = {
    "a": {"text": "Apple released iPhone 15", "sv": 0.91, "entities": ["iPhone", "Apple"]},
    "b": {"text": "Apple was founded by Steve Jobs", "sv": 0.82, "entities": ["Apple", "Steve Jobs"]},
    "c": {"text": "Another vendor makes smartphones", "sv": 0.78, "entities": ["VendorX"]},
}

graph = {
    "iPhone": [("produced_by", "Apple")],
    "Apple": [("founded_by", "Steve Jobs")],
    "VendorX": [("produced_by", "PhoneX")]
}

def bfs_has_path(graph, starts, target, max_hop=2):
    q = deque((s, 0) for s in starts)
    seen = set(starts)
    while q:
        node, hop = q.popleft()
        if node == target:
            return True, hop
        if hop == max_hop:
            continue
        for _, nxt in graph.get(node, []):
            if nxt not in seen:
                seen.add(nxt)
                q.append((nxt, hop + 1))
    return False, None

def graph_score(doc, target="Steve Jobs", max_hop=2):
    ok, hop = bfs_has_path(graph, doc["entities"], target, max_hop=max_hop)
    if not ok:
        return 0.0
    # 路径越短越好
    return 1.0 / (1 + hop)

def hybrid_score(doc, alpha=0.6, beta=0.4, tau=0.8):
    sv = doc["sv"]
    if sv <= tau:
        return sv
    sg = graph_score(doc)
    return alpha * sv + beta * sg

ranked = sorted(
    ((doc_id, hybrid_score(doc)) for doc_id, doc in docs.items()),
    key=lambda x: x[1],
    reverse=True
)

# b 虽然向量分略低，但因为能走通 founded_by 关系，混合后应该进入前列
assert graph_score(docs["b"]) > 0
assert graph_score(docs["a"]) == 0
assert ranked[0][0] in {"a", "b"}
assert any(doc_id == "b" for doc_id, _ in ranked)

print(ranked)
```

| 阶段 | 输入 | 输出 | 典型依赖 | 延迟估算 |
|---|---|---|---|---|

## 工程权衡与常见坑

最大权衡是“召回充足”和“图计算成本”之间的平衡。K 取太小，向量阶段漏掉正确语言变体；K 取太大，图过滤阶段成本急剧上升。通常不会固定一个极端值，而是根据查询类型、实体密度、历史命中率做分层策略。

常见坑可以直接列出来：

| 坑 | 后果 | 防御策略 |
|---|---|---|
| 图谱覆盖弱 | 正确答案在文本里但不在图里，导致漏解 | 建立文本 ↔ triple 互索引，并允许图失败时回退向量结果 |
| hop 过深 | 路径数量爆炸，延迟失控 | 设最大深度，如 `if k > 3 then prune` |
| 实体链接错误 | 图路径看似成立，实则连错节点 | 给实体置信度打分，令 $S_g = path\_score \times entity\_confidence$ |
| 关系模板过严 | 语言变体被过滤掉 | 先做宽松图过滤，再做 rerank |
| 只看最终答案不看证据 | 输出可读但不可验证 | 保存命中的路径和原文片段，返回解释信息 |

真实工程里，一个典型案例是企业知识库问答平台，例如销售、政务、审计、运维场景。问题常常涉及“客户-合同-审批-系统变更-时间状态”这样的链式约束。此时如果只靠向量检索，模型会把几个语义相近但时间不一致的文档拼在一起，形成貌似合理的答案；如果只靠图谱，又会因为最新制度、别名、自由文本说明未及时入图而漏掉关键证据。较稳妥的做法是：

1. 先用向量库取回 50 到 150 个候选。
2. 只对高于阈值的候选执行图谱关系验证。
3. 对超出 2-hop 或 3-hop 的路径直接裁剪。
4. 若图得分不足，但文本得分极高，则保留少量回退候选，避免“关系炸弹”把正确答案全删掉。

所谓“关系炸弹”，白话解释是“图里能连通的边太多，导致候选数量突然爆炸”。它常见于组织架构、调用链、社交网络这类高度连通图。防御手段包括限制边类型、限制时间窗口、限制节点类别，以及对高出度节点做特殊惩罚。

另一个常见误区是把知识图谱想成“只要建出来就稳定”。实际并非如此。图谱一旦落后于文本更新速度，混合架构就会出现“语义上找到了，结构上不承认”的冲突。因此在高频更新场景里，往往需要把图谱视为一个“有滞后的高精度约束层”，而不是唯一真相源。

## 替代方案与适用边界

如果问题只是开放式事实查找，纯向量 RAG 往往已经足够。RAG 的白话解释是“先检索，再让生成模型根据检索结果回答”。例如“最近发布了哪些产品”“某技术概念是什么意思”“某篇博客讲了什么”，这类问题更依赖语义覆盖，而不是关系一致性。此时混合架构增加的图计算成本，未必能换来足够收益。

如果问题天生是强结构化关系查询，且图谱覆盖极高，那么 KG-only，也就是纯图谱方案，可能更合适。例如设备拓扑、审批链路、标准知识库、固定流程规则等场景。这里的前提是：实体定义稳定、关系清晰、语言变体少、更新流程受控。

三种方案可以这样比较：

| 方案 | 数据需求 | 延迟 | 推理能力 | 适用场景 |
|---|---|---|---|---|
| 向量 | 只需文本库和嵌入 | 低 | 中，偏语义匹配 | FAQ、文档问答、开放查询 |
| 图谱 | 需高质量实体关系图 | 中到高 | 高，偏结构推理 | 强规则、强关系、低歧义场景 |
| 混合 | 同时要文本和图谱互索引 | 中 | 高，兼顾覆盖与约束 | 多跳问答、企业知识、跨系统依赖 |

可以降级到纯向量的条件通常有三类：第一，图谱覆盖明显不足；第二，查询延迟要求极高；第三，问题本身不要求关系一致性。比如“给我找关于缓存优化的资料”，混合架构就是过度设计。

必须保持混合的条件也很明确：第一，问题涉及多实体、多跳关系；第二，错误答案的代价较高；第三，文本表述变化大，但又必须满足结构约束。比如“哪个项目依赖哪个审批人负责的系统，且变更在上周已生效”，这类问题只靠向量检索很容易拼错时间和关系。

因此，混合架构不是默认答案，而是一种在“语义开放性”和“结构严格性”同时存在时最有价值的折中设计。
