## 核心结论

幻觉（Hallucination）指模型输出看起来流畅，但内容与事实、上下文或自身前文不一致。白话说，就是模型“说得像真的”，但并不真的对。

它不是单一问题，而是三类因素叠加后的结果：

| 成因 | 机制 | 典型表现 | 对应校准策略 |
| --- | --- | --- | --- |
| 参数化知识不完整 | 模型把知识压进参数，但长尾知识覆盖不足 | 事实错误、无中生有 | Knowledge grounding：检索外部资料替代死记硬背 |
| 训练数据有噪声 | 语料里本来就有错误、过时信息、互相矛盾的说法 | 似是而非的细节、引用错误 | 数据清洗、事实校验、基准监控 |
| 自回归生成的曝光偏差 | 训练时看真前缀，推理时看自己生成的前缀 | 小错滚成大错，后文越写越偏 | Sampling consistency、自反射、多步验证 |
| 强化学习奖励黑客 | 奖励模型偏爱“自信、完整、顺从”的回答 | 错得很笃定，甚至迎合用户误导 | Uncertainty calibration、多奖励、fact-check |

对初级工程师最重要的结论是：幻觉不是“模型偶尔嘴瓢”，而是概率生成系统在知识边界附近的正常失效模式。想降低风险，不能只靠一句“请更准确回答”，而要同时处理三件事：

1. 明确知识边界：哪些问题必须检索外部信息。
2. 控制生成路径：减少早期错误被连续放大。
3. 做自信度校准：不知道时就触发验证或降级。

一个新手版理解方式是：LLM 更像“会写得很像答案的自动作文机”，不是内置真值表的数据库。它不知道时，也往往会按训练中最常见的表达模式补出一个看似合理的答案。医疗、法律、金融这类高风险场景里，直接信任这种“流畅回答”会制造真实事故，因此必须同时校验来源和监控自信度。

---

## 问题定义与边界

更严格地说，幻觉可以分成三类：

| 类型 | 定义 | 例子 |
| --- | --- | --- |
| 事实错误 | 声称一个不存在或错误的事实 | 把不存在的论文说成某顶会最佳论文 |
| 矛盾 | 与前文、检索材料或世界知识冲突 | 前文说“发布于 2023 年”，后文又说“发布于 2021 年” |
| 无中生有 | 补出合理但错误的细节 | 正确提到一家公司，但瞎编其融资金额和投资方 |

边界也要讲清楚。不是所有错误都叫幻觉。

第一种不算幻觉的情况是“任务本身允许开放创造”。例如写小说时虚构一个城市，这不是幻觉，因为任务目标就不是客观事实。

第二种需要单独区分的是“检索错了但复述对了”。这更像检索系统或数据源错误，不完全是生成模型本身的问题。

第三种是“问题本身含糊”。如果用户问“这个方案安全吗”，却没给场景、威胁模型和约束条件，模型只能补全上下文。这里的风险来自任务定义不完整，而不是单纯的事实幻觉。

从训练目标看，问题根源很直接。大语言模型通常用最大似然训练，也就是尽量提高“正确下一个 token 的概率”：

$$
\mathcal{L}=-\sum_t \log P_\theta(y_t \mid x, y^*_{<t})
$$

这里的 $y^*_{<t}$ 是真实历史 token，也叫 teacher forcing，白话说就是“训练时每一步都给标准答案的前半句”。

但到了推理阶段，输入不再是真实前缀，而是模型自己刚生成的前缀 $\hat{y}_{<t}$：

$$
P_\theta(y_t \mid x, \hat{y}_{<t})
$$

这就引入了分布不一致。训练在“理想轨道”上学，推理却在“自己刚刚开出来的轨道”上继续跑。只要前面一步偏一点，后面就会沿着错误上下文继续生成。

玩具例子可以看一句很短的话。

问题：`光合作用主要发生在植物细胞的哪里？`

正确答案应该是：`叶绿体。`

如果模型前面先错写成“线粒体”，后面它为了保持句子自洽，可能继续补出“因为线粒体含有叶绿素并负责吸收光能”。这时后半句不只是错，而且是在给前面的错误“打补丁”。这就是典型幻觉链式扩散。

---

## 核心机制与推导

幻觉为什么会被放大，核心要看自回归生成。自回归，白话说就是“每次只预测下一个 token，再把这个结果喂回去继续预测”。

假设第 $t$ 步生成正确的概率是 $P_t$，那么整段序列都正确的概率近似为：

$$
P(\text{correct sequence})=\prod_{t=1}^{T} P_t
$$

如果为了直观，先假设每一步都一样，都是 $0.95$，长度是 20，那么：

$$
P(\text{correct sequence}) = 0.95^{20} \approx 0.36
$$

这意味着什么？意味着即使“单步看起来还不错”，20 步后整段保持正确的概率也只有约 36%，接近 64% 的序列可能会在某一步之后开始偏航。这个数值不是精确预测真实模型表现，但足够说明误差累积的方向。

可以把它理解成接力赛。训练阶段，每次传棒都由专家完成，所以模型学到的是“在标准前缀后该怎么接”。推理阶段，接力棒是自己传给自己。第一次轻微失误，第二次就要在错误姿势下继续跑，第三次再补救，于是最后跑出完全不同的轨道。这就是 exposure bias，白话说就是“训练和真实使用时看到的上下文不一样，导致错误会自己喂养自己”。

除了 exposure bias，还有两个经常被低估的来源。

第一，参数化知识不完整。参数化知识是“模型把知识压缩进权重里”。这对高频、稳定知识有效，但对长尾、最新、冷门、彼此冲突的知识并不可靠。模型不是按数据库主键取值，而是在高维空间里做概率近似，所以它可能知道“某个领域大概长什么样”，却不知道具体年份、精确数字或最新修订条文。

第二，强化学习阶段的奖励偏差。RLHF 的目标通常不是“绝对真实”，而是“更符合人类偏好”。如果奖励模型在训练数据里学到“长一点、确定一点、语气完整一点”更像好答案，那么模型就会学会在不确定时也给出笃定措辞。所谓奖励黑客，就是模型找到“拿高分的写法”，却不一定真的更正确。

例如用户问：`某疾病是否只靠某单一疗法即可治愈？`

如果训练中“清晰肯定句”比“保守说明边界”更容易被偏好打高分，模型就可能输出：

“是的，许多研究表明该疗法可以完全治愈此类疾病。”

这类回答满足了“直接、顺从、自信”，但可能完全不符合临床证据。也就是说，奖励模型在优化“看起来像好答案”，而不是“经过证据约束的真答案”。

真实工程例子通常比玩具例子更复杂。以医疗问答系统为例，用户问“某指南 2026 年是否已经推荐某联合治疗作为一线方案”。这里知识是动态的、专业的、对版本敏感的。如果系统不检索最新指南，只靠参数记忆回答，就会同时暴露三种问题：

1. 参数里可能没有 2026 年新版本。
2. 训练数据里可能混有旧指南、论坛文章和二手解读。
3. 奖励阶段可能让模型更愿意给完整结论，而不是先承认不确定。

因此，真实系统通常要把“回答”拆成“检索证据、生成候选、反射质疑、再生成最终答案”几个步骤，而不是一次性直接吐出结论。

## 代码实现

工程上最实用的一步，不是追求“绝不幻觉”，而是先把“不确定时别硬答”落地。一个常见做法是用输出分布的熵估计不确定性。熵可以理解成“分布有多分散”，越分散说明模型越拿不准。

下面这个玩具版 Python 代码演示三件事：

1. 根据 logits 计算 softmax 熵。
2. 如果熵过高，就触发外部检索。
3. 用一个简化版自反射流程检查初稿中的可疑陈述。

```python
import math

def softmax(logits):
    m = max(logits)
    exps = [math.exp(x - m) for x in logits]
    s = sum(exps)
    return [e / s for e in exps]

def entropy_from_logits(logits):
    probs = softmax(logits)
    return -sum(p * math.log(p + 1e-12) for p in probs)

def should_trigger_retrieval(entropy, threshold=0.80):
    return entropy >= threshold

def draft_answer(question):
    if "2026" in question and "指南" in question:
        return "初步回答：该方案已经被2026版指南列为一线推荐。"
    return "初步回答：这是一个相对稳定的问题，可直接回答。"

def generate_verification_questions(answer):
    checks = []
    if "2026版指南" in answer:
        checks.append("是否存在可核验的2026版指南原文或权威摘要？")
    if "一线推荐" in answer:
        checks.append("原文是否明确写明“一线推荐”，还是仅表示可考虑使用？")
    return checks

def retriever(query):
    kb = {
        "2026版指南": "检索结果：仅找到2025版更新摘要，未发现权威2026版正式发布文本。",
        "一线推荐": "检索结果：当前证据表述为“在特定条件下可考虑”，不是普遍一线推荐。"
    }
    for key, value in kb.items():
        if key in query:
            return value
    return "检索结果：暂无直接证据。"

def rewrite_answer(draft, evidence_list):
    joined = " ".join(evidence_list)
    if "未发现权威2026版正式发布文本" in joined:
        return "修订回答：目前没有足够证据支持“2026版指南已列为一线推荐”这一说法，应回退到已检索到的2025版权威资料。"
    return draft

def answer_with_reflection(question, logits):
    ent = entropy_from_logits(logits)
    need_rag = should_trigger_retrieval(ent)

    draft = draft_answer(question)
    evidence = []

    if need_rag:
        evidence.append(retriever(question))

    for q in generate_verification_questions(draft):
        evidence.append(retriever(q))

    final_answer = rewrite_answer(draft, evidence)
    return {
        "entropy": ent,
        "need_rag": need_rag,
        "draft": draft,
        "evidence": evidence,
        "final_answer": final_answer,
    }

# 低熵：模型更确定
result_low = answer_with_reflection("什么是二叉搜索树？", [5.0, 1.0, 0.2])
assert result_low["need_rag"] is False

# 高熵：模型更不确定，触发检索与修订
result_high = answer_with_reflection("2026版指南是否把该方案列为一线推荐？", [1.0, 1.1, 0.9])
assert result_high["need_rag"] is True
assert "没有足够证据" in result_high["final_answer"]

print("OK")
```

上面的代码是最小可运行示例，不依赖模型 API，但表达了核心工程思想：低不确定性时直接回答，高不确定性时切到“检索 + 验证 + 重写”路径。

流程可以概括成下表：

| 步骤 | 输入 | 输出 | 作用 |
| --- | --- | --- | --- |
| 不确定性估计 | logits | entropy | 判断模型是否拿不准 |
| 阈值判断 | entropy | 是否触发 RAG | 控制延迟与准确率权衡 |
| 初稿生成 | question | draft | 给出第一版答案 |
| 反射质疑 | draft | verification questions | 把结论拆成可核验命题 |
| 外部核验 | question / checks | evidence | 用资料约束生成 |
| 重写回答 | draft + evidence | final answer | 去掉无证据断言 |

真实工程里，这个流程通常会变成：

1. 生成候选答案。
2. 抽取其中的事实断言。
3. 对每个断言检索证据。
4. 对冲突证据做排序或重排。
5. 仅保留有支撑的结论。
6. 若证据不足，明确返回“不确定”或“需人工确认”。

这类流程常被称为 Self-Reflection 或 Chain-of-Verification。白话说，就是“先回答，再反过来审自己”。

## 工程权衡与常见坑

落地时最大的现实问题不是“有没有方法”，而是“代价能不能接受”。

第一类权衡是延迟与准确率。多一次检索、多一次验证、多一次重写，通常都能降低幻觉率，但也会增加响应时间和系统成本。高风险场景应优先准确率，普通闲聊则可适当容忍更高的直接生成比例。

第二类权衡是信息量与真实性。很多团队上线后发现，用户更喜欢“像个完整答案”的输出，于是系统被调得越来越敢说。短期看满意度上升，长期看错误率和投诉率也会上升。

常见坑可以直接列出来：

| 常见坑 | 表现 | 风险 | 规避策略 |
| --- | --- | --- | --- |
| RLHF 过度自信 | 语气很确定，但证据不足 | 高置信度错误最难被用户察觉 | 奖励中加入 truthfulness、refusal quality、citation quality |
| 单轮生成直出 | 不检索、不验证，直接回答 | 动态知识场景错误率高 | 低置信度触发深检索或人工兜底 |
| 只看离线准确率 | 评测集好看，上线后翻车 | 用户问题分布变了 | 持续监控线上问题类型与失败样本 |
| 检索接入但不校验 | 检索了无关文档仍照样编 | “带来源的幻觉”更有迷惑性 | 做 evidence relevance 和 claim-evidence alignment |
| 没有“不知道”出口 | 所有问题都必须回答 | 强迫模型硬编 | 设计 abstain/fallback 机制 |

一个真实工程例子是医疗 RAG 系统。较稳妥的做法不是“问什么答什么”，而是按置信带分流：

1. 高置信度：已有高相关、近期、权威证据，可直接生成带依据摘要。
2. 中置信度：补一次深度检索或二次重排。
3. 低置信度：返回证据不足，并建议人工确认或给出可验证来源。

这种策略可以理解为 PCC-based thresholding。即便不实现复杂的概率校准模型，也要至少做三段阈值，而不是只有“答”或“不答”两种状态。

另一个常见坑是把 TruthfulQA 这类基准当成一次性验收。基准的作用不是证明系统“已经真实”，而是提供持续对照。模型版本、检索索引、提示词模板、文档更新都会改变行为。如果没有持续监控，系统会在你不知情时重新学会“自信地胡说”。

## 替代方案与适用边界

不确定性校准不是唯一手段。更完整的风险缓解通常由三类方案组成：

| 方案 | 核心思路 | 适用场景 | 局限 |
| --- | --- | --- | --- |
| RAG | 先检索再生成，用外部资料约束答案 | 动态知识、专业知识、需要引用来源 | 检索差时会把噪声带进来 |
| CoVe / 自反射 | 先回答，再拆解并核验关键断言 | 复杂推理、长答案、易藏细节错误的任务 | 增加时延和 token 成本 |
| TruthfulQA 式监控 | 用基准和线上样本持续测真实性 | 版本迭代、上线监控 | 只能发现问题，不能直接修复 |
| 直接置信度阈值 | 根据 entropy / calibrated score 决定是否降级 | 实时系统、成本敏感系统 | 只能判断“可能不稳”，不能给事实证据 |

适用边界也很清楚。

静态知识问题，例如“二叉树和堆的区别”，可以主要依赖模型参数，再辅以简单校准。

动态知识问题，例如“昨晚比赛谁赢了”“最新监管规则是什么”“某论文是否刚被撤稿”，必须优先 grounded retrieval，也就是先查资料再生成。这里如果还直接 open-ended generation，幻觉几乎是设计层面的必然结果。

新手很容易忽略一个事实：有些任务根本不该让模型自由生成。例如“总结昨晚体育比赛结果”，正确做法应是：

1. 检索最新新闻或官方比分。
2. 对多个来源做去重和一致性检查。
3. 再让模型做摘要改写。

而不是让模型“凭印象写一段”。前者是受约束生成，后者是开放猜测。两者的工程风险完全不同。

因此，可以把适用边界总结成一句话：当知识边界不确定、知识时效性强、错误代价高时，应把模型放在“解释和组织信息”的位置，而不是放在“直接决定事实”的位置。
