## 核心结论

边缘模型协同推理的本质，不是把整模型复制到边缘节点，而是把**同一次推理流程**切成几段，让设备、边缘服务器、云分别承担自己最合适的那一段计算。这里的“协同推理”可以直白理解为：**同一个输入，不是在一个地方算完，而是在多个地方接力算完**。

它成立的前提也很明确：目标不是只追求最低平均时延，而是在**时延、带宽、算力、内存、精度、隐私**之间找到一个能上线的平衡点。很多系统离线测试很漂亮，线上一跑就失效，通常不是模型错了，而是部署点选错了。

一个最直观的玩具例子是手机拍照识别。手机先跑前几层，把原图压成更小的中间特征；再把这个特征发给附近边缘服务器继续推理；只有极少数复杂样本才送云端复核。重点不是“把图片传出去”，而是“**先算一段，再传更小的表示**”。如果传出去的数据比原图还大，这种协同通常没有收益。

| 方案 | 时延 | 带宽占用 | 隐私风险 | 部署复杂度 |
|---|---|---:|---|---|
| 纯本地 | 通常低且稳定，但受设备算力限制 | 最低 | 最低 | 低 |
| 纯云 | 计算强，但网络往返明显 | 最高，常传原始输入 | 最高 | 中 |
| 边缘协同 | 有机会同时降低时延与带宽 | 中等，取决于激活大小 | 中等 | 最高 |

边缘协同更像一个**系统设计问题**，不是单纯的模型结构问题。模型本身再先进，如果切分点、出口阈值、网络假设不合理，收益也会被通信和调度开销吃掉。

---

## 问题定义与边界

讨论对象必须先收窄：这里说的协同推理，指的是 **DNN 或 LLM 单次推理流程的部署分配**。DNN 是深度神经网络，可以白话理解为“由很多层算子串起来的预测函数”；LLM 是大语言模型，本质上也是分层计算图，只是结构和状态更复杂。

设模型共有 $L$ 层，切分点为 $s$。前 $s$ 层在设备执行，后面层在边缘或云执行。第 $s$ 层输出的中间激活大小记为 $a_s$，这里的“激活”可以直白理解为“前半段算出来、准备交给后半段继续用的数据”。网络有效带宽为 $B$，每层计算时延由设备端和边缘端各自决定。

协同推理的边界要讲清楚，否则概念很容易混淆：

| 术语 | 含义 | 是否属于协同推理 |
|---|---|---|
| 切分 | 把一次推理拆成多段，在不同节点执行 | 是 |
| 早退出 | 在中间层提前给出结果，不跑完整模型 | 是，常与协同组合 |
| 卸载 | 把某段计算交给别处执行 | 部分属于，视是否参与同一次推理切分 |
| 云推理 | 整个模型都在云端运行 | 否，本质是集中式推理 |
| 量化 | 用低比特表示权重/激活以降低成本 | 否，属于模型压缩 |
| 剪枝 | 删除不重要参数或通道 | 否，属于模型压缩 |
| 训练 | 更新模型参数 | 否，协同推理只讨论部署时执行 |

真实工程例子可以看工业视觉质检。相机采集图像后，工控机先跑本地前段，明显正常的样本直接判定；疑似缺陷样本送车间边缘服务器复核；极少数仍不确定的样本再上云。这里关注的是**推理时计算放在哪儿**，不是重新训练模型，也不是先量化再讨论部署。

因此，协同推理回答的问题是：**给定一个固定模型和当前系统条件，怎样选择计算位置，才能让一次推理更可部署。**

---

## 核心机制与推导

最基础的机制是**层级切分**。如果第 $1 \sim s$ 层在设备执行，第 $s+1 \sim L$ 层在边缘执行，那么总时延可以写成：

$$
T(s)=\sum_{i=1}^{s} t_i^{dev}+\frac{a_s}{B}+\sum_{i=s+1}^{L} t_i^{edge}+t_{ret}
$$

其中：
- $t_i^{dev}$ 是第 $i$ 层在设备上的执行时延；
- $t_i^{edge}$ 是第 $i$ 层在边缘上的执行时延；
- $\frac{a_s}{B}$ 是传输中间激活的时间；
- $t_{ret}$ 是结果返回时间，通常比上传小得多。

协同成立的条件，不是这个式子本身，而是它必须优于纯本地或纯云：

$$
T(s) < \min(T_{local}, T_{cloud})
$$

这就是为什么“切分”不是越多越好。每多一次跨节点传输，就多一次序列化、网络排队、协议开销和失败点。

最小数值例子可以直接代入。假设前 4 层本地执行共 $8\,ms$，第 4 层特征图大小 $120\,KB$，有效带宽 $20\,MB/s$，上传时间约 $6\,ms$；边缘端后半段执行 $10\,ms$，结果返回 $1\,ms$。那么总时延为：

$$
T(4)=8+6+10+1=25\,ms
$$

如果纯本地完整运行需要 $38\,ms$，纯云上传原图和远程推理共 $42\,ms$，那么切到第 4 层就是合理的。

再进一步，可以引入**早退出**。早退出直白说，就是“样本一旦已经足够确定，就在中途直接结束，不再跑后面的层”。设第 $k$ 个出口的触发概率为 $p_k$，触发条件为 $\text{conf}_k \ge \tau_k$，其中 $\text{conf}_k$ 是当前出口置信度，$\tau_k$ 是阈值。期望时延可写成：

$$
E[T] = \sum_{k=1}^{K} p_k \left(\sum_{i=1}^{k} t_i + \frac{a_k}{B}\right) + p_{K+1}\sum_{i=1}^{L} t_i
$$

这里的意思很简单：不同样本走不同路径，平均开销是各路径开销按概率加权后的结果。

继续看刚才的数值例子。若 60% 的样本在第 4 层早退，只花本地 $8\,ms$；其余 40% 继续走协同路径 $25\,ms$，则期望时延为：

$$
E[T]=0.6\times 8 + 0.4\times 25 = 14.8\,ms
$$

新手最容易忽略的一点是：**平均更快，不代表每个样本都更快**。早退出之所以有效，是因为简单样本少跑了后半段，复杂样本仍然走完整路径。

路径分解可以画成下面这样：

```text
输入
  |
设备前段计算（1...s层）
  |
中间激活 a_s
  |
网络传输 a_s / B
  |
边缘后段计算（s+1...L层）
  |
结果返回 t_ret
  |
输出
```

这个结构对 DNN 很自然，对 LLM 也存在类似问题，只是 LLM 往往还要考虑 KV cache、token 级流水和会话状态，部署难度会更高。

## 代码实现

工程实现的核心不是复现论文伪代码，而是三件事：**枚举切分点、估计各路径时延、按规则选择最优路径**。下面给一个可运行的简化 Python 示例，包含切分搜索和早退出判断。

```python
from math import isclose

dev_time = [2, 2, 2, 2, 6, 6]          # ms
edge_time = [1, 1, 1, 1, 5, 5]         # ms
activation_kb = [800, 420, 260, 120, 90, 20]
bandwidth_kb_per_ms = 20               # 约等于 20 MB/s
return_ms = 1

def total_latency(split_layer: int) -> float:
    local_t = sum(dev_time[:split_layer])
    transfer_t = activation_kb[split_layer - 1] / bandwidth_kb_per_ms
    remote_t = sum(edge_time[split_layer:])
    return local_t + transfer_t + remote_t + return_ms

def choose_best_split():
    best_split = None
    best_t = float("inf")
    for s in range(1, len(dev_time) + 1):
        t = total_latency(s)
        if t < best_t:
            best_t = t
            best_split = s
    return best_split, best_t

def early_exit(confidence: float, threshold: float) -> bool:
    return confidence >= threshold

best_split, best_t = choose_best_split()
assert best_split == 4
assert isclose(best_t, 25.0)

simple_sample_conf = 0.92
tau = 0.90
assert early_exit(simple_sample_conf, tau) is True

hard_sample_conf = 0.73
assert early_exit(hard_sample_conf, tau) is False

expected_latency = 0.6 * 8 + 0.4 * 25
assert isclose(expected_latency, 14.8)

print(best_split, best_t, expected_latency)
```

这段代码故意保留了最关键的决策逻辑：
- `split_layer`：切分点；
- `activation_kb`：中间激活大小；
- `bandwidth_kb_per_ms`：有效带宽；
- `tau`：早退出阈值。

一个对应的配置可以长这样：

```python
config = {
    "split_layer": 4,
    "bandwidth": "20MB/s",
    "early_exit_thresholds": [0.90],
    "fallback": "cloud_when_edge_busy"
}
```

| 项 | 含义 |
|---|---|
| 输入 | 原始样本，如图像、音频、文本 token |
| 输出 | 分类结果、检测框、生成 token 等 |
| 关键变量 | `split_layer`, `activation_size`, `bandwidth`, `tau` |
| 决策规则 | 选使总时延最小且满足精度约束的路径 |
| 早退规则 | 当 `confidence >= tau` 时提前返回 |

玩具例子里，所有层时延都是固定值，真实工程不会这么稳定。真实系统通常还需要：
- 预先 profiling 每层在设备和边缘上的时延；
- 在线采样网络带宽和边缘负载；
- 用校准集确定早退出阈值，而不是拍脑袋设 `0.9`。

真实工程例子可以看车间质检线。工控机上部署轻量 backbone 前段，边缘机房有更强 GPU 负责后段。系统每隔几秒更新一次边缘负载和网络状态；当边缘繁忙时，提高本地早退比例；当网络拥塞时，改走本地轻模型兜底。这里已经不是“固定切一刀”，而是“按在线状态动态选路径”。

## 工程权衡与常见坑

第一类坑是**中间激活过大**。很多人直觉上以为“越早切越省设备算力”，但越早切通常激活越大，甚至比原图还大。比如一张压缩 JPEG 原图只有 150KB，第一层卷积后的特征图展开后却有 600KB。如果你传出去的数据比原图还大，协同基本就失去意义了。

第二类坑是**只看平均时延，不看尾延迟**。平均值好看，不代表线上稳定。对于交互式系统，用户体验更常被 $p95/p99$ 时延决定。网络偶发抖动、边缘排队、批处理干扰，都会让极端样本变慢。

第三类坑是**把置信度当成正确率**。置信度是模型“自认为有多确定”，不是“它实际上有多对”。所以早退阈值必须用校准集验证，否则会出现“很自信地错”。

第四类坑是**静态 profiling 直接上线**。离线测得的层时延和带宽只是一个快照。边缘节点一旦被别的任务抢占，原来的最优切分点可能立刻失效。

第五类坑是**把切分、量化、剪枝混成一个问题**。切分关注计算位置；量化和剪枝关注模型体积与算力成本。它们可以组合，但验证维度不同，不能混着下结论。

| 常见错误 | 后果 | 规避方法 |
|---|---|---|
| 中间激活过大 | 传输时间吞掉协同收益 | 优先选择激活显著压缩后的层作为切分点 |
| 只看平均时延 | 线上尾延迟失控 | 同时监控 `p95/p99`，按坏情况选策略 |
| 早退阈值未校准 | 精度明显下降 | 用独立校准集调 `tau`，按任务容错设阈值 |
| 在线资源变化未重估 | 最优切分点迅速过期 | 定期重估带宽、负载、层时延 |
| 把切分和量化混为一谈 | 设计目标混乱，验证失真 | 分别评估部署收益与精度影响 |

一个常见误区是：看到协同推理论文里“平均时延下降 40%”，就以为上线也能下降 40%。实际系统里，收益经常取决于最不稳定的那一段，通常是网络和共享边缘资源，而不是模型本身。

## 替代方案与适用边界

协同推理不是默认优解。它更适合下面这类场景：**附近存在可用边缘节点，网络总体可用，样本复杂度差异大，且纯本地算不动完整模型**。如果这些条件不成立，纯本地、纯云或直接量化模型，往往更实用。

先看三类典型场景。

第一类是移动端轻量分类，比如相册二分类、小型 OCR。模型本身就很小，设备可以稳定跑完，最优方案常常是纯本地。因为你为了省几毫秒去引入网络依赖，通常不划算。

第二类是离线批量分析，比如夜间日志图像归档、批量语义分析。时延不敏感，吞吐和统一管理更重要，纯云更合适。

第三类是工业质检、车载感知、AR 识别。它们往往既要比纯云更低时延，又比纯本地更强算力，而且样本复杂度差异明显。这时边缘协同最有价值。

| 场景特征 | 推荐方案 | 原因 |
|---|---|---|
| 小模型、极低延迟要求 | 纯本地 | 网络不确定性比计算更贵 |
| 高带宽但边缘能力弱 | 纯云 | 协同收益小，集中推理更简单 |
| 隐私敏感、数据不宜外发 | 本地优先，必要时协同 | 先在端侧提特征可减少原始数据暴露 |
| 样本复杂度高度不均 | 边缘协同 + 早退出 | 简单样本快速结束，复杂样本走远端强算力 |

因此，判断标准不是“部署越分散越先进”，而是“**当前环境下，哪种部署最稳、最省、最符合约束**”。协同推理的适用边界，恰恰来自它引入的额外复杂度：多节点调度、链路监控、故障回退、阈值校准、路径选择。这些都是成本。
