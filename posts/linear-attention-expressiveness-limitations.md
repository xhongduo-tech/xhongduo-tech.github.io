## 核心结论

线性注意力的本质，不是“更快的 softmax 注意力”，而是“把历史信息压进一个固定大小状态里的线性递归模型”。术语先解释：这里的“状态”就是模型在读完前面所有 token 后手里还保留的那份内部记忆，而不是完整历史本身。

它的核心递推可以写成：

$$
M_t = M_{t-1} + \phi(k_t) v_t^\top,\qquad
z_t = z_{t-1} + \phi(k_t),\qquad
o_t = \frac{M_t^\top \phi(q_t)}{z_t^\top \phi(q_t) + \varepsilon}
$$

其中 $M_t \in \mathbb{R}^{d_\phi \times d_v}$ 是记忆矩阵，$z_t \in \mathbb{R}^{d_\phi}$ 是归一化辅助状态，$q_t,k_t,v_t$ 分别是查询、键、值向量，$\phi(\cdot)$ 是特征映射。若取 $\phi(x)=x$，就回到最直观的线性形式。

结论有三点。

第一，线性注意力等价于一个状态维度固定的线性 RNN。RNN 是“每步读一个输入，再更新一次隐藏状态”的模型；这里的隐藏状态就是 $M_t,z_t$。不管上下文长度是 $128$、$4{,}096$ 还是 $1{,}000{,}000$，状态大小都不随长度增长。

第二，固定状态意味着固定容量。容量可以白话理解成“能同时装下多少彼此不干扰的信息”。复制、精确检索、键值回忆这类任务，本质上需要把很多独立条目分别记住；当需要记住的信息量接近或超过状态容量时，新信息会和旧信息叠加，查询拿到的是混合信号，而不是单条记忆。

第三，线性注意力的问题不只是训练技巧不够，而是表达力边界就在那里。softmax 注意力在查询时仍然保留“当前查询逐项查看历史”的机制；线性注意力则先把历史压成一次线性外积和，再让查询去读这个压缩态。因此在 copying、retrieval、MQAR 这类要求“按地址精确取回某一项”的任务上，原始线性注意力天然更弱。BASED 用二阶 Taylor 近似加局部窗口去补短程表达力，HGRN2 用更大的外积状态去补容量，但它们都是“修补”，不是把原始线性注意力变成完整 softmax。

## 问题定义与边界

先把问题说清楚。本文讨论的不是“线性注意力能不能工作”，而是“它在什么任务上会因为固定状态而系统性吃亏”。

最典型的是两类任务。

1. copying：前面出现一个片段，后面要求原样复现。
2. retrieval：前面存了多组 key-value，后面给一个 key，要求准确返回对应 value。

这两类任务共同要求一件事：历史中不同位置的信息要能被分开保存，并在查询时只取回目标项。术语“分开保存”白话讲，就是不能把十条记忆搅成一锅粥，再希望查询时刚好舀出其中一条。

线性注意力的问题就在这里。它把所有历史压成：

$$
M_t=\sum_{i=1}^t \phi(k_i) v_i^\top,\qquad z_t=\sum_{i=1}^t \phi(k_i)
$$

所以历史不是按“条目列表”保存，而是按“总和”保存。总和一旦形成，就只剩一个固定大小矩阵和一个固定大小向量，无法显式区分第 3 条和第 37 条是谁贡献的。

把读出公式展开，这个问题会更明显：

$$
o_t
=
\frac{M_t^\top \phi(q_t)}{z_t^\top \phi(q_t)+\varepsilon}
=
\frac{\sum_{i\le t} v_i \, \langle \phi(k_i),\phi(q_t)\rangle}
{\sum_{i\le t} \langle \phi(k_i),\phi(q_t)\rangle+\varepsilon}
$$

这个式子说明两件事。

第一，输出始终是所有历史 value 的加权混合。  
第二，若两个 key 在特征空间里方向接近，它们对同一个查询会同时拿到较大权重。

下面这张表概括边界：

| 机制 | 状态维度 | 查询时是否逐项看历史 | 记忆方式 | 对精确检索的能力 |
|---|---:|---|---|---|
| 线性注意力 | $O(d_\phi d_v + d_\phi)$，固定 | 否 | 外积和的压缩状态 | 容易受干扰 |
| softmax 注意力 | KV cache 随上下文增长 | 是 | 保留逐 token 键值条目 | 更强 |
| 扩展状态的线性 RNN | $O(s)$，$s$ 可放大 | 否 | 更大压缩状态 | 比原始线性注意力好，但仍非完整 softmax |

这里“查询时是否逐项看历史”比“交互阶”更容易理解。softmax 注意力到查询时，历史条目还在；线性注意力到查询时，历史已经先压缩过了。

再给一个最小玩具例子。设 $d_\phi=2$，状态只有一个 $2\times2$ 矩阵，也就是 4 个数。现在连续读入 3 个 token，每个 token 都往矩阵里加一个外积 $\phi(k_t)v_t^\top$。到第 3 步时，矩阵只剩 4 个混合后的数字，但历史里原本有 3 组键值信息。查询若想“只把第 1 个 token 的 value 还原出来”，本质上是在一个已经叠加混合的系统里做逆向分离。只要这些向量不是理想正交，干扰几乎不可避免。

这个边界可以再写得更硬一些。若要记住 $N$ 条彼此可区分的项目，而状态自由度与 $N$ 无关，那么当 $N$ 增大时，平均到每条项目上的可分离空间会下降。它不保证在某个固定长度上立刻失败，但保证了随着负载增加，冲突概率会上升。

因此本文的边界非常明确：

| 任务类型 | 主要要求 | 原始线性注意力是否适合 |
|---|---|---|
| 局部语言建模 | 近邻依赖、流式处理 | 通常适合 |
| 粗粒度摘要 | 历史聚合、主题压缩 | 通常适合 |
| needle 检索 | 在长历史里找一个精确条目 | 往往不够 |
| MQAR / key-value recall | 多组地址到值的精确映射 | 往往不够 |
| 法律/财务精确问答 | 条款号、数值、实体关系不能串 | 风险较高 |

所以本文讨论的不是“线性注意力有没有用”，而是“它在远距离精确回忆任务上为什么会系统性吃亏”。

## 核心机制与推导

从公式上看，线性注意力之所以快，是因为它把“每个查询都和全部历史重新算一次相似度”的过程，改写成“先把历史累加成状态，再对状态做一次投影”。

普通因果 softmax 注意力对第 $t$ 个位置的输出是：

$$
o_t=\sum_{i\le t}\alpha_{t,i} v_i,\qquad
\alpha_{t,i}=\frac{\exp(q_t^\top k_i)}{\sum_{j\le t}\exp(q_t^\top k_j)}
$$

关键难点在于 $\exp(q_t^\top k_i)$ 同时依赖当前查询 $q_t$ 和每个历史键 $k_i$。这意味着你不能在不知道未来查询的前提下，把历史无损压缩成一个固定大小状态，再要求之后所有查询都精确恢复 softmax 的行为。

线性注意力采用特征映射 $\phi(\cdot)$，把核函数写成可分形式：

$$
\kappa(q,k)\approx \phi(q)^\top \phi(k)
$$

于是输出变成：

$$
o_t=
\frac{\sum_{i\le t}\phi(q_t)^\top \phi(k_i) v_i}
{\sum_{i\le t}\phi(q_t)^\top \phi(k_i)}
=
\frac{\phi(q_t)^\top \left(\sum_{i\le t}\phi(k_i) v_i^\top\right)}
{\phi(q_t)^\top \left(\sum_{i\le t}\phi(k_i)\right)}
$$

定义

$$
M_t=\sum_{i\le t}\phi(k_i) v_i^\top,\qquad
z_t=\sum_{i\le t}\phi(k_i)
$$

就得到递推实现：

$$
M_t=M_{t-1}+\phi(k_t)v_t^\top,\qquad
z_t=z_{t-1}+\phi(k_t)
$$

这一步的关键不是“算得更巧”，而是“存的东西变了”。softmax 的历史是条目级保存，线性注意力的历史是和式级保存。

为什么和式会损失表达力？因为和式只能保留线性叠加，不能保留“这份 value 到底来自哪一个 token”这类身份信息。只要多个历史项在特征空间里投到相近方向，它们就会在状态里共享同一批槽位。

把分子再展开一次最直观：

$$
M_t^\top \phi(q_t)=\sum_{i\le t} v_i \, \langle \phi(k_i),\phi(q_t)\rangle
$$

查询看到的是一堆 value 的加权和，而不是“先定位某一个 i，再取 $v_i$”。这也是为什么它做平均、平滑、聚合很自然，但做“按地址精确命中”更难。

再看 softmax 的另一面。若从 Taylor 展开角度写：

$$
e^{q^\top k} = 1 + q^\top k + \frac{(q^\top k)^2}{2!} + \frac{(q^\top k)^3}{3!} + \cdots
$$

许多线性化近似只显式保留低阶结构，尤其是一阶邻域附近的信息。可一旦截断到低阶，像“多个特征维度共同决定匹配”的组合关系就被弱化了。检索任务恰恰依赖这种组合结构，因为真正的地址匹配往往不是“一个方向差不多”，而是“多个条件同时成立”。

这里可以把两者差异压缩成一句公式级对比：

$$
\text{softmax: 先对每个 }(q_t,k_i)\text{ 单独算权重，再汇总}
$$

$$
\text{linear: 先把所有 }(k_i,v_i)\text{ 汇总成状态，再统一读出}
$$

两者顺序不同，能表达的函数类也不同。前者是“查询后决定读什么”；后者更像“历史先压缩，再问压缩态里还有什么”。

可以把它类比为 Hopfield 式记忆。Hopfield 网络是“把多个模式写入同一块记忆介质，再靠相似度取回”的记忆模型。线性注意力也是把多个 $\phi(k_i)v_i^\top$ 写入同一个矩阵。问题在于，同一块介质容量有限，模式变多以后就会出现串扰。术语“串扰”白话讲，就是你本来想取出 A，结果顺手也带出了 B、C 的残留成分。

真实工程例子更容易看出这个推论。假设你在做法律文档问答，输入是数千到数万 token 的合同、判例、补充条款。用户最后问：“甲方在补充协议二第 4.3 条承诺的违约金比例是多少？”这不是泛化摘要，而是精确检索。若模型把长上下文都压成固定状态，它需要在同一块记忆中同时保存大量实体名、条款号、数值和依赖关系。此时只要状态不够大，就会出现“条款号对了，数值串了”或者“实体对了，附录错了”的现象。

所以从机制上可以得到一个硬结论：只要上下文增长而状态不增长，原始线性注意力迟早会进入“记忆条目数增长快于可分离能力增长”的区间。训练可以推迟这个拐点，但不能消除这个拐点。

## 代码实现

下面先给一个最小可运行实现。它不是训练用高性能版本，而是用最少代码展示线性注意力如何把历史压成固定状态，并且如何因为叠加导致检索误差。代码依赖只有 `numpy`，复制后可直接运行。

```python
import numpy as np


def phi(x):
    # 一个最简单的非负特征映射；工程里也常见 elu(x)+1、softplus 等选择
    return np.maximum(x, 0.0) + 1.0


def build_linear_state(K, V):
    """
    K: [T, d]
    V: [T, dv]
    returns:
        M: [d, dv]
        z: [d]
    """
    Kf = phi(K)
    d = Kf.shape[1]
    dv = V.shape[1]
    M = np.zeros((d, dv), dtype=np.float64)
    z = np.zeros(d, dtype=np.float64)

    for k, v in zip(Kf, V):
        M += np.outer(k, v)
        z += k

    return M, z


def linear_read(q, M, z, eps=1e-8):
    """
    q: [d]
    returns:
        y: [dv]
    """
    qf = phi(q)
    denom = float(qf @ z) + eps
    return (qf @ M) / denom


def softmax_read(q, K, V):
    """
    只用于对比。这里直接做一次完整检索，不考虑前缀流式优化。
    """
    scores = K @ q
    scores = scores - np.max(scores)  # 数值稳定
    w = np.exp(scores)
    w = w / np.sum(w)
    return w @ V


def retrieval_demo():
    # 四条历史记忆，其中前两条 key 很接近，故意制造串扰
    K = np.array([
        [1.00, 0.00],
        [0.95, 0.05],
        [0.00, 1.00],
        [0.05, 0.95],
    ], dtype=np.float64)

    V = np.array([
        [10.0, 0.0],
        [20.0, 0.0],
        [0.0, 30.0],
        [0.0, 40.0],
    ], dtype=np.float64)

    M, z = build_linear_state(K, V)

    queries = {
        "q_for_item_1": np.array([1.00, 0.00], dtype=np.float64),
        "q_for_item_2": np.array([0.95, 0.05], dtype=np.float64),
        "q_for_item_3": np.array([0.00, 1.00], dtype=np.float64),
    }

    for name, q in queries.items():
        y_linear = linear_read(q, M, z)
        y_softmax = softmax_read(q, K, V)
        print(f"{name}")
        print("  linear :", np.round(y_linear, 4))
        print("  softmax:", np.round(y_softmax, 4))

    # 关键断言：线性注意力会把相近 key 的 value 混在一起
    y1 = linear_read(np.array([1.00, 0.00]), M, z)
    y2 = linear_read(np.array([0.95, 0.05]), M, z)
    y3 = linear_read(np.array([0.00, 1.00]), M, z)

    assert y1.shape == (2,)
    assert y2.shape == (2,)
    assert y3.shape == (2,)

    # 查询第 1 条时，结果应主要落在第 1 维，但不会精确等于 [10, 0]
    assert y1[0] > y1[1]
    assert abs(y1[0] - 10.0) > 1e-3

    # 查询第 3 条时，也会主要落在第 2 维
    assert y3[1] > y3[0]

    print("demo passed")


if __name__ == "__main__":
    retrieval_demo()
```

这段代码有三个要点。

第一，状态大小始终固定为 `d * dv + d`。不管序列有多长，构建状态时都只是反复更新 `M` 和 `z`。这正是它的内存优势来源。

第二，优势和限制来自同一个地方。因为只保留累加态，所以推理复杂度可以做到线性；但因为只有累加态，历史条目之间也会被压进同一个表示里。

第三，检索误差并不是实现 bug，而是机制结果。若两个 key 的方向接近，那么它们在 `M` 中会占据相近槽位；查询再怎么接近其中一个，都往往会把另一个一起读出来。

如果把上面的读出过程手算一下，会更直观。设查询是 $q$，那么：

$$
\hat{y}(q)=\frac{\sum_i v_i \langle \phi(k_i),\phi(q)\rangle}{\sum_i \langle \phi(k_i),\phi(q)\rangle+\varepsilon}
$$

若 $k_1$ 和 $k_2$ 很接近，则对同一个 $q$，$\langle \phi(k_1),\phi(q)\rangle$ 与 $\langle \phi(k_2),\phi(q)\rangle$ 会同时较大，于是输出近似变成：

$$
\hat{y}(q)\approx \lambda_1 v_1 + \lambda_2 v_2
$$

这就是“本来要取第 1 条，结果混进第 2 条”的来源。

一个更接近工程的伪代码如下：

```python
# phi(.) 是非负特征映射，例如 elu(x)+1 或 softplus(x)
M = zeros(d_feat, d_value)
Z = zeros(d_feat)

for x_t in stream:
    q_t, k_t, v_t = project(x_t)
    q_t = phi(q_t)
    k_t = phi(k_t)

    M += outer(k_t, v_t)
    Z += k_t

    y_t = (q_t @ M) / (q_t @ Z + eps)
```

为什么分母要用 `Z`？因为线性化以后，原来 softmax 的归一化项也必须递推保存；否则输出尺度会随着历史长度增长而漂移，晚期 token 的数值会越来越不稳定。

如果你真的要上工程，还要处理四个细节。

1. `phi(k)` 通常要求非负，否则分母容易出现抵消，稳定性会变差。
2. `eps` 不能省，否则早期前缀或稀疏键可能导致除零。
3. 长序列下常要用 chunk / scan / fused kernel 优化，但这些优化不改变表达力边界。
4. 若你拿它做检索，必须单独测 recall 类基准，不能只看困惑度或吞吐。

再补一个新手容易忽略的点：上面代码里的“固定状态”不是说模型只有一个小向量，而是说对于任意长度的历史，最终都只能压缩成同一形状的 `M` 和 `z`。这就是“容量有限”的精确定义。

## 工程权衡与常见坑

工程上最常见的误判，是把线性注意力当成“同等能力下的更省资源实现”。这通常不成立。更准确的说法是：它用更少的资源，换取了更弱的精确记忆能力。

下面这个表可以作为实际选型的起点：

| 方案 | 状态/缓存成本 | 远距精确检索 | 吞吐 | 典型问题 |
|---|---|---|---|---|
| 原始线性注意力 | 低，固定状态 | 弱 | 高 | recall 掉分、串扰 |
| softmax + KV cache | 随上下文线性增长 | 强 | 中到低 | 显存/带宽压力大 |
| BASED 类混合方案 | 中等，可调 | 中到较强 | 较高 | 实现更复杂 |
| HGRN2 / 状态扩展 RNN | 中到高 | 中到较强 | 高 | 状态设计与训练更敏感 |

常见坑有六个。

1. 只看困惑度，不测检索任务。很多模型在语言建模平均指标上还能接受，但一到 MQAR、needle-in-a-haystack、键值回忆就暴露容量不足。
2. 只加层数，不加状态。层数增加能提升变换能力，但不能改变“每层每步只写入固定大小状态”的根本约束。
3. 以为 feature map 调一下就够。特征映射能改善核近似质量，却不能把固定容量变成随上下文增长的容量。
4. 忽略局部窗口的价值。很多真实文本的强依赖先出现在近邻区域，局部 softmax 窗口能显著补回短程精确匹配。
5. 数值稳定性处理不完整。尤其在长序列里，分母尺度、状态爆炸、低精度累计误差都可能让理论问题和工程问题叠在一起。
6. 评测集与目标场景不一致。摘要任务上看起来正常，不代表“定位版本号”“找回页码”“提取条款金额”也正常。

一个真实工程例子是长文档助手。假设团队要做 8K 到 32K 上下文的企业知识库问答，预算限制又不允许完整 KV cache。若直接把注意力层替换成原始线性注意力，通常能看到显存占用下降、吞吐上升，但“精确引用页码”“找回某个版本号”“定位表格中的数值”会明显变差。此时真正有效的补法通常不是再堆几层，而是同时做三件事：

1. 增加状态维度或采用状态扩展结构。
2. 给近邻区域保留窗口化精确注意力。
3. 对 recall 任务单独评估，再决定是否采用 BASED 或 HGRN2 一类结构。

下面这张表把常见指标和风险对应起来：

| 你看到的现象 | 可能原因 | 更合适的检查方式 |
|---|---|---|
| 困惑度变化不大 | 平均语言建模仍可接受 | 加测 MQAR、needle、文档引用准确率 |
| 吞吐明显上升 | 固定状态确实省缓存 | 同时看长上下文 recall 是否下降 |
| 局部生成不错，远距问答差 | 近邻模式学到了，远距检索没学到 | 增加局部窗口与长程检索基准 |
| 加深模型后帮助有限 | 问题在容量，不只在非线性 | 扩状态而不是只扩层数 |
| 训练不稳定 | 分母、累计精度、特征映射有问题 | 检查 `phi`、`eps`、归一化与数值范围 |

因此工程上的核心 trade-off 很简单：你节省的 KV cache，不会白拿，通常要用 recall 能力、结构复杂度或训练难度来交换。

## 替代方案与适用边界

如果已经知道原始线性注意力有表达力上限，工程选择就会清楚很多。问题不是“要不要修”，而是“按什么代价修”。

第一类修补是 BASED 一类方案。它的思路不是完全回到 softmax，而是保留线性递推的高吞吐特性，同时用更高阶的近似，特别是二阶 Taylor 项，补回一部分短程表达力；再叠加滑动窗口 attention，保证近邻精确匹配。它适合“局部精确依赖很多，远距依赖需要但不必完全无损”的场景，比如代码补全、长对话、文档摘要后问答。

第二类修补是 HGRN2 或更一般的状态扩展。思路很直接：既然问题在容量不足，就把状态做大，甚至引入外积式扩展状态，让每步写入的信息不再只落到一个小矩阵里。它适合“必须长程流式处理，且不能承受完整 KV cache”的场景，比如超长日志分析、在线代理、边推理边读流数据。

第三类方案是直接扩大 RNN 状态规模。相关工作表明，只要状态足够大，RNN/线性注意力系模型在长上下文场景里可以显著逼近 Transformer 的检索能力。这类方案本质上承认一个事实：容量不够就补容量，而不是假设同样的小状态经过更聪明训练就能装下更多独立记忆。

第四类方案是保留或部分保留 softmax。最常见做法是局部窗口 softmax、分块注意力、检索增强或混合层设计。它的逻辑不是否定线性状态，而是承认某些任务必须保留条目级历史，不能全部压缩。

可以用下表快速选型：

| 方案 | 核心补法 | 优点 | 代价 | 适用场景 |
|---|---|---|---|---|
| 原始线性注意力 | 固定小状态 | 内存最低，吞吐高 | recall 弱 | 流式、粗粒度建模 |
| BASED | 二阶近似 + 滑动窗口 | 短程表达力更强 | 结构更复杂 | 需要平衡 recall 与吞吐 |
| HGRN2 | 外积状态扩展 | 提升状态容量 | 训练和调参更难 | 超长上下文、低缓存预算 |
| 大状态 RNN / 线性注意力 | 直接放大 state | 检索能力提升明确 | 状态计算变重 | 追求长上下文可用性 |
| 回到 softmax attention | 完整 KV cache | 精确检索最强 | 内存随长度涨 | 高精度优先 |

再把适用边界写成更直接的决策规则：

| 如果你的任务核心是 | 更优先考虑 |
|---|---|
| 压缩总结、局部模式、流式处理 | 原始线性注意力或轻量混合结构 |
| 近邻精确依赖很多，远距依赖中等 | BASED、窗口化混合注意力 |
| 超长上下文、低缓存预算、仍需一定 recall | HGRN2 或大状态 RNN |
| 按地址精确取回历史某一项 | softmax、检索增强、或显著放大的状态模型 |

因此适用边界可以概括为：

若任务核心是“压缩总结、局部模式、流式处理”，原始线性注意力仍然有价值。  
若任务核心是“按地址精确取回历史某一项”，应优先考虑 softmax、窗口混合结构，或显著放大的状态模型。  
若预算卡在 KV cache，但又不能接受 recall 大幅下降，优先看 BASED、HGRN2、或大状态 RNN，而不是指望原始线性注意力自行学会精确检索。

一句话收束全文：线性注意力的问题不是不够聪明，而是记忆介质太小；修补路线也因此很统一，要么补高阶近似，要么补状态容量，要么两者一起补。

## 参考资料

下面按“结论对应什么资料”来列，方便对照阅读。

- Katharopoulos et al., “Transformers are RNNs: Fast Autoregressive Transformers with Linear Attention”, ICML 2020.  
  经典线性注意力论文，给出核分解与递推实现，是“线性注意力可写成 RNN”这件事的起点。  
  链接：https://proceedings.mlr.press/v119/katharopoulos20a.html

- Beren, “Linear Attention as Iterated Hopfield Networks”, 2024.  
  技术博客，不是同行评审论文，但对“线性注意力像在固定记忆介质中叠加模式”这一视角解释得很直观。  
  链接：https://www.beren.io/2024-03-03-Linear-Attention-as-Iterated-Hopfield-Networks/

- Arora et al., “Simple linear attention language models balance the recall-throughput tradeoff”, ICML 2024.  
  讨论 recall 与 state size 的直接 trade-off，并提出 BASED；文中对 recall-intensive 任务和窗口混合结构很关键。  
  链接：https://proceedings.mlr.press/v235/arora24a.html

- Qin et al., “HGRN2: Gated Linear RNNs with State Expansion”, arXiv 2024.  
  代表性的状态扩展路线，用外积式状态扩容来提升表达力与长程能力。  
  链接：https://arxiv.org/abs/2404.07904

- Liu, Gao, Chen, “Scaling up the State Size of RNN LLMs for Long-Context Scenarios”, ACL 2025.  
  直接支持“状态规模是长上下文 recall 瓶颈之一”这一结论，文中报告了状态增大后 retrieval 能力随之提升。  
  链接：https://aclanthology.org/2025.acl-long.564/

- Mongaras, Larson, “On the Expressiveness of Softmax Attention: A Recurrent Neural Network Perspective”, TMLR 2025.  
  从 RNN 视角分析 softmax 注意力的递推表达形式，可作为“为什么 softmax 比线性注意力更强”这条论证链的补充。  
  链接：https://openreview.net/forum?id=PHcITOi3vV

建议按下面顺序读：

1. 先看 Katharopoulos et al.，理解线性注意力为什么能递推。
2. 再看 Arora et al. 与 Liu et al.，理解为什么 recall 和状态大小强相关。
3. 最后看 HGRN2 与 Mongaras-Larson，理解“怎么补”以及“softmax 为什么不容易被固定状态完全替代”。
