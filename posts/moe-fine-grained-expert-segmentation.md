## 核心结论

MoE，中文常叫“混合专家”，可以理解为“很多个前馈网络里只挑少数几个参与这次计算”的稀疏模型。DeepSeekMoE 的关键改动不是简单把专家数量变多，而是把原本少量、较大的专家切成更多、较小的细粒度专家，再把每次激活的专家数同步放大。

如果原始设计是 `16` 个大专家里选 `2` 个，那么细粒度分割后可以变成 `256` 个小专家里选 `8` 个。每个小专家的 FFN 中间维度缩小，单个专家更“窄”，但组合更自由。这样做的核心收益有两个：

1. 在总参数预算和每次激活的参数预算近似不变时，专家组合空间急剧扩大。
2. 共享专家负责通用知识，路由专家负责细分知识，减少重复学习。

玩具例子可以先这样理解：把 `16` 个“通用厨师”换成 `256` 个“只做某一类菜的小厨师”，每次不再只上 `2` 个，而是上 `8` 个。每个厨师擅长面更窄，但组合出的菜单更多，针对性更强。

下表先看最核心的差异：

| 方案 | 专家总数 | 每次激活数 | 单专家宽度 | 激活参数量 | 组合数 |
|---|---:|---:|---:|---:|---:|
| 粗粒度 MoE | 16 | 2 | $d$ | $2d$ | $C(16,2)=120$ |
| 细粒度分割 MoE | 256 | 8 | $d/4$（示意） | $8\times d/4=2d$ | $C(256,8)\approx 4.1\times 10^{14}$ |

这里“组合数”指“这次 token 可能选到哪一组专家”的数量。它不是模型性能的直接等号，但它确实给了路由层更大的可表达空间。

---

## 问题定义与边界

传统 MoE 的基本问题不是“专家不够多”，而是“专家太粗”。粗粒度，意思是单个专家承载的知识范围太宽，一个专家里混了很多并不相同的模式。这样会带来两个后果：

1. 单个专家内部知识重叠严重，不容易形成稳定专精。
2. 路由器，白话说就是“给 token 选专家的打分器”，即使选对了大方向，也很难精确到更细的知识单元。

在固定预算下，我们真正想优化的是下面两个目标：

- 总参数不明显增加。
- 每个 token 的激活计算量不明显增加。

这就是细粒度分割成立的前提。若把每个原始专家拆成 `m` 个小专家，同时把激活专家数从 `K` 增加到 `mK`，且每个小专家的中间维度缩小到原来的 `1/m`，那么激活参数量保持近似不变：

$$
K \times d \approx mK \times \frac{d}{m}
$$

这里的 $d$ 可以理解为单个专家 FFN 的中间层宽度。直观上看，就是“专家数乘上单专家大小”的总激活体积没有变。

边界也要说清楚。细粒度专家分割并不自动解决所有问题：

- 如果没有共享专家，很多细专家仍会重复学习“所有输入都常见的通用知识”。
- 如果专家切得过细，路由、通信、缓存、负载均衡的工程成本会明显上升。
- 如果任务本身不需要很强的细分专精，组合空间再大也不一定换来线性收益。

所以它适合“知识分布复杂、输入模式多样、但计算预算又必须受控”的场景，不适合把所有模型都一律改成“专家越碎越好”。

---

## 核心机制与推导

DeepSeekMoE 的两条机制是一起工作的。

第一条是细粒度专家分割。原始是从 $N$ 个专家里选 $K$ 个，现在变成从 $mN$ 个更小的专家里选 $mK$ 个。这样在激活预算不变时，路由器拥有更高分辨率的选择空间。

第二条是共享专家隔离。共享专家，白话说就是“所有 token 都固定经过的公共专家”，专门吸收通用模式；剩余专家才交给路由器做动态选择。常见写法是：

$$
h_t^l=\sum_{i=1}^{K_s}\mathrm{FFN}_i(u_t^l)+\sum_{i=K_s+1}^{mN}g_{i,t}\mathrm{FFN}_i(u_t^l)+u_t^l
$$

其中：

- $u_t^l$ 是第 $l$ 层、第 $t$ 个 token 进入 MoE 前的隐藏状态。
- $\mathrm{FFN}_i$ 是第 $i$ 个专家的前馈网络。
- $K_s$ 是共享专家数量。
- $g_{i,t}$ 是门控权重，白话说就是“这个 token 给第 $i$ 个专家分了多少权重”。
- 第二项只对 Top-$mK$ 的候选专家非零。

这个式子说明：MoE 层输出由三部分组成，固定共享知识、动态专精知识、以及残差连接。残差连接可以理解为“把输入直接绕回来一条路”，避免所有信息都压在专家路由上。

最小数值例子最能说明问题。

粗粒度方案：
$$
C(16,2)=\frac{16\times 15}{2}=120
$$

细粒度方案：
$$
C(256,8)=\frac{256!}{8!\,248!}\approx 4.1\times 10^{14}
$$

这不是几十倍的增长，而是十几个数量级的增长。它的含义不是“模型真的会把所有组合都学到”，而是“路由器有机会把不同 token 映射到更细、更稀疏、更不重叠的专家子集”。

再看一个更直白的玩具例子。假设“写 SQL”“修正英文语法”“处理日期格式”“识别日志异常”都混在同一个大专家里，这个专家就必须学很多混杂能力。细分后，可以变成：

- 专家 A 只处理 SQL 结构补全
- 专家 B 只处理 JOIN 条件检查
- 专家 C 只处理英文时态修正
- 专家 D 只处理 ISO 日期归一化

这时单个专家更容易专精，路由也更容易“精确命中”。

真实工程例子则是大规模语言模型推理。以 DeepSeek-V3 系列公开介绍为例，模型采用“`256` 个路由专家里选 `8` 个，再加共享专家”的思路，本质就是希望在超大参数规模下，只激活一小部分参数完成一次 token 计算，同时又保留足够强的差异化专家组合能力。这样大模型既能稀疏计算，又不至于因为专家太粗而浪费表达空间。

## 代码实现

下面用一个最小可运行的 Python 例子演示“共享专家常驻 + 细粒度专家 Top-k 路由”的核心流程。这里不实现真正的神经网络，只模拟路由和预算约束。

```python
from math import comb

def moe_route(scores, shared_experts, topk_routed):
    """
    scores: routed experts 的打分列表
    shared_experts: 共享专家数量，始终激活
    topk_routed: 从 routed experts 中再选多少个
    """
    routed_indices = list(range(shared_experts, shared_experts + len(scores)))
    ranked = sorted(
        zip(routed_indices, scores),
        key=lambda x: x[1],
        reverse=True
    )
    selected_routed = [idx for idx, _ in ranked[:topk_routed]]
    selected_shared = list(range(shared_experts))
    return selected_shared + selected_routed

def active_width(original_k, original_d, m):
    new_k = original_k * m
    new_d = original_d // m
    return original_k * original_d, new_k * new_d

# 玩具例子：2 个共享专家 + 8 个 routed experts 中选 3 个
scores = [0.1, 0.8, 0.4, 0.3, 0.95, 0.2, 0.7, 0.6]
selected = moe_route(scores, shared_experts=2, topk_routed=3)

# 前两个一定是共享专家
assert selected[:2] == [0, 1]

# 预算不变检查：K*d == (mK)*(d/m)
old_active, new_active = active_width(original_k=2, original_d=1024, m=4)
assert old_active == new_active == 2048

# 组合数检查
assert comb(16, 2) == 120
assert comb(256, 8) > 10**14

print("selected experts:", selected)
print("combinations:", comb(16, 2), comb(256, 8))
```

如果把这个流程翻成伪代码，核心就是：

1. 所有 token 都先经过 $K_s$ 个共享专家。
2. 路由器对细粒度专家打分。
3. 只选 Top-$mK$ 个细粒度专家参与计算。
4. 把共享专家输出、路由专家输出和残差相加。

实现时通常还会带两个工程细节：

| 模块 | 做法 | 目的 |
|---|---|---|
| 共享专家 | 常驻激活，不走竞争路由 | 承担通用知识，减少重复学习 |
| 细粒度专家 | 缩小 FFN 宽度，扩大专家数 | 提高组合灵活性和专精度 |
| 路由器 | Top-k 选择 + 负载约束 | 避免少数专家过热 |
| 分布式执行 | 专家并行或专家分片 | 把专家分散到多卡执行 |

对初学者来说，最重要的不是背具体代码，而是记住控制面和计算面的分离：路由器只负责“选谁”，专家只负责“算什么”。

## 工程权衡与常见坑

第一个坑是没有把“共享知识”和“专精知识”拆开。这样做的后果是，很多细粒度专家会反复学同样的通用模式，比如基础语法、常见实体、通用格式变换。结果不是更专精，而是更重复。

这时固定 $K_s$ 个共享专家的意义就出来了。它相当于给系统预留一块“公共底座”，让路由专家专心处理差异化模式。

第二个坑是专家太多，调度比计算更贵。细粒度分割会提升组合能力，但专家数量上去后，带来的不是免费午餐，而是：

- 路由打分矩阵更大
- 跨设备 token 分发更多
- expert cache 更碎
- load balance 更难稳定

下面这个表能说明“有无共享专家”的差异：

| 设计 | 通用知识承载 | 路由压力 | 专家重复学习 | 训练稳定性 |
|---|---|---|---|---|
| 无共享专家 | 全靠 routed experts | 高 | 高 | 较差 |
| 有共享专家 | 共享专家承担底座 | 较低 | 低 | 更稳定 |

第三个坑是只看组合数，不看有效组合数。$C(256,8)$ 很大，但如果路由器长期只偏爱少数几个专家，理论组合空间就不会变成真实可用空间。所以实际训练里通常还要配合负载均衡损失、容量限制、辅助正则项。

第四个坑是把“细粒度”理解成“越碎越好”。这不对。若 $m$ 太大，每个专家容量过小，单专家表达会过弱，导致很多模式必须依赖更多专家拼接才能完成，反而增加路由不稳定性。实际工程里，$m$ 和 $K$ 往往要一起调，目标是让单专家仍有足够表达能力，同时组合空间明显扩张。

---

## 替代方案与适用边界

细粒度专家分割不是唯一方案。它更像是在“预算不变但希望专精更强”这个目标下的一种很有效折中。

| 策略 | 组合数 | 训练稳定性 | 路由开销 | 适合场景 |
|---|---|---|---|---|
| 纯大专家 + Top-k | 较低 | 较高 | 较低 | 通用能力优先，系统简单 |
| 细粒度专家分割 | 很高 | 中等，依赖良好训练技巧 | 较高 | 复杂分布、多模式输入 |
| 细粒度 + 共享专家 | 很高 | 更高 | 较高 | 大规模语言模型主流做法 |
| 任务分组式专家 | 中等 | 取决于分组质量 | 中等 | 已知任务边界清晰 |

还是用“厨师”这个新手版本来对比最直观：

- `16` 个大厨：每个人都能做很多菜，管理简单，但点菜时选择不够精细。
- `256` 个细厨：每个人只做少量菜，搭配更灵活，但排班、调度、后厨协作更复杂。

所以什么时候选哪种？

- 如果你优先追求系统简单、路由稳定、实现成本低，粗粒度大专家仍然合理。
- 如果你的输入分布非常杂，既有代码、数学、自然语言、格式变换，又希望在相同计算预算下提高专家专精，细粒度分割更合适。
- 如果资源很紧，先压低 $m$，不要一上来把专家切得过碎。
- 如果通用模式非常强，必须引入共享专家，否则 routed experts 会浪费大量容量在重复基础能力上。
