## 核心结论

计算机使用场景里的 GUI 导航规划，核心问题不是“模型能不能点中按钮”，而是“能不能把一个长任务拆成一串可执行、可验证、可纠偏的子目标”。这里的“规划”，先可以理解成：在执行前，先决定任务骨架；在执行中，再根据当前界面状态不断修正骨架里的细节。

对跨窗口、跨应用、多步骤任务，最稳的方案通常不是纯全局规划，也不是纯局部规划，而是混合规划：

1. 先生成一份粗粒度全局计划，确定主线
2. 到每个子目标内部，再按当前屏幕状态做局部决策
3. 一旦发现偏离，只在当前子目标附近重规划，而不是把整条任务推倒重来

这个结论的原因很直接。全局规划负责“方向正确”，局部规划负责“落地可行”。只有全局规划，前面一步偏了，后面会沿着错误路径持续执行；只有局部规划，每一步都可能合理，但整条链未必能收敛到最终目标。

可以把 GUI 任务拆成“目标层”和“动作层”两层：

| 层次 | 关注点 | 典型表达 |
| --- | --- | --- |
| 目标层 | 当前到底要达到什么状态 | “浏览器已打开”“表单页已加载”“结果文件已生成” |
| 动作层 | 为了达到该状态，要执行什么操作 | “点击 Chrome 图标”“按 `Ctrl+L`”“输入网址并回车” |

真正稳定的 Agent，应该主要围绕“目标层”规划，而不是把整条链写成死板的动作清单。因为动作会受窗口位置、分辨率、弹窗、滚动条状态影响，但目标状态通常更稳定、更容易验证。

文献和公开基准里常见的趋势也支持这个判断。不同论文的任务长度、任务定义、模型能力和评测口径不同，绝对数值会波动，但趋势基本一致：一次写死整条链的策略在长任务中会快速变脆，带局部纠偏的层次化方法更稳。

| 策略 | 典型做法 | 长链 GUI 任务表现 | 主要原因 |
| --- | --- | --- | --- |
| 纯全局规划 | 一次性列完全部步骤后顺序执行 | 约 8% 到 11% | 前面一步偏离，后续误差持续放大 |
| 纯局部规划 | 每步看屏幕再决定下一步 | 短任务可用，长任务不稳定 | 缺少跨应用、跨阶段的全局结构 |
| 混合规划 | 粗粒度全局 + 每步局部 replanning | 约 22% 到 26% | 既保留任务骨架，也能在线修正 |

这里的数值不应当被理解成一个统一标准答案，而应理解成经验区间：在真实桌面环境里，任务一长、应用一多、状态一复杂，混合规划通常比单一策略稳定。

## 问题定义与边界

GUI 导航规划可以表述为：

> 给定用户目标，Agent 需要在真实图形界面环境中，通过鼠标、键盘、滚动、切换窗口、读屏和验证状态等操作，按顺序完成一组子目标，最终让任务满足可检查的完成条件。

这个定义里有三个关键词最重要。

第一是“真实图形界面环境”。  
这意味着输入不是结构化 API，而是截图、控件树、窗口标题、历史动作、界面文本等不稳定信号。Agent 面对的不是一个函数调用，而是一个会变化的桌面。

第二是“子目标”。  
子目标不是“点击某个按钮”，而是“让系统进入某个可验证状态”。例如：

| 任务阶段 | 子目标例子 | 为什么要单独拆出来 |
| --- | --- | --- |
| 浏览器阶段 | 打开 Chrome，进入目标网页，读取表单字段 | 涉及网页定位、页面加载、信息抽取 |
| 办公软件阶段 | 打开 LibreOffice Calc，定位目标单元格并填写数据 | 涉及表格定位、跨应用信息映射 |
| 开发工具阶段 | 打开 VS Code，生成并保存报告文件 | 涉及文件路径、编辑状态、保存结果验证 |

第三是“验证条件”。  
一个任务不能只靠“我觉得做完了”来判断。它必须有可检查的结束条件，例如：

- 文本文件已存在于指定目录
- 表格中的指定单元格已写入正确内容
- 浏览器当前 URL 属于目标域名
- 某个弹窗已关闭
- 某个导出文件大小大于 0

对初学者，可以先看一个玩具例子。任务是：

> 打开浏览器，搜索某城市天气，把城市名复制到记事本并保存。

如果只从动作角度写，可能是：

1. 打开浏览器
2. 点击搜索框
3. 输入城市名
4. 回车
5. 复制文本
6. 打开记事本
7. 粘贴
8. 保存

但更适合规划的写法是：

1. 浏览器已打开
2. 搜索结果页已加载
3. 城市名已复制到剪贴板
4. 记事本已打开
5. 结果已保存到文件

第二种写法更适合 GUI Agent。因为界面变化时，“点击搜索框”可能失效，但“搜索结果页已加载”这个状态仍然可以重新寻找实现路径。

这个问题与传统 API 自动化有本质区别。二者的边界如下：

| 对比项 | API 自动化 | GUI 导航规划 |
| --- | --- | --- |
| 输入形式 | 明确参数 | 截图、控件树、文本、历史动作 |
| 交互对象 | 稳定接口 | 动态界面 |
| 成功条件 | 返回值或状态码 | 视觉状态和系统状态共同决定 |
| 失败来源 | 参数错误、服务异常 | 定位失准、遮挡、弹窗、滚动、上下文丢失 |
| 可恢复性 | 通常较好 | 若无重规划，容易连锁失败 |

因此，GUI 规划最关键的约束通常有三类：

| 约束 | 具体表现 | 直接结果 |
| --- | --- | --- |
| 界面易变 | 按钮位置变化、窗口大小变化、主题不同、滚动位置不同 | grounding 失准，动作目标错位 |
| 状态隐含 | 任务是否完成不能只看一个控件 | 容易误判下一步 |
| 误差累积 | 一次误点会改变后续整个界面 | 长链任务成功率快速下降 |

这里的 `grounding` 可以先理解成“把语言目标或动作意图，对齐到当前 GUI 上的具体元素”。例如，“点保存按钮”并不是一句完整动作，Agent 还要把“保存按钮”在当前屏幕上定位出来。

这也是为什么纯全局规划会按乘积衰减。若第 $i$ 个子目标的失败概率为 $\epsilon_i$`，`则纯全局链条的近似成功率可以写为：

$$
S_{\text{global}}=\prod_{i=1}^{n}(1-\epsilon_i)
$$

它表达的是一个非常朴素的事实：整条链想成功，每一段都不能出错。只要有一段失败，后面的动作很可能就在错误状态上继续展开。

例如，一个任务有 10 个子目标，每个子目标看起来只有 10% 的失败概率，那么总体成功率不是 90%，而是：

$$
S_{\text{global}} = 0.9^{10} \approx 34.9\%
$$

这个结果往往会让新手意外，因为单步看起来“还行”的系统，一旦放到长链任务里，成功率会迅速下降。

## 核心机制与推导

先看为什么“一次性把 50 步都规划完”通常不可靠。

全局规划隐含了一个很强的假设：  
在时刻 $t=0$ 写下的动作序列，到了 $t=20$、$t=30$ 仍然适用。

真实 GUI 环境通常不满足这个假设。原因包括：

- 窗口可能被新弹窗遮挡
- 页面内容可能延迟加载
- 登录态可能中途变化
- 搜索结果排序可能变化
- 滚动位置和焦点控件可能不同
- 应用可能因为性能波动而进入中间状态

也就是说，计划依赖的“世界状态”会随时间漂移。这个漂移不是例外，而是 GUI 自动化的常态。

如果把每一步的平均失败概率近似为同一个常数 $\epsilon$，则纯全局规划的成功率可写为：

$$
S_{\text{global}}=(1-\epsilon)^n
$$

其中，$n$ 是子目标数。这个式子的含义很简单：任务越长，乘积越小。

下面给一个直观表：

| 子目标数 $n$ | 单步成功率 | 总成功率 $(1-\epsilon)^n$ |
| --- | --- | --- |
| 5 | 90% | $0.9^5 \approx 59.0\%$ |
| 10 | 90% | $0.9^{10} \approx 34.9\%$ |
| 20 | 90% | $0.9^{20} \approx 12.2\%$ |
| 50 | 90% | $0.9^{50} \approx 0.5\%$ |

所以长链 GUI 任务的难点，从来不是某一步会不会点，而是误差会不会被后续步骤放大。

局部规划解决的是另一个维度的问题。它的思想是：不提前承诺整条动作序列，而是每一步都重新看当前界面，再决定下一动作。形式上可以写成：

$$
a_t=\mathcal{L}(s_t,o_t)
$$

其中：

- $a_t$：时刻 $t$ 要执行的动作
- $s_t$：内部状态，例如当前任务阶段、历史摘要、已完成子目标
- $o_t$：当前观察，例如截图、可访问性树、窗口标题、最近动作结果

这里的“可访问性树”可以理解成操作系统或应用暴露出来的结构化界面信息，其中常见字段包括：

| 字段类型 | 例子 |
| --- | --- |
| 控件角色 | button、textbox、checkbox、menuitem |
| 可见文本 | “保存”“导出”“确定” |
| 状态 | 是否选中、是否可点击、是否聚焦 |
| 层级关系 | 某输入框属于哪个对话框、某按钮属于哪个工具栏 |

局部规划的优点是实时、灵活、对界面漂移更稳。但它也有明显缺点：容易只看眼前，不看终局。

例如任务是：

> Chrome 抓取表单字段 → Calc 填表 → VS Code 生成摘要报告

如果只做局部最优，Agent 可能会在浏览器里反复检查字段，或者在 Calc 中持续修表，却忘了整个任务最终还要求生成报告文件。也就是说，“下一步最合理”不等于“整条任务最合理”。

因此，混合规划才是更通用的工程方案。它将任务拆成两种时间尺度：

| 规划层 | 决策频率 | 负责内容 |
| --- | --- | --- |
| 全局层 | 低频 | 决定大阶段和子目标顺序 |
| 局部层 | 高频 | 决定当前界面上的具体动作 |

一个典型执行循环可以写成：

1. 全局 planner 生成粗粒度子目标序列
2. 当前子目标执行时，局部 planner 根据实时观察选择动作
3. 每执行一步，都用验证器判断是否进入目标状态
4. 若偏离，只在当前子目标附近重规划
5. 若多次局部失败，再升级到子目标级或全局级重规划

从控制论角度看，纯全局规划更像开环控制，混合规划更像分层闭环控制：

| 阶段 | 纯全局 | 混合规划 |
| --- | --- | --- |
| 任务开始 | 生成全量步骤 | 生成子目标框架 |
| 执行中偏离 | 继续沿旧计划执行 | 观察当前界面并局部重规划 |
| 偏离后果 | 错误向后传播 | 错误被限制在当前阶段 |
| 长任务表现 | 快速崩塌 | 具备恢复能力 |

工程上，经常用一个近似直觉式来表达混合规划的收益：

$$
S_{\text{mix}} \approx S_{\text{global}} + (1-S_{\text{global}})\times S_{\text{recover}}
$$

其中：

- $S_{\text{global}}$：全局计划直接成功的概率
- $S_{\text{recover}}$：当计划偏离后，局部纠偏把流程拉回主线的概率

这个式子不是严格推导，而是工程近似。它说明了混合规划的本质优势：  
即使主线第一次没有完全走通，系统仍然保留恢复机会。

如果进一步把任务拆成多个阶段，还可以写成更细的近似：

$$
S_{\text{mix}} \approx \prod_{k=1}^{m}\Big[(1-\delta_k)+\delta_k r_k\Big]
$$

其中：

- $m$：子目标阶段数
- $\delta_k$：第 $k$ 个阶段发生偏离的概率
- $r_k$：该阶段偏离后成功恢复的概率

这个表达比单纯乘积更贴近真实 GUI 系统，因为它显式把“偏离后恢复”纳入成功率计算。

放到真实基准如 OSWorld 这类任务里，这个差异会更加明显。OSWorld 的难点不是某个单步动作，而是跨应用连续性。例如：

- 先在浏览器里找到信息
- 再切换到办公软件录入
- 再切换到编辑器整理结果
- 最后把文件保存到指定目录

没有全局骨架，Agent 不知道什么时候该离开当前应用；没有局部重规划，Agent 又会因为一个弹窗、一个未加载完成的页面、一个错误焦点把整条链带偏。

所以，混合规划不是“可选优化”，而是长链 GUI 任务从“演示可跑”走向“工程可用”的核心结构。

## 代码实现

工程上，一个最小可运行版本通常可以拆成五层：

1. `plan_global(goal)`：把用户目标拆成子目标
2. `inspect_screen(env)`：读取当前界面状态
3. `plan_local(subgoal, obs)`：按当前观察生成下一动作
4. `execute_action(env, action)`：执行动作并更新环境
5. `verify_subgoal(subgoal, obs)`：判断子目标是否完成

关键点不在“动作有多复杂”，而在“每个子目标都必须可验证”。如果没有验证器，系统不知道什么时候该继续、什么时候该重试、什么时候该升级重规划。

下面给出一个可以直接运行的 Python 示例。它不依赖真实 GUI，而是用状态机模拟“全局规划 + 局部纠偏”的工作方式。相比只给几段片段代码，这个版本包含完整执行循环、失败注入、状态验证和结果打印，读者可以直接保存为 `planner_demo.py` 运行。

```python
from __future__ import annotations

from dataclasses import dataclass, field
from typing import Dict, List, Optional, Tuple


@dataclass
class Env:
    chrome_open: bool = False
    page_loaded: bool = False
    calc_open: bool = False
    form_filled: bool = False
    popup_blocking: bool = False
    action_log: List[str] = field(default_factory=list)


def plan_global(goal: str) -> List[str]:
    if "填表" not in goal:
        raise ValueError("当前玩具环境只支持包含“填表”的目标")
    return [
        "open_chrome",
        "load_form_page",
        "open_calc",
        "fill_form",
    ]


def inspect_screen(env: Env) -> Dict[str, bool]:
    return {
        "chrome_open": env.chrome_open,
        "page_loaded": env.page_loaded,
        "calc_open": env.calc_open,
        "form_filled": env.form_filled,
        "popup_blocking": env.popup_blocking,
    }


def verify_subgoal(step: str, obs: Dict[str, bool]) -> bool:
    checks = {
        "open_chrome": obs["chrome_open"],
        "load_form_page": obs["page_loaded"],
        "open_calc": obs["calc_open"],
        "fill_form": obs["form_filled"],
    }
    return checks[step]


def plan_local(step: str, obs: Dict[str, bool]) -> str:
    if obs["popup_blocking"]:
        return "close_popup"

    if step == "open_chrome":
        return "open_chrome"

    if step == "load_form_page":
        if not obs["chrome_open"]:
            return "open_chrome"
        return "load_form_page"

    if step == "open_calc":
        return "open_calc"

    if step == "fill_form":
        if not obs["page_loaded"]:
            return "load_form_page"
        if not obs["calc_open"]:
            return "open_calc"
        return "fill_form"

    raise ValueError(f"unknown step: {step}")


def execute_action(env: Env, action: str) -> Tuple[bool, str]:
    env.action_log.append(action)

    if action == "close_popup":
        env.popup_blocking = False
        return True, "popup closed"

    if env.popup_blocking:
        return False, "action blocked by popup"

    if action == "open_chrome":
        env.chrome_open = True
        return True, "chrome opened"

    if action == "load_form_page":
        if not env.chrome_open:
            return False, "chrome is not open"
        env.page_loaded = True
        return True, "page loaded"

    if action == "open_calc":
        env.calc_open = True
        return True, "calc opened"

    if action == "fill_form":
        if not env.page_loaded:
            return False, "page not loaded"
        if not env.calc_open:
            return False, "calc is not open"
        env.form_filled = True
        return True, "form filled"

    return False, f"unknown action: {action}"


def run(goal: str, env: Env, max_attempts_per_step: int = 5) -> Dict[str, object]:
    steps = plan_global(goal)

    for step in steps:
        for _ in range(max_attempts_per_step):
            obs = inspect_screen(env)
            if verify_subgoal(step, obs):
                break

            action = plan_local(step, obs)
            ok, message = execute_action(env, action)

            # 演示：执行动作后再次检查，形成局部闭环
            obs = inspect_screen(env)
            if verify_subgoal(step, obs):
                break

            # 当前动作没把系统推进到目标状态，就继续在本子目标内重规划
            if not ok:
                continue
        else:
            raise RuntimeError(f"step failed after retries: {step}")

    return {
        "final_obs": inspect_screen(env),
        "action_log": env.action_log,
    }


def demo_normal_flow() -> None:
    env = Env()
    result = run("打开网页并在 Calc 中填表", env)
    assert result["final_obs"]["form_filled"] is True
    print("normal flow:", result)


def demo_recovery_flow() -> None:
    env = Env(calc_open=True, popup_blocking=True)
    result = run("打开网页并在 Calc 中填表", env)
    assert result["final_obs"]["form_filled"] is True
    print("recovery flow:", result)


if __name__ == "__main__":
    demo_normal_flow()
    demo_recovery_flow()
```

这段代码有几个值得初学者注意的点。

第一，`plan_global()` 只负责给出子目标顺序。  
它不关心“这一刻屏幕上有什么”，也不负责决定具体点击什么位置。它只回答一个问题：任务骨架是什么。

第二，`plan_local()` 根据当前观察补齐缺失前提。  
例如执行 `fill_form` 前，如果发现网页还没加载，局部规划器不会盲目执行填写动作，而是先回去补 `load_form_page`。这就是局部纠偏。

第三，`verify_subgoal()` 直接围绕状态写。  
这比“是否刚刚执行了某个动作”更可靠。因为在 GUI 环境里，动作执行了不等于状态真的达成了。

第四，执行循环是一个小闭环。  
每次动作后都重新观测，再判断是否进入目标状态。这种闭环在真实 GUI 里非常重要，否则系统会在无效动作上不断前进。

如果运行这个程序，两个演示都能成功：

- `demo_normal_flow()`：正常执行
- `demo_recovery_flow()`：先人为制造弹窗阻塞和部分环境偏差，再靠局部重规划恢复

这个玩具环境当然没有真实视觉模型、鼠标坐标、OCR 和窗口管理，但它保留了桌面 Agent 中最重要的规划逻辑：  
全局计划决定任务顺序，局部规划负责在当前状态下找到能把任务拉回主线的下一动作。

把它映射到真实系统，可以得到一个更完整的框架：

```python
subgoals = plan_global(user_goal)

for subgoal in subgoals:
    retry_count = 0

    while True:
        obs = inspect_screen()                 # 截图 + OCR + a11y tree + 历史摘要
        if verify_subgoal(subgoal, obs):
            break

        docs = retrieve_ui_docs(obs)          # 检索界面文档/经验
        action = plan_local(subgoal, obs, docs)
        status = execute_action(action)

        if status == "success":
            continue

        retry_count += 1
        if retry_count < LOCAL_RETRY_LIMIT:
            continue

        # 子目标内局部恢复失败，升级到子目标级重规划
        subgoal = replan_subgoal(subgoal, obs, docs)
        retry_count = 0
```

这里多出来的三个模块在实战里尤其重要：

| 模块 | 作用 | 没有它会怎样 |
| --- | --- | --- |
| `retrieve_ui_docs` | 检索历史界面知识和操作经验 | 每次都从零猜按钮和路径 |
| `verify_subgoal` | 判断当前状态是否真的达成 | 系统不知道何时该继续 |
| `replan_subgoal` | 当局部动作连续失败时调整当前子目标 | 一直卡在错误局部循环里 |

很多初学者会把“代码实现”理解成“把模型接上鼠标键盘”。这其实只解决了动作执行问题，没有解决规划问题。真正决定系统稳定性的，不是能不能点击，而是：

- 子目标是否定义正确
- 状态是否可验证
- 局部失败后是否有恢复路径
- 历史经验是否能被利用

这才是 GUI 导航规划代码设计的核心。

## 工程权衡与常见坑

混合规划的难点，不是多写一个 `replan()` 函数，而是要让系统持续维护四类信息：

1. 当前任务在哪个阶段
2. 当前屏幕到底处于什么状态
3. 过去做过什么且结果如何
4. 出错后应该在什么粒度上恢复

如果这四类信息缺一项，系统都容易看起来“能跑”，但一进真实桌面就迅速失稳。

下面是工程上最常见的坑：

| 常见坑 | 现象 | 根因 | 常见缓解 |
| --- | --- | --- | --- |
| 界面模态漂移 | 弹窗出现后仍继续点原页面按钮 | 当前交互层已经变化 | 每步先检测模态窗口，再做决策 |
| 历史信息缺失 | 第二次做同类任务仍像第一次摸索 | 没有经验记忆 | 保留动作摘要、截图日志、失败原因 |
| 重复点击 | Agent 一直点同一位置 | 没验证动作是否生效 | 执行动作后强制做状态检查 |
| 子目标过细 | 规划树很长且难维护 | 把动作误写成目标 | 用“可验证状态”定义子目标 |
| 子目标过粗 | 本地 replanner 无法修复 | 一个目标里混入太多状态变化 | 以页面切换、应用切换、结果生成作为分界 |
| 验证条件模糊 | 系统不知道是否完成 | 缺少明确成功判据 | 每个子目标都定义布尔验证器 |
| 记忆污染 | 旧任务经验干扰新任务 | 记忆没有按应用、版本、页面做隔离 | 给经验加上下文标签和过期机制 |

这里有一个特别重要、但最容易被忽视的点：  
**规划应围绕状态，而不是围绕动作。**

例如，下面这组写法不够好：

- 点击菜单
- 点击导入
- 点击确认
- 滚动到底部
- 点击保存

因为它们都是“手段”，不是“目标”。一旦菜单位置变化、窗口布局变化、按钮文字变化，这条动作链就会失效。

更稳的写法是：

- 已进入导入页
- 文件已被选中
- 导入参数已确认
- 结果区已展示完整内容
- 导出文件已保存

这类状态更适合被验证，也更适合在失败后重新寻找替代动作。

对初学者，可以用一句简单标准判断一个子目标写得好不好：

> 如果这个子目标无法通过当前观察来判断真假，它大概率写得不够好。

例如：

| 写法 | 是否推荐 | 原因 |
| --- | --- | --- |
| “点击保存按钮” | 不推荐 | 这是动作，不是完成状态 |
| “文件已保存到指定目录” | 推荐 | 可以验证是否真的完成 |
| “滚动到中间位置” | 不推荐 | 缺乏明确判据 |
| “目标字段已出现在可见区域” | 推荐 | 可通过 OCR 或控件树验证 |

另一个工程常见误区，是把“记忆”理解成“把所有截图都存下来”。  
这通常会造成两个问题：

- 记忆太长，检索效率低
- 噪声太多，模型读不出重点

更有效的做法是使用压缩过的过程记忆，也就是常见的 `narrative memory`。它可以理解成一段可检索的文字化摘要，例如：

| 时间 | 记忆摘要 |
| --- | --- |
| `t1` | 在浏览器中打开目标站点，主页加载成功 |
| `t2` | 点击导入按钮无反应，屏幕右上角出现权限弹窗 |
| `t3` | 关闭弹窗后再次点击导入，进入文件选择页 |
| `t4` | 选择 `sales.csv` 后出现列映射页面 |

这种记忆的价值不在于保留全部细节，而在于给后续规划提供“试过什么、什么有效、什么无效”的上下文。

再看一个跨应用真实例子。任务是：

> 在 Chrome 中抓取销售字段，在 LibreOffice Calc 中填表，再在 VS Code 中生成摘要报告。

如果缺少跨应用历史串联，Agent 到第三阶段出错时很难定位问题来自哪里。因为错误可能来自：

- 浏览器阶段：字段提取不完整
- Calc 阶段：列映射错位
- VS Code 阶段：模板文件未打开或路径错误

所以工程上至少要维护一条跨应用任务轨迹：

| 阶段 | 应记录的关键状态 |
| --- | --- |
| 浏览器 | 数据来源页面、抽取字段、字段是否校验通过 |
| Calc | 写入列、写入行、是否出现格式警告 |
| VS Code | 打开的文件路径、模板是否加载、输出文件是否生成 |

没有这类轨迹，系统出了错只能“从头再试”；有了轨迹，系统才能在局部定位并恢复。

AppAgent 一类方法给出的经验也很重要：  
第一次先探索和记录，第二次再部署执行。它不意味着规划问题被完全解决，而是说明 GUI 任务里“界面知识”本身就是系统能力的一部分。对企业重复流程而言，这种“先学界面、后执行”的模式通常非常有价值，因为同一流程会被反复运行，初次探索成本可以在后续多次执行中摊薄。

## 替代方案与适用边界

三类规划策略并不是绝对优劣关系，而是对应不同任务结构、不同环境稳定性和不同工程成本。

先看一个总表：

| 策略 | 适合任务长度 | 界面稳定性要求 | 误差容忍度 | 典型场景 |
| --- | --- | --- | --- | --- |
| 纯全局 | 短到中等 | 高 | 低 | 固定模板、固定版本、低波动流程 |
| 纯局部 | 很短 | 中等 | 中等 | 单页面、单应用、即时响应任务 |
| 混合 | 中到长 | 中低 | 高 | 跨应用、多窗口、长链工作流 |

### 1. 纯全局规划何时可用

纯全局规划适合满足下面几个条件的流程：

- 任务步骤固定
- 页面和控件位置稳定
- 分辨率和系统版本差异小
- 弹窗和异常路径很少
- 失败成本低，可以直接重跑

例如，公司内网中的固定报表导出流程，在固定浏览器版本、固定分辨率下执行，预先录好一套步骤可能就足够。这类场景强调执行速度和实现简单，不强调鲁棒性上限。

但它的边界也很明显：  
只要页面结构变化、登录态变化、弹窗出现或者加载时序波动，原本的动作序列就可能全部失效。

### 2. 纯局部规划何时可用

纯局部规划适合任务目标短、依赖当前可见状态即可决定下一步的场景。例如：

- 单页面网页表单填写
- 简单设置页中的开关调整
- 单个弹窗里的确认操作
- 聊天窗口里的即时回复

这类任务的共同点是：  
全局结构很短，未来阶段少，局部观察几乎足以支撑全部决策。

它的优势是系统简单，不需要维护复杂子目标图；缺点是容易缺少长期约束。例如多个候选页面都“看起来可以点”，但只有其中一个通向真正终点时，纯局部策略就容易偏航。

### 3. 混合规划何时更合适

混合规划适合下面这些典型条件：

- 任务跨多个应用
- 任务长度超过十几步
- 中间存在可能打断流程的弹窗、登录页、权限确认
- 结果必须写入文件、表格或其他持久化对象
- 部分步骤需要回溯和补前提

例如下面这类任务，几乎天然适合混合规划：

| 任务 | 为什么需要混合规划 |
| --- | --- |
| 浏览器抓取数据并录入表格 | 需要跨应用切换，同时要应对网页加载和表格定位 |
| 下载附件并改名保存到指定目录 | 需要处理浏览器下载、文件管理器、路径验证 |
| 在 IDE 中修改代码后运行并复制日志 | 需要编辑、执行、观察输出，再决定下一步 |

混合策略的优势是鲁棒，但代价也更高：

| 代价 | 含义 |
| --- | --- |
| 需要维护子目标层次 | 否则全局计划会失控 |
| 需要状态验证器 | 否则不知道何时该结束或重试 |
| 需要界面知识库 | 否则每次都要从零探索 |
| 需要历史记忆管理 | 否则经验会变成噪声 |
| 需要多级重规划策略 | 否则局部失败会陷入死循环 |

因此，工程上可以用一个很实用的选择标准：

- 单应用、少步骤、页面稳定：优先局部规划
- 固定流程、模板化强、环境高度可控：可以使用全局规划
- 跨应用、长链、存在弹窗和状态漂移：优先混合规划

如果还想再压缩成一句判断规则，可以写成：

> 只要任务中存在“先做 A，结果会影响后面 B、C、D，而且中间任何一步都可能被界面打断”，就应该认真考虑混合规划。

从工程角度看，桌面 Agent 的重点不是追求“永远一次规划成功”，而是把失败限制在局部，并且把恢复路径写进系统。真正可用的 Agent，不是永不偏航，而是偏航后仍能回到主线。

## 参考资料

- **OS Agents: A Survey on MLLM-Based Agents for General Computing Devices Use**  
  说明：系统梳理基于多模态大模型的 OS Agent 组成，包括感知、grounding、规划、记忆与执行，并区分 global planning、iterative planning、hierarchical planning 等范式。  
  适合阅读的原因：如果你刚接触这个方向，这篇综述最适合先建立术语表和模块地图。  
  用途：定义、任务分解、规划框架。

- **OSWorld: Benchmarking Multimodal Agents for Open-Ended Tasks in Real Computer Environments**  
  说明：提出真实操作系统环境下的开放式任务基准，覆盖网页、桌面应用、文件系统和跨应用工作流。该工作把“看起来会点”与“能在真实系统里稳定完成任务”区分开来。  
  适合阅读的原因：它能帮助理解为什么 GUI Agent 的难点不在单步动作，而在长程、多应用、可验证完成条件。  
  用途：任务边界、评测方式、工程压力来源。

- **Agent S: An Open Agentic Framework that Uses Computers Like a Human**  
  说明：强调 experience-augmented hierarchical planning，把层次化规划、经验检索和界面控制接口结合起来，在公开基准上展示出比简单单层策略更好的长任务表现。  
  适合阅读的原因：它展示了“全局子目标 + 局部执行 + 经验回收”这一工程组合为什么有效。  
  用途：混合规划机制、层次化执行。

- **AppAgent: Multimodal Agents as Smartphone Users**  
  说明：提出 exploration phase 与 deployment phase 两阶段方法。第一阶段先探索界面并沉淀文档，第二阶段再检索这些文档辅助执行。虽然主要面向手机 GUI，但“先积累界面知识，再执行复杂任务”的思路可迁移到桌面环境。  
  适合阅读的原因：它很好地说明了 GUI 自动化中“界面知识”本身也是能力，不只是附属信息。  
  用途：工程实践、界面文档检索、经验记忆。

- **OSWorld 相关分析与社区综述页面**  
  说明：常见内容包括 GUI grounding 失败、长程规划失效、跨应用状态中断、记忆污染等问题的现象归纳。  
  使用建议：这类材料适合补充案例和经验，不适合替代原始论文。引用时应优先回到论文或官方项目页确认。  
  用途：现象总结、工程经验对照。

- **建议的阅读顺序**  
  对初学者更友好的顺序通常是：  
  1. 先读 `OSWorld`，理解任务为什么难  
  2. 再读 `OS Agents Survey`，补齐术语和模块框架  
  3. 接着读 `Agent S`，理解层次化与经验增强  
  4. 最后读 `AppAgent`，理解探索式界面知识积累
