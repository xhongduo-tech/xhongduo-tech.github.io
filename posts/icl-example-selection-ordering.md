## 核心结论

上下文学习，简称 ICL，就是模型不更新参数，只靠提示词里的几个示例临时学会任务。它的一个关键事实是：**同一批示例，换一个顺序，准确率可能就不是“小幅波动”，而是直接相差 30% 以上**。这说明示例排序不是排版细节，而是模型输入的一部分。

Lu 等人的实验给出的结论很直接：有些排列能把模型推到接近当时强基线的水平，有些排列却会退化到接近随机猜测。对初学者最重要的理解是，ICL 不是“给几个例子就行”，而是“给哪些例子、按什么顺序给、要不要先做校准”，这三件事共同决定结果。

下面这个玩具例子能先建立直觉。假设同样是 4 个情感分类示例，只是顺序不同：

| 策略 | 示例顺序特征 | 准确率 |
|---|---|---:|
| 策略 A | 标签分布均衡，且与查询更相似的示例靠后 | 82% |
| 策略 B | 同一批示例，但把不相似或标签冲突的示例放在靠后位置 | 47% |

82% 和 47% 的差距，不是采样噪声能解释的级别，而是输入结构改变了模型的注意力分配。对于零基础读者，可以先记一句最重要的话：**ICL 里“最后放什么”和“前面铺什么”会改变模型到底在模仿谁。**

## 问题定义与边界

先把问题说清楚。设有一个查询输入 $x$，比如一句待分类评论；有一个演示集合 $D=\{d_1,\dots,d_k\}$，每个演示都包含“输入+标签”或“问题+答案”；把这些示例排成某个顺序 $\pi$ 后，得到提示模板 $t_\pi$。模型最终输出的是：

$$
p(y \mid x, t_\pi)
$$

这里的意思很朴素：在“查询 $x$ + 当前这组按某种顺序摆放的示例”条件下，模型认为标签 $y$ 的概率是多少。

关键点在于，**排列 $\pi$ 不是外部装饰，而是概率条件的一部分**。也就是说，只要顺序变了，模型看到的上下文就变了，输出分布也会变。

为什么顺序会有这么大影响？一个重要边界来自模型架构。

- CausalLM，自回归语言模型，就是“当前位置只能看前面内容”的模型。
- PrefixLM，前缀语言模型，就是“某一段前缀内部可以更自由互相看见”的模型。

对白话理解来说，CausalLM 更容易出现“近因偏差”，也就是离查询更近的示例更容易主导输出；PrefixLM 因为允许更强的跨示例交互，这种偏差通常更弱。于是，同样的提示，在不同架构上顺序敏感性并不一样。

再给一个很小的例子。假设有 3 个情感示例：

1. “这电影很无聊” → 负面  
2. “画面一般，但配乐不错” → 中性偏正  
3. “节奏紧凑，值得推荐” → 正面  

如果查询是“表演自然，剧情有力”，在 CausalLM 里，把第 3 条放到最靠近查询的位置，模型更容易输出正面；如果把负面示例放在最后，它可能会被过度放大。初学者可以先把它理解成一句话：**在自回归模型中，后面的示例往往更“响”。**

但顺序不是唯一问题。模型还有先验偏置。比如没有任何内容时，它也可能天然偏向输出某个标签。为此需要校准。一个常见写法是：

$$
q(y \mid x,t)=p(y \mid x,t)-p(y \mid x_{cf},t)
$$

这里 $x_{cf}$ 是 content-free input，意思是“内容为空或无语义的输入”，比如 `N/A`、空字符串、占位符。这个式子的直觉是：先测模型在“没有真实内容”时有多偏向某个标签，再把这个偏向从真实预测里扣掉。这样做不是为了变复杂，而是为了防止模型把“提示模板偏爱某标签”误当成“样本本身属于该标签”。

## 核心机制与推导

要理解排序效应，先要区分三个层次：选择、排序、校准。

第一层是示例选择。不是所有候选示例都适合当前查询。与查询语义接近、标签可靠的示例通常更有帮助。

第二层是示例排序。即使选中了同一批示例，不同顺序也会让注意力分布不同。Lu 等人的工作用熵来做无监督判断。熵可以理解成“输出分布有多不确定”。若某个排列下，模型在探测样本上的输出更稳定、更集中，则这个排列往往更好。

分类分布熵的基本形式是：

$$
H(\pi)=-\sum_y p(y\mid x,t_\pi)\log p(y\mid x,t_\pi)
$$

如果 $H(\pi)$ 很高，说明模型摇摆大；如果较低且在多个探测样本上都稳定，说明这个排列给出的决策边界更清楚。你可以把 GlobalE 和 LocalE 理解成两种评估方式：

- GlobalE：看排列在更广泛 probing set 上的整体稳定性。
- LocalE：更关注与当前任务或当前查询邻近区域的稳定性。

第三层是局部困惑度排序。困惑度，perplexity，就是模型读某段文本时觉得“顺不顺”“像不像它熟悉分布”的指标，数值越低通常表示越自然、越一致。NeurIPS 2024 的 Local Perplexity Ranking，简称 LPR，核心思想不是“全局找最低困惑度”，而是**先在语义邻域内找候选，再在这个局部空间里按困惑度筛掉噪声示例**。

可写成一个简单目标：

$$
z^\*=\arg\min_{z\in S(x)} \mathrm{Perplexity}(z)
$$

这里 $S(x)$ 是查询 $x$ 的语义邻居集合。白话就是：不要拿全库一起比，而是先找和查询相近的一小圈候选，再优先保留其中模型读起来更一致的示例。这样做的好处是能减少两类错误：

- 语义不相似，但表面形式很常见。
- 标签被噪声污染，导致模型学错映射。

把“熵”和“困惑度”放在一起，就能解释为什么实践中常用“先筛、再排、再校准”的流程。

- 熵偏向评估“这个顺序是否稳定”。
- 困惑度偏向评估“这个示例是否像当前查询的局部同类”。
- 校准偏向处理“模型是不是本来就偏爱某标签”。

一个更完整的视角是：模型对某个查询的最终判断，不只是内容相似度函数，而是注意力结构、语言先验、标签偏置共同作用的结果。只调其中一项，往往还不够稳。

## 代码实现

下面给一个可运行的 Python 玩具实现，不依赖深度学习库，只模拟“排序评分 + 局部困惑度筛选 + 校准”这三个环节。代码的目标不是复现论文全部细节，而是让流程能跑通。

```python
from math import log, exp
from itertools import permutations

def entropy(probs):
    return -sum(p * log(p) for p in probs if p > 0)

def softmax(xs):
    exps = [exp(x) for x in xs]
    s = sum(exps)
    return [x / s for x in exps]

def calibrate(raw_probs, bias_probs):
    # 用减法去掉 content-free 偏置，再做 softmax 归一化
    adjusted = [r - b for r, b in zip(raw_probs, bias_probs)]
    return softmax(adjusted)

def score_permutation(perm, query_label="pos"):
    """
    玩具评分：
    - 越靠后的示例权重越高，模拟 recency bias
    - 与查询标签一致的示例增加得分
    - 标签切换过于频繁会造成一点不稳定惩罚
    """
    score = 0.0
    switch_penalty = 0.0
    last_label = None

    for i, demo in enumerate(perm):
        weight = i + 1
        if demo["label"] == query_label:
            score += 0.9 * weight
        else:
            score -= 0.7 * weight

        if last_label is not None and last_label != demo["label"]:
            switch_penalty += 0.15
        last_label = demo["label"]

    return score - switch_penalty

def local_perplexity_ranking(candidates, top_k=3):
    # 困惑度越低越好；这里只在局部候选中选
    ranked = sorted(candidates, key=lambda x: x["ppl"])
    return ranked[:top_k]

# 候选示例，sim 表示与查询的语义相似度，ppl 表示局部困惑度
pool = [
    {"id": "a", "label": "neg", "sim": 0.62, "ppl": 18.0},
    {"id": "b", "label": "pos", "sim": 0.91, "ppl": 11.0},
    {"id": "c", "label": "neg", "sim": 0.58, "ppl": 25.0},
    {"id": "d", "label": "pos", "sim": 0.88, "ppl": 10.5},
    {"id": "e", "label": "pos", "sim": 0.79, "ppl": 13.0},
]

# 第一步：先取局部语义邻域
neighbors = [x for x in pool if x["sim"] >= 0.75]

# 第二步：在邻域里做 Local Perplexity Ranking
selected = local_perplexity_ranking(neighbors, top_k=3)
assert [x["id"] for x in selected] == ["d", "b", "e"]

# 第三步：遍历排列，挑一个更优顺序
best_perm = None
best_score = float("-inf")
for perm in permutations(selected):
    s = score_permutation(perm, query_label="pos")
    if s > best_score:
        best_score = s
        best_perm = perm

assert best_perm is not None
assert len(best_perm) == 3

# 第四步：模拟模型原始输出与 content-free 偏置
raw_probs = [0.42, 0.58]   # [neg, pos]
bias_probs = [0.30, 0.70]  # 模型天然偏向 pos
calibrated = calibrate(raw_probs, bias_probs)

# 校准后，概率仍然合法
assert abs(sum(calibrated) - 1.0) < 1e-9
assert all(0.0 <= p <= 1.0 for p in calibrated)

print("selected:", [x["id"] for x in selected])
print("best order:", [x["id"] for x in best_perm])
print("calibrated:", calibrated)
```

这个例子里有两个值得观察的点。

第一，`selected` 不是从全库里直接挑，而是先按相似度进邻域，再按困惑度排序。这样做模拟了 LPR 的局部性假设：**只有放在“相似任务子空间”里比较困惑度，才更有意义。**

第二，`score_permutation` 里让越靠后的示例权重越高，模拟 CausalLM 的近因偏差。真实模型当然比这复杂得多，但对新手来说，这个玩具机制已经足够解释为什么“同样 3 个示例，顺序不同，结果就不同”。

如果换成真实工程流程，通常会是下面这样：

1. 从标注库中召回与查询最相似的一批候选示例。  
2. 用局部困惑度筛掉不稳定或疑似噪声的示例。  
3. 生成若干种排列，对 probing set 计算 GlobalE 或 LocalE。  
4. 用 content-free input 做 contextual calibration 或 CBM。  
5. 对若干优质排列分别推理，再投票或聚合。

真实工程例子可以看情感分类或客服工单路由。比如 SST5 五分类任务中，某些 LLaMA 系列因自回归遮罩更容易受顺序影响，部分样本在不同排列下会从“完全错误”变成“部分正确”或“完全正确”。这时，一个常见策略不是赌单次最优，而是**生成 5 个候选顺序，分别打分，再多数投票**。它牺牲一些延迟，换来明显更稳的准确率曲线。

## 工程权衡与常见坑

真正落地时，最常见的错误不是“不会算熵”，而是把 ICL 当成静态模板问题。下面这张表总结几个高频坑。

| 问题 | 常见做法 | 影响 | 更稳的做法 |
|---|---|---|---|
| 只按原始 $p(y\mid x,t)$ 选顺序 | 直接取概率最大的提示 | 容易被模型先验带偏 | 先做 calibration 再比较 |
| 只看全局相似度 | 余弦相似度 TopK 直接塞进 prompt | 可能混入标签噪声 | 在局部邻域内再做 perplexity 排序 |
| 只跑一个顺序 | 选一个“看起来合理”的排列 | 结果方差大 | 多顺序打分后投票 |
| 忽略模型架构 | 同一策略复用到所有模型 | 在 CausalLM 上不稳 | 针对架构调整排序和聚合策略 |

第一个坑是过度自信。模型有时会给出很高的标签概率，但这不代表排序真的优。它可能只是“模板 + 标签词”触发了某种先验偏好。这个问题在类别不平衡、标签词长度差异明显、或某个标签在预训练语料中更常见时更突出。所以很多论文都强调：**概率先校准，再拿来排名。**

第二个坑是误把低困惑度当成绝对好。困惑度低只说明文本形式更像模型熟悉的分布，不等于标签就一定可靠。如果一个错误标签样本写得非常工整，它也可能有低困惑度。因此 LPR 的合理用法是“局部筛噪”，不是“全局真理判定”。

第三个坑是忽略成本。多顺序投票能稳，但会增加推理延迟。举个工程化估算：如果一次请求原本只跑 1 个 prompt，耗时 400ms；现在改成 5 个顺序并行，再做投票，延迟可能升到 700ms 到 1.2s，成本也近似线性增加。是否值得，取决于业务目标。如果是低风险内容分类，这个成本可能不划算；如果是客服路由、风控初筛、医学文档预分类，稳定性通常更值钱。

第四个坑是把“首因偏差”和“近因偏差”混成一个词。它们都属于位置偏差，但含义不同：

- 首因偏差：前面示例先建立了解释框架，影响后续理解。
- 近因偏差：离查询更近的示例对最终输出影响更强。

在 CausalLM 上，近因偏差通常更明显；在允许更强前缀交互的设置里，首因偏差有时仍然存在。因此排序策略不应只靠“把最像的样本放最后”这一条经验，而应结合模型类型和任务验证。

## 替代方案与适用边界

如果把“示例选择与排序”看成一个模块，那么它不是唯一方案，但在不改模型参数的前提下，它通常是成本最低、收益较稳定的一条线。

一个基础替代方案是纯 TopK 相似度检索。优点是简单，速度快；缺点是对噪声标签和顺序不敏感，遇到 domain shift 时容易失效。

第二个方案是 TopK + LPR。也就是先按向量相似度召回，再在局部候选里用困惑度重排。它适合下面几种情况：

- 示例库有一定噪声。
- 任务是生成或长文本理解，单纯标签相似度不够。
- 查询分布变化较大，希望保留局部一致性。

第三个方案是 CBM 或 contextual calibration + LPR。CBM 可以理解成“先估计当前模板和标签词自带的偏置，再对真实预测做去偏”。这个组合更适合概率分数要被直接拿来比较、筛选、排序的场景，例如自动构造 prompt、候选提示 A/B 排名、批量路由策略。

可以用一张表概括：

| 方案 | 优点 | 缺点 | 适用边界 |
|---|---|---|---|
| 纯 TopK | 简单、快 | 对噪声和顺序敏感 | 数据干净、延迟严格受限 |
| TopK + LPR | 对噪声更稳 | 要多算困惑度 | 候选库较大、标签质量一般 |
| CBM + LPR | 排序更稳、偏置更小 | 流程更复杂、成本更高 | 需要概率可比性和鲁棒性 |
| 多顺序投票 | 方差最低 | 推理成本最高 | 高风险、高价值任务 |

这里还要强调一个边界：**当任务已经允许做参数微调时，ICL 的示例排序收益会相对下降，但不会完全消失。** 因为即使模型经过微调，推理时的上下文结构仍然会影响注意力路径。只是这时排序不再是主导因素，数据质量和任务格式会更重要。

因此，适用边界可以概括为：

- 不改权重、只改提示时，选择与排序是核心优化手段。
- 有噪声示例库时，LPR 的价值更高。
- 需要跨 prompt 稳定比较概率时，校准不可省。
- 对延迟极敏感的在线系统，优先用轻量召回 + 少量候选顺序，而不是大规模全排列搜索。
