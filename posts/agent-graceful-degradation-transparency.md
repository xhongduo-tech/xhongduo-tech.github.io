## 核心结论

Agent 的优雅降级，核心不是“尽量回答”，而是“按可信程度决定回答方式”。生产级 Agent 在无法完整完成任务时，应该严格落到三级策略：

| 阶段 | 条件 | 系统输出 | 用户感知 |
| --- | --- | --- | --- |
| 完整完成 | $confidence \ge T_{full}$ | 交付结果，附来源 | 用户直接使用 |
| 部分完成 | $T_{partial} \le confidence < T_{full}$ | 交付片段，明确标注质量、来源、缺口 | 用户知道哪里可用、哪里待确认 |
| 明确拒绝 | $confidence < T_{partial}$ | 不编造答案，说明原因并建议转人工 | 用户知道系统被什么阻塞 |

“优雅降级”这个术语，白话讲就是系统在出问题时，不突然坏掉，也不假装没坏，而是退到更安全、更诚实的工作模式。

这件事的目标不是让 Agent 看起来更聪明，而是让它在不确定时仍然可审计、可解释、可接管。对用户而言，最危险的不是“失败”，而是“看起来成功，实际上错了”。静默失败会诱发两类严重后果：一类是幻觉输出，即模型编造不存在的事实；另一类是隐式副作用，即工具重试导致重复发邮件、重复下单、重复写库。

研究给出的方向很一致：透明沟通比“装作顺滑”更能提升满意度。尤其在复杂任务中，用户并不要求 Agent 永远成功，但要求它在失败时说清楚“为什么失败”和“接下来怎么办”。

## 问题定义与边界

这篇文章讨论的不是模型回答质量本身，而是 Agent 在真实任务执行中的失败处理。这里的 Agent 指能调用外部工具、访问系统状态、执行多步任务的智能体，而不是单轮聊天机器人。

边界要先划清。一个任务失败，通常不是因为“模型突然变笨”，而是因为以下几个环节中的某一个失效：

| 失效类型 | 白话解释 | 是否适合降级 |
| --- | --- | --- |
| 工具不可用 | API 超时、权限不足、网络错误 | 适合 |
| 信息不完整 | 缺少必要参数或上下文 | 适合 |
| 置信度偏低 | 系统判断当前结果不可靠 | 适合 |
| 高风险合规场景 | 医疗、金融、法律等结果不能模糊使用 | 通常应直接拒绝或转人工 |

“置信度”这个术语，白话讲就是系统对自己当前结果可靠程度的估计，不等于真相，但可以作为决策信号。

玩具例子很直观。假设用户让 Agent 做一件小事：“帮我查今天北京到上海的高铁票，并整理出最早三班。”如果查票工具超时，但缓存里还有 10 分钟前的数据，那么系统可以返回“部分完成”：列出缓存中的车次，同时标注“数据非实时，需确认”。这就是降级。若系统完全拿不到数据，还编出三班车，那就不是降级，而是幻觉。

真实工程例子更能说明边界。用户要求：“帮我创建明天下午 3 点的会议，并给参会人发邮件。”这个任务通常要串联 `calendar.create` 和 `email.send` 两个工具。如果日历创建成功，但邮件发送接口超时，系统不能对用户说“已完成”，因为状态并不完整；也不能沉默重试到用户毫无感知。正确行为是：告诉用户会议已创建、邮件尚未发送、失败原因是邮件服务超时，并立即给出“重试”或“转人工”的入口。

## 核心机制与推导

优雅降级的核心机制可以压缩成一个双阈值策略。设任务置信度为 $c$，完整交付阈值为 $T_{full}$，部分交付阈值为 $T_{partial}$，则策略为：

$$
Policy(c)=
\begin{cases}
Full, & c \ge T_{full} \\\\
Partial, & T_{partial} \le c < T_{full} \\\\
Reject, & c < T_{partial}
\end{cases}
$$

这套规则的关键不在公式，而在“每次状态变化都要显式告知用户”。

如果只做阈值判断，不做透明展示，系统仍然可能伤害用户体验。真正可用的设计通常包含三个 UI 元素：

| 元素 | 作用 | 必要原因 |
| --- | --- | --- |
| 状态横幅 | 显示 `Idle / Executing / Partial / Blocked` | 让用户知道任务处在哪个阶段 |
| 实时日志 | 展示已调用工具、失败点、重试次数 | 让用户知道系统到底做了什么 |
| 下一步动作 | 提供“重试 / 转人工 / 修改参数” | 让用户知道现在能做什么 |

“状态横幅”这个术语，白话讲就是界面顶部一块持续显示当前任务状态的区域；“实时日志”是按时间顺序显示系统动作的记录。

可以把整个过程理解成一个受控状态机。状态机，白话讲就是系统只能在有限几个状态里切换，并且切换条件是明确定义的。它的价值在于：系统不会一会儿说成功、一会儿又偷偷重试，所有变化都有记录。

一个简单推导是：若系统做了 $n$ 步工具调用，则最终可交付程度不是由最后一步单独决定，而是由整条链路中最弱的一环决定。粗略写法可以是：

$$
confidence_{task} \approx \min(confidence_1, confidence_2, \dots, confidence_n)
$$

这不是唯一公式，但它传达一个重要工程事实：多工具链路里，只要关键步骤之一不可靠，整任务就不该被标记为“完整完成”。

玩具例子可以继续沿用。查高铁票任务里，`search_ticket` 的置信度若只有 0.82，而你的系统定义 $T_{full}=0.95$、$T_{partial}=0.80$，那么它就应该进入“部分完成”。界面不该写“查询成功”，而应写“部分完成：结果来自缓存，非实时”。

真实工程例子里，会议创建的置信度可能是 0.99，邮件发送因为接口超时只有 0.30。对整个任务而言，不能因为第一步成功就把任务标绿。正确做法是把任务拆成子结果展示：会议已创建，邮件未完成，整体状态为“部分完成”或“阻塞”，并允许用户无缝接管。

## 代码实现

下面给出一个可运行的 Python 玩具实现。它不依赖外部框架，但把三级降级、质量标注、人工接管入口都编码进去了。

```python
from dataclasses import dataclass

T_FULL = 0.95
T_PARTIAL = 0.80

@dataclass
class Result:
    status: str
    message: str
    allow_handoff: bool

def output_policy(confidence: float, tool_name: str, has_partial_data: bool) -> Result:
    assert 0.0 <= confidence <= 1.0

    if confidence >= T_FULL:
        return Result(
            status="completed",
            message=f"{tool_name} 已完成，可直接交付",
            allow_handoff=False,
        )

    if confidence >= T_PARTIAL and has_partial_data:
        return Result(
            status="partial",
            message=f"{tool_name} 部分完成，需标注来源与置信度 {confidence:.0%}",
            allow_handoff=True,
        )

    return Result(
        status="blocked",
        message=f"{tool_name} 无法可靠完成，需说明原因并建议转人工",
        allow_handoff=True,
    )

# 玩具例子
r1 = output_policy(0.97, "search_ticket", True)
assert r1.status == "completed"
assert r1.allow_handoff is False

r2 = output_policy(0.82, "search_ticket", True)
assert r2.status == "partial"
assert r2.allow_handoff is True

r3 = output_policy(0.42, "send_email", False)
assert r3.status == "blocked"
assert "转人工" in r3.message
```

这段代码表达了三个原则。

第一，状态不是文案层的装饰，而是业务决策结果。`completed`、`partial`、`blocked` 应该直接驱动 UI，而不是让前端自己猜。

第二，部分结果必须满足“有可用片段”这个前提。若只有低置信度、没有任何可交付内容，就不能假装部分完成。

第三，人工接管不是异常路径，而是正式设计的一部分。`allow_handoff=True` 表示系统已经准备好把上下文、失败原因、当前中间结果一起交给人工。

再看一个更接近工程落地的伪接口设计：

| 字段 | 含义 | 示例 |
| --- | --- | --- |
| `status` | 当前任务级状态 | `partial` |
| `reason` | 降级原因 | `tool_timeout` |
| `confidence` | 当前结果置信度 | `0.82` |
| `artifacts` | 已拿到的中间结果 | 已创建的会议 ID |
| `next_actions` | 用户可立即执行的动作 | `retry`, `handoff` |

真实工程里，前后端通常要约定：当 `status=partial` 时，界面必须同时显示三件事，分别是当前可用结果、剩余缺口、下一步动作。否则“部分完成”这四个字没有实际意义。

## 工程权衡与常见坑

最大的工程权衡，是“成功率看起来高”与“系统真实可信”之间的冲突。很多团队会本能地偏向前者，于是开始隐藏失败、后台重试、合并状态，最后把错误从“显式失败”变成“隐式误导”。

常见坑可以直接列出来：

| 误区 | 实际后果 | 正确做法 |
| --- | --- | --- |
| 暗中重试，不告诉用户 | 用户以为系统卡死，或结果突然跳变 | 显示 `retry 2/5` 和预计等待时间 |
| 工具失败但仍输出完整文案 | 诱发幻觉和错误执行 | 降级为部分结果或拒绝 |
| 只展示结果，不展示依据 | 用户无法判断是否可信 | 显示来源、时间戳、缺口 |
| 转人工时丢上下文 | 用户被迫重复描述问题 | 结构化传递历史、参数、失败点 |

“结构化传递”这个术语，白话讲就是把上下文按固定字段打包，而不是让人工去翻聊天记录猜发生了什么。

一个很典型的坑是重复副作用。比如邮件接口超时后，后台自动重试 5 次。如果前端没有任何提示，用户可能会再次手动点击“发送”，最后导致收件人收到多封重复邮件。这个问题不是单纯靠重试策略能解决的，必须靠透明度解决。正确做法是状态横幅显示“阻塞：邮件服务超时，正在第 3 次重试”，并提供“取消重试”与“转人工”按钮。

另一个坑是把“部分完成”当作模糊免责。部分完成不是一句空话，而是一个严格格式：哪些部分完成了、哪些没完成、风险在哪里、用户下一步能做什么。如果只写“系统暂未完全完成，请稍后重试”，这不叫透明，而是把责任继续推给用户。

## 替代方案与适用边界

三级降级不是唯一方案，但它适合大多数中低风险、多工具、多步骤任务。若场景不同，可以选择替代方案。

| 方案 | 适用场景 | 优点 | 边界 |
| --- | --- | --- | --- |
| 缓存回退 | 查询型任务、短时工具故障 | 响应快，用户还能拿到近似结果 | 必须标注“非实时” |
| 规则回退 | 高重复、低复杂任务 | 稳定、成本低 | 无法覆盖长尾问题 |
| 人在环路监控 | 人力有限但可随时介入 | 降低人工成本 | 不适合必须事前审批的场景 |
| 直接人工接管 | 高风险或低置信度任务 | 最安全 | 成本高、扩展性差 |

“人在环路”这个术语，白话讲就是系统先自动运行，但人工在旁边监控，一旦触发条件就介入。

什么时候不应该使用“部分完成”？答案很明确：高风险且结果不可拆分的场景。例如医疗建议、法律判断、金融合规审批。这些任务一旦错误，部分结果本身也可能误导用户。在这种边界下，低置信度时更合理的策略是直接拒绝，并明确写明原因，再引导人工处理。

真实工程例子是金融客服。用户请求“帮我确认这笔大额转账是否符合风控要求并直接提交”。如果风控工具只返回了 60% 的置信度，那么系统不应该输出“初步看可以提交”。更安全的做法是：直接阻塞，展示“当前证据不足，无法自动提交”，并把已收集到的账户信息、规则命中项、失败日志一并交给人工审核。

所以，优雅降级不是“永远给点什么”，而是“在系统能力边界内，给出最诚实、最可操作的结果”。
