## 核心结论

MMMU 是一个面向多模态大模型的综合评测集。多模态的白话解释是：模型不是只读文字，还要同时处理图片、图表、公式、结构图，再结合文字题干作答。它的目标不是测“会不会看图”这一件事，而是一起检查三类能力：

| 维度 | 白话解释 | 在 MMMU 中的典型表现 | 常见指标 |
| --- | --- | --- | --- |
| 感知 Perception | 先把图里有什么看明白 | 识别坐标轴、器件标号、化学结构、医学图像特征 | 子项准确率 |
| 知识 Knowledge | 知道这门学科在说什么 | 理解电路、医学、法律、艺术史等领域概念 | 子项准确率 |
| 推理 Reasoning | 把看见的内容和知识串起来 | 根据图文联合条件排除错误选项、推导数值 | 子项准确率 |

MMMU 的关键价值有三点。

第一，它不是单学科小测，而是跨医学、工程、艺术等多个学科的大学级题目集合，所以更接近“真实复杂任务”的输入结构。

第二，它强调 zero-shot。zero-shot 的白话解释是：不给模型额外训练，也不给专门题库适配，只用统一提示词直接答题。这样测出来的结果更像模型原始能力，而不是“刷题能力”。

第三，它的总分通常写成准确率，但这个总分不能直接等同于“模型真的会做多模态推理”。如果一个模型主要靠 OCR，也就是把图里的字抄出来，再利用文字线索猜答案，它可能在某些题上得分不低，但这不代表它真的理解图表、专业图像或结构关系。

玩具例子可以这样理解：给模型一张地图和一句问题，“图中这个区域降雨最多是在什么时候？”它至少要做三件事：先看懂图例和颜色，再知道月份或季节的表达方式，最后把地区、时间和图中信号对上。只会读字不够，只会看色块也不够。

## 问题定义与边界

MMMU 的问题定义可以压缩成一句话：在统一 zero-shot 设定下，给模型图像与文本混合输入，要求它在多学科大学级问题上直接作答，用准确率衡量综合表现。

这里有几个边界必须说清楚。

第一，题目来源不是随便抓的图片问答，而是来自真实教材、考试、习题或学科材料。题型覆盖地图、流程图、表格、工程图、化学结构式、医学影像、艺术作品等多种异构图像。异构图像的白话解释是：图片类型差别很大，不是都长得像自然照片。

第二，它评的是“图文联合理解”。如果题目给出电路图和一段说明文字，模型需要同时利用两者。只看图可能缺少约束条件，只看字可能不知道题目指的是哪个元件。

第三，它默认不允许在这个 benchmark 上训练或微调。微调的白话解释是：拿这套题或相似数据继续训练模型，让模型专门适应这个任务。MMMU 的设计初衷正是避免“为了刷榜而学题”。

第四，它适合比较“统一提示、统一评测流程”下的综合多模态能力，不适合回答所有工程问题。比如你想测一个系统在长文档检索、多轮交互、工具调用、外部数据库连接下的表现，MMMU 就不是完整答案。

下表可以帮助区分题型与所需能力：

| 题目类型 | 输入长什么样 | 主要依赖能力 | 仅靠 OCR 是否够用 |
| --- | --- | --- | --- |
| 地图/地理图表 | 地图、色块、图例、地名 | 感知 + 基础知识 + 推理 | 往往不够 |
| 化学结构式 | 键角、官能团、分子结构 | 感知 + 专业知识 | 通常不够 |
| 工程图/电路图 | 标注尺寸、元件符号、连线 | 高精度感知 + 工程知识 | 大多不够 |
| 医学图像 | CT、病理图、解剖示意 | 专业视觉理解 + 医学知识 | 基本不够 |
| 课本截图/OCR 题 | 图里主要是文字 | OCR + 文本理解 | 可能足够 |

一个新手版例子：题目给出一张简单电路图，旁边写“哪个元件控制电流大小”。如果图中用标准符号画了电阻、开关和电源，模型至少要知道电阻符号是什么。这里“知道电阻符号”不是纯视觉识别，而是视觉符号和工程知识的绑定。

所以，MMMU 的边界很明确：它评的是大学级、跨学科、图文混合、统一 prompt、zero-shot 条件下的综合答题能力，不是任何多模态系统都能用一套分数解释清楚。

## 核心机制与推导

MMMU 常用整体准确率作为主指标，但更有解释力的是把它拆成技能树。技能树的白话解释是：把一个总能力拆成几类基础能力，再分别看短板在哪里。

常见拆分是：

- 感知：30%
- 知识：35%
- 推理：35%

可以写成一个简化公式：

$$
Accuracy = 0.3 \times Perception + 0.35 \times Knowledge + 0.35 \times Reasoning
$$

这个公式不是说官方一定逐题按这个线性加权打分，而是一个非常有用的分析框架：总准确率可以被视作三类能力贡献的组合。如果一个模型视觉编码器很强，但专业知识薄弱，那么它在图像识别题上可能不错，在医学、法律、工程交叉题上却会明显掉分。

再看评测流程，本质上有三步：

1. 图文联合理解  
2. 禁止训练或微调  
3. 用统一 prompt 直接答题  

为什么这三步重要？因为它们把“模型本身能力”和“工程外挂”尽量分开了。

玩具例子：题目问“图里显示的化学键角是多少”。模型需要先看懂结构图中哪个原子连接到哪个原子，这对应感知；再知道化学里常见构型和键角范围，这对应知识；最后把局部结构归类并得出数值或选项，这对应推理。三者缺一不可。

如果进一步形式化一点，可以把单题是否答对记为 $y_i \in \{0,1\}$，全数据集共有 $N$ 题，那么总体准确率是：

$$
Acc = \frac{1}{N}\sum_{i=1}^{N} y_i
$$

但工程上更有价值的是按题型分桶。例如：

- OCR 依赖题
- 图表理解题
- 专业图像题

设三类题的准确率分别为 $a_{ocr}, a_{chart}, a_{domain}$，占比分别为 $w_{ocr}, w_{chart}, w_{domain}$，则可以写成：

$$
Acc = w_{ocr}a_{ocr} + w_{chart}a_{chart} + w_{domain}a_{domain}
$$

这说明一个重要事实：总分高，不一定说明模型“真的懂图”。如果 $w_{ocr}$ 较大，而模型在 OCR 题上特别强，即便它在图表解析和专业图像题上很弱，总体分数也可能看起来还行。也正因为这个原因，MMMU 结果最好按题型和技能维度同时解释。

真实工程例子：Tech & Engineering 题里常见工程图纸、器件标号、尺寸标注。模型若把一张高分辨率图强行缩到很小尺寸，很多细节会丢失。此时失败并不一定是“不会推理”，而可能是前面的视觉感知输入已经坏了。也就是说，评测结果常常是“模型能力”和“输入处理链路”共同决定的。

## 代码实现

MMMU 的工程实现核心不复杂，复杂的是细节。通常流程是：读取题目元数据，加载图像与文本，拼接统一 prompt，调用模型，解析模型输出，再和标准答案比较。

下面先给一个玩具级、可运行的 Python 例子。它不调用真实多模态模型，只演示评测框架里最关键的两个动作：统一 prompt 构造和动态候选项处理。

```python
from typing import List, Dict

def build_prompt(question: str, choices: List[str]) -> str:
    lines = [
        "You are given an image and a question.",
        "Choose the best answer from the options.",
        "Return only the option letter."
    ]
    option_letters = "ABCDEFGHIJKLMNOPQRSTUVWXYZ"
    for i, choice in enumerate(choices):
        lines.append(f"{option_letters[i]}. {choice}")
    lines.append(f"Question: {question}")
    return "\n".join(lines)

def parse_answer(output: str, num_choices: int) -> str:
    valid = "ABCDEFGHIJKLMNOPQRSTUVWXYZ"[:num_choices]
    text = output.strip().upper()
    for ch in text:
        if ch in valid:
            return ch
    raise ValueError("No valid option found")

def evaluate(samples: List[Dict]) -> float:
    correct = 0
    for sample in samples:
        prompt = build_prompt(sample["question"], sample["choices"])
        assert "Return only the option letter." in prompt

        # 这里用假模型代替真实模型调用
        model_output = sample["mock_output"]
        pred = parse_answer(model_output, len(sample["choices"]))
        if pred == sample["answer"]:
            correct += 1

    return correct / len(samples)

samples = [
    {
        "question": "In the circuit image, which component mainly limits current?",
        "choices": ["Battery", "Resistor", "Switch", "Capacitor"],
        "answer": "B",
        "mock_output": "B"
    },
    {
        "question": "Which month has the highest rainfall in the map?",
        "choices": ["March", "June", "August", "December"],
        "answer": "C",
        "mock_output": "Option C"
    }
]

acc = evaluate(samples)
assert abs(acc - 1.0) < 1e-9
print("accuracy =", acc)
```

这个例子说明两件事。

第一，统一 prompt 很重要。所有题目尽量走同一个接口，减少“为了某类题临时改提示词”带来的偏差。

第二，候选项数量必须动态支持。基础 MMMU 常见为多选题，MMMU-Pro 把选项数扩展到 10 个，并且加入更强干扰项，甚至有纯视觉输入题。工程代码如果把选项数写死为 4，会直接失效。

伪代码结构通常像这样：

```python
def run_one_sample(model, image, question, choices):
    prompt = build_prompt(question, choices)
    response = model.generate(
        image=image,
        text=prompt,
        max_new_tokens=16
    )
    pred = parse_answer(response, len(choices))
    return pred
```

如果要兼容更强版本，接口最好从一开始就不要假设“必须有文本解释”或“必须有 4 个选项”：

```python
def run_multimodal_eval(model, image=None, text=None, choices=None):
    choices = choices or []
    prompt = build_prompt(text or "", choices)
    response = model.generate(image=image, text=prompt)
    return parse_answer(response, len(choices))
```

真实工程例子：如果你评测的是一个带视觉编码器的系统，图像输入阶段往往才是决定上限的地方。比如工程图纸上有很小的尺寸数字，医学图像有局部病灶，化学结构图里有细小键型差异。此时代码不能只做“把整张图缩到固定大小再送进去”。更稳妥的做法通常包括：

| 工程动作 | 作用 | 典型收益 | 风险 |
| --- | --- | --- | --- |
| 高分辨率输入 | 保留细节 | 提升小字、细线、标注识别 | 计算更贵 |
| 多裁剪 | 同时看全局和局部 | 提升图纸、表格、医学图 | 需要聚合策略 |
| 动态分辨率 | 按图像内容分配 token | 兼顾全图与细节 | 实现复杂 |
| 题型分流 | 不同题走不同前处理 | 结果更稳定 | 容易破坏统一评测 |

在 benchmark 语境下，这些技巧既是必要工程优化，也是解释分数时必须披露的条件。否则“同一个模型”的分数差异，可能只是来自不同的输入处理链路。

## 工程权衡与常见坑

MMMU 最常见的误读是：把总准确率直接当成“多模态理解能力”的单一真相。这个结论通常站不住。

第一个坑是 OCR 题和真正的视觉理解题混在一起。OCR 的白话解释是：先把图片里的文字提取出来，再像做普通阅读理解一样答题。如果一个模型 OCR 很强，它会在一些题上表现不错，但这不代表它会读图表关系、会看医学图、会识别化学结构。

第二个坑是图像前处理不当。视觉编码器通常有 token 限制。token 限制的白话解释是：模型一次能处理的视觉信息量有限，图像太大就得压缩、切块或抽样。如果前处理与模型训练时的分辨率策略不匹配，细节会系统性丢失，最终表现为“似乎不会做题”。

第三个坑是把错误都归咎于推理。实际上，很多题错在更前面：图没看清、标注读错、裁剪裁偏。后面的知识和推理再强也救不回来。

下面这个表更适合工程排查：

| 题型 | 容易出现的假高分/假低分 | 典型原因 | 解决方向 |
| --- | --- | --- | --- |
| OCR 依赖题 | 假高分 | 靠文字线索猜对，不是真懂图 | 单独统计 OCR 子集 |
| 图表理解题 | 假低分 | 坐标轴、图例、颜色映射丢失 | 提升分辨率，保留全图 |
| 专业图像题 | 假低分 | 领域图像分布与自然图不同 | 强化专业视觉编码或专门分析 |
| 工程图/图纸题 | 假低分 | 小字、细线、尺寸标注看不清 | 多裁剪 + 高分辨率 |
| 多选干扰题 | 假低分 | 选项扩展后靠排除法更难 | 支持更多候选项与更稳定解析 |

新手版例子很典型：一题图片里是大段印刷文字，模型把字读出来后就答对了；另一题是化学结构图，图里几乎没有可直接抄的文字，模型就突然失分。这不是“模型忽然变笨”，而是题目真正开始要求视觉结构理解了。

真实工程例子：在工程图纸场景中，一个尺寸标注如果从 `0.8 mm` 被压缩后读成 `0.3 mm`，后续所有推理都会错，而且错得很合理。你看输出会觉得模型逻辑通顺，但前提事实已经错了。多模态系统里这种“前端感知错误伪装成后端推理错误”的情况非常常见。

因此，使用 MMMU 时建议至少做三层报表：

1. 总准确率  
2. 按学科或题型拆分准确率  
3. 按输入链路条件记录结果，例如分辨率、裁剪策略、是否多图拼接  

只有这样，分数才有可解释性。

## 替代方案与适用边界

如果你只需要一个跨学科、图文混合、统一 zero-shot 条件下的综合评测，MMMU 很合适。但如果你关心“结果是否更抗刷榜、更多干扰项、是否能测纯视觉能力”，那 MMMU-Pro 往往更合适。

MMMU-Pro 的核心变化可以概括为两点：

- 选项从常见的 4 个扩到 10 个
- 引入纯视觉输入等更强设置

这意味着它对“靠模板猜答案”或“轻度文字线索”更不友好，能更明显地拉开真正视觉理解能力与投机式答题策略的差距。

可以用下表快速区分：

| Benchmark | 核心特点 | 更适合测什么 | 不适合什么 |
| --- | --- | --- | --- |
| MMMU | 多学科、图文混合、zero-shot 综合评测 | 综合多模态能力基线 | 细粒度纯视觉专项 |
| MMMU-Pro | 更强干扰项、更多选项、含纯视觉设置 | 更稳健的多模态能力检验 | 与旧榜单直接横比 |
| 其他单模态 benchmark | 只测文字或只测图像某一侧 | 单项能力诊断 | 图文联合理解 |

新手版选择建议可以直接说：

- 想测模型在工程图纸、教材截图、图文题干混合下的综合答题能力，用 MMMU。
- 想减少“靠四选一碰运气”或“轻度 OCR”带来的水分，优先看 MMMU-Pro。
- 想单独测 OCR、图表问答、医学影像、长文档检索，就要找更专项的 benchmark，不能只看 MMMU。

它的适用边界也必须明确：

第一，它主要适用于统一 prompt 的 zero-shot 比较。如果你的系统允许专门提示词工程、题型路由、外部工具调用、检索增强，那么结果已经不再只是“基础模型能力”。

第二，它不是生产系统全链路评测。真实系统还有延迟、成本、失败重试、上下文长度、图片上传质量、接口稳定性等约束，MMMU 只覆盖其中一部分。

第三，它不是某个行业上线前的充分验证。比如医疗、法律、工程设计这类高风险场景，需要更垂直、更高质量、带人工审查标准的数据集。MMMU 可以作为预筛选，但不能直接替代行业验收。
