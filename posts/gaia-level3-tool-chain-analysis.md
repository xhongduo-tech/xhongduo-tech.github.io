## 核心结论

GAIA Level-3 的难点，不是“模型会不会答题”，而是“系统能不能把一条长工具链稳定走完”。这里的工具链，指把搜索、文件读取、表格解析、代码执行、数值计算、答案格式化等多个外部能力按顺序串起来完成同一任务。

对零基础读者，可以先把它理解成一条流水线：先查资料，再取字段，再计算，再验证，再提交答案。单步看都不难，但只要其中一步把格式、单位、文件名、字段名弄错，后面的步骤就会被整体带偏。

GAIA 这类任务最适合用下面这个近似式理解：

$$
P_{total}=\prod_{i=1}^{n} p_i
$$

也就是总成功率约等于每一步成功率的连乘。若每一步成功率都是 $0.9$，那么 6 步链的总成功率只有：

$$
0.9^6 \approx 53.1\%
$$

8 步链则降到：

$$
0.9^8 \approx 43.0\%
$$

这就是为什么“单步能力看起来不错”的 Agent，一旦进入 GAIA Level-3，就会明显掉速。

下面这张表先给出一个直观对比：

| 对比项 | 理想短链任务 | GAIA Level-3 长链任务 |
|---|---:|---:|
| 典型步数 | 1-2 步 | 5-10 步起，公开教学材料里常把更长链路视作 Level-3 |
| 工具种类 | 1 种 | 2-5 种甚至更多 |
| 中间结果数量 | 少 | 多，且彼此依赖 |
| 失败位置 | 多在当前步 | 常在第 3-6 步暴露，但根因在更早一步 |
| 主要风险 | 模型不会做 | 模型会做，但链路没保真 |

还有一个必须澄清的时间点。2023 年 GAIA 论文中的基线，`GPT-4 + plugins` 总分约 15%，人类约 92%。2024 年一些早期通用 Agent 公开评测里，很多系统仍在很低区间徘徊，Level-3 常被视为“几乎过不去”的难关。但到 2025-06 的 GAIA 官方榜单，公开提交的顶尖系统 Level-3 已经到 65.31%；到 2026-03，榜单头部公开分数已超过 85%。所以，“当前最优系统 Level-3 通过率不足 5%”这个说法只适用于更早期的历史背景，不适用于 2025 年后的公开榜单。

## 问题定义与边界

GAIA 是一个面向通用 AI 助手的基准测试。基准测试，白话说，就是一套统一题目，用来测不同系统到底能不能完成真实任务。它考的不只是知识问答，还考推理、多模态理解、网页浏览、文件处理和工具使用。

GAIA Level-3 可以概括成一句话：任务目标通常很简单，但完成路径很长，且路径中必须跨工具、跨格式、跨状态地持续保持正确。

这类题目通常有几个共同特征：

| 维度 | Level-2 常见表现 | Level-3 常见表现 |
|---|---|---|
| 步数 | 多步，但链路较短 | 更长链路，常需持续规划与修正 |
| 工具 | 多工具，但切换有限 | 多工具组合更密集 |
| 文件处理 | 偶尔涉及 | 更常见，且常与搜索/计算耦合 |
| 验证点 | 可以少量人工补救 | 必须系统化校验 |
| 失败恢复 | 容易重来 | 重来成本高，需检查点 |

如果用一个新手版例子来理解，可以想象这样一个任务：

1. 搜索某公司年报。
2. 找到营收字段。
3. 下载附件表格。
4. 解析出季度数据。
5. 计算同比增速。
6. 按指定格式提交最终答案。

这就是典型的长链任务。你不能跳过，也不能把“看起来差不多”的中间结果直接往后传。

步数增长为什么危险，可以继续用近似式看：

$$
P(n)\propto 0.9^n
$$

这里的意思不是所有任务都恰好每步 90%，而是说明一个趋势：只要链路足够长，哪怕单步看起来已经很强，总体成功率也会快速下滑。

边界也要讲清楚。不是所有复杂任务都属于 Level-3。一个任务如果只是“问题难”，但基本不需要跨工具、不需要中间状态管理，它更像高难推理题，而不一定是典型 Level-3。真正的 Level-3 更像“流程系统题”，核心挑战是编排，不只是思考。

## 核心机制与推导

理解 Level-3，最重要的是把任务看成一张有依赖关系的图，而不是一句大提示词。

节点，白话说，就是流水线中的一个处理单元。比如：

- 节点 A：网页搜索
- 节点 B：文件下载
- 节点 C：表格解析
- 节点 D：数值计算
- 节点 E：答案格式化
- 节点 F：提交与验证

可以用一个简化图表示：

```text
[搜索]
  ↓
[定位来源页]
  ↓
[下载文件]
  ↓
[解析字段]
  ↓
[计算结果]
  ↓
[格式校验]
  ↓
[提交答案]
```

这条链为什么容易断？因为每个节点都不只是“产出文本”，而是产出“后续工具可消费的数据结构”。

例如：

- 搜索节点输出的不是“我觉得这个网页可能有用”，而应该是 `url + 可信来源 + 页面标题`
- 解析节点输出的不是“这里像是营收”，而应该是 `{"revenue": 123.4, "unit": "million USD", "period": "2024Q4"}`
- 计算节点输出的不是“增长挺多”，而应该是 `{"yoy": 0.183, "formula": "(a-b)/b"}`

一旦结构不稳定，错误就会传播。传播，白话说，就是上一步的小错会变成下一步的大错。比如第二步把单位从“百万美元”误当成“美元”，第三步公式是对的，最后答案仍然会错，而且你很难从最终答案反推出根因。

玩具例子可以非常小：

- 第一步搜索得到“某城市人口 12.5 million”
- 第二步解析成 `12.5`
- 第三步计算时误以为单位是“人”而不是“百万”
- 第四步再做密度计算，整个量级错了 $10^6$ 倍

模型没有“不会算”，链路是被中间语义错位拖垮了。

从工程角度，可以把总失败拆成三类：

| 失败类型 | 含义 | 典型后果 |
|---|---|---|
| 选择错误 | 工具或来源选错 | 后面全在错误分支上努力 |
| 结构错误 | 字段、格式、单位错 | 后续工具无法消费或静默误算 |
| 状态错误 | 丢上下文、覆盖旧结果 | 重复工作、回滚失败、最终答非所问 |

真实工程例子比玩具例子更能说明问题。比如一个投研 Agent 要输出“某企业过去 4 个季度自由现金流趋势”：

1. 先搜 IR 页面。
2. 再定位财报 PDF。
3. 从 PDF 中提取现金流量表。
4. 把“经营现金流”和“资本开支”字段取出。
5. 计算自由现金流 $FCF = CFO - CapEx$。
6. 再整理成季度趋势说明。
7. 最后输出符合风控模板的结论。

这里任何一步都可能把后续全部拖偏。最常见的不是模型完全找不到，而是：
- 找到的 PDF 版本不对。
- 提取字段时把半年累计值当成单季度值。
- `CapEx` 被识别成正值或负值方向相反。
- 最终输出的数字正确，但格式不符合验收器要求。

这就是 Level-3 的核心机制：问题不是局部能力不足，而是链路保真不足。

## 代码实现

工程上最稳的做法，不是让一个大模型“一口气想完”，而是把每一步封装成带校验的模块。校验，白话说，就是先检查数据像不像我们预期的样子，再决定要不要继续往下走。

先看一个最小可运行的 Python 玩具实现。它模拟“搜索 → 解析 → 计算 → 格式化”四步，每步都返回统一结构，并用 `assert` 做检查点。

```python
from dataclasses import dataclass
from typing import Any, Dict


@dataclass
class StepResult:
    ok: bool
    payload: Dict[str, Any]
    error: str = ""


def search() -> StepResult:
    # 模拟工具返回
    return StepResult(
        ok=True,
        payload={
            "source": "official_report",
            "revenue": 120.0,
            "cost": 90.0,
            "unit": "million_usd",
        },
    )


def parse(result: StepResult) -> StepResult:
    if not result.ok:
        return result
    data = result.payload
    required = {"revenue", "cost", "unit"}
    if not required.issubset(data):
        return StepResult(False, {}, "missing_fields")
    if data["unit"] != "million_usd":
        return StepResult(False, {}, "unexpected_unit")
    return StepResult(True, data)


def compute_margin(result: StepResult) -> StepResult:
    if not result.ok:
        return result
    revenue = result.payload["revenue"]
    cost = result.payload["cost"]
    if revenue <= 0:
        return StepResult(False, {}, "invalid_revenue")
    margin = (revenue - cost) / revenue
    return StepResult(True, {"margin": margin, "unit": "ratio"})


def format_answer(result: StepResult) -> StepResult:
    if not result.ok:
        return result
    margin = result.payload["margin"]
    return StepResult(True, {"final_answer": f"{margin:.2%}"})


r1 = search()
r2 = parse(r1)
r3 = compute_margin(r2)
r4 = format_answer(r3)

assert r1.ok
assert r2.ok
assert r3.ok
assert r4.payload["final_answer"] == "25.00%"
```

这个例子很小，但它体现了 Level-3 编排的三个必要结构：

| 结构 | 作用 | 最低要求 |
|---|---|---|
| 统一返回值 | 让后续步骤知道是否继续 | `ok / payload / error` |
| 输入校验 | 防止脏数据进入下一步 | 字段、类型、单位检查 |
| 检查点 | 让系统能回退 | 每个关键节点都能停下判断 |

如果把它扩展成接近真实 Agent 的伪代码，大致会是这样：

```text
run(task):
  state = init_state(task)

  r1 = web_search(task)
  checkpoint("search", r1, schema=["url", "title", "source_type"])

  r2 = fetch_or_open_file(r1.payload)
  checkpoint("file", r2, schema=["file_path", "mime_type"])

  r3 = parse_structured_data(r2.payload)
  checkpoint("parse", r3, schema=["value", "unit", "period"])

  r4 = compute(r3.payload)
  checkpoint("compute", r4, schema=["answer", "formula"])

  r5 = validate_final_answer(r4.payload)
  if not r5.ok:
      fallback_to_alternate_source()
      retry_from("search" or "parse")

  return r5
```

真实工程例子可以再具体一点。假设你做一个法务文档 Agent，要回答“合同中提前终止通知期是多少天”：

- 搜索模块返回候选文档列表
- 文档解析模块只接受 `pdf/docx/txt`
- 条款抽取模块必须返回 `{"clause_id", "text_span", "days"}`
- 冲突消解模块检查多个条款是否互相矛盾
- 最终输出模块只允许数字或极短短语

这里 schema 很关键。schema，白话说，就是“这份数据必须长成什么样”的约束。没有 schema，后续模块只能猜；一旦靠猜，长链就会越来越不稳定。

## 工程权衡与常见坑

工程上的核心权衡是：检查越多，成本越高；检查越少，链路越脆。

如果给每一步都加严格验证，链会变长，延迟会上升，调用成本也会增加。但如果不验证，错误会在后面放大，最后你会花更大的代价排查整条链。

可以把两种风险写成一个很粗糙但有用的比较：

$$
Cost \approx C_{tool}\cdot n + C_{check}\cdot k
$$

$$
Risk \approx 1-\prod_{i=1}^{n} p_i
$$

其中 $n$ 是工具步数，$k$ 是检查点数量。检查点增加会推高成本，但通常能提高某些关键步的成功率 $p_i$，因此总风险可能下降。

常见坑大致如下：

| 坑 | 现象 | 规避方法 |
|---|---|---|
| 无格式校验 | 第 4 步才发现字段名不对 | 每步做 schema 检查 |
| 单位未对齐 | 数字看着合理，最终量级错误 | 强制检查单位与量纲 |
| 工具选错 | 下载了网页却当结构化数据解析 | 给工具维护功能元数据 |
| 来源不可信 | 搜到了二手转述，后面全错 | 优先官方源，记录来源级别 |
| 缺少回滚 | 某步失败后只能整链重跑 | 设置检查点和重试起点 |
| 最终格式不合规 | 内容正确但验收失败 | 单独做 final answer validator |

新手最容易低估的是“工具选错”这一类。比如把“网页下载”当成“网页解析”用，拿到的只是原始 HTML 或截图，后续却按结构化表格去读，系统不会立刻崩，而是会在更后面的计算阶段出现莫名其妙的空值、错值或幻觉补全。

另一个坑是“静默错误”。静默，白话说，就是系统没有报错，但结果已经坏了。比如：

- CSV 第一行被当成数据行
- 日期 `03/04/2025` 被错认成不同格式
- 百分比 `18%` 被读成 `18` 而不是 `0.18`

这类错误比直接报错更难处理，因为它会伪装成“还能继续”。

## 替代方案与适用边界

并不是所有任务都值得做成一条完整自动长链。很多时候，把长链拆开，成功率反而更高。

最常见的替代方案有三类：

| 方案 | 特点 | 成本 | 成功率稳定性 | 适用场景 |
|---|---|---:|---:|---|
| 单体流水 | 一口气跑完 | 低 | 低 | 短链、低风险任务 |
| 分段 + 复核 | 每段完成后存快照 | 中 | 高 | 多工具长链 |
| 人类在环 | 关键节点人工确认 | 高 | 最高 | 高风险、高价值任务 |

“分段 + 复核”通常是 GAIA Level-3 更现实的工程解。做法很简单：把 6 步链拆成两个 3 步子链，每段结束都保存中间状态，只有前段校验通过，后段才启动。

一个简化流程可以写成：

```text
子链 A: 搜索 -> 下载 -> 解析
  成功? 是 -> 保存快照A
  成功? 否 -> 更换来源/工具重试

子链 B: 基于快照A -> 计算 -> 格式化 -> 提交
  成功? 是 -> 输出答案
  成功? 否 -> 回滚到快照A，不必全链重跑
```

这种方案的意义是把“整条链一次性成功”的要求，改成“每一段局部成功后再继续”。它不改变任务难度，但减少了错误传播半径。

还有一种常见替代，是引入“中间缓存 + 格式化服务”。缓存，白话说，就是把中间结果存下来，后面失败时不用重做；格式化服务则专门负责把自由文本变成严格结构化数据。对于 Level-3，这种职责拆分往往比让一个模型兼做“理解 + 计算 + 校验 + 输出”更稳。

适用边界也要明确：

- 短链、低风险、可容忍偶发错误的任务，用单体流水就够。
- 需要跨网页、文档、表格、代码执行的任务，应优先做分段编排。
- 金融、法务、医疗等高风险任务，不应把最终确认完全交给无监督长链。

结论很直接：GAIA Level-3 不是“更强一点的提示工程”，而是“需要状态管理、格式约束、失败恢复的系统工程”。
