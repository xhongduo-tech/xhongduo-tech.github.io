## 核心结论

Yi-VL 是 01.AI 基于 LLaVA 路线做的双语视觉语言模型。视觉语言模型，白话说，就是“同一个模型既看图也读字，再直接用自然语言回答”。它把图像先交给 `CLIP ViT-H/14` 编码，再用一个两层 `MLP` 投影到语言模型能接收的向量空间，最后交给 `Yi-6B-Chat` 或 `Yi-34B-Chat` 生成中文或英文回答。官方 README 明确给出的能力边界是：支持单图、多轮图文对话、中文和英文、448×448 分辨率输入。

这套设计成立，不是因为模型“真正理解”了图片，而是因为训练把“视觉特征”和“语言语义空间”对齐了。对齐，白话说，就是让“图片里的向量”和“词的向量”落到可比较、可共同推理的空间里。Yi-VL 采用三阶段训练：先在 224×224 上训练视觉编码器和投影层，再升到 448×448 提升细节识别，最后解冻整模型做多模态对话微调。这个顺序的目的很明确：先把图像接进语言空间，再补细节，最后再学聊天，不然训练会更不稳定。

对初级工程师最重要的结论有三个。第一，Yi-VL 的强项是“单图 + 问答/摘要”，不是多图推理，也不是视频。第二，448×448 是训练分辨率，高于这个分辨率不会自动带来额外知识，低于这个分辨率则可能丢细节。第三，复杂画面下依然会 hallucination。hallucination，白话说，就是模型会一本正经地说出图里并不存在的内容。这决定了它适合做客服辅助、图像摘要、OCR 问答草稿，不适合单独承担高风险结论。

## 问题定义与边界

Yi-VL 解决的问题可以写成一个很具体的函数：

$$
(\text{image}, \text{text context}) \rightarrow \text{text response}
$$

其中 `image` 当前按官方说明是单张图片，`text context` 是多轮对话历史，输出是中文或英文文本。这里的“多轮”很关键，它不是一次性图像描述器，而是一个可以围绕同一张图连续追问的系统。

把边界先说清楚，比罗列能力更重要。

| 项目 | Yi-VL 支持 | Yi-VL 不擅长或不支持 | 工程含义 |
| --- | --- | --- | --- |
| 输入模态 | 单张图片 + 文本 | 多图联合推理、视频 | 多视角任务要自己拆成多轮 |
| 语言 | 中文、英文 | 其他语言不应默认可用 | 面向中英客服较合适 |
| 输出形态 | 问答、描述、摘要 | 高确定性判决 | 最好做人机协作 |
| 图像分辨率 | 统一处理到 448×448 | 超高分辨率细节保真 | 要主动裁图、放大关键区域 |
| 场景复杂度 | 中等复杂图像理解 | 多目标拥挤场景 | 需加校验和置信度策略 |

玩具例子先看一个最小场景。你给模型一张衣服照片，再问：“这件衣服有口袋吗？” 这时任务边界很窄：只需要识别衣服局部结构并输出一句自然语言。这类问题通常是 Yi-VL 的舒适区。

真实工程例子是跨境电商客服。用户上传商品图，系统先生成中英双语摘要，例如“黑色连帽外套，前侧有拉链和两个侧袋”，然后客服继续追问：“口袋是拉链口袋还是开放口袋？”“英文详情页里该写 hooded jacket 还是 zip hoodie？” 这类系统最适合用 Yi-VL 做“统一图文入口”，因为一个模型就能同时处理图像理解和双语回答，不用在 OCR、图像标签、机器翻译、FAQ 检索之间来回切换。

但要注意，它的单图限制很硬。商品有正面图、背面图、内衬图三张时，Yi-VL 并不会天然把三张图联合成一个稳定结论。正确做法通常是拆为三次单图推理，再在业务层做聚合。否则模型可能把不同图片的信息混在一起。

## 核心机制与推导

Yi-VL 的核心可以压缩成三步：视觉编码、特征投影、语言生成。

### 1. 视觉编码

ViT，Vision Transformer，白话说，就是把图片切成很多小块，当成“视觉词”来处理。对于 448×448 输入，`ViT-H/14` 的 patch size 是 14，因此每张图会被切成：

$$
\frac{448}{14}\times\frac{448}{14}=32\times32=1024
$$

个 patch token。可以先把它理解成“1024 个局部观察结果”。

设视觉编码器输出为：

$$
F_{vit}\in\mathbb{R}^{N\times D}
$$

其中 $N$ 是视觉 token 数量，$D$ 是每个 token 的隐藏维度。

### 2. 特征投影

语言模型并不能直接吃 ViT 的输出，所以中间要有 projector。projector，白话说，就是“把视觉向量翻译成语言模型能接上的向量格式”。官方给出的结构是带 LayerNorm 的两层 MLP，因此可以写成：

$$
F_{proj}=\mathrm{MLP}(\mathrm{LN}(F_{vit}))
$$

这一步不是做分类，而是做空间对齐。你可以把它理解为：视觉侧说的是“像素切块后的统计表示”，语言侧说的是“词向量序列”，投影层负责把前者变成后者能参与统一注意力计算的表示。

### 3. 与语言模型融合并生成

这里有一个容易讲错的点。很多科普会笼统说“视觉特征通过 cross-attention 送进 LLM”。更准确的说法是：在 LLaVA 路线中，投影后的视觉特征通常被当作视觉 token，与文本 token 一起送入语言模型，在 Transformer 的注意力层中共同参与上下文建模。对新手来说，记住“视觉 token 进入语言上下文”比记“是不是严格单独一层 cross-attention”更重要，也更接近实际实现。

如果用户当前提问的文本 token 记为 $Q$，视觉 token 记为 $K,V$ 的一个来源，那么注意力的基本形式仍然是：

$$
\mathrm{Attention}(Q,K,V)=\mathrm{softmax}\left(\frac{QK^\top}{\sqrt{d}}\right)V
$$

白话解释就是：语言模型在生成下一个词时，会给图片里不同区域不同权重。问“口袋在哪里”，与问“这是什么颜色”，关注的视觉 token 会不同。

下面这张表可以把训练顺序和模块职责一次看清：

| 模块 | 输入 | 输出 | 训练阶段变化 | 主要职责 |
| --- | --- | --- | --- | --- |
| CLIP ViT-H/14 | 224 或 448 图像 | 视觉 patch 特征 $F_{vit}$ | Stage 1/2 训练，Stage 3 继续训练 | 提取图像局部与整体特征 |
| LayerNorm + 2-layer MLP | $F_{vit}$ | $F_{proj}$ | 三阶段都参与 | 把视觉特征映射到语言空间 |
| Yi-6B/34B-Chat | 文本 token + 视觉 token | 文本输出 | Stage 1/2 冻结，Stage 3 解冻 | 生成中英问答和摘要 |

玩具例子可以这样理解。假设图片里只有三个区域：帽子、拉链、口袋。模型在内部并不会写出这三个中文词，但可以把它近似想成三个视觉 token。用户问“Does this jacket have pockets?” 时，和“pockets”语义最接近的视觉 token 权重会更高，因此输出更可能围绕口袋展开，而不是去描述帽子。

官方 README 给出 Stage 3 大约使用 100 万图文对、global batch size 为 256、训练 2 个 epoch。更新步数约为：

$$
\frac{1{,}000{,}000\times 2}{256}\approx 7812.5
$$

这说明最后一阶段不是在“重新学会看图”，而是在已有对齐基础上，把整模型调成更适合多模态聊天的状态。

## 代码实现

实际部署时，最常见思路不是从零实现 Yi-VL，而是沿用 LLaVA 风格的推理流程：图片预处理、插入视觉 token、拼接对话模板、调用生成接口。下面的代码不是官方完整推理脚本，而是一个可运行的“机制最小版”，帮助你把关键数字和边界条件真正算清楚。

```python
from math import ceil

def vit_patch_count(image_size: int, patch_size: int = 14) -> int:
    assert image_size % patch_size == 0, "image_size 必须能被 patch_size 整除"
    side = image_size // patch_size
    return side * side

def training_updates(num_pairs: int, epochs: int, global_batch_size: int) -> int:
    assert num_pairs > 0 and epochs > 0 and global_batch_size > 0
    return ceil(num_pairs * epochs / global_batch_size)

def resize_policy(width: int, height: int, target: int = 448) -> tuple[int, int]:
    # 官方边界可简化理解为：推理时统一缩放到 448x448
    assert width > 0 and height > 0
    return (target, target)

patches_448 = vit_patch_count(448, 14)
assert patches_448 == 1024

stage3_steps = training_updates(1_000_000, 2, 256)
assert stage3_steps == 7813

assert resize_policy(1600, 900) == (448, 448)
assert resize_policy(224, 224) == (448, 448)

print("448 输入的 patch token 数:", patches_448)
print("Stage 3 近似更新步数:", stage3_steps)
print("推理时统一缩放策略:", resize_policy(800, 600))
```

如果你要写业务侧伪代码，真正需要关心的是下面四步，而不是底层矩阵乘法细节：

```python
# 伪代码：展示推理管线而非完整官方 API
model = load_yivl_model("01-ai/Yi-VL-6B")
image = load_image("jacket.jpg")
image_tensor = preprocess(image, size=448)

messages = [
    {"role": "system", "content": "You are a bilingual vision assistant. Answer in Chinese or English according to the user."},
    {"role": "user", "content": "<image>\n这件衣服有口袋吗？"}
]

output = model.generate(image=image_tensor, messages=messages)
print(output)
```

这里最容易被忽略的工程点有两个。

第一，提示词只决定输出风格，不决定视觉事实。你可以要求“先中文后英文回答”，但不能靠 prompt 让模糊图像突然变清楚。

第二，图像预处理比很多人想的重要。既然模型训练时强调 448×448，那么业务图像最好在进入模型前就做裁切和主体居中，而不是直接把一张超宽横图粗暴塞进去。否则 448 的像素预算会被背景浪费掉。

真实工程例子里，一个常见的服务链是：

1. 商品主图先经过主体检测与裁切。
2. 把裁切结果缩放到 448×448。
3. Yi-VL 生成中英摘要和属性问答草稿。
4. 规则系统校验高风险字段，例如“材质”“是否有帽”“是否有拉链”。
5. 最终结果写入商品详情页或客服辅助面板。

这个链路里，Yi-VL 是“生成和理解核心”，不是“最终裁决器”。

## 工程权衡与常见坑

Yi-VL 真正的工程价值，不在 benchmark 名次，而在你是否理解它的约束。

| 风险点 | 现象 | 典型原因 | 建议策略 |
| --- | --- | --- | --- |
| 分辨率损失 | 小字、细口袋、纹理识别差 | 所有输入都被压到 448×448 | 先裁关键区域，再送模型 |
| 多目标混淆 | 指错物体、漏掉小物体 | 画面里对象太多 | 单图拆区域，多轮追问 |
| hallucination | 说出不存在的结构 | 生成模型会补全缺失信息 | 增加规则校验或人工复核 |
| 多图任务失败 | 正反面信息混在一起 | 模型原生只支持单图 | 业务层做逐图推理与汇总 |
| 语言漂移 | 本该中文却夹英文术语 | 提示词不稳定、上下文太长 | 固定 system prompt 和输出模板 |

最常见的坑，是把“448×448 支持细粒度理解”误读成“任何大图细节都能被保住”。不是这样。官方说的是模型在 448×448 上训练，并非说它具备任意高分辨率细节恢复能力。如果原图是 3000×3000，而目标信息只占左下角一个很小标签，那么直接缩到 448×448 会把标签压没。正确做法是先检测或人工裁出标签区域，再送模型。

第二个坑，是把单图问答错用成多图汇总。比如客服连续上传“正面图”“背面图”“细节图”，然后问“这件衣服有几个口袋？” 如果你不在业务层标明“这是第几张图、这张图展示哪个视角”，模型就可能把前后文混在一起，给出看似流畅但事实不稳定的答案。

第三个坑，是把 Yi-VL 用在高风险场景里直接给最终判定。比如医学影像、工业质检、保险定损，这些任务共同特点是“错一点都不行”。Yi-VL 可以做初步描述或疑点召回，但不应单独负责最终结论。因为它的训练目标本质上是生成式对齐，不是可证明正确的判别式系统。

一个实用的 hallucination 提前检测策略是：同一张图让模型回答两次，一次开放式，一次约束式。比如先问“请描述这件衣服”，再问“只回答：有口袋/没有口袋/无法判断”。如果两次结论冲突，就打低置信度，进入人工复核。

## 替代方案与适用边界

如果你的任务刚好是“单图、中英、多轮问答、成本可控”，Yi-VL 很合适。但只要需求超出这个边界，就应该换思路，而不是硬堆 prompt。

| 方案 | 更适合什么任务 | 优点 | 代价 |
| --- | --- | --- | --- |
| Yi-VL | 单图双语问答、图像摘要、客服辅助 | 架构简单，中文英文都能用 | 单图限制明显，仍会 hallucination |
| 多图/视频专用模型 | 多视角商品、监控视频、长时序理解 | 原生支持跨帧或跨图融合 | 部署复杂，算力更高 |
| 高分辨率多模态模型 | 文档解析、UI、表格、小字识别 | 对细粒度区域更友好 | 显存和延迟更高 |
| Yi-VL + 检索/RAG | 需要结合商品库、知识库回答 | 能把“看图”与“查知识”结合 | 系统复杂度上升 |
| Yi-VL + verifier | 高风险回答前做二次验证 | 可降低虚构输出 | 需要额外模型和标注策略 |

RAG，Retrieval-Augmented Generation，白话说，就是“先查资料，再生成答案”。它不能替代视觉编码，但能约束文本事实。例如电商里图片能看出“像是一件连帽外套”，但具体材质、SKU、库存语言必须从商品库里查。此时更稳的路径是：Yi-VL 负责从图里抽取候选属性，RAG 负责拿知识库做确认，最后再生成客服答复。

医学影像是一个清晰的适用边界反例。Yi-VL 可以用来生成“初步观察草稿”，例如“图中存在明显高密度区域，建议进一步检查”，但不应该直接输出疾病诊断。因为这里需要的是经验证的专业判读流程，而不是一个通用视觉语言模型的开放式生成能力。

所以，Yi-VL 最适合的定位不是“万能看图模型”，而是“一个把图片和中英对话接到同一接口上的通用层”。只要你接受它的单图、448 分辨率、生成式误差这三个前提，它就很有工程价值；如果你不能接受，就该选多图、高分辨率、检索增强或专用判别模型。
