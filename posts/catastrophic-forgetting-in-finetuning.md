## 核心结论

灾难性遗忘，指模型在学习新任务后，明显丢失了预训练阶段或旧任务阶段已经获得的能力。白话说，就是模型把“原来会的东西”改没了。它不是模型突然失效，而是参数在持续更新时，被新任务的梯度反复推向另一片区域，导致旧能力依赖的表示被覆盖。

微调里最基础的更新规则是：

$$
\theta_{t+1} = \theta_t - \eta \nabla_\theta L_{\text{new}}(\theta_t)
$$

其中 $\theta$ 是模型参数，$\eta$ 是学习率，表示每一步改多大；$\nabla_\theta L_{\text{new}}$ 是新任务的梯度，表示“为了新任务更好，参数应该往哪里走”。如果训练过程只看新任务，这个方向未必兼容旧能力。

“完全不忘”通常不是现实目标。工程上真正要做的是控制权衡：新任务收益要拿到，但旧能力损失要可测、可接受、可回退。

| 目标 | 结果 |
|---|---|
| 只追新任务指标 | 新任务上升，旧能力下降 |
| 同时约束旧能力 | 新任务略降，整体更稳 |

玩具例子：一个小模型原本能做加法和乘法，你拿“只包含乘法”的数据继续训练。乘法准确率可能升高，但加法开始出错。这不是因为模型“不聪明”，而是同一组参数被重新调整后，原来支撑加法的参数关系被破坏了。

真实工程例子：一个通用大模型原本能做摘要、问答、代码补全。你用客服工单数据做全量微调后，客服回复变得更像企业话术，但摘要质量下降，代码补全更容易跑偏，通用常识问答也更保守。这就是典型的微调遗忘。

## 问题定义与边界

这里讨论的是“微调中的灾难性遗忘”，不是所有性能波动都算遗忘。先把边界划清，否则会把不同问题混成一种。

操作上，一个常见定义是：在微调前后，用同一组保留测试集比较旧能力分数，下降量就是遗忘量：

$$
\text{Forgetting} = \text{Score}_{\text{before}} - \text{Score}_{\text{after}}
$$

这里的保留集，指专门留出来监控旧能力的一组数据。白话说，它是“防止你只盯着新任务自我感觉良好”的基准尺。

需要区分三类现象：

1. 旧任务指标下降。这最接近标准意义上的遗忘。
2. 通用能力下降。比如摘要、推理、代码、常识整体退化，这也是遗忘，但范围更广。
3. 局部能力下降。比如只在某类术语、某个领域样本上变差，这可能是分布偏移，不一定是遗忘。

| 现象 | 典型表现 | 是否属于遗忘 |
|---|---|---|
| 旧任务掉分 | 原能力明显下降 | 是 |
| 新任务训练集高、验证集低 | 对新任务过拟合 | 不一定 |
| 某些样本类型变差 | 分布偏移 | 不一定 |

过拟合，指模型把训练样本记得太死，导致新任务验证集表现差。分布偏移，指训练和测试数据长得不一样。两者都可能伴随性能下降，但原因不同。

新手容易犯的错误是：只看“新任务更好了没有”，不看“旧能力掉了多少”。例如客服微调后，客服回答准确率从 78% 提到 90%，看起来成功；但如果通用摘要从 85% 掉到 68%，工程上就不能说它无条件成功，因为模型总效用变差了。

还有一个边界：如果你的目标本来就是把通用模型改造成单用途模型，那么某些旧能力下降是可以接受的。遗忘不是绝对坏事，关键在于它是否超出你的业务容忍度。

## 核心机制与推导

灾难性遗忘的本质，是参数共享下的目标冲突。参数共享，指同一套权重同时服务多个能力。白话说，模型没有“客服专用脑区”和“摘要专用脑区”完全隔离地存东西，很多能力复用同一批参数。

假设旧任务最优参数在 $\theta^\*$ 附近，而新任务梯度持续把参数往另一侧推。只要新任务更新方向和旧任务保持能力所需的方向不一致，旧能力就会被挤掉。可以把它理解成：新任务要求参数往右，旧任务要求别动甚至往左；如果训练时只看新任务，系统就会一直右移。

缓解思路通常分三类。

第一类是正则化约束。正则化，指在原损失外再加一个惩罚项，防止参数改得太离谱。EWC（Elastic Weight Consolidation）是代表方法：

$$
L = L_{\text{new}}(\theta) + \frac{\lambda}{2}\sum_i F_i(\theta_i-\theta_i^*)^2
$$

这里 $\theta_i^\*$ 是旧任务阶段的参数，$F_i$ 表示第 $i$ 个参数的重要性，通常用 Fisher 信息近似。白话说，重要参数像“不能随便动的螺丝”，越重要，偏离成本越高。

第二类是 replay，也叫经验回放。回放，指把旧分布样本混回训练过程，让模型不要只看新任务。目标函数可写成：

$$
L = (1-\beta)L_{\text{new}} + \beta L_{\text{replay}}
$$

其中 $\beta$ 控制旧数据占比。$\beta$ 越大，保旧能力越强，但新任务适应可能更慢。

第三类是 PEFT，也就是参数高效微调。参数高效微调，指冻结大部分预训练参数，只训练一小部分新增或可插拔参数。其抽象形式可以写成：

$$
\theta = \theta_0 + \Delta\theta_\phi
$$

$\theta_0$ 是冻结的原始参数，$\Delta\theta_\phi$ 是小规模可训练增量。LoRA 就属于这一类。因为底座参数不大动，所以天然比全量微调更不容易忘，但“不容易忘”不等于“不会忘”，因为输出行为仍然会被增量参数改写。

| 方法 | 核心思路 | 代价 |
|---|---|---|
| 正则化 | 约束重要参数别偏太多 | 需要估计重要性 |
| Replay | 让旧分布继续参与训练 | 需要保存旧数据 |
| PEFT | 只训练少量增量参数 | 能力上限受限于可训练容量 |

玩具例子可以把冲突写得更直白。假设一个二维参数模型，旧任务喜欢 $\theta=(1,1)$，新任务喜欢 $\theta=(3,3)$。如果你只优化新任务，参数会一路向 $(3,3)$ 走；如果旧任务在 $(1,1)$ 附近才工作好，那旧能力就会显著下降。EWC 的作用，就是给“偏离 $(1,1)$”加成本；replay 的作用，是持续提醒模型“别忘了旧题”；PEFT 的作用，是尽量少碰底座，只用小改动去适配新任务。

## 代码实现

代码层面，重点不是把训练脚本堆满，而是把“如何监控遗忘”和“如何组合约束”明确写出来。下面这个最小例子不依赖深度学习框架，用线性模型模拟“新任务学习”和“旧能力保留”的冲突，能直接运行。

```python
import math

def mse(xs, ys, w, b):
    return sum((w * x + b - y) ** 2 for x, y in zip(xs, ys)) / len(xs)

def grad_mse(xs, ys, w, b):
    dw = sum(2 * x * (w * x + b - y) for x, y in zip(xs, ys)) / len(xs)
    db = sum(2 * (w * x + b - y) for x, y in zip(xs, ys)) / len(xs)
    return dw, db

def train_step(
    w, b,
    new_batch,
    replay_batch,
    old_params,
    fisher,
    lr=0.05,
    beta=0.3,
    lam=5.0
):
    new_x, new_y = new_batch
    rep_x, rep_y = replay_batch

    dw_new, db_new = grad_mse(new_x, new_y, w, b)
    dw_rep, db_rep = grad_mse(rep_x, rep_y, w, b)

    # EWC: 对偏离旧参数的变化加惩罚
    w_old, b_old = old_params
    fw, fb = fisher
    dw_ewc = lam * fw * (w - w_old)
    db_ewc = lam * fb * (b - b_old)

    dw = (1 - beta) * dw_new + beta * dw_rep + dw_ewc
    db = (1 - beta) * db_new + beta * db_rep + db_ewc

    w = w - lr * dw
    b = b - lr * db
    return w, b

def eval_score(xs, ys, w, b):
    # 分数越高越好，这里用 1 / (1 + loss) 做简单映射
    loss = mse(xs, ys, w, b)
    return 1.0 / (1.0 + loss)

# 旧任务: y = 2x + 1
old_x = [0, 1, 2, 3]
old_y = [1, 3, 5, 7]

# 新任务: y = -x + 4，和旧任务方向冲突
new_x = [0, 1, 2, 3]
new_y = [4, 3, 2, 1]

# 预训练后参数，假设已拟合旧任务
w, b = 2.0, 1.0
old_score_before = eval_score(old_x, old_y, w, b)

# 保存旧参数和重要性估计
old_params = (w, b)
fisher = (1.0, 1.0)

for _ in range(80):
    w, b = train_step(
        w, b,
        new_batch=(new_x, new_y),
        replay_batch=(old_x, old_y),
        old_params=old_params,
        fisher=fisher,
        lr=0.03,
        beta=0.35,
        lam=3.0
    )

new_score_after = eval_score(new_x, new_y, w, b)
old_score_after = eval_score(old_x, old_y, w, b)
forgetting = old_score_before - old_score_after

assert new_score_after > 0.35
assert old_score_after > 0.20
assert forgetting >= 0.0

print("new_task_eval =", round(new_score_after, 4))
print("retain_eval   =", round(old_score_after, 4))
print("forgetting    =", round(forgetting, 4))
```

这个例子里有四个核心部件：

1. `new_loss`：新任务损失，对应 `grad_mse(new_x, new_y, ...)`。
2. `replay_loss`：旧数据回放损失，对应 `grad_mse(rep_x, rep_y, ...)`。
3. `ewc_loss`：偏离旧参数的惩罚，对应 `dw_ewc` 和 `db_ewc`。
4. `new_task_eval` 与 `retain_eval`：分别评估新任务和保留能力。

真实工程里，把这个骨架替换成 Transformer 训练即可。比如客服微调时，每个 batch 不只取客服指令，还混入一部分通用问答、摘要、代码样本作为 replay；同时如果你有上一阶段模型，可以对关键层施加约束项，或者直接用 LoRA 减少底座漂移。

| 参数 | 作用 | 调整方向 |
|---|---|---|
| `lr` | 控制更新幅度 | 遗忘重就降 |
| `beta` | 回放占比 | 旧能力差就升 |
| `lambda` | 正则强度 | 参数漂移大就升 |
| `epochs` | 训练轮数 | 过训就降 |

训练日志也不能只打新任务分数，至少要同时打印两组：

```text
step=500  new_task_eval=0.91  retain_eval=0.82  gap=0.09
step=1000 new_task_eval=0.93  retain_eval=0.76  gap=0.17
step=1500 new_task_eval=0.94  retain_eval=0.68  gap=0.26
```

这里的 `gap` 可以定义为旧能力下降量或两类指标差值。只要 `new_task_eval` 继续涨而 `retain_eval` 持续掉，你就应该把它视为遗忘告警，而不是单纯的训练进展。

## 工程权衡与常见坑

工程上最常见的问题，不是“不会用高级算法”，而是训练设置太激进。

第一，学习率过大。学习率决定每次参数移动多远。白话说，它像扳手力度，拧太猛，旧结构容易被一起拧坏。很多遗忘并不是数据本身导致，而是参数更新幅度过大。

第二，训练轮数过长。尤其在小数据领域微调中，前几轮拿到主要收益，后几轮主要在覆盖底座。新任务还能涨一点，但保留集已经掉得很明显。

第三，replay 占比过低。很多团队只象征性混一点通用数据，比如 1% 到 2%，结果约束几乎不起作用。旧分布如果对业务重要，就不要把 replay 当成装饰项。

第四，只做 PEFT 就停止监控。PEFT 天然缓和遗忘，但并不提供数学上的“保留保证”。LoRA 也可能把输出风格、拒答边界、知识调用模式改偏。

| 常见坑 | 后果 | 规避方式 |
|---|---|---|
| 只看新任务指标 | 旧能力掉了不知道 | 维护 retain set |
| 学习率太大 | 更新过猛，遗忘加重 | 先降 `lr` |
| replay 太少 | 旧分布约束不足 | 提高旧样本占比 |
| 只做 PEFT 就放松监控 | 误以为不会遗忘 | 仍然做保留集评估 |

真实工程例子：做企业知识库问答微调时，团队往往只评“检索增强后的问答准确率”。但用户真正会混着问常识题、摘要题、代码题。上线后如果模型在知识库场景很好，却在非知识库问题上显著退化，客服工单会变多。解决方式不是争论“这算不算模型问题”，而是在验收前就建立 retain set，包含通用问答、摘要、指令跟随、代码格式遵循等维度。

还有一个坑是指标设计不完整。你不能只测单一平均分。遗忘常常先发生在边角能力上，比如长上下文摘要、格式严格输出、低频知识调用、代码风格一致性。这些能力在总分里占比小，但对线上体验影响很大。

## 替代方案与适用边界

没有一种方法适合所有场景。你要先定义业务约束，再选方案。

如果资源紧、上线快、目标只是让模型学会一个领域表达风格，LoRA 或其他 PEFT 方法通常够用。它改动小、成本低、回退也简单。但如果你要求“通用能力尽量不掉”，单独 PEFT 往往不够，需要配合 replay 或额外评估。

如果旧能力必须保留，比如一个内部助手既要懂公司流程，又要保留通用写作和代码能力，那么 replay 更直接，因为它从数据分布上持续提醒模型“旧世界还存在”。缺点是你得有旧数据，且要处理隐私、存储、采样策略。

如果你拿不到旧数据，或者任务是阶段式连续学习，EWC 这类正则化方法有价值。它不需要完整旧样本，但依赖“重要参数估计”是否稳定。任务差异太大时，这种约束可能不够强。

如果你一开始就知道模型要长期服务多个目标，那么与其一次次单任务微调，不如直接做多任务训练。多任务训练，指把多类任务一起组织进统一训练目标中。白话说，不是“先学 A 再学 B”，而是“让 A 和 B 从一开始就共存”。它通常更稳，但数据设计和采样更复杂。

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| LoRA / PEFT | 训练便宜、改动小 | 不保证完全不忘 | 资源受限的微调 |
| Replay | 保留旧分布 | 需要旧数据 | 旧能力必须保留 |
| 正则化 | 不用额外旧数据也能约束 | 重要性估计可能不稳 | 连续任务训练 |
| 多任务训练 | 天然更稳 | 数据组织更复杂 | 多目标共存 |

可以把选择标准抽象成一个总效用函数：

$$
\text{Total Utility} = \text{New Task Gain} - \gamma \cdot \text{Old Task Loss}
$$

这里 $\gamma$ 表示旧能力损失的业务代价。若 $\gamma$ 很大，比如线上产品严重依赖通用能力，那么你就不能为了几点新任务涨幅接受大幅遗忘。反过来，如果这是单场景离线分类器，旧能力几乎无业务价值，那么允许更强的任务适配是合理的。

结论很朴素：不是先问“哪种方法最先进”，而是先问“旧能力值多少钱”。
