## 核心结论

TextCNN 是一种用于文本分类的卷积神经网络：它把一句话表示成词向量矩阵，用多个不同宽度的卷积核扫描局部词片段，再用最大池化取出最强特征，最后用分类器输出类别。

它的本质是：**用不同宽度的卷积核抓局部短语特征，再用最大池化挑出最强证据完成分类**。

例如用户输入一句客服投诉：“退款失败，一直没人处理”。这类短句通常不需要模型理解复杂的长篇逻辑，只要识别出“退款失败”“没人处理”这些局部高权重短语，就能判断它大概率属于“售后问题”或“负面投诉”。TextCNN 正适合这种局部模式明显、句子较短、类别边界相对清晰的任务。

它比纯词袋模型更强，原因在于词袋模型只关心词是否出现，通常不显式建模词序；TextCNN 会看连续窗口。例如“退款 成功”和“退款 失败”共享“退款”这个词，但局部短语含义相反。卷积核扫描 2-gram 或 3-gram 时，可以把这些相邻词组合成特征。

| 卷积核尺寸 | 对应 n-gram 模式 | 可能捕捉的短语 | 最大池化作用 | 分类依据 |
|---|---|---|---|---|
| 2 | 二元短语 | “退款失败” | 取出最强负面证据 | 售后/负面 |
| 3 | 三元短语 | “一直没人处理” | 保留最明显投诉片段 | 投诉/工单 |
| 4 | 四元短语 | “申请退款一直失败” | 保留更长局部模式 | 退款异常 |

TextCNN 的核心优势是结构简单、训练快、对短文本分类很稳定。它的核心限制也很明确：它主要依赖局部窗口和最大池化，不擅长需要长距离依赖、跨句推理或全局证据整合的任务。

## 问题定义与边界

文本分类是把一段文本映射到一个预定义类别的任务。预定义类别指训练前已经确定好的标签集合，例如“正面、负面、中立”，或者“退款、物流、账号、其他”。

TextCNN 处理的是**文本分类**，不是序列标注、生成或抽取。序列标注是给每个词打标签，例如命名实体识别；生成是让模型续写文本；抽取是从文本中找出答案片段。TextCNN 的输出通常是一个类别，或者多个类别的概率。

评论情感分类是一个典型例子：

输入：“这个耳机音质不错，但是续航太差。”

输出：负面，或中立，取决于标注规则。

如果证据集中在一个短句或几个局部短语中，TextCNN 通常够用。如果证据分散在整篇文档里，例如一篇长评先夸产品、再讲售后失败、最后给出总体否定结论，TextCNN 就不合适，因为单个局部最强特征可能无法代表全文立场。

| 项目 | 说明 |
|---|---|
| 输入 | 分词后的文本序列，例如 `["退款", "失败", "一直", "没人", "处理"]` |
| 输出 | 类别概率，例如“售后问题: 0.82，物流问题: 0.11，其他: 0.07” |
| 适用场景 | 短评论、客服意图识别、新闻标题分类、垃圾短信识别 |
| 不适用场景 | 长文档立场判断、跨句推理、多轮对话理解、复杂因果判断 |

一个玩具例子是判断短信是否为垃圾短信：

“免费领取优惠券，点击链接激活”

TextCNN 可能抓到“免费领取”“点击链接”这些局部特征，从而输出“垃圾短信”。

一个真实工程例子是电商客服工单分类。用户输入“退款不到账，客服一直不回复”，系统需要把工单路由到“退款售后”队列。这里类别通常由少量关键词和短语决定，TextCNN 的局部窗口机制可以直接发挥作用。

## 核心机制与推导

TextCNN 的数据流可以按六步理解：词向量矩阵、卷积滑动、非线性激活、最大池化、特征拼接、softmax 分类。

词向量是把词映射成数字向量的表示方式。比如“退款”不再是字符串，而是一个长度为 $d$ 的向量。假设一句话长度为 $T$，每个词向量维度为 $d$，输入矩阵记为：

$X=[x_1,\dots,x_T]^\top$

其中 $x_i \in \mathbb{R}^d$，所以 $X \in \mathbb{R}^{T \times d}$。

卷积核可以理解为一个固定高度的局部模式检测器。若卷积核窗口大小为 $h$，它每次读取连续 $h$ 个词的向量，即 $X_{i:i+h-1}$。对第 $i$ 个窗口，卷积输出为：

$c_i^{(h)}=\mathrm{ReLU}(\langle W^{(h)}, X_{i:i+h-1}\rangle+b^{(h)})$

这里 $W^{(h)}$ 是窗口大小为 $h$ 的卷积核参数，$b^{(h)}$ 是偏置，$\langle \cdot,\cdot \rangle$ 表示对应元素相乘后求和。ReLU 是一种非线性激活函数，白话说就是把负数截断为 0，让模型只保留被激活的模式。

对所有位置滑动后，会得到一串特征值：

$c^{(h)}=[c_1^{(h)},c_2^{(h)},\dots,c_{T-h+1}^{(h)}]$

最大池化是从这些位置中取最大值：

$\hat c^{(h)}=\max_i c_i^{(h)}$

它的含义是：不关心某个特征出现在哪个位置，只关心它是否强烈出现过。对于短文本分类，这个假设经常有效。比如“退款失败”出现在句首还是句尾，都足以说明它是售后问题。

如果有多个窗口大小，例如 $h \in \{2,3,4\}$，每种窗口会产生一组池化后的特征。把它们拼接成向量 $z$，再输入分类器：

$p=\mathrm{softmax}(W_o z+b_o)$

softmax 是把一组分数转换成概率分布的函数。比如输出 `[2.1, 0.3, -0.5]`，经过 softmax 后可能变成 `[0.80, 0.13, 0.07]`。

继续看短句：“退款失败，一直没人处理”。

| 窗口大小 | 滑动片段 | 模型可能学到的响应 |
|---|---|---|
| 2-gram | “退款 失败” | 强烈指向退款异常 |
| 2-gram | “没人 处理” | 指向服务投诉 |
| 3-gram | “一直 没人 处理” | 更强地指向负面售后体验 |
| 3-gram | “退款 失败 一直” | 指向退款流程异常 |

最大池化会保留每个卷积核在全句中的最大响应。例如某个 2-gram 卷积核专门检测“退款异常”模式，它在“退款失败”位置输出 3.8，在其他位置输出 0.2、0.1，那么最大池化后只留下 3.8。分类器不需要知道这个证据出现在第几个词，只需要知道它出现得很强。

| 模块 | 作用 | 白话解释 |
|---|---|---|
| 多窗口卷积 | 捕捉不同长度的局部短语 | 同时看 2 个词、3 个词、4 个词的组合 |
| ReLU | 引入非线性 | 只保留被明显触发的模式 |
| 最大池化 | 提取最强证据 | 一句话里最像某类的片段被保留下来 |
| 拼接分类器 | 汇总多类局部证据 | 把不同窗口发现的证据合并后判断类别 |

## 代码实现

实现 TextCNN 通常只需要三块：文本向量化、卷积池化模块、分类头。

文本向量化负责把 token id 变成 embedding。卷积池化模块负责执行 `Conv1d -> ReLU -> MaxPool`。分类头负责把拼接后的特征映射到类别数。

下面是一个可运行的 PyTorch 代码骨架，展示核心数据流：

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


class TextCNN(nn.Module):
    def __init__(self, vocab_size, embed_dim, num_classes, kernel_sizes=(2, 3, 4), num_channels=8):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.convs = nn.ModuleList([
            nn.Conv1d(
                in_channels=embed_dim,
                out_channels=num_channels,
                kernel_size=k
            )
            for k in kernel_sizes
        ])
        self.classifier = nn.Linear(num_channels * len(kernel_sizes), num_classes)

    def forward(self, input_ids):
        # input_ids: [batch, seq_len]
        x = self.embedding(input_ids)          # [batch, seq_len, embed_dim]
        x = x.transpose(1, 2)                  # [batch, embed_dim, seq_len]

        pooled_features = []
        for conv in self.convs:
            c = F.relu(conv(x))                # [batch, channels, seq_len-k+1]
            p = F.max_pool1d(c, kernel_size=c.size(2)).squeeze(2)
            pooled_features.append(p)          # [batch, channels]

        z = torch.cat(pooled_features, dim=1)  # [batch, channels * len(kernel_sizes)]
        logits = self.classifier(z)            # [batch, num_classes]
        return logits


model = TextCNN(vocab_size=100, embed_dim=16, num_classes=3)
batch = torch.tensor([
    [1, 2, 3, 4, 5, 0],
    [6, 7, 8, 9, 0, 0],
])
logits = model(batch)

assert logits.shape == (2, 3)
assert torch.isfinite(logits).all()
```

张量形状变化如下：

| 阶段 | 张量形状 | 说明 |
|---|---|---|
| 输入 | `[batch, seq_len]` | 每个位置是一个 token id |
| embedding 后 | `[batch, seq_len, dim]` | 每个 token 变成词向量 |
| 转置后 | `[batch, dim, seq_len]` | `Conv1d` 要求通道维在中间 |
| 卷积后 | `[batch, channels, seq_len-h+1]` | 每种窗口得到一组局部响应 |
| 池化后 | `[batch, channels]` | 每个卷积核保留一个最大响应 |
| 拼接后 | `[batch, channels * kernel_count]` | 合并所有窗口特征 |
| 分类后 | `[batch, num_classes]` | 每个类别一个 logit |

训练和推理时最容易错的地方有三个。

第一是 padding。padding 是为了把不同长度的句子补到同一长度。短句补 0 后，卷积可能扫到 padding 区域。实际工程中要设置 `padding_idx=0`，并尽量让训练和推理使用一致的最大长度。对于极短文本，如果句长小于最大卷积核尺寸，要么过滤，要么补齐到至少能被最大窗口扫描。

第二是 tokenizer 一致性。tokenizer 是把文本切分成 token 的工具。训练时用字级、推理时用词级，或者训练时保留大小写、推理时全部转小写，都会导致输入分布变化。

第三是标签定义稳定性。如果“退款失败”有时标为“售后问题”，有时标为“支付问题”，TextCNN 会学到混乱边界。模型结构解决不了标注规范不一致的问题。

## 工程权衡与常见坑

TextCNN 的强项是快、稳、易训。它参数量小，推理延迟低，对中小规模数据也比较友好。在短文本分类、意图识别、标题分类这类任务中，它经常是一个性价比很高的基线模型。

它的弱项是对长依赖和复杂推理支持有限。最大池化会把每个卷积核的最强位置拿出来，但会丢掉大量位置信息和组合关系。对于“虽然前面提到退款成功，但最后说明实际没有到账”这类句子，真正结论可能依赖全句结构，而不是某个局部短语。

一个典型失败句子是：

“我不是说这个服务差，只是退款流程太慢。”

如果训练数据里“服务差”经常对应负面，TextCNN 可能强烈响应“服务差”，忽略“不是说”带来的否定关系。否定词和被否定对象距离较远时，局部窗口更容易抓错证据。

| 问题 | 现象 | 规避方式 |
|---|---|---|
| 窗口设计不合理 | 只会抓单词或过短片段，短语模式不完整 | 常用 `2,3,4` 或 `3,4,5`，根据文本长度调参 |
| 池化偏差 | 一个强特征压过其他上下文 | 尝试 top-k pooling、attention pooling，或改用 Transformer |
| 数据分布漂移 | 线上新说法识别差 | 定期抽样线上数据，做再训练和词表更新 |
| 类别不平衡 | 大类准确率高，小类召回差 | 使用 class weight、重采样、阈值调整 |
| 分词不一致 | 离线指标正常，线上效果异常 | 固化 tokenizer、词表、清洗规则和最大长度 |
| 过拟合高频词 | 看到“退款”就判售后，忽略上下文 | 增加困难负例，例如“退款已成功” |
| 句子过长 | 局部强证据无法代表全文 | 截断策略要可解释，必要时换长上下文模型 |

真实工程中，一个常见做法是先用 TextCNN 做强基线：如果它已经能满足准确率、延迟和成本要求，就不急着上更复杂模型。如果发现错误样本集中在否定、转折、跨句引用、长文本证据分散这几类问题上，再考虑升级模型。

## 替代方案与适用边界

不该用 TextCNN 的情况通常有四类：文本很长、证据分散、类别判断依赖全局结构、需要跨句推理。此时局部窗口和最大池化会变成限制。

BiLSTM 是双向长短期记忆网络，白话说就是从左到右和从右到左读句子，保留一定顺序信息。它比 TextCNN 更适合顺序关系明显的句子，但训练和推理通常更慢。

Transformer 是基于自注意力机制的模型。自注意力机制可以让每个 token 直接关注其他 token，适合建模长距离依赖和全局关系。BERT、RoBERTa、DeBERTa 等都属于 Transformer 类预训练模型。它们通常效果更强，但成本更高，数据、显存和部署复杂度也更高。

对比两个任务：

短评论情感分类：“包装破了，客服不处理。”

这个任务证据集中，TextCNN 很合适。

长文档立场判断：一篇文章先介绍政策背景，中间引用多方观点，最后在结尾表达反对态度。

这个任务需要全局理解和跨段落整合，更适合 Transformer 类模型。

| 任务特征 | TextCNN | BiLSTM | Transformer |
|---|---|---|---|
| 短文本、局部关键词明显 | 推荐，速度快且稳定 | 可用，但不一定划算 | 可用，但可能成本偏高 |
| 顺序关系重要 | 能处理一部分局部顺序 | 推荐，能建模前后依赖 | 推荐，效果通常更强 |
| 长上下文 | 不推荐，池化会丢信息 | 可用但长序列效率一般 | 推荐，尤其是长文本变体 |
| 跨句推理 | 不推荐 | 有限支持 | 推荐 |
| 小数据、低延迟部署 | 推荐 | 可用 | 需要权衡成本 |
| 证据分散在全文 | 不推荐 | 可作为中间方案 | 推荐 |

实际选择可以按这个顺序判断：如果任务是短文本分类，先试 TextCNN；如果顺序和否定关系经常出错，试 BiLSTM 或 TextCNN 加注意力；如果需要长上下文、跨句推理或更强语义理解，直接考虑 Transformer。
