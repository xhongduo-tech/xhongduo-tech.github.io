## 核心结论

零样本文本分类是在没有任务标注样本的情况下，直接用预训练模型把输入文本分到候选标签集合里。这里的“预训练模型”指已经在大量通用文本上学过语言规律的模型；“候选标签集合”指业务提前给定的一组可选类别，例如 `退款 / 物流 / 账号 / 技术故障`。

它的核心不是重新训练一个分类器，而是把“文本”和“标签”都当成语言对象，让模型判断它们之间的关系。统一写成公式就是：

$$
\hat{y}=\arg\max_i s_i
$$

其中 $s_i$ 是输入文本属于第 $i$ 个标签的分数，$\hat{y}$ 是分数最高的预测标签。

玩具例子：

输入文本：

```text
I want a refund for a broken phone.
```

候选标签：

```text
退款 / 法律 / 咨询
```

如果模型给出分数：

| 标签 | 分数 |
|---|---:|
| 退款 | 0.84 |
| 法律 | 0.27 |
| 咨询 | 0.19 |

那么预测结果就是 `退款`。这个过程不需要先收集几百条“退款”样本，也不需要专门训练一个退款分类器。

| 项目 | 传统监督分类 | 零样本文本分类 |
|---|---|---|
| 是否需要标注数据 | 需要 | 不需要 |
| 是否要训练任务分类器 | 通常需要 | 通常不需要 |
| 上线速度 | 慢 | 快 |
| 适合场景 | 标签稳定、数据充足 | 新标签、新领域、样本稀缺 |

零样本文本分类适合新业务快速试运行。它的工程价值是先把系统跑起来，尽快获得真实样本、错误案例和标签反馈，再决定是否升级为少样本学习或监督微调。

## 问题定义与边界

零样本文本分类不是让模型自由发挥，也不是让模型随便生成一个词。它是一个有明确输入、明确输出、明确候选范围的分类问题。

给定一段输入文本 $x$，以及一个固定候选标签集合：

$$
Y=\{t_1,t_2,\ldots,t_K\}
$$

模型要从 $Y$ 中选出一个或多个最合适的标签。这里的“固定候选标签集合”很重要，因为分类结果必须落在业务定义好的标签里。

| 符号 | 含义 |
|---|---|
| `x` | 输入文本 |
| `Y={t_1,...,t_K}` | 候选标签集合 |
| `t_i` | 第 `i` 个标签 |
| `h_i=template(t_i)` | 标签模板化后的文本 |
| `u=e(x)` | 输入文本向量 |
| `v_i=e(t_i)` | 标签向量 |
| `s_i` | 标签分数 |

“文本向量”是把一段文字变成一组数字，用这些数字表示语义位置。例如“退款”和“退钱”在语义空间里应该更接近，而“退款”和“法律责任”通常更远。

“模板化”是把短标签改写成完整句子。例如标签 `退款` 可以变成：

```text
这段文本的主题是退款。
```

在自然语言推断路线里，完整句子通常比单个词更适合模型判断。

零样本文本分类适合这些任务：

| 场景 | 是否适合 | 原因 |
|---|---|---|
| 文本分类 | 适合 | 标签通常可以用自然语言描述 |
| 意图识别 | 适合 | 用户表达和意图标签有语义关系 |
| 工单分流 | 适合 | 标签清晰，业务需要快速上线 |
| 主题归类 | 适合 | 主题标签通常有明确语义 |
| 高风险法律判断 | 不适合直接使用 | 需要严格证据、规则和专业校准 |
| 医疗诊断结论 | 不适合直接使用 | 错误成本高，需要专业验证 |
| 标签极多且相近 | 需要谨慎 | 相似标签容易互相干扰 |

真实工程例子：一个客服系统要给新上线的产品线做工单分流，但还没有历史标注数据。业务只给出 `退款 / 物流 / 账号 / 技术故障` 四个标签。此时可以先用零样本文本分类上线，把低置信度工单转人工，并记录人工修正结果。等积累到足够样本后，再训练更稳定的监督分类器。

边界也要说清楚：零样本依赖模型已有的语言能力、标签命名质量、模板设计和分数阈值。它能减少冷启动成本，但不能替代长期生产系统里的评估、监控和迭代。

## 核心机制与推导

零样本文本分类主要有三条路线：语义相似度、自然语言推断、生成式分类。三者看起来不同，本质目标相同：把“输入文本和标签之间的关系”转成可比较的分数。

第一条路线是语义相似度。它把输入文本和标签都编码成向量，再比较向量距离。编码器 $e(\cdot)$ 是把文本转换成向量的模型。

$$
u=e(x)
$$

$$
v_i=e(t_i)
$$

然后用余弦相似度计算分数：

$$
s_i=\cos(u,v_i)
$$

余弦相似度衡量两个向量方向是否接近，方向越接近，语义越相似。最后选择分数最高的标签：

$$
\hat{y}=\arg\max_i s_i
$$

这条路线直观、速度快、容易实现。问题是它对标签写法敏感。`法律`、`法务`、`法律咨询` 看起来接近，但在向量空间里的位置可能不同，最终分数也可能不同。

第二条路线是自然语言推断。自然语言推断，也叫 NLI，是判断一个前提句是否支持一个假设句的任务。比如前提是：

```text
I want a refund for a broken phone.
```

假设是：

```text
这段文本的主题是退款。
```

如果模型认为前提支持假设，就输出较高的 `entailment` 概率。`entailment` 的意思是“蕴含”，即前一句能够推出后一句。

对每个标签 $t_i$，先构造假设句：

$$
h_i=template(t_i)
$$

再计算：

$$
s_i=P(\text{entail} \mid \text{premise}=x,\text{hypothesis}=h_i)
$$

最后仍然取最高分：

$$
\hat{y}=\arg\max_i s_i
$$

NLI 路线的优点是解释性较好，因为它把分类变成了“这段话是否说明某个标签成立”。缺点是模板敏感。模板写成“这段文本的主题是退款”和“这是一条退款请求”，结果可能不同。

第三条路线是生成式分类。生成式模型是可以根据提示词直接生成文本的模型。它可以接收输入文本和候选标签，然后直接输出标签：

$$
\hat{y}=decode(M(prompt(x,Y)))
$$

如果要让结果更可控，可以限制模型只能在候选标签中选择，或者计算每个标签字符串的对数概率。对数概率是把多个 token 的概率相加后得到的分数，常用于比较不同输出文本的可能性。

| 路线 | 核心思想 | 优点 | 局限 |
|---|---|---|---|
| 语义相似度 | 文本和标签做向量匹配 | 简单、直观、速度快 | 对标签写法敏感 |
| NLI | 把分类转成“是否蕴含” | 解释性较好，通用性强 | 模板敏感，推理成本较高 |
| 生成式 | 让模型直接输出标签 | 灵活，可处理复杂指令 | 容易越界，需要约束 |

三条路线的推导可以按一个顺序理解。第一，标签本身也是文本，所以模型可以读取标签含义。第二，文本和标签之间可以被比较：相似度路线比较向量，NLI 路线比较蕴含关系，生成式路线比较输出概率或直接生成结果。第三，分类只是从多个分数中选择最大值。

这也是零样本分类和传统监督分类的关键差异。传统监督分类学习的是“训练样本到标签”的映射；零样本分类利用的是“输入文本和标签文本之间的语义关系”。

## 代码实现

下面是一个最小可运行的 Python 例子。它不依赖外部模型，用手写分数模拟模型输出，展示零样本文本分类的工程流程：准备输入、候选标签、分数、预测结果、阈值回退。

```python
text = "I want a refund for a broken phone."
labels = ["退款", "法律", "咨询"]

scores = {
    "退款": 0.84,
    "法律": 0.27,
    "咨询": 0.19,
}

pred = max(scores, key=scores.get)
assert pred == "退款"

threshold = 0.50
final_label = pred if scores[pred] >= threshold else "转人工"
assert final_label == "退款"

# 多标签任务：每个标签独立判断是否超过阈值
selected = [label for label, score in scores.items() if score >= threshold]
assert selected == ["退款"]
```

这个玩具例子里的 `scores` 可以来自三种实现方式。

| 实现方式 | 输入形式 | 输出形式 | 适用任务 |
|---|---|---|---|
| 相似度 | 文本 + 标签 | 标签分数 | 简单分类、标签较少 |
| NLI | premise + hypothesis | entailment 概率 | 通用零样本分类 |
| 生成式 | prompt + 候选集 | 生成标签或标签概率 | 开放式分类、复杂指令 |

一个更接近真实工程的流程如下：

```python
def normalize_labels(labels):
    """统一标签写法，避免同义标签混用。"""
    mapping = {
        "退货退款": "退款",
        "退款申请": "退款",
        "法务": "法律",
    }
    return [mapping.get(label, label) for label in labels]


def classify_with_threshold(text, labels, score_fn, threshold=0.6):
    labels = normalize_labels(labels)
    scores = {label: score_fn(text, label) for label in labels}

    pred = max(scores, key=scores.get)
    confidence = scores[pred]

    if confidence < threshold:
        return {
            "label": "转人工",
            "confidence": confidence,
            "scores": scores,
        }

    return {
        "label": pred,
        "confidence": confidence,
        "scores": scores,
    }


def fake_score_fn(text, label):
    if "refund" in text.lower() and label == "退款":
        return 0.84
    if label == "法律":
        return 0.27
    return 0.19


result = classify_with_threshold(
    text="I want a refund for a broken phone.",
    labels=["退款申请", "法律", "咨询"],
    score_fn=fake_score_fn,
    threshold=0.6,
)

assert result["label"] == "退款"
assert result["confidence"] == 0.84
```

真实接入模型时，`score_fn` 可以替换成句向量相似度、NLI pipeline，或者生成式模型的候选标签打分函数。代码结构不需要大改，仍然保留五个环节：数据准备、标签标准化、模型推理、分数后处理、阈值与人工回退。

生产系统里不要只返回一个标签。至少要记录：

| 字段 | 作用 |
|---|---|
| 原始文本 | 方便排查错误 |
| 候选标签版本 | 方便追踪标签变更影响 |
| 每个标签分数 | 方便观察混淆 |
| 最终标签 | 业务使用的结果 |
| 置信度 | 决定是否回退人工 |
| 模型版本 | 方便回归测试 |

这样做的原因很简单：零样本分类前期最重要的不是一次性做对所有样本，而是持续知道它在哪里错、为什么错、错得是否可控。

## 工程权衡与常见坑

零样本文本分类的效果，很多时候不是由模型名称单独决定，而是由标签命名、模板设计、候选集规模、阈值策略共同决定。

第一个常见坑是标签写法敏感。同一个业务含义，写成 `法律`、`法务`、`法律咨询`，分数可能不同。模型会把它们当成不同文本，而不是自动知道它们在业务里等价。规避方式是固定标签词表，不允许随意新增同义标签。

第二个坑是模板敏感。NLI 路线中，`这段文本的主题是退款` 和 `用户正在申请退款` 都是合理模板，但它们可能产生不同结果。工程上要固定模板，并用一批样本做回归测试。回归测试是每次改动后重新跑固定样本，检查结果有没有异常变化。

第三个坑是单标签和多标签混淆。单标签任务要求只选一个结果，例如一张工单只能进入一个主队列；多标签任务允许多个结果同时成立，例如一条投诉既涉及退款，也涉及物流。单标签通常用归一化后的分数比较，多标签应对每个标签独立设阈值。

| 常见坑 | 现象 | 规避方式 |
|---|---|---|
| 标签写法敏感 | 同义标签得分差异大 | 统一命名，固定词表 |
| 模板敏感 | 模板一改结果变 | 固定模板，做回归测试 |
| 单标签/多标签混淆 | 结果不符合业务逻辑 | 互斥任务取最高分，多标签用独立阈值 |
| 生成式越界 | 输出不在候选集 | 约束解码或后处理 |
| 只看最高分 | 误分长尾样本 | 设置信心阈值，低分回退人工 |

真实工程例子：客服工单系统上线初期，可以先抽检每天每个标签的 top-k 样本。top-k 是每个标签分数最高的若干样本，用来快速判断模型是否理解标签。然后观察低置信样本，看它们是文本太短、标签模糊，还是确实需要新增标签。最后再调整标签命名和阈值。

| 工程动作 | 目的 |
|---|---|
| 标签先规范 | 减少同义词导致的分数波动 |
| 模板先固定 | 避免上线后结果不可追踪 |
| 分数先监控 | 发现标签混淆和分布漂移 |
| 人工先兜底 | 控制低置信样本风险 |

不要把零样本分类当成一次性方案。它更适合做冷启动入口：先上线、记录错误、收集标注、持续替换不稳定部分。对于长期系统，最终目标通常不是永远零样本，而是把零样本阶段积累的数据转成更稳定的训练集。

## 替代方案与适用边界

零样本文本分类不是唯一选择。它和少样本分类、监督微调、规则系统各有边界。

少样本分类是用少量标注样本帮助模型理解任务。它比零样本更稳定，但需要先标一批数据。监督微调是用较多标注样本训练任务模型，稳定性通常更高，但上线更慢。规则系统是用人工规则判断类别，例如关键词命中、正则表达式、业务字段组合。规则可控，但维护成本会随业务复杂度上升。

| 方案 | 需要标注数据 | 上线速度 | 稳定性 | 适用场景 |
|---|---|---|---|---|
| 零样本分类 | 否 | 快 | 中等 | 新任务快速试运行 |
| 少样本分类 | 少量 | 较快 | 较好 | 早期迭代 |
| 监督微调 | 需要 | 慢 | 高 | 长期生产 |
| 规则系统 | 否或少量 | 快 | 依赖人工维护 | 强约束业务 |

适合零样本的任务通常有三个特征。第一，标签定义清晰，例如 `退款 / 物流 / 账号 / 技术故障`。第二，输入文本和标签之间有明显自然语言语义关系。第三，业务允许用人工兜底处理低置信样本。

不适合直接依赖零样本的任务也很明确。比如“是否构成法律责任”“是否可以开具医疗诊断结论”“是否触发金融风控拒绝”，这些任务需要领域规则、证据链和严格校准。零样本最多作为辅助线索，不能作为最终判断。

一个合理的演进路径是：

| 阶段 | 做法 | 目标 |
|---|---|---|
| 冷启动 | 零样本分类 + 人工回退 | 快速上线 |
| 早期迭代 | 抽检 + 少量标注 | 找到主要错误模式 |
| 稳定期 | 少样本或监督微调 | 提高一致性 |
| 长期运行 | 监控 + 回归测试 + 标签治理 | 控制质量波动 |

如果只是临时上线一个新工单分类器，零样本通常最省时间。如果这个分类器要长期支撑核心业务，就应该在运行中积累标注数据，并逐步升级成更稳定的方案。
