## 核心结论

LDA（Latent Dirichlet Allocation，潜在狄利克雷分配）是一种无监督话题模型：它把每篇文档表示成多个话题的概率混合，把每个话题表示成词的概率分布。

一句话定义：

- 文档表示为多个话题的概率混合。
- 每个话题表示为词的概率分布。

关系可以写成：

```text
文档 d
  |
  |  θ_d = [话题1: 0.70, 话题2: 0.30, ...]
  v
话题分布
  |
  |  每个词先选择一个潜在话题 z_dn
  v
词分布 φ_k
  |
  |  从对应话题中生成具体词 w_dn
  v
词：篮球 / 球队 / 芯片 / 算法 / ...
```

LDA 的本质不是“给文档打一个固定类别”，而是反推文本背后的潜在话题结构。潜在话题指的是没有人工标注、但可以从词共现模式中推断出来的主题。例如很多文章里同时出现“篮球、球队、比赛、得分”，模型可能学出一个偏体育的话题；很多文章里同时出现“芯片、算法、模型、算力”，模型可能学出一个偏技术的话题。

玩具例子：一篇报道同时写了“篮球比赛”和“芯片发布”。普通分类模型可能被迫输出一个标签，比如“体育”或“科技”；LDA 更自然的输出是：

```text
体育 70% + 技术 30%
```

这更接近真实文本。很多文章并不是单一主题，而是多个主题的混合。

LDA 适合长文档、语料规模中等以上、并且希望获得“可解释话题”的场景。它不适合把每篇文档强行归成单一类别，也不适合直接解决复杂语义理解问题。

## 问题定义与边界

LDA 解决的问题是：给定一批文档，在没有人工标签的情况下，发现这些文档中反复出现的潜在话题，并估计每篇文档由哪些话题组成。

它不是监督分类。监督分类指的是已有标签，例如“体育、财经、科技”，模型学习从文本到标签的映射。LDA 没有标签输入，它只看词在文档中的共现关系，然后学习出若干个话题。输出结果是概率分布，不是硬标签。

LDA 的典型输入是分词后的文档集合。分词是把文本切成词或 token 的过程，例如“芯片发布会讨论算法优化”可以切成“芯片 / 发布会 / 讨论 / 算法 / 优化”。LDA 通常使用词袋表示。词袋表示指只统计词出现次数，不考虑词的顺序。

| 边界项 | 内容 | 说明 |
|---|---|---|
| 输入 | 文档集合 | 多篇文本，例如新闻、论文、商品评论 |
| 输入 | 词袋表示 | 每篇文档中每个词出现了多少次 |
| 输入 | 词表 | 全部保留下来的词，例如“篮球、球队、芯片、算法” |
| 输入 | 话题数 `K` | 预先指定要学习多少个话题 |
| 输出 | `θ_d` | 第 `d` 篇文档的话题分布 |
| 输出 | `φ_k` | 第 `k` 个话题的词分布 |
| 输出 | 每个词的潜在话题归属 | 训练中推断每个词位置更可能来自哪个话题 |
| 不解决 | 语义深度理解 | 不理解复杂上下文、隐喻、指代 |
| 不解决 | 上下文歧义消解 | “苹果”是公司还是水果，需要更多语义模型处理 |
| 不解决 | 监督分类 | 如果标签已知，分类模型通常更直接 |

真实工程例子：给 1 万篇新闻，LDA 可能发现“体育、财经、科技、娱乐”等话题，并给出每篇新闻的话题比例。比如一篇文章的 `θ_d` 可能是：

```text
科技 0.62，财经 0.25，娱乐 0.08，体育 0.05
```

这可以用于内容分析、推荐特征、搜索聚类和编辑后台的文章探索。

但如果只给一条 10 个词的短微博，LDA 结果通常不稳定。原因是词太少，共现信息不足，模型很难判断这些词是稳定主题，还是偶然噪声。

## 核心机制与推导

LDA 的机制要先从“生成过程”理解。生成过程是模型假设文本如何被写出来。真实训练时我们拿到的是已经写好的文本，模型要反过来推断隐藏的话题结构。

直观过程如下：

1. 对每篇文档，先抽一个话题分布 `θ_d`。
2. 对每个话题，先抽一个词分布 `φ_k`。
3. 写文档中的每个词时，先根据 `θ_d` 选择一个话题 `z_dn`。
4. 再根据这个话题的词分布 `φ_k` 选择具体词 `w_dn`。

其中 `θ_d` 是文档的话题分布。比如 `[0.6, 0.4]` 表示这篇文档 60% 偏话题 1，40% 偏话题 2。`φ_k` 是话题的词分布。比如某个话题中，“篮球”的概率高，“芯片”的概率低，那么这个话题可能偏体育。

公式写法如下：

$$
\theta_d \sim \text{Dir}(\alpha)
$$

$$
\phi_k \sim \text{Dir}(\beta)
$$

$$
z_{dn} \sim \text{Cat}(\theta_d)
$$

$$
w_{dn} \sim \text{Cat}(\phi_{z_{dn}})
$$

`Dir` 是 Dirichlet 分布，白话说就是“生成一组加起来等于 1 的概率”。`Cat` 是 categorical distribution，白话说就是“按概率从多个类别里抽一个”。

`α` 控制文档中的话题是否稀疏。稀疏指大部分概率集中在少数几个话题上。`α` 越小，一篇文档越倾向于只包含少数话题；`α` 越大，一篇文档越倾向于混合更多话题。

`β` 控制话题中的词分布是否集中。`β` 越小，一个话题越倾向于只由少数高频关键词代表；`β` 越大，一个话题会覆盖更多词，区分度可能下降。

最小数值例子：

词表是：

```text
[篮球, 球队, 芯片, 算法]
```

设两个话题：

| 词 | 话题1 `φ1` | 话题2 `φ2` |
|---|---:|---:|
| 篮球 | 0.40 | 0.10 |
| 球队 | 0.35 | 0.10 |
| 芯片 | 0.15 | 0.35 |
| 算法 | 0.10 | 0.45 |

话题 1 偏体育，话题 2 偏技术。某篇文档的话题分布为：

```text
θ_d = [0.6, 0.4]
```

对词“篮球”，它来自话题 1 的未归一化权重是：

$$
0.6 \times 0.40 = 0.24
$$

来自话题 2 的未归一化权重是：

$$
0.4 \times 0.10 = 0.04
$$

所以“篮球”更可能来自话题 1。

对词“算法”，它来自话题 1 的未归一化权重是：

$$
0.6 \times 0.10 = 0.06
$$

来自话题 2 的未归一化权重是：

$$
0.4 \times 0.45 = 0.18
$$

所以“算法”更可能来自话题 2。

这个例子说明，同一篇文档里，不同词可以来自不同话题。文档不是一个标签，词也不是都属于同一个主题。

训练时，已知的是文档里的词 `w_dn`，未知的是每篇文档的话题分布 `θ_d`、每个话题的词分布 `φ_k`、每个词位置的潜在话题 `z_dn`。文档似然来自所有词位置的联合概率。简化地看，一篇文档的概率来自“每个词由各个话题生成的概率加权求和”：

$$
P(w_d) = \prod_n \sum_k P(z_{dn}=k \mid \theta_d)P(w_{dn}\mid \phi_k)
$$

但真实后验分布很难直接精确求解，因为每个词的潜在话题归属互相影响，组合数量很大。所以工程实现通常使用 Gibbs sampling 或变分推断近似。

Gibbs sampling 是一种采样方法，白话说就是“反复更新每个词的主题归属，让整体结果逐渐接近目标后验”。变分推断是一种优化方法，白话说就是“用一个容易计算的分布去逼近真实后验”。

## 代码实现

下面是一个最小可运行流程：文本预处理、向量化、训练 LDA、打印主题词、打印文档主题分布。示例使用 `scikit-learn`，适合快速理解参数和输出。

```python
from sklearn.feature_extraction.text import CountVectorizer
from sklearn.decomposition import LatentDirichletAllocation
import numpy as np

docs = [
    "篮球 球队 比赛 得分 篮球 球员",
    "球队 比赛 冠军 得分 篮球",
    "芯片 算法 模型 算力 芯片",
    "算法 模型 数据 训练 算力",
    "篮球 比赛 球队 芯片 发布",
    "芯片 发布 算法 数据 模型",
]

stop_words = ["的", "了", "我们", "这个", "是", "在", "和"]

vectorizer = CountVectorizer(
    tokenizer=str.split,
    token_pattern=None,
    stop_words=stop_words,
    max_df=0.95,
    min_df=1,
)

X = vectorizer.fit_transform(docs)

lda = LatentDirichletAllocation(
    n_components=2,
    doc_topic_prior=0.1,
    topic_word_prior=0.01,
    random_state=42,
    learning_method="batch",
)

theta = lda.fit_transform(X)
feature_names = np.array(vectorizer.get_feature_names_out())

assert X.shape[0] == len(docs)
assert theta.shape == (len(docs), 2)
assert np.allclose(theta.sum(axis=1), 1.0)

for topic_idx, topic_weights in enumerate(lda.components_):
    top_indices = topic_weights.argsort()[::-1][:4]
    top_words = feature_names[top_indices]
    print(f"Topic {topic_idx}:", " ".join(top_words))

print("Document-topic distribution:")
for i, row in enumerate(theta):
    print(i, np.round(row, 3))
```

这段代码中，`CountVectorizer` 把分词后的文本转成词袋矩阵 `X`。矩阵的每一行是一篇文档，每一列是一个词，值是词频。`LatentDirichletAllocation` 训练话题模型。`n_components=2` 表示学习两个话题。`doc_topic_prior` 对应 `α`，`topic_word_prior` 对应 `β`。`fit_transform(X)` 返回 `θ_d`，也就是每篇文档的话题分布。`components_` 存的是每个话题对词的权重，可以取权重最高的词作为话题解释。

| 步骤 | 输入 | 输出 | 作用 |
|---|---|---|---|
| 分词与停用词处理 | 原始文本 | token 列表或空格分隔文本 | 减少无意义词干扰 |
| 向量化 | 分词后文档 | 词袋矩阵 `X` | 把文本变成可训练的数值矩阵 |
| 训练 LDA | `X`、话题数 `K`、`α`、`β` | 训练后的模型 | 学习话题和词之间的关系 |
| 文档变换 | `X` | `θ_d` | 得到每篇文档的话题比例 |
| 提取主题词 | `components_` | 每个话题的 top 词 | 用关键词解释话题含义 |

真实工程中，如果要评估 topic coherence，一般会补充使用 Gensim。topic coherence 是话题一致性指标，白话说就是“一个话题里的高权重词放在一起是否像一个人能读懂的主题”。例如“篮球、球队、比赛、得分”一致性较强；“篮球、芯片、的、我们”一致性较弱。

## 工程权衡与常见坑

`K` 不是自动最优。`K` 太小会把不同主题混在一起，例如把“芯片发布”和“互联网财报”都压成一个模糊的科技话题。`K` 太大会把一个主题切碎，例如把体育拆成“篮球比赛”“篮球转会”“篮球得分榜”等多个难以区分的小话题。

困惑度不等于可解释性。困惑度是模型对数据拟合程度的一种指标，白话说就是“模型预测词出现的困难程度”。困惑度更低不一定代表话题更适合人读。工程中通常同时看困惑度、topic coherence、人工抽样检查。

| 问题 | 后果 | 处理方式 |
|---|---|---|
| 低频词太多 | 主题稀疏且噪声大 | 用 `min_df` 过滤低频词 |
| 停用词未清理 | 主题被功能词污染 | 维护停用词表 |
| 短文本过短 | 主题分布不稳定 | 聚合文档或换短文本模型 |
| 随机初始化不稳定 | 多次训练结果波动 | 多随机种子比较 |
| 只看困惑度 | 误判模型好坏 | 同时看 coherence 和人工解释 |
| `K` 过小 | 多个主题被合并 | 增大 `K` 并检查主题词 |
| `K` 过大 | 一个主题被切碎 | 减小 `K` 或合并相近主题 |

停用词处理尤其重要。如果把“的、了、我们、这个、是、在、和”等高频功能词保留，训练出来的话题可能变成：

```text
话题1：的 是 在 和 这个
话题2：我们 了 的 是 和
```

这种结果没有分析价值。LDA 依赖词共现统计，输入质量会直接决定话题质量。

短文本也是常见问题。LDA 原本更适合文档中有足够词共现的场景。短微博、搜索 query、短评论只有几个词，单篇文档里几乎没有稳定共现结构。处理方式通常是按用户、时间、商品、会话聚合短文本，或者改用 embedding、BERTopic 等方法。

真实工程例子：内容平台做推荐时，可以先训练 LDA，把每篇文章的 `θ_d` 作为文章特征。用户读过多篇文章后，可以把这些文章的话题分布加权平均，得到用户兴趣向量。例如用户最近读了很多“科技 0.7、财经 0.2”的文章，系统可以在召回阶段优先找相似话题文章。但这只是推荐特征之一，不能替代点击率模型、语义向量、实时行为特征。

## 替代方案与适用边界

LDA 适合“需要可解释主题”的离线分析。它的优势是输出清晰：每个话题有关键词，每篇文档有话题比例。编辑、运营、分析师可以直接检查模型是否合理。

但如果目标是最强语义表示，LDA 通常不是首选。它基于词袋表示，不直接建模词序和上下文。比如“苹果发布芯片”和“我吃了苹果”，如果只看词共现，LDA 很难像上下文语言模型那样精确区分语义。

| 方法 | 主要特点 | 适合场景 | 不适合场景 |
|---|---|---|---|
| LDA | 可解释性强，输出话题词和文档话题分布 | 主题发现、内容分析、推荐特征 | 极短文本、强语义匹配 |
| NMF | 非负矩阵分解，实现简单 | 无监督主题建模替代 | 需要生成式解释的场景 |
| BERTopic | 结合 embedding 和聚类 | 短文本、语义聚类、主题探索 | 资源受限或需要纯概率模型 |
| 词向量 / Sentence Embedding | 向量表达语义相似度 | 语义检索、推荐、召回 | 需要直接解释每个主题的场景 |
| 监督分类模型 | 标签已知时效果直接 | 文本分类、审核、路由 | 没有标签、想发现未知主题 |

新闻聚类可以用 LDA，因为新闻通常较长，主题词也容易解释。例如“球队、比赛、冠军”对应体育，“利率、股市、财报”对应财经。但如果要做语义检索，例如用户搜索“适合初学者的机器学习教程”，系统需要理解“入门、教程、基础、初学者”之间的语义关系，Sentence Embedding 或 BERT 类模型通常更合适。

适合 LDA 的情况：

| 场景 | 是否适合 | 原因 |
|---|---|---|
| 长新闻、论文、报告 | 适合 | 文档较长，共现信息足 |
| 运营后台主题分析 | 适合 | 需要可解释话题 |
| 推荐系统离线特征 | 适合 | `θ_d` 可作为内容特征 |
| 内容探索与聚类 | 适合 | 可以快速看语料结构 |
| 极短评论或 query | 不适合 | 单条文本信息太少 |
| 高精度文本分类 | 不适合 | 有标签时监督模型更直接 |
| 语义检索 | 不适合单独使用 | embedding 更擅长语义相似度 |

工程上可以把 LDA 当作“可解释的主题层”，而不是万能文本理解模型。它可以和 embedding、分类模型、推荐排序模型一起使用：LDA 提供主题比例，embedding 提供语义相似度，监督模型提供明确任务目标。
