## 核心结论

Parallel Decoding，中文常译为“并行解码”，指模型不再按“第 1 个 token 生成后才能生成第 2 个 token”的顺序工作，而是一次性并行预测整段输出，再通过若干轮修正把不可靠的位置改好。这里的 token 可以先理解为“模型处理的最小离散单位”，在文本里通常是词片段，在图像里可以是离散图像编码。

它的核心收益很直接：把原本长度为 $T$ 的串行生成，改成 1 次或少数几次并行推理，因此在机器翻译、语音识别后处理、图像 token 生成等任务上，推理速度常能达到自回归方法的 5 到 10 倍，某些场景甚至更高。

但速度不是白送的。传统自回归解码满足

$$
p(y|x)=\prod_{t=1}^{T} p(y_t|x,y_{<t})
$$

意思是“第 $t$ 个位置的输出依赖前面所有已生成内容”。并行解码通常采用更强的条件独立假设：

$$
p_{\text{NAR}}(y|x)=\prod_{t=1}^{T} p(y_t|x)
$$

这相当于把位置之间的依赖拆散。拆散后速度更快，但会出现“多模态问题”。多模态可以白话理解为：同一个输入存在多个都合理的输出版本，而模型如果把每个位置独立决策，容易拼出一个局部都像对、整体却不协调的结果。

因此，实际可用的并行解码一般不是“一次预测完就结束”，而是“一次并行预测 + 若干轮置信度驱动的迭代精炼”。新手可以把它想成一张填空卷：先把所有空一次填完，再把最不确定的几个空擦掉重填，重复几轮，最后整句变得一致。

## 问题定义与边界

并行解码要解决的问题，不是“让模型训练更快”，而是“让生成阶段更少依赖串行步骤”。生成阶段也叫 inference 或 decoding，可以白话理解为“模型真正吐出答案的过程”。

它主要适用于序列生成任务，也就是输出可以看成一串位置排列好的 token。典型任务包括：

| 任务 | 输出形式 | 为什么适合并行解码 | 成功条件 |
| --- | --- | --- | --- |
| 机器翻译 | 目标语言 token 序列 | 整句目标已知是有限长度序列 | 长度预测、知识蒸馏、迭代精炼 |
| 图像 token 生成 | 离散图像 patch/token 序列 | 大量位置可同时预测 | 置信度调度、多轮重掩码 |
| 语音相关后处理 | 音素或文本序列 | 追求低延迟 | 对齐信息较稳定 |
| 长文本开放生成 | 很长的文本序列 | 理论可做，实践难 | 长程依赖建模很难，常不划算 |

它不适合所有生成任务，边界主要有三条。

第一，输出最好有可估计长度。长度预测就是先估计输出大概有多少个 token，记作 $p(\ell|x)$。如果长度都很难猜，模型连“该填多少格”都不稳，后续并行预测就容易整体跑偏。

第二，模型最好能给出可靠置信度。置信度可以先理解为“模型觉得这个位置自己答得有多把握”。没有它，就不知道哪些位置该固定，哪些位置该重做。

第三，任务里的长距离依赖不能过强。如果一个位置强依赖几十甚至几百个前文 token，那么把条件拆散后，后续再靠几轮修补未必补得回来。

一个玩具例子很适合说明边界。输入英文 `I like apples`，假设目标中文长度预测为 4。并行模型第一轮一次输出 4 个位置：`我 / 喜欢 / 苹果 / 们`，同时给每个位置一个置信度。如果发现第 4 个位置置信度很低，就第二轮只重算这个位置，得到 `我 / 喜欢 / 苹果`。这个过程只做了 2 轮并行推理，而不是逐词做 3 到 4 次自回归推理。

所以，判断并行解码“值不值得上”，不是看概念新不新，而是看你的任务是否满足：长度相对可控、局部可并行、低置信位置能被少数几轮修复。

## 核心机制与推导

并行解码的基础版本通常写成：

$$
p(y|x)=p(\ell|x)\prod_{t=1}^{\ell} p(y_t|x)
$$

这里多出来的 $p(\ell|x)$ 是长度预测项。因为非自回归模型通常不能像自回归那样自然地“生成到停止符为止”，所以往往先预测长度 $\ell$，再一次性生成 $\ell$ 个位置。

问题在于，这个公式默认各位置条件独立。机器翻译里，同一句英文 `bank` 可能要翻成“银行”或“河岸”；到底选哪个，依赖上下文整体语义。如果各位置独立决策，就容易在同一句里混出彼此冲突的词。

为了解决这个问题，工程上常用迭代精炼。它的思想是：虽然第一轮并行预测时不看其他目标位置，但第二轮开始，可以把上一轮已经较可信的 token 当作条件，继续改低可信位置。形式上可写成：

$$
y_t^{(k)} \sim p_\theta\bigl(y_t \mid x, y_{\neg M^{(k)}}^{(k-1)}\bigr), \quad t \in M^{(k)}
$$

其中：

- $k$ 表示第几轮迭代。
- $M^{(k)}$ 是第 $k$ 轮需要重算的位置集合，也就是 mask 集合。
- $y_{\neg M^{(k)}}^{(k-1)}$ 表示上一轮已经保留、不再重算的位置。

白话解释就是：每一轮只重做那些没把握的位置，已经比较稳的位置先锁住，给下一轮提供上下文。

常见做法是 Mask-Predict 或类似的置信度调度。流程通常分 4 步：

1. 预测长度 $\ell$。
2. 初始时把所有位置都视为待预测。
3. 模型并行输出整段 token 和每个位置置信度。
4. 根据置信度排序，把最不可靠的一批位置重新 mask，进入下一轮。

一个最小玩具例子如下。目标长度是 4，第一轮输出：

| 位置 | token | 置信度 |
| --- | --- | --- |
| 1 | 我 | 0.98 |
| 2 | 去 | 0.91 |
| 3 | 银行 | 0.52 |
| 4 | 存钱 | 0.47 |

如果调度规则规定“下一轮重做最低的 50% 位置”，那第 3、4 个位置会被重新 mask。第二轮模型会在“我 / 去 / [MASK] / [MASK]”的条件下并行重算，可能得到更一致的结果：`我 / 去 / 银行 / 存款` 或 `我 / 去 / 银行 / 办业务`。这时句子的整体协调性比第一轮更强。

为什么这能工作？因为第一轮虽然粗糙，但已经给出了一个全局草稿。后续轮次不再是纯独立预测，而是在“部分已知目标”的条件下修补剩余位置，相当于逐步恢复目标侧依赖。

调度函数本身也很关键。若总轮数为 $K$，常见的 decaying mask schedule 会让每一轮重做的位置逐渐减少。例如：

$$
m_k = \left\lfloor \ell \cdot \left(1-\frac{k}{K}\right) \right\rfloor
$$

这里 $m_k$ 表示第 $k$ 轮仍需重算的位置数。它的直觉很简单：前期大范围修改，后期小范围收尾。前期像“先把大错改掉”，后期像“只修边角”。

真实工程例子是 MaskGIT。它面向离散图像 token 生成，不按像素或 patch 一个个串行生成，而是先预测整张图的大量位置，再根据置信度重掩码低可信区域，通常 4 到 10 轮就能完成。这说明并行解码并不局限于文本，本质上它是一类“少轮次、置信度驱动、局部重采样”的序列生成机制。

## 代码实现

下面给一个可运行的 Python 玩具实现。它不依赖深度学习框架，只模拟并行解码里的 3 个核心部件：长度固定、按置信度选择低可信位置、逐轮减少重算数量。

```python
from math import floor

def select_low_confidence(confidences, keep_mask, num_to_remask):
    candidates = [(c, i) for i, (c, keep) in enumerate(zip(confidences, keep_mask)) if keep]
    candidates.sort(key=lambda x: x[0])  # 置信度从低到高
    return [idx for _, idx in candidates[:num_to_remask]]

def parallel_decode_round(tokens, confidences, step_updates, max_round=4):
    """
    tokens: 当前序列
    confidences: 每个位置的置信度
    step_updates: 每轮对低置信位置的候选更新，形如:
        [
          {2: ("银行", 0.78), 3: ("存款", 0.83)},
          {3: ("办理业务", 0.92)}
        ]
    """
    seq = tokens[:]
    conf = confidences[:]
    keep_mask = [True] * len(seq)

    for k in range(max_round):
        num_to_remask = floor(len(seq) * (1 - (k + 1) / max_round))
        if num_to_remask <= 0:
            break

        remask_positions = select_low_confidence(conf, keep_mask, num_to_remask)
        if not remask_positions:
            break

        updates = step_updates[k] if k < len(step_updates) else {}
        for pos in remask_positions:
            if pos in updates:
                seq[pos], conf[pos] = updates[pos]

        # 置信度足够高的位置固定，避免来回抖动
        for i, c in enumerate(conf):
            if c >= 0.85:
                keep_mask[i] = False

    return seq, conf

tokens = ["我", "去", "银航", "存前"]
confidences = [0.98, 0.95, 0.42, 0.38]
step_updates = [
    {2: ("银行", 0.78), 3: ("存钱", 0.81)},
    {3: ("存款", 0.90)},
]

final_tokens, final_conf = parallel_decode_round(tokens, confidences, step_updates, max_round=4)

assert final_tokens[:3] == ["我", "去", "银行"]
assert final_tokens[3] in ("存钱", "存款")
assert final_conf[3] >= 0.81

print(final_tokens, final_conf)
```

这段代码表达了 3 个实现重点。

第一，低置信位置优先重算。真实模型里，置信度通常来自 softmax 最大概率、margin，或者更复杂的 calibration 方法。calibration 可以白话理解为“把模型分数修正成更接近真实可信度的分数”。

第二，高置信位置尽量锁定。否则同一位置每轮都可能被改，容易出现反复抖动，轮次增加但质量不稳定。

第三，重算数量逐轮下降。第一轮后大改，最后几轮只收尾，这样才能同时兼顾质量和速度。

如果把它写成更贴近工程的伪代码，大致是：

```python
length = predict_length(x)
tokens = ["[MASK]"] * length

for k in range(max_round):
    tokens, conf = model_predict(x, tokens)   # 并行预测所有待更新位置
    remask_num = schedule(length, k, max_round)
    remask_pos = lowest_confidence_positions(conf, remask_num)
    tokens = apply_mask(tokens, remask_pos)
    if stop_by_threshold(conf) or remask_num == 0:
        break
```

真实模型会复杂得多，还会涉及 teacher-student 蒸馏、长度候选束搜索、对齐损失等，但解码器层面的最小闭环就是这几步。

一个真实工程例子是机器翻译服务。如果用户输入平均长度 20 到 40 token 的句子，自回归解码每个位置都要依次跑一次，GPU 或 CPU 上的串行依赖会明显拉高尾延迟。改成并行解码后，可能只需 4 到 8 轮，每轮同时处理整句，大幅减少请求时延。这类收益在高并发、对延迟敏感的在线翻译场景尤其明显。

## 工程权衡与常见坑

并行解码不是“速度更快的免费午餐”。它的主要权衡可以概括为：轮次、质量、系统复杂度三者互相牵制。

| 常见坑 | 现象 | 根因 | 常见解决方案 |
| --- | --- | --- | --- |
| 多模态崩溃 | 句子局部对、整体不协调 | 条件独立假设过强 | 知识蒸馏、迭代精炼、目标对齐建模 |
| 长度预测错误 | 漏词、重复、截断 | 先验长度估计不准 | 长度候选集、length beam、插入删除机制 |
| 轮次太少 | 速度快但错误多 | 修复机会不足 | 增加 1 到 3 轮，或动态停止 |
| 轮次太多 | 质量略升但加速消失 | 过度迭代 | decaying mask schedule、置信度阈值停止 |
| 置信度不可靠 | 重算位置选错 | 概率未校准 | 温度缩放、校准训练、排名式选择 |
| 位置反复抖动 | 某些 token 来回变 | 没有固定高置信位置 | 冻结策略、最小修改约束 |

最核心的坑是多模态问题。比如英文一句话有两种都正确的译法，自回归模型会在前几个 token 选定风格后一路延续；并行模型若各位置独立，可能前半句选择译法 A，后半句选择译法 B，最后拼成不自然的混合句。文献里这通常会体现为 BLEU、chrF 等指标下降。BLEU 可以先理解为“机器翻译里衡量生成结果和参考答案重合度的经典指标”。

知识蒸馏是最常见补救手段。它的白话解释是：先让一个更强的自回归教师模型产出“更规整、更单一”的目标，再用这些目标训练并行模型。这样做的本质，是主动减少训练数据里的输出多样性，让并行模型更容易学到一个稳定模式。

另一个坑是置信度阈值设计。阈值过高，模型会把太多位置当成“不可靠”，导致轮次增加；阈值过低，又会过早冻结错误 token。实际工程里常不是用绝对阈值，而是每轮固定重算比例，比如重算最低 50%、25%、12.5% 的位置，这比绝对阈值更稳。

在图像生成里，这个问题同样明显。MaskGIT 的经验表明，4 到 10 轮通常是可接受区间。轮次太少，图像整体结构容易不完整；轮次太多，虽然局部更细，但延迟优势会被吃掉。文本任务也是同理，只是评价标准从“图像观感”变成“语义一致性和语言自然度”。

工程上还要注意一个经常被忽略的事实：并行解码的速度收益，取决于硬件是否真的能把“同一轮内的大量位置”并起来。如果序列很短，或者系统瓶颈不在解码而在网络、检索、后处理，那么算法级加速不会完整转化成端到端延迟收益。

## 替代方案与适用边界

并行解码不是唯一选择。更准确的说法是，它处在“极致质量的自回归”和“极致速度的一次性非自回归”之间，很多实际系统会选折中方案。

| 方案 | 速度 | 质量 | 实现复杂度 | 适用场景 |
| --- | --- | --- | --- | --- |
| 自回归解码 | 慢 | 通常最高 | 中 | 长文本、复杂推理、强依赖生成 |
| 并行解码/NAR | 快 | 通常低于 AR | 高 | 翻译、图像 token、低延迟生成 |
| 半自回归 | 中 | 中到较高 | 高 | 分块生成、局部并行 |
| CTC 类方法 | 很快 | 依任务而定 | 中 | 对齐较强的语音/识别任务 |
| 插入式/编辑式解码 | 中 | 中到较高 | 高 | 允许多轮编辑的结构化生成 |

半自回归可以理解为“块内并行、块间串行”。例如一次生成 4 个 token，再根据这 4 个 token 生成下一块。这比完全自回归快，也比完全并行更容易保持质量。

CTC 是另一条路线。它不强调迭代精炼，而是依赖对齐结构直接压缩或展开序列，更适合输入输出顺序比较稳定的场景，比如语音识别中的某些阶段。它不是通用文本开放生成的替代品，但在对齐问题明显时很有效。

适用边界可以简单总结为两句话。

如果你只需要快速生成短到中等长度序列，而且任务容忍少量质量损失，parallel decoding 往往值得尝试。比如在线翻译、图像 token 补全、低延迟批量生成。

如果你要生成长篇小说、复杂代码、多步推理解释，输出强依赖前文长期状态，那么自回归通常仍然更稳。因为这类任务要求模型持续维护全局一致性，而这正是条件独立假设最容易失效的地方。

再给一个新手能直接感知的例子：如果任务只是快速生成 16 个 token 的结构化短答案，并行解码通常会赢；但如果任务是连续写 2000 字故事并保持人物设定、叙事节奏和因果关系，自回归更可靠。
