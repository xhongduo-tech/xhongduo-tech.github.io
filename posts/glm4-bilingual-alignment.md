## 核心结论

GLM-4 的中英双语对齐训练，不是简单把中文数据和英文数据混在一起训练，而是把“双语能力不互相伤害”当成一个单独优化目标。更准确地说，它要同时解决三件事：预训练阶段保证两种语言都学到足够知识，SFT 阶段把两种语言都拉到一致的指令风格，RLHF 阶段在继续优化回答偏好的同时避免某一种语言能力被覆盖掉。

从公开材料看，GLM-4 这一代模型建立在约 10 万亿 token 的预训练之上，重点投入中文和英文，并在 SFT 与 RLHF 后训练后，把“中文可用、英文也强、长上下文还能工作”三件事同时做到较高水平。这一点很关键，因为很多模型在单语 benchmark 上分数不错，但一进入中英混合问答、长文档总结、工具调用场景，就会暴露语言偏好不稳定的问题。

可以把这条训练链路压缩成四步流水线：

| 阶段 | 目标 | 关键动作 | 如果做错会怎样 |
|---|---|---|---|
| 预训练 | 学知识与语言模式 | 中英多源数据清洗、去重、重采样、高质量数据加权 | 中文或英文基础能力偏弱 |
| SFT | 学会按指令说话 | 用真实人工指令对齐中英文输出风格 | 两种语言回答风格不一致 |
| RLHF | 学会更符合偏好 | 奖励模型、偏好优化、长度平衡 | 模型向单一语言偏移 |
| 正则化约束 | 防止遗忘 | KL/REG 约束参考策略 | 某一语言能力在后训练中退化 |

玩具例子可以这样理解：如果一个小模型预训练时中文 token 明显不足，SFT 又主要看英文问答，RLHF 再只按英文偏好打分，那么最后它即使还能“看懂”中文，也很容易在中文场景中输出英文式结构、术语翻译不稳，甚至直接退化成英文优先模型。GLM-4 的关键不是某一个单点技巧，而是把这种偏移在三个阶段都提前处理掉。

真实工程例子是长文档智能体。GLM-4 All Tools 面向文档检索、代码解释、Web 工具调用这类任务时，用户很可能用中文提问，但检索文档是英文，最后又要求中文总结。如果训练时没有做双语对齐，模型会在“输入语言、检索语言、输出语言”三者切换时出现漂移；而做了跨阶段的双语约束后，模型更容易保持语义一致和输出稳定。

## 问题定义与边界

这里的问题不是“如何让模型同时认识中文和英文”，而是“如何让模型在中英两种语言中保持一致的知识表达、指令遵循和偏好分布”。偏好分布可以直白地理解为：面对同一类问题，模型更倾向于用什么风格、什么长度、什么解释层次来回答。

边界也要讲清楚。GLM-4 的双语对齐训练讨论的是中英双主语种场景，不等于全面多语言统一对齐。也就是说，它的核心优化对象是中文和英文之间的能力共存，而不是几十种语言均匀覆盖。这个边界决定了训练资源的分配方式：词表、语料比例、SFT 标注和 RLHF 规则都可以更集中地为中英双语服务。

从训练阶段看，问题可以拆成三个子问题：

1. 预训练怎么分配中英语料，才能让基础语言能力不失衡。
2. SFT 怎么组织中英指令数据，才能让回答风格和任务遵循方式对齐。
3. RLHF 怎么继续提升“有帮助、真实、安全”的回答质量，同时不把已有的双语能力冲掉。

这也是为什么单看预训练语料比例是不够的。假设一个模型在预训练中把中英比例做到了 1:1，但 SFT 数据里 80% 都是英文助手问答，RLHF 偏好标注又主要根据英文评审习惯给分，那么后训练后的模型仍然可能出现英文更稳定、中文更松散的现象。原因不是它“不懂中文”，而是它后期被优化到了另一套偏好空间。

可以用一个最小化的形式表达这个目标。设预训练得到的语言知识分布为 $P_{\text{pre}}$，SFT 后策略为 $\pi_{\text{sft}}$，RLHF 后策略为 $\pi$。如果只优化奖励而不加约束，模型会朝奖励集最密集的语言区域移动。为了保持双语能力，需要在优化中保留对参考策略的接近性：

$$
\mathcal{L} = \mathcal{L}_{\text{RL}} + \beta \, D_{\text{KL}}(\pi \,\|\, \pi_{\text{sft}}) + \gamma \, \mathcal{L}_{\text{reg}}
$$

其中，$D_{\text{KL}}$ 是 KL 散度，白话解释是“新策略不要离旧策略太远”；$\mathcal{L}_{\text{reg}}$ 是额外正则项，白话解释是“对语言、长度、风格这些容易漂移的维度再加一道保险”。

一个新手容易混淆的点是：双语对齐不等于中英文数据严格 1:1。真实训练里更重要的是“有效信息密度”平衡。百科、书籍、论文、代码、问答的价值不同，低质量网页即使量大，也不能简单按 token 数量与高质量文本等价看待。因此问题定义必须包含“质量加权后的语言平衡”，而不只是原始 token 平衡。

## 核心机制与推导

GLM-4 的双语对齐机制，可以按“预训练重采样 -> 双语 SFT -> 带正则的 RLHF”理解。

### 1. 预训练：先让两种语言都学到东西

预训练阶段的任务是让模型形成足够好的基础表征。表征可以直白地理解为：模型内部如何编码词、句子、事实和推理模式。这里至少有三层工作。

第一层是清洗与去重。网页、百科、书籍、论文、代码往往同时包含复制片段、模板噪声、低质量转载和跨站重复。如果不做 exact dedup 与 fuzzy dedup，模型会在某些语言或某些站点模式上过拟合。对双语模型来说，这种伤害更大，因为重复往往并不均匀，容易把某一种语言的低质量样本放大。

第二层是高质量数据重采样。重采样的白话解释是：不是让所有样本平等进入训练，而是让更重要的样本更频繁出现。GLM-4 相关材料强调高质量百科、书籍等数据的动态加权，这说明它不是一次性定死采样概率，而是随着训练阶段变化逐步调整。这么做的原因是：训练早期模型需要广覆盖，训练中后期更需要高密度知识文本来稳定语言表达和长程依赖。

第三层是中英联合建模。联合词表或兼容词表的意义，不只是节省参数，更重要的是让跨语言共享的数字、代码、专名、数学符号、技术术语能进入相近表示空间。表示空间可以直白地理解为：语义上接近的内容，在模型内部向量上也更接近。这对中英技术问答尤其重要，因为大量术语本来就跨语言共用。

一个玩具例子如下。假设只有四类语料：中文百科、英文百科、中文论坛、英文论坛。若它们原始 token 占比是 20%、20%、30%、30%，直接训练会让论坛体比例过高。若把高质量百科权重提升到 2 倍，论坛维持 1 倍，则有效采样权重大致变成：

| 语料 | 原始占比 | 质量权重 | 有效占比趋势 |
|---|---:|---:|---:|
| 中文百科 | 20% | 2.0 | 提升 |
| 英文百科 | 20% | 2.0 | 提升 |
| 中文论坛 | 30% | 1.0 | 下降 |
| 英文论坛 | 30% | 1.0 | 下降 |

结果不是中英比例改变，而是“高质量中英内容”共同上升，这比机械调语言比例更接近真正的双语对齐目标。

### 2. SFT：把两种语言拉到同一套助手行为

SFT 是监督微调，白话解释是拿“标准问答样本”直接教模型应该怎么回答。这里最重要的不是数量，而是指令分布是否覆盖真实场景。

如果中文 SFT 数据主要是知识问答，英文 SFT 数据主要是代码解释与写作改写，那么模型会学到“中文适合短答，英文适合复杂任务”的隐含偏好。真正的双语对齐要求任务类型也尽量交叉覆盖，例如中英文都包含总结、推理、代码、抽取、拒答、多轮跟进等样本。

可以把 SFT 看成学习一个条件分布 $\pi_{\text{sft}}(y|x)$，也就是“看到输入 $x$ 时，输出 $y$ 的倾向”。若中文输入和英文输入在任务结构上不对称，模型学到的并不是“同一能力的双语版本”，而是“两套不同助手人格”。

### 3. RLHF：继续优化偏好，但不能冲掉双语能力

RLHF 的目标是让输出更符合人类偏好。这里的偏好不只包括答案对错，也包括礼貌性、信息密度、拒答边界、格式稳定性等。问题在于，一旦奖励模型或偏好数据偏向某种语言，策略优化会顺着这个方向放大偏差。

因此，GLM-4 相关实践里提到的 KL、REG、长度桶化都很关键。

长度桶化可以直白地理解为：把短回答、中回答、长回答分开比较，避免模型靠“总是更长”拿更高奖励。因为中文和英文的 token 结构不同，不做长度控制时，奖励会把语言差异和长度差异混在一起。

REG 正则项可以理解为一类补充约束，用来防止策略在奖励优化时走得过远。它不一定只约束语言标签本身，也可能通过参考奖励、输出分布、对比损失等方式间接保持稳定。

综合起来，RLHF 的核心不是把奖励做大，而是让奖励优化在一个安全半径内进行。这个思路可以写成更具体的损失：

$$
\mathcal{L}_{\text{total}}
=
-\mathbb{E}_{(x,y)\sim \pi}\big[R(x,y)\big]
+
\beta D_{\text{KL}}(\pi \,\|\, \pi_{\text{sft}})
+
\gamma \mathcal{L}_{\text{reg}}
+
\eta \mathcal{L}_{\text{len-bucket}}
$$

其中：
- $R(x,y)$：奖励模型打分，表示回答有多符合偏好。
- $\mathcal{L}_{\text{len-bucket}}$：长度桶约束，避免不同语言因长度差异被不公平优化。
- $\beta,\gamma,\eta$：权重系数，决定“追求更高奖励”和“保持原有能力”之间的平衡。

真实工程例子是中文提问、英文材料检索、中文输出总结的企业知识库问答。若 RLHF 只对中文直答任务打分，模型可能在中文闲聊上更自然，但在跨语种检索总结时更容易遗漏英文原文中的关键信息；若同时保留英文技术问答和跨语言总结样本，并加参考策略约束，就更可能维持双语检索与转换能力。

## 代码实现

下面给出一个可运行的玩具实现，用最小代码展示“双语采样 + SFT 约束 + RLHF 正则”的核心结构。它不是完整训练框架，但足够说明工程上该把约束放在哪里。

```python
from math import log

def normalize(weights):
    s = sum(weights.values())
    return {k: v / s for k, v in weights.items()}

def resample_sources(raw_mix, quality_weight):
    """
    raw_mix: 每类数据的原始token量
    quality_weight: 每类数据的质量权重
    """
    weighted = {}
    for name, tokens in raw_mix.items():
        weighted[name] = tokens * quality_weight.get(name, 1.0)
    return normalize(weighted)

def kl_divergence(p, q):
    eps = 1e-12
    total = 0.0
    for k, pv in p.items():
        qv = q.get(k, eps)
        total += pv * log((pv + eps) / (qv + eps))
    return total

def rlhf_objective(reward, policy, ref_policy, beta=0.2, gamma=0.1, reg_penalty=0.0):
    """
    一个最小化目标：越小越好
    reward: 奖励越大越好，因此损失里取负号
    beta * KL: 不要偏离SFT参考策略太远
    gamma * reg_penalty: 额外语言平衡约束
    """
    return -reward + beta * kl_divergence(policy, ref_policy) + gamma * reg_penalty

raw_mix = {
    "zh_wiki": 20,
    "en_wiki": 20,
    "zh_forum": 30,
    "en_forum": 30,
}

quality_weight = {
    "zh_wiki": 2.0,
    "en_wiki": 2.0,
    "zh_forum": 1.0,
    "en_forum": 1.0,
}

resampled = resample_sources(raw_mix, quality_weight)

# 高质量百科占比应上升
assert resampled["zh_wiki"] > 0.2
assert resampled["en_wiki"] > 0.2

# SFT参考策略：中英回答分布较平衡
sft_policy = {"zh": 0.52, "en": 0.48}

# 一个偏向英文的RLHF后策略
biased_policy = {"zh": 0.30, "en": 0.70}

# 一个更接近平衡的策略
balanced_policy = {"zh": 0.50, "en": 0.50}

loss_biased = rlhf_objective(
    reward=1.0,
    policy=biased_policy,
    ref_policy=sft_policy,
    beta=0.6,
    gamma=1.0,
    reg_penalty=0.25,
)

loss_balanced = rlhf_objective(
    reward=0.95,
    policy=balanced_policy,
    ref_policy=sft_policy,
    beta=0.6,
    gamma=1.0,
    reg_penalty=0.02,
)

# 即使奖励略低，更平衡的策略也可能是更优解
assert loss_balanced < loss_biased
print("toy bilingual alignment check passed")
```

这段代码表达了三件事。

第一，`resample_sources()` 说明高质量数据加权应该发生在预训练数据流入口，而不是等模型已经学歪了再靠后训练补救。  
第二，`kl_divergence()` 说明参考策略必须显式存在，通常它来自 SFT 模型。  
第三，`reg_penalty` 表示除了 KL 之外，还可以加入额外双语稳定项，例如语言分布偏移、长度桶偏移、风格一致性偏移。

如果把它写成更接近训练框架的伪代码，可以是：

```python
# stage 1: pretrain
for batch in bilingual_pretrain_stream:
    loss = next_token_loss(model, batch)
    update(model, loss)

# stage 2: bilingual SFT
for batch in bilingual_sft_stream:
    loss_sft = cross_entropy(model(batch.input), batch.target)
    update(model, loss_sft)

sft_model = copy_model(model)

# stage 3: RLHF with bilingual regularization
for rollout in bilingual_rlhf_stream:
    reward = reward_model(rollout)
    rl_loss = policy_gradient_loss(model, rollout, reward)
    kl = kl_to_reference(model, sft_model, rollout.prompt)
    reg = bilingual_balance_penalty(rollout) + length_bucket_penalty(rollout)
    update(model, rl_loss + 0.2 * kl + 0.1 * reg)
```

真实工程里还会多出一些关键细节：

| 组件 | 工程实现点 | 为什么重要 |
|---|---|---|
| 数据去重 | exact + fuzzy dedup | 避免某语种低质量重复样本放大 |
| 高质量重采样 | 百科、书籍、论文动态加权 | 提高知识密度与长文本稳定性 |
| 双语 SFT 混排 | 中英任务类型交叉覆盖 | 保持两种语言任务能力对称 |
| 参考策略 | 固定 SFT checkpoint | 给 RLHF 提供“不要跑偏”的锚点 |
| 长度桶 | 短中长回答分桶比较 | 降低语言长度差异的干扰 |
| REG 约束 | 语言/风格/参考奖励稳定项 | 防止灾难性遗忘 |

对零基础读者，一个实用理解方法是：预训练解决“会不会”，SFT 解决“按不按要求说”，RLHF 解决“说得好不好”，而 KL/REG 解决“优化之后别把原来会的东西忘了”。

## 工程权衡与常见坑

双语对齐训练的核心难点，不在概念，而在资源分配。

第一个权衡是语料规模与质量控制。10T 级别 token 的优势很明显，但规模扩大后，垃圾网页、重复内容、翻译腔文本也会同步增长。如果团队只盯总 token 数，不盯质量结构，那么中英双语能力会表面变强、实则变脆。很多时候，新增的不是知识，而是噪声。

第二个权衡是长上下文能力与推理成本。GLM-4 面向 128K 甚至更长上下文时，双语对齐不只是语言问题，还会变成 KV cache、注意力效率、显存占用问题。也就是说，想保住长文双语问答能力，训练和推理系统都要跟上，否则模型虽然“能吃进”长文本，真正生成时却因资源限制退化。

第三个权衡是 RLHF 提升幅度与遗忘风险。正则太弱，模型容易朝高奖励语言单边漂移；正则太强，RLHF 又可能几乎学不到新偏好。这个平衡没有一次性固定答案，通常要靠离线评测和在线反馈共同调。

下面这个表适合工程决策时快速判断：

| 维度 | 强化方向 | 好处 | 代价/风险 |
|---|---|---|---|
| 预训练 token 规模 | 更大 | 知识覆盖更广，长尾更全 | 成本高，噪声同步增加 |
| 高质量数据权重 | 更高 | 双语表达更稳，知识密度更高 | 覆盖面可能下降 |
| SFT 双语覆盖 | 更均衡 | 中英任务能力更对称 | 标注成本更高 |
| RLHF 正则强度 | 更高 | 忘却风险更低 | 偏好优化变慢 |
| 长上下文窗口 | 更长 | 更适合文档问答与 Agent | 显存和延迟压力更大 |

常见坑主要有四类。

一是把“中英混合训练”误当成“双语对齐训练”。前者只说明模型见过两种语言，后者要求它们在知识、风格、偏好上都能稳定切换。

二是高质量语料比例不足。技术类双语模型尤其依赖百科、论文、书籍、代码文档这类密度高、术语规范的文本。若被低质量网页稀释，模型会更容易生成表面流畅但术语不准的回答。

三是 SFT 任务结构不对称。比如中文样本以客服问答为主，英文样本以代码与论文总结为主，最后模型会学出“中文偏口语，英文偏技术”的不必要差异。

四是 RLHF 只看单语指标。一个典型错误是中文对齐任务做得很好，但英文技术问答分数明显掉了，团队却没有及时发现，因为评测只看中文胜率。双语模型必须保留双语评测矩阵，至少要监控中文、英文、跨语种三类任务。

一个真实工程例子是企业内部知识助手。团队往往先把中文用户反馈接到 RLHF 流程里，因为反馈更多、更容易收集。结果几轮优化后，中文满意度上升，但英文 API 文档问答能力下降，跨语言引用也更混乱。后来通过加入英文任务桶、固定 SFT 参考模型、提高 KL/REG 权重，才把单边漂移拉回来。这类问题不是算法失效，而是目标函数定义不完整。

## 替代方案与适用边界

不是每个团队都需要复现 GLM-4 级别的中英双语对齐。资源、目标场景、上线方式不同，方案也应不同。

如果资源有限，ChatGLM-6B 这一类较小模型更适合作为起点。小模型的好处不是“能力足够替代 GLM-4”，而是可以更快完成数据流程验证。也就是先确认你的中英重采样、双语 SFT、带正则的 RLHF 是否真的有效，再考虑是否值得扩大到更大参数和更长上下文。

如果业务明显偏英文，也不必机械追求中英完全对称。更现实的做法是保留中文可用能力，但把英文技术资料理解、代码解释、工具调用作为重点优化方向。不过这时必须明确，它已经不是“强双语对齐”方案，而是“英文优先、中文兜底”的业务策略。

如果场景是本地部署、隐私优先、低延迟响应，那么参数规模和上下文长度往往比 benchmark 更重要。一个 6B 量级的中英模型，只要训练数据和后训练规则设计得当，在固定领域内仍然可能比大模型 API 更稳定，因为它的行为边界可控、部署链路短。

下面给出一个场景对照表：

| 场景 | 更适合的模型路线 | 原因 |
|---|---|---|
| 长文档问答、跨语种检索、Agent 工具调用 | GLM-4 / GLM-4 All Tools | 长上下文、多工具、双语切换要求高 |
| 私有部署、预算受限、快速实验 | ChatGLM-6B 级别 | 成本低，便于验证数据与对齐流程 |
| 英文技术支持为主，中文只是辅助 | 英文优先的双语变体 | 可减少双语完全对称的训练成本 |
| 中文知识助手为主，英文文档偶尔引用 | 中文主导 + 英文检索增强 | 不必追求全量双语 benchmark 最优 |

对初级工程师，最实用的路径不是直接追求“接近 GPT-4”，而是先跑通一个小规模闭环：

1. 用中英高质量语料做重采样，验证语言基础能力不失衡。  
2. 用中英交叉任务的 SFT 数据，验证两种语言的指令风格趋于一致。  
3. 在 RLHF 中加入 KL 和长度/语言正则，验证一边优化时另一边不明显退化。  

如果这三步在 6B 模型上都做不稳，扩大到更大模型通常只会把问题放大，不会自动消失。这就是 GLM-4 双语对齐训练最值得借鉴的地方：它不是靠某个神秘技巧取胜，而是把每个阶段容易破坏双语稳定性的点都提前约束住。
