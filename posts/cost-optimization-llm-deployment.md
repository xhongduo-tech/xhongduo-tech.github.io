## 核心结论

大模型部署的成本，不是单一的“GPU 很贵”，而是三部分之和：

$$
C = H + S + O
$$

其中，$H$ 是硬件成本，白话说就是买服务器、折旧、机房配套的钱；$S$ 是云服务成本，白话说就是按需、预留、Spot 实例、存储和带宽的账单；$O$ 是运维成本，白话说就是监控、备份、告警、值班和人力维护的持续投入。

真正有效的优化，不是单纯压低某一项单价，而是在服务指标不变的前提下，提高资源利用率、减少空转，并把“基线负载”和“波峰负载”拆开处理。核心变量可以写成资源利用率：

$$
u = \frac{\text{实际计算输出}}{\text{峰值计算能力}}
$$

当 $u$ 低时，说明机器大量时间在等待、空转或低负载运行，云服务浪费会迅速放大。工程上常把这种关系近似理解为：

$$
\Delta C_S \propto (1-u)
$$

意思是：利用率越低，云服务成本里的浪费部分越大。

成本优化的主线通常只有三条：

| 主线 | 直接作用 | 典型手段 |
| --- | --- | --- |
| 提高利用率 | 减少 GPU/内存空转 | 批量推理、请求合并、模型并发调度 |
| 匹配负载波动 | 只在需要时付费 | 自动扩缩容、队列削峰填谷 |
| 降低单位资源价格 | 固定基线成本更低 | 预留实例、Savings Plan、混合部署 |

玩具例子可以先看最简单情况：一台云 GPU 每月花费 1.4 万美元，如果每天有 30% 时间空转，相当于每月有约 4200 美元没有换来有效计算。哪怕模型和用户量都没变，只要通过批量推理把空转降下来，账单就会明显收缩。

结论很直接：大模型成本优化的核心不是“更便宜的卡”，而是“更高的有效负载率 + 更合理的资源采购方式 + 持续预算治理”。

## 问题定义与边界

先明确边界。本文讨论的是“部署成本”，不是训练成本；重点是在线推理、离线批处理、混合部署这三类常见生产场景。

部署成本一般拆成三类：

| 成本项 | 具体内容 | 常见浪费来源 | 可控手段 |
| --- | --- | --- | --- |
| 硬件成本 $H$ | GPU 服务器、CPU、内存、磁盘、网络设备、折旧 | 机器买大了但长期吃不满 | 提前容量规划、提高利用率、混合部署 |
| 云服务成本 $S$ | 按需实例、预留实例、Spot、对象存储、流量 | 按需实例长期空跑、峰值配置长期保留 | 自动扩缩容、批量推理、预留锁折扣 |
| 运维成本 $O$ | 监控、日志、告警、备份、人力、故障处理 | 没有自动化、靠人工救火 | 指标体系、预算告警、成本归因 |

这里有一个重要边界：优化目标不是“账单最低”，而是“在 SLA 不变时账单更低”。

SLA，白话说是服务承诺，比如接口 95 分位延迟、可用性、错误率。假设一个问答系统要求 P95 延迟低于 800ms，那么你不能通过极端增大 batch 把延迟拖到 3 秒，再宣称“成本优化成功”。这不叫优化，这叫转移损失。

所以本文默认的优化边界是：

1. 在线服务的延迟、可用性不能明显恶化。
2. 模型输出质量不因为过度量化、过度裁剪而失真。
3. 团队要能持续观测成本，而不是月末看总账单才发现超支。

对初学者最容易混淆的一点是：成本优化不是一次性动作，而是一个闭环。你要先定义成本口径，再采集利用率，再做策略，再回看账单是否真的下降。

真实工程里，很多团队不是不会买实例，而是没有“成本归因”。比如同一套推理集群同时服务搜索、客服和摘要三个业务线，如果不做标签和归因，月底只能看到一张总账单，无法判断哪个业务在消耗资源，也就谈不上优化。

## 核心机制与推导

### 1. 为什么“利用率”是第一指标

利用率，白话说就是你买来的算力到底有多少时间在真正干活。

如果一台 GPU 理论上每秒能处理 100 个 token 的推理工作，但实际平均只跑到 45 个 token/s 的等效负载，那么：

$$
u = \frac{45}{100} = 0.45
$$

这意味着 55% 的能力没有被转成产出。对于按时间计费的云实例，这 55% 通常仍然要付钱。

一个简单近似模型是：

$$
C_S = C_{\text{base}} + C_{\text{waste}}
$$

且

$$
C_{\text{waste}} \approx k(1-u)
$$

这里 $k$ 可以理解为单位时间内的资源租赁成本。当 $u$ 从 0.45 提升到 0.75 时，浪费部分会显著下降。

### 2. 批量推理为什么能省钱

批量推理，白话说是把零散请求攒成一批一起算，减少 GPU 在小请求之间频繁切换的损耗。

玩具例子：

- 场景 A：100 个请求逐个处理，每次 batch size = 1
- 场景 B：把其中可等待的 80 个请求合并成 batch size = 16 分批执行

如果单请求模式下 GPU 平均利用率是 42%，改成批处理后达到 68%，而实例单价不变，那么同样预算下可以支撑更多吞吐；反过来看，同样吞吐下需要的实例数会下降。

但这里有边界。批量推理只适合两类任务：
- 对延迟不敏感的离线任务，比如夜间批量摘要、分类、报表生成。
- 在线但可容忍几十到几百毫秒排队的场景，比如中等并发的文本生成。

如果是强实时风控、语音交互或交易前校验，大 batch 往往会直接拖垮延迟指标。

### 3. 自动扩缩容为什么能压缩峰值浪费

自动扩缩容，白话说是负载高时自动加机器，负载低时自动减机器。

如果你的业务白天流量是夜间的 4 倍，而你为了白天高峰长期保留满配 GPU 节点，那么夜间大部分资源都在空转。自动扩缩容的价值，是让资源规模跟流量曲线同步。

可把总成本粗略写成：

$$
C(t) = n(t) \cdot p
$$

其中 $n(t)$ 是某一时刻实例数量，$p$ 是单实例单位时间价格。优化的目标不是让 $p$ 变小，而是让 $n(t)$ 更贴近真实需求。

### 4. 预留实例为什么适合基线负载

预留实例，白话说是提前承诺用量，换取更低单价。

它不适合不确定负载，但非常适合“每天都会存在”的基线流量。比如一个客服模型 24 小时都有稳定调用量，那么这部分负载用预留实例锁住折扣，通常比长期按需更便宜。

一个常见搭配是：

- 基线流量：预留实例
- 峰值流量：按需实例
- 非关键批处理：Spot 实例

这样能同时兼顾成本和稳定性。Spot，白话说是折扣很高但可能被云厂商回收的实例，因此适合可中断任务，不适合关键在线链路。

### 5. 成本治理为什么必须进入闭环

仅靠“看 GPU 利用率”还不够，因为成本问题经常不是技术问题本身，而是组织问题。完整闭环通常是：

$$
\text{指标采集} \rightarrow \text{成本归因} \rightarrow \text{预算告警} \rightarrow \text{策略调整} \rightarrow \text{复盘}
$$

其中：

- 指标采集：Prometheus、OpenTelemetry 采 GPU 利用率、内存占用、请求延迟。
- 成本归因：按团队、模型、服务、环境打标签。
- 预算告警：本周超预算 70%、85%、100% 时触发通知。
- 策略调整：改 batch、改扩缩容阈值、改实例采购方式。
- 复盘：确认账单是否真的下降，而不是只是“看起来更忙了”。

没有这个闭环，团队很容易出现“成本漂移”。成本漂移，白话说就是需求、流量、镜像、实例类型慢慢变化，但没人持续校正，结果账单在几周内悄悄失控。

## 代码实现

下面给一个可运行的 Python 玩具实现，演示“根据利用率决定批量推理、扩容和预算告警”的最小逻辑。它不是生产代码，但结构与真实系统一致：先采指标，再算利用率，再做决策。

```python
from dataclasses import dataclass

@dataclass
class Metrics:
    gpu_util: float        # 0.0 ~ 1.0
    mem_util: float        # 0.0 ~ 1.0
    queue_depth: int
    monthly_budget: float
    spent_so_far: float

def decide_action(metrics: Metrics) -> dict:
    assert 0.0 <= metrics.gpu_util <= 1.0
    assert 0.0 <= metrics.mem_util <= 1.0
    assert metrics.monthly_budget > 0
    assert metrics.spent_so_far >= 0

    budget_ratio = metrics.spent_so_far / metrics.monthly_budget

    enable_batch = metrics.gpu_util < 0.60 and metrics.queue_depth > 10
    scale_out = metrics.gpu_util > 0.85 and metrics.queue_depth > 30
    scale_in = metrics.gpu_util < 0.35 and metrics.queue_depth < 5

    budget_alert = None
    if budget_ratio >= 1.0:
        budget_alert = "hard_stop_review"
    elif budget_ratio >= 0.85:
        budget_alert = "critical"
    elif budget_ratio >= 0.70:
        budget_alert = "warning"

    return {
        "enable_batch": enable_batch,
        "scale_out": scale_out,
        "scale_in": scale_in,
        "budget_alert": budget_alert,
        "budget_ratio": round(budget_ratio, 4),
    }

# 玩具例子：低利用率但队列有积压，说明可以批量处理
m1 = Metrics(gpu_util=0.52, mem_util=0.48, queue_depth=20, monthly_budget=10000, spent_so_far=6500)
r1 = decide_action(m1)
assert r1["enable_batch"] is True
assert r1["scale_out"] is False
assert r1["budget_alert"] is None

# 高利用率且队列很深，应扩容
m2 = Metrics(gpu_util=0.91, mem_util=0.76, queue_depth=50, monthly_budget=10000, spent_so_far=7200)
r2 = decide_action(m2)
assert r2["scale_out"] is True
assert r2["budget_alert"] == "warning"

# 成本已接近上限，应触发强告警
m3 = Metrics(gpu_util=0.70, mem_util=0.68, queue_depth=12, monthly_budget=10000, spent_so_far=9100)
r3 = decide_action(m3)
assert r3["budget_alert"] == "critical"

print("all assertions passed")
```

这个例子里有三个关键判断：

| 判断 | 含义 | 工程目的 |
| --- | --- | --- |
| `gpu_util < 0.60 and queue_depth > 10` | 机器没跑满，但已有待处理请求 | 开启批量推理，提高利用率 |
| `gpu_util > 0.85 and queue_depth > 30` | 算力快打满且请求堆积 | 扩容，避免延迟失控 |
| `spent_so_far / monthly_budget > 阈值` | 本月预算接近上限 | 提醒团队调整策略 |

真实工程例子可以更完整一些。假设一个金融团队有两类任务：

- 在线问答：要求低延迟，服务内部员工查询制度和报告。
- 夜间分析：把当日文档做风险摘要和分类。

合理做法通常不是“一套集群全接”，而是分层：

1. 本地或长期预留云实例承接在线问答基线流量。
2. 夜间分析任务进入队列，统一批量执行。
3. 月度大批量回补任务使用 Spot 实例。
4. 预算系统按团队、模型、任务类型分别统计。

如果只做第 1 步，不做第 2 到第 4 步，往往会出现白天为了低延迟保留大量闲置容量，夜里又因没有批量调度而继续低效运行，最终“看起来系统稳定，实际上单位请求成本很差”。

预算告警 webhook 的最小字段一般包括：

| 字段 | 作用 |
| --- | --- |
| `service` | 哪个服务超预算 |
| `team` | 哪个团队负责 |
| `budget_limit` | 预算上限 |
| `actual_spend` | 当前已花费金额 |
| `forecast_spend` | 月底预测金额 |
| `severity` | 告警等级 |
| `window` | 时间窗口，例如 `2026-03` |

## 工程权衡与常见坑

### 1. 本地硬件 vs 云服务

本地硬件的优点是长期高负载时单位成本更低，缺点是前期投入大、弹性差。云服务的优点是弹性强、启动快，缺点是长期 24/7 运行时账单容易高。

简单判断标准是：如果业务长期稳定、合规要求高、延迟要求严，倾向本地或专有云；如果流量波动大、产品还在试错期，倾向云。

### 2. 预留实例 vs Spot

预留实例适合稳定基线，但如果你对未来用量判断错误，就会买多。Spot 很便宜，但可能被中断，所以只能承接可重试、可恢复任务。

真实工程里常见错误不是“不会选”，而是把 Spot 直接接入在线主路径。一旦实例回收，用户请求就会抖动甚至失败。

### 3. 自动扩缩容 vs 预热成本

自动扩缩容不是没有代价。GPU 节点启动、模型加载、KV cache 预热都需要时间。阈值设得太晚，扩出来也来不及；设得太早，又会导致频繁扩缩，浪费更多。

### 4. 常见坑与规避

| 常见坑 | 后果 | 规避手段 |
| --- | --- | --- |
| 没有预算告警 | 月底才发现超支 | 设 70%/85%/100% 分级告警 |
| 只看总账单，不做归因 | 不知道哪条业务在烧钱 | 给实例、命名空间、服务打标签 |
| 没有 GPU/内存利用率指标 | 看不到批量机会 | 接入 OpenTelemetry 或 Prometheus |
| 批量推理直接用于强实时链路 | 延迟恶化 | 区分在线和离线任务 |
| 所有流量都放按需实例 | 基线负载单价过高 | 基线改预留，峰值保留按需 |
| 盲目追求满载 | 延迟和稳定性变差 | 以 SLA 为约束，而不是单看利用率 |

很多团队的真正问题不是技术能力，而是没有分清“优化对象”。如果你优化的是 GPU 利用率，却让用户等待时间翻倍，那么业务成本可能反而更高，因为用户体验下降也会产生损失。

## 替代方案与适用边界

常见部署方案大致有三种：全云、全本地、混合。

| 部署模型 | 成本特点 | 适用场景 | 注意事项 |
| --- | --- | --- | --- |
| 全云 | 初期投入低，弹性最好，但长期高负载可能最贵 | 创业团队、需求不稳定、快速试验 | 必须做扩缩容和预算治理 |
| 全本地 | 长期稳定负载下单位成本低，可控性强 | 高合规、低延迟、长期固定业务 | 前期 CAPEX 高，扩容慢 |
| 混合部署 | 基线成本可控，峰值弹性较好 | 负载有明显波峰波谷的企业场景 | 调度、监控、归因更复杂 |

这里给一个简单适用边界：

- 如果你是小团队、预算紧、业务不稳定，全云通常更合理，但一定要用自动扩缩容和 Spot/预留组合。
- 如果你是金融、政务、制造等强合规场景，而且模型服务是长期核心能力，全本地或专有部署更稳。
- 如果你已经有稳定基线流量，但偶尔会出现营销活动、月末报表、夜间批量任务，混合部署通常最实用。

前面提到的金融团队案例，就是典型混合部署：

- 基线推理在本地集群，保证低延迟和合规。
- 峰值突发时接入云端 autoscale。
- 预留实例覆盖常态云需求。
- Spot 实例承接批量分析。
- 最后用预算看板和团队归因做 FinOps 闭环。

这类方案的难点不在“能不能搭起来”，而在于调度策略、成本标签和预算体系要同步设计。否则系统虽然是混合部署，管理方式却仍然是“月底看总账”，那混合带来的复杂度会吞掉优化收益。
