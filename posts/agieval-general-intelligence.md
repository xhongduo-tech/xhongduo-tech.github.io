## 核心结论

AGIEval 是一个用**人类标准化考试**构建的评测基准。标准化考试可以理解为“题型、评分规则、分数区间都固定的正式考试”，例如 SAT、GRE、公务员考试、法考等。它的核心价值不在于判断模型“会不会写得像人”，而在于判断模型“能不能稳定完成真正考人的高认知任务”。

AGIEval 的公开版本覆盖 20 种考试类型、约 8062 道客观题，题目来自数学、语言、逻辑、法律、常识等不同领域。它把模型能力拆成四个维度来观察：

| 能力维度 | 白话解释 | 典型考试示例 | 关注点 |
|---|---|---|---|
| 理解 | 看懂题目、条件和语义关系 | SAT、GRE 阅读 | 是否正确理解文本和约束 |
| 知识 | 记住并调用事实与概念 | 法考、常识类考试 | 是否具备领域知识 |
| 推理 | 根据条件一步步推出结论 | LSAT、Civil Service | 是否会逻辑分析 |
| 计算 | 做数值运算和公式变换 | SAT Math、GMAT Math | 是否会算且不出低级错 |

最直接的指标是准确率：

$$
Accuracy=\frac{Correct}{Total}
$$

例如，把 SAT 数学单科题目交给模型做，如果 20 道题答对 19 道，那么准确率就是：

$$
Accuracy=\frac{19}{20}=95\%
$$

这时就可以把模型成绩与人类平均成绩、Top 1% 成绩放在同一坐标系里比较。新手可以把它理解为：把题目拿给 AI 做，算对了多少，再和真实考生表现对比，就知道它离人类水平还有多远。

## 问题定义与边界

AGIEval 要解决的问题很明确：**评估模型是否具备完成正式人类考试的能力**。这里的“能力”主要指可验证的理解、知识、推理、计算，而不是文风、态度、流畅度。

这一定义有三个边界。

第一，只看**客观题**。客观题可以理解为“答案唯一、评分规则固定的题”，例如单选、多选、判断。这样做的原因是主观题很难统一打分。若让不同评审去改作文或论述题，分数波动会很大，模型之间也难以公平比较。

第二，AGIEval 更关心**任务完成结果**，不关心输出看起来是否“像一个很会聊天的人”。一个模型即使生成得非常自然，如果选择题经常选错，它在 AGIEval 里依然不会得到高分。

第三，它通常采用**零样本**和 **Chain-of-Thought prompting**。零样本可以理解为“不给额外示例，直接做题”；Chain-of-Thought 可以理解为“让模型先写中间推理过程，再给最终答案”。这样做的目的，是让不同模型在尽量统一的条件下比较，而不是通过复杂的手工调参把某一个模型“刷题刷高分”。

下面这个表格可以帮助理解它评什么、不评什么：

| 项目 | AGIEval 的处理方式 | 原因 |
|---|---|---|
| 客观题 | 纳入 | 可自动评分，结果可复现 |
| 主观题 | 剔除或不作为核心比较对象 | 人工评分不稳定 |
| 零样本 | 常用 | 降低任务特化工程干预 |
| CoT 提示 | 常用 | 提高复杂推理题可比性 |
| 文风流畅度 | 不作为核心指标 | 不等于解题能力 |
| 人类分数对照 | 强调 | 能直接定位模型在人类尺度上的水平 |

玩具例子很简单。假设有一组公务员考试逻辑选择题，共 10 道。模型答对 7 道，那么准确率是 70%。这已经足够说明它在该类推理任务上的水平，不需要再请老师人工阅读它的长篇解释来决定“写得像不像正确答案”。

真实工程里，这种边界很重要。比如教育行业要评估一个大模型能否用于题目解析，或者政务场景要评估它能否辅助处理规范性问答，就可以优先选取 AGIEval 中接近目标场景的客观考试题先测一轮，再决定是否进入更贵的人测或微调阶段。

## 核心机制与推导

AGIEval 的机制可以概括成四步：**考试集合 -> 模型做题 -> 按考试计分 -> 汇总到能力维度并与人类对比**。

先看最小计算单位。每一场考试都可以单独计算准确率。假设某个 SAT Math 子集有 20 题，模型答对 19 题，那么：

$$
Accuracy_{SAT\ Math}=\frac{19}{20}=95\%
$$

如果该子集同时被标注为“计算”和“推理”相关，那么它的结果就会进入这两个维度的统计。

再往上是维度聚合。最简单的理解方式是：先分别计算每个考试的准确率，再按照考试归属，把它们汇总成理解、知识、推理、计算四类能力画像。论文和公开说明强调的是可比较、可解释，因此核心思路不是做一个花哨的大一统分数，而是保留“考试级别”和“能力级别”两层视角。

可以把流程写成一个简化表达：

$$
Accuracy_{exam}=\frac{Correct_{exam}}{Total_{exam}}
$$

$$
Score_{ability}=\text{Aggregate}(\{Accuracy_{exam}\mid exam \in ability\})
$$

其中 `Aggregate` 可以理解为“按该能力维度下的考试结果做平均或汇总”。工程上最重要的不是公式形式多复杂，而是：**同一批题、同一套提示、同一套评分规则，才能让模型横向可比**。

下面给一个玩具例子。假设只有两场考试：

| 考试 | 题数 | 答对 | 准确率 | 对应维度 |
|---|---:|---:|---:|---|
| SAT Math | 20 | 19 | 95% | 计算、推理 |
| Civil Service Logic | 10 | 7 | 70% | 推理、理解 |

如果只看推理维度，模型既要做数学推导，也要做逻辑判断。此时我们不会因为它数学好，就断言“整体推理能力已经接近人类顶尖水平”；还要看不同考试上的分布是否均衡。

真实工程例子更能说明机制价值。假设一个团队要把大模型部署到教育答疑和政务助手两个场景。教育答疑更依赖数学计算与文本理解，政务助手更依赖法律常识与逻辑判断。那么团队就不该只看一个总平均分，而要分别查看 SAT/GMAT 类考试和 Civil Service/法律资格类考试的表现。如果后者明显偏低，就说明模型在“看起来会说”之外，实际处理正式规则任务的能力还不够。

## 代码实现

工程上实现一个 AGIEval 风格评测，核心流水线通常只有四步：加载考试数据、构建提示词、调用模型、自动评分。

先定义输入输出结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| exam_id | 字符串 | 考试标识，如 `sat_math` |
| ability | 列表 | 该考试对应的能力维度 |
| question | 字符串 | 题干与选项 |
| answer | 字符串 | 标准答案 |
| prompt | 字符串 | 拼接后的模型输入 |
| prediction | 字符串 | 模型输出答案 |
| reasoning | 字符串 | 模型中间推理文本 |
| accuracy | 浮点数 | 当前考试或批次准确率 |

下面是一个可运行的最小 Python 示例。它不依赖真实模型接口，而是用一个假的 `mock_model` 演示完整评分流程。

```python
from dataclasses import dataclass
from typing import List, Dict

@dataclass
class Question:
    stem: str
    options: Dict[str, str]
    answer: str

@dataclass
class Exam:
    exam_id: str
    ability: List[str]
    questions: List[Question]

def build_prompt(question: Question, zero_shot: bool = True, cot: bool = True) -> str:
    instruction = "请阅读题目并给出最终选项。"
    if zero_shot:
        instruction = "你将直接做题，不提供示例。" + instruction
    if cot:
        instruction += " 先输出简短推理，再输出 Final Answer: <选项>。"
    options_text = "\n".join([f"{k}. {v}" for k, v in question.options.items()])
    return f"{instruction}\n\n题目：{question.stem}\n{options_text}"

def mock_model(prompt: str) -> str:
    if "2 + 2" in prompt:
        return "先计算。2 + 2 = 4。\nFinal Answer: B"
    return "根据条件判断。\nFinal Answer: A"

def parse_final_answer(model_output: str) -> str:
    marker = "Final Answer:"
    if marker not in model_output:
        return ""
    return model_output.split(marker, 1)[1].strip()

def score_exam(exam: Exam) -> float:
    correct = 0
    for q in exam.questions:
        prompt = build_prompt(q, zero_shot=True, cot=True)
        output = mock_model(prompt)
        pred = parse_final_answer(output)
        if pred == q.answer:
            correct += 1
    return correct / len(exam.questions)

exam = Exam(
    exam_id="toy_math",
    ability=["calculation", "reasoning"],
    questions=[
        Question("2 + 2 等于多少？", {"A": "3", "B": "4", "C": "5", "D": "6"}, "B"),
        Question("已知三角形内角和为？", {"A": "180度", "B": "90度", "C": "360度", "D": "120度"}, "A"),
    ],
)

acc = score_exam(exam)
assert acc == 1.0
print("accuracy =", acc)
```

这个例子对应的思路就是：

```python
for exam in exams:
    for question in exam.questions:
        prompt = build_prompt(question, zero_shot=True, cot=True)
        output = model(prompt)
        prediction = parse(output)
    accuracy = correct / total
```

如果换成真实工程，`mock_model` 会被替换成大模型 API 调用函数，最后再把每场考试的 `accuracy`、原始响应、推理文本、时间戳、模型版本统一写入报告。这样才能和人类平均分或 Top 1% 基准做稳定对照。

## 工程权衡与常见坑

AGIEval 的结果很直观，但工程上最容易出问题的地方也很集中。

第一，**数据泄露**。数据泄露可以理解为“模型在训练阶段已经见过题目或答案”，这样测出来的高分不再代表真实泛化能力。若训练集里已经包含某年公务员考试原题，模型在该子集上拿高分，可能只是记住了答案。

可以用一个简化表达理解偏差：

$$
Observed\ Accuracy = True\ Signal + Leakage\ Bias
$$

其中 `True Signal` 是模型真实解题能力，`Leakage Bias` 是提前见过答案带来的虚高部分。工程上要尽量让后者接近 0。

第二，**主观题评分不一致**。主观题看起来更接近真实场景，但在基准评测里反而不稳定。没有统一标准时，不同评审会给出不同分数，模型之间的比较就会失真。这也是 AGIEval 重点保留客观题的原因。

第三，**提示词不统一**。如果一个模型用零样本，另一个模型给了多轮示例或特殊模板，那么分数不能直接比较。评测的目标是比较模型，不是比较谁的提示工程更激进。

第四，**答案抽取失败**。很多模型会输出长推理，但最终选项格式不稳定。若没有稳定的解析逻辑，可能把正确答案误判成错误答案。

| 常见坑 | 现象 | 风险 | 规避策略 |
|---|---|---|---|
| 数据泄露 | 某些考试分数异常高 | 高估模型泛化能力 | 检查来源、时间戳、去重与污染扫描 |
| 混入主观题 | 评分依赖人工理解 | 可复现性差 | 剔除主观题，单独建人工评审流程 |
| 提示词不一致 | 模型条件不同 | 横向比较失效 | 固定 zero-shot + CoT 模板 |
| 答案解析不稳 | 模型说对但没被识别 | 统计误差 | 强制 `Final Answer` 输出格式 |
| 题目预处理错误 | 选项丢失、公式乱码 | 结果无效 | 建立数据校验与回放样例 |

真实工程里，一个常见场景是法务或政务助手上线前评测。如果模型训练语料里混入过某年法考真题，它在法律资格相关子集上的高分就不可信。此时更稳妥的做法是按年份重建干净切片，或者换一套明确未被训练覆盖的题源重新评估。

## 替代方案与适用边界

AGIEval 很适合回答一个问题：**模型能否完成类似人类正式考试的高认知任务**。但它不是所有场景的最佳评测。

如果你更关心聊天体验、拒答风格、多轮对话、是否讨好用户，那么 RLHF 风格评估或评分模型评估更贴切。RLHF 可以理解为“用人类偏好把模型调得更像人们愿意接受的回答方式”。这类方法擅长评估对话体验，但不一定能精确反映做题能力。

如果你更关心百科知识覆盖，MMLU 这类多学科问答基准也很常见。TruthfulQA 则更关注模型是否会“听起来合理但事实上错误”。这些都重要，但与 AGIEval 的目标并不相同。

| 基准 | 主要目标 | 题型 | 评分方式 | 适用边界 |
|---|---|---|---|---|
| AGIEval | 对齐人类正式考试能力 | 以客观题为主 | 准确率、人类分数对照 | 教育、法律、政务等高认知任务 |
| MMLU | 多学科知识与推理覆盖 | 选择题 | 准确率 | 通用知识能力盘点 |
| TruthfulQA | 事实真实性与幻觉控制 | 问答 | 自动/人工结合 | 检查是否胡说八道 |
| RLHF/对话评测 | 对话体验与偏好对齐 | 开放式生成 | 人工偏好或评分模型 | 聊天产品、助手产品 |

对低资源语言或非英语区域，AGIEval 的思路也能迁移。做法不是照抄题库，而是保留它的评测逻辑：选择本地标准化考试、保留客观题、统一提示词、计算准确率、再与本地人类基准对比。比如要评估一个中文模型是否适合教育场景，就可以选取国内标准化考试中的客观题，按同样方式构建一个本地版本的 AGIEval 风格基准。

因此，AGIEval 的适用边界可以概括为一句话：**当你要验证“模型能不能像人一样完成正式认知任务”时，它非常合适；当你要验证“模型是否会聊天、会安抚、会写漂亮文本”时，它就不够了。**
