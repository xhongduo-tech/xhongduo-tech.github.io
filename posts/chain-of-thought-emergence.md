## 核心结论

链式思维，英文常写作 Chain-of-Thought 或 CoT，指模型把中间推理步骤显式写出来，再据此得到最终答案。它不是“只要写出思路就会更聪明”的通用开关，而是一种对模型已有推理能力的放大器。

核心结论有两条。

第一，CoT 的收益通常不是线性出现，而是在模型规模、单步推理准确率、上下文工作记忆同时达到一定条件后才会明显出现。Wei 等人的经典结果说明：在数学和符号推理任务上，足够大的模型会突然开始从“写步骤”中获益，而较小模型往往得不到同样收益。

第二，小模型做不到稳定 CoT，不是因为“不会输出更长文本”，而是因为每一步本身就不够准。CoT 会把每一步中间状态写回上下文，相当于把上下文窗口当成临时草稿纸。草稿纸本身没有纠错能力，如果前两步已经错了，后面的步骤只是在错误状态上继续推导，最终错误会累积而不是被修复。

下面这张表先给出直观对比。

| 情况 | 单步推理能力 | 工作记忆是否够用 | 开启 CoT 后常见结果 |
|---|---:|---:|---|
| 小模型 | 低 | 常常偏紧 | 中间步骤变多，但错误更早暴露并累积，准确率可能下降 |
| 中等模型 | 中 | 视任务而定 | 在短链任务上偶尔有帮助，长链任务不稳定 |
| 大模型 | 高 | 通常更充足 | 能稳定利用中间状态，多步任务准确率明显提升 |

一个玩具例子足够说明这个现象：让一个学生做四步算术题。如果他每一步都只有六成概率算对，那么把每一步都写下来，并不会自动变成正确答案；只是把错误写得更完整。只有当每一步都大概率正确时，写步骤才真正有价值。

## 问题定义与边界

这里讨论的“链式思维的涌现”，不是指模型突然学会任何复杂推理，而是指这样一个更具体的现象：

在多步任务上，模型规模较小时，给出“请一步一步思考”或少量 CoT 示例，收益很小甚至负收益；当模型规模达到某个临界区间后，CoT 开始显著提高最终正确率。

边界必须说清楚，否则结论会被误用。

第一，这里讨论的是**多步推理任务**，例如小学数学应用题、符号操作、逻辑约束满足。若任务本身是一跳检索或简单分类，CoT 未必重要。

第二，这里讨论的是**显式中间步骤写入上下文**的 CoT。若系统内部有隐藏推理、外部工具调用、程序执行器，那么机制已经不是单纯的“模型靠文本草稿完成推理”。

第三，这里讨论的是**最终正确率**，不是“回答看起来更有条理”。小模型常见的问题正是：解释写得很像样，但中间状态从第二步起已经偏了。

可以把问题压缩成两个量：

1. 单步命中率。白话解释：模型在每一个局部推理动作上做对的概率。
2. 可承受链长。白话解释：在连续做 $k$ 步时，错误还不会迅速失控的最大步数。

| 模型规模 | 单步命中率 | 可承受链长 | CoT 效果 |
|---|---:|---:|---|
| 小 | 低 | 1 到 2 步 | 常为负收益 |
| 中 | 中等 | 2 到 4 步 | 对短链偶尔有效 |
| 大 | 高 | 更长 | 能稳定利用 CoT |

中学生做四步题的例子可以直接映射到这里。若他每一步都可能错一次，那么“把四步都写出来”并没有解决本质问题。本质问题不是有没有草稿，而是草稿里的每一步是否足够可靠。

## 核心机制与推导

CoT 的本质，可以理解为“把推理状态外显到上下文里”。工作记忆，白话说就是模型在当前上下文中暂时记住并继续使用中间结果的能力。EMNLP 2024 关于 working memory 的研究，把推理上限和这种中间状态保持能力直接联系起来，这个角度很重要。

### 1. 为什么错误会连乘

假设一条推理链有 $N$ 步，每一步独立成功概率为 $s$，失败概率为 $p=1-s$。最粗略但很有用的近似是：

$$
\mathrm{Acc}(N) \approx s^N = (1-p)^N
$$

这不是精确描述所有模型行为的完整定律，但足够解释趋势：只要每一步不是非常稳，链一长，最终正确率会指数下降。

玩具例子：

- 若每步成功率 $s=0.6$，四步全对概率是 $0.6^4=0.1296$
- 若每步成功率 $s=0.8$，四步全对概率是 $0.8^4=0.4096$
- 若每步成功率 $s=0.9$，四步全对概率才到 $0.9^4=0.6561$

这说明一个关键事实：CoT 要有用，前提不是“模型能写步骤”，而是“模型写的大多数步骤都对”。

### 2. 为什么会出现阈值

把链长记为 $k$。若希望 CoT 至少不要比某个朴素基线更差，可以得到一个阈值条件。按照用户给定的研究摘要，可用一种简化判断：

$$
p < \frac{1}{k}
$$

白话解释：链越长，每一步允许的失败率就必须越低。四步链时，若失败率高于 $25\%$，整体正确率会很快塌掉。即便这个阈值只是简化近似，它也抓住了工程上最重要的关系：**链长和单步准确率之间不是可随意替换的，链长增加会迅速提高对单步可靠性的要求。**

### 3. 为什么小模型特别容易被 CoT 反噬

小模型的问题通常不是“不会生成长答案”，而是：

1. 局部推理动作不稳
2. 早期错误难以自我修复
3. 上下文容量有限时，重要状态会被后续噪声覆盖
4. 生成更多 token 会给自己喂入更多潜在错误条件

所以 CoT 对小模型常出现一个反直觉现象：答案更长、更像分析过程，但最终更不对。因为模型不是在“思考得更深”，而是在“把错误状态显式缓存起来”。

### 4. 真实工程例子

Wei 等人的工作在 GSM8K 上给出了最经典的证据：当模型规模足够大时，少量 CoT 示例能显著提升数学推理准确率。这个结果的重要性不在于某个具体百分点，而在于它证明了 CoT 不是通用 prompt 技巧，而是依赖模型本体能力的能力放大器。

工程上可以这样理解：

- 大模型：CoT 是把已有但隐含的推理能力显式展开
- 小模型：CoT 常常只是把不稳定的中间猜测写出来

## 代码实现

工程上不要先问“要不要加 step-by-step”，而要先测“这个模型的单步命中率是否够高”。

下面给一个最小可运行脚本。它做三件事：

1. 用公式估算链长为 $N$ 时的最终准确率
2. 用阈值 $p < 1/k$ 做一个非常粗的启用判断
3. 给出一个小表，帮助决定是否打开 CoT

```python
from math import isclose

def final_acc(step_success: float, chain_len: int) -> float:
    assert 0.0 <= step_success <= 1.0
    assert chain_len >= 1
    return step_success ** chain_len

def cot_gain(step_success: float, k: int) -> bool:
    """
    粗略判断：若失败率 p < 1/k，则认为 CoT 至少具备可尝试的条件。
    这是启发式规则，不是严格定理。
    """
    assert 0.0 <= step_success <= 1.0
    assert k >= 1
    p = 1.0 - step_success
    return p < 1.0 / k

def report(step_success_values, chain_lengths):
    rows = []
    for s in step_success_values:
        for n in chain_lengths:
            rows.append({
                "step_success": s,
                "chain_len": n,
                "final_acc": round(final_acc(s, n), 4),
                "enable_cot": cot_gain(s, n),
            })
    return rows

# assert: 基本正确性检查
assert isclose(final_acc(0.8, 4), 0.4096, rel_tol=1e-9)
assert cot_gain(0.8, 4) is True   # p = 0.2 < 0.25
assert cot_gain(0.6, 4) is False  # p = 0.4 > 0.25

if __name__ == "__main__":
    rows = report([0.6, 0.8, 0.9], [2, 4, 6])
    for row in rows:
        print(row)
```

用这个脚本，可以先做一个“玩具决策”。假设某模型在抽样评估里，单步成功率约为 0.6，那么：

| 单步成功率 $s$ | 链长 $N$ | 估计最终准确率 $s^N$ | 是否值得尝试 CoT |
|---|---:|---:|---|
| 0.6 | 2 | 0.36 | 短链可试，但不稳 |
| 0.6 | 4 | 0.1296 | 通常不值得 |
| 0.8 | 4 | 0.4096 | 可试，但仍需验证 |
| 0.9 | 4 | 0.6561 | 更可能稳定受益 |

真实工程里，单步成功率不是手填，而是从小批量评估中测出来。做法通常是：

1. 抽一批目标任务
2. 把问题拆成若干中间步骤或用人工标注的局部目标
3. 测模型每一步是否命中
4. 估计链长分布
5. 再决定是否打开 CoT

这比“先开再看结果”更稳，因为 CoT 会直接改变延迟、token 成本和错误传播路径。

## 工程权衡与常见坑

部署 CoT 时，真正要权衡的是准确率、延迟、成本、可观测性四件事，而不是只看“答案是否更像推理”。

### 常见坑 1：把更长答案误判为更强推理

这是最常见误判。模型写了五段分析，不等于它真的维护了五步正确状态。很多小模型在第二步就已经偏航，后面只是围绕错误前提继续圆。

### 常见坑 2：忽略 token 成本

CoT 会显著增加输出 token。token，白话说就是模型处理和生成文本时计费与延迟的基本单位。若你的线上任务量大，哪怕准确率略涨，也可能被延迟和成本抵消。

### 常见坑 3：上下文被错误状态污染

一旦早期中间结论写错，后续步骤会把这个错误当事实继续使用。这比“直接短答错”更麻烦，因为错误有了结构，模型更难在后面自发修正。

### 常见坑 4：把大模型论文结果直接迁移到轻量模型

Wei 等人的结论说明“大模型会涌现 CoT 收益”，不等于“任何模型加 step-by-step 都会更好”。这是论文外推时最常见的误用。

可以用下面这张表做工程检查。

| 方案 | CoT 示例数 | 延迟 | token 成本 | 准确率变化风险 |
|---|---:|---:|---:|---|
| 直接输出 | 0 | 低 | 低 | 基线稳定 |
| 零样本 CoT | 0，但要求“逐步思考” | 中 | 中 | 小模型常负收益 |
| 少样本 CoT | 4 到 8 | 高 | 高 | 大模型常收益明显 |
| CoT + 自检 | 4 到 8 + 复核 | 更高 | 更高 | 能缓解部分错误累积 |

一个更实用的流程是：

评估单步命中率 -> 估算链长分布 -> 只在满足阈值的任务上启用 CoT -> 监控最终正确率与 token 开销 -> 回退负收益场景。

真实工程例子可以用数学题系统来说明。若线上主要是两步以内的结构化问题，中等模型可能已经够用，CoT 带来的额外成本未必划算。若任务是 GSM8K 一类的多步算术推理，大模型配合少样本 CoT 才更可能体现优势。

## 替代方案与适用边界

当模型还没有达到稳定 CoT 的能力阈值时，不要强行让它写完整长链。更稳的做法，是减少链长、增加外部校验、把错误尽早截断。

### 1. Self-check

self-check，白话说就是先给出答案，再让模型专门检查关键步骤或约束。它不要求模型一次生成完整长链，而是把“生成”和“检查”拆开。

适用边界：单步能力一般，但对局部一致性检查还可以。

### 2. Contrastive 或多候选比较

contrastive 可以理解为“同时比较多个候选推理，保留更一致的那个”。它的优势不是让单条链更长，而是降低单次失误的影响。

适用边界：模型能生成多个候选，但单条链不稳定。

### 3. Multi-pass 分阶段推理

multi-pass，白话说就是分多轮做，每一轮只完成一个子目标。比如先抽取条件，再列式，再求值，而不是一次性写完整思路。

适用边界：任务天然可拆分，且每一步都能被验证。

### 4. 外部工具或检索

当某些中间步骤其实是可计算或可查证的，就不要让语言模型硬猜。让程序执行算术，让检索系统提供事实，模型只负责组织流程。

适用边界：任务中存在明确的外部真值来源。

| 替代机制 | 核心思路 | 适用单步命中率区间 | 链长上限 | 适用场景 |
|---|---|---|---:|---|
| Self-check | 先答再验 | 中低 | 中 | 格式检查、算术复核 |
| Contrastive | 多候选比较 | 中 | 中 | 候选可并行生成 |
| Multi-pass | 分阶段推理 | 中低到中 | 每轮短 | 可拆分任务 |
| 工具调用 | 外部执行关键步骤 | 较低也可 | 不敏感 | 算术、检索、规则计算 |

下面给一个简短伪代码，说明“少量推理 + 自检”比“长链 CoT”更适合小模型的场景。

```python
def solve_with_check(question, model):
    draft = model.answer_brief(question)
    checks = model.check_constraints(question, draft)

    if checks["arithmetic_ok"] and checks["facts_ok"]:
        return draft

    repaired = model.repair_answer(question, draft, checks)
    return repaired
```

这个思路的核心不是让模型“想更多”，而是让模型“少写不可靠中间状态，把验证前移”。当每步成功率只有 60% 左右时，把四步长链拆成两个短阶段，再加入检查点，通常比原地拉长 CoT 更稳。
