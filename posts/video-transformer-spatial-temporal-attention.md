## 核心结论

视频 Transformer 的本质，是先把视频切成一串 **token**。token 可以理解为“模型处理信息的最小单位”，通常对应某一帧中的一个图像小块。模型接下来要同时解决两个问题：一是同一帧内部哪些区域彼此相关，二是前后帧之间哪些区域在表达同一个动作或状态变化。

这件事的难点不在“有没有足够多的信息”，而在“如何用有限算力把时空关系学对”。如果把所有帧、所有 patch 一次性放进同一个全局注意力里，关系最完整，但计算量和显存成本最高；如果把时间关系和空间关系拆开学，成本会显著下降；如果再把注意力限制在局部窗口内，扩展到更长视频就更现实。

把一个 `8` 帧短视频切成每帧 `49` 个 patch，总共有 `8 × 49 = 392` 个 token。模型不是“直接看视频”，而是在这些 token 之间建立关系图：先学一帧里人物手和球拍是否相关，再学前后帧里挥拍动作如何连续。这就是视频 Transformer 真正做的事。

下面这张表先给出三类主流路线的总览：

| 架构 | 核心思路 | 复杂度趋势 | 适用场景 |
|---|---|---|---|
| 联合时空注意力 | 所有 token 直接全局互相注意 | 最高，`O(T^2N^2)` | 小规模或研究分析 |
| 分离时空注意力 | 先时间后空间，或反过来 | 中等，`O(NT^2 + TN^2)` | 通用视频任务 |
| Video Swin | 局部 3D window + shifted window | 更低，近似线性扩展 | 工程部署、长视频 |

结论可以压缩成一句话：**视频 Transformer 不是单一模型，而是一组“如何组织时空注意力”的设计选择；TimeSformer、ViViT、Video Swin 的主要差别，本质上都是时空建模完整性与工程成本之间的取舍。**

## 问题定义与边界

先把讨论对象限定清楚。本文讨论的是 **视频理解中的视频 Transformer**，也就是输入一段视频，输出分类、检测、检索或特征表示。这里不展开视频生成、视频扩散模型、音视频联合建模，也不覆盖大型多模态系统中的全部视频模块。

通常把输入表示为：

$$
X \in \mathbb{R}^{T \times N \times d}
$$

其中：

| 符号 | 含义 |
|---|---|
| `T` | 视频帧数 |
| `N` | 单帧 patch 数 |
| `L = TN` | 总 token 数 |
| `d` | token 维度 |
| `m` | window 内 token 数 |

这里的 **patch** 可以理解为“把一帧图像切成许多规则小块”，每个小块再映射成一个向量；`d` 是这个向量的长度，也就是模型内部表征的维度。

一个最小玩具例子：如果视频有 `4` 帧，每帧切成 `3×3=9` 个 patch，那么总 token 数是：

$$
L = T \times N = 4 \times 9 = 36
$$

这时问题不是“36 个 token 够不够”，而是“36 个 token 之间的关系要怎样连起来”。如果全连，时空关系最完整；如果按时间轴或空间轴拆开连，成本更低；如果只在局部窗口内连，扩展性最好。

为什么视频比图像更难？图像 Transformer 只需要建模空间关系，而视频多了时间轴。也就是说，同样一个 patch token，除了要知道“它在这一帧里和谁相关”，还要知道“它在前后帧里的变化是否构成动作”。一旦帧数变大，token 数会按 `T` 线性增长，但全局注意力的两两比较会按平方增长，这就是视频 Transformer 的核心压力来源。

因此，本文的边界很明确：**重点不是介绍所有视频模型，而是解释 Transformer 如何高效处理时空依赖，以及三种代表性架构为何要做不同的复杂度设计。**

## 核心机制与推导

先看最直接的做法：**联合时空注意力**。注意力可以理解为“让每个 token 去看所有其他 token，并给不同对象分配不同权重”。其基本形式是：

$$
A = \text{softmax}\left(\frac{QK^T}{\sqrt{d}}\right)
$$

其中 `Q`、`K`、`V` 分别是 query、key、value 投影。对于视频，如果把全部 `TN` 个 token 拉平成一个长度为 `L=TN` 的序列，那么注意力矩阵大小就是 `L×L`，复杂度近似为：

$$
O(L^2) = O((TN)^2) = O(T^2N^2)
$$

这意味着只要帧数 `T` 和单帧 patch 数 `N` 同时增大，计算会迅速失控。它的优势是关系建模最完整，因为任意两个时空位置都能直接交互；它的代价是显存和速度都最差。

### 联合注意力为什么贵

看一个具体数字。设 `T=4`、`N=9`，那么总 token 数 `L=36`。联合注意力需要比较的 token 对数大约是：

$$
36^2 = 1296
$$

这个数字在玩具例子里不大，但如果改成更接近真实设置，比如 `T=32` 帧、每帧 `14×14=196` 个 patch，那么：

$$
L = 32 \times 196 = 6272,\quad L^2 \approx 3.93 \times 10^7
$$

还没算多头注意力、batch size 和中间激活，这已经足以说明为什么“全局时空全连”不适合直接做长视频工程。

### 分离时空注意力在做什么

第二类路线是 **分离时空注意力**。分离的意思是：不一次性处理全部时空关系，而是先处理一个维度，再处理另一个维度。最常见的是先做时间注意力，再做空间注意力，或者反过来。

如果先做时间注意力，那么固定每个空间位置，沿着 `T` 帧做 self-attention。一共有 `N` 组，每组长度 `T`，复杂度约为：

$$
O(NT^2)
$$

再做空间注意力时，固定每一帧，在 `N` 个 patch 上做 self-attention。一共有 `T` 组，每组长度 `N`，复杂度约为：

$$
O(TN^2)
$$

两者相加得到：

$$
O(NT^2 + TN^2)
$$

这比联合注意力的 `O(T^2N^2)` 明显低，因为它把一个大平方问题拆成了两个小平方问题。

还是用上面的玩具例子 `T=4, N=9`：

- 联合注意力：`(TN)^2 = 36^2 = 1296`
- 分离注意力：`T·N^2 + N·T^2 = 4·81 + 9·16 = 468`

也就是说：

$$
1296 / 468 \approx 2.77
$$

约少 `2.8x`。这不是常数级优化，而是随着 `T`、`N` 增长会持续体现的结构性差异。

可以把三种机制先对照看一遍：

| 机制 | 先做什么 | 再做什么 | 核心收益 |
|---|---|---|---|
| 联合注意力 | 一次性处理全部 token | 无拆分 | 关系最完整 |
| 分离注意力 | 时间或空间之一 | 另一个维度 | 更省算力 |
| Window 注意力 | 局部窗口内注意 | shifted window / 下采样 | 更适合大视频 |

### TimeSformer、ViViT、Video Swin 分别怎么做

**TimeSformer** 的代表设计是 `Divided Space-Time`。它在同一层中显式把时间注意力和空间注意力分开做。优点是结构直观，变量容易控制，适合做时空拆分研究。它回答的问题是：既然视频有两个维度，那就不要强行一次性混在一起，而是拆开建模。

**ViViT** 是一个更广义的视频 Vision Transformer 框架。它给出了几种 factorization 方案，其中常见的 `Factorised Encoder` 可以理解为“先把每帧当成图像做空间编码，再把帧级表示拿去做时间编码”。这使它更像“帧集合 + 时序聚合”的设计，对预训练迁移更友好，也常被认为在数据量不大时更稳。

**Video Swin Transformer** 走的是第三条路线：不用全局注意力，也不只是把时间和空间拆开，而是直接把 token 限制在局部 **3D window** 中。window 可以理解为“视频块级的小邻域”，只允许邻域内部做注意力。若每个 window 大小固定为 `m` 个 token，则总复杂度约为：

$$
O(L \cdot m)
$$

当 `m` 固定时，它近似随总 token 数 `L` 线性增长。这就是为什么 Video Swin 更像工程模型：它把大规模视频的计算扩展性问题优先解决，再通过 shifted window 和层级结构补足跨窗口信息。

### shifted window 为什么有用

如果一直只在固定窗口内注意，模型会丢掉跨窗口长程依赖。**shifted window** 可以理解为“下一层把窗口边界平移一下”，这样原本分在不同窗口的 token，在下一层可能落到同一个窗口里，于是信息可以逐层跨窗传播。它不是一次性全局连接，而是通过多层局部传播近似得到更大的感受野。

因此，这三类方法不是谁“理论上更高级”，而是谁更符合约束：

- 想最大化时空关系完整性，选联合。
- 想在精度和成本之间折中，选分离。
- 想要长视频和可部署性，选 window。

## 代码实现

实现视频 Transformer 时，最容易出错的不是公式，而是张量维度。张量可以理解为“多维数组”；你以为自己在沿时间轴做注意力，实际可能在 patch 维度上做了重排，这类错误很常见。

一个标准输入通常是：

- `B×T×C×H×W`
- `B` 是 batch size
- `T` 是帧数
- `C` 是通道数
- `H, W` 是分辨率

之后流程通常是：

1. 对每帧做 patch embedding，把图像小块映射成向量。
2. 得到 `B×T×N×d` 的 token 表示。
3. 按架构选择展平、分组或窗口划分方式。
4. 做时空注意力。
5. 接分类头、检测头或检索头。

下面给一个可运行的最小 `python` 示例。它不依赖 PyTorch，只演示三种复杂度计算和 token 组织逻辑，目的是让新手把“视频进入 Transformer 后发生了什么”看清楚。

```python
from math import prod

def video_to_tokens_shape(batch, frames, channels, height, width, patch):
    assert height % patch == 0
    assert width % patch == 0
    n_h = height // patch
    n_w = width // patch
    num_patches_per_frame = n_h * n_w
    return (batch, frames, num_patches_per_frame)

def joint_attention_pairs(T, N):
    L = T * N
    return L * L

def divided_attention_pairs(T, N):
    temporal = N * (T * T)
    spatial = T * (N * N)
    return temporal + spatial

def window_attention_cost(T, N, m):
    L = T * N
    assert m > 0
    return L * m

# 玩具例子：4 帧，每帧 3x3 patch
B, T, C, H, W, P = 2, 4, 3, 48, 48, 16
shape = video_to_tokens_shape(B, T, C, H, W, P)
assert shape == (2, 4, 9)

T, N = 4, 9
joint = joint_attention_pairs(T, N)
divided = divided_attention_pairs(T, N)
window = window_attention_cost(T, N, m=8)

assert joint == 1296
assert divided == 468
assert joint / divided > 2.7
assert window == 288

print("token shape:", shape)
print("joint:", joint)
print("divided:", divided)
print("window:", window)
```

如果你用 PyTorch 风格理解，流程大致如下：

```python
# x: [B, T, C, H, W]
x = patch_embed(x)                # [B, T, N, d]

# 方案1：联合注意力
x = x.view(B, T * N, d)           # [B, L, d]
x = joint_attention(x)

# 方案2：分离时空注意力
x = temporal_attention(x, T, N)   # 在固定空间位置上沿时间建模
x = spatial_attention(x, T, N)    # 在固定帧内沿空间建模

# 方案3：Video Swin
x = window_partition_3d(x)        # 划分局部 3D window
x = window_attention(x)           # 窗口内注意力
x = shifted_window_attention(x)   # 平移窗口后再做一次

# 下游头
y = task_head(x)                  # 分类 / 检测 / 检索
```

这里最关键的是 `reshape / permute / view` 的顺序。比如时间注意力要求你把“同一空间位置 across T 帧”的 token 聚到一起；空间注意力要求你把“同一帧内部的 N 个 patch”聚到一起。如果维度重排错了，模型仍然能跑，但学到的是错误关系，这比直接报错更危险。

### 一个真实工程例子

假设你做短视频动作识别，输入是 `8` 帧、`224×224` 分辨率，patch size 取 `16`。那么每帧 patch 数为：

$$
N = (224 / 16)^2 = 14^2 = 196
$$

总 token 数为：

$$
L = 8 \times 196 = 1568
$$

如果直接上联合时空注意力，需要处理约：

$$
1568^2 = 2{,}458{,}624
$$

级别的 token 对。对单层来说这个量还可能能跑，但一旦堆多层、多头、较大 batch size，显存和训练速度压力会迅速上来。这时工程上更现实的选项通常不是“硬顶全局注意力”，而是优先考虑 TimeSformer 的分离注意力或 Video Swin 的局部窗口注意力。

## 工程权衡与常见坑

视频 Transformer 的设计很少有绝对最优，更多是约束驱动。算力、数据规模、视频长度、任务类型，都会改变最合适的结构。

第一个权衡是 **算力与建模完整性**。联合注意力最完整，但显存爆炸风险也最高。长视频、高清视频、较大 batch size 三者只要叠加两个，训练和推理压力就会明显上升。工程上更常见的顺序通常是：先降帧数，再调大 patch，再考虑换成分离式或 window 式，而不是一开始就追求最完整的全局交互。

第二个权衡是 **外观信息与动作信息**。有些数据集更像“看一张关键帧就能猜类别”，例如场景或器材线索很强的动作分类；有些数据集则更依赖时间顺序，例如 Something-Something v2，很多类别只有通过动作过程才能分开。只在 Kinetics 上表现好，不代表时间建模真的强，因为模型可能主要记住了外观偏置，而不是动作过程。

第三个权衡是 **局部建模与长程依赖**。Video Swin 的 window 注意力很省，但窗口太小会导致长程关系不够直接。它依赖 shifted window、层级合并和更深网络去逐步传播信息，因此它不是“天然拿到全局依赖”，而是“以更低成本近似构建更大感受野”。

下面这张表是常见坑的简化版：

| 常见坑 | 现象 | 规避方式 |
|---|---|---|
| 联合注意力用于长视频 | 显存和时间成本过高 | 降帧、增 patch、换分离式 / window 式 |
| 只看外观型数据集 | 误判模型时间建模能力 | 加测 Something-Something v2 |
| 认为分离式一定更差 | 实际效果可能更稳 | 用消融实验验证 |
| window 过小 | 长程依赖丢失 | 依赖 shifted window 和层级结构补偿 |

对初级工程师来说，最常见的误解有两个。

第一，认为“全局注意力一定最好”。这在理论上不成立，在工程上更不成立。因为训练预算有限时，过高的复杂度会迫使你把 batch size、分辨率、帧数都降下来，最后整体效果未必优于一个更高效的结构。

第二，认为“分离式就是在偷懒”。实际上分离式是把难问题拆解。它先让模型学“同一位置随时间如何变化”，再学“同一帧里不同位置如何组合”，这种归纳偏置在很多视频任务里是有帮助的，不是简单降配。

如果你做的是 `8` 帧短视频动作识别，显存有限，又希望尽快得到稳定基线，直接选联合注意力通常不划算。更稳妥的做法是先用 Video Swin；如果你需要分析时空拆分效果，再用 TimeSformer 做对照。这个决策不是审美问题，而是训练预算和实验目的决定的。

## 替代方案与适用边界

三种代表架构可以这样理解：

**TimeSformer** 更适合“显式研究时空分解”。它把时间和空间拆得很清楚，所以做论文、做消融、做机制分析时更方便控制变量。它的边界在于：结构虽清晰，但不是最省算力的工程解。

**ViViT** 更适合“把视频当作图像编码的自然延伸”。它提供多种 factorization 方式，特别是 `Factorised Encoder`，常被用来先继承图像 Transformer 的空间建模能力，再补上时间聚合。这种思路在数据量不大、希望复用预训练时较实用。它的边界在于：设计分支较多，具体选型依赖实验验证，不能简单说“ViViT 就是某一种固定结构”。

**Video Swin** 更适合“在现实预算里把视频 Transformer 跑起来”。局部 3D window、shifted window 和层级结构让它在速度、显存和效果之间更均衡，尤其适合长视频或部署场景。它的边界在于：长程依赖需要靠多层传播，不如全局注意力那样直接。

可以用一张适用性表做收束：

| 方法 | 优点 | 局限 | 典型选择理由 |
|---|---|---|---|
| TimeSformer | 时空分解清晰，便于分析 | 不是最省算力 | 做机制研究和消融 |
| ViViT | 结构灵活，适合预训练迁移 | 设计分支较多 | 数据不大、想稳一点 |
| Video Swin | 局部建模强，部署友好 | 需要靠层级结构补长程信息 | 工程落地和效率优先 |

再给三个实际判断例子。

一个是小数据训练。你手里只有少量标注视频，希望尽量复用已有图像预训练权重，这时 ViViT 的 `Factorised Encoder` 往往更容易稳定训练，因为它先把空间编码问题处理好，再把时间建模叠上去。

一个是线上服务。你需要低延迟、可扩展、显存可控，这时 Video Swin 往往更现实。原因不是它一定在所有指标上最好，而是它更符合工程约束。

一个是论文实验。你想比较“先时间后空间”和“先空间后时间”对结果的影响，或者想验证是否真的需要时间建模，这时 TimeSformer 更合适，因为结构更透明，实验解释更直接。

所以最终的选择标准不是“哪篇论文更有名”，而是四个变量：

- 数据规模
- 算力预算
- 视频长度
- 是否需要可解释的时空拆分

只要这四个变量没说清楚，讨论“哪个视频 Transformer 最好”基本没有意义。
