## 核心结论

DeepSeek V3 的代表性设计不是“把所有参数都算一遍”，而是“把总参数做大，但每次只激活其中一小部分”。这里的 **MoE（Mixture of Experts，专家混合）** 可以先理解成“很多子网络分工协作，但每个输入只调用少数几个子网络”。

它的核心数字可以先记住三组：

| 指标 | DeepSeek V3 | 72B 稠密模型 | 405B 稠密模型 |
|---|---:|---:|---:|
| 总参数量 | 671B | 72B | 405B |
| 单 token 激活参数 | 约 37B | 72B | 405B |
| 单 token 计算量 | 约 250 GFLOPS | 约 394 GFLOPS | 约 2448 GFLOPS |

这里的 **稠密模型（Dense Model）**，白话说就是“每一层的大部分参数都会参与每次计算”；而 **稀疏模型（Sparse Model）**，白话说就是“参数很多，但一次只用其中一部分”。

DeepSeek V3 的关键收益是：容量接近超大模型，但单次计算更接近中等规模模型。这个关系可以直接写成：

$$
\text{激活比例}=\frac{37B}{671B}\approx 5.5\%
$$

也就是说，模型保留了 671B 的总容量，但单个 token 前向时只动用约 5.5% 的参数。

玩具例子可以这样理解。假设一所学校有 257 位老师，其中 256 位是分科老师，1 位是所有学生都能上的公共老师。每个学生来问问题时，不是 257 位老师一起回答，而是系统先挑出最适合的 8 位分科老师，再加上公共老师参与。这就形成了“老师很多，但每次实际开工的人不多”的结构。

如果用一句话概括：DeepSeek V3 把“大参数容量”和“可接受的单次计算成本”拆开处理，靠的是大规模 MoE 稀疏路由，而不是单纯堆稠密参数。

## 问题定义与边界

问题本身很直接：如果继续按传统稠密路线扩模型，参数越大，每个 token 要算的东西也越多，训练和推理成本会迅速上升。MoE 想解决的是另一类问题：能不能让模型“存很多知识”，但每次“只调用和当前 token 相关的一部分知识”。

这里的 **路由（Routing）**，白话说就是“决定当前 token 该交给哪些专家处理的分配机制”。DeepSeek V3 的边界大致可以概括为：

| 边界类型 | 关键约束 |
|---|---|
| 训练边界 | 256 个路由专家 + 1 个共享专家 |
| 计算边界 | 每个 token 最多路由到 8 个专家 |
| 激活边界 | 每个 token 约激活 37B 参数 |
| 部署边界 | 一个 token 的专家执行最多跨 4 个节点 |
| 系统边界 | 必须控制 all-to-all 通信开销与负载均衡 |

为什么这些边界重要？因为 MoE 并不是“只要专家多就一定更强”。它有几个典型风险：

1. **路由崩塌**：路由器总把 token 分给少数几个专家，其他专家几乎不学东西。
2. **专家偏载**：部分专家太忙，部分专家太闲，导致吞吐下降。
3. **通信爆炸**：token 被分到很多机器上的专家，网络传输成本超过计算本身。
4. **部署复杂**：模型参数虽然稀疏激活，但系统调度、缓存、节点拓扑都更难做。

新手可以把它想成一个服务大厅。不是让 257 位服务员同时接待每位顾客，而是每位顾客只找最匹配的 8 位服务员。但前提是必须保证三件事：不能所有顾客都挤向同一组服务员，不能让顾客在太多楼层之间来回跑，也不能让大量服务员长期闲置。

所以，DeepSeek V3 讨论的不是“MoE 能不能工作”，而是“超大 MoE 在真实分布式系统里能不能稳定地工作”。

## 核心机制与推导

DeepSeek V3 的核心不是“专家多”，而是“如何让专家被稳定且均匀地用起来”。

这里的 **专家（Expert）**，白话说就是“模型里专门处理某类模式的子网络”；**共享专家（Shared Expert）** 可以理解成“所有 token 都能访问的公共通道”；**门控或 gating** 则是“最后决定每个专家贡献多少权重的打分机制”。

一个常见的抽象写法是：

$$
\text{score}_i = \text{router}_i + b_i
$$

其中：

- $\text{router}_i$ 是路由器给第 $i$ 个专家的原始分数
- $b_i$ 是专家偏置项，可以理解成“为负载均衡临时加的调节分”

然后从所有专家里选出 top-$K_r$，这里 **$K_r=8$**。也就是每个 token 最多只进 8 个路由专家。

重点在于：偏置 $b_i$ 主要用于“决定谁进入候选名单”，而最终 gating 仍然依赖原始分数或基于原始分数的归一化结果。这样做的目的，是避免均衡机制过强，反过来破坏模型真正的语义选择能力。

一个简化的动态偏置更新逻辑可以写成：

$$
b_i \leftarrow
\begin{cases}
b_i - \gamma, & \text{若专家 } i \text{ 当前负载过高}\\
b_i + \gamma, & \text{若专家 } i \text{ 当前负载过低}
\end{cases}
$$

其中 $\gamma$ 是很小的调整步长。它不是直接改模型权重，而是在路由选择阶段轻微推一把，把 token 往“还没吃饱”的专家那里引导。

玩具例子如下。假设有 4 个专家，要从中选 2 个：

| 专家 | 原始分数 $\text{router}_i$ | 偏置 $b_i$ | 选择分数 $\text{score}_i$ | 是否入选 |
|---|---:|---:|---:|---|
| E1 | 0.92 | -0.10 | 0.82 | 是 |
| E2 | 0.88 | -0.15 | 0.73 | 否 |
| E3 | 0.81 | +0.08 | 0.89 | 是 |
| E4 | 0.75 | +0.05 | 0.80 | 否 |

如果没有偏置，E1 和 E2 可能长期占优势；加入偏置后，E3 有机会被选中，从而获得训练样本，不会被永久边缘化。

再看负载变化：

| 专家 | 调整前负载占比 | 偏置更新 | 调整后趋势 |
|---|---:|---:|---|
| E1 | 40% | $-\gamma$ | 负载下降 |
| E2 | 32% | $-\gamma$ | 略降 |
| E3 | 18% | $+\gamma$ | 上升 |
| E4 | 10% | $+\gamma$ | 上升 |

这套机制的推导逻辑是：

1. 总参数大，不代表每次都要全算。
2. 既然只算少数专家，就必须先做专家选择。
3. 只靠原始分数选专家，容易让头部专家越来越忙。
4. 所以在“入围阶段”加入动态偏置，给冷门专家更多上场机会。
5. 但最后真正的贡献权重仍尽量保留原始语义信号，避免均衡机制压过模型表达能力。

真实工程例子中，这意味着一个 batch 的 token 会先经过路由器，生成专家分配，再进行 all-to-all 发送，把 token 发往对应专家所在设备。专家算完后再汇总回原始顺序。MoE 的难点并不在公式本身，而在“这个流程能否在大规模 GPU 集群上高吞吐、低抖动地跑起来”。

## 代码实现

下面用一个可运行的 Python 玩具版本演示“路由打分 + 动态 bias + top-k 选择”的核心过程。它不是 DeepSeek V3 的完整实现，但能准确表达关键机制。

```python
from math import isclose

def route_one_token(router_scores, biases, k=2):
    """
    router_scores: 路由器原始分数
    biases: 用于负载均衡的动态偏置
    返回:
      selected: 入选专家下标
      final_weights: 基于原始分数做归一化后的权重
    """
    assert len(router_scores) == len(biases)
    assert 1 <= k <= len(router_scores)

    # 1. 用 原始分数 + bias 决定谁进入 top-k
    selection_scores = [s + b for s, b in zip(router_scores, biases)]
    ranked = sorted(range(len(selection_scores)),
                    key=lambda i: selection_scores[i],
                    reverse=True)
    selected = ranked[:k]

    # 2. 最终 gating 仍用原始分数，而不是带 bias 的分数
    raw_selected = [max(router_scores[i], 0.0) for i in selected]
    total = sum(raw_selected)
    if total == 0:
        final_weights = [1.0 / k] * k
    else:
        final_weights = [v / total for v in raw_selected]

    return selected, final_weights


def update_biases(loads, avg_load, biases, gamma=0.05):
    """
    loads: 各专家当前负载
    avg_load: 期望平均负载
    负载高就减 bias，负载低就加 bias
    """
    new_biases = []
    for load, b in zip(loads, biases):
        if load > avg_load:
            new_biases.append(b - gamma)
        elif load < avg_load:
            new_biases.append(b + gamma)
        else:
            new_biases.append(b)
    return new_biases


# 玩具例子
router_scores = [0.92, 0.88, 0.81, 0.75]
biases = [-0.10, -0.15, 0.08, 0.05]

selected, weights = route_one_token(router_scores, biases, k=2)

# 由于加入 bias，E0 和 E2 会入选
assert selected == [2, 0] or selected == [0, 2]
assert isclose(sum(weights), 1.0, rel_tol=1e-9)

loads = [40, 32, 18, 10]
avg_load = sum(loads) / len(loads)
new_biases = update_biases(loads, avg_load, biases, gamma=0.05)

# 负载高的前两个专家 bias 应该下降，负载低的后两个应上升
assert new_biases[0] < biases[0]
assert new_biases[1] < biases[1]
assert new_biases[2] > biases[2]
assert new_biases[3] > biases[3]
```

这段代码对应的流程可以概括成：

1. 输入 token 表示。
2. 路由器输出每个专家的原始分数。
3. 给每个专家叠加动态 bias，用于 top-8 选择。
4. 选中的专家并行计算。
5. 使用原始分数归一化后的权重聚合结果。
6. 根据本轮负载更新 bias，为下一轮做准备。

如果写成伪代码，就是：

```python
# input token -> router -> top-8 experts -> experts run -> aggregate

token_hidden = encoder(token)

router_scores = router(token_hidden)           # 原始路由分数
selection_scores = router_scores + bias        # 只在入选阶段加 bias
topk_experts = top_k(selection_scores, k=8)    # 选择 8 个专家

expert_outputs = []
for expert_id in topk_experts:
    out = experts[expert_id](token_hidden)     # 并行专家计算
    expert_outputs.append(out)

gate = normalize(router_scores[topk_experts])  # 最终仍用原始分数
merged = weighted_sum(expert_outputs, gate)

bias = update_bias_by_load(current_load, bias) # 用本轮负载更新 bias
```

真实工程里还会多出几层复杂度：

- token 不是单个，而是整个 batch 一起路由。
- 专家可能分布在不同 GPU 或不同节点上。
- 需要把 token 重新打包，再通过 all-to-all 发到目标设备。
- 共享专家通常始终参与，保证基础表达能力不会全靠稀疏专家承担。

## 工程权衡与常见坑

MoE 真正难的地方在工程，而不是论文公式。DeepSeek V3 的一个关键信号是：它不是“小实验室级别的稀疏模型”，而是在大规模 GPU 集群上完成的可运行系统。

公开材料里常被提到的训练规模大致如下：

| 阶段 | GPU 小时 |
|---|---:|
| 预训练 | 2.664M |
| 上下文扩展 | 0.119M |
| 后训练 | 0.005M |
| 合计 | 2.788M |

如果按 2,048 块 H800 组织训练，这说明它依赖的是“稀疏架构 + 精细分布式调度”的组合，而不是只靠更多卡暴力堆出来。

这里最常见的坑有四类。

第一类是路由崩塌。少数专家持续收到大量 token，会形成“强者愈强”。表面上看准确率可能没立刻出问题，但长期会让大部分专家训练不足，等于白白存了一堆没学会东西的参数。

第二类是负载不均。MoE 的吞吐取决于最忙的那批专家，而不是平均专家。如果 256 个专家里只有十几个特别忙，整个 step 的尾延迟会被这些热点专家拖住。

第三类是通信成本过高。这里的 **all-to-all**，白话说就是“每张卡都可能要向很多别的卡发送 token 数据，同时也接收别人发来的数据”。对稠密模型来说，主要瓶颈常在矩阵乘法；对大规模 MoE 来说，通信本身可能成为主瓶颈。

第四类是节点拓扑约束。一个 token 如果被分给太多机器上的专家，网络代价会迅速上升。DeepSeek V3 类似的做法强调“一个 token 最多只跨有限节点”，可以理解成：不是让 token 在整个集群里乱飞，而是把它的活动范围限制在较小子图中。

真实工程例子可以这样理解。假设一个 token 选中了 8 个专家，如果这 8 个专家分布在 8 台不同服务器上，那么你需要 8 次远程发送和若干次汇总；如果它们被规划在最多 4 个节点内，网络扇出就更可控。前者像在全城跑 8 个窗口盖章，后者像集中在同一政务大厅的几个柜台之间流转。

一个简化对比表如下：

| 工程问题 | 典型后果 | 缓解手段 |
|---|---|---|
| 路由崩塌 | 大量专家学不到东西 | 动态 bias、辅助均衡策略 |
| 专家偏载 | step 尾延迟变长 | 容量约束、负载监控、热专家打散 |
| 通信爆炸 | GPU 空等网络 | 限制跨节点范围、优化 all-to-all |
| 部署复杂 | 调试和扩容困难 | 固定拓扑规划、专家分组、共享专家保底 |

所以，MoE 的本质权衡是：你用更复杂的系统工程，换来更高的参数容量和更低的单 token 计算成本。DeepSeek V3 说明这条路是可行的，但它不是“便宜的大模型”，而是“把成本结构重新分配的大模型”。

## 替代方案与适用边界

MoE 不是所有场景都优于稠密模型。要判断它是否值得，重点看三件事：算力形态、通信条件、延迟目标。

先看一个对比表：

| 方案 | 总参数 | 单 token 计算 | 通信复杂度 | 适合场景 |
|---|---:|---:|---|---|
| 小型稠密模型 | 小到中 | 低 | 低 | 单机部署、低延迟服务 |
| 大型稠密模型 | 很大 | 很高 | 中 | 极高质量、但算力充足 |
| 大规模 MoE | 很大 | 中等偏低 | 高 | 追求超大容量且可承受分布式通信 |

如果团队只有少量 GPU、网络一般、服务还要求很低延迟，那么小型或中型稠密模型通常更现实。原因很简单：稠密模型实现直接、部署稳定、推理路径固定，不需要处理复杂的专家路由和跨机调度。

但如果目标是“在可接受的单 token 计算量下，把模型容量继续推高”，MoE 就会更有吸引力。特别是在代码、数学、复杂推理这类任务中，模型往往受益于更大的容量上限，而不一定要求每次都激活全部参数。

可以用一个交通类比收束这个边界。小型稠密模型像自行车，启动简单、维护轻、短途灵活。大规模 MoE 像高速列车，基础设施复杂，调度要求高，但一旦轨道和站点体系搭好，单位运力会明显更强。问题不在于谁“绝对更先进”，而在于你的路网是否支持它。

所以，什么时候选稠密模型，什么时候选 MoE，可以粗略按下面判断：

| 条件 | 更适合稠密 | 更适合 MoE |
|---|---|---|
| 节点数少 | 是 | 否 |
| 网络带宽一般 | 是 | 否 |
| 延迟预算极紧 | 是 | 否 |
| 追求超大参数容量 | 否 | 是 |
| 团队擅长分布式系统 | 一般 | 是 |
| 训练预算能覆盖复杂调度 | 一般 | 是 |

结论不是“MoE 替代稠密”，而是“MoE 在超大规模阶段提供了另一条扩展路线”。DeepSeek V3 的价值，就在于把这条路线从理论设计推进到了可落地的工程系统。
