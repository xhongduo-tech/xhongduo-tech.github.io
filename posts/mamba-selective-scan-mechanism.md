## 核心结论

Mamba 的选择性扫描机制，核心不是“把状态空间模型做得更复杂”，而是把原本固定不变的更新参数改成**输入依赖**。输入依赖的准确含义是：当前 token 到来时，模型先根据这个 token 生成本步的更新规则，再用这套规则更新状态；不是整条序列共用一组固定参数。

传统 S4 一类状态空间模型，常把离散化后的更新参数视为全局共享；Mamba 则让每个位置的 $\Delta_k,B_k,C_k$ 都由当前输入 $x_k$ 生成。对单步更新，最重要的公式可以写成：

$$
\bar A_k = \exp(\Delta_k A), \qquad \bar B_k = \Delta_k B_k
$$

$$
h_k=\bar A_k h_{k-1}+\bar B_k x_k,\qquad y_k=C_k h_k
$$

这带来两个直接结果：

1. 模型可以按内容决定“记住多久、写入多少、怎么读出”。
2. 模型仍然能通过并行 scan 保持高吞吐，不必退回到完全串行的 RNN 式执行。

给新手的直观理解是：序列里的每个 token 都拿着三把旋钮。

| 旋钮 | 数学对象 | 作用 | 白话解释 |
|---|---|---|---|
| 记忆节奏 | $\Delta_k$ | 决定状态衰减速度 | 现在这条信息该慢慢保留，还是尽快忘掉 |
| 注入力度 | $B_k$ | 决定当前输入写入多少 | 当前 token 值不值得写进“记忆” |
| 读出视角 | $C_k$ | 决定怎样从状态生成输出 | 当前任务更关心状态里的哪一部分 |

因此，Mamba 的关键价值不是“完全替代注意力”，而是用更低的状态更新成本，在长序列里实现动态过滤无关信息。它解决的不是“能不能记住历史”这一个问题，而是“能不能**有选择地**记住历史，同时还保持线性时间复杂度”。

## 问题定义与边界

问题可以精确定义为：

在长序列建模中，希望系统同时满足两件事：

1. 能记住远距离依赖，而不是窗口一过就丢。
2. 不是所有输入都同等重要，模型需要按需保留信息。

传统固定 SSM 的边界在于：它虽然能高效处理长序列，但**所有 token 共用同一套状态更新规则**。这意味着不管输入是“关键信号”还是“噪声”，系统的遗忘和写入节奏都偏固定。

把这个问题说得更工程一点，就是“过滤器不够自适应”。

- 固定 SSM：每个 token 都通过同一只滤波器，滤波器参数基本不变。
- Mamba：每个 token 先决定“这一步应该怎样滤波”，再进入状态更新。

这就是“选择性”的边界：它不是让模型随意跳过递推，也不是像注意力那样显式两两比较所有 token，而是让**状态更新系数对输入敏感**。

从执行流程看，边界也很明确：

1. token 表示 $x_k$ 输入投影层
2. 投影层预测 $\Delta_k,B_k,C_k$
3. 根据 $\Delta_k$ 和连续时间参数 $A$ 做离散化
4. 执行状态更新 $h_k$
5. 从状态读出输出 $y_k$

可写成一个简化步骤流：

| 步骤 | 输入 | 输出 | 说明 |
|---|---|---|---|
| 1 | $x_k$ | 隐层投影 | 为选择性参数做准备 |
| 2 | 隐层投影 | $\Delta_k,B_k,C_k$ | 当前 token 决定本次更新方式 |
| 3 | $\Delta_k,A,B_k$ | $\bar A_k,\bar B_k$ | 连续参数转成离散更新 |
| 4 | $h_{k-1},x_k,\bar A_k,\bar B_k$ | $h_k$ | 完成记忆衰减与新信息注入 |
| 5 | $h_k,C_k$ | $y_k$ | 输出当前结果 |

这里有一个硬约束：如果每个 $h_k$ 都老老实实串行算，吞吐会被序列长度拖垮。Mamba 的工程目标不是只做“动态状态”，而是做“**可并行的动态状态**”。因此它必须能被改写成适合 parallel scan 的形式，而不是退化成普通 RNN。

玩具例子可以说明这个边界。

假设模型处理一句话：

- token1: “订单”
- token2: “编号”
- token3: “是”
- token4: “A1289”
- token5: “天气”

如果当前任务是“提取订单号”，那么“天气”应当很快被遗忘；“A1289”则应被强写入并保留一段时间。固定 SSM 对这两类 token 的处理规则差异有限，Mamba 则允许它们生成不同的 $\Delta,B,C$，从而表现出不同的记忆行为。

再看一个反例，有助于理解边界：如果任务只是短句情感分类，输入只有几十个 token，且每个 token 都大致重要，那么选择性扫描的优势可能并不明显。也就是说，Mamba 不是“任何序列任务都天然更强”，而是当“序列长且信息密度不均匀”时更有价值。

## 核心机制与推导

先看连续到离散的核心关系。状态空间模型通常从连续系统出发：

$$
\frac{dh(t)}{dt}=Ah(t)+Bx(t),\qquad y(t)=Ch(t)
$$

这里：

- $h(t)$ 是隐藏状态，表示系统在时刻 $t$ 的内部记忆。
- $x(t)$ 是输入。
- $y(t)$ 是输出。
- $A$ 决定系统自己怎样随时间演化。
- $B$ 决定输入怎样写入状态。
- $C$ 决定怎样从状态里读出输出。

离散化后，在第 $k$ 个位置得到：

$$
h_k=\bar A_k h_{k-1}+\bar B_k x_k,\qquad y_k=C_k h_k
$$

其中 Mamba 的关键变化是：

$$
\bar A_k=\exp(\Delta_k A),\qquad \bar B_k=\Delta_k B_k
$$

这里有三个输入依赖量：

- $\Delta_k$：离散步长，白话说是“这一拍走多快”。
- $B_k$：输入投影，白话说是“这一拍往状态里写多少”。
- $C_k$：输出投影，白话说是“这一拍从状态里读什么”。

于是每个 token 的状态更新都不同。它不是统一矩阵乘法的一刀切，而是位置相关的动态系统。

### 为什么 $\Delta_k$ 会影响“遗忘速度”

这是很多新手第一次看公式时最容易卡住的地方。

若 $A<0$，则：

$$
\bar A_k = \exp(\Delta_k A)
$$

因为指数函数满足：

$$
0 < \exp(\Delta_k A) < 1
\quad \text{当} \quad A<0,\ \Delta_k>0
$$

所以 $\bar A_k$ 就像“旧状态保留比例”。

- 当 $\Delta_k$ 较小，$\exp(\Delta_k A)$ 更接近 1，旧状态保留更多。
- 当 $\Delta_k$ 较大，$\exp(\Delta_k A)$ 更接近 0，旧状态衰减更快。

这就是为什么 $\Delta_k$ 常被理解为“记忆节奏”或“遗忘速度控制器”。

### 玩具例子：一维状态怎么“记忆”和“遗忘”

设一维状态，取：

$$
A=-1,\quad x_k=1,\quad h_{k-1}=0.5
$$

若当前 token 预测得到：

$$
\Delta_k=0.1,\quad B_k=0.5,\quad C_k=1
$$

则

$$
\bar A_k=\exp(\Delta_k A)=\exp(-0.1)\approx 0.9048
$$

$$
\bar B_k=\Delta_k B_k=0.1\times 0.5=0.05
$$

所以状态更新为：

$$
h_k=0.9048\times 0.5+0.05\times 1\approx 0.5024
$$

输出为：

$$
y_k=C_k h_k=1\times 0.5024=0.5024
$$

这意味着：旧状态被保留了大约 90.48%，新输入只轻微写入。也就是“这个 token 没有强烈打断已有记忆”。

如果下一 token 让步长变大：

$$
\Delta_{k+1}=1
$$

则

$$
\bar A_{k+1}=\exp(-1)\approx 0.3679
$$

此时旧状态只保留约 36.79%，遗忘明显加快。也就是说，$\Delta$ 越大，系统越倾向于快速刷新状态。

这就是新手最该抓住的一点：**Mamba 的选择性，不是先决定“算不算”，而是决定“衰减多快、注入多少、输出怎么看”**。

### 多走一步：连续两步更新

继续上面的数值，假设第二步仍有 $x_{k+1}=1$、$B_{k+1}=0.5$、$C_{k+1}=1$，但 $\Delta_{k+1}=1$，那么：

$$
\bar B_{k+1}=1\times 0.5=0.5
$$

$$
h_{k+1}=0.3679\times 0.5024+0.5\times 1\approx 0.6848
$$

这里能看到两个动作同时发生：

1. 旧状态保留得更少，因为 $\bar A_{k+1}$ 变小了。
2. 新输入写入得更多，因为 $\bar B_{k+1}$ 变大了。

所以一个更准确的表述是：$\Delta_k$ 不只改变“忘多少”，还会通过 $\bar B_k=\Delta_k B_k$ 改变“写多少”。它同时影响记忆保留和输入注入。

### 三部曲：记忆、注入、输出

可以把单步更新拆成三步：

1. 记忆：$\bar A_k h_{k-1}$
2. 注入：$\bar B_k x_k$
3. 输出：$C_k h_k$

时间线理解如下：

| 时刻 | 参数主导 | 作用 |
|---|---|---|
| 更新前 | $\Delta_k$ | 决定旧状态衰减速度 |
| 更新中 | $B_k$ | 决定当前输入写入强度 |
| 更新后 | $C_k$ | 决定从新状态读出什么 |

真实工程例子更容易看出价值。比如日志异常检测中，序列可能是：

- “service start”
- “load config”
- “db timeout”
- “retry”
- “retry”
- “fatal”

这里“db timeout”和“fatal”比“load config”更重要。理想情况下，模型应对关键异常 token 给出更强写入、更慢遗忘，或在读出时给予更高权重；对重复背景 token 则快速衰减。Mamba 的输入依赖 $\Delta,B,C$ 就是在做这件事。

这和固定 SSM 的差别可以总结为：

| 维度 | 固定 SSM | Mamba |
|---|---|---|
| $\Delta$ | 通常固定或全局共享 | 每个 token 动态生成 |
| $B,C$ | 多数情况下固定 | 随输入变化 |
| 对噪声处理 | 依赖固定动力学，适应性弱 | 可按输入动态压低写入或加快遗忘 |
| 长序列表现 | 高效，但选择性有限 | 高效且更具内容感知能力 |

### 为什么这不等于注意力

容易混淆的一点是：Mamba 也会“按内容处理”，那它是不是换一种形式做注意力？

不是。两者的核心机制不同：

| 机制 | Transformer 注意力 | Mamba 选择性扫描 |
|---|---|---|
| 核心操作 | token 两两交互 | token 驱动状态递推 |
| 信息访问 | 显式看历史 token | 隐式读写压缩状态 |
| 复杂度特征 | 常见实现对序列长度较敏感 | 目标是线性时间 |
| 选择性来源 | 注意力权重 | 输入依赖的状态更新参数 |

简化说，注意力是“我现在去看谁”，Mamba 是“我现在把什么写进状态，以及之后怎么保留它”。

## 代码实现

理论公式本身还是递推式：

$$
h_k=\bar A_k h_{k-1}+\bar B_k x_k
$$

直接实现会得到一个典型串行循环。下面给出一段**可直接运行**的 Python 示例，它只依赖标准库：

```python
import math


def selective_scan_scalar(xs, deltas, Bs, C=1.0, A=-1.0, h0=0.0):
    if not (len(xs) == len(deltas) == len(Bs)):
        raise ValueError("xs, deltas, Bs must have the same length")

    h = h0
    ys = []
    hs = []

    for x, delta, B in zip(xs, deltas, Bs):
        if delta <= 0:
            raise ValueError("delta must be positive in this toy example")

        A_bar = math.exp(delta * A)
        B_bar = delta * B
        h = A_bar * h + B_bar * x
        y = C * h

        hs.append(h)
        ys.append(y)

    return hs, ys


def main():
    xs = [1.0, 1.0]
    deltas = [0.1, 1.0]
    Bs = [0.5, 0.5]

    hs, ys = selective_scan_scalar(xs, deltas, Bs, C=1.0, A=-1.0, h0=0.5)

    print("states:", [round(v, 4) for v in hs])
    print("outputs:", [round(v, 4) for v in ys])

    # 与文中的手算结果对齐
    assert abs(hs[0] - 0.5024) < 1e-3
    assert abs(hs[1] - 0.6848) < 1e-3


if __name__ == "__main__":
    main()
```

运行结果应接近：

```text
states: [0.5024, 0.6848]
outputs: [0.5024, 0.6848]
```

这段代码能帮助理解，但它有一个明显问题：**严格串行**。因为第 $k$ 步依赖第 $k-1$ 步的状态，序列一长，吞吐就受限。

### 把递推改写成“可扫描”的形式

为了说明 Mamba 为什么还能并行，先把单步更新写成一种更适合组合的形式：

$$
h_k = a_k h_{k-1} + b_k
$$

其中：

$$
a_k=\bar A_k,\qquad b_k=\bar B_k x_k
$$

现在看相邻两步：

$$
h_1 = a_1 h_0 + b_1
$$

$$
h_2 = a_2 h_1 + b_2 = a_2(a_1 h_0 + b_1)+b_2
$$

展开后得到：

$$
h_2 = (a_2 a_1) h_0 + (a_2 b_1 + b_2)
$$

这说明两步递推可以合并成一个“复合变换”：

$$
(a_2,b_2)\circ(a_1,b_1) = (a_2 a_1,\ a_2 b_1 + b_2)
$$

关键在于，这个组合运算满足结合律，因此可以做 parallel scan。也就是说，虽然原始递推看起来像串行，但“状态转移对”的组合可以树形归约。

这正是 selective scan 能成立的数学基础。

### 简化伪代码

```python
# 输入: X [N, D]
# 输出: Y [N, D]

def mamba_block(X):
    # 1. 预测输入依赖参数
    Delta = linear_delta(X)
    B = linear_B(X)
    C = linear_C(X)

    # 2. 离散化
    A_bar = exp(Delta * A)   # 每个位置都不同
    B_bar = Delta * B

    # 3. 形成递推对
    a = A_bar
    b = B_bar * X

    # 4. 并行 selective scan
    H = parallel_scan(a, b)

    # 5. 读出
    Y = C * H
    return Y
```

这里的 `parallel_scan(a, b)` 不是普通 for 循环，而是前缀扫描。前缀扫描可以理解为：

- 先把相邻元素两两合并
- 再把更大的块继续合并
- 最后把每个位置之前的累计结果并行恢复出来

因此它比“从左到右一个个算”更适合 GPU。

### 一个最小可运行的“扫描对组合”示例

下面这段代码不是真正并行实现，但它展示了“组合规则”本身是怎么工作的：

```python
def compose(pair2, pair1):
    a2, b2 = pair2
    a1, b1 = pair1
    return a2 * a1, a2 * b1 + b2


def scan_pairs(pairs):
    out = []
    acc = (1.0, 0.0)  # 恒等变换: h -> 1*h + 0
    for pair in pairs:
        acc = compose(pair, acc)
        out.append(acc)
    return out


def main():
    pairs = [(0.9, 0.1), (0.4, 0.5), (0.8, -0.2)]
    prefix = scan_pairs(pairs)
    print(prefix)


if __name__ == "__main__":
    main()
```

输出的每个 `(a,b)` 都表示“从初始状态 $h_0$ 出发，到当前位置的整体变换”：

$$
h_k = a^{(1:k)} h_0 + b^{(1:k)}
$$

这一步是理解 selective scan 的关键桥梁：Mamba 并不是魔法般把串行递推消掉，而是把递推重写为可结合的前缀运算。

### 为什么高性能实现还需要 kernel fusion

真正高性能实现还会做 **kernel fusion**。kernel fusion 的意思是：把多个原本分开执行、分开读写显存的步骤，拼成一个更大的内核，让数据尽量留在片上高速存储里连续算完。

原因很直接：如果每一步都把中间状态写回 HBM，再从 HBM 读回来，内存带宽会成为瓶颈。HBM 是高带宽显存，但相对寄存器和 SRAM 仍然慢，尤其序列很长时，中间状态规模是 $O(N)$。

因此理想流程是：

1. 从输入投影出 $\Delta,B,C$
2. 立刻在片上完成 $\bar A,\bar B$ 计算
3. 直接进入 scan
4. 扫描结果继续用于输出投影
5. 尽量少把中间张量落到 HBM

可以用 CUDA-like 逻辑粗略表示：

```cpp
// 简化示意，不是可直接编译的 CUDA
kernel selective_scan_fused(X, A, out):
    shared tile_x, tile_delta, tile_B, tile_C
    shared tile_Abar, tile_Bbar, tile_state

    load X tile into shared memory

    // 1. 线性投影生成 Delta/B/C
    tile_delta = proj_delta(tile_x)
    tile_B = proj_B(tile_x)
    tile_C = proj_C(tile_x)

    // 2. 离散化
    tile_Abar = exp(tile_delta * A)
    tile_Bbar = tile_delta * tile_B

    // 3. 块内 scan，避免中间状态频繁写回 HBM
    tile_state = block_prefix_scan(tile_Abar, tile_Bbar, tile_x)

    // 4. 读出输出
    out = tile_C * tile_state
```

“避免中间结果写回 HBM”不是实现细节，而是性能能否成立的关键。因为 Mamba 的贡献之一，正是把动态状态更新和硬件友好的 scan 结合起来；如果只有输入依赖参数、没有高效 scan，吞吐优势就会被抵消。

真实工程例子里，这种优势在超长上下文中更明显。比如做 DNA 序列分析、长日志聚合或多小时语音转写，长度可能远大于普通注意力的舒适区。此时如果仍采用显式保存大量中间状态的串行递推，延迟和显存都很难接受；而 fused selective scan 能显著降低内存读写压力。

## 工程权衡与常见坑

Mamba 的难点不只在理论，也在内核边界条件、数值稳定性和训练行为。

一个实际问题是：当实现里启用 `delta_softplus` 或 `delta_bias` 这类稳定化处理后，$\exp(\Delta A)$ 的默认边界值可能不再等于实现所期望的“恒等传递”状态。这样在反向扫描或块边界拼接时，就可能把“本该停止传播的尾部状态”继续传下去，造成残留污染。

新手可以这样理解：扫到序列结尾后，后面的无效位置本来应该什么都不做；但如果边界填充值不对，无效位置就像还在偷偷更新状态，结果后续块拼接会被污染。

问题和处理方式可以对照看：

| 问题 | 现象 | 根因 | 常见处理 |
|---|---|---|---|
| 边界残留 | 序列尾部出现异常状态传播 | 无效位置未设置成恒等变换 | 在 kernel 中补边界安全值 |
| 反向不稳定 | 梯度在块连接处异常 | 前后向 scan 边界不一致 | 显式统一前后向边界规则 |
| 数值爆炸/衰减 | 状态过大或过小 | $\Delta$ 取值范围失控，或 $A$ 参数化不稳 | 对 $\Delta$ 做约束或稳定化 |
| 性能不达预期 | 速度接近串行 | 中间状态频繁回写 HBM | 强化 fusion，减少中间张量落盘 |
| 训练后选择性很弱 | 看起来“什么都记” | 数据没有提供明确保留/遗忘信号 | 增强任务监督，检查数据分布 |

边界扩展的伪指令可以写成：

```cpp
// 在块尾部或无效位置补“恒等转移”
if (idx >= valid_length) {
    A_bar[idx] = 1.0;   // 旧状态原样通过
    B_bar[idx] = 0.0;   // 不注入新信息
    x[idx] = 0.0;
}
```

这段逻辑的意思是：无效位置只保留恒等变换，不制造新的状态贡献。  
如果把单步写成 $h_k = A_k h_{k-1} + B_k x_k$，那么无效位置对应的安全条件就是：

$$
A_k = 1,\qquad B_k x_k = 0
$$

这样它不会改变状态：

$$
h_k = 1 \cdot h_{k-1} + 0 = h_{k-1}
$$

### 为什么 `softplus` 常用于 $\Delta$

实践里常见做法是让模型先预测一个原始值 $\hat\Delta_k$，再经过某种正值变换得到真正的步长：

$$
\Delta_k = \mathrm{softplus}(\hat\Delta_k + b_\Delta)
$$

原因很简单：

1. 希望 $\Delta_k>0$，避免无意义或不稳定的负步长。
2. 通过偏置 $b_\Delta$ 控制初始时间尺度。
3. 避免训练初期步长分布过于极端。

但这也引入一个工程事实：实现中的“默认边界行为”不再只是数学纸面上的公式问题，而是会受到参数化方式影响。所以论文公式、PyTorch 代码和 CUDA kernel 往往需要一起看，不能只盯着某一个层面。

### 一个容易被误解的点：选择性不是“天然聪明”

另一个常见坑是过度神化“选择性”。选择性扫描并不意味着模型天然懂得什么重要，真正的选择行为仍然来自训练。如果数据里关键信号稀疏、标注弱，模型可能学到保守策略，把很多 token 都当成“值得写入”。这时它仍会退化成较弱的过滤器。

所以工程上通常要同时关注三件事：

1. 参数化是否稳定，尤其是 $\Delta$ 的范围。
2. kernel 是否处理好了边界和块拼接。
3. 训练数据是否真的提供了“什么该记、什么该忘”的监督信号。

可以把它理解为三层约束：

- 数学层：公式是否稳定。
- 系统层：scan 和 kernel 是否高效且边界正确。
- 数据层：任务本身是否需要、也是否能学出“选择性”。

缺任何一层，效果都会打折。

## 替代方案与适用边界

Mamba 不是所有场景都最优，它主要适合“序列很长、并且信息重要性随位置动态变化”的问题。

如果序列只有几百个 token，普通 Transformer 往往就够了，工程生态也更成熟。选择性扫描的优势，通常在非常长的上下文下才更明显。

下表可作为粗粒度选型参考：

| 方案 | 适用场景 | 长序列成本 | 动态过滤能力 | 工程复杂度 |
|---|---|---:|---:|---:|
| Transformer | 中短序列、通用 NLP | 较高 | 强，但依赖注意力显式建图 | 中 |
| 固定 SSM | 超长序列、规则动力学较强 | 低 | 较弱 | 中 |
| Mamba | 超长序列、需要按内容调节记忆 | 低到中 | 强 | 较高 |
| Chunked LSTM | 资源受限、实现简单优先 | 中 | 中 | 低 |
| Transformer + Chunking | 想保留注意力生态又压缩成本 | 中 | 中到强 | 中 |

适用边界可以概括成三条：

1. 如果任务需要处理百万级甚至更长的序列片段，Mamba 更有吸引力。
2. 如果任务里“哪些 token 重要”强烈依赖上下文，Mamba 比固定 SSM 更自然。
3. 如果硬件或框架不支持高效 parallel scan，那么 Mamba 的理论优势会被工程成本抵消。

一个真实工程判断例子：

- 文本分类，输入 512 token：Transformer 通常够用。
- 应用日志聚合，单条样本几万到几十万事件：Mamba 更值得考虑。
- 老旧部署环境，没有成熟 scan 内核：双层 LSTM 或 Transformer + chunking 可能更务实。

再补一张更偏“决策条件”的表：

| 判断问题 | 更偏向 Mamba | 更偏向替代方案 |
|---|---|---|
| 序列是否特别长 | 是，长度是主要瓶颈 | 否，长度可控 |
| 信息重要性是否高度不均匀 | 是 | 否 |
| 是否需要成熟生态和即插即用工具链 | 否，能接受定制实现 | 是 |
| 团队是否能处理底层性能优化 | 是 | 否 |
| 是否必须保留显式 token-to-token 对齐解释 | 否 | 是，注意力更直观 |

所以，替代方案不是“谁先进用谁”，而是谁的瓶颈更匹配任务。如果你的瓶颈是上下文长度和显存带宽，Mamba 更对症；如果瓶颈是开发成本、部署稳定性和生态兼容，Transformer 仍是默认选择。

## 参考资料

1. **Gu, Dao. _Mamba: Linear-Time Sequence Modeling with Selective State Spaces_**  
   原始论文。重点看选择性状态空间的定义、输入依赖的 $\Delta,B,C$、以及 selective scan 为什么仍能保持线性时间。

2. **state-spaces/mamba 官方实现仓库**  
   重点看模型结构、选择性扫描相关实现、参数化方式，以及 issue 中关于边界条件、数值稳定性、反向传播的讨论。

3. **S4 相关论文与资料，如 _Efficiently Modeling Long Sequences with Structured State Spaces_**  
   适合补足背景：为什么状态空间模型本来就擅长长序列，Mamba 又到底改动了哪一层。

4. **Maarten Grootendorst, _A Visual Guide to Mamba and State Space Models_**  
   适合理解“状态递推”和“并行扫描”之间的关系，尤其对第一次接触 SSM 的读者更友好。

5. **DeepWiki 或类似实现导读中关于 selective scan 的页面**  
   适合在读完论文后补实现细节，特别是 scan、kernel fusion、块级并行等工程内容。

6. **GitHub issue / discussion：与 `delta_softplus`、边界 padding、scan backward 相关的讨论**  
   这些材料对理解“论文上公式成立”和“工程里高性能且稳定地跑起来”之间的差距很有帮助。

7. **阅读顺序建议**  
   如果你是第一次接触这个主题，推荐顺序是：  
   S4 背景 → Mamba 论文 → 可视化导读 → 官方实现 → issue 讨论。  
   这样更容易区分“理论定义”“模块结构”和“内核工程细节”。
