## 核心结论

循环神经网络语言模型，简称 RNN-LM，可以理解成“每读入一个词，就更新一次内部记忆，再用这份记忆预测下一个词”。它解决的核心问题是：不用固定窗口，直接建模整个历史条件下的下一个词概率 $P(w_t \mid w_{<t})$。

RNN-LM 的基本递推公式是：

$$
h_t = \tanh(Wh_{t-1} + Ux_t + b)
$$

其中 $h_t$ 是第 $t$ 个时刻的隐藏状态，可以白话理解为“当前这一步对历史的压缩记忆”；$x_t$ 是当前词的向量表示；$W,U,b$ 是全序列共享的参数。得到 $h_t$ 以后，再通过 softmax 输出下一个词的概率分布：

$$
p(w_{t+1}\mid w_{\le t}) = \text{softmax}(Vh_t + c)
$$

它比 n-gram 语言模型更强的地方在于：n-gram 只能看固定长度上下文，比如 4-gram 最多只看前 3 个词；RNN-LM 理论上可以把任意长历史折叠进隐藏状态里。2010 年 Mikolov 的工作把这种方法用于语言建模后，在 Penn Treebank 等任务上显著降低困惑度（PPL，困惑度，意思是“模型对文本有多困惑，越低越好”），相对 trigram 这类传统方法有明显优势。

玩具例子可以这样看。假设句子是“今天 天气 很 好”。当模型读到“今天”时，它只知道开头；读到“天气”时，隐藏状态会把“今天”也压进去；读到“很”时，它内部保存的是“今天 天气 很”的压缩表示。此时再预测下一个词，“好”的概率就会比无关词更高。这个“边读边记”的过程，就是 RNN-LM 的核心。

但这套方法也有硬边界。RNN 前向传播必须串行，不能像 Transformer 一样把整段文本同时并行算完；更关键的是，反向传播时梯度会沿时间步连乘，容易出现梯度消失或梯度爆炸。前者会让远处上下文学不到，后者会让训练直接不稳定。实际工程里，vanilla RNN 的有效上下文常常只有几十到一百个 token 左右，训练时通常还要配合 `clip norm=5` 这类梯度裁剪。

## 问题定义与边界

语言模型的标准目标是：给定前面的词，预测下一个词的分布。形式化写法是：

$$
P(w_1,\dots,w_T)=\prod_{t=1}^{T} P(w_t\mid w_{<t})
$$

所以语言建模本质上是在学习一连串条件概率。

传统 n-gram 的做法是人为截断历史。例如 4-gram 用：

$$
P(w_t\mid w_{<t}) \approx P(w_t\mid w_{t-3},w_{t-2},w_{t-1})
$$

它的优点是简单、可统计、推理快，但缺点也直接：固定窗口之外的信息完全丢失。像“我昨天买的那本书今天终于……”，如果关键约束出现在十几个词之前，4-gram 根本看不到。

RNN-LM 的边界更宽。它不规定最大回看步数，而是把全部历史压进一个固定维度的隐藏状态 $h_t$。这个设计带来两个直接后果：

1. 参数共享。也就是同一组 $W,U,V$ 在每个时间步重复使用，模型大小不会随着上下文长度线性增长。
2. 记忆压缩。因为不管历史多长，最后都得塞进固定维度的向量，所以它并不是真的“完整保存全部历史”，而是“尽量压缩出对预测有用的信息”。

下面这个表格可以直接看出 n-gram 和 RNN 的差别：

| 维度 | n-gram | RNN-LM |
|---|---|---|
| 上下文范围 | 固定窗口，如前 3 个词 | 理论上任意长度 |
| 参数是否共享 | 不按时间共享，主要依赖统计表 | 全时间步共享同一组参数 |
| 长距离依赖 | 很弱，窗口外直接丢失 | 能表示，但容易退化 |
| 数据稀疏性 | 明显，高阶 n-gram 更严重 | 通过词向量和共享参数缓解 |
| 训练方式 | 计数 + 平滑 | 梯度下降 + BPTT |
| 推理方式 | 快，规则明确 | 串行逐步更新 |

这里要特别强调“理论上任意长度”和“实际有效长度”不是一回事。理论上，RNN 的状态链条可以一直传；实际上，梯度一旦在长链上衰减，远处信息就很难影响参数更新。也就是说，模型结构允许长依赖，不代表训练一定学得住长依赖。

## 核心机制与推导

RNN-LM 的前向传播分三步：

1. 把当前词 $w_t$ 查表变成向量 $x_t$
2. 用上一步隐藏状态 $h_{t-1}$ 和当前输入 $x_t$ 计算新状态 $h_t$
3. 用 $h_t$ 输出下一个词分布

写成公式就是：

$$
h_t = \tanh(Wh_{t-1}+Ux_t+b)
$$

$$
o_t = Vh_t + c
$$

$$
p_{t+1} = \text{softmax}(o_t)
$$

softmax 的作用可以白话理解为“把一组任意实数分数变成总和为 1 的概率分布”。

训练时最常见的损失函数是交叉熵。交叉熵可以理解成“真实词越被模型赋高概率，损失越小”。对一个长度为 $T$ 的序列：

$$
L = -\sum_{t=1}^{T} \log P(w_t \mid w_{<t})
$$

难点不在前向，而在反向传播。RNN 训练用的是 BPTT，Backpropagation Through Time，意思是“把时间展开后做反向传播”。因为 $h_t$ 依赖 $h_{t-1}$，所以某个早期状态的梯度会被后面很多步的 Jacobian 连乘：

$$
\frac{\partial L}{\partial h_t}
=
\frac{\partial L}{\partial h_{t+1}}
\cdot
\frac{\partial h_{t+1}}{\partial h_t}
$$

继续展开就是：

$$
\frac{\partial L}{\partial h_t}
=
\frac{\partial L}{\partial h_T}
\prod_{k=t+1}^{T}
\frac{\partial h_k}{\partial h_{k-1}}
$$

这里的 Jacobian 可以理解成“当前状态变化会怎样影响下一状态”的局部线性映射。如果这些矩阵的谱半径长期小于 1，梯度就会指数衰减；如果大于 1，梯度就可能指数爆炸。谱半径可以白话理解为“一个线性变换对向量放大能力的最大主导尺度”。

最直观的办法是看简化数值。假设每一步都近似乘一个衰减因子 $\alpha$，那经过 $n$ 步后，梯度量级大约是 $\alpha^n$。

| 衰减因子 $\alpha$ | 10 步后 | 20 步后 | 50 步后 |
|---|---:|---:|---:|
| 0.9 | 0.349 | 0.122 | 0.005 |
| 0.5 | 0.00098 | 0.00000095 | 0.0000000000000009 |

这张表说明一个事实：哪怕每步只损失一点点，长链一乘就会几乎归零。比如 $\alpha=0.9$ 时，20 步后只剩约 0.12；50 步后只剩 0.005。也就是说，50 个时间步以前的信息即使结构上存在，训练时也未必能有效影响参数。这就是为什么很多 vanilla RNN 在实践里只能稳定利用几十步上下文。

玩具例子可以进一步说明。设句子为：

“如果 明天 下雨 我们 就 改成 室内 活动”

当模型看到“改成”时，真正决定后续“室内”活动概率的关键信息，可能来自 4 个词前的“下雨”。如果序列再长一点，比如中间插入 30 个无关词，RNN 理论上仍能传递“下雨”这个条件，但训练时这条依赖很可能因为梯度衰减而学不到。于是模型会更依赖最近词，而不是更关键的远处词。

梯度爆炸则是另一面。如果循环矩阵的有效放大系数偏大，梯度会在时间展开链上急剧变大，导致参数更新一步就冲飞。工程上常见做法是梯度裁剪，即 clip norm。它的意思是“如果梯度向量总长度超过阈值，就按比例缩小，但方向不变”。常用阈值就是 5 左右。

所以 vanilla RNN 的训练稳定性，通常靠三类手段：

| 问题 | 机制 | 常见处理 |
|---|---|---|
| 梯度消失 | Jacobian 连乘后持续缩小 | LSTM/GRU、正交初始化、较短截断长度 |
| 梯度爆炸 | Jacobian 连乘后持续放大 | clip norm、较小学习率 |
| 长序列难学 | 信息必须挤进固定维度状态 | 门控结构、注意力、外部缓存 |

这里的正交初始化可以理解成“让初始循环矩阵尽量保持向量长度不变”，从而减轻早期训练时的梯度退化；门控结构如 LSTM、GRU 则是在 RNN 内部增加“写入、保留、遗忘”的控制开关，让记忆路径更稳定。

## 代码实现

下面先给一个最小可运行的 Python 玩具实现。它不是高性能训练代码，但能准确展示 RNN-LM 的前向计算、softmax 概率化和损失计算过程，并包含 `assert` 检查。

```python
import numpy as np

np.random.seed(0)

vocab = ["我", "爱", "机器", "学习"]
stoi = {w: i for i, w in enumerate(vocab)}
vocab_size = len(vocab)
hidden_size = 5

# 参数初始化
E = np.random.randn(vocab_size, hidden_size) * 0.1   # embedding
W = np.random.randn(hidden_size, hidden_size) * 0.1  # recurrent
U = np.random.randn(hidden_size, hidden_size) * 0.1  # input projection
b = np.zeros((hidden_size,))
V = np.random.randn(vocab_size, hidden_size) * 0.1   # output
c = np.zeros((vocab_size,))

def softmax(x):
    x = x - np.max(x)
    ex = np.exp(x)
    return ex / ex.sum()

def rnn_step(h_prev, token_id):
    x = E[token_id]
    h = np.tanh(W @ h_prev + U @ x + b)
    logits = V @ h + c
    probs = softmax(logits)
    return h, probs

# 一个长度为3的训练样本：输入“我 爱 机器”，目标“爱 机器 学习”
inputs = [stoi["我"], stoi["爱"], stoi["机器"]]
targets = [stoi["爱"], stoi["机器"], stoi["学习"]]

h = np.zeros(hidden_size)
loss = 0.0
all_probs = []

for x_id, y_id in zip(inputs, targets):
    h, probs = rnn_step(h, x_id)
    loss += -np.log(probs[y_id] + 1e-12)
    all_probs.append(probs)

# 基本正确性检查
assert len(all_probs) == 3
for probs in all_probs:
    assert probs.shape == (vocab_size,)
    assert np.isclose(probs.sum(), 1.0)

assert loss > 0
print("loss =", float(loss))
print("next-token distribution =", all_probs[-1])
```

如果把这段代码抽象成训练框架，核心循环通常长这样：

```python
h = h0
loss = 0
for t in sequence:
    x = embedding[input_ids[t]]
    h = tanh(W @ h + U @ x + b)
    logits = V @ h + c
    loss += cross_entropy(logits, target_ids[t])

loss.backward()
grad_norm = clip_grad_norm(model.parameters(), max_grad_norm=5.0)
optimizer.step()
optimizer.zero_grad()
```

实际工程实现一般还会多出几个关键点。

第一，截断 BPTT。也就是不把一整篇长文完全展开反传，而是每次只反传最近几十步，比如 35 或 70 步。这样做会损失超长依赖，但能显著降低显存和计算成本。

第二，正交初始化。循环矩阵 $W$ 常用 orthogonal init，因为它对梯度长度更友好，尤其是在训练初期比随机高斯更稳。

第三，批处理。虽然 RNN 单条序列上是串行的，但可以把多条序列放在同一个 batch 内并行处理每个时间步，从而提高吞吐。

第四，状态管理。在语言模型训练中，常常会把上一个 batch 的最终状态作为下一个 batch 的初始状态，但要 `detach`，否则计算图会无限跨 batch 连接，导致内存爆炸。

真实工程例子是语音识别中的二阶段解码。做法通常不是让 RNN-LM 单独从头识别整句，而是先由声学模型和较轻量语言模型生成候选，再用 RNN-LM 对候选句进行重打分。Mikolov 2010 年的结果表明，这种方式在 WSJ 和 NIST 任务上能明显降低 PPL，并带来 WER 改善。原因不是 RNN 推理更快，而是它在有限数据下对顺序模式建模更强，适合做“第二遍更精细的判别”。

## 工程权衡与常见坑

RNN-LM 的优点很清楚：参数共享强、对顺序数据天然、在中小规模任务和实时解码链路里仍然实用。但它的工程代价也非常明确。

第一类问题是串行瓶颈。RNN 计算 $h_t$ 之前必须先拿到 $h_{t-1}$，这意味着同一条序列上的 token 无法完全并行。Transformer 的自注意力可以把同层全部位置一起算，RNN 不行。所以在相同数据规模、相近参数量下，RNN 训练速度通常显著慢于 Transformer，工程经验里慢一个数量级并不罕见。

第二类问题是梯度稳定性。vanilla RNN 最常见的训练故障就是：

| 常见坑 | 表现 | 原因 | 常见规避策略 |
|---|---|---|---|
| 梯度消失 | loss 降不动，长依赖学不到 | 连乘 Jacobian 持续缩小 | LSTM/GRU、orthogonal init、缩短截断长度 |
| 梯度爆炸 | loss 突然变成 `nan` 或剧烈震荡 | 连乘 Jacobian 持续放大 | clip norm、降学习率 |
| 串行训练慢 | GPU 利用率不高，吞吐低 | 时间维依赖无法并行 | 缩短序列、batch 并行、混合架构 |
| 隐状态污染 | 训练不稳定，跨样本串味 | 状态没有正确 reset/detach | 明确句边界，跨 batch 时 detach |
| 词表过大 | softmax 计算重 | 输出层复杂度高 | sampled softmax、短列表、子词切分 |

第三类问题是“理论可建模”不等于“工程上值得建模”。比如对 4k 上下文的代码补全，vanilla RNN 几乎不是合适选择，因为它既难学住这么长的依赖，又无法高效并行。反过来，在嵌入式语音解码器这类低延迟、逐帧输入的场景里，RNN 的一步一步状态更新反而很自然。

真实工程里最典型的案例仍是语音识别重打分。RNN LM 往往能把 n-gram 的困惑度进一步压低，进而把最终 WER 再往下推。Mikolov 报告里，在 WSJ/NIST 这类任务上，RNN-LM 相对经典 backoff 模型取得了明显收益。这类收益尤其常见于训练数据不算特别巨大的场景，因为 RNN 的参数共享比高阶 n-gram 更抗数据稀疏。

但代价同样明确：不能把它当成万能长上下文建模器，也不能忽略训练稳定性。如果不做梯度裁剪、不控制序列长度、不选合适初始化，训练过程很容易表现为“短句还行，长句完全不学”。

## 替代方案与适用边界

今天讨论语言模型，RNN-LM 已经不是主流大模型的核心架构。主要替代方案是 Transformer。Transformer 的核心优势是自注意力，也就是“每个位置可以直接看整段里其他位置，并且同层并行计算”。这带来两点决定性变化：

1. 训练并行度高，硬件利用率更好
2. 长距离依赖不需要沿时间链条逐步传递

所以在大规模预训练、超长上下文建模、现代 GPU 集群训练中，Transformer 基本全面占优。

下面是一个工程视角对比：

| 维度 | RNN-LM | Transformer |
|---|---|---|
| 时间步并行性 | 低，必须串行 | 高，训练时高度并行 |
| 长距离依赖 | 理论可传递，实践易衰减 | 直接全局交互，更稳定 |
| 单步在线更新 | 强 | 需要缓存 KV，系统更复杂 |
| 小模型实时部署 | 较友好 | 可做，但实现更重 |
| 大规模预训练 | 不占优 | 主流方案 |
| 典型场景 | 语音解码、低资源顺序任务 | 通用 NLP、代码、长文本 |

这不代表 RNN 没有价值。它仍然适合以下边界：

1. 流式任务。比如实时语音识别、在线对话状态跟踪。输入是逐步到来的，RNN 一步更新一次状态，接口天然匹配。
2. 资源受限部署。小型 RNN 在参数量、运行逻辑、状态维护上仍然有简单的一面。
3. 作为混合系统组件。比如和 n-gram、CTC、WFST 或轻量注意力模块组合，做重打分或局部序列建模。

一个实际对比是：如果任务是离线训练大规模文本模型，目标是高质量长上下文生成，那么应优先选 Transformer；如果任务是嵌入式语音解码器里的在线重打分，输入逐帧到达、延迟预算很紧，那么 RNN 或门控 RNN 仍然可能是更实际的工程方案。

还有一个容易混淆的点：很多人在说“RNN 不行”时，实际批评的是 vanilla RNN，不是整个循环架构家族。LSTM 和 GRU 通过门控显著缓解了梯度消失问题，在很多中小规模序列任务里仍然有效。只是当任务规模和上下文规模继续增大时，Transformer 的并行性和表达路径更有优势。
