## 核心结论

当知识图谱规模超过 10 亿三元组时，单机存储通常先撞上三类上限：内存放不下、磁盘随机访问变慢、查询并发上不去。这里的“三元组”指一条 `主语-关系-宾语` 事实，例如“用户A-购买-商品B”。分布式存储的本质，不是简单把数据拆成几份扔到多台机器，而是要在“负载均衡”和“跨机通信”之间找到一个可接受的平衡点。

工程上常见的三种策略是：

| 分割维度 | 优点 | 通信触发 | 适用场景 |
| --- | --- | --- | --- |
| 水平切分 | 实现简单，容易按容量扩容 | 一次查询访问多个实体范围时 | 实体数量大、查询偏主键 |
| 垂直切分 | 某类关系可单独优化 | 查询需要跨关系联接时 | 关系类型稳定、分析任务清晰 |
| 图分区 | 能减少跨分区边，查询延迟更低 | 图遍历碰到跨分区边时 | 关系密集、实时多跳查询 |

对新手可以用一个直观类比：把大图看成很多本笔记本。水平切分是按实体 ID 把记录分页到不同本子；垂直切分是把“购买”“关注”“评论”分别记到不同本子；图分区则尽量把经常互相关联的内容写在同一本上，减少翻本次数。Neo4j Fabric 和 JanusGraph 这类系统的作用，是在上层把这些分散的本子重新组织成一个“逻辑上完整”的图。

真正决定查询性能的不是“有几台机器”，而是两个指标：复制因子 RF 和负载因子 LF。RF 越高，说明为了查询方便而复制了越多顶点副本，写入和一致性成本越高；LF 越高，说明最忙的分区远比平均值更重，热点会拖慢整个集群。一个好的分布式知识图谱方案，目标不是把 RF 压到最低或把 LF 压到最低，而是在业务查询模式下同时把两者控制在合理范围内。

## 问题定义与边界

“大规模知识图谱分布式存储”在这篇文章中的边界很明确：图规模超过 10 亿三元组，数据分布在多个物理节点，且系统目标不是离线批处理，而是支持秒级到亚秒级查询，典型场景包括检索增强生成、风控关联查询、推荐召回和企业知识问答。

这一定义有两个重点。

第一，讨论的是“图型访问”而不是“把 RDF 文件拆到多台机器”。如果你的核心任务只是每天离线跑一次统计，HDFS 或对象存储加 Spark 就够了；只有当查询要做多跳关系遍历、邻居展开、路径过滤时，图分布式存储才真正成为主问题。

第二，讨论的是“逻辑一张图，物理多节点”。用户希望像查一张图一样发起查询，但底层实际分布在多台机器上，因此系统必须解决路由、聚合、重试和一致性边界。

典型挑战可以概括为：

| 问题维度 | 分布式需求 | 现实限制 |
| --- | --- | --- |
| 跨节点通信 | 尽量少走网络 | 多跳查询天然容易跨分区 |
| 负载均衡 | 每台机器负担接近 | 热点实体会导致倾斜 |
| 动态扩容 | 新增机器后平滑迁移 | 图数据重分区代价高 |
| 实时一致性 | 查询结果尽量新 | 跨分区事务成本高 |

一个真实工程边界是 Neo4j Fabric 的 shard 模式。它允许把一张大图拆成多个 Neo4j 数据库，每个 shard 管一块互不重叠的子图，Fabric 层负责把查询路由到目标数据库，再把结果拼回来。以 LDBC 社交图为例，可以让不同 shard 分别负责不同用户集合及其本地子图；如果用户之间存在跨 shard 关系，就通过代理节点或外部 ID 关联完成跳转。这样做的好处是每个 shard 仍然像普通 Neo4j 一样运作，但坏处是跨 shard 关系不再是“本地指针”，而变成一次远程访问问题。

所以，分布式知识图谱的核心边界不是“能不能拆”，而是“拆完以后，多跳查询还能不能快”。

## 核心机制与推导

理解分区质量，最有用的两个指标是 RF 和 LF。

复制因子 RF 定义为：

$$
RF = \frac{\text{所有分区中的顶点副本总数}}{|V|}
$$

其中 $|V|$ 是原始图中的顶点数。它衡量“为了把图拆开，你额外复制了多少点”。RF 越高，通常说明跨分区边较多，或者你为了降低查询通信而做了更多副本。

负载因子 LF 定义为：

$$
LF = \frac{\max(|E_p|)}{\operatorname{avg}(|E_p|)}
$$

其中 $|E_p|$ 表示某个分区中的边数。它衡量“最重分区相对平均值有多重”。LF 越接近 1，说明越均衡。

这两个指标往往互相拉扯。为了减少跨分区通信，你会倾向于把相关点复制到多个地方，RF 上升；为了避免单台机器太忙，你会倾向于把图切得更散，结果更多查询需要跨机，RF 和通信量又可能上升。

可以先看一个玩具例子。

假设有 8 个用户节点，两类关系是“关注”和“购买”。如果按 `user_id % 4` 做水平切分，那么用户 1、5 在分区 A，2、6 在分区 B，依此类推。这个方案的优点是每个分区点数接近；但如果真实查询经常是“看一个用户及其二跳好友”，而好友关系主要发生在同一个社群内部，那么简单按 ID 哈希就会把社群切碎。结果是点分得均匀了，但边被切断了，图遍历频繁跨机。

图分区方法的目标正好相反：尽量让高频路径留在同一分区内。所谓“edge cut”，就是被切断到不同分区的边。edge cut 少，通常意味着一次图遍历更可能留在本机完成。

再看一个数值推导。假设有 10 亿三元组，切成 4 个分区，理想情况下每个分区约承担 $2.5 \times 10^8$ 条边。若简单水平切分后，10% 的关键查询路径会碰到跨分区边，一次典型查询平均触发 10 次远程访问；若改成按社群或业务聚类的图分区，把热路径尽量放进同一分区，那么跨分区边比例可能下降，平均远程访问降到 2 次。对于延迟敏感的在线查询，这不是“快一点”，而是从“网络主导延迟”回到“本地索引主导延迟”。

下面这个趋势表可以帮助理解：

| 分区策略 | RF | LF | 查询特征 |
| --- | --- | --- | --- |
| 纯哈希水平切分 | 低到中 | 低 | 负载均衡好，但图遍历跨机多 |
| 关系垂直切分 | 低 | 中到高 | 单关系查询快，跨关系联接贵 |
| 社群导向图分区 | 中 | 中 | 多跳查询更友好 |
| 大量副本缓存热点 | 高 | 低到中 | 读快，但写入和一致性复杂 |

因此，分区设计不是单一数学最优，而是“根据查询模式选择代价分布”。如果业务 80% 是单点查实体属性，水平切分够用；如果业务 80% 是二跳、三跳关联分析，图分区通常更合理；如果某些关系类型本身就是独立工作流，垂直切分可以把系统拆成多个子图服务。

## 代码实现

下面先给一个最小可运行的 Python 玩具实现，演示“按实体 ID 水平切分”和“代理层合并查询结果”的基本思路。这里的“代理层”就是上层统一入口，它负责把请求发给多个分区，再把结果拼成一张逻辑图。

```python
from collections import defaultdict

P = 4

triples = [
    ("u1", "follow", "u2"),
    ("u1", "buy", "item9"),
    ("u2", "follow", "u3"),
    ("u3", "buy", "item7"),
    ("u5", "follow", "u1"),
]

def shard_of(entity: str, partitions: int = P) -> int:
    num = int("".join(ch for ch in entity if ch.isdigit()) or 0)
    return num % partitions

def build_shards(triples, partitions: int = P):
    shards = defaultdict(list)
    for s, p, o in triples:
        shards[shard_of(s, partitions)].append((s, p, o))
    return shards

def query_out_edges(entity: str, shards):
    # 代理层只路由到主语所在分区
    sid = shard_of(entity, P)
    return [t for t in shards[sid] if t[0] == entity]

def query_two_hops(entity: str, shards):
    result = []
    first_hop = query_out_edges(entity, shards)
    result.extend(first_hop)
    for _, _, next_entity in first_hop:
        if next_entity.startswith("u"):
            result.extend(query_out_edges(next_entity, shards))
    return result

shards = build_shards(triples)

assert shard_of("u1") == 1
assert ("u1", "buy", "item9") in query_out_edges("u1", shards)

two_hops = query_two_hops("u1", shards)
assert ("u1", "follow", "u2") in two_hops
assert ("u2", "follow", "u3") in two_hops
```

这个例子很简单，但已经暴露了真实问题：第一跳查 `u1` 只需命中一个分区，第二跳可能要再查 `u2` 所在分区。图一旦变大，远程跳转次数会迅速累积。

如果做图分区，代理层逻辑会变成“优先把相关子图放在同一分区，只有碰到 cut edge 才远程拉取”。伪代码可以写成：

```python
def route_query(seed_entities, planner, shard_clients):
    target_shards = planner.locate(seed_entities)
    partial_results = []

    for shard_id in target_shards:
        partial = shard_clients[shard_id].execute_local(seed_entities)
        partial_results.append(partial)

    return merge_subgraphs(partial_results)

def merge_subgraphs(parts):
    merged_vertices = {}
    merged_edges = set()

    for part in parts:
        for v in part["vertices"]:
            merged_vertices[v["id"]] = v
        for e in part["edges"]:
            merged_edges.add((e["src"], e["type"], e["dst"]))

    return {
        "vertices": list(merged_vertices.values()),
        "edges": list(merged_edges),
    }
```

在 JanusGraph 这类系统里，常见做法是显式分区。所谓“显式分区”，就是写入时由应用或图库明确决定顶点放到哪个分区，而不是完全交给底层随机分配。新手可以把它理解为“插入数据时就先决定这页写进哪本笔记本”。

配置层面常见关注点包括支持的后端和分区上限，例如：

```properties
storage.backend=hbase
ids.block-size=100000
cluster.max-partitions=32
```

如果你的写入规则是“同一业务实体族尽量进同一分区”，伪代码可能是：

```python
def choose_partition(entity_id: int, partition_count: int) -> int:
    return entity_id % partition_count

assert choose_partition(10, 4) == 2
assert choose_partition(11, 4) == 3
```

真实工程例子是电商知识图谱。商品、店铺、品牌、类目、用户行为事件合在一起很容易超过十亿边。如果在线查询主要是“商品 -> 品牌 -> 相似商品 -> 用户兴趣标签”的多跳路径，那么单纯按商品 ID 哈希切分往往会把热点路径切碎；更合理的做法是把高频共现关系聚到一起，让代理层只在少数边界处做跨分区补查。

## 工程权衡与常见坑

分布式图的难点不在“怎么拆”，而在“拆了以后还能不能运维”。下面这些坑在项目里非常常见。

| 坑 | 后果 | 规避方法 |
| --- | --- | --- |
| 分区规则只看容量，不看查询路径 | 跨机跳转多，延迟失控 | 先采样真实查询，再定切分策略 |
| `cluster.max-partitions` 设太大 | 小分区过多，冷热碎片严重 | 先按 1 到 2 年容量规划，不要过度超配 |
| JanusGraph 显式分区设计后难修改 | 重分布成本高 | 上线前做压测和数据回放 |
| 热点实体被切成多个远程路径 | RF 上升，查询抖动明显 | 对热点子图做聚类或副本缓存 |
| 盲目追求低 RF | 负载堆到少数节点 | 联合观察 RF、LF 和 P99 延迟 |

Neo4j Fabric 的典型权衡是：它把“多个数据库组成一张逻辑图”的工程问题做得较清晰，适合企业做跨机部署和多库路由，但跨 shard 关系不是天然本地边，所以查询写法、事务边界和结果聚合都需要更明确的设计。你可以把它理解成“多个图数据库前面加了一个统一调度层”，优点是运维边界清楚，缺点是跨 shard 图遍历不如单图自然。

JanusGraph 的典型坑更偏底层。显式分区一旦初始化，后续调整很麻烦；而且分区能力依赖支持有序分区的后端，例如 HBase 或做过相应配置的 Cassandra。很多团队一开始把分区数设得很大，以为这样更“可扩展”，结果实际写入规模远不到那个量级，造成大量冷分区、缓存命中差、扫描放大。另一类常见问题是业务热点变化。最初按组织、地区或实体 ID 设计的分区，在业务演进后可能出现新的高频路径，原先的局部最优分区会失效。

一个真实工程坑是推荐系统中的“超级节点”。例如头部商品或头部作者会和海量用户行为相连。如果不做特殊处理，这类点会让分区负载和跨分区关系同时恶化：把它放单机，会成为热点；复制到多机，RF 会飙升；拆开关联边，又会让查询频繁跨机。这里通常需要专门的热点治理策略，而不是继续依赖通用分区规则。

## 替代方案与适用边界

并不是所有“超大图”都应该直接上图库分布式方案。选择取决于你要的是“在线图查询能力”，还是“离线大规模图分析能力”。

| 方案 | 架构特点 | 伸缩性 | 配置难度 | 实时性 |
| --- | --- | --- | --- | --- |
| Neo4j Fabric | 多数据库路由，逻辑统一视图 | 中到高 | 中 | 较好 |
| JanusGraph 显式分区 | 图层控制分区，后端分布式存储 | 高 | 高 | 好 |
| Spark/GraphX 类离线图分析 | 批量作业驱动 | 很高 | 中 | 弱 |
| 普通 KV/搜索引擎加反向索引 | 非原生图，但实现简单 | 高 | 低到中 | 取决于建模 |

如果企业场景是跨地域部署、每个 shard 有独立硬件和维护边界，同时查询多为“按业务域路由后做局部图查询”，Fabric 式方案通常更实用。它更像一种“多图协调层”。

如果场景是单一超大图、关系遍历频繁、你愿意投入更多建模和运维成本去优化热点子图，那么 JanusGraph 这类显式分区方案更有发挥空间。它允许你把分区策略更深地嵌入数据模型，但代价是对底层存储、分区规划和后期迁移要求更高。

还有一个重要边界：如果你的主要工作负载是训练、离线关系发现、全图统计，而不是在线问答或实时路径检索，那么图分区做到很精细未必划算。此时把原始图落在分布式文件系统，配合批处理框架，常常更经济。

结论不是“哪种最好”，而是“哪种最符合查询模式和运维能力”。图越大，分区越不是纯技术问题，而是查询模型、业务边界和团队能力的共同结果。
