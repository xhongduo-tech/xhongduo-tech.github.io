## 核心结论

稀疏矩阵的核心思想是：矩阵里绝大多数位置都是 0，就不要为这些 0 付存储和计算成本。这里的 `nnz` 表示 non-zero entries，也就是“非零元素个数”。如果一个 $m \times n$ 矩阵只有 `nnz` 个非零项，那么压缩存储的目标就是让空间成本与 `nnz` 线性相关，而不是与总元素数 $mn$ 相关。

公式上可以直接写成：

$$
\text{Storage} \propto nnz
$$

常见的两种格式是 COO 和 CSR。

- COO（Coordinate，坐标格式）：把每个非零元素记成 `(row, col, val)` 三元组。
- CSR（Compressed Sparse Row，按行压缩格式）：把“每一行从哪里开始、到哪里结束”单独压缩成一个行指针数组。

当矩阵非常稀疏时，尤其是 $\frac{nnz}{mn} < 1\%$ 这一类场景，稀疏存储和稀疏计算通常才有明确收益。因为这时不仅空间从 $O(mn)$ 降到 $O(nnz)$，矩阵向量乘法 SpMV 也能从遍历全部元素的 $O(mn)$，降到只遍历非零元素的 $O(nnz)$。

一个最小玩具例子：

$$
A=
\begin{bmatrix}
5&0&0&0\\
0&3&0&1\\
0&0&0&0\\
2&0&0&4
\end{bmatrix}
$$

它的 COO 表示是：

`(0,0,5), (1,1,3), (1,3,1), (3,0,2), (3,3,4)`

它的 CSR 表示是：

- `row_ptr = [0,1,3,3,5]`
- `col_ind = [0,1,3,0,3]`
- `values = [5,3,1,2,4]`

这个例子只有 5 个非零项。稀疏格式记录的核心数据规模跟 5 成正比，而不是跟 16 个总位置成正比。

| 存储方式 | 主要内容 | 空间量级 |
| --- | --- | --- |
| Dense 稠密存储 | 全部 $mn$ 个元素 | $O(mn)$ |
| COO | `row[nnz] + col[nnz] + val[nnz]` | $O(nnz)$ |
| CSR | `row_ptr[m+1] + col[nnz] + val[nnz]` | $O(nnz + m)$ |

结论可以进一步压缩成一句话：矩阵越稀，越应该把“零的位置”从数据结构里删掉。

## 问题定义与边界

稀疏矩阵不是一个“数学上绝对固定”的阈值概念，而是一个“工程上值不值得压缩”的概念。直白地说，如果非零元素很少，用稀疏格式能省很多内存和计算；如果非零元素并不稀少，索引本身的额外开销会把收益吃掉。

一个常用指标是稀疏度比例：

$$
\rho = \frac{nnz}{mn}
$$

其中：

- $m$ 是行数
- $n$ 是列数
- `nnz` 是非零元素个数
- $\rho$ 表示“非零元素占总位置的比例”

经验上可以这样判断：

| 非零比例 $\rho$ | 是否优先考虑稀疏 | 原因 |
| --- | --- | --- |
| `< 1\%` | 推荐 | 存储和计算收益通常明显 |
| `1\% - 10\%` | 需要评估 | 是否收益取决于硬件、访问模式、算子类型 |
| `> 10\%` | 多数情况不推荐 | 索引开销和访存不规则可能抵消收益 |

稀疏矩阵最经典的计算任务是 SpMV，即 Sparse Matrix-Vector Multiplication，中文就是“稀疏矩阵与向量相乘”。它的定义是：

$$
y_i = \sum_{j \in I(i)} A_{ij} x_j
$$

这里的 $I(i)$ 表示第 $i$ 行所有非零列的集合。白话解释：第 $i$ 行只看那些真正有值的位置，不去碰值为 0 的列。

这一定义本身就说明了稀疏计算的边界条件：只有当“非零位置远少于总位置”时，只遍历 $I(i)$ 才有意义。如果一行几乎到处都非零，那还不如直接用稠密矩阵。

新手常见误区是把“矩阵里有很多 0”直接等同于“适合稀疏优化”。这不总是对。因为稀疏优化有两个隐含成本：

- 索引成本：你不存 0 了，但要存行号、列号、行指针。
- 访存成本：数据不再连续，CPU/GPU 更难高效预取。

一个直观数量级例子：假设矩阵大小是 `10000 × 10000`，总元素数是 $10^8$。如果非零比例是 `0.1%`，那么：

$$
nnz = 10^8 \times 0.001 = 10^5
$$

这时稠密存储要放 $10^8$ 个数，而稀疏只需围绕 $10^5$ 个非零项建结构，差距是三个数量级。SpMV 也只需要围绕这 $10^5$ 个位置做乘加，而不是把 $10^8$ 个位置全扫一遍。

所以问题定义可以收束成两点：

- 稀疏矩阵不是“有没有 0”，而是“0 是否多到值得专门编码”。
- 稀疏计算不是“数学公式变了”，而是“遍历范围从全部位置缩成了非零位置集合”。

## 核心机制与推导

### 1. COO 为什么成立

COO 最直接。每出现一个非零值，就记一条记录：

- `row[k]`：第几个非零值所在的行
- `col[k]`：第几个非零值所在的列
- `val[k]`：第几个非零值本身

于是 COO 的三个数组长度都等于 `nnz`。因此总空间量级是：

$$
O(nnz)
$$

还是用前面的 4×4 例子：

$$
A=
\begin{bmatrix}
5&0&0&0\\
0&3&0&1\\
0&0&0&0\\
2&0&0&4
\end{bmatrix}
$$

对应 COO：

| k | row[k] | col[k] | val[k] |
| --- | --- | --- | --- |
| 0 | 0 | 0 | 5 |
| 1 | 1 | 1 | 3 |
| 2 | 1 | 3 | 1 |
| 3 | 3 | 0 | 2 |
| 4 | 3 | 3 | 4 |

COO 的优点是构造简单、插入简单。缺点是按行计算时不够高效，因为行信息分散在每一条记录里。

### 2. CSR 为什么更适合按行计算

CSR 的关键想法是：如果很多非零项属于同一行，就没必要对每个元素都重复存一遍行号。改成只存一份“这一行在数组里的起止位置”。

CSR 有三个数组：

- `row_ptr[m+1]`：第 `i` 行的非零元素从哪里开始，到哪里结束
- `col_ind[nnz]`：每个非零值对应的列号
- `values[nnz]`：每个非零值

前面例子的 CSR 是：

- `row_ptr = [0,1,3,3,5]`
- `col_ind = [0,1,3,0,3]`
- `values = [5,3,1,2,4]`

含义是：

- 第 0 行的数据在区间 `[0,1)`，只有 `values[0]=5`
- 第 1 行的数据在区间 `[1,3)`，有 `values[1]=3` 和 `values[2]=1`
- 第 2 行的数据在区间 `[3,3)`，为空行
- 第 3 行的数据在区间 `[3,5)`，有 `values[3]=2` 和 `values[4]=4`

因此第 `i` 行的 SpMV 可以直接写成：

```text
for i in 0..m-1:
    for k in row_ptr[i]..row_ptr[i+1]-1:
        y[i] += values[k] * x[col_ind[k]]
```

外层循环跑 `m` 次，但内层总次数不是 `m*n`，而是所有行的非零项总和，也就是 `nnz`。因此复杂度是：

$$
O(m + nnz)
$$

通常把它简写成：

$$
O(nnz)
$$

因为当矩阵非常大时，`nnz` 通常远大于 `m`，主导成本来自遍历非零项。

### 3. 玩具例子的完整推导

令：

$$
x = [1,2,3,4]^T
$$

按 CSR 做乘法：

- 第 0 行：`5 * x[0] = 5`
- 第 1 行：`3 * x[1] + 1 * x[3] = 3*2 + 1*4 = 10`
- 第 2 行：空行，结果是 `0`
- 第 3 行：`2 * x[0] + 4 * x[3] = 2*1 + 4*4 = 18`

所以：

$$
y = Ax = [5,10,0,18]^T
$$

这个过程没有访问矩阵中的任何 0。稠密算法要做 16 次乘法检查，CSR 只围绕 5 个非零元素做有效计算。

### 4. 真实工程例子

真实工程里最典型的不是 4×4，而是“超大但很空”的矩阵。

例如长文本注意力。标准自注意力会构造一个 $n \times n$ 的注意力模式，复杂度是 $O(n^2)$。Longformer 使用滑动窗口稀疏注意力，只让每个 token 看附近宽度为 $w$ 的局部邻域，于是有效连接数从 $n^2$ 变成大约 $n \cdot w$。这本质上就是把原本稠密的注意力矩阵变成稀疏矩阵。

如果 `n = 4096`，`w = 512`，那么连接数规模从：

$$
4096^2 \approx 1.68 \times 10^7
$$

降到：

$$
4096 \times 512 \approx 2.10 \times 10^6
$$

量级下降约 8 倍。这里真正有意义的不是“矩阵名字变了”，而是注意力模式已经不再需要为所有 token 对保留数据和计算。

## 代码实现

下面用纯 Python 写一个可以运行的 COO 转 CSR，并完成 SpMV 的最小实现。代码里显式处理了“按行、按列排序”和“重复位置求和”两个工程上必须做的动作。

```python
from collections import defaultdict

def coo_to_csr(m, n, coo_entries):
    # coo_entries: list of (row, col, val)
    # 先做合法性检查，并对重复坐标做归约
    acc = defaultdict(float)
    for r, c, v in coo_entries:
        assert 0 <= r < m
        assert 0 <= c < n
        acc[(r, c)] += v

    # 按 (row, col) 排序，保证 CSR 行内有序
    items = sorted((r, c, v) for (r, c), v in acc.items() if v != 0)

    row_ptr = [0] * (m + 1)
    col_ind = []
    values = []

    current_row = 0
    count = 0

    for r, c, v in items:
        while current_row < r:
            row_ptr[current_row + 1] = count
            current_row += 1
        col_ind.append(c)
        values.append(v)
        count += 1

    while current_row < m:
        row_ptr[current_row + 1] = count
        current_row += 1

    return row_ptr, col_ind, values

def spmv_csr(m, row_ptr, col_ind, values, x):
    assert len(row_ptr) == m + 1
    assert len(col_ind) == len(values)
    y = [0.0] * m
    for i in range(m):
        s = 0.0
        for k in range(row_ptr[i], row_ptr[i + 1]):
            s += values[k] * x[col_ind[k]]
        y[i] = s
    return y

# 玩具例子
m, n = 4, 4
coo = [
    (0, 0, 5),
    (1, 1, 3),
    (1, 3, 1),
    (3, 0, 2),
    (3, 3, 4),
]

row_ptr, col_ind, values = coo_to_csr(m, n, coo)

assert row_ptr == [0, 1, 3, 3, 5]
assert col_ind == [0, 1, 3, 0, 3]
assert values == [5.0, 3.0, 1.0, 2.0, 4.0]

x = [1, 2, 3, 4]
y = spmv_csr(m, row_ptr, col_ind, values, x)

assert y == [5.0, 10.0, 0.0, 18.0]
print("CSR and SpMV work:", y)
```

这段代码有三个重点。

第一，COO 到 CSR 的转换不是简单复制数组，而是要先整理数据。因为真实输入里经常出现：

- 同一坐标重复写入
- 行内列号无序
- 某些条目值最终归约后变成 0

第二，`row_ptr` 的长度一定是 `m + 1`。这是为了让第 `i` 行的区间写成 `[row_ptr[i], row_ptr[i+1])`，最后一行也能统一处理。

第三，CSR 适合 SpMV，是因为同一行的非零值被连续放在一起了。CPU 和 GPU 在做按行扫描时更容易形成稳定访问模式。

如果对接到 GPU 库，例如 NVPL Sparse，一般会直接把 `m, n, nnz, row_ptr, col_ind, values` 交给 `nvpl_sparse_create_csr()` 之类的接口，让后续稀疏算子直接消费这些数组。这里的关键不是 API 名字，而是你喂给库的数据必须满足它的结构要求：索引合法、同一行内列唯一、最好按序排列。

## 工程权衡与常见坑

稀疏格式不是“压完就一定更快”。真实工程里，最常见的问题不是数学错误，而是“理论省了，实际反而更慢”。

先看常见坑：

| Pitfall | 后果 | 规避措施 |
| --- | --- | --- |
| COO 中同一 `(row, col)` 重复出现 | 重复累加，结果错误或性能差 | 先归约，再转 CSR |
| 行内列无序 | 某些库性能下降，甚至不满足输入要求 | 按 `(row, col)` 排序 |
| 矩阵不够稀 | 索引开销抵消收益 | 先评估 $\frac{nnz}{mn}$ |
| 非结构化稀疏过强 | GPU 访存发散，延迟不降反升 | 改用结构化稀疏或块稀疏 |
| 格式转换过于频繁 | 转换成本吃掉计算收益 | 尽量一次构建，多次复用 |

一个简化流程可以写成：

`数据预处理 -> COO/CSR 构造 -> 稀疏核执行`

### 1. 列重复不是小问题

很多初学者觉得“同一位置出现两次，反正加起来就好了”。问题在于，很多稀疏库要求输入格式本身就是规范化的。如果你把重复列直接交给底层算子：

- 有的库会把它们当作两个独立非零项
- 有的库会要求你自己先做合并
- 有的情况下结果虽对，但性能明显变差

所以工程上更稳妥的规则是：先归约、再排序、再压缩。

### 2. 稀疏不等于一定快

这是最容易踩的坑。特别是在深度学习模型剪枝之后，很多人看到大量权重被置零，就自然以为推理一定会更快。实际上如果零是“元素级、无规则、分布很散”的非结构化稀疏，GPU 上常常拿不到理想收益。

原因很简单：GPU 喜欢规则、连续、可并行批量读取的数据；而不规则稀疏会带来：

- 线程间访问地址不一致
- 缓存命中率下降
- 算子调度碎片化

结果是，虽然理论乘加次数少了，但内存访问和调度成本上升，最终端到端延迟不降反升。

一个真实工程例子是剪枝后的模型部署。假设一个线性层原本是稠密 GEMM，剪枝后 90% 权重为 0。如果只是把权重按普通 CSR 丢给通用稀疏核，可能并不会比高优化的稠密 GEMM 更快。只有在满足下面至少一个条件时，收益才更容易兑现：

- 稀疏率极高
- 模式有结构，例如 block sparse、2:4 structured sparse
- 运行时和硬件专门支持这种结构

### 3. 行压缩适合按行任务，不适合所有任务

CSR 的优势是按行访问快，所以特别适合 SpMV。反过来，如果你的主要操作是按列扫描，CSR 就不自然了。这时可能需要 CSC（Compressed Sparse Column，按列压缩）等格式。

因此，“最优稀疏格式”不是固定答案，而是由主操作类型决定的。你要先问：后续到底是做按行乘法、按列求和，还是做块操作？

## 替代方案与适用边界

稀疏矩阵和压缩存储不是孤立概念，它们在现代模型里经常以“稀疏连接模式”的形式出现。重点不是一定要落成传统 CSR 文件，而是“只保留真正参与计算的连接”。

### 1. Longformer：把全连接注意力改成局部稀疏连接

Longformer 的做法是滑动窗口注意力。每个 token 不再与所有 token 两两交互，而只看附近窗口内的 token，再加少量全局 token。

复杂度从：

$$
O(n^2)
$$

降到：

$$
O(n \cdot w)
$$

其中 `w` 是窗口宽度。

例如 `n = 4096`，`w = 512` 时：

- 全注意力：约 `16.7M` 级别连接
- 滑动窗口：约 `2.1M` 级别连接

白话解释：原来每个人和全班所有人说话，现在每个人只和附近几排说话，再保留少数班干部能全局广播。

这本质上就是一个稀疏矩阵思想：不是所有位置都保留，只保留局部窗口对应的非零模式。

### 2. MoE：不是所有专家都激活

MoE 是 Mixture of Experts，中文常译为“专家混合模型”。它的核心是：模型里有很多专家子网络，但每个 token 只路由到少数几个专家。

门控通常写成：

$$
\pi(x)=softmax(Wx)
$$

其中：

- $x$ 是输入表示
- $W$ 是门控权重
- $\pi(x)$ 是每个专家的概率分布

如果只保留 Top-K 专家，则输出是：

$$
y=\sum_{i\in TopK(\pi)} \pi_i(x) E_i(x)
$$

这里的 `TopK` 意味着只激活概率最高的 `K` 个专家，且 $K \ll E$，`E` 是专家总数。

例如有 64 个专家，每个 token 只激活前 4 个，那么单 token 的专家执行比例是：

$$
\frac{4}{64}=6.25\%
$$

也就是减少了约 93.75% 的专家调用。这个路由矩阵天然是稀疏的：大多数专家权重都被截断为 0。

### 3. 替代方案对比

| 方案 | 核心复杂度 | 适用场景 | 依赖的结构特征 |
| --- | --- | --- | --- |
| COO/CSR 稀疏存储 | SpMV 常见为 $O(nnz)$ | 科学计算、图计算、稀疏线性代数 | 显式非零索引 |
| 滑动窗口注意力 | $O(n \cdot w)$ | 长文本 Transformer | 局部邻域连接 |
| Top-K MoE | 与激活专家数 $K$ 成正比 | 大模型扩容、条件计算 | 门控稀疏激活 |
| 块稀疏/结构化稀疏 | 取决于块大小和模式 | GPU 推理优化 | 规则稀疏模式 |

边界也很清楚：

- 如果你的稀疏模式是规则局部连接，滑动窗口比通用 CSR 更自然。
- 如果你的稀疏模式体现在“只激活少数路径”，MoE 路由比矩阵压缩更关键。
- 如果你的目标是 GPU 上真实提速，结构化稀疏通常比完全不规则稀疏更容易兑现。

所以替代方案不是“谁替代谁”，而是“稀疏思想在不同任务上的不同落地形式”。
