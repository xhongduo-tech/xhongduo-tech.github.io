## 核心结论

MT-Bench 是一个面向**多轮对话**的评估基准。多轮对话，白话说，就是模型回答完第一问后，还要继续接住第二问，不能把前文忘掉。它的核心不是判断“标准答案对不对”，而是让一个强模型充当裁判，检查被测模型在连续追问中的回答质量、指令遵循、上下文保持和任务完成度。

可以把它先压缩成一条流程：

`多轮问题 -> 被测模型生成两轮回答 -> GPT-4 逐轮打分 -> 按题平均 -> 全集平均`

如果第 $i$ 题的两轮得分分别是 $s_{i,1}$ 和 $s_{i,2}$，那么题目分数与总分可以先记住：

$$
q_i = \frac{s_{i,1} + s_{i,2}}{2}, \qquad
S = \frac{1}{N}\sum_{i=1}^{N} q_i
$$

其中 $N=80$，也就是 80 个双轮问题。

它最适合做两件事：

| 用途 | 为什么适合 |
|---|---|
| 比较两个聊天模型谁更稳 | 因为会把“首轮强、追问崩”的差异放大出来 |
| 做模型迭代的回归监控 | 因为流程固定，便于看新版本是否退化 |

玩具例子：某模型第一轮能把排序算法讲清楚，第二轮用户追问“那你把空间复杂度也补上，并和快速排序对比”，模型却开始重复第一轮内容，或者忘了用户要求对比对象。单轮测试里它可能显得不错，但在 MT-Bench 里第二轮会直接掉分，最后总分明显下降。

真实工程例子：你在做客服模型。第一轮用户问“7 天无理由退货怎么走流程”，模型答得完整；第二轮用户补一句“订单号是 A123，商品已拆封”，模型却继续按“未拆封”的默认情况回答。这类错误本质上不是知识缺失，而是**对话状态没有维护住**。MT-Bench 正是用来暴露这种问题的。

## 问题定义与边界

MT-Bench 的评测对象不是“80 个孤立题目”，而是 **80 个双轮问题**。双轮问题，白话说，就是每道题都至少包含两次用户输入，第二轮往往依赖第一轮上下文。因此也可以把它看成 **160 个 turn-level prompts**。这里最容易混淆：

- `80 个双轮问题`：按“对话题目”计数
- `160 个 turn-level prompts`：按“每一轮用户输入”计数

这两种说法都能见到，但工程上真正拿来汇总的是前者，即 80 个对话单元。

题目覆盖 8 类常见聊天任务，每类 10 个双轮问题：

| 类别 | 在评什么 | 一个直观例子 |
|---|---|---|
| Writing | 写作组织能力 | 改写一段说明文，再按另一种语气重写 |
| Roleplay | 角色扮演一致性 | 扮演老师，第二轮继续沿用设定回答 |
| Extraction | 信息抽取 | 从文本里抽字段，第二轮再补充新约束 |
| Reasoning | 推理链条 | 先判断，再解释边界条件 |
| Math | 数学求解 | 第一轮算结果，第二轮换条件重算 |
| Coding | 编程能力 | 写函数后再修 bug 或补测试 |
| Knowledge I (STEM) | 理工知识 | 解释技术概念并继续追问机制 |
| Knowledge II | 人文社科知识 | 比较观点并结合上下文扩展 |

它适合评的是“聊天模型在连续对话里的表现”，不适合替代封闭式、标准答案型基准。边界可以直接看下表：

| 适合评什么 | 不适合评什么 | 原因 |
|---|---|---|
| 多轮追问是否稳定 | 单题标准答案准确率 | 它的目标是对话质量，不是客观题判分 |
| 指令遵循和上下文保持 | 代码必须可编译通过的硬约束 | GPT-4 裁判更擅长质量判断，不等于执行器 |
| 开放式任务的综合表现 | 某学科知识点的细粒度覆盖率 | 题量有限，覆盖广但不深 |
| 版本回归比较 | 绝对真值结论 | 分数依赖 judge 和 prompt 设定 |

所以，MT-Bench 不能替代 MMLU、HumanEval。MMLU 更像学科考试，HumanEval 更像带单元测试的编程题；MT-Bench 更像“连续面试追问”。

## 核心机制与推导

MT-Bench 的默认流程是 **single-answer grading**。single-answer grading，白话说，就是不拿两个模型当场对打，而是让 GPT-4 单独看某个模型的回答，直接给分。README 中推荐它作为默认模式，分数范围是 $0$ 到 $10$。

形式化写法如下：

$$
s_{i,t} \in [0,10]
$$

$$
q_i = \frac{s_{i,1} + s_{i,2}}{2}
$$

$$
S = \frac{1}{N}\sum_{i=1}^{N} q_i
$$

含义很直接：

- $s_{i,t}$：第 $i$ 题第 $t$ 轮的得分
- $q_i$：第 $i$ 题的平均分
- $S$：80 题的总平均分

玩具例子先算一遍。某题两轮分别得 `8` 和 `6`，那么：

$$
q_i = \frac{8+6}{2}=7.0
$$

如果只有 3 题，它们题目分数分别是 `7.0、9.0、5.0`，那么总分：

$$
S=\frac{7+9+5}{3}=7.0
$$

这个公式看上去简单，但它隐含了 MT-Bench 的评估重点：**第二轮不是附属项，而是和第一轮等权**。因此，一个模型哪怕首轮很强，只要追问阶段明显退化，整体分数就会被拉下来。

除了单答打分，它还支持 pairwise 模式。pairwise，白话说，就是把两个模型的答案摆在一起，让裁判判断谁更好。两种模式差异如下：

| 模式 | 输出形式 | 优点 | 局限 |
|---|---|---|---|
| 单答打分 | 0-10 分数 | 成本低，适合持续回归 | 受 judge 标尺影响更明显 |
| pairwise 胜率 | 胜/负/平或胜率 | 两模型对比更直观 | 成本更高，模型数多时更贵 |

真实工程里，这两种模式常对应两类需求：

- 你要做 nightly 回归监控，单答打分更省成本。
- 你要比较“新版本是否真的赢过旧版本”，pairwise 更直观。

还要注意，论文强调的是 **LLM-as-a-judge** 的可扩展性和较高一致性，不是“裁判永远正确”。论文报告 GPT-4 judge 与人工偏好达到超过 80% 的一致率，但这只能说明“足够有用”，不能说明“完全等同人工”。

## 代码实现

工程上理解 MT-Bench，最简单的方式不是读整套代码，而是记住三段流水线：

`gen_model_answer.py -> gen_judgment.py -> show_result.py`

对应职责如下：

| 脚本 | 负责什么 |
|---|---|
| `gen_model_answer.py` | 让被测模型回答 MT-Bench 问题 |
| `gen_judgment.py` | 调用 GPT-4 按 judge prompt 给每轮回答打分 |
| `show_result.py` | 汇总题目分、总分和类别结果 |

如果你自己复现，最关键的不是“改评分公式”，而是保证以下条件一致：

| 需要固定的东西 | 为什么 |
|---|---|
| 题目集 | 不同题集分数不可比 |
| judge prompt | 裁判标准变了，分数口径就变了 |
| judge 模型 | GPT-4 和别的 judge 不是同一标尺 |
| 采样设置 | 温度、top-p 会改变回答分布 |

下面是一个最小可运行示例，演示题目分和总分如何汇总：

```python
def score_question(turn_scores):
    assert len(turn_scores) > 0
    assert all(0 <= s <= 10 for s in turn_scores)
    return sum(turn_scores) / len(turn_scores)

def score_benchmark(question_scores):
    assert len(question_scores) > 0
    return sum(question_scores) / len(question_scores)

# 玩具例子：一题两轮分别 8 分和 6 分
q1 = score_question([8, 6])
assert q1 == 7.0

# 三题总分：7.0, 9.0, 5.0
overall = score_benchmark([7.0, 9.0, 5.0])
assert overall == 7.0

# 一个更接近 MT-Bench 的小例子
questions = [
    [8, 7],   # 写作
    [9, 8],   # 推理
    [7, 5],   # 编程：第二轮退化
]
question_scores = [score_question(x) for x in questions]
assert question_scores == [7.5, 8.5, 6.0]
assert score_benchmark(question_scores) == (7.5 + 8.5 + 6.0) / 3
```

这段代码只做了最核心的数学汇总。真正的 FastChat 实现还会处理：

- 问题读取
- 模型推理
- judge prompt 构造
- 结果写入 `jsonl`
- 按类别与按 turn 的统计展示

真实工程例子：你在做一个内部问答助手的新版本评估。上线前先跑一遍旧模型和新模型的 MT-Bench。结果总分只提升 `0.1`，看起来变化不大；但拆开后发现新模型在 `Coding` 第一轮提升很多，第二轮却下降。这说明新模型更会“开头写方案”，但在追问修订时更容易失稳。这个信息单靠一个总分看不出来。

## 工程权衡与常见坑

MT-Bench 好用，但工程上非常容易被“表面分数”误导。常见坑如下：

| 坑点 | 后果 | 规避方式 |
|---|---|---|
| 混淆 `80` 和 `160` | 报告里口径不一致，团队理解错 benchmark 大小 | 明确写“80 个双轮问题，160 个 turn-level prompts” |
| 只看总分 | 看不出退化发生在哪一类、哪一轮 | 同时看 8 类别分和 1st/2nd turn 分 |
| 过度相信 judge | 把自动评测当成绝对真值 | 结合人工抽检、业务集和线上反馈 |
| 忽略采样设置 | 新旧结果不可比 | 固定温度、top-p、max tokens、judge 配置 |

其中最值得展开的是“过度相信 judge”。MT-Bench 的核心价值是把人工偏好评估自动化，但它依赖三个前提：

- judge prompt 固定
- 评判模型固定
- 采样设置固定

任何一个变化，都可能改变分数标尺。所以 MT-Bench 更适合**相对比较**，例如比较 `v1.2` 和 `v1.3`；不适合直接说“7.8 分就是行业绝对优秀线”。

另一个常见误判是“总分变化不大，所以模型没问题”。这在多轮对话里尤其危险。举个真实工程风格的例子：

- 新模型在 `Writing`、`Knowledge` 上提升
- 但在 `Extraction`、`Coding` 的第二轮明显下降

如果你只看总体平均，可能只是从 `7.4` 变成 `7.3`，像轻微波动；但拆成类别和 turn 后，会发现退化集中在“需要记住上下文并继续操作”的任务上。这往往意味着 SFT 数据混入了更多泛化问答样本，却削弱了多轮状态维护能力。

因此，读 MT-Bench 结果时建议最少看三层：

| 观察层级 | 要回答的问题 |
|---|---|
| 总分 | 整体变好还是变差 |
| 类别分 | 哪类能力发生变化 |
| turn 分 | 退化发生在首轮还是追问轮 |

## 替代方案与适用边界

MT-Bench 不是万能评测，它和其他基准更像分工协作。

| 方案 | 看什么 | 优点 | 局限 |
|---|---|---|---|
| MT-Bench | 多轮对话质量 | 能看上下文保持和追问稳定性 | 依赖 judge，题量有限 |
| Chatbot Arena | 真实用户偏好 | 更接近开放场景使用感受 | 噪声更大，复现成本高 |
| MMLU / HumanEval | 标准题能力 | 可比性强，客观性更高 | 不擅长测多轮对话 |
| pairwise winrate | 两模型相对优劣 | 对 A/B 比较直观 | 扩展到多模型时成本高 |

适用边界可以概括成一句话：**如果你关心模型会不会在追问里失稳，用 MT-Bench；如果你关心标准答案正确率，用 MMLU 或 HumanEval；如果你关心真实用户更喜欢谁，用 Arena。**

一个完整的发布前评测组合通常是这样的：

- 用 MT-Bench 看多轮追问稳定性
- 用 Chatbot Arena 或内部 A/B 看真实偏好
- 用业务测试集看领域正确率
- 用 HumanEval 一类基准看代码任务硬能力

这样组合比“只看一个分数”可靠得多。

如果你只是在两个版本之间做快速比较，pairwise 模式很适合，因为结论直接是“新模型赢了还是输了”。如果你要长期跑周报、日报、CI 回归，single-answer grading 更高效，因为它输出统一标尺的分数，便于画趋势图。
