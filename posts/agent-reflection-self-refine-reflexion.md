## 核心结论

`Self-Refine` 和 `Reflexion` 都属于反思式代理方法。反思式代理，指系统不把第一次输出当最终答案，而是把失败、批评或环境反馈重新变成下一步输入。

它们解决的是两个不同层级的问题：

- `Self-Refine` 处理单次任务内部的修订问题，公式可以写成：`生成 -> 批评 -> 修订`
- `Reflexion` 处理多次任务之间的经验复用问题，公式可以写成：`失败 -> 反思 -> 记忆 -> 下次决策`

对白话一点的理解：

- `Self-Refine` 像“这道题刚做完，立刻检查并改答案”
- `Reflexion` 像“这次错题总结写进错题本，下次先看再做”

两者最重要的价值，不是让模型“多想一轮”，而是让失败信息变成结构化输入，推动下一次输出真正发生变化。如果没有反馈回流，额外轮数通常只会增加成本，不一定提升正确率。

| 维度 | Self-Refine | Reflexion |
|---|---|---|
| 作用范围 | 单个任务内 | 多个 episode 之间 |
| 反馈去向 | 直接作用于当前答案 | 写入记忆供后续任务使用 |
| 是否跨回合复用 | 通常不复用 | 核心就是复用 |
| 典型输入 | 当前问题、当前答案、当前批评 | 当前任务、历史记忆、环境反馈 |
| 典型输出 | 更好的当前答案 | 新的反思记忆和下一次更稳的策略 |

## 问题定义与边界

先定义几个符号：

| 记号 | 含义 |
|---|---|
| `x` | 输入任务，比如一个问题或代码需求 |
| `y_t` | 第 `t` 轮输出 |
| `f_t` | 第 `t` 轮批评或反馈 |
| `m^e` | 第 `e` 个 episode 的反思记忆 |
| `M_e` | 在第 `e` 次任务前可用的记忆集合 |

这里的 episode 可以理解为“一次完整尝试”。例如“做一道代码题并提交测试”就是一个 episode。

这两类方法关注的核心不是“模型要不要更长思考”，而是“错误信息能不能被保留下来，并对后续决策产生约束”。如果反馈不能验证、不能定位问题，反思就很容易退化成空话。

适合场景和不适合场景可以先划清边界：

| 类型 | 适合 | 不适合 |
|---|---|---|
| 任务特征 | 有明确反馈、允许多轮、结果可验证 | 反馈稀疏、结果难验证、一次性低成本 |
| 典型任务 | 代码生成、问答纠错、格式化输出、工具调用 | 开放式闲聊、单次低风险文案、无客观标准创作 |
| 主要收益 | 提升正确率、减少重复犯错 | 往往只增加延迟和 token 成本 |

一个常见误解是：只要加“请再检查一遍”，系统就等于有反思能力。这个理解不准确。真正有效的反思至少要满足两个条件：

1. 反馈要指向具体错误。
2. 反馈要能改变下一步输入。

如果任务是“写一段代码并修到能跑”，`Self-Refine` 适合在当前会话里不断修；如果任务是“连续做很多相似代码题”，`Reflexion` 更适合把上次失败原因保存下来，减少下一题重复踩坑。

## 核心机制与推导

`Self-Refine` 的闭环是单任务闭环。先产出初稿，再生成批评，再据此修订：

$$
y_0 = \pi(x)
$$

$$
f_t = Critique_\pi(x, y_{t-1})
$$

$$
y_t = Refine_\pi(x, y_{t-1}, f_t), \quad t=1,\dots,T
$$

停止条件通常是：`f_t` 已经没有新信息、答案不再变化、或外部评分达到阈值。

`Reflexion` 的闭环是跨任务记忆闭环。重点不在当前答案，而在“这次失败能不能沉淀成以后可检索的经验”：

$$
a^e = Actor_\pi(x^e, M_e)
$$

$$
r^e = Env(a^e)
$$

$$
m^e = Reflect_\pi(x^e, a^e, r^e)
$$

$$
M_{e+1} = Prune(M_e \cup \{m^e\})
$$

这里的 `Prune` 表示裁剪。裁剪就是删掉重复、错误、过时的记忆，否则记忆会污染后续决策。

可以把两者的流程简化成下面这张图：

| 阶段 | Self-Refine：单任务闭环 | Reflexion：跨 episode 闭环 |
|---|---|---|
| 1 | 生成当前答案 | 带着历史记忆执行当前任务 |
| 2 | 对当前答案做批评 | 从环境拿到成功或失败反馈 |
| 3 | 修订当前答案 | 生成一条反思总结 |
| 4 | 若未达标继续修 | 把总结写入记忆库 |
| 5 | 输出当前最好答案 | 下次任务先读取记忆再决策 |

玩具例子最容易看出差异。

任务：回答“法国首都是哪里？”

- 第 0 轮输出：`柏林`
- 批评：`国家和首都对应错误，应检查地名归属`
- 第 1 轮修订：`巴黎`

这就是 `Self-Refine`。

如果是 `Reflexion`，这次失败还会额外沉淀一条记忆：

- `先核对国家-首都映射，再给最终答案`

以后再遇到“意大利首都是什么”之类的问题，系统会先读到这条记忆，再生成答案。前者修当前题，后者防下次再错。

## 代码实现

实现 `Self-Refine` 的关键，不是把 prompt 写得多优美，而是把循环控制写清楚：生成、评估、修订、停止。

下面是一个可运行的最小 Python 玩具实现。它没有调用大模型，只是模拟“错误答案 -> 批评 -> 修订”的流程。

```python
def generate_answer(question: str) -> str:
    # 故意先返回错误答案，模拟初稿质量不足
    if "法国首都" in question:
        return "柏林"
    return "未知"

def critique(question: str, answer: str) -> str:
    if "法国首都" in question and answer != "巴黎":
        return "国家和首都对应错误，应改为巴黎"
    return ""

def refine(question: str, answer: str, feedback: str) -> str:
    if "改为巴黎" in feedback:
        return "巴黎"
    return answer

def self_refine(question: str, max_rounds: int = 3) -> str:
    answer = generate_answer(question)
    for _ in range(max_rounds):
        feedback = critique(question, answer)
        if not feedback:
            break
        new_answer = refine(question, answer, feedback)
        if new_answer == answer:
            break
        answer = new_answer
    return answer

result = self_refine("法国首都是哪里？")
assert result == "巴黎"
print(result)
```

这段代码体现了一个最小原则：批评必须能落到可执行修改，否则循环不会产生有效变化。

如果把它扩展到 `Reflexion`，重点就从“修当前答案”变成“管理记忆”：

```text
M = load_memory()
a = actor(x, M)
r = env.step(a)
m = reflect(x, a, r)
M = prune_and_store(M, m)
save_memory(M)
return a, r
```

真实工程里，最常见的例子不是常识问答，而是“代码生成 + 单测修复”：

| 步骤 | 输入 | 输出 |
|---|---|---|
| 生成代码 | 需求说明 | 初版函数 |
| 外部验证 | 单测、lint、编译器 | 失败日志 |
| Self-Refine | 代码 + 失败日志 | 修订代码 |
| Reflexion | 任务过程 + 失败原因 | 一条经验记忆 |
| 下次任务 | 新需求 + 检索到的记忆 | 更稳的首版实现 |

比如一个模型反复在列表边界条件上出错。`Self-Refine` 会在当前任务里根据 `IndexError` 修掉；`Reflexion` 会把“先判空、再访问下标”写进记忆库，供下一道相似题复用。这个差别决定了它们在工程系统中的部署位置也不同：前者更像局部修补器，后者更像经验层。

## 工程权衡与常见坑

工程上最重要的一条判断是：反馈质量决定上限。没有高质量反馈，反思模块大多只是形式正确。

常见坑可以直接列出来：

| 坑点 | 现象 | 后果 |
|---|---|---|
| 反馈太空 | 只说“再好一点” | 模型不知道改什么 |
| 自评偏差 | 模型自己说自己对 | 自洽但不准确 |
| 迭代过多 | 修很多轮 | 成本上升，收益递减 |
| 记忆污染 | 旧经验错误、重复、过时 | 后续决策被误导 |
| 过拟合评估器 | 专门迎合分数规则 | 真实任务未必更好 |

对应的规避策略也很明确：

| 问题 | 更稳的做法 |
|---|---|
| 反馈太空 | 用编译错误、单测失败、规则校验替代泛化批评 |
| 自评偏差 | 接入外部评分器或程序化验证器 |
| 迭代过多 | 通常把最大轮数设为 `T=2~4` |
| 记忆污染 | 做去重、时间衰减、任务分桶、人工抽检 |
| 只优化表面指标 | 同时看线上成功率、人工验收、真实业务指标 |

这里有一个关键权衡：`Self-Refine` 往往立竿见影，但它的风险是“局部修得很像对，其实仍然错”；`Reflexion` 的长期价值更大，但风险是“把错误经验记住了”。后者一旦记忆污染，问题会被放大，因为坏经验会反复参与后续决策。

所以工程上通常不会只依赖模型自我批评，而会尽量接入外部可验证信号，比如：

- 单元测试
- JSON Schema 校验
- API 返回码
- 检索结果一致性检查
- 人工规则或奖励模型

## 替代方案与适用边界

`Self-Refine` 和 `Reflexion` 不是通用最优解。很多任务里，外部验证、检索增强、规划器或工具调用比“让模型反思自己”更稳。

可以先看一个总表：

| 方案 | 优点 | 缺点 | 适用任务 |
|---|---|---|---|
| Self-Refine | 实现简单，适合局部修订 | 容易陷入自评偏差 | 单任务纠错、答案打磨 |
| Reflexion | 能积累经验，适合重复任务 | 需要记忆治理 | 连续 episode、相似任务复用 |
| 检索增强 RAG | 能补充外部事实 | 依赖检索质量 | 知识密集型问答 |
| 程序化校验 | 可验证性强、稳定 | 只能覆盖可形式化规则 | 代码、结构化输出、配置生成 |
| 搜索式规划 | 对复杂决策更稳 | 系统复杂度更高 | 多步工具调用、长链任务 |

一个简单的选型逻辑是：

| 判断问题 | 更合适的方向 |
|---|---|
| 当前任务能否客观验证？ | 能，就优先接外部验证 |
| 只需要修这一次答案吗？ | 是，优先 `Self-Refine` |
| 后续还会重复做相似任务吗？ | 是，考虑 `Reflexion` |
| 错误主要来自知识缺失吗？ | 是，先考虑检索增强 |
| 错误主要来自执行顺序复杂吗？ | 是，先考虑规划或搜索 |

真实系统里，最稳的做法常常不是二选一，而是组合：

- 用检索补事实
- 用工具执行动作
- 用程序校验对错
- 用 `Self-Refine` 修当前结果
- 用 `Reflexion` 存跨任务经验

因此，这两类方法真正适用的边界可以概括为：当任务存在明确反馈，而且你希望系统从反馈中持续改变行为时，它们才有明显价值。否则，直接做一次生成往往更划算。
