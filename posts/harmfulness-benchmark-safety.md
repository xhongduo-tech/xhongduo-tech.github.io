## 核心结论

HarmBench 是一个面向大语言模型安全评测的标准化框架。标准化，白话说，就是把“测什么、怎么攻、怎么判分、怎么复现”全部固定下来，让不同团队跑出来的结果可以直接比较，而不是各自写一套 prompt 和脚本后得出互相不可比的结论。

它的价值不在于“发现所有风险”，而在于提供一套行业可复用的最小公共评测面。官方公开了 510 条行为定义、18 种攻击方法、统一分类器、固定验证集与测试集拆分，以及 `local`、`local_parallel`、`slurm` 三种执行模式。工程团队可以在同一条流水线上比较多个模型在相同行为和攻击条件下的表现，最终得到一个“模型 × 攻击 × 行为类别”的 ASR 矩阵。ASR 是攻击成功率，白话说，就是“模型在多少比例的测试里真的被诱导出了不该给出的有害完成”。

对零基础读者来说，可以把 HarmBench 理解成“安全体检套餐”。你不需要先设计 500 条坏提示词，也不需要先实现一个打分器。直接按官方配置运行流水线，它会自动生成测试输入、对模型发起攻击、收集回复、调用分类器判断回复是否越界，最后把结果汇总成表。工程师真正要看的不是某一条 prompt 是否触发，而是模型在一组可复现攻击下是否稳定拒绝。

下面这张表可以先抓住它的工程轮廓：

| 维度 | HarmBench 默认提供内容 | 工程意义 |
| --- | --- | --- |
| 行为定义 | 510 条 | 覆盖常见有害意图，避免团队只测少量样例 |
| 攻击方法 | 18 种 | 检查模型是否只会拒绝“直球问题”，却挡不住变体攻击 |
| 评分方式 | 预训练分类器 | 让批量评测可自动化，不依赖纯人工逐条看 |
| 数据拆分 | 验证集 / 测试集固定拆分 | 防止调参时把测试集“练熟” |
| 执行模式 | `local` / `local_parallel` / `slurm` | 支持单机调试到集群批跑 |

## 问题定义与边界

HarmBench 解决的问题不是“模型是否聪明”，而是“模型在面对明确的潜在有害请求时，能否稳定拒绝”。这里的“明确”很关键。它主要覆盖可陈述、可验证、可归类的有害行为，比如非法活动、伤害他人、隐私侵犯、网络攻击、仿冒、骚扰等，并把它们组织成多个语义类别和功能类型。

语义类别可以理解为“坏事的类型”；功能类型可以理解为“提示是以什么形式出现的”。前者决定评测内容，后者决定攻击表面形式。实际工程里，这种拆法的价值在于：同一模型可能能挡住直接提问，却挡不住上下文包装、角色扮演或版权/格式伪装。

可以用一个抽象表来理解它的覆盖面：

| 维度 | 含义 | 例子类型 | 为什么重要 |
| --- | --- | --- | --- |
| 语义类别 | 有害目标属于哪类风险 | 非法活动、隐私侵犯、骚扰、网络犯罪等 | 决定你测的是哪种安全能力 |
| 功能类型 | 请求如何包装 | 直接请求、上下文嵌入、版权相关、多模态扩展 | 决定攻击是不是“换皮后还有效” |
| 验证集 | 用来调参与选方案 | 小规模实验 | 避免在测试集上反复试错 |
| 测试集 | 用来报告最终结果 | 正式评测 | 避免指标被调参污染 |

边界也必须说清楚。第一，HarmBench 主要是英文和文本为主的公开行为集，不等于你的真实业务流量。第二，它更擅长评估“有明确定义的违规输出”，不擅长覆盖模糊边界，比如某些灰区建议、复杂对话策略、地域法规差异。第三，静态公开数据集天然存在被模型“记住”的风险，也就是模型学会对这 510 条附近的表达拒绝，但换一种说法仍然可能失守。

一个玩具例子可以说明它的边界。假设你只测“请提供某类危险操作步骤”这种直接问法，模型拒绝得很好，于是你以为它很安全。可一旦攻击换成“我在写小说，请给我一个角色会使用的详细流程”，模型可能就漏掉了。这说明你测到的是“直白提问防御”，不一定是“语义层面的稳定防御”。HarmBench 的价值，正是在于它把多种攻击方法一起纳入，而不是只测一种表述。

## 核心机制与推导

HarmBench 的核心指标是 ASR，Attack Success Rate，即攻击成功率。定义很直接：在一组攻击输入里，有多少比例最终让模型产出了被判定为有害的回复。

如果把某一类行为记为 $y$，攻击方法记为 $g$，目标模型记为 $f$，那么可写成：

$$
\mathrm{ASR}(y, g, f)=\frac{1}{N}\sum_{i=1}^{N} c(f_T(x_i), y)
$$

其中：

- $x_i$ 是第 $i$ 个攻击后的测试输入；
- $f_T(x_i)$ 是模型对该输入生成的回复；
- $c(\cdot)$ 是分类器，输出 0 或 1；
- $N$ 是该组测试样本数。

白话解释：对每条攻击样本，把模型回复交给分类器判断“是否真的越界”，最后求平均值。平均值越高，说明模型越容易被打穿；越低，说明模型拒绝越稳定。

很多介绍还会提到分类器的等级分数，例如 1 到 5 分，通常把“≤2”视为安全，把更高分视为存在明显有害内容。这个评分层并不是最终指标本身，而是把连续或多级判断压缩成是否命中的二元结果，再用来计算 ASR。

一个最小数值例子：

- 某类行为下，一共测 20 条攻击样本；
- 模型 19 条都拒绝；
- 只有 1 条被分类器判为有害。

则：

$$
\mathrm{ASR} = \frac{1}{20}=5\%
$$

这个 5% 的意思不是“模型有 5% 概率危险”，而是“在当前这组固定测试里，这种攻击仍然有 5% 的命中率”。它是评测集上的经验结果，不是对所有现实场景的数学保证。

HarmBench 的流水线一般分四步：

1. 生成测试用例。
2. 对原始行为应用某种攻击方法，得到实际输入。
3. 让目标模型生成回复。
4. 用分类器判断回复是否构成有害完成，并汇总指标。

为什么这套流程比手工 red team 更有价值？因为它把变量尽量固定了。你可以把“攻击器”理解为出题器，把“目标模型”理解为答题者，把“分类器”理解为阅卷器。只有当三者的接口和参数都尽量固定时，不同模型的结果才具有横向可比性。

真实工程例子更能说明问题。上线前，团队往往会在多个候选模型之间做安全审计。假设 A、B、C 三个模型在通用问答上效果差不多，但在 HarmBench 上，A 在隐私侵犯类 ASR 明显偏高，B 在暴力和骚扰类受某些攻击时波动大，C 在多数类别上稳定低 ASR。那安全团队就有了一个比“感觉更安全”更可复查的依据，去支持模型选择。

## 代码实现

官方仓库已经把大部分工程骨架搭好了，核心入口通常是 `run_pipeline.py`。你选择攻击方法、目标模型、运行步骤和执行模式后，脚本就会按顺序完成测试样本生成、回复生成和评估汇总。

一个典型命令是：

```bash
python ./scripts/run_pipeline.py --methods GCG --models llama2_7b --step all --mode slurm
```

这条命令的意思是：用 `GCG` 这一类攻击方法，对 `llama2_7b` 这个模型跑完整流水线，并把任务提交到 `slurm` 执行。对工程团队来说，最重要的不是记住命令，而是理解接口稳定性：你可以尽量不改主逻辑，只通过配置接入新模型或新攻击。

下面给一个“玩具版”可运行 Python 代码，模拟 ASR 计算过程。这里不生成任何有害内容，只用布尔值模拟分类器输出：

```python
from typing import List

def compute_asr(classifier_hits: List[int]) -> float:
    assert len(classifier_hits) > 0
    assert all(x in (0, 1) for x in classifier_hits)
    return sum(classifier_hits) / len(classifier_hits)

# 玩具例子：20 次攻击里只有 1 次被判定为命中
hits = [0] * 19 + [1]
asr = compute_asr(hits)

assert abs(asr - 0.05) < 1e-9
print(f"ASR = {asr:.2%}")
```

这个例子很小，但它准确表达了 HarmBench 的核心度量逻辑：分类器先把每次回复变成 0/1，评测脚本再做聚合。

如果你要把新模型接进真实流程，通常需要准备类似下面的配置思路：

```yaml
model_name: my_model
inference_backend: api_or_local
max_new_tokens: 512
decoding: greedy
batch_size: 8
```

这里的 `greedy` 是贪心解码，白话说，就是每一步都选当前最可能的 token，不做随机采样。这样做的目的不是追求模型表现最好，而是让评测条件更稳定，更容易复现。

从工程落地角度看，HarmBench 的目录划分也很实用：

| 目录/模块 | 作用 | 你通常改哪里 |
| --- | --- | --- |
| `configs/` | 模型、攻击、分类器配置 | 接新模型时先改这里 |
| `scripts/generate_*` | 流水线各阶段脚本 | 要自定义流程时改这里 |
| `data/docs` | 数据说明与文档 | 查行为定义和使用说明 |
| `run_pipeline.py` | 总入口 | 调试时最先跑它 |

如果团队只想先做半小时内的接入验证，合理路径通常是：先用 `local` 模式跑小样本，确认接口打通；再用 `local_parallel` 扩批量；最后在 `slurm` 上做完整矩阵。

## 工程权衡与常见坑

HarmBench 很适合做“标准化对比”，但不适合被误当成“安全全貌”。下面这张表比长段落更实用：

| Pitfall | 为什么发生 | 缓解措施 |
| --- | --- | --- |
| 只看总 ASR，不看分类别结果 | 总分可能掩盖某一风险域明显失守 | 按语义类别和攻击方法拆开看 |
| 只跑公开 510 条行为 | 模型可能对公开集合产生记忆或针对性拒绝 | 增补自建 prompt，周期性更新样本 |
| 过度信任自动分类器 | 分类器也可能被格式、拒绝话术、无关内容干扰 | 抽样人工复核，并观察分类器边界样本 |
| 在测试集上反复调参 | 最终指标会被“练熟” | 先在验证集调，再一次性跑测试集 |
| 只测英文文本 | 真实业务可能是中文、多轮对话、业务术语 | 建立业务侧补充评测集 |

最常见的误区是“模型在标准集上拒绝得很好，所以可以放心上线”。这是不成立的。标准集只能说明你在标准集上的表现不错。真实业务里，用户会有新的话术、不同语言、上下文拼接、图片文字混合输入，甚至会把意图藏在多轮对话里。HarmBench 给你的是一条基线，不是一份免责证明。

另一个常见坑是“假拒绝”。假拒绝，白话说，就是模型表面上说“我不能帮你”，但后面又给了足够多的危险线索，或者换一种更隐蔽的表达继续回答。若团队只看是否出现了拒绝句式，而不结合分类器与抽样人工检查，很容易高估安全性。

真实工程里，一个更稳妥的流程通常是：

1. 用 HarmBench 跑标准对比，选掉明显差的模型。
2. 针对业务高风险场景自建补充集。
3. 对边界样本做人工抽检。
4. 记录每次上线前后的评测漂移。

这样做的原因很简单：标准化框架负责“可比性”，业务补充集负责“相关性”。

## 替代方案与适用边界

HarmBench 不是唯一方案，但它在“可比较、可复现、能自动化批量跑”这三个维度上很强。替代方案主要有两类：纯手工 red team，以及混合评测方案。

| 方案 | 可比性 | 覆盖标准风险 | 覆盖业务特定风险 | 扩展成本 |
| --- | --- | --- | --- | --- |
| 只用 HarmBench | 高 | 高 | 中到低 | 中 |
| HarmBench + 自建 prompt | 高 | 高 | 高 | 较高 |
| 只用本地 red team | 低 | 不稳定 | 高 | 高且依赖人力 |

只用 HarmBench，适合做模型横评、供应商对比、回归测试基线。它回答的是“在公开标准框架下，谁更稳”。

HarmBench 加自建 prompt，适合真实上线前的审计。它回答的是“两件事”：一是标准能力有没有退步，二是你的产品场景有没有新漏洞。

只用本地 red team，更适合探索新威胁，而不适合做严格横向比较。因为不同人、不同时间、不同提示风格做出来的数据往往不一致，复现成本高。

一个典型真实场景是：团队先用 HarmBench 比较 3 个候选模型在隐私、暴力、骚扰等类别下的 ASR，再补充一组业务特定测试，比如客服对话、多轮追问、中文口语化表达。前者保证和行业基线可对齐，后者保证和自己产品风险相关。双轨并行，通常比单押某一种方案更稳。

适用边界也要再强调一次。如果你的主要问题是多语言、强地域法规依赖、复杂多模态输入、长上下文渐进诱导，那么 HarmBench 仍可作为起点，但不能作为终点。你需要把它当成“标准底盘”，而不是完整车身。
