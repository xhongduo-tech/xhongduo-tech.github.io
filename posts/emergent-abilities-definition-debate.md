## 核心结论

“涌现能力”这个词在大模型讨论里很常见，但它最初不是泛指“模型变强了”，而是一个更严格的说法。Wei 等人在 2022 年提出：如果某种任务能力在小模型上看不到、在更大模型上突然出现，而且这种出现无法用小模型阶段的趋势直接外推，那么这类现象可以叫作涌现能力。

这个定义的关键不在“性能提高”，而在“像开关一样从没有到有”。换句话说，它把能力看成近似离散的“存在/不存在”，而不是连续分数的缓慢上升。

争议也正出在这里。Schaeffer 等人在 2023 年指出，很多所谓“突然出现”，可能不是模型内部真的发生了断层式变化，而是评测方式把连续进步压成了二值结果。最典型的情况是：模型真实能力分数 $s(\theta)$ 随参数规模 $\theta$ 平滑增长，但评测指标只在超过阈值 $\phi$ 时记为成功，即

$$
M(s)=\mathbb{1}[s\ge \phi]
$$

这时，只要小模型的 $s$ 略低于阈值、大模型的 $s$ 略高于阈值，外部曲线就会从 0 突然跳到 1，看起来像“突然学会了”，但底层改进其实可能一直是连续的。

可以先看一个玩具例子。假设模型回答某题的正确概率从 0.48 提高到 0.52。如果评测是“是否完全正确”，那么结果会从失败变成成功，像是能力突然出现；但如果看编辑距离、负对数似然或 token 级别概率，你会发现它只是跨过了一个人为阈值。

因此，讨论涌现能力时，真正需要问的不是“曲线有没有跳”，而是“跳跃来自模型机制，还是来自度量函数”。

## 问题定义与边界

先把术语说清楚。

“潜在能力分数”指模型在任务上的连续表现强度，白话说，就是它离真正会做这类题还有多近。这个分数不一定直接被用户看到，但可以用平均 log-prob、编辑距离、负对数损失等连续指标近似。

“观测指标”指最终汇报给人的评测数值，白话说，就是榜单上显示的那个成绩。它可能是连续的，也可能是离散的。精确匹配（exact match）就是离散指标：答案必须完全一样才算对。

Wei 等人的定义关注的是一种经验现象：

1. 小模型阶段，该任务几乎不表现出来。
2. 随着模型规模变大，某一点之后表现突然显著提高。
3. 这种变化很难从小模型区间直接预测。

如果用符号写，可以把模型规模记作 $\theta$，潜在能力记作 $s(\theta)$。若 $s(\theta)$ 平滑增长，但我们只通过阈值函数 $M(s)=\mathbb{1}[s\ge\phi]$ 观察它，那么只要存在

$$
s(\theta_{\text{small}})<\phi<s(\theta_{\text{large}})
$$

就会看到“前面全不会，后面突然会”的现象。

这里有一个重要边界：不是所有性能上升都能叫涌现。若指标本身就是连续的，比如平均编辑距离从 5、4、3、2、1 稳定下降，那更准确的说法是“持续改进”，而不是“突然出现”。

再看一个最小数值例子：

| 模型规模 | 潜在分数 $s$ | 阈值评测 $M(s)$ | 编辑距离 |
|---|---:|---:|---:|
| 小模型 | 0.48 | 0 | 2 |
| 大模型 | 0.52 | 1 | 1 |

如果只看 $M(s)$，你会说“大模型突然学会了”；但如果看编辑距离，就更像“答案越来越接近了”。这说明“是否涌现”部分取决于你用什么尺子量。

所以本文的讨论边界很明确：我们关注的是“能力的原始定义”和“这个定义是否会被评测指标放大”，而不是泛泛讨论“模型扩展后性能会不会提升”。

---

## 核心机制与推导

争议的核心不是模型会不会随着规模变强，而是“强到什么程度时，外部观察者才会把它判定为会”。

先看连续机制。许多任务上，模型能力随规模提升往往接近光滑曲线。在不少分析中，某些潜在分数对参数规模的对数是近似线性的，也就是可以粗略写成：

$$
s(\theta)\approx a\log \theta+b
$$

白话说，模型越大，能力越强，但这种增强通常是一点点累积的，不像跳闸那样瞬间切换。

再看离散观测。如果最终汇报指标是

$$
M(s)=\mathbb{1}[s\ge \phi]
$$

那它只关心有没有过线，不关心离线有多近。于是会出现下面这种情况：

| 参数规模 | 潜在能力 $s(\theta)$ | 二值评测结果 |
|---|---:|---:|
| 1B | 0.31 | 0 |
| 3B | 0.37 | 0 |
| 7B | 0.44 | 0 |
| 13B | 0.49 | 0 |
| 30B | 0.53 | 1 |
| 175B | 0.61 | 1 |

如果阈值 $\phi=0.5$，那么 13B 和 30B 之间确实会出现榜单上的“跳变”。但从潜在能力看，0.31 到 0.61 的变化并不神秘，它一直在增长。跳跃来自评测把“接近正确”和“完全正确”分到两个桶里。

这就是 Schaeffer 等人批评的重点：很多“涌现”可能是指标伪影。所谓“伪影”，白话说，就是观察方法自己制造出来的视觉效果，而不是底层对象真的那样变化。

可以用身高阈值再理解一次。假设“高个子”的定义是超过 170 cm。一个人从 168 cm 长到 172 cm，标签会从“不是高个子”变成“是高个子”。如果只看标签，好像发生了突然跃迁；但真实增长过程是连续的。精确匹配对语言模型就经常扮演这个“170 cm 线”。

这并不等于 Wei 等人的观察完全错误。更准确的说法是：他们发现了在某些任务和某些指标下，规模扩展会出现难以从小模型区间直接外推的显著拐点。而 Schaeffer 等人提醒，看到拐点后不能立刻把它解释为“能力机制发生离散突变”，还要先排查是否只是度量函数过于非线性。

真实工程例子通常出现在多步算术、链式推理、复杂指令跟随等任务上。比如一个模型在多步算术中，前几版常常只差最后一步或中间一位数字。如果用 exact match，它一直是 0；一旦有足够多样本刚好答对，曲线就会突然抬头。但如果同步看 token 级概率、平均编辑距离、部分步骤正确率，往往能观察到平滑改善。这说明“外部成功率跳变”和“内部表征能力断裂”不是一回事。

因此，涌现争议的真正分歧不是“模型有没有变强”，而是“能力应该被当成离散属性，还是连续变量在特定指标下的投影”。

## 代码实现

下面给一个可运行的最小 Python 例子。它不依赖任何训练框架，只模拟“潜在能力平滑增长，但二值指标出现跳变”的情况。

```python
import math

# 模型规模，单位可以理解为参数量的相对等级
sizes = [1, 3, 7, 13, 30, 65, 175]

# 连续潜在能力分数：这里用 log(size) 后再做一个缩放
def latent_score(size):
    raw = 0.18 * math.log(size) + 0.30
    return min(raw, 0.99)

# 二值评测：是否跨过“完全正确”的阈值
def exact_match(score, threshold=0.5):
    return 1 if score >= threshold else 0

# 连续误差：把“离正确答案还有多远”表示成一个平滑指标
def edit_distance_proxy(score):
    # 分数越高，误差越小
    return round((1.0 - score) * 10, 2)

rows = []
for size in sizes:
    s = round(latent_score(size), 3)
    em = exact_match(s)
    ed = edit_distance_proxy(s)
    rows.append((size, s, em, ed))

# 基本断言：潜在分数单调增加
for i in range(1, len(rows)):
    assert rows[i][1] >= rows[i - 1][1]

# 基本断言：至少存在一次“从 0 跳到 1”的 exact match
ems = [r[2] for r in rows]
assert 0 in ems and 1 in ems
jump_index = ems.index(1)
assert ems[jump_index - 1] == 0

for row in rows:
    print(f"size={row[0]:>3}, latent={row[1]:.3f}, exact_match={row[2]}, edit_distance={row[3]}")
```

这段代码会输出两条逻辑完全不同的曲线：

1. `latent_score` 平滑上升。
2. `exact_match` 在阈值处突然从 0 跳到 1。

如果你把它扩展到真实实验，建议至少同时记录三类指标：

| 指标类型 | 例子 | 作用 |
|---|---|---|
| 二值指标 | exact match、pass@1 | 看最终是否完全成功 |
| 连续误差指标 | 编辑距离、BLEU 差异、步骤错误数 | 看距离正确答案还有多远 |
| 概率指标 | log-prob、交叉熵、负对数似然 | 看模型内部置信度是否持续提高 |

玩具例子适合解释机制，真实工程例子则应该这样做：对同一任务，沿着不同模型规模或不同 checkpoint，同步记录 exact match 和连续指标，再判断“突然跃升”是否只是阈值效应。否则团队很容易把“评测曲线拐点”误读成“模型突然获得新认知模块”。

## 工程权衡与常见坑

工程上最常见的错误，是只盯着一个最硬的成功指标。比如只看 exact match、只看 pass/fail、只看是否通过单元测试。这样的指标适合做最终验收，但不适合单独拿来判断能力形成过程。

第一个坑是把“可见成功”误当成“能力首次出现”。模型在跨过阈值前，往往已经积累了大量有价值的改进，比如更稳定地召回关键词、更少漏步骤、更高的正确 token 概率。如果这些信息不记录，团队会误以为前面几个版本毫无进展。

第二个坑是错误做规模决策。真实工程里，大模型成本高，延迟高，资源调度也更复杂。如果某任务在 13B 到 175B 之间 exact match 跳得很明显，但连续指标只是稳步改善，那么“是否值得上更大模型”就不能只看那一个跳点，还要结合业务容错率判断。若业务允许轻微偏差，可通过后处理或工具调用补足，小模型未必不能用。

第三个坑是阈值本身不稳定。很多任务的“成功”标准并不是固定的。数据集清洗方式、答案标准化规则、是否允许格式差异，都会改变 $\phi$ 的实际位置。一旦阈值在不同实验间漂移，你观察到的“涌现点”也会跟着移动，这时再谈能力突然出现就很危险。

下面这个表格适合做部署前检查：

| 检查项 | 只看二值指标会怎样 | 更稳妥的做法 |
|---|---|---|
| 模型选型 | 容易高估大模型必要性 | 同看成本、延迟、连续误差 |
| 版本迭代评估 | 容易把前期进步判为“无” | 记录 token 级和样本级细分指标 |
| 结论解释 | 容易宣称“新能力涌现” | 先验证是否为阈值效应 |
| 线上监控 | 只能看到是否失败 | 同步监控错误类型与接近正确程度 |

对初级工程师来说，一个实用原则是：如果某条能力曲线只在 exact match 上突然跳，但在编辑距离、log-prob、部分步骤正确率上都很平滑，那么你首先应该怀疑“评测压缩”，而不是立即下结论说“模型突然学会了推理”。

## 替代方案与适用边界

如果不想被“伪涌现”误导，可以换几种评估思路。

第一种是连续指标优先。对生成任务，不只看答案是否全对，还看离正确答案还有多远，比如编辑距离、数值误差、步骤覆盖率。这类指标更适合分析能力形成过程。

第二种是分布视角。不要只统计均值，还看正确答案概率分布、前几候选之间的差距、不同题型的错误结构。白话说，就是看模型“越来越像会了”，还是“真的从不会突然变会”。

第三种是外推检验。如果一个任务在小模型区间表现接近对数线性增长，那么可以先做回归拟合，再看大模型点是否显著偏离预测。若没有明显偏离，只是因为二值指标跨阈值才显得陡峭，就不宜轻率称为涌现。

不同策略的适用边界可以总结为：

| 评估策略 | 适合回答的问题 | 局限 |
|---|---|---|
| 二值指标 | 最终能不能交付 | 容易制造阶跃观感 |
| 连续指标 | 能力是否持续积累 | 不一定直接对应用户体验 |
| 分布分析 | 改进来自哪里 | 成本更高，解释更复杂 |
| 外推建模 | 是否真的超出小模型趋势 | 依赖拟合假设是否合理 |

“涌现能力”这个标签在什么情况下更适合使用？至少要满足两点。第一，观测到的跃迁不能仅靠常见阈值指标解释；第二，在更平滑、更细粒度的度量下，仍然能看到与小模型趋势明显不同的结构性变化。如果做不到这两点，更谨慎的表述通常是“在某评测定义下出现显著拐点”。

所以，替代方案不是否定涌现研究，而是给它加上更严格的证据标准。真正有价值的问题不是“能不能把现象命名为涌现”，而是“这种现象到底来自模型内部机制、训练数据分布、提示方式，还是评测函数的形状”。
