## 核心结论

结构化检索增强，简称 KG-RAG，可以理解为“把知识图谱接入 RAG 的检索层”。白话说，普通 RAG 主要靠向量相似度找“像不像”，KG-RAG 额外用图中的实体和关系找“是不是、怎么连起来”。

它解决的不是“模型不知道知识”，而是“模型拿到的证据不够结构化”。纯向量检索擅长召回语义接近的段落，但对“谁导致谁、谁属于谁、A 通过什么关系连到 B”这类结构问题不稳定。纯图检索正好相反，关系精确，但语义覆盖窄，对自然语言问题的模糊表达不够友好。双通道融合的价值，就是让“广覆盖”和“高精度”同时存在。

一个新手版理解方式是：先用向量检索快速抓线索，再用图检索核对实体和关系，最后把文本证据和关系路径一起放进 prompt。这个流程相当于“先找到相关材料，再确认事实链条”，比只靠一句相似文本更稳。

下表可以直接看出三种方案的差异：

| 方案 | 主要证据单位 | 优势 | 短板 | 适合问题 |
|---|---|---|---|---|
| 纯向量检索 | 段落、句子 | 语义召回强，接入简单 | 关系不精确，多跳链路容易断 | 文本总结、FAQ、说明类问答 |
| 纯图检索 | 实体、关系、路径 | 关系精确，可解释性强 | 对自然语言表达敏感，噪声路径多 | 结构化知识查询、规则明确问答 |
| KG-RAG 双通道 | 段落 + 路径 | 同时保留语义覆盖和关系精度 | 系统复杂度更高，需维护图谱 | 多跳推理、知识密集型问答 |

可以把整体流程画成一个极简图：

```text
用户问题
  | 
  +--> 向量通道: Query -> Embedding -> TopK 文本段落
  |
  +--> 图通道: 实体识别 -> 子图扩展 -> 候选路径打分
  |
  +--> 融合生成: 文本证据 + 结构路径 -> Prompt -> LLM
```

## 问题定义与边界

KG-RAG 要解决的问题可以定义为：给定自然语言问题 $q$、文档集合 $D$、知识图谱 $G$，系统需要返回一组既相关又结构正确的证据，再生成答案。输入不是只有问题，还包括两类知识源：非结构化文本和结构化图谱。输出不是“最像的问题答案”，而是“有证据链支持的答案”。

这里的“知识图谱”是把知识表示成节点和边的结构。白话说，节点是实体，比如“肺炎”“发热”“细菌感染”，边是关系，比如“导致”“属于”“伴随症状”。它适合表达可枚举、可连接、可多跳推理的知识。

玩具例子最容易说明边界。假设用户问：

“细菌性肺炎为什么会出现发热，医生通常根据什么做初步判断？”

如果只做向量检索，系统可能找到很多包含“肺炎”“发热”“诊断”的段落，但这些段落未必构成完整链条。它可能知道“肺炎常伴发热”，也可能知道“胸片用于诊断”，却未必把“病因 -> 症状 -> 诊断依据”连成一个结构化解释。

如果有图谱，可能存在这样的链：

`细菌感染 -> 导致 -> 肺炎`
`肺炎 -> 常见症状 -> 发热`
`肺炎 -> 初步诊断依据 -> 影像学检查`

此时答案不仅能说“会发热”，还能解释“为什么发热”和“初步判断依据是什么”。

但这不意味着 KG-RAG 适合所有场景。它更适合以下边界：

| 场景特征 | 是否适合 KG-RAG | 原因 |
|---|---|---|
| 多跳关系明显 | 适合 | 图路径能表达链式因果或依赖 |
| 需要可解释证据链 | 适合 | 路径本身就是解释 |
| 领域知识稳定 | 适合 | 图谱维护成本可控 |
| 完全开放式闲聊 | 不适合 | 图谱覆盖通常不足 |
| 高度实时新闻 | 谨慎 | 图谱更新可能跟不上事实变化 |
| 纯主观创作 | 不适合 | 没有明确结构事实可检索 |

因此，KG-RAG 不是“所有 RAG 的升级版”，而是“当关系结构本身就是答案的一部分时”的升级版。

## 核心机制与推导

KG-RAG 的核心不是简单拼接两路结果，而是让图检索承担“关系过滤器”的角色。

设图检索返回的候选路径为 $w_z$，问题为 $q$，图谱为 $G$。图检索的目标可以写成：

$$
IR = \arg\max_{w_z} P(w_z \mid q, G)
$$

意思是：在图 $G$ 中，从所有可能路径里，挑出对问题 $q$ 最相关的路径。

问题在于，什么叫“最相关”？工程上通常不是单一分数，而是结构分数与语义分数的组合。一个常见写法是：

$$
Score(w_z, q) = \alpha \cdot S_{sem}(q, w_z) + \beta \cdot S_{struct}(w_z) - \gamma \cdot Hop(w_z)
$$

其中：

- $S_{sem}(q, w_z)$：语义相似度。白话说，问题和路径在语义空间里有多接近。
- $S_{struct}(w_z)$：结构可靠性。白话说，这条路径在图里是否符合已有知识模式。
- $Hop(w_z)$：跳数，也就是路径长度。一般越长越容易引入噪声。
- $\alpha,\beta,\gamma$：权重，用来平衡不同信号。

如果路径里节点和关系还有嵌入表示，例如用 TransE 或 RotatE 这类图嵌入方法，那么语义分数可以进一步细化。图嵌入可以理解为“把图里的实体和关系压成向量，好让机器计算相似度”。

融合阶段再把文本检索结果 $R_{text}$ 和图路径结果 $R_{graph}$ 一起送入生成器：

$$
A = Generator(q, R_{text}, R_{graph})
$$

更进一步，系统可以给路径加注意力权重：

$$
Context = \sum_i \lambda_i \cdot Text_i + \sum_j \mu_j \cdot Path_j
$$

这里的 $\lambda_i$ 和 $\mu_j$ 分别表示文本证据和图路径的贡献权重。白话说，不是所有段落、所有路径都同样重要，系统要给高置信度证据更高权重。

玩具例子可以这样推导。假设问题是：

“张三在哪家公司工作，这家公司属于哪个集团？”

文本通道检索到两段：

1. “张三于 2023 年加入星云科技，担任后端工程师。”
2. “星云科技是远航集团的子公司，负责云平台业务。”

图通道检索到两条边：

- `张三 -> 就职于 -> 星云科技`
- `星云科技 -> 子公司隶属 -> 远航集团`

如果只看文本，模型可能抽到第一段却漏掉第二段。如果只看图，它能回答关系，但无法补充“2023 年加入”“负责云平台业务”这类自然语言细节。双通道融合后，答案既能给出结构结论，也能带上上下文描述。

真实工程例子是客服或医疗助手。比如在多跳问答数据集 WebQSP、CWQ 一类任务中，系统先从问题里识别实体，再根据节点语义找到候选子图，再用路径结构筛掉无关链路，最后把图上下文转成 prompt 交给大模型。这类做法的关键收益不只是“答案更准”，而是“错误更容易定位”：你可以看到错在文本召回，还是错在图路径选择。

## 代码实现

下面给一个最小可运行示例，目标不是复现工业系统，而是说明“双通道检索 + 融合打分”的骨架。

先看数据结构差异：

| 通道 | 返回对象 | 典型字段 | 用途 |
|---|---|---|---|
| 向量检索 | 文本段落 | `id, text, score` | 提供自然语言证据 |
| 图检索 | 实体路径 | `nodes, relations, score` | 提供结构化关系链 |
| 融合结果 | 证据包 | `text_evidence, graph_paths` | 组织 prompt 给 LLM |

下面这段 `python` 代码模拟一个玩具 KG-RAG。它做四件事：

1. 用关键词重叠近似向量检索。
2. 用实体命中从图里找候选路径。
3. 用“语义分 + 短路径偏好”给路径打分。
4. 组装 prompt，并用 `assert` 验证关键结果。

```python
from collections import defaultdict

documents = [
    {
        "id": "d1",
        "text": "张三于2023年加入星云科技，担任后端工程师，负责云平台服务。"
    },
    {
        "id": "d2",
        "text": "星云科技是远航集团的子公司，主要提供企业级云计算产品。"
    },
    {
        "id": "d3",
        "text": "李四加入了晨海网络，负责推荐系统。"
    }
]

graph_edges = [
    ("张三", "就职于", "星云科技"),
    ("星云科技", "子公司隶属", "远航集团"),
    ("李四", "就职于", "晨海网络"),
]

def tokenize(text):
    # 简化做法：按字符和常见词片段混合处理，仅用于演示
    vocab = ["张三", "星云科技", "远航集团", "加入", "工作", "集团", "子公司", "就职"]
    found = set()
    for v in vocab:
        if v in text:
            found.add(v)
    return found

def vector_search(query, docs, topk=2):
    q_tokens = tokenize(query)
    scored = []
    for doc in docs:
        d_tokens = tokenize(doc["text"])
        score = len(q_tokens & d_tokens)
        scored.append({"id": doc["id"], "text": doc["text"], "score": score})
    scored.sort(key=lambda x: x["score"], reverse=True)
    return scored[:topk]

def graph_search(query, edges, max_hops=2):
    q_tokens = tokenize(query)
    adjacency = defaultdict(list)
    for h, r, t in edges:
        adjacency[h].append((r, t))

    paths = []

    # 从问题中命中的实体出发，做一个最多2跳的简单遍历
    for entity in q_tokens:
        if entity in adjacency:
            for r1, t1 in adjacency[entity]:
                path1 = {
                    "nodes": [entity, t1],
                    "relations": [r1],
                    "hop": 1
                }
                paths.append(path1)
                if max_hops >= 2 and t1 in adjacency:
                    for r2, t2 in adjacency[t1]:
                        path2 = {
                            "nodes": [entity, t1, t2],
                            "relations": [r1, r2],
                            "hop": 2
                        }
                        paths.append(path2)

    # 语义分：问题中命中的节点数；结构分：短路径更优
    for p in paths:
        node_match = sum(1 for n in p["nodes"] if n in q_tokens)
        semantic = node_match
        structural = 1.0 / p["hop"]
        p["score"] = 0.7 * semantic + 0.3 * structural

    paths.sort(key=lambda x: x["score"], reverse=True)
    return paths

def format_prompt(query, text_results, graph_results):
    text_block = "\n".join(
        f"- [{item['id']}] {item['text']} (score={item['score']})"
        for item in text_results
    )
    graph_block = "\n".join(
        f"- 路径: {' -> '.join(sum([[n, r] for n, r in zip(p['nodes'], p['relations'])], []) + [p['nodes'][-1]])} (score={p['score']:.2f})"
        for p in graph_results
    )
    return f"""问题: {query}

文本证据:
{text_block}

结构路径:
{graph_block}

请基于文本证据和结构路径回答，并优先保持实体关系准确。"""

query = "张三在哪家公司工作，这家公司属于哪个集团？"

text_results = vector_search(query, documents, topk=2)
graph_results = graph_search(query, graph_edges, max_hops=2)
prompt = format_prompt(query, text_results, graph_results[:2])

assert text_results[0]["id"] == "d1"
assert graph_results[0]["nodes"][0] == "张三"
assert any("远航集团" in " ".join(p["nodes"]) for p in graph_results)
assert "结构路径" in prompt
print(prompt)
```

这段代码的意义有两个。

第一，它明确了工程拆分：`vector_search` 和 `graph_search` 是两个独立模块，便于单独评估。第二，它说明融合发生在“生成前”，而不是“生成后补丁”。如果你把图路径只当附加说明，而不纳入统一 prompt，模型经常还是会优先吃掉更长的文本上下文，忽略结构关系。

真实工程里，图通道会更复杂。常见步骤是：

```text
Query
  -> 实体识别与实体链接
  -> 候选节点召回
  -> 邻域扩展 / 子图抽取
  -> 路径打分与剪枝
  -> 结构结果序列化
  -> 与文本证据联合组装 prompt
  -> LLM 生成
```

如果系统进一步做检索增强微调，例如把图路径、文本段落和答案一起做监督训练，那么模型会更稳定地区分“文本证据”和“结构证据”的角色，而不是把两者都当成普通上下文。

## 工程权衡与常见坑

KG-RAG 的主要代价不是推理公式，而是工程复杂度。你多了图谱构建、实体对齐、路径裁剪、双通道评估四套问题。

最常见的坑如下：

| 坑 | 表现 | 原因 | 常见对策 |
|---|---|---|---|
| 语义模糊 | 找到相似文本但答非所问 | 向量检索缺少关系约束 | 用图路径校验实体和关系 |
| 路径噪声 | 图里返回很多冷门链路 | 邻域扩展过大，多跳爆炸 | 限制跳数，做路径打分和剪枝 |
| 实体链接错误 | “苹果”被连到水果而不是公司 | 实体歧义 | 加实体类型、上下文别名库 |
| 提示词失衡 | 模型只看文本不看图 | prompt 中图证据权重过低 | 显式分区，给来源标记 |
| 图谱过期 | 结构正确但事实已过时 | 图谱更新不及时 | 做增量更新，给时间戳 |
| 成本过高 | 延迟和算力上升 | 双通道检索 + 图遍历 | 缓存热点子图，预计算嵌入 |

这里有一个很实际的问题：为什么只用向量不行？

因为向量相似度本质上是在估计“文本像不像用户问题”，不是在验证“关系是不是成立”。例如用户问“某药物禁忌是否与肝功能异常有关”，向量检索可能找到许多包含“药物”“肝功能”“禁忌”的段落，但无法天然保证它找到的是“禁忌关系”，而不是“副作用关系”或“适用人群关系”。

为什么只用图也不行？

因为图里的关系是离散的。用户问法往往不是图谱里关系名的原样复现。自然语言的模糊表达、缩写、代称、跨句上下文，图检索本身处理得不如文本检索自然。图很适合做精确核验，不擅长做开放召回。

融合位置也要注意。一个常见简图如下：

```text
文本TopK ----\
              >--- 权重融合 ---> Prompt
图路径TopK ----/
```

这里的权重融合不要简单做“各取前 3 条”。更稳妥的做法是：

1. 文本通道负责覆盖问题描述。
2. 图通道负责过滤关键实体关系。
3. 当两者冲突时，优先检查图路径是否来自高可信图谱，再回头检查原文出处。

如果系统需要严格可解释，还可以要求输出答案时附带“对应文本段落 ID + 图路径”。这样线上排障会轻很多。

## 替代方案与适用边界

不是所有项目都需要 KG-RAG。很多团队失败，不是因为技术路线错，而是过早引入了图谱。

一个简单决策规则是：

```text
如果问题主要是“找相关描述” -> 先试纯向量 RAG
如果问题主要是“确认实体关系” -> 考虑图检索
如果问题同时要求“语义覆盖 + 多跳关系 + 可解释” -> 上 KG-RAG
```

也可以整理成表格：

| 方案 | 适用场景 | 优点 | 局限 |
|---|---|---|---|
| 纯向量 RAG | 文本丰富、关系不关键 | 快，成本低，实现简单 | 难处理精确关系，多跳弱 |
| 纯图 QA | 数据结构清晰、查询模式固定 | 精确、可解释 | 扩展性差，对模糊问法不友好 |
| 向量 + 规则 | 电商、客服初期系统 | 上线快，可控性较高 | 规则维护成本会增长 |
| KG-RAG | 知识密集、多跳推理 | 兼顾覆盖与精度 | 架构复杂，维护成本高 |

一个新手常见误区是：只要数据是知识类，就该上知识图谱。这个判断过于粗糙。真正该问的是：

- 问题是否经常涉及两跳以上关系？
- 是否需要把“为什么这么回答”展示给用户或审核系统？
- 是否能稳定维护实体、关系和更新流程？

真实工程例子可以看电商问答。早期系统通常先做“向量检索 + 业务规则”。比如问“这款相机适合夜景吗”，纯文本可能已经够用，因为答案多来自商品文案和评测。只有当问题升级到“这个型号和上代型号的传感器、镜头卡口、兼容配件之间有什么继承关系”时，图结构的价值才会明显上升。

因此，KG-RAG 更像是“当系统需要处理结构事实时的专业工具”，而不是所有问答系统的默认起点。
