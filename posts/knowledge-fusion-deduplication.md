## 核心结论

知识融合去重要解决的不是“删掉重复行”这么简单，而是“多个来源对同一事实说法不一致时，到底保留什么”。对零基础读者，可以先记住一条主线：**先检测冲突，再估计来源可信度，最后输出真值**。如果没有这三步，多源数据一合并，结果往往只是把矛盾堆在一起。

最基础的方法是投票法，也就是“谁票多听谁的”。它简单，但容易被大量低质量来源带偏。更稳一点的方法是**源可信度加权**，白话讲就是“过去更常说对话的来源，今天的话更值得信”。再进一步，是真值发现（truth discovery），它不是一次性打分，而是让“来源可信度”和“候选值得分”互相影响、反复迭代，直到结果稳定。

玩具例子很直接。两个网站都写“张三生日”，A 说是 `1990-01-01`，B 说是 `1991-02-02`。这时系统不能把两个值都当成最终答案，而要先标记为冲突。若已知 A 过去在身份证号、毕业年份等字段上更准确，A 的可信度就更高，于是 `1990-01-01` 的得分更高，最终作为真值输出。

可以把因果链记成下面这条线：

`冲突检测 -> 可信度计算 -> 真值输出`

这条链条的核心意义是：**一致性不是输入天然就有的，而是融合系统推断出来的**。在工程里，知识融合去重不是一个“清洗附属步骤”，而是决定最终数据能不能被检索、推荐、问答、报表使用的主流程。

## 问题定义与边界

先定义问题。知识融合去重，是指在多个数据源同时描述同一实体时，把它们关于同一属性的声明收集起来，识别哪些值相同、哪些值冲突，再根据规则或算法保留唯一真值，或者在必要时保留多个真值。这里的“实体”可以理解为“被描述的对象”，比如一个人、一家公司、一篇论文；“属性”可以理解为“对象的某个字段”，比如生日、注册地、作者数。

对新手来说，可以先把它看成一个三列表：

| 实体 | 属性 | 候选值 |
|---|---|---|
| 张三 | 生日 | 1990-01-01 |
| 张三 | 生日 | 1991-02-02 |

如果这两行来自不同来源，而且字段已经对齐，那么系统就知道：这是同一实体、同一属性、不同值，属于冲突，不是普通重复。

下面是最小冲突表：

| 属性 | 来源 A | 来源 B | 状态 |
|---|---|---|---|
| 生日 | 1990-01-01 | 1991-02-02 | 冲突 |
| 性别 | 男 | 男 | 一致 |
| 国籍 | 中国 | 中国 | 一致 |

这个问题有明确边界。

第一，它更适合**结构化属性**。结构化属性的意思是值可以直接比较，比如日期、数值、单标签、枚举值。如果是长文本简介，不能简单用“相等/不等”判断冲突，通常要先做文本归一化、相似度比较，甚至语义抽取。

第二，它依赖**字段对齐**。字段对齐的白话意思是，先确认不同表里的 `birth_date`、`birthday`、`dob` 其实都是同一个属性。如果对齐错了，后面的冲突检测和真值发现都会错。

第三，它默认来源之间至少能被区分。也就是说，系统知道某个值来自哪个网站、哪个数据库、哪次采集任务。否则就无法估计来源可信度。

第四，它默认某些属性倾向于单值。例如“出生日期”一般只有一个真值；但有些属性天然可能多值，例如“别名”“经营品类”“办公地点”。这类属性不能强行只留一个答案，否则不是去重，而是误删。

所以问题本身不是“哪个字段不同”，而是：

1. 谁在说同一件事。
2. 这些说法是否冲突。
3. 如果冲突，应该如何做出可解释的选择。

## 核心机制与推导

整个机制可以拆成两层：**冲突检测**与**真值发现**。

先看冲突检测。系统要把所有记录按 `(实体, 属性)` 聚合。假设实体是张三，属性是生日，那么多个来源关于这个字段的所有候选值会组成一个集合。若集合里只有一个规范化后的值，说明一致；若有多个不同值，就进入冲突解决阶段。

设某个实体属性对的候选值集合为 $V=\{v_1,v_2,\dots,v_k\}$。如果 $k=1$，直接通过；如果 $k>1$，说明存在冲突。

接着是真值发现。它的核心思想是双向迭代：

- 来源越可信，它支持的值应该得分更高。
- 值越像真值，支持它的来源应该更可信。

这是一种“互相强化”的机制。白话讲，像电商平台既看“用户评分”，也看“用户本身是否可靠”；而用户是否可靠，又会反过来参考他过去评价过的商品是否最终被证明靠谱。

用题目给定的记号，可以写成：

$$
T_s^{(t+1)}=\frac{1}{|C_s|}\sum_{v \in C_s} P_v^{(t)}
$$

其中：

- $T_s^{(t)}$ 表示第 $t$ 轮来源 $s$ 的可信度。
- $C_s$ 表示来源 $s$ 当前支持的值集合。
- $P_v^{(t)}$ 表示第 $t$ 轮候选值 $v$ 的真值分数。

这条公式的意思很朴素：**一个来源的可信度，等于它支持的那些值整体上有多真**。

值分数的更新可以写成：

$$
P_v^{(t+1)}=\operatorname{softmax}\left(\sum_{s \mid v \in C_s} T_s^{(t)}\right)
$$

这里的 softmax 可以理解为“把原始支持强度压成 0 到 1 之间，并让所有候选值分数加起来等于 1”。如果某个值被高可信来源支持得更多，它的分数就会上升。

更具体一点，若一个属性有两个候选值 $v_1,v_2$，其原始得分分别是 $z_1,z_2$，则：

$$
P(v_i)=\frac{e^{z_i}}{e^{z_1}+e^{z_2}}
$$

这个归一化很重要，因为它把冲突值之间的竞争关系显式写出来了。一个值分数上升，通常意味着另一个值分数下降。

玩具例子如下。

- 来源 A 支持 `1990-01-01`，初始可信度 $T_A=0.9$
- 来源 B 支持 `1991-02-02`，初始可信度 $T_B=0.5$

则两个值的原始支持强度分别是：

- $z_{1990}=0.9$
- $z_{1991}=0.5$

经过 softmax：

$$
P(1990)=\frac{e^{0.9}}{e^{0.9}+e^{0.5}}, \quad
P(1991)=\frac{e^{0.5}}{e^{0.9}+e^{0.5}}
$$

因为 $0.9 > 0.5$，所以 `1990-01-01` 得分更高。下一轮里，A 因为支持了一个更高分的值，其可信度进一步提升；B 则下降。于是“可信来源支持高分值，高分值反过来抬高可信来源”的正反馈开始形成，最终系统收敛到稳定结果。

可以把这个迭代关系记成：

`来源可信度 <-> 候选值得分`

真实工程例子比“张三生日”复杂得多。假设企业知识图谱融合多个工商库、招聘平台、新闻库，字段是“总部城市”。内部主数据写“杭州”，招聘网站写“上海”，新闻正文抽取写“北京”。这时不能只看哪家先来，也不能机械投票，因为新闻抽取可能是上下文误判，招聘网站可能填的是办公地点而不是总部。系统通常会先做字段和语义对齐，再结合来源历史准确率、字段类型、时间新鲜度、是否存在因果或业务规则一致性，最后对候选值排序。这里的“因果一致性”可以白话理解为：某个值是否和系统里其他已知事实一起看时仍然说得通。

## 代码实现

下面给出一个最小可运行版本。它不追求学术上的完备，而是把工程主流程完整跑通：输入多源声明和初始可信度，检测冲突，对每个冲突组做迭代真值发现，最后输出得分最高的值。

```python
import math
from collections import defaultdict

claims = [
    {"entity": "张三", "attr": "生日", "value": "1990-01-01", "source": "A"},
    {"entity": "张三", "attr": "生日", "value": "1991-02-02", "source": "B"},
    {"entity": "张三", "attr": "性别", "value": "男", "source": "A"},
    {"entity": "张三", "attr": "性别", "value": "男", "source": "B"},
]

initial_source_weights = {
    "A": 0.9,
    "B": 0.5,
}

def softmax(score_dict):
    max_score = max(score_dict.values())
    exps = {k: math.exp(v - max_score) for k, v in score_dict.items()}
    total = sum(exps.values())
    return {k: v / total for k, v in exps.items()}

def group_claims(claims):
    groups = defaultdict(list)
    for c in claims:
        key = (c["entity"], c["attr"])
        groups[key].append(c)
    return groups

def truth_discovery_for_group(group_claims, source_weights, max_iter=50, eps=1e-8):
    values = sorted({c["value"] for c in group_claims})
    if len(values) == 1:
        return {values[0]: 1.0}, source_weights.copy()

    value_to_sources = defaultdict(set)
    source_to_values = defaultdict(set)

    for c in group_claims:
        value_to_sources[c["value"]].add(c["source"])
        source_to_values[c["source"]].add(c["value"])

    current_weights = source_weights.copy()
    value_scores = {v: 1.0 / len(values) for v in values}

    for _ in range(max_iter):
        raw_scores = {}
        for value in values:
            raw_scores[value] = sum(current_weights[s] for s in value_to_sources[value])
        new_value_scores = softmax(raw_scores)

        new_weights = current_weights.copy()
        for source, supported_values in source_to_values.items():
            avg_score = sum(new_value_scores[v] for v in supported_values) / len(supported_values)
            new_weights[source] = avg_score

        delta = 0.0
        for v in values:
            delta = max(delta, abs(new_value_scores[v] - value_scores[v]))
        for s in new_weights:
            delta = max(delta, abs(new_weights[s] - current_weights[s]))

        value_scores = new_value_scores
        current_weights = new_weights

        if delta < eps:
            break

    return value_scores, current_weights

groups = group_claims(claims)

results = {}
updated_weights = initial_source_weights.copy()

for key, group in groups.items():
    scores, updated_weights = truth_discovery_for_group(group, updated_weights)
    best_value = max(scores, key=scores.get)
    results[key] = {"scores": scores, "truth": best_value}

assert results[("张三", "生日")]["truth"] == "1990-01-01"
assert results[("张三", "性别")]["truth"] == "男"
assert results[("张三", "生日")]["scores"]["1990-01-01"] > results[("张三", "生日")]["scores"]["1991-02-02"]

print(results)
print(updated_weights)
```

这段代码有几个关键点。

第一，`group_claims` 先按 `(实体, 属性)` 分组，这一步就是冲突检测前的聚合入口。

第二，若一个组只有一个候选值，就直接返回 `1.0`，因为它没有冲突，不需要迭代。

第三，值更新时用 softmax 做归一化。这样多个候选值始终处于同一个概率空间里，便于比较。

第四，来源权重更新时，取它所支持值分数的平均值。这是题目给定公式的直接工程化版本。

第五，`eps` 是收敛阈值，白话讲就是“新一轮和上一轮几乎不变时，就停”。`max_iter` 是最大迭代次数，防止异常数据导致长期振荡。

如果要支持多值属性，最简单的改法不是强制只取一个 `best_value`，而是允许输出 top-k，或者输出所有分数高于阈值 $\tau$ 的值。例如“别名”字段可以保留多个值，“办公地点”也可能同时成立。

伪代码可以压缩成下面这样：

```python
while not converged and iter < max_iter:
    update value scores by source weights
    normalize via softmax
    update source weights by average of supported values
```

其中：

- `normalize via softmax` 的作用是让多个冲突值可比较。
- `converged` 通常表示本轮和上轮的最大差异小于阈值。
- `iter < max_iter` 用来兜底，防止永不收敛。

## 工程权衡与常见坑

知识融合去重在论文里常被写得很整齐，但工程里最容易出错的，恰恰是那些“看起来很自然”的默认假设。

先看常见坑：

| 坑 | 后果 | 处理 |
|---|---|---|
| 只做多数投票 | 大量复制同一错误值的来源会压倒少数高质量来源 | 加来源可信度、独立度和去重检测 |
| 字段未对齐就比较 | 把“出生地”和“籍贯”误当同一属性 | 先做 schema 对齐与值标准化 |
| 值未标准化 | `1990/01/01` 和 `1990-01-01` 被误判为冲突 | 统一日期、单位、大小写、别名 |
| 不设收敛条件 | 迭代耗时过长或来回振荡 | 设置 `max_iter` 和 `eps` |
| 单值算法用于多值字段 | 正确值被误删 | 对多值属性使用 top-k 或多真值模型 |
| 忽略时间因素 | 旧值压过新值 | 为来源或值加入时间衰减 |
| 把转载站当独立来源 | 虚高票数误导真值 | 做来源复制链和相似来源聚类 |

新手最容易忽略的一点，是**来源并不独立**。假设 3 个网站都写“张三生日是 1991-02-02”，但它们其实都是从同一个低质量黄页站复制来的；另一边只有 1 个政府公开数据源写 `1990-01-01`。如果只投票，错误值 `3:1` 获胜；但如果识别出前三个来源高度重复，就应该降低它们的合计影响力。

这就是“来源独立度”的作用。白话讲，它衡量“这些来源是不是在重复同一张嘴说的话”。独立度越低，总权重就应该越小。

另一个常见问题是**属性语义边界不清**。真实工程里，“总部城市”“注册地”“办公地点”“招聘地点”常被混成一类。冲突检测之前如果不先澄清字段语义，那么后面的真值发现是在错误问题上做精确计算，结果依然不可信。

真实工程例子可以看企业主数据融合。假设你要合并内部 ERP、工商注册库、第三方企业画像平台三类数据：

- ERP 里“公司地址”指发票地址。
- 工商库里“住所”指法定注册地址。
- 第三方平台里“总部”来自网页抽取。

如果系统把这三者硬合成一个字段，最终选出的“真值”可能只是数学上分数最高，但业务上并不对。工程里必须先定义字段边界，再进入冲突解决。

还有一个容易被低估的问题是**可解释性**。业务方通常不会接受“模型说这个值分高所以选它”。他们更需要这样的话：`A 来源历史准确率高，B 来源与 C 高度疑似互相转载，且 A 的值与企业成立公告一致，因此保留 A`。所以在系统设计上，最好保留每个真值的证据链，包括：

- 哪些来源支持它
- 这些来源各自权重是多少
- 是否经过时间或独立度修正
- 为什么其他值被压低

没有证据链，融合结果难以被审计，也难以修正。

## 替代方案与适用边界

不是所有场景都适合用迭代真值发现。方法选择取决于属性类型、来源规模、历史标签是否充足，以及是否需要强解释性。

下面给出一个简短对比：

| 方法 | 适用情况 | 限制 |
|---|---|---|
| 多数投票 | 来源少、质量接近、需要快速上线 | 易被重复错误源主导 |
| 可信度加权 | 已知来源历史表现 | 依赖可靠的先验权重 |
| 迭代真值发现 | 多源冲突明显、需动态估计来源可靠性 | 实现复杂，需关注收敛 |
| 规则优先 | 字段有强业务规则，如身份证校验、时间先后约束 | 泛化能力弱 |
| 多值真值发现 | 别名、标签、地点等天然多值字段 | 参数与阈值设计更复杂 |
| 人工审核兜底 | 高风险字段，如法务、医疗、合规数据 | 成本高，扩展性差 |

如果来源可信度完全无法估计，也没有足够历史真值作为监督，直接上复杂算法不一定比规则强。这种情况下，更稳的做法通常是：

1. 先做规则化冲突检测。
2. 对强规则字段直接裁决。
3. 对高冲突字段进入人工审核或半自动审核。

例如地图数据中的“所属城市”有时不是唯一答案。一个地点可能同时关联县、市、州等多层行政区。若系统强制只能输出一个值，就会把结构性多值误判成冲突。这类场景更适合多值真值发现，或者干脆调整数据模型，把“所属城市”改成“行政区层级集合”。

再比如商品知识库中的“颜色”。“黑色/深空灰/石墨色”可能在业务上属于不同粒度，而不是谁真谁假。这里冲突解决不能只看字符串，而要结合标准属性词表。否则融合系统会把“规格映射问题”误当“真值冲突问题”。

所以适用边界可以总结为：

- 当属性天然单值、来源可追踪、值可比较时，迭代真值发现很合适。
- 当属性天然多值或强依赖业务语义时，应优先改模型或改规则。
- 当字段风险高、错误代价大时，再好的自动算法也应有人工兜底。
