## 核心结论

MMMU（Massive Multi-discipline Multimodal Understanding and Reasoning Benchmark）是面向大学水平的多学科多模态理解评测，用来测试模型能否同时处理文字、图像和学科知识，并给出可靠答案。

这里的“多模态”是指输入不只有文字，还包含图片、图表、公式、地图、电路图、医学影像等视觉信息。MMMU 不是单纯的 OCR 评测。OCR 是“把图里的文字识别出来”，而 MMMU 更关心模型是否能在识别之后继续理解结构、单位、关系和学科含义。

一个典型例子是实验曲线图。图里有坐标轴、图例、三条曲线和题干说明。模型不能只读出“Method A”“Method B”“Method C”这些曲线标签，还要判断哪条曲线表现最好，判断依据是最高准确率、最低误差，还是收敛速度，并且要确认这个依据是否和题干一致。

MMMU 的核心价值在于区分两类模型：

| 能力层次 | 模型表现 | MMMU 是否能区分 |
|---|---|---|
| 只会 OCR | 能读出图中文字，但不理解图表含义 | 能 |
| 图像分类 | 能判断图片大类，但不能结合题干推理 | 能 |
| 文本问答 | 能处理题干，但忽略图片信息 | 能 |
| 图文联合推理 | 能把题干、图片、选项和学科知识合起来判断 | 能 |
| 专业多学科推理 | 能处理公式、图表、结构关系和专业概念 | 重点评测对象 |

MMMU 原始版本约 11.5K 道题，覆盖 6 大学科、30 个学科、183 个子领域，范围包括 STEM、人文、社科等方向。它的基本指标是准确率：

$$
\mathrm{Acc} = \frac{\text{正确题数}}{\text{总题数}}
$$

因此，MMMU 的一句话结论是：它不是测模型“有没有看见图中文字”，而是测模型“能不能像初级专业学习者一样，把图文信息合起来做学科判断”。

## 问题定义与边界

MMMU 要解决的问题可以定义为：给定一道包含文本题干和视觉材料的大学水平题目，让模型从候选答案中选出正确答案。

输入通常包括三部分：

| 符号 | 含义 | 白话解释 |
|---|---|---|
| $t$ | 题干文本 | 问题本身、背景说明、选项文字 |
| $v$ | 图像输入 | 图表、公式截图、地图、电路图、医学图像等 |
| $\mathcal{Y}$ | 候选答案集合 | A/B/C/D 等可选答案 |
| $y$ | 标准答案 | 人类标注的正确选项 |

输出是：

$$
\hat y \in \mathcal{Y}
$$

其中 $\hat y$ 是模型预测的答案。“预测”就是模型在多个候选项里选出它认为最可能正确的那个。

MMMU 的边界也要明确。它不是纯 OCR，不是纯文本问答，也不是普通图片分类。

| 任务 | 输入 | 输出 | 与 MMMU 的区别 |
|---|---|---|---|
| OCR | 图片中文字 | 识别出的文字 | 不要求学科推理 |
| 文本问答 | 纯文本题目 | 文本答案或选项 | 不测试图像理解 |
| 图像分类 | 单张图片 | 类别标签 | 不要求结合题干 |
| MMMU | 题干 + 图片 + 选项 | 正确选项 | 要求图文联合推理 |

玩具例子：题目给出一张电路图，电路中有电阻 $R_1$、$R_2$、$R_3$，题干问“哪个元件中的电流最大”。如果模型只做 OCR，它最多知道图里出现了这些元件名称，却不能直接得到答案。要答对，它必须理解串联、并联、电压分布和欧姆定律。

真实工程例子：一个教育助手帮助学生分析物理题。学生上传电路图和题干，系统要指出电流最大的位置，并说明依据。如果系统只提取图片文字，就无法处理电路结构；如果系统只看题干，也会漏掉关键连接关系。这类任务更接近 MMMU 的评测边界。

因此，MMMU 的问题边界可以写成：

```text
输入：t = 题干文本，v = 图像
输出：y ∈ Y，通常是四选一或多选一答案
边界：不是纯 OCR，不是纯文本问答，不是单图分类
核心：图像解析 + 文本理解 + 学科推理
```

## 核心机制与推导

MMMU 可以抽象成一个条件概率选择问题。“条件概率”是指在已知某些信息的前提下，一个答案成立的可能性。

给定题干 $t$、图像 $v$ 和候选答案集合 $\mathcal{Y}$，模型要选择概率最大的答案：

$$
\hat y = \arg\max_{y \in \mathcal{Y}} p_\theta(y \mid t, v)
$$

这里的 $p_\theta(y \mid t, v)$ 表示模型在看到题干和图像后，认为答案 $y$ 正确的概率。$\theta$ 表示模型参数，可以理解为模型内部学到的知识和规则。

如果一道题有四个选项，模型给出的概率如下：

| 选项 | 概率 |
|---|---:|
| A | 0.10 |
| B | 0.55 |
| C | 0.20 |
| D | 0.15 |

模型会选择 B，因为 0.55 最大。如果标准答案也是 B，这题记 1 分；如果标准答案不是 B，这题记 0 分。

整个数据集的准确率计算为：

$$
\mathrm{Acc} = \frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[\hat y_i = y_i]
$$

其中 $N$ 是题目总数，$\mathbf{1}[\hat y_i = y_i]$ 是指示函数。指示函数的白话解释是：条件成立就记 1，不成立就记 0。

一个 5 题的小例子：

| 题号 | 预测答案 | 标准答案 | 是否正确 |
|---|---|---|---:|
| 1 | B | B | 1 |
| 2 | C | A | 0 |
| 3 | D | D | 1 |
| 4 | A | A | 1 |
| 5 | B | C | 0 |

总共答对 3 题，因此：

$$
\mathrm{Acc} = \frac{3}{5} = 0.60
$$

也就是 60%。

从机制上看，MMMU 中一题能否答对，通常取决于三部分是否同时成功：

| 环节 | 需要解决的问题 | 常见失败方式 |
|---|---|---|
| 图像解析 | 看懂图表、公式、符号、结构 | 坐标轴读错，图例对应错 |
| 文本理解 | 理解题干问的是什么 | 忽略限定条件 |
| 学科知识 | 调用物理、医学、经济学等知识 | 概念混淆，公式用错 |
| 答案选择 | 在候选项中做最终判断 | 被文本捷径误导 |

例如，一张曲线图展示三种算法在不同训练轮数下的准确率。题干问“在相同训练预算下，哪种方法最终性能最好”。模型必须先识别横轴是训练轮数、纵轴是准确率，再把图例和曲线对应起来，最后比较末端数值。如果它只看到某个方法在前期上升最快，却忽略“最终性能”这个限定，就会答错。

## 代码实现

实现一个 MMMU 类评测流程时，最好把评测逻辑拆成几个独立步骤：读取样本、加载图片、调用模型、解析答案、计算准确率。这样做的好处是模型可以替换，评测逻辑不需要重写。

下面是一个可运行的最小 Python 玩具例子。它没有真正调用多模态模型，而是用一个假模型模拟“根据题干、图片路径和选项返回答案”的过程，重点展示评测结构。

```python
from dataclasses import dataclass
from typing import List, Dict


@dataclass
class Item:
    question: str
    image_path: str
    choices: Dict[str, str]
    answer: str
    subset: str


class ToyMultimodalModel:
    def predict(self, text: str, image_path: str, choices: Dict[str, str]) -> str:
        # 玩具规则：真实工程中这里会调用多模态模型 API 或本地模型。
        if "highest final accuracy" in text:
            return "B"
        if "largest current" in text:
            return "C"
        return "A"


def evaluate(dataset: List[Item], model: ToyMultimodalModel) -> float:
    correct = 0
    for item in dataset:
        pred = model.predict(
            text=item.question,
            image_path=item.image_path,
            choices=item.choices,
        )
        correct += int(pred == item.answer)

    return correct / len(dataset)


dataset = [
    Item(
        question="Which method has the highest final accuracy?",
        image_path="figures/curve.png",
        choices={"A": "Method A", "B": "Method B", "C": "Method C", "D": "Method D"},
        answer="B",
        subset="chart",
    ),
    Item(
        question="Which component has the largest current?",
        image_path="figures/circuit.png",
        choices={"A": "R1", "B": "R2", "C": "R3", "D": "Cannot determine"},
        answer="C",
        subset="circuit",
    ),
    Item(
        question="What is the dominant trend in the map?",
        image_path="figures/map.png",
        choices={"A": "Northward", "B": "Southward", "C": "No change", "D": "Random"},
        answer="A",
        subset="map",
    ),
]

model = ToyMultimodalModel()
acc = evaluate(dataset, model)

assert len(dataset) == 3
assert acc == 2 / 3
print(f"accuracy = {acc:.2%}")
```

真实工程里，`ToyMultimodalModel.predict` 会被替换成模型调用，例如 GPT-4V、开源多模态大模型，或者内部视觉语言模型。关键是不要把“模型调用”和“评测统计”写死在一起。

一个更接近实际的评测流程通常是：

```text
1. 读取题目 JSON
2. 校验 question、image_path、choices、answer 字段
3. 加载图片，保持原始分辨率或按统一规则缩放
4. 构造模型输入：题干 + 图片 + 选项
5. 获取模型输出
6. 解析成 A/B/C/D
7. 与标准答案比较
8. 分总体准确率和子集准确率
```

子集统计很重要。假设总体准确率是 58%，看起来只能说明模型中等水平；但如果拆开看，可能发现图表题 70%、公式题 45%、地图题 38%。这说明模型不是整体都弱，而是在特定视觉结构上存在短板。

| 子集 | 题目类型 | 工程诊断价值 |
|---|---|---|
| chart | 曲线图、柱状图、散点图 | 检查坐标轴、图例、趋势理解 |
| formula | 数学公式、物理公式 | 检查符号和推导能力 |
| map | 地图、地理分布图 | 检查空间关系理解 |
| circuit | 电路图、结构图 | 检查拓扑结构推理 |
| table | 表格截图 | 检查行列对应和数值比较 |

真实工程例子：科研助手要读取论文图表，并判断“论文正文说方法 B 最好”是否被图 3 支持。此时系统不能只识别图注，而要读取曲线、坐标轴、误差条和实验条件。MMMU 类评测可以作为模型选型前的压力测试，用来判断模型是否具备进入该场景的基础能力。

## 工程权衡与常见坑

MMMU 类评测最常见的错误，是把它降级成 OCR 任务。很多题确实包含文字，但文字只是入口，不是答案本身。

| 常见坑 | 错误做法 | 后果 | 更稳的做法 |
|---|---|---|---|
| 只做 OCR | 先识别图片文字，再丢给文本模型 | 坐标轴、曲线形状、空间关系丢失 | 直接使用图文联合模型 |
| 只靠文本捷径 | 不看图，只根据题干猜答案 | 在测试集上可能偶然有效，部署后不稳 | 强制输入图片并做消融对比 |
| 多图拆开推理 | 三张子图分别问模型 | 丢失图间引用关系 | 保留完整题目上下文 |
| 忽略单位 | 只比较数字大小 | 把 ms、s、kg、g 混用 | 解析单位并统一比较 |
| 混用评测设置 | zero-shot 和 few-shot 混在一起报结果 | 结果不可比 | 固定提示词和样本设置 |
| 只看总分 | 不拆分学科和题型 | 找不到短板 | 按学科、图像类型、难度统计 |

“zero-shot” 是指不给模型示例，直接让它答题。“few-shot” 是指先给几个示例，再让模型答题。两者结果不能直接混报，因为 few-shot 往往会改变模型行为。

多图题尤其容易出问题。比如一道题有三张子图，分别表示不同实验条件：低温、常温、高温。题干问“高温条件下哪条曲线最稳定”。如果工程实现时把三张图拆开独立送给模型，模型可能不知道哪张对应高温，也可能把子图标签和曲线标签混淆。正确做法是保留题干、所有子图和图间引用关系，让模型在完整上下文中回答。

还有一个常见问题是答案解析。模型可能输出：

```text
The answer is B.
I think option B is correct.
B) Method B
```

工程上要把这些输出统一解析成 `B`。如果解析规则过宽，可能误判；如果规则过窄，又会把正确回答判成错误。较稳的做法是要求模型最后一行只输出选项字母，并在评测代码里做严格校验。

MMMU 的难点不在单个模块，而在模块协同。图像解析错了，后面的推理会建立在错误事实上；文本理解错了，模型会回答另一个问题；学科知识错了，即使图看对了也会选错答案。

## 替代方案与适用边界

MMMU 适合评估复杂图文联合推理，但不是所有任务都需要 MMMU 级别的模型。工程选型要先看任务目标。

| 任务类型 | 适合方案 | 是否需要 MMMU 类能力 |
|---|---|---|
| 发票识别 | OCR + 规则抽取 | 通常不需要 |
| 身份证字段抽取 | OCR + 版面解析 | 通常不需要 |
| 简单图片分类 | CV 分类模型 | 不需要 |
| 商品图检索 | 图像 embedding | 不一定需要 |
| 教育题目解答 | 多模态推理模型 | 需要 |
| 论文图表理解 | 多模态推理模型 | 需要 |
| 医学影像问答 | 专业多模态模型 + 严格验证 | 需要，但 MMMU 不足以单独证明可用 |
| 科研助手 | 图文联合推理 + 检索 + 引用校验 | 需要 |

“embedding” 是把文本或图片转换成向量，方便计算相似度。它适合检索和匹配，但不等于能完成复杂推理。

发票识别是一个不必过度复杂化的例子。目标通常是抽取发票号码、金额、日期、购买方等字段。OCR 加规则抽取或轻量版面模型就能解决主要问题，不需要让大模型理解发票背后的业务逻辑。

论文图表理解则不同。比如系统要判断“一篇论文声称新方法在所有数据集上都优于基线”是否被图表支持。它必须读懂表格中的指标方向，知道 accuracy 越高越好、error rate 越低越好，还要检查是否真的“所有数据集”都成立。这已经不是 OCR，而是图文联合推理。

MMMU 的适用边界也要注意。它能帮助判断模型是否具备大学水平多学科多模态能力，但不能直接保证模型在所有专业场景中可用。医学、法律、金融、工程安全等高风险领域，还需要领域数据、专家标注、错误分析和上线监控。

可以按下面的顺序做工程判断：

| 问题 | 判断 |
|---|---|
| 只需要读出图片中文字吗 | 用 OCR |
| 只需要判断图片类别吗 | 用 CV 分类模型 |
| 需要结合题干和图片回答问题吗 | 用多模态推理模型 |
| 涉及图表、公式、专业学科知识吗 | 用 MMMU 类基准辅助选型 |
| 涉及高风险决策吗 | 还需要领域评测和人工审核 |

MMMU 的正确使用方式不是“分数越高就一定能上线”，而是把它作为模型能力诊断工具。它能告诉你模型在图表、公式、结构关系和学科知识上大致处于什么水平，再结合真实业务数据做进一步验证。
