## 核心结论

Decision Transformer，简称 DT，可以理解成“把强化学习改写成序列预测”。序列预测的白话解释是：像语言模型按上下文预测下一个词一样，DT 按“目标回报、当前状态、历史动作”预测下一个动作。

它的关键变化不是网络结构本身，而是问题定义方式变了。传统强化学习通常先学价值函数 $Q(s,a)$，也就是“这个动作值不值”；或者直接做策略梯度，优化“这套动作规则能不能拿到更高回报”。DT 不走这条路。它把一条轨迹编码成

$$
\tau = (\hat R_1, s_1, a_1, \hat R_2, s_2, a_2, \dots, \hat R_T, s_T, a_T)
$$

然后用 Transformer 自回归地预测动作：

$$
a_t \sim \pi_\theta(\cdot \mid \hat R_{\le t}, s_{\le t}, a_{<t})
$$

这里的 $\hat R_t$ 叫 return-to-go，中文常写作“剩余回报”或“从当前时刻往后的累计回报”。白话解释是：从现在开始，如果沿着这条轨迹走到底，还能拿到多少总奖励。

因此，DT 的核心结论可以压缩成三句：

| 问题 | 传统 RL 常见做法 | Decision Transformer |
| --- | --- | --- |
| 学什么 | 价值函数或策略 | 条件动作分布 |
| 输入是什么 | 状态、动作、奖励、价值估计 | $(\hat R, s, a)$ 序列 |
| 训练目标 | Bellman 误差、策略梯度 | 监督学习预测动作 |

对初学者最重要的一点是：DT 不是“更复杂的 actor-critic”，而是“把 RL 当作带条件的行为建模”。

一个最小玩具例子能直接说明它怎么工作。假设某条轨迹奖励序列是 `[2, 1, 3]`，则：

$$
\hat R_1 = 2+1+3 = 6,\quad \hat R_2 = 1+3 = 4,\quad \hat R_3 = 3
$$

训练时，模型看到的不是原始奖励流，而是：

`(6, s1, a1, 4, s2, a2, 3, s3, a3)`

它学的是：

- 给定 `(6, s1)`，应该输出什么动作 `a1`
- 给定 `(6, s1, a1, 4, s2)`，应该输出什么动作 `a2`

推理时，如果你把目标回报设成 `R_target = 6`，模型会把这个值当成条件提示，先根据 `(6, s1)` 生成 `a1`。环境返回真实奖励 `r1 = 2` 后，再把目标更新为：

$$
\hat R_{2} = \hat R_1 - r_1 = 4
$$

再继续生成下一步动作。这个“用目标回报驱动动作生成”的机制，就是 DT 最有辨识度的地方。

## 问题定义与边界

DT 主要解决的是离线强化学习问题。离线强化学习的白话解释是：训练时只能使用已经收集好的历史数据，不能一边训练一边到环境里反复试错。

这一定义很重要，因为它决定了 DT 的优势和边界都来自“只看日志数据”。如果你的场景允许大量在线探索，比如游戏 AI 可以反复自我对弈，那么 DT 不一定是首选；如果你的场景在线试错昂贵甚至危险，比如机械臂抓取、推荐系统真实流量、自动驾驶日志回放，那么 DT 就更自然。

DT 处理的数据不是零散样本，而是按时间顺序组织好的完整轨迹。每条轨迹需要能构造成：

$$
\tau = (\hat R_1, s_1, a_1, \hat R_2, s_2, a_2, \dots, \hat R_T, s_T, a_T)
$$

这里每个时间步都拆成 3 类 token：

- `return-to-go`：还剩多少目标回报
- `state`：当前环境状态
- `action`：当前应采取的动作

初学者容易误解成“DT 只是拿 Transformer 套了一下 RL”。更准确的说法是：它要求你的数据必须天然支持序列化，而且 reward 的定义要稳定，才能让“目标回报”这个条件有意义。

用一个简单比喻理解边界：行为克隆像“看很多答案，学别人平时怎么写”；离线 RL 像“只看已经交卷的历史试卷，不允许再进考场试一次”；DT 则像“在看过高分和低分试卷后，给自己设一个想达到的分数，再预测下一题该怎么答”。

下面这张表可以快速判断一个问题是不是适合 DT：

| 问题类型 | 是否适合 DT | 原因 |
| --- | --- | --- |
| 离线机器人控制 | 适合 | 在线试错成本高，日志轨迹天然可序列化 |
| 推荐系统历史日志优化 | 适合 | 有大量用户交互日志，可构造回报条件 |
| 实时博弈对抗 | 通常不优先 | 强探索需求高，环境和对手持续变化 |
| 稀疏奖励迷宫探索 | 边界较强 | 数据覆盖不足时，序列建模难补足探索缺口 |
| 多任务 reward 尺度差异大 | 有条件适合 | 需要先做 reward 标定，否则目标回报失真 |

判断边界时，至少检查四件事：

| 检查项 | 为什么重要 |
| --- | --- |
| 离线数据是否充足 | DT 不能靠在线探索补数据盲区 |
| reward 是否稳定可比较 | 目标回报必须在不同轨迹间有一致语义 |
| 目标回报是否有意义 | 如果回报定义本身很噪，条件输入就不可靠 |
| 是否允许在线试错 | 如果允许大量交互，价值学习方法可能更高效 |

真实工程例子是离线机器人抓取。假设你有几十万条机械臂抓取日志，状态包括相机特征、夹爪位置、物体位姿，动作是末端执行器控制量。因为真实设备试错昂贵，而且失败抓取可能损坏工件，所以更合理的路线不是在线瞎试，而是把历史日志转成 `(回报, 状态, 动作)` 序列做离线训练。DT 在这种场景里是“自然可用”的。相反，如果你做的是实时广告竞价、对抗游戏、动态市场决策，策略需要不断试探新状态分布，DT 的纯离线优势就会变成限制。

## 核心机制与推导

DT 的关键输入不是即时奖励 $r_t$，而是 return-to-go：

$$
\hat R_t = \sum_{t'=t}^{T} r_{t'}
$$

即时奖励的白话解释是“这一步拿了多少分”，而 return-to-go 是“从这一步开始到结尾还能累计拿多少分”。DT 选择后者，是因为它更像一个全局目标信号，而不是局部反馈。

为什么这点重要？因为如果你只把 $r_t$ 喂给模型，那么模型看到的是“刚刚这一小步的好坏”；但很多任务的关键动作，其价值要到很多步之后才显现。return-to-go 把长期目标提前编码进当前条件里，使模型在当前状态下能区分“这是为了冲高回报的动作”还是“这是低质量轨迹中的保守动作”。

训练目标也很直接：只对动作做监督，不直接预测价值函数。

$$
L = \sum_t \ell(a_t, \hat a_t)
$$

其中 $\ell$ 可以是离散动作的交叉熵，也可以是连续动作的均方误差。这里的思想很朴素：如果高回报轨迹在数据中出现过，那么在相同或相似状态下，只要给定更高的 $\hat R_t$ 作为条件，模型就会更倾向于复现高质量轨迹中的动作模式。

用前面的玩具例子展开一次完整推导。奖励序列为 `[2, 1, 3]`，则：

- $\hat R_1 = 6$
- $\hat R_2 = 4$
- $\hat R_3 = 3$

训练序列是：

`(6, s1, a1, 4, s2, a2, 3, s3, a3)`

模型学到的条件分布是：

- $p(a_1 \mid 6, s_1)$
- $p(a_2 \mid 6, s_1, a_1, 4, s_2)$
- $p(a_3 \mid 6, s_1, a_1, 4, s_2, a_2, 3, s_3)$

推理时，先设定一个目标回报 `R_target`。如果设为 6，那么第一步输入是 `(6, s1)`。执行动作后，环境给出真实奖励 $r_t$，再更新：

$$
\hat R_{t+1} = \hat R_t - r_t
$$

于是推理过程变成一个循环：

| 阶段 | 输入 | 输出/更新 |
| --- | --- | --- |
| 第 1 步 | 目标回报 $\hat R_1$、状态 $s_1$ | 生成动作 $a_1$ |
| 执行动作 | 环境执行 $a_1$ | 得到奖励 $r_1$、新状态 $s_2$ |
| 更新条件 | 用真实奖励修正目标 | $\hat R_2 = \hat R_1 - r_1$ |
| 第 2 步 | 历史序列 + $(\hat R_2, s_2)$ | 生成动作 $a_2$ |

训练和推理的差别可以压缩成下面这张流程表：

| 阶段 | 训练 | 推理 |
| --- | --- | --- |
| 条件来源 | 来自离线轨迹真实回报 | 来自人工设定的目标回报 |
| 状态来源 | 数据集中的状态序列 | 环境实时返回的状态 |
| 动作来源 | 数据集中的真实动作 | 模型生成动作 |
| 回报更新 | 预先离线计算 | 用执行后真实奖励递推 |

为什么“只监督动作”就足够？前提是数据集里已经包含不同质量的行为模式。Transformer 会在上下文中同时看到状态和目标回报，于是它学到的不是单一平均策略，而是“在这类状态下，如果目标更高，轨迹里通常出现哪类动作”。这也是 DT 相比普通行为克隆更强的地方。行为克隆不区分轨迹好坏，只学平均；DT 通过目标回报这个条件，把高回报轨迹和低回报轨迹分层了。

但这也引出一个重要约束：DT 只能从数据里抽取已有模式，不能凭空发明数据中从未出现过的高质量策略。如果训练集里根本没有接近目标回报的样本，那么把 `R_target` 设得再高，也不会神奇地产生专家行为。

## 代码实现

从实现角度看，DT 最核心的不是 Transformer 结构，而是数据管道。你需要把原始轨迹变成适合自回归建模的训练样本，这一步比“堆网络层数”更关键。

常见模块可以这样拆：

| 模块 | 作用 |
| --- | --- |
| `dataset` | 读取轨迹、状态、动作、奖励 |
| `compute_return_to_go()` | 计算每一步的 $\hat R_t$ |
| `build_context()` | 按 `(R, s, a)` 顺序构造序列 |
| `model.forward()` | Transformer 主体，输出动作预测 |
| `loss_fn()` | 对动作计算监督损失 |
| `rollout()` | 推理时滚动生成动作并更新目标回报 |

如果是批量训练，常见张量形状如下：

| 张量 | 形状 | 含义 |
| --- | --- | --- |
| `returns` | `[B, K, 1]` | 每条样本窗口内的 return-to-go |
| `states` | `[B, K, state_dim]` | 状态序列 |
| `actions` | `[B, K, action_dim]` | 动作序列 |
| `timesteps` | `[B, K]` | 时间步编码 |
| `attention_mask` | `[B, K]` 或扩展形状 | 标记有效位置 |

其中 `B` 是 batch size，`K` 是上下文窗口长度。窗口长度的白话解释是：模型每次最多回看多少步历史。

下面先给一个可运行的玩具实现，展示 return-to-go 计算、序列拼接和动作监督损失的基本思路：

```python
from typing import List, Tuple

def compute_return_to_go(rewards: List[float]) -> List[float]:
    rtg = [0.0] * len(rewards)
    running = 0.0
    for i in range(len(rewards) - 1, -1, -1):
        running += rewards[i]
        rtg[i] = running
    return rtg

def build_triplets(states: List[str], actions: List[str], rewards: List[float]) -> List[Tuple[float, str, str]]:
    assert len(states) == len(actions) == len(rewards)
    rtg = compute_return_to_go(rewards)
    return list(zip(rtg, states, actions))

def mse(pred: List[float], target: List[float]) -> float:
    assert len(pred) == len(target)
    return sum((p - t) ** 2 for p, t in zip(pred, target)) / len(pred)

# 玩具轨迹
states = ["s1", "s2", "s3"]
actions = ["a1", "a2", "a3"]
rewards = [2.0, 1.0, 3.0]

rtg = compute_return_to_go(rewards)
triplets = build_triplets(states, actions, rewards)

assert rtg == [6.0, 4.0, 3.0]
assert triplets[0] == (6.0, "s1", "a1")
assert triplets[1] == (4.0, "s2", "a2")
assert abs(mse([0.1, 0.9], [0.0, 1.0]) - 0.01) < 1e-9

print("RTG:", rtg)
print("Triplets:", triplets)
```

这段代码没有真正训练 Transformer，但把最容易写错的核心步骤抽出来了：reward 要先反向累加成 $\hat R_t$，然后再和状态、动作按时间对齐。

接着看训练伪代码。无论你是 PyTorch 还是 JAX，思路基本一致：

```python
for traj in dataset:
    rtg = compute_return_to_go(traj.rewards)
    tokens = interleave(rtg, traj.states, traj.actions)   # (R, s, a) 交错
    pred_actions = model(tokens)
    loss = action_loss(pred_actions, traj.actions)
    loss.backward()
    optimizer.step()
```

推理伪代码则是：

```python
target_return = R_target
history = []
state = env.reset()

for t in range(T):
    action = model(target_return, state, history)
    next_state, reward, done, info = env.step(action)
    target_return = target_return - reward
    history.append((target_return, state, action))
    state = next_state
    if done:
        break
```

真实工程里，模型输入通常不是把标量和向量直接硬拼成字符串，而是分别做嵌入：

- `return` 经过一个线性层映射到隐藏维度
- `state` 经过状态编码器映射到隐藏维度
- `action` 也有自己的嵌入或投影层
- 三类 token 再按 `(R, s, a)` 顺序交错送进 Transformer

一个典型的机械臂抓取例子是：`state` 包含图像编码后的特征、关节角、夹爪开合状态，`action` 是笛卡尔位姿增量或关节速度，`reward` 可能是抓取成功为 1、失败为 0，或中间加入靠近目标、稳定夹取等 shaping reward。此时工程重点不是“能不能把 Transformer 跑起来”，而是“reward 设计是否稳定、轨迹切片是否对齐、不同 episode 长度如何 padding、窗口长度 K 是否覆盖到抓取决策所需的上下文”。

## 工程权衡与常见坑

DT 的工程难点主要不在理论，而在“目标条件是否可信”。最常见的误区是把目标回报当成一个可以随便拉高的旋钮。实际上，`R_target` 最好参考训练集回报分布来设定。如果训练集大多数轨迹回报集中在 20 到 60，你硬把推理目标设成 200，模型看到的是一个严重分布外条件，动作往往会退化。

下面这张表是实际落地里最常见的坑：

| 常见坑 | 后果 | 规避方法 |
| --- | --- | --- |
| 目标回报设太高 | 动作退化、输出不稳定 | 参考训练集 return 分布设目标 |
| 把即时奖励当输入 | 条件信号错误 | 严格使用 return-to-go |
| 时间步和 token 搞混 | 序列错位、监督错配 | 固定 `(R, s, a)` 顺序 |
| 上下文太短 | 丢失长程依赖 | 按任务调大 `K` |
| reward 尺度不统一 | 目标回报失真 | 归一化、重标定或分任务训练 |
| 把 DT 当在线 RL | 面对新状态无探索能力 | 明确它是离线序列建模方法 |

先说目标回报。设得太高，等于要求模型模仿“训练中几乎没见过的超高质量轨迹”；设得太低，模型又会主动收缩到保守策略附近。对初学者来说，一个实用做法是先画训练集 return 的直方图，再从 75 分位、90 分位或专家均值附近开始尝试，而不是拍脑袋写一个数字。

再说 reward 尺度。真实工程里，不同任务的数据常常不是同一量纲。比如机器人抓取任务 A 的回报范围是 `[0, 1]`，任务 B 却是 `[0, 100]`。如果你不做标准化，推理时同样设 `R_target = 20`，在任务 A 里是荒谬高分，在任务 B 里却只是普通水平。这样“目标提示”就没有统一语义。多任务场景下，通常要做 reward normalization、按任务分桶，或者让目标回报在每个任务内部归一化后再输入。

上下文长度 `K` 也是典型权衡。太短，模型可能看不到关键历史。例如在机器人操作中，当前抓取动作是否合理，可能取决于前 10 到 20 步的接近路径；如果窗口只有 4 步，决策就可能变得短视。太长则会带来更高显存和更难优化的注意力计算。工程上通常不是追求“越长越好”，而是根据任务依赖跨度做验证。

还有一个容易忽略的问题是 token 对齐。一个时间步不是一个 token，而是三个 token：`(R_t, s_t, a_t)`。如果你在 padding、mask、位置编码或损失提取时把它们按“单步样本”处理，很容易把状态和动作错位，最终模型会学到错误监督。这类 bug 看起来不会报错，但会直接把效果打穿。

真实工程例子里，离线推荐系统也会遇到类似问题。假设你把用户一个会话的点击、停留、转化构造成轨迹，想用 DT 学“高转化目标下下一条推荐该是什么”。如果不同流量池的 reward 定义不一致，比如有的用点击加权、有的用成交额，混在一起训练时目标回报会失去可比性。结果不是模型学不到，而是“目标条件变成噪声”，推理时设高目标也无法稳定导向高质量行为。

## 替代方案与适用边界

DT 不应该孤立地看，最好放在几类常见离线方法里比较。

首先是 Behavior Cloning，简称 BC。它的白话解释是：直接模仿数据中的动作，不问这条轨迹好不好。优点是简单稳定，缺点是容易学到“平均行为”。如果数据里既有专家轨迹也有差轨迹，BC 往往会把它们混在一起学。

其次是 Conservative Q-Learning，简称 CQL。它属于显式价值学习方法，会学一个保守的 $Q$ 函数，核心目标是尽量避免对数据分布外动作过度乐观。白话解释是：宁可保守，也不要在离线数据看不到的地方乱估值。

再比如 IQL、BCQ，也都属于典型离线 RL 基线。它们与 DT 的最大区别不在“是不是深度网络”，而在“是不是显式建模价值函数”。

下面这张表可以快速对比：

| 方法 | 是否显式学价值 | 是否需要在线探索 | 是否适合高质量离线轨迹 | 主要优势 | 主要短板 |
| --- | --- | --- | --- | --- | --- |
| Behavior Cloning | 否 | 否 | 一般 | 简单稳定，训练便宜 | 不区分轨迹质量，容易学平均策略 |
| Decision Transformer | 否 | 否 | 适合 | 利用序列结构，可按目标回报条件生成 | 对目标回报和数据分布敏感 |
| CQL | 是 | 否 | 适合 | 保守，分布外动作风险控制更强 | 训练更复杂，依赖价值学习稳定性 |
| IQL | 是 | 否 | 适合 | 实践中常较稳健，不需要显式策略约束过强 | 仍依赖价值估计质量 |
| BCQ | 是 | 否 | 有条件适合 | 强调限制分布外动作 | 实现和调参复杂度较高 |

什么时候优先考虑 DT？

- 数据是离线的，且轨迹质量存在明显层次
- 目标回报有明确含义，可以作为控制信号
- 任务本身具有较强序列结构，历史上下文很重要
- 你希望用统一的序列建模框架表达状态、动作和目标

什么时候更应该看 CQL、IQL 这类方法？

- 你更关心稳健性和保守性
- 数据覆盖不完整，分布外动作风险高
- 你需要一个更明确的价值估计器来约束策略

一个直观的新手版总结是：

- BC 学“大家平时怎么做”
- CQL 学“尽量保守地选安全动作”
- DT 学“如果目标是更高回报，接下来应该怎么做”

真实工程里，如果你手上是高质量专家日志，且高回报轨迹与低回报轨迹差异明显，比如自动化仓储机械臂、固定流程工业控制、结构化推荐会话，那么 DT 很值得优先试。反过来，如果数据很脏、覆盖很差、环境还会持续变化，那么 CQL 或 IQL 往往更符合“先求稳”的目标。

DT 的适用边界可以用一句话收束：它擅长把“已有高质量轨迹中的规律”提纯成一个可条件控制的动作生成器，但它不擅长替代探索，也不擅长在几乎没有相关样本的区域凭空外推。
