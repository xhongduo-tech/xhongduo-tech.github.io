## 核心结论

RAG，中文常译为“检索增强生成”，意思是模型先查资料，再基于资料生成答案。所谓“RAG 的记忆机制”，不是把模型参数重新训练一遍，而是在“检索再生成”这条流水线外面，叠加一层可读、可写、可治理的外部记忆系统。

它成立的根本原因很直接：普通 RAG 只能回答“这次问题和知识库里哪些片段最像”，但很多真实任务需要回答“这个用户之前说过什么、上一次结论是什么、系统已经执行过哪些步骤”。这些信息不一定存在于原始知识库里，却决定了答案是否连续、个性化、可追踪。

因此，记忆增强 RAG 的核心不是“多存一点文本”，而是把不同类型的信息分层管理。常见分法是短期记忆、中期事件记忆、长期语义记忆、程序性记忆。短期记忆保存当前对话现场；事件记忆保存发生过的关键动作；语义记忆保存稳定事实；程序性记忆保存规则和流程。这样做的目的，是避免所有信息都挤进同一个 prompt，导致上下文窗口，也就是模型一次能读入的文本预算，被无效内容占满。

下面这张表先把普通 RAG 和带记忆的 RAG 的差异讲清楚：

| 维度 | 普通 RAG | 记忆增强 RAG |
|---|---|---|
| 状态 | 无状态，每轮近似重新开始 | 有状态，可延续历史交互 |
| 数据来源 | 只读外部知识库 | 外部知识库 + 可写记忆层 |
| 个性化 | 弱，常靠本轮 prompt 临时补充 | 强，可长期保留偏好与约束 |
| 连续任务能力 | 差，容易丢上下文 | 强，可追踪任务过程 |
| 治理要求 | 相对简单 | 更高，需要过期、审计、删除 |
| 失败模式 | 找不到资料或检索不准 | 除检索问题外，还会有脏记忆、旧记忆、误写入 |

一个给新手的直观例子：旅游助手第一次听到“用户去东京，只吃纯素”，普通 RAG 通常不会把这件事保留下来，下次还要再问；记忆增强 RAG 会把它写入长期偏好记忆，后续推荐餐厅时先读到这条偏好，再去检索东京餐厅资料，回答自然就连续了。

## 问题定义与边界

问题定义很明确：普通 RAG 解决的是“如何把外部资料接进模型”，但没有解决“如何让系统保留交互中产生的新信息”。如果每一轮都只从原始文档里取 Top-k 片段，然后回答完就丢弃，这个系统就无法积累用户偏好、任务进展、阶段性结论，也无法形成持续服务能力。

这里要先划清边界。记忆机制并不等于“所有内容都永久保存”。真正的工程问题是三件事：

1. 存什么。
2. 存多久。
3. 什么时候读，什么时候写，什么时候删。

“存什么”通常包括四类对象。术语第一次出现时可以先用白话解释：

| 记忆层 | 白话解释 | 保留周期 | 典型写入触发 | 控制策略 |
|---|---|---|---|---|
| 短期记忆 Short-term | 当前会话里马上会用到的话头 | 分钟到单次会话 | 每轮对话自动追加 | 固定窗口、摘要压缩 |
| 事件记忆 Episodic | 某次发生过的重要事件 | 天到周 | 任务节点、工具调用、状态变化 | 只记录关键事件，保留时间线 |
| 语义记忆 Semantic | 稳定事实与偏好 | 周到月，甚至更久 | 高置信度事实抽取 | 去重、版本化、过期刷新 |
| 程序性记忆 Procedural | 做事规则和操作步骤 | 长期 | 人工配置或审核后更新 | 强治理、低频写入 |

边界同样重要。不是所有系统都需要长期记忆。一个 FAQ 问答机器人，只要每次都从最新文档里查答案，就没有必要保存用户历史。而一个长期陪伴型学习助手，如果不记住“用户的基础水平、薄弱点、偏好表达方式”，它几乎不可能表现出稳定性。

玩具例子可以说明边界：一个诊所随访机器人，当前就诊时需要保留最近几条生命体征，这是短期记忆；“42 号病人偏好邮件联系”这种稳定偏好，应该进长期记忆；而“今天 10:30 已发送化验提醒”属于事件记忆；“药物调整必须由医生确认后再通知病人”属于程序性记忆。四类数据混在一起，会直接导致检索和治理失控。

所以，RAG 的记忆机制本质上不是一个向量库功能，而是一套“信息生命周期管理规则”。

## 核心机制与推导

从数学上看，带记忆的 RAG 并没有推翻普通 RAG 的检索逻辑。它依然先把查询 $q$ 和候选文档 $d$ 编码成向量，再按相似度排序。常见相似度函数有点积和余弦相似度：

$$
s(q,d)=q^\top d
$$

或

$$
s(q,d)=\cos(q,d)=\frac{q \cdot d}{\|q\|\|d\|}
$$

这里的“向量”可以理解为“把一句话压缩成一串数字表示，使语义相近的句子在空间里更靠近”。

训练检索器时，常见目标是 InfoNCE。它的作用可以白话理解为：让正确记忆靠近当前问题，让无关记忆远离当前问题。

$$
\mathcal{L}_{ret}=-\frac{1}{N}\sum_{i=1}^N\log\frac{\exp(s(q_i,d_i^+)/\tau)}{\exp(s(q_i,d_i^+)/\tau)+\sum_{j\neq i}\exp(s(q_i,d_j^-)/\tau)}
$$

其中，$d_i^+$ 是正例，也就是和当前问题真的相关的记忆或文档；$d_j^-$ 是负例，也就是看起来像但其实不该取回的内容；$\tau$ 是温度参数，用来控制分布的陡峭程度。

但记忆增强 RAG 比普通 RAG 多了一层“仲裁”，也就是先决定该读哪一层记忆，再做每层内的检索。一个典型读路径是：

1. 读取短期 buffer，拿到最近几轮对话。
2. 根据查询意图决定是否检索事件记忆。
3. 对长期语义记忆做向量召回。
4. 再把程序性规则作为硬约束拼进 prompt。
5. 最后交给生成模型。

这套机制能减少“上下文竞争”。上下文竞争的意思是：模型一次只能看有限 token，如果近期闲聊、长期偏好、结构化事实、操作规则全塞进去，真正重要的信息会被淹没。分层记忆的价值，就是先做筛选，再做拼接。

先看一个玩具例子。查询向量设为：

$$
q=[0.7,0.1]
$$

记忆库里有两个向量：

$$
d_1=[0.9,0.2], \quad d_2=[0.1,0.9]
$$

若用点积打分：

$$
s(q,d_1)=0.7\times0.9+0.1\times0.2=0.65
$$

$$
s(q,d_2)=0.7\times0.1+0.1\times0.9=0.16
$$

因此系统优先取回 $d_1$。如果 $d_1$ 对应“上次你提到的项目是推荐系统迁移”，那模型下一轮就能自然回答“你上次讨论的是推荐系统迁移”，而不需要重新搜索整库或让用户重复描述。

更真实一点的工程例子，是医疗或客服这类长周期任务系统。比如肝病助手要同时利用三类信息：长期文档如出院总结，短期信号如最近十几轮提问，实体级记忆如病人 ID 和约束策略。它不会把所有历史原样塞进 prompt，而是先按“当前问题是否涉及诊疗历史、是否需要个体化约束、是否命中最近事件”决定召回哪一层。这么做有两个结果：一是 prompt 更短，二是幻觉更少，因为生成模型读到的是被筛过、带来源的信息，而不是一团未经控制的历史文本。

推导到这里，可以得到一个关键判断：RAG 的记忆机制之所以有效，不是因为“模型会记忆”，而是因为系统把“检索对象”从静态文档，扩展到了动态积累的外部状态。

## 代码实现

工程上最小可行方案通常包含三部分：写入策略、读取策略、治理元数据。

写入策略决定哪些内容值得记。一个常见原则是“高价值、高置信度、可复用”才写入。比如用户长期偏好、明确事实、任务完成状态，可以进入长期记忆；闲聊句子、低置信度推断、未经确认的猜测，不应直接写入。

读取策略决定当前问题应该读哪些层。最简单做法是“最近对话 + 长期记忆 Top-k”；更稳健的做法是先做意图判断，再决定是否查事件层、语义层或规则层。

治理元数据则是防脏记忆的关键。每条记忆至少要带：时间戳、来源、类型、置信度、是否可过期。否则你知道它“被取回了”，却不知道它“该不该再信”。

下面给一个可运行的 Python 玩具实现。它不是生产级向量数据库，而是一个最小原型，用来说明读写路径和断言验证：

```python
import math
import time

CONF_THRESHOLD = 0.8

def dot(a, b):
    return sum(x * y for x, y in zip(a, b))

def norm(v):
    return math.sqrt(sum(x * x for x in v))

def cosine(a, b):
    denom = norm(a) * norm(b)
    return 0.0 if denom == 0 else dot(a, b) / denom

class MemoryStore:
    def __init__(self):
        self.items = []

    def upsert(self, text, vec, metadata):
        self.items.append({
            "text": text,
            "vec": vec,
            "metadata": metadata,
        })

    def search(self, query_vec, k=2, min_conf=0.0):
        scored = []
        for item in self.items:
            if item["metadata"]["confidence"] < min_conf:
                continue
            score = cosine(query_vec, item["vec"])
            scored.append((score, item))
        scored.sort(key=lambda x: x[0], reverse=True)
        return [item for score, item in scored[:k]]

def update_memory(store, text, vec, confidence, memory_type, source):
    if confidence >= CONF_THRESHOLD:
        store.upsert(
            text=text,
            vec=vec,
            metadata={
                "confidence": confidence,
                "memory_type": memory_type,
                "source": source,
                "timestamp": int(time.time()),
                "ttl_days": 30 if memory_type == "semantic" else 1,
            },
        )
        return True
    return False

def build_prompt(query, query_vec, recent_buffer, store):
    recalled = store.search(query_vec, k=2, min_conf=0.8)
    recalled_text = "\n".join(
        f"[{m['metadata']['memory_type']}] {m['text']}" for m in recalled
    )
    recent_text = "\n".join(recent_buffer[-3:])
    return f"{recent_text}\n{recalled_text}\nQ: {query}"

store = MemoryStore()

ok1 = update_memory(
    store,
    text="用户偏好：去东京旅行时只接受纯素餐厅",
    vec=[0.9, 0.1],
    confidence=0.95,
    memory_type="semantic",
    source="conversation",
)

ok2 = update_memory(
    store,
    text="猜测用户可能喜欢夜店",
    vec=[0.1, 0.8],
    confidence=0.42,
    memory_type="semantic",
    source="model_guess",
)

prompt = build_prompt(
    query="东京有什么适合我的餐厅？",
    query_vec=[0.85, 0.15],
    recent_buffer=[
        "用户：我下周去东京。",
        "用户：饮食上我是纯素。",
        "系统：已记录你的饮食偏好。"
    ],
    store=store,
)

assert ok1 is True
assert ok2 is False
assert "纯素餐厅" in prompt
assert "夜店" not in prompt

print(prompt)
```

这个例子说明了三件事：

1. 写入不是自动全收，而是要过置信度阈值。
2. 读取时不只看向量相似度，还可以叠加元数据过滤。
3. prompt 构造阶段已经把“近期上下文”和“长期记忆”分开了。

如果放到真实工程里，通常还会再加三层能力：

| 模块 | 最小实现 | 工程化增强 |
|---|---|---|
| 写入 | 规则阈值写入 | LLM 判定 + 人工审核 |
| 存储 | 本地列表或简单向量库 | Redis / PostgreSQL / 专用向量库 |
| 检索 | 相似度 Top-k | 分层召回、重排、权限过滤 |
| 治理 | 时间戳 | 审计日志、版本、删除与撤回 |
| 提示拼接 | 直接串接 | 模板化、去重、来源标记 |

一个真实工程例子是支持工单助手。用户第一次说“不要电话，只接受邮件跟进”，系统把它写成语义记忆；本轮工单状态“已升级二线支持”写成事件记忆；“退款必须先校验订单号”写成程序性记忆。下一次回复用户时，系统会同时读出偏好、状态和规则，于是回答不仅准确，还符合流程和沟通方式。

## 工程权衡与常见坑

记忆机制最容易被误解成“多一个库就行”。实际上，真正困难的是治理，而不是存储。

第一个坑是只读不写回。这样系统每次都要重新检索老资料，却不会积累本轮交互中出现的新约束。结果是个性化越来越弱，用户总要重复描述自己。很多所谓“有记忆”的 demo，本质只是把最近几轮对话重复拼进 prompt，不是真正的可持续记忆。

第二个坑是丢失因果与时间顺序。检索本身只关心“像不像”，不关心“先后关系对不对”。如果系统把“昨天的状态”和“今天已撤销的状态”同时取回，又没有时间约束，模型就很容易拼接出错误结论。

第三个坑是脏记忆和旧记忆。错误信息一旦写入长期层，就可能被反复召回，比单次 hallucination 更危险。一次幻觉只错一轮，脏记忆会持续错很多轮。

第四个坑是隐私和合规。很多系统把“能写入”理解成“都保存”，但偏好、身份、健康信息、行为日志的保留规则完全不同。真正上线的系统必须能证明某条记忆为什么被写入、保存多久、谁可以读取、如何删除。

下面这张表把常见坑和缓解方案对应起来：

| 问题 | 典型表现 | 后果 | 缓解方式 |
|---|---|---|---|
| 无写回 | 每轮都像第一次交流 | 无法个性化，重复提问 | 增加高置信度写入策略 |
| 因果丢失 | 旧状态和新状态混用 | 回答前后矛盾 | 事件时间线、版本号、时间过滤 |
| 治理缺失 | 不知道记忆来源与有效期 | 审计困难，法规风险 | 时间戳、来源、TTL、审计日志 |
| 脏记忆积累 | 错误信息反复被召回 | 持续性错误 | 人工确认、回收机制、置信度降权 |
| 过度写入 | 闲聊也被永久保存 | 存储膨胀，噪声污染 | 类型分层、写阈值、摘要化 |
| 过度读取 | prompt 塞满历史 | 成本升高，关键信息被淹没 | 分层召回、重排、预算限制 |

这里有一个很实际的工程权衡：你越想让系统“像真人一样记住一切”，治理成本就越高。一个支持长期陪伴的智能体，必须投入更多规则去管理记忆生命周期；而一个高合规场景的企业问答系统，反而可能故意限制长期记忆，只允许保存最小必要状态。

所以常见正确做法不是“尽量多记”，而是“只记将来值得再用的信息”。

## 替代方案与适用边界

记忆增强 RAG 不是默认最优解，它只是适合某一类问题。

如果任务是单轮问答，比如“公司报销政策是什么”，普通 RAG 往往更合适。因为系统只需从最新文档中查出答案，不需要保存历史状态，也不用承担长期存储和治理成本。

如果任务需要多步工具调用，但不允许长期保存用户数据，那么可以用 Agentic RAG，也就是“会规划、会调工具的 RAG 代理”，但它的记忆保持为只读或会话内临时状态。这样可以完成复杂任务，又避免持久化个人信息。

如果任务需要长期陪伴、个性化和连续决策，比如学习教练、医疗随访、客户成功助手，那么记忆增强 RAG 才真正必要。因为系统不仅要查资料，还要记住“这个人是谁、之前发生过什么、现在处于哪个阶段”。

三种方案可以这样比较：

| 方案 | 是否持久化 | 是否可写 | 是否需要治理 | 适合场景 |
|---|---|---|---|---|
| 普通 RAG | 否或极弱 | 否 | 低 | 单轮问答、文档检索 |
| Agentic RAG | 会话内临时状态 | 通常不做长期写入 | 中 | 多步工具编排、流程执行 |
| 记忆增强 RAG | 是 | 是 | 高 | 长周期任务、个性化服务 |

适用边界可以概括为一句话：当“历史交互本身就是答案的一部分”时，才需要记忆增强 RAG。

再给一个对比鲜明的例子。FAQ 机器人回答“退货政策是什么”，普通 RAG 就够；个人健身教练则需要记住用户膝盖旧伤、最近训练负荷、饮食偏好和坚持情况，否则它给出的计划就不是连续服务，只是一次性建议。

因此，选型标准不是“哪个更高级”，而是“问题是否真的需要跨轮状态”。
