## 核心结论

指令数据构造的目标，不是“让模型多看一些问答”，而是把自然语料、任务描述和少量人工种子任务，稳定转成可训练的 `指令-输入-响应` 三元组。指令微调，白话说就是让模型学会“看懂要求再做事”的监督训练；它的上限很大程度取决于训练集中任务覆盖、表达多样性和噪声控制。

Self-Instruct 的核心价值是提供了一条近似“自举”的路线：先准备少量 seed tasks，即人工写的种子任务；再用模型模仿这些种子继续生成新指令、输入和输出；最后用去重、规则过滤和人工或程序验证，把低质量样本挡在训练池外。公开实现中，一个常见起点是约 175 个种子任务，经迭代后形成约 52k 指令和 82k 实例。

一个最小玩具例子可以这样理解。已有 seed 指令是“我想要摘要客户投诉”。模板会把它扩展成更具体的任务，如“阅读以下客户投诉并提取 3 条核心问题”。模型生成候选输入和输出后，再计算与已有指令的 ROUGE-L 相似度。ROUGE-L，白话说就是“看两个文本最长公共子序列有多像”。若相似度过高，比如 $\max S \ge 0.7$，就不把它当新任务，避免训练集只是换个说法反复出现。

| 阶段 | 输入 | 产出 | 主要风险 | 控制手段 |
|---|---|---|---|---|
| seed | 175 个左右人工任务 | 初始任务池 | 覆盖不足 | 保证分类、生成、多轮都有样本 |
| 模板提示 | seed + few-shot 示例 | 候选新指令 | 风格塌缩 | 多模板、多类别分桶 |
| 生成 | 新指令 | 输入/输出实例 | 幻觉、格式错 | 类型感知模板 |
| 去重 | 候选指令集 | 去重后指令 | 重复表达混入 | ROUGE-L 阈值 |
| 过滤 | 指令+实例 | 训练池 | 错答、脏数据 | 规则过滤、人工或执行验证 |

## 问题定义与边界

问题可以精确定义为：从自然语料或少量人工 seed 任务出发，自动构造可用于监督微调的高质量指令数据集，使模型学到“任务意图 + 输入条件 + 目标输出”的对应关系。

这里有几个边界必须先说清。

第一，Self-Instruct 的经典设定不是完全零人工，而是“只在起点写少量 seed，不再大规模人工编写新指令”。这和传统标注式数据生产不同，后者是人工持续写任务、写答案，质量高但扩展慢。

第二，去重不是可选项，而是定义任务分布的核心约束。设一个候选指令 $x$ 与已有指令集合 $\mathcal{D}$ 中任意指令 $d$ 的相似度为：

$$
S(x, \mathcal{D}) = \max_{d \in \mathcal{D}} \text{ROUGE-L}(x, d)
$$

只有当 $S(x, \mathcal{D}) < 0.7$ 时，$x$ 才作为新样本进入池中。这个阈值不是数学定律，而是经验上“去掉明显改写、保留语义新样本”的折中。

第三，模板类别决定生成顺序。分类任务，白话说就是从有限标签里选一个；生成任务是自由生成文本；多轮任务则要求上下文连续。因此不能用一套模板处理全部任务。比如“判断投诉类别”更适合先生成输入再生成标签，“写客服回复”则更适合直接生成约束明确的响应。

一个常见边界例子是：从 175 个种子任务中抽取 8-shot 提示，即一次给模型看 8 个示例，让它模仿格式继续写。如果新指令和已有指令的 ROUGE-L 已达到 0.7 以上，就算措辞不同，也不应再入库。

| 边界项 | 典型设定 | 含义 |
|---|---|---|
| 输入来源 | 少量人工 seed + 模型生成 | 不持续人工扩写大盘任务 |
| 去重策略 | $\max \text{ROUGE-L} < 0.7$ | 保证任务新颖性 |
| 模板类别 | 生成 / 分类 / 多轮 | 决定提示词结构 |
| 验证手段 | 规则、人工、执行验证 | 降低错答与无效样本 |
| 输出目标 | 指令-输入-响应三元组 | 直接服务 SFT 训练 |

## 核心机制与推导

Self-Instruct 的机制本质上是迭代自举。自举，白话说就是先靠小规模高质量样本起步，再让系统用自己的产出扩充自己。流程可以写成：

1. 从 seed 池采样若干任务。
2. 用 few-shot 模板让模型生成新指令。
3. 对新指令做去重与合法性过滤。
4. 为保留下来的指令生成输入与输出实例。
5. 再做一次实例级过滤。
6. 合格样本加入池中，进入下一轮。

如果只看表面，这像“模型自己给自己出题”。真正有效的部分在于分层约束：指令层控制覆盖，实例层控制可训练性，验证层控制正确性。

对客服机器人做玩具例子时，可以拆成两段模板。第一段模板生成“提取客户意图”的分类任务，例如把“快递晚到了三天”映射成“物流投诉”。第二段模板生成“写回复”的生成任务，例如要求“先致歉，再说明处理路径，禁止承诺无法兑现的补偿”。这比一次性让模型同时生成复杂多轮对话更稳定，因为分类和生成的信号被分开了。

真实工程例子则更接近 AIEV-Instruct 一类方法。它把 agent，白话说就是能分步骤交互执行任务的系统，放进数据生产环节。一个 agent 扮演提问者，另一个扮演执行者；执行者产出代码或回复后，系统再把结果送入 execution verification，也就是程序执行或单元测试验证。只有“能运行、能通过检查”的样本才保留。这个思想最先在代码场景非常突出，但泛化到客服、表单填报、工具调用也成立：能验证，就别只看文本像不像。

| 模板类 | 触发场景 | 产出 | 验证方式 |
|---|---|---|---|
| 分类模板 | 意图识别、标签判断 | 指令 + 输入 + 标签 | 标签集合校验、长度校验 |
| 生成模板 | 摘要、改写、回复 | 指令 + 输入 + 文本输出 | 格式规则、关键词约束 |
| 多轮模板 | 客服对话、工具调用 | 多轮上下文 + 回复 | 状态机检查、回合一致性 |
| 执行验证模板 | 代码、SQL、API 调用 | 可执行结果 | 单元测试、沙箱执行 |

## 代码实现

工程上通常拆成四个模块：模板定义、模型调用、去重过滤、验证接口。下面给一个可运行的 Python 玩具实现，只保留 Self-Instruct 最关键的骨架。

```python
from typing import List, Dict

def lcs_len(a: str, b: str) -> int:
    dp = [[0] * (len(b) + 1) for _ in range(len(a) + 1)]
    for i, ca in enumerate(a, 1):
        for j, cb in enumerate(b, 1):
            if ca == cb:
                dp[i][j] = dp[i - 1][j - 1] + 1
            else:
                dp[i][j] = max(dp[i - 1][j], dp[i][j - 1])
    return dp[-1][-1]

def rouge_l(a: str, b: str) -> float:
    if not a or not b:
        return 0.0
    lcs = lcs_len(a, b)
    return lcs / max(len(a), len(b))

def is_novel(candidate: str, pool: List[str], threshold: float = 0.7) -> bool:
    if not pool:
        return True
    max_score = max(rouge_l(candidate, x) for x in pool)
    return max_score < threshold

seed_tasks = [
    "摘要客户投诉",
    "识别客户意图",
]

generated_candidates = [
    "总结客户投诉内容",
    "根据投诉文本识别客户意图",
    "为客户投诉生成安抚回复",
]

accepted = []
for cand in generated_candidates:
    if is_novel(cand, seed_tasks + accepted, threshold=0.7):
        accepted.append(cand)

assert rouge_l("摘要客户投诉", "摘要客户投诉") == 1.0
assert is_novel("为客户投诉生成安抚回复", seed_tasks, 0.7) is True
assert len(accepted) >= 1

print(accepted)
```

它没有真正调用大模型，但已经体现了核心逻辑：从 `seed_tasks` 出发，生成候选指令，再用 ROUGE-L 做新颖性筛选。

伪代码层面，完整流程通常是：

```python
seed_tasks = load_json("seed_tasks.json")
pool = list(seed_tasks)

for round_id in range(num_rounds):
    prompt = render_template(pool, task_type="generation", k_shot=8)
    new_instructions = llm_generate(prompt)

    for inst in new_instructions:
        if not rouge_dedup(inst, pool, threshold=0.7):
            continue

        task_type = classify_instruction(inst)   # 分类 or 生成
        instance = generate_instance(inst, task_type)
        if not rule_filter(instance):
            continue
        if not verify(instance):                 # 人工/agent/执行验证
            continue

        pool.append(instance)
```

一个常见 JSON 结构如下：

```json
{
  "instruction": "阅读投诉内容并提取客户核心诉求",
  "input": "我买的耳机左耳没有声音，申请换货三天还没处理。",
  "output": "客户核心诉求是尽快完成换货，并确认售后进度。",
  "task_type": "generation"
}
```

## 工程权衡与常见坑

最大权衡是“多样性”和“噪声率”之间的冲突。多样性越高，模型学到的覆盖越广；但模板越松、生成越自由，错误样本也越多。Self-Instruct 官方仓库明确提醒，模型生成数据不可避免包含错误，并在随机检查中发现约 46% 的数据点可能存在问题。这不是说方法失效，而是说明“自动生成”必须配“强过滤”。

最常见的坑有四个。

第一，不做去重。结果是同一任务被改写成几十个表述混进训练集，表面规模增大，实际任务熵下降。任务熵，白话说就是任务分布的信息量；重复越多，熵越低，泛化越差。

第二，不区分任务类型。把分类、生成、多轮混在一套模板里，模型会频繁输出格式错误，例如该输出标签时写成长段解释。

第三，只验证文本流畅，不验证可执行性。对代码、SQL、工具调用类任务，这会保留大量“看起来合理、实际上不能跑”的样本。

第四，单一模板生成太久。模型会逐渐学会模板习惯，而不是任务本身，出现自我退化，即越生成越像自己之前的风格，覆盖反而缩窄。

| 设置 | 数据规模观感 | 实际质量 | 常见后果 |
|---|---|---|---|
| 无去重、无验证 | 很大 | 低 | 重复堆积、错答增多 |
| 有去重、无验证 | 中等 | 中 | 文本好看但不可执行 |
| 无去重、有验证 | 中等偏大 | 中 | 正确但任务分布重复 |
| 有去重、有验证 | 较稳 | 高 | 成本更高，但训练更有效 |

对新手最重要的一条经验是：如果不设 $\max \text{ROUGE-L} < 0.7$ 一类的阈值，“总结投诉内容”“概括投诉内容”“提炼投诉重点”会被重复收进训练集。模型看起来学了很多摘要任务，实际上只是反复见到同一种任务。

## 替代方案与适用边界

如果目标是快速做大规模通用数据，自动生成路线最有优势；如果目标是垂直场景高准确率，混合路线通常更稳。

Ensemble-Instruct 是一个重要替代方向。它用多模板和多模型混合生成，并把任务先分成“需要输入”和“不需要输入”两类，再做输出共识筛选。共识筛选，白话说就是让多个模型各写一版，只保留相互接近的答案。这个方法对中小模型尤其重要，因为小模型直接跑原版 Self-Instruct 往往更容易塌缩。

另一类替代方案是人工或半人工混合。比如客服场景，先让标注人员写高价值任务骨架，再让模型补充长尾变体；产品问答场景，则优先采样真实工单和人工写好的标准回复。规模会受限，但数据分布更接近生产环境。

| 方案 | 适合场景 | 成本 | 质量 | 边界 |
|---|---|---|---|---|
| 纯自动 Self-Instruct | 通用能力扩充、冷启动 | 低到中 | 中 | 噪声需强过滤 |
| Ensemble-Instruct | 中小模型、模板敏感场景 | 中 | 中到高 | 需要多模型协作 |
| 人工/自动混合 | 客服、金融、医疗等垂直域 | 中到高 | 高 | 规模不如纯自动 |
| 全人工标注 | 高风险、高合规场景 | 高 | 最高 | 扩展速度慢 |

因此，适用边界可以简单记成一句话：想要广覆盖，优先自动生成；想要强可靠，必须引入人工或执行验证；想要兼顾成本和质量，混合式通常是工程最优解。
