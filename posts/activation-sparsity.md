## 核心结论

Activation sparsity，中文可叫“激活稀疏性”，指的是一层神经网络算完激活函数后，输出里有很多元素是 `0`，或者小到可以按 `0` 处理。它的直接意义不是“模型参数更少”，而是“后续层不必对这些无效位置继续做乘加”。

对初学者可以先记一句直白解释：某些神经元在当前输入下没有工作，因为它们的输出被激活函数压成了 `0`。如果下一层知道这些位置是 `0`，就可以跳过对应计算。

以 ReLU 为例，定义是：

$$
\mathrm{ReLU}(x)=\max(0,x)
$$

这意味着所有负数输入都会直接变成 `0`。如果一层输出经过 ReLU 后有一半位置为 `0`，那么下一层理论上只需要处理剩下一半非零位置。稀疏率通常写成：

$$
s = 1 - \frac{\|a\|_0}{n}
$$

其中，$\|a\|_0$ 表示向量里非零元素的个数，$n$ 是总元素数。比如长度为 8 的激活向量里有 3 个非零值，则稀疏率是 $1-\frac{3}{8}=62.5\%$。

但结论必须补全一半：Activation sparsity 只有在“算子实现、数据布局、硬件执行路径”都支持稀疏执行时，才会真的转成延迟收益。否则它只是数学上少算，系统上未必更快。

下面这张表先把 dense 和 sparse 的核心差异放清楚：

| 项目 | Dense 计算 | Sparse 激活计算 |
|---|---|---|
| 处理对象 | 所有激活位置都参与计算 | 只处理非零激活位置 |
| 参数数量 | 不变 | 不变 |
| 理论乘加次数 | 固定 | 随非零数下降 |
| 访存模式 | 连续、规则 | 可能不连续，需要索引 |
| 加速前提 | 通用 BLAS/GEMM 即可 | 需要稀疏 kernel 或合适实现 |
| 典型风险 | 算得多但规则 | 少算了，但索引和访存开销可能变大 |

一个玩具例子可以先建立直觉。假设某层输出经过 ReLU 后得到：

$$
a=[3,0,0,5]^T
$$

如果下一层要计算 $y = Wa + b$，那么第 2、3 个位置对结果没有任何贡献，因为乘上 `0` 还是 `0`。本来要做 4 个位置的乘加，现在只需要处理第 1 和第 4 个位置。这就是“自动关掉一部分神经元”的含义。

真实工程里，Transformer 的 FFN（前馈网络）是最常见场景。中间激活维度通常很大，如果激活函数产生大量零值，理论上可以跳过很多无效乘法。在 ReLU 类模型上，合理实现下常见的量级是 30% 到 50% 的计算节省，但是否能稳定落成实际延迟收益，取决于硬件和稀疏执行路径。

## 问题定义与边界

要先把几个容易混淆的概念拆开。

Activation sparsity 是“激活输出稀疏”；weight sparsity 是“权重矩阵稀疏”；gradient sparsity 是“梯度更新稀疏”。它们作用在不同对象上，所以优化方法也不同。

“结构化稀疏”这个词，白话解释是“稀疏有固定形状，硬件更容易识别”；“无结构稀疏”是“零散地到处为 0，看起来更随机”。这两种都可能出现在权重或激活上，但工程含义差别很大。

| 概念 | 稀疏发生在哪 | 白话解释 | 主要优化目标 | 工程难点 |
|---|---|---|---|---|
| Activation sparsity | 激活输出 | 某次输入下很多神经元没工作 | 推理时少算后续层 | 稀疏模式随输入变化 |
| Weight sparsity | 模型参数 | 模型里很多权重本身就是 0 | 减参数、减乘加 | 训练和部署要一起适配 |
| Structured sparsity | 稀疏形状有规则 | 按块、按列、按通道变 0 | 更易映射到硬件 | 约束更强，精度风险更明确 |
| Unstructured sparsity | 稀疏位置零散 | 哪些位置为 0 不固定 | 理论压缩比高 | 索引和访存开销大 |

这篇文章讨论的是 activation sparsity，核心边界有三个。

第一，它不是“模型变小”。参数矩阵 $W$ 还在那里，模型存储大小通常不变。变的是某次输入下，有没有机会跳过一部分参与计算的列。

第二，它不是“只要出现 0 就一定快”。如果稀疏率不够高，或者非零位置分布很乱，系统可能需要先扫描、压缩、维护索引，再做不连续访存。这样额外成本可能比直接 dense GEMM 还高。

第三，它的收益高度依赖场景。对新手最重要的一句判断是：只有当“很多神经元当前不工作”，并且“下一层真的知道怎么跳过它们”，activation sparsity 才会变成真实加速。

一个常见误区是把“少做乘法”直接等同于“整体更快”。这在现代硬件上不成立，因为速度还由内存带宽、cache 命中率、kernel 启动成本、batch size 等因素共同决定。尤其在 GPU 上，规则的大矩阵乘法通常已经被优化得非常极致，稀疏路径如果破坏了规则性，未必占优。

## 核心机制与推导

先从最简单的形式开始。设某层线性输出为 $x$，激活后得到：

$$
a = f(x) = \mathrm{ReLU}(x)
$$

由于 ReLU 会把所有负值压成 `0`，所以 $a$ 中通常会出现大量零值。若 $a \in \mathbb{R}^n$，稀疏率定义为：

$$
s = 1 - \frac{\|a\|_0}{n}
$$

这里 $\|a\|_0$ 不是严格意义上的“范数”，而是“非零元素个数”的记号。它表达的就是：总共有多少位置真正还在工作。

下一层若是线性层：

$$
y = Wa + b
$$

其中 $W \in \mathbb{R}^{m \times n}$，$a \in \mathbb{R}^n$，则按列展开可写成：

$$
y = \sum_{i=1}^{n} W_{:,i} a_i + b
$$

如果某个 $a_i = 0$，那么对应项 $W_{:,i}a_i$ 整列贡献都是 `0`，可以直接去掉，于是改写为：

$$
y = \sum_{i:a_i\neq 0} W_{:,i} a_i + b
$$

这就是 activation sparsity 能“跳过列乘加”的数学来源。

看一个明确的数值例子。设：

$$
a = [2,0,-1,0]^T
$$

$$
W=
\begin{bmatrix}
1 & 2 & 5 & 9\\
2 & 4 & 6 & 8
\end{bmatrix}, \quad b=[0,0]^T
$$

dense 算法会做：

$$
y = W a
= 2W_{:,1} + 0W_{:,2} + (-1)W_{:,3} + 0W_{:,4}
$$

代入后得到：

$$
y = 2
\begin{bmatrix}
1\\
2
\end{bmatrix}
-
\begin{bmatrix}
5\\
6
\end{bmatrix}
=
\begin{bmatrix}
-3\\
-2
\end{bmatrix}
$$

dense 路径会把 4 列都读出来，并做 8 次乘法；sparse 路径只需要处理第 1 列和第 3 列，共 4 次乘法。这里“少算”很直观，但更关键的是“少读了第 2、4 列对应的数据”。

把压缩过程写成表更清楚。原始激活是：

| 原始位置 `i` | 0 | 1 | 2 | 3 |
|---|---|---|---|---|
| 激活值 `a_i` | 2 | 0 | -1 | 0 |

压缩后通常变成两份数据：

| 压缩后内容 | 值 |
|---|---|
| `values` | `[2, -1]` |
| `indices` | `[0, 2]` |

执行时不再遍历整个长度为 4 的向量，而是只遍历 `values[k]` 和 `indices[k]`，并访问 `W[:, indices[k]]`。这就是“压缩非零值 + 保存索引”的基本机制。

这里要强调一个工程事实：压缩本身也有成本。你要先扫描激活，找出哪些位置非零；还要保存索引；后续访问矩阵列时，地址可能不连续。也就是说，activation sparsity 的收益来自：

$$
\text{省下的乘加与访存} > \text{扫描、索引、分支、乱序访存的额外成本}
$$

如果这个不等式不成立，就不会加速。

真实工程例子是 LLM 的 FFN 层。设输入 token 经过上投影后进入 ReLU 或 squared-ReLU，得到中间激活 $a$。若 $a$ 很稀疏，则下投影层的计算可以重写为只对非零激活求和。这类场景在 `batch size = 1` 的在线解码阶段尤其有意义，因为此时系统更容易受访存和单步延迟限制，而不是受大 batch 的矩阵吞吐限制。

## 代码实现

教学版实现先看最小逻辑。关键不是“把 0 过滤掉”这句话本身，而是“过滤后如何让后续访问矩阵时仍然正确”。

```python
def dense_linear(a, W, b):
    out_dim = len(W)
    in_dim = len(a)
    y = b[:]
    for r in range(out_dim):
        acc = y[r]
        for c in range(in_dim):
            acc += W[r][c] * a[c]
        y[r] = acc
    return y


def sparse_linear(a, W, b):
    out_dim = len(W)
    y = b[:]

    values = []
    indices = []
    for i, v in enumerate(a):
        if v != 0:
            values.append(v)
            indices.append(i)

    for k, v in enumerate(values):
        i = indices[k]
        for r in range(out_dim):
            y[r] += W[r][i] * v
    return y


a = [2, 0, -1, 0]
W = [
    [1, 2, 5, 9],
    [2, 4, 6, 8],
]
b = [0, 0]

dense = dense_linear(a, W, b)
sparse = sparse_linear(a, W, b)

assert dense == [-3, -2]
assert sparse == [-3, -2]
assert dense == sparse
```

这段代码能运行，也说明了一个基础事实：sparse 实现首先要提取非零值和索引，然后才能正确访问 `W[:, i]`。

如果写成更接近伪代码的形式，就是：

```python
def sparse_linear(a, W, b):
    y = b.copy()
    for i, v in enumerate(a):
        if v != 0:
            y += W[:, i] * v
    return y
```

dense 与 sparse 的代码路径差别可以概括成下表：

| 维度 | Dense 实现 | Sparse 实现 |
|---|---|---|
| 遍历方式 | 遍历全部输入维度 | 只遍历非零索引 |
| 数据结构 | 普通向量 | `values + indices` |
| 矩阵访问 | 连续、固定模式 | 按索引取列，可能不连续 |
| 控制流 | 几乎无分支 | 需要判断或预压缩 |
| 工程依赖 | 常规 GEMM 库 | 稀疏 kernel 或定制实现 |

但教学版到工程版之间还有很长一段距离。真正部署时，往往不会手写 Python 循环，而是交给支持稀疏执行的 kernel、图编译器或硬件库处理。因为手写循环虽然概念正确，但性能通常极差。

工程上至少要考虑三件事。

第一，零值阈值。浮点数里很多值不是精确 `0`，而是“非常接近 `0`”。因此实际系统常用阈值规则，比如 `abs(v) < eps` 就视为零。

第二，布局选择。是按元素存索引，还是按块压缩，决定了访存是否连续。块状稀疏通常更容易映射到硬件。

第三，与后续算子融合。如果上一层激活和下一层线性层能在同一个 kernel 里处理，那么不必先把完整激活写回显存，再读出来筛零，收益才更容易落地。

## 工程权衡与常见坑

Activation sparsity 最大的误区，是把“稀疏”理解成“天然快”。实际情况相反：无结构稀疏经常很难跑赢 dense。

对初学者可以用一句很直白的话理解：如果每次都要先找哪些位置是 `0`，再决定跳过哪些计算，那么“找 0 的成本”本身可能就已经不小。尤其当真正为 `0` 的位置不够多时，系统做了一堆判断，最后省下的计算却很有限。

下面这张表可以作为经验判断框架：

| 维度 | 更可能快的情况 | 更可能不快的情况 |
|---|---|---|
| 稀疏率 | 很高，例如超过 50% 且稳定 | 只有少量零值，波动大 |
| 结构化程度 | 块稀疏、列稀疏等规则模式 | 完全无结构、位置随机 |
| 硬件支持 | 库和硬件有专门稀疏路径 | 只能自己做索引遍历 |
| Batch size | 小 batch、单步解码 | 大 batch、吞吐优先场景 |

常见坑主要有五类。

第一，只看单个 GEMM benchmark，不看端到端延迟。真实系统里，前后还有激活、归一化、残差、数据搬运和 kernel 启动。单个算子省下 40%，不代表整条链路省下 40%。

第二，忽略 batch size。大 batch 时，dense GEMM 往往能把硬件吃满，规则访问的优势很大；小 batch 时，稀疏跳过才更可能带来收益。

第三，把无结构稀疏当成免费午餐。无结构稀疏压缩率高，但访存最差；结构化稀疏压缩率可能略低，但更容易真正变快。这是典型的“数学最优”不等于“系统最优”。

第四，只统计 FLOPs，不统计 memory traffic。很多推理瓶颈其实不是乘法次数，而是数据搬运。激活稀疏如果不能减少显存读写，收益会被高估。

第五，不区分训练和推理。训练阶段要保留更多中间状态，反向传播也更复杂；推理阶段尤其是在线解码，才更适合把 activation sparsity 变成低延迟优化。

真实工程里，一个比较可靠的做法是：先量化稀疏率分布，再测端到端延迟，再决定是否投入稀疏 kernel 适配。顺序不能反。否则很容易在“理论很美”的优化上花掉大量工程时间，最终收益不稳定。

## 替代方案与适用边界

如果 activation sparsity 不足以稳定获益，通常不会继续死磕，而是看有没有更容易落地的替代路线。

ReLU 或 squared-ReLU 这类方法，本质上是在“激活函数层面更自然地产生稀疏”；结构化稀疏则是在“模型形状层面主动施加规则”。前者更灵活，后者更容易映射到硬件。

| 方法 | 主要收益 | 实现难度 | 适用场景 | 风险 |
|---|---|---|---|---|
| Activation sparsity | 减少无效乘加 | 中到高 | ReLU 类模型、小 batch 推理 | 稀疏模式不稳定，依赖实现 |
| 结构化稀疏 | 更容易被硬件利用 | 高 | 需要明确部署目标的模型 | 训练约束强，可能掉点 |
| 量化 | 降低带宽和存储 | 中 | 推理部署最常见 | 精度校准成本 |
| 低秩分解 | 降低矩阵乘规模 | 中到高 | 大线性层压缩 | 近似误差可能明显 |
| 算子融合 | 减少中间访存和 kernel 开销 | 中到高 | 工程部署优化 | 平台适配复杂 |

适用边界也要分场景看。

训练阶段，activation sparsity 的收益往往不如推理直接，因为训练还涉及反向传播、梯度存储和优化器状态。离线批量推理时，大 batch dense kernel 已经很强，稀疏优化未必是首选。在线解码时，单 token 延迟更重要，activation sparsity 才更可能成为有效手段。

再给一个对比理解：

- 如果你的目标是“模型自然产生大量零激活”，可以考虑 ReLU、squared-ReLU 这类路线。
- 如果你的目标是“硬件稳定识别并加速”，结构化稀疏、量化、融合通常更容易形成稳定收益。

所以 activation sparsity 更适合这样的问题：模型本身已经使用 ReLU 类激活，中间层维度大，推理是小 batch 或在线场景，并且你手上有支持稀疏执行的实现路径。若这些条件缺一半以上，替代方案通常更现实。
