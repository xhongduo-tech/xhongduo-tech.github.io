## 核心结论

MoE，Mixture of Experts，指“很多专家子网络里，每次只激活少数几个”的模型结构。它的负载均衡问题，本质上不是“有些专家忙一点”这么简单，而是路由器会把已经经常被选中的专家继续推高分，形成正反馈，最终让少数专家吞掉大部分 token，其余专家几乎拿不到梯度，出现“专家坍缩”。

负载均衡的目标，就是在这条正反馈链上加约束，让每个专家都能持续收到训练样本。这样做不是为了机械平均，而是为了保护专家多样性。只有不同专家都被真正训练过，MoE 才能在同样计算预算下，比普通稠密模型学到更多模式。

先把结论说清楚：负载均衡优化的不是“看起来平均”，而是“让更多专家真正参与学习”。如果只有少数专家长期工作，MoE 的参数规模虽然大，但可用容量其实很小，训练出来的效果会更像“少量热点专家 + 大量闲置参数”，而不是一个真正具备分工能力的专家系统。

玩具例子可以这样理解：排队点餐时，服务员总把订单交给当前最快的厨师。这个厨师因为接单最多，练习也最多，于是越来越快；其他厨师因为几乎没有订单，越来越跟不上。长期看，厨房不是“自动优化”了，而是退化成“只有一个人真在干活”。负载均衡就是强制让其他厨师也持续接到订单，避免系统过早锁死。

这个问题在大模型里更严重。因为 token 数量极大、训练轮数极多，早期一个很小的路由偏差，经过几万步甚至几十万步训练后，会被放大成明显的专家失衡。你看到的表面现象是“某些专家很强”，但更准确的说法是“某些专家因为一直有数据，才有机会变强”。

下表给出“无负载均衡”和“有负载均衡”时常见的差异：

| 现象 | 无负载均衡 | 有负载均衡 |
| --- | --- | --- |
| 专家选择频率 | 少数专家极高，其他专家极低 | 更接近均匀分布 |
| 梯度强度 | 高频专家持续增强 | 梯度分散到更多专家 |
| 专家能力演化 | 强者更强，弱者长期学不到 | 专家逐步形成分工 |
| 通信与调度 | 容易出现热点专家和 all-to-all 倾斜 | 负载更平滑，吞吐更稳 |
| 训练风险 | 专家坍缩、通信热点、泛化下降 | 收敛更稳，容量利用更充分 |

## 问题定义与边界

路由器，router，指“决定一个 token 送去哪个专家”的小网络。它先给每个专家打分，再通过 softmax 变成概率，最后取 Top-K 专家执行。负载均衡讨论的不是“概率本身”，而是“概率”和“真实分配结果”之间的长期偏差。

这里先区分三个容易混淆的概念：

| 概念 | 含义 | 是否可微 | 是否直接决定专家计算 |
| --- | --- | --- | --- |
| logits | 路由器对每个专家的原始打分 | 是 | 否 |
| probabilities | softmax 后的概率分布 | 是 | 间接 |
| assignments | Top-K 后真实选中的专家 | 通常不可微或近似可微 | 是 |

很多新手第一次看 MoE 时，会把“概率高”直接等同于“分配多”。但两者并不完全一样。因为实际执行阶段通常有 Top-K、容量限制、丢 token 或重路由等机制，最终落到每个专家身上的真实流量，和 softmax 概率并不总是一致。

通常要区分两个量：

| 指标 | 含义 | 监控意义 | 常见异常阈值 |
| --- | --- | --- | --- |
| $f_i$ | 专家 $i$ 的实际分配占比 | 看真实流量是否过载 | 单专家长期 $>0.5$ 需警报 |
| $P_i$ | 路由器对专家 $i$ 的平均概率质量 | 看路由偏好是否过尖 | 单专家长期 $>0.5$ 需排查 |
| $f_i-P_i$ | 实际负载与预测偏好的差值 | 看路由与分配是否失配 | $|f_i-P_i|>0.3$ 常需处理 |

这三个量分别回答三个不同问题：

| 问题 | 主要看哪个量 |
| --- | --- |
| 路由器是不是偏爱少数专家 | $P_i$ |
| 真实流量是不是已经失衡 | $f_i$ |
| 概率和实际分配是不是脱节 | $f_i-P_i$ |

这里的边界很重要。负载均衡不是要求所有专家永远严格按 $1/N$ 平均。真实训练中，不同 token 的难度和类型本来就不同，允许短期偏斜；真正危险的是某类 token 长期只流向极少数专家，导致剩余专家参数几乎不更新。此时模型表面上还是 MoE，实际却在退化成“小部分稠密网络 + 一堆闲置参数”。

另一个边界是：负载均衡优化的是“训练可用性”和“专家利用率”，不是直接优化最终任务指标。辅助约束太弱，专家会坍缩；辅助约束太强，所有专家又会变得太像，专门化能力下降。所以它是一个约束问题，不是越强越好。

如果把路由器想成轮盘，$P_i$ 是轮盘给某个专家的权重，$f_i$ 是最后实际落到这个专家上的球数。负载均衡做的事，不是让每个格子完全一样，而是防止某个格子越来越大，最后几乎所有球都只落进去。

工程上还要再加一个边界：负载均衡不只影响“学得好不好”，还影响“跑得稳不稳”。当热点专家集中在少数设备上时，all-to-all 通信、显存峰值、局部排队都会恶化。所以它既是优化问题，也是系统问题。

## 核心机制与推导

MoE 负载不均的核心机制可以概括成三步：

1. 路由器给某些专家更高分。
2. 这些专家被选中的次数更多，拿到更多训练样本和梯度。
3. 因为训练更多，它们对类似 token 的表现继续变好，下一轮又更容易被选中。

这就是正反馈。它可以用一条很短的因果链表示：

$$
\text{更高打分} \rightarrow \text{更多分配} \rightarrow \text{更多梯度} \rightarrow \text{更强专家} \rightarrow \text{下一轮更高打分}
$$

如果没有外部约束，这条链条会自然把流量推向少数热点专家。

数学上，常见的辅助负载均衡损失写成：

$$
L_\text{load}=N\sum_{i=1}^{N} f_i P_i
$$

其中，$N$ 是专家数，$f_i$ 是专家 $i$ 的实际负载占比，$P_i$ 是该专家在当前 batch 的平均路由概率。

这个式子的直觉并不复杂。若某个专家同时“实际分配很多”且“概率偏好也很高”，那么 $f_i P_i$ 就会很大；若所有专家都差不多均衡，那么每一项都接近 $\frac{1}{N^2}$，总和就接近理想值。

理想均匀时，$f_i=P_i=\frac{1}{N}$，于是：

$$
L_\text{load}=N\sum_{i=1}^{N}\frac{1}{N}\cdot\frac{1}{N}=1
$$

所以这个量越接近 1，说明负载越均衡；越大，说明少数专家越集中。

为什么它最小值是 1，可以再展开一步。因为当两个分布都均匀时：

$$
\sum_{i=1}^{N} f_iP_i=\sum_{i=1}^{N}\frac{1}{N^2}=\frac{1}{N}
$$

再乘上外面的 $N$，就得到 1。工程上常把“是否显著高于 1”当作是否失衡的粗略信号。

看一个玩具例子。4 个专家时，某个 batch 统计到：

$$
f=[0.7,0.1,0.1,0.1],\quad P=[0.8,0.05,0.05,0.1]
$$

代入得：

$$
L_\text{load}=4(0.7\times0.8+0.1\times0.05+0.1\times0.05+0.1\times0.1)=2.34
$$

2.34 明显高于理想值 1，说明第 1 个专家同时在“被偏爱”和“被过载”。

再看一个接近均衡的例子：

$$
f=[0.28,0.24,0.25,0.23],\quad P=[0.30,0.23,0.24,0.23]
$$

则：

$$
L_\text{load}=4(0.28\times0.30+0.24\times0.23+0.25\times0.24+0.23\times0.23)\approx 1.01
$$

这说明即使不是严格均匀，只要专家负载接近均衡，$L_\text{load}$ 也会非常接近 1。

训练时通常把总损失写为：

$$
L_\text{total}=L_\text{task}+\alpha L_\text{load}
$$

其中 $\alpha$ 是辅助损失系数。$\alpha$ 越大，模型越被迫把流量摊开；但过大时，专家会被推向“平均化”，专长减弱。

这个式子可以这样理解：

| 项 | 作用 |
| --- | --- |
| $L_\text{task}$ | 让模型完成主任务，比如语言建模、分类、生成 |
| $L_\text{load}$ | 惩罚负载过度集中 |
| $\alpha$ | 控制“主任务”和“负载均衡”之间的权重 |

第二类方法是 Noisy Top-K。它在路由 logits 上加随机噪声：

$$
h_\text{noisy}(x)=h(x)+\epsilon,\quad \epsilon\sim\mathcal{N}(0,\sigma^2)
$$

这里 $h(x)$ 是原始打分，$\epsilon$ 是均值为 0 的高斯噪声。很多实现会让 $\sigma$ 固定，也有实现会让它可学习。白话说，就是在“差一点就落选”的专家之间加入少量随机性，让早期略强的专家不要永远把其他专家压死。它尤其适合训练前期，因为前期专家能力还没真正拉开。

噪声为什么有用，可以从排序角度理解。Top-K 最怕“始终是同一批人排在最前面”。一旦加入小噪声，原本分数非常接近的几个专家就会轮流入选，未被充分训练的专家也能拿到一部分样本。这种机制本质上是在增加探索，避免系统过早进入局部稳定状态。

第三类方法是温度调节。softmax 带温度的形式是：

$$
\mathrm{softmax}_T(x_i)=\frac{\exp(x_i/T)}{\sum_j\exp(x_j/T)}
$$

温度 $T$ 控制分布尖锐度。这里要注意一个容易写反的事实：$T$ 越大，分布越平缓；$T$ 越小，分布越尖锐。也就是说，如果目标是缓解过度集中，通常是适当提高温度，而不是降低温度。提高温度会削弱“赢家通吃”，但也可能降低专家专门化速度。

看一个非常短的数值例子。设两个专家 logits 为 $[4,2]$：

- 当 $T=1$ 时，softmax 约为 $[0.881, 0.119]$
- 当 $T=2$ 时，softmax 约为 $[0.731, 0.269]$
- 当 $T=0.5$ 时，softmax 约为 $[0.982, 0.018]$

这个例子说明，温度升高会把分布拉平，温度降低会让高分专家更占优势。

下表展示温度和噪声对分布尖锐度的典型影响：

| 配置 | 分布尖锐度 | 负载均衡效果 | 代价 |
| --- | --- | --- | --- |
| $T=0.7$，无噪声 | 很尖锐 | 差 | 专家更快分化，但更易坍缩 |
| $T=1.0$，无噪声 | 中等 | 一般 | 默认基线 |
| $T=1.5$，无噪声 | 更平缓 | 较好 | 专门化减慢 |
| $T=1.0$，小噪声 | 中等偏平 | 较好 | 训练更稳 |
| $T=1.5$，小噪声 | 平缓 | 好 | 可能牺牲峰值性能 |

真实工程例子是大模型训练中的热点专家问题。假设一个 64 专家的 MoE 层里，和“通用语法”“常见事实”相关的 token 占多数，那么少数处理通用模式的专家会被频繁命中，形成通信热点，GPU 间 all-to-all 传输也会更不均匀。此时负载均衡不仅影响精度，还直接影响吞吐和训练稳定性。

如果把这一节压缩成一句话，就是：辅助损失负责“惩罚过载”，噪声负责“增加探索”，温度负责“控制分布尖锐度”，三者共同作用于同一个问题，即抑制路由正反馈。

## 代码实现

下面用一个可运行的 Python 玩具实现，演示如何计算路由概率、实际分配比例、噪声路由和负载均衡损失。代码只依赖标准库，可以直接保存为 `moe_load_balance_demo.py` 运行。

```python
import math
import random
from typing import List, Tuple


def softmax(xs: List[float], temperature: float = 1.0) -> List[float]:
    if temperature <= 0:
        raise ValueError("temperature must be > 0")

    scaled = [x / temperature for x in xs]
    m = max(scaled)
    exps = [math.exp(x - m) for x in scaled]
    total = sum(exps)
    return [x / total for x in exps]


def topk_indices(probs: List[float], k: int) -> List[int]:
    if not 1 <= k <= len(probs):
        raise ValueError("k must be in [1, num_experts]")
    return sorted(range(len(probs)), key=lambda i: probs[i], reverse=True)[:k]


def route_batch(
    logits_batch: List[List[float]],
    num_experts: int,
    k: int = 1,
    temperature: float = 1.0,
    noise_std: float = 0.0,
    seed: int = 0,
) -> Tuple[List[float], List[float], float]:
    rng = random.Random(seed)

    assigned = [0 for _ in range(num_experts)]
    prob_mass = [0.0 for _ in range(num_experts)]

    for logits in logits_batch:
        if len(logits) != num_experts:
            raise ValueError("logit length does not match num_experts")

        noisy_logits = [x + rng.gauss(0.0, noise_std) for x in logits]
        probs = softmax(noisy_logits, temperature=temperature)

        for i, p in enumerate(probs):
            prob_mass[i] += p

        selected = topk_indices(probs, k)
        for idx in selected:
            assigned[idx] += 1

    total_tokens = len(logits_batch)
    total_assignments = total_tokens * k

    f = [count / total_assignments for count in assigned]
    P = [mass / total_tokens for mass in prob_mass]
    load_loss = num_experts * sum(fi * pi for fi, pi in zip(f, P))
    return f, P, load_loss


def max_imbalance(values: List[float], target: float) -> float:
    return max(abs(v - target) for v in values)


if __name__ == "__main__":
    # 12 个 token，4 个专家。专家 0 在基线配置下会明显更热门。
    logits_batch = [
        [3.2, 1.1, 0.8, 0.7],
        [3.0, 1.2, 0.5, 0.4],
        [2.8, 1.1, 0.9, 0.3],
        [3.5, 0.8, 0.6, 0.5],
        [3.1, 1.0, 0.7, 0.6],
        [2.9, 1.3, 0.8, 0.2],
        [3.3, 1.1, 0.4, 0.3],
        [3.4, 0.9, 0.5, 0.4],
        [3.0, 1.0, 0.9, 0.6],
        [3.1, 1.2, 0.6, 0.5],
        [2.7, 1.4, 0.8, 0.4],
        [3.2, 1.0, 0.7, 0.5],
    ]

    baseline = route_batch(
        logits_batch,
        num_experts=4,
        k=1,
        temperature=1.0,
        noise_std=0.0,
        seed=42,
    )
    balanced = route_batch(
        logits_batch,
        num_experts=4,
        k=1,
        temperature=1.5,
        noise_std=0.3,
        seed=42,
    )

    f0, P0, loss0 = baseline
    f1, P1, loss1 = balanced

    assert abs(sum(f0) - 1.0) < 1e-9
    assert abs(sum(P0) - 1.0) < 1e-9
    assert abs(sum(f1) - 1.0) < 1e-9
    assert abs(sum(P1) - 1.0) < 1e-9
    assert loss0 >= 1.0
    assert loss1 >= 1.0

    target = 1.0 / 4
    print("baseline")
    print("  f =", [round(x, 4) for x in f0])
    print("  P =", [round(x, 4) for x in P0])
    print("  load_loss =", round(loss0, 4))
    print("  max |f_i-1/N| =", round(max_imbalance(f0, target), 4))

    print("balanced")
    print("  f =", [round(x, 4) for x in f1])
    print("  P =", [round(x, 4) for x in P1])
    print("  load_loss =", round(loss1, 4))
    print("  max |f_i-1/N| =", round(max_imbalance(f1, target), 4))
```

这段代码对应训练中的三个关键步骤：

1. `softmax(noisy_logits, temperature)` 先处理温度和噪声。
2. `topk_indices` 模拟 Top-K 路由。
3. `f` 和 `P` 分别统计真实分配占比与平均概率，再计算 `load_loss`。

如果你直接运行，大概率会看到这种趋势：

| 配置 | 现象 |
| --- | --- |
| 基线：`T=1.0`，无噪声 | 专家 0 的 $f_i$ 和 $P_i$ 明显更高，`load_loss` 更大 |
| 平衡：`T=1.5`，有噪声 | 分布变平，`load_loss` 更接近 1 |

这里要强调一个新手常见误区：温度和噪声不保证“每次都更均衡”，它们只是提高整体上更均衡的概率。因为有随机性，单个 batch 可能仍然偏斜，所以工程上看的是长期统计，而不是单步结果。

如果换成训练框架里的伪代码，核心结构通常是：

```python
logits = router(x)                                # [batch, num_experts]
noisy_logits = logits + randn_like(logits) * noise_std
probs = softmax(noisy_logits / temperature, dim=-1)
topk_idx = topk(probs, k)

assigned_frac = count_assignments(topk_idx) / (batch_size * k)   # 对应 f_i
mean_prob = probs.mean(dim=0)                                    # 对应 P_i

load_loss = num_experts * (assigned_frac * mean_prob).sum()
total_loss = task_loss + alpha * load_loss
```

如果进一步接近真实工程，还会出现容量限制。设每个专家最多接收 `capacity` 个 token，则伪代码会再多一步：

```python
topk_idx = topk(probs, k)
topk_idx = apply_capacity_limit(topk_idx, capacity)
```

一旦加入容量限制，就更需要同时看 $P_i$ 和 $f_i$。因为此时路由器可能非常偏爱某个专家，但真实分配已经被容量裁掉，表现为“概率很偏，实际流量被迫截断”。如果只看一个指标，很容易误判。

建议日志里至少记录这些量：

| 日志项 | 作用 |
| --- | --- |
| `max(f_i)` | 看最热点专家是否过载 |
| `entropy(P)` | 看路由分布是否过尖 |
| `load_loss` | 看整体负载均衡程度 |
| `drop_rate` 或 `overflow_rate` | 看容量限制是否在丢 token |
| 每个专家的 token 数量 | 看是否存在长期闲置专家 |

这里 `entropy(P)` 指概率分布熵。熵越低，说明路由越尖锐；熵越高，说明路由越平缓。它不是标准负载均衡指标，但很适合做辅助诊断。

## 工程权衡与常见坑

负载均衡最常见的误区，是把它当成一个固定超参数问题。实际上它更像一个动态控制问题，要结合训练阶段看。

下表列出常见配置失误：

| 配置失误 | 典型表现 | 原因 | 建议整改 |
| --- | --- | --- | --- |
| $\alpha$ 太小 | 少数专家长期过载 | 辅助约束压不住正反馈 | 逐步提高 $\alpha$，观察 $f_i$ 曲线 |
| $\alpha$ 太大 | 专家输出趋同，主任务变差 | 过度平均化，压制专门化 | 设上下限，必要时退回较小值 |
| 不加噪声 | 早期就锁定热点专家 | 初始微小优势被放大 | 训练前期启用 Noisy Top-K |
| 温度过低 | 路由极尖锐 | softmax 赢家通吃 | 提高 $T$ 到 1.2 到 1.5 区间试验 |
| 只看总 loss | 训练看似正常，实际专家坍缩 | 负载问题被主任务掩盖 | 单独监控 $f_i$、$P_i$、最大负载 |
| 只做概率均衡 | 实际通信仍有热点 | 概率与真实分配不等价 | 同时统计路由结果和设备流量 |

一个典型坑是：训练初期设 $\alpha=0.1$，发现负载还是很偏，于是直接调到很大，比如 1.0。结果是过载缓解了，但专家学出来越来越像，最终主任务指标下降。更稳妥的做法通常是逐步调大，例如从 0.1 到 0.2，再到 0.3，同时看 $f_i$ 的最大值是否从 0.6 以上回落到 0.25 到 0.35 区间。

为什么不能一下子把 $\alpha$ 拉很大，可以从梯度角度理解。总损失是：

$$
L_\text{total}=L_\text{task}+\alpha L_\text{load}
$$

当 $\alpha$ 过大时，优化器会把很多更新都用来“摊平流量”，而不是“提升任务效果”。这会导致路由器优先学会“别太偏”，但没有学会“该把什么 token 送给什么专家”。结果就是负载看着更漂亮，任务表现反而变差。

另一个坑是把温度理解反了。很多人会误以为“降低温度能让分布更均匀”。事实相反，降低温度会让高分专家更占优势；如果目的是缓解负载集中，应考虑适当提高温度，或者只在训练早期用较高温度，后期再慢慢退火，让专家重新形成更稳定的分工。

一个实用做法是分阶段控制：

| 训练阶段 | 常见策略 |
| --- | --- |
| 前期 | 较高温度 + 小噪声 + 较强均衡约束，优先防止早期锁死 |
| 中期 | 逐步降低温度或噪声，让专家开始形成分工 |
| 后期 | 适度减弱均衡约束，避免过度平均化压制专长 |

还有一个工程现实是：仅靠 softmax + Top-K，梯度集中几乎是必然趋势。所以很多系统会把“噪声、温度、辅助损失、偏置更新”组合使用，而不是只押一个手段。

再补两个常见坑：

| 坑 | 后果 | 原因 |
| --- | --- | --- |
| 只看 batch 内均衡 | 长期仍然坍缩 | 单个 batch 偶然均衡，不代表长期均衡 |
| 忽略设备级热点 | 模型指标正常但吞吐很差 | 专家分布与并行切分方式耦合 |

所以实际监控往往有两层：

1. 模型层：看 $f_i$、$P_i$、`load_loss`、熵。
2. 系统层：看每张卡的 token 数、通信时间、overflow、尾延迟。

如果只做前一层，你会知道“路由偏了”；如果只做后一层，你会知道“系统慢了”；两层一起看，才知道慢是因为路由偏，还是因为并行切分、带宽、容量配置本身有问题。

## 替代方案与适用边界

标准辅助损失并不是唯一方案。它的优点是直接、清晰、容易实现；缺点是它会把额外梯度注入训练目标，可能干扰主任务。

一种替代思路是 Loss-Free Balancing。它不显式加辅助损失，而是维护一个路由偏置 bias，根据每个专家最近是否过载来更新偏置。白话说，就是不改主任务优化方向，而是在路由入口处做“流量阀门”。这种方法适合主任务很敏感、不希望被辅助 loss 扰动的场景。

它的直觉可以写成：

$$
\text{adjusted\_logit}_i = \text{logit}_i + b_i
$$

其中 $b_i$ 是随时间更新的偏置。如果某个专家最近太忙，就把它的偏置调低一点；如果某个专家长期闲置，就把它的偏置调高一点。这样做改变的是“路由前的倾向”，而不是“主损失里的梯度方向”。

一个简化的更新思路可以写成：

$$
b_i \leftarrow b_i - \eta(f_i-\tfrac{1}{N})
$$

其中 $\eta$ 是偏置更新步长。若 $f_i>\frac{1}{N}$，说明专家 $i$ 过载，则降低其偏置；若 $f_i<\frac{1}{N}$，说明它偏闲，则提高偏置。这个公式不一定是具体实现，但能帮助理解它的控制逻辑。

另一种重要方案是 DeepSeekMoE 的共享专家设计。共享专家，shared experts，指“无论路由结果如何都始终激活的一组专家”。它的核心思想是把通用知识从竞争式路由里隔离出来：常见模式、基础语言能力、通用事实，由共享专家稳定承接；其余路由专家只学习差异化模式。这样做的好处是，热门 token 不再全部挤占可路由专家，负载压力显著下降。

这类结构可以粗略理解成：

$$
y = y_\text{shared} + y_\text{routed}
$$

其中：

- $y_\text{shared}$ 由共享专家产生，处理通用模式
- $y_\text{routed}$ 由路由专家产生，处理差异化模式

这样一来，常见 token 不必全靠“抢热门专家”完成，热点被部分分流，路由专家也更容易形成真正的专长。

真实工程例子是低延迟推理系统。假设在线服务对尾延迟非常敏感，那么你不希望每次都因为个别热点专家过载而出现路由抖动和通信峰值。保留若干常开共享专家，可以让“人人都会碰到的通用 token”有稳定去处，而差异化专家只处理真正需要专长的部分。这样既减少热点，也降低调度复杂度。

下表对比几种方案的适用边界：

| 方案 | 主要机制 | 是否干扰主 loss | 是否需要持续监控 | 适合场景 |
| --- | --- | --- | --- | --- |
| 标准 Aux Loss | 加 $L_\text{load}$ 惩罚过载 | 会 | 需要 | 训练基线、实现简单 |
| Noise + Temperature | 平滑路由分布、增加探索 | 轻度 | 需要 | 训练前期、防早期坍缩 |
| Loss-Free Balancing | 用 bias 调整路由倾向 | 较少 | 需要 | 主任务敏感、想减少辅助梯度 |
| DeepSeekMoE | 共享专家承接通用知识 | 间接 | 需要 | 大规模训练、在线推理、热点明显 |

再补一张“怎么选”的经验表：

| 场景 | 优先方案 |
| --- | --- |
| 刚搭一个 MoE 基线，先跑通训练 | 标准 Aux Loss |
| 前期经常出现专家快速锁死 | Noise + Temperature |
| 主任务对辅助损失很敏感 | Loss-Free Balancing |
| 热点 token 很多，通信压力大 | 共享专家或结构性改造 |

适用边界也要说清楚。若专家数很少、token 类型又比较单一，复杂的共享专家或 bias 机制未必划算；若训练规模很大、all-to-all 成本高、热点明显，单靠一个辅助损失通常不够，结构性方案更有价值。

还有一个判断标准是“问题到底出在优化，还是出在结构”：

| 现象 | 更可能的问题 |
| --- | --- |
| 路由早期就锁死，后面一直偏 | 优化问题，先看噪声、温度、辅助约束 |
| 长期热点始终集中在通用 token | 结构问题，共享专家更有效 |
| 主任务一加辅助损失就掉点 | 可考虑 Loss-Free Balancing |

这也是为什么工程里很少存在“一招通吃”的负载均衡方案。不同方法解决的是不同层面的偏斜。
