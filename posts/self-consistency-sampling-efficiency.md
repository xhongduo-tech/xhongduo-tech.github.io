## 核心结论

自洽性采样（Self-Consistency, SC）可以理解为“让模型对同一道题独立思考多次，再用投票决定最终答案”。它通常比单次推理更稳，但代价也直接按采样次数线性上升。标准做法常把每题预算固定成 10 到 40 次，很多实现里要到 40 次以上才进入稳定区，这意味着大量算力被花在本来就很容易的题上。

更高效的做法不是“把所有题都少采几次”，而是“按题目难度动态分配预算”。核心策略包括三类：

| 方法 | 直观做法 | 平均采样数 | 准确率目标 |
|---|---|---:|---:|
| 固定 SC | 每题都采满 40 次再投票 | 40 | 作为基线 |
| Early-Stopping / ESC | 出现连续共识或高置信度就提前停 | 8-18 常见 | 保持接近基线 |
| Importance-Weighted / ReASC | 对高置信样本赋更大权重，难题再继续采样 | 12-16 常见 | 维持 95%+ 基线精度 |

如果目标是“在保持原始 SC 判决精度 95% 以上的前提下尽量省算力”，那么自适应采样通常比固定 40 次更合理。一个新手视角的理解是：过去固定要采 40 次，相当于所有题都要排满 40 个号；Adaptive/ESC 会先观察队伍里是不是已经明显形成共识，如果连续几次都给出同一答案，就提前离开队伍。容易题可能只采 9 次就结束，复杂题再继续追加预算。

这里的收益来自“预算重分配”而不是“算法魔法”。简单题本来就会快速收敛，继续采样只是在重复确认；复杂题才需要更多票数。把省下来的预算集中给难题，整体准确率更容易守住，而平均成本会明显下降，常见可降到原来的 30% 到 40%。

## 问题定义与边界

先定义问题。自洽性采样的目标不是生成一条“最漂亮”的推理链，而是估计“最终答案的多数意见”。形式上，模型每次采样输出一个答案 $a_i$，标准 SC 用多数投票得到最终答案：

$$
\hat{a}=\arg\max_a\sum_{i=1}^{N}\mathbf{1}(a_i=a)
$$

其中 $\mathbf{1}(a_i=a)$ 是指示函数，可以理解为“第 $i$ 次采样是否投给答案 $a$”。如果是，就记 1；不是，就记 0。

问题边界也要讲清楚。这里讨论的是“在不明显牺牲 SC 准确率的情况下减少平均采样数”，不是把 SC 改造成另一种推理框架，也不是追求开放式生成的文采或长文本质量。优化对象主要是有较明确终答案的任务，例如数学题、选择题、短答案推理题。

一个最直观的新手解释是：SC 像“用投票决定答案”，普通做法是“每次都投 40 张票”，优化后是“看到多数人已经稳定投 A，就提前公布结果”。

实际工程里需要先设定边界条件，否则“省采样”会变成“偷准确率”。常见边界如下：

| 边界项 | 典型设定 | 含义 |
|---|---|---|
| 精度阈值 | 不低于固定 SC 的 95% 或 97% | 节省成本不能明显伤害结果 |
| 最大预算 | 20、30 或 40 | 防止难题无限追加采样 |
| 最小采样数 | 3 或 5 | 防止只看一两票就过早停 |
| 提前停止条件 | 连续一致窗口 / 后验置信度超阈值 | 判断是否已形成足够共识 |
| 置信阈值 | 0.90、0.95、0.98 | 控制保守程度 |

这里的“后验置信度”可以理解为“根据目前已经观察到的投票结果，当前主导答案最终仍会赢的把握”。它不是模型原始 logits 的简单拷贝，而是把“出现次数”转成“概率判断”。

一个玩具例子可以说明问题。假设某类简单题上，正确答案 A 的真实采样概率约为 0.8，错误答案 B 的概率约为 0.15，其他答案合计 0.05。固定 SC 会机械地采 40 次；但实际上只要前 9 次里已经出现明显多数，继续采样带来的收益就很小。如果此时已经连续 3 次得到 A，且 A 的累计票数远超其他答案，就可以提前结束。

但边界也很明确：如果任务是开放式写作、复杂规划、代码生成后还需执行验证，答案空间不再是小而离散的集合，单纯的表面投票会失效。这时需要语义聚类、执行验证、模型自评等更强的判决器，不能直接照搬多数票方案。

## 核心机制与推导

先看标准 SC 为什么有效。设模型在独立采样下，对同一道题会产生多个候选答案，真实最优答案的采样概率是 $p_1$，第二常见错误答案的采样概率是 $p_2$，且 $p_1>p_2$。当采样次数 $N$ 增大时，多数投票选错的概率会快速下降。一个常见误差界写成：

$$
\text{err}(N;q)\lesssim \exp(-N\cdot m), \quad m=(\sqrt{p_1}-\sqrt{p_2})^2
$$

这里的 $m$ 可以理解为“主答案和次答案的分离度”。$m$ 越大，说明正确答案比错误答案更占优势，误差随采样数下降得越快；$m$ 越小，说明题更难，更多采样才有意义。

这个式子直接给出一个工程结论：不是所有题都值得采 40 次。因为不同题的 $m$ 不同。简单题的 $m$ 大，少量样本就够；难题的 $m$ 小，必须继续追加预算。自适应采样本质上是在“在线估计这道题到底难不难”。

### 1. Early-Stopping：先看是否已经形成共识

Early-Stopping 的想法最直接。每来一个样本，就更新答案计数器。如果最近 $w$ 次都给出同一答案，或者当前第一名的领先优势已经足够大，就停止采样。所谓“共识窗口”，就是最近连续若干票是否一致。它的白话解释是：不用等全班 40 个人都举手，只要后排几个人也连续给出同一个答案，就说明风向已经很稳定了。

玩具例子如下。默认最大预算 40，窗口宽度设为 3：

- 前 1 到 6 次答案：A, B, A, A, A, A
- 最近 3 次连续为 A
- 此时 A 的累计票数也明显领先

那么第 6 次后就可以停，而不必继续采到 40 次。若另一个题的前 10 次答案是 A, B, C, A, B, A, C, B, A, C，则没有形成稳定主导答案，预算应继续。

### 2. Beta / Dirichlet 后验：把“票数”变成“置信度”

如果只有二分类，可以用 Beta 分布；多分类更常见，用 Dirichlet 分布。它们的作用是：根据目前观察到的计数，估计每个答案成为最终主导答案的概率。术语“后验”就是“看完已有样本后更新过的概率判断”。

假设答案集合有 $K$ 类，每类计数为 $n_1,\dots,n_K$，用对称先验 $\alpha$，则后验为：

$$
(p_1,\dots,p_K)\sim \text{Dirichlet}(n_1+\alpha,\dots,n_K+\alpha)
$$

此时可以计算“当前第一名最终保持领先”的概率。如果这个概率超过阈值，比如 0.95，就提前停止。

新手版理解：Dirichlet 后验不是神秘数学，它只是把“目前 A 出现了 8 次，B 出现了 2 次，C 出现了 1 次”转成“我们对 A 最终胜出的把握已经很高”。

### 3. Importance Weighting：不是每张票都一样重

标准 SC 默认每个样本权重都相同：

$$
\text{score}(a)=\sum_{i=1}^{N}\mathbf{1}(a_i=a)
$$

重要性加权（importance weighting）改成：

$$
\text{score}(a)=\sum_{i=1}^{N} w_i \cdot \mathbf{1}(a_i=a)
$$

其中 $w_i$ 是第 $i$ 个样本的权重，可以来自模型自评置信度、答案一致性、格式完整性，或者一个外部校准器。它的白话解释是：不是每一票都等价，有些票来自“模型自己也很确定”的推理链，就让它更重一些。

这对 ReASC 一类方法特别重要。因为很多题目里，少量高质量样本比大量低质量样本更有信息量。加权后，即使总采样数减少，最终判决也可能仍接近固定 SC。

### 4. 两阶段预算：先快速筛，再把钱花在难题上

真实工程里常见两阶段策略：

1. Stage 1：只采 1 次或很少几次，判断题目是否“高把握可直接输出”。
2. Stage 2：只有剩余不确定样本才进入多轮 SC，并采用加权投票或后验早停。

这可以理解为一个漏斗。前面用便宜规则快速筛掉大部分容易题，后面把完整预算留给难题。对大规模在线服务尤其重要，因为平均延迟和平均 token 成本往往比单题最优更关键。

一个真实工程例子是小学数学推理流水线。假设基于 4B 级别模型部署 GSM8K 风格问答，Stage 1 先用单样本答案加自评置信度判断是否直接输出；若置信度足够高，则不再重复采样。只有低置信样本进入 Stage 2，再做自洽性采样和加权投票。这样大部分“算式结构很标准”的题会在第一阶段结束，真正复杂的多步题才消耗多次推理预算。

## 代码实现

下面先给一个最小可运行实现。它不依赖真实大模型，而是用一个模拟采样器来演示 Early-Stopping、后验近似和加权投票如何组合。重点是控制流，而不是模型接口本身。

```python
from collections import Counter
from math import exp

def calibrated_weight(conf, temperature=1.2, floor=0.2):
    # 简单温度校准：把过高置信度压平，并保留最低 uniform 票
    conf = max(0.0, min(1.0, conf))
    adjusted = conf ** (1.0 / temperature)
    return floor + (1.0 - floor) * adjusted

def posterior_confidence(counts):
    # 一个工程近似：用“第一名占比 + 领先边际”代替严格 Dirichlet 积分
    total = sum(counts.values())
    if total == 0:
        return 0.0
    ranked = counts.most_common(2)
    top_answer, top_count = ranked[0]
    second_count = ranked[1][1] if len(ranked) > 1 else 0
    share = top_count / total
    margin = (top_count - second_count) / max(total, 1)
    return 0.6 * share + 0.4 * max(0.0, margin)

def adaptive_self_consistency(samples, max_budget=40, min_budget=3, window=3, conf_th=0.95):
    counts = Counter()
    weighted = Counter()
    recent = []

    for i, (answer, confidence) in enumerate(samples[:max_budget], start=1):
        counts[answer] += 1
        weighted[answer] += calibrated_weight(confidence)
        recent.append(answer)
        if len(recent) > window:
            recent.pop(0)

        top_answer, _ = counts.most_common(1)[0]
        same_window = len(recent) == window and len(set(recent)) == 1
        post_conf = posterior_confidence(counts)

        if i >= min_budget and (same_window or post_conf >= conf_th):
            return top_answer, i, counts, weighted

    final_answer, _ = weighted.most_common(1)[0]
    return final_answer, min(len(samples), max_budget), counts, weighted

# 玩具例子：容易题，前几次很快收敛到 A
toy_samples = [
    ("A", 0.72),
    ("B", 0.55),
    ("A", 0.81),
    ("A", 0.84),
    ("A", 0.88),
    ("A", 0.90),
]
answer, used, counts, weighted = adaptive_self_consistency(toy_samples, max_budget=40, window=3, conf_th=0.93)

assert answer == "A"
assert used <= 6
assert counts["A"] > counts["B"]

# 难题例子：没有快速共识，应继续更多采样
hard_samples = [
    ("A", 0.55), ("B", 0.58), ("A", 0.57), ("C", 0.51),
    ("B", 0.56), ("A", 0.60), ("A", 0.63), ("B", 0.61),
    ("A", 0.67), ("A", 0.70),
]
answer2, used2, counts2, weighted2 = adaptive_self_consistency(hard_samples, max_budget=10, window=3, conf_th=0.96)

assert answer2 in {"A", "B", "C"}
assert used2 >= 3
assert sum(counts2.values()) == used2
```

上面的实现刻意保留了三个关键分支：

| 分支 | 作用 | 为什么需要 |
|---|---|---|
| `min_budget` | 至少先采几次再判断 | 防止前两票偶然一致就停 |
| `window` | 最近若干次是否连续一致 | 低成本检测“局部共识” |
| `conf_th` | 当前主导答案是否足够稳 | 防止窗口一致但整体样本太少 |

如果写成更接近生产代码的伪代码，流程通常是这样：

```python
def reasc_pipeline(question):
    answer, score = model.sample_once(question)

    if score >= stage1_threshold:
        return answer

    counter = {}
    weighted_counter = {}
    recent = []

    while total_samples < max_budget:
        answer, score = model.sample_once(question)
        update_counter(counter, answer)
        update_weighted_counter(weighted_counter, answer, calibrate(score))
        update_recent(recent, answer)

        if window_consensus(recent, width=3):
            break
        if posterior_confidence(counter) > 0.95:
            break

    return argmax(weighted_counter)
```

这里的 Stage 1 是两段式设计的核心。它的目的不是“直接替代 SC”，而是低成本吃掉一批非常容易的样本。Stage 2 则保证剩余问题仍有稳健判决能力。

真实工程例子可以进一步具体化。假设你在一个批处理推理系统里，每天要跑 10 万道带标准答案的数学题。固定 SC=40 的成本非常高，因为每题都要完整生成 40 条推理链。若改成：

- 单样本高置信直接放行
- 低置信样本进入 SC
- SC 中采用窗口共识 + 后验阈值早停
- 最终使用置信度加权投票

那么容易题可能在 1 到 5 次内结束，中等题在 8 到 12 次结束，只有少量难题才接近 30 到 40 次。平均成本因此会显著下降，而整体准确率仍可贴近原始固定 SC。

## 工程权衡与常见坑

最大风险不是“采样太少”，而是“错误地以为已经够了”。自适应机制一旦阈值设置不当，就会把偶发一致误判成稳定共识。

最常见的坑如下：

| 坑 | 表现 | 缓解 |
|---|---|---|
| 提前停止阈值太低 | 两三个错误样本恰好一致就停 | 设 `min_budget`，同时要求窗口和后验都过线 |
| 窗口太小 | `window=2` 时特别容易误停 | 常从 3 或 4 起调 |
| 置信度未校准 | 错误答案被高分放大 | 温度校准、Platt scaling、保留 uniform 票 |
| 全靠模型自评 | 模型会“自信地错” | 自评只作一部分权重，不作唯一信号 |
| 难题预算封顶过低 | 真正需要 25 次的题被迫在 10 次结束 | 为长尾难题保留足够 `max_budget` |
| 答案字符串匹配过硬 | “12”“12.0”“十二”被当成不同票 | 先做规范化或语义归一 |
| 开放式任务直接多数投票 | 正确答案表达不同导致票被打散 | 用语义聚类或 judge model 合并近义答案 |

一个新手案例很典型。若 ESC 把窗口设为 2，前两轮碰巧都输出同一个错误答案，就会立刻终止。这个错误不是因为 SC 无效，而是因为判决证据不足。解决方式通常有三层：

1. 至少要求最小采样数，例如 3 或 5。
2. 连续一致之外，再叠加“当前票差是否足够大”。
3. 不直接使用原始置信度，而是先校准。

下面给一个“保留 uniform 票”的简单钩子。它的思想是：即使模型很自信，也不要让单票权重无限大，始终保留一部分平均投票成分，避免错得太极端。

```python
def mixed_weight(confidence, alpha=0.7):
    # alpha 越大越依赖校准置信度；剩余部分保留均匀票
    calibrated = calibrated_weight(confidence, temperature=1.3, floor=0.1)
    return alpha * calibrated + (1 - alpha) * 1.0

assert 0.3 <= mixed_weight(0.0) <= 1.0
assert mixed_weight(0.9) > mixed_weight(0.2)
```

还有一个常被忽略的问题是“答案频率低但其实正确”。这在开放式问题、程序合成、复杂规划里很常见。正确答案可能不是出现最多的字符串，因为模型会用不同表述表达同一结论，或者正确推理链更长、更难采到。这种情况下，仅靠硬投票会低估少数正确答案，必须引入语义投票、执行验证或模型自评。

因此，工程上不要把“Adaptive SC”理解成一个固定公式，而应理解成一个判决框架：你需要组合预算控制、答案归一、置信度校准和难题回退机制。

## 替代方案与适用边界

不同方案的核心差异，不在名字，而在两个问题上：

1. 预算是固定还是动态。
2. 是否依赖可用且可校准的置信度信号。

下面给一个新手对照表：

| 方案 | 适用边界 | 典型效果 |
|---|---|---|
| 标准 SC | 预算固定、实现最简单、置信度不可得 | 成本高，但稳定、可作强基线 |
| ESC / Early-Stopping | 有明显共识窗口、答案空间较集中 | 易题大幅降采样，常保留接近 SC 准确率 |
| Blend-ASC / ASC | 允许动态预算，希望按题难度分层 | 比统一预算更省，适合批量推理 |
| ReASC | 有模型自评或外部校准分数可用 | 两阶段最实用，常见 60% 到 70% 成本下降 |
| Soft-SC / CISC 类 | 开放式、答案表达多样、频次未必代表正确 | 更适合语义层面的“共识”而非字面投票 |

可以再用一个角度理解它们：

| 维度 | 标准 SC | ESC | ReASC |
|---|---|---|---|
| 是否需要置信度 | 否 | 可选 | 是，且最好可校准 |
| 是否动态预算 | 否 | 是 | 是 |
| 是否适合低延迟服务 | 一般 | 较适合 | 很适合 |
| 是否怕答案表达分散 | 是 | 是 | 仍怕，需要配归一化 |
| 是否保留强基线价值 | 是 | 是 | 是，但实现更复杂 |

什么时候应该保留标准 SC？至少有三种情况：

1. 你拿不到任何可信的置信度分数，也没有稳定的早停规则。
2. 任务量不大，成本不是瓶颈，但你需要一个简单可复现的研究基线。
3. 答案归一和校准还没做好，贸然上自适应机制容易把系统复杂度拉高，却不一定真正省钱。

什么时候更推荐自适应方案？当你已经确认以下条件成立：

- 大部分样本其实是容易题
- 平均时延或推理成本是核心约束
- 可以拿到一个至少“排序大致可靠”的置信度信号
- 有开发能力做离线校准和阈值调优

对于开放式或合成任务，标准 SC、ESC、ReASC 都可能不足。因为“正确答案频次高”这个假设会弱化。此时更合适的路线是：

- 先语义聚类，再投票
- 引入 judge model 做答案比较
- 对代码/计划执行结果做外部验证
- 把“最终是否正确”从字符串匹配升级为任务完成度判定

所以，“自洽性的采样效率优化”不是一句“少采样也一样准”就能概括。更准确的说法是：在离散答案、可观测共识、可做校准的前提下，自适应采样能把预算集中到真正有争议的样本上，从而以更低成本逼近固定 SC 的稳定性。

## 参考资料

- Adaptive-Consistency / “Let’s Sample Step by Step”
  核心贡献：证明动态停采和分步采样可以显著提升采样效率，在接近原始 SC 准确率的情况下大幅减少样本数。
  适用章节：`核心结论`、`核心机制与推导`、`代码实现`
  备注：适合先读，用来建立“为什么简单题不该固定采满预算”的直觉。

- Self-Consistency Sampling 概览
  核心贡献：系统整理标准 SC、ASC、ESC、Blend-ASC 的误差界、预算分配思路和实验现象。
  适用章节：`问题定义与边界`、`核心机制与推导`、`替代方案与适用边界`
  备注：适合用来统一术语，尤其是理解误差随采样数下降的公式框架。

- ReASC / Reliability-Aware Adaptive Self-Consistency 评述
  核心贡献：给出两阶段决策和置信度加权投票的工程路径，说明如何在实际流水线里把大部分简单题提前结束。
  适用章节：`代码实现`、`工程权衡与常见坑`
  备注：如果你想落地在线推理服务，这篇比纯理论材料更接近工程设计。

- 关于 Soft-SC、CISC 一类扩展方案的资料
  核心贡献：说明当“正确答案未必是最高频字符串”时，应转向语义一致性、自评或外部验证。
  适用章节：`工程权衡与常见坑`、`替代方案与适用边界`
  备注：适合补足开放式任务场景，避免把多数投票误用到不适合的任务上。

新手提示：建议把“Adaptive-Consistency”与“ReASC 工程回顾”放在一起读。前者回答“为什么动态采样有效”，后者回答“怎样把它做成真的系统”。只读前者容易停在公式，只读后者又容易看不到误差界和预算分配的理论基础。
