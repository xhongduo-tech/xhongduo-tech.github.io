## 核心结论

CMRC 2018 是一个中文机器阅读理解数据集，任务形式是 **span extraction**，也就是“答案必须是原文里一段连续文本，模型不能自由生成”。它的重要性不在于规模极大，而在于定义清楚、格式稳定、评测标准统一，适合用来验证中文问答模型是否真的学会了“读段落并定位答案”。

它的输入由三部分组成：`context`、`question`、`answers`。`context` 是上下文段落，白话解释就是“拿来查答案的原文”；`question` 是问题；`answers` 里记录标准答案文本和起始偏移。模型的输出不是一句新话，而是原文中的一个位置对 $(s,e)$，再映射为 `context[s:e]` 对应的连续片段。

CMRC 2018 对新手特别友好，因为它把“阅读理解”拆成了一个可计算的问题：先判断答案是否在文中，再判断从哪里开始、到哪里结束。对工程实践也有价值，因为 BERT、RoBERTa 这类预训练模型可以直接套在这个任务上微调，训练和评估流程都比较标准。

数据集还有一个容易被忽略但很关键的点：除了常规训练集、开发集、测试集，还单独设计了 **challenge set**，也就是“专门更难的一组题”，用于考察跨句推理而不是简单字符串匹配。这意味着 CMRC 2018 既能做快速 baseline，也能做一定程度的深度理解检验。

---

## 问题定义与边界

CMRC 2018 的任务边界很明确：它不是开放式问答，不是聊天生成，也不是检索系统。它只回答一个问题：**给定一段中文上下文，答案是否能作为连续片段从中抽取出来**。

如果把段落长度记为 $L$，答案可以表示为：

$$
\text{answer} = \text{context}[s:e], \quad 0 \le s \le e < L
$$

这里的 $s$ 是起始位置，$e$ 是结束位置。白话解释就是“从原文第几个字符开始截，到第几个字符结束”。

这一点决定了它的适用边界：

| 维度 | CMRC 2018 的设定 | 直接含义 |
| ---- | ---- | ---- |
| 输入范围 | 单篇段落级上下文 | 不负责跨文档检索 |
| 输出形式 | 原文连续片段 | 不适合自由生成答案 |
| 标注方式 | `text` + `answer_start` | 必须保证字符偏移准确 |
| 目标能力 | 篇章理解 + 精确定位 | 同时考查理解和对齐 |

一个玩具例子最容易理解这个任务。

上下文里有一句：“12月的消费者信心指数为78.1，较上月下降。”  
问题是：“12月的消费者信心指数是多少？”  
标准答案是：“78.1”。

这里模型不需要总结，也不需要改写，更不需要输出“指数为78.1”。它只需要在原文中找到 `78.1` 这个 span。对初学者来说，这个例子足够说明：所谓阅读理解，在这个数据集里本质上是“读懂问题后做精确定位”。

再看一个官方风格的真实样本类型。比如上下文介绍《战国无双3》的开发背景，问题是“《战国无双3》是由哪两个公司合作开发的？”。答案不是一句解释，而是原文中的公司名片段。即使标注里出现多个相同答案，本质也还是一个连续 span，而不是多步生成。

所以，CMRC 2018 的核心边界可以概括为一句话：**它测的是“定位式理解”，不是“生成式作答”**。

## 核心机制与推导

大多数 span-extraction 模型都会输出两个分布：一个预测起点，一个预测终点。白话解释就是“先猜答案从哪里开始，再猜到哪里结束”。

设上下文经过分词或子词切分后有 $n$ 个位置，模型会输出：

- 起点概率分布 $P_{start}(i)$
- 终点概率分布 $P_{end}(j)$

然后在满足 $i \le j$ 的前提下，选择分数最高的 span：

$$
(s,e) = \arg\max_{0 \le i \le j < n} \left( P_{start}(i) \times P_{end}(j) \right)
$$

有些实现直接用对数概率求和，本质一样，都是在找“最像答案边界的一段区间”。

为什么要拆成起点和终点两个预测，而不是直接预测整段文本？因为候选答案空间太大。若直接枚举所有字符串，组合数量会爆炸；拆成两个边界后，问题变成在序列上做位置分类，训练更稳定，也更容易和 Transformer 编码器结合。

评估时常用两个指标：

$$
EM =
\begin{cases}
1, & \text{预测答案与任一标准答案完全一致} \\
0, & \text{否则}
\end{cases}
$$

$$
F1 = \frac{2PR}{P+R}
$$

其中 $P$ 是 precision，白话解释就是“你预测出来的内容里有多少是对的”；$R$ 是 recall，白话解释就是“标准答案里有多少被你覆盖到了”。

这两个指标一起用，是因为它们衡量的东西不同：

- EM 看“是否完全命中”
- F1 看“是否部分重叠且大体正确”

举一个最小推导例子。标准答案是“工商协进会”，模型预测成“协进会”。

- EM = 0，因为不完全一致
- 若按字符看，重叠部分是“协进会”，长度为 3
- 预测长度为 3，标准长度为 5
- 所以 $P=3/3=1$，$R=3/5$
- 得到 $F1=2 \times 1 \times 3/5 \div (1+3/5)=0.75$

这个例子说明，F1 比 EM 更宽容，能反映“错了一部分但不是全错”。在工程里，这很重要，因为很多错误不是完全没找到答案，而是边界偏了一个词或两个字。

CMRC 2018 官方排名通常按 EM 和 F1 的均值比较，平局再看 F1、EM。这种设计的含义是：系统既要追求完全正确，也不能只靠保守输出短片段来刷某一个指标。

## 代码实现

工程上最常见的做法，是直接用 Hugging Face 的 `datasets` 读取 `hfl/cmrc2018`，再把 `context`、`question`、`answers` 转成模型训练所需的起止标签。

先看一个最小可运行例子。下面代码不依赖外网数据，也能说明 span 的基本逻辑和 EM/F1 的计算方式。

```python
from collections import Counter

def extract_span(context: str, start: int, end: int) -> str:
    assert 0 <= start <= end < len(context)
    return context[start:end + 1]

def char_f1(pred: str, gold: str) -> float:
    pred_counter = Counter(pred)
    gold_counter = Counter(gold)
    overlap = sum((pred_counter & gold_counter).values())
    if overlap == 0:
        return 0.0
    precision = overlap / len(pred)
    recall = overlap / len(gold)
    return 2 * precision * recall / (precision + recall)

def exact_match(pred: str, golds: list[str]) -> int:
    return int(pred in golds)

context = "12月的消费者信心指数为78.1，较上月下降。"
gold = "78.1"
start = context.index(gold)
end = start + len(gold) - 1

pred = extract_span(context, start, end)

assert pred == "78.1"
assert exact_match(pred, [gold]) == 1
assert round(char_f1("协进会", "工商协进会"), 2) == 0.75

print("pred =", pred)
print("start =", start, "end =", end)
```

这个例子说明三件事：

1. span 任务最后一定要能回到原文切片。
2. 偏移必须精确到字符级。
3. EM 和 F1 可以在很小的代码里验证。

如果进入真实训练流程，通常会做以下几步：

1. 读取样本。
2. 用 tokenizer 把 `question` 和 `context` 编码成 token 序列。
3. 用 offset mapping 把字符级 `answer_start` 映射到 token 级起止位置。
4. 训练模型预测 `start_logits` 和 `end_logits`。
5. 后处理时把 token 位置再映射回原文片段。

真实工程例子可以这样理解。假设你在做一个中文搜索助手，前一步检索器已经从知识库里召回一段最相关的说明文档，例如支付接口文档的一段文本。现在用户问：“退款结果通知的重试周期是多少？” 这个问题如果答案就在当前段落里，CMRC 2018 式模型就适合接在检索后面做“精确抽取”。它的价值不在于会聊天，而在于能稳定地把“15分钟”或“24小时内”这种关键字段准确截出来。

数据规模上，官方公开说明的切分大致如下：

| 集合 | 段落数 | 问题数 | 每题答案数 |
| ---- | ------ | ------ | ---------- |
| 训练集 | 2,403 | 10,142 | 1 |
| 开发集 | 848 | 3,219 | 3 |
| 预测试/测试/挑战 | 470 / 1,248 | 2,000 / 4,895 | 3 |

这个表有一个工程含义：训练时只有单答案，评估时往往是多答案匹配，所以你不能把“训练集每题一个答案”误解为“任务天然只有唯一表达方式”。

## 工程权衡与常见坑

第一个常见坑是 **字符偏移错位**。`answer_start` 记录的是原始上下文里的字符位置。只要你对 `context` 做了任何会改长度的处理，比如删空格、统一全角半角、简繁转换、正则清洗，偏移就可能失效。模型看起来还能训练，但评估时 EM/F1 会明显下降，因为你切出来的文本不再是标准答案对应的位置。

第二个坑是 **字符级标注和 token 级训练之间的对齐**。中文里一个汉字通常对应一个字符，但 tokenizer 不一定一字一 token，尤其遇到英文、数字、标点混排时，经常会被切成多个子词。工程上必须依赖 offset mapping，而不是自己拍脑袋按字符数去换算 token 位置。

第三个坑是 **训练目标和评估目标不完全对称**。训练集每题通常只有 1 个答案，但开发和测试可能有 3 个参考答案。白话解释就是“训练时老师只给一个标准说法，考试时允许多种等价说法”。如果你在后处理阶段只盯着单一形式，可能会低估模型表现；如果你在训练前人为合并多种改写，又可能引入噪声。

第四个坑是 **challenge set 不能靠表面匹配蒙混过关**。有些样本需要先理解上一句的主体，再到下一句找属性值；有些需要把时间、事件和数值拼起来。只做局部关键词匹配的系统，在普通开发集上可能还行，在 challenge set 上往往掉得很明显。

第五个坑是 **答案长度偏置**。如果模型过度偏向短答案，常会把“中华人民共和国国务院”截成“国务院”；如果偏向长答案，又会把答案前后的修饰语一起吞进去。EM 对这种边界偏差很敏感，所以后处理时常需要限制最大答案长度，并结合无答案阈值、标点截断等启发式规则。

可以把这些风险总结成一张表：

| 风险点 | 典型表现 | 后果 |
| ---- | ---- | ---- |
| 文本清洗改长度 | `answer_start` 不再对齐 | 标签错、训练无效 |
| token 对齐错误 | 起止位置落在错误 token | EM/F1 同时下降 |
| 只做表面匹配 | challenge 表现差 | 泛化不足 |
| 后处理过粗 | 截短或截长 | F1 尚可，EM 偏低 |
| 忽略多参考答案 | 评估实现不一致 | 离线指标失真 |

## 替代方案与适用边界

如果你的任务就是“给定一段文本，从中抽答案”，CMRC 2018 非常合适。它定义干净、成本低、适合快速试验中文预训练模型。

但如果场景变了，就要换数据或换任务形式。

| 数据集 | 核心场景 | 规模/特点 |
| ------ | -------- | -------- |
| CMRC 2018 | 中文 span-extraction MRC | 约 2 万问题，段落级抽取，有 challenge set |
| DRCD | 类似 SQuAD 的中文阅读理解 | 30k+ 问题，维基百科段落覆盖更密 |
| DuReader | 搜索与多文档问答 | 200k 问题、420k 答案、1M 文档，更贴近搜索日志 |

它们的适用边界可以这样判断：

- 如果你在做教学、基线复现、模型对比实验，优先用 CMRC 2018。
- 如果你仍然做中文抽取式问答，但想补更多样本，可以把 DRCD 当作补充训练源。
- 如果你的真实输入不是单段百科，而是搜索结果页、论坛帖子、商品说明、帮助中心文档，DuReader 这类更贴近检索场景的数据往往更合适。

举一个真实工程判断。假设你要做“企业内部知识库问答”：

- 如果前面已经有检索器把段落召回得很准，后面只需要抽取字段式答案，CMRC 2018 风格最合适。
- 如果用户问题经常要综合多篇文档，例如“退款政策和发票政策一起看，最终限制是什么”，那就超出了单段 span extraction 的边界，应该考虑检索增强生成或多跳推理方案。
- 如果答案经常不在原文里，而需要归纳总结，比如“这份制度的核心要求是什么”，那也不该强行套 CMRC 2018，因为它测的不是摘要生成。

所以，CMRC 2018 最准确的定位不是“通用中文问答圣杯”，而是“**中文抽取式阅读理解的标准练习场**”。
