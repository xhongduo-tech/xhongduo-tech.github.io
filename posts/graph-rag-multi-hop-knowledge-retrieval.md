## 核心结论

Graph RAG 是在传统 RAG 之上增加“图结构检索”的方法。这里的“图结构”可以先理解为：把文档里的实体当作点，把实体之间的关系当作边，形成一个可遍历的知识网络。它解决的不是“找不到一段相关文本”，而是“答案分散在多段、多个文档、多个逻辑环节里，单次向量召回拼不完整”。

传统 RAG 更像“找最像的问题片段”，擅长单跳检索。单跳的意思是：模型通常只从一次召回结果里取证据，很难稳定地把“法规 A 约束案例 B，案例 B 又影响结论 C”这种链路拼出来。Graph RAG 先把知识组织成实体关系图，再按社区做摘要，查询时先选社区，再在社区内或邻域内多跳展开，因此更适合跨文档推理、全局梳理、因果链分析。

微软公开案例里，Graph RAG 在全局问题上的全面性相对朴素 RAG 提升大约 20 到 30 个百分点。这个“全面性”可以直白理解为：回答有没有把关键相关信息尽量覆盖到。代价也很明确，索引阶段要做实体抽取、关系抽取、社区发现、社区摘要，构建成本通常比普通向量索引高一个数量级，实际工程里常见到 5 到 10 倍，重度实现里甚至更高。

先看一个玩具例子。假设有三段文档：

| 文档 | 内容片段 | 抽取出的事实 |
|---|---|---|
| D1 | 法条第 12 条要求平台保存日志 | 平台 -> 需要遵守 -> 法条12 |
| D2 | 案例甲因日志缺失被处罚 | 案例甲 -> 违反 -> 法条12 |
| D3 | 处罚理由强调“无法追溯交易链” | 无法追溯交易链 -> 支撑 -> 处罚决定 |

问题是：“为什么法条 12 会在案例甲中导致处罚？”  
普通 RAG 可能召回 D2，因为“案例甲”“处罚”最像；Graph RAG 会把“法条 12”“案例甲”“处罚理由”连成链，先找到相关社区，再把链路一起交给模型，所以更容易答出完整推理：法条要求保留日志，日志缺失导致无法追溯交易链，因此构成处罚依据。

真实工程例子更典型。法律研究、科研综述、企业合规、内部知识库问答，都会遇到“答案不在一段里，而在多个实体关系链里”的问题。此时，先按社区定位，再拼接社区摘要，通常比“召回 top-k 文本片段直接问模型”更稳。

## 问题定义与边界

Graph RAG 的目标很具体：让检索系统支持多跳、多层级的知识探索，而不是只做单次相似度匹配。多跳可以直白理解为：先找到一个相关实体，再沿着它的关系继续找下一个实体，直到拼出完整证据链。

其核心数据结构可以写成：

$$
G=(V,E)
$$

其中，$V$ 是实体集合，比如“法院”“法条 12”“案例甲”“交易日志”；$E$ 是关系集合，比如“适用”“违反”“支持”“引用”。如果进一步做社区划分，还会得到层级社区：

$$
C_0 \supset C_1 \supset \cdots \supset C_L
$$

这表示图可以被分成不同粒度的社区。这里的“社区”可以先理解为：图里联系更紧密的一簇实体和关系。高层社区更粗，适合做全局摘要；低层社区更细，适合做局部追踪。

边界也必须说清楚。Graph RAG 不是所有 RAG 系统的默认升级版，它适合的是“高价值、知识结构相对稳定、问题常常跨文档”的场景。如果你的语料是高频更新的 FAQ、客服对话、商品详情页，向量 RAG 往往更便宜也更直接。因为 Graph RAG 的预处理成本高，而且知识图一旦老化，错误会被放大到社区摘要层。

再看一个新手可理解的例子。把一段判决文字拆成：

- 法院
- 判决结果
- 依据条款
- 争议事实

这些实体在图里形成链路：`法院 -> 作出 -> 判决结果 -> 依据 -> 条款`。当你问“该裁决和条款之间的推理是什么”时，系统不是只匹配“条款”两个字，而是从包含这些节点的社区中读取整体逻辑，再组织成回答。也就是说，它处理的是“关系解释”问题，不只是“片段定位”问题。

因此，Graph RAG 的问题定义可以总结成一句话：它面向的是“证据分散、关系重要、需要跨节点推理”的检索增强生成任务，而不是一切问答任务。

## 核心机制与推导

Graph RAG 通常分成两个阶段：索引阶段和查询阶段。

索引阶段先从原始文本里抽取实体和关系，构建知识图 $G=(V,E)$。然后对图做社区发现，常见方法是 Leiden。这里的“Leiden”可以先理解为：一种把图自动拆成若干高内聚群组的聚类算法。得到社区后，再为每个社区生成摘要 $s(c)$，形成“社区报告”。

这样做的关键原因是：直接把整张图喂给大模型不可行，太大、太乱、太贵；而社区摘要相当于把局部图结构压缩成可读文本，供后续快速筛选。

查询阶段有两种常见路径。

第一种是全局搜索。先让系统判断哪些社区最可能回答当前问题。常用记号可以写成：

$$
r(q,c)=\text{LLM\_rate}(q,s(c))
$$

其中，$q$ 是用户问题，$c$ 是某个社区，$s(c)$ 是该社区摘要，$r(q,c)$ 表示这个社区对问题的相关分数。然后选出分数最高的一组社区：

$$
S(q)\approx \arg\max_c r(q,c)
$$

再把这些社区摘要拼接起来，交给大模型生成答案。这个过程本质上是“先看地图，再查细节”。

第二种是局部搜索。若问题已经指向明确实体，比如“某案例为何引用某法规”，系统先定位实体节点 $v$，再沿邻域展开：

$$
N(v)=\{u \mid (v,u)\in E\}
$$

这里的 $N(v)$ 表示实体 $v$ 的邻居集合，也就是和它直接有关系的节点。继续按跳数扩展，就能得到多跳路径。路径不一定显式输出给用户，但会作为证据链参与答案生成。

可以把整个过程理解成一条流水线：

| 阶段 | 输入 | 处理 | 输出 |
|---|---|---|---|
| 构图 | 原始文档 | 实体/关系抽取 | 知识图 $G$ |
| 聚类 | 知识图 | Leiden 社区发现 | 层级社区 $C_l$ |
| 摘要 | 社区节点与边 | LLM 生成社区报告 | $s(c)$ |
| 查询打分 | 问题 + 社区摘要 | 相关性评分 | $r(q,c)$ |
| 生成 | Top 社区摘要 + 问题 | LLM 推理与回答 | 最终答案 |

玩具例子可以更具体一点。假设问题是：“技术 A 为什么在成本上优于技术 B，但在延迟上劣势更明显？”  
如果资料分散在三篇文档里：

- 文档 1 讲 A 的部署方式
- 文档 2 讲 B 的推理链路
- 文档 3 讲两者在不同负载下的延迟和成本

普通 RAG 可能先召回“成本比较”或“延迟比较”其中一部分。Graph RAG 则可能先找到包含“A 部署结构”“B 推理路径”“性能评测”的两个社区，评分后把两个社区摘要一起送入 prompt，模型再据此拼接出跨文档对比结论。

真实工程例子常见于法规研究。研究团队会对数千份判例做实体抽取，例如案件主体、法条、裁判理由、先例引用。之后用社区把“相似法规链”“相似处罚逻辑”“相似事实模式”组织起来。用户问“某条法律如何在两个案例之间形成裁判逻辑延续”时，系统不需要逐页搜索，而是先选出含法规与判例关系的社区，再结合局部邻域展开，答案连贯性会明显高于普通 top-k 文本拼接。

## 代码实现

下面给出一个最小可运行的 Python 玩具实现。它不依赖真实 LLM，只模拟 Graph RAG 的四步：构图、社区划分、社区摘要、社区排序。目的是把流程讲清楚。

```python
from collections import defaultdict

triples = [
    ("法条12", "约束", "日志保存"),
    ("案例甲", "违反", "法条12"),
    ("处罚决定", "依据", "案例甲"),
    ("处罚决定", "理由", "无法追溯交易链"),
    ("日志保存", "缺失导致", "无法追溯交易链"),
    ("技术A", "成本低于", "技术B"),
    ("技术A", "延迟高于", "技术B"),
]

def build_graph(triples):
    graph = defaultdict(list)
    for s, r, o in triples:
        graph[s].append((r, o))
        graph[o].append((f"被{r}", s))
    return dict(graph)

def simple_communities(graph):
    legal_nodes = {"法条12", "日志保存", "案例甲", "处罚决定", "无法追溯交易链"}
    tech_nodes = {"技术A", "技术B"}
    communities = {
        "legal": [n for n in graph if n in legal_nodes],
        "tech": [n for n in graph if n in tech_nodes],
    }
    return communities

def summarize_community(name, nodes, graph):
    facts = []
    for node in nodes:
        for rel, other in graph[node]:
            if other in nodes:
                facts.append(f"{node}-{rel}-{other}")
    facts = sorted(set(facts))
    return f"{name} 社区包含节点: {', '.join(nodes)}; 关键关系: {'; '.join(facts)}"

def score_query(query, summary):
    query_terms = set(query)
    return sum(ch in summary for ch in query_terms)

def rank_communities(query, summaries):
    ranked = sorted(
        summaries.items(),
        key=lambda item: score_query(query, item[1]),
        reverse=True
    )
    return ranked

graph = build_graph(triples)
communities = simple_communities(graph)
summaries = {
    name: summarize_community(name, nodes, graph)
    for name, nodes in communities.items()
}
ranked = rank_communities("案例甲为什么被处罚", summaries)

assert "legal" in communities
assert ranked[0][0] == "legal"
assert "案例甲" in summaries["legal"]

print(ranked[0][1])
```

这段代码对应真实系统中的四个环节：

| 步骤 | 玩具实现 | 真实工程实现 |
|---|---|---|
| 构图 | `build_graph` | LLM / OpenIE / NER + RE 抽取实体关系 |
| 社区划分 | `simple_communities` | Leiden 等图聚类算法 |
| 社区摘要 | `summarize_community` | 调用 LLM 生成层级社区报告 |
| 查询排序 | `rank_communities` | LLM 打分器、向量召回器或混合排序器 |

如果写成更接近生产的伪代码，大致是下面这样：

```python
def index_pipeline(docs):
    graph = build_graph_from_docs(docs)          # 实体+关系抽取
    communities = leiden_cluster(graph)          # 层级社区发现
    summaries = {}
    for c in communities:
        summaries[c.id] = llm_summarize(c)       # 生成社区报告
    persist(graph, communities, summaries)
    return graph, communities, summaries

def query_pipeline(query, graph, communities, summaries, mode="global"):
    if mode == "global":
        ranked = rank_communities(query, summaries)
        selected = [cid for cid, _ in ranked[:3]]
        context = "\n".join(summaries[cid] for cid in selected)
    else:
        seed_entities = extract_entities_from_query(query)
        subgraph = expand_neighbors(graph, seed_entities, hops=2)
        context = summarize_subgraph(subgraph)
    return llm_answer(context, query)
```

真实工程里要再加三层能力。

第一层是异步批处理。因为索引最贵，文档应分批构图、分批摘要、失败可重试。  
第二层是缓存。社区摘要、查询评分、中间子图都应缓存，否则成本会失控。  
第三层是双模式检索。全局模式回答“总体趋势、整体差异、全域总结”；局部模式回答“某实体与某法规、某论文与某结论之间的具体关系”。

一个真实工程例子是企业合规系统。原始材料包括制度文件、历史处罚案例、法规原文、会议纪要。系统先抽取“部门”“流程”“控制要求”“处罚条款”“责任人”等实体，再构图和聚类。问“为什么某流程在 2024 年整改后仍然存在合规风险”时，Graph RAG 可以把“整改动作”“对应法规”“历史处罚原因”“未闭环节点”连起来，而不是只给出某一条制度摘录。

## 工程权衡与常见坑

Graph RAG 的优势建立在昂贵的前处理之上，所以工程里最重要的不是“能不能做出来”，而是“值不值得做”。

先看常见问题与缓解方式：

| 问题 | 表现 | 常见原因 | 缓解方案 |
|---|---|---|---|
| 索引成本高 | 建库慢、花费高 | 每篇文档都要抽取与摘要 | 先做核心语料，分批异步更新，必要时用 LazyGraph |
| 实体抽取误差 | 图谱噪声大，社区失真 | 命名不统一、关系抽取漂移 | 建实体规范表，抽样人工复核，加入规则校验 |
| 社区过大 | 摘要空泛，回答泛化 | 聚类层级过粗 | 限制社区大小，分层摘要，高层总览低层细化 |
| 摘要不聚焦 | 回答啰嗦但不准 | 摘要模板不稳定 | 固定模板，要求“实体-关系-结论-证据”结构 |
| 查询偏移 | 选错社区 | 社区排序器不稳 | 混合排序：LLM 打分 + 向量召回 + 图邻域约束 |

最常见的误判是：把 Graph RAG 当作“向量 RAG 的更强版本”，直接对全量语料一把梭。这样通常会在两个地方出问题。

第一，索引账单失控。普通 RAG 可能只是切块、嵌入、入库；Graph RAG 还要额外做结构化抽取和多层摘要。在一些重处理流程里，每篇文档成本可能达到传统 RAG 的 100 到 1000 倍。即便保守实现，5 到 10 倍也并不罕见。

第二，图谱质量低于预期。知识图不是天然可靠的，抽取质量差时，“关系”会变成噪声源。噪声一旦进入社区，后续摘要就会把错误结构正规化，最后回答反而比纯向量检索更像“逻辑完整但事实不准”。

一个适合新手理解的工程策略是“小图先行”。比如你有上千篇判例，不要第一天就全量构图。先挑最核心的 50 篇，验证三件事：

- 实体抽取得是否稳定
- 社区摘要是否真的提升回答质量
- 单位成本能否接受

如果这三件事都成立，再扩到 500 篇、2000 篇。否则，多半应该退回混合方案，而不是强推全量 Graph RAG。

另一个常见坑是忽略层级。Graph RAG 不只是“有图”，更重要的是“有层级社区”。如果只构图不做层级摘要，查询时仍然会陷入上下文过载。高层社区负责全局概览，低层社区负责局部证据，两者缺一不可。

## 替代方案与适用边界

不是所有多跳问题都必须上 Graph RAG。工程上至少有三类可选方案：

| 方案 | 成本 | 多跳能力 | 适合场景 |
|---|---|---|---|
| 传统 RAG | 低 | 弱，通常依赖单次召回 | FAQ、产品文档、更新频繁的知识库 |
| Graph RAG | 高 | 强，支持社区级与邻域级多跳 | 法规、科研、复杂合规、跨文档综述 |
| LazyGraph | 中 | 中到强，按需生成部分图与摘要 | 价值高但预算有限、问题分布不稳定的系统 |

传统 RAG 的优势是便宜、简单、上线快。只要问题大多能在单段或相邻片段中回答，它仍然是默认选择。  
Graph RAG 的优势是“关系显式化”。当问题依赖法规链、因果链、引用链、先后演化链时，它明显更稳。  
LazyGraph 是折中方案。这里的“Lazy”可以理解为：不是提前把所有社区摘要都建好，而是在查询触发时只对相关局部生成图和摘要。这样能压低冷启动成本，但首问延迟会更高。

再给一个场景对比。

产品 FAQ 系统通常每天更新，文档量大、结构松散，问题大多是“这个功能在哪里”“退款规则是什么”。这类问题用向量 RAG 就够了，Graph RAG 的建图成本通常不划算。

法规白皮书或科研综述则不同。文档相对稳定，但问题常常是“某条法规如何影响另一个条款的解释”“某篇论文的方法为什么在后续工作中被替代”。这类问题天然依赖跨章节、跨文档逻辑链，更适合 Graph RAG。

因此，适用边界可以压缩成一句话：如果问题价值高、知识结构稳定、推理链长，Graph RAG 值得；如果文档更新快、问题短平快、成本敏感，优先传统 RAG 或混合检索。
