## 核心结论

MATH 是一个专门测“竞赛级数学推理”的评测集。这里的“竞赛级”可以先理解为：题目不再是日常算术或模板套公式，而是接近 AMC、AIME 这类数学竞赛中需要多步推导、分类讨论、构造与证明意识的问题。它覆盖代数、数论、几何、组合等高难领域，难度明显高于 GSM8K 这类以基础算术推理为主的数据集。

它的评测口径看起来简单，实际很严格：模型需要在解题过程中给出推导，并在结尾用 LaTeX 的 `\boxed{...}` 写出最终答案；评测端再抽取这个盒子里的内容，做格式压缩后执行 exact match，也就是“字符串完全一致才算对”。这意味着 MATH 不只测“结论对不对”，还测“答案写法是否符合规范”。

最典型的玩具例子是：

- 标准答案：`\boxed{\frac{1}{2}}`
- 模型输出 A：`\boxed{\frac{1}{2}}`，通过
- 模型输出 B：`\boxed{0.5}`，失败

这不是因为 $0.5 \neq \frac{1}{2}$，而是因为正式评分通常看规范化后的字符串是否相同，而不是无限放宽到任意语义等价。对白话解释一句：评测系统像一个很严格的阅卷程序，它先按规定格式抓答案，再按规定规则比较，超出规则的“合理写法”也可能被判错。

一个简化流程可以写成：

`数据集题目 -> 模型生成完整解答 -> 提取 \boxed{...} -> 去格式噪声并压缩 -> exact match -> PASS/FAIL`

真实工程里，MATH 常被放进模型发布流水线。每次新 checkpoint 训练完成后，都会重新跑一遍，确认“竞赛级推理能力”没有退步。因为它对格式极其敏感，所以一个成熟的评测实现不会只做原始字符串比对，而会加入基础规范化、失败样本抽检和回归分析，避免把格式噪声误当成能力退化。

## 问题定义与边界

MATH 的核心任务不是“给一个数字”，而是“解一道高难数学题，并按竞赛答案规范输出最终答案”。这里的“边界”很重要，因为很多初学者会把它误解成普通问答评测。

先看它覆盖什么。MATH 由约 12,500 道竞赛数学题组成，通常按学科和难度组织，典型学科包括代数、数论、几何、组合等。这里的“难度”可以理解为题目所需推理深度，而不是字面长度。很多题目即使答案只是一个分数，过程也可能涉及多步恒等变形、构造辅助量或分类讨论。

下面这张表可以帮助快速建立边界感：

| 维度 | MATH | GSM8K |
|---|---|---|
| 目标能力 | 竞赛级数学推理 | 基础算术文字题推理 |
| 常见题型 | 代数、数论、几何、组合 | 四则运算、比例、基础逻辑 |
| 解题过程 | 多步推导，经常需要技巧 | 通常是稳定模板展开 |
| 答案形式 | 强调 LaTeX 与 `\boxed{}` | 多为自然语言或简单数值 |
| 评分敏感点 | 数学正确性 + 格式规范 | 结论正确性更重要 |

如果题目是“求方程解”，一个合格回答通常应是：

1. 写出推导步骤；
2. 说明如何化简、代入或排除无效解；
3. 最后输出 `\boxed{\frac{3}{4}}`。

如果只写 `0.75`，即便数学上等价，也可能不合格。如果写成中文“答案是四分之三”，同样通常不合格。原因不是评测“故意刁难”，而是 benchmark 需要稳定、自动化、可复现的评分规则。

从工程角度，MATH 的问题定义可以压缩成一句话：给定一组高难数学题，要求模型生成可读推导，并把最终答案放入规范的 LaTeX `\boxed{}` 中，评测端对该最终答案做标准化后精确匹配。

它的边界也要说清楚：

| 维度 | 属于 MATH 评测关注点 | 不属于核心关注点 |
|---|---|---|
| 推理深度 | 是 | 否 |
| 最终答案规范 | 是 | 否 |
| 自然语言文采 | 否 | 是 |
| 长篇解释是否优美 | 否 | 是 |
| 与人类阅卷一致到完全语义层面 | 不一定 | 是 |

也就是说，MATH 更像一个“高压缩、高约束”的数学能力测试，而不是一个“宽松理解语义”的开放问答系统。

## 核心机制与推导

MATH 的评分机制可以形式化写成：

$$
\text{answer\_canonical} = \mathrm{compress}(\mathrm{strip\_format}(\mathrm{extract\_boxed}(y)))
$$

其中：

- `extract_boxed(y)`：从输出 $y$ 中提取 `\boxed{...}` 里的最终答案；
- `strip_format`：去掉不影响主要内容的格式命令，比如 `\left`、`\right`、多余空格、某些间距命令；
- `compress`：把可忽略的表示差异压缩成单行、稳定字符串；
- 最终再和标准答案的规范化结果做 exact match。

更直接地写：

$$
\text{score} =
\begin{cases}
1, & \text{if } \mathrm{canonicalize}(y_{\text{model}})=\mathrm{canonicalize}(y_{\text{gold}}) \\
0, & \text{otherwise}
\end{cases}
$$

这套机制为什么对格式敏感？因为它默认“规范答案空间”是有限且可枚举的，而不是开放语义空间。对白话解释一句：机器阅卷最怕“看起来差不多，实际上程序难以稳定判断”的答案，所以它宁可把规则定死。

看一个玩具例子：

- 标准：`\boxed{\frac{1}{2}}`
- 经过抽取：`\frac{1}{2}`
- 去格式：保留为 `\frac{1}{2}`
- 压缩后：`\frac12`

如果模型写成 `\boxed{\left(\frac{1}{2}\right)}`，未经额外处理时：

- 抽取：`\left(\frac{1}{2}\right)`
- 去掉 `\left/\right` 后：`(\frac{1}{2})`
- 压缩后：`(\frac12)`

这与标准答案 `\frac12` 仍然不同，所以会失败。也就是说，评测不是“看你是不是差不多”，而是“看你有没有落在允许的标准表示集合里”。

一个更贴近工程实现的伪代码是：

```text
function canonicalize(text):
    ans = extract_last_boxed(text)
    ans = remove_latex_spacing(ans)
    ans = remove_left_right(ans)
    ans = collapse_spaces(ans)
    ans = normalize_frac(ans)
    return ans

function score(model_text, gold_text):
    return canonicalize(model_text) == canonicalize(gold_text)
```

这里每一步都对应一个工程判断：

- 为什么提取最后一个 `\boxed{}`：因为模型中间可能举例或草稿，最后一个盒子更可能是最终答案。
- 为什么只做有限规范化：因为规范化做得太强，会把本不该等同的表达硬合并。
- 为什么不默认把 `0.5` 视为 `\frac{1}{2}`：因为一旦打开更强语义归一化，集合、根式、参数解、多项式因式分解形式都会立刻变复杂。

真实工程例子是模型发布流水线。假设一个团队每次训练出新 checkpoint，都在夜间自动跑 MATH。流水线会：

1. 读取题目与标准答案；
2. 调用模型生成解答；
3. 自动抽取 `\boxed{}`；
4. 计算通过率；
5. 把与上一个 checkpoint 的差异做回归分析。

如果这一步只做原始字符串比较，很多本来能力没变的样本会因为 `\left`、空格、`\dfrac`、括号包裹等噪声被误判，导致团队错误地以为模型退步了。

## 代码实现

下面给一个可运行的 Python 玩具实现。它不是完整工业版本，但足以说明提取、规范化和比对的核心思路。

```python
import re

def extract_last_boxed(text: str) -> str | None:
    # 先找简单的 \boxed{...}，适合演示与大多数规则输出
    matches = re.findall(r'\\boxed\{([^{}]*)\}', text, flags=re.UNICODE)
    return matches[-1] if matches else None

def normalize_frac(expr: str) -> str:
    # 把 \frac{1}{2} 压成 \frac12，减少花括号差异
    expr = re.sub(r'\\frac\s*\{\s*([^{}]+)\s*\}\s*\{\s*([^{}]+)\s*\}', r'\\frac\1\2', expr)
    return expr

def strip_format(expr: str) -> str:
    expr = expr.strip()
    expr = expr.replace('\\left', '')
    expr = expr.replace('\\right', '')
    expr = expr.replace('\\,', '')
    expr = expr.replace('\\!', '')
    expr = re.sub(r'\s+', '', expr)
    return expr

def unwrap_outer_parens(expr: str) -> str:
    # 只去掉最外层一对括号，避免过度处理
    if len(expr) >= 2 and expr[0] == '(' and expr[-1] == ')':
        depth = 0
        for i, ch in enumerate(expr):
            if ch == '(':
                depth += 1
            elif ch == ')':
                depth -= 1
            if depth == 0 and i != len(expr) - 1:
                return expr
        return expr[1:-1]
    return expr

def canonicalize(text: str) -> str | None:
    ans = extract_last_boxed(text)
    if ans is None:
        return None
    ans = strip_format(ans)
    ans = normalize_frac(ans)
    ans = unwrap_outer_parens(ans)
    return ans

def score(model_text: str, gold_text: str) -> bool:
    return canonicalize(model_text) == canonicalize(gold_text)

gold = r"最终答案是 \boxed{\frac{1}{2}}"
ok = r"经过推导可得 \boxed{\frac{1}{2}}"
fail = r"经过推导可得 \boxed{0.5}"
recover = r"经过推导可得 \boxed{\left(\frac{1}{2}\right)}"

assert canonicalize(gold) == r"\frac12"
assert canonicalize(ok) == r"\frac12"
assert canonicalize(fail) == "0.5"
assert canonicalize(recover) == r"\frac12"
assert score(ok, gold) is True
assert score(fail, gold) is False
assert score(recover, gold) is True

print("all tests passed")
```

这段代码里最值得初学者注意的是两个点。

第一，正则 `\\boxed\{([^{}]*)\}` 只适合简单情况。它对白话可以理解成：去抓一个最外层结构简单、内部没有继续嵌套花括号的盒子答案。演示足够，但真实工程里经常要处理更复杂的 LaTeX 结构，比如集合、矩阵、分段表达式，此时更稳妥的做法是写一个小型括号扫描器，而不是完全依赖正则。

第二，规范化必须克制。比如这里把外层括号剥掉，是因为 `(\frac12)` 和 `\frac12` 在很多单值答案任务里可以看作等价；但如果题目答案本来就要求区间、集合或有序对，盲目去括号可能把正确结构破坏掉。

如果把这段代码接到真实评测脚本中，最小闭环就是：

1. 从题库读取标准答案；
2. 从模型输出里抽 `\boxed{}`；
3. 调用 `canonicalize`；
4. 比较是否一致；
5. 保存失败样本供人工抽检。

## 工程权衡与常见坑

MATH 采用 exact match 的优点很明显：规则简单、自动化程度高、跨模型对比稳定。只要所有人都按同一协议提交答案，分数就是可复现的。这对 benchmark 很关键。

但它的代价也很直接：格式噪声会污染指标。所谓“格式噪声”，就是结论没错，只是表达方式偏离了评测允许的写法。比如 `\left(\frac{1}{2}\right)`、`\frac 1 2`、`0.5`、`{1 \over 2}`，都可能与标准答案在数学上等价，却在评分程序里得到不同结果。

下面这张表更适合工程决策：

| 坑/格式噪声 | 预处理策略 | 剩余风险 |
|---|---|---|
| `\left` / `\right` | 直接删除 | 某些结构中括号是否保留仍有歧义 |
| 多余空格、`\ , \!` | 压缩空白与间距命令 | 低风险，通常可安全处理 |
| `\frac{1}{2}` vs `\frac 1 2` | 统一成稳定分数表示 | 对复杂嵌套分数要小心 |
| 外层括号包裹 | 只在单值答案场景剥一层 | 集合、区间、有序对可能误伤 |
| `\frac{1}{2}` vs `0.5` | 内部调试可做数值归一化 | 正式榜单常不应默认等同 |
| 集合答案元素顺序不同 | 排序后比较 | 需先确认题目是否允许无序 |
| 标签本身写法不一致 | 人工抽检与修订基准 | 无法靠程序完全消除 |

这里有一个很实际的工程判断：你是在做“正式 benchmark 评分”，还是在做“内部模型调试”。

- 正式 benchmark：规则要保守，少做会改变评测定义的归一化。
- 内部调试：可以更宽松，帮助研发判断真实能力，而不是被格式问题干扰。

真实工程例子是失败样本审计。假设一次新模型上线前跑 MATH，分数比上一版低 1.8 个点。团队如果只看总分，可能会直接判定模型退步；但抽样看 100 个失败样本后发现，其中相当一部分只是：

- 把 `\frac{1}{2}` 写成了 `0.5`
- 多包了一层括号
- 少写了 `\boxed{}`
- 用了 `\dfrac` 而基准答案是 `\frac`

这时问题就不是“数学推理能力下降”，而是“输出协议偏移”。如果不做人工抽检，团队会在错误方向上改模型。

因此，较稳健的实践通常是：

1. 保留正式 exact-match 分数；
2. 在旁路增加符号化规范分数或宽松分数；
3. 对回归最大的样本做人工抽样；
4. 区分“能力退化”和“格式退化”。

## 替代方案与适用边界

如果 exact match 过于严格，有没有替代方案？有，但每一种都会引入新的边界问题。

第一类替代方案是 semantic normalization，也就是“语义归一化”。白话解释就是：不看你写得像不像标准答案字符串，而是先把答案转成更抽象的数学对象再比较。例如把 `\frac{1}{2}` 和 `0.5` 都转成有理数 $\frac{1}{2}$，再比较是否相等。这样可以大幅降低格式噪声。

它的优点是对研发友好，特别适合内部诊断。缺点是实现复杂，而且不同题型差异很大：

- 单个有理数，容易归一化；
- 根式，稍复杂；
- 多项式，要考虑展开式与因式分解式是否等价；
- 集合与区间，要考虑顺序、重数、边界闭开；
- 几何题中的符号答案，可能根本不能简单数值化。

第二类替代方案是“宽松匹配 + 人工 reviewer”。流程可以理解为：

`模型输出 -> 规则预处理 -> 尝试语义比较 -> 不确定样本交给人工复核`

它更贴近真实能力评估，但缺点也明显：成本更高，可重复性变差，榜单可证明性下降。benchmark 之所以偏爱 exact match，就是因为它“硬”，但硬规则更容易达成公开、公平、自动化。

可以用一个对比表总结：

| 方案 | 优点 | 缺点 | 适用边界 |
|---|---|---|---|
| 严格 exact match | 可复现、便于公开对比 | 对格式敏感 | 正式 benchmark、发布对比 |
| 规范化 exact match | 降低常见噪声 | 规则设计需谨慎 | 内部回归分析、自动流水线 |
| 语义归一化 | 更接近数学等价 | 实现复杂，易引入新偏差 | 内部能力诊断 |
| 人工复核 | 最接近人工判断 | 成本高，扩展性差 | 小规模审计、争议样本 |

因此，一个务实结论是：

- 如果你在做正式 MATH 跑分，仍应坚持 `\boxed{}` + exact match，把评测定义保持稳定。
- 如果你在做模型迭代和错误定位，可以加一条更宽松的归一化通道，把 `\frac{1}{2}` 和 `0.5` 视为同值，以便识别“格式问题”和“推理问题”。
- 如果任务开始大量出现集合、参数范围、表达式恒等等复杂答案，单纯字符串匹配的适用边界就快到了，应考虑更强的符号化系统或人工抽检。
