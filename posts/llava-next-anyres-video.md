## 核心结论

LLaVA-NeXT 的核心不是把图像统一缩放到固定尺寸后再送进模型，而是使用 `AnyRes`。`AnyRes` 可以理解为“按输入图像的长宽和细节密度，动态选择切块方式”，把一张高分辨率图拆成多个子图分别编码，再把这些视觉 token 依次送入大语言模型。这样做的直接目标，是在有限上下文长度内，尽量保住小字、表格线、长图局部结构这类容易在缩放中丢失的信息。

视频版本延续的是同一套思想。区别不在“模型突然学会了时间”，而在输入组织方式从“单张图的多个空间块”扩展成“多帧的视觉序列”。视频路径通常会同时做三件事：抽帧、空间压缩、长上下文适配。抽帧是“少看一些帧”，空间压缩是“每帧少放一些 token”，长上下文适配是“让 LLM 能处理更长序列”。三者缺一不可。

一个最直观的玩具例子是长网页截图。若直接把一张超长截图缩到固定尺寸，正文的小字号、侧边栏、代码块行号会一起变糊。`AnyRes` 的做法更像先把网页切成几段，每段各自编码，再拼成序列交给 LLM，因此 OCR 和局部理解通常更稳。视频也是同一个逻辑，只是把“空间切块”再叠加到“时间采样”上。

从工程视角看，LLaVA-NeXT 的本质是 token 预算分配问题。设视觉 token 长度为 $L_{\text{vis}}$，文本 token 长度为 $L_{\text{text}}$，模型最大上下文为 $L_{\max}$，则必须满足：

$$
L_{\text{vis}}=\sum_{k=1}^{mn}T_k,\quad L_{\text{total}}=L_{\text{vis}}+L_{\text{text}}\le L_{\max}
$$

其中 $T_k$ 表示第 $k$ 个图像块对应的 token 数。视频里同样成立，只不过视觉长度会近似随帧数线性增长：

$$
L_{\text{vis}} \approx N\cdot T_{\text{frame}}
$$

这里的 $N$ 是抽样后的帧数，$T_{\text{frame}}$ 是每帧压缩后的 token 数。结论很直接：你不可能同时无限增加帧数、保留每帧高分辨率细节、还不超上下文长度。

下表先给出图像版、视频版和 token 预算约束的最小对比：

| 维度 | 图像版 LLaVA-NeXT | 视频版 LLaVA-NeXT | 共同约束 |
|---|---|---|---|
| 主要矛盾 | 高分辨率细节保留 | 时长与细节同时保留 | 上下文长度有限 |
| 关键手段 | AnyRes 动态切块 | 抽帧 + 空间压缩 | 视觉 token 不能失控 |
| 输入形态 | 子图序列 | 帧序列 + 子图/压缩序列 | 最终都要并入 LLM |
| 常见任务 | OCR、长图、表格截图 | 视频问答、视频质检 | 都受 $L_{\max}$ 限制 |

官方公开结果中，7B、13B、34B 展示的是一条性能与成本之间的折中曲线。7B 更接近效率点，13B 常常是更均衡的拐点，34B 在复杂推理上更强，但边际收益开始变小。这个“效率点/均衡点/性能点”的说法，是基于官方表格做的工程解读，不是论文里的原句。

## 问题定义与边界

LLaVA-NeXT 解决的问题，可以概括为一句话：多模态模型在面对高分辨率图像、长图、OCR、视频问答时，必须在“细节保留”和“上下文长度”之间做取舍。这里的上下文长度，白话说就是“模型一次能读进去的 token 总量上限”。图像太大时，统一缩放会丢细节；视频太长时，逐帧保留全部信息会让 token 爆炸。

对零基础读者来说，可以先记住两个典型场景。

第一个是图像场景。你问模型：“这张海报最后一行小字写了什么？”如果模型把整张海报粗暴缩成固定尺寸，小字可能已经不可读。此时问题不在语言理解，而在视觉输入前就已经损失了关键信息。

第二个是视频场景。你问模型：“这段 3 分钟视频最后 20 秒发生了什么？”如果把整段视频按高密度帧率原样送入模型，视觉 token 会迅速占满上下文，文本问题本身反而没多少空间。这不是模型“不聪明”，而是预算不够。

因此，LLaVA-NeXT 的边界也要说清楚。它不是一个无限长度视频模型，不是纯粹为长时序建模设计的框架，也不是把视频理解问题彻底转化成时间推理问题的专用架构。它更像是在现有视觉编码器和 LLM 组合框架内，尽可能高效地安排空间与时间信息。

下表给出常见任务与适用边界：

| 任务类型 | 是否适合 | 原因 |
|---|---|---|
| 长图 OCR | 适合 | 小字和局部结构需要保留，AnyRes 有明显价值 |
| 表格截图理解 | 适合 | 表格线、单元格对齐在缩放后容易丢失 |
| 长网页截图问答 | 适合 | 长宽比异常，固定缩放损失大 |
| 短视频问答 | 适合 | 抽帧后仍能覆盖主要事件 |
| 视频质检/巡检 | 较适合 | 可通过稀疏采样抓异常片段 |
| 长视频全量时序建模 | 不完全适合 | 帧数增长会线性吞噬上下文 |
| 精细动作持续时间建模 | 边界较强 | 更依赖专门的时序模型 |

视频的限制可以直接写成：

$$
L_{\text{vis}} \approx N\cdot T_{\text{frame}}
$$

它的含义非常朴素：如果每帧要 144 个 token，16 帧就是 2304 个；32 帧就会变成 4608 个。帧数翻倍，视觉长度也几乎翻倍。只要 $L_{\max}$ 固定，帧数增加就一定会挤压文本空间，或者迫使你进一步压缩每帧细节。

所以，问题定义不能写成“LLaVA-NeXT 让模型能看任意分辨率和任意长度视频”，这不准确。更准确的说法是：它提供了一种更合理的 token 分配方式，让模型在固定预算下更少丢掉有价值的视觉信息。

## 核心机制与推导

先看图像版。`AnyRes` 的第一步是选网格。网格可以理解为“把图像切成几块”的方案，例如 `2×2`、`1×3`、`4×1`。这些方案不是为了美观，而是为了适应不同长宽比和内容分布。横向长图更适合 `1×n`，纵向长图更适合 `n×1`，近似方图则更适合 `2×2` 这类均衡切法。

设输入图像经过网格选择后，被切成 $m\times n$ 个子图。每个子图都送入同一个视觉编码器。视觉编码器，白话说就是“把像素变成模型能处理的向量序列”的网络。随后，所有子图产生的视觉 token 会按顺序拼接，再通过投影层对齐到 LLM 的输入空间，与文本 token 合并后一起送入语言模型。

网格方案可以这样理解：

| 网格方案 | 子图数量 | 适合形态 | 对细节保留的影响 |
|---|---:|---|---|
| `2×2` | 4 | 接近方形的大图 | 平衡全局与局部 |
| `1×2` | 2 | 横向长图 | 比整体缩放更稳 |
| `1×3` | 3 | 更长的横图 | 小字保留更好，但 token 更多 |
| `1×4` | 4 | 超长横图 | 细节最稳，但预算压力更高 |
| `2×1` | 2 | 纵向长图 | 适合海报、长截图 |
| `3×1` | 3 | 更长的纵图 | 适合网页长截图 |
| `4×1` | 4 | 超长纵图 | 细节保留最好，代价也最高 |

这一步的推导其实不复杂。设第 $k$ 个子图产出 $T_k$ 个 token，那么总视觉长度就是：

$$
L_{\text{vis}}=\sum_{k=1}^{mn}T_k
$$

再加上文本长度，得到：

$$
L_{\text{total}}=L_{\text{vis}}+L_{\text{text}}\le L_{\max}
$$

所以 `AnyRes` 的价值不是“白送细节”，而是“用更多视觉 token 购买更多局部清晰度”。代价就是预算消耗上升。

视频版是在这个基础上再加一个时间维度。设采样后的视频帧集合为：

$$
V=\{v_t\}_{t=1}^{N}
$$

每一帧 $v_t$ 都可以走图像版的处理路径：必要时切块，经过视觉编码器，再做空间压缩。空间压缩的意思是“把每帧的视觉 token 数降下来”，例如把原本的 $24\times24$ token 网格压成 $12\times12$。白话说，就是在每帧内部先做一次降采样，避免单帧过重。

一个常见的玩具例子如下。若每帧压缩后保留 $12\times12$ 个 token，则：

$$
12\times12=144
$$

若抽样得到 16 帧，则总视觉 token 约为：

$$
16\times144=2304
$$

如果文本问题和系统提示一共再占 700 token，总长度大约 3004，仍可能落在 4096 的上下文窗口内。反过来，如果不压缩，每帧保留 $24\times24=576$ 个 token，那么 16 帧就变成 9216 个视觉 token，连视觉部分都塞不下，更不用说文本。

因此视频版的核心链路其实很清楚：

1. 先抽帧，控制 $N$。
2. 再做每帧空间压缩，控制 $T_{\text{frame}}$。
3. 需要时做位置编码扩展，保证更长序列可用。

这里的位置编码，白话说就是“告诉模型每个 token 在序列中相对位置”的机制。RoPE linear scaling 的作用，是让模型在更长输入长度上仍能工作。它不能凭空增加信息，但能避免模型在序列超长时明显退化。

下面用简化伪代码串起来：

```python
def process_image_or_video(inputs, text_tokens, is_video):
    visual_tokens = []

    if is_video:
        frames = sample_frames(inputs)
        for frame in frames:
            grid = select_anyres_grid(frame)
            tiles = split_into_tiles(frame, grid)
            feats = vision_encoder(tiles)
            feats = spatial_pooling(feats)
            visual_tokens.extend(feats)
    else:
        grid = select_anyres_grid(inputs)
        tiles = split_into_tiles(inputs, grid)
        feats = vision_encoder(tiles)
        visual_tokens.extend(feats)

    llm_inputs = concat(visual_tokens, text_tokens)
    return llm(llm_inputs)
```

这个流程说明了一个事实：图像版和视频版共享大部分组件，真正新增的是时间维度控制和更严格的 token 压缩。

真实工程例子可以看视频质检。假设你要检查一段工厂摄像头视频中“最后 20 秒是否出现人员未佩戴安全帽”。如果把整段视频高密度输入，成本很高且不稳定。更合理的实现是先按 1 fps 或事件触发抽样，再对每帧压缩 token，最后把问题文本拼接进去。这种做法未必捕捉每个细粒度动作，但对“是否出现过异常”“最后一段发生了什么”这类问答很有效。

## 代码实现

实现时最重要的不是某个单独模块，而是整条链路要一起设计。图像路径和视频路径共用的组件通常包括：视觉编码器、视觉到语言的投影层、LLM 主体、token 拼接逻辑。`AnyRes` 特有的是网格选择与切图逻辑；视频额外加入的是抽帧、每帧压缩、长上下文位置适配。

下表先看模块职责：

| 模块 | 职责 |
|---|---|
| `sample_frames` | 控制时间维度，决定保留多少帧 |
| `anyres_split` | 控制空间细节，决定如何切块 |
| `vision_encoder` | 把图像或子图编码成视觉表征 |
| `spatial_pooling` | 压缩每帧或每块的 token 数 |
| `rope_scaling` | 让更长输入长度仍能工作 |
| `concat` | 拼接视觉 token 与文本 token |
| `llm` | 执行最终问答与推理 |

代码层面最容易犯的错误，是把这些模块割裂理解。例如只做抽帧，不做空间压缩，视觉长度仍然可能超标；只做压缩，不控制帧数，长视频照样塞不进上下文；只把序列拉长，不做位置适配，模型在长输入上会退化。

下面给出一个可运行的 Python 玩具实现，用来演示 token 预算检查：

```python
def estimate_video_tokens(num_frames, pooled_h=12, pooled_w=12, text_tokens=700, max_len=4096):
    tokens_per_frame = pooled_h * pooled_w
    visual_tokens = num_frames * tokens_per_frame
    total_tokens = visual_tokens + text_tokens
    return {
        "tokens_per_frame": tokens_per_frame,
        "visual_tokens": visual_tokens,
        "total_tokens": total_tokens,
        "fits": total_tokens <= max_len,
    }

case_ok = estimate_video_tokens(num_frames=16, pooled_h=12, pooled_w=12, text_tokens=700, max_len=4096)
assert case_ok["tokens_per_frame"] == 144
assert case_ok["visual_tokens"] == 2304
assert case_ok["total_tokens"] == 3004
assert case_ok["fits"] is True

case_overflow = estimate_video_tokens(num_frames=16, pooled_h=24, pooled_w=24, text_tokens=700, max_len=4096)
assert case_overflow["tokens_per_frame"] == 576
assert case_overflow["visual_tokens"] == 9216
assert case_overflow["fits"] is False
```

这个例子虽然简单，但反映了真实实现中的核心约束：预算必须先算。否则系统上线后最常见的结果不是“慢一点”，而是回答质量明显波动，甚至出现无意义输出。

一个更贴近工程的伪代码如下：

```python
frames = sample_frames(video, fps=1)
visual_tokens = []

for frame in frames:
    tiles = anyres_split(frame)
    feats = vision_encoder(tiles)
    feats = spatial_pooling(feats)
    visual_tokens.append(feats)

text_tokens = tokenize(question)
inputs = concat(visual_tokens, text_tokens)
inputs = rope_scaling(inputs, target_ctx=8192)
outputs = llm(inputs)
```

这里需要注意两个实现细节。

第一，`AnyRes` 不一定意味着每一帧都切成很多块。若视频本身分辨率不高，或者任务更偏全局语义，未必需要激进切块。第二，`rope_scaling` 不是补救一切的万能开关。它解决的是长度适配，不解决信息密度不足或关键帧漏采的问题。

## 工程权衡与常见坑

LLaVA-NeXT 的工程权衡主要集中在三个维度：模型规模、token 预算、任务类型。模型规模越大，通常推理能力越强，但吞吐、显存、延迟都会更差。token 预算越宽松，视觉保真度越高，但成本更大。任务如果偏 OCR 和局部细节，空间分辨率优先；如果偏视频事件定位，时间采样优先。

7B、13B、34B 的对比不能简单理解成“参数越大越好”。因为真实效果还受底座 LLM、训练数据、视觉编码器、训练策略影响。也就是说，这不是一个完全同构的纯参数缩放实验。比较它们时，更合理的做法是看“单位成本换来多少增益”，而不是只盯着分数高低。

下表总结常见坑：

| 常见坑 | 问题本质 | 规避方法 |
|---|---|---|
| 只看参数规模 | 忽略 backbone 和数据差异 | 同时看架构、训练集、任务类型 |
| 只做抽帧 | 每帧 token 仍可能过大 | 抽帧后再做空间压缩 |
| 多帧一定更好 | 帧数上升会稀释预算 | 先算 token 预算再决定帧数 |
| 不做 RoPE scaling | 长序列位置表示失效 | 需要时启用长度适配 |
| AnyRes 开得越大越好 | 子图越多，成本越高 | 根据长宽比和任务选择网格 |
| 把视频当图像堆叠 | 忽略时间覆盖问题 | 明确采样策略与问题窗口 |

其中最关键的一条，仍然是：

$$
L_{\text{vis}} \approx N\cdot T_{\text{frame}}
$$

它说明了为什么“不断加帧”常常是错误方向。预算固定时，若 $N$ 增大，你只能接受两种结果之一：要么总长度超限，要么每帧被压得更狠，导致局部细节消失。对于“是否有人出现”“某时刻屏幕显示了什么”这种任务，后者尤其危险，因为关键证据可能正好藏在被压掉的小区域里。

一个新手常犯的误区是：为了让模型“看得更清楚”，不断提高帧数和分辨率。实际上，正确做法通常是先问任务的主矛盾是什么。若任务是长网页截图 OCR，应该优先空间切块；若任务是“最后 20 秒发生了什么”，应该优先控制时间采样并保留该时间窗口内的有效帧。

## 替代方案与适用边界

如果任务主要是普通图像问答，且图片分辨率不高，固定分辨率视觉编码方案通常已经足够。它实现简单、吞吐稳定、部署成本低。问题在于，一旦遇到长图、表格截图、海报小字、代码截图，统一缩放就会开始明显掉点。

`AnyRes` 的优势在于空间细节保留。它适合长网页截图、OCR、文档图、表格图、信息密集的高分辨率图片。这里的“适合”不是说它一定最强，而是说在“仍使用通用视觉编码器 + LLM”的路线下，它是一种很自然且有效的增强手段。

视频路径的“抽帧 + 压缩”方案则适合视频问答、视频质检、监控片段摘要、演示视频理解。它的强项是让通用多模态框架在有限成本下处理视频，而不是做最强的细粒度时序建模。若任务核心变成“动作持续了多久”“两个相似动作的先后关系”“长时间事件链推理”，更偏专门时序建模的视频模型往往更合适。

下表给出替代路线对比：

| 方案 | 优点 | 缺点 | 更适合的任务 |
|---|---|---|---|
| 固定分辨率视觉编码 | 实现简单，推理稳定 | 高分辨率细节损失大 | 普通图像问答 |
| AnyRes | 长图和高分辨率细节保留更好 | token 开销上升 | OCR、表格、长截图 |
| 抽帧 + 压缩的视频方案 | 能在有限预算下处理视频 | 长时序信息仍会丢失 | 视频问答、视频质检 |
| 纯时序视频模型 | 时序建模能力更强 | 系统复杂度更高，通用性未必最好 | 动作识别、事件序列建模 |

无论选哪条路线，都绕不开同一个约束：最大上下文长度 $L_{\max}$。区别只是不同方案把预算花在不同地方。图像任务更强调“每个 token 要尽量保住空间细节”；视频任务更强调“有限 token 要覆盖尽量多的关键时间点”。这也是为什么图像和视频不能混成一个问题讨论。它们共享预算约束，但主矛盾不同。
