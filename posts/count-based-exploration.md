## 核心结论

基于计数的探索（count-based exploration）是一类最直接的探索策略：用状态访问次数 $N(s)$ 估计“这个状态有多新”，访问越少，探索奖励越高。这里的“新”不是“更接近目标”，而是“历史上更少见”。

在强化学习里，智能体每一步通常拿到两部分奖励：

$$
r_t = r_{\text{env}, t} + r_{\text{novel}}(s_t)
$$

其中 $r_{\text{env}, t}$ 是环境主奖励，表示任务本身给的分数；$r_{\text{novel}}(s_t)$ 是新颖性奖励，表示“这个状态值不值得因为稀有而再看一眼”。最常见的写法是：

$$
r_{\text{novel}}(s) = \frac{\beta}{\sqrt{N(s) + c}}
$$

$\beta$ 是探索强度系数，也就是“额外奖励放多大”；$c$ 是平滑项，作用是避免除零，并控制初始奖励尺度。

这类方法最适合稀疏奖励任务。稀疏奖励的意思是，环境很少明确告诉你“做对了”，比如走到终点才给分，中间几乎没有反馈。此时，count bonus 可以先把智能体推向“没去过的区域”，让它先覆盖状态空间，再逐步碰到真正的任务奖励。

一个最直观的判断标准是：如果你的状态可以稳定离散化、聚类或哈希，count-based exploration 往往是第一批应该尝试的探索方法。它简单、便宜、容易诊断，但在连续高维视觉状态里，精确计数会失效，必须依赖哈希、特征空间计数或伪计数近似。

## 问题定义与边界

先把问题说清楚。本文讨论的不是“哪个状态更有价值”，而是“哪个状态更少被访问”。价值和新颖性不是一回事。

- 状态（state）：智能体当前所处情形的数学表示。白话说，就是“现在看到的全部关键信息”。
- 新颖性（novelty）：一个状态在历史轨迹中有多少见。白话说，就是“这个地方是不是第一次来，或者很少来”。

如果环境是小型网格世界，状态可以直接写成坐标 $(x, y)$，那么计数非常自然：每到一个格子，就把这个格子的访问次数加一。问题在于，很多真实任务不是这样。

如果状态是连续值，比如机器人位置、速度、关节角度，那么“完全一样”的状态几乎不会重复出现。若直接对原始状态精确计数，结果通常是每个状态都只出现一次，$N(s)$ 基本恒等于 1，这时新颖性奖励失去区分能力。

如果状态是图像，比如 Atari 游戏画面或摄像头输入，麻烦更大。两帧画面即使语义上几乎相同，只要像素有一点变化，就会被当成不同状态。因此必须先把状态压缩成更稳定的表示，再做计数。

下面这张表给出边界条件：

| 状态类型 | 典型表示 | 计数方式 | 优点 | 局限 |
| --- | --- | --- | --- | --- |
| 离散状态 | 网格坐标、有限棋盘局面 | 直接统计 $N(s)$ | 精确、可解释、实现最简单 | 只适用于小规模离散空间 |
| 连续低维状态 | 位置、速度、角度 | 先离散化或特征分桶，再统计 $N(\phi(s))$ | 成本低，适合控制任务 | 分桶方式会强烈影响效果 |
| 高维图像状态 | 像素帧、视觉观测 | 哈希计数、特征空间计数、伪计数 $\hat N(s)$ | 可扩展到深度 RL | 近似误差大，碰撞与泛化难平衡 |

这里的 $\phi(s)$ 表示状态映射。状态映射的白话意思是：先把原始状态变成一个更适合统计的“键”。如果原始状态太细，就先压缩；如果原始状态太噪，就先抽特征。

一个新手常见误解是：既然探索奖励鼓励去低频状态，那它是不是在奖励“乱走”？答案是部分正确。它确实奖励“去没去过的地方”，但这正是它的用途。count-based exploration 从来不是直接教智能体“哪条路通向目标”，而是先提升覆盖率，让智能体更有机会碰到目标信号。

## 核心机制与推导

### 1. 从访问次数到奖励衰减

核心量只有一个：状态访问次数 $N(s)$。

- 第一次访问时，$N(s)$ 小，新颖性奖励高。
- 重复访问时，$N(s)$ 变大，新颖性奖励衰减。
- 如果某个区域已经很熟，新颖性奖励会自然接近 0。

常见公式是：

$$
r_{\text{novel}}(s) = \frac{\beta}{\sqrt{N(s) + c}}
$$

为什么是平方根衰减，而不是 $1/N(s)$ 或常数奖励？因为平方根衰减比较平衡。

- 如果衰减太慢，比如常数奖励，智能体会长期沉迷刷“探索分”。
- 如果衰减太快，比如 $1/N(s)$，很多状态只访问几次后就没有驱动力了。
- $\sqrt{N(s)}$ 提供了一种中间强度：前几次访问下降明显，后面逐渐变平。

看一个玩具例子。设 $\beta = 1, c = 0$，某状态被访问多次时的奖励如下：

| 访问次数 $N(s)$ | 奖励 $1/\sqrt{N(s)}$ |
| --- | --- |
| 1 | 1.00 |
| 2 | 0.71 |
| 4 | 0.50 |
| 9 | 0.33 |
| 16 | 0.25 |

这个表说明一件事：新颖性不是二元判断，不是“新”或“旧”，而是连续衰减。第一次最值钱，之后越来越不值钱。

### 2. 总奖励为什么能推动探索

策略优化时，智能体并不知道“未来哪里有终点”，它只能跟着累计回报上升的方向走。如果总奖励改成

$$
r_t = r_{\text{env}, t} + \frac{\beta}{\sqrt{N(s_t) + c}}
$$

那么在环境主奖励很稀少时，策略梯度或 Q 学习至少还能从“访问低频状态”这件事上拿到训练信号。换句话说，count bonus 把“覆盖未知区域”变成了一个可优化目标。

这就是它在稀疏奖励环境里有效的原因。它不是替代任务目标，而是在任务目标太弱时，补一条更容易学到的辅助信号。

### 3. 高维状态为什么要近似计数

在深度强化学习里，状态往往是高维向量或图像，无法直接做 $N(s)$。于是公式变成：

$$
r_{\text{novel}}(s) = \frac{\beta}{\sqrt{N(\phi(s)) + c}}
$$

这里 $\phi(s)$ 是状态编码，也就是把状态映射到一个桶、一个特征键，或一个近似邻域。原理是：不再问“这个原始状态是否出现过”，而是问“这个编码后的状态簇是否出现过”。

一种常见做法是 SimHash。它先从原始状态提取特征 $g(s)$，再做随机投影：

$$
\phi(s) = \operatorname{sgn}(A g(s))
$$

其中：

- $g(s)$：状态特征。白话说，就是从原始输入中抽出来的压缩表示。
- $A$：随机投影矩阵。白话说，就是一组固定的随机方向。
- $\operatorname{sgn}$：符号函数，把每个投影结果变成正负号，对应二进制位 0/1。

这样，原始状态会被映射成一个二进制码。如果两个状态特征很相似，它们更可能落到同一个哈希桶，共享计数 $N(\phi(s))$。这就是哈希计数的核心：用“相似状态共享桶”的方式，近似“相似状态共享新颖性”。

### 4. 真实工程例子

一个典型真实例子是稀疏奖励导航。设机器人在 3D 迷宫里移动，只有到达目标点时环境才给 +1，其余时刻奖励全是 0。训练初期，若没有探索机制，策略可能长期卡在起点附近，因为“无论往哪走都没反馈”。

加入 count bonus 后，情况会变成这样：

1. 起点附近的状态很快被访问多次，新颖性奖励迅速下降。
2. 离起点更远的区域访问次数低，奖励更高。
3. 策略因此逐步向未知房间扩展，而不是反复在熟悉区域打转。
4. 一旦某条新路径通向终点，环境主奖励开始接管，策略再把“新颖路径”收敛为“有效路径”。

Montezuma’s Revenge 之所以常被用来讨论探索，就是因为它环境奖励稀疏，靠随机试错极难碰到有效奖励。这类任务里，count-based exploration 往往比纯 epsilon-greedy 更有结构。

## 代码实现

实现可以拆成三步：生成状态键、维护计数表、把计数转成 bonus。下面给一个可运行的最小 Python 版本，先演示离散状态，再演示简单分桶。

```python
from collections import defaultdict
from math import sqrt

class CountBonus:
    def __init__(self, beta=1.0, c=0.0):
        self.beta = beta
        self.c = c
        self.counts = defaultdict(int)

    def hash_state(self, state):
        # 离散状态直接作为键
        return state

    def bonus(self, state):
        key = self.hash_state(state)
        self.counts[key] += 1
        n = self.counts[key]
        return self.beta / sqrt(n + self.c)

# 玩具例子：离散网格状态
cb = CountBonus(beta=1.0, c=0.0)

b1 = cb.bonus((0, 0))   # 第一次访问
b2 = cb.bonus((0, 0))   # 第二次访问
b3 = cb.bonus((1, 0))   # 新状态第一次访问

assert round(b1, 2) == 1.00
assert round(b2, 2) == 0.71
assert round(b3, 2) == 1.00
assert cb.counts[(0, 0)] == 2
assert cb.counts[(1, 0)] == 1

class BucketedCountBonus(CountBonus):
    def __init__(self, beta=1.0, c=1.0, bucket_size=0.5):
        super().__init__(beta=beta, c=c)
        self.bucket_size = bucket_size

    def hash_state(self, state):
        # 连续二维状态先分桶，再计数
        x, y = state
        bx = int(x / self.bucket_size)
        by = int(y / self.bucket_size)
        return (bx, by)

cb2 = BucketedCountBonus(beta=2.0, c=1.0, bucket_size=0.5)

# 0.10 和 0.20 会进入同一桶
k1 = cb2.hash_state((0.10, 0.20))
k2 = cb2.hash_state((0.24, 0.49))
k3 = cb2.hash_state((0.60, 0.20))

assert k1 == k2
assert k1 != k3

r1 = cb2.bonus((0.10, 0.20))
r2 = cb2.bonus((0.24, 0.49))
r3 = cb2.bonus((0.60, 0.20))

assert r1 > r2   # 同一桶第二次访问奖励更低
assert r3 == r1  # 新桶第一次访问，奖励相同
```

这段代码有两个要点。

第一，`bonus` 里是“先加一再计算”。这意味着第一次访问得到的是基于 $N=1$ 的奖励，而不是基于 $N=0$ 的无穷大奖励。工程上这比“先算再加一”更稳。

第二，连续状态要先映射成键。上面的 `BucketedCountBonus` 用的是最简单的分桶方法，本质上就是用 $\phi(s)$ 替代原始状态 $s$。

如果放进训练循环，骨架通常是这样：

```python
obs = env.reset()
done = False

while not done:
    action = agent.act(obs)
    next_obs, env_reward, done, info = env.step(action)

    intrinsic_reward = count_bonus.bonus(next_obs)
    train_reward = env_reward + intrinsic_reward

    agent.observe(obs, action, train_reward, next_obs, done)
    obs = next_obs
```

这里有一个工程上必须坚持的规则：训练时可以把 intrinsic reward 加进去，但评估时不能把它算进最终任务分数。原因很简单，评估要回答的是“任务完成得怎样”，不是“探索得有多热闹”。如果把 bonus 也算进去，就会把“到处乱看”误判成“任务表现更好”。

## 工程权衡与常见坑

count-based exploration 的难点不在公式，在状态表示。

| 问题 | 后果 | 规避方式 |
| --- | --- | --- |
| 连续状态直接精确计数 | 几乎每次都像第一次访问，计数失真 | 先离散化、分桶或做特征映射 |
| 哈希桶太粗 | 不同状态被混为一类，出现假共享 | 增加特征分辨率，检查碰撞率 |
| 哈希桶太细 | 计数无法复用，bonus 长期过高 | 减少桶数或压缩特征 |
| $\beta$ 过大 | 智能体沉迷探索，忽略主任务 | 按任务奖励量级调参，逐步退火 |
| $\beta$ 过小 | 新颖性信号太弱，几乎不起作用 | 提高权重，或标准化奖励尺度 |
| 训练评估不分离 | 指标失真，误以为性能提升 | 评估只看环境主奖励 |

### 1. 哈希粒度是核心超参数

“桶太粗”和“桶太细”是最典型的一对反向错误。

- 桶太粗：很多不同房间都被映射成同一个键。结果是，智能体明明到了新地方，计数系统却说“这里我来过”，于是不给足够探索奖励。
- 桶太细：几乎每个画面都单独成桶。结果是，系统始终觉得“一切都很新”，探索奖励长期居高不下，主任务反而学不好。

一个简单经验是：先观察哈希后每个 episode 的唯一桶数。如果唯一桶数接近步数总数，说明太细；如果大部分轨迹都落在很少几个桶里，说明太粗。

### 2. $\beta$ 要和任务奖励同量级比较

$\beta$ 不是越大越好。它决定 intrinsic reward 的量级。如果环境奖励通常只有 0 和 1，而你把 bonus 长期保持在 2 到 3，策略就会更在意“找新状态”而不是“完成任务”。

更稳的做法通常是：

- 先统计环境奖励的典型范围。
- 让 bonus 初始量级与环境奖励大致同阶。
- 必要时对 $\beta$ 做退火，也就是训练前期大一些，后期逐渐减弱。

### 3. 重置与计数生命周期

另一个常见坑是“计数表按什么周期清空”。如果每个 episode 都重置计数，那么奖励变成“本局内新颖性”；如果整个训练过程共享计数，那么奖励表示“全局新颖性”。

两者没有绝对对错，但含义不同。

- 本局计数适合鼓励局部覆盖，常见于短时导航。
- 全局计数更适合长期探索，能避免模型反复回到历史上早已熟悉的区域。

很多论文里的计数默认是全局的，因为目标是跨 episode 累积知识。

## 替代方案与适用边界

count-based exploration 很强，但不是默认最优。它适合“状态可聚类、任务稀疏、状态空间大但结构还能压缩”的场景。如果状态变化极快、观测噪声大、哈希很难稳定，那么单纯计数就会变脆弱。

下面做一个对比：

| 方法 | 核心信号 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- | --- |
| Count-based | 访问次数 $N(s)$ 或 $N(\phi(s))$ | 简单、高效、可解释 | 高维连续状态需要近似，哈希敏感 | 网格世界、导航、可稳定分桶的稀疏奖励任务 |
| RND | 随机目标网络上的预测误差 | 适合高维视觉输入，实现相对直接 | 可能被噪声或不可预测细节误导 | 图像状态、深度 RL 探索 |
| ICM | 动作导致的可预测状态变化误差 | 更关注“可控变化” | 训练更复杂，辅助模型可能不稳 | 需要强调 agent controllability 的任务 |
| Entropy bonus | 策略分布熵 | 直接鼓励动作多样性，优化简单 | 只保证动作随机，不保证到达新状态 | 早期策略优化、局部防塌缩 |

这里解释几个术语。

- RND（Random Network Distillation）：用随机固定网络生成目标特征，再训练预测网络去拟合，预测误差大说明状态少见。白话说，就是“见得少的状态更难被学会模仿”。
- ICM（Intrinsic Curiosity Module）：利用前向动力学预测误差做内在奖励。白话说，就是“如果某动作带来的状态变化还不熟，就奖励它”。
- Entropy bonus：在策略目标里鼓励更高熵。白话说，就是“不要太早把动作分布压成单一路径”。

如果你的问题是“智能体总在几个动作里过早收缩”，熵奖励通常更直接；如果你的问题是“状态太高维，分桶非常不稳”，RND 或 ICM 可能更实用；如果你的问题是“我需要一个便宜、可解释、容易调试的稀疏探索基线”，count-based exploration 仍然非常值得先做。

一个实用判断是：先问自己能不能定义一个稳定的 $\phi(s)$。如果可以，count-based 往往成立；如果不可以，说明“状态是否相同”本身就难定义，那基于预测误差的探索方法通常更合适。
