## 核心结论

课程学习（Curriculum Learning, CL）指的是按难度安排训练样本的顺序或权重。白话说，它像“先做基础题，再做奥数题”：不是因为难题不重要，而是因为模型先要学会稳定的底层模式，后面才有能力吸收高阶结构。

在 LLM 训练里，课程学习最有价值的场景，不是“所有任务都能更快”，而是下面这几类：

1. 多步推理任务，难度随着推理步数明显增加。
2. 稀疏监督任务，简单样本能先提供稳定梯度。
3. 数据质量不均匀的任务，先吃干净样本，再逐步放开复杂样本。
4. 容易陷入坏局部最优的训练过程，先学稳定模式有助于减少早期震荡。

它成立的核心原因是：训练早期的模型能力很弱，若一开始就让它接触大量高难度样本，梯度信号会更噪、更不稳定，优化器容易在错误方向上走很远。课程学习通过“难度节奏”控制样本暴露范围，让模型先建立低层规律，再逐步学习复杂组合关系。

但它不是万灵药。课程设计不当时，课程学习可能还不如随机混洗。最常见的失败模式是：简单题学太久、难度定义失真、人工节奏落后于模型真实能力、合成数据中的污染模式被过早强化。

| 方面 | 优势 | 典型风险 |
| --- | --- | --- |
| 收敛速度 | 早期梯度更稳定，常见于多步推理和结构化任务 | 课程过慢会拖延进入高难阶段 |
| 泛化能力 | 先学基础模式，后学复杂组合，常能减少坏局部最优 | 若“简单样本”分布过窄，可能过拟合基础题 |
| 工程可控性 | 可通过采样器、权重函数、阈值逐步放量 | 难度打分本身可能不准确 |
| 数据治理 | 可先用干净数据 warm-up，再放开复杂或噪声数据 | 若污染样本被错误标成“简单”，会更早伤害模型 |

## 问题定义与边界

课程学习的严格定义不是“按顺序喂数据”这么简单，而是：根据样本难度和模型当前能力，对训练样本的出现顺序、采样概率或损失权重进行动态控制。

这里有两个关键术语：

- 难度：样本对模型而言有多难。白话说，就是“这道题要花多少模型能力才能答对”。
- 节奏函数（pacing function）：控制训练进行到第几步时，允许模型接触多难的数据。白话说，就是“什么时候开始放难题”。

对 LLM 来说，难度通常不是单一维度。常见来源包括：

- 推理步数
- 知识密度
- 语法嵌套层数
- 上下文长度
- 标签噪声或答案歧义

一个玩具例子很容易看清边界。下面两个样本都能用于语言模型微调，但它们对能力要求不同：

| 样本示例 | 难度评分 | 主要能力要求 | 适合解决的问题 |
| --- | --- | --- | --- |
| “法国首都是哪里？” | 0.2 | 单跳知识匹配 | 基础语言对齐、短答案输出 |
| “若甲比乙大3，乙比丙大2，丙是5，求甲，并解释步骤。” | 0.8 | 多步推理、状态保持、链式展开 | 数学推理、分步解释、复杂监督 |

课程学习适合解决的是“样本难度结构明确，而且模型早学基础有帮助”的问题。它不适合的边界同样要说清：

1. 如果数据天然同质，难度差异很小，课程收益通常有限。
2. 如果难度无法可靠估计，错误课程可能比随机采样更差。
3. 如果复杂样本本身包含关键分布，而简单样本与目标分布差太远，先学简单题反而会走偏。
4. 如果任务依赖广覆盖而不是渐进能力积累，随机混洗可能更稳。

所以课程学习不是“从易到难”这句口号，而是一个关于样本排序、能力匹配和优化稳定性的工程设计问题。

## 核心机制与推导

课程学习通常可写成一个加权训练目标：

$$
L(\theta, t)=\sum_{i=1}^{N} w_i(t)\,\ell(f_\theta(x_i), y_i)
$$

其中：

- $\ell$ 是单样本损失，白话说，就是“这条样本让模型错了多少”。
- $w_i(t)$ 是第 $t$ 步时该样本的权重。
- $f_\theta$ 是当前模型。

在课程学习里，权重通常由“样本难度”和“训练节奏”共同决定。一个常见写法是：

$$
w_i(t)=g(d_i, p(t))
$$

其中 $d_i \in [0,1]$ 表示样本难度，$p(t)\in[0,1]$ 表示课程节奏。直观理解是：训练前期 $p(t)$ 小，只放低难度；训练后期 $p(t)$ 逐渐增大，高难样本才被更多使用。

一个简单的门控写法是：

$$
w_i(t)=\mathbf{1}[d_i \le p(t)]
$$

即只有难度不超过当前阈值的样本才进入训练。另一个更平滑的写法是：

$$
w_i(t)=\sigma(\alpha (p(t)-d_i))
$$

这里 $\sigma$ 是 Sigmoid 函数，白话说，它把“是否放行”从硬切换变成软切换，减少阈值抖动。

题目里给的数值例子可以直接说明这种机制。设两个样本：

- 样本 A：常识问答，难度 $d_A=0.2$
- 样本 B：多步数学推理，难度 $d_B=0.8$

若前 10 个 mini-batch 的 pacing 固定为 $p(t)=0.3$，之后逐步升到 1，那么前期可以把有效权重近似理解为：

$$
w_A \approx 0.3 \times 0.2 = 0.06,\quad
w_B \approx 0.3 \times 0.8 = 0.24
$$

如果进一步采用阈值门控，样本 A 会稳定进入训练，样本 B 则可能只少量出现或完全不出现。等到后期 $p(t)=0.8$ 时，高难样本才真正被解锁：

$$
w_B \approx 0.8 \times 0.8 = 0.64
$$

这背后的优化逻辑是：早期模型表示能力不足时，强推高难样本会让损失面更陡、更噪；先用低难样本训练，相当于先在更平滑的区域建立参数基础，再进入复杂区域。

若把 pacing 画成折线图，最常见有三种思路：

1. 线性增长：每训练一段时间，阈值均匀提高。
2. 分段增长：先长时间 warm-up，再集中放量难题。
3. competence-aware：根据验证集表现提高阈值。这里 competence 是“模型当前能力”，白话说，就是“模型已经学到什么程度”。

第三种通常更适合工程系统，因为模型能力提升并不总是线性的。固定日程常见的问题是：模型已经准备好学难题了，但课程还没放开；或者模型还没站稳，难题已经提前进来。

## 代码实现

课程学习在工程上最常见的落点不是改模型结构，而是改数据管道。最小实现通常只需要三部分：

1. 给每条样本打一个 difficulty 分数。
2. 定义 pacing 函数 $p(t)$。
3. 在 sampler 或 loss 上按 $d_i$ 和 $p(t)$ 做门控或加权。

先看一个玩具例子。假设我们有 6 条样本，难度分别是 `[0.1, 0.2, 0.4, 0.6, 0.8, 0.9]`。训练初期阈值只到 `0.3`，那么只能采到前两条；中期阈值到 `0.6`，前四条可用；后期阈值到 `1.0`，全量可用。这个机制已经能表达最基本的课程学习。

下面给一个可运行的 Python 例子，演示 mini-batch 级别的 difficulty gating 和 loss weighting：

```python
from math import isclose

samples = [
    {"id": "s1", "difficulty": 0.1, "base_loss": 0.5},
    {"id": "s2", "difficulty": 0.2, "base_loss": 0.6},
    {"id": "s3", "difficulty": 0.4, "base_loss": 0.8},
    {"id": "s4", "difficulty": 0.6, "base_loss": 1.0},
    {"id": "s5", "difficulty": 0.8, "base_loss": 1.2},
]

def pacing(step, warmup=5, total=20):
    if step <= warmup:
        return 0.3
    growth = (step - warmup) / (total - warmup)
    return min(1.0, 0.3 + 0.7 * growth)

def active_samples(step, dataset):
    p = pacing(step)
    return [s for s in dataset if s["difficulty"] <= p]

def weighted_loss(step, dataset):
    p = pacing(step)
    active = active_samples(step, dataset)
    loss = 0.0
    for s in active:
        weight = p * s["difficulty"]
        loss += weight * s["base_loss"]
    return loss

early = active_samples(3, samples)
mid = active_samples(10, samples)
late = active_samples(20, samples)

assert [s["id"] for s in early] == ["s1", "s2"]
assert [s["id"] for s in mid] == ["s1", "s2", "s3", "s4"]
assert [s["id"] for s in late] == ["s1", "s2", "s3", "s4", "s5"]

assert isclose(pacing(3), 0.3)
assert pacing(20) == 1.0
assert weighted_loss(20, samples) > weighted_loss(3, samples)

print("curriculum learning toy example passed")
```

真实训练里，一般有两种接入方式。

第一种是改采样器。逻辑是：当前 batch 只从满足 $d_i \le p(t)$ 的样本池里采样。这种方式直观、容易解释，但会让高难样本在前期完全不可见。

第二种是改损失权重。逻辑是：所有样本都可见，但难样本前期权重低，后期再升高。这样能减少“完全看不到高难分布”的风险。

伪代码如下：

```python
for step, batch in enumerate(loader):
    p = pacing(step)
    logits = model(batch["input_ids"])
    per_token_loss = compute_loss(logits, batch["labels"], reduction="none")
    per_sample_loss = reduce_to_sample_loss(per_token_loss)

    difficulty = batch["difficulty"]  # [B]
    weights = (difficulty <= p).float() * (p * difficulty)

    loss = (per_sample_loss * weights).sum() / weights.sum().clamp_min(1.0)
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()
```

真实工程例子是多步数学推理训练。比如一个 Synthetic Math 数据集，样本可按推理步数分层：

- 1 步题：变量代入、单次运算
- 2 到 3 步题：中间状态需要保持
- 4 步以上题：链式依赖明显增加

一个合理流程通常是：

1. 先用 1 步题 warm-up，让模型学会格式、答案模式、基础变换。
2. 再逐步解锁 2 到 3 步题，建立中间状态表示。
3. 最后才大规模放开 4 步以上题，训练长链条推理。

这类设计的重点不是“高难题更高级”，而是“高难题依赖低难能力作为前置条件”。

## 工程权衡与常见坑

课程学习最容易犯的错，是把“从易到难”理解成一条固定日程表。真实训练里，模型能力提升的速度不是常数，数据难度定义也经常不稳定。

新手版本可以这样理解：如果一直只给简单题，模型当然会把简单题做得越来越好，但这不等于它会做复杂题。很多时候，复杂题需要的是不同的中间表示和更长的依赖链。如果这些能力始终没有被训练，最后泛化甚至可能不如随机混洗。

下面这个表格更接近工程现实：

| 方案 | 优点 | 主要问题 | 对策 |
| --- | --- | --- | --- |
| 静态课程 | 实现简单，可控性强 | 节奏容易落后于模型能力 | 定期重估难度阈值 |
| 动态 pacing | 更匹配模型当前 competence | 实现更复杂，需要验证指标 | 用 held-out 集表现驱动阈值 |
| 完全随机 baseline | 简单、稳健、覆盖全面 | 早期梯度噪声大，难样本拖累优化 | 用于对照，避免过度迷信 CL |

常见坑主要有五类。

第一，难度定义错了。  
“长样本”不一定比“短样本”难；“代码题”也不一定比“问答题”难。若难度指标与真实学习难度错位，课程就会把数据排错顺序。

第二，简单数据分布过窄。  
如果所谓“简单样本”几乎全是模板化句式，模型会先学会模板，不一定学会任务本身。后期再切到复杂样本时，损失会重新暴涨。

第三，课程过硬。  
前期完全屏蔽高难样本，虽然稳定，但也可能造成分布突变。更稳妥的方式通常是“低频暴露”而不是“完全隔离”。

第四，数据污染被提早放大。  
污染指训练数据里混入不该有的答案线索、重复模式或泄漏。白话说，就是模型不是学会了推理，而是记住了脏规律。如果这些样本被标成“简单”并在早期大量出现，课程学习会更快把错误模式固化。

第五，没有与随机 baseline 对照。  
很多团队把课程学习做出效果，其实是因为顺手清洗了数据、改了 batch 构成或调了训练时长。如果没有严格对照，很难说明收益真来自 curriculum。

## 替代方案与适用边界

课程学习最直接的替代方案是随机采样。随机采样的优点非常朴素：实现最简单，不需要定义难度，不需要调节节奏，也不会因为课程设计错误而系统性偏移。

在一些复杂合成任务里，随机采样甚至可能与课程学习相当，或者更好。原因通常不是“课程思想错了”，而是工程实现中的难度标签、污染分布和节奏控制没有做好。

一个典型对照可以这样理解。设有两种训练策略：

- 课程学习：先 warm-up 一步推理题，再按步数逐步解锁多步题。
- 纯随机：从第一步开始就把所有步数的题混在一起训练。

前者的优势在于优化稳定、低层能力先成形；后者的优势在于分布覆盖完整、不会因为课程偏差错过关键模式。

在题目给出的 Synthetic Math 场景里，先 warm-up 再逐步解锁多步题，相比纯随机采样能带来约 2% 到 3% 的验证准确率提升。这说明在“难度结构清晰、污染可控、步数分层明确”的任务里，课程学习确实有效。但这类结果不应被误读为“所有 LLM 训练都该先易后难”。如果任务的难度维度混乱，或者高难样本本身才是目标分布核心，随机采样会更稳。

还有一些混合策略常比“纯课程”更实用：

| 方案 | 核心思路 | 适用边界 |
| --- | --- | --- |
| Curriculum Learning | 按难度逐步放开样本 | 难度结构清晰、能力可分层 |
| Random Sampling | 全难度随机混洗 | 难度难定义、需广覆盖 |
| Hybrid Strategy | 先短 warm-up，再随机混合或软加权 | 既想稳住早期，又不想后期分布偏移 |
| CL + Instruction Tuning | 先做难度课程，再做指令跟随对齐 | 既关心任务能力，也关心输出行为 |
| CL + Soft Prompt/Adapter | 课程主要作用于特定模块或阶段 | 参数高效微调、资源受限场景 |

工程上最常见的结论不是“选一个最好方法”，而是：

1. 先跑随机 baseline。
2. 再做一个轻量课程版本，比如 5% 到 10% 训练步数的 warm-up。
3. 若收益稳定，再考虑 competence-aware 的动态版本。

这比一开始就设计复杂 curriculum 更可靠，因为课程学习真正难的不是写公式，而是把“难度”定义成一个能反映模型学习现实的量。
