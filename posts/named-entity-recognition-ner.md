## 核心结论

命名实体识别，英文是 Named Entity Recognition，简称 NER，意思是“从一段自然语言里找出有名字、可指代现实对象的片段，并判断它属于什么类型”。对知识图谱来说，它通常是节点抽取的入口；对搜索、问答、风控、病历结构化来说，它通常是最早的一层信息抽取模块。

NER 不是“把句子看懂”这么宽泛的任务，而是一个边界明确的结构化预测问题：给定文本，输出若干个实体片段，每个片段都有起止位置和类型。形式上可以写成：

$$
E=\{(start, end, type)\}
$$

其中 `start` 和 `end` 表示实体在文本中的边界，`type` 表示类别，比如 `PER`（人名）、`ORG`（机构）、`LOC`（地点）、`DISEASE`（疾病）等。

对初学者最重要的结论有三条：

| 结论 | 含义 | 工程影响 |
|---|---|---|
| NER 不只找词，还要定边界和类型 | “苹果”可能是公司，也可能是水果 | 边界错或类型错都会影响下游 |
| 平面实体适合序列标注 | 实体之间不互相包含时，用 BIO/BIES 较直接 | 新闻、通用资讯场景常够用 |
| 嵌套实体要换建模方式 | 一个实体内部再包含另一个实体时，单层标签不够 | 医疗、金融、法律更常见 |

先看一个玩具例子。

句子：`张三在清华大学读书`

目标输出可能是：

- `张三 -> PER`
- `清华大学 -> ORG`

这里实体互不包含，属于平面实体。用序列标注，也就是“给每个字或词打标签”的方法，通常足够。

再看一个更接近真实问题的例子。

句子：`我在北京大学人民医院就诊`

这里可能同时需要抽出：

- `北京大学 -> ORG`
- `北京大学人民医院 -> HOSP`

这就是嵌套实体。所谓嵌套，就是“一个实体完整包含另一个实体”。单层 BIO 或 BIES 标签无法在同一位置同时表达两层边界，因此不能硬套平面 NER 的方案。

所以，初级工程师需要先建立一个判断标准：你面对的是平面实体问题，还是嵌套实体问题；你要的是“能跑”，还是“对复杂文本稳定”。这个判断，决定了模型结构、标注规范、评测口径和算力成本。

## 问题定义与边界

NER 的输入是非结构化文本，输出是实体集合，而不是单个分类标签。设输入文本为：

$$
x=(x_1, x_2, \dots, x_n)
$$

输出实体集合为：

$$
E=\{(i,j,c)\mid 1\le i\le j\le n, c\in C\}
$$

这里 `(i,j)` 是边界，`c` 是实体类型集合 `C` 中的一个类别。

这件事看起来简单，真正落地时却容易和其他任务混淆。下面这张表先把边界切清楚。

| 任务 | 输入 | 输出 | 关注点 |
|---|---|---|---|
| 词性标注 | 文本 | 每个词的词性 | 语法功能 |
| NER | 文本 | 实体边界 + 类型 | 文本中哪些片段是实体 |
| 实体链接 | 文本 + 实体提及 | 知识库中的具体实体 ID | “苹果”到底是 Apple 公司还是水果 |
| 关系抽取 | 文本 + 实体 | 实体间关系 | “张三就职于清华大学”中的“就职于” |
| 共指消解 | 文档 | 指向同一对象的提及集合 | “马斯克”“他”“特斯拉 CEO”是否同指一人 |

一个常见误区是把 NER 当成“实体分类”。其实分类只是后半段，前半段的边界定位往往更难。比如：

- `苹果发布新手机`
- `我买了一个苹果`

前一句里，“苹果”更可能是公司；后一句里，“苹果”更可能是水果。也就是说，NER 必须用上下文来判断类型，而不是见词就打固定标签。

再看边界问题。句子：

`上海交通大学医学院附属瑞金医院`

可能存在多种标注目标：

- 只抽最长机构名：`上海交通大学医学院附属瑞金医院`
- 同时抽全称和内部机构：`上海交通大学`、`上海交通大学医学院`、`上海交通大学医学院附属瑞金医院`
- 还要抽别名或标准化简称：`瑞金医院`

这不是模型先天“对错”的问题，而是任务定义不同。也就是说，NER 的边界并不总是自然唯一，它受到业务目标约束。你做知识图谱节点抽取，通常宁可多保留候选；你做在线高精召回，可能更偏向严格边界和低误报。

可以把信息抽取链路理解成下面这样：

`文本 -> NER -> 实体链接 -> 关系抽取 -> 图谱入库`

其中 NER 是入口。一旦 NER 把边界切错，下游错误会级联传播。比如把 `北京大学人民医院` 错切成 `北京大学` 和 `人民医院`，实体链接阶段就可能把后者错误链接到别的医院，最终图谱里形成错误节点。

因此，问题定义必须先回答四个边界问题：

| 问题 | 典型表现 | 是否影响建模 |
|---|---|---|
| 是否允许嵌套 | 长机构名内部含短机构名 | 直接决定是否能用单层 BIO |
| 是否允许重叠 | 两个实体共享部分文本但不完全包含 | 决定是否需要 span 或 pointer |
| 是否跨句 | 句内没完整实体，跨句才完整指代 | 需要文档级建模或共指辅助 |
| 是否做标准化 | “北大”“北京大学”是否视作同一实体 | 这通常属于实体链接，不是纯 NER |

对零基础读者，一个实用记忆法是：NER 负责“从文本里圈出什么是实体”，实体链接负责“这个实体在现实中到底是谁”。

## 核心机制与推导

### 1. 平面 NER：序列标注

序列标注，意思是“把输入序列中的每个位置都打一个标签”。最常见的标签体系是 BIO 或 BIES。

- `B`：Begin，实体开头
- `I`：Inside，实体内部
- `O`：Outside，不属于任何实体
- `E`：End，实体结尾
- `S`：Single，单字或单词实体

例如句子：`张三在清华大学读书`

按字级标注可能是：

| 字 | 张 | 三 | 在 | 清 | 华 | 大 | 学 | 读 | 书 |
|---|---|---|---|---|---|---|---|---|---|
| 标签 | B-PER | E-PER | O | B-ORG | I-ORG | I-ORG | E-ORG | O | O |

这种做法的优点是直接、线性、好训练。缺点是每个位置只能有一个标签，所以天然更适合平面实体。

### 2. 为什么要加 CRF

CRF，条件随机场，白话解释是“一个会考虑标签之间相邻约束的解码层”。如果只靠每个位置单独分类，模型可能输出非法序列，比如 `I-ORG` 前面没有 `B-ORG`。CRF 的价值就是把“单点分数”和“标签转移是否合法”一起考虑。

BiLSTM-CRF 的经典打分公式是：

$$
S(x,y)=\sum_i \left[T_{y_{i-1},y_i}+P_{i,y_i}\right]
$$

其中：

- $P_{i,y_i}$ 是位置 $i$ 取标签 $y_i$ 的发射分数，可以理解为编码器对当前标签的偏好
- $T_{y_{i-1},y_i}$ 是从前一个标签转到当前标签的转移分数，可以理解为标签相邻是否合法、常不常见

给定文本 `x`，标签序列 `y` 的条件概率为：

$$
p(y|x)=\frac{\exp(S(x,y))}{\sum_{y'}\exp(S(x,y'))}
$$

训练时最大化正确标签序列的概率，解码时用 Viterbi 算法找最高分路径。

它的核心收益不是“表达能力突然变强”，而是“输出更规整”。对平面 NER，这通常能带来更稳定的边界。

### 3. 为什么嵌套实体会让 BIO 失效

看句子：`北京大学人民医院`

如果要同时输出：

- `北京大学 -> ORG`
- `北京大学人民医院 -> HOSP`

那么开头几个字既属于短实体，也属于长实体。但单层 BIO 标注要求每个位置只有一个标签，这就冲突了。你无法同时写出：

- `北 -> B-ORG`
- `北 -> B-HOSP`

所以问题不是“CRF 不够强”，而是“输出空间被设计死了”。

这也是很多初学者容易误解的地方：嵌套 NER 不是把平面模型训得更久就能解决，而是要换表示方式。

### 4. span classification：按区间分类

span，中文常叫“片段”或“区间”。span classification 的思路是：不再给每个位置只贴一个标签，而是直接枚举文本中的候选区间 `(i,j)`，判断这个区间是不是实体、是什么类型。

其基本形式可以写成：

$$
s(b,c)=f(h_i,h_j,h_{i:j})
$$

其中：

- $b=(i,j)$ 是候选边界
- $c$ 是类别
- $h_i,h_j,h_{i:j}$ 是由编码器得到的表示，分别表示起点、终点和区间内部上下文
- $f$ 是一个打分函数

如果 `s(b,c)` 超过阈值 $\tau$，就保留这个实体候选。

玩具例子：

`北京大学人民医院`

假设候选打分如下：

| 候选 span | 类型 | 分数 |
|---|---|---|
| `(1,4)` | ORG | 2.4 |
| `(1,8)` | HOSP | 3.1 |
| `(5,8)` | ORG | -0.6 |

若阈值 $\tau=0$，前两个保留，第三个丢弃。这样就能自然表示嵌套。

span 方法的本质优势是：输出对象从“位置标签”变成“边界区间”，因此不会被单层标签空间卡死。

### 5. pointer network：单独预测边界

pointer，指针机制，白话解释是“模型不直接输出整段文本，而是预测起点和终点在哪”。它常用于把实体抽取拆成两个边界预测问题，或者把多类实体转成起止位置选择问题。

这类方法在边界更灵活的场景中有优势，尤其是当实体长度变化大、嵌套复杂时，可以避免固定标签体系过于僵硬。但工程上它通常比纯序列标注更难调，阈值、去重、冲突消解都要设计。

### 6. layered tagging：分层标注

layered，分层，意思是“每一层只预测一组不重叠实体，预测完再继续下一层”。它试图把嵌套问题拆成多个平面问题。

例如：

- 第一层抽：`北京大学`
- 第二层抽：`北京大学人民医院`

这种方法保留了序列标注的很多优点，但实现上要处理层数选择、误差传播和层间依赖。内层如果错了，外层经常也会被带偏。

### 7. 机制对比

| 方法 | 输出表示 | 能否处理嵌套 | 优点 | 主要问题 |
|---|---|---|---|---|
| BIO/BIES | 每位置一个标签 | 否 | 简单、高效 | 不能表示重叠或嵌套 |
| BiLSTM-CRF | 带转移约束的标签序列 | 否 | 标签合法性更强 | 仍是单层输出 |
| Span Classification | 区间 + 类型 | 是 | 表达能力强 | 候选数多，通常是 $O(n^2)$ |
| Pointer Network | 起点/终点指针 | 是 | 边界灵活 | 解码与冲突消解更复杂 |
| Layered Tagging | 多层平面标注 | 是 | 可复用序列标注组件 | 层间误差传播明显 |

真实工程里，现代方案往往不是“只用模型”。更常见的组合是：

`词典召回 + 预训练语言模型编码 + span 或 pointer 解码 + 共指辅助`

这里“共指”指“文档中多个提及是否指向同一对象”。它本身不是 NER，但在长文档场景下能减少重复切分错误，让同一个实体在多处出现时更一致。

## 代码实现

下面先给一个最小可运行的平面 NER 解码例子。它不训练模型，只演示“如何从 BIO 标签恢复实体”。这一步非常重要，因为很多工程问题就出在解码规则不严格。

```python
from typing import List, Tuple

def decode_bio(chars: List[str], tags: List[str]) -> List[Tuple[str, str, int, int]]:
    assert len(chars) == len(tags)
    entities = []
    start = None
    ent_type = None

    for i, tag in enumerate(tags):
        if tag == "O":
            if start is not None:
                entities.append(("".join(chars[start:i]), ent_type, start, i - 1))
                start = None
                ent_type = None
            continue

        prefix, cur_type = tag.split("-", 1)

        if prefix == "B":
            if start is not None:
                entities.append(("".join(chars[start:i]), ent_type, start, i - 1))
            start = i
            ent_type = cur_type

        elif prefix == "I":
            if start is None or ent_type != cur_type:
                # 非法 I，按新实体开头纠正
                if start is not None:
                    entities.append(("".join(chars[start:i]), ent_type, start, i - 1))
                start = i
                ent_type = cur_type
        else:
            raise ValueError(f"Unsupported tag: {tag}")

    if start is not None:
        entities.append(("".join(chars[start:]), ent_type, start, len(chars) - 1))

    return entities


chars = list("张三在清华大学读书")
tags = ["B-PER", "I-PER", "O", "B-ORG", "I-ORG", "I-ORG", "I-ORG", "O", "O"]

result = decode_bio(chars, tags)
assert result == [("张三", "PER", 0, 1), ("清华大学", "ORG", 3, 6)]
print(result)
```

这个玩具例子说明了平面 NER 的基本链路：

1. 文本按字或词切分
2. 编码器为每个位置输出标签分数
3. 解码器得到一条 BIO 标签序列
4. 根据标签恢复实体边界和类型

如果是 BiLSTM-CRF，那么第 2 步到第 3 步之间多了一层“全局最优路径搜索”。

再看嵌套实体的最小实现思路。这里用 span 解码演示“一个文本里保留多个相互包含的实体”。

```python
from typing import List, Tuple

def decode_spans(
    text: str,
    candidates: List[Tuple[int, int, str, float]],
    threshold: float = 0.0
) -> List[Tuple[str, str, int, int, float]]:
    kept = []
    for start, end, ent_type, score in candidates:
        assert 0 <= start <= end < len(text)
        if score > threshold:
            kept.append((text[start:end + 1], ent_type, start, end, score))

    kept.sort(key=lambda x: (x[2], x[3], -x[4]))
    return kept


text = "北京大学人民医院"
candidates = [
    (0, 3, "ORG", 2.6),   # 北京大学
    (0, 7, "HOSP", 3.1),  # 北京大学人民医院
    (4, 7, "ORG", -0.4),  # 人民医院，假设分数不够
]

result = decode_spans(text, candidates, threshold=0.0)
assert result == [
    ("北京大学", "ORG", 0, 3, 2.6),
    ("北京大学人民医院", "HOSP", 0, 7, 3.1),
]
print(result)
```

这个例子展示了 span 方法的核心差异：输出不再是一条标签链，而是一组候选区间。只要候选生成和分类器足够合理，就能保留嵌套结构。

下面给一个更完整的实现框架对比。

| 维度 | 序列标注 | Span 分类 |
|---|---|---|
| 输入 | 字/词序列 | 字/词序列 |
| 标签空间 | BIO/BIES | `(start, end, type)` |
| 解码对象 | 单条标签路径 | 候选区间集合 |
| 是否天然支持嵌套 | 否 | 是 |
| 复杂度 | 通常接近线性或线性乘标签数 | 候选通常是 $O(n^2)$ |

真实工程例子：医疗病历结构化。

文本片段：

`患者于上海交通大学医学院附属瑞金医院行冠脉造影检查。该院随后开具阿托伐他汀。`

一个可用系统通常不是“单模型端到端”就结束，而是下面这条链路：

1. 词典模块先召回医院名、药品名、疾病名候选
2. 预训练模型对上下文编码，解决“同形异义”和边界歧义
3. span 分类器决定哪些区间是真实体
4. 文档级共指模块判断“该院”是否指向前文的“上海交通大学医学院附属瑞金医院”
5. 最终再送到实体链接模块，对接标准医疗知识库

这个例子说明，工程里的 NER 更像一个“系统组件”，而不是一段独立的分类代码。

## 工程权衡与常见坑

方法选择本质上是表达能力、速度、标注成本、上线稳定性之间的权衡。

### 1. 主要权衡

| 方案 | 优势 | 成本 | 适合场景 |
|---|---|---|---|
| BIO/BIES | 简单、快、成熟 | 无法表示嵌套 | 新闻、人名地名机构名抽取 |
| CRF 增强序列标注 | 边界更稳、非法标签更少 | 仍不能处理嵌套 | 通用平面 NER |
| Span 分类 | 嵌套表达自然 | 候选多、算力高 | 医疗、金融、法律 |
| Pointer | 边界灵活 | 解码复杂 | 长实体、复杂边界 |
| 分层标注 | 可复用平面组件 | 误差逐层传播 | 嵌套较浅的场景 |

span 方法最典型的代价是候选数增长。长度为 $n$ 的序列，候选区间数量大约是：

$$
\frac{n(n+1)}{2}
$$

也就是 $O(n^2)$。文本一长，候选会迅速增多。因此工程里几乎都会做长度剪枝，比如只保留长度不超过 10 或 20 的 span，或者先用词典和边界打分过滤一遍。

### 2. 常见坑

| 问题 | 后果 | 规避策略 |
|---|---|---|
| 标注规范不一致 | 模型学不到稳定边界 | 先定义“最长匹配还是最小可用片段” |
| 只看 token-F1 | 边界错半个字也可能被忽略 | 同时看 strict span+type F1 |
| 嵌套场景硬用 BIO | 内层和外层实体互相冲突 | 改成 span、pointer 或 layered |
| 负样本过多 | 模型倾向于全预测成非实体 | 重采样、难例挖掘、阈值校准 |
| 词典直接当真值 | 误报多，类型污染严重 | 词典只做候选，不直接当最终答案 |
| 长文本不剪枝 | 训练和推理明显变慢 | 最大长度约束、候选过滤 |
| 切分错误传到下游 | 实体链接、关系抽取一起变差 | 把边界质量作为一级指标监控 |

这里要特别解释两个评测概念。

- `token-F1`：按字或词位置计算标签对不对
- `strict span+type F1`：实体的起点、终点、类型必须全部正确才算命中

工程上更应该看第二个。因为知识图谱和实体链接关注的是“这个实体片段是否可直接入库”，不是“标签大体差不多”。

举个简单例子：

- 真值：`北京大学人民医院 -> HOSP`
- 预测：`北京大学 -> ORG`

从某些位置标签视角看，前半段可能“部分对了”；但对实体链接系统来说，这就是完全错误的节点。

### 3. 错误传播问题

NER 是入口任务，所以入口错误会传染。尤其在知识图谱场景里，错误通常按下面路径扩散：

`边界错 -> 实体链接错 -> 关系挂错节点 -> 图谱污染`

这也是为什么很多团队上线时宁可先保守一点，降低误报，再逐步补召回。因为错实体入库的清理成本，往往高于漏掉一部分实体。

---

## 替代方案与适用边界

没有一种 NER 方法适合所有场景。更实用的思路是按文本结构和业务目标选方法。

### 1. 场景与推荐方法

| 场景 | 推荐方法 | 原因 |
|---|---|---|
| 新闻标题、人名地名机构名 | BIO/BIES + CRF | 平面实体为主，速度快，维护简单 |
| 客服工单、短文本搜索召回 | 词典 + 序列标注 | 业务词表稳定，词典能补召回 |
| 医疗病历、药品说明书 | Span 分类 + 词典 + 共指 | 嵌套多、长实体多、跨句别名多 |
| 法律条文、合同文本 | Span 或 Pointer | 实体长、边界复杂、重叠较常见 |
| 金融公告、研报 | Span + 文档级建模 | 机构名长，简称与全称反复出现 |
| 科研论文 | Layered 或 Span | 术语嵌套明显，句子长 |

### 2. 一个简单决策流程

可以按下面的顺序判断：

1. 如果实体基本不嵌套，优先试序列标注。
2. 如果实体经常“实体里套实体”，优先试 span 或 layered。
3. 如果边界很长且变化大，考虑 pointer。
4. 如果同一实体跨句重复出现，并且别名很多，加入共指辅助。
5. 如果业务词表稳定且漏召回成本高，加入词典匹配，但只作为候选来源。

### 3. 替代方案不是“更先进”，而是“更适配”

很多人会把“预训练大模型 + 复杂解码”理解成一定更好，这不严谨。比如新闻标题抽取中，`BIO + CRF` 往往已经足够，继续上复杂结构，可能只增加延迟和维护成本。

相反，在医疗、法律、金融这些嵌套密集场景里，如果还坚持单层 BIO，成本并不是“模型简单”，而是“下游反复修错”。

所以适用边界可以总结为：

- 平面实体为主：序列标注优先
- 嵌套实体明显：span、pointer、layered 优先
- 文档级一致性重要：加共指和别名归并
- 高召回要求：加词典，但交给上下文模型做最终判定

最终目标不是追求某个论文结构，而是让“实体边界正确、类型稳定、下游可用”。
