## 核心结论

RetNet（Retentive Network）要解决的不是“能不能建模上下文”，而是“能不能像 Transformer 一样并行训练，同时在推理时不再背着越来越大的 KV cache”。它的核心做法是用 **retention** 取代标准 attention。这里的 retention 可以先白话理解为：**把历史信息按距离做指数衰减后，再累积成一个可递推的状态**。

对初学者，最重要的结论有三条：

1. 训练时，RetNet 仍然能像 Transformer 一样并行处理整段序列，适合 GPU。
2. 推理时，RetNet 不需要保存所有历史 token 的 key/value，只维护一个递归状态，因此缓存复杂度接近 $O(1)$。
3. 它提供三种等价或近等价的工作方式：并行、循环、分块循环。前两者分别对应“训练友好”和“部署友好”，第三种是工程折中。

下面这张表先给出全局判断。

| 维度 | Transformer + KV Cache | RetNet |
|---|---|---|
| 训练并行性 | 强 | 强 |
| 自回归推理缓存 | 随序列长度线性增长 | 只存递归状态，基本与序列长度无关 |
| 长上下文推理显存 | 高 | 低 |
| 长序列单步延迟 | 随上下文增长而恶化 | 更稳定 |
| 实现成熟度 | 很高 | 较新，工程细节更多 |
| 适合场景 | 短序列、离线批处理 | 实时生成、长上下文、显存敏感服务 |

一个新手版本的直觉是：Transformer 在解码时要把前面所有 token 的 KV 都留下来；RetNet 则把“历史贡献”压缩进一个状态里，下一步只更新这个状态，不再回看整段历史。

---

## 问题定义与边界

问题的边界很明确：讨论的是 **自回归生成**，也就是模型按时间顺序一个 token 一个 token 地往后写。这里最痛的工程瓶颈不是训练，而是部署。

标准 Transformer 在第 $t$ 步生成时，需要把前面 $1 \sim t-1$ 步的 key 和 value 都缓存起来。若层数为 $L$、头数为 $H$、每头维度为 $d$、序列长度为 $T$，那么 KV cache 的规模近似随 $T$ 线性增长：

$$
\text{KV memory} \propto L \cdot T \cdot H \cdot d
$$

这意味着两件事：

1. 上下文越长，显存越高。
2. 单步生成虽然不是重新算整段，但仍要与越来越长的历史 KV 交互，延迟会持续上升。

RetNet 的目标不是完全否定 attention，而是把 attention 中“历史 token 对当前 token 的影响”改写成一种可递推的形式。这个形式必须满足三个约束：

| 约束 | 含义 |
|---|---|
| 因果性 | 当前位置只能看见过去，不能偷看未来 |
| 可并行训练 | 不能为了低延迟推理，牺牲掉训练时的 GPU 并行效率 |
| 可递归推理 | 生成下一个 token 时，只更新一个小状态，而不是保留整段历史 |

这里的“只存一份状态”不是说整个模型没有层和头，而是说 **每层每头维护固定形状的状态，而不是长度为 $T$ 的历史列表**。因此它解决的是缓存增长问题，不是把所有计算都变成常数。

一个真实工程例子：做实时客服对话，用户会连续输入很多轮消息。若使用标准 KV cache Transformer，单个会话越聊越长，显存占用也越大；若服务同时承载很多会话，显存很快被上下文拖满。RetNet 的设计目标就是把这类“会话越长越贵”的问题压平。

## 核心机制与推导

RetNet 的核心公式通常写成并行形式：

$$
Y = (QK^{*} \circ D)V
$$

符号逐个解释：

- $Q,K,V$：和 attention 一样，分别是 query、key、value 投影。
- $\circ$：逐元素乘法。
- $D$：距离衰减矩阵，白话说就是“离当前越远，权重越小”。
- $K^*$：带共轭的 key，和 xPos 一类相对位置处理有关，用来保留位置关系。

如果只看结构，这个式子像 attention，但和 softmax attention 有两个关键区别：

1. 权重来自可分解的衰减结构，而不是完整的 softmax 归一化。
2. 这个结构可以重写成递归式，因此能在推理时只维护状态。

递归形式写成：

$$
h_t = \gamma h_{t-1} + k_t v_t^\top
$$

$$
o_t = q_t^\top h_t
$$

这里：

- $\gamma$ 是 decay，白话说就是“历史记忆保留多少”。
- $h_t$ 是状态，记录到第 $t$ 步为止的历史压缩结果。
- $k_t v_t^\top$ 是当前 token 对状态的新贡献。
- $o_t$ 是当前位置输出。

这两个式子很关键，因为它们解释了为什么 RetNet 可以不存整段 KV。历史不再以列表形式存在，而是被压缩进 $h_t$。

### 玩具例子

设只有一个历史 token，且：

- $q = 0.5$
- $k = 1$
- $v = 2$
- 衰减 $D = 0.9$

并行视角下，先算 retention score：

$$
0.5 \times 1 \times 0.9 = 0.45
$$

再乘 $v$：

$$
0.45 \times 2 = 0.9
$$

递归视角下，若前一状态 $h_{t-1}=0$，则先更新：

$$
h_t = 0.9 \cdot 0 + 1 \cdot 2 = 2
$$

再与当前 query 结合。若把当前距离衰减已经吸收到状态或 query/key 的位置处理中，最终输出同样对应到 $0.9$ 这一结果。这个例子要表达的不是所有细节都能简化成标量，而是：**并行算和递归算可以对齐到同一机制**。

### 并行、循环、分块三种模式

| 模式 | 计算方式 | 适用阶段 | 关键价值 |
|---|---|---|---|
| 并行 | 一次性算整段 $Q,K,V,D$ | 训练 | GPU 利用率高 |
| 循环 | 逐步更新 $h_t$ | 在线推理 | 低延迟、低缓存 |
| 分块循环 | 块内并行，块间递归 | 长序列训练/推理 | 在吞吐和状态管理间折中 |

分块模式可以理解为：把序列切成若干 chunk。chunk 内部仍然并行计算，chunk 之间通过递归状态传递。这比纯循环更适合 GPU，也比纯并行更接近真实部署形态。

一个真实工程例子是 8k 上下文推理。若完全采用标准 KV cache，历史越长，缓存越重；若完全逐 token 循环，GPU 吞吐又可能吃亏。分块模式允许系统一次处理一段，比如 128 或 256 个 token，同时把跨块记忆压缩进状态，适合长文档生成或长会话服务。

## 代码实现

下面先给一个最小可运行的 Python 玩具实现，只演示单头、标量版本的递归 retention。它不是论文中的完整实现，但足够帮助理解“状态递推”和“结果可核对”。

```python
def recurrent_retention(qs, ks, vs, decay):
    state = 0.0
    outputs = []
    for q, k, v in zip(qs, ks, vs):
        state = decay * state + k * v
        outputs.append(q * state)
    return outputs

# 玩具例子
qs = [0.5]
ks = [1.0]
vs = [2.0]
outs = recurrent_retention(qs, ks, vs, decay=0.9)
assert len(outs) == 1
assert abs(outs[0] - 1.0) < 1e-9  # 这里是最简递归版，不含额外距离项

# 两步序列，检查状态递推是否正确
qs = [1.0, 1.0]
ks = [2.0, 3.0]
vs = [4.0, 5.0]
outs = recurrent_retention(qs, ks, vs, decay=0.5)

# 第一步: state = 0.5*0 + 2*4 = 8
assert abs(outs[0] - 8.0) < 1e-9

# 第二步: state = 0.5*8 + 3*5 = 19
assert abs(outs[1] - 19.0) < 1e-9
```

上面代码里的 `state = decay * state + k * v` 就是最核心的一行。白话解释如下：

- `decay * state`：旧记忆按比例保留。
- `k * v`：当前 token 把新的信息写入状态。
- `q * state`：当前 query 从累积状态里读出自己需要的信息。

如果写成更接近工程实现的伪代码，会是下面这样：

```python
def retention_parallel(Q, K, V, D):
    # Q: [T, d], K: [T, d], V: [T, dv], D: [T, T]
    scores = (Q @ K.T) * D
    Y = scores @ V
    return Y

def retention_recurrent(q_t, k_t, v_t, state, decay):
    # state: [d, dv]
    state = decay * state + outer(k_t, v_t)
    y_t = q_t @ state
    return y_t, state
```

真实模型会比这个复杂得多，因为还要处理：

| 组件 | 作用 |
|---|---|
| multi-scale decay | 不同 head 使用不同衰减速度，覆盖短期到长期依赖 |
| gate | 控制 retention 输出与其他分支的融合强度 |
| normalization | 稳定不同 head、不同时间步的数值范围 |
| residual connection | 保持深层网络可训练 |
| xPos/相对位置处理 | 让距离信息进入权重，而不破坏递归表达 |

可以把多头理解成“多种时间尺度的记忆通道”。有的 head 衰减快，擅长近距离局部模式；有的 head 衰减慢，擅长保留长期依赖。若所有 head 都用同一个 decay，模型会缺少尺度分工。

## 工程权衡与常见坑

RetNet 的优点很明确，但工程上并不是“把 attention 替换掉”就结束了。它真正难的地方在稳定性。

### 常见坑与修复

| 常见坑 | 现象 | 修复思路 |
|---|---|---|
| 所有 head 共用一个 decay | 模型只能记一种时间尺度，效果差且不稳 | 按 head 设置多尺度 decay |
| 忽略 gate | 某些层输出过强或过弱 | 给 retention 分支加门控控制幅度 |
| 少做归一化 | 不同 head 方差飘移，训练震荡 | 在残差前后加 GroupNorm/合适归一化 |
| 分块模式只管块内，不管跨块 decay | 长距离信息累计失真 | 分开处理 intra-chunk 与 cross-chunk 衰减 |
| 直接照搬 attention 超参 | 学习率、初始化、残差比例不匹配 | 单独调 decay、norm、gate 相关超参 |

这里最容易被初学者忽略的是：RetNet 虽然看起来像“更省缓存的 attention”，但它本质上更像 **带结构约束的状态空间递推**。状态一旦设计不好，就会出现两类典型问题：

1. **记不住**：decay 太小，历史信息很快消失。
2. **压不住**：decay 太大、归一化不足，状态方差不断累积，训练发散。

从公式上看，递归状态是：

$$
h_t = \gamma h_{t-1} + k_t v_t^\top
$$

若 $\gamma$ 接近 1，而 $k_t v_t^\top$ 的尺度又没有被控制，$h_t$ 的数值就可能持续变大。这就是为什么论文和后续实现都强调多尺度衰减、归一化和门控。

一个真实工程例子：做长对话客服或代码补全时，系统往往要求既能处理近期局部依赖，又要保留很早之前的角色设定、工具调用状态或变量上下文。工程上常见做法不是让所有 head 都“尽量长记忆”，而是故意把 decay 做成分层次的。有的 head 快速遗忘，保证局部模式敏感；有的 head 慢速遗忘，保留远距离依赖。这样整体更稳定。

另一个要点是分块模式。分块不是简单把序列截断，而是要同时保证：

- chunk 内部的并行计算正确。
- chunk 之间的递归状态连续。
- 跨 chunk 的 decay 与块内 decay 一致或可控。

如果只在块内算 retention，而跨块状态没有正确衔接，模型会在块边界出现“记忆断层”。

## 替代方案与适用边界

RetNet 不是所有场景下都优于 Transformer。更准确的说法是：它在“长上下文、低延迟、显存敏感”的部署条件下很有吸引力，但在短上下文或成熟生态要求高的场景下，传统方案仍然更省心。

| 方案 | 缓存方式 | 位置建模 | 训练/推理特征 | 适用场景 |
|---|---|---|---|---|
| RetNet | 递归状态 | 明确衰减与相对位置机制 | 训练并行，推理低缓存 | 实时生成、长上下文服务 |
| Transformer + KV Cache | 保存全部历史 KV | 绝对/相对位置都成熟 | 实现成熟，推理缓存大 | 短序列、离线批处理 |
| Performer 等 linear attention | 近似 attention，无完整 KV 历史 | 依实现而定 | 强调线性复杂度 | 关注近似加速、可接受近似误差的场景 |

这里顺带解释一下 **linear attention**：白话说，就是通过数学重写或核技巧，把原本二次复杂度的 attention 变成近似线性复杂度。RetNet 和它们有相似目标，但路径不同。RetNet 更强调三种模式的一致性，以及明确的递归状态表达。

### 什么时候不该优先用 RetNet

1. 输入很短，比如 256 到 512 token。
2. 主要是离线批量推理，不在乎单请求实时延迟。
3. 团队已经高度依赖成熟 Transformer 推理栈，迁移成本高。
4. 任务对 attention 可解释性、已有工具链兼容性要求很强。

### 什么时候该认真考虑 RetNet

1. 在线对话系统，单会话上下文不断增长。
2. 长文档生成、长代码补全、长链式推理。
3. GPU 显存紧张，但又不能牺牲太多模型容量。
4. 系统更关注 token-by-token 延迟，而不是只看离线吞吐。

一个新手判断法则很简单：如果你只有 512 token 输入，而且是离线批处理，传统 Transformer 更现实；如果你在做实时客服、长文档助手、8k 甚至更长上下文推理，RetNet 的价值就会明显上升。
