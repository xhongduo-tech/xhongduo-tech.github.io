## 核心结论

RLHF 中的奖励模型本质上是一个打分器。打分器的意思是：给定用户输入 $x$ 和模型回答 $y$，输出一个标量分数 $R(x,y)\in\mathbb{R}$，这个分数用来估计“人类会不会更喜欢这个回答”。

它不是直接学习“标准答案”，而是学习“相对偏好”。相对偏好的意思是：对同一个 prompt，人类不一定能写出唯一正确答案，但通常能判断回答 A 和回答 B 哪个更好。于是训练数据不是单条样本，而是成对样本 $(x,y_w,y_l)$，其中 $y_w$ 是人类更喜欢的回答，$y_l$ 是较差回答。

RLHF 第二阶段训练奖励模型的核心目标可以写成：

$$
\mathcal{L} = -\mathbb{E}\left[\log \sigma\left(R(x,y_w)-R(x,y_l)\right)\right]
$$

这里的 $\sigma$ 是 sigmoid 函数，也就是把任意实数压到 $(0,1)$ 区间的函数，用来表示“偏好概率”。

直观上，这个损失函数只关心一件事：让偏好回答的分数高于次优回答，而且差得越明显越好。这样做的价值是，后续策略优化阶段不再需要每次都找人打分，而是把奖励模型当作一个近似的人类评委。

一个玩具例子可以先建立直觉：

| prompt | 回答 A | 回答 B | 人类偏好 | 奖励模型希望学到什么 |
|---|---|---|---|---|
| “解释什么是缓存” | “缓存是临时存储，用于减少重复计算或重复访问慢资源。” | “缓存就是电脑更快，因为很高级。” | A 优于 B | 给 A 更高分，给 B 更低分 |

如果这种比较在大量 prompt 上重复出现，奖励模型就会逐步学到“清晰、准确、完整”对应更高分，“空泛、含糊、跑题”对应更低分。

## 问题定义与边界

先把对象定义清楚。

奖励模型定义为：

$$
R: \mathcal{X}\times\mathcal{Y}\rightarrow \mathbb{R}
$$

其中：

- $\mathcal{X}$ 是 prompt 空间，也就是用户可能提出的问题集合。
- $\mathcal{Y}$ 是 response 空间，也就是模型可能生成的回答集合。
- 输出是实数分数，数值越大代表越可能被人类偏好。

这里要注意，奖励模型不是“真实性证明器”，也不是“最终质量的绝对标准”。它只是对训练分布上的人类偏好做拟合。拟合的意思是：用有限数据逼近一个真实但未知的评价函数。

边界主要有四个。

| 维度 | 奖励模型负责什么 | 不负责什么 |
|---|---|---|
| 监督信号 | 学习人类偏好比较 | 学习唯一标准答案 |
| 输出形式 | 标量分数 | 可解释的逐项评分报告 |
| 数据依赖 | 依赖偏好对数据覆盖 | 不能自动发现未标注价值观 |
| 泛化范围 | 对相似任务可泛化 | 对完全新分布不可靠 |

一个新手常见误解是：只要训练了奖励模型，它就知道“什么是好回答”。这不准确。更准确的说法是：它知道“在给定标注规则和数据分布下，什么样的回答更可能被标注员选中”。

再看一个具体例子。假设 prompt 是“写一段两句的 Redis 简介”，我们收集到两条回答：

- $y_1$：“Redis 是内存数据库，支持多种数据结构，常用于缓存和计数场景。”
- $y_2$：“Redis 很快，在互联网里很火，很多公司都在用。”

如果标注员更偏好 $y_1$，奖励模型不会学到“Redis 的全部知识”，而是学到在这个任务里，准确、具体、贴合要求的表达更容易赢过空泛表达。

因此，奖励模型的有效边界有两个前提：

1. 偏好标签本身必须相对稳定。比如“更准确”“更有帮助”通常比“更有创意”更容易标。
2. prompt 分布要足够广。否则模型可能只学会某种文风，而不是任务本质。

## 核心机制与推导

奖励模型常用 Bradley-Terry 模型来建模偏好概率。这个模型的直觉非常直接：两个回答谁更可能被选中，取决于它们分数差。

定义：

$$
P(y_w \succ y_l \mid x)=\sigma\left(R(x,y_w)-R(x,y_l)\right)
$$

意思是：在同一个 prompt $x$ 下，回答 $y_w$ 胜过 $y_l$ 的概率，由两者分数差经过 sigmoid 后得到。

为什么是“差值”而不是各自绝对值？因为偏好数据只告诉我们“谁更好”，没有告诉我们“绝对好到几分”。所以排序关系比绝对刻度更稳定。

把对数似然写出来：

$$
\log P(y_w \succ y_l \mid x)=\log \sigma\left(R(x,y_w)-R(x,y_l)\right)
$$

训练时我们希望最大化这个概率，也就是最大化对数似然；等价地，最小化负对数似然：

$$
\mathcal{L} = -\log \sigma\left(R(x,y_w)-R(x,y_l)\right)
$$

对整个数据集求期望，就得到前面的总体损失。

这个式子为什么有效，可以从梯度方向理解。设

$$
\Delta = R(x,y_w)-R(x,y_l)
$$

则损失是：

$$
\mathcal{L}=-\log \sigma(\Delta)
$$

当 $\Delta$ 变大时，$\sigma(\Delta)$ 更接近 1，损失变小。于是优化器会推动两件事同时发生：

- 提高偏好回答 $R(x,y_w)$
- 压低次优回答 $R(x,y_l)$

这就是“偏好对比学习”的核心。

看一个数值玩具例子：

| $R(x,y_w)$ | $R(x,y_l)$ | 差值 $\Delta$ | $\sigma(\Delta)$ | 损失 $-\log \sigma(\Delta)$ |
|---|---:|---:|---:|---:|
| 1.0 | 0.5 | 0.5 | 0.622 | 0.474 |
| 1.4 | 0.2 | 1.2 | 0.769 | 0.262 |
| 2.0 | 0.0 | 2.0 | 0.881 | 0.127 |

从表里能看到，差值从 0.5 增加到 1.2 后，模型对“好回答应当获胜”的信心更强，损失显著下降。

这里还有一个工程上重要但常被忽略的点：奖励模型通常不是从零训练，而是从预训练语言模型初始化。初始化的意思是：先继承语言模型已经学到的语义表示能力，再把最后输出头换成标量回归头。这样做的原因很现实，因为“判断两个回答哪个好”本质上仍然需要理解语言内容、事实关系、指令约束和上下文一致性。

真实工程例子是 RLHF 第三阶段。比如用 PPO 做策略优化时，策略模型生成回答 $y$，然后把 $(x,y)$ 送入奖励模型得到 reward，PPO 再根据这个 reward 更新策略。此时奖励模型等价于一个可批量调用、可微分之外独立运行的人类偏好近似器。

## 代码实现

一个最小实现可以抽象成三步：

1. 用预训练编码器把 `(prompt, response)` 编码成隐藏状态。
2. 用一个线性层把隐藏状态映射成标量分数。
3. 对偏好对计算 Bradley-Terry loss。

数据流可以写成：

```text
偏好对 (x, y_w, y_l)
    -> 编码器
    -> 两个标量分数 R(x,y_w), R(x,y_l)
    -> diff = R_w - R_l
    -> loss = -log(sigmoid(diff))
    -> 反向传播更新参数
```

下面是一个可运行的简化 `python` 例子。它没有依赖深度学习框架，只演示损失计算和训练目标的方向。

```python
import math

def sigmoid(x: float) -> float:
    return 1.0 / (1.0 + math.exp(-x))

def bt_loss(r_w: float, r_l: float) -> float:
    p = sigmoid(r_w - r_l)
    return -math.log(p)

# 玩具样本：偏好回答分数应该更高
r_w_1, r_l_1 = 1.0, 0.5
loss_1 = bt_loss(r_w_1, r_l_1)

# 假设一次更新后，偏好回答更高、次优回答更低
r_w_2, r_l_2 = 1.4, 0.2
loss_2 = bt_loss(r_w_2, r_l_2)

assert round(sigmoid(0.5), 3) == 0.622
assert round(loss_1, 3) == 0.474
assert round(sigmoid(1.2), 3) == 0.769
assert round(loss_2, 3) == 0.262
assert loss_2 < loss_1

print("loss_1 =", round(loss_1, 3))
print("loss_2 =", round(loss_2, 3))
```

如果换成神经网络形式，结构通常接近下面这样：

```python
class RewardModel:
    def __init__(self, backbone, hidden_size):
        self.backbone = backbone          # 预训练语言模型
        self.value_head = Linear(hidden_size, 1)  # 标量输出头

    def forward(self, input_ids, attention_mask):
        h = self.backbone(input_ids, attention_mask)
        last_hidden = h[:, -1, :]        # 这里只是示意，真实实现常用更稳定的聚合方式
        reward = self.value_head(last_hidden)
        return reward.squeeze(-1)

def reward_loss(model, batch):
    r_w = model(batch["chosen_ids"], batch["chosen_mask"])
    r_l = model(batch["rejected_ids"], batch["rejected_mask"])
    diff = r_w - r_l
    loss = -log_sigmoid(diff).mean()
    return loss
```

这里有两个实现细节需要明确。

第一，输入不是只喂 response，而是拼接后的 `(prompt, response)`。原因是同一句回答在不同 prompt 下质量可能完全不同。

第二，很多实现不只训练最后一层，也会微调整个 backbone，或者只微调上层若干层。只训练最后一层更省算力，但表达能力可能不足；全量微调效果更强，但更容易过拟合和引入不稳定。

一个真实工程例子是客服问答优化。假设同一个用户问题“退款流程怎么走”，系统生成两个候选：

- 候选 A：礼貌、准确、分步骤说明退款入口和时限
- 候选 B：礼貌但缺少关键限制条件，甚至遗漏自动退款失败后的人工通道

标注员会稳定偏好 A。奖励模型学到这一模式后，后续 PPO 或其他策略优化阶段就能偏向生成“完整且可执行”的回答，而不是只追求语气友好。

## 工程权衡与常见坑

奖励模型最关键的风险不是“训练不起来”，而是“训练起来了，但学歪了”。

最典型的问题是奖励黑客。奖励黑客的意思是：策略模型不是在提升真实质量，而是在利用奖励模型的漏洞骗高分。

例如，如果训练数据里“礼貌措辞”与“高质量回答”高度相关，策略模型可能学会反复输出“感谢你的提问”“希望这对你有帮助”等高频模式，即使内容已经跑题，奖励模型仍给出不错分数。这说明奖励模型学到的是表面相关特征，而不是任务目标本身。

常见问题可以总结如下：

| 问题 | 典型表现 | 缓解措施 |
|---|---|---|
| 奖励黑客 | 输出模式化高分词，但内容变差 | 加强人工抽检，加入对抗样本，周期性重训 |
| 覆盖不足 | 在少数 prompt 上分数高，换任务就失效 | 扩展任务分布，按场景分层采样 |
| 长度偏置 | 总给更长或更短回答更高分 | 做长度归一化，加入长度平衡样本 |
| 风格过拟合 | 偏爱某种语气、格式、模板 | 增加多风格标注数据 |
| 标注漂移 | 不同标注员标准不一致 | 明确 rubric，做重叠标注和一致性检查 |

还可以引入额外正则项来约束策略模型，避免它过度朝奖励模型漏洞收缩。常见写法包括熵正则。熵的意思是：鼓励输出分布保持一定不确定性，避免过早塌缩到单一模式。

$$
J(\pi)=\mathbb{E}_{y\sim \pi(\cdot|x)}[R(x,y)] + \beta H(\pi(\cdot|x))
$$

其中 $H(\pi)$ 是策略熵，$\beta$ 控制探索强度。

在很多 RLHF 实现里，还会加入 KL 惩罚。KL 的意思是：约束新策略不要偏离参考模型太远，防止为了追奖励分数而语言风格崩坏。

$$
J(\pi)=\mathbb{E}[R(x,y)] - \lambda \, \mathrm{KL}(\pi \,\|\, \pi_{\text{ref}})
$$

对初级工程师来说，最重要的判断标准是：如果奖励模型上线后，自动指标升高但人工盲测下降，优先怀疑奖励黑客，而不是先相信训练日志。

还有一个现实问题是数据成本。偏好对比标注看似比写标准答案轻，但当任务非常专业时，标注员仍需要足够背景知识。例如代码审查、法律问答、医疗建议都很难靠通用标注员稳定判断。

## 替代方案与适用边界

奖励模型不是唯一方案。它适合“质量主观、规则难写、规模较大”的任务，但不适合所有场景。

先做一个对比：

| 方案 | 优点 | 缺点 | 更适合的场景 |
|---|---|---|---|
| 奖励模型 | 可规模化替代人工比较，适合复杂主观任务 | 需要偏好数据，存在奖励黑客 | 开放式对话、总结、写作辅助 |
| 直接人工评分 | 评价最直接，误差最可控 | 成本高，吞吐低，难在线训练 | 小规模高价值任务 |
| 规则评估 | 成本低、可解释、稳定 | 覆盖窄，难处理主观质量 | 格式检查、单元测试、关键词约束 |
| SFT 监督微调 | 实现简单，训练稳定 | 学的是模仿，不是偏好排序 | 有较高质量示范答案时 |

一个简单经验边界是：

- 如果任务可以写出明确规则，比如“JSON 是否合法”“代码是否通过测试”，优先规则评估。
- 如果数据量很小，比如偏好数据少于几千条，而且任务高价值，优先人工直接评分。
- 如果任务是开放式主观生成，候选答案很多、规则写不全，奖励模型才开始显示优势。

举两个对比例子。

玩具例子：让模型输出一个合法邮箱地址。这个任务不需要奖励模型，因为规则检查足够精确。正则表达式或解析器就能判断结果。

真实工程例子：让模型做产品客服多轮对话。这里“是否有帮助”“是否礼貌”“是否遗漏风险提示”很难完全规则化，直接每轮都找人评分成本又太高。这时奖励模型最有价值，因为它把大量人工相对偏好压缩成一个可自动调用的评分器。

因此，切换策略的关键不是“RLHF 是否先进”，而是“任务能否被稳定评价”。能稳定规则化，就优先规则；不能规则化但能稳定比较，奖励模型就是合理方案；连比较标准都不稳定，就要先解决标注规范，而不是急着训练模型。

1. Christiano et al., *Deep Reinforcement Learning from Human Preferences*（2017）  
   贡献：提出用人类偏好对训练奖励模型，再用奖励模型替代人工提供强化学习信号，是现代 RLHF 的核心起点。

2. Leike et al., *Scalable agent alignment via reward modeling*（2018）  
   贡献：系统讨论奖励建模在对齐中的作用，强调把复杂目标拆成可比较、可标注的监督信号。

3. Ouyang et al., *Training language models to follow instructions with human feedback*（2022）  
   贡献：把 SFT、奖励模型、PPO 串成可落地的语言模型对齐流程，展示了 RLHF 在指令跟随中的规模化效果。

4. Bradley and Terry, *Rank Analysis of Incomplete Block Designs*（1952）  
   贡献：给出基于成对比较的 Bradley-Terry 排序模型，奖励模型常用的偏好损失直接建立在这个统计模型上。
