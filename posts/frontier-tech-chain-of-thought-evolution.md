## 核心结论

思维链的进化，本质上不是“让模型写更多字”，而是把原本藏在参数里的隐式推理，变成可以观察、采样、筛选、回退的显式中间状态。

这条路线可以概括为三步：

| 阶段 | 触发方式 | 中间状态数量 | 控制手段 | 典型适用场景 |
| --- | --- | --- | --- | --- |
| Zero-shot CoT | 用一句提示触发，例如“step by step” | 1 条线性链 | 主要靠提示词 | 简单数学、基础逻辑 |
| Few-shot CoT | 给出若干带步骤的示例 | 1 条更稳定的链 | 靠示例约束格式与节奏 | 模板化推理、标准题型 |
| Tree of Thoughts | 每一步生成多个候选思路 | 多分支树结构 | 采样、评分、剪枝、回溯 | 规划、组合搜索、博弈 |
| Graph of Thoughts | 允许不同分支复用、合并、跳转 | 图结构 | 节点复用、跨路径连接、记忆共享 | 多轮复杂任务、长程依赖推理 |

对零基础读者，可以先这样理解：

- Zero-shot CoT 像是要求模型“把草稿纸写出来”。
- Few-shot CoT 像是先给它看几份标准解题过程，再让它照着写。
- Tree of Thoughts 不再只走一条路，而是每一步都先想几种可能，再保留更有希望的分支。
- Graph of Thoughts 进一步允许“以前想过的中间结论”被别的分支拿来复用，而不是从头再算。

玩具例子很直观。假设问题是“用 3、3、8、1 算出 24”。线性 CoT 可能先试 `3 × 8 = 24`，但马上卡住，因为还剩 `3` 和 `1` 没用上。Tree of Thoughts 会同时考虑“乘、加、减”等多种候选，再通过评分保留更可能通向目标的分支，最终更容易找到 `(3 + 1) × (3 + 3) = 24`。

所以，思维链的进化目标很明确：不是追求更长的答案，而是追求更可控的搜索过程。

## 问题定义与边界

这里讨论的“思维链”，指的是模型在求解过程中显式写出的中间步骤。白话解释是：模型不只给结果，还把它“此刻在想什么”写成若干可读的状态。

为了把问题说清楚，可以定义两个对象：

- $s_t$：第 $t$ 步的当前状态。白话解释是“当前已经做到哪一步了”。
- $c_i$：基于当前状态产生的第 $i$ 个候选思路。白话解释是“下一步可能怎么走”。

在这个框架下，思维链的核心问题不是“模型会不会说”，而是三件事：

1. 能否从 $s_t$ 生成有价值的候选 $c_i$。
2. 能否评价哪些 $c_i$ 值得继续扩展。
3. 能否在错误路径上及时回退，而不是一路错到底。

生成候选思路时，常见写法是：

$$
P(c_i \mid s_t) \propto \exp\left(\frac{\text{LLM\_logits}(s_t + c_i)}{\tau}\right)
$$

这里的 $\tau$ 是温度。白话解释是：温度越高，模型越敢尝试不那么确定的思路；温度越低，越偏向最稳妥的那个选项。

有了候选之后，还需要一个评分器：

$$
E(s_t, c_i)
$$

它表示“从当前状态走这一步，值不值得”。评分器可以是另一个模型，也可以是规则、约束检查器、程序执行结果，或者几者组合。

为什么从 CoT 会继续进化到树和图？因为线性链默认只走一条路径，复杂任务里这往往不够。若每一步平均产生 $b$ 个候选，搜索深度为 $d$，理论搜索空间约为：

$$
O(b^d)
$$

这说明两件事：

- 优点：模型终于可以像搜索算法一样探索多种可能。
- 代价：分支一多，token 成本和错误面也会迅速膨胀。

所以它有明确边界。思维链进化主要适用于下面几类任务：

| 任务类型 | 是否适合扩展为树/图 | 原因 |
| --- | --- | --- |
| 简单事实问答 | 通常不适合 | 没有明显中间搜索空间 |
| 基础算术与短逻辑 | 适合 Zero-shot/Few-shot CoT | 一条链往往够用 |
| 数学证明、规划、组合搜索 | 很适合 ToT | 需要试错、回溯、比较多个分支 |
| 多轮复杂决策、上下文复用 | 很适合 GoT | 需要节点复用与跨路径连接 |

因此，本文的边界很明确：讨论的是“复杂推理如何从单一路径升级为可搜索结构”，不讨论单纯的知识召回，也不讨论所有提示词技巧。

## 核心机制与推导

Zero-shot CoT 的关键贡献，是第一次稳定地把“分步推理”显式触发出来。比如一句 “Let’s think step by step”，作用不是增加知识，而是迫使模型把本来压缩在隐藏表示里的推理过程展开成一串文本状态。

Few-shot CoT 再往前走一步。它不只要求“分步”，还通过示例告诉模型：什么样的中间状态是合理的。白话解释是，模型开始不仅学“要写步骤”，还学“步骤应该怎么写”。

真正的结构跃迁发生在 Tree of Thoughts。它把“下一步只有一个输出”改成“下一步可以有多个候选，再从中挑选”。

其基本循环可以写成：

1. 从当前状态 $s_t$ 采样 $b$ 个候选 $c_1,\dots,c_b$。
2. 用评分器计算每个候选的 $E(s_t,c_i)$。
3. 保留 top-k 候选进入下一层。
4. 若到达目标则结束；若分数过低则剪枝；必要时回溯。

可以用一个简化公式表示扩展策略：

$$
\mathcal{S}_{t+1} = \text{TopK}\left(\left\{(s_t \oplus c_i, E(s_t, c_i))\right\}_{i=1}^{b}\right)
$$

其中 $\oplus$ 表示把候选动作接到当前状态后得到新状态。白话解释是：从“当前解题草稿”接上“下一步尝试”，就得到新的草稿版本。

还是看 24 点这个玩具例子。假设当前数字是 `3, 3, 8, 1`。

第一层候选可以写成：

| 层级 | 当前状态 | 候选思路 | 说明 |
| --- | --- | --- | --- |
| 0 | `3,3,8,1` | `3×8=24` | 看起来接近目标，但没处理剩余数字 |
| 0 | `3,3,8,1` | `3+1=4` | 先凑结构，方便后续乘法 |
| 0 | `3,3,8,1` | `8-3=5` | 可尝试，但离 24 结构较远 |

如果评分器只看“离 24 近不近”，它可能误保留 `3×8=24`。但如果评分器还检查“剩余数字能否继续组合”，就会发现 `3+1=4` 更有潜力，因为接下来可以得到：

```text
根节点: [3, 3, 8, 1]
├─ c1: 3×8=24      -> 剩余 [24, 3, 1]，难继续
├─ c2: 3+1=4       -> 剩余 [4, 3, 8]
│  └─ c2.1: 3+3=6  -> 这里等价于另一分支先组合两个 3
│      └─ 4×6=24   -> 成功
└─ c3: 8-3=5       -> 剩余 [5, 3, 1]，较弱
```

这里能看出 ToT 与普通 CoT 的本质差异：CoT 假设“一次生成的一条链”足够好；ToT 假设“局部最像正确答案的那一步”未必真能通向全局解，因此必须让多个中间状态竞争。

Graph of Thoughts 则比树更进一步。树要求每个分支独立向下长，图允许不同路径共享节点。白话解释是：如果两个分支都推导到同一个中间结论，就没必要重复算两遍，可以把它们并到同一个节点上。这在长程规划、跨轮对话、复杂工作流中特别重要，因为很多局部子问题会反复出现。

从推导角度看，树关注“分支展开”，图关注“状态复用”。前者提升探索能力，后者提升计算效率和跨路径一致性。

## 代码实现

下面给一个可运行的 Python 玩具实现。它不调用大模型，而是用“枚举下一步算术操作 + 评分 + 剪枝”的方式模拟 Tree of Thoughts。这样更容易看懂核心结构。

```python
from itertools import combinations

TARGET = 24

def sample_candidates(state):
    """
    state: tuple[int, ...]
    返回若干候选状态，每个候选包含:
    - expr: 当前形成的表达式
    - next_state: 执行一步后剩余的数字状态
    """
    candidates = []
    for i, j in combinations(range(len(state)), 2):
        a, b = state[i], state[j]
        rest = [state[k] for k in range(len(state)) if k not in (i, j)]

        ops = [
            (a + b, f"({a}+{b})"),
            (a - b, f"({a}-{b})"),
            (b - a, f"({b}-{a})"),
            (a * b, f"({a}*{b})"),
        ]
        if b != 0 and a % b == 0:
            ops.append((a // b, f"({a}/{b})"))
        if a != 0 and b % a == 0:
            ops.append((b // a, f"({b}/{a})"))

        for value, expr in ops:
            next_state = tuple(sorted(rest + [value]))
            candidates.append((expr, next_state))
    return candidates

def evaluate(state):
    """
    分数越小越好:
    - 若只剩一个数字，直接看离目标有多远
    - 若还没结束，看当前数字中是否容易组合出 24
    """
    if len(state) == 1:
        return abs(state[0] - TARGET)
    return min(abs(x - TARGET) for x in state)

def solve_tot(initial_state, beam_width=5, max_depth=4):
    beams = [([], tuple(sorted(initial_state)))]
    visited = set()

    for _ in range(max_depth):
        new_beams = []
        for path, state in beams:
            if state in visited:
                continue
            visited.add(state)

            if len(state) == 1 and state[0] == TARGET:
                return path, state

            for expr, next_state in sample_candidates(state):
                new_path = path + [expr]
                score = evaluate(next_state)
                new_beams.append((score, new_path, next_state))

        new_beams.sort(key=lambda x: x[0])
        beams = [(path, state) for score, path, state in new_beams[:beam_width]]

    for path, state in beams:
        if len(state) == 1 and state[0] == TARGET:
            return path, state
    return None, None

path, state = solve_tot((3, 3, 8, 1), beam_width=8, max_depth=4)
assert state == (24,), f"expected 24, got {state}"
assert path is not None and len(path) > 0
print("Found:", " -> ".join(path), "=", state[0])
```

这段代码对应的结构非常接近 ToT：

| 函数 | 输入 | 输出 | 作用 |
| --- | --- | --- | --- |
| `sample_candidates` | 当前状态 | 若干候选新状态 | 生成多个“下一步可能怎么走” |
| `evaluate` | 新状态 | 分数 | 评估哪个分支更值得保留 |
| `solve_tot` | 初始状态 | 一条成功路径 | 执行“生成-评分-保留”的循环 |

新手可以把它理解成一个“会回头看、会挑路走”的解题器：

- `state` 是当前还剩哪些数字。
- `sample_candidates` 相当于把所有可能的一步都试出来。
- `evaluate` 相当于给每种思路打分。
- `beam_width` 表示每一层只保留最有希望的前几条路，这就是剪枝。
- `visited` 是缓存，避免重复走到同一个状态。

真实工程里，`sample_candidates` 不再是枚举算术操作，而是让大模型生成多个自然语言或程序式候选；`evaluate` 也不只是简单分数，而可能结合以下信号：

- 规则校验是否通过
- 外部工具执行是否成功
- 结果与约束是否冲突
- 另一模型给出的 verifier 分数
- 多条路径之间是否自洽

一个真实工程例子是无人机路径规划。此时状态 $s_t$ 不是“剩余数字”，而是“当前位置、目标点、障碍物信息、电量、飞行约束”；候选 $c_i$ 不是“加减乘除”，而是“下一段航路方案”。Tree of Thoughts 的价值在于：目标突然变化时，系统不必把整条路径推翻重算，而是可以回到某个中间状态重新展开分支。

## 工程权衡与常见坑

思维链结构越复杂，可控性越强，但成本也越高。这不是副作用，而是设计本身带来的交换。

| 收益 | 成本 | 常见坑 | 规避办法 |
| --- | --- | --- | --- |
| 数学与规划任务成功率上升 | token 消耗增加 | 每层都扩展导致费用爆炸 | top-k 剪枝、深度限制、缓存 |
| 可以回溯错误路径 | 推理延迟上升 | 在线场景超时 | 分层预算、先粗排后精排 |
| 中间状态可解释 | 幻觉面扩大 | 错误步骤越写越像真 | verifier、自洽投票、规则检查 |
| 可做多路径比较 | 实现复杂度提高 | 评分器本身不可靠 | 多信号融合，不依赖单一评分器 |

先看成本。若树深为 6，每层分支数稍大，实际调用次数会很快膨胀，所以很多系统会感觉“准了一些，但慢了很多”。给定研究摘要里提到，深度 6 的 Tree of Thoughts 可能带来约 4 倍时延，这类量级在工程上是完全可能出现的。

再看更关键的问题：幻觉。这里的幻觉，白话解释是“模型写出了看起来连贯、其实不成立的中间过程”。CoT 的一个常见误区是，以为“步骤更多”就等于“更可靠”。事实恰好相反：如果评分与验证不足，长链只是在放大错误。

常见坑主要有四类：

1. 把“会解释”误当成“会推理”
   很多系统看到模型能写出漂亮步骤，就默认它真的在算。其实它可能只是生成了形式上像推理的文本。

2. 评分器目标错位
   比如 24 点问题里，如果评分器只看“当前值离 24 多近”，就会偏爱 `3×8=24` 这种局部最优假象，却错过真正可完成的路径。

3. 分支太多但没有强剪枝
   如果每一步都盲目展开，搜索空间会迅速变成 $O(b^d)$，不仅贵，还会让错误分支淹没正确分支。

4. 缺少状态缓存与复用
   很多分支会到达相同中间状态，若不缓存，就会重复花 token 和时间。

真实工程例子更能说明问题。给定摘要提到，2026 年无人机路径规划场景中，ToT 在生产流量 A/B 测试下带来 28% 决策质量提升。它有效，不是因为“模型写得更长”，而是因为系统允许路径候选回溯，并在目标变动时重新展开局部搜索。这类收益通常来自结构化搜索，而不是提示词本身。

另一个例子是舆情推理或用户调查分析。此类任务经常包含跳跃逻辑：一条证据链不足以覆盖全部因果关系，所以会采样多条路径，再按贡献评分器融合，这就是 Random Forest of Thoughts 的思路。它适合多视角聚合，但风险也明显：若不同路径共享同一错误前提，投票也可能集体偏离。

工程上比较稳妥的做法通常是：

- 先用短 CoT 获得低成本候选。
- 只在高价值样本上启用树搜索。
- 强制引入 verifier 或程序检查器。
- 记录中间状态与失败轨迹，做 trace distillation。白话解释是：把高质量推理轨迹蒸馏成更短、更稳定的模式，供以后直接复用。

## 替代方案与适用边界

不是所有问题都值得上 Tree of Thoughts 或 Graph of Thoughts。选择结构时，关键不在“模型能不能”，而在“任务是否真的存在需要搜索的中间空间”。

| 方案 | 适用场景 | 优点 | 风险 |
| --- | --- | --- | --- |
| 直接回答 | 简单事实查询、固定知识问答 | 最便宜、最快 | 遇到复杂约束时不稳 |
| Zero-shot CoT | 基础算术、短逻辑题 | 实现简单，常有明显增益 | 步骤不稳定，容易一路错下去 |
| Few-shot CoT | 有固定模板的推理任务 | 中间状态更规范 | 示例迁移性有限 |
| 短 CoT + 检查器 | 中低复杂度任务 | 成本低，可靠性较高 | 搜索能力有限 |
| Tree of Thoughts | 数学、规划、组合搜索 | 可采样、回溯、剪枝 | 成本与时延显著增加 |
| Graph of Thoughts | 多轮复杂任务、状态复用明显 | 可合并节点，减少重复推理 | 图管理和评分更复杂 |
| Random Forest of Thoughts | 多视角证据整合、跳跃逻辑 | 多路径鲁棒性更好 | 容易把噪声也集成进去 |

可以用一个简单决策原则：

- 如果问题主要是“知道不知道”，优先直接回答或短 CoT。
- 如果问题主要是“按固定套路推导”，优先 Few-shot CoT。
- 如果问题主要是“需要试多条路、允许回头改”，才考虑 ToT。
- 如果不同分支会反复共享子结论，再考虑 GoT。
- 如果任务本身天然需要多视角融合，比如复杂舆情、长链归因，再考虑 Random Forest of Thoughts。

一个小任务例子是“某篇论文发表于哪一年”。这里没必要扩展成树，因为问题几乎没有搜索空间，增加步骤只会增加噪声。

一个中等任务例子是“根据日志判断一次请求失败最可能的原因”。这时可以用 Few-shot CoT 加检查器：先让模型按固定模板列出可能原因，再由规则检查器验证日志字段是否支持该结论。

一个复杂任务例子是“为无人机在动态障碍环境中规划航线”。这里单一路径明显不够，因为外部条件在变化，必须保留多个候选并支持回溯。若历史局部路径还能被多个计划复用，就进一步接近 Graph of Thoughts 的适用边界。

因此，思维链的进化不是线性替代关系，而是结构复杂度随任务复杂度上升的过程。简单问题优先简单结构，复杂问题再引入树和图，这才是工程上正确的使用方式。
