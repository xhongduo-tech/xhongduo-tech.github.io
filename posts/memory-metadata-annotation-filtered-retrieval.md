## 核心结论

在 Agent 的记忆系统里，只做向量相似度检索通常不够。向量检索擅长回答“语义上像不像”，但它不擅长回答“是不是这个用户的、是不是最近的、是不是高可信来源、是不是当前主题下的”。元数据，白话说，就是“描述这条记忆属于谁、来自哪里、什么时候产生、是否可靠”的附加信息。把时间范围、来源类型、置信度、主题标签这些元数据过滤条件叠加到向量检索上，通常能明显提升检索精度。

一个常用写法是把最终得分记成：

$$
s_i=\cos(q, v_i), \quad F_i \in \{0,1\}, \quad S_i = s_i \cdot F_i
$$

其中，$q$ 是查询向量，白话说就是“把当前问题编码成一个向量”；$v_i$ 是第 $i$ 条记忆的向量；$s_i$ 是语义相似度；$F_i$ 是过滤器，只要时间、来源、置信度、标签中有一项不满足，$F_i=0$，这条记忆就直接退出排序。

玩具例子最直观。假设系统先找到了 5 条语义最相近的记忆：

| 记忆 | 相似度 | 7 天内 | 来源=客服知识库 | 置信度≥0.8 | 标签=业务策略 | 是否保留 |
|---|---:|---|---|---|---|---|
| A | 0.92 | 否 | 是 | 是 | 否 | 否 |
| B | 0.89 | 是 | 是 | 是 | 是 | 是 |
| C | 0.88 | 是 | 是 | 是 | 是 | 是 |
| D | 0.85 | 是 | 否 | 是 | 是 | 否 |
| E | 0.81 | 是 | 是 | 否 | 是 | 否 |

如果只看纯向量 top-5，真正有用的只有 B、C，精确率是 $2/5=40\%$。如果加上元数据过滤，最终只保留 B、C，返回集精确率就是 $2/2=100\%$。这就是“先找相似，再做边界约束”的核心收益。

真实工程里，这个收益通常不是理论上的。多租户 Agent、客服机器人、知识助手、合规审查助手都要面对“语义相近但上下文错配”的问题。已有工程实践报告显示，合理使用向量搜索加标量过滤的混合查询，检索精确率可提升约 25%，而延迟增加通常控制在 15% 以内，甚至在索引设计合理时还可能下降，因为搜索空间先被收窄了。

## 问题定义与边界

问题定义可以写得很具体：当一个 Agent 从记忆库里检索候选内容时，目标不是找到“最像”的内容，而是找到“最像且属于当前上下文”的内容。

这里的“上下文”通常包含四类边界：

| 维度 | 含义 | 常见错配 |
|---|---|---|
| 时间 | 记忆是否仍然有效 | 返回过期票据、旧版本规则 |
| 来源 | 记忆来自哪个系统或通道 | 把用户聊天记录和官方知识库混在一起 |
| 置信度 | 这条记忆是否足够可靠 | 低质量抽取结果进入回答 |
| 标签 | 主题或业务域归属 | 财务问题召回营销材料 |

白话说，纯向量检索只能回答“这段文字看起来像不像你的问题”，但很多工程问题真正关心的是“这条内容能不能拿来用”。

例如用户问：“给我财务合规相关的审批依据。”如果只做向量检索，系统可能会召回“市场预算投放策略”，因为“预算、审批、流程”这些词在语义空间里也接近。但从任务角度看，这些内容属于错误主题。加上 `namespace=当前组织`、`tags 包含 财务+合规`、`created_at 在制度有效期内` 之后，检索才会落到正确范围。

这个方法也有适用边界，不是所有场景都值得上复杂元数据过滤。

| 场景 | 是否适合元数据过滤 | 原因 |
|---|---|---|
| 多租户知识系统 | 适合 | 必须隔离用户和组织数据 |
| 长期运行的 Agent 记忆 | 适合 | 时间衰减和来源混杂明显 |
| 高风险回答，如财务、法务、合规 | 适合 | 错召回成本高 |
| 小规模单主题语料 | 不一定需要 | 纯向量检索已足够简单有效 |
| 元数据本身很脏、不稳定 | 效果有限 | 过滤条件可能反而误伤结果 |

所以边界很明确：元数据过滤不是替代向量检索，而是给向量检索加上“业务约束层”。如果语料单一、主题稳定、没有租户隔离，纯向量检索可能已经够用；但只要进入真实业务系统，元数据几乎总会变成必要条件。

## 核心机制与推导

一次完整的记忆检索，通常可以拆成两个阶段：

`Candidate Pool -> Metadata Filter -> Final Ranking`

第一阶段是候选生成。系统根据查询向量 $q$，从向量库中找出与之最接近的一批记忆向量 $v_i$。这一步解决“语义相关性”。

第二阶段是元数据过滤。系统根据时间范围、来源类型、置信度、标签、租户、权限等条件，为每条候选生成一个过滤值 $F_i$。这一步解决“上下文一致性”。

形式化地写：

$$
s_i = \cos(q, v_i)
$$

$$
F_i =
\begin{cases}
1, & \text{若 } m_i \text{ 满足时间、来源、置信度、标签等条件} \\
0, & \text{否则}
\end{cases}
$$

$$
S_i = s_i \cdot F_i
$$

如果 $F_i=0$，即使语义相似度很高，这条记忆也不会进入最终排序。这个设计很重要，因为它说明元数据过滤不是“轻微调权”，而是“硬边界控制”。

对新手可以这样理解：向量检索像先从一堆纸片里挑出“看起来内容接近”的 8 张，元数据过滤像再检查每张纸片上的标签，“是不是最近 7 天的”“是不是客服知识库来的”“是不是高可信”。标签不对的纸片，不参加最后比较。

这里还有一个工程细节：过滤是先做还是后做。理想情况是由向量数据库在查询阶段直接利用标量索引或 JSON path 索引，把不满足条件的对象提前排除，这叫混合查询。Pinecone 支持把 metadata filter 直接放进查询请求；Milvus 支持 filtered search，也支持在过滤表达式复杂时使用 iterative filtering。这样做的意义是，系统不会先在全量数据里找一堆相似向量，再在应用层大规模剔除，而是从底层搜索空间就开始收缩。

一个典型的候选收敛过程如下：

| 阶段 | 数量 | 说明 |
|---|---:|---|
| 全量记忆 | 1,000,000 | 所有向量和元数据 |
| 按 `namespace` 过滤 | 80,000 | 只看当前组织 |
| 按 `tags=财务/合规` 过滤 | 6,000 | 只保留目标主题 |
| 按时间范围过滤 | 900 | 只保留最近有效记录 |
| 向量 top-k 排序 | 20 | 语义最相近的候选 |
| 应用层置信度再筛一轮 | 5 | 最终返回 |

这里可以看到，元数据过滤的真正价值不是只改变最后一步，而是改变“谁有资格进入最后一步”。

## 代码实现

下面给一个可运行的 Python 玩具实现。它不依赖外部向量库，只模拟“先语义检索，再元数据过滤”的流程。`assert` 用来验证过滤后的结果确实只剩目标记忆。

```python
from math import sqrt
from datetime import datetime

def cosine(a, b):
    dot = sum(x * y for x, y in zip(a, b))
    na = sqrt(sum(x * x for x in a))
    nb = sqrt(sum(x * x for x in b))
    return dot / (na * nb)

def passes_filter(item, namespace, required_tag, min_confidence, start_date):
    # 元数据过滤：租户、标签、置信度、时间范围
    if item["namespace"] != namespace:
        return False
    if required_tag not in item["tags"]:
        return False
    if item["confidence"] < min_confidence:
        return False
    if item["created_at"] < start_date:
        return False
    return True

records = [
    {
        "id": "A",
        "vector": [0.9, 0.1, 0.0],
        "namespace": "org-1",
        "tags": ["marketing"],
        "confidence": 0.95,
        "created_at": "2026-03-01",
    },
    {
        "id": "B",
        "vector": [0.8, 0.2, 0.0],
        "namespace": "org-1",
        "tags": ["financial", "compliance"],
        "confidence": 0.91,
        "created_at": "2026-03-10",
    },
    {
        "id": "C",
        "vector": [0.79, 0.21, 0.0],
        "namespace": "org-1",
        "tags": ["financial", "compliance"],
        "confidence": 0.88,
        "created_at": "2026-03-09",
    },
    {
        "id": "D",
        "vector": [0.78, 0.22, 0.0],
        "namespace": "org-2",
        "tags": ["financial", "compliance"],
        "confidence": 0.93,
        "created_at": "2026-03-10",
    },
]

query = [1.0, 0.0, 0.0]

# 第一步：纯向量排序，故意 over-sample
scored = []
for item in records:
    scored.append((item["id"], cosine(query, item["vector"]), item))

scored.sort(key=lambda x: x[1], reverse=True)
top_candidates = scored[:4]

# 第二步：元数据过滤
start_date = "2026-03-08"
filtered = [
    (item_id, score)
    for item_id, score, item in top_candidates
    if passes_filter(
        item,
        namespace="org-1",
        required_tag="financial",
        min_confidence=0.8,
        start_date=start_date,
    )
]

result_ids = [item_id for item_id, _ in filtered]

assert result_ids == ["B", "C"]
assert len(top_candidates) == 4
assert len(filtered) == 2

print(filtered)
```

这个例子体现了两个实现原则。

第一，先 `over-sample`，再过滤。`over-sample`，白话说，就是“先多拿一点候选，避免过滤后结果不够”。如果你只取纯向量 top-2，而这两条里恰好一条是过期数据，过滤后就可能只剩 1 条。实际工程里常见做法是目标返回 `k=10`，底层先查 `top_k=30` 或 `top_k=50`。

第二，尽量把能下推到向量库的过滤条件放到底层执行，只把难以下推的业务逻辑放到应用层。一个常见查询结构如下：

```python
query_payload = {
    "vector": [0.13, 0.88, -0.02, 0.44],
    "top_k": 40,
    "filter": {
        "namespace": {"$eq": "org-1"},
        "tags": {"$in": ["financial", "compliance"]},
        "created_at": {"$gte": "2026-03-01"},
        "source_type": {"$eq": "kb"},
    }
}

# 应用层再补充无法直接索引的逻辑
def refine(candidates):
    return [
        c for c in candidates
        if c["confidence"] >= 0.8 and c.get("is_deleted") is not True
    ]
```

真实工程例子可以看多租户 RAG 机器人。假设一个企业助手同时接入聊天历史、客服知识库、审批制度库。用户问“本月差旅报销里，哪些条目要走合规复核？”此时系统至少需要：

| 过滤项 | 作用 |
|---|---|
| `namespace=当前企业` | 防止跨企业数据泄漏 |
| `source_type in [policy, kb]` | 排除随意聊天记录 |
| `tags 包含 financial/compliance` | 限定业务主题 |
| `created_at 在制度有效期内` | 排除过期政策 |
| `confidence >= 0.8` | 排除低质量抽取内容 |

没有这些条件，语义上相似的“预算审批”“营销投放报销”“旧版财务规范”都可能被召回，最后污染回答。

## 工程权衡与常见坑

元数据过滤不是“写几个条件就完事”，它真正难的地方在索引设计、表达式复杂度和返回稳定性。

先看常见坑：

| 错误用法 | 风险 | 规避策略 |
|---|---|---|
| 用超大的 `$in`，一次传几千个 ID | 请求体膨胀，过滤解析变慢 | 改成按组织、角色、项目分组过滤 |
| 过滤表达式里堆很多 `OR` | 查询规划复杂，延迟抖动明显 | 预聚合标签，缓存常用过滤模板 |
| 没有 scalar index 或 JSON path index | 可能退化成全表扫描 | 为常用字段单独建标量索引 |
| 只做严格过滤，不做 over-sample | 过滤后结果不足 | 底层多取几倍候选 |
| 元数据脏数据多 | 错误过滤或误伤召回 | 建立字段规范和回填机制 |

一个典型误区是把元数据当成“附属字段”，结果写入时根本没设计索引。这样系统虽然语法上支持过滤，但执行时要扫描大量对象，延迟会明显上升。Milvus 的实践里，如果缺少合适的 scalar index 或 JSON path index，过滤可能退化得很严重；而一旦索引配置到位，过滤成本可以大幅下降。Pinecone 的经验也类似，过滤字段如果设计得足够结构化，single-stage filter 不一定会拖慢请求，反而可能因为候选集收缩而减少整体查询时间。

另一个坑是标签体系失控。很多团队一开始只有 `finance`、`compliance`、`marketing` 三个标签，后来不断增长为几十个甚至上百个，并且命名不统一，如 `financial`、`finance-doc`、`fin-policy` 混用。结果是过滤条件越来越长，表达式越来越复杂，维护成本也越来越高。工程上更稳妥的做法是建立有限枚举，或者提前聚合组合标签，比如直接维护 `financial_compliance` 这种复合字段。

还要注意精确率和召回率的平衡。过滤条件越严格，精确率越高，但召回率可能下降。比如把时间窗口从“最近一年”缩到“最近 7 天”，可能确实减少了过期内容，但也可能把仍然有效的老规则一起过滤掉。实际系统往往需要按场景设置不同模板，而不是全局统一一个过滤器。

如果是高风险应用，建议把过滤策略配置化。比如：

- 财务助手：强时间过滤、强来源过滤、强租户过滤
- 客服助手：中等时间过滤、强来源过滤、弱主题过滤
- 个人笔记助手：弱来源过滤、弱租户过滤、强调语义召回

这比把所有场景都塞进一个固定查询表达式要稳得多。

## 替代方案与适用边界

元数据过滤不是唯一方案，它只是当前最实用的一层控制手段。根据数据规模、表达式复杂度和延迟预算，还可以考虑替代方案。

| 策略 | 适用场景 | 优点 | 局限 |
|---|---|---|---|
| 纯向量检索 | 小规模单主题语料 | 简单，维护成本低 | 上下文错配明显 |
| 向量检索 + 元数据过滤 | 多租户、长期记忆、业务系统 | 精度高，边界清晰 | 依赖元数据质量和索引 |
| 迭代过滤 | 表达式复杂、字段变化频繁 | 更灵活 | 延迟更难预测 |
| 预聚合标签 | 标签组合固定、查询模式稳定 | 表达式简单，执行快 | 灵活性下降 |
| 重排模型 reranker | 候选量小但精排要求高 | 语义判断更细 | 成本高，不能替代权限隔离 |

所谓迭代过滤，白话说，就是“不一次性把复杂过滤表达式压到底层，而是边检索边试探”。当 `OR` 特别多、标签动态变化很快时，这种方式更稳，但它通常不适合极低延迟场景。

预聚合标签是另一种常见替代方案。比如你发现业务里经常查询“财务+合规”，而不是单查 `finance` 或 `compliance`，那就可以在写入阶段直接生成一个 `topic_group=financial_compliance` 字段。这样查询时只需要一个等值过滤，而不需要多个标签组合。代价是写入逻辑更复杂，而且标签组合变化后需要回填历史数据。

还要明确一个边界：元数据过滤不能替代权限系统。`namespace` 或 `tenant_id` 过滤可以减少误召回，但真正的数据隔离还需要依赖更底层的安全设计。换句话说，过滤能提升检索正确性，但不能被误当成唯一安全措施。

对零基础到初级工程师，实践建议可以压缩成一句话：如果你的 Agent 记忆要面对多用户、多来源、长期数据和高风险回答，就不要只做纯向量 top-k，把元数据当成一等公民来设计；如果语料很小、问题很单一、没有租户边界，就先用纯向量，等错配开始出现再逐步加过滤层。
