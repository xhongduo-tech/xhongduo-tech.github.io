## 核心结论

预训练数据的配比优化，本质上是在固定训练预算下，决定“每一类数据该喂多少”。这里的“配比”就是不同数据域的采样权重，例如百科、网页、书籍、代码、数学、多语言语料各占多少 token。结论先说在前面：

1. 配比不是细枝末节，而是决定模型能力形状的主变量之一。代码和数学比例提高，通常会增强结构化推理、符号操作和程序生成；自然语言比例过低，则会伤害常识、语法敏感性和开放域理解。
2. DoReMi 的价值不在于“找一个神奇最优比例”，而在于用小代理模型先估计哪些域最吃亏，再把训练预算往这些域重分配。它优化的是最差域表现，而不是单一平均值。
3. 在公开实验里，DoReMi 用 280M 代理模型为 8B 模型搜索权重，few-shot 平均准确度相对基线提升 6.5 个百分点，并以约 $1/2.6 \approx 38.5\%$ 的训练步数达到基线效果。这说明合理重配数据，边际收益可能大于“盲目多喂数据”。
4. 实务上最后常收敛到“语言为底，代码和数学定向增强”的混料策略，而不是极端偏科。对多数通用模型，代码比例通常不应长期超过约 50%；多语言或合成语料混入时，需要保留足够真实自然语言底座。

玩具例子可以这样理解。假设你有四桶数据：百科、书籍、代码、数学。小模型先“试吃”每桶，发现代码和数学上的损失明显更高，说明这两桶是当前短板。那下一轮大模型训练时，就不是平均分配，而是给代码和数学更多采样机会。这样做不是因为它们“更重要”，而是因为它们“当前最拖后腿”。

一个真实工程例子是通用基座模型预训练。团队往往既想保留聊天、摘要、问答等通用语言能力，又想提高代码生成、公式推导、工具调用效果。这时如果只加总数据量，成本直接上升；如果只手工把代码拉高到 60% 以上，又容易把语言底座打薄。自动配比优化的作用，就是在这两类目标之间找一个可复用、可验证的折中点。

| 方案 | 训练数据策略 | few-shot 平均准确度 | 达到基线所需步数 |
|---|---|---:|---:|
| 原始基线 | 使用默认域权重 | 基线 | 100% |
| DoReMi 重配比 | 用代理模型重估域权重后重采样 | +6.5 个百分点 | 约 38.5% |

## 问题定义与边界

问题可以形式化为一句话：在固定总 token 预算下，如何调整不同数据源的采样权重，使模型在保留通用能力的同时提升目标域表现。

这里的“域”指来源或内容类型相近的一组数据，例如网页文本、论文、代码仓库、数学题、多语种新闻。这里的“预算”指训练中允许读取的 token 总量，它直接对应计算成本。

这个问题有两个常见设置。

第一种是竞争型。总量固定，只能重分配。假设总预算是 $N_{\text{total}}$，你把代码占比从 10% 提到 30%，就必须从别的域扣出 20%。这最适合回答“在同样算力下，什么比例更值”。

第二种是附加型。某个底座数据量固定，在其上额外加入目标域。比如语言数据维持 132B token 不变，再额外加代码。这更适合回答“如果我愿意多花算力，代码带来的边际收益有多大”。

新手可以把它看成调饮料：

- 竞争型：杯子总量不变，只能重调配方。
- 附加型：先保留原来那杯，再继续往里加某种成分。

边界要说清楚，否则实验很容易误读：

| 维度 | 竞争型 | 附加型 |
|---|---|---|
| 总 token | 固定 | 增加 |
| 语言 token | 可能下降 | 可固定不变 |
| 是否便于比较算力 | 是 | 否 |
| 是否便于看目标域纯增益 | 一般 | 是 |
| 常见风险 | 目标域升高导致底座退化 | 训练时长和成本膨胀 |

常见边界条件有四条：

1. 不能只看平均指标，要单独看最差域或关键任务集。
2. 语言底座要设下限，否则模型会出现“代码强、表达弱”的偏科。
3. 低资源语言不能只按原始量分配，否则会被高资源语种挤压到几乎不可见。
4. 合成数据、翻译数据、去重不充分数据都应被视为单独域，否则优化出的权重可能只是在放大噪声。

如果你的数据源本来就很单一，例如几乎全是英语网页，那“配比优化”的空间其实很小，这时扩大高质量同类语料往往更直接。

## 核心机制与推导

DoReMi 的核心思路来自 min-max 优化，也就是“模型想把损失压低，而权重分配会放大最差的域”。形式化目标是：

$$
\min_\theta \max_{\alpha\in\Delta^k}\sum_{i=1}^k \alpha_i\left[\mathbb{E}_{x\in D_i}(\ell_\theta(x)-\ell_{\text{ref}}(x))\right]
$$

符号逐个解释：

- $\theta$ 是模型参数，也就是你真正训练的大模型或代理模型。
- $\alpha$ 是各域权重，位于 simplex $\Delta^k$ 上，意思是每个权重非负且总和为 1。
- $\ell_\theta(x)$ 是样本 $x$ 上的训练损失，可理解为“这条数据当前有多难学”。
- $\ell_{\text{ref}}(x)$ 是参考模型的损失，用来做差分，衡量当前模型相对参考模型是进步还是退步。

“参考损失”这一步很重要。因为直接比较不同域的原始损失并不公平，数学和代码本来就更难，网页文本本来就更容易。减去参考模型后，优化目标更接近“哪些域是当前训练分布下被低估或学得不够的”。

直觉上，外层的 $\max_\alpha$ 会把权重往高差分损失的域上推，内层的 $\min_\theta$ 则逼着模型把这些短板补起来。所以 DoReMi 不是简单地“给难数据更高权重”，而是“给相对落后的域更高权重”。

可以看一个玩具例子：

| 域 | 当前损失 $\ell_\theta$ | 参考损失 $\ell_{\text{ref}}$ | 差分损失 | 直觉上的权重变化 |
|---|---:|---:|---:|---|
| 百科 | 1.8 | 1.9 | -0.1 | 可下调 |
| 书籍 | 2.0 | 2.0 | 0.0 | 持平 |
| 代码 | 2.7 | 2.3 | 0.4 | 应上调 |
| 数学 | 3.1 | 2.6 | 0.5 | 应上调更多 |

这张表的含义是：百科相对参考模型还学得不错，代码和数学则明显落后，所以重新采样时应该把预算往后两者倾斜。

代码比例常用一个单参数 $m$ 控制：

$$
N_{\text{code}} = m\cdot N_{\text{total}}, \qquad N_{\text{lang}} = (1-m)\cdot N_{\text{total}}
$$

如果是附加型设置，则常把语言量固定成 $N_{\text{lang}}$，总量写成：

$$
N_{\text{total}} = \frac{N_{\text{lang}}}{1-m}
$$

例如 $N_{\text{lang}}=132\text{B}$、$m=0.5$，那么总量就是 264B，其中 132B 是语言，132B 是代码。这个设置的优点是不会因为“代码加多了，语言看少了”而混淆结论；缺点是总计算量变大，和竞争型不再是同一成本线。

不同规模模型的策略也不完全一样。小模型容量有限，过多域会互相争抢表示空间，通常更需要保守配比；更大的模型对多域混合更耐受，能够吸收更多代码、多语言和数学数据，但前提仍是高质量和足够 token 下限。

## 代码实现

工程实现通常分三步：先估计各域难度，再求权重，再按权重构造训练批次。

下面给一个可运行的简化版 Python 例子。它不训练真实大模型，只演示“根据差分损失求域权重，并在预算下分 token”的核心流程。

```python
import math

def softmax(xs):
    m = max(xs)
    exps = [math.exp(x - m) for x in xs]
    s = sum(exps)
    return [x / s for x in exps]

def project_simplex(weights):
    total = sum(max(w, 0.0) for w in weights)
    if total == 0:
        return [1.0 / len(weights)] * len(weights)
    return [max(w, 0.0) / total for w in weights]

def estimate_alpha(losses, ref_losses, temperature=1.0):
    deltas = [(l - r) / temperature for l, r in zip(losses, ref_losses)]
    return project_simplex(softmax(deltas))

def allocate_tokens(alpha, total_tokens):
    raw = [a * total_tokens for a in alpha]
    rounded = [int(x) for x in raw]
    rounded[0] += total_tokens - sum(rounded)
    return rounded

domains = ["百科", "书籍", "代码", "数学"]
losses = [1.8, 2.0, 2.7, 3.1]
ref_losses = [1.9, 2.0, 2.3, 2.6]

alpha = estimate_alpha(losses, ref_losses, temperature=0.5)
tokens = allocate_tokens(alpha, total_tokens=1000)

assert len(alpha) == 4
assert abs(sum(alpha) - 1.0) < 1e-9
assert sum(tokens) == 1000
assert tokens[2] > tokens[0]   # 代码应比百科拿到更多预算
assert tokens[3] > tokens[1]   # 数学应比书籍拿到更多预算

print(dict(zip(domains, zip(alpha, tokens))))
```

上面这段代码里：

- `temperature` 控制权重分布是否过尖。温度低，权重更集中；温度高，权重更平滑。
- `estimate_alpha` 对应“max over $\alpha$”的简化近似。
- `allocate_tokens` 把权重变成实际 token 预算。

如果你同时还要控制代码总占比 $m$，可以把“域权重”和“类型占比”拆成两层。先确定代码和语言的大盘，再在代码子域、语言子域内部继续分配：

```python
def split_code_lang(total_tokens, m):
    n_code = int(total_tokens * m)
    n_lang = total_tokens - n_code
    assert n_code + n_lang == total_tokens
    return n_code, n_lang

n_code, n_lang = split_code_lang(2000, m=0.3)
assert n_code == 600
assert n_lang == 1400
```

真实工程例子会更复杂。假设你有 8 个域：网页、书籍、论文、问答、代码、数学、中文通用、低资源多语。一个常见流程是：

1. 训练参考模型和代理模型。
2. 在每个域上测差分损失。
3. 用投影梯度、指数更新或 softmax 近似求 $\alpha$。
4. 给每个域设置上限、下限和温度采样，防止权重坍缩。
5. 重写数据加载器，按新权重重采样。
6. 在大模型上正式训练，并持续监控语言、代码、数学、多语四类验证集。

简化伪代码如下：

```python
for step in range(search_steps):
    deltas = []
    for domain in domains:
        loss = eval_proxy(domain)
        ref = eval_reference(domain)
        deltas.append(loss - ref)

    alpha = project_simplex(softmax(deltas))

    alpha = apply_floor(alpha, floor=0.03)      # 防止某域被压死
    alpha = apply_ceiling(alpha, ceiling=0.35)  # 防止单域吞掉预算

final_mixture = build_sampler(alpha, mode="competitive", m=0.2)
train_large_model(final_mixture)
```

关键点不在公式本身，而在数据流。真正决定结果的，往往是域划分是否合理、验证集是否独立、采样器是否稳定，而不是优化器用 Adam 还是镜像下降。

## 工程权衡与常见坑

第一类坑是把目标域拉得过高。代码和数学确实能带来结构化推理收益，但占比过高时，语言理解、语法细节、世界知识检索会掉。原因很直接：模型的有限训练预算被更多“格式化、强约束、重复模式高”的数据占用，自然语言覆盖变薄。

第二类坑是只看平均分，不看任务拆分。很多团队看到代码任务涨了，就以为配比成功；但一拆分就会发现，语法转换、细粒度阅读理解、开放域常识反而退了。配比优化的目标应是能力曲线重塑，不是单指标冲高。

第三类坑是低资源语言或稀有域被权重优化“清零”。从纯损失最小化看，这很合理，因为它们样本少、噪声高、短期收益不明显；但从产品覆盖和长期泛化看，这是错误的。所以实践里通常要加 floor、温度采样，甚至给低资源语种保留 1B 到 2B token 级别的最低预算。

第四类坑是把附加型结果拿去替代竞争型决策。附加型里你多喂了代码，还保留了原来的语言，自然容易看到结构任务收益；但这不等于在固定算力下同样成立。两种实验回答的是不同问题，不能混用。

| 常见坑 | 现象 | 规避方式 |
|---|---|---|
| 代码占比过高 | 代码涨，语言任务退 | 给自然语言设下限，代码长期不超过约 50% |
| 只看平均指标 | 看不见偏科 | 语言、代码、数学、多语分桶评估 |
| 低资源语言被压死 | 长尾语言几乎无提升 | 温度采样 + 最低 token 保底 |
| 合成数据混太多 | 验证集分数波动大 | 保留足够真实语料，单独监控噪声域 |
| 把附加型当竞争型结论 | 预算判断失真 | 两类实验分别汇报 |

一个实用原则是：每次只改一个大因子。先扫代码比例 $m$，再扫多语言配比，再决定是否启用 DoReMi 自动权重。否则变量太多，实验结论会失真。

## 替代方案与适用边界

DoReMi 不是唯一办法，只是自动化程度较高的一种。

第一种替代方案是手工加权。比如经验上设成 70% 通用语言、20% 代码、10% 数学。它的优点是简单、便宜、上线快；缺点是强依赖专家经验，域一多就很难调，而且往往对新模型规模不稳。

第二种是混合 loss 加指标校准。也就是训练时不只调采样概率，还给不同任务或不同数据源单独加 loss 权重。它更灵活，但系统复杂度更高，对数据标注和实验设计要求也更高。

第三种是单域扩充或后续继续预训练。比如底座模型先按通用语料训练，再追加代码继续预训练。它适合目标很明确的场景，例如“我要一个更强的代码模型”；但对“同时保持多域均衡”的问题，往往不如前期混料来得自然。

新手版理解很简单：如果你只有一桶英语书，调配比没有意义，因为根本没几种料可调；但如果你有网页、书籍、代码、数学、多语五六桶数据，自动重配就开始有价值。

| 方法 | 自动化程度 | 计算成本 | 适用场景 | 主要问题 |
|---|---:|---:|---|---|
| DoReMi | 高 | 中 | 多域、多目标、预算敏感 | 需要代理模型和独立评估集 |
| 手工权重 | 低 | 低 | 域少、上线快 | 依赖经验，难扩展 |
| 单域扩充/继续预训练 | 中 | 中到高 | 目标域非常明确 | 容易牺牲通用能力 |

适用边界也要明确：

1. 数据源足够多样时，DoReMi 才有优化空间。
2. 预算紧张时，它更有价值，因为它本质上是在做“更有效地花 token”。
3. 如果任务极其单一，例如纯代码补全或纯数学推理，直接增加目标域高质量语料可能更有效。
4. 如果数据域划分本身很粗糙，自动优化出来的权重也会失真，因为它只是精准地优化了一个错误切分。
