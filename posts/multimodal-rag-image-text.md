## 核心结论

多模态检索增强生成，简称 Multimodal RAG，就是把文本证据和图像证据一起检索，再一起交给生成模型回答问题。它解决的不是“模型会不会说”，而是“模型在生成前有没有拿到足够完整、彼此一致的证据”。

它的价值不在于让模型凭空变聪明，而在于补全证据链。当问题依赖图表、截图、包装、示意图、表格截图、界面状态这类视觉信息时，纯文本 RAG 常常只能拿到“描述”，拿不到“原始视觉证据”；而只看图像又容易缺少定义、单位、时间范围、业务口径。多模态 RAG 的核心做法，是在召回阶段同时考虑文本相似度和图像相似度，在生成阶段把两类证据并列输入视觉语言模型（VLM）。

一个最直观的例子是财报问答。系统把财报段落做文本 embedding，把营收趋势图做图像 embedding。用户问“这个季度营收趋势如何”。如果只检索文本，可能只找到“公司表现稳健”这类总结句；如果只检索图像，可能知道曲线在上升，但不知道单位、同比口径和统计区间。多模态 RAG 会把图表、图例、坐标轴信息，以及对应段落一起送进模型，回答才能同时包含走势、数值、解释和来源。

短句总结：多模态 RAG 解决的不是“会不会生成”，而是“生成前有没有拿到完整证据”。

## 问题定义与边界

问题可以写成三个步骤：

1. 输入是自然语言查询 $q$。
2. 检索系统返回两类证据：文本块 $c_t$ 和图像块 $c_i$。
3. 生成模型基于图文证据联合回答，而不是只依赖参数记忆。

这里的“文本块”是被切成小段后便于检索的文本片段，例如一段产品说明、一页财报中的一个小节、一份手册中的一条操作步骤。“图像块”是单张图片、截图、图表，或者从 PDF 页面中截出的视觉区域，例如一张趋势图、一块按钮区域、一页包装背面的营养成分表。所谓“块”，本质上是检索系统的最小索引单元。

这个定义有两个关键点，新手容易混淆：

| 概念 | 含义 | 为什么重要 |
|---|---|---|
| 检索 | 从外部证据库里找材料 | 决定模型能看到什么 |
| 生成 | 基于检索结果组织答案 | 决定模型怎么表达 |
| 多模态 | 证据不只是一种形式 | 决定答案是否遗漏视觉事实 |

它适合的任务有明显边界：

| 方案 | 输入证据 | 优势 | 主要短板 | 适用场景 |
|---|---|---|---|---|
| 纯文本 RAG | 文本 | 成本低，系统简单，检索稳定 | 看不到颜色、布局、图形关系、界面状态 | 文档问答、知识库搜索、FAQ |
| 图像转文本后再 RAG | OCR 文本、图像描述文本 | 工程接入快，便于复用文本链路 | 容易丢失结构、相对位置、视觉细节 | 简单票据、清晰表单、固定模板文档 |
| 多模态 RAG | 文本 + 原始图像 | 证据更完整，适合视觉问答 | 索引、存储、重排、推理成本更高 | 图表问答、截图排障、产品包装、复杂报告分析 |

看一个更具体的玩具例子。用户拿着一张牛奶包装图，问“这款包装有哪些安全标签”。如果系统只查说明书文本，可能知道“应含巴氏杀菌标识”；但真正包装上有没有“冷藏保存”“过敏原提示”“开封后尽快饮用”，要看图。若系统只依赖 OCR，也可能把文字识别出来，但识别不到红色警示条、图标位置、醒目标识是否真实存在。多模态 RAG 会同时检索包装图和说明书段落，回答才有完整依据。

边界也要说清楚。若任务完全是纯文本，例如“解释 TCP 三次握手”，多模态 RAG 往往不值得上，因为额外的图像编码、存储、排序和上下文拼接都会增加成本。反过来，如果任务强依赖视觉证据，而系统只保留 OCR 文本，那么关键信息会在进入检索之前就丢失，后续再强的模型也补不回来。

可以用一句话概括边界：

$$
\text{是否需要多模态} \approx \text{答案是否依赖视觉事实而非仅依赖文字事实}
$$

如果答案的一部分来自“图里长什么样”，就应考虑多模态；如果答案完全来自“文里写了什么”，纯文本方案通常更合适。

## 核心机制与推导

多模态 RAG 的关键不是“把图和文都存进去”，而是让检索排序能够反映两种模态对当前问题的真实贡献。否则系统虽然存了两类数据，但排序阶段仍然可能偏向单一模态。

最常见的联合打分写法是：

$$
S_{\text{joint}} = w_t \cdot \cos(T_q, T_c) + w_i \cdot \cos(I_q, I_c)
$$

其中：

- $T_q$ 是查询的文本向量。
- $T_c$ 是候选文本块的文本向量。
- $I_q$ 是查询对应的视觉向量。常见做法是把查询改写成适合视觉检索的描述，例如把“这个季度营收趋势如何”改写成“财报中的营收趋势图、柱状图、折线图”。
- $I_c$ 是候选图像块的图像向量。
- $w_t, w_i$ 是权重，表示当前任务更偏文本还是更偏视觉，且通常满足 $w_t + w_i = 1$。

“向量”可以理解为把内容压缩成一串数字，便于计算相似度。$\cos(\cdot)$ 是余弦相似度，用来衡量两个向量方向是否接近。值越大，一般表示语义越相关。

对新手来说，公式本身不难，真正要理解的是它表达的工程含义：

| 项 | 代表什么 | 直观解释 |
|---|---|---|
| $\cos(T_q, T_c)$ | 文本相似度 | 问题和文字说明像不像 |
| $\cos(I_q, I_c)$ | 图像相似度 | 问题和图像内容像不像 |
| $w_t$ | 文本权重 | 当前问题多依赖文字定义还是文字上下文 |
| $w_i$ | 图像权重 | 当前问题多依赖图表、布局、颜色或局部标识 |

一个数值例子最容易理解。假设某候选 chunk 的文本相似度是 0.8，图像相似度是 0.6，取 $w_t=0.6, w_i=0.4$，那么：

$$
S_{\text{joint}} = 0.6 \times 0.8 + 0.4 \times 0.6 = 0.72
$$

再假设另一个候选的文本相似度是 0.65，图像相似度是 0.85，那么：

$$
S_{\text{joint}} = 0.6 \times 0.65 + 0.4 \times 0.85 = 0.73
$$

这说明第二个候选虽然文字说明一般，但视觉证据非常强，联合后反而更该被召回。

看表更直观：

| 候选块 | 文本相似度 | 图像相似度 | 权重 $(w_t,w_i)$ | 联合得分 | 解释 |
|---|---:|---:|---:|---:|---|
| A：段落强、图一般 | 0.80 | 0.60 | (0.6, 0.4) | 0.72 | 文字上下文好，视觉一般 |
| B：段落一般、图很强 | 0.65 | 0.85 | (0.6, 0.4) | 0.73 | 视觉证据强，适合图表型问题 |
| C：两边都中等 | 0.70 | 0.70 | (0.6, 0.4) | 0.70 | 稳定但不突出 |

这张表说明，多模态检索的意义不在于“选绝对最强的一边”，而在于保留跨证据的一致性。对“图中这个趋势对应哪个产品线”这类问题，只有文本没有图不够，只有图没有说明也不够。

更完整一点，可以把流程写成两阶段：

### 1. 召回阶段

先分别检索文本和图像，得到候选集：

$$
\mathcal{C}_t = \text{TopN}_t(q), \quad \mathcal{C}_i = \text{TopN}_i(q)
$$

然后合并：

$$
\mathcal{C} = \mathcal{C}_t \cup \mathcal{C}_i
$$

### 2. 重排阶段

对合并后的候选重新计算联合分数：

$$
\text{rank}(c) = f\big(\cos(T_q,T_c), \cos(I_q,I_c), \text{meta}(c)\big)
$$

其中 $\text{meta}(c)$ 可以包含页码、来源、时间、是否与同页文本绑定等元数据。工程上常见做法不是只看相似度，还会加上绑定关系和去重策略。

进一步的系统通常会做两件事：

1. 分别召回再重排  
先跑文本检索 top-n，再跑图像检索 top-n，最后合并、去重、重排。这种做法稳定，容易调试，也是工业系统里最常见的起点。

2. 联合索引或跨模态重排  
直接在共享语义空间里检索，或者用 cross-attention 模型做更细的重排。这里的 attention 可以理解为“模型在对齐图文证据时，把更多关注放到哪些局部上”。

生成阶段的输入结构也很关键。常见模板不是把图像先全部压成一段描述文本，而是保持图文并列，例如：

- `[Text Chunk]` 原文摘录
- `[Image Evidence]` 图片、图表或截图
- `[OCR]` 从图里提取出的文字
- `[Meta]` 页码、时间、来源、section_id
- `[Instruction]` 只允许基于证据作答，不足则明确说证据不足

这样做的原因很直接：文本提供定义、单位、口径和背景；图像提供布局、趋势、颜色、局部标识和位置关系。两者是互补关系，不是互相替代关系。

## 代码实现

下面给出一个可直接运行的玩具实现。它不依赖外部模型，只模拟“文本相似度”和“图像相似度”已经算好的情况，重点看联合排序、去重和提示词拼接逻辑。

```python
from dataclasses import dataclass
from typing import List, Dict, Tuple


@dataclass(frozen=True)
class Chunk:
    chunk_id: str
    doc_id: str
    page_id: int
    text_score: float
    image_score: float
    text: str
    image_desc: str
    ocr: str


def joint_score(chunk: Chunk, wt: float, wi: float) -> float:
    if not (0.0 <= wt <= 1.0 and 0.0 <= wi <= 1.0):
        raise ValueError("wt and wi must be between 0 and 1")
    if abs((wt + wi) - 1.0) > 1e-9:
        raise ValueError("wt + wi must equal 1.0")
    return wt * chunk.text_score + wi * chunk.image_score


def retrieve(chunks: List[Chunk], wt: float, wi: float, topk: int = 3) -> List[Tuple[Chunk, float]]:
    ranked = sorted(
        ((chunk, joint_score(chunk, wt, wi)) for chunk in chunks),
        key=lambda item: item[1],
        reverse=True,
    )
    return ranked[:topk]


def dedupe_by_page(ranked: List[Tuple[Chunk, float]]) -> List[Tuple[Chunk, float]]:
    seen = set()
    result: List[Tuple[Chunk, float]] = []
    for chunk, score in ranked:
        key = (chunk.doc_id, chunk.page_id)
        if key in seen:
            continue
        seen.add(key)
        result.append((chunk, score))
    return result


def build_prompt(ranked: List[Tuple[Chunk, float]]) -> str:
    sections = [
        "你是财报问答助手。请仅依据以下证据回答；若证据不足，请明确说明“证据不足”。"
    ]
    for idx, (chunk, score) in enumerate(ranked, start=1):
        sections.extend(
            [
                f"[Evidence {idx}]",
                f"Score: {score:.2f}",
                f"Doc/Page: {chunk.doc_id} / {chunk.page_id}",
                "[Text Chunk]",
                chunk.text,
                "[Image Evidence]",
                chunk.image_desc,
                "[OCR]",
                chunk.ocr,
            ]
        )
    return "\n".join(sections)


def main() -> None:
    chunks = [
        Chunk(
            chunk_id="A",
            doc_id="report_2025_q4",
            page_id=12,
            text_score=0.80,
            image_score=0.60,
            text="财报段落：本季度营收同比增长 12%，主要由订阅业务带动。",
            image_desc="折线图显示过去四个季度营收持续上升，最新季度达到阶段高点。",
            ocr="Q1 18.2, Q2 19.4, Q3 20.1, Q4 22.5（单位：亿元）",
        ),
        Chunk(
            chunk_id="B",
            doc_id="report_2025_q4",
            page_id=13,
            text_score=0.65,
            image_score=0.85,
            text="财报段落：图表见附录，详细拆分请参考各业务线收入变化。",
            image_desc="柱状图显示本季度总营收创新高，同比明显高于去年同期。",
            ocr="2024Q4 19.6, 2025Q4 22.5（单位：亿元）",
        ),
        Chunk(
            chunk_id="C",
            doc_id="report_2025_q4",
            page_id=12,
            text_score=0.70,
            image_score=0.70,
            text="财报段落：成本率略有波动，但不影响整体增长趋势。",
            image_desc="图中趋势中等，无明显异常峰值。",
            ocr="成本率 31.2%, 32.1%, 31.8%, 31.5%",
        ),
    ]

    wt, wi = 0.6, 0.4
    ranked = retrieve(chunks, wt=wt, wi=wi, topk=3)

    # 验证联合打分逻辑
    assert round(joint_score(chunks[0], wt, wi), 2) == 0.72
    assert round(joint_score(chunks[1], wt, wi), 2) == 0.73
    assert [chunk.chunk_id for chunk, _ in ranked] == ["B", "A", "C"]

    # 去重，避免同一页重复占满上下文
    deduped = dedupe_by_page(ranked)
    assert [chunk.chunk_id for chunk, _ in deduped] == ["B", "A"]

    prompt = build_prompt(deduped)
    assert "Image Evidence" in prompt
    assert "证据不足" in prompt

    print("retrieval ok")
    print("--- prompt preview ---")
    print(prompt)


if __name__ == "__main__":
    main()
```

这段代码可以直接运行，输出 `retrieval ok` 和拼接后的提示词。它虽然是玩具实现，但已经覆盖了真实系统里三个基础动作：

1. 联合打分  
不是只按文本分数排，也不是只按图像分数排，而是按任务权重组合。

2. 去重  
如果多个结果都来自同一页，就要控制重复，避免上下文预算被浪费。

3. 证据拼接  
生成时不只给一段话，而是把文本、图像描述、OCR 和元数据一起给模型。

真实工程里，流程通常是下面这样：

1. 文本切块  
把 PDF、网页、说明书切成可检索片段，并保留页码、标题、来源链接、section_id。切块不是越小越好，太小会让上下文断裂。

2. 图像切块  
把整页、图表区域、截图区域单独存储。图表、流程图、包装图通常不能只靠 OCR，因为 OCR 只能拿到字，拿不到布局关系。

3. 分别编码  
文本用文本 embedding 模型，图片用图像或图文 embedding 模型，例如 CLIP 一类的双塔模型。embedding 的作用是把内容编码成向量，便于相似度搜索。

4. 建索引  
同一个 chunk_id 下保存文本向量、图像向量和 metadata，便于后续去重、回表、拼接和审计。

5. 检索与融合  
先分别召回，再计算联合得分，必要时引入重排模型。这里常加业务规则，例如“同页图文优先”“相邻页次优先”“来源不可信则降权”。

6. 生成  
把文本原文、图像、OCR、来源元数据一起交给多模态 LLM，让模型在证据范围内作答。

伪代码可以写成：

```text
text_q = text_encoder(query)
image_q = image_encoder(query_visual_hint)

text_hits = search_text_index(text_q, topn)
image_hits = search_image_index(image_q, topn)

candidates = merge(text_hits, image_hits)

for chunk in candidates:
    score = wt * cos(text_q, chunk.text_emb) + wi * cos(image_q, chunk.image_emb)

topk = rerank(candidates, by=score)
topk = dedupe(topk, key=(doc_id, page_id))
prompt = build_prompt(topk.text, topk.image, topk.ocr, topk.meta)
answer = multimodal_llm(prompt)
```

真实工程例子：财务问答系统处理季度报告时，常见做法是把“管理层讨论”段落作为文本块，把“营收趋势图”“毛利率柱状图”作为图像块，再通过页面 ID 或 section ID 绑定。用户问“这个季度营收趋势如何，和去年同期相比有什么变化”，系统应同时返回图表和对应说明段落，而不是只返回 OCR 抽出的几个数字。否则模型可能把曲线方向说对，却把同比口径、业务范围或单位说错。

## 工程权衡与常见坑

多模态 RAG 的主要难点不是调用一个多模态模型，而是索引设计、证据同步和上下文预算控制。多数失败案例并不是模型太弱，而是上游证据组织得不好。

常见坑如下：

| 问题 | 具体表现 | 结果 | 修复方式 |
|---|---|---|---|
| 只做 OCR，不保留原图 | 图表、包装、界面截图都被转成文字 | 丢失颜色、位置、趋势、图例 | 同时保留原图和 OCR |
| 图文块没有绑定关系 | 文本块和图片块各自独立 | 生成时容易拼错证据 | 建立 `doc_id/page_id/section_id` |
| 粒度不一致 | 文本切得很细，图片切得很粗 | 排序不稳定，难以对齐 | 图文 chunk 尽量对应同一语义单元 |
| 权重固定 | 所有问题都用一组 $(w_t,w_i)$ | 视觉题和文本题都不理想 | 按问题类型动态调权 |
| top-k 不去重 | 前几个结果都来自同一页 | 浪费上下文，证据单一 | 去重、分组采样、多样性约束 |
| 上下文预算失控 | 图片过多、OCR 过长、原文过长 | 真正关键证据被挤掉 | 截断策略、摘要策略、优先级策略 |

下面逐个展开。

### 1. 只做 OCR，不保留原图

这是最常见的问题。OCR 能把图里的字变成文本，但很多视觉事实根本不是“字”：

- 红色警示框和普通黑字的风险级别不同
- 折线图上升、下降、震荡的走势不是文字本身
- 图例颜色与曲线颜色的对应关系不是纯文本
- UI 截图里按钮是灰色禁用还是蓝色可点，OCR 看不出来

因此，图像先“降级”为文本，再拿文本去检索，往往不是严格意义上的多模态，而是“带 OCR 的文本 RAG”。

### 2. 图文块没有绑定关系

如果一段文字写“如图 3 所示，本季度订阅业务增速最快”，但图 3 没有和这段话共享元数据，系统检索回来时就可能把别页的图错配过来。结果是每个证据单看都对，合在一起却错。

最基本的绑定字段通常包括：

| 字段 | 作用 |
|---|---|
| `doc_id` | 标识来自哪份文档 |
| `page_id` | 标识来自哪一页 |
| `section_id` | 标识来自哪个章节 |
| `chunk_id` | 标识具体索引单元 |
| `source_url` | 支持回溯和引用 |

### 3. 文本块切得太细，图像块切得太粗

文本切块太细时，虽然关键词容易命中，但上下文会断。图像切块太粗时，一整页都被当成一个图，真正关键的图表区域又被背景信息稀释。结果就是文本命中精确、图像命中模糊，联合得分不稳定。

一个简单经验是：图文 chunk 最好对应同一语义单元。例如“这一页中的营收趋势图 + 紧邻其下的解释段落”可以作为一个可绑定的单元，而不是把整页和整章都扔进同一个桶里。

### 4. 权重固定不变

并不是所有问题都该用同样的 $w_t,w_i$。例如：

| 问题 | 更高权重应给谁 | 原因 |
|---|---|---|
| “图里红线代表什么” | 图像 | 先识别图例和颜色对应关系 |
| “这张图对应哪个会计口径” | 文本 | 口径定义往往在说明段落里 |
| “这个按钮为什么不可点击” | 图像 + 文本 | 图像看状态，文档看规则 |
| “营收趋势和去年同期相比如何” | 图像 + 文本 | 图像给走势，文本给同比定义 |

实际工程里常见做法是先做问题分类，再按类别设置不同权重，或者训练一个小模型预测权重。

### 5. top-k 只看分数，不看多样性

如果前 5 个结果都来自同一页同一图，虽然分数很高，但信息高度重复。对生成模型来说，这等于把上下文窗口浪费在相同证据上。工程上通常需要：

- 按 `doc_id/page_id` 去重
- 限制单页最多入选数量
- 强制保留至少一条文本证据和一条图像证据
- 对相邻页做适度保留，避免章节上下文被切断

### 6. 上下文预算失控

多模态输入非常吃资源。图片本身占位置，OCR 又是一段文本，原文再来一段，元数据还要占一点。如果系统没有预算策略，就会出现“证据很多，但关键证据反而进不去”的问题。

最常见的预算控制方法是：

| 控制项 | 做法 |
|---|---|
| 图片数量 | 只保留 top-m 图像证据 |
| OCR 长度 | 截断到关键字段或关键区域 |
| 文本长度 | 优先保留定义、结论、图注、单位说明 |
| 元数据 | 只保留必要字段，避免冗余 |

工程上最重要的规避原则只有一句：不要把视觉证据先降级为文本，再假装自己做了多模态。

## 替代方案与适用边界

不是所有系统都要上多模态 RAG。合理的做法是按任务复杂度、成本预算和证据类型来选方案，而不是默认“更复杂就更高级”。

| 方案 | 成本 | 实现复杂度 | 最适合的任务 | 不适合的任务 |
|---|---:|---:|---|---|
| 纯文本 RAG | 低 | 低 | 文档知识库、FAQ、规范问答 | 图表、截图、包装、强视觉证据任务 |
| OCR/Caption 后文本 RAG | 中 | 中 | 票据识别、结构简单的扫描件、固定模板表单 | 颜色敏感、布局敏感、图形关系任务 |
| 多模态 RAG | 高 | 高 | 图文问答、视觉校验、复杂报告分析、截图排障 | 纯文本场景、极低成本场景 |

替代方案主要有两类。

第一类是纯文本 RAG。产品说明书页面如果几乎全是正文，没有图表、没有布局含义，那么直接做文本检索更快、更便宜，也更容易维护。此时上多模态的边际收益很低。

第二类是“图像先 OCR 再索引”。这是一种折中方案，适合内容简单、结构强、成本敏感的任务，比如表单、发票、固定模板清单。因为这些图像中的关键信息大部分能被文字化，损失相对可接受。

但一旦任务依赖下面这些信息，就应优先保留图 embedding，而不是只保留 OCR 文本：

| 视觉信息类型 | 例子 |
|---|---|
| 颜色区分 | 红线与蓝线代表不同产品线 |
| 曲线走势 | 上升、下降、拐点、波动区间 |
| 相对位置 | 标签在左上还是右下，警示语靠近哪个区域 |
| 局部标识 | 图标、箭头、按钮状态、异常框 |
| 版式结构 | 标题、图注、表头、说明框的层级关系 |
| 图例与注释关系 | 哪条曲线对应哪个说明 |

换句话说，如果答案的一部分存在于“怎么排、怎么画、怎么标”里，而不只是“写了什么”，就不要退回纯文本方案。

一个简单判断法是：

$$
\text{若视觉损失} > \text{工程节省}，则应保留原始图像链路
$$

这不是严格数学定律，但适合做工程决策。比如图表问答系统里，丢掉曲线走势造成的准确率下降，通常远大于少做一个图像索引所节省的成本。
