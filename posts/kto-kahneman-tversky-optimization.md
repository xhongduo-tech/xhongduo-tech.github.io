## 核心结论

KTO，Kahneman-Tversky Optimization，可以把它理解成“用单条好/差反馈训练模型的对齐方法”。它不要求标注员在两个回答之间做成对比较，只需要对某个回答给出“好”或“差”的二分类反馈，就能推动模型朝更符合人类偏好的方向更新。

它和传统偏好对方法的区别，不在于“是不是做对齐”，而在于“把人类反馈建模成什么”。传统 RLHF 中常见的偏好学习，会把问题写成“回答 A 是否优于回答 B”；KTO 则把问题写成“当前回答相对一个参考点，是收益还是损失”。这个参考点来自参考分布的平均奖励，因此 KTO 直接借用了前景理论中的“损失规避”思想。

对零基础读者，先记住一句话：告诉模型“这个回答好”，就像对它说“这条路是对的，继续走”；告诉模型“这个回答差”，就像说“这条路偏了，退回来”。KTO 的价值就在于，很多真实场景里我们拿不到稳定、便宜、隐私友好的成对偏好数据，但往往能拿到大量稀疏的单条反馈。

下表先给出最实用的对比：

| 维度 | KTO | 传统偏好对 RLHF / DPO |
|---|---|---|
| 反馈形式 | 单条回答的“好/差” | 两个回答中选更好 |
| 标注成本 | 低 | 较高 |
| 是否必须成对数据 | 否 | 是 |
| 对稀疏反馈适应性 | 高 | 一般 |
| 对标注噪声的敏感性 | 在多数负样本场景更稳 | 成对冲突时容易放大不一致 |
| 适合场景 | 设备端、隐私场景、日志反馈、弱监督 | 有完整偏好对数据的集中训练 |

## 问题定义与边界

KTO 解决的问题，不是“怎样让模型学会所有人类价值”，而是一个更具体的问题：当我们只有单条输出的正负反馈，而没有高质量偏好对时，怎样训练模型的输出分布，使其更接近人类偏好。

这里有几个边界要说清楚。

第一，KTO 建模的是输出级偏好，不是事实真值。一个回答被标成“好”，只表示它在当前任务和标注标准下更可接受，不表示它在客观上一定正确。对齐算法优化的是“人类偏好信号”，不是“世界真理”。

第二，KTO 不是直接做排序学习，而是在 HALO 框架里做效用最大化。HALO，Human-Aware Loss，可以直白理解成“把人类感受写进损失函数的训练方式”。它不是只问“谁比谁好”，而是问“这个输出相对参考点让人感觉是赚了还是亏了”。

第三，KTO 依赖参考分布 $Q$。参考分布可以理解成“模型平时大概会生成什么”的基准集合。模型对某个输出给出的奖励，如果高于这个基准的平均值，就落在“收益区”；低于平均值，就落在“损失区”。前景理论认为，人类对损失往往更敏感，因此负反馈不只是简单减分，而是可能产生更强的校正力。

一个玩具例子：

假设你在训练客服模型。用户只会点“满意”或“不满意”，不会对两个答案做投票。  
- 回答 A：礼貌、准确，被点“满意”  
- 回答 B：语气生硬、漏掉步骤，被点“不满意”

KTO 不需要再找一个与 A 或 B 配对的替代答案。它只需要知道：A 相对参考水平是收益，B 相对参考水平是损失，然后分别拉近和推远。

可以把它想成一条有参考刻度的平衡木。每个反馈都不是在“两个答案互相比”，而是在问“这个答案站在平均线之上还是之下”。只要参考线稳定，单条反馈就足以给出更新方向。

下表把 HALO/KTO 的输入输出压缩成一个工程视角：

| 元素 | 含义 | 白话解释 | 对更新方向的影响 |
|---|---|---|---|
| $x$ | 输入提示词 | 用户问题 | 决定任务上下文 |
| $y$ | 模型输出 | 当前回答 | 被评价的对象 |
| $a_{x,y}\in\{+1,-1\}$ | 反馈符号 | “好”记 +1，“差”记 -1 | 决定推近还是推远 |
| $r_\theta(x,y)$ | 当前奖励 | 模型认为这个输出有多好 | 决定更新强弱 |
| $\mathbb{E}_{y'\sim Q}[r_\theta(x,y')]$ | 参考均值 | 这类任务下的平均水平 | 决定参考点位置 |
| $v(\Delta)$ | 前景函数 | 收益和损失的感知曲线 | 决定非线性放大或压缩 |

KTO 的边界也很明确：如果你的数据天然就是高质量偏好对，例如同一 prompt 下稳定收集到多个候选答案，并且标注员一致性很高，那么 DPO 这类方法通常更直接。KTO 的优势，是在反馈稀疏、数据不成对、隐私限制强、标注口径不完全稳定时，仍然能工作。

## 核心机制与推导

KTO 的核心目标可以写成：

$$
f(\pi_\theta,\pi_{\text{ref}})=\mathbb{E}_{x,y\sim D}\left[a_{x,y}\,v\left(r_\theta(x,y)-\mathbb{E}_{y'\sim Q}[r_\theta(x,y')]\right)\right]+C_D
$$

这条式子里最关键的是中间那一项：

$$
\Delta = r_\theta(x,y)-\mathbb{E}_{y'\sim Q}[r_\theta(x,y')]
$$

$\Delta$ 可以理解成“当前回答比参考平均线高多少或低多少”。如果 $\Delta>0$，说明它处于收益区；如果 $\Delta<0$，说明它处于损失区。

再看 $a_{x,y}$。它只有两个取值：
- $+1$：正向反馈，这个回答被认为“可以保留甚至强化”
- $-1$：负向反馈，这个回答被认为“应该远离”

最后是 $v(\Delta)$。这是前景理论里的价值函数，通常是一条 S 形曲线。白话解释：人类对“赚一点”和“亏一点”的感觉不对称，亏损往往更刺痛，所以损失区的斜率和敏感度可以与收益区不同。

一个常见的简化写法是：

$$
v(\Delta)=
\begin{cases}
\Delta^\alpha, & \Delta \ge 0 \\
-\lambda(-\Delta)^\beta, & \Delta < 0
\end{cases}
$$

其中：
- $\alpha$ 控制收益区曲率，常取小于 1，例如 0.9
- $\beta$ 控制损失区曲率，常取小于 1，例如 0.6
- $\lambda$ 是损失厌恶系数，常取大于 1，表示“同样幅度的损失更痛”

这条式子的直觉是：
- 当奖励高于平均时，模型得到正向强化，但边际收益递减
- 当奖励低于平均时，模型受到更敏感的惩罚，从而更快远离明显差的输出

这就是“Kahneman-Tversky 效应”在训练中的作用：不是线性加减分，而是把人类对收益与损失的不对称感受写进优化目标。

看一个最小数值例子。假设参考均值为 $1.2$。

1. 样本被标为“好”，当前奖励 $r_\theta=1.6$  
   则 $\Delta=1.6-1.2=0.4$  
   若 $\alpha=0.9$，则
   $$
   v(0.4)=0.4^{0.9}\approx 0.43
   $$
   因为 $a=+1$，所以目标中增加一个正项，推动模型提升这类输出概率。

2. 样本被标为“差”，当前奖励 $r_\theta=0.8$  
   则 $\Delta=0.8-1.2=-0.4$  
   若 $\beta=0.6,\lambda=1.5$，则
   $$
   v(-0.4)=-1.5\times 0.4^{0.6}\approx -0.87
   $$
   因为 $a=-1$，目标中的乘积会形成强烈的“远离差输出”信号。

这里容易混淆的一点是：反馈符号 $a$ 和价值函数的正负，不是重复表达同一件事。  
- $a$ 表示“人类标签的方向”  
- $v(\Delta)$ 表示“相对参考点的感知强度”

两者结合，才构成完整梯度信号。

一个真实工程例子是设备端或联邦微调。比如输入法、邮件助手、代码补全工具，常见的反馈不是“回答 A 比 B 好”，而是“用户接受了这个建议”或“用户立刻删掉了这个建议”。这些日志天然就是单条正负反馈，而且由于隐私和分布碎片化，难以回收成完整偏好对。KTO 在这里比依赖成对比较的方法更自然，因为它顺着数据采集方式训练，不强行改造标注流程。

## 代码实现

工程上，KTO 可以插入现有奖励建模或 DPO 类训练流程。最小数据流通常是：

1. 采集单条输出的二分类反馈
2. 将“好/差”映射为 $a=\pm1$
3. 计算当前样本奖励 $r_\theta(x,y)$
4. 估计参考分布上的平均奖励 $\mathbb{E}_{y'\sim Q}[r_\theta(x,y')]$
5. 计算 $\Delta$ 与前景函数 $v(\Delta)$
6. 用 $-a\cdot v(\Delta)$ 构造最小化损失，反向传播更新参数

下面先给一个可运行的玩具版 Python 代码，不依赖深度学习框架，重点看公式如何落地：

```python
import math

def prospect_value(delta: float, alpha: float = 0.9, beta: float = 0.6, lam: float = 1.5) -> float:
    if delta >= 0:
        return delta ** alpha
    return -lam * ((-delta) ** beta)

def kto_objective(reward: float, ref_mean: float, label: int) -> float:
    assert label in (+1, -1)
    delta = reward - ref_mean
    return label * prospect_value(delta)

good_obj = kto_objective(reward=1.6, ref_mean=1.2, label=+1)
bad_obj = kto_objective(reward=0.8, ref_mean=1.2, label=-1)

assert good_obj > 0
assert bad_obj > 0  # 负样本配合损失区价值后，目标仍鼓励“远离坏输出”

# 对训练来说，一般最小化负目标
loss_good = -good_obj
loss_bad = -bad_obj

assert loss_good < 0
assert loss_bad < 0

print(round(good_obj, 4), round(bad_obj, 4))
```

如果放进 PyTorch，通常会把奖励写成“当前策略相对参考策略的对数概率比”：

$$
r_\theta(x,y)=\beta\log\frac{\pi_\theta(y|x)}{\pi_{\text{ref}}(y|x)}
$$

这里的 $\beta$ 是温度或缩放系数，白话解释是“控制奖励信号有多激进”。

下面是一个更接近训练代码的片段：

```python
import torch

def prospect_value_torch(delta, alpha=0.9, beta=0.6, lam=1.5):
    pos = torch.pow(torch.clamp(delta, min=0.0), alpha)
    neg = -lam * torch.pow(torch.clamp(-delta, min=0.0), beta)
    return torch.where(delta >= 0, pos, neg)

def kto_loss(policy_logp, ref_logp, ref_mean_reward, labels, beta_scale=0.1):
    """
    policy_logp: 当前策略对输出 y 的 log prob, shape [B]
    ref_logp: 参考策略对同一输出 y 的 log prob, shape [B]
    ref_mean_reward: 参考奖励均值, shape [B] or scalar
    labels: +1/-1, shape [B]
    """
    labels = labels.to(policy_logp.dtype)
    reward = beta_scale * (policy_logp - ref_logp)
    delta = reward - ref_mean_reward
    value = prospect_value_torch(delta)
    objective = labels * value
    loss = -objective.mean()
    return loss

policy_logp = torch.tensor([-1.2, -2.0, -0.7])
ref_logp = torch.tensor([-1.5, -1.7, -0.8])
labels = torch.tensor([1, -1, 1])
ref_mean = torch.tensor(0.0)

loss = kto_loss(policy_logp, ref_logp, ref_mean, labels)
assert torch.isfinite(loss)
loss.backward = lambda: None  # 这里只做结构演示
print(float(loss))
```

工程里还有两个实现点经常被忽略。

第一，参考均值怎么估计。最简单的做法是批内均值或滑动均值；更稳的做法是按任务类型、提示模板、领域分桶估计，否则“写代码”和“做摘要”的参考线混在一起，会让 $\Delta$ 失真。

第二，标签语义必须稳定。`+1` 和 `-1` 不能混入“中立”“不确定”“风格不喜欢但内容正确”这类模糊状态。KTO 吃的是二分类信号，边界不清，梯度方向就会混乱。

下表给出实现时常见变量：

| 变量 | 作用 | 常见来源 |
|---|---|---|
| `labels` | 单条好/差反馈 | 人工标注、点击日志、接受/拒绝 |
| `policy_logp` | 当前策略概率 | 当前模型前向输出 |
| `ref_logp` | 参考策略概率 | 冻结参考模型 |
| `ref_mean_reward` | 参考奖励均值 | 批内统计、滑动平均、分桶均值 |
| `alpha/beta/lam` | 前景函数参数 | 经验设定或验证集调优 |

## 工程权衡与常见坑

KTO 的强项是反馈便宜，但代价是它更依赖反馈语义清晰。便宜不等于随便采。下面这些坑，比公式本身更决定训练效果。

| 常见坑 | 现象 | 原因 | 规避策略 |
|---|---|---|---|
| 标签噪声大 | 同类样本方向来回震荡 | “好/差”标准不一致 | 先写明确标注准则，必要时先做人审校准 |
| 参考分布偏移 | 某些任务整体被误判成差 | 参考均值不代表当前任务 | 按领域或任务分桶估计参考均值 |
| 负样本过多 | 模型学会保守拒答 | 远离坏输出太强，靠近好输出不足 | 控制正负样本比例或单独调损失权重 |
| 二分类边界模糊 | 风格问题和事实问题混在一起 | 标签语义不单一 | 将“安全性”“正确性”“格式性”拆开标 |
| KTOO/KTOR 选择不当 | 训练稳定性差 | 目标形式与数据分布不匹配 | 先做小规模离线对比再定方案 |

这里提到的 KTOO 和 KTOR，可以把它们理解成 KTO 在不同参考构造或目标组织方式下的变体。对初学者最重要的不是记住缩写，而是知道：不同实现细节会改变参考点和梯度稳定性，因此不能只看论文公式，必须看你手头数据的分布。

一个典型新手误区，是把任何负反馈都当成“应该强烈惩罚”。这并不总是对。真实工程里，负反馈可能来自三类不同原因：
- 内容错误
- 风格不符合偏好
- 用户当时没读完，不一定真差

如果把这三类混成同一个 `-1`，损失区的敏感放大会把噪声一起放大。前景函数本来是为了更符合人类效用，不是为了替代数据清洗。

再说一个真实工程例子。假设你在做代码助手：
- 用户接受补全并直接运行，通常是强正反馈
- 用户立刻删除补全，通常是强负反馈
- 用户部分修改后再提交，这是弱反馈，不能粗暴归类

如果你把“部分修改后接受”也标成 `-1`，模型会错误打压那些“结构正确但命名不合团队习惯”的补全。结果不是更安全，而是更保守、更短、更不愿给有用建议。

所以 KTO 的工程核心并不是“前景理论多高级”，而是“你有没有把单条反馈定义得足够干净”。

## 替代方案与适用边界

如果你有大量高质量偏好对，DPO 往往是更自然的选择。它直接优化“偏好答案的概率高于非偏好答案”，不必额外引入单条标签语义设计。对有完整对比数据的场景，它像一张更清晰的导航图。

如果你能在线采样、训练预算足、还需要长期信用分配，那么 PPO 一类 RLHF 方案仍然有位置。它更像全流程强化学习：采样、打分、更新、再采样。代价是实现复杂、训练不稳定点更多。

KTO 的适用边界，可以压缩成一句话：当你只有单条好/差反馈，或者把反馈收集成成对数据的代价太高时，KTO 很合适。

| 方法 | 数据形式 | 训练复杂度 | 对噪声敏感性 | 适配场景 |
|---|---|---|---|---|
| KTO | 单条正负反馈 | 中 | 对边界模糊敏感，但对不成对数据友好 | 设备端、日志反馈、隐私场景 |
| DPO | 偏好对 | 中 | 依赖成对标注质量 | 集中式偏好数据、对比明确任务 |
| PPO | 奖励模型 + 在线采样 | 高 | 训练链路长，调参复杂 | 高预算、需要策略探索的场景 |

还可以考虑混合方案。比如先用 KTO 吃下大规模廉价单条反馈，把模型从“明显不合格”拉到“基本可用”；再用少量高质量偏好对做 DPO 精修。这样的分阶段训练，在工程上常比“只押一种方法”更稳。

对新手可以这样理解：
- 有大量对比数据时，DPO 像有完整导航
- 只能拿到“好/差”标签时，KTO 像单杆拐杖
- 需要持续试错和探索时，PPO 更像自己边走边画地图

没有哪一种方法在所有场景都最好。KTO 不是 DPO 的全面替代，而是把“对齐所需的数据门槛”降了下来。
