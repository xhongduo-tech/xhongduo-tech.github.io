## 核心结论

推理服务限流的目标，不是“把更多请求塞进去”，而是先守住系统的稳定边界，再决定哪些请求可以被处理。这里的稳定边界，指的是延迟、队列长度、显存占用、下游副本负载这些会直接决定服务是否还能工作的指标。

对推理服务来说，限流本质上是在请求入口对“谁发来的、在多长时间内、最多能发多少”做约束。更准确地说，它约束的是“身份 + 时间 + 容量”三元组，而不是一个孤立的 QPS 数字。术语首次出现时可以这样理解：QPS 是“每秒请求数”，只是观察流量快慢的一个指标，不等于完整的限流规则。

一个实用方案通常至少有两层：

1. 本地快速限流：用令牌桶吸收突发，保护单个网关或副本不被瞬时打穿。
2. 全局配额限流：用滑动窗口或全局计数约束租户、API Key 或全局总量。

可以把它抽象成一条简单链路：

`入口流量 -> 限流判定 -> 放行 / 排队 / 拒绝 -> 推理副本`

玩具例子：某推理网关高峰时先用本地令牌桶接住一小波突发，只有租户总量真的超出配额时才统一返回 `429 Too Many Requests`。白话讲，就是门口先放一个小缓冲区，平时快速通过，人一下子太多时先挡一部分，避免里面直接挤爆。

真实工程例子：一个多副本 LLM 推理服务，单个租户预算是 `600 req/min`。如果只在每个副本本地各限 `600 req/min`，3 个副本就可能实际放行 `1800 req/min`。系统表面上“每台机器都守规矩”，但全集群已经失控。

| 目标取向 | 策略倾向 | 结果 | 风险 |
| --- | --- | --- | --- |
| 稳定性优先 | 提前拒绝、限制队列、严格配额 | P95/P99 延迟更稳 | 吞吐可能下降 |
| 吞吐优先 | 放宽入口、加长排队、少拒绝 | 短期吞吐更高 | 队列膨胀、尾延迟恶化 |

## 问题定义与边界

推理服务限流不是“把每秒请求数压到某个数字以下”这么简单。它真正回答的是：某个身份在某段时间内，是否还能继续消耗系统容量。

这里要先划清边界。限流处理的是请求入口；排队处理的是已接收请求的等待顺序；调度处理的是请求如何分配到副本或 batch；缓存处理的是能否减少真实计算。四者相关，但不是同一层问题。很多系统失控，不是因为算法不会写，而是把入口限流、队列调度、模型执行混在一起，最后哪一层出问题都说不清。

滑动窗口的核心统计式是：

$$
C(t)=\sum 1[t_i \in (t-W, t]]
$$

其中，$W$ 是窗口长度，$t_i$ 是第 $i$ 个请求的到达时间，$C(t)$ 表示最近 $W$ 秒内真实发生的请求数。白话讲，就是“别看整点，直接看现在往前数这段时间里到底来了多少次”。

同样是 `1000 req/min`，按不同对象限流，结果完全不同：

| 限流对象 | 含义 | 典型用途 | 风险 |
| --- | --- | --- | --- |
| 用户 | 针对单个终端用户 | 防止个人刷接口 | 用户身份难稳定识别 |
| 租户 | 针对企业/团队账号 | SaaS 配额 | 需要跨用户聚合 |
| API Key | 针对调用凭证 | 平台开放接口 | Key 泄漏会放大风险 |
| 全局 | 针对整个服务 | 保护系统总容量 | 容易误伤低流量租户 |
| 副本本地 | 针对单个实例 | 快速保护机器 | 不能代表全局 |

为什么“本地限流”不能自动等价为“全局限流”？因为本地看到的只是局部流量。假设 3 个副本各自独立计数，每台都认为“我没超”，全集群总量却可能已经超标 3 倍。这就是局部正确、全局错误。

## 核心机制与推导

常见限流机制里，令牌桶、漏桶、滑动窗口分别解决不同问题。

令牌桶中的“令牌”可以白话理解为“预先攒下来的通行证”。其公式是：

$$
T(t)=\min(B, T(t_0)+r\cdot \Delta t)-c
$$

其中，$r$ 是补充速率，$B$ 是桶容量，$c$ 是本次请求消耗的令牌数。它适合允许短时突发，因为平时没用掉的额度可以积攒。

玩具例子：令牌桶配置为 `r=5 req/s, B=10`。在 `t=0` 突然来了 12 个请求，如果桶里一开始满了，就先放行 10 个，后 2 个只能等待补充令牌，或者直接被拒绝。白话讲，就是先存 10 张通行证，来得猛的时候先花库存。

漏桶的“漏”可以理解为“固定速度往外处理”。公式是：

$$
q(t)=\max(0, q(t_0)+a-r\cdot \Delta t)
$$

其中，$q(t)$ 是队列长度，$a$ 是这段时间到达量，$r$ 是泄露或处理速率。它更适合平滑输出，因为无论输入多抖，输出都会尽量匀速。

滑动窗口更适合精确配额统计，因为它直接数历史事件，不依赖自然分钟边界。比如 `W=60s, L=300`，如果最近 60 秒已经有 300 个请求，第 301 个立刻拒绝。重点在于“最近 60 秒”，不是“这一分钟”。

三种机制可以这样对比：

| 机制 | 核心作用 | 优点 | 缺点 | 更适合什么 |
| --- | --- | --- | --- | --- |
| 令牌桶 | 控制突发 | 快、简单、允许短时峰值 | 不擅长精确历史统计 | 入口保护 |
| 漏桶 | 平滑输出 | 输出稳定 | 容易把压力变成长队列 | 下游保护 |
| 滑动窗口 | 真实计数 | 配额精确、公平性更好 | 存储和计算更贵 | 租户级配额 |

为什么固定窗口会出现“双倍突发”？因为它按自然边界清零。假设规则是每分钟最多 100 次：某租户在 `12:00:59` 发 100 次，又在 `12:01:00` 立刻再发 100 次，理论上 2 秒内进了 200 次，但两个自然分钟都没超标。滑动窗口不会有这个问题，因为它一直看“最近 60 秒”，不会在边界处突然归零。

所以一个常见组合是：入口先用令牌桶吸收尖峰，租户配额再用滑动窗口做精确判断。如果下游副本很脆弱，再在副本前加漏桶或有限排队，让进入模型执行层的流量更平稳。

## 代码实现

代码实现时，先分清“本地快速限流”和“全局一致限流”。

本地快速限流的职责是快，通常用内存实现，纳秒到微秒级完成判断。全局一致限流的职责是准，通常要借助 Redis 或独立限流服务来共享状态。

先看一个可运行的 Python 令牌桶玩具实现：

```python
from dataclasses import dataclass

@dataclass
class TokenBucket:
    rate: float
    capacity: float
    tokens: float
    last_ts: float

    def allow(self, now: float, cost: float = 1.0) -> bool:
        elapsed = max(0.0, now - self.last_ts)
        self.tokens = min(self.capacity, self.tokens + elapsed * self.rate)
        self.last_ts = now
        if self.tokens >= cost:
            self.tokens -= cost
            return True
        return False

bucket = TokenBucket(rate=5.0, capacity=10.0, tokens=10.0, last_ts=0.0)

allowed = sum(bucket.allow(0.0) for _ in range(12))
assert allowed == 10

assert bucket.allow(0.2) is True   # 补回 1 个令牌
assert bucket.allow(0.2) is False  # 同一时刻没有继续补充
```

对应的本地伪代码非常简单：

```js
function allowRequest(now) {
  refill(now)
  if (tokens >= cost) {
    tokens -= cost
    return true
  }
  return false
}
```

真实工程例子：推理网关收到请求后，先检查本地令牌桶。如果本地已经没有令牌，说明当前实例已接近自己的安全边界，可以立即拒绝或短暂排队；如果本地通过，再去 Redis 检查租户级滑动窗口配额。两层都通过，才真正转发到推理副本。

Redis 滑动窗口常见做法是用有序集合按时间戳存请求，再在同一个原子脚本里清理窗口外数据、统计窗口内数量、插入当前请求。关键是“原子”，也就是这几步必须像一次不可分割的事务执行。

```lua
-- 先清理窗口外请求，再计数并判断是否超限
-- key: tenant:{id}
-- argv: now_ms, window_ms, limit, member_id

redis.call("ZREMRANGEBYSCORE", KEYS[1], 0, ARGV[1] - ARGV[2])
local count = redis.call("ZCARD", KEYS[1])

if count >= tonumber(ARGV[3]) then
  return 0
end

redis.call("ZADD", KEYS[1], ARGV[1], ARGV[4])
redis.call("PEXPIRE", KEYS[1], ARGV[2])
return 1
```

为什么 `INCR` 和 `EXPIRE` 分开做会有竞态？因为两个命令之间如果线程切换、进程崩溃或网络抖动，可能出现“计数加了但没设过期”的脏状态，导致窗口永久膨胀；也可能多个并发请求都先看到“未超限”，再同时加一，最终超发。

实现方式可以粗略比较如下：

| 方式 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| 内存计数 | 极快、成本低 | 仅本地有效 | 单实例保护 |
| Redis 计数 | 易共享 | 并发下需谨慎设计 | 中小规模全局配额 |
| Lua 原子脚本 | 保证读改写一致 | Redis 成为关键依赖 | 高频全局限流 |
| 全局限流服务 | 统一治理、策略丰富 | 成本和复杂度高 | 多集群、多租户平台 |

## 工程权衡与常见坑

限流不是越严格越好。过严会让系统看起来“很稳”，但用户大量收到 `429`；过松会把问题从拒绝率转成高延迟、超时和副本雪崩。真正要守的是体验边界，而不是某个漂亮的理论值。

真实工程里最常见的坑，不是不会写令牌桶，而是不会处理分布式现实。

第一个坑是把本地当全局。3 个副本每个本地限 `600 req/min`，集群就可能实际放行 `1800 req/min`。白话讲，每个门卫都只数自己门口的人，没人统计整栋楼。

第二个坑是把漏桶当万能方案。漏桶确实能把流量压平，但如果队列无限长，用户只是从“被拒绝”变成“排很久后超时”。从系统视角看，曲线平了；从用户视角看，体验仍然坏。

第三个坑是时钟漂移。多节点系统中，如果每台机器时间不一致，滑动窗口会出现误判。有的节点提前“过期”旧请求，有的节点还把旧请求算在窗口里，最终表现为一部分请求被误拒，一部分请求被误放。

常见坑与规避方式可以整理为：

| 常见坑 | 表现 | 后果 | 规避方式 |
| --- | --- | --- | --- |
| 本地当全局 | 每副本都未超，但总量超限 | 全局配额失效 | 共享状态或全局服务 |
| 固定窗口抖动 | 边界处双倍突发 | 瞬时打穿下游 | 滑动窗口或令牌桶 |
| 非原子更新 | 计数与过期分离 | 超发或脏数据 | Lua/事务/原子接口 |
| 时钟漂移 | 节点判断不一致 | 误限流 | 统一时间源 |
| 队列过长 | 拒绝率低但延迟高 | 用户超时 | 限制最大队列深度 |

判断到底是“限流策略问题”还是“容量不足问题”，可以看这几个指标：

- `q(t)`：队列长度是否持续增长而不回落。
- P95/P99 延迟：尾延迟是否先于错误率恶化。
- 拒绝率：是否突然升高，且与容量利用率不匹配。
- GPU/显存利用率：是否已经逼近物理上限。
- 每租户放行量：是否出现个别大租户挤占全局容量。

一个简单诊断清单是：如果拒绝率高但副本很空，通常是策略过严或配额错误；如果拒绝率不高但 P99 爆炸，通常是队列太长或容量不足；如果单租户流量异常放大，则优先检查是否只做了本地限流。

## 替代方案与适用边界

没有一种限流算法能同时最优解决公平、低延迟、强一致、低成本这四个目标。工程上要先回答“你最怕什么”，再选工具。

如果最怕入口突发把网关打穿，优先令牌桶。它快，而且允许合理突发。
如果最怕单个租户超用，优先滑动窗口或全局计数。它更接近真实历史。
如果最怕下游副本被瞬时洪峰打坏，优先漏桶或受控排队。
如果最怕系统复杂度太高，宁可接受一定误差，也可以先用固定窗口做简单版本，但要明确它的边界抖动问题。

“门口用快速拦截，后台用精确记账”是常见组合：API 网关层用本地令牌桶，租户配额层用滑动窗口或全局限流服务。对于推理服务，还常常要把排队长度、batch size、超时策略一起考虑，因为模型推理不是纯 CPU Web 请求，它会受显存、批处理和解码时长影响。

| 方案 | 优点 | 缺点 | 适用边界 |
| --- | --- | --- | --- |
| 令牌桶 | 允许突发、实现简单 | 配额精度一般 | 入口层 |
| 漏桶 | 输出平滑 | 容易堆队列 | 下游保护 |
| 固定窗口 | 最简单 | 边界抖动明显 | 低成本粗粒度控制 |
| 滑动窗口 | 统计精确 | 成本更高 | 租户级配额 |
| 全局限流服务 | 一致性更强 | 系统复杂 | 多副本、多集群 |
| 纯排队 | 少拒绝 | 高延迟风险 | 请求可等待场景 |

何时优先选“简单拒绝”，何时优先选“平滑排队”？

- 当请求不可等待，例如在线聊天首 token 延迟敏感，优先简单拒绝，避免把用户拖进长尾延迟。
- 当请求可等待，例如离线批量生成或低优先级任务，可以短时间排队，换取更高资源利用率。

再往前一步，很多场景需要的不是“限流单件工具”，而是“限流 + 熔断 + 降级”组合。熔断可以理解为“发现下游明显异常时直接断开”；降级可以理解为“在容量紧张时提供更便宜的服务路径”。例如把大模型切到小模型、缩短最大生成长度、关闭高成本选项。这些都比单纯放任排队更稳。
