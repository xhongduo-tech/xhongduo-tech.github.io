## 核心结论

InternLM-Math 的核心价值，不是“又一个会做数学题的模型”，而是把三类原本分开的能力放进同一条推理轨迹里：自然语言推理、可执行代码、形式化证明。自然语言推理就是用普通文本拆步骤，便于解释；可执行代码就是把中间计算交给 Python 一类工具，减少心算误差；形式化证明就是把结论翻译成 Lean 4 这类定理证明器可检查的表达，得到“机器可核查”的正确性信号。

这意味着模型不再只输出一个答案，而是在同一轮生成中动态决定什么时候“继续讲道理”，什么时候“先算一下”，什么时候“把当前结论送去验证”。论文里把这种交错过程称为 RICO，Reasoning Interleaved with Coding，直译就是“推理与代码交错进行”。

它的统一性体现在两个层面。

第一，统一输出格式。Chain-of-Thought，简称 CoT，就是逐步文字推理；Program-of-Thought，简称 PoT，就是把某一步改写成程序执行。InternLM-Math 不是把两者拆成两个子系统，而是让模型在一个 seq2seq 轨迹里混合生成它们，再接上 Lean 4 证明片段。

第二，统一训练信号。不是只看最后答对没答对，而是把结果奖励、过程奖励、Lean 验证奖励一起用于训练。用一个简化公式表示就是：

$$
R_{\text{total}} = R_{\text{ORM}} + R_{\text{PRM}} + \mathbb{1}(\text{Lean success})
$$

其中 ORM 可以理解为结果对不对，PRM 可以理解为过程是否合理，$\mathbb{1}(\text{Lean success})$ 表示 Lean 证明成功时给一个额外奖励。

一个玩具例子就能说明这个框架。题目是“$1+1$ 等于多少”。普通模型可能直接答 2。InternLM-Math 的理想轨迹更像这样：

1. 先写自然语言：两个单位相加得到两个单位。
2. 再写代码：`print(1 + 1)`，执行结果为 2。
3. 最后写 Lean 语句去验证 `1 + 1 = 2`。

重点不在于这道题有多简单，而在于三种动作被放进同一个闭环：推理、执行、验证。这个闭环正是它区别于普通数学模型的地方。

| 模块 | 作用 | 说明 |
| --- | --- | --- |
| CoT 推理 | 生成自然语言步骤 | 负责拆分子问题，保留解释性 |
| PoT / 代码解释器 | 执行可计算步骤 | 负责数值计算、符号计算、中间结果核对 |
| Lean 验证 | 形式化检查当前结论 | 负责把“看起来对”变成“可机器核查” |
| 奖励模型 | 评价整条轨迹 | 同时评价结果、过程、证明是否成功 |

## 问题定义与边界

InternLM-Math 解决的问题，不只是“提高做题准确率”，而是构建一个可核查的数学推理系统。可核查，白话说，就是模型不仅给答案，还要给出一条能被外部工具检查的路径。这个目标比传统的问答更严格，因为它要求模型产出的中间过程也尽量可靠。

它覆盖的是两类任务。

一类是 informal math，也就是自然语言数学题，例如 GSM8K、MATH。这类题的输入和输出都接近日常文字。另一类是 formal math，也就是形式化数学，例如 MiniF2F。形式化数学要求把命题写成 Lean 可接受的严格语法，不能靠“人类读起来差不多”蒙混过关。

因此，InternLM-Math 的真实目标不是单点优化，而是跨两种任务形态建立统一接口：

- 对自然语言题，模型要能解释、计算、求解。
- 对形式化题，模型要能构造证明或至少给出可验证的中间命题。
- 对混合任务，模型要能在解释、执行、验证之间切换。

这里的边界也很明显。

第一，Lean 验证不是万能校验器。Lean 擅长验证你已经表达清楚的命题，但如果自然语言步骤本身就模糊，或者中间定义没补全，Lean 往往无法直接接手。它验证的是“形式化后的命题”，不是“所有人类语义上的模糊直觉”。

第二，代码执行也不是越多越好。很多初级读者容易误以为“能调 Python 就一定更强”。其实不是。对简单逻辑题，过早调用代码会增加 token 开销，也可能让模型把本来该清楚解释的步骤压缩掉，最后得到“答案对了，过程不可读”。

第三，训练奖励会影响模型偏好。如果训练里代码执行成功、Lean 成功证明的奖励很强，模型可能倾向于优先走工具路径，而不是先写稳定的语言推理。这就是为什么官方已知问题里会提到：某些中文或中英混杂题，模型更容易直接触发代码解释器。

可以用一个真实工程例子说明边界。假设题目是：“一辆车以每小时 60 公里行驶 2.5 小时，路程是多少？”人类很容易写出 $60 \times 2.5 = 150$。但如果模型直接输出 150，没有显式给出速度、时间、路程三者关系，那么后续 Lean 想验证“路程 = 速度 × 时间”时，可能缺少必要定义。也就是说，答案正确不等于轨迹可验证。

## 核心机制与推导

InternLM-Math 的关键机制可以概括成一句话：让模型在同一条生成轨迹里，按需在 CoT 和 PoT 之间切换，并把 Lean 成功与否作为强化学习信号的一部分。

先看推理闭环。RICO 的基本循环不是“先完整思考，再完整算，再完整验证”，而是局部交错：

1. 先生成一小段自然语言推理。
2. 判断当前步骤是否适合交给程序执行。
3. 插入代码并执行。
4. 读取执行结果。
5. 继续生成下一段推理。
6. 在合适时生成 Lean 命题或证明片段。
7. 如果 Lean 成功，给这条轨迹更高奖励。

这个机制的实质，是把“思考”从单一文本过程变成带外部观察的交互过程。观察，白话说，就是模型能看到执行器或证明器返回的反馈，而不是只能盲猜下一步。

一个玩具例子是求三角形面积。题目给定底 `base=6`、高 `height=4`。

- CoT 步骤：三角形面积公式是底乘高除以二。
- PoT 步骤：执行 `0.5 * 6 * 4`。
- 执行结果：12。
- Lean 步骤：把“面积等于底高的一半”翻译成形式化命题。

这里 CoT 提供结构，PoT 提供数值，Lean 提供验证。三者不是替代关系，而是分工关系。

如果把它写成抽象流程，可以表示为：

$$
s_{t+1} = f\bigl(s_t, y_t, o_t\bigr)
$$

其中 $s_t$ 是当前推理状态，$y_t$ 是模型本轮输出的文本或代码，$o_t$ 是外部工具观察结果。传统纯文本 CoT 近似只有 $s_t \rightarrow y_t$；InternLM-Math 多了 $o_t$，也就是工具反馈。

奖励设计则解释了为什么模型会学会这种切换。只给最终答案奖励，模型容易学出“结果导向但过程混乱”的行为；只给过程奖励，模型可能写很多看似工整但无效的步骤。把两者叠加，再加上 Lean 成功信号，目标就变成：既要答对，也要过程可靠，还要能被外部验证。

这三个奖励各自对应不同偏差：

| 奖励项 | 主要解决的问题 | 风险 |
| --- | --- | --- |
| ORM | 最终答案错 | 容易忽略中间过程质量 |
| PRM | 步骤胡编、跳步 | 可能鼓励表面上“像推理”的冗长文本 |
| Lean 奖励 | 结论不可验证 | 受限于形式化覆盖范围 |

真实工程例子是 MiniF2F 这类形式化数学基准。MiniF2F 可以理解为“把高中到大学初步数学题变成定理证明器可检查的任务集合”。在这种任务上，单纯能“说得像”远远不够，必须给出 Lean 可以接受的形式化结果。InternLM-Math 的改进点，正是在候选轨迹里优先保留那些最终能通过 Lean 的路径。于是模型不只是在模仿解题文本，而是在逐渐偏向“可证明的解题文本”。

## 代码实现

如果从工程实现看，InternLM-Math 的思路并不神秘。最小可复现版本就是把一个问题组织成四段：问题、文字推理、代码执行、形式化验证。模型不一定每次都用到所有段，但模板要容纳这些位置。

一个简化模板如下：

```text
Question: {problem}

Chain-of-Thought:
1. ...

Python Code:
```python
...
```

Lean Verification:
theorem ...
```

下面给一个可运行的 Python 玩具实现。它不是真正的 InternLM-Math 训练代码，而是把“推理 -> 执行 -> 验证”的基本闭环写成最小示例。`assert` 的作用是程序化断言，白话说，就是自动检查结果是否满足预期。

```python
from fractions import Fraction

def rico_fraction_add(a_num, a_den, b_num, b_den):
    # CoT: 先把问题转成统一分母
    lcm = a_den * b_den
    left = Fraction(a_num, a_den)
    right = Fraction(b_num, b_den)

    # PoT: 用可执行对象完成精确分数计算
    result = left + right

    # "Lean-like" verification: 这里用 assert 模拟可核查约束
    assert result == Fraction(a_num * b_den + b_num * a_den, lcm)
    return result

toy = rico_fraction_add(4, 5, 3, 10)
assert toy == Fraction(11, 10)
print(toy)  # 11/10
```

这个例子对应一道简单分数题：$\frac{4}{5} + \frac{3}{10}$。玩具点在于它没有真的调用 Lean，只是用 `assert` 模拟“形式化检查”的思想。但它展示了统一框架最关键的设计：不要把“解释”和“计算”完全割裂。

再看一个更接近真实工程的例子。假设要做一个数学 agent，流程通常是：

1. 模型先生成若干自然语言步骤。
2. 检测哪些步骤适合转成代码。
3. 调用 Python 执行器求中间值。
4. 将执行结果回填到上下文。
5. 对关键结论生成 Lean 命题。
6. 用 Lean 成功与否对候选轨迹排序。

伪代码如下：

```python
def solve_with_rico(model, problem, executor, verifier):
    state = {"problem": problem, "trace": []}

    for _ in range(5):
        reasoning = model.generate_reasoning(state)
        state["trace"].append(("cot", reasoning))

        if model.need_code(reasoning):
            code = model.generate_code(state)
            result = executor(code)
            state["trace"].append(("pot", code, result))

        if model.need_verification(state):
            theorem = model.generate_lean(state)
            ok = verifier(theorem)
            state["trace"].append(("lean", theorem, ok))
            if ok:
                break

    return state
```

真正的官方实现当然更复杂，涉及 SFT、RLHF、奖励模型和推理模板，但工程骨架就是这个思路。重点不是某个 API，而是统一状态机：模型输出的不再只是“答案文本”，而是“带执行和验证副作用的轨迹”。

## 工程权衡与常见坑

这个框架很强，但不是没有代价。最大代价是复杂度上升。你不是只维护一个语言模型，而是在维护“模型 + 执行器 + 证明器 + 奖励策略”的组合系统。任何一层出问题，最终体验都会下降。

常见坑主要有三类。

第一类是跳步。模型为了追求最终答案，可能省略必要中间步骤。这在普通聊天里问题不大，但在统一数学框架里是致命的，因为代码执行和 Lean 验证都依赖前面步骤足够明确。缺一步定义，后面就无法接。

第二类是工具滥用。某些题本来只需两句逻辑说明，模型却直接生成大段代码。这通常不是“更严谨”，而是奖励偏好或模板触发导致的路径偏移。对用户来说，结果是解释性变差，推理成本反而更高。

第三类是形式化落差。自然语言结论“基本正确”，不代表 Lean 能接受。Lean 对语法、类型、定理前提都很严格。很多看起来只差一点的结论，在形式化系统里就是完全不同的命题。

下面这张表可以直接作为工程排障清单。

| 坑 | 典型表现 | 根因 | 规避方式 |
| --- | --- | --- | --- |
| 跳过计算步骤 | 直接给答案，后续无法验证 | 结果奖励过强，过程约束不足 | 提示词强制 step-by-step，训练中提高过程奖励权重 |
| 代码解释器滥用 | 简单题也先写长代码 | 工具调用模板触发过早 | 增加“是否需要代码”的分类步骤 |
| Lean 验证失败率高 | 语言步骤看似合理，形式化却卡住 | 中间定义、变量约束不完整 | 在生成阶段显式补齐假设与符号定义 |
| 中英混合题不稳定 | 输出风格漂移，工具调用异常 | 数据分布不一致 | 做语言归一化，统一模板语言 |

真实工程里还有一个经常被低估的问题：延迟。一次完整的 RICO 轨迹，可能包含多轮生成、代码执行、Lean 校验。相比纯文本回答，它的成本更高，响应也更慢。所以它更适合高价值、高可靠性场景，例如数学证明、教育批改、自动构造可验证解题记录；不一定适合所有聊天式问答。

## 替代方案与适用边界

如果任务目标只是“把题算对”，并不一定需要 InternLM-Math 这种统一框架。更轻量的 CoT 或 PoT 系统通常已经够用。比如只做算术、代数代换、短链路应用题，使用普通数学微调模型加一个 Python 工具调用层，成本会更低。

替代路线大致有三类。

第一类是纯 CoT。优点是简单、可解释、部署便宜。缺点是中间计算容易出错，而且无法天然获得形式化验证。

第二类是 CoT + 代码解释器。优点是数值计算更可靠，适合竞赛题、应用题、表格题。缺点是“可执行”不等于“可证明”，在正式证明任务上仍然不够。

第三类是专门的 theorem prover agent，例如围绕 Lean、HOL 或其他证明器构建的系统。优点是验证强，缺点是对自然语言题的兼容性较差，工程门槛也高。

对零基础到初级工程师来说，可以用一句话判断适用边界：

- 只要结果：优先普通数学模型或 Python 工具增强。
- 要过程可解释：加入 CoT 或 PRM。
- 要机器可核查：再加入 Lean 或其他形式化证明器。
- 要三者统一：才需要 InternLM-Math 这类框架。

下面这个对比足够实用。

| 方案 | 优势 | 不足 | 适用场景 |
| --- | --- | --- | --- |
| 纯 CoT | 部署最简单，解释自然 | 容易算错，无法自动验证 | 教学解释、简单问答 |
| CoT + Python | 计算准确率更高 | 缺少形式化证明 | 应用题、竞赛题、数据题 |
| 证明器 Agent | 验证最严格 | 自然语言适配弱 | 形式化数学、证明搜索 |
| InternLM-Math | 推理、执行、验证统一 | 成本高，系统复杂 | 高可靠数学 agent、可核查推理 |

如果只是做一道分数题，比如 $\frac{4}{5} + \frac{3}{10}$，用 Python 求值完全足够；如果要在 MiniF2F 这类基准上追求“通过证明器的成功率”，那统一框架就开始体现价值。也就是说，InternLM-Math 不是所有数学任务的默认答案，而是“当你同时要解释、计算、验证”时的更强方案。
