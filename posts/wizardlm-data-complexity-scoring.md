## 核心结论

WizardLM 数据复杂度评分的核心不是“造更多数据”，而是“筛更值钱的数据”。它用模型给指令样本打复杂度分数，优先保留高复杂度样本做监督微调，也就是 SFT。SFT 是用“指令-答案”样本继续训练模型，让模型更稳定地按照人类指令输出。

复杂度评分要解决的问题是：在训练预算固定时，哪些样本更值得模型学习。预算固定通常指 GPU 时间、训练 token 数、人工清洗成本都有限。此时，筛选高复杂度数据的目标是提高单位 token 的信息密度，让同样的数据量包含更多推理步骤、更多约束、更复杂的输入结构。

新手版例子：`“把这段话总结成 3 句”` 通常复杂度较低，因为目标单一、输出结构简单、推理步骤少。

对比版例子：`“先提取文中 5 个论点，再按支持/反对分类，最后生成 200 字摘要”` 通常复杂度更高，因为它同时包含信息抽取、分类、格式控制和长度约束。

| 类型 | 典型任务 | 训练价值 |
|---|---|---|
| 简单样本 | 问答、单句改写、短摘要 | 基础能力 |
| 复杂样本 | 多步骤推理、多约束生成、长输入处理 | 高密度学习 |
| 低质复杂样本 | 很长但含糊、约束冲突、不可回答 | 不应直接训练 |

玩具例子：有 100 条训练样本，其中 80 条都是“翻译这句话”“总结这段话”，另外 20 条是“抽取观点、分组、排序、生成表格”。如果只能训练 20 条，复杂度评分会倾向于保留后 20 条，因为它们覆盖了更多任务结构。

真实工程例子：做一个客服大模型时，候选数据里可能有大量简单问候和单轮问答。复杂度评分可以把“多轮上下文 + 用户限制 + 订单状态判断 + 合规话术输出”的样本筛出来，使模型更快学到实际客服场景中最难的部分。

## 问题定义与边界

这里的复杂度不是“文本更长”，而是“完成任务所需的推理步骤、约束数量、输入结构复杂度”。推理步骤是指模型为了完成任务必须经过的中间判断，例如先抽取、再分类、最后生成。约束是指输出必须满足的条件，例如字数、格式、语气、字段顺序。输入结构复杂度是指输入中是否有表格、多段文本、多角色对话、代码、公式等结构。

可以把一条指令记为 $x$，复杂度函数记为：

$$
d(x) = J(x), \quad J(x) \in \{1, 2, ..., 10\}
$$

其中 $J$ 是裁判模型，也就是专门负责打分的模型。分数越高，表示指令越复杂。

工程上也可以用强弱模型差距做抽象：

$$
gap(x) = s_{strong}(x) - s_{weak}(x)
$$

其中 $s_{strong}(x)$ 是强模型在样本 $x$ 上的回答质量分，$s_{weak}(x)$ 是弱模型的回答质量分。弱模型是能力较弱或参数较少的模型，强模型是能力更强的模型。差距越大，说明这个样本更可能需要高级能力。

新手版例子：两条指令长度差不多，一条是“把这句话翻译成英文”，另一条是“把这句话翻译成英文，保留专业术语，并用表格列出原文、译文、术语解释”。后者更复杂，因为任务从单步翻译变成了多约束结构化输出。

边界版例子：一条指令很长，但只是反复说“请认真回答、请详细回答、请不要出错”。这不一定是真复杂，因为它没有增加实质任务步骤。

| 维度 | 复杂度 | 质量 |
|---|---|---|
| 含义 | 任务有多难 | 样本是否可靠、清晰、可答 |
| 作用 | 决定是否优先训练 | 决定是否值得保留 |
| 风险 | 过高可能偏离基础任务 | 低质量样本会污染训练 |
| 判断重点 | 步骤、约束、结构 | 正确性、清晰度、无冲突 |

复杂度评分只解决“训练样本筛选”，不自动保证样本质量，也不保证答案一定正确。一个样本可以很复杂，但答案是错的；也可以很简单，但非常干净、适合训练基础指令跟随能力。

## 核心机制与推导

WizardLM 的论文机制是让裁判模型给指令打分。裁判模型读入一条指令，输出 1 到 10 的难度分。这个分数可以用于排序、过滤、分桶，也可以用于后续生成更复杂的新指令。分桶是把样本按分数区间分组，例如 1-3 分为简单，4-7 分为中等，8-10 分为复杂。

原论文形式可以写成：

$$
d(x) = J(x)
$$

$$
J(x) \in \{1, ..., 10\}
$$

其中 $d(x)$ 是样本 $x$ 的复杂度，$J(x)$ 是裁判模型给出的分数。

工程上可以把复杂度评分进一步抽象为强弱模型响应差距：

$$
gap(x) = s_{strong}(x) - s_{weak}(x)
$$

过滤规则可以写成：

$$
\text{if } gap(x) \ge \tau \text{ keep, else drop or down-weight}
$$

其中 $\tau$ 是阈值。阈值是人为设定的分界线，用来决定样本是否保留。down-weight 是降权，意思是不完全丢弃样本，但降低它在训练中的出现概率或损失权重。

新手版例子：题目 A 中，弱模型答 3 分、强模型答 8 分，差距为 5。说明弱模型明显做不好，而强模型能完成，这类样本更可能包含复杂推理或多约束处理。

进阶版例子：题目 B 中，弱模型答 6 分、强模型答 7 分，差距为 1。说明两个模型都能差不多完成，样本可能只是轻微难一点，复杂度较低。

| 样本 | 弱模型分 | 强模型分 | gap | 处理 |
|---|---:|---:|---:|---|
| A | 3 | 8 | 5 | 保留 |
| B | 6 | 7 | 1 | 丢弃或降权 |
| C | 2 | 3 | 1 | 先检查质量，可能不可答 |
| D | 8 | 9 | 1 | 可能高质量但不区分强弱模型 |

这里有一个重要细节：`gap` 大不一定等于绝对复杂。样本 C 中，强模型也只有 3 分，可能说明题目本身含糊、缺信息或答案不可判定。真实系统通常要同时看复杂度和质量分，而不是只看差距。

真实工程例子：在百万级候选指令池中，可以先用弱模型和强模型分别生成答案，再用裁判模型对两个答案打分。若强模型明显优于弱模型，并且强模型答案质量达到最低标准，就把该指令标记为高价值训练样本。这样可以把预算集中到“弱模型欠缺、强模型能示范”的区域。

## 代码实现

实现目标不是训练一个新的裁判模型，而是批量给候选指令评分并排序，再按阈值或比例筛选。最小流程通常有四步：读取数据、调用 judge、保存分数、按规则过滤样本。

下面代码是可运行的 Python 玩具实现。它不调用真实大模型，而是用规则模拟 `d(x)` 和 `gap(x)` 两种接口，便于理解完整流程。

```python
from dataclasses import dataclass

@dataclass
class Sample:
    instruction: str
    answer: str = ""

class RuleJudge:
    def complexity(self, x: str) -> int:
        score = 1
        keywords = ["先", "再", "最后", "表格", "分类", "提取", "约束", "摘要"]
        score += sum(1 for kw in keywords if kw in x)
        if len(x) > 40:
            score += 2
        return max(1, min(score, 10))

    def weak_score(self, x: str) -> int:
        return max(1, 8 - self.complexity(x) // 2)

    def strong_score(self, x: str) -> int:
        return min(10, 6 + self.complexity(x) // 2)

def score_by_d(sample: Sample, judge: RuleJudge) -> int:
    return judge.complexity(sample.instruction)

def score_by_gap(sample: Sample, judge: RuleJudge) -> int:
    return judge.strong_score(sample.instruction) - judge.weak_score(sample.instruction)

def filter_top_ratio(samples, judge, keep_ratio=0.2, mode="d"):
    assert 0 < keep_ratio <= 1
    scorer = score_by_d if mode == "d" else score_by_gap

    scored = [(sample, scorer(sample, judge)) for sample in samples]
    scored.sort(key=lambda item: item[1], reverse=True)

    keep_n = max(1, int(len(scored) * keep_ratio))
    return scored[:keep_n]

def filter_by_gap_threshold(samples, judge, tau=4):
    kept = []
    for sample in samples:
        gap = score_by_gap(sample, judge)
        if gap >= tau:
            kept.append((sample, gap))
    return kept

samples = [
    Sample("把这段话总结成 3 句"),
    Sample("先提取文中 5 个论点，再按支持/反对分类，最后生成 200 字摘要"),
    Sample("把这句话翻译成英文"),
    Sample("阅读用户投诉、判断责任方、输出处理建议，并用表格列出依据"),
]

judge = RuleJudge()

top = filter_top_ratio(samples, judge, keep_ratio=0.5, mode="d")
assert len(top) == 2
assert "先提取" in top[0][0].instruction or "阅读用户投诉" in top[0][0].instruction

kept = filter_by_gap_threshold(samples, judge, tau=4)
assert all(gap >= 4 for _, gap in kept)
```

| 输入 | 输出 | 说明 |
|---|---|---|
| 候选指令集 | 每条复杂度分 | 用于排序和筛选 |
| top 20% | 高复杂度样本 | 用于后续 SFT |
| $\tau = 4$ | 保留/丢弃 | 用于二分类式过滤 |
| 抽检样本 | 人工审核结果 | 校准 judge 是否稳定 |

工程版通常会把 `score_by_d` 和 `score_by_gap` 做成两个可切换接口。论文风格更接近 `d(x)`，实现简单，适合直接按难度分排序。工程风格的 `gap(x)` 更适合发现弱模型短板，但成本更高，因为它需要弱模型、强模型和裁判模型共同参与。

## 工程权衡与常见坑

高分不等于高质量。长、乱、含糊的样本可能被打成高复杂度，但并不值得训练。质量过滤应该在复杂度筛选之前或同时进行。可回答性是指题目是否有足够信息得到合理答案；格式过滤是指样本是否满足 JSON、表格、字段完整性等基本要求。

新手版例子：一个样本要求“生成 1000 字长文”，但主题模糊、约束冲突，例如同时要求“只写 50 字”和“必须超过 1000 字”。这类样本即使看起来复杂，也不应该直接保留。

实战版例子：可以保留 80% 高复杂度样本 + 20% 简单样本。高复杂度样本提升多步骤任务能力，简单样本维持基础对话、短问答、翻译、摘要等常用能力。

| 常见坑 | 直接后果 | 规避方式 |
|---|---|---|
| 高分低质 | 训练被噪声污染 | 先做可回答性和格式过滤 |
| 只留高复杂度 | 基础能力下降 | 混入简单样本 |
| 评分不稳定 | 数据选择前后不一致 | 固定 prompt、temperature、judge |
| 只看长度 | 把啰嗦误判为复杂 | 增加步骤数和约束数特征 |
| 忽略分布 | 模型偏向少数任务 | 按任务类型分桶采样 |

还有一个常见问题是裁判偏差。裁判偏差是指 judge 模型对某些写法、领域或格式有系统性偏好。例如它可能偏爱长指令，或者偏爱带“分析、推理、步骤”字样的指令。解决方式不是完全相信分数，而是做抽检、分桶统计和小规模训练验证。

真实工程中，复杂度评分更像数据治理工具，而不是单一真理来源。合理流程是：先去重，再做质量过滤，然后做复杂度评分，最后按任务类型、语言、长度、领域进行配比。只按一个分数从高到低取样，容易得到分布很窄的数据集。

## 替代方案与适用边界

复杂度评分不是唯一的数据筛选方法。还可以用长度过滤、困惑度、覆盖率、人工标注、多模型投票等方法。困惑度是语言模型对文本“意外程度”的度量，通常越低表示模型越容易预测。覆盖率是样本集合对任务类型、领域、格式的覆盖程度。

复杂度评分最适合“已经有大量候选指令，但不知道该优先学哪些”的场景。如果数据本来就少，单纯追求高复杂度未必有效。几千条高质量指令更需要先保证干净、完整、答案正确，而不是过早筛掉简单样本。

| 方法 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 复杂度评分 | 适合筛高价值样本 | 依赖 judge 稳定性 | 大规模候选指令池 |
| 长度过滤 | 实现简单 | 不代表真实复杂度 | 快速粗筛 |
| 人工标注 | 质量高 | 成本高 | 小规模高价值数据 |
| 多模型投票 | 更稳健 | 计算成本高 | 对稳定性要求高 |
| 覆盖率采样 | 保持任务分布 | 需要标签体系 | 多任务数据集构建 |

新手版例子：如果你只有几千条高质量指令，先检查答案是否正确、格式是否统一、分类是否均衡。此时复杂度评分不是第一优先级。

进阶版例子：如果你有百万级候选指令，复杂度评分适合做第一轮粗筛。可以先筛出 top 20% 高复杂度样本，再按任务类型做分层采样。分层采样是先把样本分到不同类别，再从每个类别按比例取样，避免某一类任务占满训练集。

适用边界可以概括为三点。第一，候选数据要足够多，否则筛选收益不明显。第二，judge 要足够稳定，否则排序本身不可信。第三，最终训练集不能只追求难，还要保留基础能力、领域覆盖和答案质量。
