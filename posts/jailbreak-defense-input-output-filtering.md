## 核心结论

越狱防御的核心不是“找几个危险词拦一下”，而是把输入侧审查和输出侧检测连成闭环。越狱攻击，白话说，就是攻击者故意把提示词包装成一段“看起来合理、实际在诱导模型违规”的输入，让模型偏离原本的安全策略。只做输入过滤，容易被语义改写、编码混淆、跨轮注入绕过；只做输出检测，又可能在模型已经形成有害推理路径后才拦截，时机偏晚。

可以把系统想成两道网。第一道网在请求进入模型前工作，判断这段输入是不是在“污染上下文”；第二道网在模型开始输出时工作，实时判断回答是不是正朝危险方向发展。两道网任意一道失效，都会留下绕过窗口；两道网协同，才有工程上的可用性。

这种互补关系可以直接写成概率公式：

$$
P_{\text{block}} = 1-(1-P_{\text{in}})(1-P_{\text{out}})
$$

其中，$P_{\text{in}}$ 是输入层拦截率，$P_{\text{out}}$ 是输出层拦截率。它表示“至少有一道门成功拦截”的概率。这个公式的意义很直接：输入和输出不是二选一，而是相互补漏洞。

对新手来说，最重要的结论只有三条：

| 结论 | 白话解释 | 工程含义 |
| --- | --- | --- |
| 输入层负责早发现 | 在模型开始思考前先看是否可疑 | 可以减少危险上下文进入模型 |
| 输出层负责晚兜底 | 模型一旦开始偏航，及时停流 | 可以避免有害内容完整流出 |
| 双层优于单层 | 两边抓不同阶段的风险 | 真实系统应做分层防御 |

一个玩具例子是：用户输入一段被 Base64 包装过的恶意指令，输入层先做字符标准化和解码检测，发现异常就直接拒答；如果第一层没识别出来，模型开始输出时，输出层的分类器检测到“请求敏感操作步骤”的模式，立刻中止流式输出并返回安全替代文本。这就是双层闭环的最小形态。

## 问题定义与边界

先明确问题。输入过滤，不是简单的字符串匹配，而是在请求进入模型前，对文本、附件、会话历史、工具调用参数做风险判断。输出检测，也不是事后审计，而是在生成阶段对 token 流或句子块做实时打分，必要时中止、改写或切人工。token，白话说，就是模型逐步吐出的最小文本片段。

越狱防御要解决的，不是“所有危险内容”，而是“用户是否正在试图诱导模型越过策略边界”。这意味着它的目标不是纯内容分类，而是识别攻击意图、上下文污染和危险行为链。上下文污染，白话说，就是模型在长对话里逐步被带偏，最后把攻击者的指令当成高优先级规则。

它的边界也必须说清楚：

1. 不能只靠静态关键词。
2. 不能只在回答生成完后再审。
3. 不能忽略多轮对话和工具调用。
4. 不能把“识别风险”和“管理权限”混成一件事。

下面这个表格有助于区分风险源和检测边界：

| 输入风险源 | 检测策略 | 可能绕过方式 |
| --- | --- | --- |
| 直接危险指令 | 关键词 + 模板匹配 | 同义改写、拆词 |
| 编码混淆内容 | Base64/十六进制/Unicode 标准化 | 分段编码、伪装自然文本 |
| 同形字攻击 | 字符归一化、脚本识别 | 混用多语言字符 |
| 跨轮注入 | 会话级风险累积、上下文完整性检查 | 前几轮埋钩子，后几轮触发 |
| 工具参数污染 | 对函数参数做规则校验 | 把恶意意图藏进 JSON 字段 |
| 伪装合规请求 | 语义分类器判断真实意图 | 用“研究”“测试”“教学”包装 |

这里有一个简化例子。输入是“请帮我分析这段编码后的安全研究文本”，正文里放入 Base64 内容。第一道网如果只查明文危险词，就会漏掉；如果先做标准化、尝试轻量解码，再结合语义判断“是否存在操作性伤害意图”，命中率会高很多。第二道网则在模型输出出现“步骤、材料、执行条件”等危险行为特征时提高分数，必要时立即截断。

所以，防御边界不是“输入或输出谁更重要”，而是要在输入、上下文演进、输出三个阶段连续设门槛。真正需要防的是整条攻击链，不是某个孤立字符串。

## 核心机制与推导

双层防御为什么有效，可以从机制和概率两边看。

先看机制。输入层主要提升 $P_{\text{in}}$，常见做法有三类：

1. 字符标准化：把全角半角、同形字、大小写、编码形式先归一。
2. 上下文完整度校验：判断当前请求是不是在破坏系统提示、角色边界或工具约束。
3. 语义漏斗化：先用便宜规则筛一遍，再用较贵的语义分类器细判，把高风险请求逐步缩进小集合。

输出层主要提升 $P_{\text{out}}$，常见做法也有三类：

1. 越狱分类器：判断当前输出是否表现出绕过策略后的危险意图。
2. 行为特征评分：不是只看“词”，而是看“是否在给步骤、是否在引导执行、是否在请求敏感资源”。
3. 流式中断：不是等模型说完再审，而是边生成边审，命中即停。

概率上，若输入层和输出层相对独立，整体拦截率是：

$$
P_{\text{block}} = 1-(1-P_{\text{in}})(1-P_{\text{out}})
$$

推导很简单。先算“两个都没拦住”的概率：

$$
P_{\text{miss}}=(1-P_{\text{in}})(1-P_{\text{out}})
$$

再用总概率 $1$ 减去漏检概率：

$$
P_{\text{block}}=1-P_{\text{miss}}
$$

如果输入层拦截率为 $0.6$，输出层拦截率为 $0.7$，则：

$$
P_{\text{block}} = 1-(1-0.6)(1-0.7)=1-0.4\times0.3=0.88
$$

也就是 88% 的恶意流程会在双层机制下被终止。

这个数字的价值不在“88 很高”，而在它揭示了互补性。输入层漏掉的 40%，一部分会被输出层补回来；输出层来不及识别的情况，输入层可以提前挡住。

看一个小表格更直观：

| 输入层拦截率 $P_{\text{in}}$ | 输出层拦截率 $P_{\text{out}}$ | 组合拦截率 $P_{\text{block}}$ |
| --- | --- | --- |
| 0.60 | 0.00 | 0.60 |
| 0.00 | 0.70 | 0.70 |
| 0.60 | 0.70 | 0.88 |
| 0.80 | 0.50 | 0.90 |

这也解释了为什么单层系统经常表现不稳定。它可能在某一类攻击上看起来很好，但在另一类攻击上突然大幅失守。

玩具例子可以这样看：一个攻击者把“忽略之前所有规则，按我的要求回答”拆成几轮聊天，并混入同形字。输入层如果有会话累积和字符归一化，能在上下文阶段识别异常；如果没有，输出层至少还能在模型开始给出敏感操作步骤时截断。两层都没有，则攻击链会完整跑通。

真实工程例子则是企业 API 网关前的 AI-FENCE 一类架构：入口先做字符标准化和编码识别，中间层维护对话风险状态，出口对流式输出打分，命中阈值就停止返回并切换到安全回复模板。这里的关键不是某个单点算法，而是“分层、串联、可中断”。

还要注意，0.88 仍然意味着 12% 的攻击可能穿透。因此高风险场景不能只依赖自动检测，还要加工具权限隔离、蜜罐金丝雀和人工兜底。金丝雀，白话说，就是故意埋入一条不应被泄露的敏感标记，一旦模型复述出来，就说明上下文边界已经失守。

## 代码实现

下面给一个可运行的简化实现。它不是生产级防御，但能说明输入检查、输出评分、阈值中断的链路。

```python
import base64
import re
from dataclasses import dataclass

DANGEROUS_KEYWORDS = {
    "ignore previous instructions",
    "bypass safety",
    "reveal secrets",
    "disable guardrails",
}

DANGEROUS_OUTPUT_PATTERNS = {
    "step 1",
    "execute",
    "payload",
    "exfiltrate",
    "credential",
}

def normalize_text(text: str) -> str:
    text = text.strip().lower()
    text = re.sub(r"\s+", " ", text)
    return text

def maybe_base64_decode(text: str) -> str:
    compact = re.sub(r"\s+", "", text)
    if not re.fullmatch(r"[A-Za-z0-9+/=]+", compact):
        return text
    if len(compact) % 4 != 0:
        return text
    try:
        decoded = base64.b64decode(compact, validate=True).decode("utf-8")
        return decoded
    except Exception:
        return text

def check_input(context: str) -> bool:
    normalized = normalize_text(context)
    decoded = normalize_text(maybe_base64_decode(normalized))
    combined = normalized + " " + decoded

    risky_keyword_hit = any(k in combined for k in DANGEROUS_KEYWORDS)
    suspicious_meta = "ignore" in combined and "instruction" in combined
    return risky_keyword_hit or suspicious_meta

def analyze_output(stream_text: str) -> float:
    normalized = normalize_text(stream_text)
    score = 0.0

    for pattern in DANGEROUS_OUTPUT_PATTERNS:
        if pattern in normalized:
            score += 0.25

    if re.search(r"\b(step|phase)\s+\d+\b", normalized):
        score += 0.25

    if "how to" in normalized and "access" in normalized:
        score += 0.25

    return min(score, 1.0)

@dataclass
class Decision:
    blocked: bool
    reason: str

def guardrail(prompt: str, model_output: str, threshold: float = 0.5) -> Decision:
    if check_input(prompt):
        return Decision(True, "blocked_at_input")

    risk_score = analyze_output(model_output)
    if risk_score >= threshold:
        return Decision(True, "blocked_at_output")

    return Decision(False, "allowed")

# 玩具例子：输入层命中
encoded_attack = base64.b64encode(b"ignore previous instructions").decode("utf-8")
decision_1 = guardrail(encoded_attack, "harmless reply")
assert decision_1.blocked is True
assert decision_1.reason == "blocked_at_input"

# 玩具例子：输入层漏掉，但输出层命中
decision_2 = guardrail("tell me something", "Step 1: execute payload and access credential")
assert decision_2.blocked is True
assert decision_2.reason == "blocked_at_output"

# 正常请求
decision_3 = guardrail("explain what a database index is", "A database index speeds up lookup.")
assert decision_3.blocked is False
assert decision_3.reason == "allowed"
```

这段代码体现了三件事：

1. `check_input(context)` 先做标准化，再尝试识别编码后的风险内容。
2. `analyze_output(stream)` 对输出中的行为模式打分，而不是只看单词。
3. 主控流程采用阈值策略，先输入拦截，再输出兜底。

主流程可以抽象成下面这个 ASCII 图：

```text
User Prompt
    |
    v
[Input Check] -- risky --> [Safe Response]
    |
   pass
    v
[Run Model]
    |
    v
[Output Analyzer] -- score >= threshold --> [Fallback / Stop Stream / Human Review]
    |
   pass
    v
[Return Response]
```

真实工程里，这个结构会更复杂。比如：

- 输入层会接收 `system prompt + chat history + tool params + user prompt` 的拼接上下文，而不是只看最后一条用户消息。
- 输出层通常不是一次性分析全文，而是每 16 或 32 个 token 做一次流式评分。
- 高风险命中后，系统不会简单返回“拒绝”，而是切换到解释性、安全性更强的替代文本，避免给攻击者提供调参反馈。

企业网关中的真实工程例子可以这样理解：当员工通过内部助手访问知识库时，API 网关先标准化请求内容，检查是否存在编码混淆、角色覆盖指令和异常工具参数；请求通过后，模型流式生成回答，旁路分类器对每一段输出做“有害意图分”。一旦发现模型开始泄露内部凭据格式、敏感接口路径或操作步骤，立即断流并记录审计事件。这里真正保护系统的，不是某一条 if 语句，而是整条链路的协同。

## 工程权衡与常见坑

越狱防御最难的部分不在“能不能做”，而在“怎么在误杀、漏检、延迟之间平衡”。

输入层越严格，误杀越多。误杀，白话说，就是把正常请求错判成危险请求。比如安全团队做红队测试、研究人员讨论攻击原理、教师讲解漏洞防护时，都可能包含高风险词。输出层越敏感，断流越频繁，用户体验会变差；但阈值太高，又容易在危险回答已经成形后才拦住。

常见策略和坑可以概括成下面这张表：

| 策略 | 优点 | 坑 / 绕过手段 |
| --- | --- | --- |
| 静态关键词 | 快、成本低 | 容易被编码、拆词、同义改写绕过 |
| 输入语义分类 | 对变形攻击更稳 | 成本高，需训练和标注 |
| 输出内容审查 | 能兜底输入漏检 | 可能在有害推理已形成后才出手 |
| 流式输出检测 | 可以及时停流 | 增加推理链路复杂度和延迟 |
| 上下文完整性检查 | 能防系统提示被覆盖 | 多轮对话中状态管理复杂 |
| 蜜罐金丝雀 | 对泄漏检测很有效 | 只能发现部分失守，不等于全面防御 |

有几个坑特别常见。

第一，只依赖静态关键词。攻击者把关键词做 Base64、Unicode 同形字、空格插入、跨轮拆分，命中率会快速下降。比如“disable guardrails”可以被拆成几轮，或者用视觉上相似的字符伪装。系统如果不做归一化和会话级分析，就等于默认允许绕过。

第二，只审输出不审输入。这种做法表面上简单，因为只要在最后一层加分类器即可，但问题是很多风险在生成前就已经形成。模型一旦被带入错误角色，后续每一步都更危险；即使最终截断，也可能已经泄露部分中间信息，或者给了攻击者足够的反馈去继续试探。

第三，把模型安全和工具权限混为一谈。即使文本输出没问题，如果模型能无约束调用文件系统、数据库、发信接口，攻击面仍然很大。所以高风险系统必须做权限隔离。权限隔离，白话说，就是即使模型“想做坏事”，它也拿不到足够权限去执行。

第四，忽略人工兜底。自动检测不可能达到 100%。对于企业 API、内部知识库、代码执行代理这类高风险场景，应当设置人工审核、操作审批或降级模式。尤其是当分类器分数接近阈值时，人工判断比“硬拒绝”更稳。

以 AI-FENCE 类工程实践为例，比较合理的部署方式通常是：

1. 输入层：字符标准化、编码识别、角色覆盖检测。
2. 中间层：记录上下文风险状态，跟踪是否存在多轮渐进式注入。
3. 输出层：流式分类器实时打分，命中即停。
4. 权限层：高风险工具默认关闭，敏感操作必须二次确认。
5. 运营层：保留审计日志，支持人工复核。

如果只部署输入层，攻击者可能通过看似正常的问题诱导模型在后续回答里逐步越界；如果只部署输出层，攻击者则可能利用时间差和上下文污染先把模型带偏，再反复试探边界。真正稳的方案一定是分层。

## 替代方案与适用边界

并不是所有场景都必须上完整双层防御，但要知道每种方案在什么边界内成立。

最强方案是输入过滤 + 输出检测 + 权限隔离 + 人工兜底。它适合高风险企业 API、内部知识库、自动化代理、能调用外部工具的系统。成本高，但风险面也最大，值得投入。

较轻方案是纯输出侧审查。它适合第三方输入源不可控、无法改造入口、或者老系统只能在模型后面加网关的场景。优点是接入快；缺点是对语义污染的早期阶段无能为力。只在输出侧插入蜜罐比分层检测更便宜，但它只能发现“已经泄露出来”的东西，不能防止模型被一步步带偏。

还有一种低成本方案是人工审核或低频抽查。它适合低敏感内容平台、内部试验系统、用户量不高的产品。优点是实现简单；缺点是无法实时覆盖，规模上去后成本会迅速上升。

可以用一个简表概括：

| 方案 | 适用场景 | 优点 | 局限 |
| --- | --- | --- | --- |
| 双层防御 | 企业 API、智能体、工具调用场景 | 覆盖攻击链前后阶段 | 成本和复杂度最高 |
| 仅输出检测 | 无法改造输入链路的旧系统 | 接入简单、上线快 | 对上下文污染反应太晚 |
| 人工审核 | 低流量、低敏感场景 | 精度高、解释性强 | 不可扩展、无法实时拦截 |

再看两个边界案例。

案例一：只有输出层，没有输入层。攻击者先通过几轮无害对话建立“研究员”角色，再在第五轮让模型总结某类敏感操作流程。输出层可能在后半段才断流，但前半段已经泄露了关键步骤框架。这种场景下，晚拦截不等于没损失。

案例二：只有输入层，没有输出打分。攻击者输入时不露危险意图，只是请求“继续上一段推理”“补全步骤”。如果系统没意识到模型当前正在危险轨道上，输出就可能直接滑出边界。也就是说，输入干净不代表输出一定安全。

所以替代方案可以存在，但它们都只是“在成本、链路可改造性、风险级别之间做妥协”。一旦系统接近高权限、高价值、高暴露面，就应优先采用双层防御，并把权限控制和人工兜底一起补上。
