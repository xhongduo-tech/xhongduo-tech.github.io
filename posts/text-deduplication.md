## 核心结论

文本去重与查重，是把文档转换成稳定、可比较的表示后，判断它们是“完全相同”还是“足够相似”。

最简单的情况是精确去重：两篇文章如果每个字都一样，对全文计算哈希值后，哈希一致就判定重复。哈希是把任意长度输入压成固定长度字符串的函数，常用于快速比较内容是否完全一致。

更常见的是模糊去重：两篇文章只改了标题、删了几句话、替换了几个词，全文哈希会完全不同。这时要先把文本拆成小片段，例如连续 5 个字组成一个 `shingle`。`shingle` 是文本中的固定长度片段，用来表示文档的局部内容。然后计算两篇文档的片段重叠程度，超过阈值就认为近重复。

工程上通常使用两段式流程：

```text
标准化 -> 切分 -> 签名 -> 召回 -> 精排
```

“召回”是先快速找出可能重复的候选文档；“精排”是对候选文档做更准确但更慢的相似度计算。这样可以避免在千万篇文档之间两两比较。

| 类型 | 判断依据 | 能处理改写 | 典型方法 | 适用场景 |
|---|---|---:|---|---|
| 精确去重 | 全文是否完全一致 | 否 | MD5/SHA 哈希 | 文件归档、日志去重 |
| 模糊去重 | 相似度是否超过阈值 | 是 | shingle + MinHash/SimHash | 网页清洗、语料去重 |
| 查重/抄袭检测 | 是否存在可疑复用来源 | 是，且可重排 | 分块对齐 + 语义相似度 | 论文、作业、内容审核 |

新手版可以这样理解：两篇文章如果每个字都一样，直接判定重复；如果只改了几个词，就要先拆成小片段，计算片段重叠比例，再决定是否算重复。

## 问题定义与边界

文本去重解决的是“是否重复”，查重解决的是“是否高度相似”，但两者都依赖文本表示和阈值判断。阈值是业务设定的分界线，例如相似度大于 0.8 判定为近重复。

需要先区分应用边界。文档归档、网页清洗、训练集去重、抄袭检测看起来都在“找重复”，但目标不同，容忍误差也不同。

| 场景 | 目标 | 允许改写 | 典型方法 |
|---|---|---:|---|
| 精确去重 | 找完全相同文本 | 否 | 全文哈希 |
| 近重复去重 | 找高度相似文本 | 是 | shingle + MinHash/SimHash |
| 抄袭检测 | 找疑似来源文本 | 是，且可重排 | 分块对齐 + 语义/来源复核 |
| 训练集去重 | 降低重复样本和数据污染 | 是 | ExactSubstr + NearDup |
| 搜索引擎网页清洗 | 避免重复页面占据索引 | 是 | 指纹签名 + LSH |

完全重复像同一篇新闻被复制粘贴 10 次。近重复像同一篇新闻被改了标题、调了顺序、换了几个同义词。抄袭检测更难，因为它还要考虑改写、插入、删减和段落重排。

这里有一个重要边界：相似不等于抄袭。两段文字相似，可能来自引用、模板、公共定义、同一新闻源，也可能是抄袭。去重系统可以输出“高度相似”，但抄袭结论通常还需要来源检索、段落对齐和人工复核。

## 核心机制与推导

文本去重的基本路线是：先把文本标准化，再切成 `shingle` 或向量，最后用签名近似表示整篇文档。签名是文档的压缩表示，目标是用更短的数据保留相似度信息。

### 3.1 文本标准化

标准化是把同一含义或同一文本的不同写法统一起来。常见操作包括：

| 操作 | 例子 | 目的 |
|---|---|---|
| 大小写统一 | `Hello` -> `hello` | 避免大小写造成差异 |
| 空白统一 | 多个空格 -> 一个空格 | 避免排版差异 |
| Unicode 统一 | 全角/半角处理 | 避免编码差异 |
| 标点处理 | 删除或归一化标点 | 降低格式影响 |
| HTML 清洗 | 去掉广告、导航、脚注 | 保留正文 |

如果标准化不一致，同一篇文章可能得到不同签名，后续算法再好也会失效。

### 3.2 `shingle` 与集合表示

`shingle` 是连续的字符片段或词片段。例如文本 `abcdef`，字符级 3-shingle 是：

```text
abc, bcd, cde, def
```

一篇文档可以表示成集合 $S_d$。集合是不关心顺序、只关心元素是否存在的数据结构。用集合表示后，两篇文档的相似度就可以转化为“片段重叠比例”。

玩具例子：

```text
A = {a, b, c}
B = {b, c, d}
```

它们重叠的是 `{b, c}`，总共出现过的元素是 `{a, b, c, d}`，所以相似度是 `2/4 = 0.5`。

### 3.3 Jaccard 相似度

Jaccard 相似度衡量两个集合的重叠比例：

$$
J(d_1,d_2)=\frac{|S_{d_1}\cap S_{d_2}|}{|S_{d_1}\cup S_{d_2}|}
$$

其中 $\cap$ 表示交集，$\cup$ 表示并集。交集是两边都有的元素，并集是两边所有出现过的元素。

精确去重可以写成：

$$
H(d_1)=H(d_2)
$$

如果两个文档的全文哈希相等，就判为重复。这个判断非常快，但只能处理完全相同文本。

### 3.4 MinHash 的直觉与推导

MinHash 是一种用短签名近似 Jaccard 相似度的方法。它的核心直觉是：如果两个集合越相似，在随机排序下，它们最小元素相同的概率越高。

MinHash 的估计形式是：

$$
\hat J = \frac{1}{k}\sum_{j=1}^{k} 1[m_j(d_1)=m_j(d_2)]
$$

其中 $k$ 是签名长度，$m_j(d)$ 是第 $j$ 个哈希函数下文档 $d$ 的最小哈希值，$1[\cdot]$ 是条件成立时取 1、否则取 0 的指示函数。

新手版例子：如果做 4 维 MinHash，4 个位置里有 2 个一样，就能近似认为两篇文档相似度大约是 `2/4 = 0.5`。

### 3.5 SimHash 的直觉与推导

SimHash 是另一种局部敏感哈希。局部敏感哈希是让相似对象更可能得到相同或接近签名的哈希方法。

MinHash 更适合集合相似，SimHash 更适合向量相似。向量是把文本转换成一串数字，例如词频向量或语义向量。SimHash 的核心关系是：

$$
P[h_r(x)=h_r(y)] = 1 - \frac{\theta}{\pi}
$$

其中 $\theta$ 是两个向量的夹角。夹角越小，语义或特征方向越接近，哈希位相同的概率越高。

### 3.6 LSH 召回与精排

LSH 是 Locality-Sensitive Hashing，中文常译为局部敏感哈希。它用于大规模候选召回：不用两两比较所有文档，而是把签名相近的文档放进同一个桶。

标准流程如下：

```text
原始文本
  ↓
标准化：统一大小写、空白、编码、正文抽取
  ↓
切分：字符 shingle、词 shingle、段落块
  ↓
签名：全文哈希、MinHash、SimHash、语义向量
  ↓
召回：LSH 分桶、倒排索引、向量检索
  ↓
精排：Jaccard、编辑距离、段落对齐、人工复核
```

| 机制 | 输入表示 | 相似度含义 | 常见用途 |
|---|---|---|---|
| 全文哈希 | 完整字符串 | 是否完全一致 | 精确去重 |
| Jaccard | shingle 集合 | 局部片段重叠比例 | 近重复判定 |
| MinHash | shingle 集合签名 | 近似 Jaccard | 大规模网页去重 |
| SimHash | 特征向量签名 | 近似夹角相似 | 大规模相似检索 |
| 语义向量 | 模型编码向量 | 语义接近程度 | 改写查重 |

真实工程例子是大模型预训练语料去重。网页语料中常有镜像页面、模板页面、重复新闻和复制内容。如果不去重，模型会反复看到同一段文本，更容易记忆训练样本，也可能让评估集和训练集发生重叠。Google 的相关研究报告过，C4 中存在 near-duplicate，且有单条样本重复超过 200,000 次。去重后，训练更高效，记忆风险和数据污染风险也会下降。

## 代码实现

最小可运行思路是先实现标准化、切分、Jaccard 相似度和阈值判断。这个版本不实现完整 MinHash，但展示了去重管线的核心。

伪代码如下：

```text
输入两篇文本
统一大小写和空白
切成固定长度 shingle
计算两个 shingle 集合的相似度
如果相似度超过阈值，输出“重复”
否则输出“不重复”
```

可运行 Python 示例：

```python
import re
import unicodedata

def normalize(text):
    text = unicodedata.normalize("NFKC", text)
    text = text.lower()
    text = re.sub(r"\s+", " ", text)
    return text.strip()

def shingles(text, k=5):
    text = normalize(text)
    if len(text) < k:
        return {text} if text else set()
    return {text[i:i + k] for i in range(len(text) - k + 1)}

def jaccard(a, b):
    if not a and not b:
        return 1.0
    if not a or not b:
        return 0.0
    return len(a & b) / len(a | b)

def is_near_duplicate(t1, t2, threshold=0.8, k=5):
    s1 = shingles(t1, k=k)
    s2 = shingles(t2, k=k)
    score = jaccard(s1, s2)
    return score >= threshold, score

a = "文本去重识别并移除高度相似或重复的文档。"
b = "文本去重可以识别并移除高度相似或重复的文档。"
c = "今天北京天气晴朗，适合户外运动。"

dup, score = is_near_duplicate(a, b, threshold=0.55, k=3)
assert dup is True
assert score > 0.55

dup2, score2 = is_near_duplicate(a, c, threshold=0.55, k=3)
assert dup2 is False
assert score2 < 0.55

print(round(score, 3), round(score2, 3))
```

阈值判断示例：

| 文本对 | Jaccard 分数 | 阈值 | 结论 |
|---|---:|---:|---|
| 完全复制 | 1.00 | 0.80 | 重复 |
| 改了少量词 | 0.85 | 0.80 | 近重复 |
| 同主题不同写法 | 0.45 | 0.80 | 不重复 |
| 无关文本 | 0.05 | 0.80 | 不重复 |

工程版通常会拆成六步：

| 步骤 | 输入 | 输出 | 注意点 |
|---|---|---|---|
| 文本预处理 | HTML、Markdown、纯文本 | 干净正文 | 广告、导航、脚注要处理 |
| shingle 生成 | 正文 | 片段集合 | 中文可用字符级或词级 |
| 签名生成 | 片段集合/向量 | MinHash/SimHash | 签名越长越准但越慢 |
| 候选召回 | 签名 | 候选文档 ID | 召回阶段允许少量误报 |
| 精排判定 | 候选文本对 | 相似度分数 | 用 Jaccard、编辑距离或段落对齐 |
| 结果落库 | 判定结果 | 去重组/复核任务 | 保留来源和证据 |

一个真实工程例子：内容平台每天抓取 100 万篇网页。系统先对清洗后的正文做 SHA-256 精确去重，删除完全重复页面；再对剩余文档计算 5-shingle MinHash，用 LSH 找候选；最后对候选对计算精确 Jaccard。相似度超过 0.85 的页面进入同一个重复组，只保留来源更可靠、发布时间更早或正文更完整的版本。

## 工程权衡与常见坑

去重系统的主要难点，不是公式本身，而是标准化不一致、阈值不稳、误报漏报平衡和大规模计算成本。

误报是把不重复文本错判为重复。漏报是把重复文本漏掉。两者通常不能同时降到最低：阈值调低，召回更多，误报增加；阈值调高，结果更准，漏报增加。

```text
标准化错误 -> 同文不同签名 -> 漏报
shingle 太短 -> 普通短语大量重叠 -> 误报
shingle 太长 -> 小幅改写破坏片段 -> 漏报
阈值太低 -> 候选过多 -> 误报和成本上升
阈值太高 -> 改写文本逃过检测 -> 漏报
```

| 常见问题 | 结果 | 规避方式 |
|---|---|---|
| 只做全文哈希 | 漏掉改写重复 | shingle + MinHash/SimHash + 精排 |
| 标准化不统一 | 同文不同签名 | 统一大小写、空白、标点、Unicode |
| shingle 太短 | 误报多 | 按语料抽样标定 |
| shingle 太长 | 漏报多 | 分场景调窗口 |
| 把相似当抄袭 | 结论过度 | 段落对齐、来源检索、人工复核 |
| 训练/测试混入重复 | 评估失真 | 切分前后都做污染检查 |

新手版要记住四点：只用全文哈希，会漏掉“改了几个词”的重复；`shingle` 太短，会把很多不相干文本判成相似；`shingle` 太长，又会把改写文本漏掉；所以通常要先抽样测试，再决定阈值。

阈值调参建议：

| 业务目标 | 推荐策略 | 原因 |
|---|---|---|
| 训练语料清洗 | 召回优先，阈值略低 | 重复数据会影响模型质量 |
| 法务抄袭检测 | 证据优先，阈值不能单独定案 | 需要可解释证据链 |
| 搜索索引去重 | 平衡召回和精度 | 既要少重复，也不能误删好页面 |
| 日志/归档去重 | 精度优先 | 错删原始记录代价高 |
| 内容推荐去重 | 用户体验优先 | 近似重复内容会降低列表质量 |

训练集去重还有一个特殊问题：数据泄漏。数据泄漏是训练阶段看到了本该只在测试阶段出现的信息。比如测试集文章片段已经存在于训练集中，模型评估分数就可能虚高。因此大模型语料不仅要在训练集内部去重，还要检查训练集、验证集、测试集之间的重叠。

近重复与抄袭也不能混为一谈。近重复是统计判断，抄袭是语义、来源、结构和规则共同判断。系统可以提供相似段落、来源链接、重叠比例，但最终结论要结合业务规则。

## 替代方案与适用边界

没有一种方法适合所有场景。全文哈希适合完全重复，MinHash 适合集合相似，SimHash 适合向量相似，抄袭检测还需要更强的语义和结构信息。

新手版可以这样选：电子表格里如果两行完全一样，用哈希最快；如果是两个长网页，正文相似但标题和广告不同，用 MinHash 更合适；如果是语义接近但词面差别大，可能需要 SimHash、语义嵌入或人工复核。

| 方法 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 全文哈希 | 快、简单 | 只能找完全相同 | 精确去重 |
| MinHash | 适合集合相似 | 依赖 shingle 设计 | 网页、文档近重复 |
| SimHash | 适合向量近似 | 可解释性较弱 | 大规模相似检索 |
| 编辑距离 | 对局部改动敏感 | 计算成本高 | 短文本、精查 |
| 语义嵌入 | 能捕捉改写 | 成本高、阈值难定 | 抄袭/语义查重 |
| 人工复核 | 判断可靠 | 成本最高 | 高风险结论 |

适用边界对照：

| 对象 | 重复模式 | 推荐方法 | 不建议 |
|---|---|---|---|
| 短文本标题 | 少量字变化影响大 | 编辑距离、语义向量 | 只用长 shingle |
| 长网页正文 | 模板、广告、正文复用 | 正文抽取 + MinHash | 直接对 HTML 哈希 |
| 代码文件 | 格式变化、变量改名 | AST/Token 相似度 | 只看字符相似 |
| 训练语料 | 大量重复、跨集合污染 | 精确 + 近重复 + 污染检查 | 只在训练后评估 |
| 论文查重 | 改写、引用、段落移动 | 分块对齐 + 来源复核 | 只用一个总分定案 |

编辑距离是把一个字符串变成另一个字符串所需的最少编辑次数，包括插入、删除、替换。它适合短文本精查，不适合直接在大规模长文档上两两计算。

语义嵌入是用模型把文本编码成向量。它能发现“词面不同但意思接近”的文本，例如“模型记住训练样本”和“模型对训练文本产生记忆”。但语义向量也更难解释，阈值更难稳定，且成本更高。

实际系统常用组合方案：

```text
精确哈希删除完全重复
  ↓
MinHash/SimHash 找近重复候选
  ↓
Jaccard/编辑距离/段落对齐精查
  ↓
高风险场景进入人工复核
```
