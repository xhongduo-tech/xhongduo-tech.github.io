## 核心结论

PIQA（Physical Interaction Question Answering）是一个面向日常物理常识推理的二选一问答数据集。它测试的不是“术语记住了没有”，而是“这个目标在现实世界里能不能这样完成”。这里的“物理常识”可以先理解成：人在不做实验的前提下，凭日常经验也能做出的可行性判断，比如“玻璃杯能装水”“纸杯接近明火容易损坏”“锤子适合敲钉子，尺子通常不适合”。

PIQA 的任务形式非常固定：给定一个 `goal`，再给出两个候选方案 `sol1` 和 `sol2`，模型必须选出唯一可行的那个。形式化写法如下：

$$
x = (goal, sol1, sol2), \qquad y \in \{0,1\}
$$

其中常见约定是：

- `label = 0` 表示 `sol1` 更可行
- `label = 1` 表示 `sol2` 更可行

也可以把它看成一个二分类问题。模型输入是三段文本，输出是两个类别中的一个：

$$
\hat{y} = \arg\max_{i \in \{0,1\}} P(y=i \mid x)
$$

这类任务的价值，在于它能暴露模型是否真正理解“物体能做什么”“动作会带来什么后果”“方案是否稳定、安全、符合材料属性”。如果模型只会做语言模式匹配，而不会推断支撑、摩擦、热量、硬度、形状、容积这些约束，它在 PIQA 上通常会出错。

一个最小“玩具例子”如下。

| 目标 | 方案 A | 方案 B | label |
|---|---|---|---|
| 将湿毛巾拧干 | 先揉搓再甩 | 直接放在桌上静置 | A 可行 |

这道题的关键不在词汇，而在动作后果。静置不会主动把水排出去；甩动会让一部分水在惯性作用下离开毛巾，所以 A 更符合目标。

再看两个更贴近日常操作的例子：

| 目标 | 方案 A | 方案 B | 正确判断 |
|---|---|---|---|
| 把钉子打进木板 | 用锤子敲钉子头部 | 用塑料尺平拍钉子侧面 | A |
| 把热锅从灶台上移开 | 戴隔热手套端起锅柄 | 直接徒手抓住锅沿 | A |

PIQA 的意义不在于题面是否华丽，而在于它强迫模型从“词像不像答案”转向“世界是否真的这样运行”。这也是它常被用来评估大模型、机器人规划模块、家庭自动化助手基础常识能力的原因。官方论文中，人类在该数据集上的准确率约为 95%，而早期强基线模型明显更低，这说明“物理可行性判断”并不是纯语言统计可以轻易替代的能力。

## 问题定义与边界

PIQA 的单条样本通常包含四个字段：`goal`、`sol1`、`sol2`、`label`。可以把它理解成一个标准的二分类任务，只不过类别不是“正例/负例”，而是“哪个方案物理上更可行”。

一个标准条目可以写成下面这样：

| 字段 | 内容 |
|---|---|
| `goal` | 把罐子里水煮沸 |
| `sol1` | 将水倒入耐热容器后加热 |
| `sol2` | 先把容器完全密封再持续加热 |
| `label` | `0` |

如果采用 `label = 0` 表示 `sol1` 可行，那么这题的含义就是：更合理的方案是使用能承受加热的容器并正常加热；“先完全密封再持续加热”可能导致压力积累，不是这里想要的标准可行解。

公开实现里常见的数据划分如下：

| split | 数量 | 作用 |
|---|---:|---|
| train | 16113 | 训练模型参数 |
| validation | 1838 | 调参与早停 |
| test | 3084 | 最终评测 |

这里有两个边界必须说清。

第一，PIQA 判断的是“可行性”，不是“最优性”。

“可行性”指的是：这个方案在现实中能把目标完成，且不明显违背基本物理约束。  
“最优性”则是在可行的方案里继续比较谁更省时、省力、省钱、更安全。

例如：

| 场景 | 可行性问题 | 最优性问题 |
|---|---|---|
| 搅拌糖水 | 用勺子搅拌能否完成目标 | 木勺、金属勺、电动搅拌器哪种更省力 |
| 打钉子 | 锤子敲击是否可行 | 哪种锤头、角度、力度效率最高 |
| 搬热锅 | 隔热手套是否可行 | 哪种手套材料耐温更高、更舒适 |

很多模型的一个系统性错误，就是把任务做成“选更高级的”“选更像教科书的”“选更现代的”，而不是判断“能不能做成”。这会直接偏离 PIQA 的定义。

第二，PIQA 判断的是“语言描述下的日常物理常识”，不是严格的数值物理计算。

它通常不要求模型算出：

- 精确受力大小
- 材料熔点的具体数值
- 热传导方程的求解结果
- 流体动力学的定量模拟

它要求的是更基础的层面：材料会不会坏、动作方向对不对、支撑是否稳定、工具是否匹配、风险是否明显。

例如：

| 目标 | 方案 A | 方案 B | 需要的知识 |
|---|---|---|---|
| 把书立起来 | 用书挡支撑 | 用湿毛巾垫底后竖放 | 支撑、摩擦、稳定性 |
| 避免烫伤 | 用布隔开热锅 | 用手直接摸锅盖 | 导热与安全常识 |
| 擦干玻璃 | 用干布擦拭 | 再泼一杯泥水 | 动作后果与目标一致性 |

再举一个边界清晰的例子：

| 目标 | 方案 A | 方案 B | 最终可行性标注 |
|---|---|---|---|
| 把钉子打进木板 | 用锤子敲钉子头部 | 用塑料尺平拍钉子侧面 | A |

这题不要求你知道锤子的品牌、质量、施力角度，也不要求你估算多少牛顿。它只要求模型知道：锤子敲击钉头是符合常识的，塑料尺拍钉子侧面通常既不稳定，也很难把力有效传到目标方向上。

对新手来说，可以把 PIQA 记成一句话：  
它问的不是“哪种方法最好”，而是“哪种方法至少能做成，而且不会明显违背现实世界的基本约束”。

## 核心机制与推导

PIQA 不是把众包题目原样堆起来就结束了。它的重要机制之一，是用对抗过滤去减少“词汇捷径”。公开资料里常见的说法是使用 AFLite（Adversarial Filtering Lite）。

这里可以先用最朴素的话理解 AFLite：  
先让一批较简单的模型去做题，如果某些题只靠表面词汇就能稳定猜中，那这些题对真正的推理能力帮助不大，就倾向于被筛掉；保留下来的题，应该更难被“背词频”解决。

为什么需要这一步？因为许多数据集都会出现偏置。比如：

- 某些词一出现，正确答案经常在 `sol1`
- 某种句式一出现，错误答案经常更长
- 某些“听起来专业”的词容易被模型误当成正确线索

如果不做过滤，模型就可能学会“押题技巧”，而不是学会物理常识本身。

可以用一个非官方、但便于理解的抽象公式描述这种思路：

$$
\text{keep}(x) \propto 1 - \text{easy\_bias}(x)
$$

其中：

- $\text{easy\_bias}(x)$ 越大，表示样本越容易被浅层模式猜中
- $\text{keep}(x)$ 越大，表示样本越值得保留

更直观一点，也可以把浅层模型对样本的“可猜性”写成：

$$
\text{easy\_bias}(x) = \frac{1}{M}\sum_{m=1}^{M}\mathbf{1}\big[f_m(x)=y\big]
$$

这里：

- $M$ 是若干个简单模型或弱分类器
- $f_m(x)$ 是第 $m$ 个模型对样本 $x$ 的预测
- $\mathbf{1}[\cdot]$ 是指示函数，猜对记作 1，否则记作 0

如果一个样本被很多浅层模型轻易猜中，它更可能带有明显的表面偏置；如果这些模型经常猜不对，那么这个样本更可能要求真实的推理。

看一个简化例子：

| 候选 | 表面词汇情况 | 简单模型是否容易猜对 | AFLite 倾向 |
|---|---|---|---|
| A：把钉子固定，方案一“敲钉子头部”，方案二“敲木板边缘” | 两个方案都含“敲” | 不容易只靠词判断 | 保留 |
| B：让东西着火，方案一含“点燃”，方案二含“放进水里” | 关键词极强 | 很容易靠词猜 | 可能过滤 |

这一步的目标，不是让题目变“刁钻”，而是减少模型利用数据集漏洞。PIQA 希望模型不要只学到“哪个词像正确答案”，而要学到“哪个方案在现实里站得住”。

对新手来说，可以把这个过程理解成两层筛选：

1. 众包先产生“看起来合理”的两个方案。
2. 过滤步骤再剔除“太容易被表面特征识别”的样本。

这样留下来的题，通常更要求模型比较动作后果，而不是比较措辞漂亮程度。

下面给一个新手更容易理解的玩具推导。

假设目标是“把玻璃窗擦干净”：

- 方案 A：用干净抹布擦拭
- 方案 B：往玻璃上再泼一杯泥水

若模型只看词汇相关性，它可能只注意到两个方案都和“玻璃”“水”“清洁动作”有关；但真正的判断应该看目标是否被满足。A 会减少污渍和水迹，B 会增加污渍，因此 A 才是可行解。

从状态变化角度写，可以更清楚：

$$
s_{t+1} = T(s_t, a)
$$

其中：

- $s_t$ 表示当前世界状态
- $a$ 表示方案里的动作
- $T$ 表示动作导致的状态转移

PIQA 虽然不显式给出状态图，但本质上要求模型做的，就是比较两个动作谁更可能把状态推向目标：

$$
\text{choose } a^\* = \arg\max_{a \in \{sol1, sol2\}} \text{GoalSatisfied}(T(s_t, a))
$$

这也是为什么 PIQA 对机器人规划、家庭自动化、具身智能前端模块有参考价值。机器人在执行前，首先就要回答一个比动作控制更早的问题：  
“这个方案会不会把世界推向目标状态？”  
如果这个判断本身不稳，后续视觉识别、路径规划、机械执行都会被错误决策拖累。

PIQA 的另一特点，是题目来源偏向日常操作场景，灵感来自 Instructables 这类实操文本。它关注的是“怎样利用现有物体完成一个具体动作”，而不是百科定义问答。因此它更像“行动前判断”，而不是“概念回忆”。

## 代码实现

工程上，PIQA 最常见的做法，是把 `goal`、`sol1`、`sol2` 拼成模型输入，然后输出两个类别的分数。这里的“拼接”可以先理解成：把分散字段组织成模型一次能读懂的上下文。

如果是预训练语言模型微调，常见形式如下：

```python
input_text = f"Goal: {goal}\nOption1: {sol1}\nOption2: {sol2}"
logits = model(input_text)
loss = cross_entropy(logits, label)
```

若 logits 为 $z_0, z_1$，则 softmax 概率为：

$$
P(y=i\mid x) = \frac{e^{z_i}}{e^{z_0}+e^{z_1}}
$$

最终预测类别为：

$$
\hat{y} = \arg\max_{i \in \{0,1\}} z_i
$$

为了把数据结构、标签约定、评测方式讲清楚，下面先给一个可直接运行的 Python 示例。它不训练神经网络，而是用非常粗糙的规则模拟“可行性判断”。目的不是追求高准确率，而是让新手先把任务接口和评测流程跑通。

```python
from __future__ import annotations

from dataclasses import dataclass
from typing import Iterable, List


@dataclass(frozen=True)
class PIQASample:
    goal: str
    sol1: str
    sol2: str
    label: int  # 0 -> sol1 feasible, 1 -> sol2 feasible


POSITIVE_CUES = {
    "锤子": 3,
    "敲钉子头": 4,
    "隔热手套": 4,
    "抹布": 2,
    "擦拭": 2,
    "加热": 2,
    "耐热容器": 3,
    "支撑": 2,
    "固定": 2,
    "拧": 2,
    "甩": 2,
}

NEGATIVE_CUES = {
    "徒手抓": -5,
    "塑料尺": -3,
    "拍钉子侧面": -4,
    "静置": -2,
    "泼泥水": -4,
    "完全密封再加热": -5,
    "塑料杯挡火花": -4,
    "直接碰火": -5,
}


def score_solution(goal: str, solution: str) -> int:
    text = f"{goal} {solution}"
    score = 0

    for cue, value in POSITIVE_CUES.items():
        if cue in text:
            score += value

    for cue, value in NEGATIVE_CUES.items():
        if cue in text:
            score += value

    return score


def predict_label(sample: PIQASample) -> int:
    s1 = score_solution(sample.goal, sample.sol1)
    s2 = score_solution(sample.goal, sample.sol2)
    return 0 if s1 >= s2 else 1


def accuracy(samples: Iterable[PIQASample]) -> float:
    samples = list(samples)
    correct = sum(int(predict_label(x) == x.label) for x in samples)
    return correct / len(samples)


def explain_prediction(sample: PIQASample) -> str:
    s1 = score_solution(sample.goal, sample.sol1)
    s2 = score_solution(sample.goal, sample.sol2)
    pred = predict_label(sample)
    chosen = "sol1" if pred == 0 else "sol2"
    return (
        f"goal={sample.goal!r}\n"
        f"sol1 score={s1}: {sample.sol1}\n"
        f"sol2 score={s2}: {sample.sol2}\n"
        f"pred={chosen}, gold={sample.label}\n"
    )


def main() -> None:
    dataset: List[PIQASample] = [
        PIQASample(
            goal="将湿毛巾拧干",
            sol1="先揉搓再甩",
            sol2="直接放在桌上静置",
            label=0,
        ),
        PIQASample(
            goal="把钉子打进木板",
            sol1="用锤子敲钉子头部",
            sol2="用塑料尺平拍钉子侧面",
            label=0,
        ),
        PIQASample(
            goal="把热锅从灶台上移开",
            sol1="戴隔热手套握住锅柄后端走",
            sol2="直接徒手抓住锅沿抬起来",
            label=0,
        ),
        PIQASample(
            goal="把玻璃窗擦干净",
            sol1="用干净抹布擦拭表面",
            sol2="往玻璃上再泼一杯泥水",
            label=0,
        ),
        PIQASample(
            goal="把水煮开",
            sol1="将水倒入耐热容器后加热",
            sol2="把容器完全密封再持续加热",
            label=0,
        ),
    ]

    for sample in dataset:
        print(explain_prediction(sample))

    acc = accuracy(dataset)
    print(f"toy accuracy = {acc:.2%}")


if __name__ == "__main__":
    main()
```

这段代码可以直接运行。它体现了四个关键点：

1. 数据单位就是 `goal + sol1 + sol2 + label`
2. 输出是二分类标签
3. 评测本质是比较两个方案谁更可行
4. 标签语义必须固定，否则训练和评测会错位

运行后你会得到类似输出：

```text
goal='将湿毛巾拧干'
sol1 score=4: 先揉搓再甩
sol2 score=-2: 直接放在桌上静置
pred=sol1, gold=0
...
toy accuracy = 100.00%
```

当然，这个“规则打分器”不是真实模型，只是教学用骨架。它的局限非常明显：

| 局限 | 具体表现 |
|---|---|
| 依赖手工关键词 | 看不到关键词就无法泛化 |
| 没有真正建模因果 | 不能理解复杂动作链条 |
| 不会处理同义表达 | “拿布擦”与“抹布擦拭”可能被当成不同模式 |
| 无法利用上下文细节 | 目标一变，规则可能立刻失效 |

如果想做更接近真实工程的原型，最小流程通常如下：

```python
def build_input(goal: str, sol1: str, sol2: str) -> str:
    return (
        f"Task: choose the physically feasible solution.\n"
        f"Goal: {goal}\n"
        f"Option 1: {sol1}\n"
        f"Option 2: {sol2}"
    )

# 伪代码
text = build_input(goal, sol1, sol2)
logits = encoder_classifier(text)   # shape: [2]
pred = int(logits.argmax())
```

如果你用 Hugging Face `datasets` 读取 PIQA，字段通常也是这四列：

```python
sample = {
    "goal": "how do you dry a wet towel",
    "sol1": "twist and swing it",
    "sol2": "leave it on the table",
    "label": 0,
}
```

对新手来说，最重要的工程纪律只有一条：  
**把 `label=0 -> sol1`、`label=1 -> sol2` 写死在代码和测试里，不要靠记忆。**

推荐至少补三类断言：

```python
assert predict_label(
    PIQASample("把钉子打进木板", "用锤子敲钉子头部", "用塑料尺平拍钉子侧面", 0)
) == 0

assert predict_label(
    PIQASample("把玻璃窗擦干净", "用干净抹布擦拭表面", "往玻璃上再泼一杯泥水", 0)
) == 0

assert PIQASample("x", "a", "b", 1).label in (0, 1)
```

在真实项目里，一个常见错误不是模型太弱，而是数据脚本把 `sol1/sol2` 或 `label` 对反了。这个错误一旦发生，训练损失可能照样下降，但评测结果会严重失真，而且很难第一时间发现。

## 工程权衡与常见坑

PIQA 最常见的问题，不是模型“不会读字”，而是模型把任务理解错了。下面几类坑在真实工程里最常见。

第一类坑是关键词陷阱。

所谓“关键词陷阱”，就是模型把某些看起来像正确操作的词当成答案信号，却没有检查它在当前场景里是否真的成立。

例如：

| 场景 | 错误直觉 | 实际判断 |
|---|---|---|
| 用塑料杯挡住火花 | “挡住”“火花”都像相关操作 | 不可行，塑料可能受热变形或熔化 |
| 把热锅移开 | “抓住”“拿开”都和目标有关 | 徒手抓锅沿不可行，存在明显烫伤风险 |
| 清理玻璃 | “再加水”看起来像清洗动作 | 泼泥水会让目标更难完成 |

第二类坑是把“可行”误当成“最优”。

PIQA 只要求区分“能做”和“不能做”，不要求“最好”。如果你把模型目标隐含地改成“选最先进方案”“选最省力方案”，模型就会学歪。

| 错误设定 | 后果 |
|---|---|
| 把“高效”当成“正确” | 模型可能错判那些虽然笨但能完成目标的方法 |
| 把“专业工具”当成“正确” | 模型会低估生活中非典型但可行的方案 |
| 把“更安全”机械等同于“唯一可行” | 模型可能忽视多个可行解之间的差异 |

第三类坑是材料属性缺失。

PIQA 中很多判断都依赖“材料会怎样”。如果模型没有材料概念，误判会很频繁。

| 材料属性 | 常见后果 | 典型误判 |
|---|---|---|
| 塑料怕高温 | 变形、熔化 | 用塑料容器直接接触明火 |
| 纸会吸水 | 软化、破损 | 用纸稳固承重或长期盛液体 |
| 玻璃易碎 | 冲击下破裂 | 把玻璃当敲击工具 |
| 金属导热快 | 可能烫手 | 直接抓热金属边缘 |

第四类坑是只看当前动作，不看动作链条。

有些方案局部动作看起来相关，但完整动作链并不能导向目标。

例如“把桌面清理干净”：

- 用纸巾擦掉污渍，目标状态是更干净
- 用报纸盖住污渍，视觉上被遮住，但污渍没有消失

这类题如果只看“动作像不像清理”，就会错；必须看动作之后状态如何变化。

第五类坑是目标约束被忽略。

同一个动作在不同目标下，结论会变。  
“把书页弄干”与“把书页弄干且保持平整”并不是同一道题。

| 目标 | 同一个动作的判断 |
|---|---|
| 只要求变干 | 通风晾晒可能可行 |
| 要求变干且不变形 | 暴晒或高温可能不再是好答案 |

把这些问题并列起来，可以更容易诊断：

| 坑 | 典型表现 | 对策 |
|---|---|---|
| 关键词陷阱 | 看见“固定”“挡住”“加热”就高分 | 显式比较动作后果，而不是词本身 |
| 最优化误区 | 选择“更高级”而不是“能完成” | 在训练和评测时明确目标是 feasibility |
| 材料属性缺失 | 忽略塑料会熔化、纸会吸水、玻璃会碎 | 补充材料常识，或引入外部知识 |
| 单步替代多步 | 只看动作，不看状态变化 | 增加中间状态推理 |
| 目标约束丢失 | 忽略“安全”“平整”“不损坏”等限制 | 把目标条件完整保留进输入 |

如果目标是提升工程可用性，常见增强方向包括：

- 使用更完整的输入模板，把目标约束、方案细节、潜在风险一起输入
- 给模型增加外部常识来源，补足材料、工具、动作之间的关系
- 引入中间推理步骤，先判断“材料属性”“受力方向”“安全性”“动作后果”，再做最终二选一
- 做错误分析时按类型切分，而不是只看总准确率

但也要注意，系统越复杂，排错越难。对于一个基础评测任务，先把下面几件事做对，通常比盲目叠加复杂模块更重要：

1. 标签语义固定。
2. 输入模板一致。
3. 评测脚本无歧义。
4. 错误案例可回溯。

## 替代方案与适用边界

PIQA 很有用，但它不是“物理智能”的全部。它本质上仍然是纯文本、二选一任务，因此更适合评估“语言描述下的物理常识判断”，不适合直接证明模型已经具备真实世界操作能力。

一个常见替代方向是多模态扩展。这里的“多模态”可以先理解成：不仅给文字，还给图片、视频、深度信息或传感器数据，让模型同时利用语言和视觉线索。

例如：

- 文字版：如何把螺丝拧紧
- 多模态版：给出螺丝刀类型、螺丝头照片、目标位置视频，再判断方案可行性

这类方案更接近机器人和家庭自动化的真实场景，但它不能完全替代 PIQA。原因很简单：视觉输入会替模型补充很多环境细节，而 PIQA 的价值恰恰在于检查“只靠语言描述时，你是否已经具备基础物理常识”。

下面是几个常见替代方向及其启用时机：

| 替代方案 | 核心思路 | 优点 | 何时启用 |
|---|---|---|---|
| 视觉增强 | 给每条方案补图片或视频 | 更接近真实场景 | 机器人指令理解、家庭场景执行 |
| 常识图谱/规则库 | 显式加入材料、工具、动作关系 | 可解释性更强 | 错误主要来自材料属性缺失 |
| 中间推理链 | 先判断受力、稳定性、安全性，再决策 | 更容易做错误诊断 | 需要分析错因而不只看分数 |
| 大模型提示工程 | 用 prompt 让模型逐步思考 | 启动快、改造成本低 | 原型验证、低成本 baseline |
| 真实仿真/机器人测试 | 在模拟器或实体设备上执行 | 最接近终局能力 | 已通过文本与视觉层面的前置筛查 |

一个实用的边界判断是：

- 如果你要评估“模型有没有基本物理常识”，PIQA 很合适
- 如果你要评估“机器人能不能真的完成任务”，PIQA 只能算前置筛查，不是最终测试
- 如果你要评估“模型能否利用视觉场景做判断”，那就应该引入多模态基准
- 如果你要定位“模型为什么错”，那就需要中间推理或规则诊断，而不是只看最终标签

真实工程流程通常更像下面这样：

1. 先在 PIQA 这类纯文本基准上测基础准确率。
2. 再做带视觉输入的方案判断。
3. 再进入动作规划、路径规划和安全约束。
4. 最后才做真实执行或仿真执行。

这个顺序的原因很直接：前一层都判断不稳，后一层只会把错误放大。  
对零基础工程师来说，可以记住一句话：

**PIQA 测的是“会不会判断一个方案是否站得住”，不是“系统最终能不能把任务做完”。**
