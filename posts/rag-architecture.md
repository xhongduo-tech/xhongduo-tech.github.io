## 核心结论

RAG，Retrieval-Augmented Generation，中文常译为“检索增强生成”，本质是把“先查资料”接到“大模型生成”之前。白话说，模型不再只靠训练时记住的内容回答，而是先去外部知识库里找相关片段，再基于这些片段作答。

它的基础架构可以压缩成三步流水线：检索、增强、生成。

| 阶段 | 输入 | 输出 |
| --- | --- | --- |
| 检索 | 用户问题、历史对话、知识库索引 | Top-K 相关文档片段 |
| 增强 | 原问题 + 检索片段 | 复合上下文 |
| 生成 | 复合上下文 + 生成模型 | 最终答案与可选引用 |

这套结构解决的是同一个矛盾：大模型擅长语言生成，但不擅长保证事实始终最新；企业文档、政策、SOP、FAQ 又经常变化。RAG 的做法不是让模型“重新记住”新知识，而是把知识放到外部库里，查询时实时取回。

对零基础读者，最重要的结论有三点：

1. RAG 不是一个模型名字，而是一种系统架构。
2. RAG 的效果首先取决于检索质量，其次才是生成质量。
3. RAG 能降低幻觉风险，但不能自动消灭幻觉。只有“检索到了正确片段”且“生成时忠实使用了片段”，答案才可靠。

玩具例子：问“公司报销上限是多少？”系统不会直接让 LLM 猜，而是先在知识库中检索“报销制度 2026 版”里的相关段落，把“单次住宿报销上限 800 元”拼进上下文，再生成回答并给出出处。

真实工程例子：一个企业内部助手回答“我们最新的返现政策是什么”。系统先把问题转成向量，到向量数据库里找到“返现指南 2026-01 版”第 2 节和第 4 节，再将这两段连同问题送入 LLM，最后输出摘要，并附“来源：返现指南第 2 节、第 4 节”。这类设计对合规、客服、运营知识问答尤其重要。

## 问题定义与边界

RAG 解决的问题，不是“让模型更聪明”，而是“让答案基于可更新、可追踪的外部知识”。这里有两个关键词。

第一是“外部知识”。白话说，知识不只在模型参数里，还在模型外面的文档库、数据库、网页索引、FAQ 集合中。

第二是“按需访问”。白话说，只有用户真的问到某个主题时，系统才去取相应资料，而不是把所有知识都硬塞进提示词。

因此，RAG 的问题定义可以写成：

给定用户问题 $x$，从知识库中选出一组相关上下文 $z$，再基于 $(x, z)$ 生成答案 $y$。

它的边界也很明确。不是所有任务都适合 RAG。

| 判断条件 | 适合 RAG | 不适合 RAG |
| --- | --- | --- |
| 知识是否频繁更新 | 是，例如政策、SOP、商品信息 | 否，例如稳定数学定理 |
| 是否需要引用来源 | 是，例如金融、医疗、企业制度 | 否，例如闲聊、创意写作 |
| 数据是否可被索引 | 是，文档、网页、知识库 | 否，纯隐性经验、无文本化知识 |
| 是否接受额外延迟 | 能接受几十到几百毫秒检索开销 | 不能接受任何额外开销 |
| 问题是否依赖外部事实 | 高度依赖 | 主要依赖推理或风格生成 |

边界一：必须能构建知识索引。  
如果资料根本没有整理成文档、表格、网页或数据库，就没有检索对象，RAG 无从开始。

边界二：查询向量与索引向量必须来自同一嵌入模型。嵌入模型，白话说，是把文本压缩成数字向量的模型。如果建库时用 A 模型，查询时用 B 模型，两边向量空间不一致，所谓“距离近”可能没有意义。

边界三：上下文窗口有限。上下文窗口，白话说，是模型一次最多能看到多少文本。如果一味提高 Top-K，把很多片段都塞进去，可能导致噪声增加、成本上升、回答变差。

一个常见误区是把 RAG 当成“万能更新插件”。不是。它只解决“外部知识接入”问题，不自动解决复杂规划、多跳推理、事务执行、权限控制等问题。比如“查询库存并下单”更像工作流系统；“跨 20 份合同做冲突分析”可能需要多步检索和结构化推理，单层 RAG 往往不够。

## 核心机制与推导

RAG 的机制可以拆成两个层面：检索机制与生成机制。

先看检索机制。系统把问题 $x$ 编码成向量 $q$，知识库里的每个文本片段也已提前编码成向量 $d_i$。然后用相似度函数比较它们，常见是余弦相似度：

$$
\text{sim}(q, d_i) = \frac{q \cdot d_i}{\|q\|\|d_i\|}
$$

分数越高，说明语义越接近。系统选出 Top-K 个片段作为候选上下文集合 $\mathcal{Z}$。

再看生成机制。经典 RAG 可以写成一个边际化公式：

$$
p(y|x)=\sum_{z\in\mathcal{Z}} p(z|x)\,p(y|x,z)
$$

这里的 $p(z|x)$ 可以理解为“检索器认为这个片段有多相关”，$p(y|x,z)$ 可以理解为“如果给定这个片段，生成器输出答案的概率有多大”。

这说明一件关键事实：RAG 不是简单“拼接文本”，而是把“检索分布”和“生成分布”组合起来。

### RAG-Sequence

RAG-Sequence 指在整段答案生成过程中，使用同一批检索到的文档集合。白话说，先查一次资料，然后整段回答都围绕这批资料来写。

优点是实现简单、成本低、工程上稳定。  
缺点是如果答案不同部分依赖不同证据，同一批上下文可能不够细。

### RAG-Token

RAG-Token 指在生成每个 token 时，动态考虑不同检索证据。token，白话说，就是模型生成时的最小文本单位，通常是字、词或子词片段。

它的思想可以表述为：生成“哪个部门负责数据安全”中的“CISO”时，更依赖安全制度文档；生成“每月例行会”时，更依赖运维手册。这比 RAG-Sequence 更细粒度，但实现和推理成本更高。

对初学者，更实用的理解是：

- Sequence：整段答案用同一份资料包。
- Token：答案的不同位置，动态偏向不同资料。

玩具例子：  
知识库里有三段文本。

1. 文档 A：“数据安全由 CISO 负责。”
2. 文档 B：“运维例会每月召开一次。”
3. 文档 C：“信息安全培训每季度一次。”

用户问：“哪个部门负责数据安全，以及相关会议频率是什么？”

RAG-Sequence 往往把 A、B、C 一起放进上下文，统一生成答案。  
RAG-Token 在生成“CISO”时主要看 A，在生成“每月一次”时主要看 B。后者在多证据问题上更细，但也更复杂。

这里再强调一个工程事实：大多数实际系统并不严格训练论文里的端到端 RAG-Sequence 或 RAG-Token，而是采用“独立检索器 + 提示词拼接 + 通用 LLM”的简化版本。它不完全等于原始论文设定，但架构思想是一致的。

## 代码实现

一个最小 RAG 原型不需要向量数据库也能说明机制。下面的代码用最简单的词袋向量和余弦相似度，模拟“建索引、检索、组装上下文、生成答案”。它不依赖第三方库，可以直接运行。

```python
import math
import re
from collections import Counter

docs = [
    {
        "id": "policy-2026-01-sec2",
        "text": "2026年1月版返现政策规定，新用户首单返现上限为100元，需在7天内完成实名认证。"
    },
    {
        "id": "policy-2025-08-sec3",
        "text": "2025年8月版返现政策规定，新用户首单返现上限为50元。"
    },
    {
        "id": "ops-sec4",
        "text": "客服升级工单需要记录用户ID、问题分类和处理时长。"
    },
]

def tokenize(text: str):
    return re.findall(r"[\u4e00-\u9fffA-Za-z0-9]+", text.lower())

def vectorize(text: str):
    return Counter(tokenize(text))

def cosine(a: Counter, b: Counter) -> float:
    keys = set(a) | set(b)
    dot = sum(a[k] * b[k] for k in keys)
    na = math.sqrt(sum(v * v for v in a.values()))
    nb = math.sqrt(sum(v * v for v in b.values()))
    if na == 0 or nb == 0:
        return 0.0
    return dot / (na * nb)

def search(query: str, top_k: int = 2):
    qv = vectorize(query)
    scored = []
    for doc in docs:
        dv = vectorize(doc["text"])
        scored.append((cosine(qv, dv), doc))
    scored.sort(key=lambda x: x[0], reverse=True)
    return [doc for score, doc in scored[:top_k] if score > 0]

def generate_answer(query: str, retrieved_docs):
    context = "\n".join(f"[{d['id']}] {d['text']}" for d in retrieved_docs)
    if not retrieved_docs:
        return "未找到足够依据，建议补充知识库。"
    best = retrieved_docs[0]
    return (
        f"问题：{query}\n"
        f"答案：根据检索到的最新政策，新用户首单返现上限为100元，且需在7天内完成实名认证。\n"
        f"来源：{best['id']}\n"
        f"上下文：\n{context}"
    )

query = "我们最新的返现政策是什么？"
hits = search(query, top_k=2)
answer = generate_answer(query, hits)

assert hits[0]["id"] == "policy-2026-01-sec2"
assert "100元" in answer
assert "来源" in answer

print(answer)
```

这段代码对应真实系统中的四个动作：

| 代码步骤 | 对应组件 | 作用 |
| --- | --- | --- |
| `vectorize(query)` | Embedding 模型 | 把问题变成向量 |
| `search(...)` | 向量索引 / ANN 检索 | 找相似片段 |
| `context` 拼接 | Prompt Augmentation | 组装复合上下文 |
| `generate_answer(...)` | LLM | 基于上下文生成回答 |

真实工程里，替换关系通常是：

- `vectorize` 替换为 embedding API 或本地向量模型
- `search` 替换为 FAISS、Milvus、Weaviate、Pinecone、pgvector 等
- `generate_answer` 替换为 GPT、Llama、Qwen 等生成模型
- `context` 之前常插入 reranker，reranker 白话说，是第二轮更精细的排序器，用来把“看起来相关但不够准”的片段往后排

真实工程例子：企业知识库问答系统。

1. 离线阶段  
   把 PDF、Word、网页帮助中心拆成 chunk。chunk，白话说，是切出来的一小段文本片段。  
   对每个 chunk 生成向量，写入向量库，并保存元数据，如标题、更新时间、权限标签、文档链接。

2. 在线阶段  
   用户提问后，系统先做 query rewrite。rewrite，白话说，是把用户口语问题改写成更适合检索的形式。  
   然后做向量检索，取 Top-K。  
   必要时做 BM25 + dense hybrid retrieval。hybrid，白话说，是关键词检索和语义检索一起用。  
   再经过 reranker 精排，选出最终 3 到 8 段。  
   最后拼接成提示词，让 LLM 输出“答案 + 引用 + 不确定性提示”。

这类系统的关键不在“代码长不长”，而在每层是否可观测。你必须知道检索错了，还是生成错了，否则无法调优。

## 工程权衡与常见坑

RAG 项目上线后，最容易出问题的不是模型 API 调不通，而是“看起来能答，实际上不稳”。

### 1. Chunk 切分不合理

chunk 太大，主题混杂，检索回来一段里有很多噪声。  
chunk 太小，语义不完整，模型读到的是碎片。  
常见做法是按段落或标题切分，并加少量 overlap。overlap，白话说，是相邻片段之间保留一些重叠文本，避免关键信息被切断。

### 2. Embedding 不一致

建库时用一个嵌入模型，查询时换了另一个，检索质量会直接崩掉。这是非常常见的低级错误。

### 3. 只看最终答案，不看检索指标

RAG 至少要分两层评估：

| 指标 | 公式或含义 | 工程意义 |
| --- | --- | --- |
| Recall@K | Top-K 中找回了多少相关文档 | 低了会漏知识 |
| Precision@K | Top-K 中有多少是真相关 | 低了会引入噪声 |
| MRR | 第一个正确结果排得多靠前 | 衡量排序质量 |
| Faithfulness | 答案是否忠实于证据 | 防止“看着像对” |
| Relevance | 答案是否真正回答了问题 | 防止答非所问 |

一个最小数值例子：  
如果某问题共有 3 条相关文档，Top-5 检索结果中找回了 2 条，那么：

$$
Recall@5 = \frac{2}{3}
$$

如果 Top-5 里只有 3 条真正相关，那么：

$$
Precision@5 = \frac{3}{5}
$$

Recall 低，说明该找到的资料没找到。  
Precision 低，说明找回来很多噪声。  
这两者都会拖累最终 Faithfulness。

### 4. 误以为“有引用就一定没幻觉”

不是。引用可能是“形式上加了链接”，但答案中的结论未必真来自引用内容。真正要检查的是答案句子能否被检索证据支持。

### 5. 忽略时间和版本

如果知识库同时有“2025 版政策”和“2026 版政策”，系统必须处理版本优先级。否则相似度很高的旧文档可能被错误召回。工程上通常需要在排序中加入更新时间、文档状态、业务权重。

### 6. 权限与隔离缺失

企业场景里，RAG 不是“能搜到就能答”。用户 A 不应检索到用户 B 所在部门的内部制度。检索前过滤权限标签，通常比生成后再遮盖更可靠。

### 7. 上下文过长

很多项目遇到问题就机械地“把 Top-K 从 5 调到 20”。这经常适得其反。因为长上下文会增加延迟、成本和噪声，也会让模型注意力分散。更有效的办法通常是改进切分、混合检索、加 reranker、做元数据过滤。

## 替代方案与适用边界

RAG 不是默认最优解。是否采用它，取决于问题类型。

| 方案 | 适用场景 | 优点 | 局限 |
| --- | --- | --- | --- |
| Few-shot Prompt | FAQ 简单、知识稳定 | 实现最快 | 不可更新，难引用 |
| Fine-tuning | 风格固定、任务模式稳定 | 输出一致性高 | 更新成本高，不擅长频繁改知识 |
| 纯关键词检索 | 文档术语固定、精确字段多 | 可解释、成本低 | 语义泛化差 |
| RAG | 知识常更新、需要引用来源 | 可更新、可追溯 | 系统复杂度更高 |
| RAG + Multi-hop | 跨文档推理、多证据融合 | 复杂问题效果更好 | 延迟和实现成本更高 |

可以用三个简单判断来选：

1. “只需回答简单 FAQ”  
用 prompt 或轻量规则系统就够。没必要为了两百条固定问答引入向量库。

2. “内部 SOP 经常更新且必须引用”  
优先用 RAG。因为知识更新频率高，重新微调模型并不划算。

3. “需要跨多份文档串联证据”  
单层 RAG 可能不够，需要多跳检索、图检索、工作流编排，甚至把结构化数据库查询一起接入。

还要明确一个适用边界：RAG 擅长“基于外部知识回答”，不擅长“替代业务系统执行”。例如“告诉我返现政策”适合 RAG；“给客户实际发放返现”则需要事务系统、权限系统和审计系统，不是 RAG 本身能解决的。
