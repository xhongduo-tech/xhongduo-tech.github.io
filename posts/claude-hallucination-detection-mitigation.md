## 核心结论

Claude 的幻觉控制，核心不是“让模型更会说”，而是“让模型在没证据时少说”。这里的“幻觉”指模型生成了看起来合理、但没有被输入材料或现实证据支持的命题。Claude 采用 Constitutional AI，简称 CAI，也就是先给模型一套明确的行为原则，再让它对自己的回答做批判和改写，其中最关键的一条是诚实性原则：不确定时承认不确定，而不是补齐一个貌似完整的答案。

这件事的工程含义很直接。传统训练信号往往偏向“流畅、完整、自信”，Claude 则更强调 calibrated uncertainty，也就是“校准后的不确定性表达”，白话就是模型知道自己不知道，并把这种不知道明确说出来。这样做不会让幻觉消失，但会把最危险的“自信错误”压低。

如果把长文任务拆开看，Claude 的优势通常不是一次性生成一篇毫无错误的内容，而是更容易配合“先引用、再作答、再校验”的流程。对零基础工程师来说，可以把它理解成一个会先自查的助手：它不是更聪明地猜，而是更严格地问自己“这句话有证据吗”。

| 训练信号 | 模型倾向 | 下游效果 |
| --- | --- | --- |
| 追求流畅 | 优先补全完整答案 | 回答更顺，但更容易编造细节 |
| 保持诚实 | 优先暴露证据不足 | 回答可能更保守，但事实性更高 |
| 奖励自信 | 少说“不知道” | 用户体验短期更顺，长期风险更高 |
| 奖励校准 | 在证据不足时降置信度 | 更适合审计、检索、分析类任务 |

## 问题定义与边界

讨论 Claude 的幻觉，先要把问题边界说清楚。幻觉不是“模型说错一句话”这么简单，而是模型输出了一个原子命题。原子命题指可以单独判断真假的最小事实单元，比如“某公司成立于 2015 年”就是一个原子命题。如果这个命题既不被输入文本支持，也不被现实事实支持，就属于幻觉。

常见有两类：

| 类型 | 定义 | 典型表现 |
| --- | --- | --- |
| 事实性幻觉 | 与外部现实不符 | 编造年份、人物、指标 |
| 忠实性幻觉 | 与给定输入不符 | 原文没写，模型硬加结论；原文说 A，模型写成非 A |

可以用一个简化公式表示输出的幻觉率：

$$
H(y)=\frac{1}{M}\sum_{i=1}^{M}\text{Hallucination}(a_i)
$$

其中，$y$ 是模型回答，$\phi(y)=\{a_1,\dots,a_M\}$ 表示把回答拆成 $M$ 个原子命题；若某个命题没有证据支撑，则 $\text{Hallucination}(a_i)=1$，否则为 0。对应的事实性可以写成：

$$
F(y)=1-H(y)
$$

这套定义很重要，因为长文本里“句子正确”不等于“命题正确”。一句话可能包含两个正确点和一个错误点，句子级判断会把它当作“整体大致对”，命题级判断才会把风险暴露出来。

| 评估粒度 | 适合场景 | 风险 |
| --- | --- | --- |
| 句子级 | 快速抽样检查 | 容易漏掉混在句子里的局部错误 |
| 命题级 | 审计、摘要、法规场景 | 成本更高，但更接近真实风险 |

玩具例子：用户给出一段产品说明，原文只写了“支持本地部署”和“支持 API 访问”。如果模型总结成“支持本地部署、API 访问和自动扩缩容”，前两项可能是真的，第三项就是忠实性幻觉，因为输入里根本没有这个证据。

真实工程例子：在安全审计报告里，如果原始扫描结果显示“22 端口已关闭”，模型却写成“22 端口开放但风险可接受”，这不是措辞问题，而是直接改变了资产优先级和整改顺序。此时“我不确定，需要更多引用”比“自信写错”安全得多。

## 核心机制与推导

Claude 的核心机制可以概括成一句话：把“诚实地承认不知道”训练成一个高奖励动作，而不是低奖励动作。

传统生成模型更容易优化成下面的目标：给定上下文后，输出最流畅、最像正确答案的文本。这会带来一个副作用，模型在证据缺失时仍倾向继续补全，因为从语言建模角度看，“继续说”通常比“停下来并承认不确定”更像高概率序列。

CAI 的不同之处在于，它增加了一层宪法式约束。所谓“宪法”，白话解释就是一组先写好的判断规则，比如“不要把猜测说成事实”“缺少证据时明确说明限制”。模型先生成，再按照这些规则自我批判，再改写输出。于是训练目标从单纯优化“像答案”，变成同时优化“像答案且符合诚实原则”。

把这个过程写成命题级视角更清楚。设回答被拆成：

$$
\phi(y)=\{a_1,a_2,\dots,a_M\}
$$

如果其中有 3 个命题没有证据支撑，那么：

$$
H(y)=\frac{3}{10}=0.3,\quad F(y)=1-0.3=0.7
$$

这就是一个很适合新手理解的玩具例子。假设 Claude 想输出 10 条判断，但系统提示里明确要求“每条判断必须能对应引用，否则标记为不确定”。那么它更可能把那 3 条改写成“原文未提供足够证据”，而不是硬补齐。结果不是让模型更沉默，而是让它把不确定暴露出来。

可以把两类训练目标对比为：

| 目标 | 优先级最高的行为 | 对幻觉率的影响 |
| --- | --- | --- |
| 传统自信训练 | 输出完整、自然、连续 | 容易在证据空洞处补全 |
| 校准训练目标 | 输出可验证、证据对齐、允许保留 | 更容易降低 $H(y)$ |

对长文本任务，Claude 常见的有效机制不是“直接写最终答案”，而是“quote + verify”。quote 指先找引用，verify 指再做核验。因为长文里最常见的问题不是完全胡编，而是掺入 2 到 5 个貌似合理但站不住脚的细节。它们在阅读体验上非常平滑，但在事实性上已经破坏了整篇内容。

把这个机制说得再白一点：Claude 不是先假设自己知道全部，再写；而是先把“我知道什么”单独列出来，再基于这些已知部分回答。这个顺序变化，就是降低幻觉的关键。

## 代码实现

工程里不要把“少幻觉”理解成一个 prompt 技巧，而要理解成一条流水线。最稳妥的结构是三层：引用提取、命题校验、再生成。

先看一个可运行的 Python 玩具实现。它不依赖任何模型，只演示“把回答拆成命题并统计幻觉率”的最小逻辑。

```python
from dataclasses import dataclass

@dataclass
class Claim:
    text: str
    supported: bool

def hallucination_rate(claims):
    assert len(claims) > 0
    unsupported = sum(1 for c in claims if not c.supported)
    return unsupported / len(claims)

def factuality(claims):
    return 1 - hallucination_rate(claims)

claims = [
    Claim("支持本地部署", True),
    Claim("支持API访问", True),
    Claim("支持自动扩缩容", False),
    Claim("默认开启审计日志", False),
]

h = hallucination_rate(claims)
f = factuality(claims)

assert round(h, 2) == 0.50
assert round(f, 2) == 0.50
```

上面这段代码对应的就是公式 $H(y)$ 和 $F(y)$。真实系统当然不会人工给每个命题标 `supported`，而是让检索器、规则器或第二个校验器去判断。

Claude 风格的服务端流程通常可以设计成下面这样：

```python
def fetchReferences(document_chunks, question):
    # 先只返回与问题直接相关的证据片段
    return [
        {"quote": "22 port is closed", "source": "scan_report_12"},
        {"quote": "audit log disabled by default", "source": "policy_doc_3"},
    ]

def generateAnswer(references, question):
    # 模型只能基于 references 作答；没有证据的地方必须显式标注不确定
    return {
        "references": references,
        "answer": [
            {"claim": "22端口已关闭", "source": "scan_report_12"},
            {"claim": "默认开启审计日志", "source": None, "uncertain": True},
        ]
    }

def verifyClaims(answer_items, references):
    ref_texts = {r["quote"] for r in references}
    verified = []
    for item in answer_items:
        ok = item.get("uncertain") or any(
            token in " ".join(ref_texts) for token in item["claim"].split()
        )
        verified.append({**item, "verified": ok})
    return verified

refs = fetchReferences([], "系统是否合规？")
result = generateAnswer(refs, "系统是否合规？")
checked = verifyClaims(result["answer"], refs)

assert checked[0]["verified"] is True
assert checked[1]["uncertain"] is True
```

推荐的输出 schema 也应该显式保留证据字段，而不是只返回一段自然语言：

| 字段 | 作用 | 所在层 |
| --- | --- | --- |
| `references` | 存放引用片段和来源 | 提取层 |
| `answer.claim` | 单条结论 | 生成层 |
| `source` | 结论对应证据 | 对齐层 |
| `uncertain` | 标记证据不足 | 校准层 |
| `verified` | 后处理校验结果 | 校验层 |

真实工程例子：做法规审计助手时，前端先把长文档按段落切块，服务端先调用 `fetchReferences()` 只提取“与当前条款判断相关”的原文，再让 Claude 输出“结论 + 证据 + 不确定项”。最后一个 verifier 检查每句是否能映射到引用。凡是不能映射的，降级为“需人工确认”，而不是直接放进正式报告。

## 工程权衡与常见坑

第一，长上下文不是幻觉解药。很多人以为“上下文够长，就不会编”。这是错误的。长上下文只能提高模型拿到证据的概率，不能保证它严格使用证据。只要生成阶段仍然偏好流畅补全，就可能在 2K 字回答里混入 2 到 5 个看起来顺滑、但并无依据的细节。

第二，只做检索，不做核验，也不够。RAG 的意思是“检索增强生成”，白话就是先找资料再回答。但如果检索到了 10 段内容，模型依然可能把第 3 段和第 7 段拼错，或者把没写出来的结论脑补出来。所以必须有 quote + verify，而不是只有 quote。

第三，不能把“自信”当作质量代理。很多反馈数据会奖励“回答快、回答满、少拒答”，这会反向诱导模型压低自己的不确定性表达。结果就是元认知腐蚀，意思是模型越来越不愿意承认自己不知道。

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 自信错误 | 语气非常确定，但证据缺失 | 强制输出 uncertainty 字段 |
| 引用错配 | 引用存在，但不支持该结论 | 做命题级对齐校验 |
| 长文漂移 | 前文正确，后文开始补细节 | 分段回答，段后复核 |
| 检索幻觉 | 检索到了相关材料，但总结越界 | 限制“只能基于引用回答” |
| 反馈污染 | 训练中过度奖励完整度 | 定期抽查“拒答是否被错误惩罚” |

常见坑可以直接编号记住：

1. 只扩上下文，不做证据约束。
2. 只让模型“引用”，不验证引用是否真支持结论。
3. 把“不知道”当作体验差，导致系统级压制不确定表达。
4. 在审计、运维、合规场景里，把草稿回答直接当最终输出。
5. 评估只看整体可读性，不看命题级错误率。

## 替代方案与适用边界

Claude 的诚实性校准并不是唯一方案。另一条常见路线是 RLHF 加事实校验器，也就是用人类偏好强化学习，再外挂一个 fact-checking 模块。它的优点是容易兼顾“好用”和“可控”，缺点是如果奖励函数仍偏向流畅，自信错误仍然会残留。

适用边界要按业务风险分。低风险聊天应用可以容忍较高幻觉率，因为错一句常识，代价通常只是体验下降。高风险场景不行，尤其是安全、医疗、金融、法规这些领域，哪怕只有 5% 的关键错误，也足以改变流程决策。

| 场景 | 可接受幻觉率 | 推荐流程 |
| --- | --- | --- |
| 普通知识问答 | 较高 | 直接回答，可选引用 |
| 教学辅导 | 中等 | 回答后附关键出处 |
| 内容摘要 | 中低 | 先抽引用，再摘要 |
| 安全审计 | 很低 | 引用 + 校验 + 人工复核 |
| 法规合规 | 极低 | 引用 + 不确定项上浮 + 人工签字 |

可以把协作流程理解成一张极简流程图：

`文档输入 -> 引用提取 -> Claude 生成结论 -> 命题校验 -> 人工确认 -> 正式输出`

新手对照理解也很直接。普通知识问答里，你可以让 Claude 直接给答案；但如果你要生成审计报告，就必须插入“引用+人工签字”步骤。前者追求效率，后者追求责任闭环，这两类任务不应该用同一条生成链路。

所以，Claude 的优势边界是：当你愿意把“不确定也要说出来”当成系统目标时，它比单纯追求流畅的模型更容易工作；但如果你要求“零幻觉且全自动”，任何大模型都还达不到这个标准。
