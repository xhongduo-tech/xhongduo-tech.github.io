## 核心结论

边缘部署里的模型剪枝，核心不是“让很多权重变成 0”，而是“把完整结构单元删掉，让网络形状真的变小”。这里的“结构单元”可以是卷积通道、注意力头、FFN 隐藏维度，或者整层。只有张量形状发生变化，后端推理框架才可能把模型当作更小的 dense 网络来执行，进而真正减少 `MACs`、激活显存和端侧延迟。

一句最重要的判断规则是：`参数变少 ≠ 延迟变快`。参数量是模型文件大小和部分存储成本的指标，但边缘设备真正关心的是 `latency`、`peak memory`、功耗以及推理引擎能否稳定支持。`latency` 是单次推理耗时，白话说就是“这次推理到底要等多久”；`peak memory` 是峰值显存或内存，白话说就是“推理过程中最占内存的瞬间需要多大空间”。

先给一个新手版本的直觉例子。假设一层卷积原本有 128 个输出通道，你把其中一部分权重置零，参数数目下降了，但这层在很多框架里仍然会按 `128` 通道的密集矩阵去算，硬件并不会自动跳过这些零。相反，如果你把这层真正改成 `96` 个输出通道，同时让下一层输入也从 `128` 改成 `96`，那么计算图中的矩阵维度整体缩小，才更可能获得实际加速。

结构化剪枝和非结构化剪枝的差别可以先用一张表看清楚：

| 方案 | 剪掉的对象 | 张量形状是否改变 | 是否容易获得真实加速 | 典型问题 |
| --- | --- | --- | --- | --- |
| 结构化剪枝 | 通道、head、FFN 维度、层 | 是 | 相对容易 | 可能精度下降，需要重建网络 |
| 非结构化剪枝 | 单个权重 | 否，通常不变 | 通常不容易 | 很多后端仍按 dense 执行 |
| 稀疏加速 | 特定稀疏模式配合硬件 | 视模式而定 | 取决于后端支持 | 硬件和编译器约束强 |
| 模型压缩 | 广义概念，含剪枝/量化/蒸馏等 | 不一定 | 不一定 | 容易把“变小”和“变快”混为一谈 |

如果只记住一句话，那就是：边缘部署剪枝的有效前提，是删掉真实影响访存和算子的结构，而不是只减少参数数目。

## 问题定义与边界

在边缘部署语境下，目标不是追论文里的单一精度指标，而是在给定设备上稳定降低延迟、内存占用和功耗，同时保证模型还能部署、还能维护。这里的“边缘设备”指手机、摄像头 SoC、Jetson、NPU 板卡、DLA 等资源更紧、后端限制更多的硬件环境。

因此，问题定义必须收紧：我们讨论的不是“如何让模型更稀疏”，而是“如何让模型在目标设备上更快、更省内存，而且后端真的吃得到这种变化”。

下面这张边界表很重要：

| 目标 | 可剪对象 | 为什么可行 | 不适合直接等同的东西 |
| --- | --- | --- | --- |
| 降低延迟 | 卷积通道、attention head、FFN 维度、整层 | 会改变算子输入输出维度 | 单个权重置零 |
| 降低显存 | 通道、层、隐藏维度 | 激活张量和中间结果变小 | 只有参数更少但激活不变 |
| 降低计算量 | 通道、head、FFN 维度 | `MACs` 直接下降 | 无后端支持的任意稀疏 |
| 提升部署可用性 | 结构规整的瘦身 | 更容易映射到 dense kernel | 需要特殊稀疏 kernel 的方案 |

几个术语需要先分清。

结构化剪枝：删除完整结构单元，比如删掉第 17 个卷积输出通道，或者删掉第 3 个注意力头。白话说，它是在“缩网络骨架”。

非结构化剪枝：把单个权重置零，但保留原始张量形状。白话说，它是在“骨架不变的前提下，把里面很多数抹掉”。

稀疏加速：利用稀疏矩阵或特定稀疏模式，让硬件跳过无效计算。白话说，它不是天然成立的，需要后端明确支持。

模型压缩：让模型更小、更省资源的一类总称，剪枝、量化、蒸馏、低秩分解都算。白话说，它是大类，不等于某一种方法。

一个常见误解是：模型在手机上跑得慢，就说明“参数太多”。这不一定对。很多时候真正的问题是模型形状不适合设备，或者后端无法把你的稀疏模式映射成高效 kernel。比如同样是参数减少 30%，一种做法是把 30% 权重置零，另一种做法是把通道从 128 改成 96。前者很可能只是“更稀疏但同样形状的矩阵”，后者才是“更小的 dense 网络”。

玩具例子可以这样理解。你有一个 `Conv2d(64, 128, 3)`，输出给下一层 `Conv2d(128, 256, 3)`。如果只是把第一层里 32% 的权重打零，两层接口还是 `128` 通道，后面的层完全不知道你做过稀疏化。反过来，如果你把第一层输出和第二层输入都改成 `96`，整个链路才真的瘦下来。

所以本文边界很明确：重点讨论结构化剪枝，尤其是卷积通道、Transformer 的注意力头和 FFN 维度剪枝；不把任意稀疏权重直接当成边缘加速方案。

## 核心机制与推导

结构化剪枝通常分三步：先给结构单元打分，再按重要性删掉一部分，最后微调恢复精度。这里的“打分”就是估计一个通道、一个头、一个隐藏单元对任务有多重要。白话说，就是先判断“谁更像可删的冗余部分”。

卷积层最基础的计算量公式是：

$$
MAC = H_{out} \cdot W_{out} \cdot C_{out} \cdot (C_{in} / g) \cdot k^2
$$

其中 `MAC` 是乘加次数，白话说就是“这层一共要做多少次乘法加法”；`g` 是分组数，普通卷积可看成 `g=1`；`k` 是卷积核边长。

如果结构化剪枝后，输入通道保留比例是 $\alpha_{in}$，输出通道保留比例是 $\alpha_{out}$，那么计算量近似变为：

$$
MAC' \approx \alpha_{in} \cdot \alpha_{out} \cdot MAC
$$

这条式子非常关键，因为它解释了为什么“输入和输出都变小”会产生乘性收益，而不是线性收益。

举一个最短的玩具例子。原来一层卷积有：

- $H_{out} = W_{out} = 32$
- $C_{in} = 64$
- $C_{out} = 128$
- $k = 3$
- $g = 1$

则原始计算量是：

$$
MAC = 32 \cdot 32 \cdot 128 \cdot 64 \cdot 9 = 75,497,472
$$

如果把输入通道和输出通道都剪掉 25%，即变成 `48` 和 `96`，那么：

$$
MAC' = 32 \cdot 32 \cdot 96 \cdot 48 \cdot 9 = 42,467,328
$$

这时 $\alpha_{in}=\alpha_{out}=0.75$，于是：

$$
MAC' \approx 0.75 \cdot 0.75 \cdot MAC = 0.5625 \cdot MAC
$$

也就是说，计算量只剩原来的 `56.25%`。这比“只少 25%”更大，因为输入和输出两边都在缩。

训练阶段常见的门控目标写法是：

$$
L = L_{task} + \lambda \sum_c |\gamma_c|
$$

这里 $\gamma_c$ 常来自 BatchNorm 的缩放因子。`BatchNorm` 是一种让训练更稳定的归一化层，白话说可以把它理解为“顺手给每个通道再乘一个可学习系数”。当某些通道的 $|\gamma_c|$ 很小，就说明网络几乎不依赖这些通道，于是它们成为优先剪掉的候选。

下面这张表把关键推导含义压缩一下：

| 符号或对象 | 含义 | 对计算量的影响 | 在 Transformer 中的对应物 |
| --- | --- | --- | --- |
| $\alpha_{in}$ | 输入通道保留比例 | 越小，后续每个输出位置读取的数据越少 | 保留的 head 输入投影维度、FFN 输入相关维度 |
| $\alpha_{out}$ | 输出通道保留比例 | 越小，产出的特征图越少 | 保留的 head 数、FFN hidden 单元比例 |
| $\alpha_{in}\alpha_{out}$ | 乘性缩减因子 | 输入和输出同时剪，计算量乘性下降 | 同样成立 |
| 通道 | 卷积里的结构单元 | 改变卷积权重张量形状 | 类比到 attention head 或 FFN 单元 |
| head | 多头注意力里的子空间 | 影响 Q/K/V 与输出投影维度 | 是 Transformer 的结构单元 |
| FFN hidden | 前馈层中间维度 | 影响两层线性映射的宽度 | 类似卷积通道宽度 |

真实工程例子更能说明问题。以 Jetson Orin 上的视觉模型部署为例，backbone 常常占主要延迟。工程上常见做法不是盲目追求最高稀疏率，而是：

1. 先按 stage 评估哪些卷积块最耗时。
2. 对这些块做通道级结构化剪枝。
3. 把剪后通道数对齐到硬件友好的粒度，比如 8、16 或 32 的倍数。
4. 导出成 ONNX 或 TensorRT 可接受的 dense 结构。
5. 再测真实 `latency` 和 `peak memory`。

如果某层从 `64 -> 50` 通道，理论上变小了，但底层 kernel 对 `50` 这种维度不友好，实际速度未必比 `64 -> 48` 更好。这就是“理论 MACs 下降”和“真实端侧延迟下降”之间的差距。

## 代码实现

代码层面最关键的一点是：剪枝不是“加一个 mask 就结束”，而是要真正修改模块定义、张量维度和权重切片。否则你只是得到了一个带很多零的旧模型，而不是一个新的、更小的 dense 模型。

一个最小落地流程可以写成伪代码：

1. 统计通道重要性  
2. 选择保留索引  
3. 重建网络结构  
4. 迁移权重  
5. 微调恢复精度  
6. 端侧实测延迟  

先看一个基于 BatchNorm `γ` 值做通道打分的最小可运行示例：

```python
import numpy as np

def rank_channels_by_bn_gamma(gamma):
    gamma = np.asarray(gamma, dtype=float)
    scores = np.abs(gamma)
    order = np.argsort(-scores)  # descending
    return scores, order

gamma = np.array([0.1, -0.9, 0.03, 0.5, -0.2])
scores, order = rank_channels_by_bn_gamma(gamma)

assert scores.tolist() == [0.1, 0.9, 0.03, 0.5, 0.2]
assert order.tolist() == [1, 3, 4, 0, 2]
```

这段代码表达的是一个很朴素的想法：绝对值大的通道优先保留，绝对值小的通道优先删除。它不是唯一标准，但足够说明结构化剪枝的基本入口。

再看卷积层权重裁剪的核心逻辑。卷积权重通常形状为 `(C_out, C_in, k, k)`。如果你想把旧层 `Conv2d(64, 128, 3)` 改成新层 `Conv2d(48, 96, 3)`，就必须同时裁剪输出维和输入维：

```python
import numpy as np

def prune_conv_weight(weight, keep_out_idx, keep_in_idx):
    weight = np.asarray(weight)
    pruned = weight[np.ix_(keep_out_idx, keep_in_idx, range(weight.shape[2]), range(weight.shape[3]))]
    return pruned

# old weight: [C_out=4, C_in=3, k=3, k=3]
weight = np.arange(4 * 3 * 3 * 3).reshape(4, 3, 3, 3)

keep_out_idx = [0, 2]
keep_in_idx = [1, 2]

new_weight = prune_conv_weight(weight, keep_out_idx, keep_in_idx)

assert new_weight.shape == (2, 2, 3, 3)
assert np.array_equal(new_weight[0, 0], weight[0, 1])
assert np.array_equal(new_weight[1, 1], weight[2, 2])
```

这个例子虽然是 `numpy` 写的玩具版本，但它说明了真正的结构变化：新权重张量的形状已经变了。部署时真正有意义的是这个“新张量”，而不是在旧张量上乘一个稀疏掩码。

下面给一个更完整的最小示意，把“算 MACs”和“剪枝后是否真的下降”连接起来：

```python
def conv_mac(h_out, w_out, c_in, c_out, k, groups=1):
    return h_out * w_out * c_out * (c_in // groups) * (k ** 2)

orig = conv_mac(32, 32, 64, 128, 3)
pruned = conv_mac(32, 32, 48, 96, 3)

assert orig == 75497472
assert pruned == 42467328
assert pruned / orig == 0.5625
```

微调阶段也不能省。剪掉结构单元以后，模型容量下降，特征分布会变，通常需要一段恢复训练。下面是一个抽象化的微调训练循环：

```python
def finetune_one_epoch(model, dataloader, optimizer, loss_fn):
    model.train()
    total_loss = 0.0
    total_steps = 0

    for x, y in dataloader:
        pred = model(x)
        loss = loss_fn(pred, y)
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

        total_loss += float(loss.detach().cpu())
        total_steps += 1

    avg_loss = total_loss / max(total_steps, 1)
    assert avg_loss >= 0.0
    return avg_loss
```

真实项目中，这一步通常会配合蒸馏。`蒸馏` 是让小模型去模仿大模型输出分布的方法，白话说就是“让被剪小的学生模型向原模型学行为”，这样更容易把精度拉回来。

如果把上面的代码逻辑翻译成工程动作，就是：

- 用 BN `γ`、梯度敏感度或 Taylor 近似给通道打分。
- 选出每层要保留的通道索引。
- 新建剪后模块，例如把 `Conv2d(64, 128, 3)` 改成 `Conv2d(48, 96, 3)`。
- 从旧权重按索引切片，把保留通道迁移到新层。
- 若有 BN、残差分支、下游线性层，也同步修改维度。
- 微调若干 epoch，再在目标端侧实测。

真实工程例子里，以一个轻量检测模型为例，如果 backbone 某 stage 原本 4 个 block 都是 `128` 通道，常见做法不是统一暴力裁成 `64`，而是先通过 profiling 看哪几个 block 延迟占比最高，再分层设定不同保留率，比如 `0.9 / 0.8 / 0.75 / 0.75`。这样通常比整网等比例裁剪更稳，因为不同层对精度和延迟的敏感度并不一样。

## 工程权衡与常见坑

结构化剪枝在论文里看起来像“删掉冗余”，但在工程里真正难的不是删，而是“删完以后硬件是否真能兑现收益”。同样少 20% 的 `MACs`，在一块 GPU、一个手机 NPU 和一个 CPU 上，收益可能完全不同。

最常见的误区是只看参数量，不看端侧延迟。参数更少常常意味着模型文件更小，但端侧速度还受内存访问、kernel 选择、算子融合和设备对齐影响。某些模型参数不大，但激活图很大，真正拖慢推理的是中间特征图搬运，而不是权重文件本身。

常见坑与规避可以直接列成表：

| 常见坑 | 现象 | 原因 | 规避方式 |
| --- | --- | --- | --- |
| 只看参数数 | 参数下降很多，延迟几乎不变 | 形状没变，仍按 dense 执行 | 用端侧实测 `latency` 作为主目标 |
| 剪太狠导致精度塌陷 | 验证集性能明显下滑 | 关键层被过度裁剪 | 分层剪枝、保留关键 stage、剪后微调 |
| 多分支网络引入额外 `memory copy` | 理论更小，实测更慢 | 分支维度不一致导致额外重排 | 统一分支维度，导出前整理通道顺序 |
| 后端不支持稀疏模式 | 稀疏权重没有加速 | 引擎只优化 dense 或特定稀疏 | 先确认 TensorRT / DLA / NPU 支持情况 |
| 通道数不对齐硬件粒度 | MACs 更低但速度更差 | 底层 kernel 对 8/16/32 对齐更友好 | 把通道数裁到硬件友好倍数 |
| 只剪单层不看全链路 | 局部变快，总体不明显 | 真正瓶颈不在该层 | 先做 profiling 再决定剪哪层 |

这里再给一个新手很容易踩的例子。原始网络某层通道数是 `64`，你把它剪成 `50`，从纸面上看减少了约 22%。但如果目标后端最擅长跑 `8` 或 `16` 对齐的矩阵，`48` 可能比 `50` 更快，甚至 `56` 也可能比 `50` 更快。原因不是数学公式错了，而是部署系统不是只看公式，它还要看 kernel 是否适配、内存布局是否规整。

部署检查清单可以直接按下面四条过：

- 是否真的减少 `latency`，而不是只减少理论 `MACs`
- 是否真的减少 `peak memory`，特别是激活显存而不只是权重体积
- 是否符合 TensorRT、DLA 或目标引擎的算子与形状约束
- 是否需要把通道数对齐到硬件粒度，比如 8、16、32 的倍数

真实工程例子可以再具体一点。假设你在 Jetson Orin 上部署一个 INT8 分类模型。你把 backbone 通道普遍裁了 30%，但发现总延迟只下降 8%。排查后发现三件事：

1. 被裁最狠的是一些本来就不耗时的小层。
2. 大层虽然参数降了，但通道数变成了非对齐值，TensorRT 没选到最优 tactics。
3. 末端 `concat` 分支因为维度不一致，增加了额外的 tensor reformat。

这时正确做法不是继续加大剪枝率，而是回到 profiling 结果，重新决定“剪哪里”和“剪成什么形状”。

## 替代方案与适用边界

结构化剪枝不是唯一方案，但它在“需要真实端侧加速，同时又希望继续使用高效 dense 算子”的场景里通常最实用。它的优点是部署链路相对直观：输出的是一个更小的常规网络，而不是一个必须依赖特殊稀疏 kernel 的网络。

不过，不同压缩方法解决的问题并不一样。下面这张对比表适合拿来做方案初筛：

| 方案 | 主要收益 | 是否容易获得真实加速 | 部署复杂度 | 适合场景 |
| --- | --- | --- | --- | --- |
| 结构化剪枝 | 降低 `MACs`、激活显存、延迟 | 较容易 | 中等 | 追求端侧真实加速 |
| 量化 | 降低模型体积、带宽、部分硬件上显著提速 | 取决于硬件支持，通常较好 | 中等 | 文件变小、INT8/NPU 加速 |
| 蒸馏 | 尽量保精度 | 本身不直接加速 | 中等 | 配合剪枝或量化恢复性能 |
| 低秩分解 | 减少某些线性/卷积层参数和计算 | 有时可行 | 较高 | 特定大矩阵冗余明显 |
| 非结构化稀疏 | 提高稀疏率、减参数 | 通常不容易 | 较高 | 后端明确支持稀疏算子时 |
| 直接换轻量架构 | 从源头控制复杂度 | 容易 | 低到中等 | 允许重训或改模型设计 |

可以把这些方案理解成三个不同目标：

- 想要真实端侧加速，优先考虑结构化剪枝。
- 想要尽可能保精度，可结合蒸馏。
- 想要最小模型体积，可结合量化。

这里的“量化”是把浮点数权重和激活映射到更低位宽，比如 INT8。白话说，就是“用更短的数字表示同样的张量”。如果你的主要目标是模型文件更小、内存带宽更低，而且硬件对 INT8 支持成熟，那么量化常常比剪枝更直接。

反过来，如果你的目标是“后端对稀疏不友好，但我还想让网络实打实变快”，结构化剪枝通常比非结构化稀疏更可靠。因为后者往往只是让权重矩阵更空，却没有改变 dense kernel 看到的形状。

再给一个边界判断。

玩具例子：一个小 CNN 部署到树莓派，CPU 对稀疏优化很弱。这时把 30% 权重打零，基本不值得期待；把卷积通道从 `64/128` 改成 `48/96`，更有可能得到可见收益。

真实工程例子：一个 Transformer 编码器要在移动端语音设备运行。如果瓶颈主要在 FFN，两种常见选择是删一部分 attention head，或缩减 FFN hidden 维度。经验上，很多任务里 FFN 宽度往往比 head 数更占算力，因此优先压缩 FFN hidden 常更有效；但如果某些层的 head 明显冗余，删 head 也能换来可部署的降耗收益。最终仍要看真实 profiling，而不是只看论文中的平均结论。

所以，结构化剪枝的适用边界可以概括为：当你希望输出仍然是标准 dense 网络，并且愿意为“重建结构 + 微调 + 端侧验证”付出工程成本时，它最有价值；当你只想快速压缩文件，或者硬件已对量化高度优化时，量化可能优先级更高。
