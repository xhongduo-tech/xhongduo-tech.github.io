## 核心结论

Claude 的 Token 采样策略，本质上是在“更稳定”与“更多样”之间调节下一步该选哪个 token。Token 可以理解为模型一次生成时选择的最小文本单元，可能是一个字、词、标点，或者词的一部分。

Claude 常见的三个采样参数是 `temperature`、`top-p`、`top-k`：

| 参数 | 作用 | 典型值 | 推荐场景 |
| --- | --- | --- | --- |
| `temperature` | 调整概率分布的陡峭程度 | `0`、`0.3`、`0.7`、`0.8` | 控制整体随机性 |
| `top-p` | 只在累计概率达到阈值的候选集合中采样 | `0.9`、`0.95`、`0.99` | 过滤长尾低概率 token |
| `top-k` | 只在概率最高的前 K 个候选中采样 | `20`、`40`、`100` | 限制候选规模 |

对初级工程师最重要的结论有三点。

第一，`temperature` 决定“敢不敢偏离最稳妥答案”。当 `temperature=0` 时，可以把它理解为接近贪心解码，也就是每一步都尽量选当前概率最大的 token；当 `temperature=1` 时，基本按原始分布采样。

第二，`top-p` 与 `top-k` 不是让模型“更聪明”，而是限制它“可以从哪些候选里选”。`top-p=0.9` 表示只保留累计概率达到 90% 的那一簇候选；`top-k=40` 表示最多只看前 40 个候选。

第三，工程上默认配置通常不是追求极端，而是追求稳定平衡。常见经验值是 `temperature≈0.7`、`top-p=0.99`、`top-k=40`。这类组合不会把模型压得太死，也不会把长尾噪声完全放开。

一个直接的使用建议是：写 FAQ、知识库问答、摘要这类偏事实任务，优先把 `temperature` 降到 `0.2~0.3`，并把 `top-p` 收到 `0.9`；写宣传文案、标题草稿、创意段落这类任务，再把 `temperature` 提到 `0.8` 左右。先调一个参数，再调第二个参数，比三个一起改更容易得到可解释结果。

## 问题定义与边界

采样策略要解决的问题，不是“模型会不会说话”，而是“模型在多个可能 token 之间如何做选择”。这一步发生在每一个生成位置上，所以参数配置会持续影响整段输出的风格、稳定性和错误模式。

这里先定义边界。

1. 本文讨论的是解码阶段，也就是模型已经算出一组候选 token 概率之后，系统如何从中选出下一个 token。
2. 本文不讨论训练过程、模型结构差异、提示词工程本身，只讨论采样参数对输出分布的影响。
3. 本文说的“确定性”不是绝对复现。确定性可以理解为“多次运行时输出更接近”，不是“每次逐字完全一样”。

对新手最容易混淆的一点是：`temperature=0` 不等于“永远 100% 可复现”。原因很简单，真实系统里还可能存在并发、批处理、数值实现、服务端细节等因素。所以温度设为 0 只是把采样策略推向最保守，不代表系统层面彻底没有非确定性。

下面这张表把边界先定清楚：

| 参数 | 含义 | 常见范围 | 边界解释 |
| --- | --- | --- | --- |
| `temperature` | 重塑概率分布 | 通常在 `[0, 2]` 附近 | 越低越保守，越高越发散 |
| `top-p` | 累计概率截断 | `(0, 1]` | 越低越只保留高概率簇 |
| `top-k` | 固定数量截断 | 正整数 | 越小越只看头部候选 |

玩具例子可以先看一句话补全：输入是“法国的首都是”。如果 `temperature=0`，模型大概率直接选“巴黎”；如果 `temperature=0.8` 且 `top-p=0.99`，它仍大概率选“巴黎”，但在更开放任务里，会更愿意尝试不同表达。也就是说，采样参数主要影响“存在多种合理续写”时的选择空间。

真实工程例子更典型。知识库问答系统要回答“退款期限是多少天”，你希望它优先复用已有事实表达，而不是换很多说法甚至带入多余推测。这时应把候选空间压小。相反，给营销团队生成活动标题时，任务目标本来就允许多样化，适当提高温度更合理。

## 核心机制与推导

先看温度。模型原始输出通常是一组 logits，logit 可以理解为“还没归一化的打分”。这些打分经过 softmax 变成概率分布。温度做的事情，就是在 softmax 前后等价地重塑这组分布。

如果原始概率是 $p_i$，温度为 $T$，那么重标定后的分布可以写成：

$$
p'_i = \frac{p_i^{1/T}}{\sum_j p_j^{1/T}}
$$

这个式子有三个直接结论：

1. 当 $T=1$ 时，$p'_i=p_i$，分布不变。
2. 当 $T<1$ 时，高概率项被进一步放大，低概率项被压缩。
3. 当 $T>1$ 时，分布被拉平，更多低概率项获得被采样机会。

再看 `top-k`。它的定义很直接：先按概率从高到低排序，只保留前 $K$ 个，再对这 $K$ 个候选重新归一化后采样。数学上可以写成：

$$
S_k = \{i \mid i \text{ 属于排序后前 } K \text{ 个候选}\}
$$

采样只在集合 $S_k$ 内进行。

`top-p` 叫 nucleus sampling，中文常译为“核采样”或“累积概率采样”。它不是固定保留几个候选，而是动态地找一个最小集合，使累计概率达到阈值 $p$：

$$
m = \min \left\{m \;\middle|\; \sum_{i=1}^{m} p'_i \ge p \right\}
$$

然后只在前 $m$ 个候选里采样。它的关键优点是“自适应”：如果当前分布很尖锐，只保留很少几个 token；如果当前分布较平，就保留更多 token。

下面用一个玩具例子说明三者如何叠加。假设当前三个候选 token 的原始概率是：

- A: 0.5
- B: 0.3
- C: 0.2

温度为 1 时，分布不变。温度降到 0.5 时，相当于做平方再归一化：

- A: $0.5^2 = 0.25$
- B: $0.3^2 = 0.09$
- C: $0.2^2 = 0.04$

归一化后大约得到：

| 候选 | 原始概率 | `T=0.5` 后近似概率 | `top-p=0.9` 是否保留 | `top-k=2` 是否保留 |
| --- | --- | --- | --- | --- |
| A | 0.50 | 0.658 | 是 | 是 |
| B | 0.30 | 0.237 | 是 | 是 |
| C | 0.20 | 0.105 | 否 | 否 |

为什么 `top-p=0.9` 时 C 被去掉？因为排序后 A+B 的累计概率已经约为 $0.658+0.237=0.895$，再加一点边界处理，通常只要再看是否需要纳入下一个 token 才能达到阈值。实际实现会按系统规则决定是否把边界 token 一并纳入，但直观上已经能看出：长尾候选被明显压缩了。

真实工程里，这个机制的意义非常直接。比如在客服 FAQ 场景中，正确答案通常集中在高概率头部，长尾 token 往往对应更花哨但不必要的表达，甚至会引入错误推断。把 `temperature` 降低，并把 `top-p` 收紧到 `0.9`，等于强制模型在“高置信概率簇”内找答案。

反过来，创意写作时你不希望模型永远选最稳妥的续写，因为那样容易模板化。提高温度，相当于把原本第二、第三梯队的候选重新放回竞争区间。这不是让模型“胡说”，而是扩大探索半径。

## 代码实现

下面先用一个纯 Python 的可运行示例，演示温度缩放与 `top-p`、`top-k` 的效果。这个例子不依赖任何外部库。

```python
from math import isclose

def apply_temperature(probs, temperature):
    assert temperature > 0
    scaled = [p ** (1 / temperature) for p in probs]
    s = sum(scaled)
    return [x / s for x in scaled]

def apply_top_k(probs, k):
    assert k >= 1
    indexed = list(enumerate(probs))
    indexed.sort(key=lambda x: x[1], reverse=True)
    kept = indexed[:k]
    total = sum(p for _, p in kept)
    result = [0.0] * len(probs)
    for idx, p in kept:
        result[idx] = p / total
    return result

def apply_top_p(probs, top_p):
    assert 0 < top_p <= 1
    indexed = list(enumerate(probs))
    indexed.sort(key=lambda x: x[1], reverse=True)

    kept = []
    cumulative = 0.0
    for idx, p in indexed:
        kept.append((idx, p))
        cumulative += p
        if cumulative >= top_p:
            break

    total = sum(p for _, p in kept)
    result = [0.0] * len(probs)
    for idx, p in kept:
        result[idx] = p / total
    return result

probs = [0.5, 0.3, 0.2]

t05 = apply_temperature(probs, 0.5)
assert isclose(sum(t05), 1.0, rel_tol=1e-9)
assert t05[0] > probs[0]   # 高概率项被放大
assert t05[2] < probs[2]   # 低概率项被压缩

topk2 = apply_top_k(t05, 2)
assert isclose(sum(topk2), 1.0, rel_tol=1e-9)
assert topk2[2] == 0.0

t1 = apply_temperature(probs, 1.0)
assert all(isclose(a, b, rel_tol=1e-9) for a, b in zip(probs, t1))

topp = apply_top_p(t05, 0.9)
assert isclose(sum(topp), 1.0, rel_tol=1e-9)
assert topp[0] > 0 and topp[1] > 0

print("temperature:", [round(x, 3) for x in t05])
print("top-k=2:", [round(x, 3) for x in topk2])
print("top-p=0.9:", [round(x, 3) for x in topp])
```

如果要在 Claude 的 Python SDK 里使用这些参数，思路也是一样的：把它们作为生成参数传给 API。下面是结构化示意代码，重点是参数关系，不是具体版本兼容细节。

```python
from anthropic import Anthropic

client = Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=300,
    temperature=0.3,  # 新手建议：先只调这个参数
    top_p=0.9,        # 把候选限制在高概率簇
    top_k=40,         # 保留头部候选，避免长尾噪声
    messages=[
        {
            "role": "user",
            "content": "请根据知识库内容回答：退款期限是多少天？"
        }
    ],
)

text = response.content[0].text
assert isinstance(text, str)
assert len(text) > 0
print(text)
```

调试建议只有一句：先固定 `top_p` 和 `top_k`，只改 `temperature`；如果输出仍不稳，再收紧 `top_p`。这样你能知道结果变化到底是谁造成的。

真实工程例子是 RAG 问答。RAG 可以理解为“先检索资料，再让模型基于资料回答”。在这种场景里，检索阶段已经提供了事实来源，生成阶段的目标不是“想新说法”，而是“别偏离证据”。所以常见做法是：

- `temperature=0~0.3`
- `top_p=0.9`
- `top_k` 保持默认或中等值

这套配置的核心不是追求语言华丽，而是减少不必要的分布尾部探索。

## 工程权衡与常见坑

采样参数没有绝对最优值，只有任务目标下的取舍。最常见的权衡，是一致性、创造性、可复现性三者不能同时拉满。

下面这张表列出工程里最常见的误区：

| 常见坑 | 现象 | 原因 | 规避策略 |
| --- | --- | --- | --- |
| 以为 `temperature=0` 就绝对一致 | 多次请求仍有细微差异 | 系统层面可能仍有非确定性 | 把它理解为“更接近贪心”，不是“绝对锁死” |
| 同时极端调低 `temperature` 和 `top-p` | 输出僵硬、重复、局部跳变 | 候选空间过小 | 先调一个，再观察 |
| 忽略默认 `top-p` 较高 | 事实任务偶发长尾噪声 | 候选簇过宽 | 问答/摘要先试 `top_p=0.9` |
| 把 `top-k` 调得过小 | 语句不自然、卡顿 | 固定截断过硬 | 除非明确需要，否则先保留默认 |
| 只看单次结果调参 | 参数效果不稳定、误判 | 采样问题本来就要看分布 | 同一提示跑多次比较趋势 |

这里重点解释两个坑。

第一个坑是“低温度就一定更真实”。不完全对。低温度只能让模型更偏向它自己认为高概率的 token，但如果模型内部本来就把错误答案判成高概率，低温度只会更稳定地输出错误。所以采样参数能控制随机性，不能替代检索、约束、评测。

第二个坑是“高温度配低 `top-p`”。这类组合看起来像是“两头兼顾”，实际上常常会让模型在一个很窄但又被高温度拉平的候选集合里剧烈跳动。简单说，候选集很小，但内部排名又被温度打散了，结果就是输出有时怪、有时跳。

真实工程里可以这样理解：

- 法律条款摘要：希望每次说法接近，宁可少一点修辞。参数应偏保守。
- 广告标语草拟：希望一轮里多给几种不同说法。参数应偏开放。
- 代码生成：希望语法和 API 名字稳定，不希望频繁换实现路径。通常低温度更安全。

如果你的业务目标是“用户会拿输出直接执行”，比如 SQL 片段、配置文件、脚本模板，那么一致性优先级应高于文采。反过来，如果输出只是供人类挑选、修改、再加工，比如文案备选，采样空间可以更大。

## 替代方案与适用边界

不是所有场景都需要同时动 `temperature`、`top-p`、`top-k`。对大多数团队，最稳妥的策略是“少调几个旋钮”。

先给出一个场景表：

| 场景 | 推荐参数组合 | 说明 |
| --- | --- | --- |
| FAQ/知识库问答 | `temperature=0.2~0.3`, `top_p=0.9`, `top_k` 默认 | 先保事实稳定性 |
| 摘要/结构化提取 | `temperature=0~0.2`, `top_p=0.9` | 尽量减少发散 |
| 代码生成 | `temperature=0~0.2`, `top_p<=0.9`, `top_k` 默认 | 保持语法与调用路径稳定 |
| 标题生成/营销文案 | `temperature=0.7~0.9`, `top_p=0.9~0.95` | 允许更多表达变化 |
| 头脑风暴/创意写作 | `temperature=0.8~1.0`, `top_p=0.95~0.99` | 扩大探索空间 |

几个可替代思路如下。

第一，只调 `temperature`。这是最适合新手的做法，因为它最直观。想更保守，就从 `0.7` 降到 `0.3`；想更开放，就从 `0.7` 提到 `0.8`。如果你还没建立参数感觉，不要一开始就同时动三个。

第二，只调 `top-p`。这适合“我不想改变总体风格，只想砍掉尾部噪声”的情况。比如模型整体表达没问题，但偶尔会在事实题里冒出多余修饰或不必要推断，这时把 `top-p` 从 `0.99` 收到 `0.9` 往往比猛降温度更平滑。

第三，少碰 `top-k`。`top-k` 是最硬的截断，适合非常明确想限制候选规模的场景，但它对语言自然度的伤害也更直接。很多业务里保留默认值即可。

它们的适用边界也要说清楚：

- 如果任务本身没有明确标准答案，单纯压低采样参数不会自动得到“更好结果”，只会得到“更像平均答案”的结果。
- 如果任务强依赖上下文事实，采样参数只能做风格控制，不能替代事实来源。
- 如果任务要求近似复现，除了调低温度，还要控制 prompt、上下文、系统设置以及调用条件。

所以，采样参数最适合解决的是“在合理答案集合里怎么选”，不适合解决“模型到底知不知道正确答案”。
