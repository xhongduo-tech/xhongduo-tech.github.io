## 核心结论

生成式问答是一类把“问题 + 上下文”直接映射成答案文本的问答方法。它不要求答案必须是原文中的连续片段，而是由模型逐个生成 token，最终组成一段自然语言回答。

这里的 token 可以先理解成“模型处理文本时使用的最小单位”，可能是一个字、一个词，也可能是一个词的一部分。生成式问答的关键不是从文档里复制答案，而是在证据约束下生成答案。

| 类型 | 输出方式 | 适合场景 |
|---|---|---|
| 抽取式问答 | 从文档中选一段原文 | 文本中答案位置明确 |
| 生成式问答 | 逐 token 生成答案 | 跨文档整合、改写、归纳 |
| 文章主结论 | 先检索，再生成 | 知识库、开放域、对话问答 |

一个真实工程例子是客服知识库问答。用户问“退款多久到账”，系统不只是返回某条 FAQ 原文，而是先检索退款政策、订单状态、支付渠道规则和工单说明，再生成回答：

“通常 3-5 个工作日到账，具体取决于支付渠道。当前订单已提交退款，预计会按原支付方式退回。”

这个回答可能没有在任何一篇文档中原样出现，但它必须能被检索到的证据支持。生成式问答的优势就在这里：它可以整合多段证据，并用更自然的语言输出答案。

它的边界也在这里：生成不是自由编造。证据不足时，系统应该回答“不确定”“当前资料不足”或请求补充信息，而不是生成一个看起来流畅但没有依据的答案。

## 问题定义与边界

先把问题定义清楚，才能避免把生成式问答、检索、摘要和对话系统混在一起。

生成式问答的输入通常包含两部分：用户问题和证据集合。证据集合可以来自搜索引擎、向量数据库、企业知识库、FAQ、数据库查询结果，也可以来自当前对话历史。

定义如下：

- 输入问题：`q`
- 证据集合：`D={d_1,...,d_k}`
- 输出答案：`y=(y_1,...,y_T)`
- 边界：答案不要求是原文片段，但必须受证据约束

| 项 | 含义 |
|---|---|
| `q` | 用户问题 |
| `D` | 检索到的文档、段落或 passage |
| `y` | 模型生成的答案序列 |

passage 是一小段可被模型读取的文本，通常由长文档切分而来。比如一篇退款政策有 3000 字，系统可能把它切成多个 300-500 字的 passage，再从中找出和问题最相关的部分。

新手版理解是：你先把相关资料找出来，再让模型根据资料写答案。它不是让模型闭眼回答。

生成式问答和摘要也不同。摘要的输入通常是一篇或多篇文档，目标是压缩主要内容；问答的输入包含明确问题，目标是回答这个问题。生成式问答和普通聊天也不同。聊天系统可以围绕上下文自由互动，而知识型问答需要更强调证据来源和事实一致性。

一个玩具例子：

问题：`谁发明了电话？`

检索到的证据：

- `d_1`: “亚历山大·格拉汉姆·贝尔因电话发明相关专利而知名。”
- `d_2`: “电话的发展还涉及多位发明者和工程师。”

生成式问答可以输出：

“通常认为亚历山大·格拉汉姆·贝尔与电话发明密切相关，尤其是他的电话专利。”

这个答案不是简单复制第一段，而是基于证据做了更稳妥的表述。它没有把复杂历史简化成绝对唯一答案，也没有脱离证据扩写无关内容。

## 核心机制与推导

生成式问答之所以叫“生成式”，是因为答案被看作一个序列。模型不是一次性吐出完整答案，而是在已有问题、证据和已生成内容的条件下，逐 token 预测下一个 token。

设问题为 `q`，证据集合为 `D={d_1,...,d_k}`，答案序列为 `y=(y_1,...,y_T)`。生成目标可以写成：

$$
p(y|q,D)=\prod_{t=1}^{T} p(y_t | y_{<t}, q, D)
$$

这个公式的意思是：完整答案的概率，等于每一步生成正确 token 的条件概率相乘。`y_{<t}` 表示第 `t` 步之前已经生成的内容。

训练时，模型知道标准答案 `y^*`。训练目标是让模型在每一步都更倾向于生成标准答案中的下一个 token。常见损失函数是负对数似然：

$$
L=-\sum_{t=1}^{T} \log p(y_t^* | y_{<t}^*, q, D)
$$

负对数似然可以白话理解为：模型给正确答案的概率越低，惩罚越大；概率越高，惩罚越小。

最小数值例子：

问题是“谁发明了电话”，标准答案第一个 token 是“贝尔”。如果模型给出的概率是：

- `P(贝尔)=0.72`
- `P(爱迪生)=0.10`
- `P(其他)=0.18`

那么这一步的损失是：

$$
-\log(0.72)\approx 0.33
$$

如果第二步正确 token 的概率是 `0.85`，那么两步总损失约为：

$$
-\log(0.72)-\log(0.85)\approx 0.33+0.16=0.49
$$

工程上，生成式问答通常不是单独一个模型完成全部工作，而是两阶段架构：先检索，再生成。

| 步骤 | 作用 |
|---|---|
| 检索 | 找到候选证据 |
| 编码 | 将问题与证据表示成模型可用向量 |
| 解码 | 逐 token 生成答案 |
| 约束 | 尽量让答案贴合证据 |

编码是把文本转成模型内部向量表示的过程。解码是模型根据这些表示逐步生成输出文本的过程。T5、BART 等 seq2seq 模型都属于常见实现基础。seq2seq 是 sequence-to-sequence 的缩写，意思是“输入一个序列，输出另一个序列”。

两阶段结构解释了一个重要事实：召回质量会强烈影响最终答案。召回是检索系统把相关文档找出来的能力。如果问题是“谁发明了电话”，检索结果里有贝尔相关证据，模型才更可能在第一步输出“贝尔”，然后继续补全答案。如果检索结果全是“电报发展史”，生成器很可能生成偏题答案。

## 代码实现

最小可运行链路不需要从训练大模型开始。先理解数据流：问题进入系统，检索器找证据，生成器基于问题和证据生成答案，后处理模块返回答案和来源。

伪代码可以理解成三步走：先搜资料，再喂给模型，再把答案和来源一起返回。

```python
def answer_question(question):
    passages = retrieve_top_k(question, k=5)      # 1. 检索
    context = format_context(passages)            # 2. 组织上下文
    answer = seq2seq_generate(question, context)  # 3. 生成
    return {
        "answer": answer,
        "evidence": passages
    }
```

下面是一个不依赖外部库的玩具实现。它不是真正的大模型，但完整展示了生成式问答系统的数据流：检索、组织上下文、生成、返回证据。

```python
from math import log

KB = [
    {
        "id": "refund_policy",
        "text": "退款通常在3-5个工作日内按原支付方式到账。银行卡可能更慢。"
    },
    {
        "id": "order_status",
        "text": "订单A1001已提交退款申请，当前状态为退款处理中。"
    },
    {
        "id": "shipping_policy",
        "text": "已发货订单需要先完成退货入库，再进入退款流程。"
    }
]

def retrieve_top_k(question, k=2):
    keywords = set(question.replace("？", "").replace("，", "").split())
    scored = []
    for doc in KB:
        score = sum(1 for word in keywords if word and word in doc["text"])
        if "退款" in question and "退款" in doc["text"]:
            score += 2
        scored.append((score, doc))
    scored.sort(key=lambda x: x[0], reverse=True)
    return [doc for score, doc in scored[:k] if score > 0]

def format_context(passages):
    return "\n".join(f"[{p['id']}] {p['text']}" for p in passages)

def seq2seq_generate(question, context):
    if "退款" in question and "3-5个工作日" in context:
        if "退款处理中" in context:
            return "退款通常3-5个工作日到账；当前订单已提交退款，状态为退款处理中。"
        return "退款通常3-5个工作日按原支付方式到账。"
    return "当前证据不足，无法给出确定答案。"

def answer_question(question):
    passages = retrieve_top_k(question, k=3)
    context = format_context(passages)
    answer = seq2seq_generate(question, context)
    return {"answer": answer, "evidence": passages}

result = answer_question("退款 多久 到账？")
assert "3-5个工作日" in result["answer"]
assert any(p["id"] == "refund_policy" for p in result["evidence"])

loss = -log(0.72) - log(0.85)
assert round(loss, 2) == 0.49
```

流程可以写成：

```text
用户问题
  ↓
检索器 Retriever
  ↓
top-k 文档 / passage
  ↓
生成器 Generator
  ↓
最终答案 + 来源
```

| 模块 | 输入 | 输出 |
|---|---|---|
| Retriever | `q` | `D` |
| Generator | `q, D` | `y` |
| Post-process | `y, D` | 结构化答案 |

真实系统里，`retrieve_top_k` 可能由 BM25、向量检索或混合检索实现。BM25 是一种按关键词相关性排序的传统检索算法。向量检索是把文本表示成向量，再按语义相似度找相近内容的方法。生成器可以是 T5、BART，也可以是更大的指令模型。

后处理也很重要。它通常负责控制答案长度、附上引用、过滤低置信度输出、把答案包装成接口需要的 JSON。对工程系统来说，返回“答案 + 来源”比只返回答案更可靠。

## 工程权衡与常见坑

生成式问答的主要风险不是“不会生成”，而是“生成得很像但不对”。这种问题通常叫幻觉。幻觉是指模型生成了看似合理但没有证据支持的内容。

客服问答里，如果只检索到一条过时政策，模型即使语言很流畅，也会把错误政策说得很完整。这说明“召回错了，生成再强也救不回来”。

| 常见坑 | 现象 | 规避方法 |
|---|---|---|
| 幻觉 | 编造不存在的信息 | 强制引用证据，限制生成范围 |
| 证据截断 | 长文档放不进上下文 | 分块、重排、top-k、FiD |
| 格式漂移 | 训练短答，线上变长答 | 模板约束、长度控制 |
| 评估失真 | EM 不认同义改写 | 同时看 EM、F1、人工评估 |
| 检索错配 | 搜到无关文档 | 先提升召回，再调生成 |

FiD 是 Fusion-in-Decoder 的缩写，可以理解为“让解码器融合多个文档证据”的生成式问答结构。它适合多文档问答，但计算成本更高。

EM 是 Exact Match，表示预测答案和标准答案是否完全一致。F1 是一种同时考虑答案重叠准确率和召回率的指标。生成式问答里，EM 经常过于严格。例如标准答案是“亚历山大·格拉汉姆·贝尔”，模型输出“贝尔”，语义可能可接受，但 EM 会判错。

| 排查顺序 | 重点 |
|---|---|
| 1 | 检索是否命中 |
| 2 | 上下文是否完整 |
| 3 | 模型是否忠于证据 |
| 4 | 输出是否符合格式 |

性能问题也常常先出在检索和上下文处理，而不是生成器本身。长文档切块太粗，会导致关键句被截断；切块太细，会导致上下文碎片过多。top-k 太小，可能漏掉关键证据；top-k 太大，可能引入噪声并挤占上下文长度。

一个工程上常见的处理方式是：

| 问题 | 处理方式 |
|---|---|
| 文档太长 | 按标题、段落、语义边界分块 |
| 召回不稳 | 关键词检索 + 向量检索混合 |
| 证据太多 | 重排模型筛选 top-n |
| 答案无引用 | 要求输出引用 passage id |
| 答案过长 | 设置最大 token 数和模板 |

生成式问答不是把模型接上知识库就结束。可靠系统通常需要检索评估、证据覆盖率检查、答案一致性检查、人工抽检和日志回放。越是面向真实用户，越不能只看答案是否“像人写的”。

## 替代方案与适用边界

生成式问答不是所有场景的最优解。它擅长归纳、改写、跨文档合并，但不一定适合强约束抽取和结构化查询。

如果用户问“订单号是多少”，更直接的方法是字段查询或抽取式问答。因为订单号是明确字段，不需要生成模型发挥语言组织能力。生成模型反而可能改写、漏字或输出错误格式。

如果用户问“这份政策对哪些用户不适用”，生成式问答更有优势。因为它可能需要综合适用范围、例外条款、用户类型和时间限制，再组织成一段清晰回答。

| 方案 | 优点 | 缺点 | 适用边界 |
|---|---|---|---|
| 抽取式 QA | 稳定、可定位证据 | 不擅长改写归纳 | 事实片段明确 |
| 生成式 QA | 表达自然、可融合信息 | 易幻觉 | 开放问答、对话 |
| 检索增强生成 | 平衡灵活与准确 | 系统复杂 | 知识库问答 |
| 规则/查询系统 | 可控、可审计 | 覆盖面窄 | 结构化数据 |

检索增强生成通常叫 RAG，意思是 Retrieval-Augmented Generation，即“先检索，再生成”。严格说，很多工程中的生成式问答都应该做成 RAG，而不是只依赖模型参数里的记忆。

选择方案时可以按问题类型判断：

| 问题类型 | 推荐方案 |
|---|---|
| “订单号是多少” | 数据库查询或字段抽取 |
| “合同中付款期限是哪一段” | 抽取式 QA |
| “退款多久到账” | RAG 或生成式 QA |
| “这三份政策有什么差异” | 生成式 QA |
| “用户是否满足某条规则” | 规则系统 + 证据解释 |

对零基础到初级工程师来说，最重要的判断标准是：如果答案必须精确定位到原文片段，优先考虑抽取式方法；如果答案需要综合多段信息并改写成自然语言，生成式问答更合适；如果系统要上线给真实用户，优先采用“检索 + 生成 + 引用 + 拒答”的结构，而不是让模型直接回答。

生成式问答的正确定位不是替代所有问答系统，而是在证据约束下提升答案表达能力。它让系统能回答更复杂的问题，但同时也要求工程上认真处理检索、证据、评估和边界。
