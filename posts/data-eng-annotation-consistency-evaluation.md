## 核心结论

标注一致性评估，解决的不是“标签对不对”，而是“不同标注者面对同一份样本时，能不能稳定地给出相近判断”。这里的“稳定”可以理解为：换一个人来标，结果不会大幅漂移。对监督学习、评测集构建、RLHF 偏好数据、语音质量评分这类任务，一致性几乎就是数据质量的上限。因为如果人和人之间都经常达不成共识，模型学到的就很可能是噪声，而不是规则。

关键点在于，不能只看原始一致率。原始一致率是“表面上有多少次打了一样的标签”，但它会把“随机碰巧一样”也算进去。标注一致性指标做的事，本质上是把这部分随机对齐扣掉，得到“净一致率”。最常见的三个量是：

| 指标 | 适用标注者数量 | 适用数据类型 | 是否可处理等级差异 |
| --- | --- | --- | --- |
| Cohen's Kappa | 2 人 | 名义分类为主 | 一般不处理，需加权变体 |
| Fleiss' Kappa | 3 人及以上 | 名义分类 | 通常不处理等级距离 |
| Krippendorff's Alpha | 2 人及以上 | 名义、序数、区间、比率 | 可以 |

它们都在比较两件事：实际观察到的一致程度，和随机情况下本来就会发生的一致程度。以 Cohen's Kappa 为例：

$$
\kappa = \frac{p_o - p_e}{1 - p_e}
$$

其中，$p_o$ 是观测一致率，$p_e$ 是预期一致率。可以把它理解成：

$$
\text{净一致率}=\frac{\text{原始一致率}-\text{随机一致率}}{1-\text{随机一致率}}
$$

玩具例子最能说明问题。假设两位标注者对 4 条文本做 A/B 二分类：

| 样本 | 标注者1 | 标注者2 |
| --- | --- | --- |
| 1 | A | A |
| 2 | A | B |
| 3 | B | B |
| 4 | B | B |

两人有 3 次一致，所以 $p_o=3/4=0.75$。如果只看 75%，新手很容易觉得“已经不错了”。但再看类别分布：标注者1 的 A/B 比例是 2/2，标注者2 的 A/B 比例是 1/3，所以随机情况下碰巧一致的概率为：

$$
p_e = P(A)\_1P(A)\_2 + P(B)\_1P(B)\_2 = \frac{2}{4}\cdot\frac{1}{4} + \frac{2}{4}\cdot\frac{3}{4}=0.5
$$

于是：

$$
\kappa = \frac{0.75-0.5}{1-0.5}=0.5
$$

这说明“75% 表面一致”扣掉随机因素后，只剩中等水平。结论不是“数据不能用”，而是“指南还不够清楚，或者任务定义存在歧义，需要继续校准”。

## 问题定义与边界

标注一致性只回答一个问题：多个标注者对同一样本是否达成一致。它不直接衡量模型性能，不等于准确率，也不等于召回率。准确率要和“真值”比较；一致性只看“人和人之间”是否接近。

对零基础读者，最容易混淆的是三个概念：

| 概念 | 比较对象 | 关注问题 |
| --- | --- | --- |
| 原始一致率 | 标注者 vs 标注者 | 两个人表面上有多少次选了一样的标签 |
| 准确率 | 预测 vs 真值 | 模型或标注结果是否正确 |
| 一致性系数 | 观测一致 vs 随机一致 | 扣掉“乱猜也可能一样”的成分后，还剩多少真实共识 |

原始一致率很直观。比如两个同事分别独立审核 100 条评论是否违规，有 80 条给出相同标签，那么原始一致率是 80%。但如果任务是简单二分类，而且两个人都倾向于把大多数样本标成“正常”，那么这 80% 里面可能有不少只是因为类别本来就不平衡。

因此，判断任务是否明确，流程不是“直接看一致率”，而是：

| 步骤 | 含义 |
| --- | --- |
| 原始一致率 $p_o$ | 先看表面上有多少次打成一样 |
| 预期一致率 $p_e$ | 再估计随机情况下本来会有多少次一样 |
| 净一致率 | 最后看扣除随机对齐之后还剩多少 |

如果是最简单的二分类，且双方各自随机以 50% 概率选 A 或 B，那么纯随机一致率就是：

$$
0.5\times0.5 + 0.5\times0.5 = 0.5
$$

也就是说，两个人哪怕根本没理解任务，仅靠随机猜，也会有 50% 的表面一致。所以“60% 一致”在这种场景下未必值得高兴。

适用边界也要说清楚。并不是所有项目都必须做一致性评估。

| 适用条件 | 为什么适用 |
| --- | --- |
| 同一批样本由多位标注者独立标注 | 才能比较人和人是否一致 |
| 分类、评分、偏好选择任务 | 有明确标签空间，能统计一致/不一致 |
| 数据集要进入训练或评测主流程 | 一致性直接影响数据可信度 |
| 要评估指南质量或标注团队稳定性 | 指标能反映流程是否稳定 |

| 不适用条件 | 为什么不适用 |
| --- | --- |
| 只有单人标注 | 没有比较对象 |
| 纯自动规则生成标签 | 关注的是规则正确性，不是多人一致 |
| 开放式创作任务 | “一致”本身可能不是目标 |
| 样本没有重复标注 | 无法计算标注者间一致性 |

所以，一致性评估是“多人标注质量控制工具”，不是通用质量分数。

## 核心机制与推导

### 1. Cohen's Kappa：两位标注者的净一致率

Cohen's Kappa 适用于两位标注者、离散标签。它的核心思想很直接：把观测一致率 $p_o$ 里的随机成分 $p_e$ 扣掉，再按上限归一化。

还是看刚才的 4 条文本玩具例子。两位标注者的标签分别是：

- 标注者1：A, A, B, B
- 标注者2：A, B, B, B

第一步，算观测一致率：

$$
p_o=\frac{3}{4}=0.75
$$

第二步，按各自边际分布计算预期一致率。所谓“边际分布”，就是每个标注者总体上有多少比例选 A，多少比例选 B。这里：

- 标注者1：$P(A)=0.5, P(B)=0.5$
- 标注者2：$P(A)=0.25, P(B)=0.75$

于是：

$$
p_e = \sum_c P_1(c)P_2(c)=0.5\times0.25+0.5\times0.75=0.5
$$

第三步，代入 Kappa：

$$
\kappa=\frac{0.75-0.5}{1-0.5}=0.5
$$

它的输出范围通常在 $[-1,1]$。$1$ 表示完全一致，$0$ 表示和随机一致没有区别，负值表示比随机还差，通常意味着标注定义或流程出现严重问题。

### 2. Fleiss' Kappa：多位标注者的样本级一致率

当标注者不止两位时，常见做法是 Fleiss' Kappa。它不再直接对“两个具体人”计算，而是先看每个样本内部的一致程度，再汇总到整体。

设某个样本有 $n$ 位标注者，类别 $j$ 上的票数是 $n_{ij}$，那么该样本的一致率为：

$$
P_i=\frac{1}{n(n-1)}\sum_j n_{ij}(n_{ij}-1)
$$

白话解释：把同一样本内部所有“成对比较”的机会列出来，看有多少对给了相同标签。整体观测一致率 $\bar P$ 是所有样本的平均；总体预期一致率 $P_e$ 根据全局类别占比计算。最后：

$$
\kappa_{Fleiss}=\frac{\bar P-P_e}{1-P_e}
$$

它适合“每个样本由 3 名或更多标注者独立打标签”的项目，例如内容审核、多轮偏好标注、问答质量评级。

### 3. Krippendorff's Alpha：从“一致率”转成“不一致度”

Krippendorff's Alpha 更通用。它不一定从“一致了多少”出发，而是从“不一致了多少”出发：

$$
\alpha = 1-\frac{D_o}{D_e}
$$

其中，$D_o$ 是实际观察到的不一致度，$D_e$ 是随机情况下的预期不一致度。为什么这样设计重要？因为有些任务不是简单分类，而是 1 到 5 的序数评分。此时“1 和 2 的分歧”显然比“1 和 5 的分歧”轻，简单地记成“同意/不同意”会损失信息。

比如 3 位标注者对一段 TTS 音频打自然度分数：

- 样本甲：4, 4, 5
- 样本乙：1, 3, 5

如果只按“是否相同”统计，甲和乙都会有分歧；但工程上两者完全不同。甲说明大家都觉得接近“好”，乙说明对音频质量理解完全分裂。Krippendorff's Alpha 可以给评分差定义距离，例如序数任务里常用平方差：

$$
\delta(c,c')=(c-c')^2
$$

这样，$(4,5)$ 的差距是 1，$(1,5)$ 的差距是 16。Alpha 会把这种“分歧程度”纳入统计，而不是仅仅记录“相同/不同”。

三种指标放在一起看，会更容易选型：

| 指标 | 输入 | 主要思想 | 输出范围 |
| --- | --- | --- | --- |
| Cohen's Kappa | 两位标注者标签向量 | 观测一致率减去随机一致率 | 通常 $[-1,1]$ |
| Fleiss' Kappa | 样本 × 多标注者计数矩阵 | 样本级配对一致率做整体归一化 | 通常 $[-1,1]$ |
| Krippendorff's Alpha | 可含缺失值的标注矩阵 | 实际不一致度与预期不一致度之比 | 通常 $[-1,1]$ |

## 代码实现

工程里最常落地的是先实现 Cohen's Kappa，再在需要时扩展到 Fleiss 或 Alpha。原因很简单：它最容易验证，也最适合做标注流程的第一层监控。

下面是一个可运行的 Python 版本，输入是两位标注者的标签列表，输出是 $\kappa$。代码里的每一步都对应前面的公式。

```python
from collections import Counter

def cohens_kappa(labels_a, labels_b):
    """
    输入:
        labels_a, labels_b: 两位标注者对同一批样本的标签列表
    输出:
        kappa: Cohen's Kappa
    """
    assert len(labels_a) == len(labels_b), "两个列表长度必须相同"
    assert len(labels_a) > 0, "样本数必须大于 0"

    n = len(labels_a)

    # p_o: 原始观测一致率
    observed_agreement = sum(a == b for a, b in zip(labels_a, labels_b)) / n

    # 统计两个标注者各自的标签分布，用来计算 p_e
    count_a = Counter(labels_a)
    count_b = Counter(labels_b)

    # 所有出现过的标签并集
    categories = set(count_a) | set(count_b)

    # p_e: 随机情况下的预期一致率
    expected_agreement = 0.0
    for c in categories:
        p_a = count_a[c] / n
        p_b = count_b[c] / n
        expected_agreement += p_a * p_b

    # 如果 1 - p_e == 0，说明类别分布极端，Kappa 不可分辨
    assert (1 - expected_agreement) > 0, "预期一致率为 1，Kappa 无法计算"

    kappa = (observed_agreement - expected_agreement) / (1 - expected_agreement)
    return kappa

# 玩具例子
rater1 = ["A", "A", "B", "B"]
rater2 = ["A", "B", "B", "B"]

k = cohens_kappa(rater1, rater2)
assert round(k, 4) == 0.5

# 一个更高一致性的例子
rater3 = ["A", "A", "B", "B", "A", "B"]
rater4 = ["A", "A", "B", "B", "A", "A"]
k2 = cohens_kappa(rater3, rater4)
assert k2 > 0.6

def quality_advice(kappa):
    if kappa < 0.6:
        return "一致性偏低：优先检查标注指南、样本歧义和培训质量"
    elif kappa < 0.8:
        return "一致性可用：建议继续抽样复核并迭代指南"
    else:
        return "一致性较高：可以进入更大规模标注或训练流程"

print("toy kappa =", k)
print("better kappa =", round(k2, 4))
print(quality_advice(k))
```

这段代码的可复用点在于：先统计标签分布，再统一计算观测一致率和预期一致率。对工程流水线来说，它可以接在标注平台导出的 CSV 之后，用于日常监控。

如果扩展到 Fleiss' Kappa，输入会从“两个列表”变成“每个样本在各类别上的票数矩阵”。如果扩展到 Krippendorff's Alpha，核心变化是：不再只统计“是否相同”，而是引入距离函数 $\delta(c,c')$，把“差一点”和“差很多”区分开。

一个真实工程例子是内容安全团队的评论审核。假设每条评论由 3 位审核员标记为“正常 / 低风险 / 高风险”。工程上可以这样组织输入：

| 样本ID | 正常 | 低风险 | 高风险 |
| --- | --- | --- | --- |
| x1 | 3 | 0 | 0 |
| x2 | 1 | 2 | 0 |
| x3 | 0 | 1 | 2 |

这种矩阵天然适合 Fleiss' Kappa。若任务升级为 1 到 5 的严重程度评分，则应优先考虑 Krippendorff's Alpha，而不是继续强行把问题压扁成“同意/不同意”。

## 工程权衡与常见坑

低一致性不一定等于“标注员做错了”。更常见的情况是：任务定义本身就模糊，或者规则没有写到足够可执行。所谓“可执行”，就是不同人拿到指南后，能根据相同条件做出大致相同判断。

真实工程例子：TTS 质量评估中，经常会让多位专家对“自然度”“清晰度”“韵律”打分。如果某一轮自然度评分的 Alpha 明显下滑，正确动作通常不是立刻扩大抽样，甚至不是立即更换标注员，而是先回头检查三件事：

1. “自然度”的定义是否区分了发音错误、停连不自然、情感不匹配。
2. 标注示例是否覆盖了边界样本，比如“音质很好但情感偏冷”的音频该打几分。
3. 标注者培训时是否统一过尺度，避免有人把 3 分当“合格”，有人把 3 分当“偏差”。

如果这些前置条件没做好，多算十次一致性也没有意义，因为你只是在更稳定地测量同一套混乱标准。

常见坑可以归纳如下：

| 常见坑 | 典型表现 | 更合理的处理 |
| --- | --- | --- |
| 任务本身模糊 | 同一类样本长期争议大 | 缩窄任务定义，拆分标签 |
| 指南描述含糊 | 标注者总在边界样本分歧 | 补充反例、边界例子和优先级规则 |
| 标注者背景差异大 | 经验丰富者和新手打分完全不同 | 做培训和校准会，先小批量试标 |
| 类别极不平衡 | 原始一致率高，但 Kappa 很低 | 同时看类别分布，避免被“全选主类”误导 |
| 只算指标不改流程 | 每周都报低分，但问题重复出现 | 建立“低一致性 -> 复盘 -> 改指南 -> 重测”闭环 |
| 分歧处理随意 | 有时多数投票，有时主管拍板 | 明确升级路径：多数投票、专家裁决、样本下线 |

一个常见反面案例是：团队发现 Kappa 只有 0.42，于是继续抽更多样本、增加更多标注者，但指南完全没改。结果只是更高成本地确认“大家确实没共识”。一致性指标是诊断工具，不是自动修复工具。

实际流程更像这样：

| 阶段 | 目标 |
| --- | --- |
| 小批量试标 | 找出最容易产生分歧的样本 |
| 计算一致性 | 判断问题是偶发还是系统性 |
| 复盘分歧 | 归因到任务定义、样本质量、培训或标签设计 |
| 更新指南 | 增加规则、反例、优先级和例外说明 |
| 再评估 | 验证修改是否真的提升一致性 |

分歧样本的处理也要制度化。最常见的是多数投票，即票多者为最终标签；但这只适合低风险、低争议场景。对于高风险样本，例如医疗文本、金融投诉、模型安全红队数据，更稳妥的是专家裁决。专家裁决可以理解为：把高争议样本升级给更熟悉业务边界的人处理，并顺便把裁决理由写回指南。

## 替代方案与适用边界

不是每个项目都需要完整地上 Kappa 或 Alpha。选择方法时，真正要看的不是“哪个指标更高级”，而是任务复杂度、预算和误差容忍度。

先给一个实用对比：

| 方法 | 标注者数要求 | 适合任务 | 优点 | 局限 |
| --- | --- | --- | --- | --- |
| Cohen's Kappa | 2 人 | 二人分类复核 | 简单、直观、易落地 | 只适合两人 |
| Fleiss' Kappa | 3 人及以上 | 多人分类审核 | 适合批量质检 | 主要面向名义分类 |
| Krippendorff's Alpha | 2 人及以上 | 评分、序数、连续尺度 | 通用性强，可处理等级差 | 计算和解释更复杂 |
| 多数投票 + 置信度 | 2 人及以上 | 预算有限的数据生产 | 操作简单，便于上线 | 不能直接反映“规则是否清晰” |
| 蒙特卡洛一致性模拟 | 灵活 | 复杂聚合或不规则抽样 | 可做鲁棒性分析 | 实施成本高，不适合入门项目 |

对新手团队，一个常见的预算受限方案是“多数投票 + 置信度聚合”。流程可以很简单：

1. 每个样本找 3 个标注者。
2. 最终标签采用多数投票。
3. 统计投票分布作为置信度，例如 3:0 记为高置信，2:1 记为中置信。
4. 把 2:1 样本优先送复核，把 3:0 样本直接入库。

这套方案的价值在于工程上便宜、直接，而且能快速把高争议样本筛出来。它的缺点是：它能帮你“选出一个最终标签”，但不能充分回答“为什么大家总在这里分歧”。因此，如果项目已经进入稳定生产阶段，或者数据会直接影响高成本训练，就不该只停留在多数投票。

什么时候必须引入 Krippendorff's Alpha？一个清晰边界是：当标签不是无序类别，而是有顺序或距离意义的评分。比如：

- 1 到 5 的回答质量评分
- 1 到 7 的用户满意度评分
- 音频 MOS 打分
- 毒性强弱等级

这时若仍然只算 Kappa 式的“相同/不同”，会把“4 和 5”与“1 和 5”同样记成不一致，损失太大。Alpha 更适合，因为它允许你定义不一致距离。

相反，如果项目只是做简单日志分类，且最终只是用来做自动化报表，标签本身风险低、可接受少量噪声，那么用重复抽样加原始一致率监控就可能已经足够，不必一开始就引入更复杂的统计量。

因此，选型原则可以压缩成一句话：先看任务是不是多人标注，再看标签有没有等级结构，最后看你是要“做质量诊断”，还是只是要“产出最终标签”。
