## 核心结论

文本聚类是把一批无标签文本自动分组的方法：输入是文档集合，输出是若干簇，每个簇内的文本语义更接近。它的目标不是“预测已有标签”，而是在没有人工标注时发现数据结构，常用于探索性分析、话题归纳、工单看板和粗粒度去重。

一句总流程可以写成：

$$
\text{文本集合} \rightarrow \text{向量表示} \rightarrow \text{相似度/距离计算} \rightarrow \text{分组或主题分布}
$$

新手版例子：客服系统里有“登录失败”“验证码收不到”“账号被锁”“退款失败”“订单未到账”这些工单。文本聚类可以把前三条自动放到“账号登录问题”附近，把后两条放到“支付订单问题”附近。这个过程不需要先人工标注“登录类”“退款类”，模型只根据文本内容的相似性分组。

常见方法可以先按定位理解：

| 方法 | 定位 | 输出形式 | 典型用途 |
|---|---|---|---|
| K-Means | 硬聚类 | 每篇文档一个簇编号 | 大量文本快速分组 |
| 层次聚类 | 分层合并 | 树状结构或簇编号 | 小中规模文本的层级分析 |
| LDA | 主题发现，不是严格聚类 | 每篇文档的主题比例 | 解释语料中有哪些主题 |

硬聚类是指每篇文档只能属于一个簇。主题发现是指判断文档中混合了哪些主题，以及各主题占比。LDA 常被放在“文本聚类”章节里讲，但更准确地说，它是主题模型，不是严格的单标签聚类算法。

## 问题定义与边界

文本聚类的输入是一批无标签文本，输出是若干个簇。簇就是一组相似对象的集合。在文本场景里，一个簇通常表示一批语义接近、词汇接近或业务含义接近的文档。

输入输出可以写成：

```text
文本集合
  -> 向量集合
  -> 簇编号 / 主题分布
```

例如用户给出 1000 条客服工单，没有人工分类标签，只想知道大致有哪几类问题，这就是聚类问题。如果这些工单已经有“退款 / 登录 / 物流”标签，并希望训练模型给新工单打标签，那是分类问题。

分类是从已知标签中学习映射关系；聚类是在没有标签时寻找数据内部结构。相似检索是给定一条查询文本，找最相似的若干条文本；它不一定要把全量数据分成若干组。摘要是压缩内容，不负责分组。

| 对比项 | 输入 | 输出 | 是否需要人工标签 | 主要目标 |
|---|---|---|---|---|
| 聚类 | 无标签文本集合 | 簇编号 | 不需要 | 自动分组 |
| 分类 | 带标签训练集 + 新文本 | 预定义标签 | 需要 | 预测类别 |
| 主题模型 | 文档集合 | 主题词 + 文档主题比例 | 不需要 | 解释主题结构 |
| 相似检索 | 查询文本 + 文档库 | 相似文档排序 | 不一定 | 找近邻文本 |

LDA 的边界尤其要明确。LDA 会输出“这篇文档有 60% 支付主题、30% 订单主题、10% 账号主题”这类比例。它适合解释主题混合关系，但不适合直接当成严格簇标签。工程上可以取最大主题作为近似标签，但这只是后处理，不是 LDA 本身的目标。

## 核心机制与推导

文本聚类的第一步是把文本变成向量。向量是由数字组成的表示形式，算法不能直接理解自然语言，只能处理数字。

TF-IDF 是最经典的文本向量化方法。TF 是词频，表示一个词在当前文档中出现得多不多；IDF 是逆文档频率，表示这个词在整个语料里是否稀有。TF-IDF 的直觉是：一个词在当前文档中频繁出现，同时在其他文档中不常见，它就更能代表这篇文档。它适合稀疏词袋表示。稀疏是指向量维度很多，但大部分位置都是 0。

词向量和句向量是现代文本表示方法。词向量把词映射成稠密向量，句向量把整句话或整篇短文本映射成稠密向量。稠密是指大部分维度都有数值。它们更强调语义接近性，例如“退款失败”和“无法退钱”字面不完全相同，但语义接近。

新手版例子：两篇文档都提到“退款”和“订单”，TF-IDF 会让它们在关键词维度上更接近。K-Means 会倾向于把它们分到同一簇。LDA 会说它们都含有“支付主题”和“订单主题”，但比例可能不同。

K-Means 的核心是找 $K$ 个中心点，并让每篇文档归到最近的中心。$K$ 是预先指定的簇数。它的目标函数是最小化类内平方和：

$$
\min_{\{C_k\},\{\mu_k\}} \sum_{i=1}^{N}\|x_i-\mu_{z_i}\|^2,\quad \mu_k=\frac{1}{|C_k|}\sum_{i:z_i=k}x_i
$$

其中 $x_i$ 是第 $i$ 篇文档的向量，$z_i$ 是它所属的簇编号，$C_k$ 是第 $k$ 个簇，$\mu_k$ 是第 $k$ 个簇的中心。K-Means 的迭代过程很直接：先初始化中心，再把每个点分配给最近中心，然后重新计算中心，直到结果稳定。

玩具例子：把 3 篇文档简化成 2 维向量：

| 文档 | 向量 | 含义 |
|---|---:|---|
| $d_1$ | $(1,0)$ | 登录失败 |
| $d_2$ | $(0.8,0.2)$ | 验证码登录失败 |
| $d_3$ | $(0,1)$ | 退款失败 |

设 $K=2$，初始中心取 $c_1=d_1$，$c_2=d_3$：

| 文档 | 到 $c_1$ 的距离平方 | 到 $c_2$ 的距离平方 | 分配结果 |
|---|---:|---:|---|
| $d_1$ | 0 | 2 | 簇 1 |
| $d_2$ | 0.08 | 1.28 | 簇 1 |
| $d_3$ | 2 | 0 | 簇 2 |

更新后 $\mu_1=((1,0)+(0.8,0.2))/2=(0.9,0.1)$，$\mu_2=(0,1)$。这说明前两篇更接近，会被放到同一组。

层次聚类从每篇文档单独成簇开始，按距离不断合并最近的簇，最终形成树状结构。linkage 距离是两个簇之间距离的定义方式。单连接看两个簇中最近两个点的距离，全连接看最远两个点的距离，平均连接看所有点对距离的平均值。单连接容易形成链式结构，全连接更紧凑，平均连接通常更稳定。

LDA 的核心不是找中心，而是用生成过程解释文本。生成式方法是指先假设数据如何被生成，再反推隐藏结构。LDA 假设每篇文档由多个主题混合生成，每个主题又由一组词的概率分布构成：

$$
\theta_d \sim \mathrm{Dir}(\alpha),\quad \phi_k \sim \mathrm{Dir}(\beta),\quad z_{dn}\sim \mathrm{Cat}(\theta_d),\quad w_{dn}\sim \mathrm{Cat}(\phi_{z_{dn}})
$$

其中 $\theta_d$ 是文档 $d$ 的主题分布，$\phi_k$ 是主题 $k$ 的词分布，$z_{dn}$ 是第 $d$ 篇文档第 $n$ 个词的主题，$w_{dn}$ 是实际生成的词。

| 方法 | 是否需要预先指定簇数 | 输出 | 是否适合解释性分析 |
|---|---|---|---|
| K-Means | 需要指定 $K$ | 硬分组 | 中等，需看中心词 |
| 层次聚类 | 可指定簇数或距离阈值 | 层级结构 / 硬分组 | 较好，可看合并树 |
| LDA | 需要指定主题数 | 主题比例 | 较好，可看主题词 |

## 代码实现

工程上最常见的起步方式是先用 scikit-learn 做基线：清洗文本，做分词，用 TF-IDF 向量化，再跑 K-Means 或层次聚类。中文场景要先处理分词、停用词和低频词，否则模型看到的是噪声。

下面代码用一个很小的工单集合演示完整流程。为了让示例最小可运行，分词用一个简单词表函数模拟；真实项目中通常会换成 jieba、pkuseg、HanLP 或业务词典分词。

```python
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.cluster import KMeans, AgglomerativeClustering

texts = [
    "登录失败 验证码 收不到",
    "账号 被锁 登录 不了",
    "退款失败 订单 未到账",
]

def tokenize(text):
    # 示例文本已经用空格切好词；真实中文项目应替换为正式分词器。
    return text.split()

vectorizer = TfidfVectorizer(
    tokenizer=tokenize,
    token_pattern=None,
    lowercase=False,
)

X = vectorizer.fit_transform(texts)

kmeans = KMeans(n_clusters=2, random_state=0, n_init=20)
kmeans_labels = kmeans.fit_predict(X)

dense_X = X.toarray()
hier = AgglomerativeClustering(n_clusters=2, linkage="average")
hier_labels = hier.fit_predict(dense_X)

print("features:", vectorizer.get_feature_names_out().tolist())
print("kmeans:", list(zip(texts, kmeans_labels)))
print("hierarchical:", list(zip(texts, hier_labels)))

# 断言不依赖具体簇编号，只验证前两条更接近，第三条不同。
assert kmeans_labels[0] == kmeans_labels[1]
assert kmeans_labels[2] != kmeans_labels[0]
assert hier_labels[0] == hier_labels[1]
assert hier_labels[2] != hier_labels[0]
```

代码流程可以拆成五步：

| 步骤 | 输入 | 输出 | 说明 |
|---|---|---|---|
| 输入文本 | 原始工单 | 文本列表 | 每条文本是一篇短文档 |
| 预处理 | 文本列表 | 分词结果 | 清洗符号、分词、去停用词 |
| 向量化 | 分词结果 | TF-IDF 矩阵 | 每篇文档变成一个向量 |
| 聚类 | 向量矩阵 | 簇编号 | K-Means 或层次聚类 |
| 解释结果 | 簇编号 + 原文 | 分组视图 | 看每簇高频词和样本文本 |

LDA 的代码结构类似，但输出含义不同。它通常接 CountVectorizer 或 TF 向量，而不是直接输出一个严格簇标签：

```python
from sklearn.feature_extraction.text import CountVectorizer
from sklearn.decomposition import LatentDirichletAllocation

docs = [
    "登录 失败 验证码 收不到",
    "账号 登录 被锁",
    "退款 失败 订单 未到账",
    "订单 支付 退款 异常",
]

count_vectorizer = CountVectorizer(tokenizer=str.split, token_pattern=None)
X_count = count_vectorizer.fit_transform(docs)

lda = LatentDirichletAllocation(n_components=2, random_state=0)
doc_topic = lda.fit_transform(X_count)

terms = count_vectorizer.get_feature_names_out()
for topic_id, weights in enumerate(lda.components_):
    top_terms = [terms[i] for i in weights.argsort()[-3:][::-1]]
    print(topic_id, top_terms)

assert doc_topic.shape == (4, 2)
assert abs(doc_topic[0].sum() - 1.0) < 1e-6
```

真实工程例子：客服工单系统每天进入 10 万条短文本。第一版可以用“中文分词 + TF-IDF + MiniBatchKMeans”做聚类看板，把工单分到“登录异常”“支付退款”“物流配送”“账号安全”等粗粒度簇。运营人员每天查看每个簇的数量变化、代表文本和关键词。如果目标是分析历史问题主要集中在哪些主题，可以另跑 LDA，输出“支付、账号、安全、物流”等主题词和每篇工单的主题比例。

## 工程权衡与常见坑

文本聚类的失败大多来自表示和距离，而不是算法名称。把文本表示错了，再换十个聚类算法也不会稳定。原始词频会偏向长文档，因为长文本天然有更多词；高维稀疏文本里，欧氏距离容易受维度和稀疏性影响；K-Means 对初始化敏感，不同随机种子可能得到不同结果。

中文文本还需要额外处理。分词是把连续中文句子切成词的过程。停用词是“的、了、是”这类高频但信息量低的词。低频词是出现次数极少、通常不稳定的词。如果“订单已取消”“订单取消中”“取消订单失败”没有正确分词，模型可能只看到很多碎片，最后把不相关内容混在一起。

| 问题 | 表现 | 规避方式 |
|---|---|---|
| 长文档主导 | 长文本之间更容易聚在一起 | 使用 TF-IDF、向量归一化或按段落处理 |
| K 值难选 | 簇太少会混杂，簇太多会碎片化 | 用肘部法、轮廓系数和人工抽样评估 |
| 初始化敏感 | 每次运行结果不一致 | 固定随机种子，多次运行取稳定结果 |
| 距离度量不合适 | 语义近的文本被分开 | 稀疏文本优先考虑 cosine 相似度 |
| LDA 不是硬分类 | 一篇文档同时属于多个主题 | 用主题比例解释，不直接当单标签 |
| 层次聚类成本高 | 数据一大就慢，占内存 | 先采样、先粗聚类，或换 K-Means |
| 分词质量差 | 主题词混乱，簇不可解释 | 加业务词典、停用词表和低频过滤 |
| 噪声字段混入 | 时间、编号、模板句影响聚类 | 清洗订单号、手机号、URL、重复模板 |

一个实用原则是：

$$
\text{稳定聚类} \approx \text{合适的文本表示} + \text{合理的距离度量} + \text{可解释的人工抽样检查}
$$

稀疏 TF-IDF 文本常优先考虑 cosine 相似度。cosine 相似度衡量两个向量方向是否接近，能减弱文本长度差异。结果不稳时，不要只看一次输出，应多次随机种子运行，比较簇大小、关键词和样本文本是否稳定。

还有一个常见误区是把聚类结果当成真实类别。聚类给出的簇只是算法根据当前表示和距离得到的结构，不等于业务真相。工程上通常要抽样检查每个簇的代表文本，给簇命名，再决定是否进入报表或下游系统。

## 替代方案与适用边界

选方法时先看任务目标。想要硬分组，优先考虑 K-Means 或层次聚类。想解释主题结构，考虑 LDA。想捕捉更强语义，考虑句向量加聚类。句向量可以来自 BERT、Sentence-BERT 或企业内部向量模型，它更适合“字面不同但意思接近”的文本。

| 任务目标 | 推荐方法 | 不推荐原因 |
|---|---|---|
| 大量短文本快速分组 | TF-IDF + K-Means | 层次聚类在大规模下成本高 |
| 少量文本看层级结构 | 层次聚类 | K-Means 不能直接给树状关系 |
| 解释语料主题 | LDA | K-Means 的中心不一定有清晰主题语义 |
| 语义表达要求高 | 句向量 + K-Means / HDBSCAN | 纯 TF-IDF 对同义改写不敏感 |
| 已有稳定标签 | 文本分类模型 | 聚类不会主动学习标签含义 |
| 只找相似案例 | 向量检索 | 聚类会强制全局分组，目标不同 |

简短决策树：

```text
是否已有标签？
  是 -> 优先做分类
  否 -> 是否需要主题解释？
       是 -> LDA 或主题建模
       否 -> 是否需要硬分组？
            是 -> K-Means / 层次聚类
            否 -> 相似检索或可视化分析
```

客服场景可以这样选：想做“工单分组看板”，适合 K-Means；想看“历史问题主要集中在哪些主题”，适合 LDA；想分析少量重点工单并查看层级结构，适合层次聚类；想把新工单分到已有“退款、登录、物流”标签，应该做分类，而不是聚类。

数据规模也决定边界。层次聚类通常需要维护样本之间的距离关系，样本量很大时容易慢和占内存。K-Means 更适合大规模，但必须指定 $K$，并且默认更偏向球形簇。LDA 适合主题解释，但短文本上可能不稳定，常需要合并文本、调主题数或加强预处理。

最终可以按一句话落地：没有标签、要看自然分组，用聚类；没有标签、要解释主题混合，用 LDA；有标签、要预测类别，用分类；只想找相似文本，用检索。
