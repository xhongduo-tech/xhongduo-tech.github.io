## 核心结论

多目标排序的融合，指的是把多个业务目标压到同一个排序决策里。这里的“目标”就是系统希望同时变好的指标，比如点击率、转化率、观看时长、7日留存。排序系统最终只能输出一个顺序，所以必须把多个目标先变成一个可比较的统一分数，或者先满足约束再做排序。

最常见的做法是标量化。标量化的意思是，把多个目标压成一个标量，也就是一个单独数字。最基础形式就是加权求和：

$$
S_{final}=\sum_{i=1}^{n} w_i \cdot S_i,\quad \sum_{i=1}^{n} w_i=1
$$

其中 $S_i$ 是第 $i$ 个目标的预测分，$w_i$ 是该目标的权重。它简单、可控、易上线，但不能自动解决目标冲突。目标冲突的意思是，一个指标上升时，另一个指标可能下降，例如点击率上升但留存下降。

当目标之间关系更复杂时，常见增强方式有三类：

| 方法 | 核心思想 | 适合场景 |
|---|---|---|
| 加权求和 | 直接线性融合多个目标分 | 目标少、业务偏好明确 |
| 分层级联 | 先过硬约束，再在候选内优化次目标 | 商业规则强、风险控制严格 |
| 门控融合 / MMoE / PLE | 用模型动态决定不同目标的贡献或共享程度 | 特征复杂、目标冲突明显 |

一个最小玩具例子可以直接说明权重的作用。假设某内容的点击评分是 0.6，留存评分是 0.5：

- 若点击和留存权重都是 0.5，则最终分数是 $0.5 \times 0.6 + 0.5 \times 0.5 = 0.55$
- 若留存权重提高到 0.7，点击权重降到 0.3，则最终分数是 $0.3 \times 0.6 + 0.7 \times 0.5 = 0.53$

这说明权重调整会改变排序偏好。这里必须注意，权重偏向长期目标，不一定让单条样本分数变高，它真正改变的是“哪些样本更容易排前面”。如果另一个候选项留存更高，它在新权重下就更可能上升。很多工程误区就出在这里：把“单样本得分变化”误认为“整体业务收益变化”。

真实工程里，多目标排序不是纯数学问题，而是业务约束问题。点击、转化、时长、留存之间常常存在拉扯。仅靠一个固定公式通常不够，必须结合量纲归一化、目标约束、在线实验和长期指标监控一起做。

## 问题定义与边界

多目标排序的输入，通常是同一个候选内容在多个目标下的预估分。例如：

- 点击率预估 $\hat{p}_{click}$
- 转化率预估 $\hat{p}_{cvr}$
- 观看时长预估 $\hat{t}_{watch}$
- 留存收益预估 $\hat{r}_{retain}$

这里“预估”就是模型根据用户、内容、上下文特征，对未来行为做出的概率或数值判断。

输出则是一个统一排序分 $S_{final}$，系统据此决定谁排前面。也就是说，多目标排序不是同时输出四个榜单，而是必须把四个目标协调成一个最终决策。

问题边界主要有三点。

第一，目标可能冲突。比如短视频推荐里，标题党内容可能提升点击，但用户看完不满意，导致次日留存下降。此时“点击更高”不等于“系统更好”。

第二，目标量纲不一致。量纲就是数值单位和尺度。点击率是 $[0,1]$ 的概率，观看时长可能是几十秒或几百秒，GMV 可能是金额。如果不做归一化，数值大的目标会天然支配融合结果。

第三，目标时效不同。点击是短反馈，几分钟内就能观察；留存是长反馈，可能要到第 1 天、第 7 天才能看到。短期目标训练快、波动小，长期目标训练慢、噪声大，因此在联合优化里容易被压制。

下面这个表把常见目标和冲突形式列出来：

| 目标类型 | 常见量纲 | 典型冲突 | 常见应对 |
|---|---|---|---|
| 点击率 CTR | 概率 | 点击上升，留存下降 | 权重控制、加入负反馈特征 |
| 转化率 CVR | 概率 | 转化提升，内容丰富度下降 | 分层级联、商业位约束 |
| 观看时长 | 秒 / 分钟 | 时长提升，完播率下降 | 分段建模、归一化 |
| 留存 | 概率 / 增量值 | 留存提升，短期点击下降 | 长短期分层、多窗口监控 |

因此，多目标排序主要适用于推荐、搜索、广告这类“只能给一个顺序、但业务有多个目标”的场景。若业务本身只有单一目标，例如只关心点击的简单召回排序，或者候选规模很小且规则足够明确，就没有必要上来就做复杂多目标融合。

还要区分“多目标预测”和“多目标排序”。前者是分别建模多个任务，后者是在最终排序阶段把这些任务融合成一个决策。很多系统先有多任务预测模型，再有独立融合层；也有系统直接在一个联合模型里输出可排序分，但本质上仍然要回答同一个问题：多个目标如何取舍。

## 核心机制与推导

### 1. 线性加权

最常见的融合公式是：

$$
S_{final}=\sum_{i=1}^{n} w_i \cdot S_i
$$

如果有点击、转化、留存三个目标，则可以写成：

$$
S_{final}=w_1S_{click}+w_2S_{cvr}+w_3S_{retain}
$$

它的优点是简单透明，业务方可以直接理解“留存更重要，就把 $w_3$ 调高”。缺点也同样明显：它假设不同目标之间的边际影响近似线性，而真实业务里常常不是。

例如，对低质量内容来说，点击高但留存极差，系统未必希望它仅凭一个高点击分冲到前面。因此会出现非线性交互。

### 2. 乘法与门控加权

当希望“某个目标太差时应整体受罚”，可用乘法型融合。例如：

$$
S_{final}=S_{click}^{\alpha}\cdot S_{retain}^{\beta}
$$

乘法的特点是，一项很差会显著拖低总分。它比线性求和更强调“短板效应”。

进一步的做法是门控。门控的意思是，不再给所有样本固定权重，而是让模型根据上下文 $x$ 动态输出权重：

$$
S_{final}=\sum_{i=1}^{n} g_i(x)\cdot S_i,\quad \sum_{i=1}^{n} g_i(x)=1
$$

这里的 $g_i(x)$ 可以理解为“当前这个用户、这个场景下，第 $i$ 个目标应该占多大比重”。比如新用户阶段更重点击和新鲜感，老用户阶段更重留存和稳定满意度。

如果用一句白话解释，固定权重像写死的配方，门控像现场调配。不同请求进来，模型会自己决定该更偏向哪种目标。

用户要求中的“调音台”比喻可以保留，但必须明确它只是帮助理解，不是机制定义。可以这样理解：点击、转化、留存像三个声道，融合层像调音台，不同场景调不同音量；但真正上线时，这个“音量”要么是业务设定的权重，要么是模型学出的门控值。

### 3. 共享表示：MMoE 与 PLE

当多目标不只是“最后一层融合”，而是从特征提取阶段就存在关联时，会使用多任务学习结构，例如 MMoE 和 PLE。

“共享表示”就是多个任务共用一部分底层特征表达，以减少重复学习并利用任务间共性。

MMoE（Multi-gate Mixture-of-Experts）可以理解为“多个专家网络 + 每个任务一个门控网络”。专家网络负责提取不同类型的特征模式，门控网络负责决定某个任务更该信哪几个专家。它适合任务相关但不完全一致的场景。

PLE（Progressive Layered Extraction）是在 MMoE 基础上更进一步，把共享专家和任务专属专家分开，并且按层逐步提取。它适合目标冲突更明显的情况，因为它显式区分“共享知识”和“任务私有知识”。

一个简化结构图如下：

```text
输入特征 x
   |
共享底层表示
   |
+-----------------------------+
| 共享专家 E_shared_1 ... k   |
| 点击专家 E_click_1 ... m    |
| 留存专家 E_retain_1 ... m   |
| 转化专家 E_cvr_1 ... m      |
+-----------------------------+
   |         |          |
 gate_click gate_retain gate_cvr
   |         |          |
 tower_click tower_retain tower_cvr
   |         |          |
 S_click    S_retain    S_cvr
      \        |        /
       \       |       /
        最终融合层 S_final
```

如果用一个新手能理解的玩具例子，多个目标模型像不同口味的饮料：一个偏甜，一个偏酸，一个偏苦。门控不是“凭感觉混合”，而是根据当前用户口味动态给比例。PLE比简单加权更进一步，它先决定原料该从哪些桶里拿，再决定最后怎么混。

### 4. 约束化与 Pareto 视角

并不是所有多目标问题都要压成单一公式。有时业务更适合“先满足约束，再优化主目标”。

例如：

$$
\max S_{click}
$$

subject to

$$
S_{retain} \ge \tau
$$

其中 $\tau$ 是留存阈值。这个思路叫约束化优化。它适合“某个指标不能跌穿底线”的业务。

进一步，如果多个目标都很重要，且很难提前给出稳定权重，可以从 Pareto front，也就是帕累托前沿的角度搜索策略。它的含义是：如果一个方案想让某个目标继续变好，就必须牺牲至少一个其他目标，那么它就在候选前沿上。

形式化地说，方案 $A$ 支配方案 $B$，当且仅当：

- 对所有目标，$A$ 都不差于 $B$
- 至少一个目标上，$A$ 严格优于 $B$

不被任何其他方案支配的集合，就是 Pareto front。工程上常见做法不是在线实时求解，而是离线扫描多组权重、门控结构或策略参数，得到一组候选，再由业务选点上线做实验。

## 代码实现

下面先给一个最小可运行的 Python 例子。它展示三个核心动作：

1. 对不同目标分做归一化
2. 用固定权重做线性融合
3. 用简单门控做动态融合

```python
from math import exp

def minmax(x, lo, hi):
    assert hi > lo
    return (x - lo) / (hi - lo)

def softmax(xs):
    exps = [exp(x) for x in xs]
    s = sum(exps)
    return [v / s for v in exps]

def weighted_score(scores, weights):
    assert set(scores.keys()) == set(weights.keys())
    total_w = sum(weights.values())
    assert abs(total_w - 1.0) < 1e-9
    return sum(scores[k] * weights[k] for k in scores)

def gate_weights(user_age_days, is_new_user):
    # 一个玩具门控：新用户更看点击，老用户更看留存
    raw = {
        "click": 1.2 if is_new_user else 0.6,
        "retain": 0.4 if is_new_user else 1.3,
        "watch_time": 0.8 + min(user_age_days / 30.0, 0.5),
    }
    keys = ["click", "retain", "watch_time"]
    probs = softmax([raw[k] for k in keys])
    return {k: p for k, p in zip(keys, probs)}

# 候选内容的原始预估
raw_scores = {
    "click": 0.6,        # 概率
    "retain": 0.5,       # 概率
    "watch_time": 120.0, # 秒
}

# 归一化后再融合，避免量纲不一致
scores = {
    "click": raw_scores["click"],
    "retain": raw_scores["retain"],
    "watch_time": minmax(raw_scores["watch_time"], 0.0, 300.0),
}

fixed_weights = {
    "click": 0.5,
    "retain": 0.3,
    "watch_time": 0.2,
}

s1 = weighted_score(scores, fixed_weights)
assert 0.0 <= s1 <= 1.0

dynamic_weights = gate_weights(user_age_days=20, is_new_user=False)
assert abs(sum(dynamic_weights.values()) - 1.0) < 1e-9

s2 = weighted_score(scores, dynamic_weights)
assert 0.0 <= s2 <= 1.0

# 题目里的最小例子
toy_scores = {"click": 0.6, "retain": 0.5}
toy_weights = {"click": 0.5, "retain": 0.5}
assert abs(weighted_score(toy_scores, toy_weights) - 0.55) < 1e-9
```

这个例子故意保持简单，但已经覆盖了线上实现的基本骨架。真正工程里，一般会拆成三层：

| 层次 | 作用 | 常见实现 |
|---|---|---|
| 多目标预估层 | 输出各任务的 $S_i$ | 多任务网络、独立 tower |
| 融合层 | 计算统一排序分 | 加权、乘法、门控 |
| 策略层 | 做规则、约束、重排 | 保底、打散、商业约束 |

伪代码可以写成这样：

```python
def rank_score(x, model, fusion_config):
    scores = {
        "click": model.predict_click(x),
        "cvr": model.predict_cvr(x),
        "retain": model.predict_retain(x),
    }

    scores = normalize(scores, fusion_config.norm)

    if fusion_config.type == "weighted_sum":
        weights = fusion_config.weights
    elif fusion_config.type == "gate":
        weights = model.predict_gate(x)  # 小型全连接网络或门控塔
    else:
        raise ValueError("unknown fusion type")

    final_score = sum(weights[k] * scores[k] for k in scores)
    return final_score, scores, weights
```

这里的“可插拔性”非常关键。权重最好支持三种来源：

- 静态配置：适合快速实验
- 分桶配置：按频道、人群、场景配置不同权重
- 模型输出：由 gate 动态学习

添加新目标时，不能只在融合公式里多加一项，还要同步检查四件事：

- 是否有稳定标签
- 是否完成量纲处理
- 是否有线上监控拆解
- 是否定义了和现有目标的约束关系

真实工程例子可以看视频推荐。系统往往会同时建模点击、完播、观看时长、次日留存。实践中常见做法是：

- 用 PLE 抽取共享与专属特征
- 每个目标输出独立分数
- 最终按场景使用固定权重或 gate 融合
- 在线监控不仅看总点击，还看完播、留存、投诉率等拆解指标

如果线上只看一个“综合分”涨没涨，通常很快就会失真，因为你无法判断提升来自哪个目标，也无法定位谁在压制谁。

## 工程权衡与常见坑

多目标排序最难的不是把公式写出来，而是让它在线上长期稳定工作。下面这些坑几乎都会遇到。

| 坑 / 风险 | 观察信号 | 应对方式 |
|---|---|---|
| 目标冲突 | 点击涨、留存跌 | 调整权重，加入约束或 Pareto 搜索 |
| 量纲不一致 | 时长目标长期主导排序 | 归一化、分位数截断、标准化 |
| 短期压制长期 | 短期实验正向，长期回访变差 | 拉长观察窗，长期指标设底线 |
| 目标标签延迟 | 留存模型更新慢、波动大 | 多窗口训练、蒸馏短期代理目标 |
| 门控塌缩 | gate 长期只偏向一个目标 | 加熵正则、监控权重分布 |
| 配置复杂失控 | 不同场景权重过多难维护 | 建立统一配置中心和实验命名 |

### 1. 冲突不是异常，而是常态

很多团队第一次做多目标时，会把点击、时长、留存都放进同一个分数，然后期待“大家一起涨”。这通常不成立。推荐系统本质上是在有限曝光位里做取舍，取舍就意味着冲突。

一个常见新手现象是：点击上涨 3%，团队马上庆祝；一周后发现留存下降，才意识到问题。正确做法不是“先把点击做起来再说”，而是从一开始就定义主目标、护栏指标和不可跌破约束。

“护栏指标”就是不能只看主指标，还要盯住的防退化指标。例如主目标是点击，护栏可以是留存、投诉率、完播率。

### 2. 量纲必须先处理

点击率和留存率通常都在 0 到 1 之间，而观看时长是秒级数字。如果直接写：

$$
S = 0.4 \cdot CTR + 0.3 \cdot Retain + 0.3 \cdot WatchTime
$$

那即使 $WatchTime$ 只有几十秒，它也会直接碾压概率项，公式看起来是“30% 权重”，实际根本不是。

因此工程上常见处理包括：

- Min-Max 归一化
- Z-score 标准化
- 分位数归一化
- 对重尾指标取 $\log(1+x)$

这里“重尾”是指大多数样本很小，少数样本极大，比如观看时长、消费金额。

### 3. 长期目标容易被短期目标抢资源

这是多任务系统里很典型的问题。点击标签多、噪声小、反馈快，模型很容易优先学点击。留存标签稀疏、延迟长，训练时天然吃亏。结果是共享层越来越偏向短期信号。

MMoE 和 PLE 的价值就在这里。它们不是保证冲突消失，而是给模型一个更合理的参数分配方式，让长期目标至少能保留一部分专属表示空间。

真实工程里，如果用了 PLE，还应该监控这些信号：

- 各目标 tower 的 loss 曲线
- gate 在不同人群、场景下的分布
- 长期目标样本覆盖率
- AB 实验中各目标的拆解效果

如果 gate 长期只输出“点击占 90%”，那说明系统虽然名义上是多目标，实际上已经退化成单目标。

### 4. 线上实验必须看分解结果

做 AB 测试时，不应只看总收益或总分。更有用的视角是：

- 综合排序收益是否上涨
- 每个子目标分别怎么变
- 哪些人群受益，哪些人群受损
- 指标变化是短周期还是长周期
- 是否有分布漂移，例如某类内容被过度放大

一个典型真实工程例子是视频推荐的 PLE 融合。系统上线后发现：综合播放时长上涨，但 7 日留存下降。排查后发现 gate 对“高点击、低满意度”的内容赋权过高，导致模型不断放大短期刺激内容。修复策略可能包括：

- 给留存目标更高保底权重
- 对低满意内容增加惩罚项
- 重新采样长期正样本
- 在策略层增加用户疲劳控制

这类问题如果没有分解监控，通常会被误判成“模型效果波动”。

## 替代方案与适用边界

加权融合不是唯一答案，它只是最容易落地的答案。真正选型时，应该根据目标关系和业务约束来定。

| 策略 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| 固定加权 | 目标少、偏好稳定 | 简单透明、上线快 | 难适应场景差异 |
| 动态门控 | 用户和场景差异大 | 更灵活，能学到条件偏好 | 可解释性下降，训练更复杂 |
| 分层级联 | 有硬约束或风控规则 | 易控制，风险小 | 可能牺牲全局最优 |
| Pareto 搜索 | 权重难定、目标冲突强 | 能系统探索权衡边界 | 计算和实验成本高 |
| 优先级调度 | 某目标极端重要 | 决策明确 | 其他目标优化空间受限 |
| 强化学习 reward shaping | 长期收益主导、反馈链路长 | 更适合序列决策 | 建模和归因难度高 |

### 1. 何时坚持加权融合

如果满足下面三个条件，优先用加权融合通常是对的：

- 目标数量不多，2 到 4 个为主
- 业务已经知道大致优先级
- 团队需要快速实验和可解释性

这是大多数团队的第一站。因为只有先把简单方案跑稳，后面的门控、Pareto、强化学习才有比较基线。

### 2. 何时考虑 Pareto front

当团队怎么调权重都不满意时，往往不是“还没调够”，而是“问题结构已经不适合手工调权重”。此时可以离线生成多组配置，评估它们在多个目标上的表现，取非支配解集合，再挑几个代表点做在线实验。

Pareto 的判定条件可以简单理解为：

- 如果方案 A 在所有目标上都不差于方案 B
- 且至少一个目标更好
- 那么 B 没有继续保留的价值

这种方法适合“想系统性看清权衡边界”的阶段，不适合拿来替代所有线上实时打分。

### 3. 何时使用优先级调度或约束化

如果某个目标极端重要，例如广告转化、风控通过率、内容安全，那么平均加权经常不是好方案。此时更常见的做法是：

- 先筛掉不满足底线的候选
- 再在剩余集合里优化次级目标

例如电商排序里，如果商业转化是核心，且履约风险不能超阈值，那么通常会先做约束，再优化点击和时长，而不是让所有指标在一个权重公式里“公平竞争”。

### 4. 强化学习不是默认升级路线

很多文章会把多目标排序最终引到强化学习，但工程上要谨慎。强化学习适合处理序列决策和长期收益，比如“今天推什么，影响未来 7 天活跃”。但它要求更稳定的反馈归因、更严格的线上控制和更高的实验成本。

因此，一条更务实的路线通常是：

1. 固定权重打基础
2. 动态门控解决场景差异
3. Pareto 或约束化解决权重难定问题
4. 只有在长期目标特别关键时，再考虑 reward shaping 或强化学习
