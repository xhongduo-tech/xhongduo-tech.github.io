## 核心结论

Agent 的工作记忆可以理解为“模型当前这次推理真正看得到的那一小块信息”。它不是无限仓库，而是受上下文窗口硬限制的活跃区。窗口一满，新信息想进入，就必须有旧信息被淘汰。

这件事的关键不在“记住多少”，而在“留下什么”。如果只按最近使用时间保留，系统会偏向保留刚出现但价值不高的信息；如果只按访问次数保留，又容易让旧热点长期霸占位置。更有效的做法是给每条记忆计算重要性分数，把“与当前任务强相关、虽然不一定最新但必须保留”的事实留下来。

对复杂任务，重要性评分通常优于纯 LRU。原因不是它更“聪明”这个空话，而是它显式建模了三件事：相关性、时效性、频率。已有评测报告显示，带特征的价值感知淘汰，相比 LRU 的命中率提升大约在 $8.4\%\sim19.2\%$，在一些复杂任务里可对应约 9% 左右的任务完成率改善。

一个新手最容易理解的例子是对话助手。用户先说“我对青霉素过敏”，后面又聊了很多无关内容。如果系统只保留最新消息，这条关键约束可能被挤掉；如果系统把“过敏史”打高分，即使它不是最新消息，也会继续保留，从而避免后续推荐错误药物。

## 问题定义与边界

这里说的“工作记忆”，指的是 Agent 在当前轮推理中能直接读取并参与计算的上下文内容。白话说，就是模型眼前这张有限大小的草稿纸。超过这张纸的信息，不是彻底消失，而是不能直接参与本轮推理，除非先被检索回来。

从工程角度看，容量约束首先来自上下文窗口大小，常用近似写法是：

$$
\text{Capacity} \approx \text{token budget}
$$

如果讨论底层表示，也可以写成“窗口令牌数乘以每个令牌的表示维度”，但真正影响系统行为的不是向量维度本身，而是当前可装入窗口的有效信息量。有效信息量低，往往不是因为窗口太小，而是因为淘汰策略把无价值内容留了太久。

下面先把边界说清楚：

| 项目 | 含义 | 触发点 | 对结果的影响 |
|---|---|---|---|
| 工作记忆容量 | 当前推理能直接访问的信息上限 | 上下文接近满载 | 超出后必须删、压缩或归档 |
| 淘汰策略 | 决定谁被留下、谁被移出 | 新记忆写入时 | 直接影响命中率与任务成功率 |
| LRU | 最近最少使用 | 只看“多久没用过” | 简单，但可能删掉关键事实 |
| LFU | 最少使用频率 | 只看“用过多少次” | 稳定，但容易固化旧热点 |
| Importance | 重要性评分 | 同时看价值、时效、频率 | 更适合复杂多步任务 |

可以把它类比成电脑缓存，但不要把类比当定义。准确说法是：缓存和工作记忆都面临容量约束，都需要决定淘汰顺序；不同点在于 Agent 的“价值”不仅来自访问模式，还来自任务语义。

玩具例子很简单。假设工作记忆容量只能放 2 条信息，当前已有：

1. “用户项目语言是 Python”
2. “用户喜欢浅色主题”

这时新信息进来：

3. “部署环境禁止外网访问”

如果系统只按最近使用保留，前两条谁没被最近提到谁就可能出局；但对一个要生成部署脚本的 Agent 来说，第 3 条的工程约束明显更关键。也就是说，淘汰问题本质上不是“旧不旧”，而是“值不值得继续占位置”。

## 核心机制与推导

重要性评分的目标，是把“记忆价值”从模糊经验变成可以计算、可以排序的分数。一个常见公式是：

$$
I = I_0 \times \frac{1}{1+\Delta t/3600} \times (1+f)
$$

其中：

- $I_0$ 是基础重要度，表示这条记忆初始有多重要。白话说，就是“刚写入时它值多少钱”。
- $\Delta t$ 是距离上次访问的秒数。
- $f$ 是频率因子，常见写法为 $f=\min(1,\text{accesses}/10)$，表示访问次数越多，分值越高，但增长有上限。

这个式子的含义很直接：

1. 基础价值高的记忆，不应因为时间过去就立刻消失。
2. 长时间没被访问，分数应自然下降。
3. 多次被用到的记忆，说明它对任务轨迹有持续贡献，应当被加权保留。

但只算重要性还不够，因为“重要”不等于“当前相关”。因此在检索或淘汰时，通常还会把语义相关性和新近性一起纳入：

$$
score = 0.4 \cdot relevance + 0.3 \cdot recency + 0.3 \cdot I
$$

这里：

- `relevance` 是语义相关性，表示与当前问题像不像。
- `recency` 是新近性，表示是否刚刚被提到。
- `I` 是上面的重要性，表示长期价值。

这三个量一起工作，解决的是三个不同问题：相关性防止“答非所问”，新近性防止“上下文脱节”，重要性防止“关键约束被时间噪声挤掉”。

看一个容量为 2 的玩具例子。现在有三条候选记忆：

- A：基础重要度 0.9，5 分钟前访问，访问 4 次
- B：基础重要度 0.4，1 小时前访问，访问 1 次
- C：基础重要度 0.7，2 分钟前访问，访问 2 次

按上式计算：

| 记忆 | $I_0$ | $\Delta t$ | access | $f$ | 重要性 $I$ |
|---|---:|---:|---:|---:|---:|
| A | 0.9 | 300s | 4 | 0.4 | $0.9 \times \frac{1}{1+300/3600} \times 1.4 \approx 1.16$ |
| B | 0.4 | 3600s | 1 | 0.1 | $0.4 \times \frac{1}{2} \times 1.1 = 0.22$ |
| C | 0.7 | 120s | 2 | 0.2 | $0.7 \times \frac{1}{1+120/3600} \times 1.2 \approx 0.81$ |

因为容量只有 2，最低分的 B 被淘汰，A 和 C 被保留。

这个例子说明两点：

1. A 虽然不是最新，但因为基础价值高、被多次访问，仍然值得保留。
2. B 即使曾经有用，只要价值低、时间久、访问少，就应该让位。

真实工程例子更能说明问题。假设你在做一个多轮代码代理，任务是“修复线上支付服务超时”。对话里会不断出现日志片段、临时排查结果、命令输出。如果系统只按最新消息保留，很容易把真正关键的约束挤出去，比如“数据库连接池上限被运维锁死，不能临时调大”。这条信息不是每轮都出现，但它决定了解法边界。如果被淘汰，Agent 后面可能反复提出无效方案，导致上下文漂移。

所以，复杂任务里真正有价值的不是“最新”，而是“能持续约束决策路径的信息”。

## 代码实现

下面给一个最小可运行版本。它不依赖外部库，只演示三件事：记忆条目结构、重要性计算、容量超限时的淘汰。

```python
from dataclasses import dataclass
from time import time

@dataclass
class MemoryItem:
    key: str
    content: str
    base_importance: float
    last_access_ts: float
    access_count: int
    relevance: float  # 与当前任务的语义相关性，0~1

def importance(item: MemoryItem, now: float) -> float:
    delta_t = max(0.0, now - item.last_access_ts)
    freq = min(1.0, item.access_count / 10.0)
    return item.base_importance * (1.0 / (1.0 + delta_t / 3600.0)) * (1.0 + freq)

def recency(item: MemoryItem, now: float) -> float:
    delta_t = max(0.0, now - item.last_access_ts)
    return 1.0 / (1.0 + delta_t / 3600.0)

def eviction_score(item: MemoryItem, now: float) -> float:
    # 先算长期价值 importance，再和当前相关性、时效性合并
    imp = importance(item, now)
    rec = recency(item, now)
    return 0.4 * item.relevance + 0.3 * rec + 0.3 * imp

def retain_top_k(items, k: int, now: float):
    ranked = sorted(items, key=lambda x: eviction_score(x, now), reverse=True)
    return ranked[:k]

now = time()

items = [
    MemoryItem("A", "用户对青霉素过敏", 0.9, now - 300, 4, 0.95),
    MemoryItem("B", "用户刚才问过天气", 0.4, now - 3600, 1, 0.20),
    MemoryItem("C", "当前任务是生成处方建议", 0.8, now - 120, 2, 0.90),
]

kept = retain_top_k(items, 2, now)
kept_keys = {item.key for item in kept}

assert "B" not in kept_keys
assert kept_keys == {"A", "C"}
```

这段代码里最核心的顺序是：

1. 先给每条记忆算长期重要性。
2. 再结合当前任务相关性和时效性形成总分。
3. 容量超限时，保留高分，淘汰低分。

如果直接按时间排序，代码会更短，但行为会更差。因为“最近出现过”并不等于“后面还必须用”。

真实工程里通常还会再加两层：

| 模块 | 作用 | 常见做法 |
|---|---|---|
| 写入打分 | 决定新记忆初始价值 | 规则打分、模型打分、人工标签 |
| 活跃集淘汰 | 保证窗口内内容精简 | importance 排序后截断 |
| 归档层 | 避免直接删除可恢复信息 | 落到短期库或向量库 |
| 周期衰减 | 避免旧热点永久占位 | 定时重算 importance |

一个真实工程场景是分层记忆。比如把评分高于 7/10 的事件直接进入工作记忆，其余进入短期或长期存储；工作记忆满了以后，不是立刻硬删，而是把低分内容下沉归档。这样系统既能保持当前窗口干净，又不至于把以后可能有用的信息彻底丢掉。

## 工程权衡与常见坑

重要性策略不是“打个分数就结束”，它的难点在于持续维护。常见问题如下：

| 常见坑 | 现象 | 后果 | 对策 |
|---|---|---|---|
| 只看时间 | 新消息不断覆盖旧消息 | 关键约束被挤出窗口 | 引入 importance，不只看 recency |
| 只看频率 | 旧热点长期霸占 | 当前任务需要的信息进不来 | 对频率加上时间衰减 |
| 越多越好 | 什么都塞进工作记忆 | 检索噪声上升，推理变慢 | 控制活跃集大小，超限归档 |
| 初始分全靠拍脑袋 | 分数不稳定 | 淘汰行为不可预测 | 建规则模板，逐步校准 |
| 不做复审 | 历史垃圾长期累积 | context drift，上下文漂移 | 周期性衰减、人工抽检 |

这里最常见的误区是“多记一点总没坏处”。这在静态数据库里可能成立，但在工作记忆里通常不成立。工作记忆不是仓库，而是推理操作台。操作台上放满无关工具，不会提高效率，只会增加干扰。

再看一个真实工程例子。某团队做客服 Agent，最初策略很简单：所有用户消息按时间顺序保留，超出窗口就删最早的一条。短期看实现很快，长期问题很明显。用户一开始提供的“企业版合同号”和“发票抬头必须走香港主体”会在几十轮后被淘汰，而后面的寒暄、重复确认却还留在窗口里。结果是系统能记住最近礼貌措辞，却忘了最影响业务结果的事实。

改进方案通常不是一步到位上复杂模型，而是三件很务实的事：

1. 给关键字段设重要性阈值，比如身份、权限、约束、禁忌项默认高分。
2. 对低频、低反馈、长期不相关的内容做自动衰减。
3. 定期把低分内容归档，必要时再检索回流。

很多系统失败，不是因为没有“记忆”，而是没有“记忆卫生”。所谓记忆卫生，白话说就是持续清理，让活跃区只留下真正还在影响决策的内容。

## 替代方案与适用边界

不是所有 Agent 都需要重要性评分。策略应当和任务复杂度匹配。

| 策略 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| FIFO | 实现最简单 | 完全不看价值 | 一次性、低风险流水任务 |
| LRU | 成本低，行为直观 | 会删掉不常访问但关键的信息 | 简单对话、短链路任务 |
| LFU | 能保留常用知识 | 旧热点容易固化 | FAQ、固定问答机器人 |
| Importance | 更贴近任务价值 | 需要设计打分与校准 | 多步推理、工具调用、长任务 |

如果是简单 FAQ Agent，LFU 往往就够用。因为这类系统问题模式稳定，热门问答重复率高，访问次数本身就是不错的信号。

但如果是多步规划、代码修复、医疗问询、企业流程代理这类任务，单一策略就容易失真。更稳妥的做法是“重要性评分 + 分层记忆”。

一个简化流程可以写成：

```text
新事件进入
   |
计算 importance
   |
是否 >= 7/10 ?
   | yes                    | no
进入工作记忆              进入短期/长期存储
   |
工作记忆超限?
   | yes
淘汰最低分条目
   |
低分条目归档而非直接删除
```

这个分层方法的价值在于：工作记忆只放最影响当前决策的内容，短期记忆保存近期上下文，长期记忆保存可检索历史。这样做的边界也很清楚：

1. 如果任务很短，分层收益有限，简单 LRU 可能更省成本。
2. 如果打分体系不稳定，importance 反而会引入噪声。
3. 如果没有归档与检索能力，只做淘汰不做回流，系统会变成“删除更聪明”，而不是“记忆更可靠”。

所以，重要性策略并不是万能方案。它最适合的前提是：任务链路较长、关键约束可能跨多轮持续生效、错误遗忘的代价较高。
