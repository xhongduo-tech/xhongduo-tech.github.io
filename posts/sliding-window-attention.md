## 核心结论

Sliding Window Attention（滑动窗口注意力）是一种把注意力计算限制在局部邻域的机制。白话说，就是每个 token 不再和整段文本里的所有 token 两两比较，而是只看自己附近的一小段窗口。

它解决的核心问题很直接：标准自注意力的计算量和显存占用会随着序列长度按平方增长，长度一长就很贵；滑动窗口把这个代价改成和窗口大小成正比。若序列长度为 $n$、窗口宽度为 $w$，则：

$$
\text{Full Attention} = O(n^2), \quad \text{Sliding Window Attention} = O(n \cdot w)
$$

当 $w \ll n$ 时，收益非常明显。

更重要的是，窗口局部并不等于只能看局部。因为 Transformer 会堆很多层，第 1 层把局部信息混合后传给第 2 层，第 2 层再继续向外传播，所以有效感受野会随着层数扩大。常见的近似表达是：

$$
R_\ell \approx \ell \times w
$$

其中 $R_\ell$ 表示第 $\ell$ 层后，一个 token 能间接接触到的上下文范围。

玩具例子：如果每层窗口宽度是 4，第一层每个位置只能直接看前后 2 个词；第二层虽然仍只看前后 2 个位置，但它看到的是“上一层已经混过局部信息的表示”，于是更远的信息会逐层传过来。这像波纹扩散，不是一次跳到全局，而是分层传递到全局。

真实工程例子：Mistral 一类模型在长文本处理中就采用了滑动窗口注意力。它不是让每一步都保留全部历史，而是把 KV Cache（键值缓存，白话说就是把历史 token 的 key/value 存起来，避免重复计算）也限制在最近窗口内，从而把推理阶段的内存占用和窗口大小绑定，而不是和全部上下文长度绑定。

## 问题定义与边界

问题定义可以写成一句话：在尽量保留上下文建模能力的前提下，降低长序列注意力的计算和内存成本。

标准自注意力的代价来自“所有位置两两打分”。如果序列有 $n$ 个 token，就要构造一个 $n \times n$ 的注意力矩阵。长度翻倍，矩阵面积接近变成 4 倍。这在 8K、16K、32K 上下文时会迅速变成瓶颈。

滑动窗口注意力的边界也很清楚：它不是免费拿到全局信息，而是拿“每层局部、跨层传播”的能力换掉“单层直接全局连接”的能力。因此需要同时看三个参数：

| 参数 | 含义 | 直接影响 |
|---|---|---|
| $n$ | 序列总长度 | 总 token 数，决定任务规模 |
| $w$ | 窗口宽度 | 每个 token 每层最多直接关注多少邻居 |
| $\ell$ | 层数 | 信息能传播多远，决定间接全局覆盖能力 |

一个最小数值对比很直观。设 $n=4096$：

| 方案 | 每个 token 可见范围 | 近似比较次数 |
|---|---|---|
| Full Attention | 全部 4096 个位置 | $4096^2 = 16,777,216$ |
| Sliding Window，$w=512$ | 最近 512 个位置 | $4096 \times 512 = 2,097,152$ |

这大约节省 8 倍计算。这里“比较次数”不是精确 FLOPs，只是帮助理解复杂度量级。

边界条件主要有两个。

第一，窗口宽度是单层边界。一个 token 在单层里看不到窗口外的信息。如果模型层数不够，远距离依赖就可能传播不过来。

第二，推理时的 KV Cache 也要遵守窗口边界。若你在训练时用滑动窗口、推理时却把所有历史 key/value 全保留，那么计算图虽然是局部的，缓存内存却还是全历史增长，空间优势会被破坏。

所以，Sliding Window Attention 适合的场景不是“任何任务都比全注意力更好”，而是“长上下文很多、局部相关性强、远程依赖可以靠层叠传播”的任务。

## 核心机制与推导

先看标准注意力。对于位置 $i$ 的 query 向量 $q_i$，它要和所有 key 向量 $k_j$ 计算分数：

$$
\alpha_{ij} = \frac{q_i k_j^\top}{\sqrt{d}}
$$

然后对所有位置做 softmax，再加权求和 value：

$$
y_i = \sum_{j=1}^{n} \text{softmax}(\alpha_{ij}) v_j
$$

问题在于，$j$ 从 1 跑到 $n$，于是每一行都有 $n$ 个分数，整张矩阵有 $n^2$ 个元素。

滑动窗口注意力做的事情很简单：把 $j$ 的范围裁掉。对于位置 $i$，只允许它关注窗口内的位置：

$$
j \in [i-w+1, i] \quad \text{（因果场景）}
$$

或者在双向编码任务里，也可以写成中心窗口：

$$
j \in [i-\lfloor w/2 \rfloor,\; i+\lfloor w/2 \rfloor]
$$

于是原本是稠密的 $n \times n$ 矩阵，现在变成一张稀疏带状矩阵。行数仍然是 $n$，但每行只有大约 $w$ 个有效位置，因此总复杂度从 $O(n^2)$ 下降为 $O(n \cdot w)$。

玩具例子：假设句子长度为 10，窗口宽度为 4。第 7 个 token 不再和 1 到 10 所有位置比较，而只和局部 4 个位置比较，比如 4 到 7，或 5 到 8，具体取决于是否因果、是否居中。单层看起来它“视野很窄”，但下一层输入不是原始 token，而是已经融合过局部上下文的新表示，因此第二层又能继续向外扩。

这就是感受野推导的核心。若每层都允许宽度为 $w$ 的局部信息交换，那么经过 $\ell$ 层后，信息最远能传播的范围大致是：

$$
R_\ell \approx \ell \times w
$$

这不是说第 $\ell$ 层突然直接连到 $\ell \times w$ 之外的位置，而是说通过层与层之间的中间表示，信息可以逐跳传递。

例如，设 $w=4096$、$\ell=32$，则：

$$
R_{32} = 32 \times 4096 = 131072
$$

这个量级已经远大于 8K 上下文，因此即便单层只看局部，多层堆叠后仍能覆盖整个 8K 序列。这也是很多长上下文模型可行的关键，不是单层全局，而是多层传播。

再看解码阶段。自回归生成时，第 $t$ 步只会新增一个 token。为了避免每次重算全部历史，模型会缓存先前 token 的 key/value，这就是 KV Cache。标准做法是历史越长，缓存越长；滑动窗口做法是只保留最近 $w$ 个 key/value。

可以把它理解为一个滚动缓冲区：

| 解码步 | 保留的 KV 范围 |
|---|---|
| 第 100 步 | 例如保留 1 到 100（若 $w$ 还没满） |
| 第 5000 步，$w=4096$ | 只保留 905 到 5000 |
| 第 5001 步 | 丢弃 905，保留 906 到 5001 |

这样，缓存内存从“随序列长度持续增长”变成“被窗口宽度封顶”。因此训练和推理都与 $w$ 强绑定，而不是与完整历史长度强绑定。

真实工程例子：做长文档问答时，模型输入可能是 8K token。若使用全注意力，每层都要处理一个 8K × 8K 的注意力图；若使用 $w=4096$ 的滑动窗口，每层每个 token 最多只和最近 4096 个位置交互，再结合 32 层传播，既保留长程信息覆盖能力，又显著降低了计算和缓存压力。

## 代码实现

下面先给一个最小可运行的 Python 版本。它不是高性能实现，而是用于说明“窗口 mask 如何限制注意力范围”。

```python
import math

def causal_sliding_window_mask(n: int, w: int):
    assert n > 0 and w > 0
    mask = [[False] * n for _ in range(n)]
    for i in range(n):
        start = max(0, i - w + 1)
        for j in range(start, i + 1):
            mask[i][j] = True
    return mask

def visible_indices(i: int, n: int, w: int):
    start = max(0, i - w + 1)
    return list(range(start, i + 1))

def rolling_kv_update(cache, new_item, w: int):
    cache.append(new_item)
    if len(cache) > w:
        cache.pop(0)
    return cache

# 窗口 mask 正确性
mask = causal_sliding_window_mask(n=6, w=3)
assert visible_indices(0, 6, 3) == [0]
assert visible_indices(2, 6, 3) == [0, 1, 2]
assert visible_indices(5, 6, 3) == [3, 4, 5]

# 检查 mask 的最后一行只允许看最近 3 个位置
assert mask[5] == [False, False, False, True, True, True]

# 检查滚动 KV cache 始终不超过窗口大小
cache = []
for x in [10, 11, 12, 13, 14]:
    rolling_kv_update(cache, x, w=3)
assert cache == [12, 13, 14]

print("ok")
```

上面代码做了三件事：

1. 构造因果滑动窗口 mask。
2. 证明某个位置只能看到最近 $w$ 个 token。
3. 用一个列表模拟滚动 KV Cache，超过窗口就弹掉最旧项。

如果写成更接近深度学习框架的伪代码，核心就是：

```python
for i in range(n):
    start = max(0, i - w + 1)
    end = i + 1
    attn_scores[i, start:end] = q[i] @ k[start:end].T
    attn_scores[i, :start] = -inf
    attn_scores[i, end:] = -inf
```

这里 `-inf` 的作用是做 mask。白话说，就是把窗口外的位置强行设成“不可能被选中”，这样 softmax 后这些位置的权重会变成 0。

若是批量张量实现，通常不会逐 token 写 for 循环，而是先构造一个布尔 mask，再一次性交给框架内核。逻辑仍然相同：只有窗口内的位置可见。

KV Cache 的滚动更新也可以写成伪代码：

```python
keys.append(new_key)
values.append(new_value)

if len(keys) > w:
    keys = keys[-w:]
    values = values[-w:]
```

真实工程例子：在自回归推理服务里，如果你为每一层、每一个 head 都保留全历史 KV，那么并发一高，显存会被缓存迅速吃满。采用滑动窗口后，服务端可以把缓存大小稳定控制在“层数 × head 数 × 窗口宽度”的量级，而不会随着对话轮数无限膨胀。

## 工程权衡与常见坑

滑动窗口注意力不是“窗口越大越好”，而是一个明确的工程权衡。

| 窗口宽度 $w$ | 层数 $\ell$ | 单层计算 | 间接感受野 | 典型问题 |
|---|---|---|---|---|
| 小 | 小 | 低 | 小 | 远程依赖传不过去 |
| 小 | 大 | 中 | 中到大 | 层数深，训练和延迟上升 |
| 大 | 小 | 中到高 | 中到大 | 计算优势变弱 |
| 大 | 大 | 高 | 很大 | 接近回到重模型配置 |

第一个常见坑，是把“单层可见范围”误当成“模型最大上下文”。这不对。单层只能直接看窗口内，但多层会传播，因此最大可利用上下文往往远大于窗口本身。若忽略这一点，会错误低估模型能力。

第二个常见坑，是训练用滑动窗口，推理却保留全历史 KV Cache。这样做的结果是：理论上你在算局部注意力，实际上缓存内存却按历史长度增长，空间收益基本丢失。规避方式很简单，缓存必须随窗口滚动裁剪。

第三个常见坑，是窗口太窄但层数不够。比如窗口只有 256，模型层数也不深，那么隔着几千 token 的依赖几乎不可能有效传递。此时模型可能在局部语言建模上还行，但在长距离引用、跨段落总结、前后呼应检索上明显变差。

第四个常见坑，是忽略边界效应。序列开头和结尾的 token 可见邻居更少，实际有效窗口不完整。这会让不同位置的信息接收能力不完全一致。工程上常用的处理包括：加全局 token、加特殊摘要位、混合局部与稀疏全局连接。

下面给一个“坑与规避”简表：

| 常见坑 | 后果 | 规避方式 |
|---|---|---|
| 窗口太小 | 长距离依赖丢失 | 增大 $w$ 或增加层数 |
| KV Cache 不滚动 | 推理显存失控 | 严格只保留最近 $w$ 项 |
| 误把窗口当最大上下文 | 错误评估模型能力 | 用层叠感受野分析 |
| 全任务都用纯局部窗口 | 某些全局任务效果差 | 局部窗口与全局 token 混合 |

工程上最重要的判断标准不是“复杂度公式更漂亮”，而是“任务中的关键信息是否主要局部出现”。代码补全、长文生成、文档续写通常有较强局部性；但某些检索增强推理、跨文档排序、全局约束满足问题，可能更依赖直接全局访问。

## 替代方案与适用边界

Sliding Window Attention 只是长文本建模的一条路线，不是唯一路线。

| 方案 | 核心思想 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| Sliding Window | 只看局部窗口 | 简单、稳定、易工程化 | 远程依赖靠层叠传播 | 长文本生成、局部相关任务 |
| Global + Local | 局部窗口加少量全局 token | 兼顾局部效率和全局锚点 | 设计更复杂 | 文档问答、摘要、跨段落任务 |
| Performer 等线性注意力 | 用核技巧近似全注意力 | 理论上更线性 | 近似误差与实现复杂度更高 | 超长序列实验场景 |
| Chunked Attention | 分块处理，块内全连接 | 实现相对直接 | 块间连接受限 | 分段文档处理 |

Longformer 一类方法常加入 global tokens。白话说，就是让少数特殊位置拥有“全局广播权”，例如标题位、分类位、检索位，它们能跨窗口收集和分发信息。这比纯局部窗口更适合需要全局聚合的任务。

Performer 之类线性注意力方法走的是另一条路：不是直接裁掉远处连接，而是对 full attention 做近似，把二次复杂度变成线性或近线性。这类方法在理论上很吸引人，但工程上往往更依赖数值稳定性和近似质量，不一定像滑动窗口那样容易落地。

Chunked attention 则是按块切分序列，块内计算全连接，块间只做有限交互。它和滑动窗口的共同点是都在限制连接稀疏性，不同点是窗口是连续滚动的，分块通常有更强的边界感。

适用边界可以总结为一句话：如果任务要求“每个位置都可能直接依赖任意远的任意位置”，而且这种依赖对精度非常敏感，那么纯 Sliding Window Attention 可能不是最佳选择；如果任务的主要信息流是局部连续的，远程依赖可以接受多层逐步传播，那么它是非常强的工程折中。

玩具例子：一句短文本分类，序列只有几十个 token，全注意力完全够用，滑动窗口没什么必要。  
真实工程例子：8K 到 32K 的长文生成、长对话续写、代码仓库级补全，局部上下文密度高，窗口化注意力往往比全注意力更现实。
