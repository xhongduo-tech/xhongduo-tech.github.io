## 核心结论

协同过滤的定义很直接：它不理解内容本身，而是利用“谁和谁行为相近”或“什么和什么常被同一批人一起喜欢”来做推荐。对初级工程师来说，可以先记住两条主线：

1. User-CF：先找和目标用户相似的用户，再把这些邻居喜欢、但目标用户还没看过的物品推荐出来。
2. Item-CF：先找和目标用户已交互物品相似的物品，再把这些相似物品作为候选推荐出来。

两者的核心不是“把矩阵补满”，而是只在稀疏数据上做局部推断。评分矩阵通常极稀疏，很多系统的已观测交互只占全部用户-物品对的 $1\%$ 甚至更少，因此必须只遍历非零项，不能做全量笛卡尔积计算。

User-CF 和 Item-CF 的差别可以先看成“找相似的人”与“找相似的东西”：

| 维度 | User-CF | Item-CF |
|---|---|---|
| 相似对象 | 用户与用户 | 物品与物品 |
| 直觉 | 志同道合的人喜欢什么 | 和你看过的东西类似的还有什么 |
| 优点 | 个性化强，能反映细粒度兴趣 | 相似物品较稳定，易缓存 |
| 缺点 | 用户活跃度波动大，邻居不稳定 | 新物品冷启动明显 |
| 常见用途 | 长期兴趣微调 | 实时召回、相关推荐 |

玩具例子可以这样理解。用户 A 对电影 1、2 打分分别为 4、5；用户 B 对电影 1、2 打分分别为 5、4，并且 B 还喜欢电影 3。那么 User-CF 会说：A 和 B 很像，所以电影 3 可以推给 A。反过来，如果 A 看过电影 2，而很多看过电影 2 的人也常看电影 4，那么 Item-CF 会说：电影 4 和电影 2 相似，可以推给 A。

## 问题定义与边界

输入是一个用户-物品评分矩阵 $R$。这里“评分矩阵”就是一个二维表，行表示用户，列表示物品，单元格 $r_{u,i}$ 表示用户 $u$ 对物品 $i$ 的评分或偏好强度。目标是预测未知位置的值 $\hat r_{u,i}$，或者至少给出候选排序分数。

问题的数学表达是：

$$
R \in \mathbb{R}^{|U| \times |I|}, \quad \text{已知少量 } r_{u,i}, \quad \text{预测未知 } \hat r_{u,i}
$$

稀疏度是这个问题的第一约束。若共有 $|U|$ 个用户、$|I|$ 个物品、观测到 $|\Omega|$ 条评分，则矩阵密度为：

$$
\text{density} = \frac{|\Omega|}{|U|\times|I|}
$$

稀疏度通常写成：

$$
\text{sparsity} = 1 - \text{density}
$$

如果系统有 100 万用户、10 万个物品、1000 万条行为，那么密度只有：

$$
\frac{10^7}{10^6 \times 10^5} = 10^{-4} = 0.01\%
$$

这意味着 99.99% 的位置是空白。结论很明确：不能把全部空格都当成 0 参与普通矩阵运算，也不能对所有用户对、物品对暴力求相似度。

问题边界如下：

| 维度 | 内容 |
|---|---|
| 输入 | 显式评分、点赞、点击、播放、收藏等交互 |
| 输出 | 预测评分或 Top-N 推荐列表 |
| 关键约束 | 高稀疏、低延迟、可增量更新 |
| 异常情况 | 新用户、新物品、共现过少、热门偏置 |
| 不解决的问题 | 内容理解、跨域语义迁移、绝对因果解释 |

这里要特别区分“可观测数据”和“真实偏好”。用户没点某个视频，不代表不喜欢，可能只是没看到。所以协同过滤更准确的说法是：它在已观测共现里寻找统计规律，而不是理解用户本心。

冷启动是第二个边界。新用户没有历史行为，新物品没有交互记录，纯协同过滤几乎无法直接处理。最常见做法不是硬算，而是退回热门推荐、规则推荐或内容推荐。

最低共现阈值是第三个边界。所谓“共现”，就是两个用户都评价过同一个物品，或两个物品都被同一用户交互过。若只靠 1 次交叉就判定“高度相似”，结果通常不可靠。

## 核心机制与推导

协同过滤的核心机制可以压缩成四步：

1. 找共同交互集合。
2. 计算相似度。
3. 选出 Top-K 邻居。
4. 用邻居的信息预测目标分数。

“邻居”就是与目标用户或目标物品最相近的一小批对象。它不是图论里的严格概念，这里只是“最像的几个”。

### 1. 相似度公式

对用户 $u,v$，记共同评分集合为 $I_{uv}$。常见相似度有三类。

余弦相似度：

$$
\text{sim}_{cos}(u,v)=
\frac{\sum_{i\in I_{uv}}r_{ui}r_{vi}}
{\sqrt{\sum_{i\in I_{uv}}r_{ui}^2}\sqrt{\sum_{i\in I_{uv}}r_{vi}^2}}
$$

它衡量两个向量方向是否接近。白话解释：如果两个人对共同看过的内容高低变化一致，余弦会高。

皮尔逊相关系数：

$$
\text{sim}_{pcc}(u,v)=
\frac{\sum_{i\in I_{uv}}(r_{ui}-\bar r_u)(r_{vi}-\bar r_v)}
{\sqrt{\sum_{i\in I_{uv}}(r_{ui}-\bar r_u)^2}\sqrt{\sum_{i\in I_{uv}}(r_{vi}-\bar r_v)^2}}
$$

这里 $\bar r_u$ 是用户平均分。它先消除“有人总打高分、有人总打低分”的偏置。白话解释：它比较的是“偏离各自平均水平的模式是否一致”。

Jaccard 相似度：

$$
\text{sim}_{jac}(u,v)=\frac{|I_{uv}|}{|I_u \cup I_v|}
$$

它不看分值，只看是否共同交互。白话解释：交集越大、并集越小，两个对象越像。

### 2. 预测公式

对 User-CF，一个经典预测公式是：

$$
\hat r_{u,j} = \bar r_u +
\frac{\sum_{v\in N(u)} \text{sim}(u,v)(r_{v,j}-\bar r_v)}
{\sum_{v\in N(u)}|\text{sim}(u,v)|}
$$

其中 $N(u)$ 是用户 $u$ 的邻居集合。这个式子的含义是：先从用户自己的平均口味出发，再叠加相似邻居在目标物品上的偏离程度。

如果是 Item-CF，形式也类似，只是把“相似用户的偏好”换成“用户已看过物品的相似物品加权和”。

### 3. 玩具例子：为什么共现不足会误导

设用户 A 对电影 1、2 的评分为 4、5；用户 B 只对电影 1 打 5 分。

此时共同评分集合 $I_{AB}=\{1\}$。

余弦相似度：

$$
\text{sim}_{cos}(A,B)=\frac{4\times 5}{\sqrt{4^2}\sqrt{5^2}}=1
$$

结果看起来是“完全相似”。但这其实是假的，因为只有一个交叉点。皮尔逊在这里只基于一个点，分母为 0，通常无定义。Jaccard 则更保守：

$$
\text{sim}_{jac}(A,B)=\frac{1}{2}
$$

这说明一个工程事实：相似度公式本身不够，必须配合最小共现数、收缩项或归一化。否则会把“样本太少”误判成“关系很强”。

### 4. 真实工程例子：流媒体推荐

在流媒体平台里，用户播放矩阵通常极稀疏，直接做全量 User-CF 代价很高。一个常见方案是：

1. 离线构建 Item-CF 相似度表。
2. 线上根据用户最近播放的 20 个视频，查每个视频的相似视频集合。
3. 合并、去重、加权，形成召回候选。
4. 再用 User-CF 或排序模型做个性化微调。

原因是物品相似度比用户相似度更稳定。视频 A 和视频 B 的受众重叠，不会像用户最近一周的兴趣那样剧烈波动，因此更适合离线缓存。

## 代码实现

实现时最重要的不是公式，而是数据结构。评分矩阵必须按稀疏结构存储。最简单的方式是两个字典：

- `user_ratings[u][i] = score`
- `item_users[i][u] = score`

前者便于按用户遍历，后者便于按物品倒排。所谓“倒排表”，就是从物品反查哪些用户交互过它；搜索引擎里也有同名概念，本质都是为了避免全量扫描。

下面给出一个可运行的 Python 版本，演示 User-CF 预测。代码只遍历非零评分，并包含 `assert`。

```python
from math import sqrt

ratings = {
    "A": {"m1": 4.0, "m2": 5.0},
    "B": {"m1": 5.0, "m2": 4.0, "m3": 5.0},
    "C": {"m1": 1.0, "m2": 2.0, "m3": 1.0},
}

def user_mean(user_ratings):
    return sum(user_ratings.values()) / len(user_ratings)

def pearson(u_ratings, v_ratings, min_common=2):
    common = set(u_ratings) & set(v_ratings)
    if len(common) < min_common:
        return 0.0

    u_mean = user_mean(u_ratings)
    v_mean = user_mean(v_ratings)

    num = sum((u_ratings[i] - u_mean) * (v_ratings[i] - v_mean) for i in common)
    den_u = sqrt(sum((u_ratings[i] - u_mean) ** 2 for i in common))
    den_v = sqrt(sum((v_ratings[i] - v_mean) ** 2 for i in common))

    if den_u == 0 or den_v == 0:
        return 0.0
    return num / (den_u * den_v)

def predict_user_cf(target_user, target_item, ratings, topk=2):
    target_mean = user_mean(ratings[target_user])
    sims = []

    # 先找共同评分，再算 sim，再用公式预测
    for other_user, other_ratings in ratings.items():
        if other_user == target_user:
            continue
        if target_item not in other_ratings:
            continue
        sim = pearson(ratings[target_user], other_ratings)
        if sim != 0:
            sims.append((other_user, sim))

    sims.sort(key=lambda x: x[1], reverse=True)
    neighbors = sims[:topk]

    if not neighbors:
        return target_mean

    num = 0.0
    den = 0.0
    for other_user, sim in neighbors:
        other_mean = user_mean(ratings[other_user])
        num += sim * (ratings[other_user][target_item] - other_mean)
        den += abs(sim)

    return target_mean if den == 0 else target_mean + num / den

pred = predict_user_cf("A", "m3", ratings)
assert pred > 4.0
assert pred <= 5.5
print(round(pred, 4))
```

这个例子里，A 和 B 的口味方向接近，B 又喜欢 `m3`，因此 A 对 `m3` 的预测分会偏高。

如果要把流程写成伪代码，可以压缩成这样：

```text
输入: 稀疏评分表 ratings
输出: 用户 u 对物品 j 的预测分

1. 从 ratings[u] 取出用户 u 已评分物品
2. 找所有对 j 有评分的其他用户 v
3. 对每个 v:
   3.1 取 u 和 v 的共同评分集合
   3.2 若共同评分数 < 阈值, 跳过
   3.3 计算 sim(u, v)
4. 选 sim 最高的 top-K 邻居
5. 按加权平均公式计算预测分
6. 若没有可用邻居, 回退到用户均值或全局均值
```

真正落地时，通常还会预计算：

- 用户平均分
- 物品平均分
- 每个用户的评分数
- 每个物品的交互用户列表
- Top-K 邻居缓存

这样做的目的只有一个：避免每次线上请求都重新扫描稀疏矩阵。

## 工程权衡与常见坑

协同过滤最容易写出“能跑但不可靠”的版本。常见问题如下。

| 问题 | 原因 | 规避措施 |
|---|---|---|
| 伪高相似度 | 共现太少，1-2 个交叉点就算出高分 | 设置最小共现阈值，加入 Jaccard 或收缩因子 |
| 计算爆炸 | 全量用户对或物品对是 $O(n^2)$ | 用倒排表先筛候选，再算精确相似度 |
| 热门偏置 | 热门物品和谁都能共现 | 对热门项降权，或限制单物品贡献 |
| 冷启动 | 新用户/新物品无历史 | 回退到热门、内容特征、人工规则 |
| 分值尺度不一致 | 有人习惯打高分，有人习惯打低分 | 用 Pearson 或均值中心化 |
| 线上延迟高 | 请求时临时算邻居 | 离线预计算 Item-CF，相似表缓存 |

复杂度也要明确。若直接比较所有用户对，复杂度近似为 $O(|U|^2 \cdot \bar m)$，其中 $\bar m$ 是平均共同项处理成本；若比较所有物品对，则是 $O(|I|^2 \cdot \bar n)$。这在大规模系统里通常不可接受。

更可行的方式是借助倒排表。以 Item-CF 为例，遍历每个用户看过的物品集合，对其中物品两两共现计数。这样算的不是“所有物品对”，而是“真实被同一用户同时看过的物品对”。在稀疏场景下，后者通常小很多。

真实工程里，一条常见架构是：

| 层次 | 做法 |
|---|---|
| 离线层 | 统计共现、构建 Item-CF 相似度表 |
| 近线层 | 每小时或每天增量更新热门物品和新相似对 |
| 在线层 | 根据用户最近行为查相似表，快速召回候选 |
| 排序层 | 再用更复杂模型做最终排序 |

流媒体场景就是典型案例。平台先离线计算“看过 A 的用户也常看 B”的物品相似表，再在线上把用户最近看过的片单映射到候选池。User-CF 可以作为补充信号，用于长期兴趣微调，例如偏好纪录片、偏好 90 年代港片、偏好某类讲解视频。这种分工比纯 User-CF 更稳。

## 替代方案与适用边界

基于最近邻的协同过滤不是过时方法，它只是有很清晰的适用边界。

第一类替代方案是隐式反馈协同过滤。所谓“隐式反馈”，就是用户没有明确打分，但你能看到点击、播放时长、收藏、加购等行为。对白话解释来说，就是“用行为强弱代替打分”。如果没有显式评分，可以把点击序列构成共现图，把同看频率当作相似度基础，再用 Jaccard 或余弦近似。

第二类替代方案是混合推荐，即 CF + 内容特征。内容特征就是物品自身属性，例如视频标签、文章主题、商品品牌。它能处理新物品冷启动，因为即使没人看过新物品，也知道它属于什么类型。

第三类替代方案是矩阵分解，例如 ALS。矩阵分解的思想是：把用户和物品都映射到低维隐向量空间，再用向量内积预测偏好。这里“隐向量”可以理解成机器自动学出的兴趣坐标，不是人工定义的标签。

对比如下：

| 方案 | 优点 | 限制 | 适用场景 |
|---|---|---|---|
| 邻居式 CF | 可解释、实现简单、增量更新方便 | 稀疏时不稳，计算邻居成本高 | 中小规模系统、召回层、教学与原型 |
| 混合推荐 | 能缓解冷启动 | 需要内容特征工程 | 新物品多、业务规则强 |
| 矩阵分解/ALS | 适合大规模稀疏数据，效果更稳 | 可解释性差，训练链路更复杂 | 大规模工业系统、长期排序优化 |

因此可以给出一条实用判断线：

- 如果系统还在早期，数据量不大，且需要快速上线可解释推荐，用邻居式 CF。
- 如果新内容很多、冷启动明显，加入内容特征做混合方案。
- 如果交互规模很大、行为日志丰富，并且可以接受训练链路，逐步过渡到矩阵分解或更复杂模型。

对“零基础到初级工程师”最重要的一点是：不要把协同过滤理解成某一个公式，而要把它理解成一套工程管线。公式决定相似度，数据结构决定算不算得动，回退策略决定系统在坏数据下是否还能稳定工作。
