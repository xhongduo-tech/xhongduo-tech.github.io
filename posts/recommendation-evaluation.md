## 核心结论

推荐系统评测不能只看一个分数，而要拆成两个阶段：离线评测负责低成本筛选候选模型，在线 A/B 负责验证真实业务价值。离线阶段主要回答“这个模型在历史数据上排得是否更合理”，在线阶段主要回答“这个模型上线后，用户是否真的更愿意点、更愿意买、平台是否真的赚更多”。

工程上最常见的正确流程是：先在切好的历史日志上比较 NDCG@10、MRR、AUC 或 GAUC，筛掉明显差的版本；再把最有希望的版本放到 1% 到 5% 小流量实验里，看 CTR、CVR、GMV、停留时长是否同向改善。离线指标快、可重复，但带有历史偏差；在线指标真实，但成本高、节奏慢，还要承担业务风险。两者缺一不可。

下面这张表可以直接作为评测框架：

阶段 | 数据源 | 典型指标 | 主要价值
--- | --- | --- | ---
离线 | 历史曝光、点击、购买日志，或人工标注数据 | AUC、GAUC、NDCG@k、MRR、Hit Rate | 迭代快、可复现、低成本筛选
在线 | 生产环境真实流量中的 A/B 实验 | CTR、CVR、GMV、停留时长、留存 | 验证真实收益，决定是否放量
连接点 | 候选模型从离线进入在线 | 离线增益是否转成线上增益 | 防止“历史上更优，线上反而变差”

一个新手工程师最实用的做法，不是追求“万能指标”，而是建立一条稳健流水线：离线先看排序质量，在线再看真实行为。

## 问题定义与边界

推荐系统的目标不是“预测一个分数”，而是“把用户更可能感兴趣的内容排到更前面”。这里的“排序”可以理解成：系统给用户展示一个列表，越靠前的位置越容易被看到，因此前几名的质量通常比后几名更重要。

离线评测的边界是：它只基于已有日志。也就是说，模型是在过去发生过的曝光和反馈上被检验。这样做很快，但它无法知道“如果给用户看了过去从未展示过的候选物品，会发生什么”。

在线评测的边界是：它直接面对真实用户，因此能测到真实点击、下单、停留，但它也会受到流量波动、节日、活动、库存、价格、页面改版等因素影响。在线结果更接近真相，但更难解释，也更难重复。

几类核心指标的含义可以先用一句白话固定下来：

- AUC：衡量正样本整体排在负样本前面的概率，白话就是“相关的东西是否普遍被排得更靠前”。
- GAUC：按用户聚合的 AUC，白话就是“先分别看每个用户的排序，再按用户贡献加权平均”，避免少数大用户把整体 AUC 带偏。
- NDCG@k：强调前几位排序质量的指标，白话就是“越应该排前面的结果，如果真的排前了，得分越高”。
- MRR：看第一个相关结果出现得多靠前，白话就是“用户最快第几条能看到想要的东西”。
- Hit Rate：是否至少命中一个相关结果，白话就是“列表里有没有给到用户想看的内容”。

在线常见指标也要区分：

- CTR（点击率）= 点击数 / 曝光数，表示用户是否愿意点。
- CVR（转化率）= 转化数 / 点击数，表示点击后是否愿意进一步购买、注册或完成目标动作。
- GMV（成交总额）表示推荐带来的总交易金额。
- 停留时长表示用户是否愿意继续消费内容。

如果一个模型让 CTR 升高但 CVR 下滑，说明它可能只是把“吸引点击”的内容排前了，却没有提升真实价值。这就是评测边界：排序变好，不一定业务变好。

## 核心机制与推导

推荐评测的核心机制是“前排权重更高”和“离线近似线上，但不能替代线上”。

先看 NDCG。它基于 DCG（折损累计增益）：

$$
\text{DCG@k}=\sum_{j=1}^{k}\frac{2^{rel_j}-1}{\log_2(j+1)}
$$

其中 $rel_j$ 是第 $j$ 个位置结果的相关性。分母中的对数折损表示：越靠后的位置，贡献越小。再用理想排序的 DCG 做归一化，得到：

$$
\text{NDCG@k}=\frac{\text{DCG@k}}{\text{IDCG@k}}
$$

所以 NDCG 的含义很直接：如果高相关内容出现在前几位，得分更高。它适合推荐列表、搜索重排这类“位置敏感”的任务。

MRR 的公式更简单：

$$
\text{MRR}=\frac{1}{m}\sum_{i=1}^{m}\frac{1}{rank_i}
$$

这里 $rank_i$ 是第 $i$ 个请求里，第一个相关结果出现的位置。MRR 只关心“第一个命中的位置”，因此很适合“用户只要尽快找到一个满意结果”的场景，比如站内搜索、问答检索。

GAUC 的作用是处理“不同用户样本量不均衡”的问题：

$$
\mathrm{GAUC}=\frac{\sum_u w_u\cdot \mathrm{AUC}_u}{\sum_u w_u}
$$

通常 $w_u$ 取用户的正样本数或曝光权重。它的白话含义是：先分别评价每个用户的排序，再按用户贡献加权汇总。这样比把所有样本直接混在一起更稳，因为推荐系统本质上是“按用户分发”的任务。

A/B 测试的核心是判断“差异是否可能只是随机波动”。对于 CTR 这类比例指标，常见 z 检验可以写成：

$$
Z=\frac{p_{var}-p_{ctrl}}{\sqrt{p(1-p)(1/n_{var}+1/n_{ctrl})}}
$$

其中 $p$ 是合并后的点击率估计，$n$ 是曝光数。若双侧检验取显著性水平 $\alpha=0.05$，通常把 $|Z|>1.96$ 作为显著差异的近似阈值。

玩具例子可以直接看出离线与在线的区别。假设一个用户有 5 个候选商品，真实相关性从高到低是 `[3,2,0,1,0]`。模型 A 排成 `[3,2,1,0,0]`，模型 B 排成 `[2,3,0,1,0]`。两者都把好商品排到了前面，但 A 把最相关商品放在第 1 位，因此 NDCG 会更高。如果线上曝光后，模型 A 的 CTR 从 3.0% 升到 3.2%，CVR 也保持不变或上升，那么它才真正通过了线上验证。

真实工程例子更典型。电商首页重排模型离线把 NDCG@10 从 0.684 提高到 0.693，相对提升约 0.9%。这个幅度在离线里已经算有意义，但不能直接全量。原因是历史日志只反映旧排序下的曝光分布，新模型可能把此前曝光不足的商品推前，用户真实反应并不一定与历史一致。正确做法是先灰度 5% 流量，看 CTR、CVR、GMV 是否同向。如果 CTR 提升但 GMV 下跌，说明模型可能偏向低价、高吸引但低成交的商品，这时不能放量。

## 代码实现

下面的代码把离线指标和在线显著性检验放在一起，适合初学者在本地先跑通评测链路。

```python
import math

def ndcg_at_k(relevances, k):
    rel = relevances[:k]
    dcg = sum((2 ** r - 1) / math.log2(i + 2) for i, r in enumerate(rel))
    ideal = sorted(relevances, reverse=True)[:k]
    idcg = sum((2 ** r - 1) / math.log2(i + 2) for i, r in enumerate(ideal))
    return dcg / idcg if idcg > 0 else 0.0

def mrr(binary_relevances):
    for idx, r in enumerate(binary_relevances, start=1):
        if r > 0:
            return 1.0 / idx
    return 0.0

def hit_rate(binary_relevances, k):
    return 1.0 if any(r > 0 for r in binary_relevances[:k]) else 0.0

def ab_z_test(click_ctrl, imp_ctrl, click_var, imp_var):
    p_ctrl = click_ctrl / imp_ctrl
    p_var = click_var / imp_var
    pooled = (click_ctrl + click_var) / (imp_ctrl + imp_var)
    se = math.sqrt(pooled * (1 - pooled) * (1 / imp_ctrl + 1 / imp_var))
    return (p_var - p_ctrl) / se

# 玩具例子：离线排序
model_a = [3, 2, 1, 0, 0]
model_b = [2, 3, 0, 1, 0]

assert ndcg_at_k(model_a, 5) > ndcg_at_k(model_b, 5)
assert mrr([0, 0, 1, 0]) == 1 / 3
assert hit_rate([0, 0, 1, 0], 2) == 0.0
assert hit_rate([0, 0, 1, 0], 3) == 1.0

# 在线例子：A/B CTR 显著性
z = ab_z_test(click_ctrl=240, imp_ctrl=2000, click_var=300, imp_var=2000)
assert z > 1.96

print("NDCG(A):", round(ndcg_at_k(model_a, 5), 4))
print("NDCG(B):", round(ndcg_at_k(model_b, 5), 4))
print("Z-score:", round(z, 4))
```

这段代码的工程意义不是“替代成熟平台”，而是帮助你先把评测逻辑想清楚：

1. 离线按用户或请求生成一个候选列表，计算 NDCG@k、MRR、Hit Rate。
2. 选出离线最稳的版本进入线上小流量。
3. 收集曝光、点击、转化后，用比例检验判断提升是否显著。

如果你的日志是按用户组织的，通常会这样汇总：每个用户先算一次 NDCG@10 和 MRR，再对所有用户求平均；如果做 GAUC，则对每个用户先算 AUC，再按用户权重加权平均。这一步比把全量样本直接打平更符合推荐任务的结构。

## 工程权衡与常见坑

离线指标和在线指标之间最大的矛盾，是“历史最优”不等于“真实最优”。这是因为日志来自旧策略，旧策略决定了什么被曝光、什么有机会被点击。新模型离线得分高，有可能只是更擅长拟合旧曝光分布。

常见坑和对策如下：

坑 | 现象 | 工程对策
--- | --- | ---
只看离线分数 | NDCG、GAUC 提升，线上 CTR 反而下降 | 必须经过小流量 A/B，再决定是否放量
只看 CTR | 点击涨了，但 CVR、GMV 下滑 | 建北极星指标和护栏指标，至少同时看 CTR、CVR、GMV
样本不够 | 实验结果今天涨明天下跌，结论反复 | 事先估算最小样本量，避免过早停止实验
实验分流不稳 | 新老用户比例不同，结果失真 | 做稳定随机分桶，保持流量口径一致
忽略时序因素 | 大促、节假日导致指标异常 | 跨完整业务周期观察，避免只看单日
只优化头部用户 | 活跃用户变好，长尾用户变差 | 分人群看指标，尤其看新用户、低活跃用户

统计功效也经常被忽略。所谓统计功效，可以白话理解成“实验有多大把握检测到真实差异”。如果 CTR 从 3.0% 提升到 3.2%，绝对只差 0.2 个百分点，但要稳定检出这个差异，往往需要很大的曝光量。样本不足时，即使模型真实更好，也可能因为波动太大而“检不出来”。

另一个常见误区是把 A/B 测试当成“只要显著就上线”。这也不对。显著性只回答“差异不像随机产生”，不回答“差异是否值得”。比如 CTR 提升 0.1%，但计算成本翻倍、延迟增加 40ms、GMV 无改善，这种改动未必值得。工程决策一定是“统计显著 + 业务收益 + 系统成本”三者一起看。

## 替代方案与适用边界

当流量很小，标准 A/B 很难快速出结论，这时可以在离线和全量 A/B 之间增加一层中间验证。

一种常见替代方案是多臂老虎机（Multi-Armed Bandit）。它的白话含义是：系统在实验过程中动态地把更多流量分给当前表现更好的方案，而不是像标准 A/B 那样固定 50:50。它适合低流量站点或探索期改版，因为能减少把大量流量长期分给差方案的成本。但 bandit 的代价是分析更复杂，结果解释不如固定分流直观。

另一种替代方案是用户研究或人工评审。对于冷启动推荐、内容安全、解释性排序，先做人工打分能快速发现明显问题，比如重复推荐、标题党、低质内容上浮。这类方法不能替代真实流量，但适合在上线前挡掉明显风险。

在多目标场景里，离线阶段也不应只看单一排序指标。比如电商重排同时关心点击和成交，就可以在离线中同时看 NDCG、覆盖率、多样性、预估收入；在线再验证 CTR、CVR、GMV 是否一致改善。一个实用流程是：

离线多指标排查 -> 小流量 bandit 或分层灰度验证 -> 稳定后进入标准 A/B -> 指标达标再全量。

什么时候应该打断推进？有三个典型信号：

1. 离线分数提升很小，但线上成本显著增加。
2. 线上 CTR 提升，CVR 或 GMV 下滑。
3. 结果只在单一人群有效，对核心人群或长尾人群反而恶化。

这说明模型可能不是“整体更优”，而只是“对局部样本更优”。这时应回到特征、目标函数、采样策略和流量切片上重新分析，而不是继续放量。
