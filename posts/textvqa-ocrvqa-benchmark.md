## 核心结论

TextVQA 和 OCR-VQA 都是文本视觉问答基准：它们评测视觉语言模型是否能先识别图像中的文字，再把这些文字作为可复制的信息来回答问题。

这类任务的重点不是“看图猜答案”，而是“读出图中文字后回答”。例如图中有一个限速牌写着 `30`，问题是 `speed limit?`，模型应该从图中文字里复制 `30`，而不是根据常识猜一个常见限速值。对新手来说，可以理解为：关键不是“知道限速一般是多少”，而是“真的把路牌上的数字读出来”。

TextVQA 和 OCR-VQA 的差异主要来自图像场景。TextVQA 更偏自然场景文字，例如街道招牌、包装、屏幕、车牌、白板；OCR-VQA 更偏书封面、标题页等版式更规整、文本密度更高的图像。

| 基准 | 核心目标 | 常见图像 | 主要答案来源 | 典型难点 |
|---|---|---|---|---|
| TextVQA | 读取自然场景文字并回答 | 招牌、包装、路牌、屏幕 | 图中局部文字 | 文字小、倾斜、遮挡、背景复杂 |
| OCR-VQA | 读取书封面等规整文本并回答 | 书封面、标题页 | 标题、作者、出版信息 | 文本较长、多词答案、版式结构 |
| 通用 VQA | 理解图像内容并回答 | 物体、动作、场景 | 视觉语义和常识 | 不一定需要读文字 |

“看图猜答案”和“读图中文字再回答”的流程不同：

```text
看图猜答案:
图像 -> 视觉特征 -> 语言先验 -> 答案

读图中文字再回答:
图像 -> OCR 识别文字 -> 融合问题与文字 -> 复制或生成答案
```

这一区别决定了评测意义。一个模型能回答“这是什么颜色”，不代表它能回答“标签上写的价格是多少”。TextVQA 和 OCR-VQA 专门把问题压到图中文字上，迫使模型暴露 OCR、文字定位、文本融合和答案复制能力的短板。

## 问题定义与边界

TextVQA 关注自然场景中的可见文本理解。自然场景文字是指出现在真实拍摄图像里的文字，例如咖啡店招牌、路牌、瓶身标签、电视屏幕、白板笔记。典型问题是：“图中咖啡店叫什么名字？”答案来自招牌上的店名。

OCR-VQA 更接近封面阅读或弱文档理解。弱文档理解是指图像有明显文本版式，但还没有复杂到完整合同、发票、表格解析。典型问题是：“这本书的作者是谁？”答案来自书封面上的作者名。对新手来说，TextVQA 像街上拍照识别路牌，OCR-VQA 像看书封面找标题和作者。

| 任务 | 白话定义 | 适用场景 | 主要难点 |
|---|---|---|---|
| TextVQA | 看真实图片里的字并回答问题 | 街景、货架、包装、屏幕、招牌 | 文字位置不稳定，背景噪声大 |
| OCR-VQA | 看封面或规整页面里的字并回答问题 | 书封面、标题页、出版物图片 | 文本更长，多字段抽取更多 |
| 适用场景 | 答案主要藏在图中文字里 | 商品巡检、门店拍照、封面检索 | OCR 与问答链路都要稳定 |
| 主要难点 | 先读对，再答对 | 小字、错字、漏字、排序、复制 | 语言模型不能弥补全部 OCR 错误 |

边界需要说清楚。

第一，它不等同于通用 VQA。通用 VQA 是视觉问答，问题可能是“图里有几个人”“猫在做什么”。TextVQA/OCR-VQA 的答案往往直接来自图中文字。

第二，它不等同于纯 OCR。OCR 是光学字符识别，白话说就是把图片里的字转成文本。TextVQA/OCR-VQA 不只是把所有文字读出来，还要根据问题选择相关文字并组织答案。

第三，它不等同于完整文档问答。文档问答通常处理扫描件、PDF、表格、合同、票据，强依赖版面结构、字段关系和跨段落检索。OCR-VQA 与文档问答有工程相似性，但场景通常更轻量。

任务边界可以写成：

```text
图像 -> OCR -> 文本融合 -> 回答
```

其中 OCR 负责读字，文本融合负责把图像、问题、OCR token 和位置关系放在一起理解。OCR token 是从图像里识别出的文字片段，可以是单词、数字、短语或字符。位置框是这个 token 在图像中的坐标，用来表达“这个词在哪里”。

玩具例子：一张图里只有一个路牌，上面写 `30`，问题是“限速是多少？”只要 OCR 读到 `30`，答案就很直接。

真实工程例子：门店巡检系统拍摄货架促销牌，问题是“这个商品的促销价是多少？”如果价格牌写 `19.9`，系统必须从图片中准确读出价格，而不是根据商品类别或历史价格猜测。

## 核心机制与推导

这类任务可以统一理解为“编码 + 指针-生成混合解码”。

编码是把图像、问题、视觉区域和 OCR 结果变成模型内部表示。视觉区域是图像中可能有用的局部区域，例如商品、招牌、封面标题区域。指针-生成混合解码是指模型回答时有两个出口：一个出口从固定词表生成词，另一个出口从 OCR tokens 中复制词。

统一公式如下：

$$
H=f_{\text{enc}}(I,Q,V,O)
$$

$$
\alpha_t=\mathrm{softmax}(h_t^\top K_O)
$$

$$
p(y_t=w)=\lambda_t\,p_{\text{gen}}(w\mid h_t,H)+(1-\lambda_t)\sum_{j:o_j=w}\alpha_{t,j}
$$

| 符号 | 含义 | 白话解释 |
|---|---|---|
| $I$ | 图像 | 原始图片 |
| $Q$ | 问题 | 用户问的文本 |
| $V$ | 视觉区域 | 图像中被检测出的局部区域 |
| $O$ | OCR tokens 与框 | 读出的文字及其位置 |
| $Y$ | 答案序列 | 最终输出的答案 |
| $H$ | 编码结果 | 模型融合后的内部表示 |
| $h_t$ | 第 $t$ 步解码状态 | 当前准备输出答案词时的状态 |
| $\alpha_{t,j}$ | 第 $t$ 步对第 $j$ 个 OCR token 的注意力 | 当前最想复制哪个图中文字 |
| $\lambda_t$ | 生成通道权重 | 越大越偏生成，越小越偏复制 |

这个公式的关键是 $\lambda_t$。当 $\lambda_t$ 大时，模型更依赖固定词表生成；当 $\lambda_t$ 小时，模型更依赖 OCR 复制。LoRRA 和 M4C 这类方法的核心思想，就是让模型不仅能“说出答案”，还能“指向并复制图中文字”。

数值例子：

图中 OCR 识别到 `30`，问题是 `speed limit?`。设：

$$
p_{\text{gen}}(\text{30})=0.05
$$

$$
\alpha_{1,30}=0.80
$$

$$
\lambda_1=0.2
$$

则：

$$
p(y_1=\text{30})=0.2\times0.05+0.8\times0.80=0.65
$$

这说明即使模型从固定词表生成 `30` 的概率很低，只要复制通道强，最终仍然可以输出 `30`。

| 输出方式 | 来源 | 适合答案 | 风险 |
|---|---|---|---|
| 生成 | 固定词表 | `yes`、`red`、`book` 等常见词 | 难输出罕见名称、数字、专有名词 |
| 复制 | OCR tokens | 价格、门牌号、书名、作者名 | OCR 错了就会复制错 |
| 混合 | 生成 + 复制 | 多词答案或带解释答案 | 需要正确学习何时复制 |

新手版解释是：模型回答时有两个“出口”。一个出口是自己造词，一个出口是直接抄图里的字。遇到图中文字问题时，通常需要走“抄图里的字”这条路。

## 代码实现

工程上通常不是直接把整张图送进大模型就结束，而是先用专用 OCR 提取文本和框，再把 OCR 结果连同图像特征和问题一起送入 VLM。VLM 是视觉语言模型，白话说就是能同时处理图片和文字的模型。

典型流程如下：

```text
高分辨率图片
  -> OCR 提取文本与框
  -> 视觉编码器提取图像特征
  -> 融合问题、视觉特征、OCR tokens、OCR boxes
  -> 指针头复制相关 OCR token
  -> 输出答案
```

伪代码可以写成：

```python
image = load_image(path)
ocr_tokens, ocr_boxes = run_ocr(image)
visual_feats = vision_encoder(image)

answer = vlm.answer(
    question=question,
    visual_feats=visual_feats,
    ocr_tokens=ocr_tokens,
    ocr_boxes=ocr_boxes,
)
```

一个可运行的简化版本如下。它不实现真实 VLM，只演示“生成 + 复制”的概率合并逻辑：

```python
def pointer_generator_prob(word, lambda_t, p_gen, ocr_tokens, attention):
    copy_prob = sum(
        att for token, att in zip(ocr_tokens, attention)
        if token == word
    )
    return lambda_t * p_gen.get(word, 0.0) + (1 - lambda_t) * copy_prob


ocr_tokens = ["speed", "limit", "30"]
attention = [0.05, 0.15, 0.80]
p_gen = {"30": 0.05, "40": 0.20}

prob_30 = pointer_generator_prob(
    word="30",
    lambda_t=0.2,
    p_gen=p_gen,
    ocr_tokens=ocr_tokens,
    attention=attention,
)

assert abs(prob_30 - 0.65) < 1e-9
assert prob_30 > pointer_generator_prob("40", 0.2, p_gen, ocr_tokens, attention)
print(prob_30)
```

| 数据结构 | 示例 | 作用 |
|---|---|---|
| 图像输入 | `store.jpg` | 提供视觉内容 |
| OCR token | `["SALE", "19.9"]` | 提供可复制文字 |
| OCR box | `[(10,20,80,40), ...]` | 提供文字位置 |
| 问题文本 | `促销价是多少？` | 决定读取目标 |
| 输出答案 | `19.9` | 最终回答 |

只传 OCR 文本、不传框，会损失布局信息。比如书封面上同时有标题、副标题、作者、推荐语，只看文本列表可能不知道哪个词在标题区，哪个词在作者区。位置框能帮助模型利用“上方大字通常是标题”“作者名常在标题附近或下方”这类空间线索。

真实工程中，门店巡检可以采用这样的链路：高分辨率拍照，OCR 模型抽取价格、商品名和促销语，VLM 根据问题选择相关 token，然后通过复制机制输出价格字符串。这里专用 OCR 很关键，因为价格牌常有反光、倾斜、遮挡、字体夸张和小数点模糊，纯 VLM 很容易把 `19.9` 看成 `18.9` 或漏掉小数点。

## 工程权衡与常见坑

性能瓶颈往往不在语言模型本身，而在 OCR 质量、分辨率、版式对齐和答案复制机制。模型不是不会答，而是先读错了字。OCR 把 `19.9` 识别成 `18.9`，再强的 VLM 也只能基于错误输入回答，除非它重新从图像中纠错。

| 常见坑 | 具体表现 | 后果 | 规避方式 |
|---|---|---|---|
| 混淆 TextVQA 与 OCR-VQA | 用同一套假设处理街景和书封面 | 错估模型能力 | 分开看场景分布和答案类型 |
| 只传 OCR 文本不传框 | 丢掉文字位置 | 标题、作者、价格关系混乱 | 保留 box、排序和区域特征 |
| 只靠固定词表 | 无法输出罕见词、数字、专名 | 价格、书名、门牌号答错 | 使用复制或指针机制 |
| 不支持多词答案 | 只复制一个 token | 书名、作者名截断 | 支持序列解码和多 token 复制 |
| 忽略文本归一化 | 大小写、标点、空格不一致 | 评测分数异常 | 按评测规范做 normalization |
| 盲目堆大语言模型 | OCR 错误没有减少 | 成本升高但收益有限 | 先查 OCR、分辨率、框排序 |

评测也有独立风险。VQA soft accuracy 是 VQA 任务常用的软准确率，白话说就是答案不只和一个标准答案比，而是和多个人类答案集合比，匹配人数越多得分越高。

| 评测点 | 含义 | 注意事项 |
|---|---|---|
| 文本归一化 | 对大小写、冠词、标点等做统一处理 | 训练、验证、提交要一致 |
| 软准确率 | 多个人类答案共同决定得分 | 不能只按精确字符串匹配理解结果 |
| 多答案匹配 | 同一问题可能有多个可接受答案 | 需要处理同义、缩写和格式差异 |

故障排查顺序建议如下：

```text
答案错误
  -> 检查 OCR 是否读对
  -> 提高输入分辨率
  -> 更换或微调 OCR
  -> 检查 OCR box、排序和置信度
  -> 检查复制机制是否生效
  -> 最后再考虑微调 VLM
```

这套顺序很重要。许多项目一开始就微调大模型，但真正问题是图片压缩过重、OCR 不适合业务字体、或 OCR token 排序乱了。对于价格、编号、书名这类高精度答案，输入链路错误会直接决定上限。

## 替代方案与适用边界

并非所有“图里有字”的任务都必须使用 TextVQA/OCR-VQA 式指针-生成架构。选择方案时，要看答案是否主要来自图中文字、文本是否密集、版式是否复杂、精度要求是否高。

| 方案 | 做法 | 适合场景 | 不适合场景 |
|---|---|---|---|
| 纯 OCR + 规则 | OCR 后用正则或规则抽取 | 固定模板、价格牌、编号牌 | 问题多变、语义复杂 |
| OCR + VLM 融合 | OCR token、框、图像和问题一起建模 | TextVQA、OCR-VQA、门店巡检 | 极复杂表格或长文档 |
| 文档理解模型 | 建模版面、字段、表格和长文本 | 发票、合同、表格、PDF | 简单街景问答可能过重 |
| 通用 VLM 直接回答 | 直接输入图像和问题 | 文本稀疏、低精度容忍 | 数字、专名、密集文本 |

| 条件 | 更合适的选择 |
|---|---|
| 文本稀疏，问题主要依赖场景语义 | 通用 VLM |
| 文本稀疏，但答案必须精确复制 | OCR + VLM 融合 |
| 文本密集，版式规整 | OCR-VQA 风格或文档理解模型 |
| 文本密集，含表格和字段关系 | 文档理解模型 |
| 答案是单词或数字 | 复制机制收益明显 |
| 答案是长标题或多词短语 | 需要多 token 解码 |
| 低精度容忍 | 通用 VLM 可先试 |
| 高精度要求 | 专用 OCR + 校验规则 + 融合模型 |

决策树可以写成：

```text
问题答案主要来自图中文字？
  -> 否：可用通用 VLM
  -> 是：文本是否密集且版式复杂？
        -> 是：考虑文档理解模型
        -> 否：使用 OCR + VLM 融合

是否要求价格、编号、姓名等精确复制？
  -> 是：必须检查 OCR 质量和复制机制
  -> 否：可接受生成式回答
```

专业版例子：书封面标题、作者名适合 OCR-VQA 风格方法，因为它们通常来自封面文字，版式相对规整。发票、合同、表格则更接近文档理解问题，因为答案不仅来自文字，还来自字段关系、表格结构和跨区域对应。

新手版解释是：不是所有“图里有字”的任务都要同一种模型。看路牌、看书封面、看发票，难点不同。看路牌要读准少量文字；看书封面要找标题和作者；看发票要理解字段、金额、日期和表格结构。

工程上可以先用最小可行方案验证：如果 OCR + 简单规则已经稳定，就不必上复杂 VLM；如果问题表达多变、答案位置不固定，再引入 OCR + VLM 融合；如果页面结构复杂到表格、字段和长文本推理，就转向文档理解模型。
