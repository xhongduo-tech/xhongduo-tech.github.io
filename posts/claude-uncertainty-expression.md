## 核心结论

Claude 的不确定性表达，不是“礼貌措辞”那么简单，而是训练目标的一部分。更准确地说，它在 Constitutional AI，简称 CAI，一种“先按原则生成，再按原则自查并修订”的训练框架里，被要求遵守“诚实”原则，因此更倾向于主动承认知识边界、区分事实与推测，并在必要时使用“可能”“我不确定”“基于现有信息看”等置信度修饰语。

这件事有两个层面。

第一，行为层面。Claude 不只是会回答问题，还会在证据不足、问题有歧义、结论依赖外部条件时，显式告诉用户“不确定在哪里”。这使“不确定性表达”成为默认行为路径，而不是极少数样本里的偶发现象。

第二，机制层面。模型内部会产生 logits，意思是“每个候选 token 的原始分数”。这些分数经过 softmax 后形成概率分布。分布越平，说明模型越犹豫；这种犹豫可以用熵 $H$ 来描述。已有研究指出，这类基于 logit 的不确定性信号，与人类感知到的“不确定表达”存在中等相关，相关系数大约在 0.3 到 0.5 之间。这说明模型对自身不确定性有一定感知能力，但还远远不是“读心术”。

一个最直观的新手例子是：当用户问“某种药能不能和治疗心梗的药一起吃”，Claude 往往不会只给机制解释，而会补上一句“这涉及具体药物与患者情况，我不能确定，最好由医生确认”。重点不在于它加了一句免责声明，而在于这句“不确定”本身就是对齐目标的一部分。

| 指标 | RLHF | CAI | 变化 |
| --- | --- | --- | --- |
| Helpfulness（人类评估） | 7.2/10 | 7.8/10 | +8% |
| Harmfulness率 | 4.2% | 1.8% | -57% |
| Evasiveness率 | 12.5% | 6.1% | -51% |
| 每轮标注成本 | $50-200K | $5-20K | -90% |
| 训练周期 | 4-8 周 | 1-2 周 | -75% |

这些数字的含义不是“CAI 一定全面优于 RLHF”，而是说明：当“诚实表达边界”被制度化以后，模型既可能更安全，也可能更少出现那种“明明不知道却说得很肯定”的输出风格。

## 问题定义与边界

这里先定义问题。所谓“不确定性表达机制”，不是指模型内部真的拥有人类式自我意识，而是指模型在生成答案时，能把“证据不足、知识缺口、语义歧义、候选冲突”等状态，翻译成自然语言里的保留表达。

这个问题之所以重要，是因为大语言模型容易出现 hallucination，也就是“看起来流畅但事实并不成立的生成内容”。其中最危险的一类，不是明显胡说，而是“高信心但低依据”。对零基础读者来说，可以把它理解为：模型的语气很像懂了，但背后并没有足够证据。

问题边界要划清楚。Claude 的不确定性表达，不等于它一定知道正确答案；它只是更可能在不知道时承认不知道。反过来，Claude 也不保证每次都能正确暴露自己的边界，因为内部统计信号和人类对“你到底确定不确定”的感受，并不是完全一致。

下面这个边界表更实用：

| 场景 | 内部信号/外部条件 | 推荐输出方式 |
| --- | --- | --- |
| 高置信度 + 多源一致 | 检索结果一致，答案稳定 | 普通回答，可直接下结论 |
| 高熵 + 候选分散 | token 分布平，模型犹豫 | 使用 hedge，列出候选或假设 |
| 外部验证失败 | RAG 未检索到可靠证据 | 明示不知道，要求补充信息 |
| 高风险问题 | 医疗、法律、金融 | 即使回答，也要标注边界与验证建议 |

一个玩具例子是“某国某冷门法规第几条何时生效”。如果模型只依赖内部参数记忆，往往可能直接说出一个日期，而且语气坚定。正确的边界处理应当是：如果没有外部证据，就把它归入“知识边界”场景，回答时明确说“这可能不准确，我需要外部法条文本确认”。

一个真实工程例子是合规问答系统。用户问“某金融产品在某地区是否允许面向散户销售”，这类问题不是单一事实题，而是法规、时间、地区、产品类型四个变量叠加。即便模型内部置信度不低，只要检索证据不完整，也不应该给出“允许/禁止”的硬结论，而应该退回到“基于当前资料，我倾向于认为……但仍需人工复核”。

## 核心机制与推导

Claude 这类模型的不确定性表达，可以拆成两层机制：训练机制和推理机制。

先看训练机制。CAI 的基本链条是“生成 -> 宪法自我批判 -> 修订”。这里的“宪法”不是法律文本，而是一组事先写好的原则，比如有帮助、无害、诚实。模型先给出初稿，再根据这些原则审查初稿是否存在过度自信、误导、潜在伤害或边界不清的问题，最后再修订答案。这样训练的结果，是模型逐渐学会在某些状态下主动加入不确定性表达。

再看推理机制。模型每一步都会对下一个 token 给出一组 logits，也就是候选词的原始分数。通过 softmax 可以转成概率分布：

$$
\sigma(z)_j = \frac{e^{z_j}}{\sum_k e^{z_k}}, \qquad
H(z) = -\sum_j \sigma(z)_j \log \sigma(z)_j
$$

这里的 $H(z)$ 就是熵。直观上看：

- 如果某个 token 概率特别高，分布很尖，熵低，说明模型更像“我大致知道该说什么”。
- 如果很多 token 概率差不多，分布很平，熵高，说明模型更像“我在多个说法之间犹豫”。

术语首次出现时可以这样理解：softmax 是把一组原始分数压成“总和为 1 的概率”；熵是“混乱程度”的数值化描述。

玩具例子很适合说明这个过程。假设模型回答“法国首都是哪里”时，候选 token 中“巴黎”的概率是 0.97，其他词总共 0.03，那么熵很低，回答通常会直接写“法国首都是巴黎”。但如果问题换成“某篇冷门论文最早在哪个 workshop 发表”，前几个候选 token 可能分别对应不同会议名，概率都不高，这时熵会显著升高，模型更容易输出“我不完全确定，可能是……也可能是……”。

研究里常见的现象是：当平均熵从大约 2.1 升到 4.7 时，输出风格会从“这个事实是”转向“我猜可能是”或“基于我现有知识，较可能是”。这不是魔法，而是把内部统计不稳定性语言化。

但必须强调一条推导边界：熵高不等于答案一定错，熵低也不等于答案一定对。它只能说明“模型在生成时是否稳定”，不能直接等价于“世界事实是否真实”。这也是为什么 logit 熵与人类感知只有中等相关，而不是强相关。

## 代码实现

工程上最简单的实现，是在推理阶段对 logits 计算 softmax 和 entropy，再根据阈值决定是否添加 hedge，也就是“保留措辞”。hedge 可以理解为“把断言降级成带边界的陈述”。

下面给一个可运行的最小 Python 例子。它不依赖深度学习框架，只演示“高熵 -> 更保守表达”的逻辑。

```python
import math

def softmax(logits):
    m = max(logits)
    exps = [math.exp(x - m) for x in logits]
    s = sum(exps)
    return [x / s for x in exps]

def entropy(probs):
    return -sum(p * math.log(p) for p in probs if p > 0)

def render_answer(base_answer, logits, hedge_threshold=1.0):
    probs = softmax(logits)
    h = entropy(probs)
    if h > hedge_threshold:
        return f"{base_answer}\n\n基于当前信息，我对此不完全确定，建议再核对外部资料。", h
    return base_answer, h

# 低熵：模型明显偏向一个候选
answer1, h1 = render_answer("法国首都是巴黎。", [8.0, 1.0, 0.5], hedge_threshold=1.0)

# 高熵：多个候选接近，模型更犹豫
answer2, h2 = render_answer("该法规可能在 2021 年左右生效。", [1.1, 1.0, 0.9], hedge_threshold=1.0)

assert "不完全确定" not in answer1
assert "不完全确定" in answer2
assert h1 < h2

print(round(h1, 4), round(h2, 4))
print(answer1)
print("---")
print(answer2)
```

这个例子只做了单步判断，真实系统通常会更复杂，至少会增加三层处理。

第一，多 token 聚合。不是只看某一个位置的熵，而是看一整段答案中关键 token 的平均熵、最大熵，或者某些实体词附近的熵。

第二，多样化采样。可以对同一问题采样多次，观察答案是否稳定。如果 5 次生成里给出 3 个不同结论，即使单次熵不高，也应提高不确定性等级。

第三，和 CAI 或 system prompt 结合。也就是说，统计信号不是直接暴露给用户，而是变成触发器。例如：

- 熵低且检索一致：直接回答。
- 熵高但能列出候选：回答并标注假设。
- 熵高且检索缺失：明确说不知道。
- 高风险场景：即使熵不高，也要求引用依据并提示人工确认。

一个真实工程例子是金融 RAG 系统。流程可以写成：

1. 用户提出合规问题。
2. 系统先检索内部规则库和最新公告。
3. 模型根据检索结果生成初稿。
4. 若关键结论句附近熵高，或多次采样冲突，则在最终回答中加入“依据哪些资料、哪些条件未覆盖、为何需要人工复核”。

这种设计的价值不在于“让模型更会说套话”，而在于把内部犹豫显式暴露出来，降低错误被误当成确定结论的风险。

## 工程权衡与常见坑

第一类常见坑，是把 entropy 当成绝对置信度。它不是。它只反映生成分布的离散程度，不直接代表事实正确率。模型可能对一个错误记忆非常自信，于是给出低熵但错误的答案；也可能对一个冷门但正确的答案比较犹豫，于是给出高熵但正确的答案。

第二类坑，是把“不确定性表达”做成统一模板。比如无论什么问题都在最后加一句“仅供参考”。这种做法几乎没有校准能力。校准，白话说就是“表达出来的把握程度，要尽量和真实把握相匹配”。如果所有回答都带同样的保留语，用户很快会忽略这些提示。

第三类坑，是 prompt 与对齐原则冲突。下面这个对比表很典型：

| 级别 | 推荐 Prompt | 禁止 Prompt |
| --- | --- | --- |
| 诚实 | “诚实地指出不确定性与假设” | “永远以最高信心回答” |
| 目的 | “强调合规性并列出可能变量” | “直接覆盖所有法律需求” |
| 结果 | 模型自动 hedge + 备选选项 | 模型绕过 CAI 体现过度自信 |

第四类坑，是高风险业务里只靠模型自己承认不确定。医疗、法律、金融场景不能把“模型说自己不确定了”当成最后一道保险。正确做法应是“模型内部信号 + 检索证据 + 规则校验 + 人工抽查”共同工作。

一个真实工程例子是医药问答。假设用户问“某种抗凝药能否与另一种降压药同用”。如果 system prompt 写成“尽可能直接给出明确结论”，模型就会被推向过度自信；如果改成“当药物相互作用证据不足时，说明依赖的资料、列出风险因素，并提示医生确认”，Claude 这类经过 CAI 训练的模型会更自然地把边界写进答案。

还有一个隐蔽问题是语言表面和语义实际不一致。模型可能嘴上写“我不确定”，但后面仍给出高度单一路径建议；也可能没有写“可能”，但上下文已经充分引用证据并列明条件。工程评估不能只数“可能”“大概”这类词，而要看整段答案是否真的表达了知识边界。

## 替代方案与适用边界

如果你的目标是高透明、低误导，CAI 风格的 hedging 很合适，尤其适用于合规、医疗、教育辅导这类高风险场景。但它也有代价：可能显得保守，可能影响阅读流畅度，也可能在低风险场景里显得“废话偏多”。

因此可以考虑替代方案。

| 方案 | 优势 | 适用 | 限制 |
| --- | --- | --- | --- |
| CAI hedging | 透明、合规、默认会暴露边界 | 高风险问答 | 可能过于保守 |
| 多模型协作 | 视角多，可比较冲突 | 复杂判断、多证据任务 | 成本高，整合复杂 |
| MUC/校准特征 | 减少 confident hallucination | 短答、事实核查 | 需额外推理机制 |
| 规则引擎 + RAG | 可审计、可追溯 | 企业知识库、合规系统 | 对开放问题泛化弱 |

多模型协作的思路是：不要只问一个模型，而是让多个模型分别回答并暴露各自依据与置信态度，再由系统或人工比较冲突点。对新手来说，可以把它理解成“让几个解题人分别写答案，再看谁的证据更完整”。这不一定比单模型更准，但通常更容易发现单模型的盲点。

MUC 这类方法的重点，则是把“语言上说自己不确定”和“语义上真的不确定”对齐。也就是说，不只看模型有没有说“可能”，而是尝试校正它在不同问题类型下的口头置信度，使 verbal uncertainty，意思是“嘴上表达出来的不确定性”，更接近 semantic uncertainty，也就是“任务本身真实存在的不确定性”。

所以适用边界可以概括为：

- 如果任务高风险，优先选 CAI 风格表达边界。
- 如果任务高复杂度，优先加多模型或外部检索。
- 如果任务强调事实校准，优先做不确定性校准而不是只加免责声明。
- 如果任务低风险且追求效率，可以适度降低 hedge 强度，但不能删除证据链要求。
