## 核心结论

位置偏差指“排在前面的内容更容易被看见和点击”，也就是位置本身会改变点击率。对推荐系统来说，这会把“用户真正喜欢”与“只是排得靠前”混在一起。

选择偏差指“系统只能看到已经曝光的样本反馈”，也就是没被展示的物品没有点击、也没有不点击，直接从训练数据里消失。两者叠加后，模型很容易把高位曝光学成高质量信号，再把高分内容继续放到更前面，形成反馈循环。

一个直接判断标准是：如果同一批内容只因为展示位置不同，CTR 就差了数倍，那么原始点击数据已经不再等于用户偏好。此时需要做两件事：

| 现象 | 模型容易产生的误解 | 后果 |
|---|---|---|
| 位置 1 点击远高于位置 5 | 以为位置 1 上的物品质量更高 | 排序越来越依赖历史高位样本 |
| 只观测曝光样本 | 以为未曝光物品“不受欢迎” | 长尾内容永远学不到 |
| 预测 pCTR 高于真实 CTR | 以为模型更准，实际只是更自信 | 在线收益不稳定，策略失真 |

工程上最常见的两类修正手段是：

1. 用 IPW（Inverse Propensity Weighting，逆倾向加权）对不同位置的样本重新加权。
2. 对线上 pCTR 做校准，让“预测点击率”和“真实点击率”重新对齐。

如果不做这两件事，推荐系统通常会表现为：头部内容越来越强，后位内容越来越没有机会，离线 AUC 可能很好看，但在线 CTR、GMV 或用户满意度不稳定。

## 问题定义与边界

先给出最核心的拆分公式：

$$
C_{u,i}=E_{u,i}\cdot R_{u,i}
$$

其中：

- $C_{u,i}$ 是点击事件，表示用户 $u$ 是否点击了物品 $i$。
- $E_{u,i}$ 是曝光或被看见事件，白话说就是“用户有没有真的看到它”。
- $R_{u,i}$ 是真实兴趣，白话说就是“即使位置公平，用户是否愿意点它”。

这个公式的重点不是数学技巧，而是提醒我们：点击不等于兴趣。点击成立，前提是先被看到；而“是否被看到”往往被位置强烈影响。

对零基础读者，可以把问题理解成一个列表浏览过程。用户通常从上往下扫列表，前几个条目更容易进入视线，后面的条目即使本身更适合，也可能根本没有机会被看到。于是日志里留下的大量点击，大概率集中在前几位。

玩具例子如下。假设一个购物 App 的列表有 5 个位置：

| 位置 | 曝光后被认真看到的概率 | 真实兴趣相同情况下的观测点击倾向 |
|---|---:|---:|
| 1 | 0.90 | 高 |
| 2 | 0.70 | 较高 |
| 3 | 0.45 | 中 |
| 4 | 0.25 | 低 |
| 5 | 0.10 | 很低 |

如果 5 个商品对用户的真实吸引力完全一样，只因为位置不同，位置 1 也会拿到更多点击。此时原始 CTR 不能直接解释为“商品质量”。

这里要明确边界：

1. 如果没有随机曝光、没有探索桶、也没有额外日志去估计曝光概率，那么位置效应和兴趣效应在数据里是耦合的，很难纯靠监督学习自动分开。
2. 如果系统能记录位置、页码、停留时长、可见区域，或者拿到一小部分随机排序流量，就能近似估计 $P(E_{u,i}=1)$，从而做去偏。
3. 本文讨论的是点击反馈场景，重点解决“展示位置改变点击分布”的问题。它不直接解决文案误导、封面质量波动、库存约束等别的偏差来源。

所以，问题定义可以压缩成一句话：推荐系统真正想学的是兴趣 $R$，但训练数据直接给出的只有点击 $C$，而点击又被曝光 $E$ 扭曲了。

## 核心机制与推导

最常见的思路是 IPW。它的直觉很简单：越不容易被看见的位置，留下来的样本越稀缺，所以这些样本在训练时要“更大声一些”。

设某个样本的曝光概率是：

$$
p_{u,i}=\mathbb{P}(E_{u,i}=1)
$$

则对应权重定义为：

$$
w_{u,i}=\frac{1}{p_{u,i}}
$$

训练目标可写成：

$$
\mathcal{L}=\sum_{(u,i)} w_{u,i}\,\ell(\hat{y}_{u,i}, C_{u,i})
$$

其中：

- $\hat{y}_{u,i}$ 是模型预测的 pCTR，白话说就是“模型估计这个用户会点的概率”。
- $\ell$ 是损失函数，常见是二分类交叉熵。
- $w_{u,i}$ 越大，说明这个样本来自低曝光区域，训练时应被更重视。

### 为什么这样能纠偏

如果某些位置天然曝光少，那么这些位置的正负样本在原始日志里都更少。直接训练时，模型会被高曝光位置主导。IPW 相当于做一次“样本再平衡”：让不同曝光概率下的样本，对总体目标的贡献更接近真实分布。

看一个最小数值例子。假设只有两个位置：

| 位置 | 曝光概率 $P(E=1)$ | 原始观测 CTR | IPW 权重 $1/P(E=1)$ |
|---|---:|---:|---:|
| 1 | 0.30 | 0.30 | 3.33 |
| 5 | 0.05 | 0.05 | 20.00 |

如果只看原始点击，位置 1 明显更“好”。但这不一定说明位置 1 上的内容更优，也可能只是它更容易被看到。IPW 之后，位置 5 的样本会被放大约 6 倍，相当于告诉模型：这些后位样本少，不是因为内容差，而是因为它们更少被看到。

这里的直观含义是：

- 原始点击率偏向“谁更容易得到机会”。
- IPW 想恢复“如果机会更公平，真实兴趣会怎样分布”。

### 从偏差到反馈循环

位置偏差最危险的地方不是一次训练误差，而是多轮迭代后的闭环：

1. 初始排序把某些内容放前面。
2. 前面内容获得更多曝光和点击。
3. 训练数据里它们看起来更优。
4. 下一个模型更相信它们。
5. 排序继续把它们放前面。

这样一来，模型学到的是“历史位置优势”，不是“稳定用户偏好”。这也是很多系统会出现“强者恒强、长尾失声”的根本原因。

### pCTR 为什么还要校准

即使做了排序去偏，模型输出的 pCTR 也可能不等于真实 CTR。原因很简单：训练目标、采样方式、负样本构造、流量分布和线上环境都可能发生偏移。

例如模型预测某类广告的点击率是 0.12，但线上真实 CTR 只有 0.08。这种问题叫“校准误差”，白话说就是“概率数字看上去像概率，实际上并不准”。

常见校准目标是让：

$$
\mathbb{E}[\text{真实 CTR} \mid \hat{y}=p] \approx p
$$

也就是预测为 10% 的样本，长期统计下来真实点击也应接近 10%。

## 代码实现

下面给出一个可运行的简化 Python 例子。它不依赖深度学习框架，重点展示三件事：

1. 如何根据位置估计曝光概率。
2. 如何构造 IPW 权重并做裁剪。
3. 如何比较原始 CTR 与校准后的聚合估计。

```python
from math import log

# 历史上估计出来的位置曝光概率
# 白话解释：排得越后，真正被用户看到的概率越低
position_prob_seen = {
    1: 0.30,
    2: 0.20,
    3: 0.12,
    4: 0.08,
    5: 0.05,
}

def clipped_ipw_weight(position, max_weight=10.0, min_prob=0.02):
    prob_seen = max(position_prob_seen[position], min_prob)
    weight = 1.0 / prob_seen
    return min(weight, max_weight)

def bce(pred, label):
    eps = 1e-12
    pred = min(max(pred, eps), 1 - eps)
    return -(label * log(pred) + (1 - label) * log(1 - pred))

# 一个玩具数据集：同类物品分布在不同位置
# click=1 表示发生点击
samples = [
    {"position": 1, "pred": 0.28, "click": 1},
    {"position": 1, "pred": 0.25, "click": 0},
    {"position": 2, "pred": 0.22, "click": 0},
    {"position": 3, "pred": 0.20, "click": 1},
    {"position": 4, "pred": 0.18, "click": 0},
    {"position": 5, "pred": 0.15, "click": 1},
]

plain_loss = 0.0
weighted_loss = 0.0
plain_ctr = sum(x["click"] for x in samples) / len(samples)

weighted_click_sum = 0.0
weighted_exposure_sum = 0.0

for x in samples:
    w = clipped_ipw_weight(x["position"])
    plain_loss += bce(x["pred"], x["click"])
    weighted_loss += w * bce(x["pred"], x["click"])
    weighted_click_sum += w * x["click"]
    weighted_exposure_sum += w

ipw_ctr = weighted_click_sum / weighted_exposure_sum

# 原始损失和加权损失都应该是正数
assert plain_loss > 0
assert weighted_loss > 0

# 后位样本权重应大于前位样本
assert clipped_ipw_weight(5) > clipped_ipw_weight(1)

# 加权后的CTR估计仍然是概率
assert 0.0 <= ipw_ctr <= 1.0
assert 0.0 <= plain_ctr <= 1.0

print("plain_ctr =", round(plain_ctr, 4))
print("ipw_ctr   =", round(ipw_ctr, 4))
print("loss      =", round(plain_loss, 4))
print("ipw_loss  =", round(weighted_loss, 4))
```

上面代码展示的是“按位置加权”的最小实现。在真实训练中，常见写法是：

```python
loss = 0.0
for batch in dataloader:
    pred = model(batch.features)
    base_loss = binary_cross_entropy(pred, batch.click, reduction="none")
    weights = [clipped_ipw_weight(pos) for pos in batch.position]
    loss = (base_loss * tensor(weights)).mean()
    loss.backward()
    optimizer.step()
```

### 真实工程例子

一个电商推荐系统通常会这样落地：

1. 日志层记录 `request_id`、`item_id`、`position`、`page_no`、`click`、`dwell_time`。
2. 从随机桶或历史统计中估计每个位置的 `ProbSeen`。
3. 训练阶段把 `position -> weight` 映射进 loss。
4. 线上服务继续输出原始 pCTR，但在重排或出价前接一个校准器。
5. 每天或每周重新估计位置曝光率和校准参数。

例如 Instacart 一类广告系统，会把主模型输出的 pCTR 先作为一个基础信号，再训练额外校准器，把它调整到更接近真实 Actr。这里的 Actr 可以理解为“实际点击率统计值”。这样做的目的不是替代排序模型，而是让概率更可信，便于后续出价、预算分配和收益估算。

## 工程权衡与常见坑

IPW 的优点是直观、容易接入已有训练流程，但它的主要问题是高方差。白话说：后位样本本来就少，再把它们乘上很大的权重，噪声也会一起被放大。

| 常见坑 | 现象 | 原因 | 缓解策略 |
|---|---|---|---|
| 权重爆炸 | loss 抖动，训练不稳定 | $P(E=1)$ 太小，导致 $1/P(E=1)$ 过大 | 设最小曝光概率、最大权重裁剪、做归一化 |
| 曝光率估计错误 | 去偏后反而更差 | 用位置 CTR 直接代替曝光概率，定义混淆 | 尽量用随机桶、可见性日志或单独 ProbSeen 模型 |
| 校准器过拟合 | 离线很好，线上漂移 | 校准集过小或和线上流量不一致 | 用时间切分校准集，定期重训 |
| 数据延迟 | 线上结果和离线不一致 | 点击回传晚，标签不完整 | 固定回溯窗口，延迟补标 |
| 只看 AUC | 概率不准但排序看似不错 | AUC 不衡量概率刻度是否准确 | 同时看 Calibration Curve、ECE、分桶 CTR |
| 去偏过度 | 长尾被异常放大 | 极少量后位样本被赋予过大影响 | 权重平滑、分层采样、最小样本数保护 |

对新人来说，最容易犯的错是把“位置 CTR”直接当成“曝光概率”。这两个量不是一回事：

- 曝光概率描述“用户有没有看到”。
- CTR 描述“看到后有没有点击”。

如果把 CTR 直接当曝光率，会把“内容本身吸引力”也混进权重，导致纠偏方向出错。更稳妥的做法是：单独估计 ProbSeen，或者至少用随机流量近似测出位置可见率。

另一个常见坑是“校准以后就万事大吉”。实际上校准不是一次性动作。线上页面样式、用户群、节假日、广告供给变化，都会让真实 CTR 分布漂移。如果校准器长期不更新，预测值又会逐渐失真。

## 替代方案与适用边界

IPW 不是唯一方案。实际工程里至少有三类常见思路：

| 方案 | 需要什么输入 | 优点 | 局限 | 适用场景 |
|---|---|---|---|---|
| IPW | 曝光概率或其近似估计 | 易接入现有训练流程，解释性强 | 权重高方差，依赖曝光率估计质量 | 有位置日志、能拿到随机桶 |
| 转移学习校准 | 主模型 pCTR + 真实点击统计 | 不必重写排序主干，适合线上快速修正 | 只能修概率，不直接解决曝光偏差来源 | 线上 pCTR 偏乐观，需要概率对齐 |
| PAL/概率分离 | 分别建模 ProbSeen 与 CTR | 结构清晰，把“看见”和“喜欢”拆开 | 实现复杂，依赖更多特征与日志 | 大型推荐系统、长期演进架构 |

### 什么时候优先用 IPW

如果系统已经有较好的位置、可见性、随机桶日志，IPW 往往是第一选择。原因是它和现有训练管线兼容，只需在 loss 层增加权重即可，改造成本较低。

### 什么时候更适合转移学习校准

如果现在线上问题主要不是“排序完全错了”，而是“pCTR 数字不准，影响出价或业务阈值”，那么单独加一个校准器会更高效。它相当于在主模型之后补一个概率映射层，把预测值拉回真实 CTR 区间。

### 什么时候考虑 PAL 或概率分离

当系统已经比较复杂，需要长期稳定地区分“用户看到没”和“用户喜不喜欢”时，最好把这两个概率拆开建模。常见写法是：

$$
P(\text{Click}) = P(\text{Seen}) \times P(\text{Click} \mid \text{Seen})
$$

其中：

- $P(\text{Seen})$ 建模位置和可见性。
- $P(\text{Click} \mid \text{Seen})$ 建模真实吸引力。

这种方案更接近业务机制，但成本也更高，需要更完整的特征、更细的日志和更复杂的训练维护流程。

最后给出一个简单选择原则：

1. 有随机曝光数据，先做 IPW。
2. 概率不准影响业务决策，再做在线校准。
3. 系统规模大、偏差问题长期存在，再考虑 ProbSeen 与 CTR 分离建模。
4. 如果完全没有可观测曝光机制，也没有探索流量，那么任何“去偏”都只能是近似，最好引入探索策略，例如多臂老虎机，为模型补充更公平的样本。
