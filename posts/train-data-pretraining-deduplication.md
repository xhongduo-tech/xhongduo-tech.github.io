## 核心结论

预训练数据去重的本质，是把训练集中“被重复计算很多次的内容”压回合理权重。这里的“去重”不是只删掉完全一样的文本，还包括删掉语义上几乎等价的改写版本。对白话解释就是：如果一批语料反复在说同一件事，模型就会把这件事学得过重，浪费算力，还可能把偏见、错误和泄露内容一起放大。

一个实用结论是，多阶段去重通常比单阶段更稳。常见流程不是一步到位，而是：

```text
阶段 ① 哈希去重 -> ② LSH 近似 -> ③ embedding 相似度筛选
```

再展开成更完整的简化流程图，可以写成：

```text
原始语料
  |
  v
[文档/段落切分]
  |
  v
[哈希去重]
  |  删除完全相同文本
  v
[MinHash / LSH / Bloom]
  |  快速找近似重复候选
  v
[Embedding 表示]
  |
  v
[相似度计算 / 聚类]
  |  只保留每簇 k 个代表
  v
[去重后的训练集]
```

新手版可以直接记一句话：先用文档或段落哈希过滤完全相同文本，再用语义 embedding 聚类删掉意思一致的句子，剩下的数据更丰富，不会让模型反复学习同一个观点。

为什么这套方法成立？因为预训练目标本质上是在估计一个数据分布。若重复样本过多，模型看到的不是“真实分布”，而是“被重复样本放大的分布”。去重就是在修正这个分布，使训练更接近“覆盖更广、重复更少、权重更平衡”的目标集合。

## 问题定义与边界

预训练数据去重复，指的是基于 exact similarity 或 semantic similarity 比较语料块，避免某些内容在训练集中被多次加权。这里 first appearance 的两个术语先解释清楚：

- exact similarity：精确相似，白话就是字面上几乎一模一样。
- semantic similarity：语义相似，白话就是写法不同，但表达的是同一件事。

去重不是一个单一动作，而是一个带边界的设计问题。至少要先回答四个问题：

| 维度 | 需要明确什么 | 常见选项 |
|---|---|---|
| 粒度 | 在哪一层比较相似 | 文档 / 段落 / 句子 |
| 表示 | 用什么方式表示文本 | 原文字符串 / n-gram / embedding |
| 阈值 | 多像才算重复 | 完全相等 / Jaccard 阈值 / cos 阈值 |
| 保留策略 | 重复组里留下谁 | 留最短 / 留最长 / 留最高质量 / 每簇留 k 个 |

这里“粒度”尤其关键。粒度的白话解释是：你到底把什么当成一个比较单位。文档级去重适合删整篇转载，段落级去重适合删模板化内容，句子级去重更细，但计算量也更高。

一个常见边界例子是 Common Crawl 这类网页语料。假设先把网页切成段落，然后对每个段落做 embedding。如果两个段落相似度满足：

$$
\cos(E_i, E_j) \ge 0.93
$$

就只保留一个代表段落。这里的 $E_i$ 是第 $i$ 个段落的向量表示，白话讲就是把一段文字压成一个可比较的数字坐标点。这个策略的含义不是“两个段落必须完全一样”，而是“只要语义接近到阈值以上，就视为重复”。

实际工程里，边界还要加两层限制：

| 粒度 | 相似度度量 | 阈值示例 | 保留策略 |
|---|---|---|---|
| 文档 | 哈希完全匹配 | 相等 | 留首次出现 |
| 段落 | MinHash / Jaccard | $\ge 0.8$ | 留质量分高者 |
| 段落 | cosine similarity | $\ge 0.93$ | 每簇留 1 个 |
| 句子 | cosine similarity | $\ge 0.95$ | 每簇留 top-k |

这张表背后的工程含义是：不同层级承担不同任务。文档层负责高吞吐删硬重复，段落层负责处理转载与模板，句子层才处理真正难的“改写但同义”。

## 核心机制与推导

核心机制可以概括为三步：表示、检索、保留。

第一步是表示。把文本映射成 embedding，也就是把文字变成向量。向量的白话解释是：一种可以计算“距离和方向”的数字表示。设文本 $i$ 的 embedding 为 $E_i$，文本 $j$ 的 embedding 为 $E_j$，则常用相似度定义为：

$$
S(i,j)=\cos(E_i,E_j)=\frac{E_i \cdot E_j}{\|E_i\|\|E_j\|}
$$

当 $S(i,j)$ 越接近 1，说明两段文本方向越一致，也就是语义越像。

第二步是检索。因为真实语料是千万级到亿级，不能对所有样本两两比较，否则复杂度接近 $O(N^2)$。所以工程上先用哈希、LSH 或近邻检索把候选集合缩小，再在候选内精算相似度。

第三步是保留。不是“发现重复后全删”，而是在重复簇里保留代表样本。假设某个聚类内样本间都满足 $S(i,j)\ge \tau$，那么可以只保留最多 $k$ 个代表样本。$\tau$ 是阈值，白话就是“多像才算一类”；$k$ 是容量上限，白话就是“每类最多留几个”。

因此，一个常见的策略是：

1. 先做 exact dedup，删掉完全一样的文档。
2. 再做 approximate dedup，找到高概率近似重复候选。
3. 对候选做 embedding 聚类。
4. 每个聚类只保留 top-k 个代表。

去重比例可以定义为：

$$
r=\frac{N_{\text{removed}}}{N_{\text{total}}}
$$

这里 $r$ 表示整体剪枝强度。它不只是一个统计数字，还直接影响训练预算。如果 $r$ 很高，说明数据被大幅压缩，训练成本会下降；但若 $r$ 过高，也可能意味着多样性受损。

玩具例子可以帮助理解。假设有 6 个段落：

- A: “Python 是解释型语言。”
- B: “Python 是一种解释执行的编程语言。”
- C: “猫有四条腿。”
- D: “猫通常有四条腿。”
- E: “数据库事务强调原子性。”
- F: “今天上海下雨了。”

如果 embedding 后发现 A 与 B 的相似度是 0.96，C 与 D 的相似度是 0.95，而其他组合都低于 0.8，那么在 $\tau=0.93$、每簇保留 1 个的策略下，A/B 留一个，C/D 留一个，E/F 原样保留。最后从 6 条变成 4 条。这里删掉的不是“信息量”，而是“重复表达”。

再看一个经常被引用的数值级例子。假设有 120M 条预训练样本，经过 SemDeDup 这类语义去重方法，在 $\tau=0.93$ 下识别出 60M 条可删样本，则：

$$
r=\frac{60M}{120M}=0.5
$$

也就是去重比率为 50%。如果训练轮数、序列长度和模型规模保持不变，训练 token 总量会近似减半，计算成本也接近减半。若下游性能基本保持，说明原始数据里很大一部分只是重复权重，不是新增知识。

这类方法成立的前提，是重复样本对训练目标的边际贡献递减。白话讲：第一遍学到知识很有价值，第二十遍看到同一句话的价值就很低。尤其在网页语料中，转载、镜像、列表聚合、模板页、SEO 改写会制造大量“形式不同但内容高度重合”的文本。语义去重就是在消灭这部分冗余。

对应的流程示意可以写成：

```text
文本 -> embedding -> 近邻搜索 / 聚类 -> 相似簇 -> 保留 k 代表
```

## 代码实现

代码实现通常按阶段组织：① URL 或正文哈希；② MinHashLSH 或 Bloom 近似；③ embedding 聚类。关键不是某个算法名字，而是每一步都要留下日志和统计，让阈值可调、结果可回滚。

先看新手版伪码：

```text
for doc in corpus:
    hash_doc(doc)
    if hash_seen:
        skip
    else:
        emit_to_candidate_pool(doc)

for cluster in semantic_clusters(candidate_pool):
    keep top-k docs
    drop others
```

下面给一个可运行的 Python 玩具实现。它不依赖深度学习库，用词袋向量模拟 embedding，用余弦相似度做段落级去重。真实工程会换成更强的向量模型和 ANN 检索，但流程是一样的。

```python
from math import sqrt
from collections import Counter, defaultdict

docs = [
    {"id": "a", "text": "python is an interpreted language"},
    {"id": "b", "text": "python is a language interpreted at runtime"},
    {"id": "c", "text": "cats usually have four legs"},
    {"id": "d", "text": "cats have four legs"},
    {"id": "e", "text": "database transactions require atomicity"},
    {"id": "f", "text": "database transactions require atomicity"},  # exact duplicate
]

def normalize(text: str):
    return text.lower().split()

def exact_hash(text: str):
    return " ".join(normalize(text))

def vectorize(text: str):
    return Counter(normalize(text))

def cosine(v1, v2):
    keys = set(v1) | set(v2)
    dot = sum(v1.get(k, 0) * v2.get(k, 0) for k in keys)
    n1 = sqrt(sum(v * v for v in v1.values()))
    n2 = sqrt(sum(v * v for v in v2.values()))
    if n1 == 0 or n2 == 0:
        return 0.0
    return dot / (n1 * n2)

# stage 1: exact dedup
seen = set()
stage1 = []
for doc in docs:
    h = exact_hash(doc["text"])
    if h not in seen:
        seen.add(h)
        stage1.append(doc)

assert len(stage1) == 5  # f 被删掉

# stage 2: semantic dedup with threshold
tau = 0.6
kept = []
removed = set()

for i, doc_i in enumerate(stage1):
    if doc_i["id"] in removed:
        continue
    kept.append(doc_i)
    vi = vectorize(doc_i["text"])
    for j in range(i + 1, len(stage1)):
        doc_j = stage1[j]
        if doc_j["id"] in removed:
            continue
        vj = vectorize(doc_j["text"])
        if cosine(vi, vj) >= tau:
            removed.add(doc_j["id"])

assert [d["id"] for d in kept] == ["a", "c", "e"]
assert removed == {"b", "d"}

n_total = len(docs)
n_removed = n_total - len(kept)
r = n_removed / n_total

assert abs(r - 0.5) < 1e-9
print("kept:", [d["id"] for d in kept], "dedup_ratio:", r)
```

这个例子里的三个阶段对应真实系统里的三个思想：

| 阶段 | 目标 | 典型做法 |
|---|---|---|
| 阶段 1 | 删完全重复 | URL 哈希、正文哈希 |
| 阶段 2 | 找近似候选 | MinHash、LSH、Bloom、ANN |
| 阶段 3 | 删语义重复 | embedding、聚类、top-k 保留 |

真实工程例子通常会再加上质量分。比如一个语义簇里有 12 条近似重复段落，可以按质量分、语言检测可信度、来源白名单、长度完整性排序，最后只留前 1 到 3 条，而不是随机留一条。否则会把“坏版本”留下来，把“好版本”删掉。

日志设计也很重要。至少记录这些指标：

- 去重前后样本数、token 数、文档数
- 各阶段删除量
- 各语言、各来源、各主题的删除比例
- 人工抽样误删率
- 阈值变动后的性能回放结果

如果这些日志缺失，后面就无法判断是“阈值设置不对”，还是“某个来源原本就重复严重”。

## 工程权衡与常见坑

去重最大的工程难点，不是“能不能删”，而是“删到什么程度刚好”。阈值 $\tau$ 过低，会把本该保留的变体误判为重复；阈值过高，又会放过大量重复内容。

可以先看一个对照表：

| 情况 | 多样性 | 重复风险 | 训练成本 | 常见后果 |
|---|---|---|---|---|
| 阈值偏低 | 下降 | 低 | 低 | 误删有效变体，低资源语料受伤 |
| 阈值偏高 | 高 | 高 | 高 | 重复内容残留，观点被过度加权 |

再看是否去重的总体差异：

| 策略 | 偏见放大风险 | 泄露风险 | 算力消耗 | 数据覆盖 |
|---|---|---|---|---|
| 未去重 | 高 | 高 | 高 | 表面上高，实际冗余多 |
| 去重后 | 更可控 | 更可控 | 更低 | 覆盖更均衡 |

几个常见坑最值得提前说明。

第一，过度去重会伤害低资源语言和边缘领域。比如 Indic 语料中，同一含义常因转写、拼写习惯、形态变化出现多个版本。如果阈值设得太激进，这些“有价值的自然变体”会被当成重复删掉。结果不是更干净，而是语言多样性被主流表达吞掉。

第二，不去重会放大热门观点和模板页面。网页上最容易重复的，往往是 SEO 内容农场、站群转载、产品参数表、通用教程模板。如果不去重，模型会把这些内容学得特别牢，导致知识分布偏斜，甚至把错误信息反复强化。

第三，经典去重默认所有子集地位相同，但实际数据并不公平。主流语言、主流网站、主流主题往往样本更多，也更容易在“保留代表”阶段继续占优势。于是即便做了语义去重，最后保留下来的仍然是头部内容。FairDeDup 这类方法的思路，就是增加子集上限或公平校准，避免大语种把小语种完全挤掉。

第四，只看总去重率会误导决策。比如总去重率从 20% 提到 45%，看起来像是效率提升，但如果删掉的大部分来自教育、高质量技术文档，而保留下来的是论坛复读文本，那模型效果可能变差。去重必须和来源质量、任务目标一起看。

真实工程例子可以参考 FineWeb/FineWeb-Edu 这类大规模网页数据构建思路。它们处理多达 96 个 Common Crawl 快照时，不是简单做一次哈希，而是逐步记录 URL 去重、字符串近似去重、语义聚类去重，并结合教育领域筛选和混合采样。这样做的结果是：既减少冗余，也避免“全靠去重”把某些关键知识源删空。最终在 MMLU、ARC 这类知识密集型评测上，去重并配合采样的数据版本通常优于未去重版本。这里的启发不是“FineWeb 的参数应该照搬”，而是大规模语料必须把去重和采样联动设计。

一个实用规避清单如下：

| 常见坑 | 表现 | 规避方法 |
|---|---|---|
| 阈值太低 | 有效改写被大量删除 | 分语言、分领域单独调阈值 |
| 阈值太高 | 模板文残留严重 | 先做近似检索，再做语义精筛 |
| 只做文档级去重 | 段落模板无法删除 | 加段落级或句子级策略 |
| 随机保留代表 | 留下低质量版本 | 用质量分或来源权重排序 |
| 只看整体比例 | 局部子集被删空 | 做分层统计和人工抽样 |
| 不做回放验证 | 调参靠猜 | 固定小规模训练回放集 |

## 替代方案与适用边界

不是所有预训练项目都需要同样复杂的去重系统。策略要看数据规模、目标和可接受成本。

如果数据量较小，比如几十万到几百万文档，完全精确的哈希判重往往已经够用。因为这类数据集通常来源更可控，重复主要来自转载和备份，没必要上昂贵的语义聚类。

如果数据量巨大，来源混杂，且训练预算昂贵，多阶段方案更合适：先用哈希和 LSH 做高召回候选，再用 embedding 去掉语义重复。这里“高召回”白话讲就是宁可先多抓一些可疑样本，也别漏掉真正的重复。

如果目标是保持多样性，而不是极限压缩，那么不应一刀切只留 1 条，而可以采用“分层保留比例”或“每簇保留 top-k”。例如同一主题簇里保留 2 到 3 个不同写作风格的版本，模型仍能看到表达差异，但不会被几十篇近似文章反复训练。

跨语言或跨领域时，更适合引入 FairDeDup 一类思路。它的核心不是只问“像不像”，还问“这个子集已经保留了多少”。例如对 Indic 语料，可以把 $\tau$ 放宽，同时在每个语言子集里仅保留最多 2% 的高相似度块删除额度，避免小语种因天然表达集中而被误伤。换句话说，公平约束是对纯相似度规则的补充，不是替代。

可以用一个决策表来总结：

| 数据规模 / 目标 | 推荐策略 | 原因 |
|---|---|---|
| 小规模、来源单一 | 精确哈希去重 | 成本低，收益足够 |
| 中规模、网页抓取 | 哈希 + MinHash/LSH | 能处理转载和模板 |
| 超大规模、开放网页 | 哈希 + LSH + embedding 聚类 | 在成本与质量间平衡 |
| 跨语言、重多样性 | 语义去重 + 分层保留 + 子集上限 | 防止主流子集挤压边缘子集 |
| 高质量子集构建 | 语义去重 + 质量排序保留 | 避免留错代表样本 |

适用边界也要讲清楚。去重不能解决所有数据问题。它不能替代质量过滤，不能替代毒性检测，也不能替代版权和隐私治理。一个垃圾样本即使只有一份，仍然是垃圾；一个高质量样本即使有三个相近版本，也不一定都该删光。去重解决的是“重复权重失衡”，不是“内容正确性”本身。

因此，一个成熟的预训练数据管线通常是：

```text
抓取 -> 清洗 -> 去重 -> 质量过滤 -> 分布重平衡 -> 采样 -> 训练
```

去重处在中间位置，价值很高，但边界也很明确。
