## 核心结论

拒绝校准可以先用一句话定义：**让模型的拒绝决策与真实风险概率对齐**。这里的“对齐”不是让模型更爱拒绝，而是让它在该拒绝时拒绝，在不该拒绝时不要误伤正常请求。

安全对齐里的核心矛盾不是抽象的“安全 vs 能力”，而是很具体的决策边界问题。边界太靠左，模型会把大量无害请求判成危险，出现 `over-refusal`，也就是过度拒绝；边界太靠右，模型会放过本该拦截的请求，出现漏拒。对用户来说，前者损害可用性，后者损害安全性。

一个玩具例子最直观。用户问“怎么清理日志里的敏感字段”，这是典型合规需求。如果模型因为看到了 `log`、`sensitive` 就直接拒绝，这叫误拒。另一边，用户问“怎么绕过权限校验”，如果模型正常给出步骤，这叫漏拒。拒绝校准要做的不是让模型“更谨慎”四个字，而是把这两类错误同时压低。

| 状态 | 表现 | 直接后果 |
|---|---|---|
| 拒绝过多 | 正常请求也常被拦截 | 用户体验差，任务完成率下降 |
| 拒绝过少 | 危险请求被放行 | 风险输出增加，安全失守 |
| 校准良好 | 高风险拒绝，低风险放行，模糊场景追问 | 安全与可用性更平衡 |

## 问题定义与边界

先把问题写成一个最小决策模型。输入记为 $x$，真实标签记为 $y \in \{safe, unsafe\}$。这里的“真实标签”指人工或规则体系认定该请求是否属于应被拦截的危险请求。模型先给出一个风险分数 $r(x)\in[0,1]$，它可以理解成“模型估计这个请求有多危险”。然后用阈值 $\tau$ 做最终决策：

$$
\hat y_{refuse} = \mathbf{1}[r(x)\ge \tau]
$$

如果分数高于阈值就拒绝，否则放行或继续回答。两类关键错误率是：

$$
FRR = P(\hat y_{refuse}=1 \mid y=safe)
$$

$$
FNR = P(\hat y_{refuse}=0 \mid y=unsafe)
$$

`FRR` 是误拒率，也就是对安全请求错误拒绝的比例。`FNR` 是漏拒率，也就是对危险请求错误放行的比例。

这里要区分三个常被混淆的概念。

第一，`unsafe` 不等于“包含危险词”。危险词只是表面信号，不能直接替代风险判断。比如 `kill process` 可能是本地排障，也可能是攻击链的一步。

第二，`refuse` 不等于“系统安全”。如果模型把一切复杂请求都拒绝，表面上很安全，实际上只是用可用性换指标。

第三，`calibration` 不等于“分类更准”。校准关心的是分数和真实风险频率是否匹配，而不是只关心排对顺序。

实际系统通常至少要区分三类边界：

| 场景类型 | 含义 | 默认策略 |
|---|---|---|
| `safe` | 明确无害，有正常业务上下文 | 放行或直接回答 |
| `unsafe` | 明确危险，包含攻击、绕过、外泄等目标 | 拒绝，并给安全替代 |
| `uncertain context` | 语义模糊，上下文不足，无法直接断定 | 追问、缩小范围、改写成安全任务 |

“追问”很重要，因为它是拒绝和放行之间的第三种动作。对零基础读者来说，可以把它理解成“先补充上下文，再做风险判定”，这比一刀切更适合真实产品。

## 核心机制与推导

拒绝校准的机制可以拆成两步：**先估计风险，再做阈值决策**。如果第一步的风险分数本身不准，第二步再怎么调阈值也只能局部补救。

校准关注的是下面这个关系是否成立：

$$
P(y=unsafe \mid r(x)\approx p) \approx p
$$

意思是，如果模型给一批请求都打了大约 `0.80` 的风险分数，那么这批请求里真实危险的比例也应该接近 `0.80`。如果实际只有 `0.50`，说明模型过度自信；如果实际有 `0.95`，说明模型低估风险。

工程上常用 `ECE`，也就是 Expected Calibration Error，衡量分桶后的平均偏差：

$$
ECE=\sum_k |acc(B_k)-conf(B_k)|\cdot \frac{|B_k|}{N}
$$

这里的 `bin` 可以白话理解成“把相近分数放进同一个桶”。`acc(B_k)` 是该桶里真实危险样本比例，`conf(B_k)` 是该桶里平均预测分数。

看一个玩具例子。设阈值 $\tau=0.70$：

- A：无害请求，$r(A)=0.72$，被拒绝，属于误拒。
- B：危险请求，$r(B)=0.68$，被放行，属于漏拒。

这说明问题不只在阈值，也可能在分数本身。A 被高估，B 被低估。拒绝校准的工作，就是把这种“该高不高、该低不低”的风险估计压回合理区间。

| 风险分数区间 | 阈值比较 | 决策 | 可能错误 |
|---|---|---|---|
| 低于 $\tau$ 很多 | 明显小于阈值 | 放行 | 若真实危险则漏拒 |
| 接近 $\tau$ | 边界不稳定 | 放行、追问或拒绝 | 最容易误判 |
| 高于 $\tau$ 很多 | 明显大于阈值 | 拒绝 | 若真实安全则误拒 |

再看一个分桶示意表：

| bin | 样本数 | 平均置信度 | 真实 unsafe 比例 | 校准误差 |
|---|---:|---:|---:|---:|
| 0.0-0.2 | 20 | 0.12 | 0.10 | 0.02 |
| 0.2-0.4 | 20 | 0.31 | 0.25 | 0.06 |
| 0.4-0.6 | 20 | 0.51 | 0.45 | 0.06 |
| 0.6-0.8 | 20 | 0.71 | 0.58 | 0.13 |
| 0.8-1.0 | 20 | 0.88 | 0.62 | 0.26 |

最后一桶最值得警惕。模型经常把高风险说得很确定，但真实危险比例并没有那么高，这就是对齐后模型常见的过拒来源之一：**学会了“遇到风险信号就强烈拒绝”，却没有把风险概率估准**。

真实工程里，企业代码助手就是典型案例。用户输入“open port 8080”“kill process using 9000”“sanitize log before export”，这些词在安全系统里都敏感，但在开发、运维、合规语境里完全可能是正当需求。只看词面，会把大量正常请求推到高风险桶，导致拒绝边界整体左移。

## 代码实现

最小实现通常分三步：拿到风险分数、搜索阈值、在验证集上计算 `FRR`、`FNR` 和 `ECE`。下面这个 Python 例子可以直接运行，演示一个小型拒绝校准流程。

```python
import numpy as np

# y: 0=safe, 1=unsafe
y = np.array([0, 0, 0, 1, 1, 1, 0, 1, 0, 1])

# 模型给出的风险分数
r = np.array([0.72, 0.18, 0.35, 0.68, 0.91, 0.83, 0.44, 0.77, 0.22, 0.62])

def refuse(score, tau):
    return (score >= tau).astype(int)

def frr(y, pred):
    safe_mask = (y == 0)
    return np.mean(pred[safe_mask] == 1)

def fnr(y, pred):
    unsafe_mask = (y == 1)
    return np.mean(pred[unsafe_mask] == 0)

def ece(conf, y, bins=5):
    edges = np.linspace(0.0, 1.0, bins + 1)
    total = len(y)
    score = 0.0
    for i in range(bins):
        if i == bins - 1:
            mask = (conf >= edges[i]) & (conf <= edges[i + 1])
        else:
            mask = (conf >= edges[i]) & (conf < edges[i + 1])
        if mask.sum() == 0:
            continue
        acc = y[mask].mean()
        avg_conf = conf[mask].mean()
        score += abs(acc - avg_conf) * mask.sum() / total
    return score

taus = np.linspace(0.1, 0.9, 17)
best_tau = None
best_cost = float("inf")

for tau in taus:
    pred = refuse(r, tau)
    cur_frr = frr(y, pred)
    cur_fnr = fnr(y, pred)

    # 简单代价函数：更重视漏拒
    cost = 1.0 * cur_frr + 2.0 * cur_fnr
    if cost < best_cost:
        best_cost = cost
        best_tau = tau

pred = refuse(r, best_tau)

assert len(pred) == len(y)
assert 0.0 <= frr(y, pred) <= 1.0
assert 0.0 <= fnr(y, pred) <= 1.0
assert 0.0 <= ece(r, y, bins=5) <= 1.0

print("best_tau =", round(float(best_tau), 2))
print("FRR =", round(float(frr(y, pred)), 3))
print("FNR =", round(float(fnr(y, pred)), 3))
print("ECE =", round(float(ece(r, y, bins=5)), 3))
```

这个例子故意保留了一个常见现实：风险分数 `r` 不是完美的，所以即使你找到了更合适的阈值，也只是“在现有分数质量上做最优切分”。如果想进一步改善，往往要同时做两件事：

1. 提升风险打分器质量，比如补充 hard negatives。这里的 hard negative 可以白话理解成“长得像危险请求、其实是正常请求的难例”。
2. 在验证集上重新做校准，比如温度缩放或分场景阈值。

流程可以概括成：

| 步骤 | 输入 | 输出 |
|---|---|---|
| 输入理解 | 用户请求与上下文 | 标准化特征 |
| 风险打分 | 分类器、reward model 或 policy head | 风险分数 $r(x)$ |
| 校准/阈值 | 验证集统计 | 拒绝阈值、追问阈值 |
| 最终动作 | 风险分数与策略规则 | 拒绝、追问、放行 |

真实工程例子里，代码助手通常不会只有“拒绝/放行”两档。更常见的做法是三路输出：

- 高风险：拒绝，并给安全替代方案
- 中风险：追问上下文
- 低风险：正常回答

比如用户说“帮我 kill 掉所有相关进程”。系统如果检测到这是本地开发机排障，可以继续给 `lsof`、`ps`、`kill` 的安全说明；如果检测到这是远程批量终止生产服务，且上下文不足，则先追问范围和环境；如果用户目标明显是绕过权限或破坏服务，再进入拒绝路径。

## 工程权衡与常见坑

工程上最大的误区，是把拒绝校准误解成“调一个更安全的阈值”。真正的问题更复杂，因为它同时涉及数据分布、上下文质量、业务场景和监控方式。

先看常见坑：

| 坑 | 结果 | 原因 |
|---|---|---|
| 只看拒绝率 | 系统越来越保守 | 拒绝多不等于判得准 |
| 关键词训练不足 | 关键词触发式误拒 | 模型学到表面词，不学语义边界 |
| 单阈值全场景通用 | 场景失配 | 不同产品线风险分布不同 |

第一个坑最常见。很多团队上线后发现安全投诉下降，于是继续提高阈值或扩大规则，最后模型看起来更“稳”，但真实情况是无害请求通过率持续下滑。用户不是拿不到危险答案，而是连正常工作也做不成。

第二个坑来自训练数据。若数据里“带危险词但无害”的样本太少，模型就会把 `exploit`、`port`、`inject`、`credential` 这类词当成粗暴触发器。对新手来说，这可以理解成“模型学会了看见红灯就刹车，但没学会判断前方到底是不是路口”。

第三个坑是部署层面的。通用聊天、代码助手、医疗问答、企业知识库，这些场景的风险形态差异极大。一个在客服场景表现良好的阈值，迁移到代码助手可能马上造成大面积误拒。

因此监控必须同时覆盖安全和可用性：

| 指标 | 作用 |
|---|---|
| `FRR` | 看误拒是否伤害正常用户 |
| `FNR` | 看危险请求是否漏过 |
| `ECE` | 看风险分数是否可信 |
| 人工复核命中率 | 看高风险样本是否真的危险 |
| 红队集命中率 | 看系统对已知攻击模式是否稳定 |

如果线上允许回放，最好做分桶复盘：哪些请求被拒了、哪些被追问、哪些被放行、其中哪些后来被人工判成误拒或漏拒。这样的回放比单看聚合指标更有价值，因为你能看到边界到底偏在哪类语义上。

## 替代方案与适用边界

阈值搜索不是唯一方案，它只是最低成本、最容易落地的一层。

| 方案 | 优点 | 局限 |
|---|---|---|
| 阈值搜索 | 简单、快、实现成本低 | 强依赖验证集质量 |
| Temperature Scaling | 改善概率解释性 | 不改变样本排序 |
| Calibration-aware Fine-tuning | 能直接优化拒绝边界 | 训练成本高，数据要求高 |
| 规则/关键词过滤 | 便宜、响应快 | 误拒高，语义泛化差 |

`Temperature Scaling` 可以白话理解成“把模型分数整体压平或拉开，让概率更像真实概率”。它常用于已有排序还可以，但分数过于自信的情况。`Calibration-aware Fine-tuning` 则更进一步，直接在训练目标里把误拒与漏拒都纳入优化。

不同风险等级，推荐策略也不同：

| 风险等级 | 典型场景 | 推荐方案 |
|---|---|---|
| 低风险 | 通用问答、基础说明 | 阈值搜索 + 轻量追问 |
| 中风险 | 代码助手、企业工具、运维问答 | 分场景阈值 + 校准 + 安全替代 |
| 高风险 | 生物、医疗、攻击链、敏感执行 | 偏保守阈值 + 强规则 + 人工复核 |

这里的重点不是“高风险就一律拒绝”，而是高风险系统要更重视下游后果。如果一次漏拒成本极高，阈值可以保守；但如果产品强依赖可用性，就应尽量增加追问、上下文补全和安全改写，而不是只靠硬拒绝。

还是用代码助手举例。用户输入“kill process”。直接回答，风险太高；直接拒绝，也可能误伤正常排障。更好的做法通常是追问：“是本地开发环境排障，还是远程服务器批量操作？”这一步本质上是在补全上下文，把 `uncertain context` 转成可以判定的 `safe` 或 `unsafe`。很多误拒问题，不是模型不会分类，而是系统太早做了二元决策。
