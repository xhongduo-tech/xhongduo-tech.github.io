## 核心结论

细粒度实体识别（Fine-grained Entity Recognition, FG-NER）要解决的问题，不是“这是不是一个人”，而是“这个人具体是什么类型的人”。传统实体识别通常只给出 `PER/LOC/ORG` 这类粗粒度标签，也就是“人 / 地点 / 组织”三大类；细粒度实体识别则继续向下拆，比如“演员、导演、运动员、大学、政府机构、公司品牌”。

这件事的价值在于语义精度。语义精度的意思是，系统对文本含义的刻画更细，后续知识图谱、搜索召回、关系抽取、问答系统都更容易利用这些标签。识别出“张艺谋”是 `PER` 只能说明他是人；识别出他是“导演/编剧”，系统才能更稳定地推断“执导了什么电影”“与哪些演员共现”。

但类别一旦从 4 类扩到 100+ 类，问题难度不是线性增加，而是结构性变化。核心难点主要有三个：

| 维度 | 粗粒度 NER | 细粒度实体识别 |
|---|---|---|
| 标签规模 | 通常 3 到 10 类 | 常见是几十到上百类，如 FIGER 的 112 类 |
| 决策目标 | 判断大类 | 判断具体语义角色 |
| 数据分布 | 相对均衡 | 明显长尾，少数类样本极少 |
| 标签关系 | 多为平级 | 常带层次结构，如 人物→艺术家→演员 |
| 错误代价 | 错把人当地点 | 错把导演当演员，语义更接近但业务仍可能出错 |

工程上可行的主线方案通常是三步组合：

1. 用层次化分类利用标签树，而不是把 112 类当成互不相关的平面标签。
2. 用负采样平衡训练，缓解长尾分布导致的多数类支配。
3. 用类型约束解码，把不可能的组合在预测阶段直接剪掉。

在 FIGER 这类数据集上，这条路线已经能达到可用水平，公开结果中 Micro-F1 可到约 68.7%。这不是“问题已经解决”，而是说明细粒度标签已经足以进入真实抽取链路，而不只是论文玩具任务。

## 问题定义与边界

先定义任务。实体识别中的“实体”，可以理解为文本里指向某个具体对象的词或短语，比如“北京大学”“乔布斯”“iPhone”。细粒度实体识别关注的是：给已经识别出的实体片段分配更细的类型标签。

这里要先划清边界。

第一，细粒度实体识别不一定负责“找边界”。“边界”是指实体在句子里的起止位置。例如“电影导演张艺谋”里，实体可能是“张艺谋”，也可能是“电影导演张艺谋”中的一部分。很多系统会把任务拆成两步：先做实体检测，再做细粒度类型判定。

第二，它往往是多标签任务。多标签的意思是，一个实体可以同时属于多个类型。比如“张艺谋”可以同时是“人物”“艺术家”“导演”“编剧”。这和传统单标签分类不同，后者通常只允许一个答案。

第三，它常常依赖类型层级。层级的意思是，标签之间不是平铺的，而是有父子关系：

$$
人物 \rightarrow 艺术家 \rightarrow 导演
$$

如果系统预测了“导演”，那通常也应该隐含满足“人物”和“艺术家”这两个更上层的约束。

一个玩具例子可以直接看出差异：

句子：`张艺谋执导了《英雄》。`

- 粗粒度 NER：`张艺谋 -> PER`
- 细粒度实体识别：`张艺谋 -> {人物, 艺术家, 导演, 编剧}`

这个例子里，“人物”是正确但不够用的答案；“导演”才是后续关系抽取真正想要的类型信号。

再看一个边界问题：如果业务只需要“人/地/组织”做高亮展示，那细粒度识别往往没有必要。标签越细，标注成本越高，模型越容易受噪声影响。也就是说，FG-NER 不是默认更好，而是适合那些确实需要精细语义的系统。

## 核心机制与推导

细粒度实体识别的难点，主要来自长尾分布、层次结构和解码约束。

### 1. 长尾分布为什么难

长尾分布的意思是，少数常见类别有大量样本，而大量细分类别样本极少。比如“人物”可能出现成千上万次，“电影制片人”可能只出现几十次。模型训练时，梯度更新会被高频类主导，结果就是模型越来越会预测大类和热门类，却学不会稀有类。

这会导致一种典型坏现象：系统总是愿意预测“人物”，但不愿意进一步落到“导演”或“运动员”。

所以训练时需要负采样。负采样的白话解释是：不要把所有非真标签都一股脑当作负样本，而是挑一部分更有代表性的负例来学。否则 112 类里，绝大多数类都会成为负例，训练信号会严重失衡。

### 2. 层次结构为什么重要

如果直接把 112 个类型看成平面标签，模型要从零学习“导演”和“演员”都属于“艺术家”，样本利用率很低。层次化分类的做法是，把预测过程拆成逐层决策：

1. 先判断是不是“人物”
2. 再判断是不是“艺术家”
3. 再判断是不是“导演”

这种做法的优势有两个：

- 参数共享更自然。学到“导演”的样本，也能反哺“艺术家”和“人物”这两层。
- 候选空间更小。若上层不是“人物”，下层就没必要再考虑“导演”。

### 3. 类型约束解码为什么有效

解码可以理解为“把模型分数变成最终标签”的最后一步。类型约束解码的核心思想是：预测结果必须满足层级一致性。

例如模型原始分数可能给出：

- `导演 = 0.82`
- `艺术家 = 0.31`
- `人物 = 0.40`

这在平面多标签里似乎可以接受，但在层级标签里是不合理的。因为“导演”不应脱离“艺术家”和“人物”单独成立。类型约束解码会把这种不一致修正成一条合法路径，或者直接丢弃不合法的子类型。

### 4. Micro-F1 为什么常用

FG-NER 常用 Micro-F1 做总体评估。原因是标签很多、且分布不均，逐类平均很容易受稀有类波动影响。Micro-F1 先把所有类别、所有样本的预测汇总，再统一计算精确率和召回率。

设全局真阳性、假阳性、假阴性分别为 $TP, FP, FN$，则：

$$
Precision = \frac{TP}{TP + FP}
$$

$$
Recall = \frac{TP}{TP + FN}
$$

$$
Micro\text{-}F1 = \frac{2 \cdot Precision \cdot Recall}{Precision + Recall}
$$

玩具例子如下。假设一个批次里：

- 预测对了 3 个标签，即 $TP=3$
- 多报了 2 个标签，即 $FP=2$
- 漏掉了 1 个标签，即 $FN=1$

则：

$$
Precision = \frac{3}{3+2}=0.6
$$

$$
Recall = \frac{3}{3+1}=0.75
$$

$$
Micro\text{-}F1 = \frac{2 \times 0.6 \times 0.75}{0.6 + 0.75} \approx 0.67
$$

这个指标的含义很直接：它衡量的是整个系统在所有标签上的总体命中质量，而不是只看头部类。

### 5. 一个真实工程例子

在知识图谱构建里，假设你要从影视新闻中抽取“人物 - 作品 - 职业”关系。

句子：`导演张艺谋与演员巩俐出席了电影《红高粱》修复版首映。`

如果只有粗粒度标签，系统可能得到：

- 张艺谋 -> 人物
- 巩俐 -> 人物
- 红高粱 -> 作品

这不足以区分“谁是导演，谁是演员”。关系抽取模型只能依赖上下文硬推，鲁棒性较差。

如果有细粒度类型：

- 张艺谋 -> {人物, 艺术家, 导演, 编剧}
- 巩俐 -> {人物, 艺术家, 演员}
- 红高粱 -> {作品, 电影}

那么后续系统就能把候选关系空间大幅缩小。例如“导演-执导-电影”是高概率关系，“演员-执导-电影”则可以降权甚至过滤。这就是细粒度类型在工程链路里的真正价值：它不是终点，而是下游推理的约束条件。

## 代码实现

下面给一个最小可运行示例，演示三件事：

1. 用层级约束过滤非法标签
2. 用简单负采样构造训练负例
3. 统计全局 Micro-F1

```python
from collections import defaultdict
import random

# 标签层级：child -> parent
PARENT = {
    "artist": "person",
    "director": "artist",
    "screenwriter": "artist",
    "athlete": "person",
    "company": "organization",
}

ALL_LABELS = {"person", "artist", "director", "screenwriter", "athlete", "organization", "company"}

def close_under_hierarchy(labels):
    """补全祖先标签，例如 director -> artist -> person"""
    result = set(labels)
    added = True
    while added:
        added = False
        for label in list(result):
            parent = PARENT.get(label)
            if parent and parent not in result:
                result.add(parent)
                added = True
    return result

def constrained_decode(scores, threshold=0.5):
    """
    先按阈值选标签，再做层级闭包，保证父标签存在。
    scores: dict[label] = prob
    """
    chosen = {label for label, score in scores.items() if score >= threshold}
    return close_under_hierarchy(chosen)

def sample_negatives(gold_labels, k=2):
    """
    简化版负采样：从非真标签中随机采样 k 个负例
    """
    candidates = list(ALL_LABELS - set(gold_labels))
    random.seed(7)
    return set(random.sample(candidates, min(k, len(candidates))))

def micro_f1(gold_sets, pred_sets):
    tp = fp = fn = 0
    for gold, pred in zip(gold_sets, pred_sets):
        tp += len(gold & pred)
        fp += len(pred - gold)
        fn += len(gold - pred)

    precision = tp / (tp + fp) if tp + fp else 0.0
    recall = tp / (tp + fn) if tp + fn else 0.0
    f1 = 2 * precision * recall / (precision + recall) if precision + recall else 0.0
    return precision, recall, f1

# 玩具数据
gold_zhang = close_under_hierarchy({"director", "screenwriter"})
gold_company = close_under_hierarchy({"company"})

scores_zhang = {
    "director": 0.86,
    "screenwriter": 0.61,
    "artist": 0.22,   # 即使父类分数低，也由约束补齐
    "person": 0.15,
    "athlete": 0.12,
}

scores_company = {
    "company": 0.71,
    "organization": 0.33,
    "person": 0.55,   # 一个明显误报
}

pred_zhang = constrained_decode(scores_zhang, threshold=0.5)
pred_company = constrained_decode(scores_company, threshold=0.5)

precision, recall, f1 = micro_f1(
    [gold_zhang, gold_company],
    [pred_zhang, pred_company]
)

# 基本断言
assert "person" in pred_zhang and "artist" in pred_zhang and "director" in pred_zhang
assert "organization" in pred_company and "company" in pred_company
assert round(precision, 4) == 0.8571
assert round(recall, 4) == 1.0
assert round(f1, 4) == 0.9231

# 负采样示例
negatives = sample_negatives(gold_zhang, k=2)
assert len(negatives) == 2
assert negatives.isdisjoint(gold_zhang)

print("pred_zhang =", pred_zhang)
print("pred_company =", pred_company)
print("Micro-F1 =", round(f1, 4))
print("sampled negatives =", negatives)
```

这段代码是教学版，不是生产版，但已经体现了 FG-NER 的核心结构。

第一，`close_under_hierarchy` 实现了层级闭包。闭包的意思是：只要命中子类型，就自动补全它的祖先类型。

第二，`constrained_decode` 模拟类型约束解码。真实系统里可能用动态规划、图约束、层级 CRF 或打分重排序，这里只演示最基本思想。

第三，`sample_negatives` 展示负采样。真正工程里往往不会纯随机，而会加入“难负例”。难负例的意思是，那些和真标签很接近、最容易混淆的错误类型，比如“导演”对“演员”，“公司”对“政府机构”。

如果把这段逻辑嵌进真实训练流程，通常结构如下：

| 模块 | 作用 | 常见实现 |
|---|---|---|
| 编码器 | 把实体上下文转成向量 | BERT、RoBERTa、指令模型编码 |
| 标签层级 | 存放父子关系 | 树、DAG、邻接表 |
| 训练采样 | 处理长尾 | class-balanced、难负采样、重加权 |
| 解码器 | 输出合法标签集 | 阈值+闭包、层级排序、约束搜索 |
| 评估器 | 统一比较效果 | Micro-F1、Macro-F1、层级一致率 |

## 工程权衡与常见坑

第一类坑是标签设计过细，但标注规范不稳定。比如“企业家”和“公司创始人”是否并列，或者后者是否应视为关系而非类型。如果标签定义不清，模型性能再高也只是学会了标注员偏好。

第二类坑是远监督噪声。远监督的意思是，用知识库自动对齐文本生成训练数据，而不是人工逐条标注。它的优点是便宜，缺点是噪声大。某个实体在知识库里有“导演”标签，不代表它在当前句子里真的以“导演”身份出现。于是模型会把“实体全局属性”误学成“句子局部表达”。

第三类坑是阈值统一设置。多标签系统常给每个标签一个分数，再统一用 0.5 截断。但长尾任务里，不同标签的分数分布差异很大。热门类可能 0.7 才可信，冷门类 0.35 已经很强。一个全局阈值往往不合适。

第四类坑是只看 Micro-F1。Micro-F1 对总体效果很有用，但它天然更关注高频标签。如果业务很在意少数类，比如“药物”“病毒”“法规条款”，那还要同时看每类召回、Macro-F1 或业务关键类的专门指标。

第五类坑是忽略层级一致性。模型可能输出“导演”但不输出“人物”，或者同时输出互斥的兄弟标签。若不做约束解码，下游系统要自己处理这些矛盾，整个链路会变脆。

真实工程里，常见折中方案是：

1. 先把标签树压到业务真正需要的深度，而不是盲目追求 100+ 类。
2. 训练时对低频类做重采样或损失重加权。
3. 推理时引入层级约束，宁可少报一点，也不要输出结构性错误。
4. 评估时同时看总体指标和关键类指标。

## 替代方案与适用边界

细粒度实体识别不是唯一选择。是否采用它，要看系统目标。

一种替代方案是“粗粒度 NER + 关系抽取”。也就是先只识别人、地点、组织，再把“导演”“创始人”“校长”这类信息交给关系模型处理。这个方案适合标签体系不稳定、但关系集合相对固定的场景。

另一种替代方案是“实体链接”。实体链接的意思是，把文本里的实体直接对齐到知识库条目，例如把“张艺谋”对齐到某个唯一 ID。只要知识库里已有丰富属性，就能间接得到其细粒度类型。它更适合头部实体丰富、知识库覆盖高的场景；对新实体和长尾实体则不稳。

还有一种替代方案是“开放类型生成”。即不从固定 112 类里选，而是让大模型直接生成类型短语，比如“电影导演”“中国女演员”。它的优点是开放，缺点是评估难、规范化难、上线一致性差。对于知识图谱等强约束系统，通常仍要回到受控标签集合。

可以用下面这个表做决策：

| 方案 | 适合场景 | 优点 | 风险 |
|---|---|---|---|
| 粗粒度 NER | 只需基础实体识别 | 成本低，稳定 | 语义不够细 |
| 细粒度实体识别 | 需要可控细标签 | 便于下游结构化利用 | 长尾与噪声问题明显 |
| 实体链接 | 知识库覆盖高 | 可直接复用库内属性 | 新实体难处理 |
| 开放类型生成 | 探索式分析、弱约束系统 | 灵活，覆盖广 | 难评估，难标准化 |

因此，FG-NER 的适用边界很明确：

- 当你需要稳定、可控、可评估的细类型标签时，它很合适。
- 当你只需要“是不是个人名”，它通常太重。
- 当知识库已足够强，实体链接可能更直接。
- 当标签体系无法预先定义，开放生成更灵活，但工程可控性更差。
