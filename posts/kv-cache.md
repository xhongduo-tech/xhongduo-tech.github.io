## KV Cache 的意义，是把自回归推理从重复计算改成增量计算

自回归生成时，第 `t` 步预测下一个 token 需要利用前 `t-1` 个位置的注意力信息。如果每一步都从头重新计算整个前缀的 Key 和 Value，推理成本会被大量重复计算吞掉。KV Cache 的作用，就是把这些历史中间结果缓存下来，只对新 token 做增量计算。

---

## 不缓存时为什么浪费

对一个长度逐渐增长的前缀：

- 第 1 步算 1 个 token 的 K/V
- 第 2 步又重复算前 2 个
- 第 3 步又重复算前 3 个

这样累计下来，许多历史位置被反复重算。  
从系统视角看，模型并不是不会算，而是在做没有必要的重复劳动。

---

## 使用 KV Cache 后发生了什么

缓存后，历史 token 的 `K` 和 `V` 在第一次计算后就被保留：

- 新一步只为最新 token 计算 `q_t, k_t, v_t`
- `q_t` 再与缓存中的全部 `K` 做注意力
- 输出使用缓存中的全部 `V`

这样，前缀处理从“每步重算全历史”变成“每步只追加最新状态”。

---

## 时间与空间的交换

KV Cache 的收益是显著降低重复计算，但代价是额外内存。缓存大小大致与以下量线性相关：

```text
batch × seq_len × n_layers × n_kv_heads × d_head × 2
```

序列越长、层越多、batch 越大，显存压力越重。  
这就是为什么在大模型推理系统里，KV Cache 往往比权重本身更早成为瓶颈。

---

## 为什么它对长上下文推理尤其重要

当上下文扩展到 8K、32K 甚至更长时：

- 不使用缓存，自回归延迟会急剧恶化
- 使用缓存，计算仍是增量式，但显存占用会快速上升

因此现代推理优化很多都围绕 KV Cache 展开，例如：

- GQA / MQA 减少 KV 头数
- PagedAttention 优化缓存布局
- 前缀共享复用相同 prompt 的缓存块

---

## 小结

KV Cache 的核心价值，是把自回归解码从重复的全前缀计算改写成只对新增 token 进行增量更新。它显著降低了推理延迟，却把问题转移到显存占用与缓存管理上。

因此，KV Cache 不是一个小优化，而是现代 LLM 推理系统的基础设施。你越重视长上下文和高并发，它就越成为首要系统设计对象。
