## 核心结论

边缘模型性能优化，是在端侧设备上同时优化 `时延、内存、功耗、算子支持` 的工程过程。这里的“端侧”就是手机、平板、摄像头、工控盒子这类本地设备；“优化”也不只是把模型参数量变小，而是让模型尽量走设备最擅长的低精度和硬件加速路径。

最重要的判断是：端侧性能瓶颈通常不是单一因素，而是 `计算 + 内存带宽 + 数据搬运 + 前后处理 + 算子兼容` 的叠加结果。只要其中一环退回慢路径，整体收益就会被吃掉。

总耗时可以先用一个足够实用的拆解式理解：

$$
T_{total} = T_{compute} + T_{memory} + T_{copy} + T_{pre/post}
$$

其中：
- `T_compute` 是算子真正执行的时间。
- `T_memory` 是读写权重和激活的时间，白话说就是“芯片等数据”的时间。
- `T_copy` 是 CPU、GPU、NPU 之间搬数据的时间。
- `T_pre/post` 是前处理和后处理时间，比如图像缩放、归一化、NMS、OCR 解码。

一个新手最容易理解的玩具例子是模型体积。假设模型有 1000 万个参数：
- FP32 权重大小约为 $10^7 \times 4\text{B} = 40\text{MB}$
- INT8 权重大小约为 $10^7 \times 1\text{B} = 10\text{MB}$

这不是单纯“文件更小”，而是权重更快加载、更容易放进缓存，也更可能直接走整数运算单元。端侧优化的核心不是某个单点技巧，而是“模型格式、算子图、运行时、硬件路径”一起对齐。

| 优化对象 | 直接收益 | 常见副作用 |
|---|---|---|
| 权重量化 | 模型更小、加载更快 | 精度可能下降 |
| 激活量化 | 低精度端到端执行 | 校准要求更高 |
| 算子替换/改写 | 提高加速器覆盖率 | 需要改模型结构 |
| delegate / accelerator | 吞吐和时延提升 | 兼容性受限 |
| 前后处理优化 | 直接降低端到端时延 | 需要做链路 profiling |

## 问题定义与边界

本文讨论的是“端侧推理性能优化”，不是训练加速，也不是云端批处理推理。

“推理”就是模型已经训练好，现在拿输入去算输出；“端侧推理”就是这件事发生在用户设备本地，而不是服务器。它的目标不是单一的最高吞吐，而是一个多目标约束问题。

真实工程里至少有四个硬约束：

| 约束 | 白话解释 | 典型指标 |
|---|---|---|
| 时延 | 用户等多久出结果 | 首帧时间、P50/P95 |
| 内存 | 设备能不能装下并稳定运行 | 峰值 RSS、模型体积 |
| 功耗 | 会不会发热、掉电太快 | 平均功耗、热降频 |
| 算子支持 | 能不能走加速器而不是回退 | delegate 覆盖率 |

边界也要划清：

| 场景 | 关注点 | 是否属于本文范围 |
|---|---|---|
| 手机端实时 OCR | 首帧时延、功耗、发热 | 是 |
| 摄像头端目标检测 | 稳定帧率、内存占用 | 是 |
| 云端大模型服务 | 吞吐、批大小、GPU 利用率 | 否 |
| 模型训练加速 | 反向传播、收敛速度 | 否 |
| 离线批量标注 | 总吞吐、成本 | 部分不同，不作为重点 |

真实工程例子可以用手机相册里的实时 OCR。用户真正关心的是：
- 点开照片后能不能马上开始识别
- 滑动多张图时会不会卡顿
- 连续使用几分钟后会不会明显发热
- 没网时能不能照样工作

所以边缘模型性能优化不是“谁 benchmark 分数高”，而是“在指定设备、指定温度、指定业务链路下，能不能稳定满足目标 SLA”。如果瓶颈在图像解码、YUV 转 RGB、裁剪、NMS 或文本后处理，那么只量化模型本体通常解决不了根问题。

## 核心机制与推导

量化的本质，是用更低比特数近似表示原来的浮点数。这里的“比特数”就是数值存储所占的位宽，位宽越小，通常存储和搬运越便宜。

最常见的 8-bit 线性量化可以写成：

$$
q = \text{round}(x / s) + z
$$

$$
x \approx s(q - z)
$$

其中：
- `x` 是原始浮点值
- `q` 是量化后的整数值
- `s` 是 `scale`，也就是缩放因子，白话说是“整数单位对应多大的真实数值”
- `z` 是 `zero-point`，白话说是“让 0 在整数空间里也能被准确表示的位置”

为什么这样能省资源？先看存储：

$$
M_{INT8} \approx \frac{M_{FP32}}{4}
$$

因为 FP32 通常占 32 bit，INT8 占 8 bit，所以同样参数个数下，理论体积约为四分之一。

| 项目 | FP32 | INT8 |
|---|---:|---:|
| 单参数占用 | 4B | 1B |
| 1000 万参数体积 | 40MB | 10MB |
| 理论体积比例 | 1 | 1/4 |

但端到端更快，不只是体积缩小。真正的原因有三层：

1. 更少的数据移动  
权重和激活更小，内存带宽压力下降，`T_memory` 往往会明显下降。

2. 更适合硬件执行  
很多移动端 CPU、DSP、NPU 对 INT8 有专门指令或执行单元，`T_compute` 会下降。

3. 更容易提高缓存命中  
模型块更小，进入 cache 的概率更高，减少主存访问。

这里有一个常见误解：只要量化成 INT8，就一定更快。这个结论不成立。因为性能收益是否真的落地，取决于“整条执行图”是否都能走低精度路径。

如果图里某个关键算子不被加速器支持，就会出现 fallback，也就是“回退到备用执行路径”，通常意味着跑回 CPU 或回到浮点实现。此时可能发生两件事：
- `T_compute` 虽然在部分层下降了
- 但 `T_copy` 和额外调度开销上升了

结果就是总时延不降反升。

玩具例子：一个小 CNN 的大部分卷积都支持 INT8，但中间插了一个不支持的自定义激活层。前面 NPU 算完，数据拷回 CPU，CPU 用浮点算这层，再拷回 NPU。单看那个激活层本身很便宜，但两次内存搬运和同步足以把前面省下的时间吃掉。

这也是为什么边缘优化通常遵循下面的因果链：

| 步骤 | 目标 | 如果失败会怎样 |
|---|---|---|
| 校准数据准备 | 让激活范围估计合理 | 精度明显下降 |
| 模型量化 | 降低位宽 | 得到混合图而非全量低精度 |
| 算子覆盖率检查 | 确保主要子图可加速 | fallback 增多 |
| 设备侧 profiling | 找到真实瓶颈 | 优化方向跑偏 |
| 端到端验证 | 检查用户收益 | 内核变快但应用没变快 |

再看一个真实工程例子。手机端 OCR 往往不是一个单模型问题，而是一条流水线：
1. 相机或相册解码图片
2. resize、归一化、旋转校正
3. 文本检测模型
4. 文本识别模型
5. 解码和版面拼接

如果你只把识别模型从 FP32 量化到 INT8，识别阶段从 30ms 降到 15ms，但前后处理仍然要 60ms，总时间只是从 90ms 降到 75ms。收益只有 16.7%，远小于“模型层面 2 倍加速”的直觉。

所以核心推导不是“INT8 更小所以更快”这么简单，而是：

$$
\Delta T_{total} = \Delta T_{compute} + \Delta T_{memory} + \Delta T_{copy} + \Delta T_{pre/post}
$$

只有当负收益项没有被别的项抵消时，用户才能真正感知到快。

## 代码实现

工程上更稳的顺序是：先验证覆盖率，再量化，再测设备端端到端时延，而不是一上来就盲目压低精度。

下面给一个最小可运行的 Python 玩具例子，演示线性量化、反量化和误差检查。这个例子不是完整框架代码，但它把最关键的数值机制讲清楚了。

```python
import math

def calc_scale_zero_point(x_min, x_max, qmin=-128, qmax=127):
    assert x_min < x_max
    scale = (x_max - x_min) / (qmax - qmin)
    zero_point = round(qmin - x_min / scale)
    zero_point = max(qmin, min(qmax, zero_point))
    return scale, zero_point

def quantize(values, scale, zero_point, qmin=-128, qmax=127):
    qs = []
    for x in values:
        q = round(x / scale) + zero_point
        q = max(qmin, min(qmax, q))
        qs.append(int(q))
    return qs

def dequantize(qvalues, scale, zero_point):
    return [scale * (q - zero_point) for q in qvalues]

xs = [-1.0, -0.4, 0.0, 0.2, 0.9]
scale, zp = calc_scale_zero_point(min(xs), max(xs))
qs = quantize(xs, scale, zp)
x_hat = dequantize(qs, scale, zp)

assert len(qs) == len(xs)
assert all(-128 <= q <= 127 for q in qs)
assert abs(x_hat[2] - 0.0) < 0.02
assert max(abs(a - b) for a, b in zip(xs, x_hat)) < 0.02

params = 10_000_000
fp32_mb = params * 4 / 1_000_000
int8_mb = params * 1 / 1_000_000
assert fp32_mb == 40.0
assert int8_mb == 10.0

print("scale=", scale, "zero_point=", zp)
print("quantized=", qs)
print("dequantized=", [round(v, 4) for v in x_hat])
```

如果进入真实框架，流程通常长这样：

```python
# 伪代码：展示工程顺序
converter = ...
converter.optimizations = [...]
converter.representative_dataset = representative_dataset
converter.target_spec.supported_ops = [...]
quant_model = converter.convert()

# 然后做三类验证：
# 1. 精度回归
# 2. 算子覆盖率检查
# 3. 真机 profiling
```

这里有两个关键词必须理解：

- `representative dataset`：代表性数据集。白话说，就是“拿一小批真实输入，告诉量化工具线上数据大概长什么样”。它主要用于估计激活范围。
- `delegate`：委托执行后端。白话说，就是“把某些算子子图交给 GPU、NPU、Core ML 或其他硬件去跑”的机制。

一个相对稳妥的实现 checklist 如下：

| 检查项 | 目的 |
|---|---|
| 是否拿真实样本做校准 | 降低激活量化误差 |
| 是否检查主要算子支持率 | 避免关键层 fallback |
| 是否确认输入输出仍为目标格式 | 避免多余拷贝和转换 |
| 是否在真实设备上测 | 桌面端结果不能替代手机 |
| 是否测端到端链路 | 防止只优化内核不优化业务 |

真实工程例子：你在 Android 端落一个文档扫描 OCR，正确姿势通常不是“直接量化模型就上线”，而是：
1. 先导出浮点模型，在目标手机上跑基线
2. 统计检测模型和识别模型各自耗时
3. 看前处理是否占比过高，比如图像 resize 或透视变换
4. 对主模型做 INT8 PTQ
5. 检查 delegate 覆盖率，确认主要卷积和矩阵乘都走加速器
6. 做精度回归，特别看弱光、模糊、倾斜样本
7. 再决定是否需要 QAT 或结构改写

这个顺序的核心价值是：先定位，再压缩。否则你很容易优化错对象。

## 工程权衡与常见坑

边缘部署里的最大误区，是把“模型数值优化”误当成“应用性能优化”。前者只是后者的一部分。

常见坑可以直接列成表：

| 坑 | 后果 | 规避方式 |
|---|---|---|
| 校准集不代表真实分布 | 精度掉点明显 | 用真实 representative dataset |
| 算子不支持低精度 | fallback 到 CPU | 先查覆盖率，再改结构 |
| 只看模型内核耗时 | 应用端感知不明显 | 做全链路 profiling |
| 输入输出频繁转格式 | 拷贝成本高 | 尽量复用设备侧 buffer |
| 激进低比特量化 | 精度崩掉 | 先从 INT8 开始 |
| 忽略热设计 | 长时间运行后降频 | 做持续压测，不只看冷启动 |

这里展开两个最常见的问题。

第一，校准集质量不够。  
后训练量化依赖样本估计激活分布。如果你拿的是“干净、居中、亮度稳定”的测试图去校准，但线上真实图片包含逆光、抖动、截断文本，那么量化范围会估错，导致某些层被截断得很严重。白话说，就是“你按理想世界定刻度，结果真实世界超出刻度了”。

第二，端到端瓶颈不在模型。  
比如一个 OCR 应用里，模型只有 20ms，但图像解码 15ms、裁剪 10ms、文本后处理 18ms、UI 线程同步 8ms。此时就算模型减半到 10ms，总收益也有限。很多团队做了一个月量化，最后只省 10ms，根因不是量化没用，而是主要瓶颈本来就不在模型。

还有一个容易被忽略的坑：首帧时延和稳态时延不是一回事。  
首帧常常包含模型加载、delegate 初始化、缓存预热；稳态则更多受热降频和内存碎片影响。只测一次平均值，结论通常不可靠。

## 替代方案与适用边界

不是所有模型都适合直接做激进量化。任务越精度敏感，越要保守决策。

常见替代方案如下：

| 方案 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| PTQ（后训练量化） | 快速落地 | 成本低、改动小 | 可能掉精度 |
| QAT（量化感知训练） | 精度敏感任务 | 精度更稳 | 训练成本高 |
| 混合精度 | 部分层敏感 | 折中收益较好 | 图更复杂 |
| 剪枝 | 结构冗余明显 | 参数和计算一起降 | 需重新训练 |
| 蒸馏 | 需要更小学生模型 | 性能和精度可兼顾 | 训练链路复杂 |
| 算子融合/结构改写 | 加速器覆盖率低 | 直接改善部署路径 | 需要懂模型图 |

可以用一个简单决策规则收束：

1. 如果目标设备支持完整 INT8，且 PTQ 精度可接受，优先 PTQ。
2. 如果 PTQ 掉点明显，但模型业务价值高，转 QAT。
3. 如果主要问题是算子不支持，先改结构，不要先硬上更低 bit。
4. 如果瓶颈在解码、后处理、UI 或数据搬运，先做全链路优化。
5. 如果模型本身仍太大，再考虑剪枝、蒸馏或更轻量架构。

一个典型边界例子是高精度检测任务。某些小目标检测模型在 INT8 后 mAP 明显下降，这时不要执着于“全量 INT8 必须成功”。更合理的做法是：
- 先看是否某些头部层或输出层更敏感
- 尝试混合精度
- 或者回到训练侧做 QAT
- 如果还是不行，就说明该任务的可压缩空间有限

换句话说，边缘模型性能优化不是一条固定配方，而是一组受设备、任务、运行时共同约束的工程决策。
