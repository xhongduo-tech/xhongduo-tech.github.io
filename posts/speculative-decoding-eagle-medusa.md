## 核心结论

投机解码的目标不是“让模型更会预测”，而是“让一次目标模型前向尽量确认更多 token”。Medusa、Eagle、树形验证分别解决三个不同问题：

1. Medusa 解决“草稿从哪里来”：在目标模型上额外加多个解码头，直接预测未来几个位置的候选 token，不再依赖独立小模型。
2. Eagle 解决“草稿依据什么预测”：直接复用目标模型中间层隐藏态。隐藏态就是模型内部对当前上下文的压缩表示，可以理解为“模型已经想出来但还没投影成词表概率的特征”。
3. 树形验证解决“怎么一次验证多条路径”：把多个候选序列组织成树，一次前向并行打分，而不是逐条回放。

三者叠加后的核心收益是：目标模型每做一次高成本前向，就可能确认多个连续 token，而不是只产出一个 token。在接受率可控时，工程上常见收益是 2 到 3 倍延迟下降，前提是分支数、树深、KV cache 和验证开销被控制在合理范围内。

一个新手能理解的玩具图景是：传统生成像排队买票，每个人都单独问一次；Medusa 和 Eagle 先让“助手”把后面几步草拟出来，树形验证再让“真正的出票员”一次核对多个人的订单。真正昂贵的是出票员的确认动作，所以只要一次能确认更多单，整体就会变快。

| 方案 | 每次目标模型前向确认多少 token | 是否需要独立草稿模型 | 并行验证能力 |
|---|---:|---|---|
| 串行 Greedy/Sampling | 1 | 否 | 无 |
| 传统 speculative decoding | 通常大于 1 | 是 | 中等 |
| Medusa + Eagle + 树形验证 | 通常大于 1，且可多路径 | 否 | 高 |

## 问题定义与边界

问题定义可以写得很精确：在尽量保持输出分布接近原始目标模型的条件下，减少生成同样长度文本所需的目标模型前向次数。

这里的“接近原始目标模型”很重要。因为部署优化不是训练新模型，而是在不明显改变答案质量、风格和稳定性的前提下，把推理变快。如果为了速度让分布偏差太大，本质上就是换了一个行为不同的模型。

它的边界也很明确：

| 维度 | 适用条件 | 不适用情况 |
|---|---|---|
| 模型类型 | 自回归语言模型 | 非自回归生成、完全黑盒 API |
| 结构访问 | 可读中间隐藏态，可加头或改推理图 | 完全不能改结构 |
| 推理系统 | 支持批量验证、支持 KV cache 复用 | 只能严格逐 token 调用 |
| 优化目标 | 延迟敏感、吞吐也重要 | 只关心最简实现 |

传统 Greedy 解码是每一步都算一次：
$$
x_t = \arg\max P(x_t \mid x_{<t})
$$

投机解码则拆成两个阶段：

1. 草稿阶段先提出未来若干 token 的候选。
2. 验证阶段由目标模型统一确认可接受的最长前缀。

所以它优化的是“前向次数”，不是简单优化单次矩阵乘法速度。对于大模型来说，前向本身很贵，尤其是长上下文下的注意力和内存访存更贵，因此减少前向轮数通常比微调几个 kernel 更有效。

一个真实工程边界例子是：如果你用的是可改造的 Llama/Vicuna，本地部署在 TensorRT-LLM 或 Triton 上，那么 Medusa/Eagle 有现实价值；如果你调用的是云厂商封闭 API，只能传 prompt 收 token，无法访问隐藏态，也无法构造树形验证图，那就不能用 Eagle，只能退回到“外部小模型草稿 + 黑盒验证”这类更弱方案。

## 核心机制与推导

先看 Medusa。它在目标模型顶层或接近输出层的位置加多个轻量草稿头。草稿头就是额外的预测分支，可以理解为“并行猜未来第 1、2、3 个位置分别可能是什么”。如果第 $i$ 个头保留 $k_i$ 个候选，那么理论上的路径数是：

$$
N_{\text{path}} = \prod_i k_i
$$

这就是它强的地方，也是风险来源。强在于一次能提出很多未来路径；风险在于分支数会笛卡尔爆炸。

再看 Eagle。它不再把草稿理解为“由另一个小模型生成”，而是直接用目标模型的中间特征来预测。设第 $L$ 层隐藏特征为 $f_t$，则草稿分布可写为：

$$
P_{\text{draft}}(x_t \mid f_t, x_{<t})
$$

这里的隐藏特征 $f_t$ 可以白话理解为：模型在第 $t$ 步已经抽象出的语义状态，只差最后几层映射成具体 token 概率。Eagle 的关键判断是，这些特征已经足够强，可以直接拿来做草稿预测，因此比“再跑一个独立小模型”更省重复计算。

然后是树形验证。树形验证不是新的语言模型，而是一种注意力组织方式。它把多个候选路径拼成一棵树，让共享前缀的节点共享计算。比如两层草稿头分别取 top-2 和 top-3，那么路径数是 $2\times3=6$。如果逐条验证，要把相同前缀重复算多次；树形验证会让前缀只算一次，再对子分支展开。

可以把结构画成简图：

```text
prefix x_{<t}
   |
  h1a
 / | \
a1 a2 a3
   |
  h1b
 / | \
b1 b2 b3
```

这里 `h1a`、`h1b` 是第一层候选，不同第一层候选下面再接第二层候选。共享前缀只保留一份 KV 表示，这就是树形验证的核心节省来源。

接受率常用一个近似关系解释：

$$
\alpha \approx A^d
$$

其中：

| 变量 | 含义 |
|---|---|
| $A$ | 单步草稿正确率，指草稿与目标模型可接受结果的一致程度 |
| $d$ | 草稿深度，指一次想向前猜多少步 |
| $\alpha$ | 整条路径或连续前缀的接受率 |

这个式子不是严格通用定理，而是工程上常用的独立近似：如果每一步都以概率 $A$ 对齐，那么连续 $d$ 步都对齐的概率大约是 $A^d$。它直接说明一个现实问题：树越深，接受率往往下降得很快。

玩具例子：若 $A=0.8$，则

- $d=1$ 时，$\alpha \approx 0.8$
- $d=2$ 时，$\alpha \approx 0.64$
- $d=4$ 时，$\alpha \approx 0.4096$

这说明“多猜几步”不是总有利。你虽然减少了前向轮数，但如果深度太大，频繁回退会把收益吃掉。

真实工程例子：在问答场景里，模型生成“根据合同条款，乙方应在收到通知后十个工作日内完成整改”。这类文本局部模式强，连续 token 可预测性高，草稿接受率往往较好；但在代码补全、数学推导或低频专有名词场景里，分支不确定性会迅速升高，$A$ 降低，树深收益就会变差。

## 代码实现

工程实现通常分成训练侧和推理侧。

训练侧，Medusa 常见做法是冻结主模型，仅训练新增草稿头。冻结主模型的意思是“不改原模型参数，只训练附加模块”，这样能保持原始模型行为稳定。Eagle 也是类似思路：主模型主体不动，只训练从隐藏态到草稿分布的轻量预测器。

推理侧，流程通常是：

1. 用 Medusa 头或 Eagle 头生成多个候选。
2. 按层展开成候选树。
3. 用树形 attention 把候选编码成一个批次。
4. 目标模型一次前向输出各节点 logits。
5. 从根往下找最长可接受前缀，提交这些 token。
6. 对未接受部分回退，进入下一轮。

下面给一个可运行的 Python 玩具实现。它不依赖深度学习框架，只模拟“草稿准确率、树深、接受 token 数”的关系，便于理解参数如何影响收益。

```python
import math
import random

def simulate_speculative(rounds=5000, A=0.8, depth=3, seed=0):
    random.seed(seed)
    accepted = 0
    forward_calls = 0

    for _ in range(rounds):
        forward_calls += 1
        step_accept = 0
        for _ in range(depth):
            if random.random() < A:
                step_accept += 1
            else:
                break
        accepted += max(1, step_accept)  # 至少产出 1 个 token，失败则回退成普通一步

    avg_tokens_per_forward = accepted / forward_calls
    return avg_tokens_per_forward

def approx_alpha(A, d):
    return A ** d

v1 = simulate_speculative(A=0.9, depth=2, seed=42)
v2 = simulate_speculative(A=0.6, depth=4, seed=42)

assert v1 > 1.0
assert v1 > v2
assert math.isclose(approx_alpha(0.8, 3), 0.512)

print(round(v1, 3), round(v2, 3), round(approx_alpha(0.8, 3), 3))
```

这个脚本说明三件事：

1. 接受率高时，一次前向平均能确认超过 1 个 token。
2. 深度不是越大越好，低准确率下深树反而吃亏。
3. $\alpha \approx A^d$ 能作为调参时的第一近似。

伪代码流程如下：

```python
def decode_step(prefix_tokens, hidden_feature):
    drafts = medusa_or_eagle_predict(prefix_tokens, hidden_feature, topk=[4, 2, 2])
    tree = build_candidate_tree(drafts)
    verify_logits = target_model_tree_verify(prefix_tokens, tree)
    accepted_path = select_longest_accepted_prefix(tree, verify_logits, accept_thresh=0.7)
    return accepted_path
```

推理参数一般这样调：

| 参数 | 作用 | 调优建议 |
|---|---|---|
| `top-k` | 每层保留多少候选 | 从小开始，先控分支数 |
| `tree depth d` | 一次草拟几步 | 通常先试 2 到 4 |
| `accept_thresh` | 接受阈值 | 结合质量回归和延迟一起调 |
| `temperature` | 草稿发散度 | 草稿侧宜偏低，减少无效分支 |
| `max_paths` | 树总路径上限 | 必须设硬限制，防止爆炸 |

一个真实工程例子是：在 Triton 或 TensorRT-LLM 推理脚本中，先对基础模型做 Medusa/Eagle 转换，再在服务配置中开启 speculative decoding。上线时通常还要加监控项，包括平均接受长度、回退比例、每轮验证节点数、P50/P95 延迟，以及显存峰值。如果这些指标不监控，系统可能表面“启用了投机解码”，实际却因为回退过多而比串行更慢。

## 工程权衡与常见坑

第一类坑是分支爆炸。若两个头各取 top-10，理论上第二层就有 $10 \times 10 = 100$ 条路径。树形验证虽然能共享前缀，但叶子节点过多时，验证图本身也会变得昂贵。此时不是“更并行”，而是“把验证阶段做成了新的瓶颈”。

第二类坑是特征质量不足。Eagle 的前提是隐藏态足够表达下一步信息。如果抽取层位不合适，或者轻量头训练不足，那么草稿分布会偏，$A$ 下降，导致：

$$
\alpha \approx A^d
$$

快速恶化。比如 $A=0.55, d=4$ 时，$\alpha \approx 0.0915$，意味着大多数深路径都要回退。

第三类坑是只看平均加速，不看尾延迟。在线服务真正关心的是 P95、P99。某些请求接受率极高，某些请求却频繁回退，最后平均值好看，但尾部请求很差，用户体验仍然不稳定。

第四类坑是忽略 KV cache 和显存。树形验证节省前向轮次，但会引入更多节点状态管理。若实现不正确，共享前缀没有真正复用 KV，或者 batch packing 不合理，显存和访存会反向拖慢速度。

| 常见坑 | 结果 | 规避方法 |
|---|---|---|
| 草稿候选过多 | 验证成本超过串行 | 限制 `top-k`、限制总路径数 |
| 树深过大 | 接受率迅速下降 | 从小深度开始，按业务分桶调 |
| Eagle 特征质量差 | 回退频繁 | 冻结主模型，单独微调轻量头 |
| 温度过高 | 无效分支增多 | 草稿侧使用更保守采样 |
| 只看平均吞吐 | 尾延迟恶化 | 同时监控 P95/P99 和回退率 |

新手版直观例子：如果你让草稿头一次吐出 `10 x 10` 组合，相当于同时递给老师 100 份作业要批。老师确实只来一次，但他一次要批太多份，最后总时间并不会更少。

实践上更稳的策略通常是“小树高接受率”优先，而不是“大树赌命中”。因为目标是稳定减少前向轮数，不是把候选空间堆到最大。

## 替代方案与适用边界

最常见替代方案是传统 speculative sampling：用一个独立小模型先生成草稿，再用大模型验证。它的优点是对目标模型侵入小，不需要改内部结构；缺点是无法复用目标模型隐藏特征，草稿模型与目标模型之间常有分布偏差。

另一类替代是 early exit 或 streaming 优化。early exit 指模型在中途层就提前输出结果，适合结构可训练且任务容忍近似误差的场景；streaming 更偏系统侧，把等待时间和传输时间压缩，但不直接减少目标模型前向次数。

| 方案 | 是否需要改目标模型 | 加速潜力 | 实现复杂度 | 适用边界 |
|---|---|---|---|---|
| Eagle + Medusa + 树形验证 | 高 | 高 | 高 | 可改模型与推理图 |
| 独立草稿模型 speculative sampling | 低到中 | 中 | 中 | 模型较黑盒但可验证 |
| 纯串行解码 | 否 | 低 | 低 | 最稳、最简单 |
| Early exit / 近似输出 | 高 | 中 | 高 | 可接受分布变化 |

如果目标模型是封闭系统，不能拿到隐藏态，也不能插入多头，那 Eagle 基本不可用。此时可行路线是外置小模型草稿，再把草稿送给大模型逐段验证。如果部署团队无法维护复杂推理图，或者业务文本高度不稳定、接受率长期偏低，那么继续用串行解码可能反而是更可靠的工程选择。

判断是否值得上这套方案，可以用三个问题：

1. 你的模型和服务框架能否访问隐藏态并支持树形验证？
2. 你的业务文本是否具有足够高的局部可预测性？
3. 你的团队是否能持续监控接受率、回退率和显存开销？

只要这三个问题里有两个回答是否定的，就不应该默认上最复杂的组合方案。
