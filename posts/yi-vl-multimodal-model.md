## 核心结论

Yi-VL 是 01.AI 基于 `Yi-6B/34B-Chat` 语言模型和 `CLIP ViT-H/14` 视觉编码器构建的双语多模态模型。多模态模型，指的是一个模型可以同时处理多种输入形式，例如图片和文本。Yi-VL 的目标不是简单把英文图文能力迁移到中文，而是让模型直接具备中文图文理解、中文图中文字识别和中文场景推理能力。

新手版本可以这样理解：模型先看图，再把图变成一串向量，最后像聊天一样回答问题。比如输入一张发票截图，并问“总金额是多少”，模型可能回答“总金额是 128.50 元”。

它的核心结构可以写成两步：

$$
h = P(V(I))
$$

$$
y = L([x; h])
$$

其中，$I$ 是图像，$V$ 是视觉编码器，负责把图像变成视觉特征；$P$ 是投影模块，负责把视觉特征变成语言模型能理解的表示；$x$ 是文本提示；$L$ 是 Yi 语言模型；$y$ 是最终回答。

| 模型 | 语言底座 | 视觉编码器 | CMMMU test | MMMU | 工程含义 |
|---|---:|---|---:|---:|---|
| Yi-VL-6B | Yi-6B-Chat | CLIP ViT-H/14 | 35.0 | 37.8 | 推理成本较低，适合轻量实验 |
| Yi-VL-34B | Yi-34B-Chat | CLIP ViT-H/14 | 36.5 | 41.6 | 效果更强，但成本明显更高 |

这个表说明一个直接结论：34B 版本比 6B 版本更强，但不是数量级飞跃。对于真实业务，是否选择 34B，不能只看“模型更大”，还要看延迟、显存、部署成本和错误容忍度。

## 问题定义与边界

Yi-VL 主要解决的是“单图 + 文本提示”的图文任务。单图，指一次输入一张图片；文本提示，指用户附带的问题或指令。典型任务包括图文问答、图片内容描述、图中文字识别、中文截图理解、票据问答和图文推理。

例如：

| 输入 | 问题 | Yi-VL 适配度 |
|---|---|---|
| 一张中文发票截图 | “这张发票的总金额是多少？” | 较适合 |
| 一张网页截图 | “按钮上写的是什么？” | 较适合 |
| 一张图表 | “销售额最高的月份是哪一个？” | 适合，但要注意图中文字和坐标轴 |
| 三张订单截图 | “哪两张是同一笔交易？” | 不适合单独使用 |
| 超高清合同扫描件 | “逐字提取全文” | 不应当只依赖 Yi-VL |

边界要说清楚。Yi-VL 不是通用视觉系统，也不是专门 OCR 引擎。OCR，指 Optical Character Recognition，即从图片中识别文字。Yi-VL 支持图中文字识别，但这不等于它在所有小字、密集表格、低清扫描件上都能达到专业 OCR 的稳定性。

公开材料也没有披露 Yi-VL 精确的中英文训练数据百分比。能确认的是，Yi 系列语言底座使用中英混合语料，Yi-VL 训练中加入双语图文数据。因此可以说它面向中英双语场景优化，但不应编造“中文占比 40%”之类的数字。

| 能做什么 | 不能做什么 | 需要额外工具补足什么 |
|---|---|---|
| 回答一张图里有什么 | 多图联合推理 | 多图检索、排序或专门多图模型 |
| 识别中文截图中的主要文字 | 稳定识别极小字号 | 裁剪、放大、OCR |
| 理解票据、报表、页面截图 | 保证结构化字段 100% 正确 | OCR + 字段规则 + 人工复核 |
| 根据图文做简单推理 | 长链路视觉推理 | 更强多模态模型或任务拆解 |

真实工程例子：一个报销系统要从发票截图里抽取金额、日期、购买方名称。Yi-VL 可以作为问答式抽取模块，例如问“发票总金额是多少”。但如果这个金额要直接进入财务系统，最好先用 OCR 识别文字，再让 LLM 做字段校验，而不是只相信一次多模态回答。

## 核心机制与推导

Yi-VL 的结构可以拆成三段：视觉编码器 `V`、投影模块 `P`、语言模型 `L`。

视觉编码器，指把图片转换成向量表示的模型。Yi-VL 使用 `CLIP ViT-H/14` 初始化视觉编码器。ViT 是 Vision Transformer 的缩写，它把图片切成小块，再像处理文本 token 一样处理图像块。投影模块，指把一种向量空间映射到另一种向量空间的小网络。Yi-VL 使用两层 MLP 做投影。MLP 是多层感知机，可以理解为多层全连接神经网络。语言模型 `L` 使用 `Yi-6B-Chat` 或 `Yi-34B-Chat` 初始化，负责根据图像表示和文本提示生成回答。

推理链路是：

1. 图片 $I$ 输入视觉编码器，得到视觉特征 $V(I)$。
2. 视觉特征输入投影模块，得到语言空间表示 $h = P(V(I))$。
3. 把文本提示 $x$ 和图像表示 $h$ 拼接，输入语言模型。
4. 语言模型生成答案 $y = L([x; h])$。

玩具例子：假设图片是一张只写着“合计 128.50 元”的小票。视觉编码器不直接输出“128.50”，而是输出一组向量。投影模块把这组向量转成 Yi 语言模型能接收的格式。用户问“总金额是多少”，语言模型结合问题和图像向量，输出“总金额是 128.50 元”。

训练也分阶段进行，不是一次性把所有参数混在一起训练。

| 阶段 | 训练对象 | 图像分辨率 | 目标 |
|---|---|---:|---|
| 阶段 1 | ViT + 投影模块，语言模型冻结 | 224×224 | 学会把视觉信息对齐到语言空间 |
| 阶段 2 | ViT + 投影模块，语言模型冻结 | 448×448 | 提升细节识别能力 |
| 阶段 3 | ViT + 投影模块 + 语言模型 | 448×448 | 做多模态对话微调 |

冻结，指训练时不更新某部分参数。前两阶段冻结语言模型，是为了先让视觉部分学会“把图像翻译成语言模型能懂的向量”。最后再解冻全模型，让视觉和语言一起适应多模态对话。

可以把训练过程直白理解为：先训练一个会“看图记笔记”的模块，再让它看更清楚的图，最后让整个系统学会按用户问题回答。

## 代码实现

工程上理解 Yi-VL，不需要从零训练一个 6B 或 34B 模型。重点是理解推理链路：图像预处理、视觉编码、特征投影、拼接文本、语言模型生成。

流程图可以写成：

```text
image
  -> resize / normalize
  -> vision_encoder: V(I)
  -> projector: P(V(I))
  -> concat with prompt: [x; h]
  -> language_model.generate
  -> answer
```

下面是一段可运行的 Python 玩具实现。它不是真实 Yi-VL 权重，只模拟“视觉编码器 + 投影模块 + 语言模型接口”的数据流。

```python
import numpy as np

def resize_to_448(image):
    # 玩具实现：真实工程会用 PIL / torchvision 做 resize 和 normalize
    return np.zeros((448, 448, 3), dtype=np.float32)

def vision_encoder(image):
    # 模拟 V(I)：把图片编码成 4 个视觉 token，每个 token 维度为 3
    assert image.shape == (448, 448, 3)
    return np.array([
        [0.8, 0.1, 0.0],
        [0.7, 0.2, 0.1],
        [0.1, 0.9, 0.2],
        [0.0, 0.8, 0.3],
    ], dtype=np.float32)

def projector(vision_feat):
    # 模拟 P(V(I))：投影到语言模型输入空间
    weight = np.array([
        [1.0, 0.0],
        [0.0, 1.0],
        [0.5, 0.5],
    ], dtype=np.float32)
    return vision_feat @ weight

def language_model_generate(prompt, image_tokens):
    # 玩具规则：如果视觉 token 的第一维均值较高，假设图片包含金额区域
    assert image_tokens.shape == (4, 2)
    if "总金额" in prompt and image_tokens[:, 0].mean() > 0.4:
        return "总金额是 128.50 元"
    return "无法确定"

raw_image = np.ones((900, 1200, 3), dtype=np.float32)
image = resize_to_448(raw_image)
vision_feat = vision_encoder(image)       # V(I)
h = projector(vision_feat)                # P(V(I))
answer = language_model_generate("这张图的总金额是多少？", h)

assert h.shape == (4, 2)
assert answer == "总金额是 128.50 元"
print(answer)
```

接口设计可以拆成下面这样：

| 模块 | 输入 | 输出 | 工程作用 |
|---|---|---|---|
| `image` | 原始图片路径或图片数组 | 像素矩阵 | 提供视觉输入 |
| `prompt` | 用户问题 | 文本 token | 指定任务目标 |
| `vision_encoder` | 预处理后的图片 | 视觉特征 | 提取图像信息 |
| `projector` | 视觉特征 | 语言空间 token | 对齐视觉和语言 |
| `language_model` | 文本 token + 图像 token | 文本答案 | 生成自然语言回答 |

真实代码通常不会手写这些模块，而是调用官方仓库或 Hugging Face 模型加载逻辑。但无论接口如何封装，核心都是同一条链路：$I \rightarrow V(I) \rightarrow P(V(I)) \rightarrow L([x; h])$。

## 工程权衡与常见坑

Yi-VL 的一个关键工程限制是输入会被处理到 `448×448`。这比很多传统图像分类模型更细，但对票据、合同、报表、网页截图里的小字仍然不够。原图如果是 3000×2000 的发票扫描件，缩到 448×448 后，小字号、表格线、角落备注都可能丢失。

新手版本可以直接记住：如果原图是一张压缩后的报销单，先裁剪金额区域再问模型，比直接整图输入更稳。

| 问题现象 | 原因 | 规避方式 |
|---|---|---|
| 把 `8` 识别成 `3` | 小字在缩放后模糊 | 裁剪金额区域，提升局部分辨率 |
| 把两行金额混在一起 | 表格密集，目标不明确 | 提问时指定字段和区域 |
| 回答图里不存在的内容 | 多模态模型存在幻觉 | 和 OCR 结果或人工规则交叉校验 |
| 多张图之间判断错误 | Yi-VL 主要支持单图输入 | 改用多图模型或先拆成单图结果再汇总 |
| 误以为中文 OCR 极强 | 支持中文识别不等于专门 OCR 高分 | 对关键字段使用 OCR 工具链 |
| 高分辨率原图没有带来收益 | 推理时仍会调整到 448×448 | 对关键区域分块输入 |

真实工程例子：客服系统收到用户上传的支付截图，要判断金额是否为 128.50 元。直接把整张聊天截图丢给模型，模型可能看到多个数字：时间、订单号、余额、金额。更稳的做法是先定位支付金额区域，再把局部截图交给模型，或者使用 OCR 提取候选数字，再让模型结合上下文判断哪个是付款金额。

另一个常见坑是把 benchmark 分数当成业务准确率。MMMU 和 CMMMU 衡量的是多学科多模态理解能力，不等于发票金额抽取准确率。一个模型在 CMMMU 上表现好，只说明它在中文多模态问答和推理上有竞争力，不说明它可以替代财务 OCR 系统。

34B 版本也不是无条件更好。它通常需要更高显存、更长延迟和更复杂部署。官方模型卡给出的推理硬件建议里，6B 可以在 RTX 3090、RTX 4090、A10、A30 这类单卡上尝试；34B 则需要更高资源，例如多张 RTX 4090 或 A800 级别资源。对于在线业务，延迟和吞吐可能比几分 benchmark 提升更重要。

## 替代方案与适用边界

选择 Yi-VL 前，要先判断任务是“聊天式图文理解”，还是“高精度结构化抽取”。

发票抽取优先走 `OCR → 结构化字段 → LLM 校验`。聊天式图文问答才更适合 Yi-VL。例如“这张中文截图表达了什么”“这个界面报错原因可能是什么”“图里总金额是多少”，这类任务适合 Yi-VL。相反，如果任务是把 1 万张发票批量抽成财务字段，纯多模态问答不是最稳方案。

| 方案 | 优势 | 短板 | 适用任务 |
|---|---|---|---|
| Yi-VL | 中文图文问答能力强，接口直观 | 单图限制，可能幻觉，小字不稳 | 截图理解、票据问答、图文对话 |
| OCR 工具链 + LLM | 字段抽取更可控，便于校验 | 工程链路更长，需要规则和版式处理 | 发票、合同、表格、证件抽取 |
| 更强通用多模态模型 | 推理能力和泛化能力可能更强 | 成本更高，私有化部署更难 | 多图推理、复杂视觉问答、长链路分析 |
| 专用视觉模型 | 在单一任务上稳定 | 迁移性弱，需要标注和训练 | 票据检测、表格结构识别、目标检测 |

适用边界可以用两个例子区分。

只看一张中文截图，问“这里的错误提示是什么意思”，Yi-VL 合适。这个任务需要理解图片里的文字和上下文，不一定要求逐字结构化输出。

要比较三张截图，判断哪个版本的页面转化路径更短，就不应该硬用单图模型。更合理的做法是分别分析每张图，再用文本模型汇总；或者直接使用支持多图输入的多模态模型。

工程选型的核心不是“哪个模型最新”，而是“错误能不能被后处理发现”。如果金额、身份证号、合同条款这类字段错了会造成业务损失，就应该引入 OCR、规则校验、置信度、人审或多模型交叉验证。Yi-VL 更适合作为理解和辅助判断模块，而不是所有视觉任务的唯一真相来源。
