## 核心结论

微调数据配比，是指在多任务微调中，给每个任务分配多少采样概率、训练 step 或 token 预算。它本质上不是“平均分数据”，而是在有限训练预算下，决定每个任务拿到多少梯度更新机会。

梯度更新机会，白话说就是模型参数被某类数据纠正的次数。同样 10 个训练 step，如果安全拒答只被抽到 1 次，它学得就慢；如果安全拒答被抽到 5 次，客服主任务能学到的机会就变少。配比的作用就是先决定“谁多学，谁少学”。

多任务微调的数据配比没有固定最优解。它要在三个目标之间取舍：

| 目标 | 含义 | 典型风险 |
|---|---|---|
| 大任务能力 | 保住样本最多、业务最常见任务的效果 | 小任务被淹没 |
| 小任务覆盖 | 让低频但重要任务也有训练机会 | 小任务重复过多 |
| 训练稳定性 | 避免某类任务主导梯度方向 | 需要更多监控和调参 |

常见策略可以先归纳成三类：

| 策略 | 采样倾向 | 优点 | 风险 |
|---|---|---|---|
| 按数据量比例 | 大任务多、小任务少 | 简单、稳定 | 小任务可能学不到 |
| 任务均衡 | 每个任务一样多 | 小任务存在感强 | 容易重复过拟合小任务 |
| 反比/强化小任务 | 小任务更多 | 小任务提升快 | 大任务容易退化 |

FLAN 和 T0 这类多任务指令微调实验给出的经验很直接：任务多样性通常比单纯堆某一个任务的数据量更重要。但这不等于所有任务都应该平均采样，而是说明在有限预算下，要主动给不同任务分配学习机会。

## 问题定义与边界

这里讨论的“数据配比”，指多任务微调中，各任务的采样概率、训练步数或 token 预算分配，不是单任务内部的样本随机打乱。

多任务微调，白话说就是多个任务共享同一个模型参数一起训练。例如客服模型同时学习“意图分类”“槽位抽取”“安全拒答”。这三类数据的输入输出格式、难度、业务优先级都不同，不能简单把所有样本混在一起，然后默认模型会自动平衡。

需要区分三个概念：

| 概念 | 白话解释 | 例子 |
|---|---|---|
| 任务级配比 | 决定每类任务被抽中的概率 | 意图分类 60%，槽位抽取 25%，安全拒答 15% |
| 样本级采样 | 在某个任务内部抽哪一条样本 | 已经选中安全拒答后，再抽一条拒答样本 |
| token budget 归一化 | 按实际 token 消耗分配资源 | 长文档任务一条样本可能抵得上几十条短问答 |

本文的边界如下：

| 情况 | 是否属于本文讨论范围 |
|---|---|
| 多任务联合微调 | 是 |
| 单任务随机打乱 | 否 |
| 仅调 batch size | 否 |
| 任务间采样概率调整 | 是 |
| 只做 loss 权重调整但不改采样 | 部分相关，但应单独说明 |

边界很重要。只调整 batch size，不一定改变任务之间的相对训练机会；只做 loss weighting，也就是给不同任务的损失乘不同系数，改变的是梯度强度，不一定解决“小任务根本抽不到”的覆盖问题。

## 核心机制与推导

训练 step 是有限资源。每个 step 只能处理一个 batch，或者一个混合 batch。某个任务被采样得越多，它对模型参数的影响就越大。因此，采样概率就是训练资源分配。

先看一个玩具例子。任务 A 有 1000 条数据，任务 B 有 100 条数据。如果按数据量采样，A 和 B 的比例是 1000:100，A 大约被抽到 90.9%，B 只有 9.1%。如果按任务均衡采样，A 和 B 各占 50%。你改的不是数据本身，而是模型“看到谁”的次数。

一个统一写法是：

$$p_i = \frac{n_i^\alpha d_i^\beta}{\sum_j n_j^\alpha d_j^\beta}$$

其中：

| 符号 | 含义 | 直觉 |
|---|---|---|
| `p_i` | 任务 `i` 的采样概率 | 这个任务被抽中的频率 |
| `n_i` | 任务数据量 | 数据越多，天然更容易被看见 |
| `d_i` | 难度权重 | 越难的任务越值得多给资源 |
| `α` | 数据量偏置系数 | 控制大任务是否占优 |
| `β` | 难度偏置系数 | 控制难任务是否被强化 |

`α=1` 时接近按数据量比例采样；`α=0` 时所有任务不看数据量，接近任务均衡采样；`α<0` 时会强化小任务。`β` 用来引入难度权重。难度权重可以来自 EMA loss、验证集错误率或 hard example 比例。EMA 是指数移动平均，白话说就是用近期多次指标的平滑结果，避免被某一次异常 loss 带偏。

对应的训练目标可以写成：

$$L = \sum_i p_i \mathbb{E}_{(x,y)\sim D_i}[\ell_i(f(x), y)]$$

这里 `L` 是总损失，表示训练希望最小化的目标；`D_i` 是任务 `i` 的数据分布；`ℓ_i` 是任务 `i` 的损失函数；`f(x)` 是模型对输入 `x` 的输出。这个式子的直觉是：总目标不是简单把所有任务相加，而是按 `p_i` 加权后相加。采样概率越高的任务，对总训练目标的影响越大。

真实工程例子可以看客服多任务微调：意图分类 50 万条、槽位抽取 5 万条、安全拒答 5 千条。如果按数据量比例采样，安全拒答大约只占 0.9%。如果训练 10 万步，安全拒答只得到约 900 步，通常不够。工程上常见做法是把 `α` 设成 0.3 到 0.7 之间，用温度采样压平头部任务，再给安全任务一个额外难度权重，同时监控意图分类是否回退。

## 代码实现

实现上通常分成两步：先算每个任务的采样权重，再在训练循环中按权重抽任务或构造混合 batch。你不是直接把 100 万条数据重新排序，而是先给每个任务分一个“抽签概率”，每次训练时按这个概率决定下一步学哪个任务。

| 步骤 | 输入 | 输出 |
|---|---|---|
| 统计任务规模 | 各任务样本数 | `n_i` |
| 估计任务难度 | loss / val / hard ratio | `d_i` |
| 计算采样概率 | `n_i, d_i, α, β` | `p_i` |
| 进入训练循环 | `p_i` | 任务采样结果 |
| 周期更新 | 新的 loss / 指标 | 新的 `p_i` |

下面是一个最小可运行例子：

```python
import random
from collections import Counter

def compute_task_probs(tasks, alpha=0.5, beta=0.0):
    scores = []
    for t in tasks:
        score = (t["num_samples"] ** alpha) * (t["difficulty"] ** beta)
        scores.append(score)

    total = sum(scores)
    return [s / total for s in scores]

def sample_task(tasks, probs):
    return random.choices(tasks, weights=probs, k=1)[0]

tasks = [
    {"name": "intent", "num_samples": 500_000, "difficulty": 1.0},
    {"name": "slot", "num_samples": 50_000, "difficulty": 1.2},
    {"name": "safety", "num_samples": 5_000, "difficulty": 2.0},
]

probs = compute_task_probs(tasks, alpha=0.5, beta=1.0)

assert abs(sum(probs) - 1.0) < 1e-9
assert probs[0] > probs[1] > probs[2]  # 数据量仍然占主导

random.seed(7)
draws = Counter(sample_task(tasks, probs)["name"] for _ in range(10_000))

assert draws["intent"] > draws["slot"]
assert draws["slot"] > draws["safety"]

print(probs)
print(draws)
```

这个例子里，`alpha=0.5` 表示不完全按数据量比例采样，而是对大任务做压平；`beta=1.0` 表示难度权重直接参与采样分数。安全任务的数据少，但因为难度权重高，它不会像纯数据量采样那样被完全淹没。

更完整的训练流程通常是：固定每个 epoch 或每 N 个 step 重新估计一次权重；训练时按权重做 weighted sampling；额外记录每个任务的有效 token 数、采样次数、训练 loss 和验证集指标。有效 token 数，白话说就是模型实际处理了多少文本单位，比样本条数更接近真实训练成本。

## 工程权衡与常见坑

纯均衡采样会放大小任务噪声，纯按数据量采样会让小任务几乎没有训练机会。真正的问题不是“多还是少”，而是“多到什么程度刚好”。

例如安全拒答只有 5000 条，如果每个 step 都抽它，它会很快记住训练集里的固定说法，生成结果变窄；如果完全不抽它，它又学不会拒答边界。合理做法通常是设置采样上限、按阶段调整权重，并用独立验证集观察它是否真的提升。

| 常见坑 | 现象 | 规避方式 |
|---|---|---|
| 纯均衡采样 | 小任务重复过多 | 设采样上限、混合温度采样 |
| 纯反比采样 | 大任务能力回退 | 加下限、分阶段退火 |
| 直接用训练 loss 当难度 | 噪声任务被误判为困难 | 用 EMA 或验证集指标 |
| 只按样本数不按 token 数 | 长文本任务被低估 | 改成 token budget 归一化 |
| 权重长期不更新 | 训练后期失衡 | 周期性重算权重 |

样本数和 token 数不能混为一谈。100 条短问答，不等于 100 条长文档。一个摘要任务的单条样本可能有 3000 token，一个意图分类样本可能只有 50 token。如果只看样本条数，长文本任务会被低估，实际训练时却消耗大量算力。更稳妥的做法是用 token budget 做归一化，例如把 `n_i` 替换成任务总 token 数，或者限制每个任务每轮最多消耗多少 token。

监控指标至少包括：

| 监控项 | 看什么 |
|---|---|
| 各任务验证集指标 | 是否有任务回退 |
| 训练 loss 曲线 | 是否某任务被噪声主导 |
| 任务采样次数 | 是否极端倾斜 |
| token 占比 | 是否和预期一致 |

还有一个常见误区：把“难学”和“数据脏”混在一起。训练 loss 高可能说明任务确实难，也可能说明标注不一致、输入输出格式混乱、答案本身有冲突。盲目提高这类任务权重，会让模型反复学习噪声，反而拖累整体效果。

## 替代方案与适用边界

数据配比不是唯一手段。很多场景还可以用 loss weighting、curriculum learning、分阶段训练、adapter 路由等方式补充。选择哪种方案，取决于任务之间是否冲突、数据是否稀缺、是否要求稳定继承大任务能力。

| 方案 | 适合场景 | 优点 | 缺点 |
|---|---|---|---|
| 数据配比 | 多任务共享训练 | 简单直接 | 需要调参 |
| Loss weighting | 采样难改、但希望改目标 | 实现方便 | 不一定解决覆盖问题 |
| 分阶段训练 | 任务优先级很明确 | 可控性强 | 可能产生遗忘 |
| Curriculum learning | 任务有明显难度梯度 | 收敛平稳 | 设计复杂 |
| Token 级归一化 | 长短样本差异大 | 更公平 | 统计成本高 |

Loss weighting 是给不同任务的 loss 乘权重。例如安全任务仍然只抽到 1% 的 batch，但每次出现时 loss 乘以 5。它能放大梯度强度，但不能解决覆盖不足：如果任务太少被抽中，模型仍然看不到足够多样的样本。

分阶段训练适合优先级很明确的场景。例如客服模型可以先学意图分类和槽位抽取，再单独加一阶段强化安全拒答。这比一开始把所有任务平均混在一起更可控，但也可能造成灾难性遗忘。灾难性遗忘，白话说就是模型在学新任务时，把旧任务能力冲掉了。

适用边界需要明确：

| 场景 | 配比是否足够 |
|---|---|
| 任务只是数据量不同，目标不冲突 | 通常足够 |
| 小任务很重要但样本少 | 需要配比加权和验证监控 |
| 任务强冲突 | 单纯调配比往往不够 |
| 任务高度相似 | 过度精细配比收益有限 |
| 小任务噪声很大 | 盲目加权会适得其反 |

当任务强冲突时，例如“尽量回答用户问题”和“严格拒绝高风险请求”在边界样本上冲突，单纯调采样概率可能不够，需要更清晰的数据标注规则、偏好优化、拒答策略或模型路由。当任务高度相似时，过度设计复杂配比也可能没有明显收益，简单温度采样已经足够。

最终判断标准不是某个公式是否漂亮，而是各任务验证集是否达到业务目标，并且关键能力没有明显回退。
