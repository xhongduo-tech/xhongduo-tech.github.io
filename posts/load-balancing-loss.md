## 核心结论

Load balancing loss 是 Mixture of Experts，简称 MoE，即“很多专家网络里只激活少数几个”的模型结构，在训练时常加的辅助损失。它不直接优化下一个词预测或分类精度，而是约束路由器，不要把大多数 token 都送到少数几个专家。

它的目标很明确：主任务 loss 负责“做对事”，load balancing loss 负责“别让计算资源只被少数专家占满”。没有这项约束时，路由器很容易反复偏爱几个早期表现稍好的专家，形成专家崩塌，也就是“少数专家长期过载，多数专家几乎没有训练信号”。

结论可以压缩成三点：

1. 它本质上是一个分布约束，鼓励“实际接收 token 的比例”和“路由器给出的平均概率”更接近均匀。
2. 它只能是辅助项，不能压过主任务。实践里通常从 $\alpha \approx 0.01$ 起步，再结合专家利用率曲线调。
3. 它不是越强越好。过强会把系统推向“形式上的均衡”，却牺牲主任务，甚至导致 capacity，也就是“单个专家可接收 token 的上限”，频繁溢出。

篮球赛的比喻只能帮助建立直觉：路由器像裁判，主任务是赢球，辅助 loss 像裁判吹哨，防止某几个人包办所有投篮。吹得太少，替补永远摸不到球；吹得太多，战术被打乱。

| 设置 | 专家利用率 | 主任务 loss 变化 | 典型现象 |
|---|---:|---|---|
| 没有辅助 loss | 很不均匀 | 早期可能下降快，后期不稳 | 少数专家过载，长尾专家闲置 |
| 辅助 loss 太弱 | 略有改善 | 基本不变 | 崩塌减轻但仍明显 |
| 辅助 loss 适中 | 较均匀 | 通常最稳 | 专家都得到训练，容量利用合理 |
| 辅助 loss 过强 | 表面均匀 | 明显变差 | 为均衡而均衡，路由质量下降 |

---

## 问题定义与边界

先定义问题。token 是“模型一次处理的最小离散单元”，可以理解为词片或子词。专家崩塌指的是：在一个 batch 内，绝大多数 token 被送到极少数专家，其他专家几乎不工作。

玩具例子最容易看清楚。假设有 8 个 token、3 个专家，最终分配结果是 $4,3,1$。这说明专家 1 和专家 2 很忙，专家 3 很闲。即使主任务还能收敛，这种状态仍然有两个问题：

1. 已经付出的模型参数成本没有被充分利用。
2. 闲置专家几乎拿不到梯度，后面更难追上，崩塌会自我强化。

这里要区分两个量：

| 量 | 含义 | 是否可导 | 作用 |
|---|---|---|---|
| 实际负载 $f_i$ | 专家 $i$ 实际接收 token 的比例 | 通常不传梯度 | 反映结果 |
| 平均路由概率 $P_i$ | 路由器对专家 $i$ 的平均偏好 | 传梯度 | 决定优化方向 |

边界也要说清楚。load balancing loss 解决的是“分配过于偏斜”，不是解决所有路由问题。下面这些因素都会影响它的有效性：

| 参数 | 影响方向 | 典型风险 |
|---|---|---|
| $\alpha$ 辅助损失系数 | 决定约束强弱 | 太小无效，太大伤主任务 |
| top-k 路由 | 每个 token 激活几个专家 | k 越大，路由更平滑但算力更高 |
| 专家数 $E$ | 增加容量 | 专家变多后更容易负载不均 |
| capacity factor | 专家容量冗余比例 | 太小丢 token，太大浪费显存 |
| 路由温度 | 控制概率尖锐度 | 太低更易集中到少数专家 |

所以它的适用边界是：当系统确实存在非均匀激活，并且你希望多数专家都获得有效训练时，它很有价值；如果路由已经足够平滑，或者你主要瓶颈在通信、显存和溢出，不应只靠它硬调。

---

## 核心机制与推导

核心公式常写成：

$$
L_{\text{balance}} = E \sum_{i=1}^{E} f_i P_i
$$

其中：

- $E$ 是专家数。
- $f_i$ 是专家 $i$ 实际接收 token 的比例。
- $P_i$ 是专家 $i$ 的平均路由概率。

“归一化”就是把式子乘上 $E$，使完全均衡时 loss 恰好等于 1。因为如果每个专家都均匀分到 $\frac{1}{E}$，那么：

$$
L_{\text{balance}} = E \sum_{i=1}^{E} \frac{1}{E}\cdot\frac{1}{E} = E \cdot E \cdot \frac{1}{E^2}=1
$$

这带来一个很实用的解释：  
当 $L_{\text{balance}}$ 接近 1，说明负载比较均衡；显著大于 1，通常说明概率和实际负载在头部专家上共同集中。

看一个 step-by-step 的玩具例子。3 个专家，实际比例和平均概率分别为：

- $f=\{0.5, 0.3, 0.2\}$
- $P=\{0.4, 0.3, 0.3\}$

| 专家 | $f_i$ | $P_i$ | $f_iP_i$ |
|---|---:|---:|---:|
| 1 | 0.5 | 0.4 | 0.20 |
| 2 | 0.3 | 0.3 | 0.09 |
| 3 | 0.2 | 0.3 | 0.06 |

于是：

$$
L_{\text{balance}} = 3 \times (0.20 + 0.09 + 0.06) = 1.05
$$

如果把归一化系数省略，只看 $\sum_i f_iP_i$，结果是 $0.35$。两种写法都能见到，但工程里常保留归一化，因为更好解释和监控。

为什么这个式子有效？因为当一个专家同时满足“实际收到很多 token”和“路由器本来也很偏爱它”时，$f_iP_i$ 会变大。多个头部专家把和式抬高后，反向传播就会推动 $P_i$ 下降，让路由概率别那么集中。注意，优化对象是 $P_i$，不是硬选中的离散结果本身。

## 代码实现

标准实现里，$f_i$ 只是统计量，通常要 `detach`。白话解释就是：实际分配结果只拿来“记账”，不要让它把离散 top-k 选择中的噪声直接反传回去。真正接收梯度的是 $P_i$，也就是连续的路由概率。

```python
import numpy as np

def load_balance_loss(router_probs, selected_experts):
    """
    router_probs: [B, E], 每个 token 对每个专家的概率
    selected_experts: [B], top-1 选中的专家索引
    """
    B, E = router_probs.shape

    # f: 实际负载比例，作为统计量，不参与梯度
    counts = np.bincount(selected_experts, minlength=E).astype(np.float64)
    f = counts / B

    # P: 平均路由概率，真实训练里这一项会接收梯度
    p = router_probs.mean(axis=0)

    loss = E * np.sum(f * p)
    return loss, f, p

# 玩具例子
router_probs = np.array([
    [0.70, 0.20, 0.10],
    [0.60, 0.30, 0.10],
    [0.55, 0.25, 0.20],
    [0.52, 0.28, 0.20],
    [0.20, 0.60, 0.20],
    [0.25, 0.55, 0.20],
    [0.15, 0.25, 0.60],
    [0.18, 0.20, 0.62],
])

selected = np.argmax(router_probs, axis=1)
loss, f, p = load_balance_loss(router_probs, selected)

assert selected.tolist() == [0, 0, 0, 0, 1, 1, 2, 2]
assert np.allclose(f, np.array([0.5, 0.25, 0.25]))
assert loss >= 1.0
print(round(loss, 4), f, p.round(4))
```

在真实训练代码里，总损失通常写成：

$$
L_{\text{total}} = L_{\text{task}} + \alpha \cdot L_{\text{balance}}
$$

伪代码如下：

```python
router_probs = router(tokens)                 # [B, E]
selected_experts = topk(router_probs, k=1)    # 离散选择
f = compute_token_fraction(selected_experts).detach()
p = router_probs.mean(dim=0)
balance_loss = num_experts * (f * p).sum()
total_loss = task_loss + alpha * balance_loss
```

真实工程例子可以参考大规模语言模型中的 top-2 MoE。比如 8 个专家的层，如果训练日志显示两个专家长期占到 55% 以上 token，而其余六个专家平分剩余负载，那么即使验证集 perplexity 还在下降，也说明容量没有被有效使用。这时一般不是先改网络结构，而是先检查 $\alpha$、capacity factor 和路由温度。

## 工程权衡与常见坑

第一类坑是 $\alpha$ 的量级。它太小，系统看起来“加了辅助 loss”，但专家利用率曲线几乎没变化；它太大，主任务 loss 会抬升，因为模型被迫把 token 分给并不适合的专家。

第二类坑是只看平均，不看溢出。某些训练日志里专家利用率很漂亮，但 dropped tokens，也就是“因为专家容量已满而被丢弃或重路由的 token”，其实在上升。这通常意味着你把均衡做出来了，却把容量管理做坏了。

第三类坑是让 $f_i$ 传梯度。因为 top-k 选择本身是离散的，直接通过它反传，梯度信号会很噪，训练不稳定。更常见也更稳的做法，是只让 $P_i$ 这条连续路径更新。

第四类坑是忽略训练阶段差异。训练早期路由器尚未形成稳定偏好，辅助 loss 应该“缓慢介入”；训练中后期，如果专家已经均衡，再继续强化它，收益会变小，副作用会变大。

| $\alpha$ | capacity factor | 专家利用率 | 主 loss 趋势 | 辅 loss 趋势 | 典型判断 |
|---:|---:|---|---|---|---|
| 0 | 1.0 | 很低且偏斜 | 早降后抖动 | 无 | 高风险崩塌 |
| 0.005 | 1.0 | 略改善 | 基本稳定 | 轻微下降 | 可作为 warm-up |
| 0.01 | 1.1-1.25 | 明显改善 | 通常最稳 | 稳定接近 1 | 常见起点 |
| 0.05 | 1.0 | 很均匀 | 往往变差 | 很低或过快下降 | 过度均衡风险 |

一个可执行的监控策略是：每 5k 步记录一次专家利用率、负载熵、主 loss、辅 loss、丢 token 比例。如果利用率仍低于 60%，而主 loss 没明显恶化，可以小幅上调 $\alpha$；如果主 loss 明显变差或 dropped tokens 上升，就优先调 capacity factor 或温度，而不是继续加大辅助损失。

## 替代方案与适用边界

load balancing loss 不是唯一办法。它更像一个“保险装置”，在路由天然容易偏斜时提供稳定器。

| 方法 | 适用场景 | 优点 | 风险 |
|---|---|---|---|
| 辅助 loss | 专家明显失衡 | 直接约束负载分布 | 系数难调，可能伤主任务 |
| capacity-only | 主要问题是溢出 | 实现简单，直接限制上限 | 不能主动改善路由偏好 |
| temperature scheduling | 路由概率过尖锐 | 可平滑早期分布 | 温度过高会损伤专家选择性 |
| top-k 调整 | 稀疏性与稳定性冲突 | 直接影响路由结构 | 计算成本变化明显 |

如果系统已经用了温度调度，让 router 概率在 warm-up 阶段更平缓，那么辅助 loss 可以弱一些，比如 $\alpha=0.005$，甚至只在 warm-up 后开启。反过来，如果你用的是极低温度、强 top-1、较小 capacity 的激进稀疏配置，单靠辅助 loss 往往不够，必须和容量设计一起看。

在一些实践里，像 Mixtral 这类模型更强调 top-k 路由、容量控制和经验调参，辅助 loss 主要用于防止极端非均匀激活，而不是追求绝对平均。原因很简单：真实任务中，最优路由本来就未必均匀。一个领域专家更擅长代码，一个更擅长数学，允许适度不均衡通常比机械平均更合理。

所以它的适用边界是：当你需要防止崩塌时，它很有效；当你已经有稳定的路由温度、容量调度和良好的专家 specialization，也就是“专家逐渐形成各自擅长的子任务”，它应当弱化，而不是主导优化。
