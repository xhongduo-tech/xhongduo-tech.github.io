## 核心结论

嵌套实体识别（Nested NER）解决的不是“文本里有没有实体”，而是“同一段文本里能不能同时保留多层实体边界”。更具体地说，一个较长实体内部，允许再包含一个或多个较短实体。传统 flat NER 通常把任务建成单一路径序列标注：每个位置只能落一个标签状态，因此一旦外层实体和内层实体共享文本片段，模型就经常被迫二选一，结果是内部实体、外部实体，或者两者中的一部分被压掉。

最直观的例子是“北京大学信息科学学院”。如果按 flat NER 做，模型往往只能输出一个整体组织名，或者只输出其中一段；但在嵌套 NER 里，可以同时识别：

| 文本片段 | 起止边界 | 实体类型 | 是否允许共存 |
| --- | --- | --- | --- |
| 北京大学 | 0-3 | 学校/组织 | 是 |
| 信息科学学院 | 4-9 | 学院/组织 | 是 |
| 北京大学信息科学学院 | 0-9 | 机构全称/组织 | 是 |

这里的关键不是“多输出几个答案”，而是**输出空间本身变了**。flat NER 假设一段文本只对应一条合法标签路径；嵌套 NER 则允许多个 span 同时成立，只要它们的边界和类型都合法。

工程上常见的三类做法是：

| 方法 | 核心思想 | 适合场景 |
| --- | --- | --- |
| 层叠标注器 | 一层一层识别，从内到外或从外到内 | 嵌套层数较浅、结构较稳定 |
| 超图解码 | 用超边表示多种重叠和包含关系 | 合法结构复杂、需要统一约束 |
| Biaffine NER | 先找边界 span，再做类别判断 | 边界和分类可解耦、易做多任务增强 |

为什么这三类方法经常优于 flat NER？原因不神秘：它们都在不同程度上摆脱了“必须走单条 BIO 路径”的限制。尤其在 ACE2004、ACE2005、GENIA 这类带明显嵌套结构的数据集上，嵌套建模通常比 flat NER 更能召回内部实体；其中 span-based 或 Biaffine 一类方法，因为直接对边界建模，往往对复杂机构名、医学术语边界更敏感。

可以把核心结论压缩成一句话：

**嵌套 NER 的重点不是换一个更大的编码器，而是换一种允许“多层边界共存”的输出与解码方式。**

## 问题定义与边界

先给出精确定义。实体识别的目标，是从句子中找出一个实体集合 $E$。每个实体可以写成：

$$
e_i = (s_i, t_i, l_i)
$$

其中：

- $s_i$：实体起始位置
- $t_i$：实体结束位置
- $l_i$：实体类别标签

白话说，就是“从哪里开始、到哪里结束、它属于什么类型”。

例如句子“中信银行总部发布公告”，如果按字符位置标注，可能出现两个实体：

- $e_1 = (0, 3, \text{BANK})$，表示“中信银行”
- $e_2 = (0, 5, \text{ORG\_UNIT})$，表示“中信银行总部”

这两个实体共享同一个起点，但终点不同，因此一个是另一个的内部实体。

很多入门者第一次接触嵌套 NER 时，容易把几种结构混在一起。先分清：

| 结构类型 | 形式 | 例子 | 是否属于本文重点 |
| --- | --- | --- | --- |
| 嵌套实体 | 一个实体完整包含另一个实体 | “北京大学信息科学学院”包含“北京大学” | 是 |
| 重叠但不包含 | 两个实体有交集，但一个不完整落在另一个内部 | 某些复杂事件片段 | 部分相关 |
| 非连续实体 | 一个实体由不连续片段组成 | “A ... and ... B”式结构 | 否 |
| 关系抽取 | 判断两个实体之间的关系 | “公司-总部位于-城市” | 否，但强相关 |

严格说，flat NER 常见假设是：输出实体之间不能发生交叉或包含。文献里常见的冲突关系写成：

$$
s_i < s_j \le t_i < t_j
$$

这表示两个实体边界发生交叉，序列标注很难自然表达。对嵌套场景，更常见的是包含关系：

$$
s_i \le s_j \le t_j \le t_i
$$

如果再加上至少有一个边界严格不同，就表示 $e_j$ 落在 $e_i$ 内部。也就是说，嵌套 NER 允许多个合法实体共享部分文本区间。

这会直接改变建模方式。为什么 BIO/BIOES 在这里天然吃亏？因为它的基本思想是：给每个 token 分配一个标签，整句对应一条标签序列。可一旦同一个 token 同时属于内层实体和外层实体，单标签序列就无法无损表达。

看一个更贴近工程的例子。“中信银行总部”至少对应两层价值：

1. “中信银行”是机构主体，适合进入机构知识库或做主体归一化。
2. “中信银行总部”是更具体的组织单元，适合事件定位、组织层级分析、风险归因。

如果系统只保留长实体，下游做主体聚合时会丢失“中信银行”这层基础对象；如果只保留短实体，事件抽取时又无法知道“总部”这个组织单元。很多业务问题并不是“长实体和短实体谁更对”，而是“两层都要留”。

所以，嵌套 NER 主要适用于这些语料：

| 领域 | 为什么容易出现嵌套 |
| --- | --- |
| 新闻事件 | 机构全称、地点层级、职位短语经常互相包含 |
| 生物医学 | 专业术语层次深，短术语经常嵌入长术语 |
| 法律法规 | 机构、法规名称、条款对象边界复杂 |
| 金融公告 | 机构主体、组织单元、产品全称经常叠层出现 |

反过来说，如果你的文本更像“快递单号查询”“天气怎么样”“买手机推荐”这类短问句，绝大多数实体都是单层片段，那嵌套建模可能并不划算。模型复杂度上去了，但收益未必明显。

这里可以先记住一个判断准则：

**如果内部实体被丢掉，会直接影响下游检索、归一化、事件抽取或知识对齐，那么就值得认真做嵌套 NER。**

## 核心机制与推导

### 1. 层叠标注器

层叠标注器可以理解成“把多个 flat NER 叠起来”。第一层先识别一批实体，第二层继续在上一层结果基础上识别其他层，通常是从内到外，也可以从外到内。

典型结构是 `BiLSTM + CRF` 多层堆叠：

- `BiLSTM`：双向长短期记忆网络，同时编码左侧和右侧上下文
- `CRF`：条件随机场，在标签序列上施加全局约束，避免非法标签跳转

一个常见流程是：

1. 编码整句，得到每个 token 的上下文表示。
2. 第 1 层 CRF 输出一批实体。
3. 把这一层预测到的实体信息重新编码回 token 表示。
4. 第 2 层继续预测更外层或更难的一层。
5. 重复直到达到最大层数，或模型判断无需继续。

如果拿“某银行总部”举例，内到外的层叠过程可以写成：

| 层数 | 输入理解 | 输出实体 |
| --- | --- | --- |
| 第 1 层 | 看原始句子 | “某银行” |
| 第 2 层 | 看原始句子 + 第 1 层实体提示 | “某银行总部” |

这种方法为什么容易理解？因为它保留了序列标注框架，很多已有 NER 系统只需要做增量修改就能迁移过来。对嵌套深度很浅的场景，比如大部分文本只出现 2 层以内包含关系，层叠模型常常是可落地的第一步。

但它的局限也很清楚：

| 局限 | 本质原因 |
| --- | --- |
| 需要预估最大嵌套层数 | 模型本身是“层数驱动”的 |
| 层间误差会传递 | 前一层漏掉实体，后一层常常也受影响 |
| 深层嵌套不稳定 | 层数一多，训练和推理复杂度都上升 |
| 结构表达有限 | 对复杂重叠、交叉关系不够自然 |

所以，层叠模型适合“结构不太乱”的嵌套问题。它不是最强表达，但通常是最容易解释、最容易调试的一类方案。

---

### 2. 超图解码

超图（hypergraph）可以理解成“边不再只连接两个点，而是能连接多个状态节点”。普通图擅长描述一对一关系；超图更适合表达“一个局部结构同时满足多种连接关系”。

放到嵌套 NER 里，超图方法的核心不是更换编码器，而是把实体的开始、结束、内部、类别等状态编码成一个结构化图，再通过图上的合法路径或合法子结构进行解码。这样做的价值是：

- 允许多个实体共享局部片段
- 允许包含关系被显式表示
- 可以把“哪些结构合法”直接写进解码空间

对新手来说，可以把它理解成：

**BIO 是在一条线性轨道上走路，超图是在一个结构化路网里找合法通路。**

举个直观的类比。对于“北京大学信息科学学院”，如果用单条标签路径表示，你必须在“北京大学”和“北京大学信息科学学院”之间做让步；而超图方法允许两条结构同时成立，因为它编码的是“边界之间的兼容关系”，不是“每个位置最终只能选一个标签”。

超图方法的优势主要在两点：

| 优势 | 解释 |
| --- | --- |
| 结构表达强 | 适合显式描述嵌套、重叠等复杂边界关系 |
| 合法性控制强 | 可以在解码层约束哪些组合允许输出 |

但代价也高：

| 成本 | 解释 |
| --- | --- |
| 实现复杂 | 图结构设计、状态定义、转移规则都不简单 |
| 调试困难 | 错误可能来自编码器、打分函数、图结构、解码器任一层 |
| 工程门槛高 | 对团队要求高，不适合快速原型 |

因此，超图方法更适合两类场景：

1. 任务价值很高，允许投入较大实现成本。
2. 你确实需要严格结构约束，而不是只想“多召回一些内部实体”。

如果只是常规业务抽取，团队资源有限，超图往往不是第一选择；如果是高价值信息抽取系统，且结构错误代价高，它就有存在意义。

---

### 3. Biaffine NER

Biaffine NER 可以理解成“先把所有可能的片段 span 看成候选实体，再对每个候选片段打分”。它不直接按 token 一个个贴 BIO 标签，而是把问题改成：

1. 从位置 $i$ 到位置 $j$ 的片段，像不像一个实体？
2. 如果像，它属于哪一类？

这是它与序列标注最本质的区别。

核心打分公式通常写成：

$$
s_{i,j}={h_i^{start}}^T U h_j^{end} + W(h_i^{start}\oplus h_j^{end}) + b
$$

其中：

- $h_i^{start}$：位置 $i$ 作为起点时的表示
- $h_j^{end}$：位置 $j$ 作为终点时的表示
- $U$：双线性参数矩阵，用来建模起点和终点的交互
- $W$：线性项参数
- $b$：偏置项
- $\oplus$：向量拼接

可以把这个公式拆成两部分理解：

| 项 | 作用 |
| --- | --- |
| ${h_i^{start}}^T U h_j^{end}$ | 建模“这个起点和这个终点搭配得像不像一个实体” |
| $W(h_i^{start}\oplus h_j^{end}) + b$ | 提供附加线性信息，补充整体偏置 |

为什么 Biaffine 对嵌套更友好？因为它不再要求所有实体共享一条标签路径。只要这些 span 合法，它们就可以同时存在。例如：

- “北京大学”
- “信息科学学院”
- “北京大学信息科学学院”

这三个都是不同的 span。它们可以共存，不必竞争同一位置的唯一标签槽位。

把它和 flat NER 对比，会更容易看清差别：

| 维度 | Flat NER | Biaffine / Span-based NER |
| --- | --- | --- |
| 基本对象 | token 标签 | span 候选片段 |
| 输出形式 | 一条标签序列 | 一组被打分后的片段 |
| 对嵌套的兼容性 | 弱 | 强 |
| 主要难点 | 标签设计与转移约束 | 候选数多、负样本不平衡 |

真实工程里，Biaffine 很少单独存在，通常会和以下策略一起用：

| 配套策略 | 作用 |
| --- | --- |
| 预训练编码器（BERT 等） | 提高上下文表示质量 |
| 边界辅助任务 | 提高起止位置召回 |
| 长度裁剪 | 控制候选 span 数量 |
| 候选剪枝 | 减少无效 span |
| 多任务损失 | 同时优化边界与类别学习 |

训练时总损失常写成：

$$
\mathcal{L} = \mathcal{L}_{boundary} + \lambda \mathcal{L}_{label}
$$

如果还引入候选筛选或嵌套辅助监督，也可能写成：

$$
\mathcal{L} = \mathcal{L}_{boundary} + \lambda_1 \mathcal{L}_{label} + \lambda_2 \mathcal{L}_{aux}
$$

其中：

- $\mathcal{L}_{boundary}$：边界检测损失
- $\mathcal{L}_{label}$：类别分类损失
- $\mathcal{L}_{aux}$：辅助任务损失，例如是否为嵌套实体、span 质量评分等
- $\lambda_1,\lambda_2$：权重系数

这里有一个非常实用的工程认识：

**很多嵌套 NER 的错误，先不是“类别错了”，而是“边界候选根本没被召回”。**

所以做 Biaffine 时，先把边界学稳，通常比一开始堆复杂类别头更重要。

## 代码实现

下面给一个可运行的简化版 Python 例子，演示“枚举 start/end 组合，计算 span 分数，超过阈值后进入类别判断”的完整流程。这个版本不依赖任何深度学习框架，只用标准库，目的是帮助理解解码逻辑，而不是追求模型效果。

为了让例子真正可运行，这里把文本按字符切开，这样“北京大学”“信息科学学院”“北京大学信息科学学院”三个 span 都能被枚举出来。

```python
from dataclasses import dataclass
from typing import Dict, List, Tuple

text = "北京大学信息科学学院"
tokens = list(text)  # 按字符切分，便于演示 span 枚举

# 每个位置的“起点表示”和“终点表示”
# 真实模型里这些向量来自 BERT / BiLSTM 编码器，这里手工构造。
start_vecs: Dict[int, List[float]] = {
    0: [1.3, 0.1],  # 北
    1: [0.2, 0.1],
    2: [0.2, 0.1],
    3: [0.2, 0.1],  # 学
    4: [0.9, 0.7],  # 信
    5: [0.2, 0.2],
    6: [0.2, 0.2],
    7: [0.2, 0.2],
    8: [0.2, 0.2],
    9: [0.2, 0.2],
}

end_vecs: Dict[int, List[float]] = {
    0: [0.2, 0.1],
    1: [0.2, 0.1],
    2: [0.2, 0.1],
    3: [1.2, 0.2],  # 学，方便“北京大学”得高分
    4: [0.3, 0.1],
    5: [0.3, 0.2],
    6: [0.3, 0.2],
    7: [0.3, 0.2],
    8: [0.3, 0.2],
    9: [0.8, 1.1],  # 院，方便长 span 和“信息科学学院”得高分
}

# 玩具 biaffine 参数
U = [
    [0.9, 0.2],
    [0.1, 0.8],
]
W = [0.05, 0.08, 0.04, 0.07]
b = -0.25

@dataclass
class Entity:
    text: str
    start: int
    end: int
    boundary_score: float
    label: str
    label_score: float

def dot(a: List[float], b: List[float]) -> float:
    return sum(x * y for x, y in zip(a, b))

def matmul_vec(v: List[float], m: List[List[float]]) -> List[float]:
    # 计算 v^T * M
    return [
        v[0] * m[0][0] + v[1] * m[1][0],
        v[0] * m[0][1] + v[1] * m[1][1],
    ]

def biaffine_score(hs: List[float], he: List[float]) -> float:
    bilinear = dot(matmul_vec(hs, U), he)
    linear = dot(W, hs + he)
    return bilinear + linear + b

def label_score(span_text: str) -> Dict[str, float]:
    # 真实场景中这里会是一个分类器；这里用规则模拟类别分数
    if span_text == "北京大学信息科学学院":
        return {"ORG_FULL": 0.97, "SCHOOL": 0.15, "COLLEGE": 0.21, "O": 0.01}
    if span_text == "北京大学":
        return {"ORG_FULL": 0.06, "SCHOOL": 0.95, "COLLEGE": 0.05, "O": 0.02}
    if span_text == "信息科学学院":
        return {"ORG_FULL": 0.04, "SCHOOL": 0.08, "COLLEGE": 0.96, "O": 0.03}
    return {"O": 0.99}

def best_label(scores: Dict[str, float]) -> Tuple[str, float]:
    label = max(scores, key=scores.get)
    return label, scores[label]

def decode(tokens: List[str], threshold: float = 0.95, max_span_len: int = 10) -> List[Entity]:
    results: List[Entity] = []
    n = len(tokens)

    for i in range(n):
        for j in range(i, min(n, i + max_span_len)):
            span_text = "".join(tokens[i:j + 1])
            boundary = biaffine_score(start_vecs[i], end_vecs[j])

            if boundary < threshold:
                continue

            scores = label_score(span_text)
            label, cls_score = best_label(scores)

            if label == "O":
                continue

            results.append(
                Entity(
                    text=span_text,
                    start=i,
                    end=j,
                    boundary_score=round(boundary, 3),
                    label=label,
                    label_score=round(cls_score, 3),
                )
            )

    return results

entities = decode(tokens)

# 简单验证：三个嵌套实体都能同时输出
texts = {e.text for e in entities}
assert "北京大学" in texts
assert "信息科学学院" in texts
assert "北京大学信息科学学院" in texts

for e in entities:
    print(
        f"text={e.text}, span=({e.start},{e.end}), "
        f"boundary={e.boundary_score}, label={e.label}, label_score={e.label_score}"
    )
```

如果运行，输出会类似下面这样：

```text
text=北京大学, span=(0,3), boundary=1.298, label=SCHOOL, label_score=0.95
text=北京大学信息科学学院, span=(0,9), boundary=1.989, label=ORG_FULL, label_score=0.97
text=信息科学学院, span=(4,9), boundary=1.497, label=COLLEGE, label_score=0.96
```

这个玩具实现故意保留了三个关键事实：

1. `span` 是枚举出来的，不是只能走一条标签路径。
2. “北京大学”和“北京大学信息科学学院”可以同时存在。
3. 边界打分和类别判断是两步决策。

对新手来说，最值得看懂的是 `decode()` 这段逻辑：

| 步骤 | 代码对应 | 含义 |
| --- | --- | --- |
| 枚举候选 | `for i ... for j ...` | 把所有可能起止边界组合成候选 span |
| 边界筛选 | `if boundary < threshold: continue` | 先判断这段像不像实体 |
| 类型判断 | `label_score(span_text)` | 再判断它属于什么类别 |
| 过滤非实体 | `if label == "O": continue` | 去掉最终不成立的候选 |
| 保留多个结果 | `results.append(...)` | 不要求候选彼此互斥 |

放回真实模型后，流水线通常变成：

```text
输入 token
-> 编码器（BiLSTM / BERT）
-> 起点投影、终点投影
-> Biaffine span scoring
-> span 分类器
-> 候选剪枝 / 结构化解码
-> 输出实体集合
```

真实工程中还有两个必须面对的问题。

第一，复杂度问题。  
长度为 $n$ 的句子，所有连续 span 数量是：

$$
\frac{n(n+1)}{2}
$$

也就是 $O(n^2)$。句子一长，候选数很快上去，所以通常会限制最大 span 长度、先做边界候选筛选，或者只保留高分起点和终点。

第二，样本不平衡问题。  
正例实体很少，负例 span 极多。比如一个 100 字句子，连续 span 就有 5050 个，但真正的实体可能只有个位数或十几个。如果不处理，模型容易学成“全部判非实体也挺准”。

因此实际训练里，经常会额外做：

| 工程策略 | 作用 |
| --- | --- |
| `max_span_len` 限制 | 减少无意义长片段 |
| 负样本下采样 | 缓解类别极不平衡 |
| 边界候选剪枝 | 提升推理速度 |
| focal loss / class weight | 强化少数正例学习 |
| 后处理去噪 | 删掉不可信短 span 或冲突候选 |

再看一个贴近业务的例子。句子里有“中信银行总部发布公告”，一个合理输出是：

| Span | 类型 | 下游用途 |
| --- | --- | --- |
| 中信银行 | 机构主体 | 对齐机构库、做主体归一化 |
| 中信银行总部 | 组织单元/机构地点 | 事件发生位置、组织层级分析 |

如果只保留长实体，下游主体归一化会受损；如果只保留短实体，事件粒度又不够。嵌套 NER 的工程价值就在这里，不在于“模型更高级”，而在于“信息保留得更完整”。

## 工程权衡与常见坑

嵌套 NER 的难点不只在模型，还在数据、解码、阈值和标注规范。很多项目效果差，并不是因为模型原理不对，而是某个工程环节把召回压掉了。

下面先看常见坑：

| 常见坑 | 现象 | 根因 | 规避策略 |
| --- | --- | --- | --- |
| 扁平化输出 | 只识别外层，漏内部实体 | 仍用单一路径序列标注 | 改用 span-based、层叠或超图解码 |
| 低召回 | 实体类别准，但漏很多边界 | 边界监督不足 | 加 boundary loss、多任务训练 |
| 训练不收敛 | span 数爆炸，正负样本极不平衡 | 枚举空间过大 | 限制最大长度、做负采样 |
| 推理过慢 | 长文本上组合过多 | $O(n^2)$ span 枚举 | 先做边界候选剪枝 |
| 标注不一致 | 同类文本有时标内层、有时不标 | 规范不清 | 先统一 annotation guideline |

先看第一个最常见的问题：**明明任务是嵌套 NER，系统却仍然是 flat 输出。**

例如“某银行总部”这类短语，单层 CRF 常见结果只有：

- 识别“某银行总部”，漏掉“某银行”
- 或识别“某银行”，漏掉“某银行总部”

这通常不是“模型偶然失误”，而是**输出空间本身不允许两者同时成立**。这类问题靠多训练几轮、堆更多数据，收益通常有限，因为模型从定义上就没有地方存放第二层实体。

第二个常见误区是：**以为用了 Biaffine，嵌套问题就自动解决了。**  
并不会。Biaffine 只是把 span 建模做得更直接，它不保证边界一定能召回，也不保证短实体不会被长实体分数掩盖。

在中文机构名里，这个问题尤其明显。比如：

- “中国银行”
- “中国银行北京分行”

长 span 往往上下文更完整，因此得分更高。如果阈值过高、负样本构造过于粗暴、短实体样本不够，模型就会保留长实体，压掉内部短实体。

工程上常见的解决办法有：

| 问题 | 可操作方案 |
| --- | --- |
| 内层短实体召回低 | 增加短 span 正例权重，单独监控短实体 F1 |
| 长文本推理慢 | 先预测潜在起点/终点，再做 span 组合 |
| 类别很多且不平衡 | 用分层分类或类别重加权 |
| 标注不一致 | 先写清“何时保留内层实体”的规范 |

这里有两个很实用的经验。

第一，先把“有没有这个边界”学稳，再学类别。  
很多项目一开始就盯着类别 confusion matrix，看 PER/ORG/LOC 谁混谁。但嵌套 NER 常见的真正瓶颈是：内部实体根本没进入候选集合。边界没召回，再好的分类头也没用。

第二，标注规范比模型花样更重要。  
例如“北京大学信息科学学院”到底要不要同时标“北京大学”和“信息科学学院”？如果标注员 A 会标，标注员 B 不会标，模型学到的就不是任务规律，而是标注噪声。

建议至少写清这些规则：

| 规则问题 | 必须回答什么 |
| --- | --- |
| 内层实体是否总要保留 | 还是只在有独立语义时保留 |
| 组织全称与组织单元如何区分 | 如“总部”“分行”“研究院”怎么标 |
| 专名缩写是否单独标 | 如“北大”与“北京大学” |
| 并列结构怎么处理 | 如“清华大学和北京大学信息学院” |

如果语料里嵌套层数不深，层叠模型通常更容易落地；如果实体长度差异大、类别多且边界复杂，Biaffine 或其他 span-based 方案往往更稳；如果你必须保证输出结构严格合法，才有必要考虑超图。

## 替代方案与适用边界

没有一种方案在所有数据集上都占优，真正决定选择的，是语料结构、错误代价和工程预算。

| 方案 | 优点 | 缺点 | 适用边界 |
| --- | --- | --- | --- |
| Flat NER | 实现简单、推理快 | 无法保留嵌套 | 基本无嵌套的数据 |
| 层叠模型 | 思路直观、易解释 | 需设层数，误差可能层间传递 | 嵌套深度有限、结构规律明显 |
| 超图解码 | 结构表达强、支持复杂约束 | 实现复杂、调试成本高 | 高价值任务、对结构合法性要求高 |
| Biaffine/span-based | 边界和分类解耦，适配预训练编码器 | 候选 span 多，需剪枝 | 边界复杂、类别较多的工程场景 |

把这个选择问题再说得更直接一些：

### 什么时候可以继续用 Flat NER

如果你的业务文本里，99% 的实体都不会互相包含，或者即使漏掉内部实体也不影响下游任务，那么 flat NER 仍然是性价比最高的选择。比如很多电商标题分类、简单客服槽位抽取、基础搜索词识别，都属于这类场景。

### 什么时候优先考虑层叠模型

如果你的嵌套层数通常只有 2 到 3 层，而且结构很稳定，例如：

- 机构主体 → 组织单元
- 药物名 → 药物组合名
- 学校名 → 学院名

那么层叠模型往往更容易上手。因为它和传统序列标注体系接近，部署、解释、调试的认知成本最低。

### 什么时候优先考虑 Biaffine / Span-based

如果你具备以下特征，通常应优先考虑 span-based：

- 实体边界复杂，长短差异大
- 希望直接接入 BERT 等预训练编码器
- 需要边界监督、多任务训练、候选打分
- 对内部实体召回比较敏感

新闻、金融公告、生物医学文本，通常都落在这类范围里。

### 什么时候才值得上超图

只有当你明确需要“结构约束的强表达力”，而且团队能承担较高实现成本时，超图才值得认真投入。否则很多时候 Biaffine 已经能解决 80% 的工程问题，而代价小得多。

从公开经验看：

- `ACE2004 / ACE2005` 这类新闻事件语料里，组织名、地点名、职位相关实体经常互相包含，span-based 方法通常能更稳定地召回内部实体。
- `GENIA` 这类生物医学语料里，术语层级更深、边界更碎，flat NER 的损失更明显，结构化方法通常更占优。
- 如果业务文本更像“短句、弱上下文、单层槽位抽取”，复杂嵌套方案往往收益有限。

可以把选择逻辑压缩成一句话：

- 能预估嵌套层数，就先考虑层叠模型。
- 需要严格结构约束，就考虑超图。
- 需要和现代编码器、候选打分、多任务监督结合，就优先考虑 Biaffine。

最后补一个经常被忽视的现实判断：

**模型选择不是论文排名问题，而是下游任务需要保留哪一层信息的问题。**  
如果内部实体被漏掉并不会影响业务，那就没必要为“更复杂的建模”付出复杂度；如果它会影响知识对齐、事件归因、法律要素抽取，那嵌套建模就是必要成本。

| 资料 | 关键贡献 | 对本文的作用 |

| Alex Ju, Miwa, Ananiadou, 2018, *A Neural Layered Model for Nested Named Entity Recognition* | 用分层神经网络逐层识别嵌套实体，代表“层叠标注器”路线 | 支撑“层叠标注器”部分 |

| 多类 span-based / biaffine NER 工作 | 直接对 $(start, end)$ 候选 span 打分，并把边界检测与类别判断解耦 | 支撑“Biaffine NER”部分 |

| GENIA 数据集 | 生物医学领域常用嵌套实体评测基准，嵌套密度高 | 支撑“生物医学文本适配性”判断 |
| 嵌套 NER 综述类论文或教程 | 汇总 layered、hypergraph、span-based 等方法优缺点 | 支撑整体比较框架 |

如果你要继续深入，建议按下面顺序看：

1. 先看 layered 模型，理解为什么单层序列标注会丢实体。
2. 再看 hypergraph，理解“结构合法性”为什么能进入解码器。
3. 最后看 span-based / biaffine，理解为什么现代工程更偏向边界打分与分类解耦。

对工程实践者来说，真正有用的问题不是“哪篇论文最好”，而是：

- 你的数据里嵌套到底有多深？
- 下游任务是否真的需要保留内部实体？
- 标注规范是否允许同一片段保留多层结果？
- 你能否接受 $O(n^2)$ 候选枚举带来的训练与推理代价？

把这几个问题先答清楚，再选模型，通常比先追论文榜单更有效。
