## 核心结论

多模态位置编码的核心任务，不是给不同模态“编号”，而是把不同模态的坐标系统放进同一套可计算的几何表示里。文本是 1D 顺序，图像是 2D 网格，视频是 3D 时空坐标；如果仍然只用单轴位置编号，模型会把“出现顺序”误当成“真实位置”。

M-RoPE（Multimodal Rotary Positional Embedding，多模态旋转位置编码，白话解释：把每个轴的位置都变成一个旋转角度）延续了 RoPE 的关键优点：注意力主要依赖相对位移，而不是绝对编号。它的做法是给每个坐标轴单独分配旋转子空间，再把这些旋转组合进 query 和 key。这样，文本位置 $t=3$ 可以写成一个 1D 旋转 $R_t$，图像 patch 坐标 $(y=1,x=2)$ 可以写成一个 2D 组合旋转 $R_{(y,x)}$，两者进入统一注意力后，模型感知的是 $\Delta t,\Delta y,\Delta x$，而不是“谁先拼进序列”。

一句话概括：从 1D 到 N-D 的扩展，本质上是把“位置”从线性序号改成多轴坐标，并保持 RoPE 的相对不变性。

---

## 问题定义与边界

先把问题说清楚。Transformer 的自注意力本身不认识顺序，它只看到一堆向量。位置编码的作用，是把“第几个词”“图像中第几行第几列”“视频中第几帧”这些结构信息写进向量里。

单模态时问题相对简单：

| 模态 | 自然坐标维度 | 常见位置表示 | 主要目标 |
| --- | --- | --- | --- |
| 文本 | 1D | token index | 保持顺序与相对距离 |
| 图像 | 2D | patch $(y,x)$ | 保持空间邻近关系 |
| 视频 | 3D | $(t,y,x)$ | 同时保持时间与空间关系 |

难点出现在跨模态统一建模。假设把文本 token 和图像 patch 直接拼成一个长序列：

- 文本 token 位置：$0,1,2,3$
- 图像 patch 位置：$4,5,6,\dots$

这种做法的错误在于，图像 patch 的“位置”变成了“拼接后排第几个”，而不再是它真实的二维坐标。比如左上角 patch 和右下角 patch 的差异，本应来自 $(\Delta y,\Delta x)$，结果却被折叠成一个单一序号差。

玩具例子可以直接看出问题。设文本有 4 个 token，图像切成 $2\times2$ 共 4 个 patch：

| 元素 | 真实位置 | 直接拼接后的编号 |
| --- | --- | --- |
| 文本 token3 | $t=3$ | 3 |
| 图像 patch(0,0) | $(0,0)$ | 4 |
| 图像 patch(0,1) | $(0,1)$ | 5 |
| 图像 patch(1,0) | $(1,0)$ | 6 |
| 图像 patch(1,1) | $(1,1)$ | 7 |

此时 patch(0,1) 与 patch(1,0) 的欧式关系完全不同，但线性编号都只是在“相邻”或“差 1”。这会破坏空间结构。

本文的边界也要说明：

- 讨论对象是规则网格数据：文本序列、规则 patch 图像、规则采样视频帧。
- 重点是 RoPE 的 N-D 扩展，不展开稀疏点云、任意图结构、非均匀采样轨迹。
- 讨论的是统一 Transformer 中的共享注意力，不覆盖单独视觉编码器内部的所有变体。

所以，本文回答的问题是：如何把 1D 文本、2D 图像、3D 视频映射到同一个旋转空间，同时保留 RoPE 的相对位置性质。

---

## 核心机制与推导

RoPE 的原理可以先用一句白话讲清楚：它不是把位置“加”到向量上，而是按位置把向量“旋转”一下。这样两个位置不同的 token，在做点积时，会自然带出相对位移。

### 1. 从 1D RoPE 开始

标准 RoPE 会把向量按两维一组分块。每一组都做一次二维旋转。对第 $m$ 个维对，角频率定义为：

$$
\theta_m = \text{base}^{-2m/d}
$$

若位置为 $p$，则该维对的旋转角度为 $p\theta_m$。旋转矩阵写成：

$$
R(p\theta_m)=
\begin{bmatrix}
\cos(p\theta_m) & -\sin(p\theta_m) \\
\sin(p\theta_m) & \cos(p\theta_m)
\end{bmatrix}
$$

把所有维对拼起来，就是整个位置 $p$ 的旋转 $R_p$。

对 query 和 key 应用后：

$$
q' = R_x q,\quad k' = R_y k
$$

注意力中的核心项变成：

$$
{q'}^\top k' = q^\top R_x^\top R_y k
$$

因为旋转矩阵满足：

$$
R_x^\top R_y = R_{y-x}
$$

所以注意力只依赖相对位移 $y-x$。这就是“相对不变性”。

### 2. 扩展到 N-D

多维扩展的关键思想是：每个轴单独旋转，再做直和组合。设坐标向量为

$$
x=[x_1,x_2,\dots,x_n]
$$

其中 $x_1$ 可能是文本位置，或视频时间轴；$x_2,x_3$ 可能是图像的高宽坐标。每个轴都有自己的旋转生成元 $A_i$。生成元可以理解为“决定怎么转的局部规则”。组合形式写作：

$$
R_x=\bigoplus_i \exp(x_i A_i)
$$

这里的 $\bigoplus$ 表示直和，白话解释：不同轴占用不同的维度子块，各转各的，最后拼在一起。

更具体一点，对第 $i$ 个轴的第 $m$ 个维对，频率定义为：

$$
\theta_{i,m} = \text{base}_i^{-2m/d_i}
$$

于是该轴的旋转角度是 $x_i\theta_{i,m}$。不同轴可以用不同的 $\text{base}_i$，这是工程上非常重要的一点，因为文本、图像、视频的分辨率尺度根本不同。

### 3. 为什么跨模态仍然只看相对位移

如果一个文本 token 的坐标是 $x=(t)$，一个图像 patch 的坐标是 $y=(y,x)$，在统一实现里通常会扩成同维坐标，例如：

- 文本：$(t,0,0)$
- 图像：$(0,y,x)$
- 视频：$(f,y,x)$

那么：

$$
q' = R_{(t,0,0)} q,\quad k' = R_{(0,y,x)} k
$$

它们的相对关系由

$$
R_{(t,0,0)}^\top R_{(0,y,x)} = R_{(-t,y,x)}
$$

决定。也就是说，注意力里真正留下的是跨轴相对量，而不是“文本先拼、图像后拼”的人工顺序。

### 4. 最小数值例子

设 $d_{\text{head}}=4$，所以有两个维对；设 $\text{base}=10^4$。

对于文本位置 $t=3$：

- 第 0 个维对频率：$\theta_0=10^0=1$
- 第 1 个维对频率：$\theta_1=10^{-2/4}=10^{-0.5}=0.01$

于是角度分别为：

- $3\times1=3$
- $3\times0.01=0.03$

对应旋转分量就是：

$$
(\cos 3,\sin 3),\quad (\cos 0.03,\sin 0.03)
$$

对于图像 patch $(y=1,x=2)$，如果把前两个维留给 $y$ 轴、后两个维留给 $x$ 轴，那么：

- $y$ 轴角度：$1\times1=1$
- $x$ 轴角度：$2\times1=2$

得到：

$$
R_{(1,2)} = R_y \oplus R_x
$$

此时两个 patch $(1,2)$ 与 $(1,3)$ 的注意力差异只与 $\Delta x=1$ 有关；与它们被排在序列中的第几个 token 无关。

这个性质正是多模态共享注意力最需要的：统一空间，但不丢真实坐标。

## 代码实现

工程上最常见的做法，是在生成 query/key 后、进入 attention 前应用多轴旋转。输入不再只是一个位置编号，而是一个坐标 tuple。

先给一个新手可运行版本。它不是训练代码，但完整展示了“按轴分配维度并应用旋转”的最小实现。

```python
import math

def rotate_pair(x0, x1, angle):
    c = math.cos(angle)
    s = math.sin(angle)
    return x0 * c - x1 * s, x0 * s + x1 * c

def apply_axis_rope(vec, coord, base=10000.0):
    """
    对单个轴应用 RoPE。
    vec 长度必须为偶数，按两维一组旋转。
    """
    assert len(vec) % 2 == 0
    out = list(vec)
    d = len(vec)
    for m in range(d // 2):
        theta = base ** (-2.0 * m / d)
        angle = coord * theta
        i = 2 * m
        out[i], out[i + 1] = rotate_pair(vec[i], vec[i + 1], angle)
    return out

def apply_mrope(vec, coords, axis_dims, axis_bases):
    """
    coords: 每个轴的坐标，如 (t,) 或 (y, x) 或 (f, y, x)
    axis_dims: 每个轴占用多少维，且每项必须为偶数
    axis_bases: 每个轴对应的 base
    """
    assert len(coords) == len(axis_dims) == len(axis_bases)
    assert sum(axis_dims) == len(vec)

    out = []
    offset = 0
    for coord, dim, base in zip(coords, axis_dims, axis_bases):
        assert dim % 2 == 0
        chunk = vec[offset:offset + dim]
        out.extend(apply_axis_rope(chunk, coord, base=base))
        offset += dim
    return out

# 玩具例子：文本 token，坐标只有 t
q_text = [1.0, 0.0, 2.0, 0.0]
text_rot = apply_mrope(q_text, coords=(3,), axis_dims=[4], axis_bases=[10000.0])
assert len(text_rot) == 4

# 图像 patch，坐标是 (y, x)，每个轴分 2 维
q_img = [1.0, 0.0, 1.0, 0.0]
img_rot_12 = apply_mrope(q_img, coords=(1, 2), axis_dims=[2, 2], axis_bases=[1000.0, 1000.0])
img_rot_13 = apply_mrope(q_img, coords=(1, 3), axis_dims=[2, 2], axis_bases=[1000.0, 1000.0])

# x 坐标不同，结果应该不同
assert img_rot_12 != img_rot_13

# 同一输入同一坐标，结果必须一致
img_rot_12_b = apply_mrope(q_img, coords=(1, 2), axis_dims=[2, 2], axis_bases=[1000.0, 1000.0])
assert img_rot_12 == img_rot_12_b

print("M-RoPE toy example passed.")
```

上面这个版本有三个关键点：

| 代码部件 | 作用 | 对应概念 |
| --- | --- | --- |
| `coords` | 输入坐标 tuple | $(t)$、$(y,x)$、$(f,y,x)$ |
| `axis_dims` | 每个轴占用的子空间维度 | 不同轴独立旋转 |
| `axis_bases` | 每个轴自己的频率基数 | 不同模态可用不同频谱 |

如果写成伪代码，逻辑更直接：

```python
def apply_mrope(q, coord):
    # coord 例如 (t,) 或 (y, x)
    for axis in coord:
        q_axis = slice_for_this_axis(q)
        cos, sin = build_freq_cache(axis)
        q_axis = rotary_embed(q_axis, cos, sin)
    return q
```

新手版理解可以这样记：

- 文本 `[(t,)]` 只有一个轴。
- 图像 `[(y, x)]` 有两个轴。
- 视频 `[(f, y, x)]` 有三个轴。
- 每个轴都像“普通 RoPE”那样转，只是现在要转多次，而且转的是各自那一小段维度。

真实工程例子是视觉语言模型。以 Qwen2-VL 一类方案为代表，常把视觉 token 绑定到 $(i,x,y)$ 或 $(t,y,x)$ 三元组，其中时间轴和空间轴使用不同频率设计。直观上：

- 空间轴需要更高频，才能区分相邻 patch 的细粒度位置。
- 时间轴需要更低频，才能覆盖更长的视频跨度，避免很快进入高频振荡区。

这类设计不是“为了形式统一”，而是直接影响 grounding。grounding 可以理解为“把语言和视觉位置对上号”的能力，比如“红色按钮在左上角”这类定位任务。

## 工程权衡与常见坑

多模态位置编码最容易出错的地方，不在公式，而在实现假设。

### 1. 直接拼 token 序号是最常见错误

错误做法是把文本和图像全部展平成一个长序列，再对全序列套普通 1D RoPE。这样做实现最省事，但代价也最大：空间位置被折叠成序号偏移。

| 错误做法 | 后果 | 优选做法 |
| --- | --- | --- |
| 文本和图像直接拼接后共用 1D index | 图像二维结构丢失 | 为每个轴单独分配坐标 |
| 所有轴共用同一个 base | 高频或低频覆盖失衡 | 时间轴、空间轴分别设 base |
| 所有模态共享同一维度切分 | 某些轴表达能力不足 | 按任务重要性分配维数 |
| 把 modality token 当位置对齐手段 | 只标记“是什么模态” | 额外保留真实坐标编码 |

### 2. 频率设计不是越高越好

频率决定不同距离能否被分开。高频更擅长区分局部细节，低频更擅长覆盖长距离依赖。可以把它理解成“显微镜”和“望远镜”的取舍，但这里要落回技术表达：高频分辨率高，低频作用范围大。

常见建议如下：

| 轴类型 | 建议频率倾向 | 原因 |
| --- | --- | --- |
| 文本轴 | 中等 | 序列长度中等，语义顺序稳定 |
| 图像空间轴 | 偏高频 | patch 很密，需要细粒度定位 |
| 视频时间轴 | 偏低频 | 帧跨度长，需要长期依赖 |
| 高分辨率遥感/文档图像 | 更高频或更多维 | 局部细节非常关键 |

对初学者，一个实用判断标准是：

- 如果模型看不清“小区域差别”，优先检查空间轴频率是否太低。
- 如果模型记不住“前几秒发生过什么”，优先检查时间轴频率是否太高。

### 3. 轴间维度分配也有成本

设一个 attention head 只有 64 维，你不能无限给每个轴分配子空间。若视频用 3 个轴，平均分可能每轴只有 20 多维；如果再加 modality 轴、层级轴，单轴表达力会继续下降。

所以工程上通常要做取舍：

- 文本+图像任务：优先保证空间轴维度。
- 视频问答任务：时间轴不能太少。
- OCR 或文档理解：二维位置往往比纯时间更重要。

### 4. 真实工程例子：Qwen2-VL 类方案

这类模型中的经验通常是：

- 给视觉 token 明确绑定时空坐标。
- 空间轴频率更密，强化定位能力。
- 时间轴频率更缓，保持远距离帧关联。
- 继续复用 RoPE 体系，减少从零学习位置结构的成本。

这类设计之所以有效，是因为视觉 grounding 本质上依赖“相对位置还原”。如果一句文本说“第二张图右下角的蓝色框”，模型必须同时处理文本顺序、图像空间、图间顺序。M-RoPE 把这三者统一进一个可计算框架里。

## 替代方案与适用边界

M-RoPE 不是唯一方案，只是“统一共享注意力且要保留相对位移”时非常合适。

### 1. 常见替代方案

| 方案 | 是否保持相对性 | 可扩展轴数 | 训练成本 | 适用情况 |
| --- | --- | --- | --- | --- |
| M-RoPE | 是 | 高 | 中 | 文本+图像+视频统一建模 |
| 绝对坐标 embedding 相加 | 弱 | 中 | 低 | 简单多模态原型 |
| learnable grid | 部分 | 中 | 高 | 固定分辨率视觉任务 |
| modality token + 1D RoPE | 否 | 低 | 低 | 只想区分模态，不强求空间关系 |
| 相对位置 bias | 部分 | 中 | 中 | 局部窗口或特定结构注意力 |

需要特别区分两件事：

- `modality token` 只回答“这是文本还是图像”。
- 多轴位置编码回答“它在什么位置，与别的位置相差多少”。

前者不能替代后者。比如在 patch 前插入一个 `[IMG]` token，模型知道后面是图像，但并不知道 patch 的二维几何结构。

### 2. 何时不该硬用 M-RoPE

M-RoPE 最适合规则坐标系。如果数据不是规则网格，就要谨慎：

- 点云是稀疏且不规则的，直接套 $(x,y,z)$ 网格未必合理。
- 图结构没有天然连续坐标，可能更适合图位置编码。
- 非均匀时间采样视频，帧间隔不等，简单整数帧号会失真。

如果任务只有图像，没有文本跨模态需求，那么 2D RoPE 往往已经足够。只有图像和文本要进入同一个注意力空间时，多模态统一 RoPE 的价值才会明显上升。

可以用一句简单决策规则：

- 只有文本：1D RoPE。
- 只有图像：2D RoPE。
- 文本+图像：优先考虑 M-RoPE。
- 文本+视频：使用时间轴和空间轴拆分的 M-RoPE。
- 非规则几何：先确认坐标是否真的可离散成规则轴。
