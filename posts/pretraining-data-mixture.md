## 核心结论

预训练数据配比，指的是为不同数据域分配采样权重。数据域就是来源或任务属性相近的一组语料，例如网页、代码、数学、百科和多语言文本。核心结论有三条。

第一，有限 token 预算下，模型能力不是“谁多喂谁就更强”，而是取决于不同域之间是否形成互补。网页语料提供覆盖面，代码语料提供强结构模式，数学语料提供符号操作，多语言语料提供语言迁移。单一域过采样，通常会把某一项能力拉高，但会压缩其他能力。

第二，动态配比通常优于静态配比。DoReMi 的做法是先用一个小代理模型估计“哪个域还值得继续学”，再把权重转给大模型。公开结果里，基于 280M 代理模型学出的域权重，用于 8B 主模型训练，平均精度提升约 6.5%，并且比均匀或静态基线更快达到同等效果，速度约为 2.6 倍。

第三，代码数据虽然在总 token 中常常只占小比例，但对推理能力有迁移价值。迁移，意思是某个任务上学到的模式，能帮助另一个相关任务。代码里的变量绑定、分步执行和条件结构，会迁移到数学与复合推理任务，所以“代码少量混入”往往比“只堆数学题”更稳。

| 场景 | 静态配比 | 动态配比 |
|---|---:|---:|
| 网页域早期收敛后继续采样 | 容易浪费 token | 会自动降权 |
| 代码域占比小但仍有收益 | 容易被低估 | 可因收益被抬权 |
| 最差域表现 | 常被平均数掩盖 | 会被显式关注 |

玩具例子：如果你只有 100 个训练片段，网页 80、代码 10、数学 10，看起来很均衡；但如果网页域已经学会、代码域仍在下降，那么下一轮继续给网页 80 个样本就是低效。动态方法会把一部分网页样本让给代码和数学。

真实工程例子：Dolma 的 3T token 中，Common Crawl 约占 81%，GitHub 代码约占 13%，剩余来自 Reddit、Wikipedia、Semantic Scholar 等。这个结构说明，即使是“次要域”，一旦规模达到数百亿 token，也足以决定模型的推理和工程能力上限。

## 问题定义与边界

预训练数据配比要解决的问题，是在固定训练预算内，决定“每种数据该抽多少”。预算，指总 token 数、GPU 时长和清洗成本三者共同形成的约束。

边界也很明确。第一，不能把“领域权重优化”理解成只追某个榜单分数。预训练目标是泛化，也就是模型在没见过的新任务上仍然表现稳定。第二，不能把低资源语言直接稀释掉。低资源语言，意思是可用高质量文本很少的语言。如果只按总量采样，它们几乎一定被高资源语言吞没。第三，不能让噪声域被错误放大。噪声域，意思是重复、错标、乱码或低信息密度数据。

下面这个表能直观看到“有限预算内仍需保留多域”的原因。

| 数据集/域 | 典型来源 | 作用 | 风险 |
|---|---|---|---|
| 网页 | Common Crawl, FineWeb | 提供广覆盖知识与语言模式 | 重复多、噪声高 |
| 代码 | GitHub | 提供结构化推理与执行模式 | 许可证、模板重复 |
| 百科/论文 | Wikipedia, Semantic Scholar | 提供高密度事实与术语 | 覆盖面窄 |
| 社区文本 | Reddit, QA | 提供对话与非正式表达 | 质量波动大 |
| 多语言 | 多语网页、平行或单语语料 | 提供跨语言迁移 | 低资源语言极易被淹没 |

如果把 Dolma 这样的混合数据集看成一个预算分配器，那么它表达的不是“网页最重要”，而是“网页负责打底，代码和其他域负责补能力缺口”。这也是数据配比的边界条件：目标不是追求某个域的纯度，而是在总预算下形成能力组合。

## 核心机制与推导

DoReMi 的核心不是“先拍脑袋定配比”，而是先比较两个模型在每个域上的学习收益。这里的代理模型，就是一个更小、更便宜、用于试错的模型；参考模型，则是固定配比训练出来的对照组。

它使用的核心量是超额损失：

$$
E_d = L^{proxy}_d - L^{ref}_d
$$

其中，$L_d$ 表示域 $d$ 上的损失。损失可以理解为“模型在该域上还没学会多少”。如果某个域的超额损失更大，说明当前策略在这个域上还没达到参考水平，或者该域仍有继续优化空间。

DoReMi 在 Group-DRO 框架下优化：

$$
\max_{w \in \Delta} \sum_d w_d E_d
$$

这里的 $w_d$ 是域权重，$\Delta$ 表示所有权重非负且总和为 1。直白地说，这个目标会把注意力集中到“最差、最没学好、但仍值得学”的域上。它带有 minimax 思想，也就是优先修正最短板，而不是单纯追求平均值。

玩具例子：假设只有三个域，网页、代码、数学。某轮评估得到超额损失分别是 0.02、0.08、0.05。那代码域最“亏”，系统就应该提高代码采样权重，而不是继续按原比例灌网页。

真实工程例子：在 The Pile 这类多域数据上，均匀训练往往让大域天然占优势；而 DoReMi 会把训练中仍在持续收益的域抬权，把已经饱和或噪声偏高的域降权。于是同样的训练 token，最终换来更好的最差域表现。

一个简化流程可以写成：

| 步骤 | 输入 | 输出 |
|---|---|---|
| 1. 训练参考模型 | 固定配比数据 | $L^{ref}_d$ |
| 2. 训练代理模型 | 当前候选配比 | $L^{proxy}_d$ |
| 3. 计算超额损失 | 两组域损失 | $E_d$ |
| 4. 更新域权重 | $E_d$ | 新的 $w_d$ |
| 5. 重采样主训练集 | $w_d$ | 新批次 token |

## 代码实现

实现上最小需要四个部件：域级损失统计、权重更新、权重归一化、按权重重采样。下面给出一个可运行的玩具实现，它不训练真正模型，但把 DoReMi 的重心逻辑保留下来了。

```python
from math import exp

domains = ["web", "code", "math", "multilingual"]

# 参考模型和代理模型在各域上的损失
ref_loss = {
    "web": 1.20,
    "code": 1.40,
    "math": 1.35,
    "multilingual": 1.50,
}

proxy_loss = {
    "web": 1.18,
    "code": 1.55,
    "math": 1.42,
    "multilingual": 1.62,
}

def excess_loss(proxy, ref):
    return {d: proxy[d] - ref[d] for d in proxy}

def softmax_weight(values, temperature=8.0):
    scores = {k: exp(v * temperature) for k, v in values.items()}
    total = sum(scores.values())
    return {k: scores[k] / total for k in scores}

def rehydrate_low_resource(weights, removed_ratio, floor=0.08):
    # 对低资源语言做保底，避免被高资源域完全挤出
    adjusted = {}
    for d, w in weights.items():
        boost = removed_ratio.get(d, 0.0)
        adjusted[d] = max(w * (1.0 + boost), floor if d == "multilingual" else 0.0)
    total = sum(adjusted.values())
    return {d: w / total for d, w in adjusted.items()}

e = excess_loss(proxy_loss, ref_loss)
weights = softmax_weight(e)
weights = rehydrate_low_resource(weights, {"multilingual": 0.20})

assert abs(sum(weights.values()) - 1.0) < 1e-9
assert weights["code"] > weights["web"]
assert weights["multilingual"] > 0.0

print("excess_loss =", e)
print("weights =", weights)
```

如果把这段逻辑接到真实训练系统里，常见接口会长这样：

| 字段 | 含义 |
|---|---|
| `domain` | 数据域名，如 `web/code/math/multilingual` |
| `loss` | 当前 batch 或滑动窗口损失 |
| `count_tokens` | 已采样 token 数 |
| `dedup_ratio` | 去重后保留比例 |
| `lang_confidence` | 语言识别置信度 |
| `sample_weight` | 下一轮采样概率 |

伪代码如下：

```python
for step in training_steps:
    domain_losses = eval_proxy_and_ref(loaders)
    excess = {d: domain_losses["proxy"][d] - domain_losses["ref"][d] for d in domains}
    weights = normalize(group_dro_update(excess))
    weights = apply_low_resource_rehydrate(weights, lang_stats)
    batch = resample_next_tokens(loaders, weights)
    train_main_model(batch)
    log(weights)
```

这套逻辑的关键不是公式本身，而是“域级反馈闭环”：先看每个域是否仍有学习收益，再决定下一批 token 怎么抽。

## 工程权衡与常见坑

工程里最常见的错误，是把“某个能力变强”误认为“整体推理更强”。例如只加数学数据，通常能提升加法、乘法这类封闭题，但在需要多步语言理解的 asdiv CoT 上，可能明显变差，甚至掉到接近 0。原因不是数学无用，而是数据分布过窄，模型学到的是“题型模板”，不是稳定的复合推理。

相反，代码数据常常带来更稳的迁移。代码片段对简单算术帮助不一定明显，但对含条件、变量和步骤拆解的任务，往往能带来额外提升。研究里有类似现象：代码混入比单纯堆数学更稳定，asdiv 这类任务可提升约 3%。

第二类坑出现在多语言。低资源语言如果只做一次语言识别，然后按总量采样，结果通常是被彻底边缘化。更稳的做法是设语言置信阈值、保留 `_removed` 集合、在训练采样时做 rehydrate。rehydrate，意思是把清洗阶段被压掉但仍有潜在价值的低资源样本，以受控权重重新补回训练流。

| 坑 | 直接后果 | 规避方法 |
|---|---|---|
| 只堆数学数据 | 算术变强，复合推理变差 | 混入代码与自然语言解释 |
| 网页域长期固定高占比 | 已饱和域继续耗 token | 用动态权重降采样 |
| 低资源语言只按量采样 | 语言覆盖崩塌 | 阈值、去重、rehydrate |
| 代码仓库不去重 | 模板重复污染训练 | 仓库级和片段级双重去重 |
| 用平均 loss 决策 | 短板域被掩盖 | 看域级损失与最差域表现 |

真实工程例子：FineWeb2 面向多语言场景时，不是简单把所有语言混在一起，而是经过多轮 Common Crawl 处理、语言识别、过滤、去重和再采样，覆盖 1,868 个语言-脚本组合，并为数百个低资源语言保留最低可用规模。这说明多语数据工程的重点不是“收得多”，而是“保得住”。

## 替代方案与适用边界

如果团队没有代理模型、没有额外评估预算，静态配比仍然是合理基线。静态配比就是在训练前一次性确定权重，例如网页 80%、代码 10%、学术 5%、多语 5%。它的优点是简单、稳定、易复现，适合小模型、预算紧、先跑通流程的场景。

但静态配比的边界也明显：它无法根据训练过程中的真实收益修正权重。域已经饱和，仍然继续采；域还有收益，却因为初始占比低而长期得不到足够 token。

还有一类替代方案是“先做多语数据工程，再做普通训练”，例如 FineWeb2 路线。它更关注语料收集、语言识别、阈值与重采样，适合目标是扩大语言覆盖而不是追求每轮动态最优的团队。

| 方案 | 优点 | 缺点 | 适用边界 |
|---|---|---|---|
| 静态配比 | 实现简单，便于复现 | 无法随训练动态修正 | 小模型、基线训练 |
| DoReMi 动态配比 | 能提升最差域与训练效率 | 需要代理模型和域评估 | 中大模型、持续迭代 |
| FineWeb2 式多语工程 | 多语言覆盖强，低资源更稳 | 主要解决数据构成，不直接给动态权重 | 多语优先场景 |

所以，实践中的顺序通常不是“只能选一个”，而是先做高质量数据构成，再决定是否上动态配比。数据清洗决定下限，权重优化决定上限。
