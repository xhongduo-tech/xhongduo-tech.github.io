## 核心结论

DeepSeek-V3 的训练数据配方可以概括为两句话。第一，它先用约 14.8T 的高质量 token 做大规模预训练，先把通用语言、知识表达和基本世界模型学稳。第二，它不是把所有数据一锅煮，而是分阶段调整配方：前期以通用文本为主，后期显著提升数学、代码、STEM 与长文本占比，再配合长上下文扩展，把模型能力从普通问答推到 128K 上下文下的技术推理。

这里的“token”可以理解为模型训练时使用的最小文本单位，通常比“词”更细。这里的“长上下文”指模型一次能同时看到的文本窗口，128K 表示可以同时处理非常长的合同、报告或代码仓片段。

新手视角可以把它想成两阶段喂料。第一阶段主要喂百科、网页、书籍、论文这类“泛知识语料”，其中公开数据约占 60%。第二阶段换更偏“技术强化”的食谱，增加数学公式、代码片段、科学文本和超长文档。这样训练出来的模型，既能回答通用问题，也更有机会在几十页文档里追溯条款引用、变量定义或前后逻辑。

| 训练阶段 | 数据重心 | 主要目标 | 对能力的直接影响 |
| --- | --- | --- | --- |
| 第一阶段 | 通用网页、书籍、学术文本 | 建立语言能力与知识覆盖 | 通用问答、基础写作、事实表达 |
| 第二阶段 | 数学、代码、STEM、长文档 | 强化推理与长程依赖 | 技术问答、代码补全、长文审查 |
| 长上下文扩展 | 32K 到 128K 长文 | 扩大可见范围 | 跨章节检索、交叉引用、长链推理 |

---

## 问题定义与边界

这套配方要解决的核心问题，不是“让模型知道更多词”，而是“让模型在很长的上下文里还能稳定理解、定位、推理”。比如合同审查场景，模型不只要认识“违约”“赔偿”“保密”这些词，还要能在第 12 页找到定义条款，再回到第 37 页判断引用是否一致。

问题的边界也很明确。数据越多不等于效果越好，因为训练数据里存在重复页面、镜像站点、模板化文档、低质量抓取内容和有害文本。如果重复太高，模型会把容量浪费在重复模式上；如果去重太狠，又可能把行业术语、标准条款和长文中必要的固定表达误删掉。

可以用一个简化公式描述这种权衡。设 $R$ 为去重率，表示被删掉的重复比例；设 $P$ 为保留下来的数据比例，则有：

$$
P = 1 - R
$$

但训练真正关心的不是只剩多少，而是保留下来的覆盖度 $C$。覆盖度可以粗略写成：

$$
C = f(P, D)
$$

其中 $D$ 表示数据多样性。也就是说，删掉重复并不自动提升效果，只有在“保留多样性”的前提下去重，覆盖度才会上升。

玩具例子很简单。假设有 100 篇文档，其中 40 篇几乎是同一篇网页转载，20 篇是不同版本的教程，40 篇是独立内容。如果只看总量，100 篇很多；但如果不去重，模型会反复学同一段话。反过来，如果把所有措辞相近的内容都删掉，连合法保留的技术术语也会丢失。

真实工程例子是合同风险审查。第一阶段通用语料保证模型至少知道“赔偿上限”“不可抗力”“通知义务”这些概念。第二阶段长文档训练则让模型在 128K 上下文内追溯“本协议第 3.2 节所述责任限制”的实际定义位置，否则它只能局部理解，做不到跨页核对。

## 核心机制与推导

DeepSeek-V3 在数据组织上有一个关键机制：PSM 与 FIM。FIM，Fill-in-the-Middle，中间填空，意思是不给模型完整顺序文本，而是挖掉中间一段，让它根据前缀和后缀还原缺失内容。PSM，Prefix-Suffix-Middle，可以理解为把文档切成“前缀、后缀、答案中段”三部分，强迫模型学会文档级补全，而不是只会从左往右续写。

典型格式是：

`<|fim_begin|>pre<|fim_hole|>suf<|fim_end|>middle`

这里 `pre` 是前缀，`suf` 是后缀，`middle` 是被挖空后要求模型预测的内容。它的意义在于，真实工程文本经常不是“顺序续写”问题，而是“已知上下文两端，恢复中间逻辑”问题。代码补全、合同引用恢复、论文段落衔接，都属于这一类。

玩具例子可以看一段合同：

- 前缀：甲方应在收到发票后 30 日内付款。
- 后缀：若逾期超过 15 日，乙方有权暂停服务。
- 中间缺失：逾期付款应按日万分之五支付违约金。

把它改造成 PSM/FIM 后，模型训练的不是单向续写，而是“在前后条件都给定时，恢复中间约束”。这比普通 next-token prediction 更接近真实文档理解。

再往上叠加，多 Token 预测是一次预测多个后续 token，而不是只预测下一个 token。它可以理解为把“每步只走一小格”改成“每步走几格”，提高训练和解码效率。MoE，Mixture of Experts，专家混合，是把模型内部拆成多个专家子网络，每次只激活一部分。白话说，就是不是所有参数都同时工作，而是按输入类型路由到更合适的“专家”，从而在总参数很大时保持计算可控。

| 机制 | 白话解释 | 主要作用 | 对长上下文或推理的贡献 |
| --- | --- | --- | --- |
| FIM | 挖掉中间让模型补全 | 学文档内部依赖 | 提升代码、文档补全能力 |
| PSM | 用前后夹住中间答案 | 学跨段落关联 | 更适合长文档逻辑恢复 |
| 多 Token 预测 | 一次预测多个 token | 提高训练/生成效率 | 长序列训练更经济 |
| MoE 路由 | 让不同输入走不同专家 | 提高容量利用率 | STEM、代码、通用任务兼顾 |

从推导上看，长上下文能力不是单一开关，而是数据分布、样本格式、训练目标共同作用的结果。只有长文档样本，没有 FIM/PSM，模型可能“看得见很多字”但不会跨段恢复逻辑；只有 FIM/PSM，没有后期 STEM 与长文强化，模型又容易停留在短文本技巧层面。

## 代码实现

从工程管道看，数据流可以理解成四步：原始文档收集、清洗与去重、转成 PSM/FIM 训练样本、送入多 Token + MoE 训练。长上下文扩展不是一开始就直接 128K 全开，而是分两段做：先 32K，再 128K。公开披露的数字是 32K 阶段额外训练约 630B token，128K 阶段再训练约 209B token。

新手可以把这理解为“先让模型适应中长文，再让它适应超长文”。直接把超长文一股脑丢进去，成本高，也不稳定。先 32K，再 128K，更像渐进式扩容。

下面是一个可运行的玩具版 Python 例子，演示如何把文档切成 FIM 样本：

```python
def build_fim_sample(text: str, start: int, end: int) -> str:
    assert 0 <= start < end <= len(text)
    prefix = text[:start]
    middle = text[start:end]
    suffix = text[end:]
    sample = f"<|fim_begin|>{prefix}<|fim_hole|>{suffix}<|fim_end|>{middle}"
    return sample

doc = "甲方应在收到发票后30日内付款，逾期付款应支付违约金，若逾期超过15日，乙方有权暂停服务。"
sample = build_fim_sample(doc, 15, 28)

assert sample.startswith("<|fim_begin|>")
assert "<|fim_hole|>" in sample
assert "<|fim_end|>" in sample
assert "逾期付款应支付违约金" in sample

print(sample)
```

如果把它放大到训练批次，伪代码流程大致如下：

```python
def make_batch(raw_docs, context_limit):
    batch = []
    for doc in raw_docs:
        if len(doc) > context_limit:
            doc = doc[:context_limit]
        # 简化示意：从中间切一段做 FIM
        start = len(doc) // 3
        end = start + max(8, len(doc) // 6)
        sample = build_fim_sample(doc, start, min(end, len(doc)))
        batch.append(sample)
    assert len(batch) == len(raw_docs)
    return batch

docs = [
    "定义：服务中断超过4小时构成重大违约，责任限制条款见第12节。",
    "若接口返回429，客户端应执行指数退避，并记录重试次数。"
]
batch = make_batch(docs, context_limit=64)
assert len(batch) == 2
assert all("<|fim_begin|>" in x for x in batch)
```

真实工程例子可以这样理解：先把 32K 以内的技术报告、长代码文件、法律文书切成训练样本，让模型学会在“长但还可控”的窗口里保持定位能力；然后再把 128K 级别的大文档、附录、跨章节文本引入，强化超长范围内的检索和一致性。

| 阶段 | 额外训练 token 量 | 上下文长度 | 作用 |
| --- | --- | --- | --- |
| 长上下文阶段一 | 630B | 32K | 先适应中长文，降低直接扩到 128K 的不稳定性 |
| 长上下文阶段二 | 209B | 128K | 强化超长文档理解与跨页推理 |

## 工程权衡与常见坑

真正难的部分不是“多抓点数据”，而是“如何删掉坏重复，又不删掉有价值的重复”。行业文本里常见固定表达，比如“本协议自双方签署之日起生效”，在不同合同里会重复出现。单层去重如果只按 URL 或全文哈希删，很容易把结构相近但语义必要的样本一并误杀。

因此更合理的做法是多层去重。研究摘要里提到的五层机制分别是 URL、SimHash、MinHash-LSH、句向量+FAISS、动态滑窗。这里的 SimHash 是一种近似指纹算法，用来快速找“很像”的文本；MinHash-LSH 是一种近似集合匹配方法，适合找重叠片段；FAISS 是向量检索库，用来在语义层面找相似句子。

| 去重层 | 处理对象 | 主要作用 | 常见风险 |
| --- | --- | --- | --- |
| URL 去重 | 完全相同来源链接 | 删除明显镜像页 | 不识别改写转载 |
| SimHash | 高相似文本 | 快速过滤近重复 | 对长文局部重复不敏感 |
| MinHash-LSH | n-gram 重叠片段 | 识别段落级重复 | 可能误伤术语密集文本 |
| 句向量 + FAISS | 语义相近句段 | 补足表面改写重复 | 语义阈值难调 |
| 动态滑窗 | 长文局部窗口 | 处理长文内部重复 | 计算和存储成本更高 |

数学上可以把“去重收益”和“多样性损失”想成一条先升后降的曲线。设训练收益为：

$$
G(R) = aR - bR^2
$$

其中 $R$ 是去重率，$a,b>0$。前半段随着去重率上升，收益增加，因为模型少看垃圾重复；后半段收益下降，因为有价值样本也被删了。这说明去重不是越狠越好，而是要找到“重复削减”和“领域覆盖”之间的平衡点。

真实工程坑有两个最常见。第一，长文训练样本很贵，存储、切分、采样、打包都会放大成本。第二，若后期 STEM/代码样本占比拉得太高，可能损伤通用写作风格与开放领域问答表现。因此配方必须是“后期强化”，而不是“全程偏科”。

## 替代方案与适用边界

不是所有团队都需要完整复刻 DeepSeek-V3 的全流程。如果资源有限，只做 32K 阶段、减少 128K 长文训练，是可以成立的折中方案。代价是模型更适合常规问答、中等长度技术文档和一般代码任务，但在超长合同、跨附录报告、仓库级代码理解上会明显变弱。

另一个替代方向是动态长上下文调度。也就是不是所有样本都强制做长文训练，而是根据任务类型切换 PSM/FIM 比例、上下文长度和长文采样频率。这样可以降低训练成本，但对数据调度器和评估集设计要求更高。

如果推理成本敏感，还可以采用更轻量的 MoE 或更小的 dense 模型。dense 模型指每次前向都激活全部参数，结构更直接，但在同等总容量下通常不如 MoE 高效。

| 方案 | 适用场景 | 优点 | 限制 |
| --- | --- | --- | --- |
| 只做到 32K | 常规问答、一般技术助手 | 成本较低，训练更稳 | 超长文跨页推理弱 |
| 32K + 128K 全流程 | 合同审查、长报告分析、仓库级理解 | 长文能力最强 | 数据与算力成本高 |
| 动态长文调度 | 样本类型差异大 | 更节省资源 | 调度复杂，难评估 |
| 轻量 MoE / 小模型 | 在线成本敏感场景 | 推理便宜 | 总容量与上限受限 |

下面给一个简化配置片段，演示如何切换阶段：

```python
train_config = {
    "stage": "32k",
    "context_length": 32768,
    "extra_tokens_billion": 630,
    "use_fim": True,
    "moe_enabled": True,
}

def switch_stage(cfg, stage):
    if stage == "32k":
        cfg["stage"] = "32k"
        cfg["context_length"] = 32768
        cfg["extra_tokens_billion"] = 630
    elif stage == "128k":
        cfg["stage"] = "128k"
        cfg["context_length"] = 131072
        cfg["extra_tokens_billion"] = 209
    else:
        raise ValueError("unknown stage")
    return cfg

cfg = switch_stage(train_config, "128k")
assert cfg["context_length"] == 131072
assert cfg["extra_tokens_billion"] == 209
```

结论很直接：如果目标只是做“回答问题的模型”，通用高质量语料已经很重要；如果目标是做“能处理复杂长文和技术推理的模型”，就必须在后期显式增加数学、代码、STEM 和长文本，并配合更细致的去重与样本格式设计。
