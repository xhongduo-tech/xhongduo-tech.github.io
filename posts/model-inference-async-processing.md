## 核心结论

模型推理异步处理的本质，是把“请求接入”和“模型执行”解耦。同步模式是“请求来了，当场算完再返回”；异步模式是“先收下请求，给一个 `job_id`，后台排队执行，结果稍后查询或回调获取”。

它解决的主要问题，不是单次推理更快，而是系统入口不被长任务拖死。更准确地说，异步改善的是入口吞吐、连接占用、线程占用、失败隔离和高峰期可用性，不直接改善模型精度，也不减少一次推理真正需要的 GPU 计算时间。

“解耦”可以先用白话理解：前台负责收单，后台负责干活。前台不再等模型执行结束，因此 API 网关、Web 线程、连接池不会长期被一个慢请求绑住。

| 模式 | 处理方式 | 影响对象 | 典型场景 |
|---|---|---|---|
| 同步 | 收到请求后立即执行并等待结果 | 用户等待时间、连接时长 | 短问答、轻量分类、低延迟检索 |
| 异步 | 先接收请求并返回任务编号，后台再执行 | 系统可用性、入口吞吐、削峰能力 | 长文本生成、长音频转写、批量审核 |
| 流式 | 请求建立后边生成边返回部分结果 | 首字延迟、交互体验 | 聊天生成、实时摘要、逐步输出 |
| 批处理 | 多个请求聚合后一起执行 | GPU 利用率、单位成本 | 离线评估、大批量嵌入、夜间任务 |

结论边界也要说清：异步不能消灭执行时间，只是把“用户同步等待”改成“后台排队和后续取回”。如果业务要求“提交后必须立刻拿到最终答案”，异步往往不是首选。

## 问题定义与边界

这里讨论的“异步处理”，不是泛指程序里用了协程、线程池或 `async/await`。这里的重点是推理请求生命周期被拆成两个阶段：

1. 提交阶段：接收输入、生成任务、写入队列、快速返回。
2. 执行阶段：后台 worker 消费任务、调用模型、保存结果、通知调用方。

“一次推理请求”指的是：一份输入，从被系统接收开始，到拿到最终结果或失败状态结束的完整过程。这个过程不一定只包含一次函数调用，还可能包含排队、重试、状态落库、回调通知。

一个新手容易混淆的点是：“接口先返回了”不等于“系统吞吐变高了”。前台异步返回，只说明入口线程释放得更早；真正能否处理更多任务，取决于后台 worker 数量、模型服务率、队列长度和结果存储是否扛得住。

玩具例子：用户提交一段 5 万字文档，让模型生成结构化总结。同步接口需要一直挂着，可能 30 秒后才返回。异步接口则立即返回：

```json
{"job_id":"a123","status":"pending"}
```

之后用户轮询 `GET /infer/a123`，或者由系统通过 webhook 回调通知结果。这个例子里，用户“更早拿到响应”只是拿到了任务编号，不是拿到了总结内容。

| 维度 | 同步模式 | 异步模式 |
|---|---|---|
| 输入 | 请求体直接进入模型执行 | 请求体先写入任务记录 |
| 处理 | 当前连接内完成 | 进入队列后后台处理 |
| 输出 | 直接返回最终结果 | 返回 `job_id`，稍后查询或回调 |
| 是否可取消 | 较难，中途常已占用执行资源 | 可设计为任务级取消 |
| 是否可流式返回 | 可以，尤其适合生成类任务 | 可以，但要额外设计结果通道 |
| 失败处理 | 请求直接报错 | 可重试、可补偿、可延迟处理 |

哪些任务天然适合异步：

| 任务类型 | 原因 |
|---|---|
| 长文本生成 | 单次执行时间长，容易拖住入口 |
| 长音频转写 | 文件大、切片多、执行耗时高 |
| 批量内容审核 | 任务量大，允许稍后取回 |
| 大规模嵌入生成 | 更关心吞吐和成本，而非单次实时性 |

哪些任务不适合异步：

| 场景 | 原因 |
|---|---|
| 用户点击后必须立刻看到完整答案 | 异步会增加交互步骤 |
| 强事务一致流程 | 后台延迟执行会破坏同步提交语义 |
| 极短请求且低延迟是核心指标 | 引入队列和状态机会放大复杂度 |

流式输出和异步也不是同义词。流式是“边算边发”；异步是“先收后算”。一个系统可以同步且流式，也可以异步但不流式，还可以异步提交后在结果阶段提供流式拉取。

## 核心机制与推导

一个典型异步推理系统通常有四层：

1. 提交层：HTTP API、鉴权、参数校验、生成 `job_id`。
2. 队列层：缓存待执行任务，完成削峰填谷。
3. 执行层：worker 拉取任务并调用模型。
4. 结果层：保存状态、结果、错误信息，并支持查询或回调。

“队列”可以先用白话理解：它是排队区，不负责做事，只负责决定谁先做、谁后做、是否重试。

“worker”可以理解成后台工人。它真正消耗 GPU 或 CPU 资源，决定系统的实际处理能力。

定义几个常用符号：

| 符号 | 含义 |
|---|---|
| $\lambda$ | 请求到达率，每秒进入多少任务 |
| $\mu$ | 单个 worker 的服务率，每秒能完成多少任务 |
| $c$ | worker 数量 |
| $\rho$ | 系统利用率，表示忙碌程度 |
| $T_{submit}$ | 提交层耗时 |
| $T_{queue}$ | 排队耗时 |
| $T_{exec}$ | 模型执行耗时 |
| $T_{cb}$ | 查询或回调取回结果的附加耗时 |

系统稳定性的一个基本条件是：

$$
\rho = \frac{\lambda}{c\mu}
$$

当 $\rho < 1$ 时，长期平均上系统才有可能稳定；当 $\rho \ge 1$ 时，任务进入速度大于或等于处理速度，队列会持续积压。这里的“利用率”白话理解就是：后台工人到底忙到什么程度，1 代表满负荷。

请求总响应时间可以拆成：

$$
T_{resp} = T_{submit} + T_{queue} + T_{exec} + T_{cb}
$$

异步真正优化的是让前台用户先感知到很小的 $T_{submit}$，而不是让 $T_{exec}$ 神奇消失。若高峰期队列很长，最终完成时间依然可能很差。

再看 Little 定律：

$$
L = \lambda W
$$

其中 $L$ 是系统里的平均任务数，$W$ 是任务平均停留时间。白话讲：任务来得越快，或者每个任务待得越久，系统里堆着的任务就越多。

最小数值例子：

- 每个请求真正执行模型要 500ms，因此单 worker 服务率 $\mu = 2$/秒。
- 前台受理只要 10ms，因此用户很快能拿到 `job_id`。
- 如果每秒来了 3 个请求，则 $\lambda = 3$/秒。

情况 1：只有 1 个 worker，$c = 1$。

$$
\rho = \frac{3}{1 \times 2} = 1.5
$$

$\rho > 1$，说明系统一定积压。前台看起来“很快返回”，但后台队列会越来越长，最终用户查询结果时还是慢。

情况 2：增加到 2 个 worker，$c = 2$。

$$
\rho = \frac{3}{2 \times 2} = 0.75
$$

这时长期平均可稳定，排队显著下降。这里可以直接看到：异步本身不增加算力，真正让系统稳住的是“解耦 + 足够的后台服务能力”。

时延拆分也可以列表看：

| 阶段 | 典型耗时 | 说明 |
|---|---|---|
| 前台受理 | 5ms - 30ms | 鉴权、校验、写任务记录 |
| 排队等待 | 0ms - 数十秒 | 高峰期最不稳定的部分 |
| 模型执行 | 100ms - 数分钟 | 真正的计算成本 |
| 结果取回 | 1ms - 数百ms | 查询存储或发送回调 |

真实工程例子：一个长音频转写服务接收 30 分钟录音。上传结束后，API 立即返回任务号；后台先切片，再并行转写，再合并文本和时间戳。这里真正耗时的不是“API 返回”，而是切片、推理、重试、合并四段流水线。异步的价值在于：上传入口和转写流水线分离，服务不会因为单个长音频把前台线程占满。

## 代码实现

实现异步推理，核心不是把函数改成 `async`，而是把“提交路径”和“执行路径”拆开，并引入完整状态机。

常见任务状态机如下：

| 状态 | 含义 |
|---|---|
| `pending` | 已接收，等待执行 |
| `running` | worker 已领取，正在执行 |
| `success` | 执行成功，结果可读取 |
| `failed` | 执行失败，可附带错误原因 |
| `canceled` | 被用户或系统取消 |

最小流程可以写成：

| 步骤 | 行为 |
|---|---|
| 1 | `POST /infer` 接收输入 |
| 2 | 生成 `job_id`，写任务表与队列 |
| 3 | 立即返回 `job_id` |
| 4 | worker 从队列取任务 |
| 5 | 更新状态为 `running` |
| 6 | 执行模型推理 |
| 7 | 保存结果或错误，更新最终状态 |
| 8 | 用户轮询 `GET /infer/{job_id}` 或接收回调 |

下面给一个可运行的 Python 玩具实现，重点是机制，不依赖真实 HTTP 服务：

```python
import time
import uuid
from queue import Queue, Empty
from threading import Thread, Lock

class AsyncInferService:
    def __init__(self):
        self.jobs = {}
        self.queue = Queue()
        self.lock = Lock()
        self.worker = Thread(target=self._worker_loop, daemon=True)
        self.worker.start()

    def submit(self, payload, request_id=None):
        dedup_key = request_id or str(uuid.uuid4())
        with self.lock:
            for job_id, job in self.jobs.items():
                if job["dedup_key"] == dedup_key:
                    return job_id
            job_id = str(uuid.uuid4())
            self.jobs[job_id] = {
                "status": "pending",
                "payload": payload,
                "result": None,
                "error": None,
                "dedup_key": dedup_key,
            }
        self.queue.put(job_id)
        return job_id

    def get(self, job_id):
        return self.jobs[job_id]

    def cancel(self, job_id):
        with self.lock:
            if self.jobs[job_id]["status"] == "pending":
                self.jobs[job_id]["status"] = "canceled"
                return True
        return False

    def _infer(self, payload):
        time.sleep(0.05)
        text = payload["text"]
        return {"length": len(text), "upper": text.upper()}

    def _worker_loop(self):
        while True:
            try:
                job_id = self.queue.get(timeout=0.1)
            except Empty:
                continue

            with self.lock:
                job = self.jobs.get(job_id)
                if not job or job["status"] == "canceled":
                    self.queue.task_done()
                    continue
                job["status"] = "running"

            try:
                result = self._infer(job["payload"])
                with self.lock:
                    job["status"] = "success"
                    job["result"] = result
            except Exception as e:
                with self.lock:
                    job["status"] = "failed"
                    job["error"] = str(e)
            finally:
                self.queue.task_done()

service = AsyncInferService()
job_id = service.submit({"text": "hello"}, request_id="req-1")
same_job_id = service.submit({"text": "hello"}, request_id="req-1")

assert job_id == same_job_id  # 幂等去重：重复提交不重复建任务

for _ in range(20):
    status = service.get(job_id)["status"]
    if status == "success":
        break
    time.sleep(0.02)

job = service.get(job_id)
assert job["status"] == "success"
assert job["result"]["length"] == 5
assert job["result"]["upper"] == "HELLO"
```

这段代码体现了四个关键点：

1. 提交和执行分离。
2. `job_id` 作为任务句柄。
3. 状态机显式可查。
4. 用 `request_id` 做幂等去重，避免客户端重试导致重复执行。

工程里还要多补几层：

- 队列选择：内存队列适合单机实验；持久队列适合生产，worker 崩溃后任务不丢。
- 结果存储：至少要存状态、结果、错误码、创建时间、更新时间。
- 重试语义：只有幂等任务才能安全重放。
- 取消语义：已经开始占用 GPU 的任务，取消不一定能立即生效，通常是“尽快停止”。
- 部分结果：生成类任务如果想支持中间结果，需要额外的 token 流存储或事件通道。

“前台只负责收单和发票号，后台像工厂生产线一样慢慢做”这个类比对新手够用了，但工程上还必须再加一句：收单系统和生产线之间要有可观测、可恢复、可限流的协议，否则系统高峰时会直接失控。

## 工程权衡与常见坑

异步不是免费收益。它把一部分前台阻塞问题，换成了后台状态管理问题。你少等了一个长连接，但多了队列、重试、取消、幂等、结果过期、告警和监控。

真实工程例子：白天业务高峰时，请求平均到达率从每秒 20 个升到 60 个，但后台只配置了 20 个 worker，每个 worker 平均每秒完成 2 个请求，总服务能力是每秒 40 个。平均延迟在低峰时很好，高峰时前台仍然能立刻返回 `job_id`，但 P95/P99 完成时延急剧恶化，因为积压开始出现。此时如果只看“API 平均响应 15ms”，会误以为系统很健康，实际上用户要很久才拿到结果。

| 坑 | 现象 | 规避 |
|---|---|---|
| 把异步误当并行 | 前台返回很快，但整体完成量没变 | 分开看提交延迟和完成吞吐 |
| 队列无上限 | 内存上涨、尾延迟失控、任务过期 | 设队列上限、拒绝策略、限流 |
| 忽略取消 | 用户走了，后台还在白算 | 设计取消状态和检查点 |
| 只看平均延迟 | 平均值正常，P95/P99 很差 | 强制监控尾延迟和队列长度 |
| 没有幂等设计 | 客户端重试造成重复推理 | 用请求指纹或 `request_id` 去重 |
| 结果不设过期 | 存储被历史结果撑爆 | 设置 TTL 和归档策略 |
| worker 崩溃不重放 | 任务永久卡在 `running` | 心跳、租约、超时回收 |
| 共享状态乱写 | 结果覆盖、顺序错乱 | 用状态机和版本控制更新 |

工程权衡通常落在这几组问题上：

| 决策点 | 低复杂度方案 | 高可靠方案 |
|---|---|---|
| 队列 | 内存队列 | 持久消息队列 |
| 结果存储 | 进程内字典 | Redis / 数据库 / 对象存储 |
| 失败处理 | 直接失败 | 分级重试 + 死信队列 |
| 取消 | 不支持 | 任务级取消 + 执行中断 |
| 观测 | 只看接口日志 | 看队列、状态机、尾延迟、丢失率 |

一个常见误判是：模型函数本身已经很慢，所以只要做异步就行。实际上生产事故常出在外围，不在模型本体。比如 webhook 回调超时、结果库写满、任务重试风暴、状态表热点更新，这些都会把系统拖垮。

## 替代方案与适用边界

异步不是唯一解。很多场景里，更有效的办法是批处理、连续批处理、多副本扩容、模型蒸馏，或者直接换更小的模型。

“连续批处理”可以先用白话理解：不是等一整批凑齐再一起跑，而是让请求在执行过程中动态插入批次，尽量提高 GPU 利用率。它常见于大模型在线生成服务。

下面对比几种方案：

| 方案 | 是否立即返回最终结果 | 是否支持长任务 | 是否允许部分结果 | 是否偏向高吞吐 | 是否对尾延迟敏感 |
|---|---|---|---|---|---|
| 同步 | 是 | 一般较差 | 一般不支持 | 中 | 高 |
| 异步 | 否 | 强 | 支持，但要额外设计 | 强 | 中 |
| 流式 | 否，先返回部分结果 | 中 | 强 | 中 | 高 |
| 批处理 | 否 | 强 | 不强调 | 很强 | 低 |
| 连续批处理 | 是或部分是 | 中 | 可配合流式 | 很强 | 中 |

实时问答和长音频转写就是两个典型边界：

- 实时问答：用户发出问题后，希望立刻看到答案的增长过程。这种场景更适合同步加流式，或者同步加连续批处理。纯异步会多出“提交任务再查询”的交互步骤，体验通常更差。
- 长音频转写：用户上传一个小时录音，本来就接受等待几分钟。此时异步更合适，因为任务长、可切片、可重试，也更适合与对象存储、消息队列、回调机制配合。

如果你的业务满足下面任一条件，优先考虑异步：

| 条件 | 原因 |
|---|---|
| 单次任务执行时间长 | 避免入口长期占用 |
| 允许稍后拿结果 | 业务可接受任务式交互 |
| 高峰流量明显波动 | 队列可用于削峰 |
| 失败后需要自动重试 | 后台任务系统更容易补偿 |
| 任务可取消或分阶段执行 | 状态机能发挥价值 |

如果你的业务满足下面任一条件，异步可能不是首选：

| 条件 | 原因 |
|---|---|
| 必须一次请求内拿到最终答案 | 异步增加系统和交互复杂度 |
| 对首字延迟极敏感 | 更适合同步流式 |
| 请求极短且稳定 | 队列带来的额外开销可能不值 |
| 强事务一致要求高 | 后台执行难保持同步语义 |

最后可以收束成一句话：异步适合“任务长、允许稍后取回、入口稳定性比即时完成更重要”的推理系统；不适合“必须立刻给最终结果、交互回路极短、同步语义很强”的系统。
