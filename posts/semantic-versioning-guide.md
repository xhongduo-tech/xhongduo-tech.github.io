## 核心结论

语义化版本控制（Semantic Versioning，简称 SemVer，意思是“用版本号表达兼容性含义”）的核心目标，不是把版本号排得整齐，而是让使用者一眼看出这次升级有没有风险。它把版本号写成：

$$
\text{MAJOR.MINOR.PATCH}
$$

其中 `MAJOR` 是主版本，表示破坏性变更；`MINOR` 是次版本，表示向后兼容的新功能；`PATCH` 是补丁版本，表示向后兼容的缺陷修复。

从 `0.1.0` 到 `1.0.0` 的本质，不是“功能终于做完了”，而是“维护者开始对外承诺：核心接口进入稳定阶段”。可以把它理解成承诺等级：

| 版本阶段 | 含义 | 使用者应如何理解 |
|---|---|---|
| `0.x.y` | 试验阶段 | 接口可能频繁变化，不宜假设长期稳定 |
| `1.x.y` | 稳定阶段 | 升级规则应按兼容性严格执行 |
| `2.x.y` 及以上 | 稳定基础上的重大演进 | 有明确破坏性升级，需要迁移 |

一个最小判断表如下：

| 变化类型 | 该升哪一位 |
|---|---|
| breaking change，破坏兼容 | `MAJOR++` |
| feature added，新增兼容功能 | `MINOR++` |
| bug fix，兼容性修复 | `PATCH++` |

玩具例子很简单：`0.3.2 -> 0.3.3`，通常是修 bug；`0.3.2 -> 0.4.0`，通常是新增功能；`0.9.0 -> 1.0.0`，表示维护者认为接口规则已经足够稳定，值得外部长期依赖。

## 问题定义与边界

语义化版本控制要解决的问题，是把“升级会不会把旧代码弄坏”这个信息编码进版本号。版本号因此不是内部流水号，而是一种沟通协议。

这里的“兼容性”可以先用一句白话解释：旧代码在不修改调用方式的前提下，升级后还能正常工作，就叫向后兼容。反过来，如果升级后旧代码要改，甚至直接报错，就是破坏兼容。

例如，一个 API 原来返回：

```json
{ "name": "Alice" }
```

后来改成：

```json
{ "full_name": "Alice" }
```

如果调用方仍然读取 `name`，代码就会失效。这不是普通迭代，而是破坏兼容，应该升主版本。

相反，如果只是把错误提示从 `"bad request"` 改成 `"email is required"`，参数和返回结构都没变，那么用户代码通常不需要修改，这更接近补丁版本升级。

本文边界很明确：只讨论对外可见版本号的语义，不讨论内部构建号、日期号、提交哈希，也不讨论营销式命名，比如“2026 春季版”这种不直接表达兼容性的标识。

| 变化类型 | 旧代码是否可能继续工作 | 对应版本位 |
|---|---|---|
| 修复 bug | 是 | `PATCH` |
| 新增功能 | 是 | `MINOR` |
| 破坏兼容 | 否 | `MAJOR` |

对零基础读者来说，最重要的认识只有一句：版本号不是告诉你“作者改了多少”，而是告诉你“你升级后要不要改代码”。

## 核心机制与推导

SemVer 的判断逻辑不是拍脑袋，而是从兼容性推导出来的。只要先问一个问题：旧代码升级后还能不能工作？后续决策就清楚了。

规则可以写成：

$$
\text{breaking change} \Rightarrow \text{MAJOR} = \text{MAJOR} + 1,\ \text{MINOR}=0,\ \text{PATCH}=0
$$

$$
\text{compatible feature} \Rightarrow \text{MINOR} = \text{MINOR} + 1,\ \text{PATCH}=0
$$

$$
\text{compatible fix} \Rightarrow \text{PATCH} = \text{PATCH} + 1
$$

对应到人能执行的判断流程，就是：

```text
if breaking_change:
    major += 1; minor = 0; patch = 0
elif new_feature:
    minor += 1; patch = 0
else:
    patch += 1
```

一个玩具例子：

- `0.1.0 -> 0.1.1`：修复输入为空时崩溃的问题，调用方式不变，这是补丁升级。
- `0.1.0 -> 0.2.0`：新增一个可选参数 `timeout`，旧调用仍能跑，这是次版本升级。
- `0.1.0 -> 1.0.0`：接口文档、返回结构、异常规则都收敛下来，维护者愿意承担稳定承诺，这是进入稳定期。

一个更典型的机制例子是函数签名变化。假设原始接口是：

```js
createUser(name)
```

后来改成：

```js
createUser({ name, email })
```

旧调用 `createUser("Alice")` 会直接失效，因此这是主版本升级条件，而不是“只是参数更合理了，所以升个 minor”。

| 变更 | 兼容旧代码 | 升级规则 |
|---|---|---|
| 改参数类型 | 否 | 主版本 |
| 新增可选字段 | 是 | 次版本 |
| 修复边界错误 | 是 | 补丁版本 |

需要额外强调的是 `0.x` 阶段。规范允许 `0.y.z` 被视为“初始开发阶段”，这意味着接口仍可能变化。现实里很多项目会在 `0.x` 期间快速试错，这是合理的；但如果一个库已经被外部广泛依赖，还长期停在 `0.x`，就会让使用者难以判断它到底有没有稳定承诺。

## 代码实现

把版本规则落到代码里，重点不是算法复杂，而是判断口径稳定。下面是一个最小可运行的 Python 实现：输入当前版本号和变更类型，输出下一个版本号。

```python
def bump_version(version: str, change_type: str) -> str:
    major, minor, patch = map(int, version.split("."))

    if change_type == "breaking":
        return f"{major + 1}.0.0"
    if change_type == "feature":
        return f"{major}.{minor + 1}.0"
    if change_type == "fix":
        return f"{major}.{minor}.{patch + 1}"
    raise ValueError("change_type must be breaking, feature, or fix")


assert bump_version("0.1.0", "fix") == "0.1.1"
assert bump_version("0.1.0", "feature") == "0.2.0"
assert bump_version("0.1.0", "breaking") == "1.0.0"
assert bump_version("1.4.2", "breaking") == "2.0.0"
```

这个例子很小，但已经表达了 SemVer 的核心：版本变化由变更性质决定，而不是由作者心情决定。

如果要更贴近工程实践，可以在发布脚本里先要求维护者声明变更类型，再自动计算版本号。比如 CI（持续集成，意思是“每次改动都自动执行检查和发布流程”）里要求提交信息带上 `fix`、`feat`、`breaking` 标签，再统一生成新版本和 CHANGELOG（变更日志，意思是“记录每次发布改了什么”）。

输入与输出关系可以先固定成表：

| 输入变化类型 | 示例 | 输出结果 |
|---|---|---|
| `fix` | 修复分页越界 | `1.2.3 -> 1.2.4` |
| `feature` | 新增导出接口 | `1.2.3 -> 1.3.0` |
| `breaking` | 删除旧参数 | `1.2.3 -> 2.0.0` |

真实工程例子可以看机器学习库。ML 库的“接口”不只包括 Python 函数签名，还包括模型权重格式、tokenizer 行为、默认推理逻辑。比如：

- 如果一个库把默认权重格式从 `bin` 切到 `safetensors`，并且旧加载逻辑失效，这本质上是破坏兼容。
- 如果 tokenizer 的分词行为变化，导致相同输入得到不同 token 序列，进而影响推理结果，即使函数名没变，也应按主版本处理。

这类变化之所以特殊，是因为用户依赖的不是“函数长什么样”，而是“结果是否可复现”。对 ML 生态来说，“行为兼容”往往和“API 兼容”同样重要。

## 工程权衡与常见坑

版本号治理不是越保守越好，也不是越激进越显得专业。核心权衡是：既要让升级风险被真实表达，又不能把版本管理做成没人愿意维护的形式主义。

最常见的坑有三类。

| 常见坑 | 后果 | 规避方式 |
|---|---|---|
| 把破坏性改动放到 `MINOR` | 用户升级后报错 | 先判断旧代码是否还能跑 |
| 长期停留在 `0.x` | 用户不敢依赖 | 接口稳定后尽快进入 `1.0.0` |
| `1.0.0` 后随意改 API | 信任受损 | 提供弃用期和迁移指南 |

第一个坑最常见。比如原来参数 `verbose=False` 是可选的，后来你把它改成必填；或者原来返回列表，后来改成生成器。维护者可能觉得“功能更合理了”，但对用户来说，旧代码已经坏了，这就不是 minor，而是 major。

第二个坑是“永远不发 1.0.0”。有些项目担心主版本之后责任更重，于是长期停在 `0.8.x`、`0.9.x`。结果不是更灵活，而是让用户怀疑：这个项目到底有没有稳定接口，适不适合放进生产环境。

第三个坑是误把 `1.0.0` 理解成“以后绝不会变”。这也不对。`1.0.0` 的真正含义不是“不再变化”，而是“变化要按兼容性规则表达”。你仍然可以做重大重构，但需要通过 `2.0.0`、弃用警告、迁移文档来管理风险。

对工程团队来说，比较稳妥的做法是：

- 用 `CHANGELOG` 说明每次发布改了什么。
- 先发 deprecation warning（弃用警告，意思是“这个旧接口还能用，但未来会删”）。
- 给旧 API 一个过渡期，再在下个主版本删除。
- 对真实用户升级一次，验证迁移文档是否足够。

## 替代方案与适用边界

语义化版本适合“用户关心升级风险”的场景，尤其是公共库、SDK、开放 API。这时版本号本质上是兼容性合同。

但并不是所有项目都适合 SemVer。一个一次性脚本、内部日报工具、纯静态数据包，使用日期版本或提交哈希，可能更直接。因为这类产物的核心问题不是“升级会不会坏”，而是“这是哪一天的版本”。

| 方案 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| 语义化版本 | 公共库、SDK、API | 兼容性清晰 | 需要严格治理 |
| 日期版本 | 持续发布产品 | 容易追踪时间 | 不表达兼容性 |
| 构建号 | 内部流水线 | 简单直接 | 语义弱 |

一个新手能立刻用上的判断标准是：

- 如果用户最关心“升级会不会坏”，优先用语义化版本。
- 如果用户最关心“这是哪次发布”，日期版本也许更合适。
- 如果产物根本不对外承诺稳定接口，强行套 SemVer 往往只会制造噪音。

再看一个对比例子。公开 SDK 面向大量外部开发者，最重要的信息是“从 `1.4.2` 升到 `1.5.0` 是否安全”，这时 SemVer 很有价值。内部数据处理脚本只在团队里每天跑一次，使用 `2026.04.29` 反而更清楚，因为大家更关心发布时间而不是 API 合同。

结论可以压缩成一句：如果读者关心升级风险，就优先语义化版本；如果只关心发布时间，语义版本未必最合适。
