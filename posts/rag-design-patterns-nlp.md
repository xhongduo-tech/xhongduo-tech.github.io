## 核心结论

检索增强生成，英文是 Retrieval-Augmented Generation，简称 RAG。它的白话解释是：模型先去外部资料库里找证据，再基于这些证据生成回答，而不是只靠参数里“记住”的旧知识。

RAG 解决的核心问题不是“让模型更会写”，而是“让模型在需要事实时少猜、多引证”。它特别适合知识密集型 NLP 任务，例如问答、文档摘要、代码搜索、接口说明生成、客服知识库回答。原因很直接：这些任务的正确性高度依赖外部事实，而事实会更新，模型参数不会实时更新。

RAG 的效果主要取决于三个设计变量是否匹配任务：

| 设计变量 | 本质问题 | QA 常见选择 | 摘要常见选择 | 代码搜索常见选择 |
|---|---|---|---|---|
| 检索粒度 | 一次检索多大文本块 | 句子/短段 | 章节/长段 | 函数/类/文件片段 |
| 排名策略 | 哪些片段最该进生成器 | 召回后 rerank | 混合检索后压缩 | 稀疏+向量+rerank |
| 融合方式 | 检索结果怎样交给模型 | 早融合或 FiD | 宏观 chunk + 早融合 | 早融合或后融合 |

一个最小玩具例子：产品文档被切成多个段落，用户问“这个 API 返回什么”。系统先检索出最相关的 2 段，把它们拼到 prompt 前面，再让模型作答。这样模型输出时引用的是文档里的描述，不是“猜测的 API 行为”。

三种常见融合方式可以先记住一个结论：

| 融合方式 | 输入结构 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| 早融合 Early Fusion | 直接拼成 `[query + doc1 + doc2 + ...]` | 实现最简单 | 容易超长、噪声直接进 prompt | 小规模 QA、知识库问答 |
| 后融合 Late Fusion | 分别计算每个文档条件下的答案，再加权 | 抗噪声更强 | 推理复杂，工程实现更难 | 高精度事实问答 |
| FiD | 每个片段独立编码，解码时跨片段融合 | 能保留更多证据 | 算力和显存成本高 | 多证据问答、复杂知识整合 |

## 问题定义与边界

可以把 RAG 定义为一个分阶段系统：给定查询 $q$，先从外部语料集合中检索相关片段 $d_1,d_2,\dots,d_k$，再把这些片段作为条件上下文提供给生成器，输出答案 $y$。这里“检索”就是从资料库中找最相关内容，“生成器”就是根据输入文本续写答案的大语言模型。

它和“模型直接回答”最大的差异在于信息来源：

1. 纯生成主要依赖模型参数中的历史知识。
2. RAG 依赖“参数知识 + 检索到的当前文本证据”。
3. 因此 RAG 更适合处理会变化、必须准确、最好可引用来源的问题。

一个新手友好的理解方式是：先搜索百科页取出相关段落，再把段落交给写作助手。搜索阶段负责补事实，生成阶段负责组织语言。这两个阶段职责不同，不能混为一谈。

RAG 不是所有 NLP 任务都应该默认使用。它的边界主要有四类：

| 任务类型 | 是否适合 RAG | 原因 |
|---|---|---|
| 实时知识问答 | 很适合 | 需要最新事实，模型参数可能过时 |
| 长文摘要 | 适合，但要重设计粒度 | 需要宏观结构，不能只取碎片 |
| 代码搜索与代码生成 | 很适合 | 依赖仓库上下文、接口定义、现有实现 |
| 纯创意写作 | 通常不必 | 外部检索不一定提供有效增益 |

如果只看流程，RAG 可以抽象成：

`query -> 检索 -> rerank -> 生成`

这里 rerank 是“重排序”，白话解释是：先粗筛一批候选，再用更精细但更贵的模型把最相关片段排到前面。很多系统失败，不是因为生成器差，而是因为相关证据根本没排进前几名。

不同任务通常有不同的检索粒度和 top-k 范围：

| 任务 | 推荐粒度 | 常见 top-k | 重点风险 |
|---|---|---|---|
| 问答 | 句子/短段落 | 3-5 | 证据太少会漏答，太多会引噪 |
| 摘要 | 章节/长段 | 5-10 | 片段太碎会丢主题结构 |
| 代码检索 | 函数块/类/文件摘要 | 5-20 | 只看向量相似度容易忽略符号精确匹配 |

真实工程例子：一个团队做 API 文档助手。用户问“批量上传接口失败时返回什么错误码”。如果只靠 LLM 参数回答，它可能编出常见 HTTP 状态码；如果用 RAG，它会先从最新接口文档、错误码表、SDK 注释中检索相关片段，再输出“业务错误码 + HTTP 状态码 + 触发条件”。这类问题就明显属于 RAG 的适用边界。

## 核心机制与推导

RAG 的第一步是把查询和文档片段放到同一个可比较空间中。最常见方法是向量检索，也就是把文本编码成向量，然后比较它们的方向是否接近。方向接近通常意味着语义相似。

若查询向量为 $q$，文档片段向量为 $d_i$，余弦相似度定义为：

$$
\cos(q,d_i)=\frac{q\cdot d_i}{\|q\|\|d_i\|}
$$

值越大，通常表示越相关。检索器会按这个分数从大到小取 top-k 个片段。

训练检索器时，一个典型目标是 InfoNCE loss。它的白话意思是：让正确文档离查询更近，让错误文档更远。其形式可以写成：

$$
\mathcal{L}_{\mathrm{ret}}=-\frac{1}{N}\sum_{i=1}^N\log\frac{\exp(s(q_i,d_i^+)/\tau)}{\exp(s(q_i,d_i^+)/\tau)+\sum_{j\neq i}\exp(s(q_i,d_j^-)/\tau)}
$$

其中：
- $d_i^+$ 是正样本，也就是正确证据。
- $d_j^-$ 是负样本，也就是不相关片段。
- $\tau$ 是温度参数，用来控制分布尖锐程度。

这个公式表达的不是“把分数做大”这么简单，而是“让正确答案在一堆候选里相对更突出”。因此检索质量不仅依赖编码器本身，也依赖负样本是否足够难。

再看一个玩具例子。用户问：“`create_order` API 成功时返回什么字段？” 检索得到三个候选：

| 片段 | 分数 |
|---|---|
| A: 成功返回 `order_id,status,created_at` | 0.92 |
| B: 失败时返回 `error_code,message` | 0.81 |
| C: 鉴权方式说明 | 0.65 |

如果 top-k=2，则取 A 和 B。

这时三种融合策略的差异就出现了：

1. 早融合：把 query、A、B 直接拼接到同一个 prompt。优点是实现简单，缺点是 B 虽然相关但方向偏了，可能把“失败字段”也带进答案。
2. 后融合：分别计算 $P(y|q,A)$ 和 $P(y|q,B)$，再按检索分数加权。这样 A 的影响更大，B 不容易污染主答案。
3. FiD：每个片段先分别编码，解码每一步都能参考多个片段。它比早融合更能处理多证据问题。

可以把后融合写成一个抽象形式：

$$
P(y|q,D)\approx \sum_{i=1}^{k} w_i P(y|q,d_i), \quad \sum_i w_i=1
$$

这里 $w_i$ 可以由相似度分数归一化得到。它的含义是：答案不是只看一个文档，而是对多个文档的条件分布做组合。

三种融合方式的推理差异如下：

| 方式 | 推理步骤 | 对上下文长度的压力 | 算力成本 | 典型问题 |
|---|---|---|---|---|
| 早融合 | 拼接后一次生成 | 高 | 低 | 冗余片段直接污染 prompt |
| 后融合 | 每片段独立推理再聚合 | 低 | 中到高 | 实现复杂，延迟上升 |
| FiD | 独立编码，解码跨片段注意力 | 中 | 高 | 成本大，工程门槛高 |

从机制上看，RAG 系统成败通常不在“有没有检索”，而在“检索误差如何传导到生成器”。若前 3 个候选里没有真正证据，再强的生成器也只能在错误上下文里组织语言。因此 RAG 是一个链式系统，前一步的误差会向后传播。

## 代码实现

下面给出一个可运行的最小 Python 示例。它不依赖外部向量库，而是用词集合重叠模拟检索分数，目的是把 RAG 的控制流程讲清楚：切片、检索、rerank、拼接 prompt、生成。

```python
from math import isclose

docs = [
    {"id": "a", "text": "create_order 成功时返回 order_id status created_at"},
    {"id": "b", "text": "create_order 失败时返回 error_code message"},
    {"id": "c", "text": "所有请求都需要 Authorization 头"},
]

def tokenize(text: str):
    return set(text.lower().split())

def score(query: str, doc_text: str) -> float:
    q = tokenize(query)
    d = tokenize(doc_text)
    if not q or not d:
        return 0.0
    return len(q & d) / len(q | d)

def retrieve(query: str, docs, top_k: int = 2):
    scored = []
    for doc in docs:
        s = score(query, doc["text"])
        scored.append((s, doc))
    scored.sort(key=lambda x: x[0], reverse=True)
    return scored[:top_k]

def rerank(query: str, candidates):
    # 模拟 reranker：如果 query 里含有“成功”，优先含有“成功”的片段
    boosted = []
    for s, doc in candidates:
        bonus = 0.2 if ("成功" in query and "成功" in doc["text"]) else 0.0
        boosted.append((s + bonus, doc))
    boosted.sort(key=lambda x: x[0], reverse=True)
    return boosted

def build_prompt(query: str, ranked):
    context = "\n".join([f"[证据{i+1}] {doc['text']}" for i, (_, doc) in enumerate(ranked)])
    return f"问题：{query}\n{context}\n请基于证据回答。"

query = "create_order 成功时返回什么"
topk = retrieve(query, docs, top_k=2)
ranked = rerank(query, topk)
prompt = build_prompt(query, ranked)

assert ranked[0][1]["id"] == "a"
assert "[证据1]" in prompt
assert "create_order 成功时返回" in prompt
assert isclose(score(query, docs[0]["text"]) > score(query, docs[2]["text"]), True)
```

这个例子虽然简化，但对应真实系统中的四个部件：

| 部件 | 例子里的函数 | 真实工程替代 |
|---|---|---|
| 切词/编码 | `tokenize` | embedding 模型、BM25 tokenizer |
| 初筛检索 | `retrieve` | FAISS、Elasticsearch、Milvus |
| 重排序 | `rerank` | cross-encoder、LLM reranker |
| 融合输入 | `build_prompt` | early fusion / FiD / late fusion |

如果换成更接近生产的伪代码，通常会写成：

```python
def rag_answer(query, retriever, reranker, llm, top_k=5):
    candidates = retriever.search(query, top_k=20)  # 先多召回
    ranked = reranker.rank(query, candidates)[:top_k]  # 再压到生成器可处理的数量
    prompt = render_prompt(query, ranked)  # early fusion；若用 FiD 则改成多路输入
    answer = llm.generate(prompt)
    return answer, ranked
```

这里 `top_k=20 -> rerank -> 保留5` 是常见工程模式。原因是第一阶段检索追求召回率，第二阶段重排序追求精度。如果一开始只检索 3 个，相关片段可能根本没进候选集。

真实工程例子是代码搜索。假设开发者问：“项目里哪里定义了支付重试逻辑？” 一个可用的 Code RAG 系统通常不会只搜 README，而会同时检索：
- 函数和类定义
- API 文档
- 调用链附近的代码片段
- 单元测试里的断言

然后做混合检索：BM25 负责精确匹配符号名，向量检索负责找语义相似实现，reranker 再把真正与“支付重试”相关的函数块提到前面。最后生成器才能基于仓库上下文解释“在哪、为什么、如何调用”。

## 工程权衡与常见坑

RAG 的难点主要不在概念，而在参数和任务之间的匹配。最常见的四个坑是 top-k、粒度、rerank 和 prompt 长度。

先看最关键的权衡：

| 维度 | 设得太小 | 设得太大 | 常见调法 |
|---|---|---|---|
| top-k | 漏掉关键证据 | 噪声变多、延迟上升 | 先召回 20，再 rerank 到 3-8 |
| chunk 粒度 | 片段太短，上下文不全 | 片段太长，主题混杂 | QA 用短段，摘要用长段，代码用函数块 |
| rerank 强度 | 错误证据排前 | 成本高、延迟高 | 只对前几十个候选细排 |
| prompt 长度 | 信息不足 | 截断或注意力稀释 | 长文任务改 FiD 或摘要压缩 |

一个经典坑是：为了“保险”，把 top-k 从 3 调到 10。结果 prompt 里混入多条相似但冲突的描述，模型为了把这些内容整合成一句话，反而开始编造中间解释。这就是很多人误以为“RAG 也会幻觉”的来源。更准确地说，是“低质量检索 + 过量上下文”诱发了生成器幻觉。

不同任务对粒度要求非常不同：

- QA 通常是句子或段落粒度，因为答案往往藏在局部事实里。
- 摘要更适合章节级 chunk，因为它需要整体结构，而不是零散事实点。
- 代码搜索更适合函数块、类定义或带注释的代码片段，因为代码语义往往跨多行。

可以把常见坑和规避方式压成一张表：

| 常见坑 | 现象 | 根因 | 规避措施 |
|---|---|---|---|
| top-k 过大 | 回答变啰嗦甚至矛盾 | 噪声片段进入生成器 | 设相似度阈值，先多召回再细排 |
| 粒度不匹配 | 找到的片段看似相关但答不准 | 信息窗口与任务不一致 | 按任务重设计 chunk 策略 |
| 不做 rerank | 最相关证据排在后面 | 初筛模型只会粗匹配 | 对前 20-50 个候选做重排序 |
| 过度早融合 | prompt 超长，引用混乱 | 所有证据都平铺给模型 | 用 FiD、后融合或先摘要再生成 |
| 只做向量检索 | 符号名、错误码命中差 | 语义相似不等于精确匹配 | 稀疏+致密混合检索 |
| 忽略评估 | 系统看似能答，实际上不稳定 | 只看主观样例，不看指标 | 分开评估召回率、MRR、答案准确率 |

工程上还要注意一个容易被忽略的问题：检索结果之间是否相互一致。比如文档 A 是旧版本，文档 B 是新版本，如果二者同时进入 prompt，模型并不知道哪个更权威。因此真实系统经常会加元数据过滤，例如版本号、发布时间、仓库分支、文档来源优先级。

## 替代方案与适用边界

RAG 不是唯一方案。至少还有两类常见替代路径：纯生成和纯检索。

纯生成就是不接外部知识库，直接让模型回答。它适合知识稳定、问题简单、回答容错高的场景，例如基础 FAQ、固定制度说明、写作润色。优点是链路短、成本低；缺点是无法保证事实新鲜度。

纯检索就是只返回搜索结果，不做生成。它的白话解释是：系统只负责找到证据，不负责组织答案。适合“我要看原文”的场景，例如法规检索、日志定位、代码片段查找。优点是可追溯性强；缺点是用户需要自己读和整合。

三者对比如下：

| 方案 | 适用场景 | 优势 | 局限 |
|---|---|---|---|
| 纯生成 | 知识稳定、任务简单 | 最便宜、链路最短 | 易过时，事实不可控 |
| 纯检索 | 事实查找、原文阅读 | 可追溯、实现稳 | 不会归纳、不擅长复杂表达 |
| RAG | 知识更新快、需解释与整合 | 准确性和表达能力兼顾 | 链路复杂，调参成本高 |

一个新手版对比例子：

- 公司内部 FAQ 长期不变，例如“年假规则是什么”，纯生成通常足够。
- 查询“最新支付接口参数和错误码”，RAG 更合适，因为文档经常更新。
- 查询“请把所有与退款重试相关的代码位置列出来”，纯检索可能已经满足需求；如果还要解释调用关系，再上 RAG。

RAG 的最优使用边界可以概括成一句话：当任务既需要外部事实，又需要把多条证据整理成自然语言时，RAG 最有价值。如果上下文已经完整给出，或者知识长期稳定到可以直接写进 prompt，那么增加检索层未必划算。
