## 核心结论

代码模型的指令微调对齐，主流上可以分成三条路径：纯监督微调 SFT、基于 Evol-Instruct 的“自动造更难指令再微调”、以及在 SFT 之后继续做偏好优化的 RLHF 或 DPO。SFT 的意思是“直接拿标准输入输出对训练模型”；偏好优化的意思是“不是只告诉模型标准答案，而是告诉它两个答案哪个更好”。

先给结论。

| 路径 | 代表模型 | 是否用 RLHF | 是否生成偏好对 | 代码正确性 | 指令遵循 | 安全性 | 算力/数据要求 |
|---|---|---:|---:|---|---|---|---|
| 纯 SFT | CodeLlama-Instruct | 否 | 否 | 稳定，依赖训练集覆盖 | 中等到较好 | 较弱，主要靠数据清洗 | 最低 |
| Evol-Instruct + SFT | WizardCoder | 否 | 间接生成高质量难指令，不是显式偏好对 | 往往更强，尤其复杂任务 | 较强 | 中等，仍主要靠数据分布 | 中等 |
| SFT + RLHF/GRPO | DeepSeek-Coder-Instruct 一类路线 | 是 | 是 | 可保持，但训练不稳时会波动 | 强 | 强 | 最高 |
| SFT + DPO | DeepSeek 系路线可替代方案 | 否，直接做偏好优化 | 是 | 通常比纯 SFT 更稳 | 强 | 强于纯 SFT | 低于 RLHF，高于纯 SFT |

对零基础读者，可以把三条路理解成三种训练助手写代码的方法。

- 纯 SFT：直接照说明书练习，给很多“题目 -> 标准答案”。
- Evol-Instruct：先把题卡自动升级，再筛出更难更规范的练习题继续练。
- RLHF/DPO：先按说明书练，再让“审核员”比较两个答案哪个更好，继续纠偏。

核心 trade-off 很明确。纯 SFT 最便宜、最稳定，但安全边界和拒答能力通常最弱。Evol-Instruct 擅长把模型往“更复杂的代码任务”上推，常常能提升代码完成质量。RLHF 能把行为边界卡得更细，但训练链路长、成本高、超参数敏感。DPO 可以看成更轻量的偏好优化：它保留“好答案胜过坏答案”的监督信号，但不需要单独训练 reward model，因此在代码领域通常比传统 RLHF 更数据高效，也更容易落地。

## 问题定义与边界

问题不是“让模型会写代码”这么简单，而是让模型在三个目标之间同时达标：

1. 代码正确性：生成的代码能运行、逻辑对、边界条件不明显出错。
2. 指令遵循：用户要求“只给 Python”“不要解释”“保留接口签名”时，模型真的照做。
3. 安全性：模型避免生成危险建议、泄露密钥、鼓励不安全实现或绕过限制。

这三个目标并不天然一致。一个模型可能代码能力很强，但总是多说废话；也可能很听话，但为了“听话”过度迎合，给出并不安全的实现。对齐的本质，就是在这些目标之间调分布。

边界主要有三层。

| 维度 | 纯 SFT 关注什么 | Evol-Instruct 关注什么 | RLHF/DPO 关注什么 |
|---|---|---|---|
| 数据来源 | 标准指令-答案对 | 自动演化后的更复杂指令及答案 | 偏好对：好答案 vs 坏答案 |
| 优化目标 | 拟合正确答案分布 | 拟合更高难度任务分布 | 直接优化“更偏好哪个回答” |
| 工程成本 | 低 | 中 | 中到高 |

一个常见误区是把“代码 benchmark 分数高”直接等同于“对齐做得好”。这不成立。代码 benchmark 往往只测功能正确率，但真实产品里，用户还关心输出格式、依赖选择、异常处理、许可证风险、凭证管理和是否拒绝危险请求。

可以把整体流程画成一个非常简化的链条：

| 阶段 | 输入数据 | 输出目标 |
|---|---|---|
| SFT | `(指令, 标准回答)` | 学会基本指令格式与常见代码模式 |
| Evol-Instruct | `(原始指令 -> 演化后更难指令, 生成回答)` | 提升复杂任务适应度 |
| RLHF / DPO | `(指令, 优选回答, 拒绝回答)` | 强化偏好顺序、安全边界与指令遵循 |

玩具例子可以很直观。假设用户说：“写一个去重函数，只返回代码，不要解释。”

- 纯 SFT 模型可能返回能跑的函数，但额外加三段解释。
- Evol-Instruct 训练过的模型更可能处理“保序去重、保留类型注解、处理空输入”等复杂细节。
- DPO/RLHF 继续对齐后，模型更可能既只输出代码，又避免引入不必要依赖，还能在危险请求下拒绝。

所以本文讨论的不是“谁绝对最好”，而是“在什么约束下，哪条路径更合适”。

## 核心机制与推导

先看纯 SFT。它的训练目标最直接：给定输入 $x$，最大化参考回答 $y$ 的条件概率 $p_\theta(y|x)$。直白地说，就是“让模型更像训练集里的标准答案”。这种方法稳定，因为目标函数简单，数据也最容易准备。但它只能学到“数据里出现过的偏好”，无法显式学“这个答案虽然也能跑，但我们不喜欢”。

Evol-Instruct 的关键，不是换了损失函数，而是改了训练数据的难度分布。所谓“演化”，就是把原本简单的指令自动改写成更复杂、更约束明确、更接近真实工程任务的版本。白话解释：不是让学生一直做基础题，而是把题目升级为“同样主题，但要求更多、限制更细、步骤更长”的版本。

例如原始指令可能是：

- “写一个 Python 函数，统计单词频率。”

演化后会变成：

- “写一个 Python 函数，统计英文单词频率，忽略大小写和标点，返回按频率降序、字典序升序排序的前 10 项，并补充时间复杂度说明，只使用标准库。”

这种数据改造会带来两个变化。第一，模型更习惯处理长指令和复合约束。第二，训练集会更强调“遵守细节约束”，而不是只给出一个能跑的最短答案。这也是 WizardCoder 一类路线常能提升复杂代码任务表现的原因。

再看 DPO。DPO 是 Direct Preference Optimization，中文可理解为“直接偏好优化”。它不训练单独的 reward model，而是直接让模型在两个候选回答之间拉开差距。

公式是：

$$
\mathcal{L}_{\text{DPO}} = -\log \sigma\left(\beta\left[\log p_\theta(r^+|x) - \log p_\theta(r^-|x)\right]\right)
$$

这里：

- $r^+$ 是优选回答，也就是人或规则认为更好的答案。
- $r^-$ 是拒绝回答，也就是较差答案。
- $\sigma$ 是 sigmoid，可以理解成“把分差压到 0 到 1 之间”。
- $\beta$ 是温度或间隔系数，可以理解成“偏好尺子的敏感度”。

最小数值例子如下。设：

- $\log p_\theta(r^+|x)=1.2$
- $\log p_\theta(r^-|x)=0.4$
- $\beta=0.01$

则有：

$$
\mathcal{L}_{\text{DPO}} = -\log \sigma(0.01 \times (1.2 - 0.4))
= -\log \sigma(0.008)
\approx -\log(0.502)
\approx 0.69
$$

这个值接近二分类刚开始训练时的损失，说明模型当前只是“略微偏向好答案”，还没有把两者明显拉开。训练的目标，就是继续增大 $\log p_\theta(r^+|x) - \log p_\theta(r^-|x)$。

玩具例子可以写成这样。用户输入：“生成一个连接数据库的 Python 示例。”

两个候选回答：

- 好答案 $r^+$：使用环境变量读取密码，使用参数化查询，补充异常处理。
- 坏答案 $r^-$：把账号密码硬编码在代码里，还拼接 SQL 字符串。

DPO 不需要额外训练一个“裁判模型”去给这两个答案打分，而是直接告诉主模型：在这个输入下，你应该更相信前者，而不是后者。这就是它相比 RLHF 更轻的一点。

RLHF 为什么更重？因为标准流程通常包含：训练 reward model、在线或半在线采样、再用 PPO/GRPO 之类算法更新策略模型。链路更长，监控点更多，也更容易因为奖励设计有偏导致模型“学会刷分”。

## 代码实现

如果只做 CodeLlama-Instruct 风格的纯 SFT，工程上通常就是三步：

1. 把多来源数据统一成同一种指令模板。
2. 保证 `prompt -> response` 配对质量。
3. 跑标准的 next-token 监督训练。

这条链路最像“把现有微调 pipeline 换一套数据”。

如果要做 DeepSeek-Coder 一类的 SFT + 偏好优化，则通常是在 SFT 之后追加第二阶段训练。这个阶段的输入不再是单个标准答案，而是 preference pair，也就是“同一题的好答案和坏答案”。

下面给一个可运行的最小 Python 例子。它不依赖深度学习框架，只演示 DPO 损失的数值计算逻辑。

```python
import math

def sigmoid(x: float) -> float:
    return 1.0 / (1.0 + math.exp(-x))

def dpo_loss(logp_pos: float, logp_neg: float, beta: float) -> float:
    return -math.log(sigmoid(beta * (logp_pos - logp_neg)))

loss = dpo_loss(1.2, 0.4, 0.01)
assert 0.68 < loss < 0.70

better_loss = dpo_loss(2.0, 0.4, 0.01)
assert better_loss < loss

print(round(loss, 4), round(better_loss, 4))
```

上面两个 `assert` 表达了 DPO 的核心性质：

- 当好答案分数只比坏答案高一点时，loss 大约在 `0.69` 附近。
- 当好答案分数进一步拉开时，loss 会下降。

如果换成训练框架里的伪代码，结构通常就是这样：

```python
logits_pos = model(tokens_pos).logits
logits_neg = model(tokens_neg).logits

logp_pos = token_logprob(logits_pos, labels_pos)
logp_neg = token_logprob(logits_neg, labels_neg)

loss = -torch.log(torch.sigmoid(beta * (logp_pos - logp_neg)))
loss.backward()
optimizer.step()
```

这里真正参与计算的不是“单个 token 的一个 logit”，而是整段回答在给定输入下的条件对数概率聚合值。新手容易把它误解成“直接拿两个张量相减就行”，这是不严谨的。实际实现时，通常要先把回答 token 对齐到标签，再对有效 token 的 log-prob 做求和或平均。

真实工程例子可以看金融审计 IDE 插件。你希望模型满足三条规则：

- 优先推荐维护中的安全库，而不是过时库。
- 不生成硬编码的 API key、数据库密码。
- 在涉及用户数据导出时，默认提示审计日志和权限控制。

如果只做纯 SFT，你需要在训练集中大量覆盖这些规则，而且还要保证示例足够一致。若改为 SFT + DPO，你只需要额外构造若干偏好对，例如：

- 好答案：读取 `os.environ["DB_PASSWORD"]`
- 坏答案：`password = "admin123"`

- 好答案：使用参数化 SQL
- 坏答案：字符串拼接 SQL

这种方式的优势在于，它不要求你为每种任务都写完美标准答案，而是只要能稳定区分“哪个更好”。在安全规则相对明确的业务里，这种数据准备方式非常省人力。

## 工程权衡与常见坑

先看结论。SFT 最稳，Evol-Instruct 更擅长拉高复杂任务表现，RLHF 控制力最强但最贵，DPO 是偏好优化里的高性价比方案。

常见坑可以直接列成表。

| 常见坑 | 出现在哪条路线 | 具体问题 | 规避策略 |
|---|---|---|---|
| 见指令就答，不会拒绝 | 纯 SFT | 对危险请求也按模式补全 | 加拒答样本、负样本、reject sampling |
| 只会解更难题，不一定更听话 | Evol-Instruct | 复杂度提升了，但格式约束未同步提升 | 演化时加入输出格式、边界条件约束 |
| reward model 过拟合 | RLHF | 裁判模型学到数据偏见，主模型开始“刷奖励” | 缩短 RL 链路，改用 DPO 或加强偏好数据审核 |
| 训练不稳定、超参敏感 | PPO/GRPO | 学习率、KL 约束、采样策略都会影响结果 | 先做强 SFT 基座，再小步 RL 或改用 DPO |
| 偏好对质量差 | DPO / RLHF | 好坏答案差异不清，模型学不到稳定边界 | 让 bad case 明确违反一条规则，避免“两个都差不多” |
| replay buffer 污染 | RLHF | 旧采样样本分布过时，影响后续优化 | 缩短缓存周期，定期重采样 |
| 安全规则只写在系统提示词里 | 全部 | 推理时容易被绕过 | 把关键规则写进训练数据与偏好对 |

对初级工程师来说，最重要的判断标准不是“论文里谁分高”，而是“你手头到底有什么数据”。如果你只有大量普通代码问答，没有人工偏好标注，那 DPO 就算理论上合适，也做不出来。反过来，如果你手里有专家能快速判断“哪个答案更合规”，那偏好优化就会很值。

还有一个很容易被忽略的坑：代码正确性和安全性有时会冲突。比如模型为了更安全，可能给出更保守、更冗长的实现，短期 benchmark 反而下降。这不是一定变差，而是目标函数换了。工程上要先明确优先级：你是在做刷题模型，还是在做生产助手。

## 替代方案与适用边界

可以把选择逻辑压缩成一个决策树思路。

| 条件 | 推荐方法 |
|---|---|
| 没有偏好对，算力有限，先追求可用 | 纯 SFT |
| 没有偏好对，但想提升复杂任务表现 | Evol-Instruct + SFT |
| 有偏好对，算力一般，希望快速加安全边界 | SFT + DPO |
| 有偏好对，算力充足，需要精细控制行为 | SFT + RLHF/GRPO |
| 安全要求极高且可持续标注 | SFT + DPO 起步，再视情况升级 RLHF |

几个典型场景可以直接判断。

第一类，数据丰富但安全要求不高，比如内部代码补全工具，主要任务是补文档、写脚手架、生成单元测试。这时纯 SFT 往往就够了，因为目标是覆盖常见模式，不是严格管行为边界。

第二类，你想把模型从“会写简单代码”拉到“更擅长复杂约束任务”，例如多文件修改、补齐接口、遵守现有代码风格。这时 Evol-Instruct 很有价值，因为它本质上是在补“任务复杂度分布”这块短板。

第三类，你已经上线到真实业务，用户会要求数据库访问、鉴权、日志脱敏、依赖选择。这时只靠 SFT 容易不够，因为你需要的不只是“会答”，而是“偏向正确行为”。若偏好数据能拿到，但算力和时间不够做完整 RLHF，DPO 通常是更合理的第一选择。

一个实际建议是：不要一开始就上最重的 RLHF。更稳的顺序通常是：

1. 先做强基座 SFT。
2. 用少量高质量偏好对做 DPO。
3. 只有当 DPO 仍然不足以满足控制需求时，再考虑更重的 RLHF 或 GRPO。

原因很简单。代码领域的很多偏好，其实是规则化的，例如“安全库优于过时库”“不要暴露凭证”“先给最小可运行示例再补解释”。这类偏好很适合用 DPO 编码，不一定非要走完整 reward learning。
