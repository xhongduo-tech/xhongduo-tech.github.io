## 核心结论

CommonsenseQA 是一个面向常识推理的五选一问答数据集。它不是让模型复述一段上下文，而是要求模型在没有外部段落提示的情况下，补出“概念之间本来就该知道的关系”。

它的核心价值不在于题目形式是多选题，而在于题目背后绑定了 **ConceptNet**。ConceptNet 是一个常识知识图谱，可以把“water”和“drink”、“boat”和“float”这类日常概念关系结构化表示出来。CommonsenseQA 正是围绕这些关系构造问题，因此它测的不是阅读理解，而是“模型有没有把常识链连起来”。

一个最典型的玩具例子是：

- 源概念：`water`
- 目标关系：`UsedFor`
- 问题：`What is water commonly used for?`
- 选项：`drink`, `build`, `sleep`, `fly`, `write`

这里正确答案是 `drink`。模型如果答对，不是因为题干里给了提示段落，而是因为它知道“水通常被用来喝”。

下表先看这个数据集最重要的三个维度：

| 维度 | 内容 | 说明 |
| --- | --- | --- |
| 数据规模 | 常见统计口径为 12,102 或 12,247 | 不同资料的统计口径略有差异，工程中应先确认所用版本 |
| 选项数 | 每题 5 个选项 | 随机猜测基线约为 20% |
| 目标能力 | 无上下文常识推理 | 重点测试概念关系理解，不是篇章检索 |

结论可以压缩成一句话：**CommonsenseQA 适合用来测“模型能不能在没有上下文时，靠常识关系选出正确概念”，但它不等价于真实世界全部常识能力。**

## 问题定义与边界

CommonsenseQA 的题目不是随便人工编出来的，而是从知识图谱关系出发半自动构造。所谓“源概念”，就是出题围绕的起点概念；所谓“目标概念”，就是和它通过某种关系连接的候选答案；所谓“关系”，就是 `UsedFor`、`CapableOf`、`AtLocation` 这类语义边。

例如 `CapableOf` 表示“某个东西能做什么”。白话说，就是“这个对象天然具备什么能力”。如果知识图谱里有 “boat -> CapableOf -> floating”，那么题目可能写成：

- 问题：`A boat is capable of what?`
- 选项：`floating`, `melting`, `singing`, `typing`, `boiling`

这类题的边界非常明确：

| 步骤 | 做法 | 目的 |
| --- | --- | --- |
| 1 | 从 ConceptNet 选一个源概念 | 确定题目主题 |
| 2 | 选一个目标关系 | 确定要考哪类常识 |
| 3 | 找到正确目标概念和若干干扰概念 | 形成可比较候选 |
| 4 | 人工写成自然语言题干 | 避免图谱三元组直接暴露 |
| 5 | 组成五选一问题 | 用准确率统一评测 |

这里有两个很重要的边界。

第一，它强调 **无上下文**。也就是说，模型通常不能依赖一段文章、对话历史或文档证据来找答案，只能依赖内部知识或外部检索补充。

第二，它强调 **概念关系辨析**，不是开放生成。模型不是自由回答一段解释，而是在五个概念中挑一个最符合常识关系的选项。因此它适合做标准化评测，但不直接等同于真实业务里的复杂推理。

一个真实工程例子是：你在做一个通用问答模型基准测试，想知道模型是“会背语料”，还是“真的掌握了基础常识关系”。这时加入 CommonsenseQA 很有价值，因为它能把阅读能力和常识链能力拆开看。如果模型在阅读理解上很强、在 CommonsenseQA 上很弱，通常说明它对概念关系的内部建模还不够稳定。

## 核心机制与推导

CommonsenseQA 最常用的指标是 **Accuracy**，也就是准确率。白话说，就是“总共做了多少题，其中答对了多少题”。

公式很直接：

$$
\mathrm{Accuracy}=\frac{\text{correct}}{\text{total}}
$$

如果在 50 道题里答对了 37 道，那么：

$$
\mathrm{Accuracy}=\frac{37}{50}=0.74=74\%
$$

这就是很多工程文档里会用的 `test-tiny` 思路：先抽一个很小的样本集，快速估算模型大致水平，再决定要不要跑完整验证集。

为什么五选一设置下，这个指标能较直观地反映常识能力？原因是：

1. 随机猜测基线固定，大约是 $1/5=20\%$。
2. 题目选项通常共享一定语义邻近性，干扰项不是完全胡乱填的。
3. 因此模型要显著高于 20%，就必须具备一定的概念关系辨识能力。

看一个小表格就很清楚：

| 评测集 | 题数 | 正确数 | Accuracy |
| --- | --- | --- | --- |
| test-tiny | 50 | 37 | 74% |
| dev-mini | 200 | 118 | 59% |
| random baseline | 50 | 10 左右 | 20% 左右 |

这里要注意一个推导边界：**准确率高，不一定说明模型真正“理解”了常识；但准确率低，通常能明确说明它无法稳定辨认这些常识关系。**  
原因在于多选题允许模型靠统计偏置、词频模式、选项分布获得一部分分数，但如果连中等水平都达不到，至少说明它在这类关系映射上存在明显短板。

## 代码实现

工程里最常见的做法有两种：

1. 直接使用现成评测框架，比如 EleutherAI 的 `lm-evaluation-harness`。
2. 自己写一个最小评测脚本，读取题目 JSON，构造 prompt，收集模型输出，计算准确率。

下面给一个可运行的简化 Python 例子。这个例子不依赖真实大模型，而是用一个玩具规则模型模拟评测流程，重点看“数据怎么过一遍”和“准确率怎么算”。

```python
from typing import List, Dict

samples: List[Dict] = [
    {
        "question": "What is water commonly used for?",
        "choices": ["drink", "build", "sleep", "fly", "write"],
        "answer": "drink",
    },
    {
        "question": "A boat is capable of what?",
        "choices": ["floating", "melting", "singing", "typing", "boiling"],
        "answer": "floating",
    },
    {
        "question": "Where would you usually find a pillow?",
        "choices": ["bed", "garage", "river", "cloud", "keyboard"],
        "answer": "bed",
    },
]

def predict(question: str, choices: List[str]) -> str:
    q = question.lower()
    # 玩具规则：真实工程里这里会替换成 LLM 推理或打分逻辑
    if "water" in q and "used for" in q:
        return "drink"
    if "boat" in q and "capable" in q:
        return "floating"
    if "pillow" in q and "find" in q:
        return "bed"
    return choices[0]

def evaluate(dataset: List[Dict]) -> float:
    correct = 0
    total = len(dataset)
    for item in dataset:
        pred = predict(item["question"], item["choices"])
        if pred == item["answer"]:
            correct += 1
    acc = correct / total
    return acc

acc = evaluate(samples)

assert len(samples) == 3
assert abs(acc - 1.0) < 1e-9

print(f"Accuracy: {acc:.2%}")
```

如果你接入真实模型，流程通常会变成：

- 读取 CommonsenseQA 验证集
- 把题干和五个选项拼成 prompt
- 让模型输出选项字母或对每个选项打分
- 取分数最高的选项
- 与 gold answer 比较，累计 `correct`

一个接近真实工程的例子是：你维护一个模型评测平台，需要每天对多个版本模型做回归测试。这时可以把 CommonsenseQA 封装成统一任务接口，让不同模型共用同一套 prompt 模板和 batch 配置。这样你比较的是模型能力变化，而不是脚本写法变化。

常见配置项如下：

| 配置项 | 作用 | 工程含义 |
| --- | --- | --- |
| prompt 模板 | 把题干和 5 个选项格式化 | 保证不同模型输入一致 |
| batch size | 控制一次送多少题 | 影响吞吐和显存 |
| model list | 要评测的模型集合 | 用于横向比较 |
| max tokens | 限制输出长度 | 多选题通常可以设很短 |
| scoring 方式 | 生成字母或选项打分 | 影响稳定性与复现性 |

如果用评测框架，CommonsenseQA 的优势是已经有标准任务定义，你只需要关注模型接入、prompt 版本和结果记录，不必从零写数据管道。

## 工程权衡与常见坑

CommonsenseQA 看起来简单，实际很容易把评测做偏。

第一类坑是把它当成“普通分类任务”直接硬微调。历史上不少模型即使经过微调，分数也不高，例如早期 BERT 微调结果只有五成多。这说明问题不只是分类头太弱，而是模型缺少稳定的常识链建模能力。

第二类坑是忽略多跳关系。所谓“多跳”，白话说，就是答案不是一步直达，而是需要连过两三个概念关系才能更稳地判断。比如“restaurant”和“menu”之间可能不是一句话级共现，而是由“餐馆提供点餐环境”“菜单用于点餐”这类复合常识支撑。

第三类坑是评测时让模型“看到了不该看的东西”。例如：
- prompt 里偷偷加了解释性示例，已经不再是原始无上下文设置
- 检索模块读入了训练泄漏内容
- 不同模型用了不同 prompt 模板，结果无法公平对比

下面这个表格概括了常见方案差异：

| 方法 | 机制 | 预期表现 | 典型问题 |
| --- | --- | --- | --- |
| 纯微调 | 只靠参数记忆题型与概念模式 | 实现简单 | 对稀疏常识和多跳关系不稳定 |
| 加知识检索 | 检索 ConceptNet 路径或相关事实 | 常识覆盖更强 | 检索噪声会误导模型 |
| 多跳/交叉注意力 | 显式融合题干与知识路径 | 关系链更清楚 | 系统更复杂，调参成本高 |
| Prompt 增强 | 少样本示例或解释式提示 | 部署快 | 复现性和稳定性一般 |

一个新手能听懂的真实工程例子是：  
你要做一个“题干 + 选项 + 知识图谱检索”的系统。流程可能是先从题干里抽出源概念 `water`，再从 ConceptNet 检索若干关系路径，如 `water -> UsedFor -> drink`，然后把这些路径和原题一起送进模型。模型通过 attention 机制，也就是“让不同信息片段互相对齐的计算方式”，更容易把“water”和“drink”对上，而不是被其他表面相似选项干扰。

这里的工程权衡非常直接：**纯模型方案更简单，知识增强方案更强，但也更容易引入检索噪声、延迟和维护复杂度。**

## 替代方案与适用边界

CommonsenseQA 并不是所有常识任务的通用替代品。它适合“无上下文、五选一、概念关系驱动”的场景，不适合需要长对话历史、社会互动背景或开放世界事件链的任务。

常见替代数据可以这样看：

| 数据集 | 是否需要上下文 | 主要能力 | 适用场景 | 限制 |
| --- | --- | --- | --- | --- |
| CommonsenseQA | 否 | 概念关系常识推理 | 无上下文多选评测 | 对真实对话和长链场景覆盖有限 |
| SocialIQA | 是 | 社会情境与意图理解 | 对话、人物动机判断 | 不以 ConceptNet 结构为核心 |
| ATOMIC | 通常是事件描述 | 事件因果与社会常识 | 生成式知识建模 | 更像知识补全，不是标准多选 |

如果你的业务是客服对话、角色动机判断、社交语义推断，那么 SocialIQA 往往更贴近实际，因为它允许上下文存在。  
如果你的业务是事件后果预测、常识知识补全，那么 ATOMIC 更合适，因为它更像“给定事件，补全可能原因或结果”。

对于资源有限的小模型，还可以考虑一种更务实的路线：不做 full fine-tune，而做 **knowledge-augmented prompting**。白话说，就是先检索少量外部知识，再把这些知识拼进 prompt，让模型在推理时“临时看一眼提示”。这种做法在算力、数据和部署成本上通常更轻，但上线效果会依赖检索质量。

所以适用边界可以总结成一句话：**当你要测的是“无上下文概念常识选择”，CommonsenseQA 很合适；当你要测社会互动、事件因果或长上下文推理，它就不是最优基准。**
