## 核心结论

IA³（Infused Adapter by Inhibiting and Amplifying Inner Activations，意思是“通过抑制或放大内部激活来适配模型”）不是去学习新的低秩矩阵，也不是给 Transformer 额外接一段大模块，而是在模型已有通道上学习几个很小的逐维缩放向量。它的核心写法可以压缩成一句话：

$$
h' = s \odot h
$$

其中 $h$ 是原始激活，白话就是某一层本来算出来的向量；$s$ 是可训练缩放向量，白话就是每个通道各自有一个“放大或缩小倍率”；$\odot$ 表示逐元素相乘。

这件事带来两个直接结果：

1. 参数极少。IA³ 每层通常只新增 $2d+d_{ff}$ 个参数，若 Transformer 里常见 $d_{ff}=4d$，则每层是 $6d$，随宽度线性增长，不随输入长度增长。
2. 推理很轻。它不改基础权重的主结构，只是在已有激活上做乘法门控，因此部署多个任务适配器时非常方便。

但代价也很明确：IA³ 只能重标定已有特征，不能像 LoRA 那样显式引入新的线性组合方向。白话讲，它更像调音台的滑块，而不是重新接线。任务如果只是“让模型更重视某些已有能力”，IA³ 很合适；任务如果要求模型学到明显不同的新特征，IA³ 往往先碰到上限。

| 方法 | 学习对象 | 典型可训练参数占比 | 是否引入新方向 | 推理额外开销 |
| --- | --- | ---: | --- | --- |
| 全量微调 | 全部权重 | 100% | 是 | 无 |
| IA³ | 若干缩放向量 | 可低到约 0.01% | 否 | 近似无 |
| LoRA | 低秩增量矩阵 | 常见约 0.1% 以上 | 是 | 可合并后近似无 |

## 问题定义与边界

PEFT（Parameter-Efficient Fine-Tuning，意思是“只训练少量参数的微调”）要解决的问题很直接：下游任务很多，但显存、训练时间、存储空间都不够，不能每来一个任务就把整模型重新训一遍。

以一个常见场景为例：你有一个预训练好的文本模型，要拿去做新闻分类、商品评论情感分析、FAQ 检索排序。若采用全量微调，你需要：

- 反向传播整个模型
- 为每个任务存一整份权重
- 承担更高过拟合风险，特别是在小数据集上

IA³ 的边界也因此很清楚。它适合的问题通常满足两个条件：

1. 基础模型已经具备大部分能力，下游任务主要是“重排已有能力的权重”
2. 资源约束明显，希望单任务适配成本尽量低

一个对新手最重要的判断标准是：如果你觉得模型“已经会，只是没把注意力放对地方”，IA³ 往往值得先试；如果你觉得模型“根本没学过这种结构或概念”，IA³ 往往不够。

下面这个对比表可以帮助建立直觉：

| 场景 | 全量微调 | IA³ | LoRA |
| --- | --- | --- | --- |
| 4GB 显存小卡做分类 | 压力大 | 最友好 | 通常可行 |
| 每个任务都要单独存权重 | 成本高 | 很低 | 较低 |
| 小样本下希望少过拟合 | 风险较高 | 通常更稳 | 通常较稳 |
| 任务与预训练分布差异很大 | 表达力足 | 容易见顶 | 更有机会提升 |

玩具例子可以这样理解。假设一个 4 维表示向量的四个通道分别粗略对应“数字敏感”“否定词敏感”“实体名敏感”“长距离依赖敏感”。原模型都有这些能力，但做情感分类时你更需要第二个通道，做命名实体识别时更需要第三个通道。IA³ 做的事，就是给这四个通道分别乘上不同倍率，而不是重新创造第五、第六种能力。

真实工程例子更接近 T5、T0 这类大模型多任务适配。一个团队可能同时维护十几个文本任务，如果每个任务都全量微调并保存完整模型，存储和发布都很重。IA³ 允许每个任务只保存一小组向量，基础模型保持统一，切任务时只切换缩放参数。

## 核心机制与推导

先看 Transformer 里的插入位置。Attention（注意力，意思是“让模型给不同 token 分配不同关注度”）内部会先算出 $Q,K,V$，即 query、key、value 三组表示。IA³ 通常不改 $Q$，而是对 $K,V$ 的投影结果做逐通道缩放：

$$
K' = l_K \odot (XW_K)
$$

$$
V' = l_V \odot (XW_V)
$$

其中 $X$ 是输入隐藏状态，$W_K,W_V$ 是冻结的原始投影矩阵，$l_K,l_V$ 是新增的可训练向量。

FFN（前馈网络，意思是“每个 token 各自经过的两层 MLP”）里，IA³ 常对中间激活或输出通道做缩放。可写成：

$$
h' = l_{ff} \odot h
$$

这三处门控的共同点是：只重标定已有坐标轴，不产生新的坐标轴。若把原线性层看成把输入映射到一个特征空间，IA³ 并没有改这个空间的基底，只是给每个基底方向乘系数。

参数量推导也很直接。若模型隐藏维度为 $d$，FFN 中间维度为 $d_{ff}$，则每层新增参数为：

$$
|l_K| + |l_V| + |l_{ff}| = d + d + d_{ff} = 2d + d_{ff}
$$

在常见 Transformer 里，$d_{ff}=4d$，所以：

$$
2d + d_{ff} = 6d
$$

这和 LoRA 很不一样。LoRA 的核心是：

$$
\Delta W = BA
$$

它通过低秩矩阵乘积学习一个新的权重增量，因此能引入新方向；IA³ 则只有缩放，表达空间明显更窄。

玩具例子最能看出这一点。假设 FFN 某层原输出为：

$$
h = [0.5, -1.0, 0.2, 0.8]
$$

学习到的缩放向量为：

$$
s = [1.2, 0.8, 1.0, 0.5]
$$

则新输出是：

$$
h' = s \odot h = [0.6, -0.8, 0.2, 0.4]
$$

你会发现每个通道只是被放大或压小，没有通道之间的新混合关系。如果原来“第 1 通道和第 3 通道线性组合后”才能表达某个新特征，IA³ 本身并不会创造这个组合。

真实工程里，这种机制的价值是“让模型重新分配内部预算”。例如做低资源文本分类时，预训练模型其实已经有关键词识别、主题词聚合、局部句法等能力。IA³ 可以把对任务更有用的那些内部通道权重抬高，把无关通道压低，以很小参数完成适配。

## 代码实现

IA³ 的实现重点只有两件事：

1. 原始线性层权重冻结
2. 在 attention 的 $K,V$ 和 FFN 指定位置插入可训练缩放向量

最关键的工程习惯是把缩放向量初始化为全 1。这样训练开始前模型行为与原模型完全一致，等价于从恒等映射出发，优化过程更平滑。

下面是一个可运行的 Python 玩具实现，只演示“逐通道缩放”本身，不依赖深度学习框架：

```python
from math import isclose

def ia3_scale(hidden, scale):
    assert len(hidden) == len(scale)
    return [h * s for h, s in zip(hidden, scale)]

# 玩具例子
h = [0.5, -1.0, 0.2, 0.8]
s = [1.2, 0.8, 1.0, 0.5]
out = ia3_scale(h, s)

assert len(out) == 4
assert isclose(out[0], 0.6)
assert isclose(out[1], -0.8)
assert isclose(out[2], 0.2)
assert isclose(out[3], 0.4)

# identity 初始化：scale 全 1 时，输出必须等于原输入
identity = [1.0, 1.0, 1.0, 1.0]
assert ia3_scale(h, identity) == h
print(out)
```

如果放到真实模型里，伪代码会更接近下面这样：

```python
class IA3Attention:
    def __init__(self, d_model):
        self.W_k = FrozenLinear(d_model, d_model)
        self.W_v = FrozenLinear(d_model, d_model)
        self.l_k = TrainableVector.ones(d_model)
        self.l_v = TrainableVector.ones(d_model)

    def forward(self, x):
        K = self.W_k(x)
        V = self.W_v(x)
        K = self.l_k * K
        V = self.l_v * V
        return attention_with_scaled_kv(K, V)

class IA3FFN:
    def __init__(self, d_ff):
        self.l_ff = TrainableVector.ones(d_ff)

    def forward(self, h):
        h = gelu(h)
        h = self.l_ff * h
        return h
```

真实工程例子可以想成“给一个冻结的 T5 做客服意图分类”。你不会改动大矩阵，只是在每层指定位置挂上 `l_k`、`l_v`、`l_ff` 三组向量。训练时优化器只更新这些向量，保存 checkpoint 时也只保存这些向量。这样一个基础模型可以复用到多个分类任务上。

## 工程权衡与常见坑

IA³ 之所以省，是因为它限制得很狠；也正因为限制狠，边界会更快暴露。

先看主要权衡：

| 维度 | IA³ 的优势 | IA³ 的风险 |
| --- | --- | --- |
| 参数量 | 极小，通常比 LoRA 更低 | 调整空间有限 |
| 训练成本 | 显存和存储友好 | 表达不足时无 rank 可调 |
| 部署 | 容易挂多个任务适配器 | 任务差异大时收益有限 |
| 实现复杂度 | 逻辑简单，接近乘法门控 | 插入位置选错会掉效果 |

常见坑主要有四类。

第一，误以为“参数越少越一定更好”。不是。参数少只是资源友好，不代表任务效果一定最好。如果任务需要新语义组合，例如医学问答、法律条文归因、强结构化抽取，IA³ 可能只是在已有特征上做微调，很快到天花板。

第二，学习率设太大。IA³ 是乘法门控，不是加法残差。缩放向量如果初期偏离过快，会直接把某些通道放大太多或压得太小，梯度尺度容易不稳定。实践里通常会用更保守的学习率，并依赖原模型里的 LayerNorm（层归一化，意思是“把每层数值尺度拉回稳定范围”）。

第三，初始化不是全 1。如果你把缩放向量随机初始化，模型一开始就已经偏离预训练分布，等于还没学任务就先破坏了基础能力。IA³ 最稳的起点就是 identity，即全 1 初始化。

第四，错误理解“零额外开销”。更准确的说法是“几乎不增加结构复杂度，并且可轻量部署”。数学上依然多了逐元素乘法，只是相对大模型主计算量，这部分通常可以忽略。

一个真实工程中的典型坑是：团队用 IA³ 给通用中文模型适配医疗文本分类，发现指标始终不如 LoRA。原因通常不是 IA³ 实现错了，而是医疗任务包含大量预训练阶段较少覆盖的专业术语与判别结构。此时模型需要新的组合方向，IA³ 只能调现有通道的重要性，无法像 LoRA 那样更主动地塑造新特征。

## 替代方案与适用边界

如果把 PEFT 方法按“表达力和成本”来排，IA³ 往往靠近“极低成本”这一端。

LoRA 更像在原权重旁边加一个低秩增量分支。它的成本比 IA³ 高，但能学习新的线性组合方向，所以面对分布偏移更明显、任务更复杂的场景，常常更有上限。

ReFT 这类方法可以理解为“只改特定表示或少量关键部件”，某些极端资源场景下也很有吸引力，但它的使用方式和适配目标通常更依赖具体任务设计。

下面是一个实用对比：

| 方法 | 可训练参数比例 | 表达力 | 调参成本 | 适合场景 |
| --- | ---: | --- | --- | --- |
| IA³ | 很低 | 中等偏低 | 低 | 资源极紧、任务接近预训练能力 |
| LoRA | 低 | 中高 | 中 | 通用首选，任务差异较大时更稳 |
| ReFT / selective tuning | 很低到低 | 依实现而定 | 中到高 | 特定结构控制、研究型实验 |
| 全量微调 | 最高 | 最高 | 高 | 数据充足、资源充足、追求上限 |

一个简单选择规则可以直接记住：

- 先试 IA³：当你最关心显存、存储、部署数量
- 先试 LoRA：当你最关心效果上限，且任务与预训练分布差异不小
- 考虑全量微调：当数据、算力、训练预算都足够，并且模型值得为单任务专门优化

新手常问“IA³ 会不会替代 LoRA”。答案是否定的。二者不是谁完全覆盖谁，而是在不同约束下做选择。IA³ 的优势是极致轻量；LoRA 的优势是更强表达力。真正的工程决策不是“哪个更先进”，而是“当前任务需要的是更小成本，还是更大可塑性”。
