## 核心结论

多语种数据采样与平衡，核心不是把所有语言“平均对待”，而是控制不同语言在训练中的**曝光率**。曝光率可以白话理解为：模型在训练过程中，某种语言被看到多少次。真正要解决的问题是两个目标同时成立：

1. 高资源语种继续提供高密度知识，不要因为过度平衡而浪费训练预算。
2. 低资源语种持续进入梯度更新，不要被高资源语种淹没到几乎学不到。

下面这个玩具例子最直观。假设英语有 100 万条样本，斯瓦希里语只有 1000 条：

| 方案 | 结果 |
|---|---|
| 按原始分布采样 | 英语几乎主导所有 batch，小语种训练机会极少 |
| 完全均匀采样 | 小语种被反复重复，高资源语种有效信息利用不足 |
| 温度采样 | 压平分布，但不压平成完全一样，通常最稳 |

所以，多语种训练常见的正确做法不是“每种语言一票”，而是“压平但不压平到完全一样”。这也是很多多语种预训练和翻译系统采用温度采样的原因。

## 问题定义与边界

先把问题说清楚。本文讨论的是**训练采样策略**，不是数据清洗、翻译增强、词表设计本身。训练采样策略指：当你已经有多种语言的数据后，训练循环每一步该从哪种语言抽样。

记号统一如下：

- `n_l`：语言 `l` 的样本数
- `p_l`：语言 `l` 的采样概率

如果有 10 种语言，问题不是“如何把每种语言变成一样多”，而是“训练时每种语言应该出现多少次，才能既保留跨语言共性，又不让小语种断粮”。

这件事有明确边界：

| 问题类型 | 是否属于本文重点 |
|---|---|
| 训练时语种曝光率分配 | 是 |
| tokenizer 是否适合低资源语种 | 相关，但不属于采样本身 |
| 语料去重和清洗 | 相关，但不属于采样本身 |
| loss 权重设置 | 相关，但要和采样区分 |
| 模型参数量是否足够 | 相关，但不属于采样本身 |

这里的“tokenizer”首次出现，可以白话理解为：把文本切成模型可处理小片段的工具。如果 tokenizer 对某语言切分很差，把一个常见词拆成很多碎片，那么即使你提高该语言采样率，收益也可能有限。

另一个边界是任务目标不同。预训练、多语种翻译、对齐训练、下游微调，它们对采样平衡的要求并不一样。预训练通常更重视覆盖面，微调通常更重视目标语种任务效果。也就是说，采样不是抽象公平问题，而是面向目标函数的工程分配问题。

---

## 核心机制与推导

最常见的平衡方法是**温度采样**。温度采样可以白话理解为：用一个可调参数，把原始极不平衡的数据分布压平一些，但不必压成完全平均。

常用公式是：

$$
p_l = \frac{n_l^\alpha}{\sum_k n_k^\alpha}
$$

其中：

- $n_l$ 是语言 $l$ 的样本数
- $p_l$ 是语言 $l$ 的采样概率
- $\alpha$ 控制压平程度

参数含义如下：

| 参数 | 含义 | 效果 |
|---|---|---|
| `α = 1` | 按原始分布采样 | 不平衡 |
| `0 < α < 1` | 幂次压缩 | 更平衡 |
| `α = 0` | 均匀采样 | 最平，但可能过强 |
| `α > 1` | 放大差异 | 一般不用于平衡 |

为什么这个公式有效？因为幂次会压缩数量级差异。若一个语言样本数是另一个的 1000 倍，那么：

- 在 $\alpha = 1$ 时，采样比仍然是 1000 倍
- 在 $\alpha = 0.5$ 时，采样比变成 $\sqrt{1000}$
- 在 $\alpha = 0.3$ 时，采样比变成 $1000^{0.3}$

也就是说，原本极端的不平衡被“拉近”了。

看一个最小玩具例子。假设三个语种样本数分别为 `1000 / 100 / 10`。

原始分布下：

$$
\frac{1000}{1110} \approx 90.1\%, \quad
\frac{100}{1110} \approx 9.0\%, \quad
\frac{10}{1110} \approx 0.9\%
$$

如果取 $\alpha = 0.3$，则权重大致为：

- $1000^{0.3} \approx 7.94$
- $100^{0.3} \approx 3.98$
- $10^{0.3} \approx 2.00$

归一化后约为：

| 语种样本数 | 原始占比 | `α=0.3` 后占比 |
|---|---:|---:|
| 1000 | 90.1% | 57.1% |
| 100 | 9.0% | 28.6% |
| 10 | 0.9% | 14.3% |

这正好说明了“不是平均，而是重新分配”。小语种不再几乎看不见，但大语种仍保留更高曝光。

为什么不能直接做完全均匀采样？因为样本数很小的语言会被反复重放。重放可以白话理解为：同一批数据被一遍又一遍重复使用。这样能提高出现次数，但也容易提高记忆而不是泛化。模型会更快背住小语种训练集，却不一定真正学会其分布。

真实工程例子比玩具例子更极端。多语种预训练里，英语、法语、西语可能有海量网页与图书数据，一些低资源语种只有新闻、百科和少量公开语料，规模差距可能达到 $10^2$ 到 $10^5$ 倍。此时如果直接按原始分布采样，英语会长期主导梯度；如果完全均匀采样，又会把低资源语种抬得过高，导致训练效率下降。因此很多系统会选一个中间值，例如 $\alpha \approx 0.2$ 到 $\alpha \approx 0.7$，具体数值再靠验证集调。

还要注意一个常见误解：采样平衡不等于最终能力一定平衡。因为模型能力还取决于语料质量、tokenizer 切分、脚本复杂度、领域覆盖和任务监督信号。采样只是在控制“被看见多少次”，不是单独决定“最终学会多少”。

## 代码实现

工程上，关键不是把公式写出来，而是让训练循环真的按这个概率抽样。最常见流程如下：

| 步骤 | 做什么 |
|---|---|
| 1 | 统计每个语种样本数 `n_l` |
| 2 | 设定 `α` |
| 3 | 计算语种采样概率 `p_l` |
| 4 | 将 `p_l` 映射到 sampler 权重 |
| 5 | 训练时按权重抽样 |

下面给一个可运行的 Python 例子。它做三件事：

1. 根据语种样本数计算温度采样概率。
2. 验证概率和为 1。
3. 通过随机采样观察经验分布是否接近理论分布。

```python
import random

def compute_probs(counts, alpha):
    assert counts, "counts cannot be empty"
    assert all(v > 0 for v in counts.values()), "all counts must be positive"
    assert alpha >= 0, "alpha must be non-negative"

    weights = {lang: count ** alpha for lang, count in counts.items()}
    total = sum(weights.values())
    probs = {lang: w / total for lang, w in weights.items()}
    return probs

def sample_languages(probs, steps=20000, seed=42):
    random.seed(seed)
    langs = list(probs.keys())
    weights = [probs[lang] for lang in langs]
    sampled = random.choices(langs, weights=weights, k=steps)

    freq = {lang: 0 for lang in langs}
    for lang in sampled:
        freq[lang] += 1

    empirical = {lang: c / steps for lang, c in freq.items()}
    return empirical

counts = {
    "en": 1000,
    "fr": 100,
    "sw": 10,
}

probs_raw = compute_probs(counts, alpha=1.0)
probs_balanced = compute_probs(counts, alpha=0.3)

assert abs(sum(probs_raw.values()) - 1.0) < 1e-9
assert abs(sum(probs_balanced.values()) - 1.0) < 1e-9
assert probs_raw["en"] > probs_balanced["en"]
assert probs_balanced["sw"] > probs_raw["sw"]

empirical = sample_languages(probs_balanced, steps=50000)

for lang in probs_balanced:
    assert abs(empirical[lang] - probs_balanced[lang]) < 0.03

print("raw:", probs_raw)
print("balanced:", probs_balanced)
print("empirical:", empirical)
```

如果你把这段代码跑起来，会看到 `alpha=0.3` 时，`sw` 的采样概率明显高于原始分布，但 `en` 仍是最大头。

再进一步，在真实训练中通常不是“先抽语种，再临时读文件”这么简单。更常见做法是：

1. 每条样本带上语言标签。
2. 先为语言分配权重。
3. 再把语种权重映射到样本级权重。
4. 交给 dataloader 或 sampler 处理。

原因在于训练框架通常直接面向样本，而不是面向“语种桶”。如果某语言有 100 条样本，而你给它高采样率，那么本质上就是这 100 条样本被更频繁地重复访问。

一个简化实现思路如下：

- 语种级概率决定大方向。
- 语种内部再均匀抽样，或按质量分层抽样。
- 若担心过拟合，可以限制低资源语种的最大重复率。

真实工程例子里，这个流程经常结合数据管道一起做。比如一个多语种预训练任务，样本来自 Common Crawl 清洗语料、百科语料和代码混合语料。团队会先按语言统计 token 数，再选定 $\alpha$ 生成语言级配额，然后在训练时动态混合不同 shard。这里的“shard”可以白话理解为：被切分出来便于并行读取的数据分片。这样做的好处是，不需要把所有语言先物理复制成同样大小，而是只在读取和采样阶段调整曝光率，成本更低，也更容易试验不同超参数。

## 工程权衡与常见坑

采样平衡不是越平越好。它本质上是在覆盖率和训练效率之间做折中。

最常见的坑如下：

| 坑 | 后果 | 规避方式 |
|---|---|---|
| 完全均匀采样 | 高资源语种训练预算被浪费 | 用温度采样代替硬平均 |
| 过度上采样低资源语种 | 小语种过拟合、整体收敛变慢 | 控制 `α`，监控重复率 |
| 只调采样不看 tokenizer | 小语种仍被切得很碎 | 联合检查词表和切分质量 |
| 只看总分 | 小语种退化被平均数掩盖 | 按语种拆开看验证指标 |
| 把采样和 loss 加权当成完全等价 | 实际优化行为偏差 | 分开做实验，不要想当然 |

先说第一个坑。完全均匀采样听起来最公平，但在数据极端不平衡时，往往是低效的。高资源语种通常不仅数据多，而且覆盖领域更广、表达形式更多。过早把它们压得太低，等于主动放弃大量有价值训练信号。

第二个坑是低资源语种重复率过高。重复率可以白话理解为：同一条样本在一个训练周期内被看见多少次。如果一个语种只有几百条句子，却被抬到和英语接近的 batch 占比，那么模型很可能先学会记住训练集，而不是学会这个语种的通用规律。

第三个坑经常被初学者忽略：只改采样，不改 tokenizer。比如某些拼写系统复杂或黏着性强的语言，本来就会被切成更长 token 序列。此时虽然样本条数上的曝光提高了，但 token 级效率仍未必高，甚至可能因为序列过长压缩了有效 batch 容量。

第四个坑是评估方式。多语种训练一定要看**按语种拆分的验证集指标**，不能只看总分。总分上涨，可能只是英语更强了；而你的目标语种反而退化。

建议监控以下观察项：

| 观察项 | 为什么重要 |
|---|---|
| 各语种验证集指标 | 防止总分掩盖退化 |
| 收敛速度 | 判断平衡是否过强 |
| 小语种重复率 | 判断是否接近过拟合 |
| batch 语种分布 | 检查采样是否符合设计 |
| 平均序列长度 | 看 tokenizer 和语种结构是否拖累吞吐 |

还有一个更细的工程问题：采样与 loss 加权不完全等价。表面上看，提升某语言出现次数，和在 loss 里给该语言更大权重，都像是在“强调”它。但在小 batch、随机梯度下降、动态混洗、优化器状态累积这些条件下，两者路径不同，最终效果也可能不同。简单说，前者改变了“看见什么”，后者改变了“看见后算多重”。它们不能机械视为一回事。

## 替代方案与适用边界

温度采样不是唯一方案，只是多数多语种训练里比较稳的默认起点。不同方法的适用场景如下：

| 方法 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 原始分布采样 | 实现简单，保留真实规模 | 小语种容易被淹没 | 高资源语种主导任务 |
| 完全均匀采样 | 覆盖最公平 | 可能严重过采样小语种 | 语种数少、规模差距小 |
| 温度采样 | 折中稳定 | 需要调 `α` | 多数多语种预训练 |
| 条件采样 | 能面向目标语种定向优化 | 设计更复杂 | 重点语种产品场景 |
| loss 加权 | 不必改采样管道 | 不等价于采样 | 某些已有训练框架 |
| 分阶段采样 | 可先广覆盖再偏目标语种 | 训练计划更复杂 | 预训练加下游适配 |

如果你的产品只服务 5 个重点语种，就没必要追求“100 个语种都公平”。此时更合理的策略是：预训练阶段保持一定长尾覆盖，微调阶段提高目标语种权重。也就是说，采样应该围绕业务目标设计，而不是围绕抽象公平设计。

再举一个反例。若某低资源语种总数据只有几万 token，且质量很差，那么单靠采样平衡通常救不了结果。因为模型反复看到的仍然是低质量、低覆盖、低多样性数据。这时更有效的手段可能是：

- 改善 tokenizer 或词表覆盖
- 引入相近语言迁移
- 做数据清洗与去重
- 进行翻译增强或蒸馏

因此，本文方法的适用边界很明确：

1. 它解决的是训练曝光率问题，不单独解决数据质量问题。
2. 当长尾极重时，采样必须和 tokenizer、去重、质量过滤一起设计。
3. 当目标是统一总分时，不能只服务某个单语种最优。
4. 当目标是重点市场语种时，应该允许策略性偏置，而不是追求形式平均。

一个实用建议是，把温度采样当作基线，从 `α=0.3`、`0.5`、`0.7` 这类点开始试，再根据语种级验证曲线决定是否继续压平。工程上最忌讳的不是参数没调到最优，而是先假设“越公平越好”，结果把训练预算消耗在无效重复上。
