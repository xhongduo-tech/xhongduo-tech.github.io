## 核心结论

知识图谱与 LLM 结合，主流上只有三条路径：

| 路径 | 做法 | LLM 的角色 | 图谱的角色 | 适合问题 |
| --- | --- | --- | --- | --- |
| 图谱检索增强生成 | 先查图，再生成 | 把事实组织成自然语言答案 | 提供可校验事实与路径 | 问答、检索、客服 |
| 图谱注入提示或参数 | 把三元组、子图或图嵌入直接送给模型 | 在已给定结构上归纳、解释、补全 | 提供结构化上下文 | 小规模闭域任务 |
| 图谱工具调用 | 把图查询当外部函数 | 负责决定何时调用、如何解释结果 | 负责精确查询与约束 | 需要高准确率的系统 |

这里的“知识图谱”可以先理解为“把事实存成节点和关系的网络”，例如“苹果公司-CEO-蒂姆·库克”。LLM 可以先理解为“大语言模型，擅长理解问题和组织语言，但不天然保证事实正确”。

这两者结合的核心，不是让模型“更会说”，而是让系统在回答时同时具备四个能力：有事实来源、有结构约束、可追踪路径、可独立更新。工程上最常见的收益是降低幻觉，代价则是更高的检索时延、更复杂的链路，以及更多缓存和版本管理工作。

---

## 问题定义与边界

这个问题要先限定边界：我们讨论的不是“用 LLM 自动生成一个知识图谱”，而是“图谱已经存在或可维护时，如何把它和 LLM 组合起来做问答、检索、抽取或决策支持”。

为什么要这样限定？因为二者解决的是不同问题。

- 知识图谱负责“把事实表示清楚”
- LLM 负责“把问题理解清楚，并把答案表达出来”

如果只用 LLM，系统往往有三个短板：

| 维度 | 纯 LLM | KG + LLM |
| --- | --- | --- |
| 事实可校验性 | 低，答案常只有文本 | 高，可回溯到三元组或查询路径 |
| 知识更新方式 | 常依赖再训练或外部检索 | 更新图数据即可生效 |
| 多跳关系推理 | 可能答对，但难解释 | 可显式走路径再生成 |
| 工程复杂度 | 低 | 中到高 |

“多跳”可以先理解为“答案需要经过多步关系才能找到”，例如从“张三在哪家公司任职”可能先要找到“张三认识的负责人”，再找到“负责人所在公司”，最后才能返回结果。

这类系统的边界主要由四件事决定：

| 边界项 | 含义 | 常见约束 |
| --- | --- | --- |
| 事实准确率 | 回答是否与数据一致 | 受图谱质量、检索召回、提示约束影响 |
| 上下文长度 | 能塞进模型的事实有多少 | 受 token 上限限制 |
| 检索时延 | 查图和拼装上下文要多久 | 多跳越深越慢 |
| 端到端复杂度 | 系统是否好维护 | 涉及图数据库、抽取链路、缓存、评估 |

因此，知识图谱与 LLM 的结合，不是一个“模型技巧”问题，而是一个“系统设计”问题。设计目标不是让每个问题都走图谱，而是判断哪些问题必须有结构化事实约束，哪些问题只需要文本检索就够了。

## 核心机制与推导

最常见的机制可以写成两步：

1. 从图谱中找出和问题最相关的路径或子图  
2. 把这些结构化事实作为上下文，交给 LLM 生成最终答案

设用户问题为 $q$，知识图谱为 $G$，检索出的候选路径为 $w_z$，则目标可以写成：

$$
IR=\arg\max_{w_z} P(w_z \mid q, G)
$$

这句话的意思很直接：在整个图谱里，找出“在当前问题下最相关”的那条路径或那组路径。

接着，生成阶段并不是让模型自由发挥，而是让它在已检索到的事实上生成：

$$
P(a \mid q, G) \approx P(a \mid q, w_z)
$$

其中 $a$ 是答案。这个近似表达的是：只要检索路径 $w_z$ 足够好，生成答案时就不必依赖模型内部的模糊记忆，而可以更多依赖显式事实。

### 玩具例子

图谱中有两条三元组：

- $(A, knows, B)$
- $(B, worksAt, C)$

问题是：“A 的公司是什么？”

如果只看第一条关系，答案并不存在；必须走两跳：

$$
A \rightarrow B \rightarrow C
$$

于是可用路径：

$$
w_z = [(A,knows,B),(B,worksAt,C)]
$$

把它改写成提示上下文：

- 已知事实1：A 认识 B
- 已知事实2：B 在 C 工作

再交给 LLM，它就能生成“根据现有事实，A 关联到的公司是 C”。这里的关键不是语言更华丽，而是答案来自明确路径，可追踪、可审计。

### 三条机制的本质差别

| 机制 | 事实进入模型的时机 | 约束强度 | 典型问题 |
| --- | --- | --- | --- |
| KG-RAG | 生成前检索 | 中高 | 如何从图中找对路径 |
| 图谱注入 | 提示或参数阶段 | 中 | 上下文太大时怎么压缩 |
| 工具调用 | 生成中按需查询 | 高 | 如何让模型稳定决定调用 |

其中“工具调用”可以先理解为“把图数据库查询封装成一个函数，模型需要时去调用”。这比把整个子图直接塞进 prompt 更稳，因为事实查询和语言生成被拆开了。

真实系统中，多跳检索通常不会暴力遍历整张图，而是做“候选实体识别 + 关系扩展 + 路径评分”。“路径评分”可以先理解为“判断哪条关系链最像用户真正想问的那条链”。评分可以来自规则、嵌入模型，或者 LLM 本身的重排能力。

## 代码实现

下面给一个最小可运行版本。它不是完整生产系统，但足以说明“实体识别 -> 图查询 -> 路径格式化 -> 生成答案”的主链路。

```python
from collections import defaultdict, deque

class SimpleKG:
    def __init__(self, triples):
        self.graph = defaultdict(list)
        for s, r, o in triples:
            self.graph[s].append((r, o))

    def find_path(self, start, target_rel, max_hops=3):
        queue = deque([(start, [], 0)])
        visited = set([start])

        while queue:
            node, path, hops = queue.popleft()
            if hops >= max_hops:
                continue

            for rel, nxt in self.graph[node]:
                new_path = path + [(node, rel, nxt)]
                if rel == target_rel:
                    return new_path
                if nxt not in visited:
                    visited.add(nxt)
                    queue.append((nxt, new_path, hops + 1))
        return []

class KGLLMCoupling:
    def __init__(self, kg):
        self.kg = kg

    def ner(self, question):
        # 极简示例：假设问题总是 "A 的公司是什么"
        return "A", "worksAt"

    def format_paths(self, path):
        return "\n".join([f"{s} --{r}--> {o}" for s, r, o in path])

    def generate(self, question, facts):
        # 用规则模拟 LLM 生成，只为说明接口
        if facts.strip().endswith("C"):
            return f"问题：{question}\n答案：C\n依据：\n{facts}"
        return f"问题：{question}\n答案：未找到"

    def answer_query(self, question):
        entity, target_rel = self.ner(question)
        path = self.kg.find_path(entity, target_rel)
        context = self.format_paths(path)
        return self.generate(question, context)

triples = [
    ("A", "knows", "B"),
    ("B", "worksAt", "C"),
]

kg = SimpleKG(triples)
system = KGLLMCoupling(kg)
result = system.answer_query("A 的公司是什么")

assert "答案：C" in result
assert "B --worksAt--> C" in result
print(result)
```

这个示例故意把“生成器”写成规则函数，而不是直接接入真实模型。原因是主线应当先清楚：图谱系统的关键价值不在于调用了哪个模型，而在于你是否把事实链条明确地抽出来了。

真实工程里，链路通常会扩成下面这样：

| 阶段 | 输入 | 输出 | 常见实现 |
| --- | --- | --- | --- |
| 实体抽取 | 用户问题 | 候选实体、关系 | 规则、NER、小模型 |
| 图检索 | 实体、关系、约束 | 路径、子图、元数据 | Cypher、SPARQL、图索引 |
| 上下文压缩 | 路径集合 | 可放入 prompt 的事实摘要 | 模板、排序、摘要器 |
| LLM 生成 | 问题 + 事实 | 自然语言答案 | 通用 LLM |
| 结果校验 | 答案 + 路径 | 引用、置信度、审计记录 | 规则或二次验证 |

### 真实工程例子

以企业内部问答为例。用户问：“过去一年与 A 产品兼容的数据库有哪些，且当前仍在支持期内？”

如果只用普通文本 RAG，可能会召回很多产品文档、公告和 FAQ，然后让模型自己总结。问题在于“兼容关系”和“支持期”经常分散在不同文档里，模型需要跨文档拼接，很容易漏条件。

如果使用知识图谱，可以把事实建成：

- 产品A -兼容- 数据库X
- 数据库X -支持截止日期- 2027-12-31
- 数据库X -版本- 13
- 产品A -兼容版本- 13

这样查询时先过滤“兼容关系”，再过滤“支持期 > 今天”，最后把命中的实体列表与证据路径一起交给 LLM。LLM 只负责把结构化结果说清楚，而不是自己猜。

这类系统的收益通常不体现在“平均回答更像人”，而体现在“高风险问题更不容易乱答”。

## 工程权衡与常见坑

最常见的误区，是把知识图谱当成“又一个检索源”，然后把查到的整个子图直接塞进 prompt。这样很快会遇到三个问题。

### 1. 上下文爆炸

图谱的优点是结构清楚，但它的表示很啰嗦。一个节点可能连出几十条边，一次多跳后子图会迅速膨胀。如果原样展开，token 会失控。

| 坑 | 后果 | 规避方法 |
| --- | --- | --- |
| 子图过大 | 请求超长、成本高、模型忽略重点 | 路径裁剪、事实排序、摘要压缩 |
| 无关边过多 | 噪声进入 prompt | 先做关系过滤，再做实体扩展 |
| 元数据丢失 | 答案无法审计 | 保留来源、时间戳、版本号 |

一个常见做法是只保留“支持最终答案的最短证据链”，而不是整个邻域子图。

### 2. 检索正确但生成偏离

即使路径对了，模型仍可能把事实说歪。比如图中是“产品 A 兼容数据库 X 的 13 版”，模型却写成“兼容所有版本”。

原因通常是 prompt 没有告诉模型“只能依据给定事实回答”。所以生成模板必须强约束，例如：

- 仅根据给定事实回答
- 若事实不足，明确说未知
- 答案后附依据路径

这不是形式主义，而是减少“语言补全覆盖事实边界”。

### 3. 图谱与模型版本漂移

“版本漂移”可以先理解为“图数据已经更新，但缓存、摘要或模型侧提示仍是旧的”。结果是系统查到了新事实，但回答仍像旧系统。

真实工程里至少要管三类版本：

| 对象 | 为什么要版本化 | 示例 |
| --- | --- | --- |
| 图谱数据 | 事实会更新 | 合规规则、产品状态 |
| 提示模板 | 约束策略会变 | 是否要求引用路径 |
| 检索器/重排器 | 召回行为会变 | 路径评分模型升级 |

如果这三者不联动，就会出现“同一问题今天和明天答案不同，却说不清为什么”的情况。

### 4. 时延不可控

多跳图查询、路径打分、子图摘要、模型生成都要时间。对在线业务来说，准确率不是唯一目标，99 线时延同样关键。

常见优化方式有：

- 热点问题缓存最终答案和证据路径
- 高频实体预计算邻域摘要
- 把“找路径”和“生成答案”拆成两段服务
- 对简单单跳问题走快速通道，不进入复杂图检索

经验上，系统越强调可审计，就越需要接受一定的时延成本；系统越追求低延迟，就越需要更激进的缓存和更短的证据链。

## 替代方案与适用边界

并不是所有场景都需要知识图谱。很多团队的问题，不是“缺少图谱”，而是“文本检索还没做好就过早上图”。

可以这样判断：

| 方案 | 适用场景 | 优势 | 边界 |
| --- | --- | --- | --- |
| 纯 LLM | 开放式写作、泛问答 | 最简单 | 事实不可控 |
| 文本 RAG | 文档问答、知识库搜索 | 成本低、迭代快 | 多跳关系弱 |
| KG + LLM | 合规、配置、依赖关系、主数据问答 | 可追踪、可校验 | 建图和维护成本高 |
| 混合 RAG | 文本检索 + 图检索 | 兼顾覆盖率与准确率 | 系统复杂度最高 |

### 什么时候不必上图

如果知识主要存在于连续文本里，问题也多是摘要、改写、归纳，那么文本 RAG 往往足够。比如帮助中心问答、文档摘要、周报生成，重点是覆盖面和响应速度，不一定需要图谱。

### 什么时候应该上图

当问题具备下面任一特征，就值得考虑知识图谱：

- 答案依赖明确实体关系
- 需要多跳推理
- 必须给出来源和证据链
- 知识更新频繁，但不希望频繁训模型
- 回答错误代价高

例如金融合规、设备依赖分析、企业权限关系、药品与适应症映射，这些都不是“把文档召回回来让模型总结”就能稳妥解决的问题。

因此，一个实用判断标准是：

$$
\text{是否引入 KG} \approx f(\text{错误代价}, \text{关系复杂度}, \text{审计要求}, \text{更新频率})
$$

错误代价越高、关系越复杂、审计要求越强、知识更新越快，知识图谱与 LLM 的组合就越有价值。
