## 核心结论

T0 是一种多任务提示微调方法：它把情感分析、问答、自然语言推理、摘要等多种 NLP 任务统一改写成“文本输入 -> 文本输出”的生成问题，然后在大量带 prompt 的任务上一起训练模型。

这里的 prompt，指写给模型的自然语言任务说明，例如“判断这条评论是正面还是负面”。多任务提示微调，指同一个模型同时学习多个任务、多个 prompt 模板和多个输出格式，而不是为每个任务单独训练一个分类器。

T0 的核心不是“再训练一个更大的分类器”，而是让模型学会：看到一段自然语言任务说明后，理解任务意图，并按照文本格式作答。传统方法通常为情感分析、问答、改写分别设计不同模型头；T0 方法则把它们都转换成生成文本。例如输入评论 `This movie is great`，prompt 是“这条评论情感是正面还是负面？”，模型直接输出 `positive` 或 `negative`。

| 维度 | 传统任务建模 | T0 建模 | 输出形式 | 泛化方式 |
|---|---|---|---|---|
| 情感分析 | 分类器 + 情感标签 | prompt + 文本生成 | `positive` / `negative` | 学会理解情感判断类 prompt |
| 问答 | 阅读理解模型头 | prompt + 答案文本生成 | 一段答案文本 | 学会从上下文中抽取或生成答案 |
| 自然语言推理 | 三分类模型头 | prompt + 标签文本生成 | `entailment` / `neutral` / `contradiction` | 学会判断句子关系 |
| 新任务迁移 | 通常需要重新训练或微调 | 给出新 prompt 后直接推理 | 文本答案 | 依赖多任务训练获得的任务理解能力 |

训练目标可以写成：

$$
\mathcal L(\theta)= - \mathbb E_{(x,y,p)\sim \mathcal D}\big[\log p_\theta(y\mid \text{prompt}(x,p))\big]
$$

其中 `x` 是原始输入，`p` 是 prompt 模板，`y` 是目标输出文本。T0 的重点不是让模型死记每个数据集的标签，而是让模型在大量任务描述中学习“任务说明 -> 输出格式”的映射。

## 问题定义与边界

T0 主要解决两个问题：任务碎片化和新任务迁移差。

任务碎片化，指不同 NLP 任务被做成不同输入格式、不同标签空间、不同模型头。例如情感分析输出类别，问答输出文本 span，摘要输出长文本，自然语言推理输出三分类标签。工程上每接入一个任务，就要重新设计数据格式、模型结构或评测逻辑。

新任务迁移差，指模型在已有任务上训练得不错，但遇到训练时没见过的新任务时，很难直接使用。T0 的思路是把任务统一为条件生成问题：

$$
x, p \rightarrow y
$$

其中 `x` 是输入文本，`p` 是任务说明，`y` 是答案文本。只要任务能自然地写成“给定这段输入，请输出某个文本答案”，就可以放进 T0 这类框架。

| 任务类型 | 是否可文本化 | 是否适合统一成生成 | 备注 |
|---|---:|---:|---|
| 情感分析 | 是 | 是 | 输出 `positive` / `negative` 等标签文本 |
| 客服工单分类 | 是 | 是 | 输出 `退款` / `物流` / `技术支持` 等类别 |
| 阅读理解问答 | 是 | 是 | 输出答案短语或句子 |
| 文本改写 | 是 | 是 | 本来就是文本到文本任务 |
| 命名实体抽取 | 部分是 | 部分适合 | 简单实体可文本化，复杂嵌套结构更难 |
| 实时库存调度 | 否 | 不适合 | 需要约束优化、实时数据和工具调用 |
| 代码编译与执行 | 部分是 | 单纯 T0 不够 | 需要外部解释器或运行环境验证 |

玩具例子：给定 `x = "This movie is great"`，prompt `p = "判断这条评论是 positive 还是 negative："`，目标输出 `y = "positive"`。这就是一个最小的文本化分类任务。

真实工程例子：客服系统里有一条工单：“我的快递三天没有更新物流信息，能帮我查一下吗？”可以写成 prompt：“请判断这条工单属于退款、物流还是技术支持。”目标输出是 `物流`。这类任务适合 T0，因为输入和输出都可以稳定表达成文本。

边界也要说清楚。零样本不等于不要 prompt。零样本通常指目标任务没有额外训练样本，但推理时仍然需要一个清晰的任务说明。T0 的泛化也不是魔法，它来自训练阶段覆盖了大量任务和多种模板，使模型见过足够多的“任务表述方式”。

## 核心机制与推导

T0 的机制可以拆成三步：统一任务格式、混合多任务训练、用生成概率做推理。

第一步是统一任务格式。判别任务，指从固定标签里选一个答案，例如二分类或三分类。生成任务，指模型逐 token 生成输出文本。T0 把很多判别任务改写成生成任务：不是额外接一个分类头，而是让模型生成标签文本。

例如情感分析：

| 输入 | prompt | 候选输出 | 选择规则 |
|---|---|---|---|
| `This movie is great` | `判断情感：` | `positive`, `negative` | 选条件概率更高的候选文本 |
| `The package never arrived` | `这条工单属于退款、物流还是技术支持？` | `退款`, `物流`, `技术支持` | 选生成概率最高的类别文本 |
| `A dog is running. / An animal is moving.` | `第二句话是否能由第一句话推出？` | `entailment`, `neutral`, `contradiction` | 比较候选答案概率 |

推理时可以写成：

$$
\hat y=\arg\max_y p_\theta(y\mid x,p)
$$

也就是说，对每个候选答案文本计算条件概率，选择概率最大的那个。最小数值例子如下：输入 review 是 `This movie is great`，候选输出是 `positive` 和 `negative`。如果模型给出 `log p(positive|x,p) = -0.20`，`log p(negative|x,p) = -1.80`，因为 `-0.20 > -1.80`，所以预测 `positive`。

第二步是多任务混合训练。设共有 $N$ 个数据集，每个数据集是 $D_i$，权重是 $\lambda_i$，训练目标为：

$$
\mathcal L(\theta)=\sum_{i=1}^N \lambda_i \, \mathbb E_{(x,y,p)\sim D_i}\big[-\log p_\theta(y\mid \text{prompt}(x,p))\big]
$$

这里的 $\lambda_i$ 控制不同任务在训练中的占比。权重太高的大数据集会主导训练，权重太低的小任务又学不到，因此混合策略本身就是重要工程问题。

第三步是 prompt 多样化。同一条问答样本可以有多个模板：

| 模板 | 输入形式 | 目标 |
|---|---|---|
| `根据文章回答问题：{question}\n文章：{context}` | 问题 + 文章 | 答案 |
| `请从下面文本中找出答案。\n文本：{context}\n问题：{question}` | 文章 + 问题 | 答案 |
| `Question: {question}\nContext: {context}\nAnswer:` | 英文格式 | 答案 |

模型在这些不同写法中学到：虽然表面文字不同，但任务意图相同。这个过程会增强模型对 prompt 改写的鲁棒性。

为什么这有效，可以从四点理解。第一，统一输出空间，所有任务都变成文本生成。第二，共享参数，不同任务的语言理解能力可以相互迁移。第三，多模板训练减少模型对单一说法的依赖。第四，任务理解发生在 prompt 和输入共同形成的上下文中，而不是固定在某个分类头里。

## 代码实现

T0 式实现的核心不是复杂模型结构，而是数据格式。训练数据需要同时保存输入、输出、prompt 和任务名。

| 字段 | 含义 | 示例 |
|---|---|---|
| `x` | 原始输入 | `This movie is great` |
| `y` | 标准答案文本 | `positive` |
| `p` | prompt 模板 | `Review: {x}\nSentiment:` |
| `task_name` | 任务名称 | `sentiment` |

最小训练伪代码是：

```python
prompted_input = build_prompt(x, p)
loss = -log_prob(model, prompted_input, y)
pred = generate(model, prompted_input)
```

下面是一个可运行的 Python 玩具实现。它不用真实神经网络，而是模拟“候选答案评分”这个关键机制：比较每个候选输出的对数概率，选择最高者。

```python
def build_prompt(x, template):
    return template.format(x=x)

class ToyModel:
    def logprob(self, prompted_input, candidate):
        scores = {
            ("Review: This movie is great\nSentiment:", "positive"): -0.20,
            ("Review: This movie is great\nSentiment:", "negative"): -1.80,
            ("工单：快递三天没有更新物流信息\n类别：", "物流"): -0.10,
            ("工单：快递三天没有更新物流信息\n类别：", "退款"): -2.30,
            ("工单：快递三天没有更新物流信息\n类别：", "技术支持"): -1.70,
        }
        return scores.get((prompted_input, candidate), -100.0)

def predict_by_candidates(model, x, template, candidates):
    prompted_input = build_prompt(x, template)
    scores = {
        y: model.logprob(prompted_input, y)
        for y in candidates
    }
    return max(scores, key=scores.get), scores

model = ToyModel()

pred, scores = predict_by_candidates(
    model,
    x="This movie is great",
    template="Review: {x}\nSentiment:",
    candidates=["positive", "negative"],
)
assert pred == "positive"
assert scores["positive"] > scores["negative"]

ticket_pred, ticket_scores = predict_by_candidates(
    model,
    x="快递三天没有更新物流信息",
    template="工单：{x}\n类别：",
    candidates=["退款", "物流", "技术支持"],
)
assert ticket_pred == "物流"
assert ticket_scores["物流"] > ticket_scores["退款"]
```

真实工程中，`ToyModel.logprob` 会替换成 T5、T0 或其他文本到文本模型的条件对数似然计算。生成式推理则可以直接调用 `generate`，让模型输出答案文本。但在分类任务上，候选答案评分通常更稳定，因为它避免模型生成 `positive.`、`Positive`、`it is positive` 这类难以精确匹配的变体。

工程版例子：客服工单自动路由系统可以把多个任务统一成 prompt。

| 任务 | prompt | 输出 |
|---|---|---|
| 工单分类 | `请判断工单属于退款、物流还是技术支持：{x}` | `物流` |
| 紧急程度判断 | `请判断工单紧急程度：低、中、高。工单：{x}` | `中` |
| 是否转人工 | `这条工单是否需要人工介入？回答是或否：{x}` | `是` |

关键约定有三条。第一，prompt 是输入的一部分，不能只把它当注释。第二，标签文本要标准化，例如统一小写、去掉句号、限制候选集合。第三，训练和评测的 task split 要严格分开，不能只按样本随机切分，否则同一个任务的模板可能同时出现在训练和测试中，导致虚高结果。

## 工程权衡与常见坑

T0 的效果强依赖 prompt 设计、模板覆盖、数据混合和评测方式。模型结构不是唯一重点，很多失败来自数据和格式细节。

| 问题 | 影响 | 规避方法 |
|---|---|---|
| prompt 太单一 | 模型只适应某一种问法 | 每个任务保留多个模板 |
| 输出格式不统一 | 评测时同义答案被判错 | 做答案规范化和候选约束 |
| 训练/评测泄漏 | 零样本结果虚高 | 按任务级别切分 held-out 任务 |
| 数据集权重失衡 | 大数据集压制小任务 | 使用采样权重或温度采样 |
| prompt 敏感 | 换个说法性能波动 | 多 prompt 评测，报告均值或中位数 |
| 推理精度问题 | 低精度可能改变概率排序 | 优先用 `bf16` 或 `fp32` 验证关键结果 |
| 标签语义不清 | 模型不知道标签边界 | 给出明确候选项和输出约束 |

新手容易把“零样本”理解成“随便写 prompt 都能工作”。这个理解不准确。T0 学到的是任务说明与输出格式之间的关系，但 prompt 仍然需要清楚。比如“判断是否积极”和“这条评论情绪如何”都能表达情感分析；如果训练时只有一种模板，模型可能对另一种写法不稳定。

输出格式也是高频坑。`Positive`、`positive`、`positive.`、`正面` 如果混在同一个评测集中，模型可能语义正确但字符串匹配失败。工程上通常要定义标签规范化函数：去空格、统一大小写、去末尾标点，并把同义答案映射到同一标签。

数值精度也要谨慎。分类时如果两个候选答案概率很接近，低精度计算可能改变排序。简单对照如下：

| 精度 | 适用场景 | 风险说明 |
|---|---|---|
| `fp32` | 评测、复现实验、概率排序敏感任务 | 显存占用高 |
| `bf16` | 训练和推理折中 | 通常比 `fp16` 更稳 |
| `fp16` | 显存紧张的推理 | 可能出现数值不稳定或排序变化 |

评测时也不应只看单个 prompt。更稳妥的方式是为同一任务准备多个 prompt，分别评测，然后报告平均值、中位数或最差表现。平均值反映整体性能，中位数减少极端模板影响，最差表现能暴露 prompt 脆弱性。

## 替代方案与适用边界

T0 不是所有场景的最优解。它适合任务数量多、任务可文本化、输出能规范化、并且希望获得零样本或少样本迁移的场景。如果只做一个固定分类任务，专用分类器通常更简单、更便宜、更容易上线。

| 方案 | 优点 | 缺点 | 适合场景 |
|---|---|---|---|
| 专用分类器 | 训练便宜、延迟低、评测简单 | 新任务需要重新训练 | 单一稳定分类任务 |
| 共享编码器多头模型 | 多任务共享表示，输出头清晰 | 每个任务仍需专用头和标签设计 | 任务集合固定、输出结构不同 |
| 普通指令微调 | 指令跟随能力强，覆盖面广 | 数据质量和指令分布影响大 | 通用助手、开放式问答 |
| T0 式 prompted multitask training | 统一文本到文本格式，零样本迁移强 | prompt 和标签规范成本高 | 多个 NLP 任务、希望快速接入新任务 |
| 检索增强或工具调用方案 | 能接外部知识和实时系统 | 系统复杂度更高 | 需要实时知识、数据库、计算或强约束执行 |

T0 的适用边界可以概括为四点。第一，任务数量多，单独维护多个模型或多个头成本太高。第二，标签可以文本化，例如 `退款`、`物流`、`技术支持`。第三，希望模型在未见过的新任务上有一定零样本或少样本能力。第四，可以接受 prompt 设计、模板管理和评测复杂度。

不适合的场景也很明确。如果任务要求严格 JSON schema、复杂约束搜索、实时数据库查询、库存价格联动，单纯文本生成不够。此时更合理的是把 T0 或指令模型作为语言理解层，再接结构化解析、规则系统、检索系统或工具调用。

对于零基础到初级工程师，判断是否使用 T0 可以用一个简单标准：如果你的任务能稳定写成“给定输入和一句任务说明，输出一段短文本”，T0 式方法值得考虑；如果你的任务核心是外部状态、强约束决策或毫秒级低延迟，专用模型、规则系统或工具增强方案通常更合适。
