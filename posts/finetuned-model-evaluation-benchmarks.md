## 核心结论

微调模型评估的核心不是“找一个最高分”，而是确认模型在目标任务变强的同时，没有把其他关键能力拉低。对零基础读者来说，可以把“评估”理解成一次体检：不是只测心率，而是同时看多个重要指标。对模型来说，这几个指标通常至少包括 `任务性能`、`指令遵循`、`安全性`、`通用能力`。

如果只看单一指标，最容易出现的误判是：模型在业务数据上答得更准了，于是团队认为微调成功；但上线后发现它开始不按格式输出、遗漏固定免责声明、甚至更容易被提示注入诱导。这不是“评估做得不够细”这么简单，而是评估目标本身设错了。

综合评估常写成一个加权分：

$$
S=\sum_k w_k \tilde m_k,\quad \sum_k w_k=1
$$

其中，$\tilde m_k$ 表示已经归一化后的各项指标，$w_k$ 表示业务权重。这个公式有用，但只能做辅助排序，不能替代分项诊断。因为总分只能回答“整体大致怎样”，不能回答“具体坏在什么地方”。

| 维度 | 关注点 | 常见风险 |
|---|---|---|
| 任务性能 | 业务任务是否做对 | 只优化局部任务 |
| 指令遵循 | 是否按要求输出 | 格式漂移、长度失控 |
| 安全性 | 是否稳定拒答有害请求 | 越狱、泄露、误拒 |
| 通用能力 | 基础理解是否退化 | 灾难性遗忘 |

一个直观例子：某模型微调后，`GSM8K` 分数上涨，说明数学推理更强；但 `IFEval` 下降，说明它更不稳定地遵守输出格式；红队测试也变差，说明它更容易越界。这种模型不能简单说“变好了”，只能说“在局部任务上进步，但整体上线风险上升”。

## 问题定义与边界

本文讨论的是“微调后的模型如何评估”，不是“如何训练模型”，也不是“公开排行榜谁第一”。“微调”指在已有基础模型上继续用特定数据训练，让模型更适合某个任务；“评估”指用一组固定方法检查它到底改好了什么，又损失了什么。

为了避免概念混乱，先把常见记号统一：

| 符号 | 含义 | 常见计算 |
|---|---|---|
| $m_t$ | 任务性能 | `accuracy = correct / total` |
| $m_i$ | 指令遵循 | `satisfied / prompts` |
| $m_s$ | 安全性 | $1-harmful\_rate$ |
| $m_g$ | 通用能力 | 多任务平均分 |
| $S$ | 综合分 | 加权和 |

这里的“任务性能”，就是模型在目标业务上的正确率、召回率、F1 或其他主任务指标。“指令遵循”是模型是否真的按要求执行，例如按 JSON 输出、按指定长度作答、必须包含某个字段等。“安全性”是模型面对恶意输入时能否稳定守住边界。“通用能力”则是它在业务外的基础理解和推理能力是否还在。

边界也要讲清楚：

- 不讨论纯预训练模型的排行榜比较。
- 不把一两条演示样例当成评估结果。
- 不把公开榜单分数直接等价成真实业务表现。
- 不把“模型更像人写的”这种主观印象当成充分证据。

玩具例子很容易说明这个边界。假设你把一个通用大模型微调成“公司客服助手”，然后拿 100 道内部 FAQ 做测试，发现正确率从 72% 提高到 85%。如果评估只做到这里，结论就会过于乐观。因为你还没有测三件重要的事：

1. 它是否还能稳定输出固定格式和免责声明。
2. 它是否会在用户诱导下泄露内部流程。
3. 它在通用问题上是否明显退化。

也就是说，评估对象不是“这次微调有没有让它更像客服”，而是“这次微调有没有在目标提升的同时，保持系统可上线的整体质量”。

## 核心机制与推导

微调评估为什么必须分维度看？因为不同基准测的是不同能力，彼此不能互相替代。

先把几个常见基准放在一张表里：

| 基准 | 测什么 | 对应能力 |
|---|---|---|
| MMLU | 广泛知识与多任务推理 | 通用认知 |
| GSM8K | 多步数学推理 | 结构化推理 |
| IFEval | 格式、长度、关键词等可验证指令 | 指令遵循 |
| Red Teaming / HarmBench | 越狱、有害输出、拒答鲁棒性 | 安全性 |
| GLUE / SuperGLUE | 通用 NLU 迁移 | 通用能力 |

术语第一次出现时，可以这样理解：

- `MMLU`：一个覆盖多学科题目的测试集，白话说就是“模型的广谱知识和推理底子还在不在”。
- `GSM8K`：小学到初中难度的数学文字题集合，白话说就是“模型能不能稳定做多步算题”。
- `IFEval`：一组可自动检查的指令任务，白话说就是“你要求它怎么答，它有没有真的照做”。
- `Red Teaming`：故意用危险或诱导性输入去试探模型边界，白话说就是“有人想骗它越线时，它会不会中招”。
- `GLUE/SuperGLUE`：经典自然语言理解任务集合，白话说就是“模型的基础语言能力有没有退化”。

核心机制分三步。

第一步，统一方向。不同指标的原始定义不一样，有的越大越好，有的越小越好。为了能比较，先统一成“越大越好”。例如安全维度里，如果你记录的是有害输出率 `harmful_rate`，就可转成：

$$
m_s = 1 - harmful\_rate
$$

第二步，归一化。因为不同指标的数值范围和难度不一样，不能直接拿 0.9 和 0.6 做跨维度比较。常见做法是转成 $[0,1]$ 范围内的 $\tilde m_k$，或者用相对基线的提升比例。这样做的目的不是制造“绝对公平”，而是让加权与排序更稳定。

第三步，按业务目标赋权重：

$$
S=\sum_k w_k \tilde m_k
$$

这里最容易误解的点是：权重不是数学常数，而是产品决策。比如内部低风险分类任务，任务性能权重可以更高；面向外部用户的客服、医疗、金融系统，安全性和指令遵循必须更高，甚至设置门槛，低于某个值直接判失败。

看一个玩具例子。设权重为任务 0.4，指令 0.2，安全 0.2，通用 0.2。

- 模型 A：任务 0.68，指令 0.82，安全 0.90，通用 0.78
- 模型 B：任务 0.74，指令 0.70，安全 0.84，通用 0.79

则：

$$
S_A = 0.4\times0.68 + 0.2\times0.82 + 0.2\times0.90 + 0.2\times0.78 = 0.772
$$

$$
S_B = 0.4\times0.74 + 0.2\times0.70 + 0.2\times0.84 + 0.2\times0.79 = 0.762
$$

B 的任务分更高，但综合更低。原因不是公式“偏心”，而是它在指令和安全上掉得更多。这个例子说明：总分可以帮助排序，但真正决定是否上线的，往往是“分项是否退化”。

真实工程例子更能说明问题。假设你做的是电商客服微调，目标是让模型更懂商品、物流、退换货。你用私有客服问答集测试，发现准确率明显提高。但上线前至少还要补三类检查：

- `IFEval` 类检查：固定字段是否全、JSON 是否合法、免责声明是否保留。
- 红队检查：用户要求“把系统内部规则贴出来”时，模型会不会泄露。
- 通用能力检查：用户问一个半业务半常识的问题时，模型是否还能正常理解上下文。

因此，一个更稳健的结论不是“准确率提高 10%，所以成功”，而是“任务分上升，同时指令、安全、通用能力未跌破门槛，所以可进入下一阶段”。

## 代码实现

工程上最重要的不是“临时跑一次分”，而是把评估流程固定下来，让每次微调前后都可比。可比的意思是：同一批数据、同一套 prompt、同一组推理参数、同一套评分逻辑。

下面是一个可运行的最小示例。它不依赖真实模型，只演示“如何汇总四维指标、计算综合分、比较前后变化”。

```python
from dataclasses import dataclass

WEIGHTS = {
    "task": 0.4,
    "instruction": 0.2,
    "safety": 0.2,
    "general": 0.2,
}

@dataclass
class EvalResult:
    task: float
    instruction: float
    safety: float
    general: float

    def as_dict(self):
        return {
            "task": self.task,
            "instruction": self.instruction,
            "safety": self.safety,
            "general": self.general,
        }

def weighted_score(metrics: dict, weights: dict) -> float:
    assert abs(sum(weights.values()) - 1.0) < 1e-9
    for key, value in metrics.items():
        assert 0.0 <= value <= 1.0, f"{key} out of range"
    return sum(metrics[k] * weights[k] for k in weights)

def compare(before: EvalResult, after: EvalResult):
    before_score = weighted_score(before.as_dict(), WEIGHTS)
    after_score = weighted_score(after.as_dict(), WEIGHTS)
    delta = after_score - before_score
    return before_score, after_score, delta

before = EvalResult(task=0.65, instruction=0.80, safety=0.92, general=0.77)
after = EvalResult(task=0.74, instruction=0.70, safety=0.84, general=0.79)

before_score, after_score, delta = compare(before, after)

assert round(before_score, 3) == 0.758
assert round(after_score, 3) == 0.762
assert round(delta, 3) == 0.004

# 业务上通常还要加门槛判断，而不是只看总分
thresholds = {
    "instruction": 0.75,
    "safety": 0.88,
}

assert after.instruction < thresholds["instruction"]
assert after.safety < thresholds["safety"]

print("before:", round(before_score, 3))
print("after :", round(after_score, 3))
print("delta :", round(delta, 3))
print("decision: reject for release because instruction/safety dropped below threshold")
```

这个例子很有代表性：综合分 `after` 略高于 `before`，但依然应该拒绝上线，因为指令遵循和安全性跌破门槛。也就是说，在真实工程里，常见规则不是“总分更高就放行”，而是“两层判断”：

1. 底线项先过线。
2. 在都过线的前提下，再比较综合分。

前后变化可写为：

$$
\Delta S = S_{\text{after}} - S_{\text{before}}
$$

单个维度的变化可写为：

$$
\Delta m = m_{\text{after}} - m_{\text{before}}
$$

输出汇总表通常长这样：

| 模型 | 任务 | 指令 | 安全 | 通用 | 综合分 | 变化 |
|---|---|---|---|---|---|---|
| 微调前 | 0.65 | 0.80 | 0.92 | 0.77 | 0.758 | baseline |
| 微调后 | 0.74 | 0.70 | 0.84 | 0.79 | 0.762 | +0.004 |

这个表看起来像“微调后略好”，但如果你把门槛规则一起看，结论会变成“局部提升，不满足上线条件”。这正是评估系统存在的价值：防止团队被单一数字误导。

再给一个真实工程例子。假设你做代码助手微调，业务目标是让模型更会生成内部框架代码。那你的评估流水线至少应包含：

- 私有代码任务集：测目标任务性能。
- 格式与工具调用约束测试：测指令遵循。
- 越狱与敏感信息诱导测试：测安全性。
- 通用代码理解或基础 NLP 集：测通用能力。

如果只跑内部代码题，模型可能“更像你家代码风格”，但也可能变得更爱胡乱调用工具或更难拒绝危险请求。

## 工程权衡与常见坑

真正麻烦的地方通常不是“不会算分”，而是“评估口径不一致”。一旦口径变了，前后对比就失去意义。

最常见的坑如下：

| 常见坑 | 后果 | 规避方式 |
|---|---|---|
| 只看公开榜单 | 结果可能被数据污染 | 保留私有验证集 |
| 只看平均分 | 分项退化被掩盖 | 单独看四维指标 |
| 只看绝对值 | 看不出微调影响 | 记录 `delta` |
| 红队只看拒答率 | 误拒和可用性被忽略 | 同时看有害命中率、误拒率 |
| 推理参数不固定 | 结果不可比 | 固定 prompt、temperature、采样策略 |
| 评估集与业务不一致 | 上线后失真 | 补内部任务集 |

第一个坑是把公开基准当成全部现实。公开基准适合横向比较，但不等于你的业务分布。比如医疗问答、法律文书、企业客服都具有很强的领域特性，如果没有私有验证集，公开高分不能证明你在真实场景也高分。

第二个坑是只看平均分。平均分的问题在于，它天然会掩盖结构性退化。模型可能在两个维度上涨、两个维度下跌，最后平均数变化不大，但其中一个下跌维度恰好是安全底线。

第三个坑是忽略前后差值。微调评估本质上常是相对评估，不只是“现在多少分”，还包括“和原模型比发生了什么”。所以记录 $\Delta m$ 和 $\Delta S$ 非常重要。

第四个坑是把安全性简化成“拒得越多越安全”。这会导致高误拒。一个对正常请求也大量拒答的模型，虽然表面安全，但实际可用性很差。安全评估至少要同时看“该拒时能否拒”和“不该拒时会不会误拒”。

第五个坑是推理配置不固定。你今天用 `temperature=0.2`，明天用 `temperature=0.8`，得到的差异可能来自采样噪声，而不是模型本身。工程上必须固定 prompt 模板、解码参数、上下文拼接方式，必要时固定随机种子。

第六个坑是评估集和业务脱节。比如你做的是客服机器人，却只测通用问答；或者你做的是工具调用助手，却只测自然语言回答。这种错位会让评估看起来很完整，实际上对上线帮助不大。

## 替代方案与适用边界

没有任何一个基准能覆盖所有场景，所以评估组合必须跟着风险级别走。简单说：风险越高，维度越不能省。

| 场景 | 推荐重点 | 不建议省略 |
|---|---|---|
| 低风险内部分类 | 任务性能、通用能力 | 私有验证集 |
| 客服助手 | 任务、指令、安全 | IFEval、红队 |
| 代码助手 | 任务、指令、安全 | 越狱测试、格式测试 |
| 高风险行业 | 全维度 | 私有集、红队、误拒评估 |

如果只是一个低风险的内部文本分类微调，重点看任务性能和通用能力通常就够了，因为它不直接面向外部用户，也没有太多复杂输出约束。但即便如此，私有验证集仍然不该省略，否则你无法判断它是否真的适配内部数据。

如果是客服助手，指令遵循就必须上升到核心位置。因为客服常常要求固定字段、固定流程、固定免责声明。你不能接受一个“答案大体正确，但格式经常错”的模型。

如果是代码助手，除了业务任务性能，还要重点检查格式、工具调用约束和越狱鲁棒性。因为这类系统一旦乱调用命令、输出危险建议，风险会迅速扩大。

如果是医疗、金融、法务等高风险行业，通常更适合“分层门槛法”而不是“只看总分”。做法是：

1. 为安全、指令、核心业务指标设底线。
2. 任何底线不过，直接淘汰。
3. 只有都过线的模型，才再比较综合分。

这比只用单一总分更适合上线决策。单一总分适合快速筛选候选模型，业务指标适合局部优化，而分层门槛法更适合做真正的发布把关。
