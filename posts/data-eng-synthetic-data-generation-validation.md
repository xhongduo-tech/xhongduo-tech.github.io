## 核心结论

合成数据的目标不是“像真”，而是“可用”。这里的“可用”可以先用一句白话理解：拿去训练、测试或演示时，不会把任务带偏，也不会把真实敏感数据泄出去。实际工程里，通常要同时满足四件事：

1. 标签正确。标签就是样本对应的答案或类别，比如“是否违约”“是否患病”。
2. 分布合理。分布就是数据整体长什么样，不是看单条是否自然，而是看年龄、收入、诊断码、点击率等统计规律是否接近真实。
3. 隐私可控。隐私可控的意思是合成样本不能把真实个体“背下来”再吐出来。
4. 对下游有效。下游就是最终任务，例如分类、推荐、风控、检索；如果模型效果没提升，前面指标再好也没有业务意义。

因此，评价不能只看一个数字。常见做法是把“统计距离”“泄露风险”“下游表现”三条线同时拉通。任意一条明显不达标，都不应认为这批合成数据可上线。

一个适合新手的最小验收口径可以写成表：

| 指标 | 常见阈值示例 | 解释 |
|---|---:|---|
| 标签一致率 | $\ge 99\%$ | 生成后经规则或人工核查，标签不能漂移 |
| Acceptance Rate | $\ge 85\%$ | 经过规则过滤和质量检查后，能保留下来的样本比例 |
| KL 距离 | $< 0.1$ | 衡量真实分布与合成分布差异，越小越接近 |
| JS 距离 | $< 0.1$ | 对称的分布差异指标，便于稳定比较 |
| 判别器准确率 | $< 60\%$ | 若真假分类器很容易分辨，说明合成数据过于“假”或有模式缺陷 |
| 泄露率 | 接近 0 | 检查是否复现真实记录或敏感片段 |
| TSTR/TRTS 下游性能 | 达到真实基线的 90% 以上 | 用真实任务验证这批数据是否真有用 |

玩具例子：你有一组真实患者记录，字段只有“年龄段”和“是否高风险”。你用模板生成 1000 条合成样本，发现标签一致率 100%，KL=0.05，JS=0.03，判别器准确率 55%，并且用这些数据训练的模型在真实测试集上达到真实基线的 92%。这时可以认为这批样本已经接近“初步可用”。

## 问题定义与边界

“合成数据的生成与验证”不是单个模型问题，而是一条流水线问题。更准确地说，它是在可控成本内，构造一批满足任务约束的人工样本，并通过验证流程筛掉不合格部分。

这里要先划清边界：

| 环节 | 典型方法 | 目标 |
|---|---|---|
| 生成 | 模板、规则引擎、仿真器、LLM | 产出候选样本 |
| 过滤 | 范围规则、格式规则、业务约束 | 删掉明显错误样本 |
| 判别 | 真假分类器、一致性检查器 | 找出“过假”或“标签不稳”的样本 |
| 分布检测 | KL、JS、相关矩阵、边际统计 | 检查整体统计是否偏离 |
| 隐私检测 | 近邻匹配、记忆泄露测试、成员推断 | 检查是否泄露真实个体 |
| 下游验证 | TSTR、TRTS、真实任务评估 | 检查能否提升或保持业务效果 |

可以把它理解成一条闭环：

`生成 -> 规则过滤/判别 -> 分布检测 -> 隐私检测 -> 下游验证`

每一步都有不同目标。生成负责“有东西”，验证负责“可使用”。

举一个初级工程师能落地的例子：你用大模型生成带标签的客户画像，字段包括年龄、收入、是否流失。生成后先做规则过滤，例如年龄必须在 18 到 80、收入非负、已婚状态与年龄要合理；然后人工抽检 5% 样本，看“高流失风险”标签是否与文本描述一致；再和真实数据比较年龄段、收入分桶、流失比例；最后把合成数据拿去训练一个流失模型，看真实测试集表现是否保持住。只有这几步连起来，才叫“生成+验证”。

边界也要明确。合成数据不一定适合替代真实数据本身，尤其在极少样本、稀有事件、高维时序、强因果任务里，合成数据更多是补充而不是完全替换。例如医疗中的罕见病记录，即使边际分布很像，病程转移关系也可能是错的。这时如果只看单字段分布，很容易误判。

## 核心机制与推导

核心机制可以压缩成一句话：先生成候选样本，再用统计、隐私和任务三套指标把“看起来能用”变成“证明能用”。

先看统计距离。设真实分布为 $P$，合成分布为 $Q$。KL 散度定义为：

$$
D_{\mathrm{KL}}(P\|Q)=\sum_i P(i)\log \frac{P(i)}{Q(i)}
$$

它的白话含义是：如果真实世界按 $P$ 出现，但你用 $Q$ 去近似，会损失多少信息。它不对称，因此通常还会配合 JS 散度：

$$
D_{\mathrm{JS}}(P,Q)=\frac{1}{2}D_{\mathrm{KL}}(P\|M)+\frac{1}{2}D_{\mathrm{KL}}(Q\|M), \quad M=\frac{P+Q}{2}
$$

JS 的优点是对称、数值更稳定，适合工程里横向比较不同生成方案。

玩具例子：真实分布 $P=[0.5,0.5]$，合成分布 $Q=[0.45,0.55]$。直觉上两者很接近，KL 和 JS 都会很小。这说明在“某个字段的边际统计”上，合成数据没有明显跑偏。但这还不够，因为边际对了，不代表联合关系对了，更不代表没有泄露。

所以第二条线是判别与隐私。判别器就是一个“真假分类器”，白话解释是：训练一个模型去猜一条记录是真实还是合成。如果它准确率很高，比如 90%，说明两类数据差异明显，生成质量不够；但如果准确率异常接近 100%，也要警惕另一种情况，即模型不是发现“假”，而是发现少量真实记录被记住并重现。此时要进一步做近邻检测、成员推断或重复率分析。

第三条线是下游效用。最常见的是 TSTR 和 TRTS：

| 缩写 | 含义 | 问的问题 |
|---|---|---|
| TSTR | Train on Synthetic, Test on Real | 用合成数据训练，能否泛化到真实世界 |
| TRTS | Train on Real, Test on Synthetic | 合成数据是否覆盖了真实任务结构 |

工程里更看重 TSTR，因为它直接回答“这批合成数据能不能替真实训练集分担任务”。如果 TSTR 结果只有真实基线的 60%，即使 KL/JS 很漂亮，也说明合成数据只学到了表面统计，没有学到对任务真正重要的结构。

真实工程例子：制药团队要生成患者就诊记录用于建模验证。模板能保证字段齐全，差分隐私仿真器保证不直接暴露真实患者，判别器检查真假可分性，多维分布检测检查年龄、诊断、用药、住院天数的联合统计，最后用 TSTR 看风险预测模型是否还能在真实测试集上保持性能。监管语境下，真正通过验收的不是“看起来像病历”，而是 fidelity、privacy、utility 三轴都过线。

## 代码实现

下面给出一个可直接运行的最小脚本。它不依赖 `scipy`，只用标准 `numpy`，完成四件事：计算 KL、JS、acceptance rate，并用一个简单阈值判断是否通过验收。

```python
import numpy as np

def normalize(arr):
    arr = np.array(arr, dtype=float)
    arr = arr / arr.sum()
    assert np.isclose(arr.sum(), 1.0)
    assert np.all(arr > 0)
    return arr

def kl_divergence(p, q):
    p = normalize(p)
    q = normalize(q)
    return float(np.sum(p * np.log(p / q)))

def js_divergence(p, q):
    p = normalize(p)
    q = normalize(q)
    m = 0.5 * (p + q)
    return 0.5 * kl_divergence(p, m) + 0.5 * kl_divergence(q, m)

def acceptance_rate(total_count, accepted_count):
    assert total_count > 0
    assert 0 <= accepted_count <= total_count
    return accepted_count / total_count

def is_acceptable(kl, js, discriminator_acc, leakage_rate, tstr_acc, baseline_acc, acc_rate):
    return (
        kl < 0.1
        and js < 0.1
        and discriminator_acc < 0.60
        and leakage_rate < 0.01
        and tstr_acc >= 0.90 * baseline_acc
        and acc_rate >= 0.85
    )

# 玩具例子：真实与合成的某个二分类字段分布
p_real = [0.5, 0.5]
q_syn = [0.45, 0.55]

kl = kl_divergence(p_real, q_syn)
js = js_divergence(p_real, q_syn)
acc_rate = acceptance_rate(total_count=1000, accepted_count=910)

discriminator_acc = 0.55
leakage_rate = 0.002
baseline_real_acc = 0.93
tstr_acc = 0.86  # 约为真实基线的 92.5%

print(f"KL={kl:.4f}")
print(f"JS={js:.4f}")
print(f"Acceptance Rate={acc_rate:.2%}")
print(f"Discriminator Acc={discriminator_acc:.2%}")
print(f"Leakage Rate={leakage_rate:.2%}")
print(f"TSTR / Real Baseline={tstr_acc / baseline_real_acc:.2%}")

ok = is_acceptable(
    kl, js, discriminator_acc, leakage_rate, tstr_acc, baseline_real_acc, acc_rate
)
print("PASS" if ok else "FAIL")

assert kl < 0.1
assert js < 0.1
assert acc_rate == 0.91
assert ok
```

这段代码反映的是工程上的最小思想，而不是完整系统。真实流水线通常还会增加：

| 模块 | 最小实现 | 工程增强版 |
|---|---|---|
| 分布检查 | 单字段 KL/JS | 多字段边际、相关矩阵、分桶 PSI、距离热图 |
| 标签核查 | 随机抽检 | 规则引擎 + 弱监督校验器 |
| 隐私检测 | 重复率 | 近邻距离、成员推断、k-anonymity 检查 |
| 下游验证 | 单模型准确率 | 多任务、多随机种子、置信区间 |

如果要把它接进生产流程，通常会把每个生成批次都存成一份报告：字段级距离、样本过滤原因、泄露告警、TSTR 结果、最终是否准入。这样回溯时可以知道到底是生成器坏了，还是过滤器、判别器或下游评估出了问题。

## 工程权衡与常见坑

第一个常见坑是只追求统计接近。很多团队会把 KL、JS 压得很低，觉得“已经很像真实数据了”。问题在于，统计接近不等于安全，也不等于有用。一个生成器完全可能记住训练集，再通过轻微扰动把统计做得很好看。结果是表面 fidelity 很高，实际 privacy 很差。

第二个坑是忽略任务目标。合成数据是为任务服务的，不是为图表服务的。尤其在推荐、风控、医疗预警中，某些少见但关键的模式决定模型是否有效。只看全局分布，可能掩盖长尾区域严重失真。

第三个坑是把 acceptance rate 当成越高越好。过滤通过率高，可能表示生成器很好；也可能表示过滤规则太松，导致低质量样本混进去。这个指标必须结合标签一致率和下游性能一起看。

下面的对比更直观：

| 评价方式 | 可能结论 | 风险 |
|---|---|---|
| 只看统计距离 | “合成数据很像真实数据” | 可能泄露真实样本，或缺乏任务有效性 |
| 统计 + 判别器 | “真假不易区分” | 仍可能没有下游价值 |
| 统计 + 隐私 + 下游 | “既接近、又安全、还能用” | 成本更高，但更接近可上线标准 |

真实工程里常见一种失败案例：团队不断调 KL/JS，把字段分布拟合得越来越紧，结果真假判别器准确率却升到 90%，后续检查发现模型开始复现训练集中的真实记录片段。后来他们加入泄露检测，并在训练时加大正则化和采样扰动，才把指标拉回可接受区间。这个案例说明，单一指标优化很容易把系统带到错误方向。

## 替代方案与适用边界

没有一种生成方案适合所有场景。选择方案时，先问清楚最硬的约束是什么：隐私、成本、解释性，还是下游效果。

| 方案 | 适用场景 | 优点 | 局限 |
|---|---|---|---|
| 模板生成 | 字段规则明确、结构化表单 | 简单、可控、可解释 | 多样性不足，难覆盖复杂联合分布 |
| 仿真器 | 有明确业务机制或物理过程 | 可编码机制，稳定复现 | 建模成本高，依赖领域知识 |
| LLM 生成 | 文本、半结构化、多字段描述 | 覆盖广，开发快 | 容易幻觉，标签可能漂移 |
| 差分隐私仿真器 | 隐私要求严格 | 泄露风险低 | 噪声会损伤可用性 |
| 真实数据增强 | 真实样本太少但允许扩充 | 保留任务相关性 | 受隐私和偏差限制 |

如果隐私是第一优先级，优先考虑“差分隐私仿真器 + 模板约束 + 强验证”。白话解释一下，差分隐私就是在统计层面加入可证明的噪声，使单个个体是否出现在训练集中，不会明显影响输出。

如果下游性能是硬指标，比如制药场景中的患者风险建模，那么流程应反过来设计：先定义 TSTR/TRTS 门槛，再倒推生成方法，而不是先选一个“看起来高级”的模型再补验证。

一个真实工程替代流程可以这样做：在制药数据场景中，先根据真实患者表定义字段模板和医疗约束，再用差分隐私仿真器生成候选记录；生成后做年龄、诊断码、用药组合等规则过滤；再做多维分布检测和近邻泄露检测；最后用真实标签任务做 TSTR。这个流程的优点不是“最像真实患者”，而是更容易向审计、法务和监管解释为什么它可用、可控、可追溯。

因此，适用边界也很明确：如果你的任务高度依赖因果关系、极端长尾事件或复杂时序依赖，只靠简单模板和边际分布验证通常不够，必须引入更强的过程建模或更贴近任务的评估。
