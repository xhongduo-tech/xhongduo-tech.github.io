## 核心结论

TagGPT 类方法的本质，不是让模型直接完成业务任务，而是让模型先充当自动标注器，把原始样本转换成可用于筛选、分桶、路由和重采样的结构化信号。

在指令数据清洗里，它的核心价值有两点：第一，自动补全标签；第二，用质量分数筛掉低价值样本，提升训练集密度。训练集密度指单位样本中有效学习信号的比例，密度越高，同样训练成本下模型越容易学到稳定能力。

可以把这个过程写成：

$$
f_\theta(x,p)\rightarrow(\hat{y},\hat{s})
$$

其中 $x$ 是原始样本，$p$ 是标注提示词，$f_\theta$ 是用于标注的大模型，$\hat{y}$ 是自动生成的标签，$\hat{s}$ 是自动生成的质量分数。

一个新手版本的玩具例子是：给模型一条指令“把这段话总结成 3 个要点”。TagGPT 风格流程不会只关心最终总结内容，而是先判断这条样本属于“摘要任务”“短输出格式”“中等难度”“高质量候选”，再决定保留、降权还是丢弃。这里的重点不是“答案是否已经完美”，而是“这条样本值不值得进入训练池”。

| 能力 | 作用 |
|---|---|
| 自动标签 | 标出任务类型、领域、难度、格式要求 |
| 质量打分 | 判断样本是否值得保留 |
| 数据路由 | 把样本分配到不同子集 |

真实工程例子是 SFT 数据池清洗。SFT 是 supervised fine-tuning，中文通常叫监督微调，意思是用“指令-答案”样本继续训练模型，让模型更符合预期输出方式。假设团队有 50 万条混合指令数据，里面包含代码、数学、客服、摘要、翻译、医疗问答和闲聊。如果人工逐条审核，成本很高；如果完全随机训练，低质样本和重复样本会稀释训练信号。更实际的做法是先让大模型为每条数据生成任务标签、领域标签、难度标签和质量分数，再按分桶策略抽样训练。

## 问题定义与边界

本文讨论的问题是：输入一条原始指令样本 $x=(Q,A)$，其中 $Q$ 是指令或问题，$A$ 是答案；输出自动标签 $\hat{y}$ 和质量分数 $\hat{s}$。

$$
x=(Q,A)
$$

| 概念 | 含义 |
|---|---|
| $Q$ | 指令 / 问题 |
| $A$ | 答案 |
| $\hat{y}$ | 自动生成标签 |
| $\hat{s}$ | 自动生成分数 |

“标签”是对样本属性的结构化描述。例如任务类型是摘要、问答、改写、代码生成；领域是金融、医疗、法律、教育；格式要求是短答案、列表、JSON、Markdown 表格。“质量分数”是对样本是否值得训练的数值判断，通常用于排序、过滤或加权。

需要划清边界：这里讨论的是“指令数据筛选与自动标注”，不是传统分类任务，也不是纯生成任务。传统分类任务通常有固定标签，例如把评论分成正面和负面；纯生成任务关注模型直接输出答案。TagGPT 风格方法位于中间：它让模型先阅读样本，再输出结构化元信息，之后这些元信息被数据管线使用。

TagGPT 原始论文更偏多模态标签生成，也就是给图像、视频等内容生成标签。本文采用的是可迁移机制：用大模型从非结构化样本中生成结构化标签，再通过过滤和归一化形成可用的数据资产。这套机制可以迁移到指令数据、代码数据、客服对话、数学题和安全评测数据中，但不能直接假设所有领域都适合同一套标签体系。

一个新手例子：样本是“请把以下日志解释成一句话”，答案却写成了长篇根因分析。标签系统应该识别它是“解释类任务”，不是普通“问答类任务”；同时还应标记“格式不匹配”，因为指令要求一句话，答案却没有遵守长度约束。

真实工程边界更严格。医疗数据的标签体系可能要包含风险等级、专业术语密度、诊断建议、是否需要免责声明；代码数据可能要包含语言、库、复杂度、是否可运行、是否有测试。把客服数据的标签体系直接套到医疗数据上，会得到看似整齐但实际无效的标注结果。

## 核心机制与推导

TagGPT 风格自动标注通常包含四步：候选标签生成、频次过滤、语义匹配、标签归一化。

| 步骤 | 作用 |
|---|---|
| 候选标签生成 | 让模型先给出多个可能标签 |
| 频次过滤 | 去掉偶发噪声标签 |
| 语义匹配 | 合并同义标签 |
| 归一化 | 把标签体系固定下来 |

候选标签生成是第一步。模型接收样本和提示词，输出多个可能标签。例如样本是“把这段话总结成 3 个要点”，模型可能输出“摘要”“压缩”“要点式输出”“结构化回答”“短文本生成”。这些标签不是最终结果，而是候选集合。

频次过滤是第二步。频次过滤指统计标签出现次数，删除只偶然出现、无法形成稳定类别的标签。例如“优雅表达”这种标签可能只出现几次，语义模糊，工程价值低；“摘要任务”出现很多次，且含义稳定，更适合保留。

语义匹配是第三步。语义匹配指把含义接近的标签合并，例如“摘要”“文本摘要”“压缩摘要”可以归到“摘要任务”。常见做法是用 embedding。Embedding 是把文本转换成向量的技术，语义相近的文本在向量空间里距离更近。这样可以避免标签体系膨胀到不可维护。

归一化是第四步。归一化指把模型自由生成的标签映射到固定标签表。例如最终只允许任务类型取值为“摘要、问答、翻译、代码、数学、改写、抽取、分类”。固定标签表可以让后续统计、训练采样和质量监控更稳定。

自动标注函数可以统一写成：

$$
f_\theta(x,p)\rightarrow(\hat{y},\hat{s})
$$

其中 $p$ 控制标注规范，例如要求输出 JSON、限定标签字段、解释打分理由。工程上通常会要求模型输出如下结构：

```json
{
  "task_type": "summarization",
  "domain": "general",
  "difficulty": "medium",
  "format_constraint": "3_bullets",
  "quality_score": 0.82,
  "keep": true
}
```

质量评估还有一类常见思路：比较“有指令条件下生成答案的损失”和“无指令条件下生成答案的损失”。损失是模型预测答案时的错误程度，数值越低，说明模型越容易生成目标答案。

IFD 分数可以写成：

$$
s_{\text{IFD}}(x)=\frac{L_\theta(A|Q)}{L_\theta(A)}
$$

其中 $L_\theta(A|Q)$ 表示模型在看到指令 $Q$ 后生成答案 $A$ 的损失，$L_\theta(A)$ 表示模型不看指令、只生成答案 $A$ 的损失。

如果：

$$
s_{\text{IFD}}\le 1 \Rightarrow Q \text{ 对生成 } A \text{ 有帮助}
$$

就说明指令提供了有效约束。通俗说，模型看到问题后更容易写出这个答案，说明问题和答案之间关联较强，这条样本通常更值得保留。

玩具数值例子：样本 $Q$ 是“把这段话总结成 3 个要点”，$A$ 是一个三条项目符号的摘要。如果模型算得 $L_\theta(A|Q)=1.2$，$L_\theta(A)=2.0$，则 $s_{\text{IFD}}=0.6$，可以认为指令对答案有明显帮助。若 $L_\theta(A|Q)=2.4$，$L_\theta(A)=2.0$，则 $s_{\text{IFD}}=1.2$，说明看到指令反而没有帮助，这条样本可能存在答非所问、格式错配或答案质量差的问题。

## 代码实现

下面的代码实现一个最小闭环：自动标注、质量打分、数据筛选。它不调用真实大模型，而是用规则模拟标注器，方便直接运行。真实系统里，只需要把 `mock_tag_model` 替换成大模型 API 调用，把 `loss_cond` 和 `loss_uncond` 替换成模型实际计算出的损失即可。

| 模块 | 输入 | 输出 |
|---|---|---|
| 标签生成器 | 原始样本 | 标签集合 |
| 打分器 | 样本、标签 | 质量分数 |
| 过滤器 | 标签 + 分数 | 保留 / 丢弃 |
| 复核器 | 抽样结果 | 人工修正 |

```python
def ifd_score(loss_cond, loss_uncond):
    if loss_uncond <= 0:
        raise ValueError("loss_uncond must be positive")
    return loss_cond / loss_uncond


def mock_tag_model(sample):
    q = sample["Q"]
    a = sample["A"]

    tags = []

    if "总结" in q or "摘要" in q:
        tags.append("摘要任务")
    if "3个要点" in q or "三点" in q:
        tags.append("短输出")
    if len(a) > 0:
        tags.append("有答案")
    if "\n" in a or "1." in a or "-" in a:
        tags.append("结构化回答")

    if not tags:
        tags.append("其他任务")

    return tags


def filter_sample(sample, loss_cond, loss_uncond, threshold=1.0):
    tags = mock_tag_model(sample)
    score = ifd_score(loss_cond, loss_uncond)

    result = {
        "Q": sample["Q"],
        "A": sample["A"],
        "tags": tags,
        "ifd_score": score,
        "keep": score <= threshold and "有答案" in tags,
    }
    return result


sample = {
    "Q": "把这段话总结成3个要点",
    "A": "1. 系统延迟升高。\n2. 数据库连接数过高。\n3. 需要扩容并检查慢查询。"
}

result = filter_sample(sample, loss_cond=1.2, loss_uncond=2.0)

assert result["ifd_score"] == 0.6
assert result["keep"] is True
assert "摘要任务" in result["tags"]
assert "短输出" in result["tags"]

bad_result = filter_sample(sample, loss_cond=2.4, loss_uncond=2.0)

assert bad_result["ifd_score"] == 1.2
assert bad_result["keep"] is False
```

这段代码的作用不是训练模型，而是把训练前的数据清洗流程自动化。一条指令数据进入脚本后，程序先得到标签“摘要任务、短输出、有答案、结构化回答”，再算出 IFD 分数 0.6，最后写入“保留”队列。

真实工程中，这个脚本通常还会增加四件事。第一，落盘标注结果，例如写入 JSONL。第二，记录 prompt 版本和模型版本，避免后续无法复现。第三，抽样人工复核，例如每个标签桶抽 1% 到 5%。第四，对不同数据桶设置不同保留阈值，例如数学题更重视答案正确性，客服对话更重视安全和语气。

## 工程权衡与常见坑

自动标注不等于自动正确。它能降低人工成本，但会引入模型偏差、提示词偏差和统计偏差。工程上不能把大模型输出直接当真值，而应该把它当成“高效但需要校验的弱标注”。

| 坑 | 现象 | 规避方式 |
|---|---|---|
| 标签泄漏 | 模型只是在抄类别名 | 隐藏类名，只给定义 |
| 单模型偏置 | 自己标自己打分 | 加规则校验和人工复核 |
| 标签塌缩 | 高频类占满数据 | 频次下限、分桶配额 |
| 分数不稳 | prompt 一改分数就变 | 多次采样取均值 / 多数票 |
| 覆盖不足 | 高分样本太相似 | 先多样性采样，再质量筛选 |

标签泄漏是最常见的问题。标签泄漏指提示词里提前暴露答案或类别，导致模型不是判断样本，而是在复述提示。例如提示词写“请判断这是否是数学题，样本来自数学数据集”，模型很可能输出“数学题”，但这个判断没有信息价值。更好的做法是只给类别定义，不给来源暗示。

单模型偏置也很危险。如果同一个模型既生成标签，又给自己的标签打高分，结果容易过于乐观。真实系统里通常会组合三类信号：大模型标注、规则校验、人工抽检。规则校验适合检查格式、长度、JSON 合法性、是否包含敏感词；人工抽检适合发现模型无法稳定判断的语义问题。

标签塌缩指少数高频标签占据大多数样本。例如“问答任务”过宽，最后 80% 数据都被归入问答，分桶失去意义。解决方式是限制标签粒度，设置分桶配额，并对大桶继续细分。

分数不稳来自模型版本、temperature、prompt 和上下文顺序。Temperature 是生成随机性的控制参数，值越高，输出越随机。若同一批数据今天打分和明天打分差异很大，就不能直接用单次结果做训练集筛选。更稳妥的做法是固定模型版本和 prompt，必要时多次采样取均值或多数票。

真实工程里还有一个容易忽略的问题：只看质量分数会损失覆盖。高分样本往往风格相似，训练后模型可能在常见任务上表现更好，但在长尾任务上退化。因此大规模数据筛选通常采用“两阶段策略”：先按 embedding 或标签做多样性采样，再在每个桶内做质量筛选。

## 替代方案与适用边界

如果目标只是提升训练集质量，不一定要上完整的 TagGPT 风格流程。更轻的方案包括规则过滤、启发式打分、embedding 聚类和人工小样本审核。

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 纯人工审核 | 准确 | 成本高 | 小数据集 |
| 规则过滤 | 简单稳定 | 覆盖有限 | 格式明确任务 |
| Embedding 聚类 | 多样性好 | 语义粒度粗 | 粗分桶 |
| TagGPT 风格自动标注 | 可扩展 | 依赖模型稳定性 | 大规模 SFT 数据池 |

纯人工审核适合小而贵的数据集。如果只有几千条高价值医疗问答或法律样本，直接人工审一遍可能比搭建复杂自动标注管线更划算。这里的核心约束不是技术复杂度，而是错误代价。

规则过滤适合格式明确的任务。例如代码样本必须能通过语法解析，JSON 输出必须能被解析，数学题答案必须包含最终数值。规则的优点是稳定、可解释、便宜；缺点是只能覆盖表层问题，难以判断答案是否真正有帮助。

Embedding 聚类适合做多样性控制。例如 30 万条客服对话可以先聚类成退款、物流、账号、安全、投诉等粗类别，再从每个类里抽样。它不擅长精细质量判断，但很适合作为 TagGPT 打分前的一道分桶步骤。

TagGPT 风格自动标注适合数据量大、任务类型多、人工成本高的场景。反过来，当任务分布简单、样本量小、容错低时，过度自动化会引入系统性偏差。一个真实工程判断标准是：如果人工审核成本已经高到无法覆盖主要数据桶，并且数据标签会影响训练采样，那么自动标注值得引入；如果只是几千条固定格式数据，先用规则和人工审核更直接。

更稳妥的实践路线是从轻到重：先定义标签体系和抽检标准，再用规则过滤明显坏样本，然后用大模型补标签和打分，最后用人工抽检修正阈值。不要一开始就追求全自动，因为自动标注系统本身也需要被评估。
