## 核心结论

BBH，中文常写作 BIG-Bench Hard，是从 BIG-Bench 里 200 多个任务中进一步筛出来的一组高难度推理任务。它的目标不是“广泛测什么都懂”，而是专门测“大模型在 few-shot 提示下仍然做不稳的推理题”。few-shot 的意思是“在正式题目前先给几个示例，让模型照着学格式和思路再答题”。

这组任务的核心价值有三点。第一，它把难点集中在符号推理、时间与因果判断、指代消解、括号语言、布尔表达式等需要多步中间推导的问题上，而不是知识记忆。第二，它要求输出尽量可自动判分，也就是答案空间有限、格式受控，这样才能用 exact match 稳定比较不同模型。exact match 就是“模型答案与标准答案完全一致才算对”。第三，它揭示了一个很重要的工程事实：模型“看起来会说”不等于“稳定会推理”。早期模型在 BBH 的多数任务上，即使给 few-shot 示例，表现仍明显低于人类，甚至在部分任务上低于随机猜测。

对初级工程师来说，BBH 最值得记住的不是“它很难”，而是“它把推理能力拆成可重复、可自动判分、可横向比较的任务”。这让它非常适合做模型上线前的推理验收，而不只是论文里的展示分数。

下面这张表可以先建立一个总览：

| 维度 | 说明 |
| --- | --- |
| 难度来源 | 多步推理、时间/因果/指代、算法式结构 |
| 判分策略 | 限定输出、exact-match、自动化 |
| 关注能力 | 逻辑推理、符号处理、常识推断 |
| 典型价值 | 检查模型“会不会稳定推对” |

一个最小“玩具例子”就是布尔表达式题。布尔表达式就是“只处理真和假”的规则表达式。例如题目是：

`Evaluate: (TRUE AND FALSE) OR TRUE`

人手算时先看括号：`TRUE AND FALSE = FALSE`，再算 `FALSE OR TRUE = TRUE`。因此标准答案只能是 `TRUE`。如果模型输出 `The answer is TRUE`，在很多 BBH 风格评测里都可能被判错，因为判分器要求最后结果和标准答案精确一致。这就是 BBH 的工程味道：它不奖励“像在思考”，只奖励“最终答对且格式正确”。

## 问题定义与边界

BBH 不是一个开放聊天测试集，也不是一个“让人主观打分”的基准。它更准确的定义是：从 BIG-Bench 的大量任务中，筛出一批在 few-shot 提示下依然明显困难、且适合自动判分的子任务集合。自动判分的意思是程序不需要人类再解释语义，只要比较答案文本是否符合标准，就能给出对错。

它的边界有三层。

第一层边界是任务筛选。不是所有难题都能进入 BBH。一个任务通常至少要有足够样本量，且答案能稳定判分。如果一个任务过于开放，比如“请谈谈你的看法”，这种题即使很难，也不适合 BBH，因为不同回答可能都“有道理”，不利于 exact match。

第二层边界是能力类型。BBH 更偏“推理结构”，不偏“大规模知识覆盖”。比如日期理解要求根据时间线推出某天是星期几，因果判断要求从事件前后关系判断原因和结果，括号语言要求补全成对结构，布尔表达式要求严格按逻辑规则化简。这些任务的共同点是：知识背景不重，但中间步骤不能跳。

第三层边界是评测目标。BBH 不直接告诉你模型能不能写 API、会不会调数据库、能不能做复杂软件开发。它主要回答的是另一个问题：当输入被约束成结构化推理题时，模型能否稳定地走完推导并落到正确答案上。

可以用下面这张表概括 BBH 的定义边界：

| 筛选条件 | 作用 |
| --- | --- |
| 任务数约 23~27 | 保持难点集中，便于跨模型追踪 |
| 样本量足够 | 降低偶然波动，结果更稳定 |
| 自动判分 | 避免人工解释歧义 |
| 受限输出 | 2~5 个候选或明确文本，便于 deterministic |
| few-shot 下仍难 | 真正筛出推理薄弱点 |

这里的 deterministic，中文可以理解为“同样规则下，判分过程不依赖人工临场判断”。这很关键。因为一个工程评测如果不能稳定复现，结果就不能拿来做上线门槛。

再看一个直观例子。假设有一道因果判断题：

- 事件 A：地面下雨后变湿
- 事件 B：清洁工在雨停后擦了地
- 问：地面湿更可能由哪个原因造成？

如果候选只有 `A` 和 `B`，题面描述又足够清晰，那么这类题就适合 BBH。它既需要常识，也需要按时间顺序和因果强度判断；但最终输出仍然是有限候选，容易自动判分。

因此，BBH 的问题定义可以总结成一句话：它不是要测试模型“能否聊得像人”，而是要测试模型“能否在受限输出下做对多步推理”。

## 核心机制与推导

BBH 的机制非常朴素，但正因为朴素，才有比较强的比较价值。每个子任务独立评测，先算各自的 exact-match 准确率，再对任务做简单平均，得到总分。它不追求复杂加权，也不试图发明一套很花哨的综合指标。

最常见的总分形式可以写成：

$$
\text{BBH\_score} = \frac{1}{N}\sum_{i=1}^{N}\mathbb{I}(\hat{y}_i = y_i)
$$

其中，$N$ 是任务数量，$\hat{y}_i$ 是模型输出，$y_i$ 是标准答案，$\mathbb{I}$ 是指示函数。指示函数的白话解释是：条件成立记 1，不成立记 0。也就是说，一道题不是“半对”，而是“完全对或完全错”。

很多实现里还会对答案做轻量标准化，例如去掉首尾空格、转成小写、只取最后一行。只取最后一行的原因很现实：模型经常先写一堆解释，最后才写答案，而判分器只希望抓取最终结果，避免中间文本干扰。

这套机制有三个推导上的后果。

第一，BBH 只奖励结果正确，不奖励解释看起来很合理。一个模型可能生成很长的分析过程，但只要最后落点错了，就是 0 分。反过来，一个模型即使没有展示中间过程，只要答案严格匹配，也能得分。这使 BBH 和“主观觉得模型推理得不错”形成了明显区分。

第二，任务之间通常做简单平均，而不是按样本数加权。这意味着 BBH 更关注“模型在不同推理模式上的覆盖面”，而不是“在哪个大样本任务上刷高分”。如果某模型只擅长日期题，不擅长因果题和括号语言题，简单平均会把这种短板暴露出来。

第三，输出空间越受限，越能把“推理能力”从“表达能力”里分离出来。比如同样是自然语言模型，开放生成任务会把措辞、风格、长度偏好混进去；而 BBH 刻意收紧答案格式，让比较聚焦在“推对没有”。

下面用一个玩具例子说明判分逻辑。

题目：

`Evaluate: NOT (TRUE AND FALSE)`

手工推导：

1. `TRUE AND FALSE = FALSE`
2. `NOT FALSE = TRUE`

因此答案应是 `TRUE`。

如果模型输出下面四种结果，判分往往不同：

| 模型输出 | 是否得分 | 原因 |
| --- | --- | --- |
| `TRUE` | 是 | 与标准答案完全一致 |
| `true` | 通常是 | 若判分做了大小写归一化 |
| `The answer is TRUE` | 常常否 | 不满足 exact match |
| `TRUE.` | 常常否 | 多了标点，未必被标准化处理 |

这就是 BBH 的“格式纪律”。对新手来说，这种纪律看起来苛刻，但它的好处是把评测从“主观判断”变成“程序规则”。

再看一个更接近真实任务的例子：`date_understanding`。日期理解可以理解成“根据几个时间条件，算出准确日期或星期几”。例如：

- 今天是 2026-04-04，星期六。
- 问：10 天后是星期几？

手算时就是做模运算。一个星期 7 天，所以：

$$
10 \bmod 7 = 3
$$

向后推 3 天：星期六 → 星期日 → 星期一 → 星期二，因此答案是星期二。这类题看似简单，但当题面同时加入“前天”“下下周”“两个月前的第三个周一”之类条件时，模型就容易在中间步骤丢失状态。

真实工程里，这正是 BBH 有价值的地方。很多模型在开放问答里能写出很流畅的解释，但一旦题目要求严格的最终答案，错误率会明显暴露。研究者后来广泛使用 chain-of-thought，也就是“把中间推理过程显式写出来”的提示方式，原因就在这里：如果模型不把中间状态展开，它往往跳步，最后出错。

但要注意，BBH 的标准分数通常仍按最终答案来算，而不是按中间过程是否“看起来像真的”。这也是为什么它适合作为上线前的硬指标。

## 代码实现

从工程实现看，BBH 评测其实不复杂，难点不在代码量，而在细节纪律。最小实现一般有四步：

1. 读取任务与样本。
2. 按统一模板构造 prompt。
3. 调模型拿到原始输出。
4. 取最后一行、做标准化、与标准答案 exact match。

下面给一个可运行的 Python 玩具实现。它没有依赖外部库，只模拟 BBH 的基本判分逻辑。

```python
from dataclasses import dataclass

@dataclass
class Sample:
    prompt: str
    target: str

def normalize_answer(text: str) -> str:
    lines = [line.strip() for line in text.strip().splitlines() if line.strip()]
    if not lines:
        return ""
    return lines[-1].lower()

def evaluate_exact_match(prediction: str, target: str) -> int:
    return int(normalize_answer(prediction) == normalize_answer(target))

def evaluate_bbh(samples, model_fn) -> float:
    assert len(samples) > 0
    scores = []
    for sample in samples:
        prediction = model_fn(sample.prompt)
        score = evaluate_exact_match(prediction, sample.target)
        scores.append(score)
    return sum(scores) / len(scores)

def toy_model(prompt: str) -> str:
    mapping = {
        "Evaluate: (TRUE AND FALSE) OR TRUE": "Reasoning...\nTRUE",
        "Evaluate: NOT (TRUE AND FALSE)": "TRUE",
        "Evaluate: TRUE AND FALSE": "The answer is FALSE",  # 这里会被判错
    }
    return mapping[prompt]

samples = [
    Sample("Evaluate: (TRUE AND FALSE) OR TRUE", "TRUE"),
    Sample("Evaluate: NOT (TRUE AND FALSE)", "TRUE"),
    Sample("Evaluate: TRUE AND FALSE", "FALSE"),
]

assert normalize_answer("step by step\nTRUE\n") == "true"
assert evaluate_exact_match("TRUE", "TRUE") == 1
assert evaluate_exact_match("true", "TRUE") == 1
assert evaluate_exact_match("The answer is TRUE", "TRUE") == 0

score = evaluate_bbh(samples, toy_model)
assert abs(score - (2 / 3)) < 1e-9

print("BBH-style score:", score)
```

这个例子故意保留了一个常见坑：`The answer is FALSE` 看起来语义正确，但因为标准化函数只取最后一行全文，再做 exact match，所以它和 `FALSE` 不一致，会被判错。真实 BBH 系统通常会在 prompt 里明确写“最后一行只输出答案”，就是为了降低这种格式损失。

如果要把它扩展成真实工程版，一般会多加几层结构：

| 模块 | 作用 |
| --- | --- |
| 数据加载器 | 读取任务描述、few-shot 示例、标准答案 |
| Prompt 组装器 | 保证所有样本格式一致 |
| 推理执行器 | 调用模型 API，控制温度、最大长度等 |
| 判分器 | 取最后一行、标准化、exact match |
| 汇总器 | 统计单任务准确率与总体平均 |

一个“真实工程例子”可以这样理解。假设团队要比较两个候选模型是否适合上线到“合同条款检查”系统。这个系统表面上是文本理解问题，但底层经常需要因果、指代和时间逻辑。如果模型在 BBH 的 `causal_judgement`、`date_understanding`、`boolean_expressions` 上连续失分，那么它在生产中也很可能出现“读懂了大概意思，但推断最后结论时翻车”的问题。此时，团队可以先不急着做昂贵的人审对比，而是先跑一轮 BBH 风格评测，把推理短板定量暴露出来。

实际实现时，推荐把“推理过程”和“最终答案”在 prompt 中分开约束，例如：

- 中间可以推理。
- 最后一行必须只写候选答案。
- 不要添加额外标点。
- 不要重复题目。

这不是形式主义，而是为了让评测结果真正可复现。

## 工程权衡与常见坑

BBH 的工程价值很高，但它也有明显边界和不少坑。理解这些坑，比单纯抄一个分数更重要。

第一类坑是输出格式坑。模型经常会多说一句话、带标点、带编号，或者把答案放在倒数第二行。这些都可能让 exact match 失败。很多团队第一次跑评测时，发现模型“肉眼看大多答对”，但脚本算出来准确率很低，原因往往不是知识错，而是格式没控住。

第二类坑是 prompt 漂移。prompt 漂移可以理解为“同一个任务因为提示模板略改，结果明显变化”。BBH 的不少子任务对示例顺序、答案格式、是否允许思维链暴露都很敏感。你不能把一个模型在模板 A 上的 75%，直接拿去和模板 B 上的 75% 当成同一件事。

第三类坑是把 BBH 当成“通用能力总分”。它不是。一个模型在 BBH 上高，不代表它写代码一定强；一个模型在 BBH 上中等，也不代表它知识问答就差。BBH 更像“推理单项体测”，而不是“全科高考”。

第四类坑是把 chain-of-thought 当成免费午餐。chain-of-thought 的白话解释是“要求模型把中间思路显式写出来”。它经常能提升 BBH 分数，但也会增加 token 开销、暴露更多格式噪声、放大长输出截断风险。生产环境不一定愿意为所有请求付这个成本。

下面这张表总结常见坑与规避方式：

| 坑 | 规避措施 |
| --- | --- |
| 模型输出解释+答案 | 提示中写明“最后一行只写答案” |
| 输出格式偏差 | 统一做 `strip`、大小写归一化、最后一行提取 |
| prompt 改一点分数大变 | 固定模板版本，记录实验配置 |
| 只看总分不看子任务 | 同时保存分任务结果，定位具体短板 |
| CoT 太长导致截断 | 控制最大输出长度，必要时改成短 scratchpad |
| 评测集太少就下结论 | 先做 smoke test，再跑全量任务 |

这里特别说一个新手最容易忽略的问题：随机基线。对于只有 `TRUE/FALSE` 两个选项的题，随机猜测准确率就是 50%。如果一个模型在布尔任务上只有 46%，这不是“还行”，而是“比乱猜还差”。因此，阅读 BBH 分数时，不能只看绝对值，还要看任务的候选空间。

再给一个真实工程场景。某团队在上线前比较两套推理提示模板：

- 模板 A：直接问“答案是什么？”
- 模板 B：允许简短推理，但最后一行必须只写答案

他们在 `causal_judgement`、`date_understanding`、`dyck_languages` 三类子任务上做 smoke test。结果发现，模板 A 的错误主要集中在中间状态丢失；模板 B 的准确率明显更高，但偶尔因为输出过长被截断。最后他们选择模板 B，并额外限制最大推理长度。这就是典型的工程权衡：不是“越能想越好”，而是“在成本、稳定性、准确率之间找平衡”。

## 替代方案与适用边界

如果你的目标是“广覆盖看一个模型什么都能不能做一点”，那 BBH 不是第一选择。更合适的是 BIG-Bench 主集，因为它覆盖任务更多，能力维度更广，适合做早期能力浏览和基线对比。

如果你的目标是“检验模型在高阶推理上有没有明显短板”，BBH 就更有针对性。它不像主集那样追求宽，而是追求把推理最难、最容易露底的部分集中起来测。

如果你关注的是编程、测试生成、API 使用、代码修复，那么应考虑代码专项基准，而不是指望 BBH 代替它们。因为 BBH 的题型主要仍是布尔、因果、时间、符号结构、常识和语言推理，不直接覆盖真实软件工程工作流。

可以把几类选择简单对比如下：

| 维度 | BBH | BIG-Bench 主集 |
| --- | --- | --- |
| 任务数量 | 23~27 | 200+ |
| 目标 | 精准暴露推理弱点 | 宽覆盖看泛能力 |
| 判分方式 | deterministic exact match 为主 | 多种方式并存 |
| 适用阶段 | 推理能力验收、上线前检查 | 初步摸底、做大盘比较 |
| 典型问题 | 布尔、因果、日期、括号语言 | 范围更广、异质性更高 |

因此，BBH 的最佳使用边界通常有三种。

第一，模型选型阶段。你已经有几个候选模型，想知道谁在推理稳定性上更强，可以先跑 BBH 做一轮筛选。

第二，提示工程阶段。你不是在换模型，而是在比较 prompt、scratchpad、few-shot 示例模板。此时 BBH 很适合做 A/B 测试，因为它的判分稳定，噪声小。

第三，上线前风险检查。尤其是产品包含规则判断、时间逻辑、条件分支、表单审核等环节时，BBH 风格任务能帮助你提前发现“模型会说但不会稳推”的风险。

但如果你的问题是开放创作、长文总结、情绪陪伴、代码库导航，BBH 的参考价值就会迅速下降。因为这些任务的核心难点并不在“受限输出下的精确推理”。

一句话概括替代关系：BBH 更像“推理压力测试”，BIG-Bench 主集更像“泛能力体检表”，代码基准更像“工程实操考核”。它们不是互斥，而是各自负责不同维度。
