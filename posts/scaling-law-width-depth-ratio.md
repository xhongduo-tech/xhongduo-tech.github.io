## 核心结论

在总参数量 $N$ 固定时，Transformer 不能只靠“更宽”或“更深”单边扩张。Levine 等人在 NeurIPS 2020 对自注意力网络的表达效率做了 separation rank 分析。把这类深度效率结论与 dense Transformer 的参数预算约束合起来看，得到一条很有用的设计基线：

$$
d \propto n^{2/3}
$$

其中 $d$ 是隐藏宽度，$n$ 是层数。

这条关系不是“论文里单独写下的万能定律”，而是把两件事放在一起后的工程近似最优线。第一，宽度决定单层容量，也就是一层里能容纳多少并行特征；第二，深度决定复合次数，也就是前面抽出的特征还能被继续组合多少轮。固定预算下，只加宽会让参数按 $d^2$ 迅速膨胀；只加深则会让每层被迫变窄，而且深到一定程度后新增层的表达收益开始变慢，训练稳定性也会下降。最优点通常不在极端，而在宽度与深度同步扩张的平衡带上。

对新手最实用的记忆方式是下面这句：在 dense Transformer 里，每层最贵的主项来自注意力投影和 FFN，它们都近似按 $d^2$ 增长；层数增加的成本更接近线性，所以宽度很贵，深度也不是免费。把“宽度的平方成本”和“深度的层级收益”一起平衡，就会得到接近 $d \propto n^{2/3}$ 的轨迹。工程上，GPT-3、LLaMA 一类模型通常也不会走“极宽薄层”或“极深窄塔”两种极端，而是落在相近的宽深窗口内，例如宽深比 $d/n$ 常见在约 $100:1$ 到 $170:1$。

| 调整方向 | 主要收益 | 主要问题 | 是否接近理论最优 |
| --- | --- | --- | --- |
| 只加宽 | 单层容量变强，注意力和 FFN 的表示空间变大 | 参数按 $d^2$ 膨胀，显存和过拟合风险上升 | 否 |
| 只加深 | 层级复合次数变多，能做更多步特征变换 | 每层变窄，收益递减更早出现，训练更难 | 否 |
| 沿 $d \propto n^{2/3}$ 调整 | 同时保留单层容量和多层复合能力 | 仍需结合优化、数据、硬件做校准 | 是 |

## 问题定义与边界

本文讨论的是标准 dense Transformer 主干，不包含 MoE、参数共享、长上下文专用模块、稀疏激活或特殊低秩改造。问题只有一个：在总参数预算固定时，宽度 $d$ 和深度 $n$ 应该怎样分配，才能让表达力、可训练性和工程成本同时可接受。

工程上常用的参数主项近似是：

$$
N \approx 12nd^2
$$

这个式子不是严格逐项计数，而是一个够用的量级估算。原因是标准 Transformer block 中最重的参数主要来自：

$$
W_Q, W_K, W_V, W_O \in \mathbb{R}^{d \times d}
$$

以及 FFN 的两层线性变换：

$$
W_1 \in \mathbb{R}^{d \times 4d}, \qquad W_2 \in \mathbb{R}^{4d \times d}
$$

把这些主项合起来，每层参数量近似就是常数倍的 $d^2$，再乘层数 $n$，于是得到 $N \sim nd^2$。常见的“12”来自对注意力和 FFN 主项的粗略合并；嵌入层、LayerNorm、bias、输出头等项通常低一个量级，做大模型估算时可以先略去。

把这个公式反过来写，就得到预算锁死后的直接约束：

$$
d \approx \sqrt{\frac{N}{12n}}
$$

它的含义很直接：层数一旦增加，单层宽度就必须下降，而且下降速度是平方根级别。这已经说明“深度不是免费”的，因为你为了多堆几层，必须拿单层容量去换。

Levine 一类理论结果提供的是另一个约束方向。它不是简单说“层数越多越好”，而是指出自注意力网络的深度效率存在区间性。工程讨论里常把这类结论简写为一个提示性阈值：

$$
n \lesssim \log_3 d
$$

这里必须加一句限定：这不是说一旦超过这个数量级，模型就立刻失效；更准确的理解是，当深度继续上升而宽度不足时，新增层的表达收益会进入低收益区，训练与优化代价会变得更突出。它是“边际收益开始变差”的信号，不是硬性的结构禁令。

用一个新手更容易抓住的例子来看，假设你有固定数量的“参数积木”：

- 如果把积木大都堆成很少几层，你得到的是极宽但很浅的网络。每层很强，但复合步骤少。
- 如果把积木切成很多薄层，你得到的是极深但很窄的网络。步骤很多，但每一步能力都弱。
- 真正要优化的是“每一步够强，而且总步数也够多”。

| 配置方向 | 表达力特征 | 优化稳定性 | 参数消耗速度 |
| --- | --- | --- | --- |
| 深度低、宽度高 | 单层强，层级弱 | 前期稳定，但参数效率容易变差 | 快，近似按 $d^2$ 增长 |
| 深度高、宽度低 | 层级强，单层弱 | 更依赖归一化和优化技巧 | 慢，近似按 $n$ 线性增长 |
| 宽深平衡 | 单层和层级都保留 | 最容易做出稳定折中 | 综合效率最好 |

## 核心机制与推导

核心推导分三步，不需要高深数学也能看清。

第一步，先看预算约束。由

$$
N \approx 12nd^2
$$

可得

$$
d \approx \sqrt{\frac{N}{12n}}
$$

这意味着固定参数预算下，深度每增加一倍，宽度不能等比例保留，而必须缩小到原来的 $1/\sqrt{2}$ 左右。也就是说，深度增长一定会挤压单层表示容量。

第二步，单看深度并不够。Levine 的工作强调的是，自注意力网络对多位置交互的表达效率会随着深度增加而提高，但这种提高并不是无限线性的。工程上常把这个现象用“深度阈值”表示为：

$$
L_{\text{th}} \approx \log_3 d
$$

它的实际含义是：宽度越大，网络能承受的“高效复合深度”越高；宽度不够时，层数继续增加，理论上的层级优势会越来越难转化为稳定的有效能力。

第三步，把两条约束放到一起。固定预算要求

$$
d \sim n^{-1/2}
$$

而深度效率又要求“不能把 $d$ 压得过窄，否则层数增加没有足够的单层容量支撑”。在这种双重约束下，工程设计就会落在一条中间轨迹附近。常用的近似写法是：

$$
d \propto n^{2/3}
$$

这条式子表达的不是某个精确常数，而是一种缩放方向：当你增加深度时，宽度也要一起增加，但增加得比线性更慢；反过来，当你为了预算必须减少宽度时，也不能把层数无上限地堆高。宽度的平方成本和深度的线性收益在这里被同时平衡。

把它写成更容易读的链条就是：

$$
N \approx 12nd^2
$$

$$
\text{宽度太大，参数成本按平方爆炸；深度太大，每层又会被压窄}
$$

$$
\text{深度继续增加时，必须保留足够的单层宽度支撑有效复合}
$$

$$
\Rightarrow \text{宽深应同步移动，而不是单边拉满}
$$

$$
\Rightarrow d \propto n^{2/3} \text{ 可作为固定预算下的设计基线}
$$

下面给一个玩具数值例子。假设总预算大致在 $50$B 左右，只比较趋势，不追求精确到最后一层 embedding：

| 方案 | 层数 $n$ | 宽度 $d$ | 直观判断 |
| --- | --- | --- | --- |
| A | 48 | 10000 | 偏宽，单层很强，但层级复合不够 |
| B | 80 | 8000 | 更接近平衡带，宽度和深度都没走极端 |
| C | 160 | 5600 | 偏深，每层偏窄，优化更敏感 |

这三组里，B 往往更像一个“容易训、也更符合缩放规律”的候选点。A 的问题是参数主要烧在单层容量上；C 的问题是层数堆上去了，但每层变窄，新增深度未必能稳定换来表达收益。

再看真实工程配置。GPT-3 175B 使用约 $96$ 层、隐藏维度 $12288$；LLaMA 65B 使用约 $80$ 层、隐藏维度 $8192$；LLaMA 7B 使用约 $32$ 层、隐藏维度 $4096$。这些模型都没有走“极深窄塔”或“极宽薄饼”的极端路线，而是停留在一个相对稳定的宽深窗口内。这说明工程界并不是机械照抄一个公式，而是在理论轨迹附近，再叠加训练稳定性、并行效率、显存与吞吐约束做折中。

| 组合 | 宽深比 $d/n$ | 参数趋势 | 预期问题 |
| --- | --- | --- | --- |
| $n=64, d=8192$ | 128 | 较平衡 | 常见于可训练区间 |
| $n=96, d=12288$ | 128 | 大模型常见 | 吞吐和显存压力较大 |
| $n=160, d=6144$ | 38.4 | 偏深 | 梯度流、收敛和调参更敏感 |
| $n=48, d=12288$ | 256 | 偏宽 | 参数效率偏低，容易把预算过多花在单层容量上 |

## 代码实现

下面的脚本不追求复现论文中的精确常数，也不把某条理论阈值硬编码成死规则。它做的是三件更实际的事：

1. 用 $N \approx 12nd^2$ 估算 dense Transformer 主干参数。
2. 在固定预算下枚举候选层数，并反推可用宽度。
3. 用“偏离 $d \propto n^{2/3}$ 的程度”和“是否过深进入低收益区”给候选配置打分。

代码可以直接运行。

```python
import math
from typing import Dict, List, Tuple


def estimate_params(n_layers: int, d_model: int, vocab_size: int = 50000) -> int:
    """
    粗略估算 dense Transformer 参数量。
    主体采用 N ≈ 12 * n * d^2，并额外加一个 embedding 主项。
    """
    core = 12 * n_layers * d_model * d_model
    embed = vocab_size * d_model
    return core + embed


def depth_threshold(d_model: int) -> float:
    """
    提示性深度阈值：L_th ≈ log_3(d)。
    它不是硬上限，只表示“继续加深后收益可能变慢”的区间。
    """
    return math.log(d_model, 3)


def choose_dims(
    target_params: int,
    alpha: float = 512.0,
    n_min: int = 24,
    n_max: int = 128,
    vocab_size: int = 50000,
    multiple: int = 128,
) -> Tuple[Dict, List[Dict]]:
    """
    搜索固定参数预算下的候选 (n, d) 组合。

    alpha 控制目标轨迹 d ~= alpha * n^(2/3) 的常数项。
    这里的 alpha 不是理论常数，只是把“2/3 轨迹”映射到具体量级的工程旋钮。
    """
    best = None
    candidates: List[Dict] = []

    for n in range(n_min, n_max + 1):
        # 预算反推宽度
        d_budget = math.sqrt(target_params / (12 * n))

        # 对齐到常见硬件友好的倍数
        d_model = int(d_budget // multiple) * multiple
        if d_model < multiple:
            continue

        approx_params = estimate_params(n, d_model, vocab_size=vocab_size)

        # 与 d ∝ n^(2/3) 的偏离程度
        d_target = alpha * (n ** (2 / 3))
        scale_penalty = abs(d_model - d_target) / max(d_target, 1.0)

        # 与目标预算的偏离程度
        budget_penalty = abs(approx_params - target_params) / target_params

        # 深度阈值只做告警式惩罚，不做硬过滤
        l_th = depth_threshold(d_model)
        depth_penalty = max(0.0, n / max(l_th, 1e-6) - 12.0)

        # 分数越小越好
        score = scale_penalty + 2.0 * budget_penalty + 0.05 * depth_penalty

        row = {
            "n_layers": n,
            "d_model": d_model,
            "approx_params": approx_params,
            "width_depth_ratio": round(d_model / n, 1),
            "depth_threshold": round(l_th, 2),
            "score": round(score, 4),
        }
        candidates.append(row)

        if best is None or score < best["score"]:
            best = row

    candidates.sort(key=lambda x: x["score"])
    return best, candidates[:5]


def pretty_print(title: str, result: Dict, topk: List[Dict]) -> None:
    print(f"\n[{title}]")
    print("best =", result)
    print("top candidates:")
    for item in topk:
        print(" ", item)


if __name__ == "__main__":
    best_175b, top_175b = choose_dims(
        target_params=175_000_000_000,
        alpha=512.0,
        n_min=48,
        n_max=128,
    )

    best_7b, top_7b = choose_dims(
        target_params=7_000_000_000,
        alpha=420.0,
        n_min=24,
        n_max=80,
    )

    pretty_print("175B-scale toy search", best_175b, top_175b)
    pretty_print("7B-scale toy search", best_7b, top_7b)
```

这段代码和原版相比，有三个关键修复。

第一，代码现在是可直接运行的。所有函数都有明确输入输出，`__main__` 入口也给了两个预算档位的示例。第二，它没有把 `n <= log_3(d)` 写成硬性 `assert`，因为现实中的大模型经常会超过这个数量级；更合理的做法是把它当作“过深风险提示”，而不是结构合法性的死线。第三，它把“2/3 缩放”做成了可调参数 `alpha`，因为指数可以作为设计基线，但常数项要由模型家族、tokenizer、词表、并行方式和训练习惯共同决定。

对新手来说，最需要理解的是这段程序背后的决策逻辑：

- 预算先决定“你大概买得起多宽”。
- 2/3 轨迹决定“你不该把层数和宽度拉向两个极端”。
- 深度阈值提示决定“你继续加层时，收益可能已经不再值回额外的优化复杂度”。

如果要把它接入真实训练配置系统，通常还会再加三条工程规则：

| 规则 | 作用 |
| --- | --- |
| `d_model` 对齐到 128 或 256 的倍数 | 便于张量并行和矩阵乘吞吐 |
| 当 $d/n$ 过低时自动提高稳定性配置 | 例如启用 RMSNorm、Pre-LN、更长 warmup、梯度裁剪 |
| 对多个候选点做小规模预训练试跑 | 用实际 loss、吞吐和稳定性筛掉纸面“好看但不好训”的方案 |

## 工程权衡与常见坑

过宽和过深的问题不是同一种问题，它们坏掉的方式不同。

过宽的核心代价是参数爆炸。因为主项近似满足

$$
N \propto nd^2
$$

所以只要宽度上去，FFN 和注意力矩阵会一起变大。直接后果通常有三个。第一，显存占用和优化器状态同步增加；第二，在同样显存预算下，batch size 往往被迫缩小，训练噪声更大；第三，模型更容易把容量用在记忆训练语料上，而不是稳定提升泛化。

过深的问题则更像“有效深度下降”。理论上堆层能增加复合次数，但在工程里，信号要穿过更多残差分支、归一化和注意力块。层数继续加，梯度路径更长，统计量更敏感，新增层未必真正贡献能力，反而可能变成额外的优化负担。这就是为什么很多团队宁愿保留中等深度、适当增加宽度，也不愿盲目把层数翻倍。

一个典型场景是：预算允许你把 80 层改成 160 层，但这么做后每层宽度必须明显缩小。结果往往是总计算增加了，训练更慢了，调参也更难了，但最终 loss 未必更好。问题不在于“深度一定差”，而在于你为了深度付出的宽度代价已经过大。

| 风险 | 直接原因 | 典型现象 | 防范措施 |
| --- | --- | --- | --- |
| 过宽 | $d^2$ 增长过快 | 显存暴涨、优化器状态膨胀、batch 被迫变小 | 控制宽深比，先做小模型试验 |
| 过深 | 每层被压得过窄，且层数过多 | 收敛慢、梯度不稳、不同随机种子波动大 | Pre-LN、RMSNorm、梯度裁剪、适度减层 |
| 宽深都大 | 同时触发预算和吞吐压力 | 训练成本失控、并行效率下降 | 先验证单卡吞吐和多卡扩展性 |
| 理论照抄 | 把近似规律当硬约束 | 参数表面合理，但实际并不好训 | 用理论做起点，用实验做最终裁决 |

实操上，一个保守而有效的流程是：先把宽深比放在相对稳定的窗口，例如 $64$ 到 $128$ 附近，再做一小圈网格搜索；观察的不只是最终 loss，还要看早期收敛速度、梯度范数、吞吐、显存峰值和训练中断率。很多坏配置在前几千步就会暴露，不需要等到整个预训练跑完才发现方向错了。

## 替代方案与适用边界

$d \propto n^{2/3}$ 不是万能答案，它只是 dense Transformer 在固定预算下的一条基线。如果你的目标函数、硬件环境或模型结构变了，最优宽深比也会跟着变。

第一类替代方案是 MoE。它把“总参数”和“每 token 激活参数”拆开，使得

$$
N_{\text{active}} \ll N_{\text{total}}
$$

这样做的效果是，总容量可以很大，但单步计算不必等比例上升。它确实能缓解 dense 模型里“宽度太贵”的问题，但代价会转移到专家路由、负载均衡、通信和训练稳定性上。换句话说，MoE 不是取消宽深权衡，而是把权衡改写成了另一套约束。

第二类替代方案是压缩路线，例如蒸馏、低秩分解、剪枝。它们适合预算本来就很紧的场景。如果你只能训练一个几亿到几十亿参数的模型，那么与其在“深一点还是宽一点”之间来回挤压，不如先把大模型知识蒸馏到一个更容易训练的小骨架，再在剩余预算里微调宽深比。

第三类是部署导向的工程方案，例如量化、KV cache 优化、张量并行切分。它们不会改变宽深关系的理论方向，但会改变“哪种配置在你这套硬件上最容易落地”。例如同样是 8K 宽度，有的集群更容易容纳更宽但层数更少的模型；有的集群则更适合把参数切进更多层。

| 方案 | 适用规模 | 解决的问题 | 宽深比提示 |
| --- | --- | --- | --- |
| Dense 基线 | 大多数常规模型 | 直接做宽深平衡 | 先以 $d \propto n^{2/3}$ 作为起点 |
| MoE | 中大到超大模型 | 扩大总容量而控制单步计算 | 仍需关注激活专家的实际宽度 |
| 蒸馏/低秩分解 | 小模型、部署受限场景 | 降低参数量和推理成本 | 先压缩，再重新搜索宽深比 |
| 量化/并行优化 | 推理或训练资源受限 | 提升可部署性与吞吐 | 不改变理论趋势，但改变落地最优点 |

适用边界也要说清楚。Levine 的理论分析针对的是自注意力网络的表达效率，不是对所有现代变体的统一定理。长上下文模型、状态空间模型混合架构、共享参数模型、MoE 和各种检索增强系统，都可能让最优宽深比例偏离这条基线。因此更稳妥的用法是：把 $d \propto n^{2/3}$ 当成初始设计原则，用来排除明显极端的宽深配置；最终方案仍然要由实验结果决定。
