## 核心结论

BLOOM 的多语言平衡策略，核心不是“让每种语言完全一样多”，而是“让高资源语言不再垄断训练，同时让低资源语言持续进入 batch”。它的做法可以写成语言级温度采样（temperature sampling）：

$$
p_l=\frac{n_l^\alpha}{\sum_j n_j^\alpha},\quad \alpha=0.3
$$

其中：

- $n_l$ 是语言 $l$ 在训练语料中的有效数据量，可以理解为清洗后可用于训练的 token 数或样本量。
- $p_l$ 是训练时抽到语言 $l$ 的概率。
- $\alpha$ 是“拉平力度”参数。$\alpha=1$ 表示不做重加权，完全按原始数据量采样；$0<\alpha<1$ 表示压缩大语种优势，抬高小语种曝光。

这里 $\alpha=0.3$ 的效果很强，因为幂次小于 1 时，大数会被压扁，小数会被抬高。结果不是英语消失，而是英语从“天然占多数”变成“仍然常见，但无法长期霸占训练”。对 BLOOM 这类 176B 参数、覆盖 46 种自然语言和 13 种编程语言的预训练模型，这一步直接影响模型是否还能在非英语、非拉丁字符语言上保留基本能力。

一个最小玩具例子就能看清这个机制。假设英语原始数据占 50%，某低资源语言占 1%。如果直接按原始比例采样，二者曝光比约为：

$$
\frac{50}{1}=50
$$

如果改成 $\alpha=0.3$，二者的未归一化权重变成：

$$
50^{0.3}\approx 3.23,\quad 1^{0.3}=1
$$

新的曝光比约为：

$$
\frac{50^{0.3}}{1^{0.3}}\approx 3.23
$$

也就是说，原来 50 倍的数据量差距，被压成大约 3.23 倍的采样权重差距。对初学者，可以把它理解成：大语种还是多吃几口，但不能把整桌饭都吃完；小语种虽然菜少，但每轮仍然能上桌。

还需要先纠正一个常见混淆。ROOTS 通常描述为约 1.6TB 的组合语料，不是“1.6T token”。BLOOM 的公开资料通常写作：模型训练使用 ROOTS 语料，覆盖 46 种自然语言和 13 种编程语言，总量约 1.6TB、约 350B 级别 token。讨论平衡策略时，重点不是死记总 token 数，而是看“语言采样权重如何定义，以及它如何改变 batch 中的语言出现频率”。

| 场景 | 原始占比 | 温度变换后未归一化权重 | 直观结果 |
|---|---:|---:|---|
| 英语 | 50% | $50^{0.3}\approx 3.23$ | 被明显降采样，不再长期主导 |
| 低资源语言 | 1% | $1^{0.3}=1$ | 相对曝光被抬高，能持续进入训练 |

一句话概括：BLOOM 的策略不是“均分数据”，而是“控制训练中的语言暴露分布”，让尾部语言不至于在优化过程中被完全淹没。

## 问题定义与边界

问题定义很直接：多语言预训练时，各语言的数据规模差异极大。如果直接按原始 token 比例喂数据，模型会在高资源语言上学得更快、更稳，但低资源语言几乎没有有效训练机会。这里的“有效训练机会”，不是“词表里存在这个语言”，而是“模型在很多轮参数更新中反复见到这种语言，足以让梯度真正来自它”。

把这个问题写成训练视角，更准确一些。设总训练步数为 $T$，每步从某个语言桶中抽样一次，那么语言 $l$ 的期望抽样次数是：

$$
\mathbb{E}[N_l]=T\cdot p_l
$$

如果 $p_l$ 很小，哪怕训练总步数很大，语言 $l$ 在整个训练过程中对参数更新的贡献仍然可以忽略。BLOOM 要解决的正是这个问题：不是让所有语言“看起来公平”，而是让低资源语言在长期训练里保持非零且不至于过低的更新权重。

工程上，这个目标通常可以拆成四个约束：

1. 高资源语言不能因为数据太多而长期占满 batch。
2. 低资源语言不能因为数据太少而几乎不出现。
3. 采样后的数据仍必须来自可追溯、可清洗、可审核的来源，不能为了“平衡”硬塞低质量噪声。
4. 总训练预算固定时，任何语言的上采样都意味着其他语言被相对降采样，这是硬约束，不是免费收益。

可以把这个 batch 级目标写成一个粗略但有用的约束：

$$
\Pr(l\in B_t)\ \text{should remain non-trivial for low-resource languages}
$$

其中 $B_t$ 表示第 $t$ 个 batch。意思是：对低资源语言来说，在大量 batch 中“出现的概率”不能低到接近 0，否则从优化角度看，这个语言基本没有参与训练。

一个更具体的玩具例子是：假设共有 10 种语言，英语占 50%，另外 9 种语言共享剩余 50%，其中还有几种更小的尾部语言。如果完全按原始比例采样，很多 batch 会重复出现英语和少数大语种，而尾部语言可能长时间不出现。结果是：

- tokenizer 可能支持这些语言；
- 参数却没有被足够多的样本更新；
- 模型“看起来多语”，但实际能力严重倾斜。

真实工程里的问题比玩具例子更复杂。ROOTS 不是简单地“全网抓下来直接混合”，而是先由社区整理和记录不同语言的数据来源，再与大规模抓取语料组合。这样做的原因很现实：低资源语言最怕的不是“少”，而是“少且脏”。如果一个语言只有很少数据，且里面大量是模板页、乱码、重复页面、OCR 错误或机器翻译残片，那么把它上采样，本质上是在放大噪声。

因此，本文讨论的边界要说清楚：

- 讨论对象是“多语言预训练阶段的语言级采样”。
- 不讨论 tokenizer 设计细节本身。
- 不讨论监督微调、指令微调或 RLHF。
- 不讨论专门的机器翻译系统训练。
- 不声称温度采样单独就能解决所有多语言能力问题。

温度采样解决的是“训练暴露分布”问题，不是全部问题。它能提高低资源语言被看到的概率，但不能凭空创造新知识，也不能替代高质量数据建设。

## 核心机制与推导

温度采样的核心机制只有一步：把每个语言的原始数据量 $n_l$ 先经过幂函数压缩，再做归一化。公式再次写一遍：

$$
p_l=\frac{n_l^\alpha}{\sum_j n_j^\alpha}
$$

这个公式为什么有效，关键在于 $0<\alpha<1$ 时，函数 $f(x)=x^\alpha$ 是凹函数。凹函数有一个重要性质：输入越大，继续增大的边际收益越小。换句话说：

- 大语种即使数据量非常大，经过幂变换后优势会被压缩。
- 小语种虽然绝对数据量没变，但相对权重会被明显抬高。

更直观地说，原始分布里比较的是“数据量差多少倍”，温度采样后比较的是“数据量差的幂次”。任意两个语言 $i,j$ 的采样权重比满足：

$$
\frac{p_i}{p_j}=\frac{n_i^\alpha}{n_j^\alpha}=\left(\frac{n_i}{n_j}\right)^\alpha
$$

这个式子很重要，因为它直接说明了“压缩”发生在哪里。若原始数据量差距为 $r=\frac{n_i}{n_j}$，则新差距变成 $r^\alpha$。当 $\alpha=0.3$ 时：

- 原始差距 $10$ 倍，会变成 $10^{0.3}\approx 2.00$ 倍；
- 原始差距 $100$ 倍，会变成 $100^{0.3}\approx 3.98$ 倍；
- 原始差距 $1000$ 倍，会变成 $1000^{0.3}\approx 7.94$ 倍。

这就是为什么它对极端长尾分布特别有用。原始分布里 100 倍、1000 倍的差距，在训练时会导致近乎垄断；而经过 $\alpha=0.3$ 压缩后，这些差距虽然还在，但已经变成“可管理的差距”。

看一个更完整的玩具例子。假设只有 4 种语言，原始占比分别是：

- 英语：50
- 法语：20
- 阿拉伯语：5
- 低资源语言 X：1

先做幂变换：

- $50^{0.3}\approx 3.23$
- $20^{0.3}\approx 2.46$
- $5^{0.3}\approx 1.62$
- $1^{0.3}=1$

总和约为：

$$
3.23+2.46+1.62+1.00=8.31
$$

归一化后得到：

| 语言 | 原始份额 | $n_l^{0.3}$ | 采样后份额 |
|---|---:|---:|---:|
| 英语 | 50 | 3.23 | 38.9% |
| 法语 | 20 | 2.46 | 29.6% |
| 阿拉伯语 | 5 | 1.62 | 19.5% |
| 低资源语言 X | 1 | 1.00 | 12.0% |

这个表说明三件事。

第一，温度采样不是平均采样。英语仍然是最高权重，只是不再像原始分布那样压倒性领先。  
第二，低资源语言的提升是“相对提升”，不是“生成了更多新数据”。语言 X 的采样份额从原始分布里的极小比例，被抬到了约 12%，但它实际看到的仍然是那一小池数据被更频繁重放。  
第三，$\alpha$ 的作用是连续的，不是开关。你可以通过调节 $\alpha$ 控制“保留原始分布”与“拉平长尾”之间的折中。

再看 batch 视角。如果一个训练 run 共执行 $T=10^6$ 个 step，那么上表四种语言的期望被抽中次数大致是：

$$
10^6 \times (0.389,\ 0.296,\ 0.195,\ 0.120)
$$

也就是约：

- 英语：389,000 次
- 法语：296,000 次
- 阿拉伯语：195,000 次
- 低资源语言 X：120,000 次

而按原始比例，语言 X 的出现次数会低得多。这个差异不只是“统计上更平均”，而是直接决定了训练时梯度更新由哪些语言贡献。

这也解释了为什么 BLOOM 必须把温度采样和数据治理一起看。ROOTS 的价值，不只是“有很多语言”，而是“先尽量保证每种语言都有一批可用、可追溯、可清洗的数据池，再在训练阶段用重加权改变暴露频率”。如果没有前一步，后一步只会把重复和噪声一起放大。

从优化角度看，这种设计相当于控制梯度来源的多样性。白话解释是：如果每次参数更新几乎都由英语样本驱动，模型自然更懂英语；如果 batch 中持续混入阿拉伯语、印地语、班巴拉语、沃洛夫语，甚至不同编程语言，模型参数就会被迫适配更广的输入分布。BLOOM 的多语言能力不是“参数足够大自然就有”，而是训练流里真的持续出现了这些语言。

## 代码实现

下面给出一个可运行的 Python 例子，完整模拟三件事：

1. 根据语言数据量计算温度采样概率。
2. 验证概率和为 1。
3. 进行多次抽样，观察经验频率是否接近理论概率。

这段代码不依赖深度学习框架，只展示“语言级采样”本身。

```python
from __future__ import annotations

from math import isclose
from random import Random


def compute_sampling_probs(counts: dict[str, float], alpha: float = 0.3) -> dict[str, float]:
    """
    根据各语言数据量计算温度采样概率。

    参数
    ----
    counts:
        {语言: token数或样本量}
    alpha:
        温度参数。alpha=1 表示按原始分布采样；
        0<alpha<1 表示压平高资源语言优势。

    返回
    ----
    {语言: 采样概率}
    """
    if not counts:
        raise ValueError("counts 不能为空")
    if not (0 < alpha <= 1):
        raise ValueError("alpha 必须位于 (0, 1] 区间")
    if any(value <= 0 for value in counts.values()):
        raise ValueError("所有语言的数据量必须为正数")

    powered = {lang: value ** alpha for lang, value in counts.items()}
    total = sum(powered.values())
    probs = {lang: value / total for lang, value in powered.items()}

    if not isclose(sum(probs.values()), 1.0, rel_tol=1e-12, abs_tol=1e-12):
        raise RuntimeError("概率和不为 1，计算有误")

    return probs


def sample_languages(
    probs: dict[str, float],
    num_steps: int,
    seed: int = 7,
) -> dict[str, int]:
    """
    按给定概率重复抽样，模拟训练中“每个 step 先抽语言”的过程。
    """
    if num_steps <= 0:
        raise ValueError("num_steps 必须为正整数")

    langs = list(probs.keys())
    weights = list(probs.values())
    rng = Random(seed)

    counts = {lang: 0 for lang in langs}
    for _ in range(num_steps):
        chosen = rng.choices(langs, weights=weights, k=1)[0]
        counts[chosen] += 1

    return counts


def normalize(counts: dict[str, int]) -> dict[str, float]:
    total = sum(counts.values())
    return {k: v / total for k, v in counts.items()}


if __name__ == "__main__":
    # 玩具例子：四种语言的原始数据量
    language_counts = {
        "en": 50.0,
        "fr": 20.0,
        "ar": 5.0,
        "low_resource_x": 1.0,
    }

    probs = compute_sampling_probs(language_counts, alpha=0.3)

    # 理论上应保持排序：大语种仍更常见，但不再是原始分布那种极端差距
    assert probs["en"] > probs["fr"] > probs["ar"] > probs["low_resource_x"]
    assert probs["low_resource_x"] > 0.10
    assert probs["en"] < 0.40

    sampled_counts = sample_languages(probs, num_steps=100_000, seed=42)
    empirical_probs = normalize(sampled_counts)

    print("Theoretical probabilities:")
    for lang, p in probs.items():
        print(f"  {lang:15s} {p:.4f}")

    print("\nEmpirical probabilities after 100000 samples:")
    for lang, p in empirical_probs.items():
        print(f"  {lang:15s} {p:.4f}")

    print("\nSample counts:")
    for lang, c in sampled_counts.items():
        print(f"  {lang:15s} {c}")
```

在本地运行时，你会看到经验概率接近理论概率，通常类似这样：

```text
Theoretical probabilities:
  en              0.3886
  fr              0.2959
  ar              0.1945
  low_resource_x  0.1203

Empirical probabilities after 100000 samples:
  en              0.3898
  fr              0.2938
  ar              0.1958
  low_resource_x  0.1206
```

这段代码比“只算一个公式”更接近真实训练流程，因为它展示了一个关键点：训练时关心的不是静态比例表本身，而是大量 step 上的语言出现频率。

如果要把它再向真实工程推进一步，通常还需要做四件事：

| 工程步骤 | 作用 | 为什么不能省 |
|---|---|---|
| 统计各语言有效 token 数 | 建立 $n_l$ | 没有统计，就无法定义采样权重 |
| 清洗、去重、质量过滤 | 保证样本可用 | 否则上采样会把脏数据一起放大 |
| 构建按语言分桶的数据流 | 支持语言级抽样 | 训练时通常不是直接在全量混合池里盲抽 |
| 记录每轮语言曝光日志 | 检查分布是否跑偏 | 分布漂移会直接影响训练效果 |

用伪代码写成训练循环，会更清楚：

```python
for step in range(total_steps):
    lang = sample_language(language_probs)      # 先抽语言
    doc = sample_document_from_bucket(lang)     # 再从该语言桶取文档
    tokens = tokenize_and_pack(doc)             # 切分为训练序列
    loss = model(tokens)
    loss.backward()
    optimizer.step()
```

这里最容易被忽略的一点是：`sample_language` 和 `sample_document_from_bucket` 是两个不同层次。前者控制语言暴露分布，后者决定该语言内部哪些文档被重复看到。对低资源语言来说，这两个层次都会影响过拟合风险。

再给一个新手常见误区的对照：

| 误区 | 实际情况 |
|---|---|
| “采样概率变大，说明这个语言数据变多了” | 不是。数据没变多，只是被抽到得更频繁 |
| “只要做了温度采样，低资源语言一定会学好” | 不是。它只提高曝光，不保证知识覆盖和泛化 |
| “语言比例平衡后，训练就公平了” | 不一定。数据质量、脚本复杂度、重复率仍然会影响效果 |

所以，代码实现看起来简单，但真正有效的前提是：先有合格的数据池，再做抽样重加权。

## 工程权衡与常见坑

温度采样的第一个权衡是 $\alpha$ 的选择。$\alpha$ 越小，分布越平；$\alpha$ 越大，越接近原始分布。它不是越小越好，因为你在上采样尾部语言的同时，也在更频繁地重复使用那部分有限数据。

| $\alpha$ | 分布特征 | 对低资源语言 | 主要风险 |
|---|---|---|---|
| 0.1 | 极强拉平 | 曝光大幅提升 | 重复过多，容易记忆化和过拟合 |
| 0.3 | 明显拉平 | 有稳定训练机会 | 仍然依赖高质量清洗与去重 |
| 0.6 | 轻度拉平 | 提升有限 | 高资源语言重新主导 |
| 1.0 | 不拉平 | 几乎无保护 | 训练被原始分布强烈支配 |

对初学者，一个更容易记住的理解是：

- $\alpha=0.1$ 更像“强行给小语种抢镜头”，风险是内容太少，模型反复看同一批文本。
- $\alpha=0.6$ 更像“稍微照顾一下小语种”，风险是照顾力度不够，大语种很快重新占主导。
- $\alpha=0.3$ 通常是中间地带：既明显改变暴露分布，又没有走到近似均匀采样那么激进。

第二个常见坑，是把“采样更公平”误解成“性能一定单调提升”。真实情况通常不是线性的。低资源语言曝光增加，确实可能改善零样本或少样本表现，但提升幅度取决于很多因素：

- 数据质量是否足够高；
- 数据重复率是否过高；
- tokenizer 是否对该语言分词过碎；
- 该语言和高资源语言是否存在可迁移的共享结构；
- 下游任务是不是依赖世界知识而不是纯语言建模。

也就是说，温度采样解决的是“有机会学”，不是“保证学会”。

第三个坑，是忽略极低资源场景。如果某语言只有几十 KB 到几万 token 量级，那么再高的采样权重也只是更频繁地重复同一批文本。可以把这个现象写成一个近似判断：

$$
\text{effective diversity} \ll \text{sampling frequency}
$$

这表示：真正有用的内容多样性，远小于训练中重复看到它的频率。白话讲，就是“杯子里只有几滴水，怎么放大都还是那几滴”。这类语言往往需要：

- 补充语料；
- 做跨语迁移；
- 或在预训练后做语言特定继续预训练。

第四个坑，是只做采样，不做数据治理。低资源语言的数据更容易混入：

- OCR 错误；
- 编码损坏；
- 机器翻译伪语料；
- 页面模板；
- 重复抓取；
- 大量 boilerplate 文本。

温度采样会把这些问题一起放大，所以它必须和去重、质量过滤、来源审核一起使用。否则模型看到的不是“更多该语言”，而是“更多该语言中的垃圾”。

第五个坑，是把语言采样和任务采样混为一谈。BLOOM 讨论的是预训练阶段的语言级数据混合，而不是指令微调阶段“任务 A 和任务 B 如何分配 batch”。这两个问题都叫 sampling，但目标完全不同：

| 采样层级 | 目标 | 典型对象 |
|---|---|---|
| 语言级采样 | 控制语言暴露分布 | 英语、法语、阿拉伯语、沃洛夫语 |
| 任务级采样 | 控制任务学习顺序 | 问答、摘要、翻译、代码补全 |
| 样本级采样 | 控制实例重复频率 | 单条文档、单个 prompt-response 对 |

最后一个常见误区，是只看比例表，不看训练日志。理论上算出的 $p_l$ 很漂亮，但实际训练中还会受到 shard 切分、数据加载器实现、pack 长度、过滤器、断点恢复策略等影响。真正可靠的做法是同时记录：

- 每种语言的目标采样概率；
- 实际被抽中的次数；
- 每个语言桶的重复率；
- 去重前后有效 token 变化。

否则你以为自己在做平衡，实际跑出来的分布可能已经偏了。

## 替代方案与适用边界

温度采样不是唯一方案，但它在“语言统计已知、语言数量多、资源差距大”的预训练场景里非常实用，因为实现简单、可连续调节，而且能直接作用于训练数据流。

先看几种常见替代方案：

| 方案 | 做法 | 优点 | 局限 |
|---|---|---|---|
| 温度采样 | 按 $n_l^\alpha$ 归一化 | 简单、可连续调节、适合大规模训练 | 极低资源语言仍可能重复过度 |
| 均匀采样 | 每种语言同概率 | 最大化语言曝光公平性 | 尾部语言极易被过度重放 |
| 按语系分层采样 | 先抽语系，再抽语言 | 可减少同语系大语言垄断 | 设计复杂，依赖人工分层 |
| 设置最小/最大采样上限 | 对每种语言截断权重 | 工程上容易控风险 | 阈值选择较主观 |
| 训练后继续预训练/微调 | 对重点语言单独补训 | 对目标语言补救有效 | 不能替代基础预训练覆盖 |

为什么温度采样常常是第一选择？因为它介于两个极端之间：

- 比原始分布更照顾尾部语言；
- 比均匀采样更尊重真实数据量；
- 不需要人工为几十种语言逐一写规则；
- 参数只有一个，便于调参和复现实验。

但它也有清晰边界。

如果面对的是几十种语言、每种语言至少有可用 token 统计，而且尾部语言不是“几乎空集”，温度采样通常是最稳妥的方案。  
如果某语言资源少到几乎没有有效文本，温度采样只能保证“它被看到”，不能保证“它被学会”。这时更合理的方案是先补数据，再配合语言特定继续预训练。  
如果任务目标不是通用多语能力，而是少数高价值语言的特定指标，例如法律问答、医学检索、区域客服，那么直接做目标语言继续预训练或任务微调，往往比在预训练阶段追求全局平衡更有效。

对工程团队来说，一个很实用的决策框架是：

| 你当前的问题 | 更合适的办法 |
|---|---|
| 长尾语言在预训练中几乎不出现 | 先用温度采样改善曝光 |
| 某些语言数据极少且重复严重 | 先补数据，再考虑采样 |
| 少数重点语言指标不达标 | 做语言特定继续预训练 |
| 任务是特定领域而非通用建模 | 直接做领域数据适配 |
| 需要兼顾多语覆盖和训练稳定性 | 预训练用温度采样，后续再微调重点语言 |

真实工程里，常见组合方案是：

1. 预训练阶段用 $\alpha=0.3$ 一类的温度采样，保证多语言底线覆盖；
2. 对低质量语言桶做更严格清洗，避免脏数据被放大；
3. 训练后对重点语言做继续预训练或指令微调；
4. 用下游评测确认“曝光增加”是否真的转化成“能力提升”。

所以，温度采样最适合拿来解决“基础覆盖不足”的问题，不适合被神化成一把万能钥匙。它改变的是训练机会，不是语言能力的全部来源。
