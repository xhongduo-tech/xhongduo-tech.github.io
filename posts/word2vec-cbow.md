## 核心结论

CBOW（Continuous Bag of Words，连续词袋）是 Word2Vec 里的一个基础模型。它做的事很直接：给定窗口内的上下文词，预测中间那个中心词。这里的“词袋”不是说把词丢进一个袋子里，而是说它不严格区分上下文词的顺序，只聚合这些词的信息。

如果把一句话写成词序列 $w_1,w_2,\dots,w_T$，对位置 $t$ 的中心词 $w_t$，设窗口半径为 $c$，那么上下文集合记为 $C_t$。CBOW 先把上下文词映射成向量，再取平均，得到上下文表示：
$$
v_c=\frac{1}{2c}\sum_{j=1}^{2c} W^\top \text{one\_hot}(w_{t+j}),
$$
然后用 Softmax 计算中心词概率。Softmax 可以理解为“把一组分数变成概率分布”的函数：
$$
P(w_t\mid C_t)=\frac{\exp(u_{w_t}^\top v_c)}{\sum_{w'\in V}\exp(u_{w'}^\top v_c)}.
$$

CBOW 的核心价值有三点：

| 结论 | 含义 | 工程结果 |
|---|---|---|
| 用上下文预测中心词 | 通过共现关系学习词向量 | 不需要人工标注 |
| 上下文向量通常取平均 | 模型结构简单、参数共享 | 训练速度快 |
| 窗口大小影响偏好 | 小窗口偏句法，大窗口偏语义 | 需要按任务调参 |

玩具例子：词表只有 `{A, B, C}`，窗口中上下文是 `A` 和 `B`，模型把 `A`、`B` 的向量取平均后去预测中心词 `C`。这说明 CBOW 学到的不是“某个词本身长什么样”，而是“哪些词经常一起出现”。

真实工程例子：在电商搜索日志里，用户连续搜索“红色 连衣裙 夏装”。如果把“红色”“连衣裙”作为上下文，中心词设为“夏装”，CBOW 会把这类共现模式压缩进向量里。训练完后，“夏装”“碎花裙”“短袖连衣裙”这类词可能在向量空间里更接近，可用于召回、扩展查询和冷启动推荐。

## 问题定义与边界

先把问题说清楚。CBOW 不是在做句子理解，也不是在生成文本。它只解决一个更小的问题：在固定窗口内，根据周围词预测中心词。

模型的输入和输出边界如下：

| 项目 | 定义 | 维度/形式 |
|---|---|---|
| 输入词 | 上下文中的每个词 | one-hot 向量，维度 $V$ |
| 嵌入矩阵 $W$ | 词表到低维向量的查表参数 | $V\times d$ |
| 上下文表示 $v_c$ | 多个上下文词向量的平均 | $d$ |
| 输出权重 $U$ | 用于给每个候选中心词打分 | $V\times d$ |
| 输出 | 中心词的概率分布 | $V$ 维概率 |

这里的 one-hot 向量可以理解为“只有当前位置是 1，其余全是 0 的索引表示”。它本身不包含语义，只是为了从矩阵 $W$ 中取出对应那一行词向量。

训练目标是最大化所有位置的对数似然：
$$
\max_{W,U}\sum_t \log P(w_t\mid C_t).
$$

注意边界条件：

1. CBOW 依赖分词结果。分词不好，词向量质量会直接下降。
2. 它通常忽略词序，只保留“哪些词一起出现过”。
3. 它输出的是静态词向量。静态的意思是“同一个词无论放在哪个句子里，向量都一样”。例如 “bank” 在金融和河岸语境里不会自动分开。
4. 它不擅长建模长距离依赖，窗口之外的信息通常看不到。

再看一个极简定义例子。词典是 `{A,B,C}`，上下文是 `A/B`，目标预测中心词 `C`。这个任务不要求模型理解句子，只要求模型学到：如果 `A` 和 `B` 常常围着 `C` 出现，那么 `A`、`B` 的平均向量应该能给 `C` 更高分。

---

## 核心机制与推导

CBOW 的计算链路可以拆成四步：

1. 把上下文词变成 one-hot。
2. 用共享的嵌入矩阵 $W$ 查出每个词向量。
3. 对上下文词向量求平均，得到 $v_c$。
4. 用输出矩阵 $U$ 给词表中每个词打分，再经过 Softmax 变成概率。

“共享”这个词很重要。共享的意思是：不管一个词出现在左边、右边，还是不同句子里，它都用同一张 embedding 表。这样参数更少，也能让模型把分散的共现信息累积到同一个词向量上。

设上下文词是 $w_a,w_b$，对应输入向量分别为 $v_a,v_b$，则：
$$
v_c=\frac{v_a+v_b}{2}.
$$
如果要预测词 $C$，且输出层中 $C$ 的参数向量是 $u_C$，则它的未归一化分数为：
$$
s_C=u_C^\top v_c.
$$
对所有词都算一个分数后，用 Softmax 归一化。分数越大，概率越高。

玩具推导例子：

- 设 $v_A=[1,0]$
- 设 $v_B=[0,1]$
- 则上下文平均向量为 $v_c=[0.5,0.5]$
- 若 $u_C=[0.4,0.6]$，则
$$
u_C^\top v_c = 0.4\times 0.5 + 0.6\times 0.5 = 0.5
$$

这时 $C$ 的指数分数是 $\exp(0.5)$。如果其他词的分数更低，Softmax 后 $C$ 的概率就更大。

从训练角度看，梯度会从预测误差反向传回去，同时更新 $U$ 和 $W$。如果某些上下文经常对应同一个中心词，那么这些上下文词的向量会被推向一种“更容易一起预测该词”的方向。长期训练后，语义相近或用法相近的词会在向量空间里更接近。

这也是为什么窗口大小很关键：

| 窗口设置 | 更容易学到什么 | 原因 |
|---|---|---|
| 小窗口 | 句法关系 | 邻近词通常有局部搭配 |
| 大窗口 | 主题语义 | 更远词更反映整体语境 |

例如在“我 买了 一件 红色 连衣裙 夏装”里，小窗口更容易抓住“红色-连衣裙”的局部修饰关系；大窗口则可能把“夏装”和“连衣裙”“服饰”“女装”这类主题相关性学进去。

## 代码实现

下面给一个可运行的最小 CBOW 前向实现。它不依赖深度学习框架，只演示“查表、平均、Softmax、交叉熵”的核心逻辑。

```python
import math

def softmax(xs):
    m = max(xs)
    exps = [math.exp(x - m) for x in xs]
    s = sum(exps)
    return [e / s for e in exps]

def cbow_forward(context_ids, target_id, W, U):
    # W: V x d, U: V x d
    d = len(W[0])
    vc = [0.0] * d
    for idx in context_ids:
        for j in range(d):
            vc[j] += W[idx][j]
    for j in range(d):
        vc[j] /= len(context_ids)

    scores = []
    for out_vec in U:
        score = sum(out_vec[j] * vc[j] for j in range(d))
        scores.append(score)

    probs = softmax(scores)
    loss = -math.log(probs[target_id])
    return vc, probs, loss

# 词表: A, B, C
W = [
    [1.0, 0.0],  # A
    [0.0, 1.0],  # B
    [0.2, 0.8],  # C
]

U = [
    [0.1, 0.2],  # A
    [0.0, 0.3],  # B
    [0.4, 0.6],  # C
]

context_ids = [0, 1]   # A, B
target_id = 2          # C

vc, probs, loss = cbow_forward(context_ids, target_id, W, U)

assert vc == [0.5, 0.5]
assert abs(sum(probs) - 1.0) < 1e-9
assert probs[target_id] == max(probs)
assert loss > 0

print("context vector:", vc)
print("probs:", probs)
print("loss:", round(loss, 6))
```

这段代码对应的流程是：

```text
上下文词ID -> 查 W 取词向量 -> 求平均得到 vc
vc -> 与 U 中每个输出向量做点积 -> 得到 scores
scores -> softmax -> 概率分布
概率分布 + 目标词 -> 交叉熵损失
```

如果进一步训练，最基本的循环就是：

1. 从语料生成滑动窗口样本。
2. 取上下文词 ID 和中心词 ID。
3. 前向计算概率与损失。
4. 反向传播更新 $W,U$。
5. 重复直到损失收敛。

真实工程例子可以写成搜索日志训练：

- 原始日志序列：`["红色", "连衣裙", "夏装", "女装"]`
- 若窗口半径 $c=1$，则可构造样本：
  - 上下文 `["红色", "夏装"]`，预测 `"连衣裙"`
  - 上下文 `["连衣裙", "女装"]`，预测 `"夏装"`

训练后，“夏装”和“女装”的向量可能靠近，“连衣裙”和“裙子”也可能靠近。召回阶段可直接按向量相似度找近邻词或近邻商品标签。

如果词表很大，完整 Softmax 要对所有词打分，复杂度高。工程里常用负采样（negative sampling）替代。负采样可以理解为：每次不和整个词表比较，而是只拿“1 个正确词 + 若干个随机负词”来训练二分类目标，速度会快很多。

## 工程权衡与常见坑

CBOW 简单，但工程里有几个常见问题需要提前处理。

| 常见坑 | 现象 | 规避方式 |
|---|---|---|
| 全词表 Softmax 太慢 | 词表大时训练成本高 | 负采样或层次 Softmax |
| 窗口过大 | 主题相关性增强，但语法信息被稀释 | 按任务调小窗口 |
| 窗口过小 | 只学到局部搭配，泛化差 | 适当增大窗口 |
| 高频词主导训练 | “的”“是”“the”频繁更新，稀释有效信号 | subsampling |
| 稀有词训练不足 | 低频词向量不稳定 | 增加语料或改用 Skip-gram |

subsampling 指“对高频词按概率丢弃一部分样本”。白话说，就是别让“的、了、是、the、a”这种词在训练里出现得过于频繁，否则模型会把大量更新浪费在几乎没有区分度的词上。

一个典型坑是窗口设置错误。比如做句法相似任务，希望学到“形容词修饰名词”“动词支配宾语”这类局部结构，结果把窗口开到 10。这样模型更容易学到“这篇文章和科技有关”“这段文本和服饰有关”，而不是精细搭配关系。实际观察里，大窗口经常让“主题词靠近”，但会削弱局部语法信号。

另一个坑是误把 CBOW 当成上下文敏感模型。它学到的是静态 embedding，不会因为句子不同而给同一个词不同向量。所以它适合做轻量词表示，不适合直接处理多义词消歧、复杂问答或长文本推理。

## 替代方案与适用边界

CBOW 不是唯一选择。更准确地说，它适合“低成本地学习静态词向量”，但不是所有场景都该优先用它。

| 方法 | 核心任务 | 速度 | 对低频词 | 适用场景 |
|---|---|---|---|---|
| CBOW | 上下文预测中心词 | 快 | 一般 | 大语料、快速训练 |
| Skip-gram | 中心词预测上下文 | 较慢 | 更好 | 小语料、稀有词更重要 |
| GloVe | 基于全局共现矩阵分解 | 中等 | 依赖统计质量 | 需要显式全局统计 |
| Transformer embedding | 上下文条件编码 | 慢很多 | 更强 | 复杂语义、动态表示 |

CBOW 与 Skip-gram 最容易混淆。两者的任务方向相反：

- CBOW：给上下文，猜中心词。
- Skip-gram：给中心词，猜周围词。

经验上，CBOW 更快、更稳定，适合大规模语料预训练；Skip-gram 往往对低频词更友好，因为每个中心词会生成多个训练对，稀有词能获得更细的更新信号。

GloVe 更强调全局共现统计，不是通过局部窗口的预测任务来学表示。它在某些词相似度任务上表现很好，但工程链路和数据准备方式不同。

如果任务已经进入“一个词在不同上下文中含义会变”的阶段，比如“苹果”既可能是水果，也可能是公司，那么静态向量通常不够，应该考虑双向 LSTM 或 Transformer。这类模型输出的是上下文相关表示，也就是同一个词在不同句子里的向量可以不同。

所以 CBOW 的适用边界可以总结为：

1. 目标是训练静态词向量，不追求复杂上下文建模。
2. 语料量较大，希望训练快、实现简单。
3. 下游任务是召回、聚类、相似词搜索、轻量特征输入。
4. 不要求处理细粒度歧义和长距离语义推理。
