## 核心结论

文本分类的类别不平衡处理，是在训练数据分布、损失函数和推理阈值三层同时纠偏，避免模型只学会多数类。

类别不平衡指不同类别的样本数量差异过大，例如 95 条负类、5 条正类。模型如果全预测负类，accuracy 仍然有 95%，但正类召回率是 0。这个结果在业务上通常不可接受，因为少数类往往对应投诉、欺诈、事故、违规、账号盗用等高价值或高风险事件。

不平衡处理的目标不是让整体准确率更好看，而是让模型在少数类重要时，仍能学到少数类、判出少数类、把少数类报出来。

| 环节 | 主要问题 | 常用处理 | 关注指标 |
|---|---|---|---|
| 训练 | 多数类样本主导优化目标 | 重采样、类别权重、Focal Loss | 训练损失、各类召回 |
| 验证 | 默认阈值不一定符合业务目标 | 阈值搜索、分层验证集 | macro-F1、少数类 recall、PR-AUC |
| 部署 | 误报与漏报代价不同 | 按业务目标设置阈值 | 线上召回、误报率、人工审核压力 |

一个玩具例子：二分类任务中，正类只有 5 条，负类有 95 条。模型全部预测为负类：

| 指标 | 数值 | 含义 |
|---|---:|---|
| accuracy | 95% | 95 个负类都预测对 |
| 正类 recall | 0% | 5 个正类全部漏掉 |
| 正类业务价值 | 高 | 真正需要发现的对象没有被发现 |

所以，在极端不平衡场景下，accuracy 不能单独作为结论依据。至少要看 F1、少数类召回率；如果关注排序能力，还要看 ROC-AUC 或 PR-AUC。

## 问题定义与边界

分类不平衡的本质，是不同类别样本数量差异过大，导致训练目标被多数类主导。多数类是样本数量较多的类别，少数类是样本数量较少的类别。

文本分类中常见的不平衡场景包括客服工单分类、垃圾邮件识别、欺诈检测、事故预警、违规内容审核。真实工程例子是客服工单：`退款`、`物流查询`、`密码重置` 很多，`安全事件`、`账号盗用` 很少。业务上真正不能漏的是后者，因此不能只追求整体准确率。

需要区分三类问题：

| 类型 | 白话解释 | 例子 | 处理重点 |
|---|---|---|---|
| 数据不平衡 | 样本数量不均 | 账号盗用样本远少于物流咨询 | 重采样、分层划分 |
| 代价不平衡 | 不同错误的损失不同 | 漏掉安全事件比误报更严重 | 类别权重、Focal Loss |
| 阈值不平衡 | 判定门槛不符合业务目标 | 风险分 0.35 就应进入人工审核 | 阈值移动 |

任务目标也要拆开看：

| 类别 | 样本数量 | 业务价值 | 常见错误 | 优先目标 |
|---|---:|---|---|---|
| 多数类 | 多 | 通常稳定、低风险 | 被误判为少数类 | 控制误报 |
| 少数类 | 少 | 往往高风险、高价值 | 被漏判为多数类 | 提高召回 |

适用边界如下：

| 场景 | 是否适用不平衡处理 | 原因 |
|---|---|---|
| 少数类有明确业务价值 | 适用 | 漏报代价高 |
| 只是自然长尾，但错分代价接近 | 谨慎适用 | 过度纠偏可能降低整体体验 |
| 标签质量很差 | 先清洗标签 | 加权会放大错误标签影响 |
| 少数类只有 1 到 2 条 | 单靠训练技巧不够 | 需要补数据、合并类别或规则兜底 |
| 线上类别先验明显变化 | 需要持续监控 | 训练分布和线上分布可能不一致 |

## 核心机制与推导

普通交叉熵会偏向多数类，因为总损失由所有样本求和或求平均得到。交叉熵是分类任务常用损失函数，含义是模型对真实类别越不自信，损失越大。

如果负类有 95 条、正类有 5 条，模型在负类上犯的小错误会累计很多次。即使正类更重要，普通训练目标也不会自动知道这一点。它只会优化平均损失。

类别权重的做法是让少数类样本的损失乘上更大的系数。常用平衡权重为：

$$
w_c = \frac{N}{C \cdot n_c}
$$

其中 $N$ 是总样本数，$C$ 是类别数，$n_c$ 是第 $c$ 类样本数。样本越少，权重越大。

加权交叉熵为：

$$
L = -\frac{1}{N}\sum_{i=1}^{N} w_{y_i}\log p_{i,y_i}
$$

其中 $y_i$ 是第 $i$ 个样本的真实类别，$p_{i,y_i}$ 是模型给真实类别的预测概率。白话说法是：同样错一次，少数类错得更贵。

玩具例子：二分类里正类 5 条、负类 95 条。按平衡权重：

$$
w_+=\frac{100}{2\times5}=10
$$

$$
w_-=\frac{100}{2\times95}\approx0.526
$$

如果某个正样本和某个负样本的原始交叉熵都是 0.2，加权后分别是：

| 样本 | 原始损失 | 类别权重 | 加权损失 |
|---|---:|---:|---:|
| 正类 | 0.2 | 10 | 2.0 |
| 负类 | 0.2 | 0.526 | 0.105 |

这意味着少数类对梯度的影响被显著放大。梯度是模型参数更新的方向和强度，白话说就是“这条样本对模型下一步怎么改的影响”。权重越大，这类样本越能推动模型调整。

Focal Loss 进一步处理“简单样本太多”的问题。简单样本是模型已经很容易判对的样本，例如明显的物流查询。Focal Loss 公式为：

$$
FL = -\alpha_t (1-p_t)^\gamma \log p_t
$$

其中 $p_t$ 是真实类别的预测概率，$\alpha_t$ 是类别权重，$\gamma$ 控制压低简单样本的程度。若 $p_t$ 很高，说明模型已经很自信，$(1-p_t)^\gamma$ 会变小，这条样本的损失贡献会被压低。

阈值移动发生在推理层。二分类模型通常输出一个分数 $s(x)$，可以理解为样本属于正类的概率或置信度。默认阈值常是 0.5：

$$
\hat y = 1 \quad \text{当} \quad s(x)\ge \tau
$$

当 $\tau$ 从 0.5 降到 0.3 时，更多样本会被判为正类。召回通常上升，因为少数类更容易被报出来；精度通常下降，因为误报也会增加。

| 方法 | 作用层 | 优点 | 风险 | 适用场景 |
|---|---|---|---|---|
| 过采样 | 数据层 | 增加少数类出现频率 | 复制样本可能过拟合 | 少数类较少但标签可靠 |
| 欠采样 | 数据层 | 降低多数类主导性 | 丢失多数类信息 | 多数类极多且冗余 |
| 分层划分 | 数据层 | 保持训练/验证比例稳定 | 少数类太少时仍不稳定 | 所有不平衡任务 |
| 类别权重 | 损失层 | 实现简单，不改数据 | 错误标签影响被放大 | 线性模型、树模型、深度模型 |
| Focal Loss | 损失层 | 压低大量简单样本 | 多一个超参数 $\gamma$ | 极端不平衡、易样本很多 |
| 阈值移动 | 推理层 | 不重训模型即可调整目标 | 误报和召回需要权衡 | 部署前按业务目标调参 |

三种机制可以组合使用。常见路径是：先用分层划分保证验证可靠，再用类别权重训练模型，最后在验证集上搜索阈值。如果少数类极少且多数类简单样本很多，再考虑 Focal Loss 或轻度过采样。

## 代码实现

最小可运行方案是：先分层划分训练集和验证集，只在训练集上计算类别权重；训练时让模型更重视少数类；验证时不要只看 accuracy，而要看 F1、少数类召回和分类报告；部署前在验证集上搜索阈值。

下面代码使用 `scikit-learn` 的 `LogisticRegression`。为了让例子可直接运行，文本很短，重点是流程。

```python
from sklearn.model_selection import train_test_split
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import Pipeline
from sklearn.utils.class_weight import compute_class_weight
from sklearn.metrics import f1_score, recall_score, classification_report
import numpy as np

texts = [
    "物流 查询 包裹 到哪里", "物流 快递 没更新", "退款 申请 订单", "退款 什么时候 到账",
    "修改 地址", "查询 订单", "密码 重置", "优惠券 使用", "发票 开具", "退货 流程",
] * 10 + [
    "账号 被盗 异地 登录", "安全 事件 账户 风险", "账号 盗用 资金 风险", "异常 登录 安全 预警",
    "账号 被盗 无法 登录"
]

# 0 表示普通工单，1 表示高风险工单
y = np.array([0] * 100 + [1] * 5)

X_train, X_val, y_train, y_val = train_test_split(
    texts,
    y,
    test_size=0.3,
    random_state=42,
    stratify=y
)

classes = np.array([0, 1])
weights = compute_class_weight(
    class_weight="balanced",
    classes=classes,
    y=y_train
)
class_weight = dict(zip(classes, weights))

model = Pipeline([
    ("tfidf", TfidfVectorizer()),
    ("clf", LogisticRegression(class_weight=class_weight, max_iter=1000))
])

model.fit(X_train, y_train)

proba = model.predict_proba(X_val)[:, 1]

best_threshold = 0.5
best_f1 = -1

for threshold in np.linspace(0.1, 0.9, 17):
    pred = (proba >= threshold).astype(int)
    score = f1_score(y_val, pred, pos_label=1, zero_division=0)
    if score > best_f1:
        best_f1 = score
        best_threshold = threshold

final_pred = (proba >= best_threshold).astype(int)

minority_recall = recall_score(y_val, final_pred, pos_label=1, zero_division=0)

print("class_weight:", class_weight)
print("best_threshold:", best_threshold)
print("best_f1:", best_f1)
print("minority_recall:", minority_recall)
print(classification_report(y_val, final_pred, zero_division=0))

assert set(class_weight.keys()) == {0, 1}
assert class_weight[1] > class_weight[0]
assert 0.1 <= best_threshold <= 0.9
assert len(final_pred) == len(y_val)
```

这段代码里有三个关键点。

第一，先 `train_test_split(..., stratify=y)`，再计算类别权重。分层划分是让训练集和验证集尽量保持原始类别比例。不能先对全量数据过采样再切分，否则复制出来的少数类样本可能同时出现在训练集和验证集。

第二，`compute_class_weight(class_weight="balanced")` 只使用训练集标签 `y_train`。验证集代表“没见过的数据”，不应该参与训练权重计算。

第三，阈值搜索在验证集上做。默认 0.5 不是业务真理。如果高风险工单漏报代价高，可以选择让少数类 recall 更高的阈值，但要接受更多误报。

Focal Loss 在深度学习框架中实现更常见。训练思路是：先算普通交叉熵，再乘上 $(1-p_t)^\gamma$，让已经容易判对的样本贡献变小。伪代码如下：

```python
# logits: 模型输出
# target: 真实类别
# ce: 每个样本的交叉熵
# pt: 模型给真实类别的概率
# alpha: 类别权重
# gamma: 聚焦系数

focal_loss = alpha[target] * (1 - pt) ** gamma * ce
loss = focal_loss.mean()
```

新手阶段不必一开始就写 Focal Loss。优先掌握 `class_weight + 验证集阈值搜索 + 正确指标`，通常已经能解决很多文本分类不平衡问题。

## 工程权衡与常见坑

工程上没有永远最优的不平衡方案，只有和业务目标匹配的方案。常见失败原因通常不是算法不够强，而是数据泄漏、过采样过头、指标选错、阈值没调。

最严重的坑是先对全量数据过采样，再划分训练集和验证集。过采样常会复制少数类样本。如果一个“账号被盗”的样本被复制出多份，然后一份进入训练集，一份进入验证集，模型等于在训练时已经见过验证样本。这就是数据泄漏。数据泄漏指验证或测试信息提前进入训练过程，会导致离线指标虚高，线上效果下降。

不要这样做：

1. 收集全量数据。
2. 对全量数据复制少数类样本。
3. 再切分训练集和验证集。
4. 在验证集上看到很高的 F1。
5. 上线后发现少数类召回明显下降。

正确做法是：

1. 先分层切分训练集、验证集、测试集。
2. 只在训练集内部重采样或计算类别权重。
3. 在验证集上调阈值和选模型。
4. 最后只用测试集做一次最终评估。

| 常见坑 | 后果 | 规避方式 |
|---|---|---|
| 先重采样后划分 | 验证集泄漏，指标虚高 | 先划分，再只处理训练集 |
| 少数类过采样过度 | 模型记住重复样本 | 轻度过采样、正则化、早停 |
| 多数类欠采样过多 | 丢失有效语言模式 | 控制采样比例，保留足够多数类 |
| 阈值固定 0.5 | 召回和误报不符合业务目标 | 在验证集上按目标调阈值 |
| 只看 accuracy | 少数类完全漏掉也可能高分 | 看 macro-F1、recall、PR-AUC |
| 标签噪声未清洗 | 错误样本被加权放大 | 先抽检少数类标签 |

指标选择也要和任务目标一致。

| 指标 | 白话解释 | 适用边界 |
|---|---|---|
| accuracy | 总体预测对的比例 | 类别均衡、错分代价接近时可用 |
| macro-F1 | 每个类别 F1 平均，不按样本数加权 | 关注各类都不能太差 |
| recall | 真实正类中被找出来的比例 | 漏报代价高的任务 |
| PR-AUC | precision-recall 曲线下面积 | 极端不平衡且关注正类 |
| ROC-AUC | 正负样本排序能力 | 看整体区分能力，但极端不平衡时要结合 PR-AUC |

真实工程里还要看人工处理成本。比如安全事件预警中，把阈值调低可以提高召回，但会产生更多误报。如果误报全部进入人工审核，就需要确认审核团队能否承受。否则模型指标变好了，业务流程反而被压垮。

## 替代方案与适用边界

不同方法解决的是不同层面的问题：有的方法改数据分布，有的方法改优化目标，有的方法改决策规则。选择方案要看少数类是否稀缺、是否可接受漏报、是否能接受更多误报、是否需要可解释性。

如果任务是垃圾邮件过滤，误报和漏报的代价都相对可控，可以先试类别权重和阈值调整。如果任务是安全事件预警，漏报代价更高，应优先保证少数类召回，再接受一定误报，并配合人工复核或二阶段模型。

| 方案 | 解决什么问题 | 代价 | 优先使用场景 | 不适用场景 |
|---|---|---|---|---|
| 纯重采样 | 改变训练数据比例 | 可能过拟合或丢信息 | 数据量较小、类别差异明显 | 标签噪声大、少数类极少 |
| 类别权重 | 改变损失贡献 | 可能放大错误标签 | 大多数传统分类器和深度模型 | 类别边界本身不清楚 |
| Focal Loss | 压低简单样本影响 | 需要调 $\gamma$ 和 $\alpha$ | 极端不平衡、易样本很多 | 简单线性模型流程 |
| 阈值调整 | 改变最终决策规则 | 误报和漏报此消彼长 | 已有概率分数，业务目标明确 | 模型分数未校准且不稳定 |
| 两阶段方案 | 先高召回，再精排 | 系统复杂度更高 | 安全、风控、审核 | 小项目或低风险任务 |
| 数据增强 | 扩充少数类表达 | 可能生成低质量样本 | 少数类语义明确、可人工审核 | 高精度领域且难以验证生成质量 |

决策边界可以这样看：

| 条件 | 推荐方向 | 原因 |
|---|---|---|
| 少数类几十到几百条 | 类别权重 + 阈值调优 | 简单稳定，改动小 |
| 少数类只有几条 | 补数据、合并类别、规则兜底 | 模型很难可靠泛化 |
| 多数类极多且重复 | 轻度欠采样 | 降低训练成本 |
| 漏报代价极高 | 降低阈值、两阶段召回 | 先保证发现问题 |
| 误报代价极高 | 提高阈值、加强校准 | 避免打扰正常用户 |
| 需要强可解释性 | 线性模型 + 类别权重 | 便于检查特征和阈值 |

每种方案的边界要明确：

纯重采样适合快速建立基线，但不要让复制样本污染验证集。类别权重适合作为默认起点，因为它不改变样本本身。Focal Loss 适合多数类简单样本特别多的场景，但不应在没有验证指标的情况下盲目使用。阈值调整适合部署前按业务目标做最后一层决策，但不能弥补模型完全学不到少数类的问题。两阶段方案适合高风险系统，例如先用宽松阈值召回疑似安全事件，再用更强模型或规则精排。数据增强适合少数类表达可以被可靠扩展的任务，但生成样本必须抽检，否则会把噪声写进训练集。
