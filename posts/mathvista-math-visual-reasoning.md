## 核心结论

MathVista 是一个面向视觉语言模型的数学视觉推理基准。视觉语言模型，简称 VLM，指能同时处理图像和文本输入，并生成文本回答的模型。MathVista 评测的不是单纯“看见图里有什么”，而是模型能否先理解图像，再把图里的信息转成可计算结构，最后推导出标准答案。

它的核心任务可以概括为：

**MathVista = 图像理解 + 数值/逻辑推导 + 标准答案匹配**

例如题目给一张函数图，问 `f(3)`。模型不能只回答“图里有一条直线”或“看起来是 3 附近”。它要先找到横轴上 `x=3` 的位置，再根据图上的函数关系读出或算出对应的 `y` 值。如果图像表示的是 $y=2x+1$，那么：

\[
f(3)=2\times 3+1=7
\]

答案应是 `7`。

MathVista 的价值在于它把“看懂”和“算对”区分开。一个模型可能能识别坐标轴、图例、表格文字，却仍然在斜率判断、几何关系、单位换算或多步推导上出错。论文报告中，GPT-4V 在该任务上的总体准确率为 49.9%，仍低于人类水平 10.4 个百分点，说明视觉数学推理仍是当前 VLM 的短板。

| 任务类型 | 输入 | 目标 | 是否要求数学推理 |
|---|---|---|---|
| 看图识别 | 图像 | 识别物体、文字、形状 | 否 |
| 视觉问答 | 图像 + 问题 | 回答图像相关问题 | 不一定 |
| 视觉数学推理 | 图像 + 数学问题 | 从图中取信息并计算答案 | 是 |

MathVista 的评分核心可以写成：

\[
Score=\mathbf{1}[\text{模型短答案}=\text{标准答案}]
\]

这意味着长篇解释本身不直接得分。模型最终必须给出能被抽取、规范化并与标准答案匹配的结果。

## 问题定义与边界

MathVista 的输入是一张图像 $I$ 和一个问题 $q$，输出是模型回答 $r$。评测时还会把 $r$ 抽取成短答案 $a$，再和标准答案比较。数据集可表示为：

\[
D=\{(I_i,q_i,a_i)\}_{i=1}^N
\]

其中 $I_i$ 是第 $i$ 个样本的图像，$q_i$ 是问题，$a_i$ 是标准答案。MathVista 收录 6,141 道题，图像类型包括函数图、几何图、表格、论文图、坐标轴图、实验曲线等。

新手可以这样理解：如果题目只要求“图里有什么”，那还不是 MathVista 的重点；如果题目要求“看见图里的信息后算出什么”，这才是它要测的能力。

例如：

- 函数图：给出曲线，问某个 $x$ 对应的 $y$。
- 几何图：给出三角形角度和边长，问未知角或面积。
- 表格：给出多行数据，问满足条件的最大值、平均值或差值。
- 论文图：给出实验曲线，问哪个方法提升最多。
- 坐标轴图：给出散点或折线，问趋势、斜率或交点。
- 实验曲线：给出时间和值，问某个时刻的读数。

任务边界如下：

| 范畴 | 是否属于 MathVista |
|---|---|
| 仅图像描述 | 否 |
| 仅文本数学题 | 否 |
| 图像中读取信息后再推理 | 是 |
| 表格/图表/几何/函数混合推理 | 是 |

它不等同于纯文本数学题。纯文本数学题只需要处理文字，比如“已知 $y=2x+1$，求 $x=3$ 时的 $y$”。MathVista 则可能只给你一张函数图，模型必须先从图像中读出关系。

它也不等同于一般图像描述。图像描述可能回答“图中有一个坐标系和一条直线”，但 MathVista 关心的是“这条直线在 $x=3$ 时的函数值是多少”。

一个真实工程例子是科研助手读取论文图表。用户问：“图 3 中，当训练步数为 10k 时，蓝色曲线的准确率是多少？”这不是简单 OCR，因为图中可能没有直接写出答案；也不是普通问答，因为模型需要读坐标、识别曲线、估计数值并按要求输出。

## 核心机制与推导

MathVista 的评测链路可以拆成三步：模型生成回答、抽取短答案、规范化比对。流程是：

`图像+问题 -> 模型生成 -> 答案抽取 -> 规范化 -> 正误判定 -> 汇总分数`

设模型为 $f_\theta$，其中 $\theta$ 表示模型参数。模型对第 $i$ 个样本生成回答：

\[
r_i=f_\theta(I_i,q_i)
\]

然后答案抽取器从长回答中取出短答案：

\[
\hat a_i=\mathrm{Extract}(r_i)
\]

最后计算准确率：

\[
Acc=\frac{1}{N}\sum_{i=1}^N \mathbf{1}\big[\mathrm{norm}(\hat a_i)=\mathrm{norm}(a_i)\big]
\]

这里的 $\mathrm{norm}$ 是归一化函数，意思是把不同格式统一成可比较形式。例如 `7.0` 和 `7` 可以归一化为同一个数值；`3 cm` 和 `3` 是否相同，则要看题目是否要求保留单位。

玩具例子：图像中有一条直线，关系是 $y=2x+1$，问题问 `f(3)`。正确推理过程是：

1. 从问题中确定要找的是 $x=3$。
2. 从图中确认函数关系或读出对应点。
3. 代入公式：$2\times3+1=7$。
4. 输出短答案：`7`。

如果模型把斜率看成 1，就会算成 $3+1=4$。如果模型把横轴读错，把 `x=3` 看成 `x=2`，就会算成 $2\times2+1=5$。这说明错误会沿着链路传播：图像读错，后续推理即使形式正确，结果也会错。

| 环节 | 常见错误 | 结果 |
|---|---|---|
| 读取图像 | 坐标轴/图例看错 | 后续推导基于错误输入 |
| 推理计算 | 代数或几何步骤错 | 得到错误数值 |
| 答案抽取 | 长答案没抽成短答案 | 可能判分失败 |
| 归一化 | 单位/格式不一致 | 本来对也可能判错 |

这类评测特别强调最终答案，而不是解释是否流畅。模型可以写出一段看似合理的推理，但只要最终短答案和标准答案不一致，就会被判错。反过来，如果模型解释很短，但答案正确，评分通常会认为它通过。

这也是 MathVista 对 VLM 更严格的地方。它要求模型同时稳定完成四类能力：视觉定位、符号理解、数值计算、格式输出。任意一个环节薄弱，都会影响最终准确率。

## 代码实现

实现一个最小 MathVista 风格评测器，重点不只是调用模型接口，而是把“长回答”变成“可评分短答案”。工程链路通常包括输入读取、推理生成、答案抽取、标准化评分。

伪代码如下：

```python
for sample in dataset:
    image, question, gold = sample["image"], sample["question"], sample["answer"]
    response = vlm.generate(image, question)
    pred = extract_short_answer(response)
    pred = normalize(pred)
    gold = normalize(gold)
    correct += int(pred == gold)

acc = correct / len(dataset)
```

模块拆分可以按职责设计：

| 模块 | 职责 |
|---|---|
| 数据加载器 | 读取图像、问题、标准答案 |
| 推理模块 | 调用 VLM 生成自然语言回答 |
| 答案抽取器 | 从长回答中提取短答案 |
| 归一化模块 | 统一数字、单位、格式 |
| 评测器 | 汇总准确率与分项结果 |

下面是一个可运行的 Python 玩具实现。它不调用真实 VLM，而是模拟模型输出，重点展示“长回答 -> 短答案 -> 归一化 -> 准确率”的评测逻辑。

```python
import re
from fractions import Fraction

def extract_short_answer(response: str) -> str:
    """
    从模型长回答中抽取最后一个数值。
    真实系统会更复杂，可能需要处理选项、单位、分数、文本答案。
    """
    numbers = re.findall(r"-?\d+(?:\.\d+)?", response)
    if not numbers:
        return response.strip().lower()
    return numbers[-1]

def normalize(answer: str) -> str:
    answer = answer.strip().lower()
    answer = answer.replace("cm", "").replace("meters", "").replace("meter", "")
    answer = answer.strip()

    try:
        value = float(answer)
        if value.is_integer():
            return str(int(value))
        return str(value)
    except ValueError:
        pass

    try:
        return str(float(Fraction(answer)))
    except ValueError:
        return answer

def evaluate(dataset):
    correct = 0
    details = []

    for sample in dataset:
        response = sample["model_response"]
        pred = normalize(extract_short_answer(response))
        gold = normalize(sample["answer"])
        ok = pred == gold
        correct += int(ok)
        details.append((pred, gold, ok))

    return correct / len(dataset), details

dataset = [
    {
        "image": "toy_function.png",
        "question": "The graph shows y=2x+1. What is f(3)?",
        "answer": "7",
        "model_response": "From the graph, f(x)=2x+1. So f(3)=2*3+1=7."
    },
    {
        "image": "toy_length.png",
        "question": "What is the side length?",
        "answer": "3",
        "model_response": "The side length is 3.0 cm."
    },
]

acc, details = evaluate(dataset)

assert acc == 1.0
assert details[0] == ("7", "7", True)
assert details[1] == ("3", "3", True)

print(acc)
```

这段代码说明一个关键点：评测不是只看模型“说了什么”，而是要把自然语言回答压缩成标准格式。真实 MathVista 还会面对更多复杂情况，例如选择题答案、带单位数值、近似值、几何符号、表格单元格、文本类别等。

真实工程例子中，假设一个数据分析助手要读取业务仪表盘截图。用户问：“2025 年 Q3 的转化率比 Q2 高多少个百分点？”模型需要先在图中找到 Q2 和 Q3 的数值，再做差值计算，最后输出类似 `2.4` 或 `2.4%` 的短答案。如果它只描述“Q3 比 Q2 高”，对业务问答有帮助，但对 MathVista 式评测来说还不够。

## 工程权衡与常见坑

MathVista 暴露的主要问题不是模型不会生成文字，而是模型容易看错、算错或输出格式不稳定。对初级工程师来说，要特别区分三件事：识别正确、推理正确、答案可评分。这三件事都正确，最终才可能得分。

例如题目要求读折线图某个时刻的值。不能只看曲线“差不多在 5 附近”，而要读准坐标轴刻度。如果纵轴每格是 0.5，而模型按每格 1 来读，答案就会系统性偏差。

常见坑如下：

| 坑点 | 表现 | 规避方式 |
|---|---|---|
| 细节没读准 | 坐标轴、图例、几何标记错误 | 强制定位证据到图中区域 |
| 局部对、整体错 | 中间步骤看似合理，最终答案错 | 保留逐步推导记录 |
| 单位/精度错误 | `3.0`、`3`、`3 cm` 混淆 | 统一单位与格式 |
| 过度依赖 OCR/caption | 文本识别对了但语义推理错 | 把 OCR 结果纳入校验，不直接等于答案 |

错误链条可以表示为：

| 阶段 | 示例 | 后果 |
|---|---|---|
| 图像读取错误 | 把蓝线看成红线 | 选错对象 |
| 数值定位错误 | 把 $x=3$ 读成 $x=2$ | 输入值错误 |
| 推导错误 | 把 $2x+1$ 算成 $2x-1$ | 计算结果错误 |
| 输出错误 | 回答“约等于 7 cm”，标准答案要求 `7` | 可能被格式规则判错 |

论文和榜单还显示，不同能力维度的难度并不相同。LOG 可以理解为逻辑推理题，NUM 是数值计算题，GEO 是几何题，TXT 是文本相关题。它们考察的能力不同，因此一个模型在某类题上表现好，不代表整体视觉数学能力强。

| 能力维度 | 白话解释 | 典型失败模式 |
|---|---|---|
| LOG | 根据条件做逻辑判断 | 条件组合错、比较关系错 |
| NUM | 做数值计算 | 读数不准、运算错误 |
| GEO | 理解几何图形 | 角度、平行、垂直关系识别错 |
| TXT | 读图中文字 | OCR 对但语义用错 |

工程上要权衡两类目标。第一类是评测公平性：模型输出要被统一抽取和归一化，不能因为格式差异造成大量误判。第二类是真实能力测量：不能让答案抽取器过强，以至于替模型完成了本该由模型完成的推理。

因此，比较稳妥的做法是：模型负责图像理解和推理，评测器只做格式处理。评测器可以把 `7.0` 规范成 `7`，但不应该根据模型解释重新推导答案。否则测到的就不再是模型能力，而是评测后处理能力。

## 替代方案与适用边界

MathVista 适合评测视觉数学推理，但不适合替代所有视觉任务基准。选择基准前，应先明确你要测的是“识别”“理解”还是“推理”。

| 基准/方案 | 适合测什么 | 不适合什么 |
|---|---|---|
| MathVista | 视觉数学推理 | 纯 OCR、纯分类 |
| OCR 基准 | 文字识别 | 数学推导 |
| 纯文本数学基准 | 代数/推理 | 图像理解 |
| 通用 VQA | 视觉问答 | 复杂数值推导 |

如果你要评估文档中的文字识别，OCR 基准更直接。OCR 是光学字符识别，白话说就是把图片里的文字读出来。它适合回答“这张截图里写了什么”，但不一定能测出“读完后能不能计算”。

如果你要评估纯代数能力，文本数学基准更直接。例如给出“若 $x=3$，求 $2x+1$”，这能测试代数计算，但不能测试模型是否能从图像中读出函数关系。

如果你要评估普通视觉问答，通用 VQA 更合适。例如“图中有几个人”“桌上是什么物体”，这些问题不一定涉及数学推理。

MathVista 的适用边界是：任务必须需要从图像中读取数学相关信息，并进一步做数值或逻辑推导。如果任务不需要图像中的数值推导，就不必用 MathVista。

真实工程中可以这样选择：

| 场景 | 推荐评测 |
|---|---|
| 发票、表单、合同截图识别 | OCR 或文档理解基准 |
| 只问图片中有哪些物体 | 图像分类/检测/VQA 基准 |
| 只做文本数学题 | GSM8K、MATH 等文本数学基准 |
| 读论文图表后计算差值或趋势 | MathVista 类视觉数学推理基准 |

新手可以把它理解为：不要拿识字题去测推导能力，也不要拿数学题去测图像定位能力。MathVista 的位置正好在两者交叉处，它测的是“图像里的数学问题能不能被正确解决”。
