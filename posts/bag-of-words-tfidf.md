## 核心结论

词袋模型（Bag of Words，简称 BoW，白话说就是“把一篇文本只当成一张词频统计表”）和 TF-IDF（词频-逆文档频率，白话说就是“词在当前文档里常见、但在整个语料里不那么常见时，权重更高”）解决的是“文本如何变成机器能算的数值向量”这个问题，但它们不解决“文本真正表达了什么语义”这个问题。

它们的共同特点很明确：

1. 每个词对应一个维度，文本被表示成高维稀疏向量。
2. 只保留“某个词出现了几次”或“加权后出现得有多重要”，不保留词序。
3. 不能天然识别同义词、多义词、上下文关系。
4. 适合做可解释、低成本、快速上线的基础检索或召回，不适合承担深层语义理解。

一个最直观的玩具例子是“好人”和“善人”。在 BoW 或 TF-IDF 里，这两个词只是词表中的两个不同维度。即使它们含义接近，向量空间里也没有任何自动建立的联系。模型只知道它们是两个不同字符串，不知道它们“意思差不多”。

下表对比原始词频与 TF-IDF 的能力边界：

| 维度 | 原始词频向量 | TF-IDF 加权向量 |
|---|---:|---:|
| 是否统计词出现次数 | 是 | 是 |
| 是否降低“的/是/the”这类高频词权重 | 否 | 是 |
| 是否保留词序 | 否 | 否 |
| 是否识别“好人≈善人” | 否 | 否 |
| 是否区分“银行=金融机构”与“河岸” | 否 | 否 |
| 是否适合做基础相似度计算 | 是 | 是 |

所以，TF-IDF 不是“有语义的 BoW”，它只是“更合理加权的 BoW”。也正因为这种统计表示的局限，后来才出现了 Word2Vec、GloVe，再到更强的上下文嵌入模型，用密集向量去恢复词与词之间的相似性。

## 问题定义与边界

BoW 的定义是：先建立一个词汇表，再把每篇文档映射成长度等于词汇表大小的向量，每一维记录对应词在文档中出现的次数。TF-IDF 在这个基础上继续加一层全局权重，让“在当前文档里频繁、但在整个语料里不常见”的词获得更高分。

形式上：

$$
TF(t,d)=\frac{\text{词 }t\text{ 在文档 }d\text{ 中出现次数}}{\text{文档 }d\text{ 的总词数}}
$$

$$
IDF(t)=\log\frac{N}{df(t)}
$$

其中，$N$ 是总文档数，$df(t)$ 是包含词 $t$ 的文档数。

$$
TF\text{-}IDF(t,d)=TF(t,d)\times IDF(t)
$$

它的边界也要说清楚：

| 能做什么 | 不能做什么 |
|---|---|
| 把文本转成数值向量 | 理解句法结构 |
| 计算关键词重合程度 | 理解上下文语义 |
| 做基础检索、分类特征 | 区分同义词、多义词 |
| 给出可解释的关键词权重 | 建模长距离依赖 |

“银行”是最典型的边界例子。在金融新闻里，“银行”指金融机构；在地理描述里，“河流的银行”若翻译回中文语境对应“河岸”。BoW 和 TF-IDF 都只看到同一个词项，无法根据上下文判断它到底是哪一种含义。换句话说，它们处理的是“符号出现统计”，不是“语境中的意义”。

因此，BoW/TF-IDF 更准确的定位是“统计表示方法”，不是“语义表示方法”。

---

## 核心机制与推导

先看 BoW。假设词表只有四个词：`[机器学习, 深度学习, 好人, 善人]`。两篇文档分别是：

- 文档 A：`机器学习 好人`
- 文档 B：`深度学习 善人`

那么它们的词频向量分别是：

- A: `[1, 0, 1, 0]`
- B: `[0, 1, 0, 1]`

从向量角度看，这两篇文档几乎毫无相似性。但如果站在人类理解上，“好人”和“善人”是近义词，“机器学习”和“深度学习”也强相关。BoW 无法表达这种关系，因为它把“词是否相似”这个问题完全留空了。

再看 TF-IDF。它的核心逻辑分两步：

1. TF 解决“文档长度不同”的问题。
2. IDF 解决“有些词到处都出现，所以没信息量”的问题。

一个经典玩具例子是两篇英文文档：

- $d_1$: `this is a sample document`
- $d_2$: `this is another example example example document`

设总文档数 $N=2$。

先看词 `this`：

- 在 $d_1$ 中，出现 1 次，总词数 5，所以
  $$
  TF(\text{this}, d_1)=\frac{1}{5}=0.2
  $$
- 在 $d_2$ 中，出现 1 次，总词数 7，所以
  $$
  TF(\text{this}, d_2)=\frac{1}{7}\approx 0.143
  $$
- 由于 `this` 在两篇文档都出现，所以
  $$
  IDF(\text{this})=\log\frac{2}{2}=0
  $$
- 因此两篇文档中它的 TF-IDF 都是 0。

再看词 `example`：

- 只在 $d_2$ 中出现 3 次，所以
  $$
  TF(\text{example}, d_2)=\frac{3}{7}\approx 0.429
  $$
- 只出现在 1 篇文档中，所以
  $$
  IDF(\text{example})=\log\frac{2}{1}\approx 0.301
  $$
- 最终
  $$
  TF\text{-}IDF(\text{example}, d_2)\approx 0.429 \times 0.301 \approx 0.129
  $$

这个过程说明：TF-IDF 强调“能区分文档的词”，抑制“大家都有的词”。

下面用表格把计算过程压缩展示：

| 词 | 文档 | 词频 count | TF | DF | IDF | TF-IDF |
|---|---|---:|---:|---:|---:|---:|
| this | d1 | 1 | 0.200 | 2 | 0.000 | 0.000 |
| this | d2 | 1 | 0.143 | 2 | 0.000 | 0.000 |
| example | d2 | 3 | 0.429 | 1 | 0.301 | 0.129 |

问题在于，这种“区分能力”仍然是统计层面的，不是语义层面的。它知道 `example` 更能区分文档，但它不知道 `example` 和 `sample` 在很多语境里接近。它也不知道“我爱你”和“你爱我”词集合完全相同，但含义方向不同。因为一旦文本进入词袋，词序就被删除了。

## 代码实现

下面给出一个可运行的极简 Python 实现。它做的事情是：分词、统计 TF、统计 DF、计算 IDF、生成稀疏 TF-IDF 向量，并用 `assert` 验证关键结果。

```python
import math
from collections import Counter

def tokenize(text):
    return text.lower().split()

def compute_tf(tokens):
    total = len(tokens)
    counts = Counter(tokens)
    return {term: count / total for term, count in counts.items()}

def compute_df(docs_tokens):
    df = Counter()
    for tokens in docs_tokens:
        for term in set(tokens):
            df[term] += 1
    return df

def compute_idf(docs_tokens):
    n_docs = len(docs_tokens)
    df = compute_df(docs_tokens)
    return {term: math.log(n_docs / freq) for term, freq in df.items()}

def compute_tfidf(docs):
    docs_tokens = [tokenize(doc) for doc in docs]
    idf = compute_idf(docs_tokens)

    vectors = []
    for tokens in docs_tokens:
        tf = compute_tf(tokens)
        vec = {term: tf[term] * idf[term] for term in tf}
        vectors.append(vec)
    return vectors, idf

docs = [
    "this is a sample document",
    "this is another example example example document"
]

vectors, idf = compute_tfidf(docs)

assert abs(idf["this"] - 0.0) < 1e-12
assert round(vectors[1]["example"], 3) == 0.297  # ln(2) * 3/7
assert vectors[0].get("example") is None

print(vectors)
```

注意，这里使用的是自然对数 $ \ln $。如果你换成常用对数 $\log_{10}$，`example` 的数值会变成约 `0.129`。工程里两种写法都能见到，重点不是具体底数，而是“文档越少出现，IDF 越大”这个方向不变。

如果要进一步落地到真实系统，还会加两个工程结构：

| 组件 | 作用 |
|---|---|
| 倒排索引 | 白话说就是“记录每个词出现在哪些文档里”，便于快速召回 |
| 稀疏矩阵 | 白话说就是“只存非零项”，避免为大量 0 浪费空间 |

真实工程例子：新闻推荐系统初级阶段，常用 `TF-IDF + 余弦相似度` 作为稀疏召回层。用户点过几篇“芯片出口管制”新闻后，系统提取高权重词，如“芯片”“出口”“管制”“AI”，再去倒排索引中召回包含这些词且权重较高的新闻，作为候选集。这个阶段不追求真正理解新闻，只追求低延迟、可解释、便于控制。

## 工程权衡与常见坑

BoW 和 TF-IDF 在教学里很简单，但在工程里问题很多，最核心的是“高维稀疏”与“语义缺失”。

| 常见坑 | 现象 | 后果 | 常见规避方式 |
|---|---|---|---|
| 维度诅咒 | 词表达到 10 万以上 | 存储大、余弦计算慢 | 限制词表、停用词过滤、只保留高频词 |
| 顺序丢失 | “我爱你”与“你爱我”向量相同 | 语义方向被抹平 | 加 n-gram，但会增大维度 |
| 多义词混淆 | “银行”在不同语境同一向量 | 相似度误判 | 上下文嵌入替代 |
| 同义词割裂 | “好人”“善人”毫无联系 | 召回不全 | 查询扩展、词向量补充 |
| 长尾词过多 | 很多词只出现一次 | 噪声大、泛化差 | 最小文档频率裁剪 |

所谓维度诅咒，白话说就是“特征维度多到离谱以后，计算和泛化都开始变差”。假设词表大小是 10 万，那么每篇文档都是 10 万维向量，虽然大多数维度为 0，但你在做余弦相似度时仍然要处理这些稀疏结构。数据一大，检索、排序、存储都会吃紧。

n-gram 是常见补救手段。比如 unigram 只看单词，bigram 还能区分“机器 学习”和“学习 机器”的局部顺序。但代价是维度进一步爆炸。原来 10 万个 unigram，加入 bigram 后可能涨到百万级，稀疏问题更严重。

因此，TF-IDF 在工程中常被放在“第一层”而不是“最后一层”。它适合做：

- 候选召回
- 关键词解释
- 小规模分类基线
- 无 GPU 的轻量文本系统

但如果直接把它当作最终语义表示，效果通常会在同义表达、多义表达、长文本理解上迅速见顶。

## 替代方案与适用边界

TF-IDF 的替代方向是密集嵌入。密集嵌入（dense embedding，白话说就是“把词或句子压缩成低维但有语义结构的连续向量”）不再要求“一词一维”，而是让模型通过上下文共现学习“哪些词语经常在相似环境里出现”。

Word2Vec 的核心思想就是：如果两个词经常出现在相似上下文里，它们的向量应该接近。这使得“好人”和“善人”即使不是同一个词，也可能在向量空间里靠近。后来的 BERT 类模型更进一步，让同一个词在不同句子里拥有不同表示，从而缓解多义词问题。

对比如下：

| 方法 | 计算成本 | 语义能力 | 可解释性 | 典型用途 |
|---|---|---|---|---|
| TF-IDF | 低 | 弱 | 强 | 检索初筛、基线分类 |
| Word2Vec/GloVe | 中 | 中 | 中 | 词相似度、传统 NLP 特征 |
| BERT 类嵌入 | 高 | 强 | 弱 | 语义检索、排序、问答 |

但替代不等于完全淘汰。TF-IDF 仍然有明确适用边界：

1. 机器资源有限，不能跑大模型。
2. 需要可解释性，想直接看到“哪些词导致召回”。
3. 语料相对规范，关键词本身就很强。
4. 系统延迟要求极高，需要毫秒级基础过滤。

一个真实组合方案是：先用 TF-IDF 从百万文档中召回 1000 个候选，再用 BERT 或 reranker 进行精排。前者负责“快而粗”，后者负责“慢而准”。这比让 BERT 直接全库检索更现实。

所以结论不是“TF-IDF 过时”，而是“TF-IDF 适合解决统计匹配，不适合独立承担语义理解”。
