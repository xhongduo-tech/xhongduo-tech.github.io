## 核心结论

数据清洗的质量规则，本质上是一组**可执行约束**。可执行约束的意思是：它不是“数据最好规范一点”这种描述，而是“订单日期不能为空”“金额必须大于 0”“同一个订单号不能重复”这种可以直接写进程序、逐条检查的条件。

规则的价值不在于“把脏数据改干净”这句话本身，而在于把“什么算脏”定义清楚，把“发现后怎么处理”流程化。一个成熟的数据清洗系统，通常至少包含三部分：

| 组成部分 | 作用 | 典型输出 |
|---|---|---|
| 质量规则 | 定义什么数据算异常 | 通过/失败 |
| 规则引擎 | 执行规则并产出告警 | 失败明细、告警事件 |
| 例外与复核流程 | 管理“这次先放过”的特殊情况 | 豁免记录、审批日志 |

常见规则类型包括：完整性、唯一性、一致性、范围、模式、跨字段依赖。

| 规则类型 | 白话解释 | 典型例子 |
|---|---|---|
| 完整性 | 该有的值不能缺 | `order_date` 不能为空 |
| 唯一性 | 不该重复的值不能重复 | `order_id` 全表唯一 |
| 一致性 | 不同字段或系统之间要对得上 | 订单总额等于明细求和 |
| 范围 | 数值必须落在合理区间 | 金额在 0 到 1e6 之间 |
| 模式 | 格式必须符合约定 | 手机号满足固定正则 |
| 跨字段依赖 | 一个字段是否合法取决于另一个字段 | `refund_time >= pay_time` |

仅靠缺失值填补不够。缺失值填补只是“修一条记录”，质量规则解决的是“定义标准、追踪异常、形成责任边界”。如果没有规则、例外机制和复核责任，团队很容易陷入两个极端：要么什么都放过，要么每天被大量误报淹没。

工程上，规则不应只看“抓到了多少脏数据”，还要看它是否值得长期运行。常用评估指标有：

$$
\text{命中率}=\frac{\text{通过记录数}}{\text{总记录数}}
$$

$$
\text{误报率}=\frac{\text{误报记录数}}{\text{告警总数}}
$$

$$
\text{人工负担}\approx \text{告警数}\times \text{平均复核时间}
$$

$$
\text{修复后增益}=\text{下游指标}_{修复后}-\text{下游指标}_{修复前}
$$

如果规则导致人工复核成本极高，而对下游报表、推荐模型、风控模型几乎没有净提升，这条规则通常就不值得保留。

## 问题定义与边界

数据清洗质量规则解决的问题，不是“把所有数据都变成真相”，而是“把可疑数据转成可追责的约束检查”。这里的“可追责”很关键，意思是任何一条异常都能回答四个问题：

1. 哪条规则发现的。
2. 为什么失败。
3. 是自动拦截、自动修复，还是进入人工复核。
4. 如果临时放行，谁批准、有效到什么时候。

对于零基础工程师，最容易误解的一点是：**规则不是万能的数据治理系统**。它更像一道自动门禁。门禁可以拦住明显异常，但它不能替代上游流程设计、主数据管理、埋点规范和业务定义统一。

下面这张表能帮助划清边界：

| 场景 | 规则能否直接解决 | 原因 |
|---|---|---|
| 订单日期为空 | 能 | 条件明确，可直接判定 |
| 订单号重复 | 能 | 唯一性可计算 |
| 金额为负数 | 能 | 范围规则可覆盖 |
| 用户年龄填了 999 | 能 | 范围异常可识别 |
| 两个系统对“已支付”定义不同 | 不能单靠规则解决 | 需要业务口径统一 |
| 上游表单设计导致字段长期缺失 | 不能根治 | 需要修改采集流程 |
| 自由文本评论是否“合理” | 不适合纯规则 | 语义开放，需模型或抽样复核 |

一个玩具例子很直观。假设你有一张账单表，要求每次导入至少 10 行。于是定义规则：

- `row_count >= 10`

某次导入只有 8 行。此时正确做法不是“随手补 2 行占位”，而是：
- 规则触发失败。
- 记录本次批次号。
- 暂停下游报表刷新。
- 由 QA 或数据工程师排查抽取任务、源表权限、调度日志。

这说明规则的目标不是让数据“看起来完整”，而是阻止错误被带进下游。

再看一个真实工程边界。比如训练推荐模型的样本表里，`user_id`、`item_id`、`label` 都存在，但某次 ETL 把时区处理错了，导致“点击发生时间”整体偏移 8 小时。单条记录不一定看起来异常，但会影响训练集切窗。此时简单的缺失值检查几乎无效，需要增加跨字段或时间一致性规则，例如：

- `event_time <= sample_extract_time`
- 每小时样本量与历史窗口相比不能偏离过大

这已经不是“补空值”的问题，而是质量约束和业务时间语义的结合。

## 核心机制与推导

规则引擎的基本工作流可以抽象为：

**输入数据 -> 执行规则 -> 产出失败记录 -> 分级处理 -> 例外/复核 -> 汇总指标**

同一条记录可能命中多条规则，所以系统通常需要定义优先级。优先级的白话意思是：当一条记录同时“主键为空”和“金额超范围”时，应该先把它归为更根本的问题，避免重复告警。

一个最小规则链可以这样设计：

| 执行顺序 | 规则 | 失败后动作 |
|---|---|---|
| 1 | 完整性检查 | 高优先级拦截 |
| 2 | 模式检查 | 中优先级告警 |
| 3 | 范围检查 | 中优先级告警 |
| 4 | 跨字段依赖 | 高优先级复核 |
| 5 | 跨表一致性 | 批次级阻断 |

为什么很多团队先做完整性，再做范围和一致性？因为前面的规则更基础。若主键缺失，后续去做跨表比对通常没有意义，反而浪费计算。

### 玩具例子

假设一张订单表有 10000 行，定义两条规则：

- R1: `total_price > 0`
- R2: `order_date is not null`

统计结果如下：

| 项目 | 数值 |
|---|---|
| 总记录数 | 10000 |
| 两条规则都通过 | 9800 |
| 至少触发一条规则失败 | 200 |
| 其中人工确认误报 | 50 |

则：

$$
\text{命中率}=\frac{9800}{10000}=98\%
$$

$$
\text{误报率}=\frac{50}{200}=25\%
$$

如果每天约有 200 条告警，平均每条复核 10 分钟，则：

$$
\text{每日人工负担}=200 \times 10\text{分钟}=2000\text{分钟}\approx 33.3\text{小时}
$$

这个数字很重要。它告诉你规则是否“理论上正确但工程上不可用”。如果一个团队每天只有 1 名数据分析师兼职处理质量问题，那么 33 小时/天 的人工负担显然不可接受。

### 真实工程例子

设想一个电商平台把订单数据同步到数仓，再从数仓抽训练样本给风控模型。这里可以定义三层规则：

| 层级 | 规则示例 | 目的 |
|---|---|---|
| 记录级 | `amount > 0`，`order_id` 非空 | 过滤明显脏数据 |
| 批次级 | 当日订单数不能低于过去 7 日均值的 70% | 发现采集/调度异常 |
| 业务级 | `refund_time >= pay_time`，退款金额不超过支付金额 | 保证业务语义正确 |

如果某天同步任务成功完成，但批次级规则发现“订单量只有历史均值的 40%”，说明系统层面可能没报错，但业务数据已经失真。规则此时起到的是“对业务正确性做最后一道守门”。

这也是为什么质量规则常常要和 SLI/SLO 一起监控。SLI 可以理解为可量化的服务指标，SLO 可以理解为团队承诺要维持的目标范围。比如：

- SLI：订单表完整性命中率
- SLO：过去 30 天内，完整性命中率不低于 99.5%

一旦连续多天低于阈值，就不再是“某次脏数据事件”，而是流程设计或上游系统质量退化。

## 代码实现

下面给出一个可运行的最小 Python 规则引擎。它只做三件事：

1. 定义规则。
2. 对每条记录执行检查。
3. 产出失败列表和简单指标。

```python
from dataclasses import dataclass
from typing import Callable, List, Dict, Any

Record = Dict[str, Any]

@dataclass
class Rule:
    rule_id: str
    description: str
    severity: str
    check: Callable[[Record], bool]

def run_rules(records: List[Record], rules: List[Rule]):
    failed = []
    passed_count = 0

    for row in records:
        row_failed = []
        for rule in rules:
            if not rule.check(row):
                row_failed.append({
                    "rule_id": rule.rule_id,
                    "severity": rule.severity,
                    "description": rule.description
                })

        if row_failed:
            failed.append({
                "record": row,
                "failures": row_failed
            })
        else:
            passed_count += 1

    total = len(records)
    alerts = len(failed)

    metrics = {
        "total_records": total,
        "passed_records": passed_count,
        "alerts": alerts,
        "hit_rate": passed_count / total if total else 0.0,
    }
    return failed, metrics

records = [
    {"order_id": "A001", "order_date": "2026-04-01", "amount": 120.0, "refund_amount": 0.0},
    {"order_id": "A002", "order_date": None, "amount": 88.0, "refund_amount": 0.0},
    {"order_id": "A003", "order_date": "2026-04-01", "amount": -5.0, "refund_amount": 0.0},
    {"order_id": "A004", "order_date": "2026-04-01", "amount": 100.0, "refund_amount": 120.0},
]

rules = [
    Rule(
        rule_id="not_null_order_date",
        description="订单日期不能为空",
        severity="high",
        check=lambda r: r.get("order_date") is not None
    ),
    Rule(
        rule_id="amount_positive",
        description="订单金额必须大于0",
        severity="high",
        check=lambda r: isinstance(r.get("amount"), (int, float)) and r["amount"] > 0
    ),
    Rule(
        rule_id="refund_not_exceed_amount",
        description="退款金额不能超过支付金额",
        severity="medium",
        check=lambda r: r.get("refund_amount", 0) <= r.get("amount", 0)
    ),
]

failed, metrics = run_rules(records, rules)

assert metrics["total_records"] == 4
assert metrics["passed_records"] == 1
assert metrics["alerts"] == 3
assert abs(metrics["hit_rate"] - 0.25) < 1e-9

# 三条失败记录分别来自：
# A002: 缺少订单日期
# A003: 金额为负数
# A004: 退款金额超过支付金额
assert failed[0]["record"]["order_id"] == "A002"
assert failed[1]["record"]["order_id"] == "A003"
assert failed[2]["record"]["order_id"] == "A004"
```

这个实现很简单，但已经体现了规则系统的核心结构。它适合作为“玩具例子”帮助理解最小闭环。

如果继续往工程版本演进，通常会增加下面几类能力：

| 能力 | 作用 |
|---|---|
| 批次元数据 | 记录数据来源、导入时间、批次号 |
| 规则优先级 | 决定先报哪类问题 |
| 失败明细表 | 把失败记录写入单独表，供复核 |
| 告警系统 | 发邮件、Slack、飞书、PagerDuty |
| 例外管理 | 某条记录可临时豁免，到期自动恢复检查 |
| 规则版本管理 | 规则变更可追踪、可回滚 |

可以把结构理解成下面这个伪流程：

```text
规则定义 -> 规则引擎 -> 失败记录表 -> 告警/工单 -> 人工复核 -> 豁免或修复
```

真实工程里，Google Cloud Dataplex 这类平台就提供了更完整的自动扫描、失败记录输出和日志集成能力。平台的价值不在于“检查逻辑更神奇”，而在于把调度、失败表、监控、权限和审计串起来，减少团队重复造轮子。

## 工程权衡与常见坑

规则系统最大的难点不是“怎么写 if”，而是“怎么把规则长期稳定地运行下去”。常见权衡主要有三组。

| 维度 | 偏左方案 | 偏右方案 | 代价 |
|---|---|---|---|
| 稳定性 vs 灵敏度 | 阈值宽松，少告警 | 阈值严格，多告警 | 严格规则更容易误报 |
| 自动 vs 人工 | 自动拦截/自动修复 | 人工复核后处理 | 自动化高但误伤风险大 |
| 单表 vs 跨表 | 实现简单、成本低 | 业务覆盖更完整 | 跨表规则维护更复杂 |

### 坑一：把缺失值填补当成质量治理

这是最常见的问题。比如用户地址为空，团队直接统一填成“未知”。这当然能让非空规则通过，但没有解决真正的问题：是采集表单没做必填校验，还是上游同步漏字段，还是中间转换逻辑丢了值？

如果只做填补，不回溯根因，那么系统会越来越依赖“修补”，最后谁也不知道数据到底可靠吗。

### 坑二：规则抖动导致告警失效

“抖动”指的是规则今天触发、明天恢复、后天再触发，团队长期看不到稳定规律。比如某字段本来每天波动 0% 到 10%，你却按非常激进的阈值去报错，结果每天都报警。久而久之，团队会把告警当背景噪声。

处理方法通常有三种：

1. 用滑动窗口做动态基线，而不是固定阈值。
2. 把告警分级，高风险才阻断，中低风险只进入日报。
3. 增加复核 SOP，保证告警有明确负责人和关闭时限。

一个简单 SOP 可以写成这样：

| 告警级别 | 触发条件 | 责任人 | 关闭时间 |
|---|---|---|---|
| P1 | 关键主键为空、跨表严重不一致 | 数据工程 oncall | 4 小时 |
| P2 | 批次数量异常、字段范围异常 | 业务分析师 + 数据工程师 | 1 个工作日 |
| P3 | 格式轻微异常、低风险模式错误 | 数据治理专员 | 3 个工作日 |

### 坑三：只看规则通过率，不看误报成本

一条规则把通过率从 99.8% 提高到 99.95%，听起来很好，但如果它把人工复核量从每天 20 条提升到 600 条，而且修复后对模型 AUC、报表准确率、库存决策没有明显增益，这条规则就是“统计上显得严格，工程上却不划算”。

### 坑四：没有例外机制

现实系统总会有特殊情况。比如某个历史渠道订单编号格式和新规则不一致，但这批历史订单又必须保留。若系统没有例外机制，团队只能：
- 手工改数据
- 临时改规则
- 永久关闭告警

这三种做法都不理想。更合理的是记录一条带有效期的豁免：

- 豁免对象：渠道 X 的历史订单
- 规则 ID：`order_id_pattern_v2`
- 失效时间：2026-05-01
- 审批人：某负责人

这样例外本身也成为可审计的数据。

## 替代方案与适用边界

质量规则并不是唯一方案。它适合**业务约束明确、可解释性要求高、失败后需要明确责任归属**的场景。比如订单、账单、交易、库存、设备状态、埋点字段等。

当场景更开放时，替代方案往往更合适。

| 方法 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| 显式质量规则 | 订单、账单、主数据 | 可解释、可追责 | 规则维护成本会上升 |
| 数据契约 | 上下游接口稳定的团队协作 | 问题更早暴露 | 需要组织协同 |
| 异常检测模型 | 高维、动态、难写死规则的数据 | 能发现未知异常 | 可解释性较弱 |
| 质量评分卡 | 需要高层看总体健康度 | 便于汇总展示 | 不能替代具体修复 |
| 抽样复核 | 半结构化、非结构化数据 | 成本可控 | 覆盖不全 |

一个对比很典型：

- 金融流水数据适合规则：`amount > 0`、账户号格式合规、币种在白名单内、交易时间不能晚于入库时间。
- 用户评论文本不适合堆太多规则。你很难为“这条评论是否语义异常”写出稳定规则。更合理的方案是先用模型打质量分，再抽样复核。

所以可以把适用边界总结为：

| 数据类型 | 优先方案 |
|---|---|
| 结构化、字段含义稳定 | 显式规则优先 |
| 半结构化、字段频繁变化 | 规则 + 评分卡 |
| 非结构化文本/图片/日志 | 模型评分 + 抽样复核 |
| 高风险核心数据 | 强规则 + 阻断机制 |
| 低风险探索型数据 | 轻规则 + 观察监控 |

对初级工程师来说，最实用的判断标准是：**如果你能把“错误”写成明确的布尔表达式，就优先考虑规则；如果你只能说“看起来怪”，就要考虑统计方法、模型方法或人工采样。**
