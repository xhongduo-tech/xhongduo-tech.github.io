## 核心结论

指令微调和预训练不是一回事，它们解决的是两个不同层面的问题。

预训练是在海量文本上做“下一 token 预测”。token 可以理解为模型处理文本时使用的最小片段，可能是一个字、一个词，或一个子词。这个阶段主要让模型学会语言规律、常识关系、事实共现和一般表达能力。形式化地说，预训练优化的是整段文本的自回归概率：

$$
\theta_0=\arg\min_\theta \mathbb{E}_{w_{1:T}\sim \mathcal D_{pre}}\Big[-\sum_{t=1}^{T}\log p_\theta(w_t\mid w_{<t})\Big]
$$

指令微调是在预训练之后，再用“指令 $x$ - 答案 $y$”的数据训练模型，让它更稳定地把输入看成任务，把输出组织成符合要求的答案：

$$
\theta^*=\arg\min_\theta \mathbb{E}_{(x,y)\sim \mathcal D_{inst}}\big[-\log p_\theta(y\mid x)\big]
$$

这意味着：指令微调主要激发和重排已有能力，而不是大规模注入新知识。LIMA 的核心启发也是这一点：高质量、少量的指令数据之所以有效，不是因为它们带来了很多新事实，而是因为它们把“怎么答”这件事训练得更稳定。

可以把两者的差异先压缩成一张表：

| 维度 | 预训练 | 指令微调 |
| --- | --- | --- |
| 主要目标 | 学语言分布 | 学任务映射 |
| 优化对象 | $p(w_t \mid w_{<t})$ | $p(y \mid x)$ |
| 主要获得 | 知识、语感、表达能力 | 指令遵循、格式控制、风格对齐 |
| 常见收益 | “会说” | “按要求说” |
| 对新知识的作用 | 主要来源 | 通常不是主要来源 |

玩具例子很直观。一个基础模型可能已经知道“法国首都是巴黎”，但当输入是“请用一句话回答：法国首都是哪里？”时，它未必总能只输出一句话。它可能回答成一段百科式说明。经过指令微调后，它更可能稳定输出“法国首都是巴黎。”知识没明显变多，但输出形式变得可控。

## 问题定义与边界

先把四个容易混在一起的概念分开。

`input` 是输入内容，即任务上下文，比如一段文本、一张表、一个用户问题。  
`instruction` 是对任务的明确要求，比如“请总结”“请用一句话回答”“请输出 JSON”。  
`output` 是模型最终要生成的答案。  
`knowledge` 是模型内部已经学到的事实、模式和关系，也就是它“脑子里原本有什么”。

预训练主要解决“模型能不能形成 knowledge，并据此生成自然语言”。指令微调主要解决“模型拿到 instruction 后，能不能把 knowledge 以正确形式输出出来”。

这一点的工程边界非常重要。很多团队第一次做微调时，会把“知识更新”和“行为对齐”混成一个问题。结果就是：明明要解决的是“最新价格、最新规则、最新政策”，却试图靠指令微调一次性记住。这通常不是最优路径。

下面这张表给出边界：

| 任务类型 | 是否适合主要靠指令微调 | 原因 | 更常见方案 |
| --- | --- | --- | --- |
| 回答风格统一 | 是 | 本质是输出行为约束 | 指令微调 |
| 固定格式输出 | 是 | 本质是模板稳定化 | 指令微调 |
| 多轮交互礼貌性 | 是 | 本质是对齐偏好 | 指令微调 |
| 注入新事实 | 否 | 数据量小且易遗忘 | RAG、继续预训练 |
| 实时信息回答 | 否 | 事实会变 | 检索、工具调用 |
| 外部系统结果整合 | 否 | 需要外部真值 | 工具调用 |

真实工程例子更能说明边界。假设你在做一个客服助手，想要它每次都输出四段结构：

1. 问题摘要  
2. 处理建议  
3. 是否需要人工升级  
4. 风险提示

这很适合指令微调，因为目标是稳定格式和任务响应习惯。但如果第二天产品价格、退款规则、活动政策变了，就不该指望“再微调一轮”来同步事实。更稳妥的办法是让模型先检索最新规则，再组织回答。

所以边界可以压缩成一句话：指令微调更擅长“把已有知识用成产品形态”，不擅长“持续维护一份最新知识库”。

## 核心机制与推导

理解两者关系，关键是看训练目标改变了什么。

预训练阶段，模型面对的是自然文本序列 $w_{1:T}$。它的任务是根据前文预测下一个 token。因为训练语料足够大，模型会在统计意义上学到哪些实体经常一起出现、哪些句式常见、哪些概念有因果或定义关系。这里的“学到”不是显式存表，而是参数中形成分布式表征。分布式表征可以理解为：知识不是存在一个固定单元里，而是分散编码在大量参数的联合模式中。

指令微调阶段，训练样本变成了条件生成：给定输入 $x$，输出目标答案 $y$。这时优化重点不再是“自然语料里下一个 token 最可能是什么”，而是“在这个任务条件下，什么答案最符合示例”。

再看一遍这两个目标：

$$
\theta_0=\arg\min_\theta \mathbb{E}_{w_{1:T}\sim \mathcal D_{pre}}\Big[-\sum_{t=1}^{T}\log p_\theta(w_t\mid w_{<t})\Big]
$$

$$
\theta^*=\arg\min_\theta \mathbb{E}_{(x,y)\sim \mathcal D_{inst}}\big[-\log p_\theta(y\mid x)\big]
$$

从形式上看，两者都还是语言建模，但条件不同。预训练更接近“开放世界续写”，指令微调更接近“条件约束下作答”。

为什么这会主要改变行为，而不是大规模注入新事实？原因有三个。

第一，指令数据规模通常远小于预训练语料。预训练可能是万亿级 token，指令微调常见是万级到百万级样本。两者信息量不在一个数量级上。

第二，指令数据高度结构化。它反复教模型“遇到这种提示，要用这种方式答”。这更像在重塑输出偏好，而不是重建知识底座。

第三，损失主要集中在答案部分。模型反复被奖励的是“这种题型要这样组织答案”，不是“把世界知识重新学一遍”。

可以用一个玩具概率例子来理解。假设基础模型已经知道巴黎和法国的关系。面对输入：

“请用一句话回答：法国首都是哪里？”

它可能有两种高概率输出：

- 输出 A：“法国首都是巴黎。”
- 输出 B：“法国的首都是巴黎。巴黎也是法国最大城市之一，位于欧洲西部……”

如果预训练阶段，长解释在语料里更常见，那么 B 未必比 A 概率低。指令微调提供大量“短答”“一句话回答”“只输出结论”的样本后，模型学到的是：在这种输入模式下，A 的条件概率应该更高。知识本身没变，条件分布的偏好变了。

这也是 LIMA 结论成立的机制基础：如果预训练已经让模型“知道很多”，那么少量高质量指令样本只需把“如何把知道的内容按要求说出来”校准好，就可能显著改善使用体验。

## 代码实现

最小实现上，指令微调本质仍然是条件语言建模。只是训练时通常把“提示部分”和“答案部分”拼成一个序列，并且只对答案 token 计算损失。mask 可以理解为“哪些位置参与损失计算的开关”。

先看一个数据样例：

| 原始需求 | 指令格式 | 目标输出 |
| --- | --- | --- |
| 问法国首都 | `请用一句话回答：法国首都是哪里？` | `法国首都是巴黎。` |
| 总结工单 | `请按四段格式总结以下工单：...` | `问题摘要：...` |
| 输出 JSON | `提取字段并输出 JSON：...` | `{"name":"...","price":"..."}` |

下面给一个可运行的 Python 玩具实现。它不训练真实大模型，只演示“只对答案部分算损失”这个核心思想：

```python
import math

def nll(probabilities, targets):
    loss = 0.0
    for p, t in zip(probabilities, targets):
        loss -= math.log(p[t])
    return loss / len(targets)

# 两个位置的预测概率，分别对应答案中的两个 token
# 这里把 0/1 看成两个候选 token 的索引
probs_before_tuning = [
    [0.4, 0.6],  # 第一个答案 token 的分布
    [0.3, 0.7],  # 第二个答案 token 的分布
]

probs_after_tuning = [
    [0.8, 0.2],
    [0.9, 0.1],
]

# 目标答案 token 序列，假设都应选择索引 0
targets = [0, 0]

loss_before = nll(probs_before_tuning, targets)
loss_after = nll(probs_after_tuning, targets)

assert loss_after < loss_before
assert round(loss_before, 4) == round(( -math.log(0.4) - math.log(0.3) ) / 2, 4)

print("loss_before =", round(loss_before, 4))
print("loss_after  =", round(loss_after, 4))
```

上面的 `assert` 验证了：如果微调后目标答案 token 的概率更高，损失就更低。真实大模型的训练当然复杂得多，但核心结构就是这样。

再看一个更接近真实训练流程的伪代码：

```python
for batch in dataloader:
    prompt = batch["instruction"] + batch.get("input", "")
    target = batch["output"]

    # 将 prompt 和 target 拼接成一个序列
    tokens = tokenize(prompt + target)

    # labels 中，prompt 对应位置设为 ignore_index
    labels = build_labels(prompt, target, ignore_index=-100)

    logits = model(tokens)
    loss = cross_entropy(logits[:-1], labels[1:])  # 只对答案部分生效

    loss.backward()
    optimizer.step()
    optimizer.zero_grad()
```

这里最重要的不是 API 细节，而是训练语义：

1. 模型看到了完整提示。
2. 模型生成整段文本的 logits。
3. 但损失只在答案区域计算。
4. 所以它学到的是“在这个提示条件下，答案应该怎么写”。

这也解释了为什么高质量样本比海量杂乱样本更重要。如果样本中的提示模板、答案长度、格式要求都很一致，模型就更容易学到稳定的条件映射。

## 工程权衡与常见坑

指令微调的收益通常是“输出更稳”，代价则是“如果数据设计不好，模型会被错误地稳定化”。

常见坑可以直接列出来：

| 常见坑 | 具体表现 | 本质问题 | 规避策略 |
| --- | --- | --- | --- |
| 把微调当知识注入 | 以为加几千条样本就能记住新政策 | 错把行为训练当知识更新 | 新知识优先用 RAG、工具、继续预训练 |
| 只追求数据量 | 回答变得套话、空洞、模板化 | 低质量监督放大错误偏好 | 少量高质量、去重、一致性优先 |
| 训练提示和评测提示不一致 | 训练时表现好，上线后不稳定 | 条件分布发生漂移 | 统一模板和评测协议 |
| 过度微调导致遗忘 | 通用问答能力下降 | 新分布覆盖旧能力 | 降低学习率、混入通用数据、监控遗忘 |

第一个坑最常见。很多团队上线后发现模型会“很像产品经理想要的语气”，但答错最新政策，于是误以为“微调不行”。其实不是微调不行，而是目标错配。你想解决的是事实更新，不是输出对齐。

第二个坑是数据污染。比如把严肃问答、营销文案、客服安抚话术都混在一起训练，而且没有统一模板。结果模型可能在技术问答里也突然出现“感谢您的支持，我们将持续为您服务”这种不合场景的句子。模型不是变笨了，而是学到了混乱的任务边界。

第三个坑是训练和使用场景不一致。你训练时全是“请按以下 JSON schema 输出”，上线时用户却只会输入自然语言。模型并不会自动理解你隐含的协议。训练格式如果和真实请求模式差太远，效果就会明显打折。

第四个坑是遗忘。遗忘可以理解为模型在适应新任务时，原有通用能力被覆盖。尤其在数据少、学习率高、训练轮数多时更明显。工程上通常会同时看两类指标：任务内指标和通用指标。前者看有没有对齐成功，后者看有没有把底座能力训坏。

经验上，指令微调最值得投入的场景，不是“让模型突然知道一个它原本完全不知道的世界”，而是“让模型把原本会的东西，以产品需要的方式稳定输出”。

## 替代方案与适用边界

当你知道指令微调主要解决“行为对齐”，方案选择就会更清晰。

| 方案 | 主要目标 | 是否引入新事实 | 是否依赖外部系统 | 成本特征 | 适用场景 |
| --- | --- | --- | --- | --- | --- |
| 指令微调 | 调整回答方式与任务映射 | 弱 | 否 | 训练成本中等 | 风格、格式、流程对齐 |
| 继续预训练 | 让模型更懂某类语料分布 | 强 | 否 | 数据和算力成本高 | 领域语言、术语、知识底座增强 |
| RAG | 检索外部知识再回答 | 强 | 是 | 系统复杂度中等 | 最新知识、文档问答 |
| 工具调用 | 调外部 API 或程序获取真值 | 强 | 是 | 工程集成成本高 | 查价格、查库存、执行操作 |

四种方案里，最容易被误用的是指令微调和继续预训练。

继续预训练更适合“让模型更懂这个领域本身”。例如法律、医学、芯片文档、公司内部技术语料。这里目标不是单纯换一种答题格式，而是让模型内部对术语、上下文、搭配方式都更熟。它改变的是底层分布感知。

指令微调更适合“让模型把能力用得更可控”。例如统一输出为表格、固定风险提示语、限制回答长度、形成客服处理模板。

RAG 适合“知识在外部，而且会变”。检索增强生成中的检索，就是先从资料库里找相关文档，再把文档送给模型回答。它的优势是真实、可更新、可追溯。

工具调用适合“答案不是记忆问题，而是执行问题”。比如查数据库余额、调接口取天气、调用规则引擎判断是否可退款。

真实工程里，一个成熟系统通常不是四选一，而是组合使用。比如客服系统常见组合是：

1. 用继续预训练增强领域术语理解。
2. 用指令微调固定回复结构。
3. 用 RAG 注入最新规则。
4. 用工具调用查询订单状态。

边界总结可以很直接：

- 想让模型“更像你的产品说话”，优先看指令微调。
- 想让模型“更懂你的领域文本”，优先看继续预训练。
- 想让模型“知道最新资料”，优先看 RAG。
- 想让模型“拿到外部真值并执行动作”，优先看工具调用。

不要把所有问题都丢给微调。微调是对齐工具，不是万能知识注入器。
