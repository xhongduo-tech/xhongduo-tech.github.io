## 核心结论

模型合并（Model Merging）是把多个**同源微调模型**融合成一个新模型的方法。这里的“同源”是指它们来自同一个基础模型 `base model`，也就是同一份预训练权重出发得到的不同版本。它的核心不是再训练一遍，而是直接组合这些模型的参数改动。

如果把基础模型记为 $W_0$，把第 $i$ 个微调模型记为 $W_i$，那么每个微调模型都可以看成“在基础模型上打了一层补丁”，这个补丁叫任务向量（task vector，白话解释：某个任务带来的参数改变量）：

$$
\Delta_i = W_i - W_0
$$

模型合并做的事，本质上就是按某种规则组合多个 $\Delta_i$，再加回 $W_0$。因此它解决的是一个工程问题：如何用**一份部署权重**尽量保留多个任务能力。

最常见的四类方法可以放到同一张表里理解：

| 方法 | 合并对象 | 直觉 | 优点 | 风险 |
|---|---|---|---|---|
| 线性加权 | 完整参数 $W_i$ | 直接平均多个模型 | 最简单 | 容易互相抵消 |
| Task Arithmetic | 任务向量 $\Delta_i$ | 把多个补丁叠加 | 语义清晰 | 任务冲突时退化明显 |
| DARE | 稀疏化后的 $\Delta_i$ | 随机丢掉部分更新再缩放 | 可降噪 | 超参数敏感 |
| TIES | 过滤和投票后的 $\Delta_i$ | 先裁小值，再按符号投票 | 更能处理冲突 | 实现更复杂 |

一个新手能直接理解的类比是：`base model` 像底稿，数学微调、代码微调、对话微调像三份不同颜色的批注。模型合并不是重新写一本书，而是把这些批注按规则合成一版新稿。

真实工程里，这种需求很常见。比如同一个 `Qwen` 或 `Llama` 基座上分别训出“数学”“代码”“指令跟随”三个模型，部署时如果分别维护三套权重，显存、版本管理和服务编排都会复杂很多。合并的目标就是把它们压回一个模型里，减少维护成本。

## 问题定义与边界

模型合并讨论的对象通常不是“任意两个模型”，而是“同一个基础模型派生出来的多个微调模型”。这个边界非常重要，因为合并依赖参数的逐元素对齐。参数形状、层结构、词表映射、位置编码只要有一个不兼容，结果就可能失真。

统一记号如下：

- $W_0$：基础模型参数
- $W_i$：第 $i$ 个微调模型参数
- $\Delta_i = W_i - W_0$：第 $i$ 个任务向量
- $W_{\text{merge}}$：合并后模型参数

这里的“参数”可以是全量权重，也可以是 LoRA 这类参数高效微调模块。LoRA（Low-Rank Adaptation，白话解释：只训练少量低秩矩阵而不改全量模型）场景尤其适合做合并，因为补丁天然独立、体积也更小。

是否适合直接合并，可以先看下面这张判断表：

| 检查项 | 适合合并 | 不适合合并 |
|---|---|---|
| 架构是否相同 | 同一架构 | 不同架构 |
| base 是否相同 | 同一个 base checkpoint | 不同来源 base |
| tokenizer 是否相同 | 词表和分词规则一致 | 词表或分词规则不同 |
| 位置编码方案是否一致 | 同为 RoPE 或同一实现 | 编码机制不同 |
| 参数形状是否一致 | 层名和 shape 对齐 | 层缺失或 shape 不匹配 |
| 量化状态是否一致 | 都是 FP16/BF16，或都已反量化 | 一部分量化、一部分非量化直接混 |

玩具例子可以帮助建立边界感。假设你有两个模型：

- 模型 A：在 `Qwen2.5-7B` 上做数学微调
- 模型 B：在同一个 `Qwen2.5-7B` 上做代码微调

这通常可以合并，因为它们的参数坐标系是一致的。所谓“坐标系一致”，白话讲就是每个权重位置表达的是同一种含义。

但如果模型 A 来自 `Qwen2.5-7B`，模型 B 来自 `Llama-3-8B`，即使二者参数量接近，也不应该直接合并。因为“第 100 层第 20 个通道”的语义不一定对应，硬合并只是数学上可算，不代表语义上成立。

因此，模型合并的目标也要定义清楚：它不是让每个单任务都达到各自的最优，而是尽量在一个模型中保留多个能力，并把冲突控制在可接受范围内。这个目标和“训练一个绝对最强的单任务模型”不是一回事。

## 核心机制与推导

理解模型合并，最有效的方式是把所有方法都放进“如何组合 $\Delta_i$”这个统一框架。

### 1. 线性加权

线性加权直接对完整模型做平均：

$$
W_{\text{merge}} = \sum_i \alpha_i W_i,\qquad \sum_i \alpha_i = 1
$$

它最直观，也最容易实现。问题在于，这相当于把基础模型本身也重复混进去了。如果所有模型都来自同一 $W_0$，那更自然的做法其实是先减去 base，看增量怎么组合。

### 2. Task Arithmetic

Task Arithmetic 可以直译为“任务算术”。白话讲，就是先算每个任务对 base 的增量，再把这些增量叠加回去：

$$
W_{\text{merge}} = W_0 + \sum_i \lambda_i \Delta_i
$$

这里的 $\lambda_i$ 是任务权重。它控制每个任务补丁放大还是缩小。这个方法的优点是解释性强：不是在混合多个成品模型，而是在合成多个任务带来的偏移。

### 3. DARE

DARE 可以理解为“先稀疏化再重标定”的方法。它的一个常见写法是：

$$
\hat{\Delta}_i = \frac{m_i \odot \Delta_i}{1-p}
$$

其中 $m_{i,j} \sim \text{Bernoulli}(1-p)$，表示每个参数位置以概率 $p$ 被丢弃；$\odot$ 表示逐元素相乘。白话解释是：随机删掉一部分小更新或噪声更新，再把剩下的部分按期望缩放回来。

这么做的动机是，很多微调更新并不是稳定有用的“信号”，而是带有任务偶然性的“噪声”。DARE 通过随机保留一部分更新，降低不同任务之间细碎冲突的累积。

### 4. TIES

TIES 的关键是先处理冲突，再聚合。它一般包含三步：

1. 裁剪小幅更新
2. 对每个参数位置做符号投票
3. 只保留与最终方向一致的更新

这里的“符号投票”指的是看同一位置的更新多数是正还是负。白话讲，如果数学模型想把某个参数往上推，而代码模型想往下拉，TIES 不会直接平均，而是先判断哪个方向更一致，再只保留那个方向。

下面用一个最小玩具例子说明差异。设：

$$
W_0 = [10, 10]
$$

数学模型 A 为：

$$
W_A = [12, 9] \Rightarrow \Delta_A = [2, -1]
$$

代码模型 B 为：

$$
W_B = [11, 13] \Rightarrow \Delta_B = [1, 3]
$$

若做等权线性平均：

$$
W_{\text{merge}} = \frac{W_A + W_B}{2} = [11.5, 11]
$$

若做 Task Arithmetic，且 $\lambda_A=\lambda_B=1$：

$$
W_{\text{merge}} = W_0 + \Delta_A + \Delta_B = [13, 12]
$$

两者差别很大。原因是线性平均在“成品模型空间”折中，而 Task Arithmetic 在“任务增量空间”叠加。尤其第 2 维中，A 的更新是负的，B 的更新是正的，这就是典型冲突位。TIES 会先判断这类冲突，而不是无条件平均。

真实工程例子更能说明问题。假设你有三个微调模型：

- `math-lora`：擅长公式推导
- `code-lora`：擅长 Python 和 SQL
- `chat-lora`：擅长指令跟随与格式稳定

如果直接平均完整权重，常见结果是“都会一点，但都不强”。如果先转成任务向量，再用 Task Arithmetic 得到基线，再用 TIES 或 DARE 压低冲突位，通常更容易得到一个可上线的多任务版本。

## 代码实现

工程实现的重点不是训练，而是**对齐、合成、保存、验证**。最小流程通常是：

1. 载入 `base model`
2. 载入多个微调模型或 LoRA adapter
3. 逐层计算 $\Delta$
4. 按策略合并
5. 保存新权重
6. 做回归评测

下面先看一个可运行的玩具实现，它不依赖深度学习框架，只演示“任务向量相加”和“简单冲突处理”的逻辑。

```python
from typing import List

def task_arithmetic(base: List[float], deltas: List[List[float]], weights: List[float]) -> List[float]:
    assert len(deltas) == len(weights)
    assert all(len(d) == len(base) for d in deltas)
    merged = base[:]
    for i in range(len(base)):
        merged[i] += sum(weights[j] * deltas[j][i] for j in range(len(deltas)))
    return merged

def sign_vote_merge(base: List[float], deltas: List[List[float]]) -> List[float]:
    assert all(len(d) == len(base) for d in deltas)
    merged = base[:]
    for i in range(len(base)):
        vals = [d[i] for d in deltas]
        pos = sum(v for v in vals if v > 0)
        neg = sum(v for v in vals if v < 0)
        if abs(pos) >= abs(neg):
            chosen = [v for v in vals if v > 0]
        else:
            chosen = [v for v in vals if v < 0]
        merged[i] += sum(chosen) / len(chosen) if chosen else 0.0
    return merged

W0 = [10.0, 10.0]
delta_a = [2.0, -1.0]
delta_b = [1.0, 3.0]

ta = task_arithmetic(W0, [delta_a, delta_b], [1.0, 1.0])
sv = sign_vote_merge(W0, [delta_a, delta_b])

assert ta == [13.0, 12.0]
assert sv[0] == 11.5
assert sv[1] == 13.0
```

这段代码只是教学版。真实大模型中，`base` 和 `delta` 不是长度为 2 的列表，而是数十亿参数组成的张量。张量（tensor，白话解释：高维数组）层层对应，按相同规则逐元素处理。

工程伪代码通常长这样：

```python
W_merge = W0.copy()
for layer in layers:
    deltas = [Wi[layer] - W0[layer] for Wi in models]
    merged_delta = merge_rule(deltas)
    W_merge[layer] = W0[layer] + merged_delta
```

真正落地时，配置项往往比算法本身更重要：

| 配置字段 | 作用 | 示例 |
|---|---|---|
| `base_model` | 指定共同 base | `Qwen/Qwen2.5-7B` |
| `models` | 参与合并的模型列表 | `math`, `code`, `chat` |
| `weights` | 各模型权重系数 | `[0.8, 1.0, 0.6]` |
| `merge_method` | 合并策略 | `task_arithmetic` / `ties` |
| `layer_filter` | 限定参与合并的层 | 只合并 attention/MLP |
| `dtype` | 保存精度 | `bf16` |
| `tokenizer_source` | 分词器来源 | 通常跟 base 一致 |

如果使用现成工具，`mergekit` 和 `MergeLM` 都是常见选择。它们的价值不只是提供算法实现，更重要的是把模型加载、层映射、权重导出、配置管理做成了稳定流程。

合并前建议先过一遍检查清单：

| 检查项 | 为什么重要 |
|---|---|
| 参数 shape 一致 | 否则无法逐元素合并 |
| tokenizer 一致 | 否则输入切分和输出解码都会变 |
| 层命名一致 | 否则脚本可能漏层或错层 |
| 量化状态一致 | 混合精度和量化可能引入数值偏差 |
| 评测脚本已准备 | 合并后必须立刻做回归测试 |

真实工程例子：一个团队有三套 `LoRA`，分别服务客服问答、表格抽取、SQL 生成。若三套能力都要进一个离线私有部署节点，显存预算又卡死，那么可以先在统一 base 上导出三组 adapter 增量，再用 Task Arithmetic 做第一版，用 TIES 处理明显冲突的层，最后在三类验证集和混合任务集上回归。

## 工程权衡与常见坑

模型合并最容易让人误判的地方是：它常常能让模型“看起来会更多事”，但不代表每件事都做得更好。多任务可用和单任务最优，是两个不同目标。

最常见的坑是直接平均导致抵消。假设某个参数位置上，数学模型更新为 `+0.8`，代码模型更新为 `-0.7`。平均后接近 `+0.05`，这等于把两个任务都想保留的信息削弱了。这个问题在任务方向相反时最明显，所以冲突检测比“简单平均”更重要。

另一个坑是评测方式太单一。很多人只看一个总榜分数，或者只看一个任务集。如果你的目标是“一个模型兼顾聊天、代码、数学”，那评测就必须覆盖这三类任务，还要有混合提示词场景。否则容易出现“数学涨了，代码崩了，但总分看不出来”的情况。

下面这张表是工程里最常见的坑位与对策：

| 常见坑 | 现象 | 规避方式 |
|---|---|---|
| 不同 base 直接合并 | 输出质量明显崩坏 | 只合并同源模型 |
| 直接平均完整权重 | 多项能力一起变弱 | 先试 Task Arithmetic，再试 TIES/DARE |
| 系数乱设 | 某项能力被压没 | 做小范围网格搜索 |
| 只看单任务评测 | 误判合并成功 | 加跨任务与混合任务评测 |
| tokenizer 不一致 | 输出不稳定、乱码、风格漂移 | 统一 tokenizer |
| 量化状态混乱 | 数值异常或结果不复现 | 先反量化或统一精度 |
| 层名不对齐 | 某些层没合进去 | 合并前做 state dict diff |

这里给一个真实工程中的经验判断：当任务冲突较大时，先用 `Task Arithmetic` 做基线最稳，因为它最容易解释；如果发现某些任务明显相互拖累，再尝试 `TIES` 或 `DARE`。不要一上来就堆复杂算法，因为你首先要确认问题到底出在冲突、噪声、还是兼容性。

还有一个容易被忽视的点是“层级差异”。不是所有层都同等重要。某些任务的风格更新主要集中在高层，而知识补充可能分布更广。工程上常见做法是先全量合并做基线，再试只合并某些模块，例如只合并 attention 或只合并后几层，观察哪类能力受影响最大。

## 替代方案与适用边界

模型合并不是默认最优解，它只是“同源模型能力整合”这类问题中的一条成本较低的路线。是否该用，要看你的目标函数。

如果目标是：同一个 base 上已经有多个专项微调版本，希望用最低额外训练成本把它们收敛到一个部署产物，那么模型合并很合适。它的优势是快、便宜、迭代周期短，而且通常不需要重新做大规模训练。

如果目标变成：每个任务都要尽可能逼近单模型峰值，那合并就不一定合适。因为合并天然带折中，冲突强的任务不太可能无损共存。

可以和几个常见替代方案对比：

| 方案 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| 模型合并 | 同源模型能力整合 | 部署简单，额外训练少 | 可能冲突、性能折中 |
| 保留多模型 | 任务差异大 | 单任务性能高 | 运维和显存成本高 |
| 路由 / MoE 风格 | 输入可清晰分流 | 更灵活 | 系统复杂、调度成本高 |
| 重新联合训练 | 资源充足、目标明确 | 可控性高 | 成本最高、周期最长 |

玩具例子可以这样看：

- 需求 A：一个模型同时处理小学数学题和简单 Python 代码补全  
  这两个任务都基于文本推理，且来自同一 base，合并值得一试。
- 需求 B：一个模型既做通用聊天，又做高精度法条检索生成，还做结构化工业控制指令  
  这三类任务的行为约束差异很大，如果每项都要求强稳定性，更可能需要多模型或路由方案。

真实工程里，常见的选择建议是：

- 如果目标是“一个模型覆盖数学、代码、对话”并且接受少量折中，优先考虑模型合并。
- 如果目标是“每个领域都接近各自最优”，不要默认合并，先评估多模型路由或继续联合训练。
- 如果模型来源不一致，先不要想算法细节，先把兼容性问题解决，否则后面的调参没有意义。

一句话概括边界：模型合并适合解决“同源多能力压缩成一个部署单元”的问题，不适合解决“异构模型统一成一个最优通才”的问题。
