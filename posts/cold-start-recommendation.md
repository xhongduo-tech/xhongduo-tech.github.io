## 核心结论

冷启动问题是推荐系统在新用户、新内容或新业务刚进入系统时，因为缺少行为数据而失去个性化能力的问题。行为数据可以理解为点击、停留、点赞、购买这类“系统已经观察到的反馈”。没有这些反馈，协同过滤这类依赖“相似用户喜欢相似内容”的方法就很难工作。

工程上不能把冷启动理解成“补几个热门内容”这么简单。它本质上是一个完整流程问题：输入是什么，召回怎么做，排序前怎么探索，什么时候算热身完成，评测指标看什么，指标坏了怎么回滚。只讨论“规则池”“画像池”“多臂老虎机”这些工具，而不定义输入输出、上线门槛和回滚条件，系统通常会在真实流量下失控。

最小可行方案通常是三件事同时成立：

| 模块 | 作用 | 冷启动期为什么必要 |
| --- | --- | --- |
| 人口画像映射 | 用年龄段、设备、地区、兴趣标签估计偏好 | 没有历史行为时，先用静态属性打底 |
| 规则池/热门池 | 提供稳定可用的候选内容 | 保证首页有内容，不至于空转 |
| 探索机制 | 主动给新内容、新策略少量曝光 | 否则系统永远拿不到新反馈 |

玩具例子可以直接看一个新用户首页。用户刚注册，系统只知道“25-30岁、关注 AI、来自一线城市”。这时不能做成熟用户协同，只能把“热门榜”和“画像映射池”混合。若热门榜标准化得分为 0.08，权重为 0.7；画像池标准化得分为 0.05，权重为 0.3，则最终得分为：

$$
Score_{total}=0.7\times0.08+0.3\times0.05=0.067
$$

这不是“精准推荐”，而是“先让用户看到可消费内容，并尽快收集第一批反馈”。冷启动阶段的核心目标不是立刻做到最优点击率，而是在可控风险内完成从“无数据”到“有数据”的过渡。

## 问题定义与边界

冷启动的严格定义是：由于交互矩阵稀疏，系统无法稳定估计用户偏好或内容质量，导致推荐准确度、覆盖率和可解释性同时下降。交互矩阵稀疏，白话说就是“用户和内容之间几乎还没发生过足够多的点击或消费”。

常见冷启动分三类：

| 场景 | 主体 | 可用信息 | 主要缺失 | 常用起步方法 |
| --- | --- | --- | --- | --- |
| 新用户冷启动 | 刚注册用户 | 注册信息、地域、设备、初始兴趣选择 | 历史点击、停留、转化 | 热门榜、画像映射、首屏探索 |
| 新内容冷启动 | 新上架文章/视频/商品 | 标题、类目、关键词、图像、作者信息 | CTR、完播率、收藏率 | 内容召回、规则保量、探索曝光 |
| 新业务冷启动 | 新频道、新国家、新端 | 少量种子数据、业务规则 | 稳定分布、成熟标签体系 | 迁移特征、规则先行、人工兜底 |

为什么新商品刚上架时协同过滤不生效？因为协同过滤需要“谁和谁一起被点击过”。如果一件商品只有标题、类目和属性，还没人点过，它在用户-物品交互图里几乎是孤立点。此时只能退回内容相似度，例如把“降噪耳机”先推荐给近期浏览过“蓝牙耳机”“音频设备”的人群，先拿到第一批交互。

边界也必须说清。不是“有一次点击”就算结束冷启动。更合理的定义是：当主体已经积累到足以进入主流程的最小有效反馈量，就视为完成热身。这个阈值可以是“新内容累计 100 次有效曝光且 CTR 置信区间收敛”，也可以是“新用户累计 20 次真实交互后切到常规排序”。

可以把它理解成一条简单关系：

$$
\text{数据越稀疏} \Rightarrow \text{越依赖规则与内容特征}
$$

$$
\text{反馈越充分} \Rightarrow \text{越依赖个性化模型}
$$

成熟用户协同和冷启动规则的输入差异如下：

| 推荐模式 | 主要输入 | 输出特点 |
| --- | --- | --- |
| 成熟用户协同 | 长期点击序列、共现关系、Embedding 向量 | 个性化强，波动小 |
| 冷启动规则 | 人口画像、内容属性、人工策略、探索预算 | 个性化弱，但启动快、可控 |

## 核心机制与推导

一个可落地的冷启动方案通常不是单一算法，而是“多源召回 + 过滤 + 探索 + 主排序”的流水线。

核心打分公式通常写成：

$$
Score_{total}=\sum_i \alpha_i \times norm(score_i)
$$

其中，$\alpha_i$ 是渠道权重，表示该召回源在当前场景下的重要性；`norm(score_i)` 是标准化后的分数，白话说就是把不同来源的分数映射到同一量纲，避免一个渠道因为分值范围大而压制其他渠道。

冷启动期的关键不是公式本身，而是权重重心的变化。成熟阶段，$\alpha$ 更多分给协同过滤、序列模型、深度排序等个性化来源；冷启动阶段，$\alpha$ 需要向内容特征、规则池、人气池倾斜。例如：

| 渠道 | 成熟用户权重 | 新用户权重 |
| --- | --- | --- |
| 协同过滤召回 | 0.45 | 0.05 |
| 序列兴趣召回 | 0.30 | 0.10 |
| 热门池 | 0.10 | 0.45 |
| 画像映射池 | 0.05 | 0.25 |
| 新内容探索池 | 0.10 | 0.15 |

这里的逻辑很直接：不是协同过滤“没用”，而是它在数据稀疏区间不可靠，应该暂时降权。

真实工程例子可以看新内容推荐流水线。系统先把用户最近消费轨迹编码成用户向量，把新内容的文本、图像、标签编码成内容向量。`Fresh Content Nominator` 用点积生成候选：

$$
score(u, i)=\mathbf{u}\cdot\mathbf{v}_i
$$

点积，白话说就是“用户向量和内容向量方向有多接近”。越接近，越可能相关。之后进入 `Graduation Filter`，根据曝光、点击、收藏、完播等交互判断新内容是否“毕业”到主流程。如果内容已经拿到足够反馈，就不该继续长期吃探索流量，否则会污染全局分发。

仅靠线性加权仍然不够，因为冷启动期最大的难题是“不确定性”。新内容得分低，不一定因为差，也可能只是没人见过。此时需要多臂老虎机。多臂老虎机可以理解为“边试边学的分配器”，它会在收益未知的候选之间分配流量，兼顾利用和探索。

Thompson Sampling 是常见做法。若把每个候选的点击率看成未知概率 $p$，就可以用 Beta 分布建模：

$$
p_i \sim Beta(\alpha_i, \beta_i)
$$

这里的 $\alpha_i$ 可以理解为成功次数加先验，$\beta_i$ 是失败次数加先验。每轮从各候选分布里采样，采样值高的优先展示。这样高潜内容会逐步拿到更多流量，低质量内容会自然被淘汰。

流程可以概括为：

召回 -> 基础过滤 -> 探索器挑选候选 -> DNN 排序 -> 曝光 -> 反馈回流 -> 更新探索参数 -> 达阈值后毕业到主流程

重点不在“用了 Thompson Sampling”这几个字，而在工程闭环：探索带来反馈，反馈更新不确定性，达到门槛后退出冷启动专线。

## 代码实现

下面给一个可运行的 Python 玩具实现，展示新用户场景下如何把“热门池 + 画像池 + 探索器”拼起来。这个例子不追求工业级细节，但接口划分和真实系统是一致的。

```python
from dataclasses import dataclass
import random

@dataclass
class Item:
    item_id: str
    source: str
    norm_score: float
    alpha: float
    beta_a: int
    beta_b: int

def blended_score(item: Item) -> float:
    return item.alpha * item.norm_score

def thompson_sample(item: Item) -> float:
    return random.betavariate(item.beta_a, item.beta_b)

def rank_for_new_user(items, topk=3):
    ranked = []
    for item in items:
        # 冷启动阶段：规则/画像先提供基础分，探索器提供不确定性奖励
        final_score = 0.8 * blended_score(item) + 0.2 * thompson_sample(item)
        ranked.append((item.item_id, item.source, final_score))
    ranked.sort(key=lambda x: x[2], reverse=True)
    return ranked[:topk]

items = [
    Item("hot_1", "popular", 0.08, 0.7, 20, 80),
    Item("profile_1", "profile", 0.05, 0.3, 8, 12),
    Item("fresh_1", "explore", 0.04, 0.2, 2, 3),
]

score_hot = blended_score(items[0])
score_profile = blended_score(items[1])
expected = 0.7 * 0.08 + 0.3 * 0.05

assert abs(score_hot - 0.056) < 1e-9
assert abs(score_profile - 0.015) < 1e-9
assert abs(expected - 0.071) < 1e-9

top_items = rank_for_new_user(items, topk=2)
assert len(top_items) == 2
assert all(len(x) == 3 for x in top_items)

print(top_items)
```

这个实现里有三个关键模块：

| 模块 | 输入 | 输出 | 说明 |
| --- | --- | --- | --- |
| `blended_score` | 标准化分数、渠道权重 | 基础得分 | 体现规则池/画像池优先级 |
| `thompson_sample` | 历史成功失败次数 | 探索采样值 | 给新内容不确定性奖励 |
| `rank_for_new_user` | 候选列表 | TopK 排序结果 | 新用户先命中热门+画像，再混入探索 |

如果落到真实服务接口，可以抽象成：

- `ProfileMapper(user_features) -> candidate_pool`
- `RuleRetriever(scene, budget) -> candidate_pool`
- `BanditExplorer(candidates, feedback_state) -> reranked_candidates`
- `GraduationFilter(item_stats, threshold) -> route_to_mainflow`

一个典型工程流程是：新用户进站后先拉热门池和画像池各 100 条，再插入少量新内容探索位，例如 5%。探索位不是随机乱插，而是由多臂老虎机在新内容集合里挑 Top10，之后再交给 DNN 排序模型选 TopN。这样主排序仍然统一，但探索预算受控。

## 工程权衡与常见坑

冷启动最容易出错的地方，是把“短期 CTR 最大化”当成唯一目标。探索会天然拉低一些即时点击，因为你在给不确定内容机会。现实里 1% 到 3% 的短期 CTR 损失并不罕见，但如果因此换来新用户留存提升、新内容覆盖扩大，这个代价可能是值得的。

因此评测口径不能只看 CTR。至少要补这些指标：

| 指标 | 含义 | 为什么要看 |
| --- | --- | --- |
| CTR | 点击率 | 看短期吸引力 |
| Retention | 留存率 | 看冷启动是否真的把人留住 |
| Coverage | 覆盖率 | 看系统是否只推少数热门内容 |
| ILD | 列表内多样性 | 看推荐是否过于同质化 |
| Fresh Exposure Rate | 新内容曝光占比 | 看新内容是否真正拿到流量 |

如果只盯 CTR，系统会偏向“永远推旧爆款”，新内容没有机会，长期供给会枯竭。

常见坑可以直接列出来：

| 坑 | 后果 | 规避手段 |
| --- | --- | --- |
| 把热门池当全部方案 | 首屏稳定但个性化长期很差 | 热门池只做兜底，必须叠加画像和探索 |
| 探索比例过大 | CTR 明显下跌，业务难接受 | 先从固定小流量开始，按场景调预算 |
| 没定义毕业阈值 | 新内容长期占探索位，污染主流程 | 用曝光、点击、转化阈值控制迁移 |
| 样本选择偏差 | 模型只学会推高分旧内容 | 训练集纳入探索样本并做校正 |
| 上线无自动回滚 | 策略异常时持续放大损失 | 监控告警绑定自动回滚 |

真实工程里，上线门槛和回滚条件必须写成配置，而不是口头约定。一个伪配置示意如下：

```yaml
cold_start_release:
  scene: homepage_new_user
  enable: true
  traffic_ratio: 0.1
  success_gate:
    retention_d1_lift: ">= 0.5%"
    fresh_exposure_rate: ">= baseline"
    coverage: ">= baseline"
    ctr_drop: "<= 2%"
  rollback:
    trigger:
      - cloudwatch_alarm == "ALARM"
      - deployment_state == "FAILED"
      - ctr_drop > 3%
    action: "appconfig_auto_rollback"
```

这类配置的意义是：当 CloudWatch 告警触发，或部署状态失败，或 CTR 跌破约定阈值时，AppConfig 自动回滚，避免冷启动逻辑持续污染线上指标。对初级工程师来说，最重要的观念是：推荐策略上线不是“算法发版”，而是“带监控、带熔断、带回滚的产品变更”。

## 替代方案与适用边界

冷启动没有单一标准答案，应该根据冷启动持续时间、隐私限制、特征可用性和业务成本选方案。

| 方案 | 优势 | 限制 | 适用场景 |
| --- | --- | --- | --- |
| 规则 + 画像 | 实现快、可解释、稳定 | 个性化弱，天花板低 | 新业务启动、低数据量场景 |
| 内容相似度 | 对新内容友好，不依赖交互 | 特征工程要求高 | 商品、文章、视频等内容丰富场景 |
| 多臂老虎机探索 | 能处理不确定性，自动分配流量 | 指标波动更大，监控要求高 | 新内容流入频繁的平台 |
| 外部数据增强 | 补齐冷启动特征，降低空白期 | 隐私、合规、质量风险高 | B 端或有稳定外部画像来源场景 |
| 强化学习式探索 | 长期收益优化能力强 | 训练、评估、上线复杂 | 超大规模平台，且实验基础成熟 |

规则 + 画像适合“先跑起来”；多臂老虎机适合“已有一定流量，想把探索做成系统能力”；外部数据增强适合“业务允许且数据可靠”；强化学习更适合成熟平台，不适合零到一阶段直接上。

新内容何时切回主流程，也必须明确。一个实用规则是：当内容达到最小有效反馈门槛，就退出冷启动专线。例如：

- 曝光超过 1000
- 点击超过 50
- 有效互动率达到类目基线
- 置信区间足够窄，质量估计稳定

此时应进入常规召回与排序，不要永远依赖老虎机。否则系统会把“冷启动机制”当常驻机制，造成分发体系混乱。

所以完整策略不是“给新内容一点流量”这么简单，而是：

1. 用规则、画像、内容特征把主体送进系统。
2. 用探索机制快速收集早期反馈。
3. 用毕业阈值把成熟主体迁回主流程。
4. 用监控和回滚保证策略异常时能立刻止损。
