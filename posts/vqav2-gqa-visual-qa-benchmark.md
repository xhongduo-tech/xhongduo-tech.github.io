## 核心结论

VQAv2 和 GQA 都是视觉问答基准。视觉问答，指模型输入一张图像和一个自然语言问题，然后输出答案。两者的共同目标是评估模型是否能把“看见的内容”和“读到的问题”对齐，但它们检查的能力层次不同。

VQAv2 更偏基础能力。它主要问：模型能不能识别图像中的对象、属性、动作、数量，并给出一个短答案。新手可以把它理解成：“这张图里是什么”“这个物体是什么颜色”“图里有几个人”。它的重点是基础视觉理解和语言问题匹配。

GQA 更偏结构化推理。结构化推理，指模型不是只识别一个局部对象，而是要沿着对象、属性、关系一步步推导答案。新手可以把它理解成：“先找出图中左边的杯子，再判断它旁边是什么颜色的书”。这类问题考察的不只是“有没有看见杯子”，还包括“左边”“旁边”“书的颜色”这些关系链是否能串起来。

所以，VQAv2 更像“看见就答”的基础考卷，GQA 更像“按条件一步步推”的推理考卷。一个模型在 VQAv2 上高分，不代表它已经具备稳定的组合推理能力；一个模型在 GQA 上掉分，也不必然说明它完全不会看图，而可能是多步关系推导暴露了短板。

| 基准 | 核心目标 | 题目特征 | 主要暴露的问题 |
|---|---|---|---|
| VQAv2 | 图像问答基础能力 | 简短、自然语言、答案短 | 语言偏见、基础视觉理解不足 |
| GQA | 组合推理与一致性 | 关系链、多步推导、场景图 | 推理链断裂、属性组合失败 |

## 问题定义与边界

统一来看，视觉问答任务可以写成一组符号：给定图像 $x$ 和问题 $q$，数据集中有真实答案 $a$，模型输出预测答案 $\hat a$。评测指标的核心，就是判断 $\hat a$ 与 $a$ 是否足够一致。

| 符号 | 含义 |
|---|---|
| `x` | 图像 |
| `q` | 问题 |
| `a` | 真实答案 |
| `\hat a` | 模型预测答案 |

这里的关键边界是：VQAv2 和 GQA 都叫视觉问答，但不能把它们的准确率当成同一种能力刻度。

同样是问“图里有没有狗”，在 VQAv2 里，这类题主要考模型是否能识别狗并输出 “yes” 或 “no”。在 GQA 里，类似问题可能变成“左边那只狗旁边有没有椅子”。这时模型要先定位“左边那只狗”，再找“旁边”的物体，最后判断是否存在“椅子”。如果第一步定位错了，后面即使答案格式正确，也不是可靠推理。

玩具例子可以这样理解：桌上有一个红杯子和一本蓝书。问题 A 是“杯子是什么颜色”，答案是“红色”。问题 B 是“杯子旁边的书是什么颜色”，答案是“蓝色”。A 只需要识别杯子的属性；B 需要识别杯子、判断空间关系、再读取书的属性。VQAv2 中有大量接近 A 的问题，GQA 更强调 B 这类组合问题。

真实工程例子是仓库巡检机器人。简单问题是“货架上有没有红色箱子”。如果模型能识别红色和箱子，就可能答对。复杂问题是“左侧第二层货架上，蓝色箱子右边的标签是什么”。这要求模型先定位货架层级，再定位蓝色箱子，再理解右侧关系，最后读取标签。只看基础问答准确率，可能会误判系统已经可用。

因此，VQAv2 更偏“标准视觉问答”，GQA 更偏“带结构化推理的视觉问答”。两者不能简单按同一准确率口径机械比较。

## 核心机制与推导

VQAv2 的官方评分不是简单的 exact match。exact match，指模型输出字符串必须和标准答案字符串完全相同才算对。这个做法在视觉问答里太硬，因为人类标注本身可能存在多个合理答案。例如同一张图里，一个人写 “cat”，另一个人写 “kitten”，它们语义接近，但字符串不同。

VQA 的常用评分思路是基于 10 个人类标注答案做软评分。设某题的 10 个标注答案为 $a_1,\dots,a_{10}$，模型答案是 $\hat a$，则匹配数量为：

$$
c(q)=\sum_{j=1}^{10} 1[\hat a = a_j]
$$

其中 $1[\cdot]$ 是指示函数，条件成立时为 1，否则为 0。题目级 VQA 分数为：

$$
Acc_{vqa}(q)=\min(c(q)/3, 1)
$$

这意味着，只要至少 3 个标注者给出了和模型相同的答案，模型就可以拿满分。某题 10 个标注里，4 个答案是 “cat”，6 个答案是 “dog”。模型答 “cat”，得分是 $\min(4/3, 1)=1$；模型答 “car”，匹配数为 0，得分是 0。这个机制说明 VQAv2 更看人类共识，而不是死板字符串匹配。

GQA 的问题不同。它不只关心最终答案对不对，还关心答案是否和相关问题保持一致。一致性，指模型在一个问题上给出的答案，能否支持它在蕴含问题上的回答。蕴含问题，指由原问题和答案可以直接推出的问题。

假设 $Q^+$ 是模型答对的原问题集合，$E_q$ 是由原问题 $q$ 可推出的蕴含问题集合，则一致性可以写成：

$$
Cons = \frac{1}{|Q^+|}\sum_{q\in Q^+}\frac{1}{|E_q|}\sum_{q'\in E_q}1[\hat a(q')=a(q')]
$$

如果某个原问题对应 3 个蕴含问题，模型答对 2 个，则这一组的一致性是 $2/3$。这说明 GQA 会检查“一个答案是否能带出一组正确后续判断”。

举例：原问题是“红色杯子在桌子上吗”，模型答“是”。由此可能蕴含出“图中有杯子吗”“杯子是什么颜色”“杯子在哪里”。如果模型原问题答对，但后续又说“没有杯子”，它的单题答案可能碰巧正确，但推理状态并不稳定。

| 基准 | 核心指标 | 指标在测什么 |
|---|---|---|
| VQAv2 | `Acc_vqa` | 与人类标注的一致性 |
| GQA | `Cons` / `Validity` / `Plausibility` | 逻辑一致性、可回答性、答案合理性 |

Validity 指答案是否符合问题类型。例如问颜色时，回答 “yes” 通常无效。Plausibility 指答案是否在语义上合理。例如问“苹果是什么颜色”，回答“跑步”不合理。GQA 用这些指标补充单一准确率，因为复杂视觉问答的错误不只有“答错”，还包括“答案类型错”“前后矛盾”“关系链断裂”。

## 代码实现

评测实现要先统一数据格式，再计算指标。不能直接拿模型自由文本输出和标准答案字符串硬比。自由文本输出，指模型可能回答“这是一只猫”“cat.” 或 “a cat”，而标准答案可能只是 “cat”。如果不清洗，评测结果会被大小写、标点、冠词影响。

基本流程是三步：加载数据、标准化答案、计算指标。

| 步骤 | VQAv2 | GQA |
|---|---|---|
| 1 | 读取问题与 10 个答案标注 | 读取原问题、蕴含问题、场景结构 |
| 2 | 标准化答案 | 标准化答案并构造一致性关系 |
| 3 | 计算 soft accuracy | 计算 consistency / validity / plausibility |

下面是一个最小可运行 Python 例子。它不是官方完整评测脚本，只用于说明核心计算方式。

```python
import re
from statistics import mean

def normalize_answer(text):
    text = text.lower().strip()
    text = re.sub(r"[^\w\s]", "", text)
    text = re.sub(r"\b(a|an|the)\b", " ", text)
    text = re.sub(r"\s+", " ", text).strip()
    return text

def vqa_score(pred, gt_answers):
    pred = normalize_answer(pred)
    gt_answers = [normalize_answer(a) for a in gt_answers]
    c = sum(int(pred == a) for a in gt_answers)
    return min(c / 3.0, 1.0)

def gqa_consistency(pred_map, gold_map, entailment_sets):
    scores = []
    for q, implied_questions in entailment_sets.items():
        if normalize_answer(pred_map.get(q, "")) != normalize_answer(gold_map[q]):
            continue

        implied_scores = []
        for q2 in implied_questions:
            pred = normalize_answer(pred_map.get(q2, ""))
            gold = normalize_answer(gold_map[q2])
            implied_scores.append(int(pred == gold))

        if implied_scores:
            scores.append(mean(implied_scores))

    return mean(scores) if scores else 0.0

vqa_answers = ["cat", "cat", "cat", "cat", "dog", "dog", "dog", "dog", "dog", "dog"]
assert vqa_score("Cat.", vqa_answers) == 1.0
assert vqa_score("car", vqa_answers) == 0.0

pred_map = {
    "q1": "yes",
    "q1_entail_1": "cup",
    "q1_entail_2": "red",
    "q1_entail_3": "shelf",
}
gold_map = {
    "q1": "yes",
    "q1_entail_1": "cup",
    "q1_entail_2": "red",
    "q1_entail_3": "table",
}
entailment_sets = {
    "q1": ["q1_entail_1", "q1_entail_2", "q1_entail_3"]
}

assert gqa_consistency(pred_map, gold_map, entailment_sets) == 2 / 3
```

最小实现有三个约束。第一，输入答案必须先清洗，包括大小写、标点、冠词和空白符。第二，必须区分题目级分数和整体平均分，不能把单题得分当成全局指标。第三，GQA 不能只算单题准确率，必须保留原问题和蕴含问题之间的关系映射，否则 consistency 无法计算。

工程实现中还要记录分桶结果。分桶，指按问题类型、推理深度、答案类型等维度拆开统计。比如颜色题、计数题、关系题分别统计。这样才能知道模型是“整体都弱”，还是“基础识别强但关系推理弱”。

## 工程权衡与常见坑

第一个常见坑，是把 VQA 当成简单分类任务做 exact match。模型答 “cat”，标注里有人写 “Cat.”，语义一致，但字符串不完全一致。直接 exact match 会低估模型表现。官方软评分的核心意义，就是承认人类答案存在合理差异，并用共识程度计算分数。

第二个常见坑，是只看 GQA 总准确率。总分会把不同难度的问题混在一起。简单题通常是单事实查询，例如颜色、位置、是否存在。VLM，即视觉语言模型，通常在这类题上更稳，因为它接近“局部识别 + 轻量匹配”。复杂题涉及关系链、属性组合、计数和多步推断，一旦题目从“看见就答”变成“串联约束再答”，性能往往下滑。

第三个常见坑，是混用自由生成和固定答案集。固定答案集，指模型只能从预定义答案列表中选一个；自由生成，指模型可以输出任意文本。两者的错误形态不同。固定答案集容易限制表达，自由生成容易出现同义改写、冗余解释和格式不一致。评测前必须统一答案规范。

| 常见坑 | 后果 | 规避方式 |
|---|---|---|
| 直接 exact match | 低估 VQA 表现 | 使用官方 soft scoring |
| 只看 GQA 总准确率 | 看不到推理短板 | 同时看 consistency / validity / plausibility |
| 忽略 question type 分析 | 无法定位错误来源 | 按类型、深度、拓扑分桶 |
| 混用自由生成和固定答案集 | 评测不可比 | 先统一答案规范 |
| 忽略数据泄漏 | 分数虚高 | 检查特征来源和验证集污染 |

真实工程例子仍以仓库巡检机器人为例。模型可能在“有没有红色箱子”上表现很好，因为红色和箱子都是显著视觉特征。但在“左侧第二层货架上，蓝色箱子右边的标签是什么”上失败，因为它需要同时处理空间定位、层级关系、颜色属性、邻接关系和文字识别。只看总体准确率，会掩盖这个短板。

更稳妥的做法是把评测拆成几组：存在性问题、属性问题、空间关系问题、计数问题、多步关系问题。然后分别看准确率和一致性。这样才能判断模型失败在视觉识别、语言理解、关系推理，还是答案规范化。

## 替代方案与适用边界

VQAv2 适合做基础视觉问答能力评测，但不适合单独代表组合推理能力。它可以告诉你模型是否具备基本看图答题能力，却不能充分证明模型能稳定处理多步条件。

GQA 更适合测试结构化推理，但它也不是完整的开放域视觉理解评测。开放域视觉理解，指模型不仅回答短问题，还要做长文本描述、跨图对话、细粒度解释和外部知识结合。GQA 对组合推理更敏感，但仍然受数据集构造方式、问题模板、场景图质量和答案空间影响。

| 场景 | 更适合的基准 | 原因 |
|---|---|---|
| 基础视觉问答 | VQAv2 | 看基本理解和短答案能力 |
| 组合推理 | GQA | 看多步推导和一致性 |
| 开放域描述 | 其他任务补充 | VQA/GQA 不覆盖长文本表达 |

产品原型阶段，可以先用 VQAv2 看基础可用性。比如模型是否能回答“图中有几个人”“汽车是什么颜色”“桌上有什么”。如果这些问题都不稳，继续做复杂推理评测意义不大。

推理能力验证阶段，再用 GQA 看是否能处理关系链和多步判断。比如“穿红衣服的人左边的物体是什么”“桌子下面靠近椅子的东西是什么颜色”。这些问题更接近真实系统中的条件查询。

但边界必须明确。不能把 VQAv2 的高分误读为“真正理解图像”。高分可能来自强视觉识别、语言先验和训练数据覆盖。也不能把 GQA 的低分误读为“模型完全不会推理”。GQA 的问题更难，评测口径更细，低分可能说明模型在某类关系、某种拓扑结构或某个推理深度上不稳。

对初级工程师来说，最实用的选择方式是：先用 VQAv2 判断模型是否具备基础视觉问答能力，再用 GQA 定位组合推理短板。如果业务还需要开放式长回答、文档图表理解、OCR、跨多张图的对话，就需要补充其他基准和自建测试集。
