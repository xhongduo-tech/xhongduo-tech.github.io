## 核心结论

成分句法分析（constituency parsing）是把句子解析成带标签的嵌套短语树。它预测的不是“哪个词依赖哪个词”，而是“哪些连续词先组成短语，短语再如何组成更大的短语”。

一句话定义：

**成分句法 = 短语树结构预测。**

例如句子：

> 我 喜欢 这 本 书

成分句法不会只输出“喜欢 → 书”这种词间关系，而是先识别：

```text
这 本 书  -> NP
喜欢 这 本 书 -> VP
我 + 喜欢 这 本 书 -> S
```

一棵简化树可以写成：

```text
S
├── NP: 我
└── VP
    ├── V: 喜欢
    └── NP
        ├── Det: 这
        ├── Clf: 本
        └── N: 书
```

新手可以先这样理解：**成分句法先找词组，再找词组的词组，一直合并到整句。**

它和依存句法的核心区别如下：

| 维度 | 成分句法分析 | 依存句法分析 |
|---|---|---|
| 基本单位 | 连续区间 span | 词与词之间的边 |
| 输出形式 | 带标签的短语树 | 有向依存图或依存树 |
| 关注问题 | 哪些词组成一个整体 | 哪个词支配哪个词 |
| 常见标签 | `NP`、`VP`、`PP`、`S` | `nsubj`、`obj`、`amod` |
| 适合任务 | 短语边界、层次结构、范围识别 | 词间关系、谓词论元、修饰关系 |

成分句法分析的技术核心是：**在所有合法短语树中找到全局最优的一棵树**。经典做法是 `chart + 动态规划 + CKY`；现代神经模型通常用 `BiLSTM` 或 `Transformer` 编码上下文，再用 chart decoder 解码最优树。

## 问题定义与边界

输入是一个词序列：

$$
x_{1:n} = x_1, x_2, ..., x_n
$$

输出是一棵树。树由若干个区间和标签组成。这里的 **span** 指句子中一段连续词的范围，例如 `[2, 4]` 表示从第 2 个词到第 4 个词。**label** 指这个区间的短语类别，例如 `NP` 表示名词短语，`VP` 表示动词短语，`PP` 表示介词短语。

| 名称 | 形式 | 白话解释 |
|---|---|---|
| `span` | `[i, j]` | 从第 `i` 个词到第 `j` 个词的一段连续文本 |
| `label` | `l` | 给这段文本贴的短语类型 |
| `tree` | `spans + labels` | 所有短语区间和标签组成的嵌套结构 |

以“他 在 北京 工作”为例：

```text
他 / 在 / 北京 / 工作
1    2    3     4
```

一种可能的 span 表示如下：

| span | 文本 | label | 含义 |
|---|---|---|---|
| `[1,1]` | 他 | `NP` | 主语名词短语 |
| `[2,3]` | 在 北京 | `PP` | 介词短语 |
| `[2,4]` | 在 北京 工作 | `VP` | 谓语结构 |
| `[1,4]` | 他 在 北京 工作 | `S` | 整句 |

示意图：

```text
[1,4] S: 他 在 北京 工作
├── [1,1] NP: 他
└── [2,4] VP: 在 北京 工作
    ├── [2,3] PP: 在 北京
    │   ├── [2,2] P: 在
    │   └── [3,3] NP: 北京
    └── [4,4] V: 工作
```

成分分析解决的是“哪些词先组成短语、短语如何继续合并”的问题。它不直接做依存边预测，也不直接做语义角色标注。

| 边界 | 内容 |
|---|---|
| 输入 | 已分词的词序列，或由模型内部处理的 token 序列 |
| 输出 | 带短语标签的树结构 |
| 直接解决 | 短语边界、短语类别、层次合并顺序 |
| 不直接解决 | 词间依存边、语义角色、事件抽取、实体链接 |

这里要区分“句法结构”和“语义理解”。成分树能告诉你“在 北京”是一个 `PP`，也能告诉你它在更大的 `VP` 里，但它不会直接告诉你“北京”是地点角色。地点角色属于语义角色标注或信息抽取任务。

## 核心机制与推导

成分句法分析可以看成一个打分搜索问题。模型给每个候选 span 和 label 一个分数：

$$
s(i,j,l)
$$

其中 `[i,j]` 是区间，`l` 是短语标签。整棵树的分数通常写成所有短语分数之和：

$$
S(T) = \sum_{(i,j,l)\in T} s(i,j,l)
$$

目标是在所有合法树里找到得分最高的树：

$$
T^* = \arg\max_T S(T)
$$

如果直接枚举所有树，数量会随句长快速爆炸。动态规划的作用是把重复子问题存起来。**动态规划**是把大问题拆成小问题，并缓存小问题答案的方法。

对一个区间 `[i,j]`，它可以在某个切分点 `k` 拆成左区间 `[i,k]` 和右区间 `[k+1,j]`。于是有：

$$
F[i,j] = \max_l \left( s(i,j,l) + \max_{k=i}^{j-1}(F[i,k] + F[k+1,j]) \right)
$$

其中：

$$
F[i,i] = 0
$$

`F[i,j]` 表示区间 `[i,j]` 能得到的最优分数。CKY 算法就是在一个二维 chart 上按区间长度从小到大填表。**chart** 是一个表格，里面存每个区间的最优分数和回溯指针。

玩具例子：句子只有 3 个词：

```text
x1 x2 x3
```

设叶子分数为 0，候选短语分数为：

| span | 分数 |
|---|---:|
| `[1,2]` | 1.2 |
| `[2,3]` | 0.4 |
| `[1,3]` | 2.0 |

两种二叉树分别是：

```text
((x1 x2) x3) = 1.2 + 2.0 = 3.2
(x1 (x2 x3)) = 0.4 + 2.0 = 2.4
```

所以 chart 会选择左结合结构：

```text
[1,3]
├── [1,2]
│   ├── x1
│   └── x2
└── x3
```

对应的 DP 表：

| 区间 | 最优切分 | 最优分数 |
|---|---|---:|
| `[1,1]` | 无 | 0 |
| `[2,2]` | 无 | 0 |
| `[3,3]` | 无 | 0 |
| `[1,2]` | `1 | 2` | 1.2 |
| `[2,3]` | `2 | 3` | 0.4 |
| `[1,3]` | `[1,2] | [3,3]` | 3.2 |

区间拆分流程可以写成：

```text
目标区间 [i,j]
      │
      ├── 尝试 k=i
      │   ├── 左: [i,k]
      │   └── 右: [k+1,j]
      │
      ├── 尝试 k=i+1
      │   ├── 左: [i,k]
      │   └── 右: [k+1,j]
      │
      └── 保留得分最高的切分
```

CKY 的典型复杂度是 $O(n^3)$。原因是：区间长度有 $O(n)$ 种，左端点有 $O(n)$ 种，每个区间还要枚举 $O(n)$ 个切分点。因此总量是 $O(n) \times O(n) \times O(n)$。如果还显式枚举大量标签或文法规则，复杂度会继续乘上标签或规则数量。

## 代码实现

工程实现通常分成三层：

| 模块 | 作用 |
|---|---|
| 编码器 | 为每个 token 生成上下文表示，常用 `BiLSTM` 或 `Transformer` |
| span 打分器 | 计算每个 `[i,j,l]` 的分数 `s(i,j,l)` |
| chart decoder | 用动态规划搜索全局最优树，并保存回溯指针 |

**BiLSTM** 是双向循环神经网络，会同时利用左侧和右侧上下文。**Transformer** 是基于自注意力的编码器，可以让任意两个 token 直接交互。**decoder** 是把模型分数转换成最终结构的解码模块。

下面是一个最小可运行的 CKY 风格实现。为了突出机制，代码只处理无标签二叉树；真实模型会把 `span_score(i,j)` 扩展为 `span_label_score(i,j,l)`。

```python
from dataclasses import dataclass
from typing import Optional, Tuple, Dict, List

@dataclass
class Cell:
    score: float
    split: Optional[int]

def cky_decode(tokens: List[str], span_scores: Dict[Tuple[int, int], float]):
    """
    使用 0-based 闭区间 [i, j]。
    chart[i][j] 保存区间 [i,j] 的最优分数和最佳切分点。
    """
    n = len(tokens)
    chart = [[None for _ in range(n)] for _ in range(n)]

    for i in range(n):
        chart[i][i] = Cell(score=0.0, split=None)

    for width in range(2, n + 1):
        for i in range(0, n - width + 1):
            j = i + width - 1
            best_score = float("-inf")
            best_split = None
            span_score = span_scores.get((i, j), 0.0)

            for k in range(i, j):
                left = chart[i][k].score
                right = chart[k + 1][j].score
                score = left + right + span_score

                if score > best_score:
                    best_score = score
                    best_split = k

            chart[i][j] = Cell(score=best_score, split=best_split)

    return chart

def build_tree(tokens: List[str], chart, i: int, j: int):
    if i == j:
        return tokens[i]

    k = chart[i][j].split
    left = build_tree(tokens, chart, i, k)
    right = build_tree(tokens, chart, k + 1, j)
    return (left, right)

tokens = ["x1", "x2", "x3"]
span_scores = {
    (0, 1): 1.2,
    (1, 2): 0.4,
    (0, 2): 2.0,
}

chart = cky_decode(tokens, span_scores)
tree = build_tree(tokens, chart, 0, 2)

assert chart[0][2].score == 3.2
assert tree == (("x1", "x2"), "x3")
print(tree)
```

这段代码对应的流程是：

```text
1. 初始化长度为 1 的 span
2. 枚举区间长度 width = 2..n
3. 枚举左端点 i，并计算右端点 j
4. 枚举切分点 k
5. 更新 chart[i][j] 的最优分数和 best split
6. 从 chart[0][n-1] 回溯整棵树
```

如果加入标签，`chart[i][j]` 不仅要记录切分点，还要记录最佳 label。例如 `[2,4]` 的最佳 label 可能是 `VP`，`[2,3]` 的最佳 label 可能是 `PP`。整棵树的得分就是回溯路径上所有 span-label 分数之和。

真实神经成分分析器通常不会手写 `span_scores`。它会先把句子送入编码器，然后用区间两端的向量构造 span 表示，再计算每个标签分数。训练时，模型会提高 gold tree 中正确 span 的分数，降低错误 span 的分数；推理时，chart decoder 负责找全局最优结构。

## 工程权衡与常见坑

CKY 默认适合二叉结构，常见前提是文法已经二叉化，或者转换成接近 CNF 的形式。**CNF** 是乔姆斯基范式，简单说就是把文法规则规范成“一个非终结符生成两个非终结符”或“一个非终结符生成一个词”的形式。

真实树库经常不是干净的二叉树。它可能有多叉节点、连续 unary chain、空节点、标点规则等。如果训练、推理、评测阶段的预处理不一致，模型分数看似正常，评测结果会明显偏低。

| 问题 | 影响 | 规避方式 |
|---|---|---|
| 原始树有多叉结构 | CKY 无法直接按二叉切分处理 | 训练前统一二叉化，评测前还原 |
| unary chain 不一致 | 标签路径对不上 | 固定合并或展开策略 |
| 标点处理不同 | span 边界变化 | 与树库评测脚本保持一致 |
| 空节点处理不同 | 训练树和预测树不在同一空间 | 明确删除、保留或映射策略 |
| 长句过多 | $O(n^3)$ 解码成本高 | 长度裁剪、beam、span pruning |
| 分词不一致 | 中文 span 边界偏移 | 训练和推理使用同一分词方案 |

特殊结构处理对照表：

| 结构 | 例子 | 常见处理 |
|---|---|---|
| unary chain | `S -> VP -> V` | 合并成复合标签或保留链式结构 |
| 标点 | 逗号、句号、引号 | 按评测规范保留或剔除 |
| 空节点 | 省略成分、移动痕迹 | 多数工程系统会在预处理阶段删除 |
| 多叉节点 | `A -> B C D` | 二叉化为 `A -> B A'`，`A' -> C D` |

中文任务尤其要注意分词。比如：

> 我 非常 喜欢 这 本 书

如果训练数据把它切成：

```text
我 / 非常 / 喜欢 / 这 / 本 / 书
```

而推理时切成：

```text
我 / 非常喜欢 / 这 / 本 / 书
```

那么所有从“喜欢”开始的 span 边界都会偏移，树结构和标注空间已经不是同一个问题。成分句法先依赖 token 边界，再预测短语边界；因此训练与推理必须使用同一分词方案。

一个真实工程例子是中文信息抽取。系统要从“北京大学附属医院心内科主任”中抽取机构或职位，如果没有短语边界约束，模型可能把修饰成分和实体主体混在一起。成分句法可以先给出 `NP` 候选范围，再让信息抽取模型在这些候选里判断实体边界，从而减少过长实体或截断实体。

长句是另一个现实问题。下面是一个简单的 span 长度裁剪配置思路：

```python
MAX_SENT_LEN = 120
MAX_SPAN_LEN = 40

def keep_span(i, j):
    return (j - i + 1) <= MAX_SPAN_LEN

assert keep_span(0, 39) is True
assert keep_span(0, 40) is False
```

这种裁剪会降低计算量，但也可能丢掉真正的长短语。因此它是速度和召回率之间的权衡。

## 替代方案与适用边界

成分句法适合需要短语边界和层次结构的任务。它不适合所有 NLP 问题。如果任务只关心词和词之间的直接关系，依存句法通常更直接；如果任务只需要给每个 token 打标签，序列标注更轻量；如果任务只需要抽取片段，端到端 span 模型可能更简单。

| 方法 | 输出 | 适合场景 | 局限 |
|---|---|---|---|
| 成分句法 | 短语树 | 短语边界、范围识别、层次建模 | 解码较重，依赖树库规范 |
| 依存句法 | 词间依存边 | 谓词论元、修饰关系、主谓宾关系 | 不直接给完整短语边界 |
| 序列标注 | 每个 token 的标签 | NER、分词、词性标注 | 难表达嵌套结构 |
| span 模型 | 候选片段及类别 | 实体抽取、答案抽取 | 不一定保证全局树合法 |

适用与不适用任务可以这样判断：

| 任务 | 是否适合成分句法 | 原因 |
|---|---|---|
| 实体候选边界约束 | 适合 | `NP` 可提供短语范围 |
| 否定范围识别 | 适合 | 树结构能帮助确定修饰范围 |
| 情感短语组合 | 适合 | “不 太 好”应作为结构整体建模 |
| 只抽取主谓宾 | 不一定 | 依存句法通常更直接 |
| 简单 NER | 不一定 | BIO 序列标注成本更低 |
| 超长文档实时处理 | 通常不适合 CKY | $O(n^3)$ 成本过高 |

以中文信息抽取为例，“北京大学附属医院”可能是一个完整机构名，也可能出现在更大的名词短语中。成分句法给出的 `NP` 边界可以帮助系统判断候选实体的覆盖范围，减少把外层修饰语误并入实体的问题。

以情感分析为例，“不 太 好”和“不是 不 好”不能只按单词独立打分。短语结构能帮助模型识别否定和程度副词的作用范围。这里成分树不是最终任务答案，而是给下游模型提供结构约束。

什么时候不该用 CKY？如果句子很长、延迟要求很高、任务不需要完整树，或者训练数据没有稳定的成分树标注，就不应默认上 CKY。可以优先考虑依存句法、浅层 chunking、序列标注或轻量 span classifier。工程选择的关键不是“树结构更完整”，而是“任务是否真的从完整树结构中获益”。
