## 核心结论

长上下文位置编码的核心问题，不是“把最大长度参数改大”这么简单，而是让模型在超出训练窗口时，仍然能把“第 3 个 token 比第 3000 个 token 更近”这种相对位置信号表示正确。这里的位置编码，指模型用来表示顺序关系的一套数学规则；没有它，注意力机制只知道有哪些词，不知道谁在前谁在后。

当前主流的三类策略可以概括为：

| 方案 | 主要目标 | 怎么调整 | 是否需要微调 | 常见效果 |
|---|---|---|---|---|
| PI / YaRN | 把训练过的 RoPE 角度范围复用到更长序列 | 统一缩放频率或位置角度 | 需要 | 精度最好，工程上最常见 |
| 动态 NTK | 不改模型参数，推理时按实际长度动态改缩放 | 每次推理根据当前长度重算 base 或频率 | 不需要 | 实现方便，但通常有 5% 到 10% 精度损失 |
| 位置跳跃 / PoSE | 保留块内局部细节，同时让块间覆盖更远距离 | 分块后为不同块分配跳跃 position id | 可选 | 适合极长文档，常与微调结合 |

把它们说得更直白一点：

- PI / YaRN 像把更长的尺子“压缩”回模型熟悉的刻度，再通过少量训练让模型适应这种新刻度。
- 动态 NTK 像在推理时临时调一个“放大器”，当前文本越长，位置频率就拉得越开。
- PoSE 像把一本书分成很多页块，块内页码连续，块与块之间故意跳号，让模型在低频维度上看到更远的位置范围。

工程实践里，Qwen-Long、Kimi 128K、Claude 的长上下文能力，本质上都更接近“方案一加训练配方优化”，而不是单纯在推理时改个参数。原因很简单：位置编码是注意力分数的一部分，改它等于改模型看到世界的坐标系。只靠推理时补丁，通常不如在目标长度上做少量继续训练稳定。

## 问题定义与边界

先定义问题。假设一个模型在训练时只见过长度 $L_{\text{train}}=4096$ 的序列，现在希望它处理 $L_{\text{new}}=32768$。如果它使用的是 RoPE，中文通常叫旋转位置编码，也就是把位置映射成一组旋转角，让 query 和 key 在不同位置产生不同的相对相位。

RoPE 的关键性质是：不同维度对应不同频率。高频维度负责近距离、精细的位置信号，低频维度负责远距离、缓慢变化的位置信号。问题在于，训练只覆盖到 4096 时，模型实际“学会解释”的旋转角范围也是有限的。超过这个范围后：

1. 某些维度的角度会绕圈过快。
2. 远距离位置在部分维度上会落到训练未见区间。
3. 注意力分数中的相对位置信号开始失真。

可以把它理解成“圆周尺子”。训练时模型只见过圆周上前一段刻度；现在把位置拉到更远，角度会继续沿圆周转动，某些维度很快转了很多圈。这样一来，位置 5000 和位置 9000 在某些维度上可能看起来并不比 5000 和 5001 更容易区分。

下面这个表把边界讲清楚：

| 字段 | 含义 | 典型取值 |
|---|---|---|
| 训练上下文 | 预训练或指令微调时见过的最大长度 | 4K、8K |
| 目标上下文 | 希望部署时支持的最大长度 | 32K、128K、200K |
| 允许调整 | 不重训全模型时，允许动哪些部分 | RoPE 频率、position id 生成规则、少量继续训练 |
| 不希望动的部分 | 尽量保持已有语义能力稳定 | 主体权重、短文本能力、推理吞吐 |

一个常见误解是：“既然 RoPE 是公式，那把 `max_position` 改大不就行了？”不行。因为公式能算出值，不代表模型见过这些值。模型并不是在学公式本身，而是在学“这种位置分布下如何形成正确注意力”。所以长上下文扩展的真实问题是：

$$
\text{在不完全重训模型的前提下，如何让超长位置的旋转分布仍落在模型可泛化的区间内。}
$$

这也是三种策略的共同出发点。它们都在回答同一个问题：当目标长度远大于训练长度时，应该改“频率”、改“当前推理缩放”，还是改“位置编号方式”。

玩具例子：一个 4K 模型写摘要时，前 3000 token 是文章正文，最后 1000 token 是结论。原生 RoPE 在 4K 内通常还能工作。现在把输入换成 32K 的法律合同，核心条款在第 20000 token，补充条款在第 28000 token。若位置编码没有被重新设计，模型看到的远距相位关系就可能混乱，最终表现为“能读到词，但抓不住跨页引用关系”。

真实工程例子：企业知识库问答经常把几十篇文档拼成一个长 prompt。失败时，模型不是完全答不出来，而是会出现“前文事实抓得住，后文事实漏掉”或“把两个远距离段落混成一个来源”的现象。很多时候，这就是长上下文位置编码失真带来的注意力退化，而不是检索本身出了问题。

## 核心机制与推导

### 1. PI / YaRN：把长位置压回训练角度范围

PI 是 Positional Interpolation，位置插值。它的基本思想是：如果目标长度是训练长度的 $s$ 倍，那么把所有位置按比例缩小到原窗口里，再交给 RoPE。

设训练长度为 $L_{\text{train}}$，目标长度为 $L_{\text{new}}$，则缩放因子是：

$$
s=\frac{L_{\text{new}}}{L_{\text{train}}}
$$

如果 $L_{\text{train}}=4096$，$L_{\text{new}}=32768$，那么：

$$
s=\frac{32768}{4096}=8
$$

原来位置 $p$ 的角度大致由 $p\omega_i$ 决定，其中 $\omega_i$ 是第 $i$ 个频率。PI 可以等价理解成两种方式：

1. 缩位置：把 $p$ 变成 $p/s$
2. 缩频率：把 $\omega_i$ 变成 $\omega_i'=\omega_i/s$

两者本质等价，因为：

$$
p\omega_i' = p\cdot \frac{\omega_i}{s} = \frac{p}{s}\omega_i
$$

这一步的含义是：让 32768 长度上的最大旋转角，回到 4096 长度时模型熟悉的范围内。于是模型不会一下子面对一整片完全陌生的高角度区域。

但 PI 不是白送的。因为你把所有频率都压缩了，高频也被一起压缩，局部细节会变钝。所以实际工程里通常不是“直接改完就上线”，而是做少量继续训练，让模型重新适应这个新坐标系。YaRN 可以看成 PI 的工程增强版，它在频率缩放之外，还会搭配更细致的训练配方，比如分阶段拉长序列、调整采样比例或温度，减轻短文本能力退化。

### 2. 动态 NTK：按当前长度实时改缩放

动态 NTK 的思路不同。它不要求先把模型拿去做长上下文微调，而是在推理时根据当前实际输入长度 $l$，动态算一个缩放系数。

常见写法是先定义：

$$
\hat{\alpha} = \max\left(1, \frac{l}{L_{\text{train}}}\right)
$$

它表示“当前序列相对训练长度放大了多少”。如果当前只输入了 3000 token，而训练长度是 4096，那么 $\hat{\alpha}=1$，不做修改；如果当前输入到 8192，那么 $\hat{\alpha}=2$。

进一步，在部分实现里会用它改写 RoPE 的 base，形式可以写成：

$$
\theta_i' = \left(base \cdot \hat{\alpha}^{d/(d-2)}\right)^{-2i/d}
$$

这里：

| 变量 | 含义 |
|---|---|
| $l$ | 当前推理实际长度 |
| $L_{\text{train}}$ | 原始训练长度 |
| $d$ | 头维度，也就是单个注意力头的通道数 |
| $i$ | 频率维度索引 |
| $base$ | 原始 RoPE 底数，常见为 10000 |

白话解释：当前文本越长，就把低频范围往更远处展开一些；短文本则尽量保持原始分布，不破坏模型原有能力。

它的优点是部署简单。缺点也直接：模型参数没适应过这种变化，只是推理时临时换了坐标系，所以通常会有可见精度损失。尤其当 prefill 很长、KV cache 已建立、后续继续生成时，如果不同阶段的旋转 base 处理不一致，还会引入额外误差。

### 3. PoSE：块内连续，块间跳跃

PoSE 这里指位置跳跃类思路。核心不是缩频率，而是改 position id 的生成方式。

假设把一篇 128K 长文切成若干块，每块大小为 1024。块内位置保持稠密连续，比如：

- 第 1 块：0 到 1023
- 第 2 块：4096 到 5119
- 第 3 块：8192 到 9215

也就是块与块之间故意跳跃。

为什么这样有意义？因为 RoPE 的低频维度变化慢，负责远距离结构。如果块间 position id 跳得足够远，低频维度就会感知到“这些块相隔很远”；但块内仍然连续，所以局部阅读能力基本保留。于是模型既不需要把整个频谱统一压缩，也不必完全依赖推理时动态改 base。

可以把块记为 $B_k$，每块长度为 $m$，起始偏移为 $o_k$。那么块内第 $j$ 个 token 的位置可以写成：

$$
p_{k,j}=o_k + j,\quad 0 \le j < m
$$

若相邻块的偏移差 $o_{k+1}-o_k$ 明显大于 $m$，就形成“跳跃位置”。

### 4. 三种方法的本质差别

1. PI / YaRN 改的是“全局频率尺度”，把全部位置重新映射到可训练区间。
2. 动态 NTK 改的是“当前推理时的频率底数”，按实际长度自适应。
3. PoSE 改的是“位置编号策略”，让块级距离显式拉开。

它们不是互斥关系。现实系统常常组合使用，比如先做 PI/YaRN 微调，再在数据构造或训练样本中加入块式位置策略，进一步增强超长跨度建模。

## 代码实现

下面先给一个玩具实现，用 Python 展示三种策略的核心变换。这里不实现完整 Transformer，只验证“位置如何被改写”。

```python
import math

def positional_interpolation_position(pos, train_len, target_len):
    scale = target_len / train_len
    return pos / scale

def dynamic_ntk_alpha(current_len, train_len):
    return max(1.0, current_len / train_len)

def dynamic_ntk_base(base, current_len, train_len, head_dim):
    alpha = dynamic_ntk_alpha(current_len, train_len)
    exponent = head_dim / (head_dim - 2)
    return base * (alpha ** exponent)

def pose_positions(num_blocks, block_size, stride):
    positions = []
    for k in range(num_blocks):
        start = k * stride
        block = list(range(start, start + block_size))
        positions.append(block)
    return positions

# 玩具例子：4K 训练扩到 32K
train_len = 4096
target_len = 32768

# PI: 32768 位置会被压回 4096
assert positional_interpolation_position(32768, train_len, target_len) == 4096

# 动态 NTK: 当前长度 8192 时，alpha=2
assert dynamic_ntk_alpha(8192, train_len) == 2.0
assert dynamic_ntk_alpha(2048, train_len) == 1.0

# PoSE: 块内连续，块间跳跃
blocks = pose_positions(num_blocks=2, block_size=4, stride=16)
assert blocks[0] == [0, 1, 2, 3]
assert blocks[1] == [16, 17, 18, 19]

# base 会随当前长度增大
base_4k = dynamic_ntk_base(10000, 4096, 4096, 128)
base_8k = dynamic_ntk_base(10000, 8192, 4096, 128)
assert base_8k > base_4k
```

如果把它翻译成工程伪代码，通常是下面三类流程。

### 1. PI / YaRN 的实现步骤

```python
def build_scaled_rope_frequencies(inv_freq, train_len, target_len):
    # inv_freq: 原始 RoPE 频率数组，通常按维度递减
    scale = target_len / train_len
    # 白话解释：目标长度更长，就把频率整体放慢
    return inv_freq / scale
```

落地时通常发生在模型加载阶段，或者构建 rotary embedding cache 时。然后再用长序列数据做少量训练，让模型适应新频率。

### 2. 动态 NTK 的实现步骤

```python
def update_rope_for_prefill(base, current_len, train_len, head_dim):
    alpha = max(1.0, current_len / train_len)
    new_base = base * (alpha ** (head_dim / (head_dim - 2)))
    return new_base
```

这里要注意时机。一般是在 prefill 前，根据“当前这一批输入有多长”更新；如果系统允许超长续写，还要确保 KV cache 中已经写入的 key 使用了同一套规则，否则前后 token 的位置坐标系不一致。

### 3. PoSE 的实现步骤

```python
def make_skipped_position_ids(total_len, block_size, stride):
    position_ids = []
    num_blocks = math.ceil(total_len / block_size)
    for b in range(num_blocks):
        start = b * stride
        for j in range(block_size):
            pos = start + j
            position_ids.append(pos)
            if len(position_ids) == total_len:
                return position_ids
    return position_ids
```

输入输出关系如下：

| 方法 | 输入 | 输出 |
|---|---|---|
| PI / YaRN | `train_len`, `target_len`, 原始频率数组 | 缩放后的频率数组 |
| 动态 NTK | `current_len`, `train_len`, `base` | 当前推理使用的新 base |
| PoSE | `block_size`, `stride`, `total_len` | 跳跃后的 position id 序列 |

真实工程例子：在 `llama.cpp` 这类推理框架里，动态缩放通常暴露成命令行参数，例如 `--rope-scaling`、`--yarn-ext-factor` 一类开关。它的优点是“开了就能试”，适合快速验证模型是否能从 4K 拉到 8K、16K；缺点是如果没有对应的长上下文继续训练，评测集和真实任务里通常会看到明显退化。

## 工程权衡与常见坑

最常见的工程结论是：只想临时试长输入，用动态 NTK；要稳定上线，用 PI / YaRN 加继续训练；要冲更极端长度，再考虑 PoSE 类位置策略与训练配方联动。

下面是常见问题表：

| 问题 | 原因 | 补救 |
|---|---|---|
| PI 直接上线后短文本变差 | 高频被统一压缩，局部相对位置变钝 | 用目标长度数据做少量继续训练，最好分阶段拉长 |
| 动态 NTK 长文能跑但回答变散 | 模型未在这种动态频率分布下训练过 | 只作为推理补丁，不作为高精度方案 |
| KV cache 前后不一致 | prefill 和 decode 阶段使用了不同 base | 固定同一次会话的 rope 参数，核查缓存构建逻辑 |
| PoSE 跳跃太大 | 块间距离被夸大，低频连续性反而受损 | 调整 stride，结合训练样本做校准 |
| 只看“最大支持长度”指标 | 长度能跑通不等于检索精度稳定 | 做 needle-in-a-haystack、长摘要、跨段引用测试 |

一个典型坑是 4K 模型直接做 32K PI 压缩但不微调。这样确实能“看到”32K 输入，但模型在短文任务里可能变糊。原因不复杂：原先用于表达近距离细节的高频，也被一起压慢了。对新手来说，可以把它理解成“显微镜调低了倍率”，远处能看见了，近处细节却糊了。

另一个常见坑出现在动态 NTK。很多实现是“prefill 时按总长度改一次 rope 参数”，但如果后续继续生成、继续拼接上下文，rope base 处理不严谨，旧 KV 和新 token 会混在两套坐标系里。表现出来往往不是直接报错，而是模型生成后段质量突然下降、重复、遗漏引用或事实对齐失稳。

实际落地时可以按这个简短流程做：

1. 先确定目标长度，比如 4K 扩到 32K 还是 128K。
2. 如果只是验证可行性，先试动态 NTK，快速测长文摘要和长检索。
3. 如果要上线，改成 PI / YaRN，并做 sequence length curriculum，也就是序列长度课程学习，指训练时先短后长逐步拉大窗口。
4. 如果任务包含超长文档、多块证据聚合，再评估 PoSE 或块式 position id。
5. 最后专门核查 KV cache 一致性、短文本退化和跨段检索精度。

建议至少测试这些项目：

- 4K 内短问答是否回退
- 32K 长摘要是否丢尾部信息
- 关键句跨 20K 以上距离时能否正确引用
- 连续多轮对话下 KV cache 是否稳定
- 不同 chunk 拼接顺序下答案是否敏感

## 替代方案与适用边界

如果需求只是从 4K 提到 8K、16K，动态 NTK 往往足够做原型验证。如果需求是生产级 32K、128K，主流还是 PI / YaRN 类方案加继续训练。如果需求进一步走向 200K 甚至更长，位置编码本身已经不是唯一问题，还会牵涉数据配方、训练样本结构、注意力稀疏化、检索增强与长文任务设计。

下面这个表可以直接用于选型：

| 方案 | 所需资源 | 精度影响 | 适用长度 |
|---|---|---|---|
| 动态 NTK | 最低，只改推理参数 | 中等退化 | 试验性 8K 到 32K |
| PI | 需要少量长序列继续训练 | 较稳 | 32K 常用 |
| YaRN | 比 PI 更完整的工程 recipe | 通常更稳 | 32K 到 128K |
| PoSE | 需要位置生成逻辑和训练配合 | 取决于设计 | 极长文档、块式任务 |
| 直接长窗训练 | 资源最高 | 上限最好 | 顶级闭源或高预算训练 |

一个现实边界是：位置编码只能解决“模型如何表示更远的位置”，不能单独解决“模型是否有足够记忆能力”。比如 128K 输入里放了几十篇文档，即便位置编码稳定，模型也未必能自动学会多跳检索、证据聚合和冲突消解。所以很多系统在长上下文之外，还会叠加检索、重排或摘要压缩。

QwenLong-L1.5 这类路线更接近“长上下文 post-training”。也就是不只改位置编码，还通过长序列数据合成、强化学习、记忆相关训练目标，让模型在更长窗口里学会真正利用信息。Kimi 128K 的公开分析也体现了类似思想：不是一步到位训 128K，而是通过 4K → 32K → 128K 的 curriculum 逐步拉长。对初学者来说，这可以理解成“先练短距离记忆，再练中距离，最后练长距离”，而不是第一天就要求模型处理整本书。

Claude 200K 一类超长上下文系统，则更像“从训练阶段就大量接触长文任务”。这条路最贵，但在稳定性上通常最好，因为模型不是靠补丁适应长窗口，而是在训练中真的学会了长跨度建模。

因此可以把替代路径概括为：

$$
4K \rightarrow 32K \rightarrow 128K
$$

每一步都包含两件事：

1. 调整位置编码策略
2. 用相应长度的数据重新适配模型

如果只有第 1 步，没有第 2 步，通常很难得到稳定结果。

用途：提供 PI 的核心机制、缩放因子 $s=L_{\text{new}}/L_{\text{train}}$、以及“把长位置压回训练角度范围”的基本思想。  
链接：https://www.emergentmind.com/articles/2306.15595

2. Linsight: *Dynamically NTK Scaled RoPE*  
用途：补充动态 NTK 的推理时缩放思路、当前长度因子设计，以及 KV cache 一致性的工程注意事项。  
链接：https://www.linsight.cn/c4da56c0.html

3. QwenLong-L1.5 相关论文与工程说明  
用途：支撑“长上下文能力通常仍依赖 post-training、长序列数据与训练配方”，而不是只改推理参数。  
链接：https://huggingface.co/papers/2512.12967

4. Kimi 长上下文公开资料与使用说明  
用途：作为工程案例，说明 128K 级上下文通常配合 YaRN 类思路和课程式扩展，而不是纯推理补丁。  
链接：https://kimi-ai.chat/guide/manages-context-windows/

5. 长上下文扩展的工程与评测讨论（含精度损失与常见问题）  
用途：支撑“动态缩放方便但通常存在精度损失”“KV cache 与位置缩放要一起校验”等工程结论。  
链接：https://www.mdpi.com/2741726

6. Claude 长上下文能力相关公开文档与分析  
用途：作为“高预算路线”的参照，说明直接在长文任务上训练或后训练，是获得稳定 200K 上下文能力的重要条件。  
说明：适合用于理解工程边界，不应简单等同于单一位置编码技巧。
