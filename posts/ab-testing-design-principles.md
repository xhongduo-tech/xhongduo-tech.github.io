## 核心结论

A/B 测试是随机对照实验：把流量随机分到对照组 `C` 和实验组 `T`，用线上真实行为估计一次产品或策略改动对核心指标的因果效应 `δ`。

它的目标不是比较“哪个版本总量更高”，而是估计“如果同一类用户看到新版本，指标平均会改变多少”。常用估计量是两组均值差：

$$
\hat{\delta}=\bar{Y}_T-\bar{Y}_C
$$

其中 `δ̂` 是估计出来的效果，`\bar{Y}_T` 是实验组平均指标，`\bar{Y}_C` 是对照组平均指标。

新手版解释：把用户随机分到两个组，A 组看旧版本，B 组看新版本。如果两组起点尽量一样，那么最后指标差异更可能来自新版本本身，而不是用户本来就不同。随机化是因果推断的前提。

| 实验目标 | 错误理解 |
|---|---|
| 估计新版本对核心指标的因果效应 `δ` | 看实验组总订单数是否更高 |
| 保证两组用户在实验前尽量可比 | 只要样本量大就一定可信 |
| 提前定义主指标、分流规则、停止规则 | 跑完后挑一个显著指标汇报 |
| 判断效果是否稳定、可上线 | 看到一次 p-value 小于 0.05 就上线 |

设计优先于分析。随机化是否成立、实验单元是否选对、是否存在串扰，决定了结论能不能信。统计检验只能处理“随机误差”，不能修复错误分流、错误埋点和错误实验单元。

---

## 问题定义与边界

本文默认单次实验、二元处理、主要看平均处理效应。二元处理指用户只会处在两种状态之一：旧版本或新版本。平均处理效应指新版本让一组用户的平均指标提高或降低了多少。

先定义三个符号：

| 符号 | 含义 |
|---|---|
| `Y_i` | 第 `i` 个实验单元的观测指标，例如是否下单、收入、停留时长 |
| `Z_i` | 第 `i` 个实验单元的处理变量，`Z_i=1` 表示进入实验组，`Z_i=0` 表示进入对照组 |
| `δ` | 新版本相对旧版本带来的真实平均效果 |

实验单元是被随机分配的对象。它不一定是用户，也可能是会话、订单、城市、时间块或地理块。随机化单位是实际执行随机分配的维度。两者最好一致，否则容易出现分析和分流不匹配。

| 实验单元 | 适用场景 | 风险 |
|---|---|---|
| 用户 | 页面改版、推荐卡片样式、会员弹窗 | 多设备登录、跨端污染 |
| 会话 | 搜索结果页、短期交互实验 | 同一用户多次会话相关，不完全独立 |
| 订单 | 优惠策略、履约策略 | 一个用户多单会造成相关性 |
| 时间块 | 打车调度、外卖派单、库存策略 | 受高峰期、节假日影响大 |
| 地理块 | 本地生活、物流、供需匹配 | 区域之间可能相互影响 |
| 商家或内容 | 广告、货架、内容分发 | 流量不均导致方差大 |

普通用户级 A/B 的适用边界是：组间近似独立、无明显干扰、效果相对稳定。组间独立是指实验组用户的体验不会改变对照组用户的结果。干扰是指一个单元接受处理后，会影响另一个单元的指标。

新手版解释：如果你改的是外卖派单策略，用户会影响骑手，骑手又影响订单履约。此时按用户随机会让对照组也被实验组影响，结果就不准了。因为对照组不再是真正的“旧策略环境”。

## 核心机制与推导

A/B 测试的核心估计量是两组均值差：

$$
\hat{\delta}=\bar{Y}_T-\bar{Y}_C
$$

如果随机分组成立，实验组和对照组在实验开始前的用户结构期望相同。也就是说，年龄、活跃度、购买力、设备类型等因素不会系统性偏向某一组。因此，实验后的均值差在期望上只来自处理变量 `Z_i`。这就是均值差可以作为无偏估计的原因。无偏估计的白话解释是：重复做很多次同样设计的实验，估计值的平均会接近真实效果。

二元指标是取值只有 0 和 1 的指标，例如是否点击、是否下单、是否留存。若基线转化率为 `p`，希望检测最小可检测绝对提升 `Δ`，显著性水平为 `α`，功效为 `1-β`，近似等样本、双边检验时，单组样本量可估为：

$$
n \approx \frac{2p(1-p)(z_{1-\alpha/2}+z_{1-\beta})^2}{\Delta^2}
$$

`α` 是把无效版本误判为有效的容忍概率，功效是版本真的有效时能检出的概率。`Δ` 是业务认为值得上线的最小提升，不是你希望看到的最大提升。

玩具例子：基线下单率 `p=10%`，希望检测绝对提升 `Δ=2%`，取 `α=0.05`、功效 `80%`，单组样本量约为 3528，总样本约 7056。若只收集 500 个用户，即使新版本看起来高一点，也很可能只是随机波动。

不同实验设计解决不同问题：

| 设计 | 解决的问题 | 核心做法 | 适用例子 |
|---|---|---|---|
| 标准 A/B | 单一改动，组间独立 | 用户或会话随机分流 | 新按钮是否提升下单率 |
| 正交实验 | 同时测试多个因素 | 用设计矩阵拆分主效应 | 同时测按钮颜色、文案、位置 |
| Switchback | 强串扰、供需耦合、时变系统 | 按时间块或地理块切换策略 | 打车调度、外卖派单 |
| Interleaving | 排序、搜索、推荐结果比较 | 同一页面交叉展示两套排序 | 搜索排序 A 与 B 谁更好 |

正交实验中的“正交”可以理解为因素之间尽量不互相混淆。若设计矩阵 `X` 满足 `X^TX` 近似对角，就更容易分开估计颜色、文案、位置各自的主效应。

真实工程例子：外卖平台想改派单策略。用户、骑手、商家、订单会互相影响，按用户随机会让实验组策略改变骑手位置，从而影响对照组订单。更稳妥的做法是按“城市区块 × 15 分钟”做 switchback，同一时间块整块使用同一策略，再按块聚合比较履约率、等待时长、取消率。

## 代码实现

代码实现要覆盖实验全链路，不只是“算 p-value”。至少要包含稳定分流、曝光记录、转化采集、指标聚合、SRM 检查和置信区间。SRM 是 Sample Ratio Mismatch，意思是样本比例不匹配，例如预期 50:50，实际却是 60:40。

流程如下：

| 阶段 | 目的 | 失败后果 |
|---|---|---|
| 分流 | 让同一用户稳定进入同一组 | 用户体验混乱，实验污染 |
| 曝光 | 确认用户真的看到了版本 | 把未曝光用户算入分析 |
| 采集 | 记录转化、收入等指标 | 指标缺失或偏差 |
| 校验 | 检查 SRM、埋点、过滤条件 | 坏数据进入分析 |
| 分析 | 计算 delta、置信区间、显著性 | 结论不可复现 |
| 结论 | 按预设规则决定上线、回滚或继续 | 事后挑结果 |

伪代码结构：

```js
// 1. 按 userId 做稳定分流
const group = stableHash(userId + experimentId) % 100 < 50 ? "C" : "T";

// 2. 记录曝光和转化
logExposure({ experimentId, userId, group, configVersion });
logConversion({ experimentId, userId, converted, revenue });

// 3. 聚合两组指标
const metrics = aggregateByGroup(events);

// 4. 检查 SRM
checkSampleRatio(metrics.C.count, metrics.T.count, expectedRatio = 0.5);

// 5. 计算 delta 和置信区间
const delta = metrics.T.mean - metrics.C.mean;
const ci = confidenceInterval(metrics.C, metrics.T);
```

可运行的 Python 玩具实现如下，包含稳定分流、SRM 检查、均值差和置信区间：

```python
import hashlib
import math
import random

def assign_group(user_id: str, experiment_id: str) -> str:
    key = f"{experiment_id}:{user_id}".encode()
    h = int(hashlib.md5(key).hexdigest(), 16)
    return "T" if h % 100 < 50 else "C"

def mean(xs):
    return sum(xs) / len(xs)

def analyze(rows):
    c = [r["y"] for r in rows if r["group"] == "C"]
    t = [r["y"] for r in rows if r["group"] == "T"]

    ratio = len(t) / (len(c) + len(t))
    assert 0.45 <= ratio <= 0.55, f"SRM risk: treatment ratio={ratio:.3f}"

    p_c, p_t = mean(c), mean(t)
    delta = p_t - p_c
    se = math.sqrt(p_c * (1 - p_c) / len(c) + p_t * (1 - p_t) / len(t))
    ci95 = (delta - 1.96 * se, delta + 1.96 * se)
    return delta, ci95, len(c), len(t)

random.seed(7)
rows = []
for i in range(8000):
    user_id = f"user_{i}"
    group = assign_group(user_id, "checkout_button_v2")
    base = 0.10
    lift = 0.02 if group == "T" else 0.0
    y = 1 if random.random() < base + lift else 0
    rows.append({"user_id": user_id, "group": group, "y": y})

delta, ci95, n_c, n_t = analyze(rows)

assert n_c + n_t == 8000
assert ci95[0] < delta < ci95[1]
print(round(delta, 4), tuple(round(x, 4) for x in ci95), n_c, n_t)
```

工程上最重要的是可重复和可审计。随机种子、哈希规则、实验配置版本、样本过滤条件、开始结束时间都要落地。否则同一批数据今天算一个结果，明天换个过滤条件又算出另一个结果，实验结论就无法被复核。

## 工程权衡与常见坑

最常见的问题不是统计公式错了，而是数据和实验设计坏了。SRM、埋点丢失、过滤偏差、实验配置漂移，会直接让结果失真。时间因素和多重比较也经常被低估。没有覆盖完整业务周期，或者频繁偷看结果、反复挑指标，都会夸大显著性。

新手版解释：如果周末用户和工作日用户行为差异很大，而实验只跑了 2 天，就可能把“周末效应”误认为“版本提升”。

| 问题 | 表现 | 排查顺序 | 规避办法 |
|---|---|---|---|
| SRM | 预期 50:50，实际明显偏离 | 分流日志、曝光日志、过滤条件、配置版本 | 上线前做 AA 测试，分析前先查样本比例 |
| 串扰 | 对照组也被实验策略影响 | 用户路径、共享资源、供需关系 | 改用 switchback、簇随机或更大随机单元 |
| 指标不灵敏 | 样本很多但波动仍大 | 指标方差、事件频率、归因窗口 | 选择更近因指标，使用 interleaving 或协变量调整 |
| 偷看 | 中途看到显著就停止 | 查询记录、停止规则、看板权限 | 预注册实验周期，使用序贯检验方法 |
| 多重比较 | 测很多指标后挑显著项 | 指标列表、分组维度、探索分析记录 | 预设主指标，必要时做校正 |
| 季节性 | 周末、节假日、高峰期影响结论 | 时间分布、业务周期、历史基线 | 覆盖完整周期，按时间分层分析 |
| 埋点丢失 | 曝光数、点击数异常下降 | SDK 版本、端类型、网络失败率 | 埋点验收，设置数据质量报警 |
| 配置漂移 | 实验期间规则被改动 | 配置发布时间、灰度记录 | 固化配置版本，变更需重新开实验 |

工程权衡的核心是：越严格的实验设计，通常越慢、越贵、越影响业务排期；越快的实验，通常越容易受噪声、串扰和偏差影响。不要把统计显著性当成唯一目标。一个提升 0.1% 但会增加系统复杂度和长期维护成本的方案，未必值得上线。

## 替代方案与适用边界

当普通用户级 A/B 不适用时，要按问题类型换方法，不要硬套。替代方案不是更高级，而是针对特定约束的折中。选型标准应该是：随机单元是否合理、是否存在干扰、方差是否过大、业务是否允许切换。先判断约束，再选方法。

| 判断问题 | 若答案为是 | 推荐设计 | 例子 |
|---|---|---|---|
| 是否存在强串扰 | 是 | Switchback 或簇随机 | 外卖派单、打车调度 |
| 是否明显时变 | 是 | 分层 A/B 或 switchback | 高峰期供需策略 |
| 是否同时测试多因素 | 是 | 正交实验或因子实验 | 多个运营按钮一起测 |
| 是否是排序任务 | 是 | Interleaving | 搜索排序、推荐排序 |
| 是否组间独立且单一改动 | 是 | 标准用户级 A/B | 普通页面改版 |

外卖派单更适合 switchback，因为同一时间同一区域内的订单和骑手互相影响。搜索排序更适合 interleaving，因为同一用户在同一次搜索中同时比较两套排序，方差更低。多个运营按钮一起测更适合正交实验，因为它能拆开颜色、文案、位置的影响。普通页面改版通常还是标准 A/B，因为用户之间影响较弱，分流和解释都更简单。

边界也要明确。Switchback 会引入时间粒度选择问题，时间块太短可能来不及生效，太长又降低样本量。Interleaving 主要适合排序列表，不适合两个完全不同的页面体验。正交实验适合估计主效应，但如果因素之间存在强交互，就需要更大的设计和更多样本。
