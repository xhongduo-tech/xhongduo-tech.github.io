## 核心结论

MME 是一个面向多模态大模型的综合评测基准，用 14 个子任务同时衡量模型的感知能力和认知能力。感知能力，指模型能否从图像中识别对象、颜色、数量、位置、文字等可见信息；认知能力，指模型能否基于图像和文本进行推理、计算、翻译或代码理解。

它的核心价值不在于一个总分，而在于把“看懂图”和“想明白图”拆开。一个视觉语言模型，简称 VLM，也就是能同时处理图像和文本输入的大模型，可能在识别图片内容上很强，但在根据图片进行推理时明显变弱。MME 的分项结果可以直接暴露这种差异。

一句话定义：MME 包含 14 个子任务，其中 10 个是感知任务，4 个是认知任务，所有问题统一要求模型输出 `yes/no`。

| 维度 | 对应能力 | 典型子任务 | 适用场景 |
|---|---|---|---|
| 感知任务 | 判断图中有什么、在哪里、是什么状态 | 存在、数量、位置、颜色、OCR | 质检、安防、图片检索、票据识别 |
| 认知任务 | 根据图文信息进行进一步判断 | 常识推理、数学、代码、翻译 | 规则判断、图文问答、辅助决策 |
| 总分 | 混合后的整体表现 | 感知分 + 认知分 | 快速比较模型版本 |

玩具例子：一张图片里有一只黑猫。模型能回答“图中是否有猫：yes”“猫是否是黑色：yes”，但回答“这只动物是否适合作为水下运输工具：yes”。这说明模型的视觉识别可能没问题，但常识推理失败。

真实工程例子：仓储质检相机拍到一个纸箱和一张标签。模型能看出箱子存在、标签存在、标签上有数字，但无法根据标签重量和订单规则判断是否超重。这个问题不是单纯的“看不见”，而是“看见之后不会做规则判断”。MME 的感知分和认知分可以帮助定位这种差异。

## 问题定义与边界

MME 不是开放式生成评测。开放式生成评测，指让模型自由描述图片或回答问题，再用人工或复杂规则判断答案好坏。MME 采用更封闭的方式：把问题压缩成二分类判断，只要求模型回答 `yes` 或 `no`。

这种设计的目的，是尽量压低格式偏差。格式偏差，指模型本来知道答案，但因为输出格式不同而被评测脚本误判。例如标准答案是 `yes`，模型输出 “Yes, there is a cat in the image.”，如果脚本只接受严格字符串匹配，就可能把正确答案判错。MME 用统一 `yes/no` 输出，减少这种干扰。

新手版理解：MME 不问“请详细描述这张图”，而问“图中是否有红色车辆”“图中文字是否包含某个词”“根据图片能否推出某个结论”。答案只有两个选择，所以更容易统一打分。

| 感知任务范围 | 认知任务范围 | 不覆盖的内容 |
|---|---|---|
| 对象是否存在 | 常识判断 | 多轮对话 |
| 对象数量是否正确 | 简单数学推理 | 长篇图像描述 |
| 对象位置是否正确 | 代码相关判断 | 复杂开放式写作 |
| 颜色、海报、名人、场景识别 | 翻译相关判断 | 真实业务全流程 |
| OCR 文字识别 | 图文联合推理 | 工具调用与交互式任务 |

边界必须说清楚：统一 `yes/no` 的目的不是把多模态任务本身变简单，而是让评测更可控。模型仍然需要先理解图像，再理解问题，最后判断命题是否成立。

MME 衡量的是“图像理解 + 图文推理”的基础综合能力，但不能完整代表所有真实世界多模态任务。如果目标任务需要长文本描述、连续追问、多轮澄清、拒答策略或复杂业务流程，仅靠 MME 不够。

例如客服图文问答中，用户可能先上传商品破损图片，再追问“能不能退货”，然后补充订单规则。MME 可以说明模型的基础视觉理解和部分推理能力，但不能直接证明模型能处理完整客服流程。

## 核心机制与推导

MME 的计分可以理解为“每个子任务先算准确率，再换算成 200 分”。准确率，指答对题数占总题数的比例。

记第 $t$ 个子任务的准确率为 $Acc_t$，则该子任务得分为：

$$
S_t = 200 \times Acc_t
$$

感知任务集合记为 $P$，认知任务集合记为 $C$，则：

$$
S_{perception} = \sum_{t \in P} S_t
$$

$$
S_{cognition} = \sum_{t \in C} S_t
$$

$$
S_{total} = S_{perception} + S_{cognition}
$$

MME 有 10 个感知子任务，每个满分 200，所以感知总分满分是 2000。它有 4 个认知子任务，每个满分 200，所以认知总分满分是 800。总分满分是 2800。

| 类别 | 子任务 | 测什么 |
|---|---|---|
| 感知 | Existence | 图中是否存在某个对象 |
| 感知 | Count | 对象数量是否正确 |
| 感知 | Position | 对象位置关系是否正确 |
| 感知 | Color | 对象颜色是否正确 |
| 感知 | Posters | 海报内容理解 |
| 感知 | Celebrity | 名人识别 |
| 感知 | Scene | 场景识别 |
| 感知 | Landmark | 地标识别 |
| 感知 | Artwork | 艺术作品识别 |
| 感知 | OCR | 图中文字识别 |
| 认知 | Commonsense Reasoning | 常识推理 |
| 认知 | Numerical Calculation | 数值计算 |
| 认知 | Text Translation | 文本翻译 |
| 认知 | Code Reasoning | 代码理解与推理 |

玩具数值例子：某个感知子任务有 10 道题，模型答对 8 道，则：

$$
Acc = \frac{8}{10} = 0.8
$$

$$
S = 200 \times 0.8 = 160
$$

所以该子任务得分是 160/200。

更接近工程分析的例子：如果一个模型感知分是 1790.04/2000，认知分是 643.57/800，那么不能只看两个原始分数。因为两类任务满分不同，应该先归一化。归一化，指把不同满分的分数统一换算成百分比。

$$
\frac{1790.04}{2000} = 89.50\%
$$

$$
\frac{643.57}{800} = 80.45\%
$$

两者差距是 9.05 个百分点。这说明该模型在感知任务上明显强于认知任务。直接看 raw total，也就是原始总分，容易掩盖这种结构性差异。

统一成 `yes/no` 后，可以直接统计准确率。因为每道题的输出空间只有两个标准答案，评测脚本只需要判断模型输出是否等于标准答案。这样减少了“答案语义正确但表达形式不同”的问题。

## 代码实现

MME 评测脚本的本质是三步：加载题目索引，读取模型答案，逐题比对并汇总分数。真正容易出错的地方不是公式，而是答案规范化。答案规范化，指把模型输出清洗成统一格式，例如去掉空格、统一大小写、去掉句号。

新手例子：`Yes.`、`YES`、`yes ` 在语义上都表示 `yes`。如果不做规范化，脚本可能把它们当成三个不同字符串。

| 阶段 | 输入字段 | 标准答案字段 | 输出字段 | 汇总字段 |
|---|---|---|---|---|
| 读取题目 | `id`, `task`, `category`, `question` | `gt` | 无 | 无 |
| 读取预测 | `id`, `pred` | 无 | `pred` | 无 |
| 单题计分 | `pred` | `gt` | `correct` | 无 |
| 子任务汇总 | `task`, `correct` | 无 | `task_score` | `Acc_t`, `S_t` |
| 类别汇总 | `category`, `task_score` | 无 | `perception`, `cognition` | `total` |

评测流程可以写成步骤表：

| 步骤 | 动作 | 关键检查 |
|---|---|---|
| 1 | 读取题目文件 | 每题有唯一 `id` |
| 2 | 加载模型输出 | 每题都有预测答案 |
| 3 | 规范化 `yes/no` | 大小写、空格、标点一致 |
| 4 | 逐题计分 | 预测等于标准答案记 1 |
| 5 | 按子任务汇总 | 每个子任务计算准确率 |
| 6 | 按类别汇总 | 分别计算感知、认知、总分 |

下面是一段可运行的最小 Python 示例，模拟 MME 的计分逻辑：

```python
from collections import defaultdict

TASK_CATEGORY = {
    "existence": "perception",
    "count": "perception",
    "ocr": "perception",
    "commonsense_reasoning": "cognition",
    "numerical_calculation": "cognition",
}

def normalize(ans):
    ans = ans.strip().lower()
    ans = ans.rstrip(".。!！")
    if ans in {"yes", "y"}:
        return "yes"
    if ans in {"no", "n"}:
        return "no"
    return ans

def score_item(pred, gt):
    return 1 if normalize(pred) == normalize(gt) else 0

def score_task(items):
    if not items:
        return 0.0
    acc = sum(score_item(x["pred"], x["gt"]) for x in items) / len(items)
    return 200 * acc

def evaluate(rows):
    by_task = defaultdict(list)
    for row in rows:
        by_task[row["task"]].append(row)

    task_scores = {
        task: score_task(items)
        for task, items in by_task.items()
    }

    category_scores = defaultdict(float)
    for task, score in task_scores.items():
        category = TASK_CATEGORY[task]
        category_scores[category] += score

    total = sum(task_scores.values())
    return task_scores, dict(category_scores), total

rows = [
    {"task": "existence", "pred": "Yes.", "gt": "yes"},
    {"task": "existence", "pred": "no", "gt": "yes"},
    {"task": "count", "pred": "YES", "gt": "yes"},
    {"task": "count", "pred": " no ", "gt": "no"},
    {"task": "commonsense_reasoning", "pred": "yes", "gt": "no"},
    {"task": "commonsense_reasoning", "pred": "no.", "gt": "no"},
]

task_scores, category_scores, total = evaluate(rows)

assert task_scores["existence"] == 100.0
assert task_scores["count"] == 200.0
assert task_scores["commonsense_reasoning"] == 100.0
assert category_scores["perception"] == 300.0
assert category_scores["cognition"] == 100.0
assert total == 400.0
```

这段代码对应真实评测链路的简化版。真实工程中，题目通常来自评测集文件，模型输出来自推理结果文件，脚本会按任务名聚合结果，再输出每个子任务、感知类别、认知类别和总分。

工程实现里要额外处理两类情况。第一，模型输出不是单独的 `yes/no`，而是一整句话，需要更严格的解析策略。第二，模型漏答或题号错位，这会让分数失真，必须在评测前检查题目数量和 `id` 对齐。

## 工程权衡与常见坑

MME 的最大优势是结构清晰：感知和认知分开，子任务也分开。因此它适合做模型选型、版本回归和失效定位。但它也有明显工程约束：评测提示词、输出格式、解析规则必须固定，否则分数会混入评测链路误差。

新手例子：两个模型总分接近。模型 A 感知很高、认知很低；模型 B 感知和认知都中等。只看总分会觉得它们差不多，但在业务里完全不同。如果业务是 OCR 和物体存在判断，模型 A 可能更合适；如果业务需要规则推理，模型 B 可能更稳。

真实工程例子：仓储质检更重视箱体缺件、标签 OCR、数量判断，就应优先看感知分。如果系统还要求根据箱体标签判断是否符合发货规则，就必须看认知分，尤其是常识推理和数值计算相关表现。

| 常见坑 | 后果 | 规避方式 |
|---|---|---|
| 只看总分 | 掩盖感知和认知短板 | 同时看感知分、认知分、子任务分 |
| 直接比 raw total | 不同类别满分不同，比较失真 | 按 2000 和 800 分母转成百分比 |
| 不固定输出格式 | 同一模型因格式不同被误判 | 固定提示词、固定解析规则 |
| 忽略答案规范化 | `Yes.` 和 `yes` 被当成不同答案 | 统一大小写、空格和标点 |
| 数据泄漏 | 分数虚高，失去可比性 | 只用官方评测集，避免把题目混入训练或调参 |
| 只测一次 | 采样和推理设置影响结果 | 固定温度、模型版本和推理参数 |

失效定位可以进一步拆成四类：

| 失效类型 | 典型现象 | 优先检查 |
|---|---|---|
| 感知失效 | 看错对象、颜色、数量、文字 | 图像分辨率、视觉编码器、OCR 能力 |
| 认知失效 | 看见信息但推理错误 | 推理提示词、数学能力、规则表达 |
| 格式失效 | 答案含解释导致解析失败 | 输出模板、解析脚本、规范化逻辑 |
| 数据泄漏 | 线下分数异常高但线上无提升 | 训练数据、调参数据、评测集隔离 |

公开题目混入训练或调参会破坏可比性。基准评测的意义在于用模型没见过的题目衡量泛化能力。泛化能力，指模型面对新样本时仍能正确处理的能力。如果模型提前见过测试题，分数就不能代表真实能力。

## 替代方案与适用边界

MME 适合做综合能力体检，但不能单独代表真实应用中的全部需求。它的优势是快速、结构化、容易复现；边界是封闭式、单轮、以 `yes/no` 为主。

可以把 MME 理解成标准体检：它能快速发现模型在哪类基础能力上偏弱，但不能替代专项业务测试。专项业务测试，指围绕具体产品流程、数据分布和错误成本设计的评测集。

| 基准类型 | 主要特点 | 适合回答的问题 | 局限 |
|---|---|---|---|
| MME | 感知/认知拆分，二分类判断 | 模型基础多模态能力是否均衡 | 不衡量长文本生成质量 |
| 开放式图文问答基准 | 答案更自由 | 模型能否自然回答图像问题 | 自动评分更困难 |
| 长链推理基准 | 强调多步推导 | 模型能否进行复杂推理 | 成本更高，速度更慢 |
| 场景任务基准 | 贴近业务流程 | 模型能否解决具体应用问题 | 通用可比性较弱 |
| 人工评测 | 人判断答案质量 | 复杂输出是否真正可用 | 昂贵且一致性难控制 |

适合使用 MME 的情况包括：快速选型、比较不同 VLM 的基础能力、观察新版本相对旧版本的回归、定位模型是感知弱还是认知弱。

不适合只用 MME 的情况包括：替代全部业务测试、评估多轮交互、衡量长文本描述质量、测试拒答策略、验证工具调用流程。

客服图文问答是一个典型边界场景。MME 能说明模型是否具备基本图像理解和图文推理能力，但不能直接证明它在多轮追问、售后规则、用户情绪、合规拒答中的表现。实际落地时，应把 MME 作为基础筛选，再叠加业务数据集和人工抽检。

工程上更稳妥的做法是三层评测：第一层用 MME 看基础能力结构；第二层用公开相关基准看同类任务表现；第三层用业务数据看真实链路表现。这样不会把一个综合基准误用成万能结论。
