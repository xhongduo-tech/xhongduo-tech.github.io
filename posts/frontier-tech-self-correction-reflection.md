## 核心结论

自我修正与反思，不是让模型把同一个问题反复重写，而是显式加入“批评-修复”回路。可以把它理解为一次有检查标准的二次提交：先产出初稿，再指出错因，最后按错因改稿。

它的核心收益有两个。第一，错误被分类型处理，例如“计算错”“漏边界条件”“工具调用参数错”“代码能写但跑不通”。第二，只要外部有验证信号，修复就不再靠运气，而是靠证据。这里的验证信号，指能客观判断对错的反馈，比如单元测试、编译器报错、计算器结果、规则引擎输出。

最常见的有效形态是两轮闭环：

$$
a_0 \xrightarrow{C} c_0 \xrightarrow{R} a_1 \xrightarrow{V} \text{pass/fail}
$$

其中，$a_0$ 是初始答案，$C$ 是批评器，白话说就是“找错的人”；$R$ 是修复器，白话说就是“按错误清单改稿的人”；$V$ 是验证器，白话说就是“最后拍板的人”。

下面这张表可以先把结构看清楚：

| 步骤 | 责任主体 | 输出类型 | 检验方式 |
|---|---|---|---|
| 初始作答 | 生成模型 | 草稿 $a_0$ | 暂不检验，先保留 |
| 批评 | 模型或工具 | 错误条目 $c_0$ | 看是否定位到具体错因 |
| 修复 | 模型 | 修订稿 $a_1$ | 是否覆盖批评中的问题 |
| 验证 | 测试器、规则、工具 | 通过/失败 | 是否满足客观标准 |

结论可以压缩成一句话：自我修正的价值，不在“多想一次”，而在“把错误暴露出来，再按证据修掉”。

## 问题定义与边界

严格定义下，自我修正/反思是：模型先生成答案，再用自身或外部工具显式检测错误，并依据检测结果重写输出。重点在“显式检测”和“依据反馈改写”。如果只是再次采样一次答案，那更像盲目重试，不算完整反思。

这个定义要和两种近似方法区分开：

| 方法 | 是否保留初稿 | 是否显式找错 | 是否有验证信号 | 预期效果 |
|---|---|---|---|---|
| 无修正 | 否 | 否 | 否 | 速度快，但错了就直接错 |
| 盲目重试 | 否 | 否 | 弱 | 偶尔变好，本质靠采样运气 |
| 显式批评-修复 | 是 | 是 | 强 | 在可验证任务上更稳定 |

边界也很明确。它不是万能补丁，而是“已有一定基础正确率时的放大器”。如果初答几乎全错，批评器通常只能重复题目，或者给出模糊提醒，修复器也没有足够正确的骨架可改。实践中可以把它理解为一个阈值问题：

- 当初答低于某个正确率阈值时，批评往往只会说“推理不完整”“请重新检查”，但说不出哪一步错。
- 当初答已经抓住主要结构，只是漏条件、算错一处、参数填错时，批评才有信息密度，修复才有明显收益。

玩具例子很直观。问题是“判断一个数是否为偶数”。如果初稿写成“偶数就是能被 3 整除的数”，那定义层面已经错了，后续反思大概率也救不回来。但如果初稿写成“偶数是能被 2 整除的整数”，只是代码里把 `% 2 == 0` 写成 `% 2 == 1`，这时批评器就很容易定位错误，修复也很可靠。

所以边界不是“能不能反思”，而是“有没有可修复的结构”。这也是为什么它在代码、数学、工具调用上更有效，因为这些任务的错误通常能被具体指出，而不是停留在风格层面。

## 核心机制与推导

形式化地看，反思回路可以写成：

$$
a_{t+1}=R(a_t, C(a_t))
$$

这里 $C(a_t)$ 输出的不是一个笼统评价，而是错误条目、测试结果、失败日志、规则冲突等可操作信息。是否继续迭代，由验证条件控制：

$$
V(a_t, C(a_t)) \ge \theta
$$

其中 $V$ 是验证函数，$\theta$ 是通过阈值。最简单的情况是布尔值：测试全过就是 1，否则是 0。复杂一点时，也可以是置信度、规则覆盖率、失败数是否下降等指标。

数值玩具例子如下：

1. 初稿 $a_0$ 回答“$2+3=6$”。
2. 批评器 $C$ 调用计算器，得到真实结果为 5，并生成批评：“加法结果错误”。
3. 修复器 $R$ 基于这个批评生成新答案 $a_1$：“$2+3=5$”。
4. 验证器再次调用计算器，确认通过，终止循环。

这个过程的关键不是模型突然变聪明，而是把“错在哪里”从隐含状态变成了显式对象。

再看一个更接近工程的推导。问题是“实现二分查找”。初稿常见错误不是完全不会写，而是边界处理错，比如循环条件写成 `left < right`，导致最后一个元素漏检。此时：

- $a_0$：给出基本正确的二分结构。
- $C(a_0)$：通过单元测试发现“单元素数组 `[5]` 查找 `5` 返回 -1”。
- $R(a_0, C(a_0))$：把循环和边界更新改为覆盖单元素场景。
- $V(a_1)$：所有测试通过。

可以把常见验证信号总结成表：

| 验证信号 | 来源 | 适合任务 | 信息强度 |
|---|---|---|---|
| 数值结果 | 计算器、解释器 | 数学、简单推理 | 高 |
| 单元测试 | 测试框架 | 代码生成、算法题 | 很高 |
| 编译报错 | 编译器、静态检查 | 程序修复 | 很高 |
| 工具返回值 | API、SQL、检索系统 | 工具调用 | 高 |
| 自然语言评价 | 另一个模型或自身 | 开放问答、写作 | 中到低 |

流程图可以简化为：

`a0 -> 批评 C -> 修复 R -> 验证 V -> 通过则终止，否则继续`

终止条件必须提前设定，否则容易陷入“无限自纠”。常见做法有三种：限制最大轮数，例如最多两轮或三轮；设置收益阈值，例如失败数不再下降就停止；要求必须有新的证据输入，没有新报错就不继续改。

## 代码实现

实现上最重要的原则是模块化，把“初答”“批评”“修复”“验证”拆开。这样做的原因很简单：批评器可以换成人工检查、规则引擎、测试器；验证器也可以从 `assert` 换成编译器或 API 回包。

下面是一个可运行的最小 Python 例子。它不依赖大模型，只模拟“先答错，再批评，再修复，再验证”的流程，重点是把闭环写清楚。

```python
def draft_add(a: int, b: int) -> str:
    # 模拟一个会犯错的初稿
    return f"{a}+{b}={a + b + 1}"

def critic(answer: str, a: int, b: int) -> dict:
    expected = a + b
    actual = int(answer.split("=")[1])
    if actual != expected:
        return {
            "ok": False,
            "error_type": "arithmetic_error",
            "message": f"expected {expected}, got {actual}",
        }
    return {"ok": True, "error_type": None, "message": "pass"}

def fixer(answer: str, critique: dict, a: int, b: int) -> str:
    if critique["ok"]:
        return answer
    if critique["error_type"] == "arithmetic_error":
        return f"{a}+{b}={a + b}"
    return answer

def verifier(answer: str, a: int, b: int) -> bool:
    return answer == f"{a}+{b}={a + b}"

draft = draft_add(2, 3)
critique = critic(draft, 2, 3)
revised = fixer(draft, critique, 2, 3)

assert draft == "2+3=6"
assert critique["ok"] is False
assert revised == "2+3=5"
assert verifier(revised, 2, 3)
```

如果把它抽象成新手也能直接照抄的结构，就是：

```python
draft = model.generate(question)
critique = critic.inspect(draft, tools)
revised = fixer.rewrite(draft, critique)
assert verifier.check(revised)
```

各模块的输入输出如下：

| 模块名 | 输入 | 输出 | 职责 | 依赖 |
|---|---|---|---|---|
| `generate` | 问题、上下文 | 初稿 | 先产出可检查对象 | 模型 |
| `inspect` | 初稿、工具、规则 | 批评项 | 找出可执行错误信息 | 模型/工具 |
| `rewrite` | 初稿、批评项 | 修订稿 | 按错误清单改写 | 模型 |
| `check` | 修订稿、标准答案或规则 | 通过/失败 | 判定是否终止 | 测试/规则/工具 |

真实工程里，最实用的简化版本往往不是全自动，而是“半自动反思”。也就是先让系统给出批评稿，再由人确认批评是否靠谱，最后再触发修复。这对初级工程师特别有用，因为批评稿本身就像一份“带证据的 code review”。

## 工程权衡与常见坑

工程上，自我修正几乎总能提高准确率，但代价同样明确：更高延迟、更高 token 成本、更复杂的流程控制。因此不能把它默认套在所有任务上，而是要根据任务价值和可验证性决定是否开启。

| 权衡项 | 影响 | 优化建议 | 度量方式 |
|---|---|---|---|
| 准确率 | 通常上升 | 优先用于高错误成本任务 | 通过率、修复率 |
| 成本 | 线性增加 | 限制反思轮数 | 平均 token、调用次数 |
| 延迟 | 明显增加 | 批评和验证并行化 | P50/P95 延迟 |
| 稳定性 | 依赖验证质量 | 使用强约束验证器 | 重跑一致性 |
| 可解释性 | 通常更好 | 保留批评日志 | 错误定位耗时 |

真实工程例子可以看“日志分析与补丁生成”。假设 CI 失败，初稿先猜“缺少依赖包”。如果只停在这里，价值很低，因为这只是常见猜测。引入反思回路后，流程会变成：

1. 初稿根据日志生成根因猜测。
2. 批评器读取静态检查结果、编译错误和失败测试，指出“不是缺包，而是函数签名已变更，调用方仍传 3 个参数”。
3. 修复器据此生成补丁：更新调用参数，并补一个回归测试。
4. 验证器重新跑测试，确认补丁是否真的生效。

这里的关键不在“猜得更像”，而在“把具体报错传进修复器”。这和普通问答完全不同，因为报错日志本身就是强验证信号。

常见坑主要有以下几类：

- 初稿质量过低。没有基本正确骨架时，反思只会重复指出“答案有问题”，但无法落到具体修改。
- 批评过于空泛。比如只说“需要考虑边界条件”，却不指出哪个输入触发错误，这种批评很难驱动修复。
- 没有停止条件。系统不断重写措辞，内容变化不大，但成本持续上涨。
- 验证器太弱。若验证只靠另一个模型打分，可能出现“自我认同”，即批评器和修复器一起错。
- 错误归因混乱。测试失败可能源于多处缺陷，若把所有失败都归成一个原因，修复很容易方向错误。

一个实用原则是：先保证批评可执行，再追求批评优雅。所谓“可执行”，就是批评至少要包含失败样例、具体报错、规则冲突或数值差异中的一种。

## 替代方案与适用边界

自我修正不是唯一方案。它只是“显式闭环修复”这一类方法的代表。常见替代方案包括错误模板缓存、人类审读、多模型投票、多次采样后交叉比较。它们各有价值，但不一定都具备“错误定位后再修复”的完整链条。

| 任务类型 | 是否可验证 | 推荐方法 | 说明 |
|---|---|---|---|
| 算法题、代码生成 | 强 | 批评-修复 | 测试和编译器能直接判错 |
| 数学计算、符号推理 | 强 | 批评-修复 | 计算器或规则可校验 |
| 工具调用、SQL、API 编排 | 中到强 | 批评-修复 | 返回值和执行日志可检查 |
| 文档总结、开放问答 | 弱到中 | 多次采样 + 交叉检查 | 缺强验证器，反思收益有限 |
| 创意写作、风格改写 | 弱 | 人工审读 | 对错标准主观，不适合自动闭环 |

可以用一个小例子看什么时候该切换方案。假设任务是“给一篇产品介绍文案改写得更有亲和力”。这类任务很难定义唯一正确答案，验证器也难以客观裁决。此时如果强行上批评-修复，系统可能只是在不同措辞间来回切换，收益有限。更合适的流程是多次采样，再由人选最合适的一版。

反过来，如果任务是“生成一个查询订单状态的 SQL”，而数据库会返回语法错、字段不存在、结果为空等信号，那就很适合批评-修复。因为错误不仅能被发现，还能被定位。总结起来，是否使用自我修正，主要看两件事：

1. 初稿是否已经有可修复结构。
2. 外部是否存在足够强的验证信号。

满足这两条，自我修正往往有效；缺任何一条，就要考虑改用投票、人工复核或模板化流程。
