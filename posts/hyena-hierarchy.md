## 核心结论

Hyena Hierarchy 是一种不用自注意力（self-attention，白话讲就是“让每个 token 去和所有 token 两两对比打分”）的长序列建模方法。它把核心计算改成两类交替操作：

1. 长卷积（long convolution，白话讲就是“用一条很长的滤波器在整段序列上滑动聚合历史信息”）
2. 门控（gating，白话讲就是“按输入内容给每个位置乘一个动态开关”）

这套结构的关键不只是“把 attention 换成 convolution”，而是它保留了一个很重要的性质：算子会随输入变化。Transformer 的优势不是矩阵乘本身，而是“当前输入会决定怎么读历史”。Hyena 用“卷积 + gate”的交替递推，近似保留了这种输入相关性，同时把长序列复杂度从注意力常见的 $O(L^2)$ 降到接近 $O(NL\log L)$。

这里的 $L$ 是序列长度，$N$ 是 Hyena 的递推阶数或层内 order。直观上，Hyena 更像“多轮因果滤波”，不是“一次性全局配对比较”。

| 维度 | Transformer | Hyena Hierarchy |
|---|---|---|
| 核心操作 | 自注意力 | 隐式长卷积 + 门控 |
| 典型长序列复杂度 | $O(L^2)$ | 约 $O(NL\log L)$ |
| 参数是否随序列长度增长 | 不直接增长，但显存和算力成本随 $L^2$ 激增 | 卷积核通过函数生成，参数与 $L$ 解耦 |
| 是否输入自适应 | 是 | 是，通过 gate 和递推实现 |
| 长上下文可扩展性 | 较弱 | 更强 |
| 硬件实现 | 成熟 | 依赖 FFT 与 padding 细节 |
| 适合任务 | 中短序列通用 NLP | 超长上下文、因果滤波型任务 |

玩具例子可以这样理解：你不是让每个词去看全篇所有词，而是先用一条很长的“历史汇总规则”把过去压成一个结果，再用当前位置的 gate 决定这个结果该保留多少。重复几轮后，模型就能形成比较复杂的长程依赖。

## 问题定义与边界

Hyena 要解决的问题很具体：如何在不支付 $O(L^2)$ 成本的前提下，仍然做“输入感知的长距离依赖建模”。

如果只追求线性复杂度，其实办法很多，比如固定卷积、状态空间模型、低秩近似。但很多替代方案会丢掉一个能力：输入一变，读历史的方法也要跟着变。Hyena 的设计目标不是只做“快”，而是做“快且仍然动态”。

它的基本递推写成：

$$
z_1 = v,\qquad z_{n+1} = x_n \odot (h_n * z_n),\qquad y = z_{N+1}
$$

其中：

- $v$ 是初始值向量，可以理解成输入投影后的“待传播内容”
- $x_n$ 是第 $n$ 轮 gate，可以理解成“本轮保留/抑制多少信息”
- $h_n$ 是第 $n$ 轮长卷积核
- $*$ 表示卷积
- $\odot$ 表示逐元素乘法

这个定义自带三个边界条件。

第一，默认是因果（causal）的。因果，白话讲就是“当前位置只能看过去，不能偷看未来”。这对语言建模、时间序列、公告流分析都很关键。

第二，它不是无限次递推，而是 order-$N$。$N$ 越大，信息交互路径越复杂，但实现和训练也更重。

第三，它主要解决“超长上下文下的序列压缩与传播”问题，不是显式 token-to-token 对齐问题。如果任务要求某个词必须直接定位另一个词并做细粒度关系判断，Hyena 不一定天然占优。

新手可以用“长小说”例子理解边界：

- Transformer 的做法像“第 5000 个词把前 4999 个词都重新看一遍并逐个打分”
- Hyena 的做法像“先把前 4999 个词经过一轮长滤波形成历史摘要，再用第 5000 个词控制保留哪些历史分量”

两者都能利用长历史，但路径不同。前者是显式全连接，后者是递推式因果混合。

## 核心机制与推导

Hyena 的关键不是“卷积很长”，而是“卷积核不是显式存一大串参数，而是隐式生成”。

显式长卷积的问题很直接：如果序列长度是 $L$，你真存一个长度 $L$ 的核，参数和实现都会不优雅。Hyena 采用 implicit filter parametrization，也就是隐式滤波器参数化，白话讲就是“只存一小个生成函数，需要多长的卷积核时再现场生成”。

其形式可以写成：

$$
h(t)=\text{Window}(t)\cdot \text{FFN}(\text{PosEnc}(t))
$$

这里：

- $\text{PosEnc}(t)$ 是位置编码，告诉模型“这是第几个时间步”
- $\text{FFN}$ 是前馈网络，负责把位置映射成滤波器值
- $\text{Window}(t)$ 是窗函数，负责控制衰减、平滑和频谱形状

这件事的意义很大。模型不再需要“为每个长度单独存一条核”，而是学会“一个生成核的规则”。因此参数量和序列长度解耦。

从线性代数角度，Hyena 每一轮可以写成“门控矩阵乘卷积矩阵”交替：

$$
y = D_{x_N}S_{h_N}\cdots D_{x_1}S_{h_1}v
$$

其中：

- $D_{x_i}$ 是由 gate 向量构造出的对角矩阵
- $S_{h_i}$ 是由卷积核构造出的 Toeplitz 型卷积算子

这说明它仍然是线性算子复合，但这个线性算子的系数来自输入，所以是动态线性算子。白话讲：规则本身会随输入变，不是固定模板。

### 玩具例子

设 order=1，长度 $L=4$：

- $v=[1,2,3,4]$
- $x_1=[0.5,0.5,0.5,0.5]$
- $h_1=[1,0,0,0]$

因为这个卷积核等价于“原样保留”，所以：

$$
h_1 * z_1 = z_1 = [1,2,3,4]
$$

再做 gate：

$$
z_2 = x_1 \odot (h_1 * z_1) = [0.5,1,1.5,2]
$$

这说明哪怕最简单的情形里，gate 也已经在决定“历史内容保留多少”。如果把 $h_1$ 换成一个更长的因果核，例如强调最近 3 个位置的加权和，那么它就不只是缩放，而是在“先聚合历史，再决定保留”。

### 为什么能做到 $O(NL\log L)$

长卷积直接做通常是 $O(L^2)$，因为每个输出位置都可能和整个历史核做乘加。Hyena 用 FFT（快速傅里叶变换，白话讲就是“把卷积改写成频域逐点相乘，再变回来”）加速卷积，把单次长卷积降到约 $O(L\log L)$。

因此 order-$N$ 的整体核心开销近似是 $N$ 次长卷积，加上一些逐点门控和投影，得到论文中的复杂度表达。核心信息不是常数项，而是：长度增长时，Hyena 的增长速度明显慢于二次注意力。

| 方案 | 核参数化方式 | 参数是否依赖 $L$ | 长序列实现 |
|---|---|---|---|
| 显式长卷积 | 直接存长度为 $L$ 的核 | 强相关 | 不优雅 |
| 隐式长卷积 | 用 $\text{Window}\cdot \text{FFN(PosEnc)}$ 生成 | 基本解耦 | 更适合扩展 |
| 注意力 | 不存卷积核 | 参数不随 $L$ 变，但算力随 $L^2$ 增长 | 中短序列更常见 |

真实工程例子是长篇央行公告、会议纪要、法律文档或多天日志流。这些数据经常有数万到数十万 token。此时 attention 的主要问题不是“不能算”，而是“算得太贵”。Hyena 的长卷积路径在这类任务上更有现实意义。

## 代码实现

实现上，Hyena 一层的主流程可以概括为：

1. 输入投影，得到内容分量 $v$
2. 再投影出若干组 gate $x_1,\dots,x_N$
3. 对每一轮生成隐式卷积核 $h_n$
4. 用 FFT 做因果长卷积
5. 将卷积结果与 gate 做逐元素乘法
6. 多轮递推后输出

下面给一个可运行的简化 Python 版本。它不是训练代码，而是帮助理解“因果卷积 + gate”的前向逻辑。

```python
import math

def causal_conv_direct(h, z):
    """最朴素的因果卷积: y[t] = sum_{k=0}^t h[k] * z[t-k]"""
    L = len(z)
    y = [0.0] * L
    for t in range(L):
        acc = 0.0
        for k in range(min(len(h), t + 1)):
            acc += h[k] * z[t - k]
        y[t] = acc
    return y

def hyena_order1(v, x, h):
    conv = causal_conv_direct(h, v)
    return [a * b for a, b in zip(x, conv)]

# 玩具例子
v = [1.0, 2.0, 3.0, 4.0]
x = [0.5, 0.5, 0.5, 0.5]
h = [1.0, 0.0, 0.0, 0.0]

y = hyena_order1(v, x, h)
assert y == [0.5, 1.0, 1.5, 2.0]

# 再看一个更像“历史聚合”的核：当前 + 前一个位置
h2 = [1.0, 0.25]
y2 = hyena_order1(v, x, h2)

# 因果卷积结果应为:
# t=0: 1
# t=1: 2 + 0.25*1 = 2.25
# t=2: 3 + 0.25*2 = 3.5
# t=3: 4 + 0.25*3 = 4.75
expected = [0.5, 1.125, 1.75, 2.375]
for a, b in zip(y2, expected):
    assert abs(a - b) < 1e-9
```

对应的伪代码就是：

```python
v = proj(input)
gates = gate_proj(input)   # 生成多组 gate
z = v
for n in range(order):
    h = window(t) * FFN(pos_enc(t))
    z = gates[n] * fft_conv_causal(h, z)
y = z
```

这里最容易被忽略的是“因果 FFT 卷积怎么做”。原因是 FFT 默认更接近循环卷积，不天然等于你想要的线性因果卷积。工程上通常要：

1. 把输入序列和卷积核都 pad 到至少 $2L-1$
2. 在频域相乘
3. 做逆变换
4. 只截取前 $L$ 个位置的线性卷积有效部分
5. 保证当前位置只用了过去的信息

为什么是 $2L-1$？因为两个长度为 $L$ 的序列做线性卷积，结果长度就是 $2L-1$。不补到这个长度，就容易把尾部绕回头部，变成循环卷积，因果性直接被破坏。

真实工程里，完整实现还会额外处理：

- 通道维度分组
- 不同 order 的核共享或分离
- 频域张量布局
- 复数 FFT 输出取实部
- mixed precision 下的数值稳定性

## 工程权衡与常见坑

Hyena 的难点不在公式，而在“公式变成可训练、可稳定部署的实现”。

第一类坑是滤波器退化成低频平滑器。低频，白话讲就是“只保留缓慢变化的大趋势，细节变化被抹平”。如果你只是简单用位置编码加一个小 MLP 去生成 $h(t)$，但没有窗函数、衰减项或足够合适的激活，学出来的核常常像平滑器，而不是有辨识力的长程滤波器。这样模型会“记住趋势”，却抓不住边界、高频事件和局部突变。

第二类坑是因果性在 FFT 实现里被悄悄破坏。很多实验效果异常好或异常差，最后都不是模型理论问题，而是 padding、截断或卷积方向写反了。

第三类坑是 gate 顺序写错。Hyena 不是随便“卷积一下再乘一下”都一样。$x_n \odot (h_n * z_n)$ 和 $h_n * (x_n \odot z_n)$ 一般不是同一个算子，顺序错了，模型性质就变了。

| 常见问题 | 表现 | 原因 | 规避方式 |
|---|---|---|---|
| 低频退化 | 输出过度平滑，长程依赖弱 | 核生成函数表达力不足 | 在 $h(t)$ 中加入 decay、bias、合适激活如 `sin` |
| 因果性丢失 | 训练/验证结果异常，疑似“偷看未来” | FFT padding 或截断错误 | pad 到 $2L-1$，只保留线性因果部分 |
| 卷积方向写反 | 结果与理论不符 | 把相关当卷积、时间索引反了 | 明确使用因果定义并写单元测试 |
| gate 顺序错误 | 性能明显下降 | 递推结构被改写 | 保持 $z_{n+1}=x_n\odot(h_n*z_n)$ |
| 复数处理不当 | 数值噪声大 | FFT/IFFT 后未正确取实部或类型不一致 | 统一 dtype，明确 `real` 截取 |
| 长度外推不稳 | 训练长度好，推理更长时崩 | 核生成函数未学到稳定外推 | 检查窗函数设计和位置编码尺度 |

一个典型的新手错误是：为了“简单”，把核直接设成几个可学习标量，再靠很深层数去补。这样虽然形式上还是 Hyena 风格，但你已经失去“超长隐式滤波”的核心价值，最后只得到一个表达力受限的门控卷积网络。

## 替代方案与适用边界

Hyena 不是“比 Transformer 更先进”的通用结论，它更像“在超长因果序列上非常有竞争力的一条路线”。

如果任务长度只有 512 或 2048，FlashAttention 往往已经足够好。FlashAttention 的核心优势是“让 attention 更省显存、更高效”，但本质仍是 attention。对于中短序列，它通常更简单，生态也更成熟。

如果任务强依赖显式 token-token 对齐，比如关系抽取、复杂多跳问答、实体精确指代，自注意力依然常常更直接。Hyena 能传播信息，但它不是在每一步都显式构造“谁看谁”的得分矩阵。

如果任务是超长上下文、强因果、强调历史滤波，比如：

- 央行公告和金融时间序列联合建模
- 数十万 token 的长文档压缩与分类
- 长时序传感器信号分析
- 需要类似 PQMF 风格长滤波思想的序列恢复

那么 Hyena 的路线更值得考虑。

| 架构 | 适用长度 | 典型复杂度 | 优势 | 局限 |
|---|---|---|---|---|
| Hyena | 很长，尤其万级以上 | 约 $O(NL\log L)$ | 长上下文友好，参数与长度解耦 | 实现细节多，生态较新 |
| Transformer | 中短到中长 | $O(L^2)$ | 表达直观，生态成熟 | 长序列成本高 |
| FlashAttention | 中短到较长 | 仍基于 attention，本质不改 $L^2$ | 工程效率高，显存友好 | 超长序列仍贵 |
| Sparse Attention | 长序列 | 低于全注意力，依赖稀疏模式 | 可保留部分显式对齐能力 | 稀疏模式设计影响大 |
| Linear Attention | 长序列 | 近线性 | 速度好 | 近似误差和表达限制明显 |

一个直观判断标准是：

- 如果你要的是“几十万 token 上还能跑”，优先考虑 Hyena 一类结构
- 如果你要的是“每个 token 精确挑出另一个 token 做对齐”，优先考虑 attention 系
- 如果长度不算大，先用成熟的 Transformer/FlashAttention，通常性价比更高

真实工程例子可以看利率建模或央行公报分析：文档很长，很多信息不是局部触发，而是跨很长上下文逐步累积。此时“长程因果滤波 + 动态门控”往往比“全量两两比对”更贴近成本约束。
