## 核心结论

跨模态检索的核心问题，不是“模型要不要更大”，而是“跨模态交互发生在什么阶段”。这里的跨模态，白话说就是文本和图像这类不同类型的数据要放到同一个任务里比较相关性。主流方案可以归纳为三类：

| 架构 | 交互时机 | 速度 | 精度上限 | 索引方式 | 典型场景 |
|---|---|---:|---:|---|---|
| 双塔 | 编码后才交互 | 快 | 中 | 向量索引，适合 ANN | 大规模召回 |
| 融合 | 编码时就交互 | 慢 | 高 | 很难离线建统一索引 | 小规模精排 |
| 后期交互 | 编码后保留局部交互 | 中 | 较高 | token/patch 级索引 | 召回后重排、中间层 |

最重要的工程结论只有一句：

$$
\text{最终方案}=\text{召回层}+\text{重排层}
$$

也就是说，实际系统通常不是在“双塔”和“融合”之间二选一，而是分层使用。新手版理解可以写成一句话：文搜图时，双塔先把文本和图片分别变成向量，再快速找相近候选；融合模型把文本和图片放一起仔细比对，但只能慢慢算少量结果。

如果候选库有百万到千万级，双塔几乎是召回层的默认选择；如果业务要求最终排序能区分颜色、品牌、局部物体、属性组合，通常还要接一个融合模型或 ColBERT 风格后期交互模型做重排。

## 问题定义与边界

跨模态检索要解决的问题可以形式化为：给定查询 $q$，从候选集合 $C$ 中找出最相关的内容：

$$
\arg\max_{c\in C} s(q,c)
$$

其中 $s(q,c)$ 是匹配分数，白话说就是“这对查询和候选有多像”。

这里讨论的是检索，不是生成，也不是分类。

- 检索：从现有候选库里找最相关的项。
- 生成：让模型直接生成图像、文本或答案。
- 分类：把输入映射到固定标签集合。

一句最容易混淆的边界是：一段文字找相关图片，不是让模型“画图”，也不是给图片“打标签”，而是从海量图片里找最匹配的几张。

常见任务边界如下：

| 场景 | 查询输入 | 候选输入 | 输出 |
|---|---|---|---|
| 文本搜图 | 文本 | 图片库 | Top-K 图片 |
| 图搜文 | 图片 | 文本库 | Top-K 文本 |
| 图搜图 | 图片 | 图片库 | Top-K 相似图片 |
| 多模态商品搜索 | 文本或图像 | 商品图文对 | Top-K 商品 |

本文重点讨论的是“大规模候选库下的检索架构权衡”。如果候选集合只有几十条，直接用融合模型逐一打分也可行；但当候选量到百万级时，问题就会从“能不能做”变成“能不能在延迟预算内做”。

---

## 核心机制与推导

三类架构的本质区别，是交互发生得早还是晚。

### 1. 双塔：先各自编码，再做全局相似度

双塔的意思是查询侧和候选侧各有一个编码器，分别把输入映射成向量。向量可以理解成“压缩后的语义摘要”。打分公式通常是：

$$
s(q,c)=\langle f_q(q), f_c(c)\rangle
$$

其中 $\langle \cdot,\cdot\rangle$ 是点积或余弦相似度，$f_q,f_c$ 是两侧编码器。

训练时常用对比学习。对比学习，白话说就是“拉近正确配对，推远错误配对”。典型损失为：

$$
L=-\log \frac{e^{s(q,c^+)/\tau}}{\sum_k e^{s(q,c_k)/\tau}}
$$

其中 $c^+$ 是正样本，$\tau$ 是温度参数，用来调节分布陡峭程度。

双塔为什么适合大规模检索？因为候选库可以离线编码。也就是说，图片库里的每张图片都能提前算成向量，建好 ANN 索引。ANN 是 approximate nearest neighbor，白话说就是“近似找最近向量”，速度远快于全量逐个比较。

### 2. 融合：先让两边交互，再输出匹配分数

融合模型通常把文本 token 和图像 patch 一起送进联合编码器。token 可以理解成“文本切分后的最小处理单位”，patch 可以理解成“图像切成的小块”。打分可写为：

$$
s(q,c)=g([q;c])
$$

这里 $[q;c]$ 表示把查询和候选拼在一起，$g$ 表示联合编码后的打分头。

它的优势在于细粒度对齐能力强。比如“红色跑鞋，白色中底，鞋侧有黑条”，融合模型可以在编码阶段直接让“红色”“白色中底”“黑条”这些词与图像局部区域交互，因此对复杂属性组合更敏感。

代价也很直接：不能像双塔那样把所有候选都预编码成一个统一向量后直接检索。每来一个查询，几乎都要和候选逐个做联合计算，所以全库打分成本极高。

### 3. 后期交互：保留局部表示，但避免全量联合编码

ColBERT 风格属于后期交互。它不是只保留一个全局向量，而是保留每个 token 或 patch 的表示，再在查询时做轻量局部匹配。典型打分是：

$$
s(q,c)=\sum_i \max_j \langle q_i, c_j\rangle
$$

含义是：查询中的每个 token 向量 $q_i$，都到候选 token 向量集合里找一个最像的 $c_j$，然后把这些最大值加起来。它关注的是“每个查询局部证据是否都能找到支撑”。

这是一条中间路线：

- 比双塔更细，因为它保留了 token 级信息。
- 比全融合便宜，因为候选侧可以预存局部向量，不必每次都完整联合编码。

### 4. 玩具例子：为什么后期交互能保住细节

假设一个查询有 2 个 token，候选有 3 个 token，余弦相似度矩阵是：

$$
S=\begin{bmatrix}
0.2 & 0.9 & 0.1\\
0.7 & 0.4 & 0.3
\end{bmatrix}
$$

那么 ColBERT 风格打分是：

$$
\max(0.2,0.9,0.1)+\max(0.7,0.4,0.3)=0.9+0.7=1.6
$$

这表示查询中的两个关键信号都找到了强匹配。若只看一个全局向量，可能把这两个局部证据平均掉，导致“整体像，但关键属性不全对”的候选排得太高。

可以把三者做一个直观类比：

- 双塔：先各自写摘要，再比摘要。
- 融合：把两份材料放一起逐句核对。
- ColBERT：先各自保留关键词，再让关键词两两找最像的。

机制对比如下：

| 方案 | 编码方式 | 是否可离线建索引 | 打分复杂度 | 细粒度匹配 |
|---|---|---|---|---|
| 双塔 | 全局向量 | 可以 | 低 | 弱 |
| 融合 | 联合编码 | 基本不行 | 高 | 强 |
| ColBERT | token/patch 级向量 + 后期 MaxSim | 可以，但索引更重 | 中 | 较强 |

## 代码实现

工程里应按完整链路理解：数据预处理、编码、建索引、召回、重排、评估。

先看一个最小可运行的 Python 例子。这里不依赖深度学习框架，只用手写向量来演示“双塔召回 + 重排”的流程。

```python
import math

def dot(a, b):
    return sum(x * y for x, y in zip(a, b))

def cosine(a, b):
    na = math.sqrt(dot(a, a))
    nb = math.sqrt(dot(b, b))
    if na == 0 or nb == 0:
        return 0.0
    return dot(a, b) / (na * nb)

# 候选库：假设已经由图像编码器离线编码
gallery = {
    "shoe_red_black": {
        "global_vec": [0.9, 0.1, 0.8],
        "tokens": [[0.9, 0.0], [0.0, 0.8], [0.7, 0.2]],  # 局部特征
    },
    "shoe_blue_white": {
        "global_vec": [0.8, 0.2, 0.4],
        "tokens": [[0.7, 0.1], [0.1, 0.6], [0.2, 0.3]],
    },
    "bag_black": {
        "global_vec": [0.2, 0.9, 0.1],
        "tokens": [[0.2, 0.8], [0.1, 0.7]],
    },
}

# 查询文本向量：假设由文本编码器输出
query_global = [0.95, 0.05, 0.75]
query_tokens = [[1.0, 0.0], [0.0, 1.0]]  # 例如两个关键信号：跑鞋 / 黑色细节

# 1) 双塔召回
recall_scores = {
    item_id: cosine(query_global, item["global_vec"])
    for item_id, item in gallery.items()
}
topk = sorted(recall_scores, key=recall_scores.get, reverse=True)[:2]

# 2) 后期交互重排（ColBERT 风格 MaxSim）
def maxsim_score(q_tokens, c_tokens):
    total = 0.0
    for q in q_tokens:
        total += max(cosine(q, c) for c in c_tokens)
    return total

rerank_scores = {
    item_id: maxsim_score(query_tokens, gallery[item_id]["tokens"])
    for item_id in topk
}
ranked = sorted(rerank_scores, key=rerank_scores.get, reverse=True)

assert topk[0] in {"shoe_red_black", "shoe_blue_white"}
assert ranked[0] == "shoe_red_black"
assert len(topk) == 2

print("Recall TopK:", topk)
print("Reranked:", ranked)
```

这个例子表达了一个核心事实：双塔先用全局语义把候选缩小到 Top-K，后面的重排层再利用局部信息纠正最终顺序。

如果用伪代码表示完整系统：

| 步骤 | 输入 | 输出 | 说明 |
|---|---|---|---|
| 预处理 | 图文样本 | 训练对/索引元数据 | 清洗、截断、标准化 |
| 编码 | 查询/候选 | 向量或 token 向量 | 双塔或局部表示 |
| 建索引 | 候选表示 | ANN/token 索引 | 离线完成 |
| 召回 | 查询 | Top-K 候选 | 快速缩小范围 |
| 重排 | 查询 + Top-K | 最终排序 | 用更重的模型 |
| 评估 | 排序结果 | Recall@K / MRR / NDCG | 验证检索效果 |

工程骨架通常长这样：

```python
# 1) 预编码候选
gallery_vecs = image_encoder(gallery_images)
ann_index.add(gallery_vecs)

# 2) 查询召回
q_vec = text_encoder(query_text)
topk_ids = ann_index.search(q_vec, k=100)

# 3) 重排
candidates = load_items(topk_ids)
scores = cross_encoder(query_text, candidates)
ranked = sort_by_score(scores)
```

如果换成 ColBERT 风格，会多一个“存 token 向量”的步骤：

```python
# 离线：为每个候选保存 token/patch 向量
doc_token_vecs = encoder(candidate_items)
token_index.add(doc_token_vecs)

# 在线：查询 token 与候选 token 做 MaxSim
q_token_vecs = encoder(query_text)
topk_ids = coarse_index.search(q_token_vecs, k=200)
scores = {doc_id: maxsim(q_token_vecs, load_doc_tokens(doc_id)) for doc_id in topk_ids}
ranked = sort_by_score(scores)
```

### 真实工程例子：电商“文搜图”

用户输入“黑色防水冲锋衣，帽檐有反光条”。系统通常不会直接把这句文本和全站一千万张商品图逐一做融合打分，因为在线成本不可接受。更常见的是：

1. 双塔把一千万商品图提前编码成向量并建 ANN 索引。
2. 查询文本编码后先召回 Top-200。
3. 用融合模型或 ColBERT 风格模型对这 200 个候选重新排序。
4. 最终返回 Top-20。

这样做的原因不是学术偏好，而是吞吐、延迟、精度三者无法同时取最优。

## 工程权衡与常见坑

工程选择本质上是在延迟、精度、内存之间做三角权衡。

| 维度 | 双塔 | 融合 | ColBERT |
|---|---|---|---|
| 延迟 | 低 | 高 | 中 |
| 精度上限 | 中 | 高 | 较高 |
| 内存/存储 | 中 | 低到中 | 高 |

常见坑如下：

| 问题 | 表现 | 原因 | 规避方式 |
|---|---|---|---|
| 只用双塔做最终排序 | 颜色、品牌、属性组合经常排错 | 全局向量压平局部证据 | 增加重排层 |
| 融合模型直接全库打分 | 延迟爆炸、QPS 很低 | 每个候选都要联合计算 | 改为两阶段检索 |
| ColBERT token 索引过大 | 内存和磁盘占用失控 | 每个候选保留大量局部向量 | 控制 token 数、量化压缩 |
| 负样本太容易 | 离线指标很好，线上提升不明显 | 模型只学会区分很粗的差异 | 加 hard negative mining |
| 文本截断或 patch 太少 | 长描述或局部视觉证据丢失 | 输入预算不够 | 设计裁剪策略并做消融实验 |

这里最值得强调两个误区。

第一，把双塔当最终排序器。双塔适合“先找到大致相关的候选”，但对“黑色”和“深灰色”、“低帮”和“中帮”、“有 logo”和“无 logo”这类细粒度差异，往往不够稳定。原因不是它完全看不到这些特征，而是它必须把所有信息压进一个全局向量，局部证据容易被平均。

第二，把融合模型放到全库阶段。很多初学者会认为“既然融合更准，那就全程都用融合”。这个思路在离线实验里容易成立，在真实服务里通常不可用。假设全库有 $10^7$ 个候选，每次请求都要跑 $10^7$ 次联合编码，哪怕单次打分只要 1 毫秒，总时延也完全失控。

还有一个经常被忽略的问题是训练分布。若负样本都很容易，比如“运动鞋”对应的负样本是“西红柿”，模型当然能学会；但线上真正难的是“黑色跑鞋”和“深灰训练鞋”这种近邻竞争。检索训练如果没有困难负样本，离线 Recall@K 可能很好看，线上点击和转化却不涨。

## 替代方案与适用边界

选型时不要问“哪种架构最好”，而要问“在哪个阶段、什么规模、什么目标下最好”。

适用边界可以先按规模和预算看：

| 数据规模/候选量 | 延迟预算 | 目标 | 推荐架构 |
|---|---|---|---|
| 小库，几千到几万 | 宽松 | 最终精度优先 | 融合 |
| 中库，几十万 | 中等 | 精度与成本平衡 | ColBERT 或双塔+重排 |
| 大库，百万到千万 | 严格 | 先召回再精排 | 双塔召回 + 融合/ColBERT 重排 |
| 超大库，强吞吐 | 很严格 | 稳定在线服务 | 双塔为主，轻量重排为辅 |

再按方案本身看：

| 方案 | 适合什么时候用 | 不适合什么时候用 |
|---|---|---|
| 双塔 | 大规模召回、需要 ANN、吞吐优先 | 直接做最终细粒度排序 |
| 融合 | 候选数少、要求高精度判断 | 百万级全库在线打分 |
| ColBERT | 想保留局部交互，又要比融合更快 | 内存非常紧张、索引预算很小 |
| 两阶段混合 | 大多数真实系统 | 只有极小数据集时可能过度设计 |

为什么不直接做“单模型端到端全量排序”？因为排序不是唯一目标，系统还必须满足服务约束。一个只能离线跑得动、线上无法承载的模型，在工程上并不优。

对于零基础到初级工程师，最稳妥的判断规则是：

- 先问候选库有多大。
- 再问延迟预算是多少。
- 最后问业务是否需要识别细粒度属性。

如果答案是“大库 + 低延迟 + 细粒度要求高”，那么基本就会落到“两阶段方案”。电商搜索、图文素材检索、内容推荐召回几乎都符合这个模式。
