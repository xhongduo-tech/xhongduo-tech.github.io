## 核心结论

多源数据实体对齐，指的是把数据库、百科、日志、文档等不同来源里的多条记录，判断为“同一个真实对象”的过程。这里的“真实对象”可以是人、机构、商品、设备、地址，甚至一次事件。白话说，它要解决的不是“这两条记录看起来像不像”，而是“它们在现实世界里是不是同一个东西”。

实体对齐的完整问题，至少包含五步：候选召回、字段标准化、相似度计算、匹配判定、冲突消解。很多初学者把注意力只放在“算相似度”上，这会漏掉两个更难的部分：第一，怎样把全量数据缩到可计算的候选集合；第二，匹配成功后到底保留哪份值、如何记录来源、怎样避免脏数据污染主档。

下面这张表可以先建立整体框架：

| 阶段 | 输入 | 目标 | 典型方法 | 输出 |
|---|---|---|---|---|
| 候选生成 | 多源原始记录 | 缩小比较范围 | blocking、倒排索引、规则分桶 | 候选记录对 |
| 字段标准化 | 候选记录对 | 消除格式噪声 | 清洗、分词、归一化、日期统一 | 可比较字段 |
| 相似度判断 | 标准化字段 | 给出是否同实体的证据 | 字符串相似度、规则、概率模型、分类器 | 匹配分数 |
| 冲突消解 | 已匹配记录组 | 生成主实体 | 来源加权、可信度投票、业务规则 | 合并主档 |
| 结果落库 | 主档 + 来源信息 | 可追溯、可回滚 | provenance 存储、版本管理 | 主数据视图 |

一个总公式可以把“匹配”和“合并”放在同一张图里理解。先对候选对 $(a,b)$ 计算匹配分数：

$$
S(a,b)=\sum_{k=1}^{K} w_k\, s_k(x_k^a,x_k^b),\quad w_k\ge 0,\ \sum_k w_k=1
$$

再在匹配成功后，对冲突字段做值选择：

$$
v^*=\arg\max_v \sum_{s\in \mathcal{S}(v)} r_s
$$

其中 $s_k$ 是第 $k$ 个字段的相似度，白话说就是“每个字段给出一份证据”；$r_s$ 是来源可靠度，白话说就是“这个系统的话值不值得更相信”。

玩具例子很简单。来源 A 里有“Apple Inc.”，来源 B 里有“Apple, Incorporated”，地址都指向“1 Apple Park Way”，网站都是 `apple.com`。名称写法不同，但多个字段共同支持“它们是同一家公司”，因此应该对齐。

真实工程例子更典型。医院患者主数据平台要合并挂号系统、HIS、检验系统、医保接口里的患者记录。姓名可能有别名，手机号可能换过，地址写法不统一，证件号可能缺失。对齐的目标不是做一个模糊查询结果页，而是合成一条可信患者主档，并保留“这个手机号来自检验系统、这个地址来自挂号系统、这个生日由医保接口确认”的来源信息。

结论先说清楚：实体对齐既是识别问题，也是数据治理问题。只做“识别”而不做“治理”，最后会得到一堆无法解释、无法回滚、无法审计的合并结果。

## 问题定义与边界

形式化地说，给定两个来源中的记录 $a$ 和 $b$，实体对齐要判断：

$$
M(a,b)\in\{0,1\}
$$

其中 $M(a,b)=1$ 表示两条记录指向同一真实实体，$M(a,b)=0$ 表示不是。白话说，这就是一个“是不是同一个对象”的二分类问题。

但“同一实体”不能脱离业务语义单独定义。对“患者”来说，通常要求证件号、姓名、生日、联系方式等多个证据共同支持；对“商品”来说，型号、品牌、规格、条码更关键；对“日志事件”来说，时间窗口、服务名、请求链路、资源 ID 可能比文本描述更重要。对象不同，容错边界不同。

下面先区分三个常见术语：

| 术语 | 关注点 | 常见场景 | 与本文关系 |
|---|---|---|---|
| `record linkage` | 记录级链接 | 医疗、政府、统计 | 强调跨表、跨库把记录连起来 |
| `entity resolution` | 实体级消歧与合并 | 知识图谱、主数据、搜索 | 范围更广，通常包含匹配与聚合 |
| `deduplication` | 去重 | 单库重复清理 | 更像特殊情况，常发生在单一来源内部 |

可以把它们理解成一个由窄到宽的关系：`deduplication` 常常是单库去重；`record linkage` 更强调跨源记录关联；`entity resolution` 则不仅判断，还包含合并、修复、追溯等后续动作。

边界同样重要。以下情况适合做实体对齐：

| 适合对齐的对象 | 不适合直接对齐的情况 | 常见风险 |
|---|---|---|
| 患者、客户、供应商、商品、机构、设备、地址 | 语义本身不稳定的对象，比如“热点事件”或“模糊主题” | 把“相似”误当“同一” |
| 结构相对稳定、字段可比较的记录 | 只有自由文本、没有可比较锚点的记录 | 候选过多，误匹配率高 |
| 有来源标识、时间戳、属性字段的多源记录 | 业务上本来就允许一人多号、一物多版本的场景 | 过度合并，丢失版本差异 |

必须强调一条边界：实体对齐不是万能去重，也不是简单模糊搜索。模糊搜索只要求“找出像的结果”，实体对齐要求“对现实世界同一性作出判断”。这两个目标的误差成本不同。

看一个最容易出错的例子。两条“张三”记录并不一定是同一个人：

- 挂号系统：姓名“张三”，生日“1990-03-02”，证件号后四位“1023”
- 检验系统：姓名“张三”，生日“1990-03-02”，证件号后四位“1023”

这通常可以进入高置信匹配。

但如果另一条记录只有姓名相同，手机号、地址、生日、就诊时间段都不一致，那就不能因为“同名”直接合并。对齐系统犯的最大错误之一，就是把高频姓名、高频机构名、常见商品名当成强证据。

所以问题定义应当分成两层：

1. 匹配层：判断哪些记录属于同一实体。
2. 合并层：决定主实体保留哪些值，怎样记录来源，怎样支持回滚和修复。

这两层不能混写成一个模糊函数。否则一旦误匹配，就很难追溯到底是“判错了”，还是“合并规则错了”。

## 核心机制与推导

实体对齐的核心机制通常分两层。第一层是 `blocking`，白话说就是“先把不可能相同的记录排掉”；第二层是候选对比较，白话说就是“对剩下可能相同的记录做细算分”。

如果没有 blocking，$N$ 条记录两两比较的复杂度接近 $O(N^2)$。当每个来源都有百万级记录时，这在工程上通常不可接受。blocking 的目标不是直接找出真匹配，而是尽量保证“真匹配别漏掉，同时候选数不要爆炸”。

常见 blocking 键包括：

- 证件号后四位 + 生日
- 姓名拼音首字母 + 城市
- 商品品牌 + 型号前缀
- 机构名称归一化结果 + 邮编

玩具例子：  
两条患者记录如果都满足“生日同一天，手机号后四位一致”，就进入同一个候选桶。这样全库不需要和所有记录比较，只需要和桶内记录比较。

进入候选后，再做字段级比较。设候选对为 $(a,b)$，第 $k$ 个字段的相似度函数为 $s_k$，字段权重为 $w_k$，则总体分数是：

$$
S(a,b)=\sum_{k=1}^{K} w_k\, s_k(x_k^a,x_k^b),\quad w_k\ge 0,\ \sum_k w_k=1
$$

这里的 $x_k^a$ 表示记录 $a$ 在第 $k$ 个字段上的值。比如名称字段可以用编辑距离相似度，地址字段可以用行政区和门牌拆分后的组合相似度，时间字段可以用时间差衰减函数。

题目给出的最小例子正好说明这一点。设字段相似度为：名称 $0.90$、地址 $0.80$、电话 $0.00$；权重分别为 $0.5$、$0.3$、$0.2$，则：

$$
S(a,b)=0.5\times0.90+0.3\times0.80+0.2\times0=0.69
$$

若阈值 $\tau_m=0.65$，则判为同一实体。关键不是“电话不一样也能匹配”，而是“电话的不一致只是负证据之一，不能盖过其他强证据”。对齐是证据综合，不是单字段一票否决，除非业务明确规定某字段是硬约束。

经典的概率链接模型是 Fellegi-Sunter。它把每个字段的权重写成对数似然比：

$$
w_k=\log\frac{m_k}{u_k},\quad m_k=P(\text{agree}\mid \text{match}),\ u_k=P(\text{agree}\mid \text{nonmatch})
$$

直觉上，如果一个字段在“真实匹配”中经常一致，但在“真实不匹配”中很少碰巧一致，那么这个字段的权重就高。比如身份证号这种强标识字段，通常 $m_k/u_k$ 很大；而性别、省份这类低区分度字段，权重通常较低。

判定规则也经常不是单阈值，而是双阈值：

$$
S\ge \tau_m \Rightarrow \text{match},\quad S\le \tau_u \Rightarrow \text{nonmatch}
$$

处在中间区域的候选对，可以进入人工审核、二次规则或更复杂模型。这样做的原因很现实：很多场景里最贵的不是算分，而是误合并。一条错误合并的患者主档、供应商主档、法人主档，后续修复成本很高。

下面用步骤表替代流程图：

| 步骤 | 作用 | 关键输出 |
|---|---|---|
| blocking | 把全量记录缩成候选集合 | 候选对或候选簇 |
| 字段标准化 | 统一格式、消除噪声 | 规范化字段 |
| 特征计算 | 为每个字段产出证据 | 相似度向量 |
| 相似度/分类 | 综合证据作出匹配判断 | 分数、标签、置信度 |
| 冲突消解 | 在已匹配记录中选值 | 主档字段值 |
| provenance 落库 | 记录来源、版本、置信度 | 可追溯主实体 |

真实工程例子可以看医院患者主档。blocking 阶段可以先用“生日 + 姓名首字 + 证件后四位”召回候选；字段标准化把“138-0000-0000”变成“13800000000”，把“2024/1/2”变成“2024-01-02”，把“北京海淀”“北京市海淀区”归一化到统一地址层级；相似度阶段综合姓名、生日、手机号、住址、来源系统可信度；最后合并时不只保留一个手机号，还要记录“主手机号来自医保接口，备用手机号来自挂号系统”。

这一整套机制的重点是：匹配分数是“识别证据”，而 provenance 是“治理证据”。前者回答“是不是”，后者回答“为什么这样合并、以后怎么纠错”。

## 代码实现

实现时不要把流程写成一个大函数。至少拆成五块：候选生成、标准化、特征打分、判定、合并。这样每一块都能单测，出了问题也能定位。

先给最小流程骨架：

```text
1. normalize(a), normalize(b)
2. if not blocking_pass(a, b): return no_match
3. features = compare_fields(a, b)
4. score = weighted_sum(features)
5. if score >= tau_m:
6.     merged = resolve_conflicts(a, b, source_reliability)
7.     write_back(merged, provenance, confidence)
```

模块职责可以先拆成下面这样：

| 模块 | 输入 | 输出 | 职责 |
|---|---|---|---|
| `normalize_record` | 原始记录 | 规范化记录 | 清洗姓名、电话、日期、地址 |
| `blocking_pass` | 两条规范化记录 | 布尔值 | 判断是否进入候选 |
| `compare_fields` | 两条规范化记录 | 特征字典 | 计算名称、电话、地址等相似度 |
| `weighted_score` | 特征字典 + 权重 | 分数 | 综合证据 |
| `resolve_conflicts` | 已匹配记录组 | 主档 + provenance | 选值并记录来源 |

字段标准化是新手最容易低估的部分。看三个简单例子：

| 原始值 | 标准化后 | 说明 |
|---|---|---|
| 北京-海淀区 | 北京市海淀区 | 统一行政区格式 |
| 138-0000-0000 | 13800000000 | 去分隔符 |
| 2024/1/2 | 2024-01-02 | 统一日期格式 |

下面给一个可运行的最小 Python 实现。它不是生产系统，但足以把“匹配”和“合并”分开说明清楚。

```python
import re
from difflib import SequenceMatcher

SOURCE_RELIABILITY = {
    "registration": 0.8,
    "lab": 0.7,
    "insurance": 0.95,
    "his": 0.9,
}

def normalize_text(s: str) -> str:
    return re.sub(r"\s+", "", s.strip().lower())

def normalize_phone(phone: str) -> str:
    return re.sub(r"\D", "", phone or "")

def normalize_date(date_str: str) -> str:
    return (date_str or "").replace("/", "-")

def normalize_record(record: dict) -> dict:
    return {
        "name": normalize_text(record.get("name", "")),
        "phone": normalize_phone(record.get("phone", "")),
        "birth": normalize_date(record.get("birth", "")),
        "address": normalize_text(record.get("address", "")),
        "source": record.get("source", ""),
    }

def blocking_pass(a: dict, b: dict) -> bool:
    # 真实系统里 blocking 往往更复杂，这里只做玩具示例
    birth_ok = a["birth"] and a["birth"] == b["birth"]
    phone_suffix_ok = a["phone"] and b["phone"] and a["phone"][-4:] == b["phone"][-4:]
    return birth_ok or phone_suffix_ok

def sim_text(x: str, y: str) -> float:
    if not x or not y:
        return 0.0
    return SequenceMatcher(None, x, y).ratio()

def sim_exact(x: str, y: str) -> float:
    if not x or not y:
        return 0.0
    return 1.0 if x == y else 0.0

def compare_fields(a: dict, b: dict) -> dict:
    return {
        "name": sim_text(a["name"], b["name"]),
        "birth": sim_exact(a["birth"], b["birth"]),
        "phone": sim_exact(a["phone"], b["phone"]),
        "address": sim_text(a["address"], b["address"]),
    }

def weighted_score(features: dict, weights: dict) -> float:
    return sum(features[k] * weights[k] for k in weights)

def resolve_conflicts(records: list, field: str) -> tuple[str, dict]:
    votes = {}
    provenance = {}
    for r in records:
        value = r.get(field, "")
        if not value:
            continue
        source = r["source"]
        votes[value] = votes.get(value, 0.0) + SOURCE_RELIABILITY.get(source, 0.5)
        provenance.setdefault(value, []).append(source)

    best_value = max(votes, key=votes.get)
    return best_value, {"field": field, "chosen_from": provenance[best_value], "score": votes[best_value]}

weights = {"name": 0.35, "birth": 0.30, "phone": 0.20, "address": 0.15}
tau_m = 0.70

raw_a = {
    "name": "张三",
    "phone": "138-0000-0000",
    "birth": "1990/03/02",
    "address": "北京市海淀区",
    "source": "registration",
}
raw_b = {
    "name": "张 三",
    "phone": "",
    "birth": "1990-03-02",
    "address": "北京海淀区",
    "source": "insurance",
}

a = normalize_record(raw_a)
b = normalize_record(raw_b)

assert blocking_pass(a, b) is True

features = compare_fields(a, b)
score = weighted_score(features, weights)

assert features["birth"] == 1.0
assert score > 0.70

if score >= tau_m:
    merged_address, addr_prov = resolve_conflicts([raw_a, raw_b], "address")
    assert merged_address in {"北京市海淀区", "北京海淀区"}
    assert "chosen_from" in addr_prov
```

这段代码有三个要点：

1. `blocking_pass` 先把候选缩小，不让所有记录都两两比较。
2. `compare_fields` 只负责产出证据，不直接做最终决策。
3. `resolve_conflicts` 只在已经确认匹配后运行，并且单独记录来源。

真实工程例子里，这套代码会再往前扩展两层：

- 候选生成从“两条记录判断”变成“批量分桶 + 倒排索引 + 分批计算”。
- 合并从“两条记录合并”变成“多条记录聚成一个实体簇后再选主值”。

这也是为什么工程上要把“pairwise match”和“cluster merge”分开建模。前者处理记录对，后者处理实体簇，职责不同。

## 工程权衡与常见坑

实体对齐最常见的工程失败，不是模型不够高级，而是边界没控住。尤其是 blocking、缺失值处理、provenance 落库，这三个点最容易让系统在上线后失真。

先看常见坑：

| 坑点 | 典型表现 | 后果 | 规避方式 |
|---|---|---|---|
| blocking 过严 | 要求多个字段完全一致才入桶 | 漏召回真匹配 | 分层 blocking，先粗后细 |
| blocking 过松 | 姓名同音都入桶 | 候选爆炸，算力失控 | 增加辅助键，限制桶大小 |
| 字段相关性过强 | 姓名、拼音、姓名首字母都高权重 | 重复计分，误判偏高 | 做特征去相关或降权 |
| 缺失值处理错误 | 缺失当成不一致 | 召回率下降 | 区分“未知”和“冲突” |
| provenance 缺失 | 只存合并结果，不存来源 | 无法追责和回滚 | 字段级来源留痕 |
| 脏源污染 | 低质量来源覆盖高质量来源 | 主档失真 | 来源分级、隔离更新 |

特别说一下缺失值。缺失不是“不一致”。如果检验系统没有手机号，不代表它在反对挂号系统的手机号。它只是“没有提供证据”。这一点在医疗、政务、B2B 主数据里非常重要。Li 等人在 2022 年讨论的 data-adaptive Fellegi-Sunter 模型，一个关键点就是不要粗暴地把缺失等同于 disagreement，这会系统性损伤匹配表现。

再看一个常见业务坑。一个患者手机号缺失、身份证后四位一致、生日一致，但地址差异较大。如果只看地址就拒绝，会漏掉真实同人；如果只看生日又可能误合并同名同生日的人。正确做法是：

- 把地址差异记为弱负证据。
- 把手机号缺失记为未知。
- 把生日、证件后四位、来源可信度作为强证据综合判断。

合并值时也不能简单“最后写入覆盖前值”。更稳妥的规则是来源可靠度加权投票：

$$
v^*=\arg\max_v \sum_{s\in \mathcal{S}(v)} r_s
$$

如果某个地址值被两个中高可信来源支持，而另一个地址值只来自一个低可信来源，前者更适合作为主值。这个规则的优点是可解释，缺点是依赖可靠度先验。

权衡通常体现在下面几维：

| 指标 | 提高它通常意味着什么 | 代价 |
|---|---|---|
| 召回率 | 多抓真匹配 | 误匹配风险上升 |
| 精度 | 少误合并 | 漏匹配增多 |
| 可解释性 | 规则和概率模型更清楚 | 表达能力可能不如复杂模型 |
| 维护成本 | 规则少、特征稳 | 适应新数据慢 |
| 人工成本 | 中间区间交审核 | 人力投入增加 |

真实工程例子：知识图谱中的机构对齐，经常会遇到“清华大学附属医院”“清华附院”“THU Hospital”这类别名。若系统只依赖名称相似度，容易把不同附属机构混在一起；若同时引入上级机构、地址、官网域名、邻居关系，效果会明显更稳。但这也意味着特征工程复杂度和维护成本上升。

还有一个经常被忽略的点：传递性风险。若系统判断 A=B，B=C，就可能推导出 A=C；但现实里，A=B 和 B=C 中有一个可能是误匹配。于是一个错误会在聚类阶段放大。这就是为什么很多系统会在 pairwise 匹配后增加一致性检查、must-link/cannot-link 约束，甚至保留人工修复入口。

所以工程上最重要的不是“分数算得多复杂”，而是“错误发生后能不能看见、解释、隔离、回滚”。

## 替代方案与适用边界

实体对齐没有单一最优方案。方法选择取决于数据质量、业务风险和解释性要求。

先给一个总对比表：

| 方法 | 适用边界 | 优点 | 缺点 |
|---|---|---|---|
| 规则法 | 强业务键明确、字段稳定 | 实现快、可解释 | 脆弱，适应性差 |
| Fellegi-Sunter / 概率法 | 字段结构稳定、需解释性 | 理论清楚、能量化不确定性 | 假设较强，特征表达有限 |
| 监督学习法 | 有标注数据、特征丰富 | 能学复杂模式 | 标注贵，漂移时要重训 |
| 图方法 | 关系链强、需要传递推断 | 能利用邻居结构和簇信息 | 实现复杂，调参难 |
| 人工审核 | 高风险边界样本 | 风险可控 | 成本高、扩展差 |

规则法最适合“硬标识强”的场景。比如医院患者主数据里，证件号、生日、手机号、医保号的业务语义明确，可以先用确定性规则筛出高置信匹配，再用概率模型处理剩余模糊样本。这类场景里，规则 + 概率通常比直接上深度模型更稳，因为错误代价高，而且业务方通常需要解释“为什么合并”。

Fellegi-Sunter 或其变体适合字段清晰、缺失可建模、业务需要解释性的场景。它的优势是可以把“字段一致在匹配中常见、在非匹配中少见”这件事形式化。对初中级工程师来说，这是非常值得掌握的基线方法。

监督学习法适合已有较多标注对、特征丰富且数据分布相对稳定的场景。比如商品知识融合，标题、属性、品牌、类目、图片嵌入都可以进入模型。这时可以用逻辑回归、梯度提升树，甚至双塔编码器。缺点是标注成本高，而且模型漂移后需要重新训练。

图方法适合关系信息很强的场景。比如百科人物对齐，不仅看名字和生日，还看“毕业院校”“任职机构”“合作作者”“上下游关系”。如果两个实体的邻居结构也高度一致，那么即使名字不同，也可能是同一人。图方法的强项是利用上下文，弱项是复杂度和可解释性。

人工审核不是落后方案，而是高风险系统的必要兜底。尤其当误合并代价高于漏匹配时，中间区域样本进入人工复核往往是合理设计，不是系统失败。

玩具例子可以这样看：

- 若学生信息表中“学号”全局唯一，规则法就够了。
- 若只有姓名、手机号、宿舍号，且存在缺失，就更适合概率法。
- 若还要结合课程关系、社团关系判断同一人，图方法更有价值。

真实工程例子：

- 医院患者主数据：优先规则 + 概率，原因是字段语义明确，风险高，需要解释和审计。
- 百科人物、商品别名聚合：可能要用文本嵌入或图方法，因为别名噪声大、上下文关系强。
- 安全日志事件归并：常结合时间窗口、主机 ID、调用链、告警模板，属于半结构化对齐，规则与学习方法经常混用。

最后给一个明确边界：如果数据极度稀疏、字段缺失严重、没有稳定锚点，也没有可用关系结构，那么复杂模型也很难凭空解决实体对齐。问题本身的信息量不足时，最有效的办法往往不是换模型，而是补充数据采集规范、统一标识策略、加强来源治理。
