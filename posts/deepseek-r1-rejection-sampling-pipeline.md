## 核心结论

DeepSeek-R1 的关键不在于“只做一次强化学习，模型就自然学会复杂推理”，而在于把不同训练阶段组织成一条有明确分工的数据与能力闭环：先用 SFT 冷启动建立可控输出，再用推理 RL 强化长链路解题能力，再用拒绝采样把 RL 阶段产出的高方差答案筛成高质量监督数据，随后做全场景 SFT 恢复并统一通用能力，最后再做全场景 RL 校正行为偏好、格式稳定性与交付风格。

这里的 SFT 是监督微调，意思是直接给模型“输入-标准答案”对，让模型学习应该如何回答；RL 是强化学习，意思是让模型对同一个问题多次尝试，再根据结果质量给奖励，推动模型朝高奖励方向更新。前者解决“会不会按要求输出”，后者解决“会不会在复杂任务里探索出更强策略”。

这条流水线要解决的是一个工程上反复出现的矛盾：如果模型只在数学、代码、逻辑题上做强化训练，推理能力通常会提高，但写作、翻译、问答、风格一致性等通用能力容易漂移。输出可能更会“想”，却未必更会“交付”。DeepSeek-R1 的做法不是把 RL 阶段生成的所有答案都继续拿去训练，而是先做拒绝采样。拒绝采样的核心不是“多生成”，而是“多生成之后只保留真正合格的少数答案”，把探索阶段的噪声输出压缩成可以稳定回流训练的高质量数据。

一个适合新手理解的玩具例子是：先让学生看少量高质量范文，学会题目要求、答案格式和基本表达；再让他反复做推理题，鼓励尝试不同解法；然后从 30 份草稿里只选出 1 份同时满足“推理正确、结构清晰、语言合规”的答案收入题库；最后再把这些优质推理题答案与普通写作、翻译、问答样本混合训练。这样训练出来的模型，不只是“某些题会做”，而是“能在不同任务里稳定按要求输出”。

| 阶段 | 输入数据 | 主要目标 | 直接产物 |
|---|---|---|---|
| SFT 冷启动 | 少量高质量人工/蒸馏样本 | 建立基本指令跟随、格式控制与答题习惯 | 初始可用模型 |
| 推理 RL | 推理任务 prompt + 奖励机制 | 强化长链路推理、探索解题策略 | 推理更强的 checkpoint |
| 拒绝采样 | RL checkpoint 的多次采样结果 | 过滤低质量输出，构造干净推理数据 | 高质量推理数据集 |
| 全场景 SFT | 拒绝采样数据 + 写作/翻译/问答数据 | 统一推理能力与通用能力 | 更均衡的通用模型 |
| 全场景 RL | 多场景任务 + 奖励信号 | 进一步校正行为、格式与用户偏好 | 最终 R1 能力形态 |

可以把整个过程概括成一句话：推理 RL 负责“把能力顶上去”，拒绝采样负责“把噪声压下去”，全场景 SFT 和全场景 RL 负责“把模型重新拉回可交付状态”。

## 问题定义与边界

问题定义很明确：如何让一个模型在推理、写作、翻译、问答等不同场景下同时保持稳定表现，而不是在某一项能力提升时，把其他能力、输出风格和可控性一起带偏。

这个问题的难点不在“如何让模型生成更多推理”，而在“哪些推理结果值得进入下一轮训练”。RL 阶段本质上鼓励探索。探索的好处是可以找到更强解法，代价是输出分布会变得很散。一个 RL checkpoint 往往会同时产出以下几类结果：

| 输出类型 | 是否可能高奖励 | 是否适合直接回流训练 | 原因 |
|---|---|---|---|
| 正确、结构清晰、语言稳定 | 是 | 是 | 既有内容质量，也具备监督训练价值 |
| 正确但格式混乱 | 可能 | 否 | 模型会学到不稳定模板 |
| 局部正确但答案不完整 | 可能 | 通常否 | 容易把“不完整回答”学成正常模式 |
| 语言混杂、推理痕迹失控 | 可能 | 否 | 会污染写作、翻译和问答分布 |
| 完全错误 | 否 | 否 | 直接降低训练数据质量 |

因此，边界也很明确：不是所有 RL 生成结果都能回流到训练集。若把噪声输出直接当作 SFT 标签，相当于告诉模型“这种回答就是标准答案”。污染一旦发生，影响的不是单个任务，而是整个输出分布。

可以把筛选逻辑写成：

$$
D_{RS}=\{r_i \in R \mid f(r_i)=1 \land q(r_i)\ge \tau_q \land s(r_i)\ge \tau_s\}
$$

其中：

- $R$ 是某个 prompt 经过多次采样得到的候选回答集合。
- $f(r_i)$ 是格式约束函数，判断这条答案是否满足语言、结构、长度、模板等基本要求，结果只取 0 或 1。
- $q(r_i)$ 是质量分数，可以来自规则判题器、程序执行结果、强模型评估或多者组合。
- $s(r_i)$ 是稳定性交付分数，用来衡量答案是否适合作为监督样本，例如是否简洁、是否结论完整、是否没有明显风格漂移。
- $\tau_q$ 和 $\tau_s$ 是阈值，分别表示“内容质量至少要达标”和“交付质量至少要达标”。

如果进一步写成每个 prompt 的单样本保留过程，可以表示为：

$$
r^* = \arg\max_{r_i \in R'} q(r_i), \quad R'=\{r_i \in R \mid f(r_i)=1\}
$$

当且仅当 $q(r^*) \ge \tau_q$ 且 $s(r^*) \ge \tau_s$ 时，$r^*$ 才进入最终数据池。也就是说，流程不是“先找最高分”，而是“先过滤掉不适合作为标准答案的输出，再从剩余结果里选最好的”。

例如，一个中文推理 prompt 的目标输出应当是中文、结构清楚、结论明确。如果模型给出的答案中英夹杂、段落无限膨胀、推理区和答案区混在一起，即使其中某些中间推导是对的，也不应进入 $D_{RS}$。拒绝采样要追求的不是“勉强能看懂”，而是“可以被后续 SFT 安全复用”。

对新手来说，最值得记住的是这条边界：RL 生成的答案不等于训练标签，只有同时满足“正确”和“可教”的答案，才是下一阶段真正需要的数据。

## 核心机制与推导

核心机制可以拆成两层。第一层是拒绝采样，负责把探索式生成结果转成高质量监督数据。第二层是多阶段训练编排，负责把推理能力与通用能力重新接回一条统一链路。

先看拒绝采样。

对每个推理 prompt，系统不会只生成一次答案，而是从推理 RL 的 checkpoint 采样多次。采样的意思是：在同一道题上让模型走出多个不同回答路径，因为 RL 阶段的模型对同一问题可能给出质量差异很大的结果。然后，对每个候选答案执行至少两类检查：

1. 格式检查：语言是否符合要求，结构是否完整，长度是否合理，是否满足模板约束。
2. 质量检查：结论是否正确，推理是否可靠，是否达到奖励阈值。

只有同时通过两类检查的答案，才允许进入后续训练集。这一步的目的不是扩大数据量，而是把 RL 阶段的高方差输出压缩为低噪声监督样本。

把这个过程写成概率视角会更清楚。设：

- 单次采样通过格式检查的概率为 $p_f$
- 单次采样在通过格式检查后内容也正确的条件概率为 $p_q$
- 单次采样最终可保留的概率为 $p = p_f \cdot p_q$

若对同一 prompt 采样 $n$ 次，则至少获得 1 条合格样本的概率为：

$$
P(\text{retain} \ge 1)=1-(1-p)^n
$$

这个公式说明了一个核心工程事实：单次采样合格率低，并不意味着流程无效，只要增加采样次数，拿到至少一个好答案的概率会快速上升。

举一个具体玩具例子。假设：

- 单次格式通过率 $p_f = 0.2$
- 在格式通过的前提下，内容正确率 $p_q = 0.2$
- 则单次最终保留率 $p = 0.04$

如果每个 prompt 采样 30 次，那么至少保留 1 条的概率为：

$$
1-(1-0.04)^{30} \approx 1-0.96^{30} \approx 70.6\%
$$

也就是说，即使单次成功率只有 4%，多次采样后，约七成题目仍然能得到至少一条可用样本。这正是“多生成 + 严筛选”在工程上成立的原因。

再看保留规模估算。假设有 20,000 个推理 prompt，每个 prompt 采样 30 次，总候选数是：

$$
N_{cand}=20000 \times 30 = 600000
$$

若平均最终保留率为 3%，则可得到：

$$
|D_{RS}| \approx 0.03 \times 600000 = 18000
$$

若系统采用“每题最多保留 1 条”的策略，而 20,000 道题里有 16,000 道至少找到 1 条合格答案，则：

$$
|D_{RS}| = 16000
$$

如果题库更大、采样次数更高，或者允许每题保留多条高质量变体，最终规模就可能扩展到几十万。论文解读里常见的“通过拒绝采样构造约 80 万条高质量数据”，本质上对应的就是这个机制：候选池极大，但保留标准严格。

接着看多阶段训练编排。

| 训练阶段 | 输入来源 | 主要优化对象 | 是否使用拒绝采样 |
|---|---|---|---|
| SFT 冷启动 | 少量高质量监督样本 | 基本指令遵循、输出格式、基础知识调用 | 否 |
| 推理 RL | 数学、代码、逻辑等推理任务 | 长链路推理、探索式求解、策略优化 | 否 |
| 拒绝采样构数 | RL checkpoint 的多次采样结果 | 生成可回流的高质量推理样本 | 是 |
| 全场景 SFT | 拒绝采样推理数据 + 通用任务数据 | 推理能力与写作/翻译/问答能力统一 | 间接使用 |
| 全场景 RL | 多场景任务与偏好/规则奖励 | 行为一致性、风格稳定性、交付偏好 | 可结合 |

这张表最重要的含义是：拒绝采样不是一个孤立技巧，而是连接“推理 RL”与“全场景 SFT”的桥。如果没有这座桥，RL 阶段探索出的能力很难以稳定方式进入通用模型；如果只有这座桥而没有后续全场景 SFT，模型又可能在推理增强之后继续牺牲写作、翻译和问答体验。

可以把整个机制总结成一个更容易记忆的链条：

$$
\text{SFT}_{cold} \rightarrow \text{RL}_{reasoning} \rightarrow \text{RS} \rightarrow \text{SFT}_{general} \rightarrow \text{RL}_{general}
$$

其中每一段的职责不同：

- `SFT_cold` 负责把模型拉进“能按要求回答”的状态。
- `RL_reasoning` 负责把模型推向“更会做复杂题”的状态。
- `RS` 负责筛掉探索噪声，保留可以复用的优质答案。
- `SFT_general` 负责把推理能力与通用能力整合。
- `RL_general` 负责把最终行为修到更稳、更一致。

真实工程例子可以想象成一个多语言写作助手。产品既要能做复杂分析，也要能稳定翻译、总结、写邮件。如果只做推理 RL，模型可能更会算、更会分析，但也可能变得啰嗦、风格漂移、语言混杂。把 RL 输出经过拒绝采样后，再与翻译、摘要、写作、问答语料一起做全场景 SFT，才能把“推理增强”真正变成“可交付能力”。

## 代码实现

下面用一个可直接运行的 Python 玩具实现，演示“多次采样 -> 格式检查 -> 质量判定 -> 去重 -> 保存样本 -> 与通用数据混合”的核心流程。代码仍然是简化版，但流程和真实工程的关键结构是一致的。

```python
from dataclasses import dataclass
from typing import List, Dict, Iterable
import re


@dataclass
class Sample:
    prompt: str
    answer: str
    reward: float
    source: str  # "reasoning_rs" or "general"


def generate_candidates(prompt: str, n: int) -> List[str]:
    """玩具候选生成器。真实场景里这里会调用 RL checkpoint。"""
    base = [
        "先设 x=2，代入后得到结果为 8。结论：答案是 8。",
        "Let us think step by step... 最后答案 8",
        "答案是 8" + "。" * 200,
        "先计算平方，再乘 2，得到 8。结论：答案是 8。",
        "先计算 2^2=4，再乘 2 得到 8。结论：答案是 8。"
    ]
    return [base[i % len(base)] for i in range(n)]


def detect_language(answer: str) -> str:
    chinese_chars = sum("\u4e00" <= ch <= "\u9fff" for ch in answer)
    ascii_letters = sum(ch.isascii() and ch.isalpha() for ch in answer)
    if chinese_chars > 0 and ascii_letters > 10:
        return "mixed"
    if chinese_chars > 0:
        return "zh"
    if ascii_letters > 0:
        return "en"
    return "unknown"


def is_format_ok(answer: str, expected_lang: str = "zh", max_len: int = 80) -> bool:
    if len(answer) > max_len:
        return False
    if "结论：" not in answer:
        return False
    if "\n\n\n" in answer:
        return False

    lang = detect_language(answer)
    if expected_lang == "zh" and lang != "zh":
        return False
    if expected_lang == "en" and lang != "en":
        return False
    return True


def reward(answer: str, expected_final: str = "8") -> float:
    """
    玩具打分器。
    真实工程里，这里可能是规则判题器、程序执行器、强模型评估器，或多者组合。
    """
    if expected_final not in answer:
        return 0.0

    score = 0.6
    if "先" in answer:
        score += 0.2
    if "结论：" in answer:
        score += 0.2
    return min(score, 1.0)


def normalize_text(text: str) -> str:
    return re.sub(r"\s+", "", text)


def deduplicate(samples: Iterable[Sample]) -> List[Sample]:
    seen = set()
    result = []
    for sample in samples:
        key = (sample.prompt, normalize_text(sample.answer))
        if key in seen:
            continue
        seen.add(key)
        result.append(sample)
    return result


def rejection_sampling(
    prompts: List[str],
    n_samples: int,
    tau: float,
    expected_lang: str = "zh",
    one_per_prompt: bool = True,
) -> List[Sample]:
    retained: List[Sample] = []

    for prompt in prompts:
        candidates = generate_candidates(prompt, n_samples)
        passed_for_prompt: List[Sample] = []

        for ans in candidates:
            if not is_format_ok(ans, expected_lang=expected_lang):
                continue

            w = reward(ans)
            if w < tau:
                continue

            passed_for_prompt.append(
                Sample(
                    prompt=prompt,
                    answer=ans,
                    reward=w,
                    source="reasoning_rs",
                )
            )

            if one_per_prompt:
                break

        retained.extend(passed_for_prompt)

    return deduplicate(retained)


def build_sft_dataset(
    reasoning_prompts: List[str],
    general_samples: List[Dict[str, str]],
    n_samples: int = 30,
    tau: float = 0.8,
) -> List[Sample]:
    rs_data = rejection_sampling(
        prompts=reasoning_prompts,
        n_samples=n_samples,
        tau=tau,
        expected_lang="zh",
        one_per_prompt=True,
    )

    general_data = [
        Sample(
            prompt=item["prompt"],
            answer=item["answer"],
            reward=1.0,
            source="general",
        )
        for item in general_samples
    ]

    return deduplicate(rs_data + general_data)


if __name__ == "__main__":
    dataset = build_sft_dataset(
        reasoning_prompts=[
            "求 2^2 * 2 的值",
            "已知 2 的平方再乘 2，结果是多少？"
        ],
        general_samples=[
            {
                "prompt": "把这句话翻译成英文：今天下雨。",
                "answer": "It is raining today."
            },
            {
                "prompt": "写一封请假邮件。",
                "answer": "Subject: Leave Request\n\nI would like to request one day of leave."
            }
        ],
        n_samples=30,
        tau=0.8,
    )

    assert len(dataset) == 4
    assert sum(x.source == "reasoning_rs" for x in dataset) == 2
    assert sum(x.source == "general" for x in dataset) == 2
    assert all(x.answer for x in dataset)
    assert all(x.reward >= 0.8 for x in dataset if x.source == "reasoning_rs")

    for item in dataset:
        print(f"[{item.source}] {item.prompt} -> {item.answer}")
```

这段代码体现了几个真实工程里不可省略的点。

第一，拒绝采样不是“多生成一些答案再随机挑一个”，而是显式地把生成、检查、打分、保留拆开。模型在 RL 阶段负责探索，数据管线负责筛选，二者职责不同。

第二，格式约束和质量判定必须分开。`is_format_ok` 处理的是“这个答案适不适合作为监督标签”，`reward` 处理的是“这个答案本身质量够不够高”。这两个判断常常不一致。一个答案可能内容基本对，但格式混乱；也可能格式漂亮，但结论错了。两种都不该进入训练集。

第三，数据进入全场景 SFT 之前，不能只保留推理样本。代码里的 `build_sft_dataset` 把 `reasoning_rs` 和 `general` 合并，体现的正是“推理增强后还要和通用任务统一训练”的思路。

第四，哪怕是玩具实现，也应该加入去重。因为多次采样时常见情况不是“30 个完全不同答案”，而是“几个模板在来回变体”。如果不去重，看似样本很多，实际上信息密度很低。

如果把这段玩具代码映射到真实系统，对应关系大致如下：

| 玩具函数 | 真实系统里的对应模块 | 作用 |
|---|---|---|
| `generate_candidates` | 推理 RL checkpoint 采样服务 | 对同一 prompt 生成多条候选 |
| `is_format_ok` | 规则过滤器 / 语言检测器 / 模板校验器 | 筛掉不适合回流训练的输出 |
| `reward` | 判题器 / 执行器 / 强模型评估器 | 判断内容质量是否达标 |
| `deduplicate` | 去重与相似度过滤模块 | 提升数据信息密度 |
| `build_sft_dataset` | 数据编排管线 | 混合推理数据与通用数据 |

真实工程中通常还会补几层能力：

- 并行采样：同一 prompt 同时发起多个生成任务，降低整体等待时间。
- 多级评估：规则过滤之后，再由判题器或更强模型复核。
- 长度与成本控制：长推理链可能提高正确率，但也会显著抬高生成成本。
- 失败重试：对高价值 prompt 允许追加采样预算，而不是一次失败就放弃。
- 数据配比策略：控制推理样本和写作/翻译/问答样本的混合比例，避免再次失衡。

新手读这部分代码时，最容易误解的一点是：拒绝采样的“核心难度”不在采样函数，而在过滤和评估。真正决定数据质量的不是“模型一次能写多少”，而是“系统到底保留什么”。

## 工程权衡与常见坑

拒绝采样的核心收益是显著提高数据质量，但代价也非常直接：吞吐下降、成本上升、系统复杂度增加。你会生成大量候选，却只保留很少一部分。表面看这像算力浪费，实际上这是在用额外生成成本换训练数据纯度。对生产级模型来说，这通常是值得的，因为错误监督比没有监督更危险。

可以把这件事写成一个简单成本公式。设：

- 每次采样平均成本为 $c$
- 每个 prompt 采样 $n$ 次
- 总 prompt 数为 $M$
- 最终保留率为 $\rho$

则总生成成本近似为：

$$
Cost \approx M \times n \times c
$$

而每条最终保留样本的平均成本约为：

$$
Cost_{per\ retained} \approx \frac{M \times n \times c}{M \times n \times \rho} = \frac{c}{\rho}
$$

这个式子说明了一个直观结论：保留率越低，每条有效样本越贵。所以工程上既不能盲目放宽过滤，也不能盲目收紧阈值。过宽会污染数据，过严会把样本筛没。

常见坑如下：

| 常见坑 | 现象 | 风险 | 规避策略 |
|---|---|---|---|
| 格式混乱 | 答案区、推理区、结论区混在一起 | SFT 学到不稳定模板 | 明确结构规则，增加格式判别函数 $f(r)$ |
| 中英夹杂 | 中文任务中出现英文推理短语 | 写作和翻译质量下降 | 加入语言检测与语种一致性校验 |
| 段落爆长 | 推理链极长，超出交付长度 | 成本升高，用户体验变差 | 设置长度上限与压缩策略 |
| 奖励投机 | 模型学会骗过打分器 | 高分样本实际不可用 | 用多评估器交叉验证，加入人工抽检 |
| 数据重复 | 候选只是轻微改写 | 有效信息密度低 | 做相似度去重与模板去重 |
| 推理占比过高 | 后续 SFT 基本全是推理样本 | 通用能力再次漂移 | 控制推理与通用数据配比 |
| 只看正确率 | 数学对了但风格很差 | 模型可交付性下降 | 把“正确”与“可教”分开评估 |
| 阈值过严 | 很多题一条也留不下 | 数据规模不足 | 分层阈值，按题型动态调整采样预算 |

数据清洗顺序通常可以写成：

`RL checkpoint 采样 -> 结构格式检查 -> 语言一致性检查 -> 正确性/奖励评估 -> 去重 -> 写入 D_RS -> 与通用数据混合 SFT`

这里最容易被低估的是“双重过滤”。只做奖励打分不够，因为高分答案也可能格式差、风格差、语言不稳；只做格式检查也不够，因为排版合规不代表内容正确。生产级系统需要同时保证两件事：

1. 这条答案是对的。
2. 这条答案适合被模型再次学习。

如果只满足第一条，模型可能学到奇怪表达方式；如果只满足第二条，模型可能学到漂亮但错误的答案。拒绝采样的工程价值，正是同时守住这两条线。

一个真实感更强的例子是多语言写作助手。假设产品支持“中文分析 + 英文邮件输出”。如果推理 RL 阶段大量生成“中文解释夹英文思路”的答案，再把这些内容直接混入写作训练集，模型后续就可能把这种风格当成正常写作模板。结果是：用户明明只想要一封简洁英文邮件，模型却先输出一段中英混杂的推理过程。拒绝采样的作用，就是在训练前把这种本不该交付给用户的模式切断。

可以把工程上的取舍概括成一张更直观的表：

| 目标 | 如果不过滤 | 如果过度过滤 | 合理做法 |
|---|---|---|---|
| 数据规模 | 大 | 小 | 在质量可控前提下逐步放大 |
| 数据质量 | 不稳定 | 很高但可能不足 | 用分层阈值平衡 |
| 训练稳定性 | 差 | 高 | 保证回流数据风格一致 |
| 研发成本 | 低 | 高 | 先搭最小可用过滤链，再逐步加评估器 |

对新手来说，最实用的理解方式是：拒绝采样不是“高级花活”，而是一个数据入口闸门。闸门宽了，脏数据进来；闸门窄了，没数据可用。工程难度恰恰在这个闸门如何调得既不过宽，也不过窄。

## 替代方案与适用边界

不是所有系统都必须采用“拒绝采样 + 多阶段训练”。但如果目标是把推理能力和通用能力统一在一个生产级通用助手里，这套方案有明显优势，因为它同时处理了能力提升、数据回流和行为稳定三个问题。

先看几种常见策略对比：

| 策略 | 数据质量 | 训练效率 | 通用能力稳定性 | 系统复杂度 | 适用场景 |
|---|---|---|---|---|---|
| 拒绝采样 + 多阶段训练 | 高 | 中到低 | 高 | 高 | 需要统一推理与通用能力的通用助手 |
| 全部采样直接回流 | 低到中 | 高 | 低 | 低 | 数据极度稀缺、容忍噪声的实验场景 |
| 单任务流水线 | 中 | 高 | 低到中 | 中 | 只做专项能力，如纯数学解题器 |
| 只做 SFT 不做 RL | 中 | 中 | 中到高 | 低到中 | 更重视稳定输出、较少需要复杂探索的场景 |
| 只做推理 RL 不做回流整理 | 低到中 | 中 | 低 | 中 | 研究性验证，不直接面向真实用户 |

第一种策略最稳，但成本最高。它要求你维护采样、过滤、评估、去重、数据配比、后续全场景训练等整套管线。好处是模型不只在推理 benchmark 上分数高，而且在真实产品里更可控。

第二种策略是“全部采样直接回流”。优点是简单，几乎没有额外数据工程成本；缺点也同样直接，低质量答案会混进 SFT 标签，模型会把不该学的表达、结构和风格一并学走。做小实验时，这种方法也许可接受；做面向用户的产品时，风险通常偏高。

第三种策略是“单任务流水线”。例如只做数学推理器，只关心解题准确率，不关心长文写作、翻译或问答风格。这种做法可以避开一部分多场景统一的难题，但代价是能力边界很窄。一个模型如果只在专项任务里表现好，不代表它适合作为通用助手。

第四种策略是“只做 SFT，不做 RL”。这种方法在稳定性上通常更容易控制，特别适合输出格式明确、探索价值有限的任务。但如果目标里包含复杂推理、长链路搜索、程序求解等能力，只靠 SFT 往往难以逼出 RL 带来的探索增益。

一个适合新手理解的反例是：如果把推理 RL 输出原样混入写作数据，模型很可能学到“语义堆砌、过度展开、中英混杂”的风格。这样即使它在推理基准上得分更高，写作任务却会明显变差。换句话说，拒绝采样不是锦上添花，而是在“能力增强”和“能力失控”之间划一道边界。

适用边界也要说清楚。拒绝采样更适合以下情况：

- 候选生成成本可以接受。
- 有相对可靠的格式规则与质量评估器。
- 目标是生产级模型，而不是一次性实验。
- 需要统一推理、写作、翻译、问答等多种任务。
- 愿意投入额外工程成本做数据清洗和混合训练。

不太适合的情况则包括：

- 数据极少，采样后几乎留不下样本。
- 评估器质量差，筛选结果不可信。
- 任务目标单一，只追求某一专项指标。
- 成本约束极强，无法负担大量候选生成。

可以把适用条件概括成一个简单判断：

$$
\text{Use RS} \iff \text{高质量评估可得} \land \text{采样成本可承受} \land \text{目标是统一多能力}
$$

如果评估器不可靠，拒绝采样就可能退化成“随机丢数据”；如果采样成本太高，系统无法持续扩展；如果目标本来就是单任务专项模型，复杂的多阶段编排也未必必要。

因此，拒绝采样最重要的前提不是“模型够强”，而是“你有能力判断什么答案值得保留”。没有这个前提，再复杂的多阶段训练也很难落到稳定结果上。

1. 先看整体训练流水线，建立“为什么不是单阶段训练”的全局图景。
2. 再看推理 RL 与拒绝采样之间的关系，理解为什么 RL 输出不能直接全部回流。
3. 然后看全场景 SFT 与全场景 RL，理解为什么推理增强之后还要重新统一通用能力。
4. 最后再回到公式、样本规模和工程细节，补充对保留率、成本和数据质量的直觉。

更具体地说，阅读时可以重点追踪下面几个问题：

| 阅读问题 | 为什么重要 |
|---|---|
| 冷启动 SFT 解决了什么问题 | 它决定模型是否一开始就具备可控输出和基本格式 |
| 推理 RL 强化了什么能力 | 它决定模型是否能在复杂任务上进行有效探索 |
| 为什么要拒绝采样 | 它决定 RL 阶段的高方差输出能否安全回流 |
| 为什么还要全场景 SFT | 它决定推理增强后通用能力能否恢复并统一 |
| 为什么最后还要全场景 RL | 它决定最终行为、偏好和风格是否稳定 |

这一组资料和问题合起来，能帮助读者避免两个常见误解。第一个误解是把 DeepSeek-R1 理解成“单靠 RL 就突然获得推理能力”；第二个误解是把拒绝采样理解成“只是为了多造一点数据”。更准确的理解是：DeepSeek-R1 依赖的是一条分工明确的训练流水线，而拒绝采样是把“推理能力增强”转成“可稳定训练、可稳定交付”的关键桥梁。
