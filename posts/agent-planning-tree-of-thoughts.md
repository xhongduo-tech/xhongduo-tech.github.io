## 核心结论

Tree-of-Thoughts，简称 ToT，可以直译为“思路树搜索”。它不是把 Chain-of-Thought，简称 CoT，也就是“单链条逐步推理”写得更长，而是把中间想法 $z$ 当成一个个可扩展、可比较、可回退的节点，显式组织成一棵搜索树。

对初学者可以先记一句话：CoT 是“一路写到底”，ToT 是“先想几个方向，再挑最像答案的方向继续走”。

它的核心价值不在“多生成几步文本”，而在“把分叉、比较、剪枝变成流程”。这件事只有在任务本身存在中间状态时才有意义。所谓中间状态，就是问题做到一半时，你手里会留下一个可检查、可延续的局部结果。比如数独盘面、代码补丁、路线规划、数学式子，它们都不是一句话结束，而是会经历多个会影响终局的局部选择。

ToT 常用的抽象写法是：

$$
Z_t = \{z_t^i\}_{i=1..m}
$$

表示第 $t$ 轮生成出的 $m$ 个候选思路；然后用评分函数

$$
v(x, z_t^i)
$$

衡量输入 $x$ 下候选 $z_t^i$ 的质量；最后保留前 $K$ 个：

$$
S_t = TopK(Z_t, v, K)
$$

再继续展开。整件事本质上是“显式树搜索 + 评分或投票 + 剪枝”。

一个玩具例子是“用 4、5、6、10 算出 24”。不是让模型直接一路算，而是先分叉：

- `10 - 4 = 6`
- `10 * 4 = 40`
- `5 + 6 = 11`

然后评估哪个分支更接近可达 24 的状态，再保留高分分支继续算。这里关键不是第一步一定正确，而是系统允许“先试几条路，再筛掉差的路”。

下表可以直接看出 CoT 和 ToT 的差异：

| 维度 | CoT | ToT |
|---|---|---|
| 推理结构 | 单链 | 树状分叉 |
| 中间状态 | 通常只写下来，不显式管理 | 作为节点显式保存 |
| 失败处理 | 一条路错了通常整段偏掉 | 可以丢弃坏分支、继续好分支 |
| 适合任务 | 直线型推理 | 需要搜索、回溯、比较的任务 |
| 成本 | 低 | 更高 |

所以最准确的结论是：ToT 不是“万能更强推理”，而是把大模型从“想法生成器”升级为“想法生成器 + 启发式搜索器”。当任务需要显式探索多个中间方案时，它很有价值；否则只是增加成本。

## 问题定义与边界

ToT 解决的问题，不是“模型不会思考”，而是“模型只走一条路时，容易过早承诺到错误方向”。所谓过早承诺，就是模型在前几步做了一个局部选择，后面即使还在认真推理，也只是沿着错方向越走越远。

因此，ToT 更适合这类问题：

1. 存在明确的中间状态。
2. 中间状态之间可以比较好坏。
3. 局部选择会影响最终结果。
4. 必要时需要回溯，也就是放弃当前路线换路走。

可以把它记成一句简式定义：

$$
ToT = 显式树搜索 + 评分/投票 + 剪枝
$$

如果一个任务根本没有“候选路径”这个概念，那 ToT 往往没有意义。比如“把这段文字改成固定格式摘要”，通常只要信息提取准确、模板填充正确，直接生成或普通 CoT 就够了。这里没有多少值得搜索的分叉空间，因为候选答案之间的优劣主要靠最终格式是否对齐，而不是靠多轮中间状态递进。

反过来，像代码修复、数学证明、数独、博弈走子、复杂规划，它们的中间状态都能被保存和比较，所以更适合 ToT。

| 任务特征 | 适合 ToT | 不适合 ToT |
|---|---|---|
| 是否存在中间状态 | 有，例如部分代码补丁、局部式子 | 没有，基本是一口气输出 |
| 是否有明显分叉 | 有多种候选路径 | 候选几乎等价或不存在 |
| 是否能评估中间结果 | 能，靠规则、测试、启发式 | 很难，只有最终主观感受 |
| 是否需要回溯 | 需要 | 不需要 |
| 主要瓶颈 | 找路径 | 找信息或套模板 |

举两个对比很清楚。

玩具例子：24 点。你每一步都会留下一个新的数字集合，这就是中间状态，而且明显有多条路可以比较，天然适合 ToT。

真实工程例子：代码修复 Agent。假设一个接口返回 500，可能原因有参数校验、空指针、数据库字段不兼容、缓存脏数据。这里如果只让模型直接给一个补丁，它会押注某个原因；而 ToT 可以先生成多个修复方向，再用静态检查、单测、日志规则做评分，最后只继续最有希望的分支。

新手可以用一句话判断要不要上 ToT：能不能把问题拆成几个可比较的候选方案？如果不能，就没必要。

## 核心机制与推导

ToT 的基本循环只有三步：生成、评估、选择。

第一步，生成候选。给定输入 $x$ 和当前状态，模型生成一组候选思路：

$$
Z_t = \{z_t^1, z_t^2, ..., z_t^m\}
$$

这里的 $m$ 是生成宽度，表示每轮分叉多少条路。

第二步，评估候选。对每个候选计算分数：

$$
v(x, z_t^i)
$$

这个 $v$ 可以是模型自评，也可以是投票器、规则打分器、程序验证器。验证器就是“不是靠语言感觉，而是靠外部规则真查”的组件，比如运行测试、检查表达式是否有效、检查路径是否可达。

第三步，选择候选。保留前 $K$ 个：

$$
S_t = TopK(Z_t, v, K)
$$

其中 $K$ 是保留宽度。保留后的分支继续展开，直到达到最大深度 $T$ 或命中终止条件。

简化伪代码如下：

```text
states = [initial_state]

for t in 1..T:
  candidates = []
  for s in states:
    candidates += generate(s, m)
  scores = evaluate(candidates)
  states = TopK(candidates, scores, K)

return best_terminal_state(states)
```

这里三个参数最关键：

| 参数 | 含义 | 影响 |
|---|---|---|
| $m$ | 每轮生成多少候选 | 越大越容易找到好路，但更贵 |
| $K$ | 每轮保留多少分支 | 越大越不容易错剪，但更慢 |
| $T$ | 搜索深度 | 越深越能处理长链任务，但成本上升 |

可以用一个最小数值例子把这三步走完。

输入数字：`4, 5, 6, 10`，目标：`24`。假设第 1 轮设置 $m=3, K=1$。

生成 3 个候选状态：

- A：`10 - 4 = 6`，剩余状态变成 `{6, 5, 6}`
- B：`10 * 4 = 40`，剩余状态变成 `{40, 5, 6}`
- C：`5 + 6 = 11`，剩余状态变成 `{4, 10, 11}`

评估器给分：

- $v(x, A)=0.9$
- $v(x, B)=0.2$
- $v(x, C)=0.1$

于是

$$
S_1 = TopK(\{A,B,C\}, v, 1) = \{A\}
$$

只保留 A 继续展开。下一轮再从 `{6, 5, 6}` 生成候选，比如：

- `5 * 6 = 30`，剩余 `{30, 6}`
- `5 + 6 = 11`，剩余 `{11, 6}`

若评估器认为 `30` 更接近能凑出 `24`，就保留它。再下一步得到 `30 - 6 = 24`，搜索成功。

注意一个容易误解的点：高分分支不保证正确，它只是“更值得继续试”。ToT 的本质不是神奇预测，而是更合理地分配计算预算。

从搜索角度看，CoT 像深度优先地沿一条链往下写，ToT 更像受启发式函数引导的宽度受控搜索。也正因此，它特别依赖评估器质量。树能不能长对，很多时候不取决于生成器有多能写，而取决于评分器能不能把坏路早点砍掉。

## 代码实现

官方实现里，ToT 的工程形态并不复杂。入口通常在 `run.py`，搜索主逻辑在 `bfs.py`。前者负责“选任务、设参数、启动运行”，后者负责“按生成-评估-选择的循环执行树搜索”。

对新手最重要的不是逐行啃源码，而是先建立参数和流程的对应关系。常见三个核心旋钮是：

- `n_generate_sample`：每轮生成多少候选。
- `n_evaluate_sample`：每个候选评估多少次。
- `n_select_sample`：每轮保留多少分支。

可以把它们理解为“生成宽度、评分稳健度、保留宽度”。

| 配置项 | 作用 | 直接影响 |
|---|---|---|
| `n_generate_sample` | 每轮生成多少候选 | 越大越宽、越贵 |
| `n_evaluate_sample` | 每个候选评估多少次 | 越大越稳、越慢 |
| `n_select_sample` | 每轮保留多少分支 | 越大越不容易错剪，但成本更高 |

`run.py` 的职责通常包括：

| 模块职责 | 说明 |
|---|---|
| 任务选择 | 比如 24 点、创意写作、数独等 |
| 参数注入 | 把深度、宽度、评估次数传给搜索器 |
| 运行控制 | 启动实验、记录日志、输出结果 |

`bfs.py` 的名字来自 Breadth-First Search，简称 BFS，意思是“按层扩展的搜索”。但在 ToT 实现里，它通常不是纯粹教科书式 BFS，而是“按层生成 + 打分 + Top-K 剪枝”的近似宽搜。

可把它概括成下面这样的代码框图：

```python
from typing import List, Tuple

def score_24_state(nums):
    # 启发式：离 24 越近越高分
    best = min(abs(x - 24) for x in nums)
    return -best

def generate_24_states(nums):
    a, b, *rest = nums
    return [
        [a + b, *rest],
        [a - b, *rest],
        [a * b, *rest],
    ]

def topk(states: List[list], k: int) -> List[list]:
    return sorted(states, key=score_24_state, reverse=True)[:k]

def toy_tot(initial, depth=2, branch=3, keep=1):
    states = [initial]
    for _ in range(depth):
        candidates = []
        for s in states:
            candidates.extend(generate_24_states(s)[:branch])
        states = topk(candidates, keep)
    return states[0]

result = toy_tot([10, 4, 5])
assert isinstance(result, list)
assert len(result) >= 1
```

上面这个代码不是官方源码复刻，而是一个能运行的最小版本，用来说明流程。真实仓库里，生成和评估不是手写函数，而是通过语言模型提示词完成；任务状态也不是简单整数列表，而是任务相关的文本或结构化状态。

再看一个更贴近“24 点”的可运行玩具实现。它不用大模型，而是用规则搜索模拟 ToT 的“生成-评分-剪枝”。

```python
from itertools import combinations

def generate_states(nums):
    out = []
    for i, j in combinations(range(len(nums)), 2):
        a, b = nums[i], nums[j]
        rest = [nums[k] for k in range(len(nums)) if k not in (i, j)]
        candidates = [a + b, a - b, b - a, a * b]
        if b != 0:
            candidates.append(a / b)
        if a != 0:
            candidates.append(b / a)
        for c in candidates:
            out.append(rest + [c])
    return out

def score_state(nums, target=24):
    return -min(abs(x - target) for x in nums)

def tot_search(nums, target=24, depth=3, keep=4):
    states = [nums]
    for _ in range(depth):
        next_states = []
        for s in states:
            if any(abs(x - target) < 1e-9 for x in s):
                return s
            next_states.extend(generate_states(s))
        next_states.sort(key=lambda s: score_state(s, target), reverse=True)
        dedup = []
        seen = set()
        for s in next_states:
            key = tuple(sorted(round(x, 6) for x in s))
            if key not in seen:
                seen.add(key)
                dedup.append(s)
        states = dedup[:keep]
    return states[0]

ans = tot_search([4, 5, 6, 10], keep=6)
assert any(abs(x - 24) < 1e-9 for x in ans), ans
```

这个例子说明了三件事：

1. 候选不是一条，而是一批。
2. 每轮会评分并剪枝。
3. 去重很重要，否则同一种状态会反复出现。

真实工程例子可以看代码修复 Agent。一次运行中，模型可能先生成三个补丁方案：

- 方案 A：修复空指针保护。
- 方案 B：修改 SQL 字段映射。
- 方案 C：补上缓存失效逻辑。

然后用测试器和静态检查器评估。若只有 A 和 C 通过基本检查，就把它们保留，再继续让模型补上下游修改或补测试。整个过程不是“一次性写完补丁”，而是“先产出多个补丁方向，再由验证器帮你筛路”。

## 工程权衡与常见坑

ToT 一旦进入工程系统，真正的难点通常不在“会不会搜索”，而在“评分是否可靠、成本是否可控”。

第一大风险是评估器偏差。评估器偏差，就是打分规则和真实目标不一致。比如代码修复里，三个补丁都能通过静态检查，但只有一个能通过单元测试。如果你只看静态检查分，搜索会被错误引导到假阳性分支。模型自评尤其容易出这个问题，因为它擅长解释自己的方案，却不一定真能验证自己的方案。

第二大风险是分支爆炸。搜索树节点数近似随深度指数增长。若每层扩展 $m$ 个候选、保留 $K$ 个分支，那么总成本大致会随层数快速增加。即使简化估算，调用量也会接近：

$$
\text{Cost} \propto T \times K \times m \times \text{evaluation\_cost}
$$

所以 `m`、`K`、`T` 一定要受控，不能“为了更强全拉满”。

第三大风险是候选重复。模型很容易换个措辞生成同义分支，表面上有很多候选，实际上是同一路。没有去重，预算会被浪费在重复探索上。

第四大风险是不适合搜索的任务被硬套 ToT。比如简单摘要、固定模板抽取、明确规则转换，这些问题的关键不在多路径搜索，而在信息准确和格式稳定。用 ToT 只会变慢。

下表是最常见的风险表：

| 风险 | 后果 | 规避策略 |
|---|---|---|
| 评估器偏差 | 好分支被剪掉，坏分支被放大 | 优先接入单测、规则校验、执行结果 |
| 分支爆炸 | 延迟高、成本失控 | 限制 `m`、`K`、`T`，加早停 |
| 候选重复 | 浪费预算，假宽度 | 对状态去重，对文本做规范化 |
| 任务不适配 | 复杂度上升但效果不增 | 先判断任务是否真有搜索结构 |

真实工程里，最好尽量让“外部工具”参与评估，而不是只靠模型打分。比如代码修复场景，可以把评分拆成几层：

1. 能否通过语法和静态检查。
2. 能否通过单元测试。
3. 能否满足回归测试与业务规则。
4. 最后才用模型做可读性或方案合理性补充分。

这比“让模型自己说哪个补丁更对”可靠得多。新手可以记住一句话：别只看模型自评，要尽量让外部工具帮你判分。

## 替代方案与适用边界

ToT 不是唯一方案，更不是默认方案。是否使用它，关键要看问题的瓶颈到底是“找信息”还是“找路径”。

如果问题主要是“缺事实”，那应该优先检索增强。比如回答最新 API 价格、公司高管、论文发布日期，这类问题靠搜索或 RAG，比 ToT 更直接。

如果问题主要是“有信息，但路径复杂”，ToT 才更有优势。比如策略规划、数学推理、谜题求解、代码修复。

可以和常见替代方案做一个并排对比：

| 方法 | 核心思路 | 优点 | 局限 | 更适合什么任务 |
|---|---|---|---|---|
| CoT | 单链逐步推理 | 便宜、简单 | 一条路走错就容易整体失败 | 直线型、一步步可直达的问题 |
| ReAct | 推理 + 行动 | 能查资料、调工具 | 不一定显式比较多条内部路径 | 需要调用搜索、数据库、执行器的信息任务 |
| Beam Search | 保留多条高分序列 | 结构清晰、搜索稳定 | 更依赖明确的序列评分 | 明确评分结构的序列生成 |
| ToT | 显式分叉、评估、剪枝 | 能探索多条思路并回溯 | 成本高，依赖评分器质量 | 规划、谜题、代码修复、复杂推理 |

几类边界要分清。

第一，ToT 和 CoT 的边界。若问题本来就是线性的，例如“按固定步骤把十进制转二进制”，CoT 通常足够。ToT 在这里意义不大，因为分叉很少，扩树反而浪费。

第二，ToT 和 ReAct 的边界。ReAct 强在“边想边查边做”，比如问答机器人去搜网页、查数据库、调接口。ToT 强在“在内部保留多条推理分支并筛选”。如果任务核心是外部信息获取，先用 ReAct 或检索；如果核心是多路径规划，再考虑 ToT。

第三，ToT 和规则系统的边界。若任务已有成熟求解器，比如最短路径、SAT、约束求解、标准编译器优化，有时直接用专用算法更靠谱。ToT 更适合“规则不完整，但又需要多路径试探”的场景。

所以，判断题可以简化为一句新手版问题：先问自己，问题的关键是找信息，还是找路径？前者优先检索或工具调用，后者才考虑 ToT。
