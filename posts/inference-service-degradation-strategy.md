## 核心结论

推理服务降级策略，是指在系统容量、队列长度或尾延迟逼近上限时，主动把请求从“完整高质量路径”切到“更便宜但仍可接受的路径”，用可控的局部降质换整体可用性。

它要解决的不是“怎样让每个请求都达到最好效果”，而是“怎样在高峰和故障边缘保住关键请求的 SLO”。SLO 是服务等级目标，白话说就是“系统答应用户的时延和可用性底线”。如果没有降级，系统常见结局不是少数请求变慢，而是排队扩大、尾延迟飙升、超时增多、重试放大，最后所有请求一起变差。

一个最小直觉是：高峰期不要硬扛所有请求，而要先保“必须答”的请求。对低优先级流量，可以切小模型、缩短上下文、减少候选集、命中缓存，或者直接拒绝。

| 路径 | 典型动作 | 成本 | 质量 | 目标 |
| --- | --- | --- | --- | --- |
| 完整路径 | 大模型、长上下文、全量召回、完整重排 | 高 | 高 | 正常负载下给出最佳结果 |
| 降级路径 | 小模型、短上下文、部分召回、缓存、早退出 | 低 | 中 | 过载时保住时延和核心可用性 |

可以把系统状态看成一条切换曲线：

正常负载 -> 轻度紧张 -> 进入降级 -> 强制降级/拒绝

结论只有一句：降级不是失败补丁，而是容量管理的一部分。

## 问题定义与边界

问题定义要先说清。推理服务降级并不是普通异常处理。普通异常处理面对的是“单次请求执行失败”，比如依赖超时、参数错误、实例崩溃。降级面对的是“系统整体正在接近承载极限”，典型信号包括 CPU 或 GPU 利用率长期偏高、队列长度增长、P99 时延上升、超时比例增加。

这里的关键不是单点错误，而是资源竞争。排队论里常用负载比表示压力：

$$
\rho = \lambda / \mu
$$

其中 $\lambda$ 是到达率，白话说是“每秒来了多少请求”；$\mu$ 是有效处理能力，白话说是“系统每秒真正能处理多少请求”。当 $\rho$ 接近 1 时，系统已经很紧；当 $\rho > 1$ 时，请求进入速度超过处理速度，队列会持续增长。

玩具例子很适合说明边界。假设一个问答服务的完整路径平均耗时 220 ms，SLO 是 150 ms，稳定容量是 100 rps。高峰期实际流量到 130 rps，此时：

$$
\rho = 130 / 100 = 1.3
$$

如果坚持完整路径，系统不只是“慢一点”，而是会持续积压。降级后若轻路径平均只要 80 ms，那么低优先级请求切到轻路径，高优先级保完整路径，整个系统就可能重新回到 SLO 范围内。

但降级也有边界。不是所有业务都适合降级。支付扣款、库存扣减、强一致审计，这类场景结果必须完整正确，不能因为高峰就“少算一点”。这时更适合限流、排队、熔断或直接失败，而不是输出降质结果。

| 系统状态 | 典型特征 | 合适动作 |
| --- | --- | --- |
| 正常负载 | P99 稳定，队列短，资源有余量 | 走完整路径 |
| 负载接近上限 | P99 抬升，队列增长但可控 | 启动部分降级，优先保关键流量 |
| 明显过载 | 超时增多，排队持续扩张 | 强制降级、限流、拒绝低优先级 |
| 不可恢复故障 | 核心依赖不可用或结果不可信 | 直接失败、熔断、只返回安全兜底 |

边界的核心要求有两个。第一，降级后仍要保留最低可接受质量。第二，降级路径必须真的更便宜，否则只是换个名字继续拥塞。

## 核心机制与推导

降级决策的核心不是一个简单的 `if load > x`，而是“在时延、成本、质量之间选一个满足约束的动作”。通常可把动作抽象成三类：

- `f`：完整路径
- `d`：降级路径
- `r`：拒绝或快速失败

形式化写法是：

$$
a^\* = \arg\max_{a \in \{f,d,r\}} \mathbb{E}[Q_a]
\quad \text{s.t.} \quad P(T_a \le SLO) \ge p
$$

这里 $Q_a$ 是动作带来的质量或业务效用，白话说是“结果值不值”；$T_a$ 是该动作的时延；$p$ 是 SLO 达标概率要求。意思是：在能守住 SLO 的前提下，选择期望质量最高的动作。

实际系统里，最常见的进入和恢复条件是双阈值，也叫迟滞。迟滞就是“进入和退出不用同一个门槛”，目的是避免来回抖动。

$$
\text{当 } \rho \ge \rho_h \text{ 时进入降级；当 } \rho \le \rho_l \text{ 时恢复，且 } \rho_l < \rho_h
$$

如果没有 $\rho_l < \rho_h$，系统会在边界附近频繁切换：刚好一点就恢复，恢复后又立刻过载，再次降级。这样不仅用户体验不稳定，还会让缓存命中、线程池、批处理窗口全部失去稳定性。

对 LLM 推理，还有一种更细粒度的降级是早退出。早退出就是“对简单样本少算几层，复杂样本继续算”。一个常见形式是：

$$
\max_j p_j^{(k)} \ge \tau \Rightarrow \text{在第 } k \text{ 层退出}
$$

其中 $\tau$ 是置信度阈值。白话说，如果模型在第 $k$ 层已经足够有把握，就不用继续算后面的层。这类方法适合样本难度差异大的场景，但前提是提前退出不会明显破坏质量。

真实工程例子更能说明机制。假设一个在线检索增强生成服务有四段链路：查询改写、向量召回、交叉重排、答案生成。高峰时最贵的部分通常不是整个链路平均变慢，而是某个环节拖住队列，例如交叉重排和长上下文生成。一个可行的降级顺序通常是：

1. 先缩小召回候选集，例如从 200 条降到 50 条。
2. 再关闭昂贵的重排模型，改用轻量打分。
3. 再缩短上下文窗口，减少输入 token。
4. 仍不够时切到小模型。
5. 最后拒绝低优先级流量。

这比一上来就“全站 500”强得多，因为每一步都在明确地交换成本与质量。

| 动作 | 时延 | 资源成本 | 质量 | 适用时机 |
| --- | --- | --- | --- | --- |
| `f` 完整路径 | 高 | 高 | 高 | 正常状态或关键流量 |
| `d` 降级路径 | 中低 | 中低 | 中 | 接近过载但仍需保服务 |
| `r` 拒绝/错误返回 | 最低 | 最低 | 无 | 严重过载或不可信结果 |

## 代码实现

实现重点不是把策略写死在业务逻辑里，而是把“观测信号”“决策逻辑”“执行路径”拆开。这样才能配置、灰度、回滚和观测。

下面是一个可运行的 Python 玩具实现。它演示四件事：读取负载、按迟滞切换状态、按优先级路由、记录基础决策结果。

```python
from dataclasses import dataclass

HIGH_WATERMARK = 0.85
LOW_WATERMARK = 0.70
MIN_DWELL = 3  # 最小驻留次数，防止频繁抖动

@dataclass
class Request:
    priority: str   # "critical" or "normal"
    cached: bool = False

class DegradeController:
    def __init__(self):
        self.mode = "normal"
        self.dwell = 0

    def update_mode(self, rho: float) -> str:
        if self.mode == "normal":
            if rho >= HIGH_WATERMARK:
                self.mode = "degraded"
                self.dwell = 0
        else:
            self.dwell += 1
            if rho <= LOW_WATERMARK and self.dwell >= MIN_DWELL:
                self.mode = "normal"
        return self.mode

def run_full_path(req: Request) -> str:
    return "full"

def run_light_path(req: Request) -> str:
    return "light"

def reject_or_cache(req: Request) -> str:
    return "cache" if req.cached else "reject"

def handle_request(req: Request, rho: float, controller: DegradeController) -> str:
    mode = controller.update_mode(rho)

    if req.priority == "critical":
        return run_full_path(req) if mode == "normal" else run_light_path(req)

    if mode == "normal":
        return run_full_path(req)

    if rho >= 1.0:
        return reject_or_cache(req)

    return run_light_path(req)

controller = DegradeController()

assert handle_request(Request("normal"), 0.5, controller) == "full"
assert handle_request(Request("normal"), 0.9, controller) == "light"
assert handle_request(Request("critical"), 0.95, controller) == "light"
assert handle_request(Request("normal", cached=True), 1.1, controller) == "cache"
assert handle_request(Request("normal", cached=False), 1.2, controller) == "reject"
```

这段代码故意保持简单，但结构已经有工程含义：

| 模块 | 作用 | 设计要求 |
| --- | --- | --- |
| `metrics collector` | 收集 CPU、GPU、队列、P95/P99、错误率 | 不只看 QPS |
| `policy engine` | 根据阈值和优先级做决策 | 可配置、可回滚 |
| `request router` | 把请求送到完整/轻量/拒绝路径 | 不与业务强耦合 |
| `fallback handlers` | 缓存、小模型、模板回答、快速失败 | 必须真正更轻 |
| `logging/telemetry` | 记录触发原因、命中率、质量变化 | 能复盘策略效果 |

真实工程里，策略模块通常还要支持按租户、按接口、按请求类型做分级。例如内部监控查询可直接拒绝，但付费用户的核心推理请求优先保留完整路径。这不是“业务特殊逻辑污染系统”，而是降级本来就必须和业务价值绑定。

## 工程权衡与常见坑

降级最难的地方，不是“有没有开关”，而是“什么时候降、对谁降、降多少”。错误的降级通常比没有降级更糟，因为它会制造一种“策略存在但无效”的假象。

下面这些坑非常常见：

| 常见坑 | 为什么危险 | 规避方式 |
| --- | --- | --- |
| 所有请求一视同仁 | 关键流量会被普通流量拖死 | 先做优先级分层 |
| 只看 QPS 不看资源消耗 | 同样 100 rps，请求成本可能差 10 倍 | 看 GPU、CPU、队列、token 数、候选集大小 |
| 降级路径仍卡同一瓶颈 | 切换后时延几乎不降 | 确认真正绕开瓶颈 |
| 重试风暴 | 失败后重试把过载放大 | 设重试预算和退避 |
| 正常/降级抖动 | 用户体验和指标都不稳定 | 用迟滞和最小驻留时间 |
| 客户端继续狂打 | 服务端拒绝无效，整体压力不降 | backpressure 或客户端节流 |
| 降级质量不可控 | 结果虽然返回，但业务不可用 | 先定义最低可接受质量 |

迟滞区间必须单独强调：

$$
\rho_l < \rho_h
$$

这不是数学装饰，而是稳定性条件。进入降级和恢复正常不能用同一阈值。再配合“最小驻留时间”，也就是进入某个状态后至少待一段时间，系统才不会在边缘流量下反复翻转。

另一个容易忽略的点是“降级路径必须轻到足以改变系统状态”。比如完整路径和降级路径都要访问同一个拥塞的向量检索集群，那你把生成模型换小了也未必有用。真正有效的降级，往往要减少共享瓶颈上的占用，比如减小召回规模、跳过重排、直接命中缓存、降低输出长度。

对 LLM 服务还有一个典型坑：只降模型，不降 token。实际上很多场景里成本主要跟 token 数和并发占用有关。只换模型但保留超长上下文，效果可能远低于“先截断上下文，再决定是否切小模型”。

## 替代方案与适用边界

降级不是唯一方案，更不是默认优先项。它适用于“部分结果仍然有价值”的系统，例如搜索、推荐、广告排序、RAG 问答、内容审核辅助、聊天摘要。这些场景里，结果可以有一定质量折损，但必须快速返回。

不适合降级的场景也要明确：强一致交易、账务核对、权限判定、法律或医疗中不能输出不可信结果的关键决策。这些系统更看重正确性底线，而不是“先返回一个差一点的答案”。

| 方案 | 适用场景 | 优点 | 风险 | 与降级的关系 |
| --- | --- | --- | --- | --- |
| 降级 | 部分结果仍有价值 | 可保住核心 SLO | 质量下降需可控 | 主角 |
| 限流 | 流量明显超出容量 | 简单直接 | 会丢请求 | 常与降级配合 |
| 缓存 | 热点高、结果可复用 | 成本低、见效快 | 结果可能过期 | 是常见降级动作 |
| 扩容 | 负载上升可预期 | 不损质量 | 成本高、响应慢 | 解决容量，不替代降级 |
| 排队 | 短时突发 | 保证不丢任务 | 时延增加 | 适合离线或弱实时 |
| 异步化 | 结果可稍后返回 | 解耦峰值压力 | 用户交互变复杂 | 可减少在线路径压力 |

从工程角度看，最稳妥的方案通常不是只选一个，而是组合拳：正常时依靠扩容和缓存，接近上限时轻度降级，明显过载时叠加限流和拒绝，后台再做异步补算或日志复盘。

判断是否适合做降级，可以用三个问题快速筛选：

1. 结果部分可接受吗？
2. 降级后能显著降低计算或排队成本吗？
3. 用户对延迟比对精度更敏感，或者精度损失可控吗？

三个问题里只要有两个答“否”，就要谨慎。很多团队失败，不是因为不会写策略，而是把不该降级的业务硬套进了降级框架。
