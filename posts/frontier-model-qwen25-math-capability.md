## 核心结论

Qwen2.5-Math 的提升重点不是“把参数做大”这么简单，而是把数学任务拆成一条完整的能力链路：更高质量的数学语料、更强调中间步骤的监督、更强调最终答案可验证的奖励训练，以及推理阶段的多候选采样与重排序。

官方在 2024 年 9 月 19 日发布的技术说明里给出的结论很明确：Qwen2.5-Math 提供 `1.5B / 7B / 72B` 的 base 与 Instruct 版本，支持中文和英文数学任务，重点支持两种工作模式。CoT（Chain-of-Thought，链式思维）就是把答案拆成连续小步骤；TIR（Tool-Integrated Reasoning，工具集成推理）就是模型一边写推理，一边生成可执行代码来辅助计算。对数学题来说，后者往往比单纯文字推理更稳，因为算术、符号变换、方程求解可以交给外部执行器核验。

一个最能说明问题的数字对比是：官方博客给出的 MATH 基准上，Qwen2.5-Math-7B-Instruct 在 CoT 模式下达到 `83.6`，切到 TIR 后达到 `85.3`；72B-Instruct 在 TIR 的 `RM@8` 设置下达到 `92.9`。RM（Reward Model，奖励模型）可以理解成“专门给候选解答打分的裁判模型”。这说明专门的后训练和推理时选优，确实能让同尺寸模型显著强于通用模型，甚至让 7B 级别模型越过上一代 72B 数学模型的一部分表现。

但边界也必须说清楚。Qwen2.5-Math 不是“自动会做所有数学”的系统。它的官方数学模型配置仍以 `4K context` 为主，超长证明、图形几何、复杂公式 OCR、严谨符号计算链条，通常仍要结合 Python、SymPy、几何引擎或视觉前处理。换句话说，它更像一个“擅长数学推理的语言模型内核”，不是完整的数学软件栈。

---

## 问题定义与边界

这类模型要解决的，不是普通聊天里的“顺口回答”，而是数学问答里的“结果正确且过程自洽”。这里的“自洽”很重要，意思是中间推导不能和最终答案打架。因为在数学场景里，用户通常不只看一句结论，还会检查每一步是否成立。

Qwen2.5-Math 面向的典型任务包括：小学到高中应用题、大学代数与微积分、竞赛风格证明或构造题、符号运算、代码辅助求解。这些任务的共同点是：错误往往不是“表达不顺”，而是某一步算错、某个边界条件漏掉、某个符号被误写。

可以把它的工作流抽象成下面这个式子：

$$
\text{Math QA} = \text{Prompt} + \text{Reasoning Mode} + \text{Verifier}
$$

其中 `Reasoning Mode` 可以是 CoT，也可以是 TIR；`Verifier` 则可能是奖励模型、答案规则匹配、Python 执行器，或者三者叠加。

先看一个玩具例子。题目是：

$$
4x + 5 = 2x + 11
$$

如果只用普通对话模型，它可能直接给出“$x=3$”，但不一定解释为什么。Qwen2.5-Math 更强调下面这种链路：

1. 先移项得到 $4x-2x=11-5$
2. 再化简得到 $2x=6$
3. 再除以 2 得到 $x=3$
4. 若启用 TIR，再生成一段代码验证 `x=3` 代回原式成立

这里的边界也顺带出现了。对这类短方程，CoT 已经够用；对含大量符号变换、行列式、积分、求根的题，TIR 往往更可靠，因为模型可以把“算”交给程序。

官方信息还给出两个现实约束。第一，Qwen2.5-Math 官方数学版本的上下文长度以 `4K` 为主，这意味着特别长的题干、长证明历史、跨多轮的复杂推理会更容易出现前文遗忘。第二，官方明确说这套模型主要支持中英文数学任务，不建议拿它当通用聊天模型长期替代普通 Qwen2.5。

## 核心机制与推导

Qwen2.5-Math 的训练思路可以概括成四层。

第一层是数学预训练语料。官方说明里提到，他们构建了 `Qwen Math Corpus v2`，总量从上一版的 `700B tokens` 增长到 `1T+ tokens`，来源包括网页、书籍、代码和合成数学数据。这里的“合成数据”不是随便机器生成文本，而是用已有强模型生成高质量数学样本，再经过筛选加入训练。直白地说，就是先造出更多“像好解答”的训练材料。

第二层是 SFT（Supervised Fine-Tuning，监督微调）。它的意思是：拿高质量样例直接教模型“遇到数学题应该怎么写”。在 Qwen2.5-Math 里，这一步不是只教最终答案，而是同时教 CoT 与 TIR 两种模式。这样模型学到的不只是“答什么”，还有“如何分步答”和“何时调用工具”。

第三层是 RM 与拒绝采样。RM 是奖励模型，可以把同一道题的多个候选答案排序。拒绝采样可以理解成“先多生成几份，再只保留高分样本继续训练”。这一步的作用是强化“过程靠谱且结果正确”的回答，而不是只强化表面像样的回答。

第四层是 RL（强化学习）阶段的 GRPO。GRPO（Group Relative Policy Optimization）可以理解成“把同一题的一组候选答案放在一起比较，再按相对好坏更新模型”。直觉上，它不像传统监督学习那样只告诉模型唯一标准答案，而是让模型学会“哪些推理链更优”。

用一个简化表达式表示，若同一题采样出 $N$ 个候选解 $\{y_i\}_{i=1}^N$，奖励模型给出分数 $r_i$，则组内相对优势可以写成：

$$
A_i = \frac{r_i - \mu_r}{\sigma_r + \epsilon}
$$

其中 $\mu_r$ 是这组样本的平均奖励，$\sigma_r$ 是标准差。直观上，$A_i > 0$ 说明这个候选比组内平均更好，$A_i < 0$ 说明更差。训练时会提升高优势样本的概率，同时加入 KL 惩罚，避免模型一步跳得太远。可以把目标粗略写成：

$$
\mathcal{L} \approx - \mathbb{E}\left[A_i \log \pi_\theta(y_i|x)\right] + \beta \, D_{KL}(\pi_\theta \,\|\, \pi_{\text{ref}})
$$

这不是 Qwen 文档里的逐字公式，而是对 GRPO 更新思路的简化表达。

再看一个更贴近机制的玩具例子。题目是：

$$
\int_0^1 x^2\,dx
$$

CoT 模式会倾向于这样推：

$$
\int x^2 dx = \frac{x^3}{3}, \quad
\left[\frac{x^3}{3}\right]_0^1 = \frac{1}{3}
$$

TIR 模式则可能额外写出一段可执行代码，用 SymPy 验证定积分结果。然后如果同一道题采样出 8 条候选答案，系统可以做 `RM@8`：先生成 8 条，再由奖励模型选出最靠谱的一条。这就是为什么官方报告里 `RM@8` 往往高于单次贪心输出。

真实工程例子可以这样理解：在线教育平台接到“求一元二次方程根”的请求时，不让模型只返回自然语言，而是让它同时产出解释与代码。解释给学生看，代码给系统验证。前者提升可读性，后者提升可置信度。这种“双轨输出”正是 Qwen2.5-Math 强调的设计点。

## 代码实现

如果你要把 Qwen2.5-Math 接进自己的服务，最实用的做法不是“只收一段文本答案”，而是做一个闭环：

| 环节 | 做什么 | 目的 |
|---|---|---|
| 1. 提示模型进入 TIR 模式 | 要求输出推理和 `python` 代码块 | 让模型显式区分“解释”和“计算” |
| 2. 捕捉代码块 | 从回复里提取 ```python ... ``` | 找到可执行部分 |
| 3. 安全执行 | 在受限环境运行代码 | 得到可验证结果 |
| 4. 回填执行结果 | 把 `stdout` 或结构化结果交回模型 | 让模型基于真实结果总结 |
| 5. 最终作答 | 输出自然语言答案与必要步骤 | 兼顾可读性和正确性 |

下面是一个最小可运行例子，用 SymPy 验证一次方程求解。它不依赖大模型本身，但演示了 TIR 想达到的效果：把符号计算交给程序。

```python
from sympy import symbols, Eq, solve

x = symbols('x')
eq = Eq(4 * x + 5, 2 * x + 11)
solution = solve(eq, x)

assert solution == [3]
assert 4 * solution[0] + 5 == 2 * solution[0] + 11

print(solution)
```

如果换成更接近真实服务端的写法，核心逻辑通常是“提取代码并执行，再回填结果”。伪代码如下：

```python
import re

def extract_python_block(text: str) -> str | None:
    m = re.search(r"```python\n(.*?)```", text, re.S)
    return m.group(1).strip() if m else None

def should_use_tir(problem: str) -> bool:
    keywords = ["方程", "积分", "矩阵", "求根", "sympy", "python"]
    return any(k in problem for k in keywords)

sample = """
先设方程并求解：
```python
from sympy import symbols, Eq, solve
x = symbols('x')
print(solve(Eq(4*x + 5, 2*x + 11), x))
```
"""

code = extract_python_block(sample)
assert code is not None
assert "solve" in code
```

真实工程例子是：题库批改系统收到“解方程、求导、求积分”请求后，先调用 Qwen2.5-Math-7B-Instruct 生成解释与代码，再在隔离的 Python 沙箱中执行。执行成功，就把结果回填给模型生成最终答案；执行失败，就要求模型重试或退回纯 CoT。这样做的好处是，用户看到的是完整讲解，平台拿到的是可核验答案。

## 工程权衡与常见坑

第一类坑是“以为开了 TIR，实际上没接执行器”。这是最常见误区。没有执行器时，模型只是输出“应该执行什么代码”，并没有真正运行。对于简单题，这问题不大；对复杂计算题，准确率会明显下降。TIR 的价值来自“代码被执行并把结果反馈回来”，不是来自“代码看起来很专业”。

第二类坑是把 benchmark 数字理解错。官方公开信息里比较稳定的数字是：1.5B/7B/72B-Instruct 在 MATH 上的 TIR 成绩分别为 `79.7 / 85.3 / 87.8`，72B 在 `RM@8` 的 TIR 设置下达到 `92.9`。因此如果你看到“7B 在 MATH 达到 97.6”之类说法，要先核对来源。至少按官方 2024 年 9 月 19 日博客与 GitHub README，公开主数字不是这个值。

第三类坑是资源预估过低。官方仓库给出的评测脚本示例里，1.5B 与 7B 用单卡 `CUDA_VISIBLE_DEVICES="0"`，72B 示例用 `CUDA_VISIBLE_DEVICES="0,1,2,3"`。这不等于“线上部署必须四卡”，但能说明 72B 至少不是轻量级选项。

| 场景 | 无执行器 | 有执行器 |
|---|---|---|
| 简单代数题 | 通常能答对，但偶尔中间算错 | 可直接验证，稳定性更高 |
| 积分/求根/矩阵 | 容易写出“看似正确”的错误步骤 | 可交给 SymPy/Numpy 计算 |
| 输出解释质量 | 仍然较好 | 解释质量相近，但结果更可信 |
| 系统复杂度 | 低 | 更高，需要沙箱与回填逻辑 |

还有两个经常被忽略的边界。一个是上下文限制，4K 对短题足够，但对超长证明、多轮反复纠错并不宽裕。另一个是任务错配，Qwen 官方明确不建议把 Qwen2.5-Math 当通用模型使用。数学专精带来数学收益，也意味着在闲聊、开放生成、泛领域问答上未必是最优选。

## 替代方案与适用边界

如果你的目标是“尽量低成本地把数学答题做稳”，优先级通常不是直接冲 72B，而是先判断任务类型。

| 方案 | 适合什么场景 | 优点 | 边界 |
|---|---|---|---|
| Qwen2.5-Math-7B-Instruct + CoT | 讲解型题库、低成本部署 | 单卡友好，推理清晰 | 复杂计算稳定性一般 |
| Qwen2.5-Math-7B-Instruct + TIR | 教育平台、作业批改、数学助手 | 成本和效果平衡最好 | 需要执行器与沙箱 |
| Qwen2.5-Math-72B-Instruct + RM@N | 高难竞赛题、离线高精度评测 | 上限高 | 资源开销大 |
| 普通 Qwen2.5-Instruct | 通用助手、知识问答 | 场景更泛化 | 数学专项通常不如 Math 系列 |
| 通用模型 + 外部 CAS | 已有成熟工具链的企业系统 | 易与现有系统整合 | 解释过程未必自然 |

一个简单决策规则是：

$$
\text{优先选型} =
\begin{cases}
\text{7B + CoT}, & \text{题目短且重讲解} \\
\text{7B + TIR}, & \text{题目需要计算验证} \\
\text{72B + RM@N}, & \text{高难题且资源充足}
\end{cases}
$$

和上一代 Qwen2-Math 或普通 Qwen2.5 相比，Qwen2.5-Math 的主要差异不在“会不会说数学术语”，而在“是否系统性强化了数学推理链”。官方说明里明确提到，7B-Instruct 借助 72B 合成数据和后训练后，已经能超过上一代 72B 数学模型的一部分表现。这说明在数学这个窄领域里，专精数据、奖励模型和推理时选优，往往比盲目堆参数更有效。

但也不要把它看成万能解。超长形式化证明、图形几何、需要 OCR 的试卷题、严格符号计算链，都更适合“Qwen2.5-Math + 外部工具”而不是“Qwen2.5-Math 单独硬扛”。模型负责理解题意和组织步骤，外部工具负责精确计算与验证，这才是更稳的工程边界。
