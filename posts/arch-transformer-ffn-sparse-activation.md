## 核心结论

FFN（Feed-Forward Network，前馈网络，指 Transformer 里每个 token 独立通过的两层感知机）之所以会出现稀疏激活，核心原因通常不是“模型主动丢信息”，而是激活函数把大量预激活值直接压成了 0。对 ReLU 类激活来说，单个 token 经过某一层 FFN 时，常见现象是只有少数神经元真正输出非零值，其余神经元处于“休眠”状态。

最小玩具例子直接能看清这件事。假设某层预激活向量是 $z=[-1.2, 0.3, 1.7]$，经过 ReLU 后得到：

$$
a=\mathrm{ReLU}(z)=[0, 0.3, 1.7]
$$

这里有 $2/3$ 的位置不再参与后续计算，可记为约 66% 的稀疏率。白话说，这一层只“点亮”了少数几个灯，其他灯虽然存在，但对这个 token 没有贡献。

这类稀疏不是边角现象，而是 FFN 的重要结构特征。只要活跃神经元保留了主要表达能力，模型就有机会在基本不损失效果的前提下减少有效计算量、降低 FLOPs，并为后续剪枝或稀疏算子优化提供空间。

| 激活函数 | 典型稀疏率 | 活跃神经元比例 | 直观含义 |
|---|---:|---:|---|
| ReLU | 50%~90% | 10%~50% | 负值截断为 0，天然稀疏 |
| ReLU$^2$ | 60%~95% | 5%~40% | 保留正值并放大较大激活，稀疏更强 |
| dReLU | 70%~95% | 5%~30% | 双侧门控或阈值抑制，更少神经元被点亮 |

结论可以压缩成一句话：FFN 不是让所有神经元一起工作，而是让一小部分神经元按输入内容被动态唤醒，这就是稀疏激活模式。

## 问题定义与边界

本文讨论的“FFN 稀疏激活”，特指 FFN 第一层线性投影后的激活输出中，大量元素恰好为 0 的现象。这里不讨论注意力矩阵的稀疏，也不讨论权重本身是否被剪枝。

设某一层 FFN 的预激活向量为 $z \in \mathbb{R}^N$，激活输出为 $a$。如果使用 ReLU，则：

$$
a_i = \max(0, z_i)
$$

稀疏率定义为：

$$
s=\frac{1}{N}\sum_{i=1}^N \mathbb{1}_{a_i=0}=\Pr(z_i<0)
$$

符号表如下：

| 符号 | 含义 |
|---|---|
| $z_i$ | 第 $i$ 个神经元的预激活值，也就是过激活函数前的线性输出 |
| $a_i$ | 第 $i$ 个神经元的激活输出，也就是真正传给下一层的值 |
| $s$ | 稀疏率，表示输出中为 0 的比例 |

为什么它会自然出现？因为 Transformer 通常会先对输入做规范化。规范化的白话解释是：把特征尺度拉到比较稳定的范围，避免某些维度特别大或特别小。规范化后再做线性投影，预激活值 $z_i$ 往往更接近一个以 0 为中心的分布。此时一旦经过 ReLU，所有负值都会直接变成 0，于是只有正值神经元继续“亮灯”。

可以把它理解为“按需开灯”，但这里的比喻只服务于机制理解：房间里有很多灯，电路先根据输入内容打分，只有得分为正的灯会亮。重要的是，这不是随机亮灯，而是由输入内容决定的条件计算。

本文边界再明确三点：

| 项目 | 本文是否讨论 | 说明 |
|---|---|---|
| FFN 激活稀疏 | 是 | 核心主题 |
| 注意力稀疏 | 否 | 属于另一类机制 |
| 权重剪枝 | 部分涉及 | 只在工程权衡中对比，不作为主线 |
| 非 ReLU 模型整体替换策略 | 部分涉及 | 只讨论适用边界，不展开完整迁移路线 |

## 核心机制与推导

先看最基本的 ReLU。若预激活 $z_i$ 的分布近似对称，并且中心在 0 附近，那么：

$$
s = \Pr(z_i < 0)
$$

当 $z_i \sim \mathcal{N}(0,1)$ 时：

$$
s = \Pr(z_i<0)=0.5
$$

这意味着最基础情况下，哪怕没有额外门控，理论上也会有一半神经元输出 0。

更一般地，如果 $z_i \sim \mathcal{N}(\mu,\sigma^2)$，则：

$$
s = \Pr(z_i<0)=\Phi\left(-\frac{\mu}{\sigma}\right)
$$

其中 $\Phi$ 是标准正态分布的累积分布函数。这个式子说明三件事：

1. 如果均值 $\mu$ 更偏负，稀疏率会上升。
2. 如果均值接近 0，稀疏率接近 50%。
3. 如果网络训练后让很多神经元只在少数输入上激活，那么实测稀疏率可以远高于 50%。

一个简化的概率密度示意可以这样看：

```text
            ^
            |           .-^-.
            |        .-'     '-.
------------+------|------0------|-------->
                   负区间       正区间

ReLU 只保留右侧，左侧全部压成 0
```

为什么工程里常能看到 80%、90% 甚至更高？因为真实模型里的 FFN 不只是“对称分布 + ReLU”这么简单，还叠加了参数学习后的选择性。训练会让某些神经元只对少数模式敏感，于是对大多数 token 来说它们都输出负值或接近阈值，结果就是大规模休眠。

再看门控增强。以 ReLU$^2$ 为例，可写成：

$$
a_i = (\max(0,z_i))^2
$$

它不会改变“负值归零”这件事，但会进一步压低小正值的相对影响，让真正有贡献的激活更加集中。dReLU 可以理解为带更强抑制条件的 ReLU 变体，它会扩大“被截断”区域，或者让部分本来略大于 0 的值也难以通过。于是有效活跃神经元更少，稀疏率继续上升。

一个简单推导是：若引入阈值 $\tau>0$，定义

$$
a_i = \max(0, z_i-\tau)
$$

那么稀疏率变为：

$$
s = \Pr(z_i \le \tau)
$$

若 $z_i \sim \mathcal{N}(0,1)$，则：

$$
s=\Phi(\tau)
$$

例如 $\tau=1$ 时，$s\approx 0.8413$。也就是约 84% 的神经元被压成 0。这里虽然不一定对应某个具体论文中的标准 dReLU 定义，但足够说明“阈值右移会提高稀疏率”这一机制。

真实工程例子里，这种机制的价值更明显。比如对已经训练好的大模型做低成本改造时，如果 FFN 激活替换为更稀疏的 ReLU 类方案，再通过一定规模的继续训练恢复分布，模型往往能把活跃神经元数量压得很低，同时保持主要任务精度。直白地说，模型不是“少算了很多所以坏掉”，而是“学会只算必要部分”。

## 代码实现

工程上最常见的两件事是：

1. 替换 FFN 里的激活函数。
2. 统计每层激活稀疏率，观察是否真的产生了想要的休眠模式。

下面先给一个可运行的最小 Python 例子，不依赖深度学习框架，只演示机制与统计逻辑。

```python
import math

def relu(xs):
    return [max(0.0, x) for x in xs]

def zero_ratio(xs):
    return sum(1 for x in xs if x == 0.0) / len(xs)

# 玩具例子
z = [-1.2, -0.4, 0.3]
a = relu(z)

assert a == [0.0, 0.0, 0.3]
assert abs(zero_ratio(a) - 2/3) < 1e-9

# 一个简单的阈值版激活，模拟更强稀疏
def threshold_relu(xs, tau=0.5):
    return [max(0.0, x - tau) for x in xs]

z2 = [-0.2, 0.3, 0.8, 1.5]
a2 = threshold_relu(z2, tau=0.5)

assert a2 == [0.0, 0.0, 0.30000000000000004, 1.0]
assert zero_ratio(a2) == 0.5

print("relu:", a, "zero_ratio:", zero_ratio(a))
print("threshold_relu:", a2, "zero_ratio:", zero_ratio(a2))
```

如果放到 PyTorch 风格的 FFN 中，逻辑通常长这样：

```python
import torch
import torch.nn as nn

class SparseStats:
    def __init__(self):
        self.total_zero = 0
        self.total_count = 0

    def update(self, act: torch.Tensor):
        zeros = (act == 0).sum().item()
        count = act.numel()
        self.total_zero += zeros
        self.total_count += count

    @property
    def zero_ratio(self):
        if self.total_count == 0:
            return 0.0
        return self.total_zero / self.total_count


class ThresholdReLU(nn.Module):
    def __init__(self, tau=0.0):
        super().__init__()
        self.tau = tau

    def forward(self, x):
        return torch.relu(x - self.tau)


class FFN(nn.Module):
    def __init__(self, d_model, d_ff, act=None):
        super().__init__()
        self.up_proj = nn.Linear(d_model, d_ff)
        self.act = act if act is not None else nn.ReLU()
        self.down_proj = nn.Linear(d_ff, d_model)
        self.stats = SparseStats()

    def forward(self, x):
        z = self.up_proj(x)
        act = self.act(z)
        self.stats.update(act.detach())
        zero_ratio = (act == 0).float().mean().item()
        # 训练时可写日志：logger.info(f"ffn_zero_ratio={zero_ratio:.4f}")
        return self.down_proj(act)

# 可序列化：tau 作为模块参数保存在 state_dict 之外的配置中
model = FFN(d_model=8, d_ff=16, act=ThresholdReLU(tau=0.2))
dummy = torch.randn(2, 4, 8)
out = model(dummy)

assert out.shape == (2, 4, 8)
assert 0.0 <= model.stats.zero_ratio <= 1.0
```

这里有两个实现点很关键。

第一，激活模块最好显式做成 `nn.Module`。这样模型保存和加载时结构明确，微调后的配置不容易丢。

第二，稀疏率不要只看单个 batch。单个 batch 的 `zero_ratio` 波动很大，更稳定的做法是跨多个 batch、多个层累计统计，再按层输出均值、方差、分位数。

监控字段可以设计成这样：

| 字段名 | 含义 | 替换前示例 | 替换后示例 |
|---|---|---:|---:|
| `ffn.zero_ratio` | 当前 batch 激活为 0 的比例 | 0.52 | 0.87 |
| `ffn.nonzero_mean` | 非零激活的均值 | 0.41 | 0.76 |
| `ffn.nonzero_std` | 非零激活的标准差 | 0.29 | 0.58 |
| `ffn.layerwise_p50` | 各层稀疏率中位数 | 0.55 | 0.89 |
| `ffn.layerwise_p90` | 各层稀疏率 90 分位 | 0.71 | 0.95 |

真实工程里，常见流程不是“替换完立即上线”，而是“替换激活函数 -> 小规模验证 -> 持续微调恢复分布 -> 统计层级稀疏率 -> 再决定是否做激活剪枝或稀疏加速”。

## 工程权衡与常见坑

稀疏激活看起来很美，但它不是“零越多越好”。真正的目标不是把更多神经元关掉，而是在不破坏表达能力的前提下，让有效计算集中在重要神经元上。

第一个常见坑是直接按激活频率剪神经元。所谓激活频率，就是某个神经元在数据集上被点亮的比例。频率低不等于没用，因为有些神经元专门负责少数关键模式，比如代码 token、长尾术语、特定语言结构。直接删掉，可能在平均指标上看不明显，但在细分任务上出现断崖式退化。

举个工程化例子：如果你观察到某层 90% 激活近似为 0，就直接把这些低活跃神经元裁掉而不重训练，模型往往会出现精度回落。原因不是“模型太脆弱”，而是 FFN 的表示往往带有低秩结构。白话说，很多神经元虽然不常亮，但它们之间共同组成了某种压缩表达。你删的是神经元，破坏的却可能是整体表示空间。

第二个坑是把稀疏率当成最终收益。软件日志里看到 `zero_ratio=0.92`，不等于硬件上就自动省掉 92% 的时间。若底层矩阵乘法库不会跳过零值，理论稀疏只会变成“统计上很漂亮”，未必变成真实延迟收益。

第三个坑是层间分布不均。某些层天然更稀疏，某些层承担更重的变换任务。如果统一阈值、统一剪枝比例，常常会过度压缩关键层，导致整体性能下降。

下面这个表更实用：

| 问题 | 症状 | 规避措施 |
|---|---|---|
| 激活剪枝过猛 | 验证集精度明显回落 | 先小比例剪枝，再微调恢复 |
| 只看平均稀疏率 | 少数任务突然失真 | 分任务、分层、分 token 类型统计 |
| 稀疏率高但没提速 | 端到端延迟变化很小 | 检查算子是否支持稀疏执行 |
| 层间统一阈值 | 个别层成为瓶颈 | 做分层阈值或分层预算 |
| 非零值幅度失真 | 训练不稳定或输出漂移 | 联合监控 `nonzero_mean/std` |
| 直接删低频神经元 | 长尾样本退化明显 | 保留低秩结构，配合再训练 |

更稳妥的做法通常是：先做温和剪枝，再微调恢复；或者先用幅值剪枝、低秩重建这类更可控的方法，再视情况引入激活稀疏。激活剪枝不是不能做，而是要承认它改的是“动态行为”，修复难度往往比静态权重剪枝更高。

## 替代方案与适用边界

如果目标只是“降 FLOPs、降显存、提吞吐”，激活稀疏不是唯一方案。

第一类替代方案是幅值剪枝。它按权重绝对值大小删参数。白话解释是：直接删掉数值很小、贡献可能较弱的连接。这种方法实现成熟，和现有训练管线兼容性好，恢复手段也多，但它削减的是静态参数，不一定像激活稀疏那样天然具备输入条件选择性。

第二类是 LUT 或低秩重建。LUT 可理解为查表近似，低秩重建则是把大矩阵拆成更小的矩阵乘积，用更少参数逼近原映射。这类方法对硬件更友好，也更稳定，但往往需要专门的实现路径，且对误差累积较敏感。

第三类是动态专用激活，比如 SwiGLU 一类门控激活。它不一定输出大量精确的 0，但可以通过门控机制提升选择性，常在性能与稳定性之间取得更好的平衡。代价是“零值稀疏”未必明显，硬件收益更依赖实现细节。

可以做一个定性对比：

| 方案 | 复杂度 | 恢复速度 | 设备依赖 | 适用场景 |
|---|---|---|---|---|
| ReLU 稀疏/激活替换 | 中 | 中到慢 | 中到高 | 已有 ReLU 类结构，想做条件计算 |
| 幅值剪枝 | 低到中 | 快 | 低到中 | 想快速压缩并保留通用兼容性 |
| LUT/低秩重建 | 中到高 | 中 | 中 | 追求矩阵层面的稳定压缩 |
| 动态门控激活 | 中 | 中 | 中 | 愿意改结构并继续训练 |

适用边界也要说清楚。

如果原模型本来就是 ReLU 家族激活，那么继续强化稀疏通常最自然，因为你是在原有行为上加强选择性。

如果原模型使用的是 GELU、SwiGLU 等非零稀疏激活，直接替换成 ReLU 类方案并不总是划算。原因在于模型原本的表示分布、训练超参、残差尺度都围绕旧激活设计。替换后通常需要足够多的微调 token 才能把分布重新校准，否则你省下来的 FLOPs，可能又被恢复训练成本吃掉。

定性比较一下相同 FLOP 目标下的两类路线：

1. 对 LLaMA、Falcon 这类模型做 ReLU 稀疏替换，优点是理论稀疏显著，后续有机会接稀疏算子；缺点是需要额外微调恢复。
2. 用幅值剪枝达到相同 FLOP 目标，优点是工程链路更成熟、恢复更快；缺点是未必能获得输入条件选择性的收益。

所以判断标准不应只看“谁更稀疏”，而应看你的部署环境是否真能把这种稀疏转成收益，以及你是否有预算完成替换后的再训练。
