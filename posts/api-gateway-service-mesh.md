## 核心结论

在大模型部署里，`API 网关`可以理解为“系统对外的统一入口”，负责处理所有从外部进入平台的请求；`服务网格`可以理解为“系统内部通信层”，负责处理服务之间的连接、重试、安全和观测。两者不是替代关系，而是分工关系。

如果把推理平台拆成“鉴权服务、路由服务、Embedding 服务、重排序服务、模型推理服务、日志服务”等多个微服务，那么最稳定的做法通常是：

| 组件 | 面向路径 | 关键功能 |
| --- | --- | --- |
| API 网关 | 北向流量，外部到系统 | 路由、鉴权、限流、协议转换、版本入口、审计 |
| 服务网格 | 东西向流量，服务到服务 | 服务发现、mTLS、重试、熔断、负载均衡、追踪 |
| 业务服务 | 具体功能路径 | 预处理、检索、重排、推理、后处理 |

核心收益不是“请求一定更快”，而是“治理复杂度下降”。没有网关时，客户端和服务的关系数量近似为：

$$
\text{路径}_{直连}=N\times M
$$

引入网关后变成：

$$
\text{路径}_{网关}=N+M
$$

这里的“路径”不是 URL 数量，而是需要分别维护认证、路由、限流和兼容策略的连接关系数量。系统越大，这个差异越明显。

玩具例子很直观：3 个客户端调用 2 个模型服务，直连时需要维护 $3\times2=6$ 条治理路径；接入网关后只需要 $3+2=5$ 条。这个例子数值差异不大，但真正减少的是“重复实现鉴权和策略”的次数。

## 问题定义与边界

`微服务`是把一个大系统拆成多个可独立部署的小服务。对白话解释来说，就是“不把所有逻辑塞进一个进程，而是让不同功能各自运行、各自扩缩容”。

在大模型场景里，常见拆分方式包括：

| 服务 | 职责 | 是否适合独立扩缩容 |
| --- | --- | --- |
| API 网关 | 接入治理 | 是 |
| 鉴权服务 | 用户身份、Token 校验 | 是 |
| Prompt 预处理服务 | 模板拼接、参数校验 | 是 |
| 检索服务 | 向量检索、召回 | 是 |
| 重排序服务 | 排序候选结果 | 是 |
| 推理服务 | 调用 GPU 模型 | 强烈建议 |
| 结果后处理服务 | 结构化输出、敏感词处理 | 是 |

问题的边界要先讲清楚。

第一，API 网关主要解决北向问题，也就是外部客户端如何稳定、安全地进入系统。浏览器、移动端、BFF、第三方调用方都属于这一侧。

第二，服务网格主要解决东西向问题，也就是系统内部服务如何相互调用。比如“检索服务调用重排序服务”，“编排服务调用推理服务”，“推理服务调用日志服务”。

第三，网关和网格都不能替代业务建模。它们负责的是通信治理，不负责决定“模型该不该切流”“业务该不该降级”。这些策略最终仍然要由平台设计决定。

复杂度可以用一个简化公式表达：

$$
\text{治理复杂度} \propto \text{认证关系} + \text{路由关系} + \text{限流规则} + \text{观测埋点}
$$

在直连架构下，上述几项经常跟着 $N\times M$ 一起增长。引入网关后，外部治理集中到一个入口；引入网格后，内部治理集中到基础设施层。对初级工程师来说，这个边界非常重要：不要把“所有请求都先过网关”误解成“网关负责全部通信”。

## 核心机制与推导

API 网关处理的是横切关注点。`横切关注点`的白话解释是“很多服务都要做、但不属于任何一个业务服务本身的共性工作”。典型包括：

| 功能 | 为什么放在网关 |
| --- | --- |
| 认证授权 | 避免每个后端重复验证 Token |
| 限流配额 | 避免每个服务分别实现计数器 |
| 路由转发 | 统一暴露稳定入口，不让客户端感知内部拓扑 |
| 版本控制 | `/v1/chat`、`/v2/chat` 统一在边缘切换 |
| 协议转换 | 外部 HTTP，内部 gRPC 的转换可集中处理 |
| 入口日志 | 统一生成 trace id、request id、审计日志 |

服务网格的核心机制通常基于 `sidecar`。`sidecar` 可以理解为“部署在每个服务旁边的代理进程”，业务容器发出的请求先经过它，再进入网络。这样业务代码不需要自己实现 mTLS、重试、熔断、流量镜像等能力。

一个简化推导如下。

假设系统有 $N$ 个客户端、$M$ 个服务。

1. 直连模式下，每个客户端都要知道每个服务地址。
2. 每条连接都可能需要单独配置认证、超时、重试、限流。
3. 所以治理关系接近 $N\times M$。

引入网关后：

1. 客户端只需要知道网关地址。
2. 鉴权、入口限流、版本路由收口到网关。
3. 外部关系下降为 $N+M$。

再引入服务网格后：

1. 服务之间不再由业务代码手写重试和证书逻辑。
2. mTLS 和流量策略通过控制面统一下发。
3. 业务服务关注“做什么”，网格关注“怎么可靠地连”。

这也是为什么“网关 + 网格”比“只有一个大网关”更合理。前者把职责拆成“外部接入治理”和“内部通信治理”，后者容易把一切都堆进边缘层，最后网关本身变成系统瓶颈。

下面是一个推理感知路由的思路表。传统四层负载均衡通常只看连接或轮询；但大模型推理常常需要看“模型实例当前状态”。

| 路由因素 | 白话解释 | 为什么影响推理路由 |
| --- | --- | --- |
| 关键性 | 请求是否必须低延迟返回 | 高优先级请求应避开拥堵实例 |
| 资源利用率 | GPU 显存、算力是否紧张 | 防止把请求继续打到已接近上限的实例 |
| 适配器亲和 | 实例是否已加载对应 LoRA/Adapter | 避免重复加载导致冷启动 |
| 队列深度 | 当前排队长度 | 队列长说明尾延迟可能升高 |

真实工程例子是 LLM 平台中的“多模型池”。假设同一个逻辑模型 `faq-bot` 背后有多个实际实例，其中有的实例已经加载了特定租户的 LoRA，有的没有。如果用户请求带着该租户标识，推理扩展会优先选中“已有适配器上下文”的实例，而不是简单 round-robin。这样做的原因不是概念上更优雅，而是能减少加载抖动，降低首 Token 延迟。

## 代码实现

下面先用一个可运行的 Python 玩具程序，演示“直连治理关系”和“网关治理关系”的差异。它不是生产代码，但能帮助理解为什么入口收口有效。

```python
from dataclasses import dataclass

@dataclass
class Topology:
    clients: int
    services: int

    def direct_paths(self) -> int:
        return self.clients * self.services

    def gateway_paths(self) -> int:
        return self.clients + self.services

def governance_cost(paths: int, auth_cost: int = 3, rate_limit_cost: int = 2) -> int:
    # 用一个线性模型近似治理工作量
    return paths * (auth_cost + rate_limit_cost)

toy = Topology(clients=3, services=2)

direct = toy.direct_paths()
gateway = toy.gateway_paths()

assert direct == 6
assert gateway == 5
assert governance_cost(direct) > governance_cost(gateway)

print({
    "direct_paths": direct,
    "gateway_paths": gateway,
    "direct_cost": governance_cost(direct),
    "gateway_cost": governance_cost(gateway),
})
```

这个玩具例子表达的不是“6 和 5 差很多”，而是“随着客户端和服务继续增长，集中治理的收益会越来越明显”。

再看一个简化版的网关到模型池定义。`CRD` 是“自定义资源定义”，白话说就是“让 Kubernetes 认识一种新对象”。下面这个例子把多个模型版本挂在同一个逻辑入口后面：

```yaml
apiVersion: inference.networking.x-k8s.io/v1alpha2
kind: InferenceModel
metadata:
  name: faq-bot
spec:
  modelName: faq-bot
  targetModels:
    - name: llama-3-8b
      weight: 90
    - name: llama-3-8b-slim
      weight: 10
```

这个配置的意义是：外部仍然只访问一个稳定入口 `faq-bot`，但内部可以把 90% 流量发到标准模型，把 10% 发到轻量模型，或者用于灰度验证。

如果把它放进完整链路，常见请求路径是：

1. 客户端请求 `/v1/chat/completions`
2. API 网关完成鉴权、配额校验、生成 trace id
3. 网关把请求转到编排服务或模型入口
4. 服务网格 sidecar 负责 mTLS、重试和遥测
5. 推理扩展根据队列深度、GPU 状态、LoRA 亲和性选择后端
6. 返回结果时保留统一 trace id，便于排查慢请求

真实工程里，建议至少保留这几类观测字段：

| 字段 | 来源层 | 作用 |
| --- | --- | --- |
| request_id | 网关 | 单次请求唯一标识 |
| trace_id | 网关/网格 | 贯穿全链路追踪 |
| tenant_id | 业务层 | 多租户隔离与计费 |
| model_name | 推理层 | 确认命中的逻辑模型 |
| backend_instance | 网格/推理层 | 定位具体实例 |
| queue_depth | 推理层 | 判断慢请求是否因排队导致 |

对初级工程师最重要的实现原则只有一条：入口身份、内部通信、安全、观测这四件事不要散落在每个业务服务里重复写。它们应该尽量由网关和网格承接。

## 工程权衡与常见坑

API 网关和服务网格都不是“加上就一定更先进”的组件，它们会引入额外复杂度。

先看一个对比表：

| 维度 | API 网关主要痛点 | 服务网格主要痛点 |
| --- | --- | --- |
| 可用性 | 网关失效会影响全部入口 | 控制面或策略错误会影响内部调用 |
| 复杂度 | 易堆积业务逻辑 | 配置概念多，调试门槛高 |
| 监控成本 | 入口指标多但较集中 | 全链路指标更完整但更重 |
| 首次部署时间 | 相对较短 | 相对较长 |
| 性能开销 | 增加一跳网络转发 | sidecar 带来额外资源开销 |

常见坑有五个。

第一，把业务编排写进网关。网关适合做鉴权、限流、路由，不适合承载复杂业务分支。否则它会从“入口层”变成“第二业务系统”。

第二，只部署单实例网关。统一入口意味着统一风险，至少要多副本、健康检查、自动扩缩容，并把配置变更纳入发布流程。

第三，在业务代码里继续手写重试。服务网格已经能统一处理部分重试策略；业务层再盲目重试，容易把一次失败放大成重试风暴。

第四，没有区分“可重试错误”和“不可重试错误”。例如 429、上游短暂超时可能适合有限重试，但参数校验失败不应重试。

第五，一上来就全量引入重型网格。对团队还不熟悉流量治理时，直接上复杂配置，经常会出现“问题到底在应用、sidecar、控制面还是策略”的排障困难。

更稳妥的落地方式通常是分阶段：

1. 在开发或测试集群先验证网关入口、鉴权和 trace id 透传。
2. 先用较轻量的网格能力验证 mTLS 和基本重试。
3. 再逐步引入更复杂的流量治理，例如故障注入、分区域流量、推理感知路由。
4. 最后才在生产环境扩大到核心模型流量。

## 替代方案与适用边界

不是所有系统都需要同时使用 API 网关和服务网格。

如果你的系统只有 2 到 3 个内部服务，而且调用方几乎只有内部前端或管理后台，那么直接通信可能更划算。此时多加一层网关和一层网格，得到的治理收益可能还不足以覆盖引入的运维成本。

下面是三种常见方案的边界比较：

| 方案 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| 直接通信 | 结构简单，额外开销少 | 客户端重复实现认证和路由，扩展差 | 早期原型、内部工具、小规模系统 |
| 单 API 网关 | 对外治理清晰，适合统一入口 | 内部服务治理仍分散 | 对外开放接口、服务数量开始增长 |
| API 网关 + 服务网格 | 外部与内部治理同时标准化 | 学习和运维成本最高 | 多团队、多模型、多租户生产平台 |

一个常见的真实选择路径是：

开发初期，前端或内部 UI 直接调用单个模型服务，先把产品逻辑跑通；  
准备上线时，引入 API 网关统一做鉴权、配额和版本管理；  
当内部服务数量继续增加，且出现跨服务超时、证书、重试和追踪问题时，再引入服务网格。

所以结论不是“网关和网格越早越好”，而是“当连接关系和治理策略开始系统性膨胀时，它们才真正值回成本”。如果系统还处在频繁改接口、频繁改单体逻辑的阶段，过早引入复杂基础设施反而会拖慢迭代。
