## 核心结论

离线在线一致性分析，是判断“离线评估的提升”能否稳定预测“线上真实收益提升”的方法。它不关心离线指标本身是否好看，而关心离线结论能不能指导上线决策。

线上真正关心的是策略在真实流量中的收益：

$$
V(\pi)=\mathbb{E}[r]
$$

这里 $\pi$ 表示策略，白话说就是“系统在某个用户面前选择什么内容或动作的规则”；$r$ 表示奖励，白话说就是点击、转化、停留时长、成交额这类业务结果。

离线指标提升不保证线上效果提升。一个推荐模型离线 AUC 从 0.82 提升到 0.85，线上 CTR 却没有变化甚至下降，这并不矛盾。原因可能是离线 AUC 衡量的是“历史点击样本上的排序区分能力”，线上 CTR 衡量的是“真实曝光机制下用户是否点击”。两者不是同一个问题。

| 对比项 | 离线指标 | 线上指标 |
|---|---|---|
| 数据来源 | 历史日志 | 实时流量 |
| 典型指标 | AUC、NDCG、Recall、LogLoss | CTR、CVR、GMV、停留时长 |
| 主要风险 | 被历史曝光策略带偏 | 影响真实用户体验 |
| 优点 | 成本低、迭代快、可回放 | 结论直接、业务可信 |
| 局限 | 只能近似反事实结果 | 成本高、周期长、有风险 |

真正决定一致性的，通常不是模型结构本身，而是四件事是否一致：目标函数、数据采集机制、特征处理链路和曝光策略。模型只是其中一环。

## 问题定义与边界

离线评估，是基于历史日志推断一个新策略可能带来的效果。线上实验，是把新策略放进真实流量里直接观测结果。两者的信息条件不同，所以不能直接等价。

一致性分析关注的是：离线结果是否能预测线上结果。它不是讨论模型是否收敛，也不是讨论模型能否拟合历史数据。一个模型可以很好地拟合历史点击，却在线上表现很差，因为历史点击本身来自旧策略的选择。

| 概念 | 白话解释 | 解决什么问题 | 不能解决什么问题 |
|---|---|---|---|
| 离线评估 | 用历史日志评估新模型 | 快速筛掉明显差的方案 | 不能完全替代线上实验 |
| 线上实验 | 在真实用户流量里做 A/B | 直接观测业务收益 | 成本高，不能无限试错 |
| 一致性分析 | 检查离线提升是否能映射到线上提升 | 判断离线指标是否可信 | 不能凭空生成缺失的反事实信息 |
| 适用范围 | 推荐、广告、搜索、排序、风控策略 | 历史日志较完整的策略评估 | 日志缺关键字段时效果有限 |
| 核心边界 | 没有曝光和反馈就没有可识别结论 | 明确推断范围 | 不能估计从未出现过的动作 |

推荐系统是典型场景。离线评估时，系统用历史点击数据评估排序模型；线上实验时，业务看真实 CTR、停留时长、转化率或 GMV。问题在于，历史数据里的“用户点击了什么”并不是自然发生的，它先经过旧推荐策略筛选。用户没有看到的内容，日志里通常没有点击反馈。

这就是反事实信息缺失。反事实信息，白话说就是“如果当时给用户展示另一个内容，会发生什么”。如果日志里没有这些信息，离线评估只能近似，不能直接等价于线上实验。

## 核心机制与推导

离线在线不一致主要来自三类偏差：曝光偏差、分布漂移、训练-服务偏差。

曝光偏差，是指日志只记录了旧策略展示过的内容，未展示内容没有反馈。分布漂移，是指训练或评估时的数据分布和线上真实流量分布不同。训练-服务偏差，是指训练阶段使用的特征处理和线上服务阶段不一致。

统一记号如下：

| 符号 | 含义 | 白话解释 |
|---|---|---|
| $x$ | 上下文 | 用户、场景、设备、时间等特征 |
| $a$ | 动作 | 推荐某个物品、广告或排序结果 |
| $r$ | 奖励 | 点击、转化、成交等反馈 |
| $\pi(a|x)$ | 目标策略 | 新策略在上下文 $x$ 下选择动作 $a$ 的概率 |
| $p_0(a|x)$ | 日志策略概率 | 旧策略当时展示动作 $a$ 的概率，也叫 propensity |
| $D$ | 日志数据 | 历史记录集合 |

目标是估计新策略的真实价值：

$$
V(\pi)=\mathbb{E}_{x\sim \mathcal D,\ a\sim \pi(\cdot|x)}[r]
$$

如果直接用历史日志均值评估新策略，通常会被旧策略的曝光方式带偏。更合理的方法是用逆概率加权，英文是 Inverse Propensity Weighting，简称 IPW。它的核心思想是：旧策略很少展示但新策略很想展示的动作，需要在估计中放大权重。

$$
\hat V_{\text{IPW}}(\pi)=\frac1n\sum_{i=1}^n \frac{\pi(a_i|x_i)}{p_i} r_i
$$

这里 $p_i$ 是日志中动作 $a_i$ 被旧策略展示的概率。分子是新策略选择该动作的概率，分母是旧策略选择该动作的概率。

玩具例子如下。两个上下文 $x_1,x_2$，两个动作 A、B，各出现 100 次。真实点击率为：

| 上下文 | A | B |
|---|---:|---:|
| $x_1$ | 0.8 | 0.9 |
| $x_2$ | 0.6 | 0.1 |

旧日志策略有明显曝光偏置：

| 上下文 | A 的曝光 | B 的曝光 |
|---|---:|---:|
| $x_1$ | 10 | 90 |
| $x_2$ | 90 | 10 |

观察到的点击数是：A 在 $x_1$ 有 8 次点击，在 $x_2$ 有 54 次点击；B 在 $x_1$ 有 81 次点击，在 $x_2$ 有 1 次点击。直接算日志均值：

| 动作 | 原始日志点击率 |
|---|---:|
| A | $(8+54)/100=62\%$ |
| B | $(81+1)/100=82\%$ |

原始日志会误判 B 更好。但如果目标策略希望在两个上下文上均匀评估动作效果，用 IPW 校正：

| 动作 | IPW 估计 |
|---|---:|
| A | $(8/0.1+54/0.9)/200=70\%$ |
| B | $(81/0.9+1/0.1)/200=50\%$ |

校正后恢复真实结论：A 更好。

另一个常用方法是双重稳健估计，英文是 Doubly Robust，简称 DR。它同时使用奖励模型和 propensity 校正：

$$
\hat V_{\text{DR}}(\pi)=\frac1n\sum_{i=1}^n \left[\hat m(x_i,\pi)+\frac{\pi(a_i|x_i)}{p_i}\bigl(r_i-\hat m(x_i,a_i)\bigr)\right]
$$

$\hat m(x,a)$ 是奖励模型，白话说就是“先预测某个用户看到某个物品会不会点击”。DR 的直觉是：先用模型给出一个基础估计，再用真实日志残差修正它。

无偏或近无偏估计依赖几个条件：日志策略必须有支持集覆盖，propensity 必须可用，日志不能缺关键动作。支持集覆盖，白话说就是“新策略想评估的动作，旧日志里至少出现过”。如果 $p_0(a|x)=0$，旧日志里从未给这个上下文展示过动作 $a$，那么无论 IPW 还是 DR，都无法可靠估计它的效果。

真实工程例子是信息流推荐。旧策略偏向热门内容，因此日志里热门内容曝光多、反馈多，长尾内容曝光少、反馈少。新模型如果更偏长尾，离线 NDCG 可能上升，因为它在历史点击样本上排序更好；但线上 CTR 或 GMV 可能下降，因为真实用户对这些长尾内容没有足够兴趣，或者内容质量不稳定。这时问题不是模型“学坏了”，而是离线评估没有准确反映线上策略收益。

## 代码实现

工程实现的重点不是重新训练模型，而是把日志读取、字段校验、propensity 校正、分片分析和结果对齐串成可复用流程。

最少需要这些字段：

| 字段名 | 含义 | 是否必需 | 缺失后果 |
|---|---|---:|---|
| `x` | 上下文或分片标识 | 是 | 无法做分布和分片分析 |
| `a` | 实际曝光动作 | 是 | 无法知道旧策略展示了什么 |
| `r` | 用户反馈奖励 | 是 | 无法计算策略收益 |
| `p` | 旧策略曝光概率 | 是 | 无法做 IPS / DR 校正 |
| `pi` | 新策略选择该动作的概率 | 是 | 无法评估目标策略 |
| `slice` | 用户组、设备、地区、流量位 | 建议 | 只能看总体，容易掩盖问题 |

下面是一个可运行的 Python 最小实现，展示 IPS、clipping 和分片指标。clipping，白话说就是把过大的权重截断，降低方差。

```python
from collections import defaultdict

logs = [
    {"x": "x1", "a": "A", "r": 1, "p": 0.1, "pi": 0.5, "slice": "new_user"},
    {"x": "x1", "a": "B", "r": 1, "p": 0.9, "pi": 0.5, "slice": "new_user"},
    {"x": "x2", "a": "A", "r": 1, "p": 0.9, "pi": 0.5, "slice": "old_user"},
    {"x": "x2", "a": "B", "r": 0, "p": 0.1, "pi": 0.5, "slice": "old_user"},
]

required = {"x", "a", "r", "p", "pi"}
for row in logs:
    missing = required - row.keys()
    assert not missing, f"missing fields: {missing}"
    assert 0 < row["p"] <= 1
    assert 0 <= row["pi"] <= 1
    assert row["r"] in (0, 1)

def ips_value(rows, clip=None):
    total = 0.0
    for row in rows:
        w = row["pi"] / row["p"]
        if clip is not None:
            w = min(w, clip)
        total += w * row["r"]
    return total / len(rows)

def slice_ips(rows, clip=None):
    groups = defaultdict(list)
    for row in rows:
        groups[row["slice"]].append(row)
    return {name: ips_value(items, clip=clip) for name, items in groups.items()}

raw_ctr = sum(row["r"] for row in logs) / len(logs)
ips = ips_value(logs)
ips_clipped = ips_value(logs, clip=3.0)
by_slice = slice_ips(logs, clip=3.0)

assert raw_ctr == 0.75
assert round(ips, 4) == 1.5278
assert round(ips_clipped, 4) == 1.0
assert set(by_slice.keys()) == {"new_user", "old_user"}

print("raw_ctr =", raw_ctr)
print("ips =", ips)
print("ips_clipped =", ips_clipped)
print("slice_ips =", by_slice)
```

这段代码刻意保留了一个现象：未裁剪的 IPS 可能大于 1。对于点击率这种 $[0,1]$ 奖励，这看起来反直觉，但它反映的是小样本和大权重导致的高方差，不代表点击率真的超过 100%。工程里通常会继续使用 SNIPS、clipping、分层评估或扩大探索流量来降低不稳定性。

SNIPS 是 Self-Normalized IPS，白话说就是“把加权奖励除以权重总和”，让估计更稳定，但会引入一定偏差。

## 工程权衡与常见坑

一致性分析不是追求“完全无偏”，而是在偏差和方差之间找可上线的平衡。偏差，白话说是估计结果系统性偏离真实值；方差，白话说是估计结果对样本波动非常敏感。

低 propensity 是最常见的问题之一。假设旧策略展示某动作的概率是 0.001，新策略又很想展示它，那么 IPS 权重可能接近 1000。只要这一条样本点击了，整体估计就可能被拉得很高。此时不是公式错了，而是日志对这个动作的信息太少。

| 坑 | 典型表现 | 规避方式 |
|---|---|---|
| 指标不对齐 | 离线 AUC 涨，线上 GMV 不涨 | 先定义线上主指标，再设计离线 proxy |
| 训练-服务偏差 | 离线验证很好，线上明显回落 | 统一特征处理，记录 serving 侧特征 |
| 曝光偏差 | 热门物品在离线评估中总是占优 | 记录 propensity，使用 IPS / DR / SNIPS |
| 低 propensity | 权重爆炸，结果不稳定 | clipping、归一化、增加探索流量 |
| 时间泄漏 | 离线异常好，上线失效 | 按时间切分，禁止使用未来信息 |
| 只看总体均值 | 总体提升，关键人群变差 | 做 slice analysis |
| 日志字段缺失 | 无法复现当时决策 | 补齐动作、概率、特征版本、实验桶 |
| 延迟和体验未纳入 | 指标涨但投诉率升高 | 加护栏指标 |

分片分析是工程里必须做的。分片，白话说就是按用户或场景拆开看结果，而不是只看一个总体均值。

| 分片 | 离线 IPS | 线上 CTR | 可能解释 |
|---|---:|---:|---|
| 新用户 | +4.2% | -1.1% | 新用户特征稀疏，模型过度依赖热门偏好 |
| 老用户 | +2.0% | +1.8% | 历史行为充分，离线和线上接近 |
| iOS | +1.5% | +1.4% | 链路稳定 |
| Android | +1.7% | -0.8% | 客户端埋点或特征版本可能不一致 |
| 首页流量位 | +3.1% | +2.6% | 主场景一致 |
| 详情页推荐 | +2.8% | -2.0% | 用户意图和首页不同 |

只看总体均值会掩盖局部风险。一个模型总体 CTR 提升 1%，但新用户下降 5%，可能仍然不能上线，因为新用户体验影响长期留存。

## 替代方案与适用边界

IPS 和 DR 不是万能解法。它们要求日志中有足够探索，并且能拿到可靠的曝光概率。如果某个动作在历史日志里从未出现，即 $p_0(a|x)=0$，就不能靠 IPS 或 DR 可靠估计它的效果。因为分母为 0，本质上没有数据支撑这个反事实判断。

| 方法 | 适用场景 | 前提条件 | 优点 | 局限 |
|---|---|---|---|---|
| 直接离线指标 | 快速模型筛选 | 历史日志较稳定 | 简单、便宜、迭代快 | 容易受曝光偏差影响 |
| IPS / IPW | 有 propensity 的策略评估 | 支持集覆盖，概率可靠 | 理论清晰，可校正曝光偏差 | 低 propensity 时方差高 |
| DR | 有奖励模型和 propensity | 至少一侧较可靠 | 比单纯 IPS 更稳 | 实现复杂，仍受覆盖限制 |
| 随机探索桶 | 需要收集无偏数据 | 可承受少量探索成本 | 数据质量高 | 可能损失短期收益 |
| Shadow mode | 新策略只打分不曝光 | 可记录新旧策略输出 | 风险低，便于排查链路 | 不能直接获得新曝光反馈 |
| Interleaving | 搜索、排序对比 | 可混排候选结果 | 对排序差异敏感 | 适用范围较窄 |
| 分桶实验 | 关键上线决策 | 有实验平台和足够流量 | 结论最直接 | 成本高，周期长 |
| Proxy metric 回归 | 线上指标稀疏或滞后 | proxy 与主指标稳定相关 | 便于日常迭代 | proxy 可能失效 |

什么时候不用 IPS / DR？第一，日志没有记录 propensity。第二，新策略大量选择旧日志从未展示过的动作。第三，曝光概率极低导致估计完全由少数样本支配。第四，奖励定义和线上业务目标不一致。第五，日志存在严重埋点缺失或特征版本混乱。

这时更合理的做法是补充探索流量、做随机桶实验、使用 shadow mode 检查链路，或者先采用更保守的 proxy 指标。对于高风险策略，离线一致性分析只能作为上线前证据之一，不能替代完整实验体系。

离线在线一致性的最终目标，是建立一条可信的工程判断链：离线指标为什么能代表线上目标，历史日志为什么能支持这个估计，偏差在哪里，方差有多大，哪些分片可能失效，以及上线实验应该重点观察什么。
