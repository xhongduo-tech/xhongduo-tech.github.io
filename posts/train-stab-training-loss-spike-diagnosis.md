## 核心结论

Loss 突刺，指训练过程中损失值在短时间内突然大幅上升，随后又可能回落的现象。它通常不是“模型彻底坏了”，而是参数更新在某些方向上暂时越过了稳定边界。更具体地说，当优化器沿着高曲率方向更新时，如果有效步长过大，就会把参数从低损失区域直接甩出去，表现为 loss 瞬间飙升。

这里的高曲率，可以先理解为“损失面对参数变化非常敏感”。在平坦方向上，参数改一点，loss 变化不大；在陡峭方向上，参数只动一点，loss 就会上升很多。训练后期尤其容易出现一种现象：模型整体 loss 更低了，但局部几何更尖锐了。这类区域常被概括为 LLAS，意思是“更低损失但更尖锐”的区域。进入这种区域后，只要学习率、动量、预条件缩放三者组合得不合适，就容易触发突刺。

最关键的稳定条件可以写成：

$$
\lambda_{\max} < \frac{2}{\eta}
$$

其中 $\lambda_{\max}$ 是预条件 Hessian 的最大特征值，白话说就是“当前最危险方向的弯曲程度”；$\eta$ 是学习率。如果最危险方向的曲率已经高到超过 $2/\eta$，那么这一步更新就可能从“朝谷底走”变成“直接冲出谷底”。

新手可以先记住一个图像化理解：模型原本沿着光滑山谷往下走，突然踩到一个特别陡的侧坡，因为步子太大，被弹出谷底，loss 就会从 0.18 一下跳到 0.5，等优化器后续重新找到平坦方向，loss 又回到 0.19 左右。这个短时间的暴涨，就是训练 Loss 突刺。

---

## 问题定义与边界

Loss 突刺不是所有 loss 波动都算。它有几个边界条件。

第一，它通常是短时间的异常跃升，而不是长期发散。比如模型训练时长期从 0.5、0.8、1.2 一路变差，那更像学习率整体过高或数据分布崩坏；而如果是 0.20、0.19、0.18 之后突然跳到 0.52，再过几十步又回到 0.21，这更接近本文讨论的突刺。

第二，它往往发生在已经训练到较低损失的阶段。原因不是“后期更脆弱”这么简单，而是低损失区域常常伴随更复杂的局部几何结构。模型可能在大多数方向都已经很稳定，但只要有少数高曲率方向被碰到，优化器就会在这些方向上失稳。

第三，它不等于单纯的数据脏样本。极端 batch 确实也会让 loss 抖一下，但如果你同时看到梯度范数持续抬升、Adam 二阶矩统计滞后、固定高学习率不退火，那么更可能是优化稳定性问题，而不是数据问题。

下表把常见触发条件、表现和可监控指标放在一起：

| 触发条件 | 常见表现 | 可监控指标 |
|---|---|---|
| 学习率偏高 | loss 突然跳升，随后缓慢回落 | `lr`、loss 曲线斜率 |
| 高曲率方向出现 | 某几个 step 内梯度急剧变大 | Hessian 最大特征值估计、梯度范数 |
| Adam/AdamW 的 $\beta_2$ 过大 | 二阶矩更新过慢，危险方向被低估 | `exp_avg_sq` 变化速度、有效步长 |
| 动量累积过强 | 突刺后不会立刻恢复，会拖很多步 | 动量缓冲区范数 |
| 混合精度数值不稳 | 突刺后接 NaN/Inf | loss scale、梯度是否含 NaN |
| 极端 batch 或数据异常 | 单次抖动明显，但未必持续 | per-batch loss、样本长度分布 |

一个玩具例子：某分类模型训练到第 8000 步时，最近 200 步 loss 一直在 0.18 到 0.21 之间波动。第 8001 步突然跳到 0.47，第 8002 步到 0.53，第 8010 步又回到 0.22。这种“先爆一下再回来”的模式，才是诊断 Loss 突刺的典型入口。

## 核心机制与推导

要理解为什么会突刺，可以先看一维二次函数：

$$
f(x) = \frac{1}{2}\lambda x^2
$$

这里 $\lambda$ 表示曲率，数值越大，函数越陡。对它做最基本的梯度下降：

$$
x_{t+1} = x_t - \eta \nabla f(x_t) = x_t - \eta \lambda x_t = (1-\eta\lambda)x_t
$$

要想不发散，必须满足：

$$
|1-\eta\lambda| < 1
$$

整理得到：

$$
0 < \eta\lambda < 2
\quad \Rightarrow \quad
\lambda < \frac{2}{\eta}
$$

这就是稳定阈值的来源。真实神经网络当然不是一维二次函数，而是高维、非凸、不断变化的损失面。但在某一步附近，沿着某个最危险方向局部近似成二次形式，仍然成立。所以工程上常把“预条件 Hessian 最大特征值是否逼近 $2/\eta$”当成突刺预警。

这里的“预条件”要单独解释。预条件器，白话说就是优化器对不同方向施加不同缩放。SGD 基本不做这件事，而 Adam 会根据历史梯度平方做缩放。因此实际危险的不是原始 Hessian，而是“被优化器缩放之后的 Hessian”。这就是为什么同一个模型，同一个学习率，用 SGD 可能不突刺，用 AdamW 却可能突刺。

Adam/AdamW 在大 $\beta_2$ 下更容易出问题。$\beta_2$ 是二阶矩平滑系数，白话说就是“历史梯度平方记多久”。$\beta_2$ 越大，统计越平滑，但越滞后。如果当前 batch 或当前区域的高曲率突然出现，二阶矩估计还停留在旧状态，就会错误地把步长放得过大。于是危险方向上的有效更新超出稳定边界，loss 就会爆一下。

数值例子可以直接代入。假设某阶段预条件 Hessian 的最大特征值估计为：

$$
\lambda_{\max} \approx 120
$$

当前学习率为：

$$
\eta = 0.02
$$

那么稳定阈值是：

$$
\frac{2}{\eta} = \frac{2}{0.02} = 100
$$

因为 $120 > 100$，说明当前至少有一个方向已经越界。此时即使大多数方向仍然稳定，只要更新在这个方向上分量不小，就会看到 loss 突然暴涨。之后如果动量逐渐衰减、学习率降低、或者模型重新回到较平坦区域，比如 $\lambda_{\max}$ 回到 80，那么又重新满足稳定条件，loss 就会恢复。

可以把不同曲率下的行为概括成下面这张表：

| 情况 | 数学关系 | 训练表现 |
|---|---|---|
| 低曲率方向 | $\eta\lambda \ll 2$ | 更新平稳，loss 缓慢下降 |
| 接近阈值 | $\eta\lambda \approx 2$ | loss 震荡加剧，梯度范数抬升 |
| 超过阈值 | $\eta\lambda > 2$ | 突刺、震荡，严重时 NaN |
| 回到平坦区 | $\eta\lambda < 2$ 再次成立 | loss 恢复，训练继续 |

真实工程例子是 LLM 训练。模型在长时间平稳下降后，突然在某个阶段频繁出现 loss spike。排查后常见结论不是“数据突然坏了”，而是高 $\beta_2$ 的 AdamW 在某个高曲率阶段反应过慢，使危险方向的有效步长过大。训练团队通常不会等它自然恢复，而是主动触发降学习率、重置动量或梯度裁剪，因为一次突刺如果继续放大，可能直接烧掉几小时训练资源。

## 代码实现

工程上不需要每一步都精确算 Hessian，这太贵。更实用的方法是用三个近似信号做联防：梯度范数、损失斜率、最大曲率的粗略估计。只要这些信号接近阈值，就先保护训练，再追求速度。

下面给一个可运行的 Python 玩具实现。它不是深度学习框架代码，而是把“阈值判断 + 裁剪 + 降学习率 + 动量重置”的逻辑单独演示出来。

```python
from collections import deque
from math import isfinite

class SpikeGuard:
    def __init__(self, lr, beta1=0.9, grad_clip=5.0, window=5):
        self.lr = lr
        self.beta1 = beta1
        self.grad_clip = grad_clip
        self.grad_hist = deque(maxlen=window)
        self.momentum = 0.0

    def step(self, grad, lambda_max_est):
        assert isfinite(grad)
        assert lambda_max_est > 0

        grad_norm = abs(grad)
        self.grad_hist.append(grad_norm)

        eta_limit = 2.0 / lambda_max_est
        spike_risk = self.lr > eta_limit

        # 梯度裁剪：先限制单步破坏性
        if grad_norm > self.grad_clip:
            grad = self.grad_clip if grad > 0 else -self.grad_clip
            grad_norm = abs(grad)

        # 动量更新
        self.momentum = self.beta1 * self.momentum + grad

        # 接近或越过稳定阈值时，减半学习率并重置动量
        if spike_risk:
            self.lr *= 0.5
            self.momentum = 0.0

        update = -self.lr * (grad + 0.1 * self.momentum)
        return {
            "grad_norm": grad_norm,
            "eta_limit": eta_limit,
            "lr": self.lr,
            "spike_risk": spike_risk,
            "update": update,
        }

guard = SpikeGuard(lr=0.02)

# 安全区：lambda_max = 80, 2/lambda = 0.025, 当前 lr=0.02 未越界
safe = guard.step(grad=3.0, lambda_max_est=80.0)
assert safe["spike_risk"] is False
assert abs(safe["lr"] - 0.02) < 1e-12

# 危险区：lambda_max = 120, 2/lambda = 0.0167, 当前 lr=0.02 越界
danger = guard.step(grad=6.0, lambda_max_est=120.0)
assert danger["spike_risk"] is True
assert abs(danger["lr"] - 0.01) < 1e-12
assert danger["grad_norm"] <= 5.0
```

如果把这段逻辑迁移到真实训练循环，通常要监控下面这些量：

| 参数/指标 | 含义 | 典型用途 |
|---|---|---|
| `grad_norm` | 梯度范数，表示整体更新强度 | 发现爆炸前兆 |
| `lambda_max_est` | 最大曲率方向的粗估值 | 判断是否逼近 $2/\eta$ |
| `eta_limit=2/lambda_max_est` | 当前稳定上限 | 决定是否降学习率 |
| `grad_clip` | 梯度裁剪阈值 | 防止单步更新过猛 |
| `beta1/beta2` | 动量与二阶矩平滑强度 | 调节响应速度与稳定性 |

在真实工程里，一般不会写成上面这么简化，而是接在训练 step 后面做监控。例如：

1. 每个 step 记录 `loss`、`grad_norm`、`lr`。
2. 每隔 N 步估计一次最大曲率，哪怕只是用 power iteration 做粗估计。
3. 若 `lr > 2 / lambda_max_est`，立即执行三件事：梯度裁剪、学习率减半、重置 Adam 状态中的动量或二阶矩缓存。
4. 如果连续多次触发，再进入更强的 warmup/decay 模式，而不是继续硬顶。

新手可以把这理解成开车防打滑：先看轮胎是否开始飘，也就是梯度范数是否异常；再看当前车速是否超过路面承受能力，也就是学习率是否越界；一旦打滑，就先减速，再清掉错误惯性，而不是继续猛踩油门。

## 工程权衡与常见坑

Loss 突刺诊断的核心权衡，不在“要不要监控”，而在“监控到多细”。监控越密，稳定性越好，但额外算力和日志开销也越高。对小模型，这点开销通常可以接受；对大模型，尤其是分布式训练，Hessian 相关估计会明显增加成本，因此常见做法是“高频看梯度范数，低频看曲率粗估”。

常见坑主要有下面几类：

| 坑 | 症状 | 对策 |
|---|---|---|
| 只看训练 loss，不看梯度范数 | loss 已开始抖，但没有提前告警 | 记录 `grad_norm` 滑动窗口 |
| 学习率长期固定偏高 | 训练后期频繁 spike | 增加 warmup，后期 per-step decay |
| AdamW 的 $\beta_2$ 太大 | 高曲率变化跟不上，突刺更突然 | 适度降低 $\beta_2$，提高响应速度 |
| 混合精度状态未对齐 | spike 后迅速 NaN | 核查 FP16/FP32 主权重与动量缓存 |
| 只做梯度裁剪，不调学习率 | 短期缓解，长期仍反复突刺 | 裁剪和降学习率一起用 |
| 误把数据异常当成优化异常 | 盲目重启优化器，问题反复 | 同时检查 batch 统计与数据分布 |

真实工程例子是大语言模型预训练。假设你在 8 卡或 64 卡环境里使用 AdamW，$\beta_2=0.999$，训练前期没问题，到了某个 token 区间后开始每隔几千步出现一次损失尖峰。如果此时只看平均 loss，你会误以为“还能回来，不严重”；但如果同时看梯度范数，通常会发现尖峰前一小段时间，`grad_norm` 已经持续抬升。这时最有效的组合往往是：

1. 在触发阈值时把学习率减半。
2. 对当前 step 执行梯度裁剪。
3. 必要时重置优化器中的动量状态。
4. 若仍反复触发，则缩短学习率平台期，提前进入 decay。

很多人容易忽略一点：突刺并不一定立刻造成训练失败，但它会偷偷损耗训练效率。因为每次 spike 都意味着参数被甩出稳定区域，后面要花若干步再爬回来。即使没有 NaN，累计下来也会浪费大量算力。

## 替代方案与适用边界

如果已经确定问题来自高曲率阶段的越界更新，那么替代方案的核心，不是“换个技巧试试”，而是重新管理 $\lambda_{\max}$ 和 $\eta$ 的关系。

下表是几种常见替代策略：

| 策略 | 核心思路 | 适用场景 | 局限 |
|---|---|---|---|
| 更长 warmup | 让学习率慢慢升，避免早期直接越界 | 刚进入复杂损失面阶段 | 后期 spike 不一定能解决 |
| 更强 lr decay | 高曲率阶段主动减小步长 | 中后期频繁突刺 | 收敛速度可能变慢 |
| 梯度裁剪 | 限制单步更新幅度 | 突发性大梯度 batch | 不能替代学习率调度 |
| 调低 $\beta_2$ | 提高 Adam 对新曲率的响应速度 | AdamW 滞后明显的场景 | 统计更噪，可能影响平滑性 |
| 换优化器 | 用更稳的预条件方法 | 已确认 AdamW 不稳 | 调参成本高 |
| 周期性重启优化器状态 | 清除错误惯性累积 | 突刺后恢复慢 | 可能损失部分历史信息 |

对初学者，最直接的替代方案通常是把学习率从 0.02 降到 0.01，或者延长 warmup，让优化器更慢地进入高曲率区域。这种方法虽然朴素，但常常有效，因为它直接把稳定条件中的 $\eta$ 压小了。

但要注意适用边界。如果 $\lambda_{\max}$ 本身波动非常剧烈，那么单纯降低学习率可能只能缓解，不能根治。这种场景更适合配合梯度裁剪和更密的监控。如果问题主要来自 Adam 二阶矩估计滞后，那么调低 $\beta_2$ 或切换优化器更有针对性。

还有一个现实边界：很多团队并不会在线估计真正的 Hessian 最大特征值，因为成本太高。此时“梯度范数 + loss 斜率 + 学习率计划”的组合，是最划算的近似方案。它不如曲率估计精确，但对于防止工程事故已经足够有效。
