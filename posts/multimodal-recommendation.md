## 核心结论

多模态推荐的目标，不是把“图片、文本、点击”简单堆在一起，而是把三类信号映射到同一个用户-物品表示空间，再让排序模型学习它们在不同场景下各自有多重要。这里的“表示空间”可以理解为一种统一坐标系，用户和商品都会变成向量，向量越接近，越可能发生点击、收藏或购买。

对零基础读者，最重要的结论有四个。

第一，视觉特征、文本特征、行为特征是互补关系。视觉特征负责表达款式、颜色、版型、审美；文本特征负责表达属性、用途、品牌、适用场景；行为特征负责表达用户已经表现出的短期和长期偏好。单模态很容易漏信息，多模态的价值就在于补漏。

第二，工程上最常见的编码方式是：图像用预训练 CNN 或 ViT，文本用预训练语言模型如 BERT，行为序列用 Transformer 或其他序列模型。预训练的意思是模型已经在大规模通用数据上学过基础表达能力，业务侧只需要拿它做特征提取，成本低于从零训练。

第三，真正影响效果的往往不是“有没有多模态”，而是“怎么融合”。简单拼接虽然最容易上线，但通常不能表达“哪一模态在当前用户、当前商品、当前上下文下更重要”。因此实际系统常加门控和注意力，让模型按维度、按场景分配权重。

第四，模态缺失必须单独处理。很多团队默认商品都同时有图、有标题、有描述，但真实数据经常不是这样。只要某个模态整块缺失，训练分布和线上分布就会错位。常见处理方式是图上传播补全、默认向量、蒸馏预测三类，其中图传播适合“先补齐再训练”的静态工程流程。

一个新手用户的玩具例子很直观。用户刚注册，只点了 3 个商品：一双白色运动鞋、一件浅灰卫衣、一个简短标题写着“轻量透气跑步鞋”的商品。仅靠点击 ID，系统几乎不知道这个用户偏好什么；加入 ViT 提取的图像向量、BERT 提取的标题向量，再用行为 Transformer 编码这 3 次点击的顺序，系统已经能判断他更偏运动休闲而不是商务正装。

## 问题定义与边界

多模态推荐可以定义为：在推荐模型中同时使用两种及以上模态信息，对用户兴趣和商品属性进行统一建模。模态就是信息来源的类型，例如图像、文本、音频、视频、结构化属性、行为序列。本文只讨论最常见的三类：视觉、文本、行为。

它解决的核心问题主要有三类。

| 问题 | 单模态为什么难 | 多模态怎么补 |
| --- | --- | --- |
| 商品冷启动 | 新商品没有点击日志，ID 特征几乎不可用 | 先用图像和文本获得内容向量 |
| 用户冷启动 | 新用户行为少，协同过滤信号太弱 | 用早期点击序列结合内容特征快速估计偏好 |
| 稀疏场景下的泛化 | 点击分布长尾，很多商品交互极少 | 通过内容相似性把信息从热门商品迁移到长尾商品 |

边界也要讲清楚。

第一，多模态推荐不等于“理解一切语义”。如果商品图片质量差、标题极短、行为日志有噪声，那么多模态只是在噪声上做更复杂的计算，不会自动变准。

第二，模态缺失通常是整块缺失，不是少几个维度。比如某商品根本没有图，这不是向量里丢了几列，而是整个视觉编码器无法运行。此时要么跳过该模态，要么补一个近似向量。

第三，多模态特征只能提升“可表达性”，不能替代目标设计。如果业务目标是 CTR，模型学到的是点击倾向；如果目标是 GMV 或停留时长，最优特征组合和融合方式都会变。

处理缺失模态时，图传播方法很常见。可以把商品间共现关系构成图，已知视觉向量的商品把信息传播给缺图商品。归一化邻接矩阵常写为：

$$
\tilde{R}^I = D_I^{-1/2} R^I D_I^{-1/2}
$$

传播更新为：

$$
F_m^{(k+1)} = \tilde{R}^I F_m^{(k)}
$$

对已知模态的节点，每轮都重置：

$$
F_m^{(k+1)}[B] = Y[B]
$$

这里的白话解释是：已知特征当作锚点不让它漂移，未知节点反复从邻居平均吸收信息，直到稳定。

## 核心机制与推导

多模态推荐通常分三步：先编码，再融合，最后打分。

编码阶段：

- 视觉编码器把商品图片变成向量 $v$
- 文本编码器把标题或描述变成向量 $t$
- 行为编码器把用户历史序列变成向量 $b$

融合阶段不能只做拼接，因为拼接默认每一维同权，而且不显式建模模态交互。更常见的写法是门控加逐维注意力。一个典型形式是：

$$
h = \sigma(W_v v + W_t t + W_b b) + \text{BitWiseAttn}([v,t,b])
$$

其中 $\sigma$ 是非线性函数，例如 ReLU；$W_v,W_t,W_b$ 是把不同模态投影到同一空间的矩阵；`BitWiseAttn` 表示对每一维单独学习重要性。逐维的意思是，不同维度可能对应不同语义，例如某些维度更像“颜色”，某些维度更像“运动风格”，不能统一加权。

玩具例子如下。设：

$$
v=[0.4,0.2], \quad t=[0.1,0.5], \quad b=[0.3,0.7]
$$

若 $W_v=W_t=W_b=I$，且 $\sigma(x)=\max(0,x)$，则门控主干先得到：

$$
h_{base}=v+t+b=[0.8,1.4]
$$

如果逐维注意力额外给第二维更大权重，例如输出 $[0.05,0.20]$，则最终融合向量变成 $[0.85,1.60]$。这表示系统判断“第二维代表的偏好更关键”，比如它可能对应运动风格或休闲程度。

用户历史视觉偏好的建模也不能忽略。常见方法是对用户历史点击过的图片做 Query-Key-Value Attention。白话解释是：目标商品先拿自己的向量去“查询”用户历史，历史里越相似的商品权重越高，最后汇总成一个用户视觉兴趣向量。这比“把历史图片简单平均”更合理，因为用户可能同时浏览过西装和运动鞋，但当前目标是运动鞋时，模型应优先关注历史里的运动风格样本。

真实工程例子可以看电商首页推荐。用户近一周长期偏好是数码产品，但今天连续点击了健身服、跑鞋、运动耳机。行为 Transformer 会把这种短期转向编码出来；视觉分支识别图片里的材质和风格；文本分支识别“速干”“透气”“训练”等关键词。三者融合后，系统会短期上调运动品类，但不会完全忘掉用户长期数码偏好。这就是上下文感知的具体体现。

## 代码实现

下面给一个可运行的 Python 玩具实现，演示三件事：模态对齐、简单逐维门控、缺失视觉时的图传播补全。代码不依赖深度学习框架，重点是把机制写清楚。

```python
import numpy as np

def relu(x):
    return np.maximum(0.0, x)

def row_normalize(mat):
    row_sum = mat.sum(axis=1, keepdims=True)
    row_sum[row_sum == 0] = 1.0
    return mat / row_sum

def featprop_fill(adj, features, known_mask, steps=20):
    # known_mask=True 表示该商品这个模态已知
    F = features.copy().astype(float)
    Y = F.copy()
    A = row_normalize(adj.astype(float))
    for _ in range(steps):
        F = A @ F
        F[known_mask] = Y[known_mask]
    return F

def bitwise_attn(v, t, b):
    stacked = np.stack([v, t, b], axis=0)  # [3, d]
    scores = np.abs(stacked)
    weights = scores / (scores.sum(axis=0, keepdims=True) + 1e-8)
    return (weights * stacked).sum(axis=0)

def fuse(v, t, b):
    base = relu(v + t + b)
    return base + bitwise_attn(v, t, b)

# 1) 融合示例
v = np.array([0.4, 0.2])
t = np.array([0.1, 0.5])
b = np.array([0.3, 0.7])
h = fuse(v, t, b)

assert h.shape == (2,)
assert np.all(h >= np.array([0.8, 1.4]))  # attention 只会额外增强非负分量

# 2) 缺失视觉补全
# item0, item1 已知视觉；item2 缺失，但和 item1 共现强
adj = np.array([
    [1, 1, 0],
    [1, 1, 1],
    [0, 1, 1],
], dtype=float)

visual = np.array([
    [1.0, 0.0],
    [0.8, 0.2],
    [0.0, 0.0],  # 缺失
], dtype=float)

known_mask = np.array([True, True, False])
filled_visual = featprop_fill(adj, visual, known_mask, steps=30)

assert filled_visual[2, 0] > 0.0
assert filled_visual[2, 1] > 0.0

print("fused h =", h.round(4))
print("filled visual for item2 =", filled_visual[2].round(4))
```

这段代码虽然简化，但已经对应真实系统的三个模块。

| 模块 | 玩具实现 | 工程实现 |
| --- | --- | --- |
| 视觉编码 | 直接输入向量 `v` | `visual_vec = ViT(image)` 或 `CNN(image)` |
| 文本编码 | 直接输入向量 `t` | `text_vec = BERT(title, desc)` |
| 行为编码 | 直接输入向量 `b` | `behavior_vec = Transformer(seq)` |
| 模态融合 | `fuse(v, t, b)` | MLP + 门控 + attention + ranking head |
| 缺失补全 | `featprop_fill` | 基于 item-item 图离线补全并落表 |

如果做最小可上线版本，建议把系统拆成两层。

第一层是离线特征层。图片和文本先统一编码，缺失模态在这里补齐，最终把每个商品的多模态向量写入特征仓库。

第二层是在线排序层。用户实时行为进序列模型，拉取候选商品的离线多模态向量，再做融合打分。这种拆分的好处是稳定、成本可控，而且便于排查问题。

## 工程权衡与常见坑

多模态推荐最常见的问题，不是模型搭不起来，而是“上线后效果不稳定”。

| 常见坑 | 现象 | 规避方式 |
| --- | --- | --- |
| 只拼接目标商品特征 | 推荐看起来“像”，但不贴合用户历史 | 加入历史行为和历史视觉 attention |
| 忽略模态缺失 | 训练正常，线上大量 fallback | 离线补全或显式 missing token |
| 图像和文本维度未对齐 | 某一模态主导训练，另一模态失效 | 先投影到同维空间再融合 |
| 行为序列过长 | 延迟高，短期兴趣被长期噪声淹没 | 截断、分层建模或短长期双塔 |
| 简单平均多模态 | 冷启动改善有限 | 引入门控或 bit-wise attention |
| 只看离线 AUC | 线上 CTR、GMV 提升不稳定 | 联合看分桶效果、冷启动分群、延迟成本 |

有一个很典型的真实工程坑。某电商团队把新上市运动鞋的图像向量和标题向量直接拼到 CTR 模型里，结果推荐给大量偏好正式装的用户。原因不是图像模型错了，而是融合过于粗糙。文本里“皮革”“商务”这样的信号和图片里的“白底跑鞋”信号没有被区分对待，模型只学到“鞋类整体点击率”。加入用户历史视觉 attention 和逐维门控后，偏好正式风格的用户对运动鞋的曝光显著下降，推荐分布才恢复正常。

另一个坑是把缺失模态当成零向量。零向量在数学上方便，但在训练中往往变成一种伪语义，模型会误以为“全零”本身代表某类商品。更稳妥的方法有三种：一是显式加 missing 标记；二是做图传播补全；三是让文本分支蒸馏视觉分支，缺图时预测一个近似视觉向量。

## 替代方案与适用边界

不是所有业务都需要复杂的多模态融合。选择方案时，要先看数据规模、实时性要求和团队维护能力。

| 方案 | 优点 | 缺点 | 适用边界 |
| --- | --- | --- | --- |
| 单模态拼接 | 开发快，成本低 | 表达能力弱，冷启动一般 | 小团队验证阶段 |
| 传统 CTR + 多模态离线特征 | 稳定，容易部署 | 上下文建模有限 | 大多数中型内容/电商场景 |
| CTR + FeatProp 缺失补全 | 对缺图缺文友好 | 依赖图质量，传播会平滑过度 | 数据脏、模态缺失明显的业务 |
| RL + Transformer + GNN 动态建模 | 能建模反馈循环与策略调整 | 实现复杂，训练和评估成本高 | 高交互、高频反馈平台 |

如果平台没有足够的算力和在线训练能力，最现实的路线通常是：先做“离线多模态特征 + 传统排序模型”。这是一个强基线。视觉用 ViT，文本用 BERT，行为用轻量 Transformer，缺失模态用 FeatProp 或默认向量补齐。这样已经能覆盖新商品冷启动和一部分新用户场景。

当业务进入更复杂的互动阶段，例如短视频、电商信息流、实时反馈强的平台，可以考虑更动态的方法，比如把推荐看成序列决策问题，引入 RL、GNN 和更强的多模态编码器。它的优势在于能利用“推荐后用户怎么反馈”这一闭环信号，而不是只拟合静态点击标签。但它对样本量、特征更新、在线策略评估要求都更高，不适合作为起步方案。

一句话概括边界：如果你的问题主要是“新商品没点击”和“商品资料不完整”，优先做内容特征和缺失补全；如果你的问题主要是“用户兴趣切换快、反馈强依赖上下文”，再考虑更重的动态建模。
