## 核心结论

FB15k-237 与 WN18RR 是知识图谱链接预测里最常用的两个“去逆关系泄漏”基准。这里的“逆关系泄漏”可以先理解为：训练集已经给过一种方向的答案，测试集只是把方向反过来问一遍，模型即使不理解语义，也能靠模式记忆拿高分。FB15k-237 和 WN18RR 的价值，就在于尽量拆掉这种“抄答案通道”。

链接预测，白话说，就是给定一个三元组 $(s,r,o)$ 中缺失的一端，让模型补全。例如 $(\text{Barack\_Obama}, \text{presidentOf}, ?)$。评估时不是只看“答没答对”，而是看正确实体在全部候选中的排序位置，也就是 rank。rank 越小，说明模型把正确答案排得越靠前；基于 rank 推导出的 MRR、HITS@N，才是这类任务的核心指标。

例如在 FB15k-237 中，如果测试三元组 $(\text{Barack\_Obama}, \text{presidentOf}, \text{United\_States})$ 的目标实体被排在第 3 位，那么这一条对 MRR 的贡献是 $1/3$。这不是“完全答错”，而是说明模型已经把目标压到了前列，但仍没有把语义关系解码到最优位置。

| 数据集 | 来源 | 实体数 | 关系数 | 核心修正 |
|---|---|---:|---:|---|
| FB15k-237 | Freebase 子集 | 14,541 | 237 | 从 FB15k 中移除可由逆关系直接推断的关系 |
| WN18RR | WordNet 子集 | 40,943 | 11 | 从 WN18 中移除可导致反向泄漏的关系模式 |

结论可以压缩成一句话：如果目标是评估模型是否真的学到了关系语义，而不是只会背训练集的反向映射，优先使用 FB15k-237 与 WN18RR，而不是原始 FB15k 与 WN18。

## 问题定义与边界

知识图谱，白话说，就是把事实写成“头实体-关系-尾实体”的结构化图。例如 $(\text{Paris}, \text{capitalOf}, \text{France})$。链接预测任务则是在已知一部分图结构的前提下，预测缺失边是否存在。

更具体地说，常见评估有两种形式：

1. 头实体预测：$(?, r, o)$
2. 尾实体预测：$(s, r, ?)$

模型需要对所有候选实体打分，再把正确实体排到尽可能靠前的位置。

这个任务的边界非常重要。因为如果数据集切分不严，模型可能根本没学“关系语义”，只是利用了训练集和测试集之间的镜像关系。最典型的漏洞就是原始 FB15k 和 WN18 中的大量逆关系对。

玩具例子如下：

- 训练集有：$(\text{France}, \text{hasCapital}, \text{Paris})$
- 测试集有：$(\text{Paris}, \text{capitalOf}, \text{France})$

如果 `capitalOf` 与 `hasCapital` 是严格逆关系，那么模型只要记住“见过 Paris 和 France 这对实体”，就能把测试题做出来。它不需要理解“首都”这个概念，也不需要通过上下文推理。

WN18RR 的“RR”通常被理解为 reduced/relation-reduced 的修正版，核心目标就是把这种明显可逆推断压下去。FB15k-237 也是同样思路：不再保留大量能直接靠反向关系猜中的关系集合。

| 版本 | 是否容易出现逆关系泄漏 | 适合用途 |
|---|---|---|
| FB15k / WN18 | 高 | 仅作历史对照、低门槛 baseline |
| FB15k-237 / WN18RR | 低得多 | 更严格的语义泛化评估 |

边界还要再强调一次：

> 去逆关系不等于“没有任何数据偏差”。  
> 它只表示最显眼、最容易作弊的反向对应被削弱了，不表示数据集已经完全真实地代表工业知识图谱。

所以，使用这两个基准时，应该把它们理解为“比原始版本更可信的学术评估集”，而不是“覆盖全部工程问题的通用答案”。

## 核心机制与推导

这两个数据集之所以重要，不只是因为它们“删了点边”，而是因为删边之后，评估目标被重新定义了：模型必须靠打分函数学习实体和关系在嵌入空间中的相容性。

嵌入，白话说，就是把实体和关系映射成向量，让“语义上接近”的对象在数值空间里也更容易匹配。无论是 TransE、DistMult，还是更复杂的图推理模型，本质都要回答同一个问题：给定 $(s,r,o)$，这个三元组有多合理？

常见做法是定义一个评分函数 $f(s,r,o)$。分数越高，表示三元组越可能为真。评估时，对固定的 $(s,r, ?)$，枚举全部候选实体 $o'$，比较 $f(s,r,o')$，再看真实答案 $o$ 排第几。

MRR 的公式是：

$$
\mathrm{MRR}=\frac{1}{N}\sum_{i=1}^{N}\frac{1}{\mathrm{rank}_i}
$$

其中，$N$ 是测试样本数，$\mathrm{rank}_i$ 是第 $i$ 条测试三元组中真实答案的排名。

这个公式的直觉很直接：

- rank = 1，贡献 $1$
- rank = 2，贡献 $0.5$
- rank = 3，贡献 $0.333...$
- rank 越靠后，贡献越接近 $0$

所以 MRR 不只是看“有没有进前几”，而是持续惩罚排名后移。它比单纯准确率更适合链接预测，因为候选空间通常很大。

再看 HITS@N。它可以理解为：真实答案是否落在前 $N$ 名。例如 HITS@10，就是统计有多少比例的测试样本把正确实体排进前 10。这个指标更容易解释，但没有 MRR 那么细。因为 rank=1 和 rank=10 在 HITS@10 里都算“命中”，可它们显然不一样。

一个最小例子：

| rank | 对 MRR 的贡献 | 对 HITS@1 | 对 HITS@3 |
|---|---:|---:|---:|
| 1 | 1.0 | 1 | 1 |
| 2 | 0.5 | 0 | 1 |
| 3 | 0.333 | 0 | 1 |
| 8 | 0.125 | 0 | 0 |

这就是为什么论文通常同时汇报 MRR 与 HITS@1/3/10：前者反映整体排序质量，后者反映“排进可接受范围”的比例。

真实工程例子可以这样看。假设你在做企业知识库问答，图里有“药物-治疗-疾病”“公司-位于-城市”“作者-写作-书籍”等关系。如果训练数据里保留了大量逆边，模型可能只是学会“如果见过 A 指向 B，那就把 B 指回 A”。上线后，面对没见过的新组合时，效果会明显下滑。FB15k-237/WN18RR 的意义，就是在离线评估阶段尽早暴露这种虚高。

## 代码实现

实现一套最小可运行的评估逻辑，并不复杂。核心是三步：

1. 加载实体、关系和三元组切分；
2. 用模型给 $(s,r,o)$ 打分；
3. 对候选实体排序，计算 rank，再累积 MRR 与 HITS@N。

下面先给一个玩具版 `python` 示例。这个例子不依赖深度学习框架，只模拟“候选排序 -> 指标计算”的过程，适合理解评估逻辑。

```python
from math import isclose

def rank_of_true_entity(scores, true_entity):
    # 分数越高排名越靠前
    ordered = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    for idx, (entity, _) in enumerate(ordered, start=1):
        if entity == true_entity:
            return idx
    raise ValueError("true entity not found")

def evaluate(samples):
    mrr = 0.0
    hits1 = 0
    hits3 = 0

    for scores, true_entity in samples:
        rank = rank_of_true_entity(scores, true_entity)
        mrr += 1.0 / rank
        hits1 += int(rank <= 1)
        hits3 += int(rank <= 3)

    n = len(samples)
    return {
        "MRR": mrr / n,
        "HITS@1": hits1 / n,
        "HITS@3": hits3 / n,
    }

samples = [
    ({"United_States": 0.82, "Canada": 0.63, "Kenya": 0.21}, "United_States"),  # rank=1
    ({"Paris": 0.71, "London": 0.90, "Berlin": 0.55}, "Paris"),                  # rank=2
    ({"cat": 0.30, "animal": 0.95, "organism": 0.70}, "cat"),                    # rank=3
]

metrics = evaluate(samples)

assert isclose(metrics["MRR"], (1 + 1/2 + 1/3) / 3, rel_tol=1e-9)
assert isclose(metrics["HITS@1"], 1/3, rel_tol=1e-9)
assert isclose(metrics["HITS@3"], 1.0, rel_tol=1e-9)

print(metrics)
```

这段代码对应的逻辑就是：

- 第一条样本 rank=1，贡献 $1$
- 第二条样本 rank=2，贡献 $1/2$
- 第三条样本 rank=3，贡献 $1/3$

最终 MRR 是三者平均值。

如果换成真实工程流程，通常会在测试时对每个三元组执行下面的伪代码：

```python
for triple in test_set:
    scores = [model.score(s, r, candidate) for candidate in all_entities]
    rank = sorted(scores, reverse=True).index(model.score(triple)) + 1
    running_mrr += 1 / rank
mrr = running_mrr / len(test_set)
```

但工程里还要多处理两件事。

第一，filtered setting。它的意思是：如果候选集中某个实体和当前 $(s,r,?)$ 也能组成另一个已知真三元组，就不能把它简单当负样本，否则会误罚模型。学术评测中常会区分 raw rank 和 filtered rank，后者更常用。

第二，负采样。负采样，白话说，就是训练时不可能对全部实体做完整排序，所以要人为构造一些“错误候选”来近似学习目标。这会直接影响训练速度和稳定性，但最终验证/测试阶段仍应回到更完整的排序评估。

如果你用现成库，思路通常是：

| 工具 | 作用 | 适合情况 |
|---|---|---|
| PyKEEN | 知识图谱嵌入训练与评估框架 | 想快速复现实验 |
| 自定义脚本 + NumPy/PyTorch | 完全控制采样、过滤和指标 | 想对评估协议做定制 |
| 数据卡/Hugging Face 格式数据 | 方便下载 split 和元信息 | 想先把数据接进实验管线 |

真实工程例子：团队训练一个面向科研文献图谱的嵌入模型，关系包括“作者-隶属-机构”“论文-引用-论文”“方法-应用于-任务”。如果只看训练 loss，很容易误以为模型已经收敛；但把验证 checkpoint 选择标准切到 MRR/HITS@10 后，往往会发现后期训练虽然 loss 继续下降，排序指标却开始回落。这时应以验证集 MRR/HITS 为主做早停和 checkpoint 选择，而不是只看损失函数。

## 工程权衡与常见坑

去逆关系之后，评估更可信，但工程成本也会上来。原因很简单：不能再靠“记住反向答案”拿分，模型必须在更大的候选空间里做真正排序。

FB15k-237 与 WN18RR 的一个直接差异是实体规模不同。WN18RR 的实体数更大，意味着测试时替换头实体或尾实体的候选集合也更大，完整排序更慢。对初级工程师来说，这会在两个地方出问题：一是显存和内存压力，二是评估时间明显变长。

常见坑和对策可以压缩成下面这张表：

| 常见坑 | 具体表现 | 后果 | 对策 |
|---|---|---|---|
| inverse edge 残留 | 只删测试集中的反向边，没检查 train/valid | 验证分数虚高 | 对全部 split 做统一逆关系检查脚本 |
| 只看总三元组数量 | 以为样本多就评估严格 | 忽略关系分布偏差 | 同时检查关系种类、长尾分布、实体覆盖率 |
| rank 计算成本过高 | 全实体打分太慢 | 验证周期过长，实验效率低 | 分批排序、向量化打分、必要时近似候选筛选 |
| 忽略 filtered setting | 把其他真三元组当负例 | 指标被错误拉低 | 使用 filtered MRR/HITS |
| checkpoint 选错指标 | 按 loss 保存最优模型 | 线上泛化不佳 | 以验证集 MRR/HITS 为主 |
| 混用不同 benchmark 结果 | 把 WN18 与 WN18RR 分数直接比较 | 结论失真 | 明确写清数据集版本和评测协议 |

一个很典型的失败案例是：某团队在 WN18RR 上训练时，只删除了“测试集里能直接找到训练逆边”的部分样本，但验证集没有同步处理。结果验证集仍包含大量可逆模式，MRR 看起来很好，换到真正无泄漏的私有集上却显著下降。问题不在模型结构，而在评估协议没有彻底对齐。

因此，实践建议很明确：

1. 不要只信数据集名字，自己检查 split。
2. 不要只报一个指标，至少同时报 MRR、HITS@1、HITS@10。
3. 不要只看论文默认脚本，确认它的 filtered/raw 定义。
4. 不要把“学术 benchmark 最优”直接等同于“工业可上线”。

## 替代方案与适用边界

FB15k-237 与 WN18RR 很常用，但不是所有问题都必须用它们。选 benchmark 的核心，不是“哪个最流行”，而是“哪个最接近你的风险点”。

如果你在做基础教学、模型结构调试，原始 WN18 或 FB15k 仍有价值。因为它们更容易跑出高分，适合快速验证代码是否通。但这里要非常明确：这种结果更像开发期 baseline，不能拿来证明模型具有强泛化能力。

如果你要做更严格的语义推理评估，就应继续使用去逆关系版本。若你的目标图谱更大、更接近开放域知识库，还可以考虑更复杂的 Wikidata 类数据集，或者在现有数据上进一步构造更严格的切分规则。

| 数据集类型 | 适用场景 | 优点 | 局限 |
|---|---|---|---|
| WN18 / FB15k | 代码联调、历史 baseline | 容易复现，训练快 | 逆关系泄漏明显，结果偏乐观 |
| WN18RR / FB15k-237 | 标准语义链接预测评估 | 相对更公平，社区使用广 | 仍是学术化子集，不等于真实工业图谱 |
| 更大规模 Wikidata 类数据集 | 接近开放域或工业场景 | 实体更多、结构更复杂 | 训练评估成本高，协议更难统一 |
| 私有业务图谱切分 | 面向真实上线目标 | 最贴近业务问题 | 难与公开结果横向比较 |

一个简单判断规则是：

- 如果你要回答“模型有没有基本可用性”，原始版本可做 baseline。
- 如果你要回答“模型是否真的学到关系语义”，优先 WN18RR/FB15k-237。
- 如果你要回答“模型在我的业务图谱能不能上线”，最终仍要回到你自己的数据切分。

也就是说，baseline、论文评测、生产验证，这三层边界不能混。
