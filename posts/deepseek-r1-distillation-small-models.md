## 核心结论

DeepSeek-R1 的蒸馏策略，本质上是把大模型已经学会的长链推理过程，直接当作小模型的监督学习目标。蒸馏（distillation，白话说就是“让小模型模仿大模型的输出习惯”）这里不是传统分类任务里的软标签迁移，而是把完整的“问题 + 推理轨迹 + 最终答案”作为文本序列交给学生模型做 SFT。

这件事的关键结论有三个。

第一，DeepSeek-R1 并不是把“答案”蒸馏给小模型，而是把“如何一步一步得到答案”的文本轨迹蒸馏给小模型。这样学生模型学到的不是最终结论，而是中间推理模式。

第二，公开结果说明，只做 SFT 就能显著转移推理能力。以 R1-Distill-Qwen-32B 为代表的蒸馏模型，在 AIME2024、MATH-500 这类强调多步推理的基准上取得了很强的成绩，其中 AIME2024 达到 72.6%，高于不少同级别或闭源小型推理模型。这说明“长思维链”不是只有超大模型才能保留的能力。

第三，这条路线的工程价值非常直接。671B 级教师模型适合做能力来源，不适合大规模在线部署；1.5B、7B、14B、32B、70B 这类蒸馏学生更容易部署、成本更低、延迟更可控，更接近真实产品环境。

下面这个表格可以先把结论钉住：

| 模型 | 参数规模 | 训练方式 | AIME2024 | MATH-500 | 工程意义 |
|---|---:|---|---:|---:|---|
| DeepSeek-R1 | 671B 级 | RL + 冷启动 + 后续蒸馏数据生成 | 很强 | 很强 | 教师模型，主要负责产出高质量推理轨迹 |
| R1-Distill-Qwen-1.5B | 1.5B | SFT 蒸馏 | 明显提升 | 明显提升 | 极低部署成本，适合边缘试验 |
| R1-Distill-Qwen-7B/14B | 7B/14B | SFT 蒸馏 | 持续提升 | 持续提升 | 成本与效果更平衡 |
| R1-Distill-Qwen-32B | 32B | SFT 蒸馏 | 72.6% | 高水平 | 小模型推理落地的代表点 |
| R1-Distill-Llama-70B | 70B | SFT 蒸馏 | 更强 | 更强 | 高性能部署选项 |

如果只看一句话，核心就是：DeepSeek-R1 证明了高质量长链推理可以通过大规模教师轨迹 SFT，稳定迁移到更小、更便宜、更容易部署的模型上。

## 问题定义与边界

这里要先把问题说清楚。所谓“小模型推理”，不是说小模型突然自己发现了新的数学定理，而是说它在训练时看到了大量高质量推理示范，因此在推理时能更稳定地生成“先拆解、再计算、再校验”的输出过程。

问题定义可以写成：

给定输入问题 $x$，教师模型生成包含中间推理链和最终答案的输出 $y$，目标是训练学生模型 $p_\theta(y \mid x)$，使它在类似任务上也能生成接近教师风格的完整推理序列。

这里有三个边界必须明确。

第一，这不是“让 32B 变成 671B”。参数规模、记忆容量、知识上限仍然不同。蒸馏后的学生模型更像是学会了某类做题方法，而不是完整复制教师的全部能力。

第二，这里的收益主要集中在“有明确步骤结构”的任务，比如数学题、逻辑题、代码题、规则推断题。对于开放式创作、长期规划、多工具交互等任务，蒸馏带来的提升不一定同样明显。

第三，蒸馏数据的组成决定能力边界。公开复盘里常提到的数据配比是约 80 万条，其中约 60 万条是推理样本，约 20 万条是写作、问答、翻译等非推理样本。加入非推理样本的目的，不是增强推理，而是避免学生模型只会“写很长的思考过程”，却在普通任务上退化。

可以把这个边界理解成下面这张表：

| 数据类型 | 规模 | 来源方式 | 作用 | 风险 |
|---|---:|---|---|---|
| 推理样本 | 约 600k | 教师模型生成后再过滤 | 迁移长链推理模式 | 容易让模型过度偏向“长回答” |
| 非推理样本 | 约 200k | 通用写作/QA/翻译等数据 | 保持通用语言能力 | 占比太高会稀释推理信号 |
| 拒绝采样后保留样本 | 总计约 800k | 质量过滤后的结果 | 提高监督信号纯度 | 过滤规则差会留下坏轨迹 |

玩具例子可以帮助理解边界。

题目：小明有 3 个苹果，又买了 2 个，一共有多少个？

如果只蒸馏答案，学生只会学到“5”。

如果蒸馏完整轨迹，学生会学到：
1. 初始数量是 3。
2. 新增数量是 2。
3. 总数是 $3 + 2 = 5$。
4. 答案是 5。

对于这种简单题，二者差别不大。但在多步题里，是否保留过程会直接决定学生模型能不能稳定走完整条路径。

真实工程例子则完全不同。比如一个在线教育平台要批改高中竞赛题。部署 671B 教师模型通常意味着极高的显存、吞吐和成本压力；部署 32B 蒸馏模型，则可能在较低硬件预算下仍保留较强的分步推理能力。这时，问题的本质不是“谁最强”，而是“谁在可接受成本内足够强”。

## 核心机制与推导

SFT（监督微调，白话说就是“拿标准答案逐 token 教模型写下去”）在这里的目标函数很直接：

$$
\mathcal{L}_{\mathrm{SFT}}(\theta)=-
\mathbb{E}_{(x,y)\sim \mathcal{D}}
\left[
\sum_{t=1}^{T}
\log p_\theta(y_t \mid x, y_{<t})
\right]
$$

其中：

- $x$ 是输入问题；
- $y=(y_1,\dots,y_T)$ 是教师生成的完整输出；
- $y_t$ 是第 $t$ 个 token；
- $\mathcal{D}$ 是蒸馏数据集。

公式的含义并不复杂：教师已经把整段推理写好了，学生模型只需要在每个位置预测“下一个 token 应该是什么”，把整条轨迹学下来。

为什么这种方式能迁移推理能力？关键在于推理能力在语言模型里并不只表现为“最后那个答案 token”，而是表现为一连串中间状态的语言展开。只要这些中间状态是高质量、稳定、可重复的，学生模型就可能学会相近的生成路径。

可以把它理解成条件概率链：

$$
p_\theta(y\mid x)=\prod_{t=1}^{T} p_\theta(y_t\mid x,y_{<t})
$$

只要教师轨迹足够好，学生在训练中就不断看到：

- 问题出现后通常先做哪类拆解；
- 遇到方程时先列式还是先代数变换；
- 何时进行自检；
- 何时给出最终答案。

这也是为什么“推理轨迹质量”比“单纯题目数量”更重要。80 万条乱写的 CoT 价值不高；80 万条经过拒绝采样和一致性过滤的轨迹，价值很高。

这里再给一个玩具推导例子。

假设一道两步题的教师轨迹是：

- 输入：`2x + 3 = 7`
- 轨迹：`先移项得到 2x = 4，再两边除以 2，得到 x = 2`
- 输出：`答案是 2`

如果学生只学最后一句“答案是 2”，它面对 `3x+6=15` 时未必会做。
如果学生学的是完整轨迹，它更可能抽象出一个模板：

1. 移项；
2. 化简；
3. 除系数；
4. 写出答案。

这就是所谓“长链推理模式”可以被蒸馏的原因。

但要注意，这不是严格意义上的符号推理复制。学生模型并没有显式证明系统，它只是把教师轨迹压缩进参数里，近似学习一种高频、有效的文本推理策略。因此它会继承教师的优势，也会继承教师轨迹中的噪声、冗余和风格偏差。

## 代码实现

工程上最常见的做法是把蒸馏样本整理成统一文本格式，然后用标准 SFT 训练器做微调。LoRA（低秩适配，白话说就是“只训练少量附加参数，减少显存占用”）通常会一起使用，因为全量训练 7B、14B、32B 都比较贵。

下面先给一个最小可运行的玩具 Python 例子。它不训练真正的大模型，只演示“如何把教师轨迹整理成学生训练样本”，以及为什么要保留完整推理链。

```python
from dataclasses import dataclass

@dataclass
class DistillSample:
    question: str
    reasoning: str
    answer: str

    def to_text(self) -> str:
        return (
            f"Question: {self.question}\n"
            f"<think>\n{self.reasoning}\n</think>\n"
            f"Answer: {self.answer}"
        )

sample = DistillSample(
    question="解方程 2x + 3 = 7",
    reasoning="先移项得到 2x = 4，再两边除以 2，得到 x = 2。",
    answer="x = 2"
)

text = sample.to_text()

assert "Question:" in text
assert "<think>" in text
assert "2x = 4" in text
assert text.strip().endswith("x = 2")

# 对比：只保留答案会丢失推理轨迹
answer_only = f"Question: {sample.question}\nAnswer: {sample.answer}"
assert len(text) > len(answer_only)
```

真实训练时，通常会把每条样本做成类似下面的结构：

```python
from trl import SFTTrainer, SFTConfig

trainer = SFTTrainer(
    model=model,
    train_dataset=dataset,
    args=SFTConfig(
        per_device_train_batch_size=4,
        gradient_accumulation_steps=4,
        learning_rate=2e-5,
        bf16=True,
        gradient_checkpointing=True,
        output_dir="DeepSeek-R1-Distill-Qwen-Finetuned",
        max_seq_length=32768,
    ),
)
trainer.train()
```

这段代码背后的工程点主要有四个：

| 配置项 | 作用 | 为什么重要 |
|---|---|---|
| `max_seq_length` | 限制最长上下文长度 | 推理轨迹很长，太短会被截断 |
| `gradient_checkpointing` | 以算力换显存 | 长序列训练几乎必需 |
| `bf16` | 降低显存压力 | 训练更稳，吞吐更高 |
| `LoRA/PEFT` | 降低可训练参数量 | 让中等硬件也能做蒸馏微调 |

真实工程例子可以这样理解。假设你有一个教育问答服务，题目和参考解析都比较长。你把教师生成的 `question -> think -> answer` 数据清洗后做成训练集，用 7B 或 14B 的底座做 LoRA SFT。上线时模型不需要再访问教师模型，也不需要再跑 RL，只靠一份权重就能输出比较像教师的推理风格。这就是蒸馏路线最直接的价值。

## 工程权衡与常见坑

第一类坑是数据坑。

教师模型会生成很多“看起来像推理、实际上质量一般”的长文本。拒绝采样（rejection sampling，白话说就是“先多生成，再筛掉差的”）因此很关键。如果不筛，学生会学到错误步骤、重复表达、语言混杂，最后表现成“想得很长，但不一定做对”。

第二类坑是长度坑。

如果教师轨迹按 32k 上下文准备，但训练或部署时只支持 8k，常见结果不是简单少一点信息，而是会把 `<think>` 区域截断在中间。模型看到残缺的推理模板后，往往会输出重复、跳步或突然结束。这种错误在数学题和代码题里尤其明显。

第三类坑是分布坑。

如果 80 万条数据几乎全是数学和逻辑题，学生模型在这些任务上会很强，但在普通问答、翻译、摘要上可能变得不自然。加入 20 万条非推理数据，就是为了控制这种分布偏移。换句话说，蒸馏不是只追求“更会想”，还要避免“只会一种想法”。

第四类坑是风格坑。

长链推理并不等于无限啰嗦。教师如果偏好超长回答，学生也会继承这个习惯。对于真实产品，太长的推理会增加延迟、token 成本和用户等待时间。因此上线时往往还要配合提示词、解码策略或后续微调来控制输出长度。

下面这个表格可以直接用来判断常见问题：

| 问题 | 表现 | 原因 | 处理方式 |
|---|---|---|---|
| 推理很长但正确率不高 | 看起来会想，结果常错 | 蒸馏轨迹质量不够 | 强化过滤，优先保留高正确率轨迹 |
| 普通问答退化 | 非推理任务变差 | 数据过度偏向 reasoning | 混入通用样本重新平衡 |
| 上线后频繁截断 | 输出停在中间或重复 | 上下文长度不一致 | 保证训练、推理都支持目标长度 |
| 延迟太高 | 单次响应过慢 | CoT 太长、模型太大 | 控制输出长度，或换更小学生模型 |

真实工程里最典型的场景，是团队只看 benchmark，不看部署条件。比如 32B 模型在评测上很好，但如果线上显存预算、并发需求、延迟 SLA 都很紧，那么 14B 甚至 7B 可能是更合理的选择。蒸馏路线的目标不是单纯追分，而是在成本、时延、效果之间找到能上线的平衡点。

## 替代方案与适用边界

蒸馏不是唯一方案，只是当前性价比很高的一条。

第一种替代方案是直接 RL 训练。RL（强化学习，白话说就是“根据结果好坏反复试错学习策略”）能让模型自己探索推理路径，代表性路线就是 R1-Zero 一类方法。它的优点是可能挖出更强的 emergent reasoning，缺点是训练极贵、过程不稳定、复现门槛高，而且最后还要考虑部署问题。

第二种方案是纯普通 SFT，不蒸馏长链，只用标准指令数据。这种方式最便宜、最稳，但通常无法获得同等级的复杂推理提升。

第三种方案是蒸馏后再做额外优化，比如偏好优化、分支融合、多轮反思数据补充。这类方法适合在已有蒸馏模型基础上继续提高鲁棒性，尤其适合担心分布漂移的场景。

可以把三条路线直接并排看：

| 方案 | 成本 | 推理能力上限 | 稳定性 | 部署友好度 | 适用场景 |
|---|---:|---:|---:|---:|---|
| 直接 RL 直训 | 高 | 高 | 较低 | 差 | 研究新推理能力 |
| 纯 SFT 蒸馏 | 中 | 高 | 高 | 好 | 产品化小模型推理 |
| SFT 蒸馏 + 后续优化 | 中高 | 更高 | 中高 | 较好 | 对鲁棒性要求更高的服务 |
| 普通指令 SFT | 低 | 中 | 高 | 很好 | 通用助手，不强调复杂推理 |

适用边界也要说清楚。

如果目标是研究“模型能否自己发现新的推理策略”，蒸馏不是首选，因为它主要在复制教师已经存在的能力。

如果目标是把强推理能力压缩进更小模型，蒸馏非常合适，因为它跳过了高成本 RL 探索阶段，直接利用教师现成的高质量轨迹。

如果目标是强交互、多轮工具调用、长期规划，单次 CoT 蒸馏未必够，还需要工具使用数据、环境反馈数据和更复杂的训练闭环。

所以 DeepSeek-R1 蒸馏路线最准确的定位不是“万能小模型方案”，而是“把已验证有效的长链推理，以较低成本复制到可部署学生模型上的工程方案”。
