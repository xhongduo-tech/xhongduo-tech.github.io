## 核心结论

LoRA 是一种参数高效微调方法。参数高效的意思是：不改整套大模型参数，只训练一小部分新增参数。它的基本形式是冻结原始权重 $W_0$，只学习一个低秩增量 $\Delta W$，最终权重写成 $W = W_0 + \Delta W$。

`rank r` 决定 LoRA 的容量。容量的意思是：这组增量最多能表达多少种有效更新方向。`alpha` 决定缩放强度。缩放强度的意思是：这组增量在前向计算里被放大到什么程度。两者不是独立的，因为常见实现里会用 $s=\alpha/r$ 作为缩放系数。

新手可以先记住一句话：`rank` 管“能学多少”，`alpha` 管“每次改多猛”。这也是为什么 `r=8, alpha=16` 和 `r=32, alpha=16` 不能只看 `r` 大小判断优劣。后者参数更多，但如果缩放还是 $\alpha/r$，单个方向的平均更新强度反而更小。

| 参数 | 主要作用 | 直接影响 | 常见误区 |
| --- | --- | --- | --- |
| `rank r` | 决定低秩增量容量 | 可训练参数量、表达能力、显存开销 | 认为越大一定越好 |
| `alpha` | 决定前向缩放强度 | 更新信号大小、训练稳定性 | 认为越大效果一定更强 |
| `alpha / r` | 决定默认缩放比例 | 不同 `r` 实验的可比性 | 改 `r` 时忘了它也在变 |

经验上，很多任务会从 `r=8/16/32` 起步，`alpha` 常设成 `r` 或 `2r`。这不是定律，只是一个方便的起点。真正的选择标准仍然是任务复杂度、数据量、目标模块和训练稳定性。

## 问题定义与边界

本文只回答一个问题：LoRA 里 `rank` 和 `alpha` 应该怎么选。这里的“怎么选”不是给一个万能数字，而是解释这两个参数分别控制什么、为什么会互相耦合、在工程里该怎么比较。

LoRA 的目标不是从零训练模型，而是在已有大模型上做低成本适配。适配的意思是：让通用模型更符合某个领域、任务或输出风格。比如把通用语言模型改成更会写客服摘要、更会生成 SQL、或者更懂某个垂类术语。

一个直观场景是客服摘要。原模型已经会总结文本，但不懂公司内部标签体系。你不需要重训全部权重，只需要在原模型上做局部修补。LoRA 就是在做这个“局部修补”，而 `rank` 和 `alpha` 分别决定修补的容量和力度。

本文不展开以下问题：

| 场景 | 是否是本文重点 | 原因 |
| --- | --- | --- |
| LoRA 与全参微调孰优 | 否 | 这是方案选择问题，不是参数选择问题 |
| QLoRA、量化训练 | 否 | 会影响显存与数值稳定性，但不是本文主线 |
| Adapter、Prefix Tuning 细节 | 否 | 属于替代方案比较 |
| `target_modules` 如何精细选择 | 部分涉及 | 会影响 `r/alpha` 效果，但不是全文核心 |

还需要明确一个边界：如果问题根本不在 LoRA 容量上，那么调大 `rank` 往往不会解决问题。典型情况有三类：

| 问题来源 | 现象 | 调大 `r` 是否有效 |
| --- | --- | --- |
| 数据质量差 | 训练和验证都不稳 | 通常无效 |
| `target_modules` 选错 | 加很多参数也学不到关键能力 | 通常无效 |
| 训练策略不对 | 学习率过大、过拟合、梯度不稳 | 可能掩盖问题，但不是根治 |

术语先统一一下。`W_0` 是冻结的原始权重；冻结的意思是训练时不更新它。$\Delta W$ 是新增的可训练增量。`A` 和 `B` 是 LoRA 分解后的两个小矩阵。`target_modules` 是要插入 LoRA 的模块名，比如注意力层里的 `q_proj`、`v_proj`。

## 核心机制与推导

LoRA 的核心写法是：

$$
W = W_0 + \Delta W
$$

其中，LoRA 不直接学习完整的 $\Delta W \in \mathbb{R}^{m \times n}$，而是把它写成两个低秩矩阵的乘积：

$$
\Delta W = s \cdot BA
$$

这里：

$$
A \in \mathbb{R}^{r \times n}, \quad B \in \mathbb{R}^{m \times r}
$$

常见默认缩放是：

$$
s = \frac{\alpha}{r}
$$

所以可训练参数量不是 $mn$，而是：

$$
r(n+m)
$$

“低秩”可以先理解成“只允许有限个方向的组合变化”。因为 $BA$ 的秩不会超过 $r$，所以当 $r$ 很小时，LoRA 只能在较小的子空间里改动原模型。这就是为什么 `rank` 会限制表达能力。

### 玩具例子

设原始矩阵 $W_0$ 是一个 $2 \times 3$ 的权重矩阵。如果直接训练完整增量 $\Delta W$，需要 6 个参数。现在令 `r=1`，则：

- $A$ 的形状是 $1 \times 3$
- $B$ 的形状是 $2 \times 1$

总可训练参数变成 $1 \times 3 + 2 \times 1 = 5$ 个。这个例子很小，所以压缩不明显，但已经能看出两件事：

1. 参数量来自两个小矩阵，而不是完整矩阵。
2. 因为 `r=1`，更新方向非常受限，本质上只能围绕一个主方向展开。

再给一个具体数值。设：

$$
A = [1, 2, 3], \quad
B = \begin{bmatrix}2 \\ 4\end{bmatrix}, \quad
\alpha=2, \quad r=1
$$

则缩放系数 $s=2$，并且

$$
BA =
\begin{bmatrix}
2 & 4 & 6 \\
4 & 8 & 12
\end{bmatrix}
$$

因此：

$$
\Delta W = 2 \cdot BA =
\begin{bmatrix}
4 & 8 & 12 \\
8 & 16 & 24
\end{bmatrix}
$$

这个例子说明得很直接：`rank` 决定能造出什么形状的更新，`alpha` 决定这个更新最终以多大强度加到原模型上。

### 为什么 `r` 和 `alpha` 会耦合

如果缩放固定写成 $\alpha/r$，那么只改 `r` 而不改 `alpha`，前向更新强度就会变化。

例如：

- 配置 A：`r=8, alpha=16`，缩放 $s=2$
- 配置 B：`r=32, alpha=16`，缩放 $s=0.5$

这时 B 虽然有更大的容量，但单个方向的整体缩放更小。你最后观测到的效果差异，既来自容量变化，也来自缩放变化。这个比较不是干净实验。

| 变化 | 直接结果 | 间接风险 |
| --- | --- | --- |
| 增大 `r`，固定 `alpha` | 容量变大，$\alpha/r$ 变小 | 实验结论混入缩放因素 |
| 增大 `alpha`，固定 `r` | 缩放变大 | 更新过猛，训练不稳 |
| 同时增大 `r` 和 `alpha` | 容量、缩放一起变化 | 更接近公平对比，但仍要看数据量 |

还有一个工程上会见到的变体是 rsLoRA，它把缩放改成 $\alpha/\sqrt{r}$。这说明缩放策略本身就是设计变量，不是自然常数。默认实现常用 $\alpha/r$，但并不代表它对所有任务都最优。

### 真实工程例子

假设你在做代码审查意见生成，把一个通用因果语言模型适配到公司内部代码库风格。常见做法是只在注意力投影层加 LoRA，比如 `q_proj` 和 `v_proj`。如果你发现：

- `r=8, alpha=16` 能学会格式，但细节判断差；
- `r=32, alpha=64` 在验证集上更稳，能更好识别 review 模式；
- `r=64, alpha=128` 提升很小，但训练时间和显存继续上涨；

那么更合理的结论是：任务确实需要一定容量，但收益已经开始递减。这个结论比“rank 越大越好”准确得多。

## 代码实现

工程里最常见的是通过 PEFT 配置 LoRA。下面先看一个最小可运行例子，用纯 Python 复现 LoRA 的形状、参数量和缩放关系。

```python
import numpy as np

def lora_delta(A, B, alpha):
    r = A.shape[0]
    scale = alpha / r
    return scale * (B @ A)

# 玩具例子：W0 是 2x3，r=1
A = np.array([[1.0, 2.0, 3.0]])          # shape: (1, 3)
B = np.array([[2.0], [4.0]])             # shape: (2, 1)

delta = lora_delta(A, B, alpha=2.0)

expected = np.array([
    [4.0, 8.0, 12.0],
    [8.0, 16.0, 24.0],
])

assert A.shape == (1, 3)
assert B.shape == (2, 1)
assert delta.shape == (2, 3)
assert np.allclose(delta, expected)

def lora_param_count(m, n, r):
    return r * (m + n)

assert lora_param_count(2, 3, 1) == 5
assert lora_param_count(4096, 4096, 8) < 4096 * 4096
```

上面的代码说明三件事：

1. LoRA 的增量确实是 `B @ A` 再乘缩放。
2. 参数量由 `r(m+n)` 决定。
3. 同一个大矩阵下，`r` 小时参数节省非常明显。

再看 PEFT 里的最小配置：

```python
from peft import LoraConfig

config = LoraConfig(
    r=8,
    lora_alpha=16,
    target_modules=["q_proj", "v_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)
```

这里每个字段的作用可以直接对照理解：

| 配置项 | 作用 | 对 `r/alpha` 选择的影响 |
| --- | --- | --- |
| `r` | 低秩维度 | 决定容量与参数量 |
| `lora_alpha` | 缩放因子 | 决定前向更新强度 |
| `target_modules` | LoRA 插入位置 | 决定容量是否用在关键部位 |
| `lora_dropout` | LoRA 分支 dropout | 影响泛化与稳定性 |

如果你要做小规模网格搜索，建议写成显式组合，而不是手工改参数：

```python
from peft import LoraConfig

def build_configs():
    configs = []
    for r in [8, 16, 32]:
        for alpha in [r, 2 * r]:
            configs.append(
                LoraConfig(
                    r=r,
                    lora_alpha=alpha,
                    target_modules=["q_proj", "v_proj"],
                    lora_dropout=0.05,
                    bias="none",
                    task_type="CAUSAL_LM",
                )
            )
    return configs

configs = build_configs()
assert len(configs) == 6
assert configs[0].r in [8, 16, 32]
assert configs[0].lora_alpha in [configs[0].r, 2 * configs[0].r]
```

一个实用原则是：先固定 `target_modules`，再比较 `r/alpha`。否则你同时改变模块覆盖范围和 LoRA 容量，实验很难解释。

## 工程权衡与常见坑

`rank` 不是越大越好。它提高的是可表达更新空间，但很多任务在较小 `r` 下就已经接近饱和。继续增大 `r`，收益往往递减，代价却线性增加。

`alpha` 也不是越大越好。它更像音量，不是音质。音量太小，模型几乎听不见 LoRA 分支；音量太大，更新容易过猛，训练曲线抖动，甚至过拟合。

最常见的坑有下面几类：

| 常见坑 | 实际问题 | 处理建议 |
| --- | --- | --- |
| 只改 `r` 不改 `alpha` | $\alpha/r$ 跟着变，比较失真 | 保持缩放策略一致 |
| 把 `alpha` 当成容量参数 | 误以为大 `alpha` 等于更强表达力 | 区分“容量”和“强度” |
| 任务复杂却只加少数层 | 容量没落到关键模块 | 先检查 `target_modules` |
| 数据很少却上大 `r` | 容易过拟合，成本偏高 | 先试小 `r` |
| 拿经验值当规则 | `alpha=2r` 被误当成定律 | 视任务和实现调整 |

工程上更稳的排查顺序通常是：

1. 先看 `target_modules` 是否合理。
2. 再看数据规模和任务复杂度是否支持更大 `r`。
3. 最后再用 `alpha` 调整更新强度和稳定性。

一个真实坑例是：有人把 `r` 从 8 提到 32，但 `alpha` 保持 16 不变，结果发现效果没提升，于是得出“高 rank 没用”的结论。这个结论不可靠，因为缩放从 2 降到了 0.5，训练信号已经不是同一量级。更干净的对比应该至少把 `alpha` 随 `r` 一起调整，比如比较 `8/16`、`16/32`、`32/64`。

还有一个容易被忽略的点是初始化。初始化的意思是训练刚开始时参数被设成什么值。LoRA 常见做法会让某个分支初始近似 no-op，也就是一开始几乎不改变原模型输出。这时 `alpha` 的影响不是“训练第一步就完全显现”，而是随着参数逐渐学出来后才体现为前向缩放差异。

## 替代方案与适用边界

LoRA 适合“已有大模型 + 想低成本适配”的场景，但不是唯一方案。选 `r/alpha` 之前，先确认 LoRA 本身是不是合适。

| 方案 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| LoRA | 参数少、显存省、部署灵活 | 容量受限，模块选择敏感 | 大模型领域适配、资源受限 |
| 全参微调 | 表达能力最强 | 显存和存储成本高 | 数据足、预算足、任务差异大 |
| Adapter | 结构清晰、模块化强 | 额外推理开销可能更明显 | 需要多任务切换 |
| Prefix / Prompt Tuning | 参数更少 | 对任务与模型形式更敏感 | 极低成本尝试 |

什么时候优先用小 `r`？

| 条件 | 更合理的起点 |
| --- | --- |
| 数据少 | `r=8` 或 `16` |
| 任务简单 | `r=8` 起步 |
| 只是风格迁移或术语适配 | 小 `r` 往往够用 |
| 显存紧张 | 先压低 `r`，固定关键模块 |

什么时候考虑更大 `r`？

| 条件 | 可能动作 |
| --- | --- |
| 任务复杂、跨域差异大 | 试 `r=32` 或 `64` |
| 小 `r` 明显欠拟合 | 增大 `r` 或扩大模块覆盖 |
| 模型需要学更细的新模式 | 提高容量，但同时看验证集收益 |

这里有一个边界非常重要：如果瓶颈不在容量，而在数据、模块或训练策略，那么继续增大 `r` 往往是低效的。例如医疗摘要任务效果差，原因可能不是 `r=8` 太小，而是你只给 `q_proj/v_proj` 加了 LoRA，却没有覆盖更关键的投影层；也可能是标注摘要本身不一致，模型学不到稳定目标。

所以更实用的决策逻辑不是“先把 `r` 拉大”，而是：

1. 任务简单、小样本：先试小 `r`。
2. 任务复杂、域差明显：先确认模块覆盖，再增大 `r`。
3. 训练不稳：先检查学习率、初始化和 `alpha`，不要只怪 `r`。
4. 大 `r` 提升很小：说明收益进入递减区，应该回头看数据与模块设计。
