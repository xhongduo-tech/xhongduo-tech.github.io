## 核心结论

视觉语言模型，简称 VLM，就是“同一套模型同时看图和读指令”的模型。它在 OCR 场景里的变化，不是把识字模块做得更大，而是把“检测文字、识别字符、理解版面、回答问题、输出结构”放进一个端到端系统里统一完成。

这就是近两年常说的 OCR-2.0。传统 OCR 更像一条流水线：先找文字框，再识别字符，再交给下游规则或语言模型处理。Qwen2.5-VL 和 GOT-OCR 2.0 代表的新范式，则是直接输入原始文档图像，让模型同时完成文本提取、结构理解与任务输出。输出不再局限于纯文本，还可以是 Markdown、LaTeX、表格结构、区域坐标，甚至直接回答 DocVQA、InfographicVQA、ChartQA 这类问题。

这里的关键变化有三点：

| 维度 | 传统 OCR 流水线 | OCR-2.0 / VLM |
| --- | --- | --- |
| 输入 | 文档图像，通常先缩放和切块 | 原始文档图像，可保留更多布局信息 |
| 中间表示 | 检测框、行块、字符序列 | 视觉 token 与文本 token 的统一表示 |
| 输出 | 纯文本或字符框 | 文本、表格、公式、区域定位、问答结果 |

对初学者，最重要的结论是：文档理解不是“把图片里的字抄出来”这么简单。真实任务往往要求模型同时完成三件事：读字、懂结构、按问题返回结果。信息图里“哪个部门占比最大”，财报图表里“哪个季度营收最高”，合同页里“违约金条款在哪一段”，这些任务都不是单纯 OCR，而是 OCR 与结构理解的组合。

## 问题定义与边界

先把问题讲清楚。文档理解不是一个单一任务，而是一类任务集合。它至少包括以下三层：

| 层级 | 含义 | 典型输出 |
| --- | --- | --- |
| 字符级识别 | 把图中的字读出来 | 文本串 |
| 结构级理解 | 识别段落、表格、图表、公式的组织关系 | Markdown、HTML、LaTeX、坐标 |
| 任务级推理 | 根据问题返回答案 | 自然语言答案、字段值、定位结果 |

DocVQA 是“文档视觉问答”，白话说就是“给你一页文档图片，再给你一个问题，让模型直接回答”。InfographicVQA 面向信息图，难点是颜色、图标、文本框、装饰元素混在一起。ChartQA 面向图表，模型不仅要识别坐标轴和标签，还要理解数值关系。

因此，这类任务的输入与输出可以写成一个简化形式：

$$
\text{Task} = \{\text{Image},\ \text{Prompt},\ \text{Layout},\ \text{Answer}\}
$$

如果进一步抽象，模型真正要学的是一个条件映射：

$$
P(\text{Answer} \mid \text{Image},\ \text{Prompt})
$$

这里的 `Layout` 不一定显式作为输入字段出现，但它必须被模型在内部表示出来。因为很多答案只靠 OCR 文本本身是拿不到的。

一个玩具例子可以说明这个边界。

假设一张信息图上有三个色块：

- 蓝色块写着“研发 45%”
- 绿色块写着“市场 30%”
- 灰色块写着“行政 25%”

问题是：“哪个部门占比最大？”

如果只做 OCR，你得到的可能只是三段文本：`研发 45%`、`市场 30%`、`行政 25%`。这时还需要额外逻辑把“部门”和“比例”配对，再做比较。VLM 的目标则是直接在统一表示里完成“读取文本 + 绑定关系 + 选择最大值”，输出“研发”。

所以边界很明确：

- 如果任务只是把扫描件尽量准确转成纯文本，传统 OCR 仍然有效。
- 如果任务要求理解表格列关系、图表趋势、公式结构、跨区域引用，那么只做 OCR 往往不够。
- 如果任务要求“先看图，再回答问题”，就已经进入文档理解而不是单纯字符识别。

真实工程里，很多失败案例都来自边界判断错误。团队以为自己在做 OCR，实际上需求是字段抽取或文档问答；结果系统前半段输出一堆文本，后半段却因为丢失布局关系而无法稳定工作。

## 核心机制与推导

要理解这类模型为什么有效，先看它怎么“看图”。

视觉模型会把图片切成很多小块，通常叫 patch，也就是“固定大小的小图块”。白话说，就是先把大图切成网格，再把每个网格编码成 token。Qwen2.5-VL 文档场景常提到的一个核心点，是动态分辨率。它不是强行把所有图片缩成同一个尺寸，而是尽量保留原始长宽比，再按 patch 切分。

若 patch 大小为 $14 \times 14$，则一张高为 $H$、宽为 $W$ 的图，视觉 token 数量近似为：

$$
N(H,W)=\left\lfloor \frac{H}{14} \right\rfloor \cdot \left\lfloor \frac{W}{14} \right\rfloor
$$

这意味着，图像越大、细节越多，token 数量也越多。好处是表格线、上标下标、公式分数线这类细节不会因为暴力缩放而消失；代价是计算量上升。

用一个最小数值例子说明。若输入页面尺寸为 $1280 \times 720$，则：

$$
N(1280,720)=\left\lfloor \frac{1280}{14} \right\rfloor \cdot \left\lfloor \frac{720}{14} \right\rfloor
= 91 \times 51 = 4641
$$

这 4641 个视觉 token 会与文本 prompt 一起送入模型。模型随后在统一序列里完成跨模态建模，也就是“图像 token 和文字 token 一起参与推理”。

为什么这对文档特别重要？因为文档里的语义高度依赖空间关系。下面几个例子都说明这一点：

| 文档元素 | 仅靠文本序列为什么不够 | 需要的视觉结构 |
| --- | --- | --- |
| 表格 | 同一行、同一列关系可能丢失 | 行列边界、单元格对齐 |
| 公式 | 上下标、分式、根号位置决定含义 | 二维排版关系 |
| 图表 | 标签、刻度、柱线高度共同决定答案 | 坐标轴与图元位置 |
| 多栏版面 | 阅读顺序不等于扫描顺序 | 区域布局与块结构 |

GOT-OCR 2.0 的思路更偏“专用 OCR-2.0 模型”。它把高分辨率视觉编码与长文本解码结合起来，重点优化的是：长页、多区域、复杂格式下，如何稳定输出结构化结果。你可以把它理解为“不是只回答一句话，而是更擅长把整页内容还原成一种可用格式”。

这里还有一个范式差异值得强调。传统 OCR 的链路常常是：

1. 检测文字区域
2. 逐块识别字符
3. 后处理拼行
4. 单独做表格恢复
5. 再接规则或 LLM 做问答

这种做法每一步都可能引入误差，而且误差会向后传递。比如表头识别错了一个字，下游字段抽取就可能全错；表格行列切错了，语言模型读到的文本顺序就不再对应原始结构。端到端 VLM 的价值在于，它直接优化最终任务目标，减少了多模块拼装中的语义错位。

一个真实工程例子是财报审核。财报页面里通常同时有：

- 正文段落
- 合并报表
- 柱状图或折线图
- 注释脚注
- 数学公式或记账符号

如果系统只是把每块文字识别出来，再交给下游做分析，常见问题是“正文引用的表格项”和“图表中的简称字段”无法稳定关联。统一模型则可以根据 prompt 直接输出类似“请提取营业收入表，并保持行列结构”的结果，或者回答“2024 年第四季度营收是否高于第三季度”。这比“先 OCR，再正则，再人工修补”的链路更接近工程可维护方案。

## 代码实现

下面给一个可以直接运行的 Python 玩具实现。它不调用大模型，只模拟文档理解里的一个关键点：在保留结构的前提下，根据问题从“区域化信息”中取答案。目的不是复现 VLM，而是让初学者先理解“为什么不能只存一坨文本”。

```python
from dataclasses import dataclass

@dataclass
class Region:
    kind: str
    text: str
    value: float | None = None

def answer_largest_department(regions):
    candidates = []
    for r in regions:
        if r.kind == "legend_item" and r.value is not None:
            candidates.append((r.text, r.value))
    assert candidates, "no candidates found"
    return max(candidates, key=lambda x: x[1])[0]

def count_patches(height, width, patch=14):
    return (height // patch) * (width // patch)

# 玩具例子：信息图中的三个区域
regions = [
    Region(kind="legend_item", text="研发", value=45.0),
    Region(kind="legend_item", text="市场", value=30.0),
    Region(kind="legend_item", text="行政", value=25.0),
]

assert answer_largest_department(regions) == "研发"

# 文档页 token 数量估算
assert count_patches(1280, 720) == 4641
assert count_patches(224, 224) == 256

print("largest department:", answer_largest_department(regions))
print("tokens@1280x720:", count_patches(1280, 720))
print("tokens@224x224:", count_patches(224, 224))
```

这段代码表达了两个事实。

第一，回答“哪个部门占比最大”时，模型需要的是“文本 + 区域类型 + 数值绑定”的联合表示，而不是三段无结构字符串。

第二，`1280x720` 和 `224x224` 的 token 规模差异巨大。后者算力更省，但文档细节保真度也更低。很多表格和公式问题，恰恰死在这个缩放步骤上。

如果需要接近真实推理管线，Hugging Face 的用法通常是“处理器 + 模型 + generate”三步。下面给一个面向文档 OCR-2.0 的示意代码，展示统一输入输出接口长什么样：

```python
from transformers import AutoProcessor, AutoModelForImageTextToText
import torch

model_id = "stepfun-ai/GOT-OCR-2.0-hf"

processor = AutoProcessor.from_pretrained(model_id)
model = AutoModelForImageTextToText.from_pretrained(
    model_id,
    torch_dtype=torch.float16,
    device_map="auto"
)

prompt = "请将这页财报转换为 Markdown，并保留表格结构。"
inputs = processor(
    images="report_page.png",
    text=prompt,
    return_tensors="pt"
).to(model.device)

outputs = model.generate(
    **inputs,
    max_new_tokens=1024
)

result = processor.batch_decode(outputs, skip_special_tokens=True)[0]
print(result[:200])
```

实际工程里，Qwen2.5-VL 和 GOT-OCR 2.0 的差别不只在模型名字，而在集成目标：

- 如果你的任务是“看文档后回答问题”，Qwen2.5-VL 这种通用多模态模型通常更灵活。
- 如果你的任务是“把复杂页面稳定转成结构化格式”，GOT-OCR 2.0 这类专用 OCR-2.0 模型往往更直接。

一个真实工程例子可以这样设计。假设你在做发票审核系统，输入是 PDF 截图，目标是抽取：

- 发票号
- 开票日期
- 金额表格
- 税率字段
- 备注中的异常说明

传统方案是 OCR 引擎识字、版面分析器切块、表格模型恢复网格、规则系统抽字段。新方案则可以直接把页面图像和任务提示一起送入 VLM，让模型输出 Markdown 表格或 JSON 草稿，再由后处理做字段校验。前者模块多、可控性高；后者集成简单、泛化更强。

## 工程权衡与常见坑

第一类坑是过度缩放。把所有文档统一压到 `224x224`，在自然图像分类里可能还能工作，但在文档场景往往是灾难。原因很直接：表格线会变细甚至消失，小字号脚注会模糊，公式上下标会粘连，图表刻度会难以区分。你省下了 token，代价是把关键结构信息一并删掉。

第二类坑是把“识字”和“理解”完全拆开。很多团队先跑 OCR，再把全文拼成字符串喂给 LLM，最后问“金额是多少”。这个方案在简单收据上可能有效，但在多栏论文、复杂表格、嵌套标题、带图例图表里很容易失效，因为文本顺序和版面语义不是一回事。

下面这个对照很典型：

| 选择 | 优点 | 缺点 |
| --- | --- | --- |
| 固定小分辨率输入 | 部署简单，吞吐稳定 | 表格、公式、细字丢失严重 |
| 动态分辨率输入 | 保留结构和细节 | token 数变化大，批处理更复杂 |
| 先 OCR 再 QA | 每步可解释，旧系统兼容好 | 版面关系容易断裂 |
| 端到端 VLM | 任务统一、跨结构泛化强 | 调参与显存压力更高 |

第三类坑是把“答案正确”和“可审计”混为一谈。端到端模型能直接回答问题，但在金融、法务、医疗文档里，很多时候不仅要答案，还要证据位置。也就是说，系统最好能返回“答案 + 来源区域”。否则即使答对，也不利于人工复核。

第四类坑是忽略表格与公式的特殊性。表格不是普通段落，公式也不是普通文本。比如：

- 表格里的“100”可能属于“应收账款”行，也可能属于“营业收入”行，必须依赖行列关系。
- 公式里的 $x^2$ 和 $x_2$ 只差一个位置，但语义完全不同。
- 化学式、财务报表、统计图例都对二维布局敏感。

第五类坑是把 benchmark 分数直接当成落地效果。DocVQA、InfographicVQA、ChartQA 很重要，但它们只能说明模型在公开任务上的能力，不等于你的扫描仪噪声、压缩 PDF、水印、印章、低光拍照、旋转页面都能稳定处理。工程落地必须补一层真实数据验证。

## 替代方案与适用边界

不是所有文档任务都要上端到端 VLM。正确做法是按任务复杂度选方案。

如果目标只是把排版简单、模板固定的文档转成文本，例如固定格式的快递面单、票据编号读取、标准表单字段提取，传统 OCR 加规则依然有成本优势。它的优点是吞吐高、资源消耗低、行为更可预测。

如果任务开始涉及复杂版面，尤其是下列情况，端到端 VLM 的优势会变明显：

- 多栏排版
- 表格跨页
- 公式与正文混排
- 图表问答
- 信息图理解
- 需要自然语言提问交互

可以做一个简化选择表：

| 场景 | 推荐方案 | 原因 |
| --- | --- | --- |
| 固定模板字段抽取 | 传统 OCR + 规则 | 成本低，易控 |
| 简单文档问答 | 轻量 VLM | 不必维护复杂规则 |
| 多页财报、合同、审计材料 | GOT-OCR 2.0 类方案 | 更适合长文档与结构化输出 |
| 图表、信息图、复杂视觉问答 | Qwen2.5-VL 类方案 | 通用多模态推理更强 |

还要注意资源边界。动态分辨率本质上是在拿更多 token 换更高保真，因此显存、时延、吞吐都会受影响。对于单卡资源有限的团队，常见做法不是一味追求大模型，而是：

- 先按页面复杂度分流
- 简单页用传统 OCR
- 复杂页才送入 VLM
- 对长文档分页或区域裁切
- 对结果增加规则校验和人工抽检

这比“一套最大模型跑全部请求”更符合生产系统思路。

最后再强调一次适用边界。VLM 解决的是“从图像到结构化信息”的统一建模问题，不是自动消除一切噪声。极端模糊图像、遮挡严重拍照、超长 PDF、非常专业的小众符号体系，仍然可能需要专门数据、领域微调或后处理模块配合。OCR-2.0 不是替代所有传统方法，而是在复杂文档理解场景里，把原来割裂的多阶段链路尽可能合并。
