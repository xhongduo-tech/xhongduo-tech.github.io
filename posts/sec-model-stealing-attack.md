## 核心结论

模型窃取攻击，英文常叫 *model extraction attack*，指攻击者把目标模型当作“只能输入、只能输出的黑盒服务”，通过大量 API 查询收集输入与输出，再训练一个行为接近原模型的替身模型。白话说，就是不断“问题”，把答案抄下来，最后自己做出一个会“模仿答题”的模型。

它的关键不在于拿到原模型参数，而在于拿到**足够多、足够稳定的决策信号**。这个信号可以是硬标签，如“猫”；也可以是概率分布，如 `[0.7, 0.2, 0.1]`；还可以是更细的 logit，即 softmax 之前的原始分数。返回信息越细，攻击者越容易逼近目标模型的决策边界。决策边界可以理解为“模型把不同类别分开的那条分界线”。

一个新手可直接理解的流程如下：

```text
输入样本 x
   ->
调用目标 API
   ->
获得输出标签/概率/logit
   ->
保存为训练样本 (x, y_v)
   ->
训练替身模型 F_a
   ->
评估替身与目标模型的一致率
```

表 1 展示不同输出信号给攻击者带来的信息量差异。

| 攻击输出信号 | 例子 | 信息量 | 对替身训练的帮助 |
|---|---|---:|---|
| 硬标签 | `cat` | 低 | 只能学习最终分类结果 |
| Top-k 概率 | `cat:0.72, dog:0.19` | 中 | 能学习类别间相对接近程度 |
| 全概率分布 | `[0.72,0.19,0.06,0.03]` | 高 | 能更好拟合边界附近样本 |
| Logit | `[5.1,3.8,1.2,-0.5]` | 很高 | 保留最多教师信号，最利于蒸馏 |

工程上，攻击效果通常受三类约束控制：

1. 查询成本：每次 API 调用都要花钱或消耗配额。
2. 返回粒度：只返回标签，比返回完整概率更难被窃取。
3. 访问限制：速率限制、风控、异常检测会压缩可收集样本数。

但要注意一个事实：**只要服务仍然返回可学习的决策信号，攻击通常就不会因为“看不到参数”而失效，只会因为信息不足而变慢。**

## 问题定义与边界

形式化地说，受害者模型记作 $F_v$，攻击者训练的替身模型记作 $F_a$。攻击者无法访问 $F_v$ 的参数、训练集或内部梯度，只能构造输入 $x$ 并观察输出 $F_v(x)$。这就是黑盒，白话说就是“只能问，不能拆开看”。

如果攻击者还能知道一些额外信息，例如模型架构大类、输出维度、标签空间、是否使用温度缩放，通常称为灰盒。灰盒不是拿到了完整内部实现，而是“知道一点背景资料”。黑盒与灰盒的差别，不在有没有源码，而在攻击者可利用的先验信息多少。

一个最简单的例子是二分类图片服务。你上传一张图，API 返回：

- `cat`
- 或 `cat, p=0.91`
- 或 `logits=[3.1, -0.4]`

这三种返回值都能被拿去训练替身，只是训练效率不同。

表 2 给出攻击边界的直观划分。

| 可查询信息 | 平台限制措施 | 对攻击者影响 |
|---|---|---|
| 仅标签 | 每分钟 10 次 | 可做标签蒸馏，但收敛慢 |
| Top-k 概率 | 每分钟 100 次 | 可恢复更多类别关系，攻击明显更强 |
| 全概率分布 | 无严格限流 | 黑盒近似能力强，容易训练高质量替身 |
| Logit | 有费用但无异常检测 | 攻击成本高但监督最强 |
| 输出经噪声扰动 | 有会话级风控 | 一致性下降，替身更难稳定学习 |

这里的边界有两个容易混淆的点。

第一，模型窃取不等于训练数据恢复。训练数据恢复关注“模型记住了哪些原始样本”；模型窃取关注“模型会怎么回答”。前者是数据泄露问题，后者是功能复制问题。

第二，模型窃取不一定追求逐点完全相同。很多真实攻击只要让替身在关键业务分布上达到高一致率，就已经有商业价值。例如广告审核、图像识别、风控打分、LLM 风格模仿，都不要求 100% 逐样本一致。

玩具例子可以这样理解：有一个“猫狗分类 API”，你每次上传图片，它返回“猫”或“狗”，有时还给概率。你拿 1 万张公开图片轮流去问，再把这些问答对拿来训练自己的小模型。最后，你虽然不知道原模型参数，但已经有了一个会做类似判断的替身。这就是最基础的模型窃取。

## 核心机制与推导

模型窃取的核心目标是让替身模型 $F_a$ 在查询到的输入分布上逼近目标模型 $F_v$。最常见写法是最小化下面的损失：

$$
\min_{\theta_a}\; \mathbb{E}_{x \sim \mathcal{Q}} \left[\mathcal{L}\big(F_a(x), F_v(x)\big)\right]
$$

其中：

- $\theta_a$ 是替身模型参数。
- $\mathcal{Q}$ 是攻击者选出来的查询分布，白话说就是“拿什么样的输入去问”。
- $\mathcal{L}$ 常见是交叉熵或 KL 散度。
- KL 散度可以理解为“两个概率分布差多远”。

如果 API 只返回硬标签 $y_v=\arg\max F_v(x)$，那么损失通常退化为普通监督学习：

$$
\mathcal{L}_{hard} = - \sum_{c} \mathbf{1}[c=y_v]\log F_a(x)_c
$$

如果 API 返回软标签，也就是概率分布 $p_v = F_v(x)$，则更常见的是蒸馏损失：

$$
\mathcal{L}_{soft} = - \sum_c p_v(c|x)\log F_a(c|x)
$$

或者写成 KL 散度：

$$
D_{KL}(p_v \| p_a) = \sum_c p_v(c|x)\log \frac{p_v(c|x)}{p_a(c|x)}
$$

为什么软标签更危险？因为它不只告诉你“答案是什么”，还告诉你“模型有多犹豫”。例如：

$$
F_v(x) = [0.7, 0.2, 0.1]
$$

这说明模型虽然判成第 1 类，但第 2 类也有一定可能。这个信息会让替身更快学到边界附近的结构，而不仅是最终类别。

可以把训练质量理解成两个因子的乘积：

$$
\text{攻击效果} \approx \text{查询样本代表性} \times \text{单次返回信息量}
$$

前者取决于攻击者怎么选 $x$，后者取决于平台返回标签、概率还是 logit。

常见输入构造策略如下。

| 输入构造策略 | 白话解释 | 优点 | 风险/代价 |
|---|---|---|---|
| 随机采样 | 随便找样本去问 | 简单，成本低 | 样本效率差 |
| 标签查询蒸馏 | 用标签训练替身 | 对只返标签场景有效 | 需要更多查询 |
| Logit 提取 | 用更细分数训练 | 学得快，边界更准 | 平台很少直接暴露 |
| 主动学习采样 | 专门问“最难的题” | 查询效率高 | 需要先有初始替身 |
| 高价值提示回放 | 重放高收益输入 | 针对 LLM/生成模型有效 | 容易触发风控 |

“主动学习”首次出现时可以这样理解：先用一个初步替身找出最不确定的样本，再优先去问这些样本，因为这些样本更接近决策边界，信息价值更高。

数值玩具例子如下。假设一个三分类 API 返回若干 soft-label，攻击者用 100 条样本训练逻辑回归，验证集上相对目标模型达到 92% 一致率。这个例子说明，哪怕没有原始训练标签，只要 API 返回足够好的教师信号，替身也能收敛。

更直观地看，100 条带概率的样本，往往比 100 条只有类别名的样本更“值钱”。因为每条样本都携带了更多结构信息。

真实工程例子则更常见于高价值 API。比如一个图像审核平台返回 top-5 概率：`normal 0.62, violence 0.21, porn 0.08 ...`。攻击者只要积累足够多的图片与这些概率输出，就能训练出高度相似的审核器。对外部客户来说，这意味着平台的核心识别能力被复制；对平台收入来说，这意味着用户以后可能不再需要持续调用官方 API。

## 代码实现

下面给一个最小可运行的 Python 玩具实现。它不调用真实远程 API，而是用一个“受害者模型”函数来模拟 API 返回 soft-label，再训练一个替身模型去逼近它。逻辑和真实模型窃取是一致的：采样输入，记录输出，用输出监督训练替身。

```python
import math
import random

def softmax(logits):
    m = max(logits)
    exps = [math.exp(v - m) for v in logits]
    s = sum(exps)
    return [v / s for v in exps]

# 模拟受害者 API：输入二维特征，输出三分类概率
def victim_api(x1, x2):
    logits = [
        2.0 * x1 - 1.0 * x2 + 0.3,
        -1.5 * x1 + 1.8 * x2 - 0.2,
        0.5 * x1 + 0.2 * x2 - 0.1,
    ]
    return softmax(logits)

# 生成查询样本
random.seed(7)
X = []
Y = []
for _ in range(400):
    x1 = random.uniform(-2, 2)
    x2 = random.uniform(-2, 2)
    X.append((x1, x2))
    Y.append(victim_api(x1, x2))

# 训练一个线性替身模型：三分类 softmax 回归
W = [[0.0, 0.0] for _ in range(3)]
B = [0.0, 0.0, 0.0]

def predict(x1, x2):
    logits = [W[c][0] * x1 + W[c][1] * x2 + B[c] for c in range(3)]
    return softmax(logits)

lr = 0.1
for _ in range(800):
    for (x1, x2), target in zip(X, Y):
        pred = predict(x1, x2)
        for c in range(3):
            grad = pred[c] - target[c]
            W[c][0] -= lr * grad * x1
            W[c][1] -= lr * grad * x2
            B[c] -= lr * grad

# 评估替身与受害者的一致率
def argmax(v):
    return max(range(len(v)), key=lambda i: v[i])

correct = 0
test_samples = []
for _ in range(200):
    x1 = random.uniform(-2, 2)
    x2 = random.uniform(-2, 2)
    test_samples.append((x1, x2))

for x1, x2 in test_samples:
    y_v = argmax(victim_api(x1, x2))
    y_a = argmax(predict(x1, x2))
    if y_v == y_a:
        correct += 1

acc = correct / len(test_samples)
print("agreement:", round(acc, 4))

assert acc > 0.9
assert abs(sum(predict(0.2, -0.4)) - 1.0) < 1e-6
```

这个例子故意选了一个简单线性受害者模型，所以替身很容易学会。在真实工程里，目标模型更复杂，数据分布也不一定已知，但攻击思路仍然是同一个模板。

## 工程权衡与常见坑

第一类常见误区是：**只返回硬标签就安全了**。这不准确。只返回硬标签确实降低了信息量，但并没有切断监督信号。攻击者仍可通过更大的查询量、主动学习采样、分布外探测来逼近边界。它是“减速器”，不是“绝缘层”。

第二类误区是：**服务条款禁止窃取，所以问题解决了**。这只是法律与合规层面的约束，不是技术防御。脚本化攻击不会因为 TOS 文本而自动失败。

真实工程例子可以这样描述。某公司提供图像分类 API，默认返回 top-5 概率。外部脚本持续抓取公开图片，按类别与难样本混合查询，累计数十万次请求后训练出一个本地模型，在关键验证集上与官方模型高度一致。结果是客户不再持续调用官方 API，而改为离线部署替身，造成授权收入和知识产权泄露。后续缓解措施包括：

1. 每账户速率限制。
2. 从 top-5 概率降到只返回 top-1 标签。
3. 对异常查询分布做检测。
4. 对部分接口注入受控噪声。

防御措施本质上是在三条线上做权衡：可用性、精度、泄露风险。

| 防御措施 | 降低泄露的方式 | 对正常用户的影响 |
|---|---|---|
| 只返回 top-1 标签 | 降低单次信息量 | 用户失去置信度信息 |
| 速率限制 | 压缩样本收集速度 | 高频合法用户受影响 |
| 查询异常检测 | 识别脚本化/边界探测 | 需要日志与误报处理 |
| 输出加噪声 | 降低可蒸馏性 | 预测稳定性下降 |
| 水印/输出指纹 | 便于事后归因 | 不能直接阻止窃取 |

“异常查询检测”可以简单理解为：看某个账户是不是在持续提交不符合正常业务分布的输入。比如大量接近决策边界的图像、批量轻微扰动后的相似样本、短时间高频系统化 prompt。研究里常见的思路是看查询分布是否异常集中、是否带有主动学习痕迹、是否在局部区域密集探边。

“差分隐私噪声”首次出现时可以理解为：在输出里加入经过设计的小扰动，使单个查询更难精确反映模型内部边界。它的目标不是让结果完全随机，而是在“还能用”和“别太容易被复制”之间找平衡。

一个可操作的组合流程通常是：

```text
请求进入
   ->
身份与配额校验
   ->
查询分布异常检测
   ->
决定返回粒度（标签 / top-k / full probs）
   ->
必要时加入受控噪声
   ->
记录日志并更新风控状态
```

这里最大的坑是过度防守。比如对所有用户统一加较大噪声，可能直接让高价值客户失去可用性；或者限流过严，导致实时业务不可接受。工程上不能只追求“最难被偷”，还要保证服务还能卖得出去、用得起来。

## 替代方案与适用边界

如果业务对延迟、交互方式或部署架构有调整空间，可以考虑不把“完整决策信号”直接暴露给公共 API。

表 4 给出几种替代方案。

| 替代方案 | 核心思路 | 适用场景 | 局限 |
|---|---|---|---|
| 白盒验证/可信执行 | 在受控环境运行模型，结果经签名返回 | 高敏感、高价值模型 | 成本高，部署复杂 |
| 合成数据分发 | 先给用户数据或离线能力，不直接暴露核心模型 | 低频、低实时性服务 | 不能覆盖所有真实输入 |
| 认证 token + 分级返回 | 不同身份看到不同输出粒度 | 企业 API、多租户系统 | 权限设计复杂 |
| 缓存与模板回答 | 对重复问题返回缓存或规则结果 | 高频标准化问答 | 对开放任务效果有限 |
| 本地轻量模型 + 云端复核 | 大部分请求本地处理，少数送核心模型 | 成本敏感场景 | 架构更复杂 |

低频服务里，一个很实用的策略是“合成数据 + 认证 token”。白话说，就是不要让所有用户都能无限次直接问主模型，而是先分发一部分可替代的数据、模板或小模型，只有高权限请求才打到核心模型。这样做不能完全消除窃取，但能减少高价值输出暴露次数。

对一些低灵敏度模型，例如简单文本分类、低风险推荐排序，平台甚至可以接受一定程度的功能模仿，因为商业壁垒不只来自模型本身，还来自数据更新、部署稳定性、产品集成和运维能力。这时防守目标不是“完全防偷”，而是“把复制成本抬高到不划算”。

但对审核、风控、医疗、金融定价、专有 agent 策略等高价值模型，适用边界就完全不同。只要 API 返回过于丰富的决策信号，攻击者就可能在可承受预算内做出高保真替身。此时更合理的目标是：

1. 控制信息粒度。
2. 压缩可采样规模。
3. 监控异常模式。
4. 对高价值接口做分级访问。

   核心点：经典黑盒替身模型思路，虽主要讨论对抗样本，但替身训练范式对理解模型窃取很重要。  
   用途：适合补足“为何黑盒也能学到边界”的历史背景。

5. **Tramèr et al., Stealing Machine Learning Models via Prediction APIs**  
   链接：https://www.usenix.org/conference/usenixsecurity16/technical-sessions/presentation/tramer  
   核心点：系统展示通过预测 API 窃取模型的可行性，是该方向的代表性工作。  
   用途：适合深入理解 API 设计、返回粒度与攻击成本之间的关系。
