## 核心结论

MMBench 是一个面向多模态模型的选择题评测基准。多模态模型，指同时处理图片、文字等不同输入形式的模型。它的价值不在于给模型一个简单“总分”，而在于把能力拆成可定位的细粒度维度，帮助判断模型到底弱在感知、理解还是推理。

如果一个模型总分不低，但一换选项顺序就经常答错，那它不是真正稳定地“会做题”，而是可能只记住了选项位置。MMBench 就是专门抓这种问题。

| 指标 | 说明 | 读法 |
|---|---|---|
| 总分 | 所有题目的整体准确率 | 用来快速比较模型整体水平 |
| `L1` | 一级能力大类，例如感知、推理 | 判断模型弱在大方向 |
| `L2` | 二级能力类别，例如属性识别、关系推理 | 判断弱点属于哪类任务 |
| `L3` | 三级细粒度任务，例如颜色识别、空间关系、OCR | 定位具体短板 |
| `CircularEval` | 循环移动选项后多轮评测 | 检查模型是否受选项位置影响 |

阅读 MMBench 结果时，合理顺序是：先看总分，确认模型是否处在可用候选范围；再看 `L1`，判断大类能力差异；然后看 `L2/L3`，定位具体短板；最后检查 `CircularEval` 下的稳定性，避免被单轮选择题分数误导。

核心公式是：

$$
s_i=\prod_{r=1}^{N_i}\mathbf{1}(\hat y_{i,r}=y_{i,r}),\quad
\text{Acc}=\frac{1}{Q}\sum_{i=1}^{Q}s_i
$$

其中，$s_i$ 表示第 $i$ 道题是否最终通过，$N_i$ 表示这道题被循环评测的轮数，$\hat y_{i,r}$ 表示模型在第 $r$ 轮的预测答案，$y_{i,r}$ 表示第 $r$ 轮的正确答案，$Q$ 表示题目总数。只有每一轮都答对，这道题才记为 1。

## 问题定义与边界

MMBench 评测的是多模态模型在选择题场景下的视觉-语言能力。视觉-语言能力，指模型把图片信息和文本问题结合起来理解并回答的能力。它不是开放式对话评测，也不是端到端业务上线评测。

它适合做“体检”和短板定位。例如，模型是否能识别图片中的属性，是否能理解图中物体之间的空间关系，是否能根据图片和问题做简单推理。它不适合直接回答“这个模型上线后业务是否稳定”，因为真实业务还包含长上下文、多轮追问、工具调用、异常输入、延迟、成本和安全策略。

| 能回答的问题 | 不能回答的问题 |
|---|---|
| 模型在哪类视觉-语言能力上更强 | 上线后真实业务是否稳定 |
| 是否存在位置偏差 | 是否能处理复杂交互流程 |
| 哪些子维度明显偏弱 | 延迟、成本、工具调用能力 |
| 选择题回答是否稳定 | 真实用户是否满意 |
| 图文基础理解是否过关 | 复杂业务链路是否闭环 |

MMBench 的能力划分可以按三层理解：

| 层级 | 含义 | 示例 |
|---|---|---|
| `L1` | 大类能力 | 感知、推理 |
| `L2` | 中类能力 | 属性识别、关系推理、逻辑推理 |
| `L3` | 细粒度任务 | 颜色识别、位置判断、OCR、跨实例关系 |

玩具例子：给模型一张桌子图片，问“杯子在书的左边还是右边”。如果模型答对，说明它可能具备基础空间关系判断能力。空间关系，指物体之间的上下、左右、前后、远近等位置关系。但这个结果不能说明模型能处理所有复杂图片，因为图片可能更模糊、物体更多、问题更绕。

真实工程例子：你在做图文客服系统，用户会上传票据、表单、截图。模型在 MMBench 上 OCR 和关系推理分数不错，只能说明它可能会读图中文字，并且可能会找出图中元素之间的关系。OCR，指从图片中识别文字的能力。但如果要处理真实票据，还要看它在模糊图片、多人对话、异常版式、长链路流程里的表现。MMBench 本身不能覆盖这些业务变量。

因此，MMBench 更像预筛工具。它能告诉你哪些模型值得进入下一轮业务验证，不能替代真实数据集、人工抽检和线上灰度。

## 核心机制与推导

MMBench 的核心机制是把同一道选择题做循环移位，检查模型是否对选项位置敏感。位置偏差，指模型并不真正理解题目，而是倾向于固定选择某些字母，比如总喜欢选 A 或 B。

普通选择题只问一次。如果模型碰巧选对，很难判断它是理解了题目，还是被选项位置、训练分布、提示词格式影响。`CircularEval` 的做法是：同一道题不只测一次，而是把选项顺序循环移动多次，每次重新推理。最终是否通过，不看某一轮是否猜对，而看每一轮是否都答对。

一道题有 4 个选项，正确答案语义上是 `B`。轮换后，四轮正确字母分别是 `B, C, D, A`。如果模型四轮输出是 `B, C, D, B`，最后一轮错了，这题记为 0。新手可以这样理解：不是看你蒙中一次，而是看你换了位置以后还能不能次次选对。

公式为：

$$
s_i=\prod_{r=1}^{N_i}\mathbf{1}(\hat y_{i,r}=y_{i,r})
$$

逐项解释如下：

| 符号 | 含义 |
|---|---|
| $i$ | 第几道题 |
| $N_i$ | 第 $i$ 道题的循环轮数，通常等于选项数 |
| $r$ | 当前是第几轮评测 |
| $\hat y_{i,r}$ | 模型在第 $r$ 轮输出的答案字母 |
| $y_{i,r}$ | 第 $r$ 轮的正确答案字母 |
| $\mathbf{1}(\cdot)$ | 指示函数，条件成立为 1，否则为 0 |
| $s_i$ | 第 $i$ 道题最终得分 |

因为这里使用乘法，只要某一轮答错，就会出现一个 0，整道题的 $s_i$ 就变成 0。只有所有轮次都答对，乘积才是 1。

伪流程如下：

1. 读取题目与选项
2. 对选项做循环移位
3. 每轮推理一次
4. 抽取答案字母
5. 比较每轮是否命中
6. 汇总为题目得分

| 现象 | 可能原因 |
|---|---|
| 总喜欢选 A | 位置偏差 |
| 文本回答很长但没抽出字母 | 答案抽取失败 |
| 某些轮次对某些轮次错 | 理解不稳定或受选项位置影响 |
| 原题答对，换顺序后答错 | 可能记住了局部模式 |
| 复杂关系题持续错误 | 推理能力不足 |

答案抽取器也是关键组件。答案抽取器，指把模型的自由文本输出转成标准选项字母的规则或模型。例如模型输出“我认为答案是 B，因为图中红色物体在左侧”，评测系统不能简单拿整句话和 `B` 比较，而应该抽取出 `B`。否则模型可能说对了，但系统没识别出来。

## 代码实现

代码实现的重点不是训练模型，而是实现一套稳定的评测流水线：读题、轮换选项、发起推理、抽取答案、统计准确率。真正容易出问题的地方通常不是模型本身，而是 prompt、选项重排、答案解析、日志记录和结果汇总。

新手可以把循环评测理解成：同一题反复换顺序，看看每次都能不能答对。

```python
import re

def rotate_options(options, shift):
    return options[shift:] + options[:shift]

def choice_letters(n):
    return [chr(ord("A") + i) for i in range(n)]

def correct_letter_after_rotation(options, correct_index, shift):
    n = len(options)
    new_index = (correct_index - shift) % n
    return choice_letters(n)[new_index]

def extract_choice(text):
    match = re.search(r"\b([A-D])\b", text.upper())
    return match.group(1) if match else None

def toy_model(question, rotated_options):
    # 玩具模型：根据选项文本找语义正确项，而不是固定选字母。
    for idx, option in enumerate(rotated_options):
        if option == "cat":
            return choice_letters(len(rotated_options))[idx]
    return "A"

def circular_eval_item(item):
    options = item["options"]
    correct_index = item["correct_index"]

    for r in range(len(options)):
        rotated = rotate_options(options, r)
        pred_text = toy_model(item["question"], rotated)
        pred = extract_choice(pred_text)
        gold = correct_letter_after_rotation(options, correct_index, r)
        if pred != gold:
            return 0
    return 1

dataset = [
    {
        "question": "Which option is the animal in the image?",
        "options": ["table", "cat", "car", "cup"],
        "correct_index": 1,
    }
]

score = sum(circular_eval_item(item) for item in dataset)
acc = score / len(dataset)

assert circular_eval_item(dataset[0]) == 1
assert acc == 1.0
```

更接近评测系统的伪代码如下：

```python
for item in dataset:
    ok = 1
    for r in range(len(item["options"])):
        rotated = rotate_options(item["options"], r)
        pred = model_infer(prompt(item["question"], rotated))
        ans = extract_choice(pred)
        if ans != rotated["correct_choice"]:
            ok = 0
            break
    score += ok
acc = score / len(dataset)
```

| 模块 | 作用 | 常见失败点 |
|---|---|---|
| 题目加载 | 读取题目、图片、选项 | 格式不统一 |
| 选项循环移位 | 消除位置偏差 | 正确答案映射错 |
| 模型推理 | 输出答案 | prompt 不稳定 |
| 答案抽取器 | 提取 A/B/C/D | 自由文本解析失败 |
| 结果统计 | 生成总分与分项分数 | 语言混统计 |
| 日志记录 | 保存每轮输入输出 | 无法复盘错误 |

评测配置建议固定。prompt 要固定，温度要固定，模型版本要固定，图片预处理方式要固定。温度，指控制模型输出随机性的参数；评测时通常应设为较低值，减少随机波动。每轮都应保留原始输出，因为后续排查时需要区分三类问题：模型真的不会、答案抽取失败、选项映射写错。

一个简单配置可以包括：

| 配置项 | 建议 |
|---|---|
| prompt | 固定模板，只要求输出选项字母 |
| temperature | 设为 0 或接近 0 |
| max tokens | 足够输出简短解释，但不要过长 |
| raw output | 必须保存 |
| extracted answer | 必须保存 |
| language | 中文、英文分开统计 |

## 工程权衡与常见坑

MMBench 的设计目标是尽量公平地测能力，但公平性和工程复杂度之间有明显权衡。循环评测能减少位置偏差，但会增加推理次数；答案抽取器能减少误判，但也会引入解析规则本身的复杂性。

常见错误不是“模型太差”，而是“统计方式错了”。例如把单轮准确率直接当最终能力，或者把中英结果混在一起，都会让结论失真。

| 坑点 | 后果 | 规避 |
|---|---|---|
| 只看单轮准确率 | 高估真实能力 | 用 `CircularEval` |
| 不用答案抽取器 | 自由文本被误判 | 统一抽取规则 |
| 混合中英分数 | 结果不可比 | 分开统计 |
| 不保留每轮日志 | 难定位问题 | 保存原始输出 |
| 只看总分 | 找不到短板 | 看 `L1/L2/L3` |
| 正确答案映射错误 | 分数整体异常 | 对轮换逻辑写单元测试 |
| prompt 每次变化 | 结果不可复现 | 固定评测模板 |

一个常见误区是：模型某轮输出“我认为答案是 B，因为……”被当成错误。实际上如果没有答案抽取器，它可能是对的，只是格式不是单字母。新手可理解为：模型说对了，但系统没听懂，所以被算错。

建议每轮日志至少记录：

| 字段 | 作用 |
|---|---|
| 题目 ID | 定位具体样本 |
| 原始选项 | 复原题目 |
| 轮换后的选项 | 检查循环移位是否正确 |
| 正确答案 | 检查标签映射 |
| 模型原始输出 | 分析模型真实行为 |
| 抽取后的答案 | 分析答案解析是否成功 |
| 本轮是否命中 | 汇总题目得分 |
| 能力标签 | 统计 `L1/L2/L3` 分数 |

错误来源可以按特征粗分：

| 错误来源 | 特征 |
|---|---|
| 位置偏差 | 轮换后表现波动大 |
| 抽取错误 | 模型回答长句但没被识别 |
| 真不会 | 各轮都不稳定或都错误 |
| 标签映射错误 | 多个模型在同一轮出现异常低分 |
| 输入处理错误 | 图片加载失败或题目文本缺失 |

真实工程例子：你在选一个模型处理电商售后截图。它的 MMBench 总分看起来不错，但细看发现 OCR 和属性识别高，关系推理低。这意味着它可能能读出截图里的订单号、金额、商品名，但在“哪个退款原因对应哪个商品”这类关系判断上容易出错。工程上不能只因为总分高就上线，而要补充业务样本，重点测试订单、金额、商品、原因之间的关联。

## 替代方案与适用边界

MMBench 适合做细粒度能力剖析，但不是唯一选择。如果目标是更广覆盖、更强鲁棒性或更贴近真实业务，需要搭配其他基准。替代方案通常在题型、任务形式、评测代价、可解释性上各有取舍。

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| MMBench | 细粒度、可定位短板 | 主要是选择题 | 能力体检、模型对比 |
| 开放式 VQA 基准 | 更接近真实回答 | 评分更难 | 开放问答 |
| 场景任务集 | 更贴业务 | 成本高、样本少 | 上线前验证 |
| 人工抽检集 | 能发现业务细节问题 | 扩展慢、成本高 | 关键链路验收 |
| 自动化回归集 | 可持续监控版本变化 | 依赖高质量样本 | 模型升级评估 |

VQA，指视觉问答任务，即模型根据图片回答问题。如果你要评估的是“开放式图文问答能力”，MMBench 可能不够，因为它以选择题为主。选择题容易统计，也便于横向比较，但它不能完全模拟真实用户的自由表达。如果你要评估的是“多模态综合推理和真实交互”，可能要搭配其他开放式或场景化基准一起看。

边界可以这样定：

| 目标 | 是否适合只用 MMBench | 原因 |
|---|---|---|
| 选模型做预筛 | 适合 | 能快速比较基础能力和细分短板 |
| 分析模型弱项 | 适合 | `L1/L2/L3` 可以定位能力维度 |
| 决定是否上线 | 不适合 | 缺少真实业务流程验证 |
| 评估开放式长回答 | 不充分 | 选择题无法覆盖完整生成质量 |
| 评估延迟和成本 | 不适合 | 这不是 MMBench 的核心目标 |

什么时候不用 MMBench：如果你的任务完全不是选择题形式，而且业务重点是多轮对话、工具调用、长文档理解、复杂权限逻辑或实时系统延迟，那么 MMBench 只能作为旁证，不能作为主评测。此时应优先构造业务数据集，并用真实输入、真实输出格式、真实错误成本来评估。

更稳妥的流程是：先用 MMBench 做模型预筛，再用开放式 VQA 看自然回答能力，最后用业务场景任务集做上线前验证。这样既能利用公开基准的可比性，也能避免公开分数和真实业务表现脱节。
